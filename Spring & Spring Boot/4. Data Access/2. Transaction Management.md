## Overview

Transaction management is one of the most critical aspects of enterprise applications. Spring Boot provides powerful declarative and programmatic transaction support that abstracts the complexity of managing database transactions. Understanding transactions deeply is essential for building reliable, consistent applications.

**Key Concept**: A transaction is a unit of work that either completely succeeds or completely fails, ensuring data consistency even when errors occur.

---

## 1. Transaction Fundamentals

### ACID Properties

Every transaction must guarantee ACID properties:

#### **Atomicity**

All operations within a transaction succeed or fail together - no partial updates.

```java
@Service
public class BankService {
    
    @Transactional
    public void transferMoney(Long fromAccountId, Long toAccountId, BigDecimal amount) {
        // Both operations succeed or both fail - no partial transfer
        accountRepository.debit(fromAccountId, amount);  // Operation 1
        accountRepository.credit(toAccountId, amount);   // Operation 2
        
        // If Operation 2 fails, Operation 1 is rolled back automatically
    }
}
```

**Without atomicity**: Money could be debited but not credited, causing data inconsistency.

#### **Consistency**

Database moves from one valid state to another valid state, maintaining all constraints.

```java
@Entity
@Table(name = "accounts")
public class Account {
    
    @Column(nullable = false)
    @Min(0) // Business rule: balance cannot be negative
    private BigDecimal balance;
}

@Service
public class BankService {
    
    @Transactional
    public void withdraw(Long accountId, BigDecimal amount) {
        Account account = accountRepository.findById(accountId)
            .orElseThrow(() -> new AccountNotFoundException(accountId));
        
        if (account.getBalance().compareTo(amount) < 0) {
            throw new InsufficientFundsException(); // Prevents inconsistent state
        }
        
        account.setBalance(account.getBalance().subtract(amount));
        // Transaction maintains consistency - balance never goes negative
    }
}
```

#### **Isolation**

Concurrent transactions don't interfere with each other. The isolation level determines how much transactions can see of each other's changes.

```java
// Transaction 1: Reading account balance
@Transactional(isolation = Isolation.READ_COMMITTED)
public BigDecimal getBalance(Long accountId) {
    return accountRepository.findById(accountId).getBalance();
    // Only sees committed changes from other transactions
}

// Transaction 2: Updating same account simultaneously
@Transactional
public void deposit(Long accountId, BigDecimal amount) {
    Account account = accountRepository.findById(accountId).orElseThrow();
    account.setBalance(account.getBalance().add(amount));
}
```

#### **Durability**

Once committed, changes persist even after system crashes.

```java
@Transactional
public void createOrder(Order order) {
    orderRepository.save(order);
    // After this method completes successfully,
    // the order is permanently saved even if server crashes immediately after
}
```

### Isolation Levels (Critical Understanding)

Isolation levels control what phenomena can occur when transactions run concurrently.

#### **Concurrency Phenomena**

**Dirty Read**: Reading uncommitted changes from another transaction

```java
// Transaction A (isolation = READ_UNCOMMITTED)
@Transactional(isolation = Isolation.READ_UNCOMMITTED)
public BigDecimal calculateTotalBalance() {
    // Might read balance = $1000 from Transaction B (uncommitted)
    return accountRepository.sumAllBalances();
}

// Transaction B (simultaneously)
@Transactional
public void deposit(Long accountId, BigDecimal amount) {
    Account account = accountRepository.findById(accountId).orElseThrow();
    account.setBalance(account.getBalance().add(amount)); // Balance now $1000 (uncommitted)
    // If this transaction rolls back, Transaction A read incorrect data!
}
```

**Non-Repeatable Read**: Reading same row twice gets different values

```java
// Transaction A (isolation = READ_COMMITTED)
@Transactional(isolation = Isolation.READ_COMMITTED)
public void processAccount(Long accountId) {
    Account account1 = accountRepository.findById(accountId).orElseThrow();
    BigDecimal balance1 = account1.getBalance(); // Read: $500
    
    // Transaction B commits a change here
    
    Account account2 = accountRepository.findById(accountId).orElseThrow();
    BigDecimal balance2 = account2.getBalance(); // Read: $1000 (different!)
    
    // balance1 != balance2 within same transaction
}
```

**Phantom Read**: Same query returns different rows

```java
// Transaction A (isolation = REPEATABLE_READ)
@Transactional(isolation = Isolation.REPEATABLE_READ)
public void calculateStatistics() {
    List<Account> accounts1 = accountRepository.findAll(); // Returns 10 accounts
    int count1 = accounts1.size(); // 10
    
    // Transaction B inserts a new account and commits
    
    List<Account> accounts2 = accountRepository.findAll(); // Returns 11 accounts
    int count2 = accounts2.size(); // 11 (phantom row appeared!)
}
```

#### **Isolation Level Comparison**

|Isolation Level|Dirty Read|Non-Repeatable Read|Phantom Read|Performance|
|---|---|---|---|---|
|`READ_UNCOMMITTED`|✓ Possible|✓ Possible|✓ Possible|Fastest|
|`READ_COMMITTED`|✗ Prevented|✓ Possible|✓ Possible|Fast|
|`REPEATABLE_READ`|✗ Prevented|✗ Prevented|✓ Possible|Slower|
|`SERIALIZABLE`|✗ Prevented|✗ Prevented|✗ Prevented|Slowest|

#### **READ_UNCOMMITTED** (Rarely Used)

```java
@Transactional(isolation = Isolation.READ_UNCOMMITTED)
public List<Order> getDashboardStats() {
    // Can read uncommitted changes
    // Use only for non-critical data like dashboards, reports
    // NEVER use for financial or critical data
    return orderRepository.findAll();
}
```

**Use Case**: Non-critical reporting where performance matters more than accuracy.

#### **READ_COMMITTED** (Default in Most Databases)

```java
@Transactional(isolation = Isolation.READ_COMMITTED)
public Order getOrder(Long orderId) {
    // Reads only committed data
    // Most common isolation level
    // Good balance of performance and consistency
    return orderRepository.findById(orderId).orElseThrow();
}
```

**Use Case**: Default choice for most applications. Prevents dirty reads.

#### **REPEATABLE_READ** (MySQL Default)

```java
@Transactional(isolation = Isolation.REPEATABLE_READ)
public void processPayment(Long orderId) {
    Order order = orderRepository.findById(orderId).orElseThrow();
    BigDecimal amount1 = order.getAmount(); // Read once
    
    // Business logic...
    
    order = orderRepository.findById(orderId).orElseThrow();
    BigDecimal amount2 = order.getAmount(); // Same value guaranteed
    
    // amount1 == amount2 (repeatable read)
}
```

**Use Case**: When you need consistent reads within a transaction. Financial calculations.

#### **SERIALIZABLE** (Strictest)

```java
@Transactional(isolation = Isolation.SERIALIZABLE)
public void allocateInventory(Long productId, int quantity) {
    // Transactions execute as if serial (one after another)
    // Prevents all concurrency phenomena
    // Highest consistency, lowest performance
    Product product = productRepository.findById(productId).orElseThrow();
    
    if (product.getStock() >= quantity) {
        product.setStock(product.getStock() - quantity);
        productRepository.save(product);
    }
}
```

**Use Case**: Critical operations requiring absolute consistency (inventory management, seat booking).

**Real-World Example - Concert Ticket Booking**:

```java
@Service
public class TicketService {
    
    // ✗ BAD: READ_COMMITTED allows overselling
    @Transactional(isolation = Isolation.READ_COMMITTED)
    public void bookTicketBad(Long eventId) {
        Event event = eventRepository.findById(eventId).orElseThrow();
        
        if (event.getAvailableSeats() > 0) {
            // Another transaction might see available seats too!
            event.setAvailableSeats(event.getAvailableSeats() - 1);
        }
    }
    
    // ✓ GOOD: SERIALIZABLE prevents overselling
    @Transactional(isolation = Isolation.SERIALIZABLE)
    public void bookTicketGood(Long eventId) {
        Event event = eventRepository.findById(eventId).orElseThrow();
        
        if (event.getAvailableSeats() > 0) {
            event.setAvailableSeats(event.getAvailableSeats() - 1);
            // No other transaction can interfere
        }
    }
}
```

### Propagation Levels

Propagation defines how transactions relate to each other when methods call each other.

#### **REQUIRED** (Default)

Joins existing transaction or creates new one.

```java
@Service
public class OrderService {
    
    @Autowired
    private PaymentService paymentService;
    
    @Transactional(propagation = Propagation.REQUIRED)
    public void createOrder(Order order) {
        orderRepository.save(order); // Part of Transaction A
        
        // Uses same Transaction A
        paymentService.processPayment(order); 
        
        // If processPayment fails, entire transaction rolls back
    }
}

@Service
public class PaymentService {
    
    @Transactional(propagation = Propagation.REQUIRED) // Default
    public void processPayment(Order order) {
        // Joins existing transaction from createOrder
        paymentRepository.save(new Payment(order));
    }
}
```

**Use Case**: Default choice. Most business operations should use REQUIRED.

#### **REQUIRES_NEW**

Always creates a new transaction, suspending existing one.

```java
@Service
public class OrderService {
    
    @Autowired
    private AuditService auditService;
    
    @Transactional
    public void createOrder(Order order) {
        orderRepository.save(order); // Transaction A
        
        // Creates new Transaction B (independent)
        auditService.logOrderCreation(order);
        
        // If this fails, Transaction B still commits (audit saved)
        throw new RuntimeException("Order processing failed");
    }
}

@Service
public class AuditService {
    
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void logOrderCreation(Order order) {
        // Transaction B - independent of parent
        auditRepository.save(new AuditLog("Order created: " + order.getId()));
        // This commits even if parent transaction rolls back
    }
}
```

**Use Case**: Audit logging, sending notifications that should succeed regardless of business transaction outcome.

#### **NESTED**

Creates a nested transaction (savepoint) within existing transaction.

```java
@Service
public class OrderService {
    
    @Autowired
    private LoyaltyService loyaltyService;
    
    @Transactional
    public void createOrder(Order order) {
        orderRepository.save(order); // Main transaction
        
        try {
            // Creates nested transaction (savepoint)
            loyaltyService.addPoints(order.getUserId(), order.getAmount());
        } catch (Exception e) {
            // Nested transaction rolls back, but main continues
            log.warn("Failed to add loyalty points", e);
        }
        
        // Main transaction can still commit
    }
}

@Service
public class LoyaltyService {
    
    @Transactional(propagation = Propagation.NESTED)
    public void addPoints(Long userId, BigDecimal amount) {
        // Nested transaction (savepoint)
        loyaltyRepository.addPoints(userId, calculatePoints(amount));
    }
}
```

**Use Case**: Optional operations within a transaction that shouldn't fail the main operation.

#### **SUPPORTS**

Executes within transaction if exists, otherwise non-transactional.

```java
@Service
public class ReportService {
    
    @Transactional(propagation = Propagation.SUPPORTS)
    public List<Order> getOrders() {
        // If called from transactional method, joins transaction
        // If called from non-transactional method, executes without transaction
        return orderRepository.findAll();
    }
}
```

**Use Case**: Read-only operations that work fine with or without transactions.

#### **NOT_SUPPORTED**

Executes non-transactionally, suspending existing transaction.

```java
@Service
public class EmailService {
    
    @Transactional(propagation = Propagation.NOT_SUPPORTED)
    public void sendEmail(String to, String subject, String body) {
        // Always executes without transaction
        // Even if caller has transaction
        emailClient.send(to, subject, body);
    }
}
```

**Use Case**: External service calls (email, APIs) that shouldn't participate in database transactions.

#### **MANDATORY**

Requires existing transaction, fails otherwise.

```java
@Service
public class PaymentService {
    
    @Transactional(propagation = Propagation.MANDATORY)
    public void processPayment(Order order) {
        // Must be called from within an existing transaction
        // Throws exception if no transaction exists
        paymentRepository.save(new Payment(order));
    }
}
```

**Use Case**: Internal service methods that should never be called outside a transaction.

#### **NEVER**

Must execute non-transactionally, fails if transaction exists.

```java
@Service
public class CacheService {
    
    @Transactional(propagation = Propagation.NEVER)
    public void invalidateCache(String key) {
        // Must not be called within transaction
        // Throws exception if transaction exists
        cacheManager.evict(key);
    }
}
```

**Use Case**: Operations that explicitly should not run in transactions.

### Read-Only Transactions

Optimize read-only operations.

```java
@Service
public class ProductService {
    
    // ✓ GOOD: Marks transaction as read-only
    @Transactional(readOnly = true)
    public List<Product> getAllProducts() {
        // Hibernate won't flush changes
        // Database can optimize (no write locks)
        // Better performance for reads
        return productRepository.findAll();
    }
    
    // ✗ BAD: Attempting write in read-only transaction
    @Transactional(readOnly = true)
    public void updateProductBad(Product product) {
        productRepository.save(product);
        // Throws exception or silently fails (depending on configuration)
    }
}
```

**Benefits**:

- Hibernate skips dirty checking (performance boost)
- Database can apply read optimizations
- Clearly documents method intent

## 2. @Transactional Deep Dive

### How It Works (Proxy-Based AOP)

Spring creates a proxy around your bean that intercepts method calls.

```java
// Your service
@Service
public class UserService {
    
    @Transactional
    public void createUser(User user) {
        userRepository.save(user);
    }
}

// What Spring actually creates (conceptually)
public class UserServiceProxy extends UserService {
    
    private UserService target;
    private PlatformTransactionManager transactionManager;
    
    @Override
    public void createUser(User user) {
        TransactionStatus status = transactionManager.getTransaction(definition);
        try {
            target.createUser(user); // Call actual method
            transactionManager.commit(status);
        } catch (Exception e) {
            transactionManager.rollback(status);
            throw e;
        }
    }
}
```

**Key Insight**: The proxy intercepts the call, starts a transaction, delegates to actual method, then commits or rolls back.

---

### Class-Level vs Method-Level

```java
// Class-level: Default for all methods
@Service
@Transactional(readOnly = true) // Default for all methods
public class ProductService {
    
    // Inherits @Transactional(readOnly = true)
    public Product getProduct(Long id) {
        return productRepository.findById(id).orElseThrow();
    }
    
    // Method-level overrides class-level
    @Transactional(readOnly = false) // Override for this method
    public Product createProduct(Product product) {
        return productRepository.save(product);
    }
}
```

**Best Practice**:

- Class-level for read-only services (repositories, queries)
- Method-level overrides for writes
- Explicit is better than implicit for critical methods

---

### rollbackFor and noRollbackFor

By default, Spring rolls back only on **unchecked exceptions** (RuntimeException and Error).

```java
@Service
public class PaymentService {
    
    // ✗ BAD: Checked exception doesn't trigger rollback by default
    @Transactional
    public void processPaymentBad(Order order) throws PaymentException {
        paymentRepository.save(new Payment(order));
        
        if (insufficientFunds) {
            throw new PaymentException("Insufficient funds");
            // Transaction COMMITS (PaymentException is checked)
        }
    }
    
    // ✓ GOOD: Explicitly rollback on checked exception
    @Transactional(rollbackFor = PaymentException.class)
    public void processPaymentGood(Order order) throws PaymentException {
        paymentRepository.save(new Payment(order));
        
        if (insufficientFunds) {
            throw new PaymentException("Insufficient funds");
            // Transaction ROLLS BACK
        }
    }
    
    // Advanced: Don't rollback on specific exceptions
    @Transactional(noRollbackFor = TemporaryNetworkException.class)
    public void processWithRetry(Order order) {
        try {
            externalService.charge(order);
        } catch (TemporaryNetworkException e) {
            // Transaction still commits (will retry later)
            retryQueue.add(order);
        }
    }
}
```

**When to use rollbackFor**:

- Always specify for checked exceptions
- Critical business exceptions that should rollback

**When to use noRollbackFor**:

- Temporary failures (network issues)
- Validation exceptions that shouldn't rollback partial work

### timeout Attribute

Prevents long-running transactions from locking resources.

```java
@Service
public class ReportService {
    
    // ✓ GOOD: Set timeout for long operations
    @Transactional(timeout = 30) // 30 seconds
    public Report generateReport(ReportRequest request) {
        // If this takes > 30 seconds, transaction rolls back
        return reportGenerator.generate(request);
    }
    
    // Real-world example
    @Transactional(
        timeout = 60,
        isolation = Isolation.READ_COMMITTED,
        rollbackFor = Exception.class
    )
    public void processBatchOrders(List<Order> orders) {
        // Complex batch processing with time limit
        orders.forEach(order -> {
            orderRepository.save(order);
            inventoryService.updateStock(order);
        });
    }
}
```

**Best Practice**: Always set timeout for potentially long operations to prevent resource exhaustion.

### Self-Invocation Problem (Critical!)

**@Transactional doesn't work when calling methods within the same class!**

```java
@Service
public class OrderService {
    
    // ✗ BAD: Transaction not applied!
    public void createOrderBad(Order order) {
        // Direct call bypasses proxy - NO TRANSACTION!
        this.saveOrder(order);
    }
    
    @Transactional
    private void saveOrder(Order order) {
        // Never executed in transaction when called from createOrderBad
        orderRepository.save(order);
    }
}
```

**Why?** The proxy only intercepts calls from **other beans**, not internal calls.

```java
// What happens
OrderService service = context.getBean(OrderService.class);
// service is actually OrderServiceProxy

service.createOrderBad(order);
    // Inside createOrderBad:
    this.saveOrder(order); // 'this' = real OrderService, not proxy!
```

**Solutions**:

#### **Solution 1: Inject Self (Spring 4.3+)**

```java
@Service
public class OrderService {
    
    @Autowired
    private OrderService self; // Inject proxy of itself
    
    public void createOrder(Order order) {
        // ✓ GOOD: Calls through proxy
        self.saveOrder(order);
    }
    
    @Transactional
    public void saveOrder(Order order) {
        orderRepository.save(order);
    }
}
```

#### **Solution 2: Extract to Separate Service**

```java
@Service
public class OrderService {
    
    @Autowired
    private OrderPersistenceService persistenceService;
    
    public void createOrder(Order order) {
        // ✓ GOOD: Calls different bean through proxy
        persistenceService.saveOrder(order);
    }
}

@Service
public class OrderPersistenceService {
    
    @Transactional
    public void saveOrder(Order order) {
        orderRepository.save(order);
    }
}
```

#### **Solution 3: AspectJ (Compile-Time Weaving)**

```java
// Enable AspectJ mode
@Configuration
@EnableTransactionManagement(mode = AdviceMode.ASPECTJ)
public class TransactionConfig {
}

@Service
public class OrderService {
    
    public void createOrder(Order order) {
        // Works with AspectJ - no proxy needed
        this.saveOrder(order);
    }
    
    @Transactional
    private void saveOrder(Order order) {
        orderRepository.save(order);
    }
}
```

**Best Practice**: Use Solution 2 (separate service) for clarity and maintainability.

## 3. TransactionTemplate (Programmatic Transactions)

Programmatic control when declarative @Transactional isn't sufficient.

### Basic Usage

```java
@Service
public class PaymentService {
    
    @Autowired
    private TransactionTemplate transactionTemplate;
    
    @Autowired
    private PaymentRepository paymentRepository;
    
    public Payment processPayment(Order order) {
        // Execute within transaction
        return transactionTemplate.execute(status -> {
            Payment payment = new Payment(order);
            paymentRepository.save(payment);
            
            if (paymentFailed) {
                status.setRollbackOnly(); // Manual rollback
            }
            
            return payment;
        });
    }
}
```

### Advanced Configuration

```java
@Configuration
public class TransactionConfig {
    
    @Bean
    public TransactionTemplate readOnlyTransactionTemplate(
            PlatformTransactionManager transactionManager) {
        
        TransactionTemplate template = new TransactionTemplate(transactionManager);
        template.setReadOnly(true);
        template.setIsolationLevel(TransactionDefinition.ISOLATION_READ_COMMITTED);
        template.setTimeout(30);
        return template;
    }
    
    @Bean
    public TransactionTemplate writeTransactionTemplate(
            PlatformTransactionManager transactionManager) {
        
        TransactionTemplate template = new TransactionTemplate(transactionManager);
        template.setReadOnly(false);
        template.setPropagationBehavior(TransactionDefinition.PROPAGATION_REQUIRED);
        template.setIsolationLevel(TransactionDefinition.ISOLATION_SERIALIZABLE);
        return template;
    }
}

@Service
public class OrderService {
    
    @Autowired
    @Qualifier("writeTransactionTemplate")
    private TransactionTemplate writeTemplate;
    
    @Autowired
    @Qualifier("readOnlyTransactionTemplate")
    private TransactionTemplate readTemplate;
    
    public List<Order> getOrders() {
        return readTemplate.execute(status -> orderRepository.findAll());
    }
    
    public Order createOrder(Order order) {
        return writeTemplate.execute(status -> {
            return orderRepository.save(order);
        });
    }
}
```

### When to Use Programmatic vs Declarative

#### **Use @Transactional (Declarative) When:**

- Simple, straightforward transaction boundaries
- Standard CRUD operations
- Entire method should be transactional
- You want clean, readable code

```java
// ✓ GOOD: Simple, clean
@Transactional
public Order createOrder(Order order) {
    return orderRepository.save(order);
}
```

#### **Use TransactionTemplate (Programmatic) When:**

- Conditional transaction logic
- Multiple transaction boundaries in one method
- Fine-grained control over commit/rollback
- Mixing transactional and non-transactional code

```java
// ✓ GOOD: Complex logic with multiple transaction boundaries
public void processOrders(List<Order> orders) {
    
    // Non-transactional validation
    List<Order> validOrders = orders.stream()
        .filter(this::validate)
        .collect(Collectors.toList());
    
    // Transactional processing (each order separately)
    validOrders.forEach(order -> {
        transactionTemplate.execute(status -> {
            try {
                orderRepository.save(order);
                inventoryService.updateStock(order);
                return null;
            } catch (Exception e) {
                status.setRollbackOnly();
                errorService.logError(order, e); // Non-transactional
                return null;
            }
        });
    });
}
```

## 4. Multiple Transaction Managers

When working with multiple databases or multiple data sources.

### Configuration

```java
@Configuration
@EnableTransactionManagement
public class MultipleDataSourceConfig {
    
    // Primary Database (Orders)
    @Bean
    @Primary
    @ConfigurationProperties("spring.datasource.orders")
    public DataSource ordersDataSource() {
        return DataSourceBuilder.create().build();
    }
    
    @Bean
    @Primary
    public LocalContainerEntityManagerFactoryBean ordersEntityManagerFactory(
            @Qualifier("ordersDataSource") DataSource dataSource) {
        
        LocalContainerEntityManagerFactoryBean em = new LocalContainerEntityManagerFactoryBean();
        em.setDataSource(dataSource);
        em.setPackagesToScan("com.example.orders.entity");
        em.setJpaVendorAdapter(new HibernateJpaVendorAdapter());
        return em;
    }
    
    @Bean
    @Primary
    public PlatformTransactionManager ordersTransactionManager(
            @Qualifier("ordersEntityManagerFactory") EntityManagerFactory emf) {
        return new JpaTransactionManager(emf);
    }
    
    // Secondary Database (Analytics)
    @Bean
    @ConfigurationProperties("spring.datasource.analytics")
    public DataSource analyticsDataSource() {
        return DataSourceBuilder.create().build();
    }
    
    @Bean
    public LocalContainerEntityManagerFactoryBean analyticsEntityManagerFactory(
            @Qualifier("analyticsDataSource") DataSource dataSource) {
        
        LocalContainerEntityManagerFactoryBean em = new LocalContainerEntityManagerFactoryBean();
        em.setDataSource(dataSource);
        em.setPackagesToScan("com.example.analytics.entity");
        em.setJpaVendorAdapter(new HibernateJpaVendorAdapter());
        return em;
    }
    
    @Bean
    public PlatformTransactionManager analyticsTransactionManager(
            @Qualifier("analyticsEntityManagerFactory") EntityManagerFactory emf) {
        return new JpaTransactionManager(emf);
    }
}
```

### Using Specific Transaction Manager

```java
@Service
public class OrderService {
    
    // Uses primary (ordersTransactionManager)
    @Transactional
    public void createOrder(Order order) {
        orderRepository.save(order);
    }
    
    // Uses specific transaction manager
    @Transactional(transactionManager = "analyticsTransactionManager")
    public void logAnalytics(OrderEvent event) {
        analyticsRepository.save(event);
    }
    
    // Uses both transaction managers (WARNING: Not atomic across databases!)
    public void createOrderWithAnalytics(Order order) {
        // Transaction 1: Orders database
        transactional(ordersTransactionManager).execute(() -> {
            orderRepository.save(order);
        });
        
        // Transaction 2: Analytics database (independent!)
        transactional(analyticsTransactionManager).execute(() -> {
            analyticsRepository.save(new OrderEvent(order));
        });
        
        // Problem: If Transaction 2 fails, Transaction 1 already committed!
    }
}
```

### ChainedTransactionManager (Deprecated)

**Note**: `ChainedTransactionManager` is deprecated as of Spring 5.x because it doesn't provide true atomic commits across multiple resources.

```java
// ✗ DEPRECATED: ChainedTransactionManager
@Bean
public ChainedTransactionManager transactionManager(
        @Qualifier("ordersTransactionManager") PlatformTransactionManager orders,
        @Qualifier("analyticsTransactionManager") PlatformTransactionManager analytics) {
    
    // Commits in reverse order: analytics first, then orders
    return new ChainedTransactionManager(orders, analytics);
}

// Problem: Not truly atomic!
// If orders commits but analytics fails, orders is already committed
```

**Best Practice**: Use distributed transactions (XA) or Saga pattern instead.

## 5. Distributed Transactions

### XA Transactions (Two-Phase Commit)

True atomic transactions across multiple resources (databases, message queues).

#### **Configuration**

```java
@Configuration
public class XATransactionConfig {
    
    @Bean
    public DataSource ordersDataSource() {
        MysqlXADataSource xaDataSource = new MysqlXADataSource();
        xaDataSource.setUrl("jdbc:mysql://localhost/orders");
        
        AtomikosDataSourceBean atomikosDataSource = new AtomikosDataSourceBean();
        atomikosDataSource.setXaDataSource(xaDataSource);
        atomikosDataSource.setUniqueResourceName("ordersDB");
        return atomikosDataSource;
    }
    
    @Bean
    public DataSource inventoryDataSource() {
        MysqlXADataSource xaDataSource = new MysqlXADataSource();
        xaDataSource.setUrl("jdbc:mysql://localhost/inventory");
        
        AtomikosDataSourceBean atomikosDataSource = new AtomikosDataSourceBean();
        atomikosDataSource.setXaDataSource(xaDataSource);
        atomikosDataSource.setUniqueResourceName("inventoryDB");
        return atomikosDataSource;
    }
    
    @Bean
    public PlatformTransactionManager transactionManager() {
        return new JtaTransactionManager();
    }
}
```

#### **Usage**

```java
@Service
public class OrderService {
    
    @Transactional
    public void createOrderWithInventory(Order order) {
        // Both operations are truly atomic via XA protocol
        orderRepository.save(order); // Database 1
        inventoryRepository.decrementStock(order.getProductId()); // Database 2
        
        // If either fails, both rollback atomically
    }
}
```

**Pros**:

- True ACID across multiple databases
- Simple programming model

**Cons**:

- Performance overhead (two-phase commit protocol)
- All resources must support XA
- Potential for blocking if coordinator fails
- Not suitable for microservices

### Saga Pattern (Compensating Transactions)

For microservices where XA is impractical. Each service manages its own local transaction and provides compensating actions for rollback.

#### **Choreography-Based Saga**

Services communicate via events.

```java
// Order Service
@Service
public class OrderService {
    
    @Autowired
    private ApplicationEventPublisher eventPublisher;
    
    @Transactional
    public void createOrder(Order order) {
        // Step 1: Create order (local transaction)
        order.setStatus(OrderStatus.PENDING);
        orderRepository.save(order);
        
        // Publish event
        eventPublisher.publishEvent(new OrderCreatedEvent(order));
    }
    
    // Compensation if payment fails
    @EventListener
    @Transactional
    public void handlePaymentFailed(PaymentFailedEvent event) {
        Order order = orderRepository.findById(event.getOrderId()).orElseThrow();
        order.setStatus(OrderStatus.CANCELLED);
        orderRepository.save(order);
    }
}

// Payment Service
@Service
public class PaymentService {
    
    @Autowired
    private ApplicationEventPublisher eventPublisher;
    
    @EventListener
    @Transactional
    public void handleOrderCreated(OrderCreatedEvent event) {
        try {
            // Step 2: Process payment (local transaction)
            Payment payment = new Payment(event.getOrder());
            paymentRepository.save(payment);
            
            eventPublisher.publishEvent(new PaymentSuccessEvent(event.getOrder()));
        } catch (Exception e) {
            eventPublisher.publishEvent(new PaymentFailedEvent(event.getOrderId()));
        }
    }
}

// Inventory Service
@Service
public class InventoryService {
    
    @EventListener
    @Transactional
    public void handlePaymentSuccess(PaymentSuccessEvent event) {
        // Step 3: Reserve inventory (local transaction)
        inventoryRepository.reserve(event.getOrder().getProductId());
    }
    
    // Compensation if order cancelled
    @EventListener
    @Transactional
    public void handleOrderCancelled(OrderCancelledEvent event) {
        inventoryRepository.release(event.getProductId());
    }
}
```

#### **Orchestration-Based Saga**

Central orchestrator coordinates the saga.

```java
@Service
public class OrderSagaOrchestrator {
    
    @Autowired
    private OrderService orderService;
    
    @Autowired
    private PaymentService paymentService;
    
    @Autowired
    private InventoryService inventoryService;
    
    public void executeOrderSaga(OrderRequest request) {
        Order order = null;
        Payment payment = null;
        boolean inventoryReserved = false;
        
        try {
            // Step 1: Create order
            order = orderService.createOrder(request);
            
            // Step 2: Process payment
            payment = paymentService.processPayment(order);
            
            // Step 3: Reserve inventory
            inventoryService.reserveInventory(order);
            inventoryReserved = true;
            
            // Success - commit all
            orderService.confirmOrder(order.getId());
            
        } catch (Exception e) {
            // Compensation (reverse operations)
            if (inventoryReserved) {
                inventoryService.releaseInventory(order);
            }
            if (payment != null) {
                paymentService.refundPayment(payment);
            }
            if (order != null) {
                orderService.cancelOrder(order.getId());
            }
            
            throw new SagaFailedException("Order saga failed", e);
        }
    }
}

@Service
public class OrderService {
    
    @Transactional
    public Order createOrder(OrderRequest request) {
        Order order = new Order(request);
        order.setStatus(OrderStatus.PENDING);
        return orderRepository.save(order);
    }
    
    @Transactional
    public void confirmOrder(Long orderId) {
        Order order = orderRepository.findById(orderId).orElseThrow();
        order.setStatus(OrderStatus.CONFIRMED);
        orderRepository.save(order);
    }
    
    @Transactional
    public void cancelOrder(Long orderId) {
        Order order = orderRepository.findById(orderId).orElseThrow();
        order.setStatus(OrderStatus.CANCELLED);
        orderRepository.save(order);
    }
}
```

#### **Saga State Machine (Spring State Machine)**

```java
@Configuration
@EnableStateMachine
public class OrderSagaStateMachine {
    
    @Bean
    public StateMachine<OrderState, OrderEvent> stateMachine() throws Exception {
        StateMachineBuilder.Builder<OrderState, OrderEvent> builder = 
            StateMachineBuilder.builder();
        
        builder.configureStates()
            .withStates()
                .initial(OrderState.CREATED)
                .state(OrderState.PAYMENT_PENDING)
                .state(OrderState.INVENTORY_RESERVED)
                .state(OrderState.COMPLETED)
                .state(OrderState.CANCELLED);
        
        builder.configureTransitions()
            .withExternal()
                .source(OrderState.CREATED)
                .target(OrderState.PAYMENT_PENDING)
                .event(OrderEvent.PAYMENT_INITIATED)
                .action(processPaymentAction())
            .and()
            .withExternal()
                .source(OrderState.PAYMENT_PENDING)
                .target(OrderState.INVENTORY_RESERVED)
                .event(OrderEvent.PAYMENT_SUCCESS)
                .action(reserveInventoryAction())
            .and()
            .withExternal()
                .source(OrderState.PAYMENT_PENDING)
                .target(OrderState.CANCELLED)
                .event(OrderEvent.PAYMENT_FAILED)
                .action(cancelOrderAction());
        
        return builder.build();
    }
    
    @Bean
    public Action<OrderState, OrderEvent> processPaymentAction() {
        return context -> {
            Order order = context.getMessage().getHeaders().get("order", Order.class);
            paymentService.processPayment(order);
        };
    }
}
```

**Pros**:

- Works across microservices
- No distributed transaction coordinator needed
- Scales well

**Cons**:

- Eventually consistent (not immediately)
- Complex to implement and test
- Need compensating transactions for all steps
- Need to handle duplicate messages (idempotency)

## Summary

### Transaction Fundamentals

- ✅ **ACID Properties**: Atomicity, Consistency, Isolation, Durability
- ✅ **Isolation Levels**: READ_UNCOMMITTED, READ_COMMITTED, REPEATABLE_READ, SERIALIZABLE
- ✅ **Propagation Levels**: REQUIRED, REQUIRES_NEW, NESTED, SUPPORTS, NOT_SUPPORTED, MANDATORY, NEVER
- ✅ **Read-Only**: Optimize read operations, prevent writes

### @Transactional Deep Dive

- ✅ **Proxy-Based AOP**: Spring creates proxy to intercept method calls
- ✅ **Class vs Method Level**: Method-level overrides class-level
- ✅ **rollbackFor/noRollbackFor**: Control rollback on checked exceptions
- ✅ **timeout**: Prevent long-running transactions
- ✅ **Self-Invocation Problem**: Direct calls bypass proxy - use self-injection or separate services

### Programmatic Transactions

- ✅ **TransactionTemplate**: Fine-grained control over transactions
- ✅ **When to Use**: Complex logic with multiple transaction boundaries
- ✅ **Configuration**: Custom templates for different isolation levels

### Multiple Transaction Managers

- ✅ **Multiple DataSources**: Separate transaction managers per database
- ✅ **Specify Manager**: `@Transactional(transactionManager = "...")`
- ✅ **ChainedTransactionManager**: Deprecated, not truly atomic

### Distributed Transactions

- ✅ **XA Transactions**: True ACID across multiple databases (two-phase commit)
- ✅ **Saga Pattern**: Choreography or orchestration-based compensating transactions
- ✅ **Trade-offs**: XA (consistent but slow) vs Saga (fast but eventually consistent)

---

## Best Practices

1. **Always specify rollbackFor for checked exceptions**
2. **Use read-only transactions for queries** (performance boost)
3. **Set timeouts for potentially long operations**
4. **Avoid self-invocation** - extract to separate services
5. **Use appropriate isolation level** - balance consistency vs performance
6. **Use Saga pattern for microservices** - avoid distributed XA
7. **Make transactions as short as possible** - don't call external services within transactions
8. **Use programmatic transactions for complex scenarios**
9. **Test transaction behavior** - especially rollback scenarios
10. **Monitor transaction metrics** - deadlocks, timeouts, rollbacks

---

## Common Pitfalls

❌ **Self-invocation without proxy** ❌ **Forgetting rollbackFor on checked exceptions** ❌ **Long-running transactions holding locks** ❌ **Calling external services within transactions** ❌ **Using wrong isolation level (too strict or too lenient)** ❌ **Not handling transaction timeouts** ❌ **Mixing transactional and non-transactional code incorrectly** ❌ **Not testing rollback scenarios**
