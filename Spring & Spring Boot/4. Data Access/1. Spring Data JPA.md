## Overview

Spring Data JPA simplifies data access by providing a repository abstraction over JPA (Java Persistence API). Understanding JPA fundamentals and Spring Data features is crucial for building efficient, maintainable data access layers.

**Key Concept**: JPA is the specification, Hibernate is the most popular implementation, and Spring Data JPA provides a higher-level abstraction.

---

## 1. JPA Fundamentals

### EntityManager and Persistence Context

**EntityManager** - Primary interface for interacting with persistence context.

**Persistence Context** - First-level cache that manages entity instances.

```java
@Service
public class UserService {
    
    @PersistenceContext
    private EntityManager entityManager;
    
    @Transactional
    public void demonstratePersistenceContext() {
        // 1. Create new entity (Transient state)
        User user = new User("John", "john@example.com");
        
        // 2. Persist (Transient → Persistent)
        entityManager.persist(user);
        // Entity is now in persistence context
        // Changes are tracked
        
        // 3. Modify entity
        user.setName("John Doe");
        // No explicit save needed - changes are tracked
        
        // 4. Flush - sync persistence context with database
        entityManager.flush();
        // INSERT and UPDATE statements executed
        
        // 5. Find entity - returns from persistence context (1st level cache)
        User found = entityManager.find(User.class, user.getId());
        System.out.println(found == user);  // true - same instance
        
        // 6. Detach - remove from persistence context
        entityManager.detach(user);
        user.setName("Jane");  // This change will NOT be tracked
        
        // 7. Merge - reattach to persistence context
        User managed = entityManager.merge(user);
        // Now changes are tracked again
        
        // 8. Remove - mark for deletion
        entityManager.remove(managed);
        // DELETE will be executed on flush/commit
        
        // 9. Clear - detach all entities
        entityManager.clear();
    }
}
```

**Persistence Context scope:**

```java
@Service
public class TransactionScopeExample {
    
    @PersistenceContext
    private EntityManager entityManager;
    
    @Transactional  // Persistence context created
    public void transactionalMethod() {
        User user = new User("John", "john@example.com");
        entityManager.persist(user);
        
        // user is in persistence context
        user.setName("John Doe");  // Tracked
        
    } // Transaction commits, persistence context closed
    
    public void nonTransactionalMethod() {
        // Each EntityManager operation creates its own temporary context
        User user = entityManager.find(User.class, 1L);
        user.setName("Jane");  // NOT tracked - no active transaction
        // Changes lost
    }
}
```

### Entity Lifecycle

Entities have four states:

```
┌─────────────┐
│  Transient  │  New object, not in DB, not in context
└──────┬──────┘
       │ persist()
       ▼
┌─────────────┐
│ Persistent  │  In persistence context, changes tracked
└──────┬──────┘
       │ detach() / clear() / close()
       ▼
┌─────────────┐
│  Detached   │  Was in DB, not in context, changes not tracked
└──────┬──────┘
       │ merge()
       ▼
┌─────────────┐
│ Persistent  │  Back in context
└──────┬──────┘
       │ remove()
       ▼
┌─────────────┐
│  Removed    │  Marked for deletion, will be deleted on flush
└─────────────┘
```

**Complete lifecycle example:**

```java
@Service
public class EntityLifecycleDemo {
    
    @PersistenceContext
    private EntityManager entityManager;
    
    @Transactional
    public void demonstrateLifecycle() {
        // 1. TRANSIENT - new object
        User user = new User("John", "john@example.com");
        System.out.println("Contains: " + entityManager.contains(user));  // false
        
        // 2. PERSISTENT - persist()
        entityManager.persist(user);
        System.out.println("Contains: " + entityManager.contains(user));  // true
        System.out.println("ID: " + user.getId());  // ID assigned (if IDENTITY)
        
        // Modifications are tracked
        user.setName("John Doe");
        user.setEmail("johndoe@example.com");
        // No need to call save() or update()
        
        entityManager.flush();  // INSERT and UPDATE executed
        
        // 3. DETACHED - detach()
        entityManager.detach(user);
        System.out.println("Contains: " + entityManager.contains(user));  // false
        
        user.setName("Jane");  // NOT tracked
        
        // 4. PERSISTENT - merge()
        User managed = entityManager.merge(user);
        System.out.println("Contains: " + entityManager.contains(managed));  // true
        System.out.println(user == managed);  // false - different instance
        
        // 5. REMOVED - remove()
        entityManager.remove(managed);
        System.out.println("Contains: " + entityManager.contains(managed));  // false
        
        // DELETE executed on flush/commit
    }
}
```

### First-Level Cache (Persistence Context)

Persistence context acts as a first-level cache.

```java
@Transactional
public void firstLevelCacheDemo() {
    // First access - hits database
    User user1 = entityManager.find(User.class, 1L);
    // SQL: SELECT * FROM users WHERE id = 1
    
    // Second access - from cache (no SQL)
    User user2 = entityManager.find(User.class, 1L);
    
    System.out.println(user1 == user2);  // true - same instance
    
    // Modify entity
    user1.setName("John Doe");
    
    // Both references see the change (same object)
    System.out.println(user2.getName());  // "John Doe"
}
```

**Cache benefits:**

- ✅ Reduces database hits
- ✅ Identity guarantee (same ID = same instance)
- ✅ Automatic dirty checking
- ✅ Write-behind (batching updates)

**Cache scope:**

- ⚠️ Transaction-scoped (cleared after transaction)
- ⚠️ Not shared between transactions
- ⚠️ Memory-bound (can grow large)

### Second-Level Cache (Hibernate)

Shared cache across persistence contexts.

**Enable second-level cache:**

```xml
<!-- pom.xml -->
<dependency>
    <groupId>org.hibernate</groupId>
    <artifactId>hibernate-jcache</artifactId>
</dependency>
<dependency>
    <groupId>org.ehcache</groupId>
    <artifactId>ehcache</artifactId>
</dependency>
```

**application.yml:**

```yaml
spring:
  jpa:
    properties:
      hibernate:
        cache:
          use_second_level_cache: true
          region:
            factory_class: org.hibernate.cache.jcache.JCacheRegionFactory
        javax:
          cache:
            provider: org.ehcache.jsr107.EhcacheCachingProvider
            uri: classpath:ehcache.xml
```

**Mark entities as cacheable:**

```java
@Entity
@Cacheable
@org.hibernate.annotations.Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    private String name;
    private String email;
    
    // Getters/setters...
}
```

**Cache strategies:**

|Strategy|Use Case|Isolation|
|---|---|---|
|`READ_ONLY`|Never updated (reference data)|Highest|
|`NONSTRICT_READ_WRITE`|Rarely updated, eventual consistency OK|Medium|
|`READ_WRITE`|Frequently updated, strict consistency|Low|
|`TRANSACTIONAL`|Requires JTA, strictest consistency|Lowest|

**ehcache.xml:**

```xml
<config xmlns="http://www.ehcache.org/v3">
    <cache alias="com.example.entity.User">
        <expiry>
            <ttl unit="minutes">10</ttl>
        </expiry>
        <heap unit="entries">1000</heap>
    </cache>
    
    <cache alias="com.example.entity.Order">
        <expiry>
            <ttl unit="minutes">5</ttl>
        </expiry>
        <heap unit="entries">500</heap>
    </cache>
</config>
```

**Query cache:**


## 2. Repository Pattern

### Repository Hierarchy

```
Repository<T, ID>
    ↓
CrudRepository<T, ID>
    ↓
PagingAndSortingRepository<T, ID>
    ↓
JpaRepository<T, ID>
```

**Interfaces comparison:**

|Interface|Methods|Use Case|
|---|---|---|
|`Repository`|Marker interface|Custom repositories|
|`CrudRepository`|save, findById, findAll, delete|Basic CRUD|
|`PagingAndSortingRepository`|+ findAll(Pageable), findAll(Sort)|Pagination/sorting|
|`JpaRepository`|+ flush, saveAndFlush, deleteInBatch|JPA-specific|

**Example:**

```java
// Basic CRUD
public interface UserRepository extends CrudRepository<User, Long> {
    // Inherits: save, findById, findAll, count, delete, etc.
}

// With pagination
public interface OrderRepository extends PagingAndSortingRepository<Order, Long> {
    // Inherits: + findAll(Pageable), findAll(Sort)
}

// Full JPA features
public interface ProductRepository extends JpaRepository<Product, Long> {
    // Inherits: + flush, saveAndFlush, deleteInBatch, getOne
}
```

### @Query with JPQL and Native SQL

#### JPQL Queries

```java
public interface UserRepository extends JpaRepository<User, Long> {
    
    // Simple JPQL
    @Query("SELECT u FROM User u WHERE u.email = :email")
    Optional<User> findByEmail(@Param("email") String email);
    
    // Multiple parameters
    @Query("SELECT u FROM User u WHERE u.name = :name AND u.age > :age")
    List<User> findByNameAndAgeGreaterThan(
        @Param("name") String name,
        @Param("age") int age
    );
    
    // JPQL with JOIN
    @Query("SELECT u FROM User u JOIN u.orders o WHERE o.status = :status")
    List<User> findUsersWithOrderStatus(@Param("status") OrderStatus status);
    
    // JPQL with FETCH JOIN (avoid N+1)
    @Query("SELECT u FROM User u JOIN FETCH u.orders WHERE u.id = :id")
    Optional<User> findByIdWithOrders(@Param("id") Long id);
    
    // Aggregate functions
    @Query("SELECT COUNT(u) FROM User u WHERE u.active = true")
    long countActiveUsers();
    
    @Query("SELECT AVG(u.age) FROM User u")
    Double getAverageAge();
    
    // GROUP BY
    @Query("SELECT u.country, COUNT(u) FROM User u GROUP BY u.country")
    List<Object[]> countUsersByCountry();
    
    // Constructor expression (DTO)
    @Query("SELECT new com.example.dto.UserSummary(u.id, u.name, u.email) " +
           "FROM User u WHERE u.active = true")
    List<UserSummary> findActiveUserSummaries();
    
    // Subquery
    @Query("SELECT u FROM User u WHERE u.age > " +
           "(SELECT AVG(u2.age) FROM User u2)")
    List<User> findUsersOlderThanAverage();
    
    // LIKE with parameter
    @Query("SELECT u FROM User u WHERE u.name LIKE %:searchTerm%")
    List<User> searchByName(@Param("searchTerm") String searchTerm);
    
    // IN clause
    @Query("SELECT u FROM User u WHERE u.id IN :ids")
    List<User> findByIds(@Param("ids") List<Long> ids);
    
    // Pagination with JPQL
    @Query("SELECT u FROM User u WHERE u.age > :age")
    Page<User> findByAgeGreaterThan(@Param("age") int age, Pageable pageable);
}
```

### @Modifying Queries

For UPDATE and DELETE operations.

```java
public interface UserRepository extends JpaRepository<User, Long> {
    
    // UPDATE
    @Modifying
    @Transactional
    @Query("UPDATE User u SET u.active = :active WHERE u.id = :id")
    int updateUserStatus(@Param("id") Long id, @Param("active") boolean active);
    
    @Modifying
    @Transactional
    @Query("UPDATE User u SET u.lastLoginDate = :date WHERE u.id = :id")
    void updateLastLoginDate(@Param("id") Long id, @Param("date") LocalDateTime date);
    
    // Bulk UPDATE
    @Modifying
    @Transactional
    @Query("UPDATE User u SET u.active = false WHERE u.lastLoginDate < :date")
    int deactivateInactiveUsers(@Param("date") LocalDateTime date);
    
    // DELETE
    @Modifying
    @Transactional
    @Query("DELETE FROM User u WHERE u.active = false")
    int deleteInactiveUsers();
    
    @Modifying
    @Transactional
    @Query("DELETE FROM Order o WHERE o.createdDate < :date")
    void deleteOldOrders(@Param("date") LocalDate date);
    
    // Native UPDATE
    @Modifying
    @Transactional
    @Query(value = "UPDATE users SET failed_login_attempts = 0 WHERE id = :id",
           nativeQuery = true)
    int resetFailedLoginAttempts(@Param("id") Long id);
    
    // Clear persistence context after modifying query
    @Modifying(clearAutomatically = true)
    @Transactional
    @Query("UPDATE User u SET u.version = u.version + 1")
    void incrementVersions();
}
```

**Important notes:**

- ⚠️ Always use `@Transactional` with `@Modifying`
- ⚠️ `@Modifying` queries bypass persistence context
- ⚠️ Use `clearAutomatically = true` to sync context
- ⚠️ Return type: `int`/`void` for updates, `int` for affected rows

### Specifications (Dynamic Queries)

Build dynamic queries using Criteria API.

**Enable Specifications:**

```java
public interface UserRepository extends JpaRepository<User, Long>, 
                                        JpaSpecificationExecutor<User> {
}
```

**Simple Specification:**

```java
public class UserSpecifications {
    
    public static Specification<User> hasName(String name) {
        return (root, query, cb) -> cb.equal(root.get("name"), name);
    }
    
    public static Specification<User> hasEmail(String email) {
        return (root, query, cb) -> cb.equal(root.get("email"), email);
    }
    
    public static Specification<User> isActive() {
        return (root, query, cb) -> cb.isTrue(root.get("active"));
    }
    
    public static Specification<User> ageGreaterThan(int age) {
        return (root, query, cb) -> cb.greaterThan(root.get("age"), age);
    }
}

// Usage
@Service
public class UserService {
    
    @Autowired
    private UserRepository userRepository;
    
    public List<User> searchUsers(String name, Integer minAge) {
        Specification<User> spec = Specification.where(null);
        
        if (name != null) {
            spec = spec.and(UserSpecifications.hasName(name));
        }
        
        if (minAge != null) {
            spec = spec.and(UserSpecifications.ageGreaterThan(minAge));
        }
        
        spec = spec.and(UserSpecifications.isActive());
        
        return userRepository.findAll(spec);
    }
}
```

**Complex Specifications:**

```java
public class OrderSpecifications {
    
    public static Specification<Order> hasStatus(OrderStatus status) {
        return (root, query, cb) -> cb.equal(root.get("status"), status);
    }
    
    public static Specification<Order> totalGreaterThan(BigDecimal amount) {
        return (root, query, cb) -> cb.greaterThan(root.get("total"), amount);
    }
    
    public static Specification<Order> createdBetween(LocalDate start, LocalDate end) {
        return (root, query, cb) -> cb.between(
            root.get("createdDate"), start, end
        );
    }
    
    public static Specification<Order> hasUserEmail(String email) {
        return (root, query, cb) -> {
            Join<Order, User> userJoin = root.join("user");
            return cb.equal(userJoin.get("email"), email);
        };
    }
    
    public static Specification<Order> hasItemName(String itemName) {
        return (root, query, cb) -> {
            Join<Order, OrderItem> itemsJoin = root.join("items");
            return cb.like(itemsJoin.get("productName"), "%" + itemName + "%");
        };
    }
}

// Dynamic search
public List<Order> searchOrders(OrderSearchCriteria criteria) {
    Specification<Order> spec = Specification.where(null);
    
    if (criteria.getStatus() != null) {
        spec = spec.and(OrderSpecifications.hasStatus(criteria.getStatus()));
    }
    
    if (criteria.getMinTotal() != null) {
        spec = spec.and(OrderSpecifications.totalGreaterThan(criteria.getMinTotal()));
    }
    
    if (criteria.getStartDate() != null && criteria.getEndDate() != null) {
        spec = spec.and(OrderSpecifications.createdBetween(
            criteria.getStartDate(), criteria.getEndDate()
        ));
    }
    
    if (criteria.getUserEmail() != null) {
        spec = spec.and(OrderSpecifications.hasUserEmail(criteria.getUserEmail()));
    }
    
    return orderRepository.findAll(spec, criteria.getPageable());
}
```

```java
@Service
public class ProductService {
    
    @Autowired
    private ProductRepository productRepository;

        Product probe = new Product();

        probe.setCategory(category);
        probe.setActive(active);

        ExampleMatcher matcher = ExampleMatcher.matching()
            .withIgnoreNullValues()
            .withIgnoreCase()
            .withStringMatcher(ExampleMatcher.StringMatcher.CONTAINING);

        return productRepository.findAll(example);
    }

            .withIgnorePaths("id", "createdDate")
            .withMatcher("name", match -> match.contains().ignoreCase())
            .withMatcher("price", match -> match.greaterThan());
        
        Example<Product> example = Example.of(probe, matcher);
        
        return productRepository.findAll(example);
    }
```

### @OneToOne

```java
// Address.java
```

### Cascade Types

Control how operations cascade to related entities.

|Cascade Type|Description|Use Case|
|---|---|---|
|`PERSIST`|Save child when parent is saved|New entities|
|`MERGE`|Update child when parent is merged|Updates|
|`REMOVE`|Delete child when parent is deleted|Dependent entities|
|`REFRESH`|Reload child when parent is refreshed|Sync from DB|
|`DETACH`|Detach child when parent is detached|Rare|
|`ALL`|All of the above|Full lifecycle management|

```java
@Entity
public class User {
    
    // Cascade everything
    @OneToMany(mappedBy = "user", cascade = CascadeType.ALL, orphanRemoval = true)
    private List<Order> orders = new ArrayList<>();
    
    // Only persist and merge (not remove)
    @OneToMany(mappedBy = "user", cascade = {CascadeType.PERSIST, CascadeType.MERGE})
    private List<Address> addresses = new ArrayList<>();
    
    // No cascade
    @ManyToOne(fetch = FetchType.LAZY)
    private Company company;
}
```

**orphanRemoval:**

```java
@Entity
public class Post {
    
    // Delete comments when removed from collection
    @OneToMany(mappedBy = "post", cascade = CascadeType.ALL, orphanRemoval = true)
    private List<Comment> comments = new ArrayList<>();
}

// Usage
Post post = postRepository.findById(1L).orElseThrow();
post.getComments().remove(0);  // Comment is deleted from DB
postRepository.save(post);
```

### Fetch Types (LAZY vs EAGER)

**LAZY** - Load on demand (default for collections) **EAGER** - Load immediately (default for single entities)

```java
@Entity
public class User {
    
    @OneToMany(mappedBy = "user", fetch = FetchType.LAZY)
    private List<Order> orders;  // Loaded when accessed
    
    @ManyToOne(fetch = FetchType.LAZY)
    private Company company;  // Loaded when accessed (recommended!)
    
    @OneToOne(fetch = FetchType.EAGER)
    private Profile profile;  // Loaded immediately
}
```

**Default fetch types:**

|Relationship|Default Fetch|
|---|---|
|`@OneToOne`|EAGER|
|`@ManyToOne`|EAGER|
|`@OneToMany`|LAZY|
|`@ManyToMany`|LAZY|

**Best practice:**

- ✅ Use LAZY for all associations
- ✅ Use fetch joins or entity graphs when needed
- ❌ Avoid EAGER (causes performance issues)

### N+1 Query Problem

**The Problem:**

```java
// Find all users
List<User> users = userRepository.findAll();
// SQL: SELECT * FROM users

// Access orders for each user
for (User user : users) {
    System.out.println(user.getOrders().size());
    // SQL: SELECT * FROM orders WHERE user_id = ?
    // This executes N times (once per user)!
}

// Total queries: 1 + N (N+1 problem)
```

**Solution 1: FETCH JOIN**

```java
public interface UserRepository extends JpaRepository<User, Long> {
    
    @Query("SELECT DISTINCT u FROM User u LEFT JOIN FETCH u.orders")
    List<User> findAllWithOrders();
    // SQL: SELECT * FROM users u LEFT JOIN orders o ON u.id = o.user_id
    // Single query!
}
```

**Solution 2: @EntityGraph**

```java
public interface UserRepository extends JpaRepository<User, Long> {
    
    @EntityGraph(attributePaths = {"orders"})
    List<User> findAll();
    // Single query with LEFT JOIN
    
    @EntityGraph(attributePaths = {"orders", "orders.items"})
    List<User> findAllWithOrdersAndItems();
    // Multiple associations
}
```

**Solution 3: Batch Fetching**

```java
@Entity
public class User {
    
    @OneToMany(mappedBy = "user")
    @BatchSize(size = 10)  // Fetch 10 at a time
    private List<Order> orders;
}

// application.yml
spring:
  jpa:
    properties:
      hibernate:
        default_batch_fetch_size: 10
```

## 5. Locking

### Optimistic Locking (@Version)

Prevents lost updates without holding database locks.

```java
@Entity
public class Product {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    private String name;
    private BigDecimal price;
    
    @Version
    private Long version;  // Auto-incremented on updates
    
    // Getters/setters...
}

// Usage
@Service
public class ProductService {
    
    @Transactional
    public void updatePrice(Long id, BigDecimal newPrice) {
        Product product = productRepository.findById(id).orElseThrow();
        
        product.setPrice(newPrice);
        
        productRepository.save(product);
        // If version changed (concurrent update), throws OptimisticLockException
    }
}
```

**How it works:**

```sql
-- Initial state
SELECT id, name, price, version FROM products WHERE id = 1
-- Returns: id=1, name="Widget", price=10.00, version=0

-- Update
UPDATE products 
SET name = 'Widget', price = 15.00, version = 1 
WHERE id = 1 AND version = 0

-- If version doesn't match (someone else updated), update fails
-- Throws OptimisticLockException
```

---

### Pessimistic Locking

Database-level locks (use sparingly).

```java
public interface ProductRepository extends JpaRepository<Product, Long> {
    
    // Pessimistic read (shared lock)
    @Lock(LockModeType.PESSIMISTIC_READ)
    @Query("SELECT p FROM Product p WHERE p.id = :id")
    Optional<Product> findByIdWithReadLock(@Param("id") Long id);
    
    // Pessimistic write (exclusive lock)
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT p FROM Product p WHERE p.id = :id")
    Optional<Product> findByIdWithWriteLock(@Param("id") Long id);
    
    // Pessimistic force increment (increment version)
    @Lock(LockModeType.PESSIMISTIC_FORCE_INCREMENT)
    Optional<Product> findById(Long id);
}

// Usage
@Transactional
public void reserveInventory(Long productId, int quantity) {
    // Lock product for update
    Product product = productRepository
        .findByIdWithWriteLock(productId)
        .orElseThrow();
    
    if (product.getStock() < quantity) {
        throw new InsufficientStockException();
    }
    
    product.setStock(product.getStock() - quantity);
    productRepository.save(product);
    
    // Lock released when transaction commits
}
```

**Lock modes:**

|Lock Mode|Description|Use Case|
|---|---|---|
|`PESSIMISTIC_READ`|Shared lock|Read consistency|
|`PESSIMISTIC_WRITE`|Exclusive lock|Critical updates|
|`PESSIMISTIC_FORCE_INCREMENT`|Exclusive + version increment|Force concurrency detection|
|`OPTIMISTIC`|Version check|Most cases|
|`OPTIMISTIC_FORCE_INCREMENT`|Force version increment|Ensure new version|

## 7. Custom Repository Implementations

### Fragment Interfaces

Add custom methods to repositories.

**Step 1: Define custom interface**

```java
public interface UserRepositoryCustom {
    List<User> findUsersWithComplexLogic(SearchCriteria criteria);
    void bulkUpdateStatus(List<Long> ids, UserStatus status);
}
```

**Step 2: Implement interface**

```java
@Repository
public class UserRepositoryImpl implements UserRepositoryCustom {
    
    @PersistenceContext
    private EntityManager entityManager;
    
    @Override
    public List<User> findUsersWithComplexLogic(SearchCriteria criteria) {
        CriteriaBuilder cb = entityManager.getCriteriaBuilder();
        CriteriaQuery<User> query = cb.createQuery(User.class);
        Root<User> root = query.from(User.class);
        
        List<Predicate> predicates = new ArrayList<>();
        
        if (criteria.getName() != null) {
            predicates.add(cb.like(root.get("name"), "%" + criteria.getName() + "%"));
        }
        
        if (criteria.getMinAge() != null) {
            predicates.add(cb.greaterThanOrEqualTo(root.get("age"), criteria.getMinAge()));
        }
        
        query.where(predicates.toArray(new Predicate[0]));
        
        return entityManager.createQuery(query).getResultList();
    }
    
    @Override
    @Transactional
    public void bulkUpdateStatus(List<Long> ids, UserStatus status) {
        entityManager.createQuery(
            "UPDATE User u SET u.status = :status WHERE u.id IN :ids"
        )
        .setParameter("status", status)
        .setParameter("ids", ids)
        .executeUpdate();
    }
}
```

**Step 3: Extend in main repository**

```java
public interface UserRepository extends JpaRepository<User, Long>, 
                                        UserRepositoryCustom {
    // Standard Spring Data JPA methods
    // + custom methods from UserRepositoryCustom
}
```

### Custom Base Repository

Create custom base for all repositories.

**Step 1: Define custom base interface**

```java
@NoRepositoryBean
public interface CustomRepository<T, ID> extends JpaRepository<T, ID> {
    Optional<T> findByIdWithLock(ID id);
    List<T> findAllByIds(List<ID> ids);
}
```

**Step 2: Implement base repository**

```java
public class CustomRepositoryImpl<T, ID> 
        extends SimpleJpaRepository<T, ID> 
        implements CustomRepository<T, ID> {
    
    private final EntityManager entityManager;
    
    public CustomRepositoryImpl(JpaEntityInformation<T, ?> entityInformation,
                               EntityManager entityManager) {
        super(entityInformation, entityManager);
        this.entityManager = entityManager;
    }
    
    @Override
    public Optional<T> findByIdWithLock(ID id) {
        return Optional.ofNullable(
            entityManager.find(
                getDomainClass(), 
                id, 
                LockModeType.PESSIMISTIC_WRITE
            )
        );
    }
    
    @Override
    public List<T> findAllByIds(List<ID> ids) {
        CriteriaBuilder cb = entityManager.getCriteriaBuilder();
        CriteriaQuery<T> query = cb.createQuery(getDomainClass());
        Root<T> root = query.from(getDomainClass());
        
        query.where(root.get("id").in(ids));
        
        return entityManager.createQuery(query).getResultList();
    }
}
```

**Step 3: Register custom base**

```java
@Configuration
@EnableJpaRepositories(
    repositoryBaseClass = CustomRepositoryImpl.class
)
public class JpaConfig {
}
```

**Step 4: Use in repositories**

```java
public interface UserRepository extends CustomRepository<User, Long> {
    // Inherits custom methods: findByIdWithLock, findAllByIds
    // Plus all standard JpaRepository methods
}

// Usage
Optional<User> user = userRepository.findByIdWithLock(1L);
```

## Best Practices Summary

### Repository Design ✅

1. Use JpaRepository for most cases
2. Use Specifications for complex dynamic queries
3. Use projections for read-only queries
4. Implement custom repositories for complex logic
5. Follow naming conventions for query methods

### Relationships ✅

1. Use LAZY fetch for all associations
2. Always maintain both sides of bidirectional relationships
3. Use helper methods (addChild/removeChild)
4. Be careful with CascadeType.ALL
5. Use orphanRemoval for dependent entities

### Performance ✅

1. Always watch for N+1 queries
2. Use @EntityGraph or fetch joins
3. Enable second-level cache for read-heavy entities
4. Use projections for large result sets
5. Configure batch fetching
6. Use pagination for large collections

### Locking ✅

1. Prefer optimistic locking (use @Version)
2. Use pessimistic locking only when necessary
3. Keep locked transactions short
4. Handle OptimisticLockException gracefully

### Auditing ✅

1. Use @CreatedDate, @LastModifiedDate for all entities
2. Track who made changes with @CreatedBy, @LastModifiedBy
3. Create base auditable entity class
4. Enable JPA auditing with @EnableJpaAuditing
