# Spring Boot Caching

## Overview

Caching is one of the most effective performance optimization techniques. Spring Boot provides a powerful, vendor-neutral cache abstraction that works with multiple cache providers. Understanding caching deeply helps you build high-performance applications while avoiding common pitfalls.

**Key Concept**: Caching trades memory for speed by storing frequently accessed data in fast-access storage, reducing expensive operations like database queries or external API calls.

---

## 1. Spring Cache Abstraction

### Enabling Caching

```java
/**
 * Enable Spring Caching
 */
@SpringBootApplication
@EnableCaching  // ✅ Required to activate caching
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}
```

### @Cacheable - Cache Method Results

```java
/**
 * @Cacheable: Cache method results
 */
@Service
public class ProductService {
    
    @Autowired
    private ProductRepository productRepository;
    
    // ✅ Basic caching: Cache result by method parameter
    @Cacheable("products")  // Cache name
    public Product findById(Long id) {
        System.out.println("Fetching from database: " + id);
        return productRepository.findById(id).orElseThrow();
        
        // First call: Executes method, caches result
        // Subsequent calls: Returns cached result, method NOT executed
    }
    
    // ✅ Multiple cache names
    @Cacheable(value = {"products", "items"})
    public Product findByIdMultipleCaches(Long id) {
        // Stored in both "products" and "items" caches
        return productRepository.findById(id).orElseThrow();
    }
    
    // ✅ Composite cache key (multiple parameters)
    @Cacheable("products")
    public Product findByNameAndCategory(String name, String category) {
        // Cache key: "products::name-category" (both parameters)
        return productRepository.findByNameAndCategory(name, category);
    }
    
    // ✅ Custom cache key with SpEL
    @Cacheable(value = "products", key = "#id")
    public Product findByIdCustomKey(Long id) {
        return productRepository.findById(id).orElseThrow();
    }
    
    // ✅ Complex cache key (nested properties)
    @Cacheable(value = "orders", key = "#order.id + '-' + #order.customerId")
    public Order processOrder(Order order) {
        // Cache key: "orderId-customerId"
        return orderRepository.save(order);
    }
    
    // ✅ Root object access in SpEL
    @Cacheable(value = "products", key = "#root.methodName + '-' + #id")
    public Product findByIdWithMethodName(Long id) {
        // Cache key: "findByIdWithMethodName-123"
        return productRepository.findById(id).orElseThrow();
    }
}
```

#### SpEL Context for Cache Keys

|Expression|Description|Example|
|---|---|---|
|`#paramName`|Method parameter|`#id`, `#name`|
|`#p0, #p1, ...`|Parameter by position|`#p0` (first param)|
|`#a0, #a1, ...`|Alternative parameter syntax|`#a0` (first param)|
|`#root.methodName`|Method name|`"findById"`|
|`#root.method.name`|Full method name|`"findById"`|
|`#root.target`|Target object|Service instance|
|`#root.targetClass`|Target class|`ProductService.class`|
|`#root.args[0]`|Arguments array|First argument|
|`#result`|Method result (only in `unless`)|Return value|

### @CachePut - Update Cache

```java
/**
 * @CachePut: Always execute method and update cache
 */
@Service
public class ProductService {
    
    // ✅ Update cache after creating/updating
    @CachePut(value = "products", key = "#product.id")
    public Product save(Product product) {
        System.out.println("Saving to database: " + product.getId());
        return productRepository.save(product);
        
        // Method ALWAYS executes
        // Result stored in cache (replaces existing)
    }
    
    // ✅ Update cache with result ID (not parameter ID)
    @CachePut(value = "products", key = "#result.id")
    public Product create(Product product) {
        Product saved = productRepository.save(product);
        // Cache key uses the generated ID from result
        return saved;
    }
    
    // ✅ Common pattern: Update cache when modifying
    @CachePut(value = "products", key = "#id")
    public Product updatePrice(Long id, BigDecimal newPrice) {
        Product product = productRepository.findById(id).orElseThrow();
        product.setPrice(newPrice);
        return productRepository.save(product);
        // Cache automatically updated with new price
    }
}
```

**@Cacheable vs @CachePut**:

|Aspect|@Cacheable|@CachePut|
|---|---|---|
|**Execution**|Skipped if cached|Always executes|
|**Use Case**|Read operations|Write operations|
|**Performance**|Faster (skip execution)|Slower (always runs)|
|**Cache Update**|Only on cache miss|Every call|

### @CacheEvict - Remove from Cache

```java
/**
 * @CacheEvict: Invalidate cache entries
 */
@Service
public class ProductService {
    
    // ✅ Evict single entry
    @CacheEvict(value = "products", key = "#id")
    public void delete(Long id) {
        productRepository.deleteById(id);
        // Removes specific entry from cache
    }
    
    // ✅ Evict all entries in cache
    @CacheEvict(value = "products", allEntries = true)
    public void deleteAll() {
        productRepository.deleteAll();
        // Clears entire "products" cache
    }
    
    // ✅ Evict multiple caches
    @CacheEvict(value = {"products", "items"}, allEntries = true)
    public void clearAllCaches() {
        // Clears both "products" and "items" caches
    }
    
    // ✅ beforeInvocation: Evict before method execution
    @CacheEvict(value = "products", key = "#id", beforeInvocation = true)
    public void deleteRisky(Long id) {
        productRepository.deleteById(id);
        // If method throws exception, cache still evicted
        throw new RuntimeException("Error after eviction");
    }
    
    // ❌ Default: Evict after method execution
    @CacheEvict(value = "products", key = "#id")  // beforeInvocation = false (default)
    public void deleteDefault(Long id) {
        productRepository.deleteById(id);
        throw new RuntimeException("Error!");
        // Cache NOT evicted because method failed
    }
}
```

**beforeInvocation**:

|Setting|When Evicted|Use Case|
|---|---|---|
|`true`|Before method|Critical: evict even if method fails|
|`false` (default)|After method success|Only evict if operation succeeds|

### @Caching - Multiple Operations

```java
/**
 * @Caching: Combine multiple cache operations
 */
@Service
public class ProductService {
    
    // ✅ Multiple evictions
    @Caching(evict = {
        @CacheEvict(value = "products", key = "#product.id"),
        @CacheEvict(value = "productsById", key = "#product.id"),
        @CacheEvict(value = "productsByCategory", key = "#product.category")
    })
    public Product update(Product product) {
        return productRepository.save(product);
        // Evicts from 3 different caches
    }
    
    // ✅ Evict and put simultaneously
    @Caching(
        evict = @CacheEvict(value = "products", allEntries = true),
        put = @CachePut(value = "latestProduct", key = "'latest'")
    )
    public Product createAndInvalidate(Product product) {
        Product saved = productRepository.save(product);
        // 1. Clears all products from cache
        // 2. Stores this product as "latest"
        return saved;
    }
    
    // ✅ Multiple cacheable with different keys
    @Caching(cacheable = {
        @Cacheable(value = "productsById", key = "#id"),
        @Cacheable(value = "productsCache", key = "'product-' + #id")
    })
    public Product findWithMultipleKeys(Long id) {
        // Stores in both caches with different keys
        return productRepository.findById(id).orElseThrow();
    }
}
```

### Cache Key Generation

#### Default KeyGenerator

```java
/**
 * Default cache key generation
 */

// No parameters → SimpleKey.EMPTY
@Cacheable("cache")
public String noParams() { return "value"; }  // Key: SimpleKey []

// Single parameter → parameter value
@Cacheable("cache")
public String singleParam(Long id) { return "value"; }  // Key: 123

// Multiple parameters → SimpleKey with all parameters
@Cacheable("cache")
public String multiParams(Long id, String name) { return "value"; }  
// Key: SimpleKey [123, "John"]
```

#### Custom KeyGenerator

```java
/**
 * Custom KeyGenerator Implementation
 */
@Configuration
public class CacheConfig {
    
    // ✅ Custom KeyGenerator
    @Bean("customKeyGenerator")
    public KeyGenerator keyGenerator() {
        return (target, method, params) -> {
            StringBuilder key = new StringBuilder();
            key.append(target.getClass().getSimpleName());
            key.append(".");
            key.append(method.getName());
            key.append(":");
            
            for (Object param : params) {
                key.append(param.toString()).append("-");
            }
            
            return key.toString();
            // Example: "ProductService.findById:123-"
        };
    }
}

@Service
public class ProductService {
    
    // ✅ Use custom KeyGenerator
    @Cacheable(value = "products", keyGenerator = "customKeyGenerator")
    public Product findById(Long id) {
        return productRepository.findById(id).orElseThrow();
    }
}

/**
 * Advanced: UUID-based KeyGenerator
 */
@Component("uuidKeyGenerator")
public class UuidKeyGenerator implements KeyGenerator {
    
    @Override
    public Object generate(Object target, Method method, Object... params) {
        // Generate UUID for cache keys (useful for avoiding collisions)
        UUID uuid = UUID.nameUUIDFromBytes(
            (target.getClass().getName() + method.getName() + Arrays.toString(params))
                .getBytes()
        );
        return uuid.toString();
    }
}
```

### Conditional Caching

```java
/**
 * Conditional Caching: condition and unless
 */
@Service
public class ProductService {
    
    // ✅ condition: Decides whether to cache (before method execution)
    @Cacheable(value = "products", condition = "#id > 0")
    public Product findById(Long id) {
        // Only caches if id > 0
        // Negative IDs not cached
        return productRepository.findById(id).orElseThrow();
    }
    
    // ✅ condition with multiple parameters
    @Cacheable(value = "products", condition = "#id > 0 and #name != null")
    public Product findByIdAndName(Long id, String name) {
        // Caches only if BOTH conditions true
        return productRepository.findByIdAndName(id, name);
    }
    
    // ✅ unless: Prevents caching based on result (after method execution)
    @Cacheable(value = "products", unless = "#result == null")
    public Product findByIdSafe(Long id) {
        // Don't cache null results
        return productRepository.findById(id).orElse(null);
    }
    
    // ✅ unless with complex conditions
    @Cacheable(value = "products", unless = "#result.price.compareTo(T(java.math.BigDecimal).ZERO) <= 0")
    public Product findExpensiveProduct(Long id) {
        Product product = productRepository.findById(id).orElseThrow();
        // Don't cache products with price <= 0
        return product;
    }
    
    // ✅ Combine condition and unless
    @Cacheable(
        value = "products",
        condition = "#id > 0",                    // Check before execution
        unless = "#result == null || #result.isDeleted()"  // Check after execution
    )
    public Product findActive(Long id) {
        // Only cache if id > 0 AND result is not null AND product not deleted
        return productRepository.findById(id).orElse(null);
    }
    
    // ✅ Real-world example: Cache only successful API responses
    @Cacheable(
        value = "apiResponses",
        condition = "#apiKey != null",
        unless = "#result.statusCode != 200"
    )
    public ApiResponse callExternalApi(String apiKey, String endpoint) {
        ApiResponse response = externalClient.call(endpoint, apiKey);
        // Only cache successful responses (status 200)
        return response;
    }
}
```

**condition vs unless**:

|Attribute|When Evaluated|Use Case|Can Access Result?|
|---|---|---|---|
|`condition`|**Before** method|Skip caching entirely|❌ No (`#result` not available)|
|`unless`|**After** method|Prevent caching specific results|✅ Yes (`#result` available)|

### Cache Synchronization

```java
/**
 * Cache Synchronization: Prevent cache stampede
 */
@Service
public class ProductService {
    
    // ❌ Without synchronization: Multiple threads may execute simultaneously
    @Cacheable("products")
    public Product findById(Long id) {
        // If cache miss, multiple threads execute this method simultaneously
        return expensiveDatabaseQuery(id);  // Multiple DB hits!
    }
    
    // ✅ With synchronization: Only one thread executes, others wait
    @Cacheable(value = "products", sync = true)
    public Product findByIdSync(Long id) {
        // Only ONE thread executes on cache miss
        // Other threads wait for result
        return expensiveDatabaseQuery(id);  // Single DB hit!
    }
}
```

**sync = true**:

- ✅ Prevents **cache stampede** (thundering herd)
- ✅ Multiple threads don't execute simultaneously on cache miss
- ⚠️ Supported by: Caffeine, Hazelcast, JCache
- ⚠️ NOT supported by: Simple cache (ConcurrentHashMap), Redis (default)

### Simple Cache (ConcurrentHashMap)

### Caffeine Cache

```xml
<!-- pom.xml -->
<dependency>
    <groupId>com.github.ben-manes.caffeine</groupId>
    <artifactId>caffeine</artifactId>
</dependency>
```

```java
/**
 * Caffeine: High-performance Java caching library
 */
@Configuration
@EnableCaching
public class CaffeineCacheConfig {
    
    @Bean
    public CacheManager cacheManager() {
        CaffeineCacheManager cacheManager = new CaffeineCacheManager("products", "users", "orders");
        
        // ✅ Configure cache specifications
        cacheManager.setCaffeine(Caffeine.newBuilder()
            .maximumSize(1000)                    // Max 1000 entries
            .expireAfterWrite(10, TimeUnit.MINUTES)  // TTL: 10 minutes
            .recordStats());                       // Enable statistics
        
        return cacheManager;
    }
    
    // ✅ Per-cache configuration
    @Bean
    public CacheManager multiCacheManager() {
        SimpleCacheManager cacheManager = new SimpleCacheManager();
        
        List<CaffeineCache> caches = Arrays.asList(
            // Short-lived cache
            buildCache("products", Caffeine.newBuilder()
                .maximumSize(500)
                .expireAfterWrite(5, TimeUnit.MINUTES)),
            
            // Long-lived cache
            buildCache("users", Caffeine.newBuilder()
                .maximumSize(10000)
                .expireAfterWrite(1, TimeUnit.HOURS)),
            
            // Access-based expiration
            buildCache("sessions", Caffeine.newBuilder()
                .maximumSize(1000)
                .expireAfterAccess(30, TimeUnit.MINUTES))
        );
        
        cacheManager.setCaches(caches);
        return cacheManager;
    }
    
    private CaffeineCache buildCache(String name, Caffeine<Object, Object> caffeine) {
        return new CaffeineCache(name, caffeine.build());
    }
}
```

```yaml
# application.yml (alternative configuration)
spring:
  cache:
    type: caffeine
    cache-names: products, users, orders
    caffeine:
      spec: maximumSize=1000,expireAfterWrite=10m
```

**Caffeine Features**:

|Feature|Configuration|Use Case|
|---|---|---|
|**Size-based eviction**|`maximumSize(1000)`|Limit memory usage|
|**Time-based eviction**|`expireAfterWrite(10m)`|TTL for all entries|
|**Access-based eviction**|`expireAfterAccess(30m)`|Remove unused entries|
|**Weight-based eviction**|`maximumWeight(10000)`|Custom size calculation|
|**Statistics**|`recordStats()`|Monitor hit/miss ratio|
|**Async loading**|`buildAsync()`|Non-blocking cache loads|

**Pros**:

- ✅ High performance (better than Guava)
- ✅ Advanced eviction policies
- ✅ Statistics and monitoring
- ✅ Write-through support

**Cons**:

- ❌ Not distributed
- ❌ Single JVM only

**Use Case**: Single-instance applications requiring advanced caching features

### Redis Cache

```xml
<!-- pom.xml -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-redis</artifactId>

```

```java
/**
 * Redis Cache Configuration
 */
@Configuration
@EnableCaching
public class RedisCacheConfig {
    
    @Bean
    public CacheManager cacheManager(RedisConnectionFactory connectionFactory) {
        // ✅ Default configuration
        RedisCacheManager cacheManager = RedisCacheManager.builder(connectionFactory)
            .cacheDefaults(defaultCacheConfig())
            .build();
        
        return cacheManager;
    }
    
    private RedisCacheConfiguration defaultCacheConfig() {
        return RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofMinutes(10))              // TTL: 10 minutes
            .serializeKeysWith(
                RedisSerializationContext.SerializationPair.fromSerializer(
                    new StringRedisSerializer()))          // Keys as strings
            .serializeValuesWith(
                RedisSerializationContext.SerializationPair.fromSerializer(
                    new GenericJackson2JsonRedisSerializer()))  // Values as JSON
            .disableCachingNullValues();                   // Don't cache nulls
    }
    
    // ✅ Per-cache configuration
    @Bean
    public CacheManager multiCacheManager(RedisConnectionFactory connectionFactory) {
        Map<String, RedisCacheConfiguration> cacheConfigurations = new HashMap<>();
        
        // Short-lived cache (5 minutes)
        cacheConfigurations.put("products", 
            RedisCacheConfiguration.defaultCacheConfig()
                .entryTtl(Duration.ofMinutes(5)));
        
        // Long-lived cache (1 hour)
        cacheConfigurations.put("users", 
            RedisCacheConfiguration.defaultCacheConfig()
                .entryTtl(Duration.ofHours(1)));
        
        // No expiration
        cacheConfigurations.put("config", 
            RedisCacheConfiguration.defaultCacheConfig()
                .entryTtl(Duration.ZERO));  // Never expires
        
        return RedisCacheManager.builder(connectionFactory)
            .cacheDefaults(defaultCacheConfig())
            .withInitialCacheConfigurations(cacheConfigurations)
            .build();
    }
    
    // ✅ Custom serialization
    @Bean
    public CacheManager customSerializationCacheManager(RedisConnectionFactory connectionFactory) {
        // Use Kryo for better performance
        RedisSerializer<Object> serializer = new GenericJackson2JsonRedisSerializer();
        
        RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
            .serializeValuesWith(
                RedisSerializationContext.SerializationPair.fromSerializer(serializer));
        
        return RedisCacheManager.builder(connectionFactory)
            .cacheDefaults(config)
            .build();
    }
}
```

**Pros**:

- ✅ Rich data structures
- ✅ Pub/sub support
- ✅ Atomic operations

**Cons**:

- ❌ Slower than in-memory

**Use Case**: Distributed applications, microservices, session management

```java
/**
 * Hazelcast Cache Configuration
 */
@Configuration
@EnableCaching
public class HazelcastCacheConfig {
    
    @Bean
    public Config hazelcastConfig() {
        Config config = new Config();
        config.setInstanceName("hazelcast-cache");
        
        // ✅ Network configuration
        config.getNetworkConfig()
            .setPort(5701)
            .setPortAutoIncrement(true);
        
        // ✅ Map (cache) configuration
        MapConfig mapConfig = new MapConfig();
        mapConfig.setName("products");
        mapConfig.setTimeToLiveSeconds(600);  // 10 minutes
        mapConfig.setMaxIdleSeconds(300);     // 5 minutes idle
        mapConfig.setEvictionConfig(new EvictionConfig()
            .setEvictionPolicy(EvictionPolicy.LRU)
            .setMaxSizePolicy(MaxSizePolicy.ENTRY_COUNT)
            .setSize(1000));
        
        config.addMapConfig(mapConfig);
        
        return config;
    }
    
    @Bean
    public HazelcastInstance hazelcastInstance(Config config) {
        return Hazelcast.newHazelcastInstance(config);
    }
    
    @Bean
    public CacheManager cacheManager(HazelcastInstance hazelcastInstance) {
        return new HazelcastCacheManager(hazelcastInstance);
    }
}
```

**Pros**:

- ✅ Distributed and in-memory

- ✅ Near-cache support
- ✅ Replication and backup
- ✅ Advanced eviction policies

**Cons**:

- ❌ More complex setup
- ❌ Memory usage across cluster

**Use Case**: Distributed applications requiring fast access without external cache server

### Cache Expiration Strategies

```java
/**
 * Expiration Strategies
 */
@Configuration
public class ExpirationConfig {
    
    // 1️⃣ TTL (Time-To-Live): Fixed expiration after write
    @Bean
    public CacheManager ttlCacheManager() {
        return RedisCacheManager.builder(connectionFactory)
            .cacheDefaults(
                RedisCacheConfiguration.defaultCacheConfig()
                    .entryTtl(Duration.ofMinutes(10))  // Expires 10 min after write
            ).build();
    }
    
    // 2️⃣ TTI (Time-To-Idle): Expires after period of no access
    @Bean
    public CacheManager ttiCacheManager() {
        CaffeineCacheManager cacheManager = new CaffeineCacheManager();
        cacheManager.setCaffeine(Caffeine.newBuilder()
            .expireAfterAccess(30, TimeUnit.MINUTES));  // Expires if not accessed for 30 min
        return cacheManager;
    }
    
    // 3️⃣ Custom expiration: Different TTL per entry
    @Bean
    public CacheManager customExpirationCacheManager() {
        CaffeineCacheManager cacheManager = new CaffeineCacheManager();
        cacheManager.setCaffeine(Caffeine.newBuilder()
            .expireAfter(new Expiry<Object, Object>() {
                @Override
                public long expireAfterCreate(Object key, Object value, long currentTime) {
                    // Custom logic: expire premium users cache slower
                    if (value instanceof User && ((User) value).isPremium()) {
                        return TimeUnit.HOURS.toNanos(1);  // 1 hour
                    }
                    return TimeUnit.MINUTES.toNanos(10);   // 10 minutes
                }
                
                @Override
                public long expireAfterUpdate(Object key, Object value, 
                                               long currentTime, long currentDuration) {
                    return currentDuration;  // Keep same expiration
                }
                
                @Override
                public long expireAfterRead(Object key, Object value, 
                                             long currentTime, long currentDuration) {
                    return currentDuration;  // Keep same expiration
                }
            }));
        return cacheManager;
    }
}
```

```java
/**
```

## 4. Caching Strategies

### Cache-Aside (Lazy Loading)

**Most Common Pattern**: Application manages cache explicitly.

```java
/**
 * Cache-Aside Pattern (Lazy Loading)
 */
@Service
public class ProductService {
    
    @Autowired
    private ProductRepository productRepository;
    
    @Autowired
    private CacheManager cacheManager;
    
    // ✅ Read: Check cache first, load from DB if miss
    @Cacheable("products")
    public Product findById(Long id) {
        // Spring handles this automatically:
        // 1. Check cache for key
        // 2. If found, return cached value
        // 3. If not found, execute method
        // 4. Store result in cache
        // 5. Return result
        
        System.out.println("Cache miss - loading from DB");
        return productRepository.findById(id).orElseThrow();
    }
    
    // ✅ Write: Invalidate cache on update
    @CacheEvict(value = "products", key = "#product.id")
    public Product update(Product product) {
        Product updated = productRepository.save(product);
        // Cache invalidated, next read will reload from DB
        return updated;
    }
    
    // Manual cache-aside implementation (when needed):
    public Product findByIdManual(Long id) {
        Cache cache = cacheManager.getCache("products");
        
        // 1. Try cache
        Product cached = cache.get(id, Product.class);
        if (cached != null) {
            return cached;
        }
        
        // 2. Load from DB
        Product product = productRepository.findById(id).orElseThrow();
        
        // 3. Store in cache
        cache.put(id, product);
        
        return product;
    }
}
```

**Flow**:

```
READ:  Application → Cache (miss) → Database → Cache (store) → Application
WRITE: Application → Database → Cache (invalidate)
```

**Pros**:

- ✅ Simple to understand
- ✅ Cache only what's requested
- ✅ Resilient (cache failure doesn't break app)

**Cons**:

- ❌ Cache miss penalty (latency spike)
- ❌ Cache stampede risk

### Write-Through

**Pattern**: Write to cache and database simultaneously.

```java
/**
 * Write-Through Pattern
 */
@Service
public class ProductService {
    
    // ✅ Read from cache
    @Cacheable("products")
    public Product findById(Long id) {
        return productRepository.findById(id).orElseThrow();
    }
    
    // ✅ Write to cache AND database
    @CachePut(value = "products", key = "#product.id")
    public Product save(Product product) {
        // 1. Write to database first
        Product saved = productRepository.save(product);
        
        // 2. Spring automatically updates cache with return value
        // Cache always consistent with database
        
        return saved;
    }
    
    // Manual write-through implementation:
    public Product saveManual(Product product) {
        // 1. Write to database
        Product saved = productRepository.save(product);
        
        // 2. Write to cache
        Cache cache = cacheManager.getCache("products");
        cache.put(saved.getId(), saved);
        
        return saved;
    }
}
```

**Flow**:

```
WRITE: Application → Database (write) → Cache (write) → Application
READ:  Application → Cache → Application
```

**Pros**:

- ✅ Cache always consistent
- ✅ No stale data
- ✅ Read performance

**Cons**:

- ❌ Write latency (two operations)
- ❌ Caches data that might never be read
- ❌ Wastes cache space

### Write-Behind (Write-Back)

**Pattern**: Write to cache immediately, persist to database asynchronously.

```java
/**
 * Write-Behind Pattern
 */
@Service
public class ProductService {
    
    @Autowired
    private CacheManager cacheManager;
    
    @Autowired
    private ProductRepository productRepository;
    
    @Autowired
    private TaskExecutor taskExecutor;
    
    // ✅ Write to cache immediately, DB asynchronously
    public Product save(Product product) {
        // 1. Write to cache immediately
        Cache cache = cacheManager.getCache("products");
        cache.put(product.getId(), product);
        
        // 2. Write to DB asynchronously
        taskExecutor.execute(() -> {
            productRepository.save(product);
        });
        
        return product;  // Return immediately
    }
    
    // ✅ Read from cache (fast)
    @Cacheable("products")
    public Product findById(Long id) {
        return productRepository.findById(id).orElseThrow();
    }
}

/**
 * Advanced: Batch Write-Behind
 */
@Service
public class BatchWriteBehindService {
    
    private final Queue<Product> writeQueue = new ConcurrentLinkedQueue<>();
    
    public Product save(Product product) {
        // Add to cache
        cache.put(product.getId(), product);
        
        // Add to write queue
        writeQueue.offer(product);
        
        return product;
    }
    
    // Scheduled batch writer
    @Scheduled(fixedRate = 5000)  // Every 5 seconds
    public void flushWrites() {
        List<Product> batch = new ArrayList<>();
        Product product;
        
        // Drain queue
        while ((product = writeQueue.poll()) != null) {
            batch.add(product);
        }
        
        if (!batch.isEmpty()) {
            productRepository.saveAll(batch);  // Batch write to DB
        }
    }
}
```

**Flow**:

```
WRITE: Application → Cache (write) → Application
       Cache → [Queue] → Database (async batch)
READ:  Application → Cache → Application
```

**Pros**:

- ✅ Fastest write performance
- ✅ Batch writes (efficient)
- ✅ Reduces database load

**Cons**:

- ❌ Risk of data loss (if cache crashes before DB write)
- ❌ Complex to implement correctly
- ❌ Consistency challenges

**Use Case**: High write throughput, eventual consistency acceptable (analytics, logs)

### Refresh-Ahead

**Pattern**: Proactively refresh cache before expiration.

```java
/**
 * Refresh-Ahead Pattern
 */
@Service
public class ProductService {
    
    @Autowired
    private ProductRepository productRepository;
    
    @Autowired
    private CacheManager cacheManager;
    
    // ✅ Caffeine with refresh support
    @Bean
    public CacheManager refreshAheadCacheManager() {
        CaffeineCacheManager cacheManager = new CaffeineCacheManager();
        cacheManager.setCaffeine(Caffeine.newBuilder()
            .maximumSize(1000)
            .expireAfterWrite(10, TimeUnit.MINUTES)
            .refreshAfterWrite(5, TimeUnit.MINUTES));  // Refresh after 5 min
        return cacheManager;
    }
    
    @Cacheable("products")
    public Product findById(Long id) {
        return productRepository.findById(id).orElseThrow();
    }
    
    // Manual refresh-ahead implementation:
    @Scheduled(fixedRate = 60000)  // Every minute
    public void refreshPopularProducts() {
        List<Long> popularIds = getPopularProductIds();
        
        for (Long id : popularIds) {
            // Proactively reload cache
            Product product = productRepository.findById(id).orElse(null);
            if (product != null) {
                Cache cache = cacheManager.getCache("products");
                cache.put(id, product);
            }
        }
    }
    
    private List<Long> getPopularProductIds() {
        // Logic to identify popular products (e.g., most frequently accessed)
        return productRepository.findTopByAccessCount(100);
    }
}
```

**Flow**:

```
READ: Application → Cache (hit, but near expiration) → Application
      Background: Cache → Database (refresh) → Cache (update)
```

**Pros**:

- ✅ No cache miss latency for frequently accessed data
- ✅ Always fresh data

**Cons**:

- ❌ Wastes resources refreshing unused data
- ❌ Complex to implement

**Use Case**: High-traffic items (product pages, user profiles)

## 5. Common Pitfalls

### Caching Methods with Side Effects

```java
/**
 * PITFALL: Caching methods with side effects
 */
@Service
public class BadService {
    
    // ❌ BAD: Method has side effects
    @Cacheable("users")
    public User processUser(Long id) {
        User user = userRepository.findById(id).orElseThrow();
        
        // Side effect: Sending email
        emailService.sendWelcomeEmail(user);  // ❌ Sent only on cache miss!
        
        // Side effect: Incrementing counter
        user.setLoginCount(user.getLoginCount() + 1);  // ❌ Not incremented on cache hit!
        userRepository.save(user);
        
        return user;
    }
}

@Service
public class GoodService {
    
    // ✅ GOOD: Separate caching from side effects
    public User processUser(Long id) {
        User user = findById(id);  // Cached
        
        // Always execute side effects
        emailService.sendWelcomeEmail(user);
        incrementLoginCount(id);
        
        return user;
    }
    
    @Cacheable("users")
    public User findById(Long id) {
        // Pure function: no side effects
        return userRepository.findById(id).orElseThrow();
    }
    
    @CacheEvict(value = "users", key = "#id")
    public void incrementLoginCount(Long id) {
        User user = userRepository.findById(id).orElseThrow();
        user.setLoginCount(user.getLoginCount() + 1);
        userRepository.save(user);
    }
}
```

**Rule**: Only cache **pure functions** (no side effects).

### Cache Stampede (Thundering Herd)

```java
/**
 * PITFALL: Cache stampede
 */
@Service
public class ProductService {
    
    // ❌ PROBLEM: Multiple threads execute on cache miss
    @Cacheable("products")
    public Product findById(Long id) {
        // If cache expires, many threads may execute simultaneously
        return expensiveDatabaseQuery(id);  // Multiple DB hits!
    }
    
    // ✅ SOLUTION 1: sync = true
    @Cacheable(value = "products", sync = true)
    public Product findByIdSync(Long id) {
        // Only one thread executes, others wait for result
        return expensiveDatabaseQuery(id);
    }
    
    // ✅ SOLUTION 2: Probabilistic early expiration
    @Cacheable("products")
    public Product findByIdProbabilistic(Long id) {
        return expensiveDatabaseQuery(id);
    }
    
    @Bean
    public CacheManager probabilisticCacheManager() {
        CaffeineCacheManager cacheManager = new CaffeineCacheManager();
        cacheManager.setCaffeine(Caffeine.newBuilder()
            .expireAfterWrite(10, TimeUnit.MINUTES)
            .refreshAfterWrite(8, TimeUnit.MINUTES));  // Refresh before expiration
        return cacheManager;
    }
    
    // ✅ SOLUTION 3: Lock-based refresh
    private final ConcurrentHashMap<Long, Lock> locks = new ConcurrentHashMap<>();
    
    public Product findByIdWithLock(Long id) {
        Cache cache = cacheManager.getCache("products");
        Product cached = cache.get(id, Product.class);
        
        if (cached != null) {
            return cached;
        }
        
        // Acquire lock for this ID
        Lock lock = locks.computeIfAbsent(id, k -> new ReentrantLock());
        lock.lock();
        try {
            // Double-check cache (another thread may have loaded it)
            cached = cache.get(id, Product.class);
            if (cached != null) {
                return cached;
            }
            
            // Load from database
            Product product = expensiveDatabaseQuery(id);
            cache.put(id, product);
            return product;
            
        } finally {
            lock.unlock();
        }
    }
}
```

**Cache Stampede Symptoms**:

- Sudden spike in database load
- Multiple identical queries simultaneously
- Degraded performance on cache expiration

### Stale Data

```java
/**
 * PITFALL: Stale cached data
 */
@Service
public class OrderService {
    
    // ❌ PROBLEM: Cached order becomes stale when updated elsewhere
    @Cacheable("orders")
    public Order findById(Long id) {
        return orderRepository.findById(id).orElseThrow();
    }
    
    public void updateStatus(Long id, OrderStatus status) {
        Order order = orderRepository.findById(id).orElseThrow();
        order.setStatus(status);
        orderRepository.save(order);
        // Cache still has old status! ❌
    }
    
    // ✅ SOLUTION 1: Evict cache on update
    @CacheEvict(value = "orders", key = "#id")
    public void updateStatusWithEviction(Long id, OrderStatus status) {
        Order order = orderRepository.findById(id).orElseThrow();
        order.setStatus(status);
        orderRepository.save(order);
        // Cache evicted, next read will reload ✅
    }
    
    // ✅ SOLUTION 2: Update cache on update
    @CachePut(value = "orders", key = "#id")
    public Order updateStatusWithCachePut(Long id, OrderStatus status) {
        Order order = orderRepository.findById(id).orElseThrow();
        order.setStatus(status);
        Order updated = orderRepository.save(order);
        return updated;  // Cache updated ✅
    }
    
    // ✅ SOLUTION 3: Short TTL for frequently changing data
    @Cacheable(value = "orders")
    public Order findByIdShortTTL(Long id) {
        return orderRepository.findById(id).orElseThrow();
    }
    
    @Bean
    public CacheManager shortTTLCacheManager() {
        CaffeineCacheManager cacheManager = new CaffeineCacheManager("orders");
        cacheManager.setCaffeine(Caffeine.newBuilder()
            .expireAfterWrite(30, TimeUnit.SECONDS));  // Short TTL
        return cacheManager;
    }
}
```

**Stale Data Solutions**:

|Approach|Consistency|Performance|Use Case|
|---|---|---|---|
|**No caching**|Strong|Slow|Critical data|
|**Short TTL**|Eventual|Fast reads|Frequently updated|
|**@CacheEvict**|Strong|Fast reads|Manual invalidation|
|**@CachePut**|Strong|Slower writes|Read-heavy|

## Summary & Best Practices

### Quick Reference

|Annotation|Purpose|When Executed|Cache Update|
|---|---|---|---|
|**@Cacheable**|Read from cache|Only on cache miss|After method execution|
|**@CachePut**|Update cache|Always|After method execution|
|**@CacheEvict**|Remove from cache|Always (configurable)|Before/after method|
|**@Caching**|Multiple operations|N/A|As configured|

### Cache Provider Selection

|Provider|Distributed|Performance|Setup|Use Case|
|---|---|---|---|---|
|**Simple**|❌|⚡⚡⚡⚡|✅ Easy|Dev/test, single instance|
|**Caffeine**|❌|⚡⚡⚡⚡|✅ Easy|Production, single instance|
|**Redis**|✅|⚡⚡⚡|⚠️ Moderate|Distributed apps, sessions|
|**Hazelcast**|✅|⚡⚡⚡⚡|⚠️ Complex|Embedded distributed cache|

### Best Practices

1. **Always use @EnableCaching** to activate cache support
2. **Cache pure functions only** - no side effects
3. **Use sync = true** to prevent cache stampede
4. **Set appropriate TTL** based on data volatility
5. **Use @CacheEvict** or @CachePut on write operations
6. **Monitor cache metrics** (hit rate, evictions)
7. **Use conditional caching** to avoid caching nulls
8. **Choose cache provider** based on distribution needs
9. **Avoid self-invocation** - extract to separate services
10. **Test cache behavior** - verify hits, misses, evictions

### Common Mistakes to Avoid

❌ **Caching methods with side effects**  
❌ **Not invalidating cache on updates**  
❌ **Over-caching (caching everything)**  
❌ **Under-caching (not caching expensive operations)**  
❌ **Ignoring cache stampede**  
❌ **Not monitoring cache performance**  
❌ **Self-invocation bypassing cache**  
❌ **Caching null results without condition**  
❌ **Not setting TTL (infinite cache)**  
❌ **Using wrong cache provider for use case**
