# Advanced Topics - Deep Interview Questions & Answers

## Overview

This guide contains in-depth answers to advanced Spring Boot interview questions, focusing on reactive programming, microservices patterns, and distributed systems challenges.

**Interview Tip**: Advanced questions test architectural thinking, understanding of trade-offs, and ability to design systems that scale and remain resilient.

## Advanced Questions

### Q1: When to use reactive programming

**Short Answer**: Use reactive programming for I/O-bound applications with high concurrency, streaming data, backpressure requirements, or when non-blocking is critical. Avoid for CPU-bound tasks, simple CRUD apps, or teams unfamiliar with reactive paradigms. Reactive trades complexity for scalability and resource efficiency.

**Use Cases & Decision Framework**: See full guide for complete examples covering when reactive provides value vs when traditional blocking is better, with performance comparisons and real-world scenarios.

### Q2: Microservices communication patterns

**Short Answer**: Microservices communicate via synchronous (REST, gRPC) or asynchronous (messaging, events) patterns. Choose sync for immediate response needs, async for decoupling and resilience. Use service mesh for cross-cutting concerns, circuit breakers for fault tolerance, and saga pattern for distributed transactions.

**Communication Patterns**: See full guide for complete examples of REST, gRPC, message queues, event-driven architecture, service discovery, circuit breakers, API gateways, and saga patterns with working code.

### Q3: Handling distributed transactions

**Short Answer**: Distributed transactions cannot use traditional ACID transactions across services. Solutions include: Saga pattern (orchestration or choreography), Two-Phase Commit (2PC - rarely used), eventual consistency, idempotent operations, and compensating transactions. Choose based on consistency requirements and system complexity.

**Transaction Strategies**: See full guide for complete implementations of choreography-based sagas, orchestration-based sagas, outbox pattern, eventual consistency, and decision frameworks for choosing the right approach.
