## Core Spring Questions

### Q1: Explain IoC and DI with real-world examples

**Short Answer**:

- **IoC (Inversion of Control)**: Framework controls object creation and lifecycle, not your code
- **DI (Dependency Injection)**: IoC implementation where dependencies are provided (injected) rather than created

**Detailed Explanation**:

**Traditional Programming (You Control)**:

```java
public class OrderService {
    private EmailService emailService;
    
    public OrderService() {
        this.emailService = new EmailService(); // You create it
    }
}
```

**With IoC/DI (Framework Controls)**:

```java
@Service
public class OrderService {
    private final EmailService emailService;
    
    // Spring creates and injects EmailService
    public OrderService(EmailService emailService) {
        this.emailService = emailService;
    }
}
```

**Real-World Analogy**: Think of a restaurant:

- **Without IoC**: You (the chef) must go shopping, buy ingredients, prepare everything yourself
- **With IoC**: Someone else (Spring) prepares all ingredients and brings them to your station. You just cook.

**Benefits**:

1. **Loose Coupling**: OrderService doesn't know about EmailService implementation
2. **Testability**: Easy to inject mock EmailService for testing
3. **Flexibility**: Switch implementations without changing OrderService

**Real-World Example**:

```java
// Without DI - Tightly coupled
public class PaymentService {
    private StripePaymentGateway gateway;
    
    public PaymentService() {
        this.gateway = new StripePaymentGateway(); // Hard-coded!
    }
    
    // Problem: Can't switch to PayPal without changing code
    // Problem: Can't test without calling real Stripe API
}

// With DI - Loosely coupled
public interface PaymentGateway {
    void processPayment(double amount);
}

@Service
public class StripePaymentGateway implements PaymentGateway {
    public void processPayment(double amount) {
        // Stripe implementation
    }
}

@Service
public class PayPalPaymentGateway implements PaymentGateway {
    public void processPayment(double amount) {
        // PayPal implementation
    }
}

@Service
public class PaymentService {
    private final PaymentGateway gateway;
    
    // Spring injects the implementation
    public PaymentService(PaymentGateway gateway) {
        this.gateway = gateway;
    }
    
    public void makePayment(double amount) {
        gateway.processPayment(amount);
    }
}

// Configuration - Easy to switch implementations
@Configuration
public class PaymentConfig {
    @Bean
    @Primary
    public PaymentGateway paymentGateway() {
        // Switch between Stripe and PayPal easily
        return new StripePaymentGateway();
        // return new PayPalPaymentGateway();
    }
}

// Testing - Easy to mock
@Test
public void testPaymentService() {
    PaymentGateway mockGateway = mock(PaymentGateway.class);
    PaymentService service = new PaymentService(mockGateway);
    
    service.makePayment(100.0);
    
    verify(mockGateway).processPayment(100.0);
}
```

**Types of Dependency Injection**:

**1. Constructor Injection (Recommended)**:

```java
@Service
public class OrderService {
    private final EmailService emailService;
    private final InventoryService inventoryService;
    
    // Constructor injection - immutable, required dependencies
    public OrderService(EmailService emailService, 
                       InventoryService inventoryService) {
        this.emailService = emailService;
        this.inventoryService = inventoryService;
    }
}
```

**2. Setter Injection (Optional dependencies)**:

```java
@Service
public class ReportService {
    private NotificationService notificationService;
    
    // Optional dependency
    @Autowired(required = false)
    public void setNotificationService(NotificationService service) {
        this.notificationService = service;
    }
    
    public void generateReport() {
        // ... generate report ...
        
        if (notificationService != null) {
            notificationService.notify("Report ready");
        }
    }
}
```

**3. Field Injection (Not recommended)**:

```java
@Service
public class UserService {
    @Autowired
    private UserRepository userRepository; // Avoid this!
    
    // Problems:
    // - Can't make field final (not immutable)
    // - Hard to test (need reflection)
    // - Hidden dependencies (not visible in constructor)
}
```

### Q2: How does Spring resolve circular dependencies?

**Short Answer**: Spring uses three-level cache to handle circular dependencies in singleton beans with setter/field injection. Constructor injection circular dependencies cannot be resolved.

**Detailed Explanation**:

**Circular Dependency Example**:

```java
@Service
public class ServiceA {
    @Autowired
    private ServiceB serviceB;
}

@Service
public class ServiceB {
    @Autowired
    private ServiceA serviceA;
}
```

**Spring's Three-Level Cache**:

```
1. singletonObjects (L1) - Fully initialized beans
2. earlySingletonObjects (L2) - Early bean references
3. singletonFactories (L3) - Bean factories for early reference
```

**Resolution Process**:

```
Step 1: Create ServiceA (incomplete)
        └─> Put ServiceA factory in L3 cache
Step 2: ServiceA needs ServiceB
        └─> Start creating ServiceB
Step 3: ServiceB needs ServiceA
        └─> Get ServiceA from L3 (early reference)
        └─> Return incomplete ServiceA to ServiceB
Step 4: Complete ServiceB initialization
        └─> Move ServiceB to L1 cache
Step 5: Complete ServiceA initialization (now has ServiceB)
        └─> Move ServiceA to L1 cache
```

**Visual Representation**:

```java
// Spring's internal process (simplified)
public class DefaultSingletonBeanRegistry {
    // L1: Fully initialized singletons
    private final Map<String, Object> singletonObjects = new ConcurrentHashMap<>();
    
    // L2: Early singleton references (being created)
    private final Map<String, Object> earlySingletonObjects = new HashMap<>();
    
    // L3: Factories for early references
    private final Map<String, ObjectFactory<?>> singletonFactories = new HashMap<>();
    
    public Object getSingleton(String beanName) {
        // Try L1 cache
        Object singleton = singletonObjects.get(beanName);
        if (singleton != null) return singleton;
        
        // Try L2 cache
        singleton = earlySingletonObjects.get(beanName);
        if (singleton != null) return singleton;
        
        // Try L3 cache (create early reference)
        ObjectFactory<?> factory = singletonFactories.get(beanName);
        if (factory != null) {
            singleton = factory.getObject();
            earlySingletonObjects.put(beanName, singleton);
            singletonFactories.remove(beanName);
        }
        
        return singleton;
    }
}
```

**Why Constructor Injection Fails**:

```java
// This WILL FAIL - Spring cannot resolve
@Service
public class ServiceA {
    private final ServiceB serviceB;
    
    public ServiceA(ServiceB serviceB) { // Constructor injection
        this.serviceB = serviceB;
    }
}

@Service
public class ServiceB {
    private final ServiceA serviceA;
    
    public ServiceB(ServiceA serviceA) { // Constructor injection
        this.serviceA = serviceA;
    }
}

// Error at startup:
// "The dependencies of some of the beans in the application 
//  context form a cycle:
//  
//  ┌─────┐
//  |  serviceA defined in file [ServiceA.class]
//  ↑     ↓
//  |  serviceB defined in file [ServiceB.class]
//  └─────┘"
```

**Why It Fails**:

- Constructor requires fully initialized object BEFORE the object can be created
- ServiceA constructor needs ServiceB, but ServiceB constructor needs ServiceA
- Chicken-and-egg problem - neither can be created first

**Solutions**:

**Solution 1: Use @Lazy (Recommended)**:

```java
@Service
public class ServiceA {
    private final ServiceB serviceB;
    
    // @Lazy creates a proxy, breaks the cycle
    public ServiceA(@Lazy ServiceB serviceB) {
        this.serviceB = serviceB; // Proxy injected, not real object
    }
    
    public void doSomething() {
        serviceB.execute(); // Proxy delegates to real object
    }
}

@Service
public class ServiceB {
    private final ServiceA serviceA;
    
    public ServiceB(ServiceA serviceA) {
        this.serviceA = serviceA;
    }
}
```

**Solution 2: Redesign (Best Practice)**:

```java
// Bad design - circular dependency indicates poor separation of concerns
// Ask: "Why do these services need each other?"

// Good design - extract common logic
@Service
public class OrderService {
    private final PaymentService paymentService;
    private final EmailService emailService;
    
    public void createOrder(Order order) {
        // Process order
        paymentService.processPayment(order);
        emailService.sendConfirmation(order);
    }
}

@Service
public class PaymentService {
    // No dependency on OrderService
    public void processPayment(Order order) {
        // Payment logic
    }
}

@Service
public class EmailService {
    // No dependency on OrderService
    public void sendConfirmation(Order order) {
        // Email logic
    }
}
```

**Solution 3: Use @PostConstruct + Setter**:

```java
@Service
public class ServiceA {
    private ServiceB serviceB;
    
    @Autowired
    public void setServiceB(ServiceB serviceB) {
        this.serviceB = serviceB;
    }
    
    // Can also use @PostConstruct for initialization
    @PostConstruct
    public void init() {
        // Initialization logic after all dependencies injected
    }
}

@Service
public class ServiceB {
    private ServiceA serviceA;
    
    @Autowired
    public void setServiceA(ServiceA serviceA) {
        this.serviceA = serviceA;
    }
}
```

**Solution 4: Use ApplicationContextAware**:

```java
@Service
public class ServiceA implements ApplicationContextAware {
    private ApplicationContext context;
    private ServiceB serviceB;
    
    @Override
    public void setApplicationContext(ApplicationContext context) {
        this.context = context;
    }
    
    public void doSomething() {
        // Lazy lookup of ServiceB
        if (serviceB == null) {
            serviceB = context.getBean(ServiceB.class);
        }
        serviceB.execute();
    }
}
```

**Real-World Example - Circular Dependency Bug**:

```java
// Bug discovered in production
@Service
public class UserService {
    private final AuthenticationService authService;
    
    public UserService(AuthenticationService authService) {
        this.authService = authService;
    }
    
    public User createUser(UserDto dto) {
        User user = new User(dto);
        // Save user then authenticate
        authService.authenticate(user);
        return user;
    }
}

@Service
public class AuthenticationService {
    private final UserService userService;
    
    public AuthenticationService(UserService userService) {
        this.userService = userService;
    }
    
    public void authenticate(User user) {
        // Needed to look up user details
        User fullUser = userService.findById(user.getId());
        // Authenticate...
    }
}

// Fix: Break the cycle
@Service
public class UserService {
    private final UserRepository userRepository;
    
    public User createUser(UserDto dto) {
        return userRepository.save(new User(dto));
    }
    
    public User findById(Long id) {
        return userRepository.findById(id).orElseThrow();
    }
}

@Service
public class AuthenticationService {
    private final UserRepository userRepository; // Direct dependency
    
    public void authenticate(User user) {
        User fullUser = userRepository.findById(user.getId()).orElseThrow();
        // Authenticate...
    }
}
```

### Q3: Difference between BeanFactory and ApplicationContext

**Short Answer**: ApplicationContext extends BeanFactory with more enterprise features like event propagation, internationalization, and AOP support. ApplicationContext is almost always preferred.

**Comparison Table**:

|Feature|BeanFactory|ApplicationContext|
|---|---|---|
|Lazy initialization|Yes (default)|No (default)|
|Bean post-processing|Manual|Automatic|
|Internationalization (i18n)|No|Yes|
|Event publication|No|Yes|
|AOP support|Manual|Automatic|
|Application-layer features|No|Yes|
|Startup time|Faster|Slower|
|Memory footprint|Lower|Higher|
|Enterprise features|Minimal|Full|

**Code Example**:

```java
// BeanFactory - Basic container
BeanFactory factory = new XmlBeanFactory(
    new ClassPathResource("beans.xml")
);
MyBean bean = (MyBean) factory.getBean("myBean");

// ApplicationContext - Full-featured container
ApplicationContext context = new ClassPathXmlApplicationContext("beans.xml");
MyBean bean = context.getBean(MyBean.class);

// Event publishing (ApplicationContext only)
context.publishEvent(new CustomEvent(this, "data"));
```

**Detailed Differences**:

**1. Bean Instantiation**:

```java
// BeanFactory - Lazy (on-demand)
BeanFactory factory = new XmlBeanFactory(...);
// Beans NOT created yet

MyBean bean = factory.getBean("myBean"); // Created NOW

// ApplicationContext - Eager (at startup)
ApplicationContext context = new ClassPathXmlApplicationContext("beans.xml");
// ALL singleton beans created NOW (during context initialization)

MyBean bean = context.getBean(MyBean.class); // Already exists
```

**2. BeanPostProcessor Support**:

```java
// BeanFactory - Manual registration required
BeanFactory factory = new XmlBeanFactory(...);
MyBeanPostProcessor processor = new MyBeanPostProcessor();
factory.addBeanPostProcessor(processor); // Manual!

// ApplicationContext - Automatic
ApplicationContext context = new ClassPathXmlApplicationContext("beans.xml");
// BeanPostProcessor beans automatically detected and registered
```

**3. Event Handling**:

```java
// BeanFactory - No event support
// Cannot do: factory.publishEvent(...)

// ApplicationContext - Full event support
@Component
public class MyEventListener implements ApplicationListener<CustomEvent> {
    @Override
    public void onApplicationEvent(CustomEvent event) {
        System.out.println("Event received: " + event);
    }
}

@Service
public class MyService {
    @Autowired
    private ApplicationContext context;
    
    public void doSomething() {
        // Publish event
        context.publishEvent(new CustomEvent(this, "data"));
    }
}
```

**4. Internationalization (i18n)**:

```java
// BeanFactory - No i18n support

// ApplicationContext - Built-in i18n
// messages.properties:
// welcome.message=Welcome {0}!

ApplicationContext context = ...;

String message = context.getMessage(
    "welcome.message",
    new Object[]{"John"},
    Locale.ENGLISH
);
// Output: "Welcome John!"
```

**5. Resource Loading**:

```java
// BeanFactory - Basic resource loading
Resource resource = new ClassPathResource("config.xml");

// ApplicationContext - Advanced resource loading
ApplicationContext context = ...;

// Load from classpath
Resource r1 = context.getResource("classpath:config.xml");

// Load from file system
Resource r2 = context.getResource("file:/path/to/config.xml");

// Load from URL
Resource r3 = context.getResource("https://example.com/config.xml");
```

**6. Lifecycle Management**:

```java
// ApplicationContext provides lifecycle hooks
@Component
public class MyBean implements ApplicationContextAware, DisposableBean {
    
    private ApplicationContext context;
    
    @Override
    public void setApplicationContext(ApplicationContext context) {
        this.context = context;
        // Access to application context
    }
    
    @Override
    public void destroy() {
        // Cleanup on shutdown
    }
}

// BeanFactory has limited lifecycle support
```

**When to Use Each**:

**BeanFactory**:

- Mobile applications (memory constrained)
- IoT devices with limited resources
- Microcontrollers
- You need absolute control over bean instantiation timing
- Legacy systems where you can't change the architecture

**ApplicationContext**:

- 99.99% of Spring applications
- Web applications (Spring MVC, Spring Boot)
- Enterprise applications
- Microservices
- When you need any Spring features beyond basic DI

**Real-World Example**:

```java
// In Spring Boot, you always get ApplicationContext
@SpringBootApplication
public class MyApplication {
    
    public static void main(String[] args) {
        // Returns ConfigurableApplicationContext (extends ApplicationContext)
        ApplicationContext ctx = SpringApplication.run(MyApplication.class, args);
        
        // Access environment properties
        Environment env = ctx.getEnvironment();
        String port = env.getProperty("server.port");
        
        // Publish events
        ctx.publishEvent(new ApplicationStartedEvent());
        
        // Get internationalized messages
        String msg = ctx.getMessage("welcome", null, Locale.US);
        
        // Access all beans of a type
        Map<String, MyService> services = ctx.getBeansOfType(MyService.class);
        
        // Graceful shutdown
        ((ConfigurableApplicationContext) ctx).close();
    }
}

// Using ApplicationContext in beans
@Service
public class ReportService {
    
    @Autowired
    private ApplicationContext context;
    
    public void generateReport(String type) {
        // Dynamic bean lookup based on type
        ReportGenerator generator = context.getBean(
            type + "ReportGenerator", 
            ReportGenerator.class
        );
        generator.generate();
    }
}
```

**ApplicationContext Implementations**:

```java

    "applicationContext.xml"
);

// 2. FileSystemXmlApplicationContext - Load from file system
ApplicationContext context = new FileSystemXmlApplicationContext(
    "/path/to/applicationContext.xml"
);

// 3. AnnotationConfigApplicationContext - Java config
ApplicationContext context = new AnnotationConfigApplicationContext(
    AppConfig.class
);

// 4. WebApplicationContext - Web applications
// Created automatically by Spring Boot

// 5. GenericApplicationContext - Programmatic
GenericApplicationContext context = new GenericApplicationContext();
context.registerBean(MyBean.class);
context.refresh();
```

### Q4: Bean lifecycle in detail

**Short Answer**: Spring bean lifecycle involves instantiation → dependency injection → initialization callbacks → usage → destruction callbacks.

**Complete Bean Lifecycle (15 Steps)**:

```
1. Instantiate Bean (constructor called)
         ↓
2. Populate Properties (dependency injection)
         ↓
3. Set Bean Name (BeanNameAware)
         ↓
4. Set Bean Factory (BeanFactoryAware)
         ↓
5. Set Application Context (ApplicationContextAware)
         ↓
6. Pre-Initialization (BeanPostProcessor.postProcessBeforeInitialization)
         ↓
7. @PostConstruct method
         ↓
8. InitializingBean.afterPropertiesSet()
         ↓
9. Custom init-method
         ↓
10. Post-Initialization (BeanPostProcessor.postProcessAfterInitialization)
         ↓
11. Bean Ready to Use
         ↓
12. Container Shutdown
         ↓
13. @PreDestroy method
         ↓
14. DisposableBean.destroy()
         ↓
15. Custom destroy-method
```

**Practical Example - All Lifecycle Hooks**:

```java
@Component
public class LifecycleBean implements 
        BeanNameAware, 
        BeanFactoryAware,
        ApplicationContextAware,
        InitializingBean,
        DisposableBean {
    
    private String beanName;
    private BeanFactory beanFactory;
    private ApplicationContext context;
    
    // 1. Constructor
    public LifecycleBean() {
        System.out.println("1. Constructor called");
    }
    
    // 2. Dependency Injection
    @Autowired
    private DataSource dataSource; // Injected here
    
    // 3. BeanNameAware
    @Override
    public void setBeanName(String name) {
        this.beanName = name;
        System.out.println("3. Bean name set: " + name);
    }
    
    // 4. BeanFactoryAware
    @Override
    public void setBeanFactory(BeanFactory beanFactory) {
        this.beanFactory = beanFactory;
        System.out.println("4. BeanFactory set");
    }
    
    // 5. ApplicationContextAware
    @Override
    public void setApplicationContext(ApplicationContext ctx) {
        this.context = ctx;
        System.out.println("5. ApplicationContext set");
    }
    
    // 7. @PostConstruct (Modern way - RECOMMENDED)
    @PostConstruct
    public void postConstruct() {
        System.out.println("7. @PostConstruct - Initialize resources");
        // Best place for initialization logic
    }
    
    // 8. InitializingBean.afterPropertiesSet (Old way)
    @Override
    public void afterPropertiesSet() {
        System.out.println("8. afterPropertiesSet called");
    }
    
    // 9. Custom init-method (XML or @Bean)
    public void customInit() {
        System.out.println("9. Custom init-method");
    }
    
    // 13. @PreDestroy (Modern way - RECOMMENDED)
    @PreDestroy
    public void preDestroy() {
        System.out.println("13. @PreDestroy - Cleanup resources");
        // Close connections, release resources
    }
    
    // 14. DisposableBean.destroy (Old way)
    @Override
    public void destroy() {
        System.out.println("14. destroy() called");
    }
    
    // 15. Custom destroy-method
    public void customDestroy() {
        System.out.println("15. Custom destroy-method");
    }
}

// Configuration
@Configuration
public class Config {
    @Bean(initMethod = "customInit", destroyMethod = "customDestroy")
    public LifecycleBean lifecycleBean() {
        return new LifecycleBean();
    }
}
```

**Real-World Usage Examples**:

**Example 1: Database Connection Pool**:

```java
@Service
public class DatabaseConnectionService {
    
    private HikariDataSource dataSource;
    
    @Value("${spring.datasource.url}")
    private String jdbcUrl;
    
    @Value("${spring.datasource.username}")
    private String username;

    public void init() {
        // Initialize connection pool after properties are set
        dataSource = new HikariDataSource();
        dataSource.setJdbcUrl(jdbcUrl);
        dataSource.setUsername(username);
        dataSource.setPassword(password);
        dataSource.setMaximumPoolSize(10);
        
        System.out.println("✅ Connection pool initialized");
    }
    
    @PreDestroy
    public void cleanup() {
        // Close connection pool gracefully
        if (dataSource != null && !dataSource.isClosed()) {
            dataSource.close();
            System.out.println("✅ Connection pool closed");
        }
    }
    
    public Connection getConnection() throws SQLException {
        return dataSource.getConnection();
    }
}
```

**Example 2: Scheduled Task Manager**:

```java
@Service
public class ScheduledTaskManager {
    
    private ScheduledExecutorService scheduler;
    private final List<ScheduledFuture<?>> scheduledTasks = new ArrayList<>();
    
    @PostConstruct
    public void init() {
        // Create thread pool for scheduled tasks
        scheduler = Executors.newScheduledThreadPool(5);
        
        // Schedule tasks
        ScheduledFuture<?> task1 = scheduler.scheduleAtFixedRate(
            () -> System.out.println("Periodic task"),
            0, 1, TimeUnit.MINUTES
        );
        scheduledTasks.add(task1);
        
        System.out.println("✅ Scheduled tasks started");
    }
    
    @PreDestroy
    public void cleanup() {
        // Cancel all scheduled tasks
        scheduledTasks.forEach(task -> task.cancel(false));
        
        // Shutdown scheduler gracefully
        scheduler.shutdown();
        try {
            if (!scheduler.awaitTermination(5, TimeUnit.SECONDS)) {
                scheduler.shutdownNow();
            }
        } catch (InterruptedException e) {
            scheduler.shutdownNow();
        }
        
        System.out.println("✅ Scheduled tasks stopped");
    }
}
```

**Example 3: Cache Initialization**:

```java
@Service
public class ProductCacheService {
    
    @Autowired
    private ProductRepository productRepository;
    
    private Map<Long, Product> cache;
    
    @PostConstruct
    public void warmUpCache() {
        // Warm up cache with frequently accessed products
        System.out.println("🔥 Warming up product cache...");
        
        List<Product> popularProducts = productRepository
            .findTop100ByOrderByViewsDesc();
        
        cache = popularProducts.stream()
            .collect(Collectors.toMap(
                Product::getId,
                product -> product
            ));
        
        System.out.println("✅ Cache warmed up with " + cache.size() + " products");
    }
    
    @PreDestroy
    public void clearCache() {
        if (cache != null) {
            cache.clear();
            System.out.println("✅ Cache cleared");
        }
    }
    
    public Product getProduct(Long id) {
        return cache.computeIfAbsent(id, 
            productId -> productRepository.findById(productId).orElse(null)
        );
    }
}
```

```java
@Service
public class ExternalApiClient {
    
    @Value("${external.api.url}")
    private String apiUrl;
    
    @Value("${external.api.key}")
    private String apiKey;
    
    private RestTemplate restTemplate;

        System.out.println("✅ External API client initialized");
    }
    
    @PreDestroy
    public void cleanup() {
        // Revoke access token
        if (tokenManager != null) {
            tokenManager.revokeToken();
        }
        
        System.out.println("✅ External API client cleaned up");
    }
}
```

### Q5: AOP - How does @Transactional work internally?

**Short Answer**: @Transactional works through AOP proxies created by BeanPostProcessor. When you call a @Transactional method, you're actually calling a proxy that begins transaction → calls real method → commits/rolls back.

**Detailed Explanation**:

**Proxy Creation Process**:

```
1. Spring scans for @Transactional methods
2. BeanPostProcessor creates proxy for beans with @Transactional
3. Proxy intercepts method calls
4. Proxy delegates to TransactionInterceptor
5. TransactionInterceptor manages transaction lifecycle
```

**What Really Happens**:

```java
// Your code
@Service
public class OrderService {
    
    @Transactional
    public void createOrder(Order order) {
        orderRepository.save(order);
        inventoryService.decrementStock(order);
        // If exception thrown, transaction rolls back
    }
}

// What Spring creates internally (simplified)
public class OrderService$$SpringProxy extends OrderService {
    
    private OrderService target; // Real object
    private TransactionInterceptor txInterceptor;
    
    @Override
    public void createOrder(Order order) {
        TransactionInfo txInfo = null;
        try {
            // 1. Begin transaction
            txInfo = txInterceptor.createTransactionIfNecessary(
                "createOrder", OrderService.class
            );
            
            // 2. Call real method
            target.createOrder(order);
            
            // 3. Commit transaction
            txInterceptor.commitTransactionAfterReturning(txInfo);
            
        } catch (Exception ex) {
            // 4. Rollback on exception
            txInterceptor.completeTransactionAfterThrowing(txInfo, ex);
            throw ex;
        }
    }
}
```

**The Self-Invocation Problem**:

```java
@Service
public class OrderService {
    
    @Transactional
    public void createOrder(Order order) {
        // ... order logic ...
        
        this.sendEmail(order); // ❌ PROBLEM! No transaction!
    }
    
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void sendEmail(Order order) {
        // This transaction annotation is IGNORED!
        // Because 'this' is the real object, not the proxy
    }
}

// What actually happens:
// 1. Client calls proxy.createOrder()
// 2. Proxy starts transaction
// 3. Proxy calls target.createOrder() (real object)
// 4. Inside real object: this.sendEmail()
// 5. 'this' = real object, NOT proxy
// 6. sendEmail() called directly, NO proxy interception!
// 7. @Transactional on sendEmail() is ignored!
```

**Visual Explanation**:

```
Client → Proxy (TX starts) → Real Object.createOrder()
                                     ↓
                            this.sendEmail() ← Direct call!
                                     ↓
                            Real Object.sendEmail() ← NO PROXY!
```

```java
    }
}
```

**Transaction Interceptor Flow**:

```java
public class TransactionInterceptor extends TransactionAspectSupport {
    
    public Object invoke(MethodInvocation invocation) throws Throwable {
        // 1. Get transaction definition from @Transactional
        TransactionAttribute txAttr = getTransactionAttribute(
            invocation.getMethod(), 
            targetClass
        );
        
        // 2. Determine transaction manager
        PlatformTransactionManager tm = determineTransactionManager(txAttr);
        
        // 3. Get method signature for logging
        String joinpointIdentification = methodIdentification(
            invocation.getMethod(), 
            targetClass
        );
        
        // 4. Create transaction if needed
        TransactionInfo txInfo = createTransactionIfNecessary(
            tm, 
            txAttr, 
            joinpointIdentification
        );
        
        Object retVal;
        try {
            // 5. Invoke actual method
            retVal = invocation.proceed();
            
        } catch (Throwable ex) {
            // 6. Handle exception (rollback or commit)
            completeTransactionAfterThrowing(txInfo, ex);
            throw ex;
        } finally {
            cleanupTransactionInfo(txInfo);
        }
        
        // 7. Commit transaction
        commitTransactionAfterReturning(txInfo);
        return retVal;
    }
    
    protected void completeTransactionAfterThrowing(
            TransactionInfo txInfo, Throwable ex) {
        
        if (txInfo != null && txInfo.getTransactionStatus() != null) {
            // Check if we should rollback
            if (txInfo.transactionAttribute.rollbackOn(ex)) {
                txInfo.getTransactionManager().rollback(
                    txInfo.getTransactionStatus()
                );
            } else {
                // Exception doesn't trigger rollback (e.g., checked exception)
                txInfo.getTransactionManager().commit(
                    txInfo.getTransactionStatus()
                );
            }
        }
    }
}
```

**Rollback Rules**:

```java
@Service
public class PaymentService {
    
    // Default: Rollback on RuntimeException and Error
    @Transactional
    public void processPayment1(Payment payment) {
        // Rolls back on: RuntimeException, IllegalArgumentException, etc.
        // Does NOT rollback on: IOException, SQLException (checked exceptions)
    }
    
    // Explicit rollback rules
    @Transactional(rollbackFor = {Exception.class})
    public void processPayment2(Payment payment) throws Exception {
        // Rolls back on ANY exception (including checked)
    }
    
    // Don't rollback on specific exception
    @Transactional(noRollbackFor = {PaymentDeclinedException.class})
    public void processPayment3(Payment payment) {
        // Payment declined is NOT an error - don't rollback
    }
    
    // Multiple rules
    @Transactional(
        rollbackFor = {Exception.class},
        noRollbackFor = {PaymentDeclinedException.class, InsufficientFundsException.class}
    )
    public void processPayment4(Payment payment) throws Exception {
        // Rolls back on all exceptions EXCEPT payment declined or insufficient funds
    }
}
```

### Q6: When would you use prototype scope?

**Short Answer**: Use prototype scope when you need a new instance for each request, typically for stateful objects or objects with per-request configuration. Singleton (default) is preferred for stateless services.

**Scope Comparison**:

|Scope|Lifecycle|Instances|Use Case|
|---|---|---|---|
|singleton (default)|One instance per container|1|Stateless services, repositories|
|prototype|New instance per request|Many|Stateful objects, per-user data|
|request|One per HTTP request|Many|Web request-specific data|
|session|One per HTTP session|Many|User session data|
|application|One per ServletContext|1|Application-wide shared data|
|websocket|One per WebSocket|Many|WebSocket session data|

**Prototype Bean Example**:


```java
public interface Command {
    void execute();
    void undo();

@Component
@Scope("prototype")
public class SendEmailCommand implements Command {
    private String to;
    private String subject;
    private String body;
    private boolean executed = false;
    
    @Autowired
    private EmailService emailService; // Singleton injected

        this.subject = subject;
        this.body = body;
    }
    
    @Override

        emailService.sendEmail(to, subject, body);
        executed = true;
    }
    
    @Override
    public void undo() {
        if (executed) {
            emailService.sendEmail(to, "UNDO: " + subject, 
                "Previous email cancelled");
        }
    }
}

@Component
@Scope("prototype")
public class CreateOrderCommand implements Command {
    private Order order;
    private Long orderId;
    
    @Autowired
    private OrderRepository orderRepository;
    
    public void setOrder(Order order) {
        this.order = order;
    }
    
    @Override
    public void execute() {
        Order saved = orderRepository.save(order);
        this.orderId = saved.getId();
    }
    
    @Override
    public void undo() {
        if (orderId != null) {
            orderRepository.deleteById(orderId);
        }
    }
}

@Service
public class CommandExecutor {
    @Autowired
    private ApplicationContext context;
    
    private final Stack<Command> executedCommands = new Stack<>();
    
    public void execute(Command command) {
        command.execute();
        executedCommands.push(command);
    }
    
    public void undo() {
        if (!executedCommands.isEmpty()) {
            Command command = executedCommands.pop();
            command.undo();
        }
    }
    
    public void sendEmail(String to, String subject, String body) {
        SendEmailCommand command = context.getBean(SendEmailCommand.class);
        command.configure(to, subject, body);
        execute(command);
    }
}
```

**Use Case 3: Task Executor** (Per-task state):

```java
@Component
@Scope("prototype")
public class DataImportTask implements Runnable {
    private String fileName;
    private ImportConfig config;
    private ImportProgress progress;
    
    @Autowired
    private DataImportService importService; // Singleton
    
    @Autowired
    private ProgressTracker progressTracker; // Singleton

            progress.setStatus("COMPLETED");
        } catch (Exception e) {
            progress.setStatus("FAILED");
            progress.setError(e.getMessage());
        } finally {
            progressTracker.update(progress);
        }
    }
}
```
