# Testing - Deep Interview Questions & Answers

## Overview

This guide contains in-depth answers to Spring Boot testing interview questions, focusing on test annotations, mocking strategies, and integration testing approaches.

**Interview Tip**: Testing questions evaluate understanding of test scope, performance trade-offs, and ability to write maintainable, reliable tests.

## Testing Questions

### Q1: @SpringBootTest vs @WebMvcTest

**Short Answer**:

- **@SpringBootTest**: Full application context, all beans loaded, slower, for integration tests
- **@WebMvcTest**: Only web layer (controllers), no services/repositories, faster, for controller unit tests

**Detailed Comparison**:

|Aspect|@SpringBootTest|@WebMvcTest|
|---|---|---|
|**Context**|Full application context|Web layer only|
|**Beans loaded**|All beans (@Service, @Repository, @Component)|Only @Controller, @ControllerAdvice, filters|
|**Speed**|Slower (full startup)|Faster (minimal context)|
|**Database**|Real or embedded database|No database (mock repositories)|
|**Use case**|Integration tests|Controller unit tests|
|**External dependencies**|Can use real dependencies|Must mock all dependencies|
|**Configuration**|All @Configuration classes|Only web-related configuration|
|**Test scope**|End-to-end flow|Controller logic only|

### Q2: When to use mocks vs real beans

**Short Answer**: Use **mocks** for external dependencies, slow operations, and when testing in isolation. Use **real beans** for integration tests, testing interactions between components, and when behavior is complex. Follow the test pyramid: many unit tests with mocks, fewer integration tests with real beans.

**Decision Framework**:

```
Should I mock this dependency?

External service (HTTP, database, message queue)?
├─ Yes → MOCK (slow, unreliable)
└─ No → Continue

Does it have complex state or behavior?
├─ Yes → REAL (hard to mock correctly)
└─ No → Continue

Is it slow (>100ms)?
├─ Yes → MOCK (fast tests)
└─ No → Continue

Am I testing integration?
├─ Yes → REAL (test real interactions)
└─ No → MOCK (test in isolation)

Is it a core domain object?
├─ Yes → REAL (avoid mocking domain logic)
└─ No → MOCK is fine
```

```java
    }
```

```java
// 2. Spy - Real object with selective mocking
@Spy
private UserService userService;

doReturn(user).when(userService).findById(1L);
// Only stubbed methods mocked, rest use real implementation

// 3. Fake - Simple working implementation
public class FakeUserRepository implements UserRepository {

    public Optional<User> findById(Long id) {
        return Optional.ofNullable(users.get(id));
    }
    
    public User save(User user) {
        users.put(user.getId(), user);
        return user;
    }

// Real-ish behavior, but in-memory

// 4. Stub - Returns predetermined values
when(userRepository.findById(any())).thenReturn(Optional.of(user));
// Simple predetermined response
```

**Anti-Patterns to Avoid**:

```java
// ❌ Don't mock what you don't own
@Mock
private LocalDateTime dateTime; // Don't mock JDK classes

@Mock
private List<Order> orders; // Don't mock collections

@Mock
private String username; // Don't mock primitives/value objects

// ✅ Wrap external types if needed
public class Clock {
    public LocalDateTime now() {
        return LocalDateTime.now();
    }
}

@Mock
private Clock clock; // Mock your wrapper

// ❌ Don't over-mock (makes tests fragile)
@Mock private ServiceA serviceA;
@Mock private ServiceB serviceB;
@Mock private ServiceC serviceC;
@Mock private ServiceD serviceD;
@Mock private ServiceE serviceE;
// Too many mocks = testing mock setup, not real behavior

// ✅ Use integration test instead
@SpringBootTest
// Test real interactions

// ❌ Don't mock value objects
@Mock
private OrderRequest request; // Value object - use real instance

// ✅ Create real value objects
OrderRequest request = new OrderRequest("Widget", 5);

// ❌ Don't mock everything in integration tests
@SpringBootTest
public class Test {
    @MockBean private ServiceA a;
    @MockBean private ServiceB b;
    @MockBean private ServiceC c;
    // Defeats purpose of integration test
}

// ✅ Use real beans in integration tests
@SpringBootTest
public class Test {
    @Autowired private ServiceA a; // Real
    @Autowired private ServiceB b; // Real
    @MockBean private ExternalApi api; // Mock only external
}
```

**Test Pyramid Strategy**:

```
        ▲
       /E\      E2E Tests (few)
      /   \     - @SpringBootTest with real dependencies
     /     \    - Test complete user flows
    /   I   \   Integration Tests (some)
   /         \  - @SpringBootTest with test database
  /           \ - Test layer interactions
 /             \
/     Unit      \ Unit Tests (many)
─────────────────
                  - @WebMvcTest, @DataJpaTest
                  - Mock dependencies
                  - Test individual components
                  - Fast feedback

70% Unit Tests (mocks)
20% Integration Tests (real beans)
10% E2E Tests (real everything)
```

### Q3: Integration testing strategies

**Short Answer**: Integration testing strategies include: test containers for real dependencies, embedded databases for data layer, mock external services, transactional rollback for cleanup, test fixtures for data setup, and separate test configuration. Use @SpringBootTest with appropriate slices, isolate tests, and balance coverage with execution time.
