## Data & Transactions Questions

### Q1: N+1 problem and solutions

**Short Answer**: N+1 problem occurs when you fetch a list of entities (1 query) and then fetch related entities for each item (N queries), resulting in N+1 total queries. This severely impacts performance.

**The Problem Explained**:

```java
// Entity definitions
@Entity
public class Order {
    @Id
    private Long id;
    
    @ManyToOne(fetch = FetchType.LAZY) // Lazy loading
    private Customer customer;
    
    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
    private List<OrderItem> items;
}

@Entity
public class Customer {
    @Id
    private Long id;
    private String name;
}

@Entity
public class OrderItem {
    @Id
    private Long id;
    
    @ManyToOne
    private Order order;
    
    @ManyToOne
    private Product product;
}

// Repository
public interface OrderRepository extends JpaRepository<Order, Long> {
}

// Service with N+1 problem
@Service
public class OrderService {
    
    @Autowired
    private OrderRepository orderRepository;
    
    public void printOrders() {
        List<Order> orders = orderRepository.findAll(); // 1 query
        
        for (Order order : orders) {
            // N queries - one for each order!
            System.out.println("Customer: " + order.getCustomer().getName());
            
            // Another N queries - one for each order!
            System.out.println("Items: " + order.getItems().size());
        }
    }
}

// Generated SQL:
// 1. SELECT * FROM orders                    -- 1 query
// 2. SELECT * FROM customers WHERE id = 1    -- Query for order 1
// 3. SELECT * FROM customers WHERE id = 2    -- Query for order 2
// 4. SELECT * FROM customers WHERE id = 3    -- Query for order 3
// ... (N times)
// 5. SELECT * FROM order_items WHERE order_id = 1
// 6. SELECT * FROM order_items WHERE order_id = 2
// 7. SELECT * FROM order_items WHERE order_id = 3
// ... (N times)
//
// Total: 1 + N + N = 1 + 2N queries!
// For 100 orders: 1 + 200 = 201 queries!
```

**Solution 1: JOIN FETCH (Best for single collection)**:

```java
public interface OrderRepository extends JpaRepository<Order, Long> {
    
    // Fetch orders with customers in single query
    @Query("SELECT o FROM Order o JOIN FETCH o.customer")
    List<Order> findAllWithCustomer();
    
    // Fetch orders with items in single query
    @Query("SELECT DISTINCT o FROM Order o JOIN FETCH o.items")
    List<Order> findAllWithItems();
    
    // ❌ PROBLEM: Cannot fetch multiple collections with JOIN FETCH
    // This causes MultipleBagFetchException
    @Query("SELECT DISTINCT o FROM Order o " +
           "JOIN FETCH o.customer " +
           "JOIN FETCH o.items")
    List<Order> findAllWithCustomerAndItems(); // Will fail!
}

// Usage
@Service
public class OrderService {
    
    @Autowired
    private OrderRepository orderRepository;
    
    public void printOrders() {
        // Single query fetching orders and customers
        List<Order> orders = orderRepository.findAllWithCustomer();
        
        for (Order order : orders) {
            System.out.println("Customer: " + order.getCustomer().getName());
            // No additional query - customer already loaded!
        }
    }
}

// Generated SQL:
// SELECT o.*, c.* 
// FROM orders o 
// JOIN customers c ON o.customer_id = c.id
// 
// Total: 1 query (instead of 1 + N)
```

**Solution 2: Entity Graph (Best for multiple collections)**:

```java

// Generated SQL (uses LEFT JOIN):
// SELECT o.*, c.*, i.*
// FROM orders o
// LEFT JOIN customers c ON o.customer_id = c.id
// LEFT JOIN order_items i ON i.order_id = o.id
//
// Total: 1 query
```

**Solution 3: Batch Fetching**:

```java

// 1. SELECT * FROM orders                              -- 1 query
// 2. SELECT * FROM customers WHERE id IN (1,2,3,...10) -- Batch of 10
// 3. SELECT * FROM customers WHERE id IN (11,12,...20) -- Next batch
// 4. SELECT * FROM order_items WHERE order_id IN (1,2,3,...10)
// 5. SELECT * FROM order_items WHERE order_id IN (11,12,...20)
//
// Total: 1 + ceil(N/10) + ceil(N/10) queries
// For 100 orders: 1 + 10 + 10 = 21 queries (instead of 201!)
```

**Solution 4: @Fetch(FetchMode.SUBSELECT)**:

```java
//        SELECT id FROM orders

//

```


**Solution 6: Specification with Fetch Joins**:

```java
@Service
public class OrderService {
    
    @Autowired
    private OrderRepository orderRepository;
    
    public List<Order> findOrdersWithDetails(String customerName) {
        Specification<Order> spec = (root, query, cb) -> {
            // Prevent duplicates
            query.distinct(true);
            
            // Fetch joins
            root.fetch("customer", JoinType.LEFT);
            root.fetch("items", JoinType.LEFT);
            
            // Where clause
            if (customerName != null) {
                return cb.like(root.get("customer").get("name"), 
                             "%" + customerName + "%");
            }
            return null;
        };
        
        return orderRepository.findAll(spec);
    }
}
```

### Q2: Transaction propagation scenarios

**Short Answer**: Transaction propagation defines how transactions interact when one transactional method calls another. Spring supports 7 propagation types, each handling transaction boundaries differently.

**Propagation Types**:

|Propagation|Behavior|Use Case|
|---|---|---|
|REQUIRED (default)|Join existing or create new|Most common - normal operations|
|REQUIRES_NEW|Always create new, suspend existing|Independent operation that must complete regardless|
|NESTED|Nested transaction (savepoint)|Partial rollback scenarios|
|MANDATORY|Must have existing transaction|Enforce transaction presence|
|SUPPORTS|Use transaction if exists|Read operations that can be transactional|
|NOT_SUPPORTED|Execute without transaction|Long-running operations|
|NEVER|Must NOT have transaction|Strict non-transactional operations|

**1. REQUIRED (Default)**:

```java
@Service
public class OrderService {
    
    @Autowired
    private InventoryService inventoryService;
    
    @Transactional // REQUIRED by default
    public void createOrder(Order order) {
        orderRepository.save(order);
        
        // Uses same transaction
        inventoryService.decrementStock(order);
        
        // If exception here, EVERYTHING rolls back
        emailService.sendConfirmation(order);
    }
}

@Service
public class InventoryService {
    
    @Transactional(propagation = Propagation.REQUIRED)
    public void decrementStock(Order order) {
        // Uses existing transaction from createOrder
        for (OrderItem item : order.getItems()) {
            Inventory inv = inventoryRepository.findByProductId(item.getProductId());
            inv.setQuantity(inv.getQuantity() - item.getQuantity());
            inventoryRepository.save(inv);
        }
    }
}

// Transaction flow:
// 1. createOrder() starts transaction TX1
// 2. orderRepository.save() - within TX1
// 3. decrementStock() called
// 4. decrementStock() JOINS TX1 (doesn't create new)
// 5. inventoryRepository.save() - within TX1
// 6. emailService.sendConfirmation() - within TX1
// 7. If ANY step fails, ALL changes roll back
```

**2. REQUIRES_NEW**:

```java
@Service
public class OrderService {
    
    @Autowired
    private AuditService auditService;
    
    @Transactional
    public void createOrder(Order order) {
        orderRepository.save(order);
        
        // Log audit in SEPARATE transaction
        auditService.logOrderCreation(order);
        
        // If this throws exception, order rolls back
        // but audit log is ALREADY COMMITTED
        inventoryService.decrementStock(order);
    }
}

@Service
public class AuditService {
    
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void logOrderCreation(Order order) {
        // Creates NEW transaction TX2, suspends TX1
        AuditLog log = new AuditLog();
        log.setAction("ORDER_CREATED");
        log.setOrderId(order.getId());
        log.setTimestamp(LocalDateTime.now());
        auditLogRepository.save(log);
        
        // TX2 commits here (even if outer transaction fails)
    }
}

// Transaction flow:
// 1. createOrder() starts TX1
// 2. orderRepository.save() - within TX1
// 3. logOrderCreation() called
// 4. TX1 is SUSPENDED
// 5. NEW transaction TX2 created
// 6. auditLogRepository.save() - within TX2
// 7. TX2 COMMITS (independent of TX1)
// 8. TX1 RESUMES
// 9. If decrementStock() fails, TX1 rolls back
// 10. But audit log remains (TX2 already committed)
```

**3. NESTED**:

```java
@Service
public class OrderService {
    
    @Autowired
    private NotificationService notificationService;
    
    @Transactional
    public void createOrder(Order order) {
        orderRepository.save(order);
        
        try {
            // Try to send notifications
            notificationService.sendNotifications(order);
        } catch (Exception e) {
            // Notification failed, but order should succeed
            log.warn("Notifications failed", e);
        }
        
        // Order is still committed even if notifications failed
    }
}

@Service
public class NotificationService {
    
    @Transactional(propagation = Propagation.NESTED)
    public void sendNotifications(Order order) {
        // Creates SAVEPOINT within TX1
        
        // Send email
        emailRepository.save(new EmailLog(order));
        
        // Send SMS
        smsRepository.save(new SmsLog(order));
        
        // If exception here, rolls back to SAVEPOINT
        // Main transaction continues
    }
}

// Transaction flow:
// 1. createOrder() starts TX1
// 2. orderRepository.save() - within TX1
// 3. sendNotifications() called
// 4. SAVEPOINT created within TX1
// 5. emailRepository.save() - within TX1
// 6. smsRepository.save() - within TX1
// 7. If exception in sendNotifications():
//    - Rolls back to SAVEPOINT
//    - Order save is NOT affected
// 8. Main transaction continues and commits

// Note: NESTED requires JDBC driver support for savepoints
```

**4. MANDATORY**:

```java
@Service
public class OrderService {
    
    @Transactional
    public void createOrder(Order order) {
        orderRepository.save(order);
        
        // This works - transaction exists
        auditService.logAction("ORDER_CREATED");
    }
    
    public void someNonTransactionalMethod() {
        // This will FAIL - no transaction
        auditService.logAction("SOME_ACTION");
        // Throws: IllegalTransactionStateException: 
        // No existing transaction found for transaction marked with propagation 'mandatory'
    }
}

@Service
public class AuditService {
    
    @Transactional(propagation = Propagation.MANDATORY)
    public void logAction(String action) {
        // MUST be called within existing transaction
        // Enforces that caller manages transaction
        
        AuditLog log = new AuditLog();
        log.setAction(action);
        auditLogRepository.save(log);
    }
}
```

**5. SUPPORTS**:

```java
@Service
public class ReportService {
    
    @Transactional(propagation = Propagation.SUPPORTS)
    public List<Order> getOrders() {
        // If caller has transaction: participates
        // If caller has no transaction: executes without transaction
        return orderRepository.findAll();
    }
}

@Service
public class OrderService {
    
    @Transactional
    public void processOrders() {
        // getOrders() runs within this transaction
        List<Order> orders = reportService.getOrders();
    }
    
    public void viewOrders() {
        // getOrders() runs WITHOUT transaction
        List<Order> orders = reportService.getOrders();
    }
}
```

**6. NOT_SUPPORTED**:

```java
@Service
public class ReportService {
    
    @Transactional(propagation = Propagation.NOT_SUPPORTED)
    public byte[] generateLargeReport() {
        // Always executes WITHOUT transaction
        // Even if caller has transaction, it's suspended
        
        // Long-running operation
        List<Order> orders = orderRepository.findAll();
        
        // Process 1 million orders
        return generatePdfReport(orders);
    }
}

@Service
public class OrderService {
    
    @Transactional
    public void processWithReport() {
        orderRepository.save(new Order());
        
        // Transaction is SUSPENDED here
        reportService.generateLargeReport();
        // Transaction RESUMES here
        
        orderRepository.save(new Order());
    }
}

// Use case: Long-running operations that shouldn't hold database locks
```

**7. NEVER**:

```java
@Service
public class CacheService {
    
    @Transactional(propagation = Propagation.NEVER)
    public void updateCache(String key, Object value) {
        // MUST NOT be called within transaction
        // Throws exception if transaction exists
        
        redisTemplate.opsForValue().set(key, value);
    }
}

@Service
public class OrderService {
    
    @Transactional
    public void createOrder(Order order) {
        orderRepository.save(order);
        
        // This will FAIL
        cacheService.updateCache("order:" + order.getId(), order);
        // Throws: IllegalTransactionStateException:
        // Existing transaction found for transaction marked with propagation 'never'
    }
    
    public void createOrderFixed(Order order) {
        orderRepository.save(order);
        
        // After transaction commits
        cacheService.updateCache("order:" + order.getId(), order);
    }
}
```

**Real-World Scenario: E-commerce Order Processing**:

```java
@Service
public class OrderProcessingService {
    
    @Autowired
    private OrderService orderService;
    
    @Autowired
    private PaymentService paymentService;
    
    @Autowired
    private NotificationService notificationService;
    
    @Autowired
    private AuditService auditService;
    
    @Transactional
    public void processOrder(OrderRequest request) {
        // 1. Create order (REQUIRED - joins this transaction)
        Order order = orderService.createOrder(request);
        
        // 2. Process payment (REQUIRES_NEW - independent transaction)
        // Payment must succeed even if order processing fails later
        PaymentResult payment = paymentService.processPayment(order);
        
        if (!payment.isSuccessful()) {
            throw new PaymentFailedException("Payment failed");
        }
        
        // 3. Update order status
        order.setStatus("PAID");
        orderService.updateOrder(order);
        
        try {
            // 4. Send notifications (NESTED - can fail independently)
            notificationService.sendOrderConfirmation(order);
        } catch (Exception e) {
            // Log but don't fail the order
            log.warn("Failed to send notifications", e);
        }
        
        // 5. Log audit (REQUIRES_NEW - must be recorded)
        auditService.logOrderProcessed(order);
    }
}

@Service
public class OrderService {
    @Transactional(propagation = Propagation.REQUIRED)
    public Order createOrder(OrderRequest request) { ... }
    
    @Transactional(propagation = Propagation.REQUIRED)
    public Order updateOrder(Order order) { ... }
}

@Service
public class PaymentService {
    // Independent transaction - commits even if order fails
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public PaymentResult processPayment(Order order) { ... }
}

@Service
public class NotificationService {
    // Nested - can roll back without affecting main transaction
    @Transactional(propagation = Propagation.NESTED)
    public void sendOrderConfirmation(Order order) { ... }
}

@Service
public class AuditService {
    // Independent transaction - audit must be recorded
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void logOrderProcessed(Order order) { ... }
}
```

**Propagation Decision Tree**:

```
Need transaction?
│
├─ Yes
│  │
│  ├─ Can reuse existing?
│  │  │
│  │  ├─ Yes → REQUIRED (default)
│  │  │
│  │  └─ No, need independent → REQUIRES_NEW
│  │     (e.g., audit logs, payment processing)
│  │
│  └─ Must have existing? → MANDATORY
│     (e.g., method should only be called transactionally)
│
└─ No
   │
   ├─ Optional → SUPPORTS
   │  (e.g., read operations)
   │
   ├─ Must suspend → NOT_SUPPORTED
   │  (e.g., long-running reports)
   │
   └─ Must not exist → NEVER
      (e.g., cache operations)
```

### Q3: Optimistic vs pessimistic locking

**Short Answer**:

- **Optimistic locking**: Assumes conflicts are rare, checks for conflicts at commit time using version field
- **Pessimistic locking**: Assumes conflicts are common, locks database rows immediately to prevent concurrent access

**Optimistic Locking**:

```java
@Entity
public class Product {
    @Id
    private Long id;
    
    private String name;
    private BigDecimal price;
    private int quantity;
    
    @Version // Optimistic locking
    private Long version;
    
    // When updating, JPA automatically:
    // 1. Reads current version
    // 2. Updates with WHERE clause including version
    // 3. Increments version
    // 4. If version doesn't match (someone else updated), throws OptimisticLockException
}

// Generated SQL:
// UPDATE products 
// SET name = ?, price = ?, quantity = ?, version = version + 1
// WHERE id = ? AND version = ?
//
// If WHERE returns 0 rows → OptimisticLockException
```

**Optimistic Locking Example**:

```java
@Service
public class ProductService {
    
    @Autowired
    private ProductRepository productRepository;
    
    @Transactional
    public void updatePrice(Long productId, BigDecimal newPrice) {
        Product product = productRepository.findById(productId)
            .orElseThrow();
        
        // Current version: 1
        product.setPrice(newPrice);
        
        // Save executes:
        // UPDATE products SET price = ?, version = 2 
        // WHERE id = ? AND version = 1
        productRepository.save(product);
    }
    
    // Concurrent scenario:
    // Thread 1: Reads product (version=1)
    // Thread 2: Reads product (version=1)
    // Thread 1: Updates price, version becomes 2 ✓
    // Thread 2: Tries to update, version is now 2 (not 1)
    //          UPDATE returns 0 rows → OptimisticLockException ✗
}
```

**Handling OptimisticLockException**:

```java
@Service
public class ProductService {
    
    @Autowired
    private ProductRepository productRepository;
    
    @Retryable(
        value = OptimisticLockException.class,
        maxAttempts = 3,
        backoff = @Backoff(delay = 100)
    )
    @Transactional
    public void updatePriceWithRetry(Long productId, BigDecimal newPrice) {
        Product product = productRepository.findById(productId)
            .orElseThrow();
        product.setPrice(newPrice);
        productRepository.save(product);
    }
    
    // Or manual retry
    @Transactional
    public void updatePriceManualRetry(Long productId, BigDecimal newPrice) {
        int maxRetries = 3;
        int attempt = 0;
        
        while (attempt < maxRetries) {
            try {
                Product product = productRepository.findById(productId)
                    .orElseThrow();
                product.setPrice(newPrice);
                productRepository.save(product);
                return; // Success
                
            } catch (OptimisticLockException e) {
                attempt++;
                if (attempt >= maxRetries) {
                    throw new ConcurrentUpdateException(
                        "Failed to update after " + maxRetries + " attempts", e
                    );
                }
                // Wait before retry
                try {
                    Thread.sleep(100 * attempt);
                } catch (InterruptedException ie) {
                    Thread.currentThread().interrupt();
                    throw new RuntimeException(ie);
                }
            }
        }
    }
}
```

**Pessimistic Locking**:

```java
public interface ProductRepository extends JpaRepository<Product, Long> {
    
    // Pessimistic write lock (exclusive)
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT p FROM Product p WHERE p.id = :id")
    Optional<Product> findByIdWithLock(@Param("id") Long id);
    
    // Pessimistic read lock (shared)
    @Lock(LockModeType.PESSIMISTIC_READ)
    @Query("SELECT p FROM Product p WHERE p.id = :id")
    Optional<Product> findByIdWithReadLock(@Param("id") Long id);
    
    // With timeout
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @QueryHints({@QueryHint(name = "javax.persistence.lock.timeout", value = "3000")})
    @Query("SELECT p FROM Product p WHERE p.id = :id")
    Optional<Product> findByIdWithLockAndTimeout(@Param("id") Long id);
}

// Generated SQL (PostgreSQL):
// SELECT * FROM products WHERE id = ? FOR UPDATE
//
// FOR UPDATE: Locks the row(s) until transaction commits
```

**Pessimistic Locking Example**:

```java
@Service
public class InventoryService {
    
    @Autowired
    private ProductRepository productRepository;
    
    @Transactional
    public void decrementStock(Long productId, int quantity) {
        // Lock the row immediately
        Product product = productRepository.findByIdWithLock(productId)
            .orElseThrow();
        
        // No one else can modify this product until transaction commits
        if (product.getQuantity() < quantity) {
            throw new InsufficientStockException();
        }
        
        product.setQuantity(product.getQuantity() - quantity);
        productRepository.save(product);
        
        // Lock released when transaction commits
    }
    
    // Concurrent scenario:
    // Thread 1: Locks product (SELECT ... FOR UPDATE)
    // Thread 2: Tries to lock same product → WAITS
    // Thread 1: Updates quantity, commits → lock released
    // Thread 2: Gets lock, reads updated quantity
    // Thread 2: Updates quantity, commits → lock released
}
```

**Lock Modes**:

|Lock Mode|Behavior|SQL|Use Case|
|---|---|---|---|
|PESSIMISTIC_READ|Shared lock|SELECT ... FOR SHARE|Multiple readers, prevent writers|
|PESSIMISTIC_WRITE|Exclusive lock|SELECT ... FOR UPDATE|Exclusive access|
|PESSIMISTIC_FORCE_INCREMENT|Exclusive lock + version increment|SELECT ... FOR UPDATE|Lock and track version|

**Lock Timeout Handling**:

```java
@Service
public class OrderService {
    
    @Autowired
    private ProductRepository productRepository;
    
    @Transactional
    public void reserveProduct(Long productId, int quantity) {
        try {
            // Try to lock with 3-second timeout
            Product product = productRepository
                .findByIdWithLockAndTimeout(productId)
                .orElseThrow();
            
            product.setQuantity(product.getQuantity() - quantity);
            productRepository.save(product);
            
        } catch (PessimisticLockException e) {
            throw new ProductLockedException(
                "Product is being modified by another user. Please try again.", e
            );
        } catch (LockTimeoutException e) {
            throw new ProductUnavailableException(
                "Product is currently unavailable. Please try again later.", e
            );
        }
    }
}
```

**Comparison**:

|Aspect|Optimistic|Pessimistic|
|---|---|---|
|**When to use**|Low contention|High contention|
|**Performance**|Better (no locks)|Worse (locks held)|
|**Scalability**|Better|Worse|
|**Consistency**|Eventual|Immediate|
|**Failure mode**|Exception at commit|Wait or timeout|
|**Retry needed**|Yes|No|
|**Database load**|Lower|Higher|
|**Deadlock risk**|None|Possible|

**Real-World Example: Flash Sale**:

```java
@Service
public class FlashSaleService {

    private ProductRepository productRepository;

    @Transactional
    public void purchaseWithOptimistic(Long productId, int quantity) {

            .orElseThrow();
        
        // Version: 1
        if (product.getQuantity() < quantity) {
            throw new InsufficientStockException();
        }
        
        product.setQuantity(product.getQuantity() - quantity);
        
        // 1000 concurrent requests
        // Only first one succeeds
        // Other 999 get OptimisticLockException
        productRepository.save(product);
    }
    
    // ✅ Pessimistic locking - handles high contention
    @Transactional
    public void purchaseWithPessimistic(Long productId, int quantity) {
        // Lock the row immediately
        Product product = productRepository.findByIdWithLock(productId)
            .orElseThrow();
        
        if (product.getQuantity() < quantity) {
            throw new InsufficientStockException();
        }
        
        product.setQuantity(product.getQuantity() - quantity);
        productRepository.save(product);
        
        // Requests processed serially
        // Each waits for previous to complete
        // All succeed (until stock runs out)
    }
    
    // ✅ Best: Optimistic with retry for moderate contention
    @Retryable(
        value = OptimisticLockException.class,
        maxAttempts = 5,
        backoff = @Backoff(delay = 50, multiplier = 2)
    )
    @Transactional
    public void purchaseWithRetry(Long productId, int quantity) {
        Product product = productRepository.findById(productId)
            .orElseThrow();
        
        if (product.getQuantity() < quantity) {
            throw new InsufficientStockException();
        }
        
        product.setQuantity(product.getQuantity() - quantity);
        productRepository.save(product);
    }
}
```

**Deadlock Prevention with Pessimistic Locking**:

```java
@Service
public class AccountService {
    
    // ❌ Can cause deadlock
    @Transactional
    public void transferMayDeadlock(Long fromId, Long toId, BigDecimal amount) {
        Account from = accountRepository.findByIdWithLock(fromId).orElseThrow();
        Account to = accountRepository.findByIdWithLock(toId).orElseThrow();
        
        // If Thread 1: transfer(A, B) locks A then tries B
        // And Thread 2: transfer(B, A) locks B then tries A
        // → DEADLOCK
    }
    
    // ✅ Prevent deadlock - always lock in same order
    @Transactional
    public void transferSafe(Long fromId, Long toId, BigDecimal amount) {
        // Always lock accounts in ascending ID order
        Long firstId = Math.min(fromId, toId);
        Long secondId = Math.max(fromId, toId);
        
        Account first = accountRepository.findByIdWithLock(firstId).orElseThrow();
        Account second = accountRepository.findByIdWithLock(secondId).orElseThrow();
        
        // Determine which is from/to
        Account from = first.getId().equals(fromId) ? first : second;
        Account to = first.getId().equals(toId) ? first : second;
        
        from.setBalance(from.getBalance().subtract(amount));
        to.setBalance(to.getBalance().add(amount));
        
        accountRepository.save(from);
        accountRepository.save(to);
    }
}
```

### Q4: When would @Transactional not work?

**Short Answer**: @Transactional doesn't work due to: self-invocation (calling from same class), method not public, checked exceptions (by default), proxy limitations, or wrong transaction manager configuration.

**Problem 1: Self-Invocation**:

```java
@Service
public class OrderService {
    
    @Autowired
    private OrderRepository orderRepository;
    
    // Public method - not transactional
    public void processOrder(Order order) {
        // Some logic...
        
        // ❌ @Transactional IGNORED - self-invocation!
        this.saveOrder(order);
    }
    
    @Transactional
    public void saveOrder(Order order) {
        // Transaction doesn't start!
        // Called via 'this', not proxy
        orderRepository.save(order);
    }
}

// Why it fails:
// 1. Spring creates proxy for OrderService
// 2. processOrder() called → goes through proxy ✓
// 3. Inside processOrder(): this.saveOrder()
// 4. 'this' = real object, NOT proxy
// 5. Proxy not involved → @Transactional ignored ✗
```

**Solutions for Self-Invocation**:

```java
// Solution 1: Separate service
@Service
public class OrderService {
    @Autowired
    private OrderTransactionService transactionService;
    
    public void processOrder(Order order) {
        // Calls another service → goes through proxy ✓
        transactionService.saveOrder(order);
    }
}

@Service
public class OrderTransactionService {
    @Transactional
    public void saveOrder(Order order) {
        orderRepository.save(order);
    }
}

// Solution 2: Inject self
@Service
public class OrderService {
    @Autowired
    private OrderService self; // Injected proxy
    
    public void processOrder(Order order) {
        // Use injected proxy ✓
        self.saveOrder(order);
    }
    
    @Transactional
    public void saveOrder(Order order) {
        orderRepository.save(order);
    }
}

// Solution 3: ApplicationContext
@Service
public class OrderService {
    @Autowired
    private ApplicationContext context;
    
    public void processOrder(Order order) {
        // Get proxy from context ✓
        OrderService proxy = context.getBean(OrderService.class);
        proxy.saveOrder(order);
    }
    
    @Transactional
    public void saveOrder(Order order) {
        orderRepository.save(order);
    }
}

// Solution 4: AspectJ (compile-time weaving)
@EnableTransactionManagement(mode = AdviceMode.ASPECTJ)
@Configuration
public class AppConfig {
    // No proxies - bytecode modified
    // Self-invocation works ✓
}
```

**Problem 2: Non-Public Methods**:

```java
@Service
public class OrderService {
    
    // ❌ @Transactional ignored - method not public
    @Transactional
    private void saveOrder(Order order) {
        orderRepository.save(order);
    }
    
    // ❌ @Transactional ignored - method not public
    @Transactional
    protected void updateOrder(Order order) {
        orderRepository.save(order);
    }
    
    // ✅ @Transactional works - method is public
    @Transactional
    public void deleteOrder(Long id) {
        orderRepository.deleteById(id);
    }
}

// Why: Spring AOP proxies only intercept public methods
// Solution: Make method public or use AspectJ
```

**Problem 3: Checked Exceptions (Default)**:

```java
@Service
public class OrderService {
    
    // ❌ Transaction NOT rolled back on checked exception
    @Transactional
    public void processOrder(Order order) throws Exception {
        orderRepository.save(order);
        
        // Throws checked exception
        if (order.getTotal().compareTo(BigDecimal.ZERO) <= 0) {
            throw new Exception("Invalid order total");
        }
        
        // Transaction COMMITS despite exception!
        // Order is saved in database
    }
    
    // ✅ Solution 1: Specify rollbackFor
    @Transactional(rollbackFor = Exception.class)
    public void processOrderFixed1(Order order) throws Exception {
        orderRepository.save(order);
        
        if (order.getTotal().compareTo(BigDecimal.ZERO) <= 0) {
            throw new Exception("Invalid order total");
        }
        
        // Transaction rolls back ✓
    }
    
    // ✅ Solution 2: Throw unchecked exception
    @Transactional
    public void processOrderFixed2(Order order) {
        orderRepository.save(order);
        
        if (order.getTotal().compareTo(BigDecimal.ZERO) <= 0) {
            throw new IllegalArgumentException("Invalid order total");
        }
        
        // Transaction rolls back ✓
    }
}

// Default rollback behavior:
// - RuntimeException → Rolls back ✓
// - Error → Rolls back ✓
// - Checked Exception → Commits ✗
```

**Problem 4: Catching Exceptions**:

```java
@Service
public class OrderService {
    
    // ❌ Transaction NOT rolled back - exception caught
    @Transactional
    public void processOrder(Order order) {
        orderRepository.save(order);
        
        try {
            inventoryService.decrementStock(order);
        } catch (InsufficientStockException e) {
            // Exception caught and swallowed
            log.error("Insufficient stock", e);
        }
        
        // Transaction commits even though stock wasn't decremented!
        // Order saved but inventory not updated → inconsistent state
    }
    
    // ✅ Solution 1: Rethrow exception
    @Transactional
    public void processOrderFixed1(Order order) {
        orderRepository.save(order);
        
        try {
            inventoryService.decrementStock(order);
        } catch (InsufficientStockException e) {
            log.error("Insufficient stock", e);
            throw new OrderProcessingException("Cannot process order", e);
        }
        
        // Transaction rolls back ✓
    }
    
    // ✅ Solution 2: Mark for rollback manually
    @Transactional
    public void processOrderFixed2(Order order) {
        orderRepository.save(order);
        
        try {
            inventoryService.decrementStock(order);
        } catch (InsufficientStockException e) {
            log.error("Insufficient stock", e);
            
            // Manually mark transaction for rollback
            TransactionAspectSupport.currentTransactionStatus()
                .setRollbackOnly();
        }
        
        // Transaction rolls back ✓
    }
    
    // ✅ Solution 3: Handle exception properly
    @Transactional
    public OrderResult processOrderFixed3(Order order) {
        try {
            orderRepository.save(order);
            inventoryService.decrementStock(order);
            return OrderResult.success(order);
            
        } catch (InsufficientStockException e) {
            // Let transaction roll back naturally
            throw e;
        }
    }
}
```

**Problem 5: Wrong Transaction Manager**:

```java
// Multiple data sources
@Configuration

@Service

        // Tries to use secondaryDataSource

    }
```

### Q5: Second-level cache strategy

**Short Answer**: Second-level cache (L2 cache) is a session-factory level cache shared across all sessions, caching entities, collections, and queries. It improves performance by reducing database queries but requires careful strategy selection and configuration.

**Cache Levels**:

```
┌─────────────────────────────────────────┐
│         Application Layer               │
│                                         │
│  ┌──────────────────────────────────┐  │
│  │    Query Cache (Optional)         │  │
│  │    Caches query results          │  │
│  └──────────────────────────────────┘  │
│                  │                      │
│  ┌──────────────────────────────────┐  │
│  │  Second-Level Cache (L2)         │  │
│  │  (SessionFactory scope)          │  │
│  │  Shared across all sessions      │  │
│  └──────────────────────────────────┘  │
│                  │                      │
│  ┌──────────────────────────────────┐  │
│  │  First-Level Cache (L1)          │  │
│  │  (Session/EntityManager scope)   │  │
│  │  Always enabled                  │  │
│  └──────────────────────────────────┘  │
│                  │                      │
└──────────────────┼──────────────────────┘
                   │
           ┌───────────────┐
           │   Database    │
           └───────────────┘
```

**Enabling Second-Level Cache**:

**Step 1: Add Dependencies**:

```xml
<!-- Hibernate Second-Level Cache -->
<dependency>
    <groupId>org.hibernate</groupId>
    <artifactId>hibernate-jcache</artifactId>
</dependency>

<!-- EHCache 3 (JSR-107 compliant) -->
<dependency>
    <groupId>org.ehcache</groupId>
    <artifactId>ehcache</artifactId>
</dependency>
```

**Step 2: Configure in application.yml**:

```yaml
spring:
  jpa:
    properties:
      hibernate:
        cache:
          use_second_level_cache: true
          use_query_cache: true
          region:
            factory_class: org.hibernate.cache.jcache.JCacheRegionFactory
        javax:
          cache:
            provider: org.ehcache.jsr107.EhcacheCachingProvider
            uri: classpath:ehcache.xml
```

**Step 3: EHCache Configuration** (ehcache.xml):

```xml
<config xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
        xmlns="http://www.ehcache.org/v3">
    
    <!-- Default cache configuration -->
    <cache-template name="default">
        <expiry>
            <ttl unit="minutes">10</ttl>
        </expiry>
        <heap unit="entries">1000</heap>
    </cache-template>
    
    <!-- Entity cache -->
    <cache alias="com.example.entity.Product" uses-template="default">
        <expiry>
            <ttl unit="hours">1</ttl>
        </expiry>
        <heap unit="entries">5000</heap>
    </cache>
    
    <!-- Collection cache -->
    <cache alias="com.example.entity.Product.reviews" uses-template="default">
        <expiry>
            <ttl unit="minutes">30</ttl>
        </expiry>
        <heap unit="entries">2000</heap>
    </cache>
    
    <!-- Query cache -->
    <cache alias="default-query-results-region" uses-template="default">
        <expiry>
            <ttl unit="minutes">5</ttl>
        </expiry>
    </cache>
    
    <!-- Update timestamps cache (required for query cache) -->
    <cache alias="default-update-timestamps-region">
        <expiry>
            <none/>
        </expiry>
        <heap unit="entries">5000</heap>
    </cache>
</config>
```

**Cache Strategies**:

**1. READ_ONLY** (Immutable entities):

```java
@Entity
@Cacheable
@org.hibernate.annotations.Cache(usage = CacheConcurrencyStrategy.READ_ONLY)
public class Country {
    @Id
    private Long id;
    private String code;
    private String name;
    
    // Countries don't change - perfect for READ_ONLY
}

// Use case: Reference data that never changes
// - Countries, currencies
// - Static configuration
// - Historical data

// Pros:
// - Best performance
// - No locking overhead
// - No synchronization needed

// Cons:
// - Cannot update entities
// - Must invalidate cache manually if data changes
```

**2. READ_WRITE** (Most common):

```java
@Entity
@Cacheable
@org.hibernate.annotations.Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
public class Product {
    @Id
    private Long id;
    private String name;
    private BigDecimal price;
    
    @OneToMany(mappedBy = "product")
    @org.hibernate.annotations.Cache(
        usage = CacheConcurrencyStrategy.READ_WRITE
    )
    private List<Review> reviews;
}

// Use case: Entities that are read frequently, updated occasionally
// - Products, categories
// - User profiles
// - Configuration settings

// Behavior:
// - Read: Returns from cache if available
// - Update: Invalidates cache entry
// - Soft locks prevent dirty reads during updates

// Pros:
// - Balances performance and consistency
// - Handles concurrent reads/writes

// Cons:
// - Slight overhead from soft locking
// - Brief inconsistency window during updates
```

**3. NONSTRICT_READ_WRITE** (Eventual consistency OK):

```java
@Entity
@Cacheable
@org.hibernate.annotations.Cache(usage = CacheConcurrencyStrategy.NONSTRICT_READ_WRITE)
public class BlogPost {
    @Id
    private Long id;
    private String title;
    private String content;
    private int viewCount;
    
    // View count doesn't need strict consistency
}

// Use case: Data where brief inconsistency is acceptable
// - Blog posts
// - Comment counts
// - Non-critical statistics

// Behavior:
// - No locking
// - Cache invalidated AFTER transaction commits
// - Possible stale reads

// Pros:
// - Better performance than READ_WRITE
// - No locking overhead

// Cons:
// - Can return stale data
// - Lost updates possible (though rare)
```

**4. TRANSACTIONAL** (Strict consistency):

```java
@Entity
@Cacheable
@org.hibernate.annotations.Cache(usage = CacheConcurrencyStrategy.TRANSACTIONAL)
public class BankAccount {
    @Id
    private Long id;
    private BigDecimal balance;
    private String accountNumber;
    
    // Financial data requires strict consistency
}

// Use case: Critical data requiring full ACID properties
// - Financial transactions
// - Inventory counts
// - Booking systems

// Requirements:
// - JTA transaction manager
// - XA-compliant cache provider

// Pros:
// - Full consistency
// - No stale reads
// - Coordinated with database transactions

// Cons:
// - Worst performance
// - Complex setup
// - Not all cache providers support it
```

**Query Cache**:

```java
public interface ProductRepository extends JpaRepository<Product, Long> {
    
    // Enable query cache for specific query
    @QueryHints({
        @QueryHint(name = "org.hibernate.cacheable", value = "true"),
        @QueryHint(name = "org.hibernate.cacheRegion", value = "productQueries")
    })
    @Query("SELECT p FROM Product p WHERE p.category = :category")
    List<Product> findByCategory(@Param("category") String category);
    
    // With cache mode
    @QueryHints({
        @QueryHint(name = "org.hibernate.cacheable", value = "true"),
        @QueryHint(name = "org.hibernate.cacheMode", value = "NORMAL")
    })
    List<Product> findByPriceRange(BigDecimal min, BigDecimal max);
}

// Query cache stores:
// - Query string + parameters → List of entity IDs
// - Actual entities still come from entity cache

// Example:
// 1. Query: "SELECT p FROM Product WHERE category = 'Electronics'"
// 2. Query cache: Returns [1, 5, 10, 15] (entity IDs)
// 3. Entity cache: Returns Product(1), Product(5), etc.
```

```java
@Service
public class CacheService {
    
    @PersistenceContext
    private EntityManager entityManager;
    
    // Evict specific entity from cache
    public void evictEntity(Class<?> entityClass, Long id) {
        entityManager.getEntityManagerFactory()
            .getCache()
            .evict(entityClass, id);
    }
    
    // Evict all entities of a type
    public void evictAllEntities(Class<?> entityClass) {
        entityManager.getEntityManagerFactory()
            .getCache()
            .evict(entityClass);
    }
    
    // Clear entire cache
    public void evictAll() {
        entityManager.getEntityManagerFactory()
            .getCache()
            .evictAll();
    }
    
    // Check if entity is cached
    public boolean contains(Class<?> entityClass, Long id) {
        return entityManager.getEntityManagerFactory()
            .getCache()
            .contains(entityClass, id);
    }
}

@Service
```

**Cache Monitoring**:

```java
@Service
public class CacheMonitoringService {
    
    @PersistenceContext
    private EntityManager entityManager;
    
    public CacheStatistics getCacheStatistics() {
        SessionFactory sessionFactory = entityManager
            .getEntityManagerFactory()
            .unwrap(SessionFactory.class);
        
        Statistics stats = sessionFactory.getStatistics();
        
        CacheStatistics cacheStats = new CacheStatistics();
        
        // Second-level cache stats
        cacheStats.setSecondLevelCacheHitCount(stats.getSecondLevelCacheHitCount());
        cacheStats.setSecondLevelCacheMissCount(stats.getSecondLevelCacheMissCount());
        cacheStats.setSecondLevelCachePutCount(stats.getSecondLevelCachePutCount());
        
        // Hit ratio
        long hits = stats.getSecondLevelCacheHitCount();
        long total = hits + stats.getSecondLevelCacheMissCount();
        double hitRatio = total > 0 ? (double) hits / total * 100 : 0;
        cacheStats.setHitRatio(hitRatio);
        
        // Query cache stats
        cacheStats.setQueryCacheHitCount(stats.getQueryCacheHitCount());
        cacheStats.setQueryCacheMissCount(stats.getQueryCacheMissCount());
        cacheStats.setQueryCachePutCount(stats.getQueryCachePutCount());
        
        // Region statistics
        String[] regionNames = stats.getSecondLevelCacheRegionNames();
        for (String regionName : regionNames) {
            CacheRegionStatistics regionStats = stats.getCacheRegionStatistics(regionName);
            
            RegionInfo info = new RegionInfo();
            info.setName(regionName);
            info.setHitCount(regionStats.getHitCount());
            info.setMissCount(regionStats.getMissCount());
            info.setElementCountInMemory(regionStats.getElementCountInMemory());
            info.setSizeInMemory(regionStats.getSizeInMemory());
            
            cacheStats.addRegion(info);
        }
        
        return cacheStats;
    }
    
    @Scheduled(fixedRate = 60000) // Every minute
    public void logCacheStatistics() {
        CacheStatistics stats = getCacheStatistics();
        log.info("Cache hit ratio: {}%", stats.getHitRatio());
        log.info("Cache hits: {}, misses: {}", 
            stats.getSecondLevelCacheHitCount(),
            stats.getSecondLevelCacheMissCount());
    }
}
```

**Common Pitfalls**:

**Pitfall 1: Caching associations without caching entity**:

```java
// ❌ Wrong - collection cached but entity not
@Entity
public class Product {
    @Id
    private Long id;
    
    @OneToMany(mappedBy = "product")
    @org.hibernate.annotations.Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
    private List<Review> reviews; // Cached
}

// ✅ Correct - cache both
@Entity
@Cacheable
@org.hibernate.annotations.Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
public class Product {
    @Id
    private Long id;
    
    @OneToMany(mappedBy = "product")
    @org.hibernate.annotations.Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
    private List<Review> reviews;
}
```

**Pitfall 2: Query cache without entity cache**:

```java
// Query cache stores entity IDs
// If entities aren't cached, still queries database for each entity

// ❌ Wrong - query cache alone
@QueryHints(@QueryHint(name = "org.hibernate.cacheable", value = "true"))
List<Product> findAll();

// ✅ Correct - entity cache + query cache
@Entity
@Cacheable
@Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
public class Product { }

@QueryHints(@QueryHint(name = "org.hibernate.cacheable", value = "true"))
List<Product> findAll();
```

**Pitfall 3: Caching large collections**:

```java
// ❌ Wrong - caching huge collection
@Entity
@Cacheable
public class User {
    @OneToMany(mappedBy = "user")
    @Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
    private List<Order> orders; // Could be thousands!
}

// ✅ Better - don't cache large collections
@Entity
@Cacheable
public class User {
    @OneToMany(mappedBy = "user")
    // No @Cache annotation
    private List<Order> orders;
}

// Or use pagination
List<Order> findOrdersByUser(User user, Pageable pageable);
```

**Best Practices**:

```
1. ✅ Cache immutable reference data (READ_ONLY)
   - Countries, currencies, static config

2. ✅ Cache frequently read, rarely updated (READ_WRITE)
   - Products, categories, user profiles

3. ✅ Monitor cache hit ratios
   - Aim for >80% hit ratio
   - If lower, reconsider cache strategy

4. ✅ Set appropriate TTL
   - Short TTL: Frequently changing data
   - Long TTL: Static data

5. ✅ Cache entity + associations together
   - Consistent caching strategy

6. ✅ Don't cache everything
   - Large entities/collections
   - Frequently updated data
   - User-specific data with low reuse

7. ✅ Use cache regions for fine-grained control
   - Different TTL per entity type

8. ✅ Enable statistics in development
   - Analyze what should be cached

9. ✅ Clear cache after bulk updates
   - Prevent stale data

10. ✅ Consider distributed cache for clustered apps
    - Hazelcast, Redis instead of EHCache
```
