## Overview

Reactive programming is a paradigm for building non-blocking, asynchronous, event-driven applications that can handle massive amounts of concurrent connections efficiently. Spring WebFlux provides reactive programming support using Project Reactor.

**Key Concept**: Traditional (imperative) programming blocks threads while waiting for I/O. Reactive programming uses non-blocking I/O and event-driven architecture, allowing one thread to handle thousands of concurrent requests.

## 1. Reactive Streams

### Core Interfaces

```
┌─────────────────────────────────────────────────────────────┐
│              Reactive Streams Specification                  │
└─────────────────────────────────────────────────────────────┘

Four Core Interfaces:

1. Publisher<T>
   - Produces items
   - Can have 0 to N subscribers
   - Methods: subscribe(Subscriber)

2. Subscriber<T>
   - Consumes items from Publisher
   - Methods:
     - onSubscribe(Subscription)    → Called once, receives subscription
     - onNext(T)                    → Called for each item
     - onComplete()                 → Called when done (no more items)
     - onError(Throwable)           → Called on error

3. Subscription
   - Represents connection between Publisher and Subscriber
   - Methods:
     - request(long n)              → Request n items (backpressure)
     - cancel()                     → Cancel subscription

4. Processor<T, R>
   - Both Publisher and Subscriber
   - Transforms items (T → R)
```

```java
/**
 * Reactive Streams Flow
 */

// Step 1: Create Publisher
Publisher<String> publisher = subscriber -> {
    // Called when subscriber subscribes
    subscriber.onSubscribe(new Subscription() {
        private boolean canceled = false;
        
        @Override
        public void request(long n) {
            if (!canceled) {
                // Produce n items
                for (int i = 0; i < n; i++) {
                    subscriber.onNext("Item " + i);
                }
                subscriber.onComplete();
            }
        }
        
        @Override
        public void cancel() {
            canceled = true;
        }
    });
};

// Step 2: Create Subscriber
Subscriber<String> subscriber = new Subscriber<>() {
    private Subscription subscription;
    
    @Override
    public void onSubscribe(Subscription s) {
        this.subscription = s;
        subscription.request(5);  // Request 5 items (backpressure)
    }
    
    @Override
    public void onNext(String item) {
        System.out.println("Received: " + item);
        // Can request more: subscription.request(1);
    }
    
    @Override
    public void onComplete() {
        System.out.println("Completed");
    }
    
    @Override
    public void onError(Throwable error) {
        System.err.println("Error: " + error.getMessage());
    }
};

// Step 3: Subscribe
publisher.subscribe(subscriber);

/**
 * Output:
 * Received: Item 0
 * Received: Item 1
 * Received: Item 2
 * Received: Item 3
 * Received: Item 4
 * Completed
 */
```

### Backpressure

```
┌─────────────────────────────────────────────────────────────┐
│                    Backpressure                              │
└─────────────────────────────────────────────────────────────┘

Problem: Publisher produces items faster than Subscriber can consume
Solution: Backpressure - Subscriber controls the flow

Without Backpressure (BAD):
Publisher ───────────────> [Buffer overflow!] ───> Subscriber
(Fast)                     (Unlimited queue)       (Slow)

With Backpressure (GOOD):
Publisher <──[request(10)]── Subscriber
          ──[10 items]────>
          <──[request(10)]──
          ──[10 items]────>
```

```java
/**
 * Backpressure Strategies
 */

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// Strategy 1: REQUEST - Let subscriber control (default, best)
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Flux<Integer> flux = Flux.range(1, 100);

flux.subscribe(new BaseSubscriber<Integer>() {
    @Override
    protected void hookOnSubscribe(Subscription subscription) {
        request(10);  // Request 10 items at a time
    }
    
    @Override
    protected void hookOnNext(Integer value) {
        System.out.println("Processing: " + value);
        
        if (value % 10 == 0) {
            request(10);  // Request next batch
        }
    }
});

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// Strategy 2: BUFFER - Buffer items (can cause OutOfMemoryError)
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Flux<Integer> buffered = Flux.range(1, 1000)
    .onBackpressureBuffer(100);  // Buffer up to 100 items
    // If buffer full → BufferOverflowException

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// Strategy 3: DROP - Drop items that can't be processed
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Flux<Integer> dropped = Flux.range(1, 1000)
    .onBackpressureDrop(item -> {
        System.out.println("Dropped: " + item);  // Log dropped items
    });

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// Strategy 4: LATEST - Keep only latest item
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Flux<Integer> latest = Flux.range(1, 1000)
    .onBackpressureLatest();  // Keep only latest item, drop others

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// Strategy 5: ERROR - Fail fast
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Flux<Integer> errorOnOverflow = Flux.range(1, 1000)
    .onBackpressureError();  // Throw exception if can't keep up

/**
 * Backpressure in Practice
 */
@Service
public class UserService {
    
    public Flux<User> getAllUsers() {
        return userRepository.findAll()
            // Apply backpressure strategy
            .onBackpressureBuffer(1000)  // Buffer up to 1000 users
            
            // Alternative: Let client control
            // (no operator needed, default behavior)
            
            // Alternative: Drop if slow
            // .onBackpressureDrop()
            
            .doOnNext(user -> log.info("Processing user: {}", user.getId()));
    }
}
```

## 2. Project Reactor

### Mono vs Flux

```
┌─────────────────────────────────────────────────────────────┐
│                    Mono vs Flux                              │
└─────────────────────────────────────────────────────────────┘

Mono<T>:
- Emits 0 or 1 item
- Then completes (or errors)
- Like Optional<T> or CompletableFuture<T>

Examples:
- findById() → Mono<User>      (0 or 1 user)
- save()     → Mono<User>      (1 saved user)
- count()    → Mono<Long>      (1 count)
- Mono.empty()                 (0 items)

Flux<T>:
- Emits 0 to N items
- Then completes (or errors)
- Like Stream<T> or List<T>

Examples:
- findAll()  → Flux<User>      (0 to N users)
- range()    → Flux<Integer>   (N integers)
- Flux.empty()                 (0 items)
```

```java
/**
 * Creating Mono
 */

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// From Value
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Mono<String> mono1 = Mono.just("Hello");  // Emits "Hello", completes

Mono<String> mono2 = Mono.empty();  // Completes immediately, no items

Mono<String> mono3 = Mono.error(new RuntimeException("Error"));  // Errors immediately

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// From Supplier (Lazy)
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Mono<String> mono4 = Mono.fromSupplier(() -> {
    System.out.println("Executing supplier");
    return "Hello";
});
// Supplier not executed until subscribed

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// From Callable
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Mono<String> mono5 = Mono.fromCallable(() -> {
    // Blocking operation (executed on separate thread)
    return blockingHttpCall();
});

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// From CompletableFuture
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> "Hello");
Mono<String> mono6 = Mono.fromFuture(future);

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// Defer (Lazy creation)
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Mono<String> mono7 = Mono.defer(() -> {
    System.out.println("Creating Mono");
    return Mono.just("Hello");
});
// Mono created on each subscription

/**
 * Creating Flux
 */

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// From Values
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Flux<String> flux1 = Flux.just("A", "B", "C");  // Emits 3 items

Flux<Integer> flux2 = Flux.range(1, 10);  // Emits 1, 2, 3, ..., 10

Flux<Long> flux3 = Flux.interval(Duration.ofSeconds(1));  // Emits 0, 1, 2, ... every second

Flux<String> flux4 = Flux.empty();  // Completes immediately, no items

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// From Iterable
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

List<String> list = Arrays.asList("A", "B", "C");
Flux<String> flux5 = Flux.fromIterable(list);

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// From Array
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

String[] array = {"A", "B", "C"};
Flux<String> flux6 = Flux.fromArray(array);

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// From Stream
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Stream<String> stream = Stream.of("A", "B", "C");
Flux<String> flux7 = Flux.fromStream(stream);

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// Generate (Programmatic)
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Flux<Integer> flux8 = Flux.generate(
    () -> 0,  // Initial state
    (state, sink) -> {
        sink.next(state);  // Emit current state
        if (state == 10) {
            sink.complete();  // Stop after 10
        }
        return state + 1;  // Next state
    }
);

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// Create (Advanced)
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Flux<String> flux9 = Flux.create(sink -> {
    for (int i = 0; i < 10; i++) {
        sink.next("Item " + i);
    }
    sink.complete();
});

/**
 * Subscribing
 */

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// Subscribe with Consumer
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Mono<String> mono = Mono.just("Hello");

// Basic subscription
mono.subscribe(value -> System.out.println("Received: " + value));

// With error handler
mono.subscribe(
    value -> System.out.println("Received: " + value),
    error -> System.err.println("Error: " + error)
);

// With complete handler
mono.subscribe(
    value -> System.out.println("Received: " + value),
    error -> System.err.println("Error: " + error),
    () -> System.out.println("Completed")
);

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// Block (Convert to blocking - AVOID in reactive code!)
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

String result = mono.block();  // Blocks until value received (or timeout)
String result2 = mono.block(Duration.ofSeconds(5));  // With timeout

List<String> results = Flux.just("A", "B", "C")
    .collectList()
    .block();  // Returns List<String>
```

### Operators

```java
/**
 * Transformation Operators
 */

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// map - Transform items (1-to-1)
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Flux<String> names = Flux.just("john", "jane", "bob");
Flux<String> upperCase = names.map(name -> name.toUpperCase());
// Emits: "JOHN", "JANE", "BOB"

Flux<Integer> lengths = names.map(String::length);
// Emits: 4, 4, 3

// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
// flatMap - Transform to Publisher (1-to-N, async)
// ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Flux<User> users = Flux.just(1L, 2L, 3L)
    .flatMap(id -> userRepository.findById(id));  // Each returns Mono<User>
// Emits: user1, user2, user3 (order not guaranteed)

// flatMap with concurrency limit
Flux<User> usersSequential = Flux.just(1L, 2L, 3L)
    .flatMap(id -> userRepository.findById(id), 1);  // Process 1 at a time
// Emits: user1, user2, user3 (order guaranteed)
```

```java
Flux<String> combined4 = Flux.combineLatest(source1, source2,

/**
 * Real-World Combining Example
 */
@Service
public class OrderService {
    
    // ✅ Create order: parallel operations, wait for all
    public Mono<Order> createOrder(OrderRequest request) {
        Mono<User> userMono = userRepository.findById(request.getUserId());
        Mono<Product> productMono = productRepository.findById(request.getProductId());
        Mono<Boolean> inventoryMono = inventoryService.checkStock(request.getProductId());
        
        // Execute all in parallel, combine results
        return Mono.zip(userMono, productMono, inventoryMono)
            .flatMap(tuple -> {
                User user = tuple.getT1();
                Product product = tuple.getT2();
                Boolean inStock = tuple.getT3();
                
                if (!inStock) {
                    return Mono.error(new OutOfStockException());
                }
                
                Order order = new Order(user, product);
                return orderRepository.save(order);
            });
    }
    
    // ✅ Process order: sequential operations
    public Mono<Void> processOrder(Long orderId) {
        return orderRepository.findById(orderId)
            .flatMap(order -> paymentService.processPayment(order))  // 1. Payment
            .flatMap(payment -> inventoryService.reserveStock(payment.getOrderId()))  // 2. Reserve
            .flatMap(reservation -> emailService.sendConfirmation(reservation))  // 3. Email
            .then();  // Wait for all, return Mono<Void>
    }
    
    // ✅ Get order details: parallel fetches, combine
    public Mono<OrderDetails> getOrderDetails(Long orderId) {
        Mono<Order> orderMono = orderRepository.findById(orderId);
        Mono<User> userMono = orderMono.flatMap(o -> userRepository.findById(o.getUserId()));
        Mono<Product> productMono = orderMono.flatMap(o -> productRepository.findById(o.getProductId()));
        Mono<Payment> paymentMono = paymentRepository.findByOrderId(orderId);
        
        return Mono.zip(orderMono, userMono, productMono, paymentMono)
            .map(tuple -> new OrderDetails(
                tuple.getT1(),  // Order
                tuple.getT2(),  // User
                tuple.getT3(),  // Product
                tuple.getT4()   // Payment
            ));
    }
}
```

### Hot vs Cold Publishers

```
┌─────────────────────────────────────────────────────────────┐
│              Hot vs Cold Publishers                          │
└─────────────────────────────────────────────────────────────┘

COLD Publisher:
- Starts emitting when subscribed
- Each subscriber gets ALL items from the beginning
- Independent streams per subscriber
- Examples: HTTP requests, database queries

HOT Publisher:
- Emits regardless of subscribers
- Subscribers get items from subscription point (miss earlier items)
- Shared stream across subscribers
- Examples: Mouse clicks, stock prices, sensor data
```

```java
/**
 * Cold Publisher (Default)
 */

// ✅ Cold: Each subscriber gets independent stream
Flux<Integer> coldFlux = Flux.range(1, 5)
    .map(i -> {
        System.out.println("Generating: " + i);
        return i;
    });

// Subscriber 1
coldFlux.subscribe(i -> System.out.println("Sub1: " + i));
// Output:
// Generating: 1, Sub1: 1
// Generating: 2, Sub1: 2
// ... (full sequence)

// Subscriber 2 (gets own sequence)
coldFlux.subscribe(i -> System.out.println("Sub2: " + i));
// Output:
// Generating: 1, Sub2: 1  ← Regenerates from start
// Generating: 2, Sub2: 2
// ... (full sequence)

/**
 * Hot Publisher - share()
 */

// ✅ Hot: Convert cold to hot (shares single stream)
Flux<Integer> hotFlux = Flux.range(1, 5)
    .delayElements(Duration.ofSeconds(1))
    .share();  // Convert to hot

// Start emitting
hotFlux.subscribe(i -> System.out.println("Sub1: " + i));

// Wait 2 seconds, then subscribe
Thread.sleep(2000);
hotFlux.subscribe(i -> System.out.println("Sub2: " + i));

// Output:
// Sub1: 1
// Sub1: 2
// Sub1: 3, Sub2: 3  ← Sub2 joins here, misses 1 and 2
// Sub1: 4, Sub2: 4
// Sub1: 5, Sub2: 5

/**
 * Hot Publisher - publish().autoConnect()
 */

// ✅ Hot: Start emitting after N subscribers
Flux<Integer> hotFlux2 = Flux.range(1, 5)
    .delayElements(Duration.ofSeconds(1))
    .publish()
    .autoConnect(2);  // Start after 2 subscribers

// First subscriber (waits for second)
hotFlux2.subscribe(i -> System.out.println("Sub1: " + i));

// Second subscriber (starts emission)
hotFlux2.subscribe(i -> System.out.println("Sub2: " + i));

/**
 * Hot Publisher - ConnectableFlux
 */

// ✅ Manual control over when to start emitting
ConnectableFlux<Integer> connectableFlux = Flux.range(1, 5)
    .delayElements(Duration.ofSeconds(1))
    .publish();

// Subscribe multiple subscribers
connectableFlux.subscribe(i -> System.out.println("Sub1: " + i));
connectableFlux.subscribe(i -> System.out.println("Sub2: " + i));

// Manually start emission
connectableFlux.connect();

/**
 * Hot Publisher - Processor (Advanced)
 */

// ✅ Create hot publisher with processor
Sinks.Many<String> sink = Sinks.many().multicast().onBackpressureBuffer();

// Convert to Flux
Flux<String> hotStream = sink.asFlux();

// Subscribe
hotStream.subscribe(s -> System.out.println("Sub1: " + s));

// Emit items
sink.tryEmitNext("A");  // Sub1 receives
sink.tryEmitNext("B");  // Sub1 receives

// New subscriber
hotStream.subscribe(s -> System.out.println("Sub2: " + s));

sink.tryEmitNext("C");  // Both receive
// Output:
// Sub1: A
// Sub1: B
// Sub1: C, Sub2: C  ← Sub2 misses A and B

/**
 * Real-World Hot Publisher: Stock Price Feed
 */
@Service
public class StockPriceService {
    
    private final Sinks.Many<StockPrice> priceSink = 
        Sinks.many().multicast().onBackpressureBuffer();
    
    private final Flux<StockPrice> priceStream = priceSink.asFlux();
    
    // External system pushes prices
    public void publishPrice(StockPrice price) {
        priceSink.tryEmitNext(price);
    }
    
    // Clients subscribe to hot stream
    public Flux<StockPrice> subscribeToPrices(String symbol) {
        return priceStream
            .filter(price -> price.getSymbol().equals(symbol));
    }
}

@RestController
@RequestMapping("/api/stocks")
public class StockController {
    
    @Autowired
    private StockPriceService stockPriceService;
    
    // SSE endpoint - hot stream
    @GetMapping(value = "/{symbol}/prices", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<StockPrice> streamPrices(@PathVariable String symbol) {
        return stockPriceService.subscribeToPrices(symbol);
        // Each client gets prices from subscription point onwards
    }
}

/**
 * Decision Guide: Hot vs Cold
 */

// Use COLD when:
// - Each subscriber needs all data
// - Data is specific to subscriber (e.g., user profile)
// - Reproducible operations (HTTP request, DB query)
// - Most common case

// Use HOT when:
// - Data is shared across subscribers
// - Late subscribers should miss old data
// - Real-time events (stock prices, sensor data, notifications)
// - Broadcasting to multiple clients
```

## Summary & Best Practices

### Reactive Streams Quick Reference

|Component|Purpose|Key Method|
|---|---|---|
|**Publisher**|Produces items|`subscribe(Subscriber)`|
|**Subscriber**|Consumes items|`onNext(T)`, `onComplete()`, `onError()`|
|**Subscription**|Controls flow|`request(long)`, `cancel()`|
|**Processor**|Transforms items|Implements both Publisher & Subscriber|

### Mono vs Flux

|Mono|Flux|
|---|---|
|0 or 1 item|0 to N items|
|Like `Optional<T>`|Like `Stream<T>`|
|`findById()`|`findAll()`|
|`save()`|`saveAll()`|
|`count()`|`range()`|

### Operator Categories

**Transformation**: map, flatMap, flatMapSequential, concatMap  
**Filtering**: filter, take, skip, distinct  
**Combining**: zip, merge, concat, combineLatest  
**Error Handling**: onErrorReturn, onErrorResume, retry, onErrorMap  
**Aggregation**: collectList, collectMap, reduce  
**Utility**: doOnNext, doOnError, doOnComplete, log

### Best Practices

1. **Never block** in reactive pipeline (no `.block()` in production)
2. **Use flatMap** for async operations (DB, HTTP)
3. **Use map** for sync transformations
4. **Handle errors** explicitly (onErrorResume, retry)
5. **Control backpressure** (request only what you need)
6. **Use operators** instead of subscribe callbacks
7. **Prefer cold** publishers (default, easier to reason about)
8. **Use hot** for broadcasting (stock prices, notifications)
9. **Test reactively** (StepVerifier from reactor-test)
10. **Monitor** with Reactor Context and logging

### Common Mistakes

❌ Calling `.block()` in reactive code  
❌ Not subscribing (nothing happens without subscribe)  
❌ Ignoring backpressure  
❌ Not handling errors  
❌ Using imperative patterns (loops, if-else)  
❌ Creating mutable state  
❌ Mixing blocking and non-blocking code

### When to Use Reactive

✅ **USE Reactive when:**

- High concurrency requirements
- I/O-bound operations (network, database)
- Streaming data
- Backpressure needed
- Low latency critical

❌ **AVOID Reactive when:**

- Simple CRUD applications
- CPU-bound operations
- Team not familiar with reactive
- Legacy blocking dependencies
