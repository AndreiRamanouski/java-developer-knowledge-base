## Overview

Auto-configuration is Spring Boot's "magic" that automatically configures your application based on dependencies in the classpath. Understanding how it works transforms it from magic into a powerful, predictable tool.

**Key Concept**: Spring Boot examines your classpath and existing beans, then automatically configures beans you're likely to need. You can always override these defaults.

---

## 1. How Auto-Configuration Works

### @SpringBootApplication Breakdown

```java
@SpringBootApplication
public class MyApplication {
    public static void main(String[] args) {
        SpringApplication.run(MyApplication.class, args);
    }
}
```

**This annotation is composed of three meta-annotations:**

```java
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Documented
@Inherited
@SpringBootConfiguration      // = @Configuration (Spring configuration class)
@EnableAutoConfiguration      // Enable auto-configuration magic
@ComponentScan(              // Scan for components in this package and below
    excludeFilters = {
        @Filter(type = FilterType.CUSTOM, classes = TypeExcludeFilter.class),
        @Filter(type = FilterType.CUSTOM, classes = AutoConfigurationExcludeFilter.class)
    }
)
public @interface SpringBootApplication {
    // ...
}
```

#### 1. @SpringBootConfiguration

- Meta-annotation for `@Configuration`
- Marks the class as a source of bean definitions
- Specialized form of `@Configuration` for Spring Boot

#### 2. @EnableAutoConfiguration

- Triggers the auto-configuration mechanism
- Imports `AutoConfigurationImportSelector`
- Reads `META-INF/spring.factories` from all JARs
- Applies conditional logic to decide which configurations to activate

#### 3. @ComponentScan

- Scans for `@Component`, `@Service`, `@Repository`, `@Controller`
- Default: scans the package of the annotated class and all sub-packages
- Your components are discovered automatically

---

### spring.factories and Auto-Configuration Classes

**Location**: `META-INF/spring.factories` in Spring Boot JARs

**Example from spring-boot-autoconfigure.jar:**

```properties
# Auto Configure
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
org.springframework.boot.autoconfigure.admin.SpringApplicationAdminJmxAutoConfiguration,\
org.springframework.boot.autoconfigure.aop.AopAutoConfiguration,\
org.springframework.boot.autoconfigure.amqp.RabbitAutoConfiguration,\
org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration,\
org.springframework.boot.autoconfigure.orm.jpa.HibernateJpaAutoConfiguration,\
org.springframework.boot.autoconfigure.web.servlet.DispatcherServletAutoConfiguration,\
org.springframework.boot.autoconfigure.web.servlet.WebMvcAutoConfiguration,\
# ... many more
```

**What happens at startup:**

1. Spring Boot reads all `spring.factories` files from classpath
2. Loads all classes listed under `EnableAutoConfiguration`
3. Evaluates conditional annotations on each configuration class
4. Only configures beans when conditions are met

### Conditional Annotations

Auto-configuration uses conditional annotations to decide whether to apply configuration.

#### Common Conditional Annotations

|Annotation|Activates When|
|---|---|
|`@ConditionalOnClass`|Specified class is on classpath|
|`@ConditionalOnMissingClass`|Specified class is NOT on classpath|
|`@ConditionalOnBean`|Specified bean exists in context|
|`@ConditionalOnMissingBean`|Specified bean does NOT exist|
|`@ConditionalOnProperty`|Property has a specific value|
|`@ConditionalOnResource`|Specified resource exists|
|`@ConditionalOnWebApplication`|App is a web application|
|`@ConditionalOnNotWebApplication`|App is NOT a web application|
|`@ConditionalOnExpression`|SpEL expression evaluates to true|
|`@ConditionalOnJava`|Running on specific Java version|

#### Real Example: DataSourceAutoConfiguration

```java
@Configuration(proxyBeanMethods = false)
@ConditionalOnClass({ DataSource.class, EmbeddedDatabaseType.class })
@ConditionalOnMissingBean(type = "io.r2dbc.spi.ConnectionFactory")
@EnableConfigurationProperties(DataSourceProperties.class)
@Import({ DataSourcePoolMetadataProvidersConfiguration.class,
          DataSourceInitializationConfiguration.class })
public class DataSourceAutoConfiguration {
    
    @Configuration(proxyBeanMethods = false)
    @Conditional(EmbeddedDatabaseCondition.class)
    @ConditionalOnMissingBean({ DataSource.class, XADataSource.class })
    @Import(EmbeddedDataSourceConfiguration.class)
    protected static class EmbeddedDatabaseConfiguration {
    }
    
    @Configuration(proxyBeanMethods = false)
    @ConditionalOnMissingBean(DataSource.class)
    @ConditionalOnProperty(name = "spring.datasource.type")
    static class Generic {
        
        @Bean
        DataSource dataSource(DataSourceProperties properties) {
            return properties.initializeDataSourceBuilder().build();
        }
    }
}
```

**Logic breakdown:**

1. ✅ Only if `DataSource.class` is on classpath (you added JDBC dependency)
2. ✅ Only if no `ConnectionFactory` bean exists (not reactive)
3. ✅ If no `DataSource` bean defined → create embedded database (H2/HSQLDB)
4. ✅ If `spring.datasource.type` property set → use that DataSource type
5. ❌ If you define your own `DataSource` bean → skip auto-configuration

### Auto-Configuration Order

Control when auto-configurations are processed:

```java
@Configuration
@AutoConfigureBefore(DataSourceAutoConfiguration.class)
public class MyCustomDataSourceConfig {
    // This runs BEFORE DataSourceAutoConfiguration
}

@Configuration
@AutoConfigureAfter(DataSourceAutoConfiguration.class)
public class MyDataSourcePostProcessor {
    // This runs AFTER DataSourceAutoConfiguration
}

@Configuration
@AutoConfigureOrder(Ordered.HIGHEST_PRECEDENCE)
public class HighPriorityConfig {
    // Runs early in auto-configuration
}
```

**Use cases:**

- Run before: Want to set up prerequisites
- Run after: Need to customize/enhance auto-configured beans
- Order: Fine-tune execution sequence

## 2. Creating Custom Starters

Create reusable auto-configuration modules for your organization.

### Starter Project Structure

**Naming Convention:**

- Official Spring starters: `spring-boot-starter-*`
- Custom starters: `*-spring-boot-starter`

**Example: `email-spring-boot-starter`**

```
email-spring-boot-starter/
├── pom.xml
└── src/main/
    ├── java/com/company/email/
    │   ├── EmailService.java                    (The actual service)
    │   ├── EmailProperties.java                 (Configuration properties)
    │   └── autoconfigure/
    │       └── EmailAutoConfiguration.java      (Auto-configuration class)
    └── resources/
        └── META-INF/
            └── spring.factories                  (Registers auto-configuration)
```

### Complete Custom Starter Example

#### Step 1: pom.xml

```xml
<?xml version="1.0" encoding="UTF-8"?>
<project>
    <modelVersion>4.0.0</modelVersion>
    
    <groupId>com.company</groupId>
    <artifactId>email-spring-boot-starter</artifactId>
    <version>1.0.0</version>
    <name>Email Starter</name>
    <description>Auto-configuration for email service</description>
    
    <dependencies>
        <!-- Spring Boot Auto-Configuration -->
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-autoconfigure</artifactId>
        </dependency>
        
        <!-- Configuration Processor (generates metadata for IDE autocomplete) -->
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-configuration-processor</artifactId>
            <optional>true</optional>
        </dependency>
        
        <!-- Your actual dependencies -->
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-mail</artifactId>
        </dependency>
    </dependencies>
</project>
```

#### Step 3: The Service

```java
package com.company.email;

import org.springframework.mail.SimpleMailMessage;
import org.springframework.mail.javamail.JavaMailSender;

public class EmailService {
    
    private final JavaMailSender mailSender;
    private final EmailProperties properties;
    
    public EmailService(JavaMailSender mailSender, EmailProperties properties) {
        this.mailSender = mailSender;
        this.properties = properties;
    }
    
    public void sendEmail(String to, String subject, String text) {
        SimpleMailMessage message = new SimpleMailMessage();
        message.setFrom(properties.getFrom());
        message.setTo(to);
        message.setSubject(properties.getSubjectPrefix() + subject);
        message.setText(text);
        
        mailSender.send(message);
    }
    
    public void sendNotification(String to, String notification) {
        sendEmail(to, "Notification", notification);
    }
}
```

#### Step 4: Auto-Configuration Class

```java
package com.company.email.autoconfigure;

import com.company.email.EmailProperties;
import com.company.email.EmailService;
import org.springframework.boot.autoconfigure.AutoConfiguration;
import org.springframework.boot.autoconfigure.condition.ConditionalOnClass;
import org.springframework.boot.autoconfigure.condition.ConditionalOnMissingBean;
import org.springframework.boot.autoconfigure.condition.ConditionalOnProperty;
import org.springframework.boot.autoconfigure.mail.MailSenderAutoConfiguration;
import org.springframework.boot.context.properties.EnableConfigurationProperties;
import org.springframework.context.annotation.Bean;
import org.springframework.mail.javamail.JavaMailSender;

@AutoConfiguration(after = MailSenderAutoConfiguration.class)
@ConditionalOnClass({ JavaMailSender.class, EmailService.class })
@ConditionalOnProperty(prefix = "company.email", name = "enabled", havingValue = "true", matchIfMissing = true)
@EnableConfigurationProperties(EmailProperties.class)
public class EmailAutoConfiguration {
    
    @Bean
    @ConditionalOnMissingBean
    public EmailService emailService(JavaMailSender mailSender, EmailProperties properties) {
        return new EmailService(mailSender, properties);
    }
}
```

**Conditional logic explained:**

1. Run after `MailSenderAutoConfiguration` (need JavaMailSender first)
2. Only if `JavaMailSender` and `EmailService` classes are available
3. Only if `company.email.enabled=true` (or property not set)
4. Enable `EmailProperties` for property binding
5. Create `EmailService` bean only if user hasn't defined one

#### Step 5: spring.factories Registration

**File**: `src/main/resources/META-INF/spring.factories`

```properties
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
com.company.email.autoconfigure.EmailAutoConfiguration
```

**This is critical!** Without this, Spring Boot won't discover your auto-configuration.

## 3. Overriding Auto-Configuration

### When to Override

**Good reasons:**

- Need specific configuration for your use case
- Auto-configuration is too generic
- Performance optimization
- Custom business logic integration

**Bad reasons:**

- "I want full control" (usually unnecessary)
- Not understanding what auto-configuration does
- Premature optimization

### How to Override

#### Method 1: Define Your Own Bean (Recommended)

Auto-configuration classes use `@ConditionalOnMissingBean`, so your bean takes precedence.

```java
@Configuration
public class CustomDataSourceConfig {
    
    @Bean
    public DataSource dataSource() {
        // Your custom DataSource configuration
        HikariConfig config = new HikariConfig();
        config.setJdbcUrl("jdbc:postgresql://localhost:5432/mydb");
        config.setUsername("user");
        config.setPassword("pass");
        config.setMaximumPoolSize(20);
        config.setMinimumIdle(5);
        config.setConnectionTimeout(30000);
        
        return new HikariDataSource(config);
    }
}
```

**Result**: `DataSourceAutoConfiguration` sees your bean exists and skips its configuration.

#### Method 2: Exclude Auto-Configuration Classes

**Option A: Via @SpringBootApplication**

```java
@SpringBootApplication(exclude = {
    DataSourceAutoConfiguration.class,
    HibernateJpaAutoConfiguration.class
})
public class MyApplication {
    public static void main(String[] args) {
        SpringApplication.run(MyApplication.class, args);
    }
}
```

**Option B: Via application.properties**

```properties
spring.autoconfigure.exclude=\
  org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration,\
  org.springframework.boot.autoconfigure.orm.jpa.HibernateJpaAutoConfiguration
```

**Option C: Via application.yml**

```yaml
spring:
  autoconfigure:
    exclude:
      - org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration
      - org.springframework.boot.autoconfigure.orm.jpa.HibernateJpaAutoConfiguration
```

#### Method 3: Override Properties

Often sufficient without excluding auto-configuration:

```yaml
# Override DataSource auto-configuration
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/mydb
    username: myuser
    password: mypass
    hikari:
      maximum-pool-size: 20
      minimum-idle: 5
      connection-timeout: 30000
      
# Override JPA auto-configuration
  jpa:
    hibernate:
      ddl-auto: validate
    properties:
      hibernate:
        dialect: org.hibernate.dialect.PostgreSQLDialect
        format_sql: true
        use_sql_comments: true
```

#### Method 4: Customize Auto-Configured Beans

Use `@PostConstruct` or implement interfaces:

```java
@Configuration
public class DataSourceCustomizer {
    
    @Autowired
    private DataSource dataSource;
    
    @PostConstruct
    public void customize() {
        if (dataSource instanceof HikariDataSource) {
            HikariDataSource hikari = (HikariDataSource) dataSource;
            hikari.setMaximumPoolSize(30);
            // Customize further...
        }
    }
}
```

Or use dedicated customizers:

```java
@Configuration
public class DataSourceCustomizer {
    
    @Bean
    public HikariConfigCustomizer hikariConfigCustomizer() {
        return config -> {
            config.setMaximumPoolSize(30);
            config.setConnectionTimeout(20000);
            config.setMetricRegistry(myMetricRegistry());
        };
    }
}
```

### Custom Configurations Take Precedence

**Spring Boot's precedence order (highest to lowest):**

1. **User-defined beans** in `@Configuration` classes
2. **Auto-configuration** from starters
3. **Default values** in auto-configuration

**Example:**

```java
// User configuration - HIGHEST PRECEDENCE
@Configuration
public class MyConfig {
    @Bean
    public DataSource dataSource() {
        return new HikariDataSource();  // This wins!
    }
}

// Auto-configuration - LOWER PRECEDENCE
@Configuration
@ConditionalOnMissingBean(DataSource.class)  // Checks for user bean first
public class DataSourceAutoConfiguration {
    @Bean
    public DataSource dataSource() {
        return new HikariDataSource();  // Only if user didn't define one
    }
}
```

## Complete Example: Custom vs Auto-Configuration

### Scenario: Email Service with Retry Logic

**Auto-configured starter (basic):**

```java
// In email-spring-boot-starter
@AutoConfiguration
public class EmailAutoConfiguration {
    @Bean
    @ConditionalOnMissingBean
    public EmailService emailService(JavaMailSender mailSender) {
        return new EmailService(mailSender);
    }
}
```

**Your application (enhanced with retry):**

```java
@Configuration
public class CustomEmailConfig {
    
    @Bean
    public EmailService emailService(JavaMailSender mailSender) {
        EmailService service = new EmailService(mailSender);
        
        // Wrap with retry logic
        return new RetryableEmailService(service, 3, 1000);
    }
    
    // Custom implementation
    private static class RetryableEmailService extends EmailService {
        private final EmailService delegate;
        private final int maxRetries;
        private final long retryDelay;
        
        public RetryableEmailService(EmailService delegate, int maxRetries, long retryDelay) {
            super(null);  // Not used
            this.delegate = delegate;
            this.maxRetries = maxRetries;
            this.retryDelay = retryDelay;
        }
        
        @Override
        public void sendEmail(String to, String subject, String text) {
            int attempt = 0;
            while (attempt < maxRetries) {
                try {
                    delegate.sendEmail(to, subject, text);
                    return;  // Success
                } catch (Exception e) {
                    attempt++;
                    if (attempt >= maxRetries) throw e;
                    try { Thread.sleep(retryDelay); } catch (InterruptedException ie) {}
                }
            }
        }
    }
}
```

**Result**: Auto-configuration is skipped, your custom bean is used instead.

## Best Practices

### DO ✅

1. **Use properties for configuration** - More flexible than hardcoding
2. **Rely on auto-configuration when possible** - Less code to maintain
3. **Debug auto-configuration with --debug** - Understand what's happening
4. **Use @ConditionalOnMissingBean** in custom configs - Allow override
5. **Document custom starters well** - Include property documentation
6. **Use configuration processor** - Enable IDE autocomplete
7. **Test auto-configuration** - Write tests for conditional logic
8. **Follow naming conventions** - `*-spring-boot-starter` for custom starters

### DON'T ❌

1. **Don't fight auto-configuration** - Work with it, not against it
2. **Don't exclude without understanding** - May break dependencies
3. **Don't duplicate auto-configuration** - Override specific beans only
4. **Don't ignore conditional annotations** - They prevent conflicts
5. **Don't hardcode configuration** - Use properties
6. **Don't create starter without @ConditionalOnMissingBean** - Users can't override
7. **Don't forget spring.factories** - Auto-configuration won't be discovered
8. **Don't over-customize** - Keep it simple

## Troubleshooting

### Problem: Auto-Configuration Not Applied

**Symptoms**: Expected bean not created, features not working

**Solutions**:

1. Check classpath has required dependencies

```bash
mvn dependency:tree | grep starter-name
```

2. Verify conditions with --debug

```bash
java -jar app.jar --debug
```

3. Check spring.factories registration

```bash
jar -tf starter.jar | grep spring.factories
```

4. Verify property values

```yaml
logging:
  level:
    org.springframework.boot.autoconfigure: DEBUG
```

### Problem: Wrong Bean Configured

**Symptoms**: Unexpected behavior, wrong implementation used

**Solutions**:

1. Check bean precedence

```java
@Autowired
private ApplicationContext context;

@PostConstruct
public void debug() {
    String[] beanNames = context.getBeanNamesForType(DataSource.class);
    System.out.println("DataSource beans: " + Arrays.toString(beanNames));
}
```

2. Use @Primary if multiple beans exist

```java
@Bean
@Primary
public DataSource primaryDataSource() {
    return new HikariDataSource();
}
```

3. Use @Qualifier for specific bean selection

```java
@Autowired
@Qualifier("customDataSource")
private DataSource dataSource;
```

### Problem: Property Not Applied

**Symptoms**: Configuration property ignored

**Solutions**:

1. Check property prefix and name

```java
@ConfigurationProperties(prefix = "app.email")  // Must match YAML structure
```

2. Verify @EnableConfigurationProperties

```java
@Configuration
@EnableConfigurationProperties(EmailProperties.class)  // Required!
```

3. Check property in actuator

```bash
curl http://localhost:8080/actuator/configprops
```

4. Add configuration processor dependency

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-configuration-processor</artifactId>
</dependency>
```

## Summary

### Key Takeaways

1. **@SpringBootApplication** = `@Configuration` + `@EnableAutoConfiguration` + `@ComponentScan`
2. **Auto-configuration** reads `spring.factories` and applies conditional logic
3. **Conditional annotations** make auto-configuration smart and non-intrusive
4. **--debug flag** reveals exactly what auto-configuration is doing
5. **Custom starters** package reusable auto-configuration
6. **Your beans always win** - User configuration takes precedence
7. **Override with properties** when possible - More flexible than code
