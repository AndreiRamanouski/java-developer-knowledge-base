## Overview

Spring Boot's externalized configuration system allows you to work with the same application code in different environments. Configuration can come from properties files, YAML files, environment variables, command-line arguments, and more.

**Key Concept**: Configuration is loaded from multiple sources with a well-defined precedence order, allowing you to override defaults for different environments without changing code.

---

## 1. Externalized Configuration

### Configuration Precedence (17 Levels!)

Spring Boot uses a very specific order to load configuration. **Higher numbers override lower numbers.**

|Priority|Source|Example|Use Case|
|---|---|---|---|
|**1**|Default properties|`SpringApplication.setDefaultProperties()`|Hard-coded defaults|
|**2**|`@PropertySource`|`@PropertySource("classpath:/custom.properties")`|Legacy integration|
|**3**|Config data (application.properties/yml)|`application.properties` in jar|Default configuration|
|**4**|Profile-specific config data|`application-dev.properties`|Environment-specific|
|**5**|Config data outside jar|`./config/application.properties`|External configuration|
|**6**|Profile-specific outside jar|`./config/application-prod.properties`|Production overrides|
|**7**|OS environment variables|`export SERVER_PORT=9090`|Container/cloud config|
|**8**|Java System properties|`-Dserver.port=9090`|JVM configuration|
|**9**|JNDI attributes|`java:comp/env`|Java EE integration|
|**10**|ServletContext init parameters|`web.xml` settings|Servlet config|
|**11**|ServletConfig init parameters|Servlet-specific|Servlet config|
|**12**|`SPRING_APPLICATION_JSON`|`SPRING_APPLICATION_JSON='{"server.port":9090}'`|JSON config|
|**13**|Command-line arguments|`--server.port=9090`|Runtime overrides|
|**14**|Test `@TestPropertySource`|Test-specific properties|Testing|
|**15**|Test `@SpringBootTest` properties|`@SpringBootTest(properties = "...")`|Testing|
|**16**|Devtools global settings|`~/.config/spring-boot/`|Developer machine|
|**17**|`@ConfigurationProperties` defaults|Default values in class|Code defaults|

**Practical priority (most common sources):**

```
Command-line args (highest)
  ↓
Environment variables
  ↓
application-{profile}.properties (outside jar)
  ↓
application-{profile}.properties (in jar)
  ↓
application.properties (outside jar)
  ↓
application.properties (in jar)
  ↓
@ConfigurationProperties defaults (lowest)
```

#### Real-World Example

```yaml
# application.yml (in jar) - Priority 3
server:
  port: 8080
  
# application-dev.yml (in jar) - Priority 4
server:
  port: 8081
  
# Environment variable - Priority 7
export SERVER_PORT=8082

# Command-line argument - Priority 13 (WINS!)
java -jar app.jar --server.port=8083
```

**Result**: Application runs on port **8083** (command-line wins)

### application.properties vs application.yml

Both formats are supported. Choose based on preference and team standards.

#### Properties Format

**File**: `application.properties`

```properties
# Server configuration
server.port=8080
server.servlet.context-path=/api

# Database configuration
spring.datasource.url=jdbc:postgresql://localhost:5432/mydb
spring.datasource.username=dbuser
spring.datasource.password=dbpass
spring.datasource.hikari.maximum-pool-size=10
spring.datasource.hikari.minimum-idle=5

# Logging
logging.level.root=INFO
logging.level.com.company.myapp=DEBUG
logging.pattern.console=%d{HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n

# Custom properties
app.name=My Application
app.version=1.0.0
app.features.analytics=true
app.features.notifications=false
```

**Pros:**

- Simple, flat structure
- Easier for simple configurations
- Better tool support historically
- Can use in `@PropertySource`

**Cons:**

- Repetitive prefixes
- Harder to read with deep nesting
- No native list/map support

#### YAML Format

**File**: `application.yml`

```yaml
# Server configuration
server:
  port: 8080
  servlet:
    context-path: /api

# Database configuration
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/mydb
    username: dbuser
    password: dbpass
    hikari:
      maximum-pool-size: 10
      minimum-idle: 5

# Logging
logging:
  level:
    root: INFO
    com.company.myapp: DEBUG
  pattern:
    console: "%d{HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n"

# Custom properties
app:
  name: My Application
  version: 1.0.0
  features:
    analytics: true
    notifications: false
```

**Pros:**

- Hierarchical, easier to read
- Less repetition
- Better for complex configurations
- Native list and map support

**Cons:**

- Indentation-sensitive (YAML gotcha!)
- Cannot use with `@PropertySource`
- Slightly slower to parse

#### Lists and Maps

**Properties format:**

```properties
# List
app.servers[0]=server1.company.com
app.servers[1]=server2.company.com
app.servers[2]=server3.company.com

# Map
app.database.primary.url=jdbc:postgresql://db1:5432/app
app.database.primary.username=user1
app.database.secondary.url=jdbc:postgresql://db2:5432/app
app.database.secondary.username=user2
```

**YAML format (much cleaner):**

```yaml
app:
  servers:
    - server1.company.com
    - server2.company.com
    - server3.company.com
  
  database:
    primary:
      url: jdbc:postgresql://db1:5432/app
      username: user1
    secondary:
      url: jdbc:postgresql://db2:5432/app
      username: user2
```

### Profile-Specific Properties

Profiles allow different configurations for different environments.

#### Naming Convention

- `application.properties` - Default (always loaded)
- `application-{profile}.properties` - Profile-specific
- `application.yml` - Default
- `application-{profile}.yml` - Profile-specific

#### Common Profiles

```
application.yml              # Base configuration (all environments)
application-dev.yml          # Development
application-test.yml         # Testing
application-staging.yml      # Staging
application-prod.yml         # Production
```

#### Activating Profiles

**Method 1: Command-line**

```bash
java -jar app.jar --spring.profiles.active=prod
```

**Method 2: Environment variable**

```bash
export SPRING_PROFILES_ACTIVE=prod
java -jar app.jar
```

**Method 3: In application.properties**

```properties
spring.profiles.active=dev
```

**Method 4: In code (not recommended)**

```java
@SpringBootApplication
public class MyApplication {
    public static void main(String[] args) {
        SpringApplication app = new SpringApplication(MyApplication.class);
        app.setAdditionalProfiles("dev");
        app.run(args);
    }
}
```

**Method 5: IDE configuration**

```
IntelliJ IDEA:
Run → Edit Configurations → Active profiles: dev
```

#### Multiple Profiles

Activate multiple profiles (comma-separated):

```bash
java -jar app.jar --spring.profiles.active=dev,debug,mysql
```

**Load order**: Later profiles override earlier ones

#### Profile-Specific Configuration Example

**application.yml** (base):

```yaml
app:
  name: My Application
  
spring:
  jpa:
    show-sql: false
    hibernate:
      ddl-auto: validate

logging:
  level:
    root: WARN
```

**application-dev.yml**:

```yaml
spring:
  datasource:
    url: jdbc:h2:mem:testdb
    username: sa
    password: 
  jpa:
    show-sql: true
    hibernate:
      ddl-auto: create-drop

logging:
  level:
    root: INFO
    com.company.myapp: DEBUG
```

**application-prod.yml**:

```yaml
spring:
  datasource:
    url: jdbc:postgresql://${DB_HOST}:5432/${DB_NAME}
    username: ${DB_USERNAME}
    password: ${DB_PASSWORD}
    hikari:
      maximum-pool-size: 20
      minimum-idle: 10
  jpa:
    show-sql: false
    hibernate:
      ddl-auto: validate

logging:
  level:
    root: WARN
    com.company.myapp: INFO
```

#### Profile Groups (Spring Boot 2.4+)

Group related profiles together:

```yaml
spring:
  profiles:
    group:
      production:
        - prod
        - proddb
        - prodlogging
      development:
        - dev
        - devdb
        - h2
```

Activate with:

```bash
java -jar app.jar --spring.profiles.active=production
```

This activates `prod`, `proddb`, and `prodlogging` profiles.

### Command-Line Arguments

Override any property from command line with `--` prefix.

```bash
# Override single property
java -jar app.jar --server.port=9090

# Override multiple properties
java -jar app.jar --server.port=9090 --spring.profiles.active=prod

# Override nested properties
java -jar app.jar --spring.datasource.url=jdbc:postgresql://prod-db:5432/app

# Disable command-line property parsing (rare)
SpringApplication.setAddCommandLineProperties(false)
```

**Property format**: Use dot notation, not YAML hierarchy.

### Environment Variables

Map environment variables to properties using uppercase and underscores.

#### Naming Convention

```
Property:          server.port
Env Variable:      SERVER_PORT

Property:          spring.datasource.url
Env Variable:      SPRING_DATASOURCE_URL

Property:          app.api-key
Env Variable:      APP_API_KEY  or  APP_APIKEY
```

#### Setting Environment Variables

**Linux/Mac:**

```bash
export SERVER_PORT=8080
export SPRING_PROFILES_ACTIVE=prod
export SPRING_DATASOURCE_URL=jdbc:postgresql://db:5432/app
export SPRING_DATASOURCE_USERNAME=dbuser
export SPRING_DATASOURCE_PASSWORD=secret

java -jar app.jar
```

**Windows:**

```cmd
set SERVER_PORT=8080
set SPRING_PROFILES_ACTIVE=prod
java -jar app.jar
```

**Docker:**

```dockerfile
FROM openjdk:17-jdk-slim
COPY target/myapp.jar app.jar

ENV SERVER_PORT=8080
ENV SPRING_PROFILES_ACTIVE=prod

ENTRYPOINT ["java", "-jar", "/app.jar"]
```

Or pass at runtime:

```bash
docker run -e SERVER_PORT=8080 -e SPRING_PROFILES_ACTIVE=prod myapp:latest
```

**Kubernetes:**

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: myapp
spec:
  containers:
  - name: myapp
    image: myapp:latest
    env:
    - name: SERVER_PORT
      value: "8080"
    - name: SPRING_PROFILES_ACTIVE
      value: "prod"
    - name: SPRING_DATASOURCE_PASSWORD
      valueFrom:
        secretKeyRef:
          name: db-secret
          key: password
```

### @ConfigurationProperties vs @Value

Two ways to bind configuration to your code. Each has its use case.

#### @Value - Simple Property Injection

**Use for**: Simple values, SpEL expressions, one-off properties

```java
@Component
public class MyService {
    
    @Value("${app.name}")
    private String appName;
    
    @Value("${server.port}")
    private int serverPort;
    
    @Value("${app.timeout:5000}")  // Default value: 5000
    private long timeout;
    
    @Value("${app.features.analytics:true}")  // Default: true
    private boolean analyticsEnabled;
    
    // SpEL expressions
    @Value("#{systemProperties['user.home']}")
    private String userHome;
    
    @Value("#{T(java.lang.Math).random() * 100}")
    private double randomNumber;
    
    @Value("#{${app.features}}")  // Inject map
    private Map<String, Boolean> features;
}
```

**application.yml:**

```yaml
app:
  name: My Application
  timeout: 3000
  features:
    analytics: true
    notifications: false
```

**Pros:**

- Simple and quick
- Supports SpEL expressions
- Works on fields, methods, constructor parameters

**Cons:**

- No validation
- No IDE autocomplete
- Scattered throughout codebase
- Hard to test
- No metadata generation

#### @ConfigurationProperties - Type-Safe Configuration

**Use for**: Grouped properties, complex configuration, validated configuration

```java
@ConfigurationProperties(prefix = "app")
@Component
public class AppProperties {
    
    private String name;
    private String version;
    private int timeout = 5000;  // Default value
    private Features features;
    private List<String> servers;
    private Map<String, Database> databases;
    
    // Getters and setters required!
    
    public String getName() {
        return name;
    }
    
    public void setName(String name) {
        this.name = name;
    }
    
    public String getVersion() {
        return version;
    }
    
    public void setVersion(String version) {
        this.version = version;
    }
    
    public int getTimeout() {
        return timeout;
    }
    
    public void setTimeout(int timeout) {
        this.timeout = timeout;
    }
    
    public Features getFeatures() {
        return features;
    }
    
    public void setFeatures(Features features) {
        this.features = features;
    }
    
    public List<String> getServers() {
        return servers;
    }
    
    public void setServers(List<String> servers) {
        this.servers = servers;
    }
    
    public Map<String, Database> getDatabases() {
        return databases;
    }
    
    public void setDatabases(Map<String, Database> databases) {
        this.databases = databases;
    }
    
    // Nested classes
    public static class Features {
        private boolean analytics;
        private boolean notifications;
        
        // Getters and setters
        public boolean isAnalytics() {
            return analytics;
        }
        
        public void setAnalytics(boolean analytics) {
            this.analytics = analytics;
        }
        
        public boolean isNotifications() {
            return notifications;
        }
        
        public void setNotifications(boolean notifications) {
            this.notifications = notifications;
        }
    }
    
    public static class Database {
        private String url;
        private String username;
        
        // Getters and setters
        public String getUrl() {
            return url;
        }
        
        public void setUrl(String url) {
            this.url = url;
        }
        
        public String getUsername() {
            return username;
        }
        
        public void setUsername(String username) {
            this.username = username;
        }
    }
}
```

**application.yml:**

```yaml
app:
  name: My Application
  version: 1.0.0
  timeout: 3000
  features:
    analytics: true
    notifications: false
  servers:
    - server1.company.com
    - server2.company.com
  databases:
    primary:
      url: jdbc:postgresql://db1:5432/app
      username: user1
    secondary:
      url: jdbc:postgresql://db2:5432/app
      username: user2
```

**Enable in Configuration:**

```java
@Configuration
@EnableConfigurationProperties(AppProperties.class)
public class AppConfig {
    // AppProperties is now a bean
}
```

Or use `@Component` on the properties class directly:

```java
@ConfigurationProperties(prefix = "app")
@Component  // Makes it a bean automatically
public class AppProperties {
    // ...
}
```

**Usage:**

```java
@Service
public class MyService {
    
    private final AppProperties appProperties;
    
    // Constructor injection (recommended)
    public MyService(AppProperties appProperties) {
        this.appProperties = appProperties;
    }
    
    public void doSomething() {
        String name = appProperties.getName();
        boolean analyticsEnabled = appProperties.getFeatures().isAnalytics();
        List<String> servers = appProperties.getServers();
    }
}
```

**Pros:**

- Type-safe
- IDE autocomplete
- Validation support
- Grouped logically
- Easy to test (just create instance)
- Metadata generation
- Nested structures

**Cons:**

- More verbose
- Requires getters/setters
- Cannot use SpEL expressions

#### Comparison Table

|Feature|@Value|@ConfigurationProperties|
|---|---|---|
|Simple values|✅ Excellent|✅ Good|
|Complex objects|❌ No|✅ Excellent|
|Validation|❌ No|✅ Yes|
|SpEL|✅ Yes|❌ No|
|IDE autocomplete|❌ No|✅ Yes|
|Type safety|⚠️ Limited|✅ Full|
|Testability|⚠️ Hard|✅ Easy|
|Metadata|❌ No|✅ Yes|

**When to use what:**

- **@Value**: One-off properties, SpEL needed, simple values
- **@ConfigurationProperties**: Related properties, validation needed, complex structures

### Relaxed Binding

Spring Boot is flexible with property naming. Multiple formats map to same property.

#### Supported Formats

For property `app.apiKey`:

```yaml
# All of these work!
app:
  apiKey: xyz123          # Camel case
  api-key: xyz123         # Kebab case (recommended for YAML)
  api_key: xyz123         # Snake case
  APIKEY: xyz123          # Upper case
```

Environment variable `APP_APIKEY` or `APP_API_KEY` both map to `app.apiKey`.

#### Binding Rules

**For @ConfigurationProperties:**

|Property Format|Java Field|YAML/Properties|Env Variable|
|---|---|---|---|
|`apiKey`|`apiKey`|`api-key` ✅|`API_KEY` ✅|
|`api-key`|`apiKey`|`api-key` ✅|`API_KEY` ✅|
|`api_key`|`apiKey`|`api-key` ✅|`API_KEY` ✅|

**Recommendations:**

- **Java**: Use camelCase (`apiKey`)
- **Properties/YAML**: Use kebab-case (`api-key`)
- **Environment variables**: Use UPPER_SNAKE_CASE (`API_KEY`)

#### Examples

**Java class:**

```java
@ConfigurationProperties(prefix = "app")
public class AppProperties {
    private String apiKey;           // Camel case in Java
    private int connectionTimeout;   // Camel case in Java
    private boolean sslEnabled;      // Camel case in Java
    
    // Getters and setters...
}
```

**YAML (recommended):**

```yaml
app:
  api-key: xyz123              # Kebab case
  connection-timeout: 5000     # Kebab case
  ssl-enabled: true            # Kebab case
```

**Properties file:**

```properties
app.api-key=xyz123
app.connection-timeout=5000
app.ssl-enabled=true
```

**Environment variables:**

```bash
export APP_API_KEY=xyz123
export APP_CONNECTION_TIMEOUT=5000
export APP_SSL_ENABLED=true
```

**All work perfectly!** Spring Boot maps them automatically.

#### List Binding Variations

```yaml
# Array notation
app.servers[0]=server1.com
app.servers[1]=server2.com

# Comma-separated (for environment variables)
APP_SERVERS=server1.com,server2.com

# YAML list
app:
  servers:
    - server1.com
    - server2.com
```

## 2. Configuration Validation

Validate configuration at startup to fail fast on misconfiguration.

### JSR-303 Validation on @ConfigurationProperties

**Step 1: Add dependency**

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-validation</artifactId>
</dependency>
```

**Step 2: Add @Validated and constraints**

```java
@ConfigurationProperties(prefix = "app")
@Validated  // Enable validation!
@Component
public class AppProperties {
    
    @NotBlank(message = "Application name is required")
    private String name;
    
    @Min(value = 1000, message = "Timeout must be at least 1000ms")
    @Max(value = 60000, message = "Timeout must be at most 60000ms")
    private int timeout = 5000;
    
    @Email(message = "Must be a valid email address")
    private String contactEmail;
    
    @Pattern(regexp = "^[A-Z]{3}-\\d{4}$", message = "API key format: ABC-1234")
    private String apiKey;
    
    @NotEmpty(message = "At least one server must be configured")
    private List<@NotBlank String> servers;
    
    @Valid  // Validate nested object
    @NotNull
    private Database database;
    
    // Getters and setters...
    
    public static class Database {
        
        @NotBlank(message = "Database URL is required")
        @Pattern(regexp = "^jdbc:.*", message = "Must be a valid JDBC URL")
        private String url;
        
        @NotBlank(message = "Database username is required")
        private String username;
        
        @Size(min = 8, message = "Password must be at least 8 characters")
        private String password;
        
        @Min(1)
        @Max(100)
        private int maxPoolSize = 10;
        
        // Getters and setters...
    }
}
```

**application.yml:**

```yaml
app:
  name: ""  # ❌ Fails validation (blank)
  timeout: 100  # ❌ Fails validation (< 1000)
  contact-email: invalid-email  # ❌ Fails validation
  api-key: INVALID  # ❌ Fails validation (wrong pattern)
  servers: []  # ❌ Fails validation (empty list)
  database:
    url: "not-jdbc"  # ❌ Fails validation
    username: ""  # ❌ Fails validation
    password: "short"  # ❌ Fails validation (< 8 chars)
    max-pool-size: 200  # ❌ Fails validation (> 100)
```

**Application fails to start with clear error messages!**

```
***************************
APPLICATION FAILED TO START
***************************

Failed to bind properties under 'app' to com.company.config.AppProperties

Reason: 
- app.name: Application name is required
- app.timeout: Timeout must be at least 1000ms
- app.contact-email: Must be a valid email address
- app.api-key: API key format: ABC-1234
- app.servers: At least one server must be configured
- app.database.url: Must be a valid JDBC URL
- app.database.username: Database username is required
- app.database.password: Password must be at least 8 characters
- app.database.max-pool-size: must be less than or equal to 100
```

### Common Validation Annotations

|Annotation|Usage|Example|
|---|---|---|
|`@NotNull`|Field cannot be null|`@NotNull private String name;`|
|`@NotBlank`|String cannot be null/empty/whitespace|`@NotBlank private String username;`|
|`@NotEmpty`|Collection/array cannot be null/empty|`@NotEmpty private List<String> items;`|
|`@Size(min, max)`|String/Collection size constraints|`@Size(min=8, max=20) private String password;`|
|`@Min(value)`|Number minimum value|`@Min(1) private int port;`|
|`@Max(value)`|Number maximum value|`@Max(65535) private int port;`|
|`@Email`|Valid email format|`@Email private String email;`|
|`@Pattern(regexp)`|Regex pattern match|`@Pattern(regexp="[A-Z]{2}") private String code;`|
|`@Positive`|Number > 0|`@Positive private int count;`|
|`@PositiveOrZero`|Number >= 0|`@PositiveOrZero private int retries;`|
|`@Negative`|Number < 0|`@Negative private int offset;`|
|`@NegativeOrZero`|Number <= 0|`@NegativeOrZero private int balance;`|
|`@Past`|Date in the past|`@Past private LocalDate birthDate;`|
|`@Future`|Date in the future|`@Future private LocalDate expiryDate;`|
|`@Valid`|Validate nested object|`@Valid private Database db;`|

### Custom Validators

Create custom validation logic for complex rules.

#### Example: Custom Validator for URL List

**Step 1: Create annotation**

```java
@Target({ElementType.FIELD, ElementType.PARAMETER})
@Retention(RetentionPolicy.RUNTIME)
@Constraint(validatedBy = ValidUrlListValidator.class)
public @interface ValidUrlList {
    
    String message() default "One or more URLs are invalid";
    
    Class<?>[] groups() default {};
    
    Class<? extends Payload>[] payload() default {};
}
```

**Step 2: Create validator**

```java
public class ValidUrlListValidator implements ConstraintValidator<ValidUrlList, List<String>> {
    
    private static final Pattern URL_PATTERN = Pattern.compile(
        "^(https?://)?([\\da-z\\.-]+)\\.([a-z\\.]{2,6})([/\\w \\.-]*)*/?$"
    );
    
    @Override
    public boolean isValid(List<String> urls, ConstraintValidatorContext context) {
        if (urls == null || urls.isEmpty()) {
            return false;
        }
        
        for (String url : urls) {
            if (url == null || !URL_PATTERN.matcher(url).matches()) {
                return false;
            }
        }
        
        return true;
    }
}
```

**Step 3: Use it**

```java
@ConfigurationProperties(prefix = "app")
@Validated
@Component
public class AppProperties {
    
    @ValidUrlList(message = "All server URLs must be valid")
    private List<String> servers;
    
    // Getters and setters...
}
```

#### Example: Cross-Field Validation

Validate that two fields are consistent:

```java
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Constraint(validatedBy = PasswordMatchValidator.class)
public @interface PasswordMatch {
    String message() default "Passwords do not match";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}

public class PasswordMatchValidator implements ConstraintValidator<PasswordMatch, UserProperties> {
    
    @Override
    public boolean isValid(UserProperties user, ConstraintValidatorContext context) {
        if (user.getPassword() == null || user.getPasswordConfirm() == null) {
            return false;
        }
        return user.getPassword().equals(user.getPasswordConfirm());
    }
}

@ConfigurationProperties(prefix = "user")
@Validated
@PasswordMatch  // Applied at class level
@Component
public class UserProperties {
    
    @NotBlank
    private String password;
    
    @NotBlank
    private String passwordConfirm;
    
    // Getters and setters...
}
```

## 3. Configuration Encryption

Protect sensitive configuration values.

### Jasypt Integration

**Jasypt** (Java Simplified Encryption) encrypts properties at rest.

#### Step 1: Add Dependency

```xml
<dependency>
    <groupId>com.github.ulisesbocchio</groupId>
    <artifactId>jasypt-spring-boot-starter</artifactId>
    <version>3.0.5</version>
</dependency>
```

#### Step 2: Configure Encryption Password

**DO NOT hardcode in application.yml!** Pass as environment variable or command-line argument.

```bash
# Environment variable
export JASYPT_ENCRYPTOR_PASSWORD=mySecretKey

# Or command-line
java -jar app.jar --jasypt.encryptor.password=mySecretKey
```

#### Step 3: Encrypt Sensitive Values

**Use Jasypt CLI to encrypt:**

```bash
# Download Jasypt
wget https://github.com/jasypt/jasypt/releases/download/jasypt-1.9.3/jasypt-1.9.3-dist.zip
unzip jasypt-1.9.3-dist.zip
cd jasypt-1.9.3/bin

# Encrypt a value
./encrypt.sh input="myDatabasePassword" password="mySecretKey" algorithm=PBEWithMD5AndDES

----OUTPUT----------------------
Encrypted: kL8v3zTKv5QHZw4qHIkJQg==
```

Or use Java code:

```java
import org.jasypt.encryption.pbe.StandardPBEStringEncryptor;

public class EncryptionUtil {
    public static void main(String[] args) {
        StandardPBEStringEncryptor encryptor = new StandardPBEStringEncryptor();
        encryptor.setPassword("mySecretKey");
        encryptor.setAlgorithm("PBEWithMD5AndDES");
        
        String encrypted = encryptor.encrypt("myDatabasePassword");
        System.out.println("Encrypted: " + encrypted);
        
        String decrypted = encryptor.decrypt(encrypted);
        System.out.println("Decrypted: " + decrypted);
    }
}
```

#### Step 4: Use Encrypted Values

**application.yml:**

```yaml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/mydb
    username: dbuser
    password: ENC(kL8v3zTKv5QHZw4qHIkJQg==)  # Encrypted!
    
app:
  api-key: ENC(aB3dF6gH9jK2lM5nP8qR1sT4v)
  secret-token: ENC(xY7zA8bC9dE0fG1hI2jK3lM4n)
```

**Format**: `ENC(encrypted_value)`

Jasypt automatically decrypts at runtime using the password from environment variable.

#### Step 5: Configure Jasypt (Optional)

**application.yml:**

```yaml

    string-output-type: base64

```

Or use stronger encryption:


## Best Practices

### DO ✅

1. **Use profiles** - Separate configs for dev/test/prod
2. **Externalize secrets** - Never commit passwords to VCS
3. **Use @ConfigurationProperties** - For grouped, validated configuration
4. **Enable validation** - Fail fast on misconfiguration
5. **Use kebab-case** - In YAML/properties files
6. **Document properties** - Add JavaDoc for metadata generation
7. **Use environment variables** - For deployment-specific values
8. **Encrypt sensitive data** - Use Jasypt or vault solutions
9. **Set defaults** - Provide sensible defaults in code
10. **Test configuration** - Unit test your @ConfigurationProperties classes

### DON'T ❌

1. **Don't hardcode secrets** - Use environment variables
2. **Don't commit production configs** - Use profiles and externalization
3. **Don't use @Value everywhere** - Prefer @ConfigurationProperties for related settings
4. **Don't skip validation** - Catch config errors early
5. **Don't mix formats** - Pick YAML or properties, not both
6. **Don't forget .gitignore** - Exclude `application-local.yml`
7. **Don't use production passwords in dev** - Use different values per environment
8. **Don't over-complicate** - Keep configuration simple and flat when possible

## Summary

### Key Takeaways

1. **17 levels of precedence** - Command-line > env vars > profiles > defaults
2. **YAML vs Properties** - Both work, YAML better for complex configs
3. **Profiles** - Essential for environment-specific configuration
4. **@ConfigurationProperties** - Type-safe, validated, testable
5. **Relaxed binding** - Flexible property naming (kebab, camel, snake)
6. **Validation** - JSR-303 annotations prevent misconfiguration
7. **Encryption** - Jasypt for encrypting sensitive properties
8. **Metadata** - Enable IDE autocomplete with configuration processor
