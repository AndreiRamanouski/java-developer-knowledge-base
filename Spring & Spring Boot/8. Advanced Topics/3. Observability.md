## Overview

Observability is the ability to understand the internal state of a system based on its external outputs. The three pillars of observability are **Logging** (what happened), **Metrics** (how much/how fast), and **Tracing** (where did the request go). Together, they provide complete visibility into application behavior.

**Key Concept**: Monitoring tells you _when_ something is wrong. Observability helps you understand _why_ it's wrong. Modern applications need all three pillars working together.

## 1. Logging

### SLF4J and Logback

```
┌─────────────────────────────────────────────────────────────┐
│              Logging Architecture                            │
└─────────────────────────────────────────────────────────────┘

Application Code
      ↓
   SLF4J API (Simple Logging Facade for Java)
      ↓
SLF4J Binding
      ↓
Logback Implementation (default in Spring Boot)
      ↓
Appenders (Console, File, etc.)

Benefits of SLF4J:
✅ Abstraction layer (switch implementations without code changes)
✅ Parameterized logging (better performance)
✅ Industry standard
✅ Spring Boot default
```

```java
import org.slf4j.LoggerFactory;

@Service
public class UserService {

    private static final Logger log = LoggerFactory.getLogger(UserService.class);

    public User createUser(UserRequest request) {
        // ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

        log.error("Failed to create user: {}", request.getEmail(), exception);
        
        // WARN: Recoverable issues, degraded functionality
        log.warn("Invalid email format: {}", request.getEmail());
        
        // INFO: Important business events (default level)
        log.info("Creating user: {}", request.getEmail());
        
        // DEBUG: Detailed diagnostic information
        log.debug("User request details: {}", request);
        
        // TRACE: Very detailed, often too verbose
        log.trace("Entering createUser method");
        
        // ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
        // Parameterized Logging (efficient)
        // ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
        
        // ✅ GOOD: Parameterized (only formats if level enabled)
        log.debug("Processing user {} with role {}", user.getId(), user.getRole());
        
        // ❌ BAD: String concatenation (always executes)
        log.debug("Processing user " + user.getId() + " with role " + user.getRole());
        
        // ✅ Multiple parameters
        log.info("User {} created with id {} in {}ms", 
            request.getEmail(), user.getId(), duration);
        
        // ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
        // Exception Logging
        // ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
        
        try {
            return processUser(request);
        } catch (Exception e) {
            // ✅ GOOD: Exception as last parameter (includes stack trace)
            log.error("Failed to create user: {}", request.getEmail(), e);
            
            // ❌ BAD: Exception in message (loses stack trace)
            log.error("Failed to create user: " + e.getMessage());
            
            throw e;
        }
    }
}
```

### MDC (Mapped Diagnostic Context)

```java
/**
 * MDC - Thread-local context for logging
 */
import org.slf4j.MDC;

@Component
public class RequestFilter implements Filter {
    
    private static final Logger log = LoggerFactory.getLogger(RequestFilter.class);
    
    @Override
    public void doFilter(ServletRequest request, ServletResponse response, 
                        FilterChain chain) throws IOException, ServletException {
        
        HttpServletRequest httpRequest = (HttpServletRequest) request;
        
        try {
            // ✅ Add context to MDC (available in all logs in this thread)
            MDC.put("userId", getUserId(httpRequest));
            MDC.put("sessionId", httpRequest.getSession().getId());
            MDC.put("requestId", generateRequestId());
            MDC.put("ipAddress", httpRequest.getRemoteAddr());
            
            // All logs in this thread will include MDC values
            log.info("Processing request: {}", httpRequest.getRequestURI());
            
            chain.doFilter(request, response);
            
        } finally {
            // ✅ CRITICAL: Always clear MDC (thread pool reuse!)
            MDC.clear();
        }
    }
    
    private String generateRequestId() {
        return UUID.randomUUID().toString();
    }
}

/**
 * Using MDC in Logback Pattern
 */
// logback-spring.xml
// Pattern: %d{yyyy-MM-dd HH:mm:ss} [%X{requestId}] [%X{userId}] %-5level %logger{36} - %msg%n

// Output:
// 2024-01-15 10:30:45 [abc-123] [user-456] INFO  UserService - Creating user

/**
 * MDC with Async Processing
 */
@Service
public class AsyncService {
    
    @Async
    public CompletableFuture<Void> processAsync() {
        // ❌ PROBLEM: MDC not propagated to async threads by default
        
        // ✅ SOLUTION: Copy MDC before async execution
        Map<String, String> mdcContext = MDC.getCopyOfContextMap();
        
        return CompletableFuture.runAsync(() -> {
            // Restore MDC in async thread
            if (mdcContext != null) {
                MDC.setContextMap(mdcContext);
            }
            
            try {
                log.info("Processing asynchronously");
                // Business logic
            } finally {
                MDC.clear();
            }
        });
    }
}

/**
 * MDC Decorator for Thread Pool
 */
@Configuration
public class AsyncConfig {
    
    @Bean
    public Executor taskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(10);
        executor.setMaxPoolSize(20);
        
        // Decorate to propagate MDC
        executor.setTaskDecorator(new MdcTaskDecorator());
        
        executor.initialize();
        return executor;
    }
}

public class MdcTaskDecorator implements TaskDecorator {
    
    @Override
    public Runnable decorate(Runnable runnable) {
        // Capture MDC from parent thread
        Map<String, String> contextMap = MDC.getCopyOfContextMap();
        
        return () -> {
            try {
                // Set MDC in child thread
                if (contextMap != null) {
                    MDC.setContextMap(contextMap);
                }
                runnable.run();
            } finally {
                MDC.clear();
            }
        };
    }
}
```

### Correlation IDs

```java
/**
 * Correlation ID for Distributed Tracing
 */
@Component
public class CorrelationIdFilter implements Filter {
    
    private static final String CORRELATION_ID_HEADER = "X-Correlation-ID";
    private static final String CORRELATION_ID_MDC_KEY = "correlationId";
    
    @Override
    public void doFilter(ServletRequest request, ServletResponse response, 
                        FilterChain chain) throws IOException, ServletException {
        
        HttpServletRequest httpRequest = (HttpServletRequest) request;
        HttpServletResponse httpResponse = (HttpServletResponse) response;
        
        // Get correlation ID from header or generate new
        String correlationId = httpRequest.getHeader(CORRELATION_ID_HEADER);
        if (correlationId == null || correlationId.isEmpty()) {
            correlationId = UUID.randomUUID().toString();
        }
        
        // Add to MDC (appears in all logs)
        MDC.put(CORRELATION_ID_MDC_KEY, correlationId);
        
        // Add to response headers
        httpResponse.addHeader(CORRELATION_ID_HEADER, correlationId);
        
        try {
            chain.doFilter(request, response);
        } finally {
            MDC.remove(CORRELATION_ID_MDC_KEY);
        }
    }
}

/**
 * Propagating Correlation ID to External Services
 */
@Configuration
public class RestTemplateConfig {
    
    @Bean
    public RestTemplate restTemplate() {
        RestTemplate restTemplate = new RestTemplate();
        
        // Add interceptor to propagate correlation ID
        restTemplate.getInterceptors().add((request, body, execution) -> {
            String correlationId = MDC.get("correlationId");
            if (correlationId != null) {
                request.getHeaders().add("X-Correlation-ID", correlationId);
            }
            return execution.execute(request, body);
        });
        
        return restTemplate;
    }
}

/**
 * WebClient (Reactive) with Correlation ID
 */
@Configuration
public class WebClientConfig {
    
    @Bean
    public WebClient webClient() {
        return WebClient.builder()
            .filter((request, next) -> {
                String correlationId = MDC.get("correlationId");
                if (correlationId != null) {
                    return next.exchange(
                        ClientRequest.from(request)
                            .header("X-Correlation-ID", correlationId)
                            .build()
                    );
                }
                return next.exchange(request);
            })
            .build();
    }
}
```

## 3. Tracing

### Distributed Tracing Concepts

```
┌─────────────────────────────────────────────────────────────┐
│            Distributed Tracing Terminology                   │
└─────────────────────────────────────────────────────────────┘

Trace: Complete journey of a request across all services
  │
  ├─ Span: Single unit of work (e.g., HTTP request, DB query)
  │   ├─ Span ID: Unique identifier for this span
  │   ├─ Trace ID: Identifier linking all spans in a trace
  │   ├─ Parent Span ID: ID of parent span (for hierarchy)
  │   ├─ Start time & Duration
  │   ├─ Tags: Key-value metadata (http.method, db.type)
  │   └─ Logs: Timestamped events within span
  │
  └─ Context Propagation: Passing trace info between services

Example Request Flow:
┌────────────────────────────────────────────────────────────┐
│ Trace ID: abc123                                           │
├────────────────────────────────────────────────────────────┤
│ Span 1: API Gateway (100ms)                              │
│   ├─ Span 2: User Service (80ms)                         │
│   │   ├─ Span 3: Database Query (20ms)                   │
│   │   └─ Span 4: Cache Lookup (5ms)                      │
│   └─ Span 5: Order Service (60ms)                        │
│       └─ Span 6: Payment Service (40ms)                  │
└────────────────────────────────────────────────────────────┘
```

```java
 * Custom Spans

@Service
public class UserService {
    
    @Autowired
    private Tracer tracer;
    
    public User createUser(UserRequest request) {
        // Create custom span
        Span span = tracer.nextSpan().name("createUser").start();
        
        try (Tracer.SpanInScope ws = tracer.withSpan(span)) {
            // Add tags (metadata)
            span.tag("user.email", request.getEmail());
            span.tag("user.role", request.getRole());
            
            // Add events (logs)
            span.event("validation-started");
            
            validateUser(request);
            
            span.event("validation-completed");
            
            // Database operation (auto-instrumented)
            User user = userRepository.save(toEntity(request));
            
            span.tag("user.id", user.getId().toString());
            
            return user;
            
        } finally {
            span.end();  // Complete span
        }
    }
    
    // ✅ Using annotation (simpler)
    @NewSpan(name = "searchUsers")
    public List<User> searchUsers(@SpanTag("query") String query) {

    }
```

## Summary & Best Practices

### Logging Best Practices

✅ Use SLF4J API (abstraction layer)  
✅ Parameterized logging (performance)  
✅ Appropriate log levels (ERROR, WARN, INFO, DEBUG, TRACE)  
✅ Include context (MDC, correlation IDs)  
✅ Structured logging (JSON for centralized systems)  
✅ Async appenders (performance)  
✅ Log rotation (prevent disk full)  
✅ Never log sensitive data (passwords, tokens, PII)  
✅ Clean up MDC in finally blocks  
✅ Use correlation IDs for distributed tracing

### Metrics Best Practices

✅ Use Micrometer (abstraction for metrics)  
✅ Meaningful metric names (domain-specific)  
✅ Add tags for filtering/grouping  
✅ Percentiles for latency metrics  
✅ Counters for totals, Gauges for current values  
✅ Timers for duration and rate  
✅ Monitor business metrics (not just technical)  
✅ Set up alerting (high latency, errors)  
✅ Create dashboards (Grafana)  
✅ Review metrics regularly

### Tracing Best Practices

✅ Use sampling in production (10-20%)  
✅ Meaningful span names  
✅ Add tags for important context  
✅ Propagate trace IDs to logs (correlation)  
✅ Use automatic instrumentation when possible  
✅ Custom spans for critical operations  
✅ Include trace ID in error messages  
✅ Set up trace retention policy  
✅ Monitor trace sampling rate  
✅ Review slow traces regularly
