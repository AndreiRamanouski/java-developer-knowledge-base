## Overview

Performance optimization in Spring Boot involves improving application startup time, runtime efficiency, memory usage, and throughput. This guide covers techniques from application configuration to JVM tuning and profiling tools.

**Key Concept**: Optimization is about trade-offs. Always measure before and after optimization. Profile to find bottlenecks before optimizing. Premature optimization is the root of all evil, but understanding performance characteristics is essential.

---

## 1. Application Startup Optimization

### Baseline Performance

```
┌─────────────────────────────────────────────────────────────┐
│           Typical Spring Boot Startup Time                   │
└─────────────────────────────────────────────────────────────┘

Minimal Application (REST API only):
- Startup: 2-4 seconds
- Memory: 100-200 MB

Typical Application (JPA + Security + Caching):
- Startup: 5-8 seconds
- Memory: 300-500 MB

Large Application (many dependencies):
- Startup: 10-15 seconds
- Memory: 500-800 MB

Goal: Reduce startup time for:
- Faster development feedback
- Faster container startup (Kubernetes)
- Serverless functions (AWS Lambda)
```

### Lazy Initialization

```properties
# application.properties

# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
# Lazy Initialization (Spring Boot 2.2+)
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

# Enable lazy initialization globally
spring.main.lazy-initialization=true

# Impact:
# ✅ Faster startup (beans created on-demand)
# ✅ Lower initial memory usage
# ❌ Slower first request (bean initialization happens then)
# ❌ Configuration errors discovered later (not at startup)

# Recommendation: Use in development, not production
# Production: Fast startup matters less than catching errors early
```

```java
/**
 * Selective Lazy Initialization
 */
@Configuration
public class AppConfig {
    
    // ✅ Eager initialization (default)
    @Bean
    public CriticalService criticalService() {
        return new CriticalService();
    }
    
    // ✅ Lazy initialization for specific bean
    @Bean
    @Lazy
    public ExpensiveService expensiveService() {
        return new ExpensiveService();
        // Only created when first accessed
    }
    
    // ✅ Lazy on component
    @Component
    @Lazy
    public class ReportGenerator {
        // Only created when first injected
    }
}

/**
 * Lazy vs Eager - Trade-offs
 */
/*
Eager (Default):
✅ All beans initialized at startup
✅ Configuration errors caught early
✅ Consistent performance after startup
❌ Slower startup
❌ Higher initial memory

Lazy:
✅ Fast startup
✅ Lower initial memory
✅ Only used beans are created
❌ First request slower
❌ Configuration errors caught late
❌ Unpredictable performance
*/
```

### Component Scanning Optimization

```java
/**
 * Limit Component Scanning
 */

// ❌ BAD: Scans entire classpath
@SpringBootApplication
public class Application {
}

// ✅ GOOD: Scan specific packages
@SpringBootApplication(scanBasePackages = {
    "com.example.api",
    "com.example.service",
    "com.example.repository"
})
public class Application {
}

// ✅ BETTER: Specify base package classes
@SpringBootApplication(scanBasePackageClasses = {
    ApiPackageMarker.class,
    ServicePackageMarker.class
})
public class Application {
}

/**
 * Manual Bean Registration (No Scanning)
 */
@Configuration
public class ManualBeanConfig {
    
    @Bean
    public UserService userService() {
        return new UserService();
    }
    
    @Bean
    public UserRepository userRepository() {
        return new UserRepository();
    }
    
    // No @ComponentScan → Faster startup
    // Trade-off: More manual configuration
}
```

```
- Startup: 2-5 seconds
- Memory: 300-500 MB
- Peak Performance: High (JIT compilation)
- Warmup: Required

- Startup: 0.1-0.3 seconds (10-50x faster!)
- Memory: 50-100 MB (3-5x less!)
- Peak Performance: Good (ahead-of-time compilation)

Trade-offs:
✅ Instant startup
✅ Low memory footprint
✅ Perfect for serverless, containers
❌ Longer build times (minutes)
❌ Limited reflection support
❌ Some libraries incompatible
❌ Slightly lower peak performance
```

## 2. Runtime Performance

### HikariCP Connection Pooling

```properties
# application.properties

# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
# HikariCP Configuration (default in Spring Boot)
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

# Pool size (most important setting)
spring.datasource.hikari.minimum-idle=10
spring.datasource.hikari.maximum-pool-size=20

# Rule of thumb for pool size:
# max-pool-size = (core_count * 2) + effective_spindle_count
# For 4-core CPU with SSD: (4 * 2) + 1 = 9-10 connections
# More is NOT better! Too many connections waste resources

# Connection timeout
spring.datasource.hikari.connection-timeout=30000  # 30 seconds

# Idle timeout (close idle connections)
spring.datasource.hikari.idle-timeout=600000  # 10 minutes

# Max lifetime (close connections after)
spring.datasource.hikari.max-lifetime=1800000  # 30 minutes

# Leak detection (find connection leaks)
spring.datasource.hikari.leak-detection-threshold=60000  # 60 seconds

# Validation
spring.datasource.hikari.connection-test-query=SELECT 1
spring.datasource.hikari.validation-timeout=5000

# Pool name (for monitoring)
spring.datasource.hikari.pool-name=HikariPool-API

# Register MBeans (for JMX monitoring)
spring.datasource.hikari.register-mbeans=true
```

```java
/**
 * HikariCP Monitoring
 */
@Component
public class HikariCPMonitor {
    
    @Autowired
    private DataSource dataSource;
    
    @Scheduled(fixedRate = 60000)  // Every minute
    public void logPoolStats() {
        if (dataSource instanceof HikariDataSource) {
            HikariDataSource hikariDS = (HikariDataSource) dataSource;
            HikariPoolMXBean poolMXBean = hikariDS.getHikariPoolMXBean();
            
            log.info("HikariCP Stats: " +
                "Active={}, Idle={}, Waiting={}, Total={}", 
                poolMXBean.getActiveConnections(),
                poolMXBean.getIdleConnections(),
                poolMXBean.getThreadsAwaitingConnection(),
                poolMXBean.getTotalConnections()
            );
        }
    }
}

/**
 * Common Connection Pool Issues
 */
/*
Issue 1: Pool exhaustion
Symptom: "Connection is not available, request timed out"
Cause: All connections in use, new requests wait
Solution: 
  - Increase pool size (if database can handle it)
  - Fix connection leaks (not closing connections)
  - Reduce query time
  - Add connection timeout

Issue 2: Connection leaks
Symptom: Gradual performance degradation
Cause: Connections not returned to pool
Solution:
  - Enable leak detection
  - Use try-with-resources
  - Review @Transactional boundaries

Issue 3: Pool too large
Symptom: High memory usage, context switching
Cause: Too many database connections
Solution:
  - Reduce pool size
  - Formula: (CPU cores * 2) + 1
*/
```

```properties
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

# Don't increase beyond CPU cores!

# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
# Async Thread Pool
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

```java
/**
 * Custom Thread Pool Configuration
 */
@Configuration
@EnableAsync
public class AsyncConfig {
    
    @Bean(name = "taskExecutor")
    public Executor taskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        
        // Core pool size (always alive)
        executor.setCorePoolSize(10);
        
        // Max pool size (creates up to this many threads)
        executor.setMaxPoolSize(20);
        
        // Queue capacity (waits here if all threads busy)
        executor.setQueueCapacity(500);
        
        // Thread name prefix
        executor.setThreadNamePrefix("async-task-");
        
        // Rejection policy (what to do when queue full)
        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
        // CallerRunsPolicy: Execute in caller thread
        // AbortPolicy: Throw exception (default)
        // DiscardPolicy: Silently discard
        // DiscardOldestPolicy: Discard oldest task
        
        // Shutdown behavior
        executor.setWaitForTasksToCompleteOnShutdown(true);
        executor.setAwaitTerminationSeconds(60);
        
        executor.initialize();
        return executor;
    }
}

/**
 * Using Custom Thread Pool
 */
@Service
public class EmailService {
    
    @Async("taskExecutor")
    public CompletableFuture<Void> sendEmailAsync(String to, String subject) {
        // Executes on custom thread pool
        sendEmail(to, subject);
        return CompletableFuture.completedFuture(null);
    }
}

/**
 * Thread Pool Sizing Guidelines
 */
/*
CPU-bound tasks:
- Pool size = CPU cores (or CPU cores + 1)
- Example: Image processing, encryption

I/O-bound tasks:
- Pool size = CPU cores * 2 (or more)
- Example: HTTP calls, database queries

Formula: Pool size = CPU cores * (1 + wait time / compute time)
Example: 4 cores, 90% waiting (network I/O), 10% computing
  = 4 * (1 + 9) = 40 threads
*/
```

### JVM Tuning


```
┌─────────────────────────────────────────────────────────────┐
│              Garbage Collector Comparison                    │
└─────────────────────────────────────────────────────────────┘

Serial GC:
- Single-threaded
- Pause all threads
- Best for: Small heaps (< 100 MB)

Parallel GC:
- Multi-threaded
- Pause all threads
- Best for: Throughput applications

G1GC (default Java 11+):
- Low pause times (< 200ms)
- Concurrent marking
- Best for: Most applications

ZGC (Java 15+):
- Ultra-low pause times (< 10ms)
- Concurrent everything
- Best for: Low-latency applications

Shenandoah GC:
- Low pause times (< 10ms)
- Concurrent compaction
- Best for: Low-latency applications

Recommendation: Use G1GC (default), upgrade to ZGC if latency critical
```

```java
/**
 * Application Performance Monitoring
 */
@Component
public class PerformanceMonitor {
    
    @Autowired
    private MeterRegistry meterRegistry;
    
    // ✅ Monitor method execution time
    @Around("@annotation(com.example.Monitored)")
    public Object monitorPerformance(ProceedingJoinPoint joinPoint) throws Throwable {
        String methodName = joinPoint.getSignature().getName();
        
        Timer.Sample sample = Timer.start(meterRegistry);
        
        try {
            return joinPoint.proceed();
        } finally {
            sample.stop(Timer.builder("method.execution.time")
                .tag("method", methodName)
                .register(meterRegistry));
        }
    }
    
    // ✅ Monitor database query time
    @Component
    public class DatabaseQueryMonitor {
        
        @EventListener
        public void onQueryExecuted(QueryExecutionEvent event) {
            meterRegistry.timer("database.query.time",
                "query", event.getQueryType())
                .record(event.getDuration(), TimeUnit.MILLISECONDS);
        }
    }
}

/**
```
```java
 * Runtime Memory Monitoring
 */
@Component
public class MemoryMonitor {
    
    private final Runtime runtime = Runtime.getRuntime();
    
    @Scheduled(fixedRate = 60000)  // Every minute
    public void logMemoryUsage() {
        long maxMemory = runtime.maxMemory();
        long allocatedMemory = runtime.totalMemory();
        long freeMemory = runtime.freeMemory();
        long usedMemory = allocatedMemory - freeMemory;
        
        log.info("Memory Usage: " +
            "Max={}MB, Allocated={}MB, Used={}MB, Free={}MB",
            maxMemory / 1024 / 1024,
            allocatedMemory / 1024 / 1024,
            usedMemory / 1024 / 1024,
            freeMemory / 1024 / 1024);
        
        // Alert if memory usage high
        double usagePercent = (double) usedMemory / maxMemory * 100;
        if (usagePercent > 90) {
            log.warn("High memory usage: {}%", usagePercent);
        }
    }
}

/**
 * Memory Leak Detection
 */
@Component
public class MemoryLeakDetector {
    
    private final MemoryMXBean memoryMXBean = ManagementFactory.getMemoryMXBean();
    private long lastUsedMemory = 0;
    private int increasingTrend = 0;
    
    @Scheduled(fixedRate = 300000)  // Every 5 minutes
    public void detectMemoryLeak() {
        long currentUsed = memoryMXBean.getHeapMemoryUsage().getUsed();
        
        if (currentUsed > lastUsedMemory) {
            increasingTrend++;
            
            if (increasingTrend >= 12) {  // 1 hour of increasing memory
                log.error("Potential memory leak detected! " +
                    "Memory has been increasing for 1 hour");
                
                // Trigger heap dump
                dumpHeap();
            }
        } else {
            increasingTrend = 0;
        }
        
        lastUsedMemory = currentUsed;
    }
    
    private void dumpHeap() {
        try {
            MBeanServer server = ManagementFactory.getPlatformMBeanServer();
            HotSpotDiagnosticMXBean mxBean = ManagementFactory.newPlatformMXBeanProxy(
                server, "com.sun.management:type=HotSpotDiagnostic",
                HotSpotDiagnosticMXBean.class);
            
            String filename = "leak-" + System.currentTimeMillis() + ".hprof";
            mxBean.dumpHeap(filename, true);
            
            log.info("Heap dump saved to: {}", filename);
        } catch (Exception e) {
            log.error("Failed to dump heap", e);
        }
    }
}
```

### Common Memory Leaks in Spring

```java
/**
 * Memory Leak #1: Unbounded Caches
 */
// ❌ BAD: No size limit
@Component
public class UserCache {
    private final Map<Long, User> cache = new HashMap<>();  // LEAK!
    
    public void cacheUser(User user) {
        cache.put(user.getId(), user);  // Grows forever
    }
}

// ✅ GOOD: Bounded cache
@Component
public class UserCache {
    private final Map<Long, User> cache = new LinkedHashMap<Long, User>(100, 0.75f, true) {
        @Override
        protected boolean removeEldestEntry(Map.Entry<Long, User> eldest) {
            return size() > 100;  // Max 100 entries
        }
    };
    
    // Or use Caffeine/Guava cache
    private final Cache<Long, User> cache = Caffeine.newBuilder()
        .maximumSize(1000)
        .expireAfterWrite(10, TimeUnit.MINUTES)
        .build();
}

/**
 * Memory Leak #2: ThreadLocal Not Cleaned
 */
// ❌ BAD: ThreadLocal never removed
@Component
public class RequestContext {
    private static final ThreadLocal<User> currentUser = new ThreadLocal<>();
    
    public void setCurrentUser(User user) {
        currentUser.set(user);  // LEAK in thread pool!
    }
    
    // Thread reused in pool → ThreadLocal value remains
}

// ✅ GOOD: Clean up ThreadLocal
@Component
public class RequestContext {
    private static final ThreadLocal<User> currentUser = new ThreadLocal<>();
    
    public void setCurrentUser(User user) {
        currentUser.set(user);
    }
    
    public void clear() {
        currentUser.remove();  // Always clean up!
    }
}

@Component
public class RequestInterceptor implements HandlerInterceptor {
    
    @Autowired
    private RequestContext requestContext;
    
    @Override
    public void afterCompletion(HttpServletRequest request, 
                               HttpServletResponse response,
                               Object handler, 
                               Exception ex) {
        requestContext.clear();  // Clean up after request
    }
}

/**
 * Memory Leak #3: Event Listeners Not Unregistered
 */
// ❌ BAD: Listener never removed
@Component
public class UserActivityTracker {
    
    @EventListener
    public void onUserLogin(UserLoginEvent event) {
        // Processes event
    }
    
    // If bean destroyed, listener remains registered → LEAK
}

// ✅ GOOD: Proper cleanup
@Component
public class UserActivityTracker implements DisposableBean {
    
    @Autowired
    private ApplicationEventPublisher eventPublisher;
    
    @EventListener
    public void onUserLogin(UserLoginEvent event) {
        // Process event
    }
    
    @Override
    public void destroy() {
        // Clean up resources
        // Spring handles @EventListener cleanup automatically
    }
}

/**
 * Memory Leak #4: Unclosed Resources
 */
// ❌ BAD: Connection not closed
@Service
public class ReportService {
    
    public void generateReport() {
        Connection conn = dataSource.getConnection();  // LEAK!
        // If exception occurs, connection never closed
        Statement stmt = conn.createStatement();
        ResultSet rs = stmt.executeQuery("SELECT * FROM users");
        // Process results
    }
}

// ✅ GOOD: Use try-with-resources
@Service
public class ReportService {
    
    public void generateReport() {
        try (Connection conn = dataSource.getConnection();
             Statement stmt = conn.createStatement();
             ResultSet rs = stmt.executeQuery("SELECT * FROM users")) {
            
            // Process results
            
        } catch (SQLException e) {
            log.error("Error generating report", e);
        }
        // Resources automatically closed
    }
}

/**
 * Memory Leak #5: Large Collections in Session
 */
// ❌ BAD: Large object in session
@RestController
public class UserController {
    
    @GetMapping("/users")
    public List<User> getAllUsers(HttpSession session) {
        List<User> users = userService.findAll();  // 100,000 users
        session.setAttribute("allUsers", users);  // LEAK!
        return users;
    }
}

// ✅ GOOD: Don't store large data in session
@RestController
public class UserController {
    
    @GetMapping("/users")
    public List<User> getAllUsers() {
        return userService.findAll();
        // Let client cache if needed
    }
}

/**
 * Memory Leak #6: Static Collections
 */
// ❌ BAD: Static collection grows forever
public class UserRegistry {
    private static final List<User> users = new ArrayList<>();  // LEAK!
    
    public static void register(User user) {
        users.add(user);  // Never removed
    }
}

// ✅ GOOD: Use weak references or bounded collections
public class UserRegistry {
    private static final Map<Long, WeakReference<User>> users = new ConcurrentHashMap<>();
    
    public static void register(User user) {
        users.put(user.getId(), new WeakReference<>(user));
        // Garbage collected when no strong references
    }
}
```

```bash
# G1GC Tuning (Recommended)
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

java -XX:+UseG1GC \
     -Xms4g -Xmx4g \
     -XX:MaxGCPauseMillis=200 \
     -XX:G1HeapRegionSize=16m \
     -XX:InitiatingHeapOccupancyPercent=45 \
     -XX:G1ReservePercent=10 \
     -XX:G1NewSizePercent=5 \
     -XX:G1MaxNewSizePercent=60 \
     -jar app.jar

# MaxGCPauseMillis: Target pause time (not guaranteed)
# G1HeapRegionSize: Region size (1MB-32MB, default: heap/2048)
# InitiatingHeapOccupancyPercent: When to start marking
# G1ReservePercent: Reserve space to avoid full GC
# G1NewSizePercent: Min young generation size
# G1MaxNewSizePercent: Max young generation size

# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
# ZGC Tuning (Ultra-Low Latency)
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

# ConcGCThreads: Concurrent GC threads (default: CPU cores / 8)
# ZCollectionInterval: Min time between GCs
```

```
│                  GC Tuning Guidelines                        │
└─────────────────────────────────────────────────────────────┘

Symptom: Frequent young GCs
Cause: Young generation too small
Solution: Increase -XX:G1NewSizePercent

Symptom: Long pause times
Cause: Old generation full, needs compaction
Solution: 
  - Increase heap size
  - Reduce -XX:MaxGCPauseMillis target
  - Reduce object allocation rate

Symptom: Full GCs occurring
Cause: Heap too small or memory leak
Solution:
  - Increase heap size
  - Find memory leak (heap dump)
  - Reduce -XX:InitiatingHeapOccupancyPercent

Symptom: High CPU usage from GC
Cause: Too much garbage created
Solution:
  - Optimize code (reduce allocations)
  - Increase heap size
  - Object pooling for frequently created objects

General Rules:
✅ Set -Xms = -Xmx (avoid resizing)
✅ Heap should be 25-50% of system memory
✅ Leave memory for OS and off-heap
✅ Monitor GC logs regularly
✅ Tune after measuring (don't guess)
```

## Summary & Best Practices

### Performance Optimization Checklist

**Startup Time:** ✅ Use lazy initialization (development only)  
✅ Exclude unused auto-configurations  
✅ Limit component scanning scope  
✅ Consider GraalVM native images

**Runtime Performance:** ✅ Configure HikariCP properly (pool size = CPU cores * 2 + 1)  
✅ Tune thread pools (CPU-bound: cores, I/O-bound: cores * 2+)  
✅ Enable compression (gzip)  
✅ Use caching effectively  
✅ Optimize database queries (indexes, pagination)

**JVM Tuning:** ✅ Set -Xms = -Xmx (avoid resizing)  
✅ Use G1GC (default) or ZGC (low latency)  
✅ Enable GC logging  
✅ Configure heap dumps on OOM  
✅ Monitor GC metrics

**Memory:** ✅ Profile memory usage regularly  
✅ Fix memory leaks (ThreadLocal, caches, listeners)  
✅ Use bounded caches  
✅ Close resources (try-with-resources)  
✅ Avoid large collections in sessions

**Monitoring:** ✅ Enable Actuator endpoints  
✅ Export metrics (Prometheus, Grafana)  
✅ Set up alerts (high memory, slow requests)  
✅ Profile production periodically

### Common Performance Issues

|Issue|Symptom|Solution|
|---|---|---|
|**Slow startup**|10+ seconds|Lazy init, reduce auto-config|
|**Connection pool exhausted**|Timeouts|Increase pool size, fix leaks|
|**High memory**|OOM errors|Profile, fix leaks, increase heap|
|**Slow requests**|High latency|Profile CPU, optimize queries|
|**Frequent GC**|CPU spikes|Reduce allocations, tune GC|
