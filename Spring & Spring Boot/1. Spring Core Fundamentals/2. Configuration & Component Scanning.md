## Overview

Spring's configuration system has evolved from XML to annotation-based and Java-based approaches. Understanding how @Configuration works, component scanning mechanisms, profiles, and property management is essential for building flexible Spring applications.

```
┌─────────────────────────────────────────────────────────────────┐
│           SPRING CONFIGURATION ARCHITECTURE                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Configuration Sources                                          │
│  ├── Java Config (@Configuration, @Bean)                       │
│  ├── Component Scanning (@Component, @Service, etc.)           │
│  ├── XML Config (legacy)                                       │
│  └── Property Files (.properties, .yml)                        │
│       ↓                                                         │
│  ConfigurationClassPostProcessor                                │
│  ├── Parses @Configuration classes                             │
│  ├── Creates CGLIB proxies (Full mode)                         │
│  ├── Processes @ComponentScan                                  │
│  ├── Handles @Import                                           │
│  └── Evaluates @Conditional                                    │
│       ↓                                                         │
│  BeanDefinition Registry                                        │
│  ├── Stores all bean definitions                               │
│  ├── Applies @Profile filters                                  │
│  └── Resolves property values                                  │
│       ↓                                                         │
│  Bean Creation                                                  │
│  └── Instantiate and initialize beans                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 1. Java-Based Configuration

### 1.1 @Configuration Deep Dive

**Basic @Configuration:**

```java
@Configuration
public class AppConfig {
    
    @Bean
    public UserService userService() {
        return new UserService(userRepository());
    }
    
    @Bean
    public UserRepository userRepository() {
        return new UserRepository();
    }
}
```

**CGLIB Proxies - Full Mode Magic:**

```java
/**
 * What happens behind the scenes:
 * Spring creates CGLIB proxy of AppConfig
 */
@Configuration
public class AppConfig {
    
    @Bean
    public ServiceA serviceA() {
        return new ServiceA(serviceB());  // Calls serviceB()
    }
    
    @Bean
    public ServiceB serviceB() {
        return new ServiceB();
    }
    
    @Bean
    public ServiceC serviceC() {
        return new ServiceC(serviceB());  // Calls serviceB() again
    }
}

/**
 * Without CGLIB proxy (incorrect behavior):
 * serviceA() creates ServiceB instance
 * serviceC() creates DIFFERENT ServiceB instance
 * Result: 2 ServiceB instances (wrong!)
 * 
 * With CGLIB proxy (correct behavior):
 * First call to serviceB() creates singleton
 * Second call to serviceB() returns same singleton
 * Result: 1 ServiceB instance shared by both (correct!)
 */
```

**How CGLIB Proxy Works:**

```java
/**
 * Spring generates something like this:
 */
public class AppConfig$$EnhancerBySpringCGLIB extends AppConfig {
    
    private Map<String, Object> beanCache = new HashMap<>();
    
    @Override
    public ServiceB serviceB() {
        // Check if bean already exists
        if (beanCache.containsKey("serviceB")) {
            return (ServiceB) beanCache.get("serviceB");
        }
        
        // Create new bean
        ServiceB bean = super.serviceB();
        beanCache.put("serviceB", bean);
        return bean;
    }
    
    @Override
    public ServiceA serviceA() {
        // When serviceA() calls serviceB(), it calls the proxied method
        // which returns the cached singleton
        return super.serviceA();
    }
}

/**
 * Key points:
 * 1. @Configuration class cannot be final
 * 2. @Bean methods cannot be final or private
 * 3. CGLIB proxy intercepts method calls
 * 4. Ensures singleton semantics for @Bean methods
 */
```

### 1.2 @Bean Method Inter-Dependencies

**Correct Way - Method Calls:**

```java
@Configuration
public class DatabaseConfig {
    
    @Bean
    public DataSource dataSource() {
        HikariDataSource dataSource = new HikariDataSource();
        dataSource.setJdbcUrl("jdbc:postgresql://localhost/mydb");
        dataSource.setUsername("user");
        dataSource.setPassword("pass");
        return dataSource;
    }
    
    @Bean
    public JdbcTemplate jdbcTemplate() {
        // Calling @Bean method - returns singleton
        return new JdbcTemplate(dataSource());
    }
    
    @Bean
    public UserRepository userRepository() {
        // Calling @Bean method again - same singleton
        return new UserRepository(dataSource());
    }
}
```

**Alternative - Method Parameters:**

```java
@Configuration
public class DatabaseConfig {
    
    @Bean
    public DataSource dataSource() {
        return new HikariDataSource();
    }
    
    @Bean
    public JdbcTemplate jdbcTemplate(DataSource dataSource) {
        // Spring injects the dataSource bean
        // Preferred approach - more testable
        return new JdbcTemplate(dataSource);
    }
    
    @Bean
    public UserRepository userRepository(DataSource dataSource) {
        return new UserRepository(dataSource);
    }
    
    // Can also inject multiple beans
    @Bean
    public UserService userService(
            UserRepository userRepository,
            EmailService emailService,
            @Qualifier("cacheManager") CacheManager cacheManager) {
        return new UserService(userRepository, emailService, cacheManager);
    }
}
```

**Comparison:**

```
Method Calls:
  ✓ Works within same @Configuration class
  ✓ Enforces singleton through CGLIB
  ✗ Tight coupling
  ✗ Hard to test
  ✗ Requires CGLIB proxy

Method Parameters:
  ✓ Loose coupling
  ✓ Easy to test
  ✓ Works across @Configuration classes
  ✓ More flexible
  ✓ Preferred approach
  ✗ Slightly more verbose
```

### 1.3 Lite Mode vs Full Mode @Configuration

**Full Mode (default with @Configuration):**

```java
@Configuration  // Full mode - CGLIB proxy
public class FullModeConfig {
    
    @Bean
    public ServiceA serviceA() {
        // Calling serviceB() returns singleton
        return new ServiceA(serviceB());
    }
    
    @Bean
    public ServiceB serviceB() {
        return new ServiceB();
    }
}

/**
 * Characteristics:
 * ✓ CGLIB proxy created
 * ✓ @Bean method calls intercepted
 * ✓ Singleton semantics enforced
 * ✓ Inter-bean dependencies work
 * ✗ Class cannot be final
 * ✗ Methods cannot be final/private
 * ✗ Slight performance overhead
 */
```

**Lite Mode (without @Configuration):**

```java
@Component  // Lite mode - NO CGLIB proxy
public class LiteModeConfig {
    
    @Bean
    public ServiceA serviceA() {
        // Calling serviceB() creates NEW instance!
        return new ServiceA(serviceB());  // ⚠️ Different instance
    }
    
    @Bean
    public ServiceB serviceB() {
        return new ServiceB();
    }
}

/**
 * Characteristics:
 * ✓ No CGLIB proxy
 * ✓ Class can be final
 * ✓ Methods can be final
 * ✓ Faster instantiation
 * ✗ @Bean method calls create new instances
 * ✗ Must use method parameters for dependencies
 * ✗ No singleton enforcement
 */
```

**When to Use Lite Mode:**

```java
/**
 * Use lite mode when:
 * 1. Beans don't have inter-dependencies
 * 2. Using method parameters for dependencies
 * 3. Performance is critical
 * 4. Don't need @Bean method calls
 */
@Component
public class SimpleConfig {
    
    @Bean
    public UserRepository userRepository(DataSource dataSource) {
        // All dependencies through parameters
        return new UserRepository(dataSource);
    }
    
    @Bean
    public UserService userService(UserRepository repository) {
        // No inter-bean method calls
        return new UserService(repository);
    }
}
```

**Explicit Lite Mode:**

```java
@Configuration(proxyBeanMethods = false)  // Explicit lite mode
public class LiteModeExplicit {
    
    @Bean
    public ServiceA serviceA(ServiceB serviceB) {
        // Must use parameters
        return new ServiceA(serviceB);
    }
    
    @Bean
    public ServiceB serviceB() {
        return new ServiceB();
    }
}

/**
 * Spring Boot 2.2+ uses this by default in:
 * - @SpringBootApplication
 * - Auto-configurations
 * For better startup performance
 */
```


**@Import with ImportSelector:**

```java
/**
 * Conditional import based on logic
 */
public class MyImportSelector implements ImportSelector {
    
    @Override
    public String[] selectImports(AnnotationMetadata metadata) {
        // Dynamic selection of configurations
        List<String> imports = new ArrayList<>();
        
        if (ClassUtils.isPresent("org.h2.Driver", null)) {
            imports.add("com.example.H2Config");
        }
        
        if (ClassUtils.isPresent("org.postgresql.Driver", null)) {
            imports.add("com.example.PostgresConfig");
        }
        
        return imports.toArray(new String[0]);
    }
}

@Configuration
@Import(MyImportSelector.class)
public class AppConfig {
    // Dynamically imports H2Config or PostgresConfig
}
```

**@Import with ImportBeanDefinitionRegistrar:**

```java
/**
 * Programmatically register bean definitions
 */
public class MyBeanRegistrar implements ImportBeanDefinitionRegistrar {
    
    @Override
    public void registerBeanDefinitions(
            AnnotationMetadata metadata,
            BeanDefinitionRegistry registry) {
        
        // Create bean definition
        BeanDefinition beanDefinition = BeanDefinitionBuilder
            .genericBeanDefinition(DynamicBean.class)
            .addPropertyValue("name", "dynamic")
            .getBeanDefinition();
        
        // Register it
        registry.registerBeanDefinition("dynamicBean", beanDefinition);
        
        // Can register multiple beans, conditionally
        if (someCondition()) {
            BeanDefinition another = ...;
            registry.registerBeanDefinition("anotherBean", another);
        }
    }
}

@Configuration
@Import(MyBeanRegistrar.class)
public class AppConfig {
    // Beans registered programmatically
}
```

### 1.5 Conditional Bean Registration

**@Conditional - Custom Condition:**

```java
/**
 * Custom condition
 */
public class LinuxCondition implements Condition {
    
    @Override
    public boolean matches(ConditionContext context, AnnotatedTypeMetadata metadata) {
        return context.getEnvironment()
            .getProperty("os.name")
            .toLowerCase()
            .contains("linux");
    }
}

@Configuration
public class ConditionalConfig {
    
    @Bean
    @Conditional(LinuxCondition.class)
    public FileService linuxFileService() {
        return new LinuxFileService();
    }
    
    @Bean
    @Conditional(WindowsCondition.class)
    public FileService windowsFileService() {
        return new WindowsFileService();
    }
}
```

**Spring Boot Conditional Annotations:**

```java
@Configuration
public class ConditionalBeansConfig {
    
    // Bean registered only if class present on classpath
    @Bean
    @ConditionalOnClass(DataSource.class)
    public DatabaseConfig databaseConfig() {
        return new DatabaseConfig();
    }
    
    // Bean registered only if class NOT present
    @Bean
    @ConditionalOnMissingClass("com.example.SpecialService")
    public DefaultService defaultService() {
        return new DefaultService();
    }
    
    // Bean registered only if property matches
    @Bean
    @ConditionalOnProperty(
        name = "app.feature.enabled",
        havingValue = "true",
        matchIfMissing = false
    )
    public FeatureService featureService() {
        return new FeatureService();
    }
    
    // Bean registered only if another bean NOT exists
    @Bean
    @ConditionalOnMissingBean(CustomService.class)
    public CustomService defaultCustomService() {
        return new DefaultCustomService();
    }
    
    // Bean registered only if another bean exists
    @Bean
    @ConditionalOnBean(DataSource.class)
    public JdbcTemplate jdbcTemplate(DataSource dataSource) {
        return new JdbcTemplate(dataSource);
    }
    
    // Bean registered only if expression evaluates to true
    @Bean
    @ConditionalOnExpression("${app.cache.enabled:false} and '${app.cache.type}' == 'redis'")
    public CacheManager redisCacheManager() {
        return new RedisCacheManager();
    }
    
    // Bean registered only in web environment
    @Bean
    @ConditionalOnWebApplication
    public WebConfig webConfig() {
        return new WebConfig();
    }
    
    // Bean registered only if specific profile active
    @Bean
    @ConditionalOnProfile("dev")
    public MockEmailService mockEmailService() {
        return new MockEmailService();
    }
}
```

**Combining Conditions:**

```java
@Configuration
public class ComplexConditionalConfig {
    
    @Bean
    @ConditionalOnClass(Redis.class)
    @ConditionalOnProperty(name = "cache.type", havingValue = "redis")
    @ConditionalOnMissingBean(CacheManager.class)
    public CacheManager redisCacheManager() {
        // Only registered if:
        // 1. Redis is on classpath
        // 2. cache.type=redis in properties
        // 3. No other CacheManager bean exists
        return new RedisCacheManager();
    }
}
```

**Custom Meta-Annotation:**

```java
/**
 * Create reusable conditional annotation
 */
@Target({ElementType.TYPE, ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
@Conditional(DatabaseCondition.class)
public @interface ConditionalOnDatabase {
    DatabaseType value();
}

enum DatabaseType {
    POSTGRES, MYSQL, H2
}

class DatabaseCondition implements Condition {
    @Override
    public boolean matches(ConditionContext context, AnnotatedTypeMetadata metadata) {
        Map<String, Object> attrs = metadata.getAnnotationAttributes(
            ConditionalOnDatabase.class.getName()
        );
        DatabaseType type = (DatabaseType) attrs.get("value");
        
        String driver = context.getEnvironment().getProperty("spring.datasource.driver-class-name");
        
        switch (type) {
            case POSTGRES:
                return driver != null && driver.contains("postgresql");
            case MYSQL:
                return driver != null && driver.contains("mysql");
            case H2:
                return driver != null && driver.contains("h2");
            default:
                return false;
        }
    }
}

// Usage
@Configuration
public class DatabaseSpecificConfig {
    
    @Bean
    @ConditionalOnDatabase(DatabaseType.POSTGRES)
    public PostgresSpecificService postgresService() {
        return new PostgresSpecificService();
    }
    
    @Bean
    @ConditionalOnDatabase(DatabaseType.MYSQL)
    public MySqlSpecificService mySqlService() {
        return new MySqlSpecificService();
    }
}
```

## 2. Component Scanning

### 2.1 @ComponentScan Filters

**Basic Component Scanning:**

```java
@Configuration
@ComponentScan(basePackages = "com.example")
public class AppConfig {
    // Scans com.example and all sub-packages
}

// Or with type-safe base packages
@Configuration
@ComponentScan(basePackageClasses = {
    UserService.class,  // Scans package of UserService
    OrderService.class  // Scans package of OrderService
})
public class AppConfig {
    // Type-safe - refactoring friendly
}
```

**Include Filters:**

```java
@Configuration
@ComponentScan(
    basePackages = "com.example",
    includeFilters = {
        // Include specific annotation
        @ComponentScan.Filter(
            type = FilterType.ANNOTATION,
            classes = MyCustomAnnotation.class
        ),
        
        // Include specific classes
        @ComponentScan.Filter(
            type = FilterType.ASSIGNABLE_TYPE,
            classes = {SpecialService.class, AnotherService.class}
        ),
        
        // Include by regex
        @ComponentScan.Filter(
            type = FilterType.REGEX,
            pattern = "com\\.example\\..*Service"
        ),
        
        // Include by AspectJ pattern
        @ComponentScan.Filter(
            type = FilterType.ASPECTJ,
            pattern = "com.example..*Service+"
        ),
        
        // Include by custom filter
        @ComponentScan.Filter(
            type = FilterType.CUSTOM,
            classes = MyTypeFilter.class
        )
    }
)
public class AppConfig {
}
```

**Exclude Filters:**

```java
@Configuration
@ComponentScan(
    basePackages = "com.example",
    excludeFilters = {
        // Exclude @Configuration classes
        @ComponentScan.Filter(
            type = FilterType.ANNOTATION,
            classes = Configuration.class
        ),
        
        // Exclude specific classes
        @ComponentScan.Filter(
            type = FilterType.ASSIGNABLE_TYPE,
            classes = LegacyService.class
        ),
        
        // Exclude by package
        @ComponentScan.Filter(
            type = FilterType.REGEX,
            pattern = "com\\.example\\.legacy\\..*"
        )
    }
)
public class AppConfig {
}
```

**Custom Type Filter:**

```java
public class MyTypeFilter implements TypeFilter {
    
    @Override
    public boolean match(MetadataReader metadataReader, 
                        MetadataReaderFactory metadataReaderFactory) {
        
        // Get class metadata
        ClassMetadata classMetadata = metadataReader.getClassMetadata();
        String className = classMetadata.getClassName();
        
        // Get annotation metadata
        AnnotationMetadata annotationMetadata = metadataReader.getAnnotationMetadata();
        
        // Custom logic
        // Example: Include only classes ending with "Impl"
        return className.endsWith("Impl");
        
        // Example: Include only classes with specific annotation value
        if (annotationMetadata.hasAnnotation(MyAnnotation.class.getName())) {
            Map<String, Object> attrs = annotationMetadata
                .getAnnotationAttributes(MyAnnotation.class.getName());
            return "special".equals(attrs.get("value"));
        }
        
        return false;
    }
}
```

### 2.2 Stereotype Annotations

**@Component - Generic Component:**

```java
@Component
public class GenericComponent {
    // Generic Spring-managed component
}
```

**@Service - Business Logic Layer:**

```java
@Service
public class UserService {
    // Service layer component
    // Semantically indicates business logic
}
```

**@Repository - Data Access Layer:**

```java
@Repository
public class UserRepository {
    // Repository layer component
    // Enables exception translation (SQLException -> DataAccessException)
    // Clearly indicates data access
}

/**
 * @Repository special feature:
 * PersistenceExceptionTranslationPostProcessor translates
 * JPA/Hibernate exceptions to Spring's DataAccessException
 */
```

**@Controller - Presentation Layer:**

```java
@Controller
public class UserController {
    // MVC Controller
    // Handles web requests
}

@RestController  // @Controller + @ResponseBody
public class UserRestController {
    // REST API Controller
    // Returns data directly (not view names)
}
```

**Differences:**

```
┌──────────────────────────────────────────────────────────────┐
│         STEREOTYPE ANNOTATION DIFFERENCES                    │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  Annotation    Layer         Special Feature                │
│  ────────────────────────────────────────────────────────   │
│  @Component    Any           None                           │
│  @Service      Business      None (semantic only)           │
│  @Repository   Data Access   Exception translation          │
│  @Controller   Presentation  Request mapping                │
│  @RestController Presentation @Controller + @ResponseBody   │
│                                                              │
│  All are semantically @Component                             │
│  All enable component scanning                               │
│  Use appropriate one for clarity                             │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 2.3 Lazy Initialization

**@Lazy on Bean:**

```java
@Configuration
public class LazyConfig {
    
    @Bean
    @Lazy  // Not created until first use
    public ExpensiveService expensiveService() {
        System.out.println("Creating expensive service...");
        return new ExpensiveService();
    }
    
    @Bean
    public QuickService quickService() {
        System.out.println("Creating quick service...");
        return new QuickService();
    }
}

// Application startup:
// "Creating quick service..."
// (expensiveService NOT created yet)

// First use:
ExpensiveService service = context.getBean(ExpensiveService.class);
// "Creating expensive service..."
```

**@Lazy on Component:**

```java
@Service
@Lazy
public class HeavyService {
    // Not initialized until first injection or lookup
    
    public HeavyService() {
        System.out.println("HeavyService initialized");
    }
}

@Service
public class UserService {
    
    // Lazy injection - proxy injected, actual bean created on first use
    @Autowired
    @Lazy
    private HeavyService heavyService;
    
    public UserService() {
        System.out.println("UserService initialized");
        // heavyService is proxy, not actual HeavyService yet
    }
    
    public void doSomething() {
        // First call to heavyService triggers initialization
        heavyService.doWork();  // "HeavyService initialized" printed here
    }
}
```

**Global Lazy Initialization:**

```java
// application.properties
spring.main.lazy-initialization=true

// Or programmatically
@SpringBootApplication
public class Application {
    public static void main(String[] args) {
        SpringApplication app = new SpringApplication(Application.class);
        app.setLazyInitialization(true);
        app.run(args);
    }
}

/**
 * Effects:
 * ✓ Faster startup
 * ✓ Lower initial memory
 * ✗ Slower first request
 * ✗ Configuration errors discovered later
 * ✗ Not recommended for production
 */
```

### 2.4 Component Naming Strategies

**Default Naming:**

```java
@Component
public class UserService {
    // Bean name: "userService" (class name with first letter lowercase)
}

@Component
public class XMLParser {
    // Bean name: "XMLParser" (first two letters uppercase, keeps as-is)
}
```

**Explicit Naming:**

```java
@Component("customName")
public class UserService {
    // Bean name: "customName"
}

@Service("userSvc")
public class UserService {
    // Bean name: "userSvc"
}
```

**Custom Naming Strategy:**

```java
public class MyBeanNameGenerator implements BeanNameGenerator {
    
    @Override
    public String generateBeanName(BeanDefinition definition, 
                                   BeanDefinitionRegistry registry) {
        String className = definition.getBeanClassName();
        // Custom naming: use fully qualified name
        return className;
        
        // Or use package + simple name
        // return className.substring(className.lastIndexOf('.') + 1);
        
        // Or add prefix/suffix
        // return "bean_" + className.substring(className.lastIndexOf('.') + 1);
    }
}

@Configuration
@ComponentScan(

    nameGenerator = MyBeanNameGenerator.class

}
```
