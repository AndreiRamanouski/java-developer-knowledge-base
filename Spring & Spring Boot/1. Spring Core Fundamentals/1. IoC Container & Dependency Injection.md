## Overview

The Inversion of Control (IoC) container is the heart of Spring Framework. Understanding how it works internally, how beans are created and managed, and how dependency injection operates is fundamental to mastering Spring.

```
┌─────────────────────────────────────────────────────────────────┐
│              SPRING IOC CONTAINER ARCHITECTURE                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. Configuration Phase                                         │
│     ├── Read Configuration (XML/Annotations/Java)              │
│     ├── Create BeanDefinitions                                 │
│     ├── BeanFactoryPostProcessor execution                     │
│     └── Register BeanDefinitions in BeanFactory                │
│                                                                 │
│  2. Bean Creation Phase                                         │
│     ├── Instantiate beans (constructor call)                   │
│     ├── Populate properties (dependency injection)             │
│     ├── BeanPostProcessor.postProcessBeforeInitialization()    │
│     ├── Call initialization callbacks                          │
│     │   ├── @PostConstruct                                     │
│     │   ├── InitializingBean.afterPropertiesSet()             │
│     │   └── custom init-method                                 │
│     ├── BeanPostProcessor.postProcessAfterInitialization()     │
│     └── Bean is ready for use                                  │
│                                                                 │
│  3. Bean Usage Phase                                            │
│     └── Application uses beans                                 │
│                                                                 │
│  4. Destruction Phase                                           │
│     ├── @PreDestroy                                            │
│     ├── DisposableBean.destroy()                               │
│     └── custom destroy-method                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 1. IoC Container Internals

### 1.1 BeanFactory vs ApplicationContext

**BeanFactory - The Core Interface:**

```java
// BeanFactory - Minimal container
public interface BeanFactory {
    Object getBean(String name) throws BeansException;
    <T> T getBean(String name, Class<T> requiredType) throws BeansException;
    <T> T getBean(Class<T> requiredType) throws BeansException;
    boolean containsBean(String name);
    boolean isSingleton(String name) throws NoSuchBeanDefinitionException;
    boolean isPrototype(String name) throws NoSuchBeanDefinitionException;
    // ... more methods
}
```

**When to Use BeanFactory:**

```
Use BeanFactory when:
  ✓ Resource-constrained environment (embedded systems)
  ✓ Need lazy initialization for all beans
  ✓ Minimal overhead required
  ✓ Don't need enterprise features

Limitations:
  ✗ No automatic BeanPostProcessor registration
  ✗ No event propagation
  ✗ No internationalization (i18n)
  ✗ No application-layer specific contexts
```

**ApplicationContext - Full-Featured Container:**

```java
// ApplicationContext extends BeanFactory with more features
public interface ApplicationContext extends 
    EnvironmentCapable,      // Environment and profiles
    ListableBeanFactory,     // List all beans
    HierarchicalBeanFactory, // Parent-child relationships
    MessageSource,           // Internationalization
    ApplicationEventPublisher, // Event publishing
    ResourcePatternResolver  // Resource loading
{
    // Additional methods...
}
```

**Feature Comparison:**

```
┌────────────────────────────────────────────────────────────┐
│      BEANFACTORY vs APPLICATIONCONTEXT                     │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Feature              BeanFactory    ApplicationContext   │
│  ────────────────────────────────────────────────────     │
│  Bean instantiation   Lazy           Eager (default)      │
│  BeanPostProcessor    Manual         Automatic            │
│  Event publication    No             Yes                  │
│  Internationalization No             Yes                  │
│  AOP support          No             Yes                  │
│  Resource loading     Basic          Advanced             │
│  Environment          No             Yes                  │
│  Profiles             No             Yes                  │
│  Memory footprint     Small          Larger               │
│  Use case             Embedded       Standard apps        │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 1.2 Container Lifecycle Phases

**Complete Lifecycle:**

```java
@Component
class LifecycleBean implements 
    BeanNameAware,           // 1. Aware interfaces
    BeanFactoryAware,
    ApplicationContextAware,
    InitializingBean,        // 2. Initialization callback
    DisposableBean           // 3. Destruction callback
{
    
    public LifecycleBean() {
        System.out.println("1. Constructor called");
    }
    
    @Autowired
    private SomeService someService;
    
    @Override
    public void setBeanName(String name) {
        System.out.println("2. setBeanName: " + name);
    }
    
    @Override
    public void setBeanFactory(BeanFactory beanFactory) {
        System.out.println("3. setBeanFactory");
    }
    
    @Override
    public void setApplicationContext(ApplicationContext applicationContext) {
        System.out.println("4. setApplicationContext");
    }
    
    @PostConstruct
    public void postConstruct() {
        System.out.println("5. @PostConstruct");
    }
    
    @Override
    public void afterPropertiesSet() {
        System.out.println("6. afterPropertiesSet()");
    }
    
    @PreDestroy
    public void preDestroy() {
        System.out.println("8. @PreDestroy (on shutdown)");
    }
    
    @Override
    public void destroy() {
        System.out.println("9. destroy() (on shutdown)");
    }
}
```

## 2. Dependency Injection Deep Dive

### 2.1 Constructor vs Setter vs Field Injection

**Constructor Injection (Recommended):**

```java
@Service
public class UserService {
    
    private final UserRepository userRepository;
    private final EmailService emailService;
    
    @Autowired  // Optional in Spring 4.3+ if only one constructor
    public UserService(UserRepository userRepository, 
                      EmailService emailService) {
        this.userRepository = userRepository;
        this.emailService = emailService;
    }
}
```

**Why Constructor Injection is Preferred:**

```
Advantages:
  ✓ Immutability - dependencies are final
  ✓ Required dependencies - compiler enforces them
  ✓ Testability - easy to instantiate in tests
  ✓ No reflection needed for testing
  ✓ Clear dependencies - visible in constructor
  ✓ Thread-safe - final fields are thread-safe
  ✓ Fail fast - app won't start if dependencies missing
  ✓ No NullPointerException risks
```

### 2.2 Circular Dependencies

**Understanding Circular Dependencies:**

```java
@Service
public class ServiceA {
    private final ServiceB serviceB;
    
    @Autowired
    public ServiceA(ServiceB serviceB) {
        this.serviceB = serviceB;
    }
}

@Service
public class ServiceB {
    private final ServiceA serviceA;
    
    @Autowired
    public ServiceB(ServiceA serviceA) {
        this.serviceA = serviceA;
    }
}

// Result: BeanCurrentlyInCreationException
```

**Best Practices - Avoid Circular Dependencies:**

```java
/**
 * Solution 1: Refactor to remove circular dependency
 */
@Service
public class SharedService {
    public void sharedLogic() {
        // Common functionality
    }
}

@Service
public class ServiceA {
    private final SharedService sharedService;
    
    public ServiceA(SharedService sharedService) {
        this.sharedService = sharedService;
    }
}

/**
 * Solution 2: Use @Lazy
 */
@Service
public class ServiceA {
    private final ServiceB serviceB;
    
    @Autowired
    public ServiceA(@Lazy ServiceB serviceB) {
        this.serviceB = serviceB;
    }
}
```
