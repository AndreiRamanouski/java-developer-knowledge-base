
## What is Docker Compose?

Docker Compose is a tool for defining and running multi-container Docker applications. You define your application's services in a YAML file, then start everything with a single command.

**Benefits:**

- Define entire application stack in one file
- Start/stop all services with one command
- Automatic network setup between containers
- Volume management
- Easy environment configuration
- Perfect for local development

**Basic workflow:**

```bash
# Start all services
docker compose up

# Stop all services
docker compose down

# View logs
docker compose logs -f
```

## 1. Local Development

### Multi-Container Applications

A typical application stack includes multiple services that need to work together.

#### Basic docker-compose.yml Structure

```yaml
version: '3.8'

services:
  # Service 1
  web:
    image: nginx:latest
    ports:
      - "80:80"
  
  # Service 2
  api:
    build: .
    ports:
      - "8080:8080"
  
  # Service 3
  database:
    image: postgres:15
    environment:
      POSTGRES_PASSWORD: secret

# Services can communicate using service names as hostnames
# web can reach api at http://api:8080
# api can reach database at postgresql://database:5432
```

#### Complete Multi-Container Example

```yaml
version: '3.8'

services:
  # Frontend
  frontend:
    image: node:18-alpine
    working_dir: /app
    volumes:
      - ./frontend:/app
      - /app/node_modules
    command: npm run dev
    ports:
      - "3000:3000"
    depends_on:
      - backend
    environment:
      - REACT_APP_API_URL=http://localhost:8080

  # Backend API
  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    ports:
      - "8080:8080"
    depends_on:
      - database
      - redis
    environment:
      - SPRING_DATASOURCE_URL=jdbc:postgresql://database:5432/mydb
      - SPRING_REDIS_HOST=redis
    volumes:
      - ./backend/target:/app/target

  # Database
  database:
    image: postgres:15-alpine
    environment:
      POSTGRES_DB: mydb
      POSTGRES_USER: user
      POSTGRES_PASSWORD: password
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data

  # Cache
  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    volumes:
      - redis_data:/data

volumes:
  postgres_data:
  redis_data:
```

**Usage:**

```bash
# Start all services
docker compose up -d

# Check status
docker compose ps

# View logs
docker compose logs -f backend

# Stop all services
docker compose down

# Stop and remove volumes
docker compose down -v
```

### Service Dependencies (depends_on)

Control startup order and dependencies between services.

#### Basic depends_on

```yaml
version: '3.8'

services:
  app:
    build: .
    depends_on:
      - database
      - redis
    # App starts AFTER database and redis containers are created
    # (but doesn't wait for them to be ready!)

  database:
    image: postgres:15

  redis:
    image: redis:7
```

**Important:** `depends_on` only waits for containers to **start**, not for them to be **ready**!

#### depends_on with Health Checks (Better)

```yaml
version: '3.8'

services:
  app:
    build: .
    depends_on:
      database:
        condition: service_healthy
      redis:
        condition: service_healthy
    # Now app waits until services are HEALTHY
    environment:
      - DB_HOST=database
      - REDIS_HOST=redis

  database:
    image: postgres:15
    environment:
      POSTGRES_PASSWORD: secret
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 10s

  redis:
    image: redis:7
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 3s
      retries: 5
```

#### Manual Wait Script (Alternative)

```yaml
version: '3.8'

services:
  app:
    build: .
    depends_on:
      - database
    command: >
      sh -c "
        while ! nc -z database 5432; do
          echo 'Waiting for database...';
          sleep 1;
        done;
        echo 'Database ready!';
        java -jar app.jar
      "
```

#### Multiple Dependency Conditions

```yaml
version: '3.8'

services:
  web:
    image: nginx
    depends_on:
      api:
        condition: service_healthy
      frontend:
        condition: service_started

  api:
    build: ./api
    depends_on:
      database:
        condition: service_healthy
      cache:
        condition: service_started
      queue:
        condition: service_started
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
      interval: 30s
      timeout: 3s
      retries: 3
      start_period: 40s

  frontend:
    build: ./frontend
    depends_on:
      - api

  database:
    image: postgres:15
    healthcheck:
      test: ["CMD-SHELL", "pg_isready"]
      interval: 10s

  cache:
    image: redis:7

  queue:
    image: rabbitmq:3-management
```

### Networks and Volumes

#### Networks

Docker Compose automatically creates a network for your services, but you can customize it.

**Default Network (Automatic):**

```yaml
version: '3.8'

services:
  app:
    image: myapp
  database:
    image: postgres

# Compose automatically creates a network named 'myproject_default'
# All services can reach each other by service name
# app can connect to: postgresql://database:5432
```

**Custom Networks:**

```yaml
version: '3.8'

services:
  frontend:
    image: react-app
    networks:
      - frontend-network

  backend:
    image: spring-boot-app
    networks:
      - frontend-network
      - backend-network

  database:
    image: postgres
    networks:
      - backend-network
    # Database is NOT accessible from frontend (isolated!)

networks:
  frontend-network:
    driver: bridge
  backend-network:
    driver: bridge
```

**Network Isolation Example:**

```yaml
version: '3.8'

services:
  # Public-facing services
  nginx:
    image: nginx
    networks:
      - public
      - internal
    ports:
      - "80:80"

  api:
    build: ./api
    networks:
      - internal
    # No direct external access

  database:
    image: postgres
    networks:
      - internal
    # Only accessible from internal network

networks:
  public:
    driver: bridge
  internal:
    driver: bridge
    internal: true  # No external access
```

**External Networks:**

```yaml
version: '3.8'

services:
  app:
    image: myapp
    networks:
      - shared-network

networks:
  shared-network:
    external: true
    name: company_shared_network
```

#### Volumes

Volumes persist data beyond container lifecycle.

**Named Volumes (Recommended):**

```yaml
version: '3.8'

services:
  database:
    image: postgres:15
    volumes:
      - postgres_data:/var/lib/postgresql/data
      # Data persists even after 'docker compose down'

  redis:
    image: redis:7
    volumes:
      - redis_data:/data

volumes:
  postgres_data:
    driver: local
  redis_data:
    driver: local
```

**Bind Mounts (Development):**

```yaml
version: '3.8'

services:
  app:
    build: .
    volumes:
      # Mount local directory to container
      - ./src:/app/src
      - ./target:/app/target
      # Changes in local files reflect immediately in container
```

**Volume Options:**

```yaml
version: '3.8'

services:
  app:
    image: myapp
    volumes:
      # Read-only mount
      - ./config:/app/config:ro
      
      # Read-write (default)
      - ./logs:/app/logs:rw
      
      # Named volume with options
      - data:/app/data:rw
      
      # Anonymous volume (excluded from bind mount)
      - /app/node_modules

volumes:
  data:
    driver: local
    driver_opts:
      type: none
      o: bind
      device: /path/on/host
```

**Volume Sharing Between Services:**

```yaml
version: '3.8'

services:
  app:
    build: .
    volumes:
      - shared_data:/data

  worker:
    build: ./worker
    volumes:
      - shared_data:/data
    # Both services can read/write to same volume

volumes:
  shared_data:
```

**tmpfs Mounts (In-Memory):**

```yaml
version: '3.8'

services:
  app:
    image: myapp
    tmpfs:
      - /tmp
      - /app/cache
    # Data stored in memory, lost on container stop
```

### Environment Variables

Multiple ways to configure environment variables in Compose.

#### Method 1: Inline in docker-compose.yml

```yaml
version: '3.8'

services:
  app:
    image: myapp
    environment:
      - DATABASE_HOST=database
      - DATABASE_PORT=5432
      - DATABASE_NAME=mydb
      - LOG_LEVEL=INFO
```

#### Method 2: env_file

```yaml
# docker-compose.yml
version: '3.8'

services:
  app:
    image: myapp
    env_file:
      - .env
      - .env.local
    # Loads variables from files
```

```bash
# .env file
DATABASE_HOST=database
DATABASE_PORT=5432
DATABASE_NAME=mydb
LOG_LEVEL=INFO
SECRET_KEY=my-secret
```

#### Method 3: Shell Environment Variables

```yaml
version: '3.8'

services:
  app:
    image: myapp
    environment:
      - DATABASE_PASSWORD=${DB_PASSWORD}
      - API_KEY=${API_KEY}
    # Uses variables from shell environment
```

```bash
# Set in shell before running compose
export DB_PASSWORD=secret123
export API_KEY=abc123xyz

docker compose up
```

#### Method 4: Default Values

```yaml
version: '3.8'

services:
  app:
    image: myapp
    environment:
      - DATABASE_HOST=${DB_HOST:-database}
      - DATABASE_PORT=${DB_PORT:-5432}
      - LOG_LEVEL=${LOG_LEVEL:-INFO}
    # Uses shell value if set, otherwise uses default after :-
```

#### Complete Environment Configuration Example

```yaml
version: '3.8'

services:
  app:
    build: .
    environment:
      # Database configuration
      - SPRING_DATASOURCE_URL=jdbc:postgresql://database:5432/${DB_NAME:-mydb}
      - SPRING_DATASOURCE_USERNAME=${DB_USER:-postgres}
      - SPRING_DATASOURCE_PASSWORD=${DB_PASSWORD:?DB_PASSWORD not set}
      
      # Redis configuration
      - SPRING_REDIS_HOST=redis
      - SPRING_REDIS_PORT=6379
      
      # Application configuration
      - SPRING_PROFILES_ACTIVE=${PROFILE:-dev}
      - SERVER_PORT=8080
      - LOG_LEVEL=${LOG_LEVEL:-INFO}
    env_file:
      - .env
      - .env.${PROFILE:-dev}
    depends_on:
      - database
      - redis

  database:
    image: postgres:15
    environment:
      POSTGRES_DB: ${DB_NAME:-mydb}
      POSTGRES_USER: ${DB_USER:-postgres}
      POSTGRES_PASSWORD: ${DB_PASSWORD:?DB_PASSWORD not set}

  redis:
    image: redis:7
```

**Environment File Precedence:**

```
1. Compose file environment
2. Shell environment variables
3. env_file
4. Dockerfile ENV
```

**Security Best Practices:**

```yaml
# .gitignore
.env
.env.local
.env.*.local

# .env.example (commit this)
DB_PASSWORD=changeme
API_KEY=your-key-here

# Actual .env (DO NOT commit)
DB_PASSWORD=actual-secret-password
API_KEY=actual-api-key
```

## 2. Docker Compose for Spring Boot

### App + Database + Redis/Kafka Setup

#### Spring Boot + PostgreSQL + Redis

```yaml
version: '3.8'

services:
  # Spring Boot Application
  app:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: spring-app
    ports:
      - "8080:8080"
    depends_on:
      database:
        condition: service_healthy
      redis:
        condition: service_started
    environment:
      # Database
      - SPRING_DATASOURCE_URL=jdbc:postgresql://database:5432/mydb
      - SPRING_DATASOURCE_USERNAME=postgres
      - SPRING_DATASOURCE_PASSWORD=secret
      
      # Redis
      - SPRING_REDIS_HOST=redis
      - SPRING_REDIS_PORT=6379
      
      # JPA
      - SPRING_JPA_HIBERNATE_DDL_AUTO=update
      - SPRING_JPA_SHOW_SQL=true
      
      # Profile
      - SPRING_PROFILES_ACTIVE=dev
    volumes:
      - ./logs:/app/logs
    networks:
      - app-network
    restart: unless-stopped

  # PostgreSQL Database
  database:
    image: postgres:15-alpine
    container_name: postgres-db
    environment:
      POSTGRES_DB: mydb
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: secret
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
      - ./init-scripts:/docker-entrypoint-initdb.d
    networks:
      - app-network
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 10s
      timeout: 5s
      retries: 5
    restart: unless-stopped

  # Redis Cache
  redis:
    image: redis:7-alpine
    container_name: redis-cache
    ports:
      - "6379:6379"
    volumes:
      - redis_data:/data
    networks:
      - app-network
    command: redis-server --appendonly yes
    restart: unless-stopped

volumes:
  postgres_data:
  redis_data:

networks:
  app-network:
    driver: bridge
```

#### Spring Boot + PostgreSQL + Kafka

```yaml
version: '3.8'

services:

      - SPRING_KAFKA_BOOTSTRAP_SERVERS=kafka:9092
      - SPRING_KAFKA_CONSUMER_GROUP_ID=myapp-group
      - SPRING_KAFKA_CONSUMER_AUTO_OFFSET_RESET=earliest
      
      # Profile
      - SPRING_PROFILES_ACTIVE=dev
    networks:
      - app-network

  # PostgreSQL
  database:
    image: postgres:15-alpine
    environment:
      POSTGRES_DB: mydb
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: secret
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
    networks:
      - app-network
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 10s

  # Zookeeper (required for Kafka)
  zookeeper:
    image: confluentinc/cp-zookeeper:7.5.0
    container_name: zookeeper
    environment:
      ZOOKEEPER_CLIENT_PORT: 2181
      ZOOKEEPER_TICK_TIME: 2000
    networks:
      - app-network

  # Kafka
  kafka:
    image: confluentinc/cp-kafka:7.5.0
    container_name: kafka
    depends_on:
      - zookeeper
    ports:
      - "9092:9092"
    environment:
      KAFKA_BROKER_ID: 1
      KAFKA_ZOOKEEPER_CONNECT: zookeeper:2181
      KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://kafka:9092
      KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 1
      KAFKA_TRANSACTION_STATE_LOG_MIN_ISR: 1
      KAFKA_TRANSACTION_STATE_LOG_REPLICATION_FACTOR: 1
    networks:
      - app-network

  # Kafka UI (optional but useful)
  kafka-ui:
    image: provectuslabs/kafka-ui:latest
    container_name: kafka-ui
    depends_on:
      - kafka
    ports:
      - "8090:8080"
    environment:
      KAFKA_CLUSTERS_0_NAME: local
      KAFKA_CLUSTERS_0_BOOTSTRAPSERVERS: kafka:9092
      KAFKA_CLUSTERS_0_ZOOKEEPER: zookeeper:2181
    networks:
      - app-network

volumes:
  postgres_data:

networks:
  app-network:
    driver: bridge
```

### Development vs Production Configs

Separate configurations for different environments.

#### Using Multiple Compose Files

**Base configuration** (`docker-compose.yml`):

```yaml
version: '3.8'

services:
  app:
    build: .
    environment:
      - SPRING_DATASOURCE_URL=jdbc:postgresql://database:5432/mydb
      - SPRING_REDIS_HOST=redis
    depends_on:
      - database
      - redis

  database:
    image: postgres:15-alpine
    environment:
      POSTGRES_DB: mydb

  redis:
    image: redis:7-alpine
```

**Development overrides** (`docker-compose.dev.yml`):

```yaml
version: '3.8'

services:
  app:
    build:
      target: development
    ports:
      - "8080:8080"
      - "5005:5005"  # Debug port
    volumes:
      # Hot reload
      - ./src:/app/src
      - ./target:/app/target
    environment:
      - SPRING_PROFILES_ACTIVE=dev
      - SPRING_DEVTOOLS_RESTART_ENABLED=true
      - JAVA_TOOL_OPTIONS=-agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=*:5005
    command: mvn spring-boot:run

  database:
    ports:
      - "5432:5432"
    volumes:
      - ./init-scripts:/docker-entrypoint-initdb.d

  redis:
    ports:
      - "6379:6379"

  # Additional dev tools
  adminer:
    image: adminer
    ports:
      - "8081:8080"
    depends_on:
      - database
```

**Production overrides** (`docker-compose.prod.yml`):

```yaml
version: '3.8'

services:
  app:
    build:
      target: production
    restart: always
    environment:
      - SPRING_PROFILES_ACTIVE=prod
      - JAVA_OPTS=-XX:+UseContainerSupport -XX:MaxRAMPercentage=75.0
    deploy:
      resources:
        limits:
          cpus: '2'
          memory: 2G
        reservations:
          cpus: '1'
          memory: 1G
    # No volume mounts
    # No exposed ports (behind reverse proxy)

  database:
    restart: always
    volumes:
      - postgres_data:/var/lib/postgresql/data
    # No port exposure (internal only)
    deploy:
      resources:
        limits:
          cpus: '1'
          memory: 1G

  redis:
    restart: always
    volumes:
      - redis_data:/data
    command: redis-server --requirepass ${REDIS_PASSWORD}
    deploy:
      resources:
        limits:
          cpus: '0.5'
          memory: 512M

volumes:
  postgres_data:
  redis_data:
```

**Usage:**

```bash
# Development
docker compose -f docker-compose.yml -f docker-compose.dev.yml up

# Production
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d

# Or use aliases
alias dcdev='docker compose -f docker-compose.yml -f docker-compose.dev.yml'
alias dcprod='docker compose -f docker-compose.yml -f docker-compose.prod.yml'

dcdev up
dcprod up -d
```

#### Using Profiles (Compose 1.28+)

**Single file with profiles:**

```yaml
version: '3.8'

services:
  app:
    build: .
    environment:
      - SPRING_PROFILES_ACTIVE=${PROFILE:-dev}
    depends_on:
      - database

  database:
    image: postgres:15-alpine
    environment:
      POSTGRES_DB: mydb

  # Development-only services
  adminer:
    image: adminer
    profiles: ["dev"]
    ports:
      - "8081:8080"
    depends_on:
      - database

  pgadmin:
    image: dpage/pgadmin4
    profiles: ["dev"]
    ports:
      - "5050:80"
    environment:
      PGADMIN_DEFAULT_EMAIL: admin@admin.com
      PGADMIN_DEFAULT_PASSWORD: admin

  # Testing-only service
  test-db:
    image: postgres:15-alpine
    profiles: ["test"]
    environment:
      POSTGRES_DB: testdb
    tmpfs:
      - /var/lib/postgresql/data

  # Production monitoring
  prometheus:
    image: prom/prometheus
    profiles: ["prod"]
    ports:
      - "9090:9090"

  grafana:
    image: grafana/grafana
    profiles: ["prod"]
    ports:
      - "3000:3000"
```

**Usage:**

```bash
# Development (includes adminer, pgadmin)
docker compose --profile dev up

# Test
docker compose --profile test up

# Production (includes monitoring)
docker compose --profile prod up -d

# Multiple profiles
docker compose --profile dev --profile test up
```

### Hot Reload with Volumes

Enable live code reloading during development without rebuilding containers.

#### Spring Boot DevTools with Volumes

**Dockerfile with multi-stage for development:**

```dockerfile
# Development stage
FROM maven:3.9-eclipse-temurin-17 AS development

WORKDIR /app

# Install Spring Boot DevTools
COPY pom.xml .
RUN mvn dependency:go-offline

# Don't copy src - will be mounted as volume
VOLUME /app/src
VOLUME /app/target

# Run with Maven (auto-restarts on change)
CMD ["mvn", "spring-boot:run"]

# Production stage
FROM eclipse-temurin:17-jre-jammy AS production

WORKDIR /app
COPY target/*.jar app.jar

CMD ["java", "-jar", "app.jar"]
```

**docker-compose.yml:**

```yaml
version: '3.8'

services:
  app:
    build:
      context: .
      target: development
    ports:
      - "8080:8080"
    volumes:
      # Mount source code
      - ./src:/app/src
      # Mount compiled classes
      - ./target:/app/target
      # Maven repository cache
      - maven_cache:/root/.m2
    environment:
      - SPRING_DEVTOOLS_RESTART_ENABLED=true
      - SPRING_DEVTOOLS_RESTART_POLL_INTERVAL=1s
      - SPRING_DEVTOOLS_RESTART_QUIET_PERIOD=400ms
    depends_on:
      - database

  database:
    image: postgres:15-alpine
    environment:
      POSTGRES_DB: mydb
      POSTGRES_PASSWORD: secret
    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  maven_cache:
  postgres_data:
```

**pom.xml - Enable Spring Boot DevTools:**

```xml
<dependencies>
    <!-- Spring Boot DevTools -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-devtools</artifactId>
        <scope>runtime</scope>
        <optional>true</optional>
    </dependency>
</dependencies>
```

**How it works:**

1. Edit a Java file in `./src`
2. Save the file
3. Maven detects change and recompiles
4. Spring DevTools detects new class files in `./target`
5. Application automatically restarts (2-3 seconds)

## 3. Advanced Features

### Health Checks

Health checks monitor container health and enable smart dependency management.

#### Basic Health Check

```yaml
version: '3.8'

services:
  app:
    build: .
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/actuator/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s
```

**Health check options:**

- `test`: Command to run (exit 0 = healthy, exit 1 = unhealthy)
- `interval`: How often to check
- `timeout`: Max time for check to complete
- `retries`: Consecutive failures before marking unhealthy
- `start_period`: Grace period before health checks start

#### Health Check Methods

**1. HTTP Endpoint:**

```yaml
services:
  app:
    build: .
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
      interval: 30s
      timeout: 3s
      retries: 3
```

**2. TCP Port Check:**

```yaml
services:
  database:
    image: postgres:15
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 10s
      timeout: 5s
      retries: 5
```

**3. Custom Script:**

```yaml
services:
  app:
    build: .
    healthcheck:
      test: ["CMD", "/app/healthcheck.sh"]
      interval: 30s
```

```bash
# healthcheck.sh
#!/bin/sh
if curl -f http://localhost:8080/health > /dev/null 2>&1; then
    exit 0
else
    exit 1
fi
```

**4. Multiple Checks:**

```yaml
services:
  app:
    build: .
    healthcheck:
      test: >
        CMD-SHELL
        curl -f http://localhost:8080/health &&
        curl -f http://localhost:8080/actuator/health &&
        test -f /tmp/ready
      interval: 30s
```

#### Complete Health Check Example

```yaml
version: '3.8'

services:
  # Web Application
  app:
    build: .
    depends_on:
      database:
        condition: service_healthy
      redis:
        condition: service_healthy
      kafka:
        condition: service_started
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/actuator/health"]
      interval: 30s
      timeout: 3s
      retries: 3
      start_period: 60s

  # PostgreSQL
  database:
    image: postgres:15-alpine
    environment:
      POSTGRES_PASSWORD: secret
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 10s

  # Redis
  redis:
    image: redis:7-alpine
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 3s
      retries: 5
      start_period: 5s

  # Kafka (no built-in health check)
  kafka:
    image: confluentinc/cp-kafka:7.5.0
    depends_on:
      zookeeper:
        condition: service_healthy
    environment:
      KAFKA_ZOOKEEPER_CONNECT: zookeeper:2181
    healthcheck:
      test: ["CMD", "kafka-broker-api-versions", "--bootstrap-server", "localhost:9092"]
      interval: 30s
      timeout: 10s
      retries: 5
      start_period: 60s

  # Zookeeper
  zookeeper:
    image: confluentinc/cp-zookeeper:7.5.0
    healthcheck:
      test: ["CMD", "echo", "ruok", "|", "nc", "localhost", "2181"]
      interval: 10s
      timeout: 5s
      retries: 3
```

### Resource Limits

Control CPU and memory usage for services.

#### Memory Limits

```yaml
version: '3.8'

services:
  app:
    image: myapp
    deploy:
      resources:
        limits:
          memory: 1G        # Maximum memory
        reservations:
          memory: 512M      # Minimum guaranteed memory
```

**Alternative syntax (Compose v2):**

```yaml
services:
  app:
    image: myapp
    mem_limit: 1g
    mem_reservation: 512m
```

#### CPU Limits

```yaml
version: '3.8'

services:
  app:
    image: myapp
    deploy:
      resources:
        limits:
          cpus: '2.0'       # Maximum 2 CPU cores
        reservations:
          cpus: '1.0'       # Minimum 1 CPU core
```

**Alternative syntax:**

```yaml
services:
  app:
    image: myapp
    cpus: 2.0
    cpu_shares: 1024      # Relative weight (default 1024)
```

#### Complete Resource Configuration

```yaml
version: '3.8'

services:
  # Resource-intensive application
  app:
    build: .
    deploy:
      resources:
        limits:
          cpus: '2.0'
          memory: 2G
        reservations:
          cpus: '1.0'
          memory: 1G
      restart_policy:
        condition: on-failure
        max_attempts: 3

  # Lightweight cache
  redis:
    image: redis:7-alpine
    deploy:
      resources:
        limits:
          cpus: '0.5'
          memory: 512M
        reservations:
          cpus: '0.25'
          memory: 256M

  # Database with guaranteed resources
  database:
    image: postgres:15-alpine
    deploy:
      resources:
        limits:
          cpus: '2.0'
          memory: 2G
        reservations:
          cpus: '1.0'
          memory: 1G
    volumes:
      - postgres_data:/var/lib/postgresql/data

  postgres_data:
```

### Profiles (dev, test, prod)

Profiles allow you to selectively start services based on environment.

#### Basic Profiles

```yaml
version: '3.8'

services:
  # Always started
  app:
    build: .
    profiles: []          # Empty = always started

  database:
    image: postgres:15
    profiles: []

  # Development only
  adminer:
    image: adminer
    profiles: ["dev", "debug"]
    ports:
      - "8081:8080"

  pgadmin:
    image: dpage/pgadmin4
    profiles: ["dev"]
    ports:
      - "5050:80"

  # Testing only
  test-db:
    image: postgres:15-alpine
    profiles: ["test"]
    tmpfs:
      - /var/lib/postgresql/data

  wiremock:
    image: wiremock/wiremock
    profiles: ["test"]
    ports:
      - "8090:8080"

  # Production only
  nginx:
    image: nginx
    profiles: ["prod"]
    ports:
      - "80:80"

  prometheus:
    image: prom/prometheus
    profiles: ["prod", "monitoring"]

  grafana:
    image: grafana/grafana
    profiles: ["prod", "monitoring"]
```

**Usage:**

```bash
# Default (no profile) - only app and database
docker compose up

# Development
docker compose --profile dev up

# Testing
docker compose --profile test up

# Production
docker compose --profile prod up

# Multiple profiles
docker compose --profile dev --profile monitoring up

# All profiles
docker compose --profile dev --profile test --profile prod up
```

## Summary

### Key Takeaways:

**Local Development:**

- Use docker-compose.yml to define multi-container apps
- `depends_on` with health checks ensures proper startup order
- Named volumes persist data, bind mounts enable hot reload
- Multiple methods for environment configuration

**Spring Boot Integration:**

- Complete stacks with PostgreSQL, Redis, Kafka
- Service discovery using container names
- Health checks for all components
- Proper initialization with init scripts

**Environment Management:**

- Multiple compose files for different environments
- Profiles for selective service startup
- Override files for environment-specific config
- Separate configs for dev/test/prod

**Advanced Features:**

- Health checks monitor service readiness
- Resource limits prevent overconsumption
- Profiles enable flexible deployments

- Use health checks for critical services
- Set resource limits in production
- Keep secrets in .env files (not in Git)
- Use profiles to organize services
- Enable hot reload for development
- Document your compose files well
