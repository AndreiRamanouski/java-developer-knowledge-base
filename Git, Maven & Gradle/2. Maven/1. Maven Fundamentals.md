## Overview

Apache Maven is a powerful build automation and dependency management tool that has become the de facto standard for Java projects. It follows the principle of "convention over configuration" to simplify project setup and management.

```
┌─────────────────────────────────────────────────────────────────┐
│                    MAVEN ARCHITECTURE                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Developer → pom.xml → Maven → Build Lifecycle → Artifact      │
│                           ↓                                     │
│                     Repositories                                │
│                  (Local/Central/Remote)                         │
│                           ↓                                     │
│                    Dependencies                                 │
│                     Plugins                                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

---

## 1. What is Maven?

### Core Capabilities

Maven provides four primary functions for Java projects:

1. **Build Automation**
    
    - Compiles source code
    - Runs tests
    - Packages applications (JAR, WAR, EAR)
    - Generates documentation
    - Deploys artifacts
2. **Dependency Management**
    
    - Automatically downloads dependencies
    - Manages transitive dependencies
    - Resolves version conflicts
    - Handles dependency scopes
3. **Project Structure Standardization**
    
    - Enforces consistent directory layout
    - Standard location for source, tests, resources
    - Predictable build output location
4. **Convention Over Configuration**
    
    - Sensible defaults for everything
    - Minimal configuration needed
    - Override only when necessary

### Why Use Maven?

**For Individual Developers:**

- No need to manually download JARs
- Consistent project structure across all projects
- Easy to share projects (just share pom.xml)
- Rich plugin ecosystem for common tasks

**For Teams:**

- Everyone uses same project structure
- Dependencies are versioned and tracked
- Reproducible builds across environments
- Easy CI/CD integration

**For Organizations:**

- Centralized artifact management
- Dependency governance and security scanning
- License compliance tracking
- Standardized build process

## 2. Maven Architecture

### Repository System

Maven uses a three-tier repository system:

```
┌─────────────────────────────────────────────────────────────────┐
│                   MAVEN REPOSITORY HIERARCHY                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. Local Repository (~/.m2/repository)                        │
│     ↓                                                           │
│     Maven checks local first                                   │
│     Cached downloads stay here                                 │
│     Fast access, no network needed                             │
│                                                                 │
│  2. Remote Repositories (Company/Private)                      │
│     ↓                                                           │
│     Custom/proprietary artifacts                               │
│     Nexus, Artifactory, etc.                                   │
│     Controlled by organization                                 │
│                                                                 │
│  3. Central Repository (https://repo.maven.apache.org)         │
│     ↓                                                           │
│     Public open-source libraries                               │
│     Default fallback                                           │
│     Millions of artifacts                                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### Local Repository

```bash
# Default location
~/.m2/repository/

# Example structure
~/.m2/repository/
  └── org/
      └── springframework/
          └── spring-core/
              └── 5.3.20/
                  ├── spring-core-5.3.20.jar
                  ├── spring-core-5.3.20.pom
                  └── spring-core-5.3.20-sources.jar
```

**Configuration:**

```xml
<!-- ~/.m2/settings.xml -->
<settings>
  <localRepository>/custom/path/to/repo</localRepository>
</settings>
```

#### Central Repository

```xml
<!-- Automatically configured by default -->
<!-- No configuration needed in pom.xml -->

<!-- Maven Central contains:
     - Open source libraries
     - Public frameworks
     - Common utilities
     - Community artifacts
-->
```

**Search Central Repository:**

- https://search.maven.org/
- https://mvnrepository.com/

#### Remote/Private Repositories

```xml
<!-- pom.xml -->
<repositories>
  <repository>
    <id>company-nexus</id>
    <name>Company Nexus Repository</name>
    <url>https://nexus.company.com/repository/maven-public/</url>
  </repository>
</repositories>

<!-- For plugin repositories -->
<pluginRepositories>
  <pluginRepository>
    <id>company-nexus-plugins</id>
    <url>https://nexus.company.com/repository/maven-plugins/</url>
  </pluginRepository>
</pluginRepositories>
```

**With Authentication:**

```xml
<!-- ~/.m2/settings.xml -->
<settings>
  <servers>
    <server>
      <id>company-nexus</id>
      <username>developer</username>
      <password>{encrypted-password}</password>
    </server>
  </servers>
</settings>
```

### POM (Project Object Model)

The POM is the fundamental unit of work in Maven. It's an XML file that contains information about the project and configuration details.

#### POM Hierarchy

```
┌─────────────────────────────────────────────────────────────────┐
│                      POM INHERITANCE                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Super POM (built into Maven)                                  │
│       ↓                                                         │
│  Company Parent POM (optional)                                 │
│       ↓                                                         │
│  Your Project POM                                              │
│       ↓                                                         │
│  Module POMs (if multi-module)                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### Minimal POM

```xml
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0
                             http://maven.apache.org/xsd/maven-4.0.0.xsd">
    <modelVersion>4.0.0</modelVersion>
    
    <!-- Project Coordinates (GAV) -->
    <groupId>com.company.app</groupId>
    <artifactId>my-application</artifactId>
    <version>1.0.0-SNAPSHOT</version>
    
    <!-- Basic packaging type -->
    <packaging>jar</packaging>
</project>
```

#### Complete POM Structure

```xml
<project>
    <!-- Project Coordinates -->
    <modelVersion>4.0.0</modelVersion>
    <groupId>com.company.app</groupId>
    <artifactId>my-application</artifactId>
    <version>1.0.0-SNAPSHOT</version>
    <packaging>jar</packaging>
    
    <!-- Project Information -->
    <name>My Application</name>
    <description>A sample Maven project</description>
    <url>https://github.com/company/my-app</url>
    
    <!-- Parent POM -->
    <parent>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-parent</artifactId>
        <version>2.7.5</version>
    </parent>
    
    <!-- Properties -->
    <properties>
        <java.version>17</java.version>
        <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
        <spring.version>5.3.20</spring.version>
    </properties>
    
    <!-- Dependencies -->
    <dependencies>
        <dependency>
            <groupId>org.springframework</groupId>
            <artifactId>spring-core</artifactId>
            <version>${spring.version}</version>
        </dependency>
    </dependencies>
    
    <!-- Build Configuration -->
    <build>
        <plugins>
            <plugin>
                <groupId>org.apache.maven.plugins</groupId>
                <artifactId>maven-compiler-plugin</artifactId>
                <version>3.11.0</version>
            </plugin>
        </plugins>
    </build>
    
    <!-- Profiles -->
    <profiles>
        <profile>
            <id>development</id>
            <activation>
                <activeByDefault>true</activeByDefault>
            </activation>
        </profile>
    </profiles>
</project>
```

#### Understanding GAV Coordinates

```
┌─────────────────────────────────────────────────────────────────┐
│                      GAV COORDINATES                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  G - GroupId:    com.company.app                               │
│     Identifies the organization/group                           │
│     Usually reverse domain name                                │
│                                                                 │
│  A - ArtifactId: my-application                                │
│     Identifies the specific project                            │
│     Must be unique within the group                            │
│                                                                 │
│  V - Version:    1.0.0-SNAPSHOT                                │
│     Identifies the version                                     │
│     SNAPSHOT = under development                               │
│     Without SNAPSHOT = released version                        │
│                                                                 │
│  Full Identifier: com.company.app:my-application:1.0.0-SNAPSHOT│
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**Version Conventions:**

```
1.0.0-SNAPSHOT    # Development version
1.0.0             # Released version
1.0.1             # Patch release
1.1.0             # Minor release
2.0.0             # Major release

# Semantic Versioning (SemVer)
MAJOR.MINOR.PATCH
  │     │     └─ Bug fixes
  │     └─────── New features (backward compatible)
  └───────────── Breaking changes
```

### Maven Lifecycle and Phases

Maven has three built-in build lifecycles, each with multiple phases.

#### The Three Lifecycles

```
┌─────────────────────────────────────────────────────────────────┐
│                    MAVEN LIFECYCLES                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. DEFAULT (Build)                                            │
│     validate → compile → test → package → verify → install     │
│     → deploy                                                    │
│                                                                 │
│  2. CLEAN                                                       │
│     pre-clean → clean → post-clean                             │
│                                                                 │
│  3. SITE (Documentation)                                        │
│     pre-site → site → post-site → site-deploy                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### Default Lifecycle Phases (Most Important)

```bash
# Phase execution is sequential
# Running a phase executes all previous phases

mvn validate    # Validate project is correct
mvn compile     # Compile source code
mvn test        # Run unit tests
mvn package     # Package compiled code (JAR/WAR)
mvn verify      # Run integration tests
mvn install     # Install to local repository
mvn deploy      # Deploy to remote repository
```

**Detailed Phase Breakdown:**

```
validate
  └─ Check if project is correct and all necessary information is available

compile
  └─ Compile source code in src/main/java
  └─ Output to target/classes

test-compile
  └─ Compile test source code in src/test/java
  └─ Output to target/test-classes

test
  └─ Run unit tests using Surefire plugin
  └─ Tests should not require deployment

package
  └─ Package compiled code into distributable format
  └─ JAR for libraries, WAR for web apps

verify
  └─ Run integration tests
  └─ Check package is valid and meets quality criteria

install
  └─ Install package to local repository (~/.m2/repository)
  └─ Available for other projects on same machine

deploy
  └─ Copy final package to remote repository
  └─ Share with other developers/projects
```

#### Common Maven Commands

```bash
# Clean build output
mvn clean

# Compile only
mvn compile

# Compile and run tests
mvn test

# Build JAR/WAR (skip tests)
mvn package -DskipTests

# Full build with tests
mvn clean install

# Deploy to remote repository
mvn clean deploy

# Run specific phase
mvn verify

# Combined commands
mvn clean package
mvn clean install -DskipTests
mvn clean test
```

#### Phase vs Goal

```
Phase: A step in the build lifecycle
Goal:  A specific task that can be bound to a phase

Example:
  mvn package                    # Run 'package' phase
  mvn compiler:compile           # Run 'compile' goal directly
  mvn dependency:tree            # Run 'tree' goal from dependency plugin
```

### Plugins and Goals

Plugins are the workhorses of Maven. Every task in Maven is performed by a plugin goal.

#### Plugin Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│                      PLUGIN STRUCTURE                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Maven Plugin                                                   │
│    ├── Goal 1                                                   │
│    ├── Goal 2                                                   │
│    └── Goal 3                                                   │
│                                                                 │
│  Example: maven-compiler-plugin                                │
│    ├── compiler:compile    (bound to compile phase)           │
│    └── compiler:testCompile (bound to test-compile phase)     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### Core Maven Plugins

```xml
<!-- Compiler Plugin -->
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <version>3.11.0</version>
    <configuration>
        <source>17</source>
        <target>17</target>
    </configuration>
</plugin>

<!-- Surefire Plugin (Unit Tests) -->
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-surefire-plugin</artifactId>
    <version>3.0.0</version>
</plugin>

<!-- JAR Plugin -->
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-jar-plugin</artifactId>
    <version>3.3.0</version>
    <configuration>
        <archive>
            <manifest>
                <mainClass>com.company.app.Main</mainClass>
            </manifest>
        </archive>
    </configuration>
</plugin>

<!-- Dependency Plugin -->
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-dependency-plugin</artifactId>
    <version>3.5.0</version>
</plugin>
```

#### Running Plugin Goals Directly

```bash
# View dependency tree
mvn dependency:tree

# Copy dependencies to target/dependency
mvn dependency:copy-dependencies

# Analyze dependencies
mvn dependency:analyze

# Display effective POM
mvn help:effective-pom

# Display active profiles
mvn help:active-profiles

# List all plugins
mvn help:describe -Dplugin=compiler
```

#### Plugin Configuration

```xml
<build>
    <plugins>
        <plugin>
            <groupId>org.apache.maven.plugins</groupId>
            <artifactId>maven-compiler-plugin</artifactId>
            <version>3.11.0</version>
            <configuration>
                <!-- Configuration for ALL goals -->
                <source>17</source>
                <target>17</target>
                <encoding>UTF-8</encoding>
                <showWarnings>true</showWarnings>
                <showDeprecation>true</showDeprecation>
            </configuration>
            <executions>
                <execution>
                    <!-- Configuration for SPECIFIC execution -->
                    <id>compile-java-17</id>
                    <phase>compile</phase>
                    <goals>
                        <goal>compile</goal>
                    </goals>
                    <configuration>
                        <release>17</release>
                    </configuration>
                </execution>
            </executions>
        </plugin>
    </plugins>
</build>
```

### Directory Purposes

#### src/main/java

```
Purpose: Production Java source code
Compiled to: target/classes
Package structure mirrors directory structure

Example:
  src/main/java/com/company/app/Main.java
  → Compiles to class: com.company.app.Main
  → Output: target/classes/com/company/app/Main.class
```

#### src/main/resources

```
Purpose: Application resources (non-code files)
Output: Copied to target/classes (root of classpath)

Common files:
  - application.properties    (Spring Boot configuration)
  - logback.xml              (Logging configuration)
  - messages.properties      (Internationalization)
  - schema.sql               (Database schema)
  - static/                  (Static web resources)
  - templates/               (View templates)

Access in code:
  ClassLoader.getResourceAsStream("application.properties")
  @Value("classpath:config/app.yml")
```

#### src/main/webapp

```
Purpose: Web application resources (WAR projects only)
Output: Root of WAR file

Structure:
  webapp/
    ├── WEB-INF/              # Protected (not directly accessible)
    │   ├── web.xml           # Servlet configuration
    │   ├── classes/          # Compiled classes (from src/main/java)
    │   ├── lib/              # Dependency JARs
    │   └── views/            # JSP/template files
    ├── index.html            # Public files (directly accessible)
    ├── css/
    └── js/

Note: For JAR projects (Spring Boot), use src/main/resources/static instead
```

#### src/test/java

```
Purpose: Test source code (unit & integration tests)
Compiled to: target/test-classes
Not included in final JAR/WAR

Test naming conventions:
  - *Test.java      (JUnit)
  - *Tests.java     (JUnit)
  - Test*.java      (JUnit)
  - *IT.java        (Integration Tests - Failsafe plugin)
```

#### src/test/resources

```
Purpose: Test-specific resources
Output: Copied to target/test-classes

Common files:
  - application-test.properties
  - test-data.sql
  - mock-responses.json
  - test-certificates/

Overrides src/main/resources during tests
```

#### target/

```
Purpose: Build output directory
Generated by Maven, should be in .gitignore

Contents:
  - classes/              # Compiled production code
  - test-classes/         # Compiled test code
  - generated-sources/    # Generated code (annotation processors)
  - surefire-reports/     # Unit test reports (XML/TXT)
  - failsafe-reports/     # Integration test reports
  - *.jar / *.war         # Final packaged artifact
  - maven-archiver/       # Packaging metadata
  - maven-status/         # Build status

Clean with: mvn clean
```

### Project Types and Structure

#### JAR Project (Library or Spring Boot)

```xml
<packaging>jar</packaging>
```

```
Structure:
  src/
    └── main/
        ├── java/           # Code
        └── resources/      # Config files, static resources

  target/my-app-1.0.0.jar

    ├── static/             # Static resources
    ├── templates/          # Templates
    └── META-INF/
        ├── MANIFEST.MF
        └── maven/
```

#### WAR Project (Traditional Web App)

```xml
<packaging>war</packaging>
```

```
Structure:
  src/
    └── main/
        ├── java/           # Code
        ├── resources/      # Config files
        └── webapp/         # Web resources
            └── WEB-INF/

Output:
  target/my-webapp-1.0.0.war
    ├── WEB-INF/
    │   ├── classes/        # From src/main/java
    │   ├── lib/            # Dependencies
    │   └── web.xml
    ├── index.html          # From src/main/webapp
    ├── css/
    └── js/
```

#### POM Project (Parent or Aggregator)

```xml
<packaging>pom</packaging>
```

```
Purpose: 
  - Parent POM: Share configuration across modules
  - Aggregator: Build multiple modules together

Structure:
  parent-project/
    ├── pom.xml           # Parent/Aggregator POM
    ├── module-a/
    │   └── pom.xml
    └── module-b/
        └── pom.xml
```

### Customizing Directory Structure

While not recommended, you can customize the structure:

```xml
<build>
    <!-- Override source directories -->
    <sourceDirectory>src/java</sourceDirectory>
    <testSourceDirectory>test/java</testSourceDirectory>
    
    <!-- Override resource directories -->
    <resources>
        <resource>
            <directory>src/resources</directory>
        </resource>
    </resources>
    
    <testResources>
        <testResource>
            <directory>test/resources</directory>
        </testResource>
    </testResources>
    
    <!-- Override output directory -->
    <outputDirectory>build/classes</outputDirectory>
    <testOutputDirectory>build/test-classes</testOutputDirectory>
</build>
```

**However, prefer standard layout because:**

- Everyone knows where things are
- IDEs work without configuration
- CI/CD tools expect standard layout
- Plugins assume standard structure

## 4. Convention Over Configuration

Maven's philosophy is to provide sensible defaults for everything.

### What You Get For Free

```
Without any configuration, Maven provides:

✓ Standard directory layout
✓ Compilation of src/main/java
✓ Compilation of src/test/java
✓ Test execution with JUnit
✓ JAR packaging
✓ Installation to local repository
✓ Dependency management
✓ Transitive dependency resolution
✓ Resource filtering
✓ Site generation
```

### Minimal vs Full Configuration

**Minimal POM (uses all defaults):**

```xml
<project>
    <modelVersion>4.0.0</modelVersion>
    <groupId>com.company</groupId>
    <artifactId>my-app</artifactId>
    <version>1.0.0</version>
    
    <dependencies>
        <dependency>
            <groupId>junit</groupId>
            <artifactId>junit</artifactId>
            <version>4.13.2</version>
            <scope>test</scope>
        </dependency>
```

### When to Override Defaults

Only override when you have specific requirements:

```xml
<!-- Override Java version -->
<properties>
    <maven.compiler.source>17</maven.compiler.source>
    <maven.compiler.target>17</maven.compiler.target>
</properties>

<!-- Override final artifact name -->
<build>
    <finalName>my-custom-name</finalName>
</build>

<!-- Add additional source directories -->
<build>
    <plugins>
        <plugin>
            <groupId>org.codehaus.mojo</groupId>
            <artifactId>build-helper-maven-plugin</artifactId>
            <executions>
                <execution>
                    <phase>generate-sources</phase>
                    <goals>
                        <goal>add-source</goal>
                    </goals>
                    <configuration>
                        <sources>
                            <source>src/generated/java</source>
                        </sources>
                    </configuration>
                </execution>
            </executions>
        </plugin>
    </plugins>
</build>
```

## Summary

### Key Concepts

1. **Maven = Build Tool + Dependency Manager**
    
    - Automates compilation, testing, packaging
    - Manages dependencies and their versions
    - Provides consistent project structure
2. **Three-Tier Repository System**
    
    - Local: `~/.m2/repository` (cache)
    - Remote: Company repositories (Nexus/Artifactory)
    - Central: Public open-source libraries
3. **POM is the Heart of Maven**
    
    - Defines project coordinates (GAV)
    - Declares dependencies
    - Configures build process
    - Inherits from parent POMs
4. **Build Lifecycle**
    
    - Default: validate → compile → test → package → verify → install → deploy
    - Clean: removes build output
    - Site: generates documentation
5. **Standard Directory Layout**
    
    - `src/main/java`: Production code
    - `src/main/resources`: Configuration files
    - `src/test/java`: Test code
    - `target/`: Build output
6. **Convention Over Configuration**
    
    - Minimal configuration needed
    - Override only when necessary
    - Consistent across all Maven projects

### Common Commands

```bash
mvn clean                # Clean build output
mvn compile              # Compile source code
mvn test                 # Run tests
mvn package              # Build JAR/WAR
mvn install              # Install to local repo
mvn deploy               # Deploy to remote repo
mvn dependency:tree      # View dependencies
mvn help:effective-pom   # See final POM
```
