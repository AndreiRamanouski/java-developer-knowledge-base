# High Availability and Disaster Recovery

## Overview

High Availability (HA) ensures a system remains operational despite individual component failures. Disaster Recovery (DR) ensures the system can be restored after a catastrophic event — data centre fire, region outage, ransomware, accidental deletion. HA is about preventing downtime; DR is about recovering from it. A system can have excellent HA (three-nines uptime through redundancy) and poor DR (no tested restore procedure) — and discover the gap only during an actual disaster.

**The cost of downtime:**

```
Industry downtime cost benchmarks (approximate):
  E-commerce:         $100k–$500k per hour
  Financial trading:  $5M+ per hour
  Healthcare:         patient safety risk (beyond financial)
  SaaS B2B:          customer SLA breach + churn + reputation

Availability targets:
  99%      ("two nines"):   87.6 hours/year downtime   — unacceptable for production
  99.9%    ("three nines"): 8.76 hours/year            — typical baseline
  99.95%:                   4.38 hours/year            — most cloud-native targets
  99.99%   ("four nines"):  52 minutes/year            — financial, healthcare
  99.999%  ("five nines"):  5 minutes/year             — telecoms, trading systems

Formula: downtime_per_year = (1 - availability) × 8760 hours
```

**Key metrics defined:**

```
RTO (Recovery Time Objective):
  Maximum acceptable time from outage start to service restoration.
  "If we go down at 2pm, we must be back by 2:15pm" → RTO = 15 minutes.

RPO (Recovery Point Objective):
  Maximum acceptable data loss measured in time.
  "If we fail over, we can lose at most 5 minutes of transactions" → RPO = 5 minutes.
  RPO drives backup frequency and replication strategy.

MTTR (Mean Time To Recover):
  Average time to restore service after a failure. Operational metric.

MTBF (Mean Time Between Failures):
  Average time between failures. Drives reliability engineering effort.

Availability = MTBF / (MTBF + MTTR)
  MTBF = 730 hours (once/month), MTTR = 1 hour → 99.86% availability
  Reduce MTTR: better runbooks, automation, on-call alerting
  Increase MTBF: better testing, redundancy, chaos engineering
```

## 1. Redundancy and Replication

Redundancy eliminates single points of failure by deploying multiple instances of every component. Replication ensures data exists on multiple nodes so no single node failure causes data loss.

### 1.1 Eliminating single points of failure

```
Every layer needs redundancy:

  ┌──────────────────────────────────────────────────────────────────┐
  │  DNS:        Multiple DNS providers (Route53 + Cloudflare)       │
  │              GeoDNS for regional failover                        │
  ├──────────────────────────────────────────────────────────────────┤
  │  CDN:        Global PoPs — inherently redundant                  │
  ├──────────────────────────────────────────────────────────────────┤
  │  Load Balancer: Active-Active pair (or managed: AWS ALB/NLB)     │
  │               Each LB in different Availability Zone             │
  ├──────────────────────────────────────────────────────────────────┤
  │  Application:  N stateless instances across ≥2 AZs              │
  │               Auto Scaling Group — replace failed instances      │
  ├──────────────────────────────────────────────────────────────────┤
  │  Cache:        Redis Sentinel (1 master + 2 replicas)            │
  │               or Redis Cluster (3+ shards × 2 replicas)         │
  ├──────────────────────────────────────────────────────────────────┤
  │  Database:     Primary + Synchronous replica (same region)       │
  │               + Async replica in second region (cross-region DR) │
  ├──────────────────────────────────────────────────────────────────┤
  │  Message Queue: Kafka: 3 brokers, replication factor 3           │
  │               Minimum ISR 2 — tolerate 1 broker failure          │
  ├──────────────────────────────────────────────────────────────────┤
  │  Object Storage: S3 — 11 nines durability (3 AZ replication)     │
  └──────────────────────────────────────────────────────────────────┘

Rule: N+1 redundancy for critical components (N needed + 1 spare).
      N+2 for systems that cannot tolerate even brief degradation.
```

### 1.2 Database replication modes

```java
// PostgreSQL synchronous replication — RPO = 0 (no data loss on primary failure)
// primary postgres.conf:
// synchronous_standby_names = 'FIRST 1 (replica1, replica2)'
// synchronous_commit = on
//
// Write committed only after at least 1 replica confirms WAL write to disk.
// Write latency += replica round-trip (~1ms same DC, ~5ms cross-AZ)

// PostgreSQL asynchronous replication (default) — low write latency, small RPO
// synchronous_commit = off  (or not set)
// Replica lag: typically 10-200ms in same region under load
// RPO during primary failure = replication lag at time of failure (usually < 1s)

// Spring — read from replica, write to primary
@Configuration
public class ReplicationAwareDataSource {
    @Primary @Bean
    DataSource primaryDs() {
        return DataSourceBuilder.create()
            .url("jdbc:postgresql://primary.db.internal:5432/mydb")
            .username("app").password("secret").build();
    }

    @Bean("replicaDs")
    DataSource replicaDs() {
        // Round-robin across replicas via PgBouncer or application-level routing
        return DataSourceBuilder.create()
            .url("jdbc:postgresql://replica.db.internal:5432/mydb")
            .username("app_ro").password("secret").build();
    }
}

@Transactional(readOnly = true) // routes to replica via AbstractRoutingDataSource
public List<Order> findRecentOrders(Long userId) {
    return orderRepository.findByUserIdOrderByCreatedAtDesc(userId);
}
```

### 1.3 Replication topologies

```
Primary-Replica (Single Primary):
  One writable primary → N read replicas
  Failover: promote one replica to primary
  Use: most OLTP workloads
  Risk: write SPOF (mitigated by fast failover)

Primary-Primary (Multi-Primary / Active-Active):
  Both nodes accept writes — requires conflict resolution
  Conflict resolution: last-write-wins (LWW), CRDTs, application-level merging
  Use: geographically distributed writes (each region writes locally)
  Risk: write conflicts if same row updated in both regions simultaneously
  Examples: CockroachDB, MySQL Group Replication, DynamoDB Global Tables

Read Replicas (Fan-Out):
  One primary → many replicas for read scaling
  Each replica may serve a different team or reporting query
  Risk: replication lag means reads may be stale

Cascading Replicas:
  Primary → Replica1 → Replica2 (chain)
  Reduces load on primary (only replicates to one node)
  Risk: cascade lag — Replica2 is always more stale than Replica1

Multi-Region:
  Primary in us-east → Async replica in eu-west
  Replica in eu-west can be promoted if us-east fails entirely
  RPO = replication lag (typically 100-500ms cross-continent)
```

## 2. Failover Strategies

A failover is the process of redirecting traffic from a failed component to a healthy one. The strategy determines how fast, how automatic, and how correctly this transition occurs.

### 2.1 Active-Passive failover

```
Active-Passive (Cold/Warm Standby):

  Normal operation:
    ┌──────────┐          ┌──────────────┐
    │  Active  │◀─traffic─│ Load Balancer│
    │  Primary │          └──────────────┘
    └──────────┘
    ┌──────────┐
    │ Passive  │  ← idle (cold) or warmed (warm), receiving replication
    │ Standby  │
    └──────────┘

  After failure detection:
    ┌──────────┐  ✗ failed  ┌──────────────┐
    │  Active  │            │ Load Balancer│─▶ promotes standby
    │  Primary │            └──────────────┘
    └──────────┘
    ┌──────────┐
    │ Standby  │◀─traffic─  ← now active
    │(promoted)│
    └──────────┘

Cold standby:  standby is off; must boot on failure  → RTO: minutes
Warm standby:  standby is running, synced, not serving traffic → RTO: seconds
Hot standby:   standby serves reads, instant write promotion  → RTO: < 1 second

Cost: cold < warm < hot (idle resources)
```

### 2.2 Active-Active failover

```
Active-Active:
  Both nodes serve live traffic simultaneously.
  On failure of one node, surviving nodes absorb its traffic.

  ┌──────────┐
  │ Node A   │◀─50% traffic─┐
  └──────────┘               │  ┌──────────────┐
                             ├──│ Load Balancer│
  ┌──────────┐               │  └──────────────┘
  │ Node B   │◀─50% traffic─┘
  └──────────┘

  Node A fails:
  ┌──────────┐  ✗
  └──────────┘
                             ┌──────────────┐
  ┌──────────┐◀─100% traffic─│ Load Balancer│
  │ Node B   │               └──────────────┘
  └──────────┘

Benefits: no idle capacity wasted; zero failover latency (already active)
Risk: surviving nodes must handle 2x load — capacity planning critical
Use: stateless application servers, caches, Kafka brokers
```

```java
// Application: handle failover transparently with retry + connection refresh
@Configuration
public class DataSourceFailoverConfig {

    @Bean
    public DataSource failoverAwareDataSource() {
        // PgBouncer / HAProxy in front of database handles DNS-based failover
        // Application reconnects automatically through connection pool
        HikariConfig config = new HikariConfig();
        config.setJdbcUrl("jdbc:postgresql://db-ha.internal:5432/mydb");
        config.setMaximumPoolSize(20);
        config.setConnectionTimeout(5_000);       // 5s timeout on new connections
        config.setKeepaliveTime(30_000);           // detect stale connections early
        config.setMaxLifetime(600_000);            // recycle connections every 10 min

        // On failover: old connections throw exception → pool discards them
        // → establishes fresh connections to new primary via DNS update
        config.setInitializationFailTimeout(-1);   // don't fail startup if DB unavailable
        return new HikariDataSource(config);
    }
}

// Service: retry transient DB failures during failover window
@Service
public class OrderService {

    @Retryable(
        retryFor = { DataAccessException.class, TransactionException.class },
        maxAttempts = 3,
        backoff = @Backoff(delay = 500, multiplier = 2, maxDelay = 5000)
    )
    @Transactional
    public Order createOrder(CreateOrderRequest req) {
        return orderRepository.save(Order.from(req));
        // If DB failover causes transient exception → retried 3x with backoff
        // Most failovers complete in < 5 seconds → retry succeeds
    }
}
```

### 2.4 Health check and circuit breaker in failover

```java
// Spring Boot Actuator readiness/liveness probes for Kubernetes failover
@Component
public class DatabaseReadinessIndicator implements HealthIndicator {
    private final DataSource dataSource;
    private volatile boolean isDegraded = false;

    @Override
    public Health health() {
        try (Connection c = dataSource.getConnection()) {
            c.createStatement().executeQuery("SELECT 1");
            isDegraded = false;
            return Health.up().withDetail("db", "reachable").build();
        } catch (SQLException e) {
            isDegraded = true;
            // Return DOWN → Kubernetes marks pod NotReady → LB stops routing → failover
            return Health.down().withDetail("error", e.getMessage()).build();
        }
    }
}

// Kubernetes readiness failure → pod removed from Service endpoints
// → load balancer stops sending traffic → effectively failover to healthy pods
```

## 3. Backup and Restore

A backup is useless until it is tested. Restoring from an untested backup during a disaster is the worst time to discover the backup is corrupted, incomplete, or missing critical tables.

### 3.1 Backup types and strategies

```
Full Backup:
  Complete snapshot of all data.
  Pros: simplest restore (single file/snapshot)
  Cons: large, slow, storage-intensive — impractical daily for 10+ TB databases

Incremental Backup:
  Only changes since last backup (full or incremental).
  Pros: small, fast to create
  Cons: restore requires full + all incrementals in sequence (slow, fragile chain)

Differential Backup:
  All changes since last FULL backup (not since last differential).
  Pros: restore = full + one differential (faster than incremental chain)
  Cons: grows over time until next full backup

Continuous / Point-in-Time Recovery (PITR):
  Continuous archiving of WAL (Write-Ahead Log) or binlog.
  Restore to any point in time within retention window.
  RPO = seconds (limited by WAL archival frequency)
  Used by: PostgreSQL pg_basebackup + WAL archive, MySQL binlog backup
```

### 3.2 PostgreSQL backup implementation

```bash
#!/bin/bash
# Automated PostgreSQL backup to S3 with pgBackRest

# pgBackRest configuration (/etc/pgbackrest/pgbackrest.conf)
cat > /etc/pgbackrest/pgbackrest.conf << 'EOF'
[global]
repo1-type=s3
repo1-s3-bucket=my-db-backups
repo1-s3-region=us-east-1
repo1-s3-key=AKIAIOSFODNN7EXAMPLE
repo1-s3-key-secret=wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY
repo1-retention-full=7          # keep 7 full backups
repo1-retention-diff=30         # keep 30 differential backups
repo1-cipher-type=aes-256-cbc  # encrypt backups at rest
repo1-cipher-pass=strong-passphrase

[mydb]
pg1-path=/var/lib/postgresql/14/main
EOF

# Weekly full backup (Sunday 1am)
# 0 1 * * 0 pgbackrest --stanza=mydb backup --type=full

# Daily differential backup (Mon-Sat 1am)
# 0 1 * * 1-6 pgbackrest --stanza=mydb backup --type=diff

# Continuous WAL archiving (configured in postgresql.conf):
# archive_mode = on
# archive_command = 'pgbackrest --stanza=mydb archive-push %p'
# archive_status_interval = 10  # archive every 10 seconds

# Restore to specific point in time
pgbackrest --stanza=mydb --delta \
    --type=time "--target=2024-03-15 14:30:00+00" \
    restore
# --delta: only restore changed files (faster for large DBs)
# --type=time: PITR — restore to exact timestamp
```

### 3.3 Application data backup

```java
// Transactional outbox backup: ensure events are durable
// Kubernetes CronJob for database snapshot to S3

// Spring: scheduled backup metadata logging and verification
@Component
@RequiredArgsConstructor
public class BackupVerificationService {
    private final S3Client s3;
    private final AlertService alertService;

    // Daily backup integrity check
    @Scheduled(cron = "0 0 6 * * *")  // 6am daily — after overnight backup
    public void verifyLatestBackup() {
        try {
            // Check backup exists and was created within last 25 hours
            S3Object latest = s3.listObjectsV2(req -> req
                .bucket("my-db-backups")
                .prefix("backups/")
                .maxKeys(1))
                .contents().stream()
                .max(Comparator.comparing(S3Object::lastModified))
                .orElseThrow(() -> new BackupMissingException("No backups found"));

            Duration age = Duration.between(latest.lastModified(), Instant.now());
            if (age.toHours() > 25) {
                alertService.page("BACKUP ALERT: Latest backup is " +
                    age.toHours() + " hours old — backup may have failed");
            }

            // Verify checksum matches recorded manifest
            verifyChecksum(latest);
            log.info("Backup verified: {} size={} age={}h",
                latest.key(), latest.size(), age.toHours());

        } catch (Exception e) {
            alertService.page("BACKUP VERIFICATION FAILED: " + e.getMessage());
        }
    }
}
```

### 3.4 Restore testing — the critical step

```bash
#!/bin/bash
# Automated weekly restore test to staging environment
# Run in CI/CD pipeline every Sunday — alerts if restore fails

set -e  # exit on any error

RESTORE_HOST="restore-test.internal"
BACKUP_DATE=$(date -d "yesterday" +"%Y-%m-%d")

echo "=== Restore Test: ${BACKUP_DATE} ==="

# 1. Stop any running restore-test DB
ssh ${RESTORE_HOST} "sudo systemctl stop postgresql || true"

# 2. Restore latest backup
pgbackrest --stanza=mydb \
    --pg1-path=/var/lib/postgresql/restore-test \
    --target-action=promote \
    restore

# 3. Start restored DB
ssh ${RESTORE_HOST} "sudo systemctl start postgresql"
sleep 10  # wait for startup

# 4. Run validation queries
RESULT=$(psql -h ${RESTORE_HOST} -U validator -c "
    SELECT
        (SELECT COUNT(*) FROM orders) AS order_count,
        (SELECT COUNT(*) FROM users)  AS user_count,
        (SELECT MAX(created_at) FROM orders) AS latest_order
")

echo "Validation result: ${RESULT}"

# 5. Check data freshness — latest order should be < 24h old
HOURS_OLD=$(psql -h ${RESTORE_HOST} -U validator -t -c "
    SELECT EXTRACT(EPOCH FROM (NOW() - MAX(created_at)))/3600
    FROM orders
")

if (( $(echo "$HOURS_OLD > 24" | bc -l) )); then
    echo "RESTORE TEST FAILED: Data is ${HOURS_OLD} hours old (expected < 24)"
    curl -X POST "${PAGERDUTY_WEBHOOK}" -d '{"message":"Restore test FAILED: data too stale"}'
    exit 1
fi

echo "=== Restore Test PASSED: Data ${HOURS_OLD}h old ==="
# Publish success metric to monitoring
curl -X POST "${METRICS_ENDPOINT}" \
    -d "restore_test_success=1,hours_old=${HOURS_OLD}"
```

## 4. Multi-Region Deployment

Multi-region deployment places infrastructure in geographically separate cloud regions to survive region-level failures (data centre disasters) and reduce latency for globally distributed users.

### 4.1 Multi-region patterns

```
Active-Passive (Pilot Light):
  Primary region: fully active, serving all traffic
  DR region: minimal infrastructure running (DB replica, no application servers)
  On failover: spin up application tier in DR region, promote DB replica
  RTO: 30–60 minutes (infrastructure provisioning time)
  RPO: replication lag (typically < 1 minute cross-region)
  Cost: low (minimal idle resources in DR region)
  Use: systems where 30-minute RTO is acceptable

Active-Passive (Warm Standby):
  Primary region: fully active
  DR region: full application stack running at reduced capacity (1/4 size)
  On failover: DNS flip to DR region, scale up capacity
  RTO: 5–15 minutes (DNS TTL + scale-out time)
  RPO: replication lag (< 1 minute)
  Cost: medium (reduced idle tier in DR region)
  Use: systems needing < 15 minute RTO

Active-Active (Multi-Master):
  Both regions serve live traffic simultaneously
  Users routed to nearest region by GeoDNS/Anycast
  Data replicated bi-directionally (conflict resolution required)
  On region failure: all traffic fails over to surviving region
  RTO: seconds (routing layer, not infrastructure change)
  RPO: near-zero (bi-directional sync)
  Cost: high (full infrastructure in both regions)
  Use: global consumer products, financial systems, < 30 second RTO
```

### 4.3 Data residency and cross-region consistency

```java
// Multi-region write: write to local region, replicate async to others
// Key challenge: user in EU writes data, user in US reads it before replication
// (read-your-writes consistency across regions)

@Service
@RequiredArgsConstructor
public class MultiRegionOrderService {
    private final OrderRepository localRepo;        // same-region DB
    private final RedisTemplate<String, String> redis; // distributed session store

    @Transactional
    public Order createOrder(CreateOrderRequest req, String userId) {
        Order order = localRepo.save(Order.from(req));

        // Record "sticky primary" for this user — next reads routed to this region
        // Prevents reading from replica before replication completes
        redis.opsForValue().set(
            "region:sticky:" + userId,
            currentRegion(),
            Duration.ofSeconds(30)  // 30 seconds: enough for cross-region replication
        );

        return order;
    }

    public Order findOrder(Long orderId, String userId) {
        String stickyRegion = redis.opsForValue().get("region:sticky:" + userId);

        if (currentRegion().equals(stickyRegion)) {
            // Read from primary (or local primary in this region)
            return localRepo.findById(orderId).orElseThrow();
        }
        // No sticky — safe to read from local replica
        return replicaRepo.findById(orderId).orElseThrow();
    }
}
```

## 5. Chaos Engineering

Chaos engineering deliberately injects failures into a production-like system to discover weaknesses before they cause real outages. The principle: if failures will happen in production, practise recovering from them intentionally under controlled conditions.

### 5.1 Principles

```
Core hypothesis methodology:
  1. Define steady state: measurable baseline of healthy behaviour
     (e.g. 99th percentile latency < 200ms, error rate < 0.1%)
  2. Hypothesize: "The system will remain in steady state when X fails"
  3. Introduce failure: inject the failure in production (or staging)
  4. Observe: does the system maintain steady state?
  5. Fix: if steady state violated, fix the weakness
  6. Automate: run continuously to catch regressions

Blast radius minimisation:
  Start small: single instance, then AZ, then region
  Start in staging: prove the experiment before running in production
  Feature flags: abort experiment if metrics breach thresholds
  Monitoring must be in place BEFORE chaos experiments begin
```

### 5.2 Chaos experiments by failure mode

```java
// Chaos Monkey (Simian Army) style — implemented as a Spring component
// In practice: use Chaos Mesh (K8s), Gremlin, or AWS Fault Injection Simulator

@Component
@ConditionalOnProperty("chaos.enabled")
@RequiredArgsConstructor
public class ChaosExperiments {

    // Experiment 1: Latency injection — does the system degrade gracefully?
    @Scheduled(cron = "${chaos.latency.cron:0 30 14 * * TUE}")  // Tuesday 2:30pm
    public void injectLatency() {
        String targetService = "inventory-service";
        Duration injectedDelay = Duration.ofMillis(500);
```

### 5.4 Measuring chaos experiment results

```java
// GameDay runbook: structured chaos experiment execution
@Component
public class ChaosDashboard {

    // Before experiment: capture baseline metrics
    public ExperimentBaseline captureBaseline(Duration window) {
        return ExperimentBaseline.builder()
            .p99Latency(metrics.getPercentile("http.request.duration", 99, window))
            .errorRate(metrics.getRate("http.requests.errors", window))
            .throughput(metrics.getRate("http.requests.success", window))
            .timestamp(Instant.now())
            .build();
    }

    // During/after experiment: compare to baseline
    public ExperimentResult evaluate(ExperimentBaseline baseline,
                                      String experimentName,
                                      Duration duration) {
        ExperimentBaseline during = captureBaseline(duration);

        double latencyDegradation =
            (during.getP99Latency() - baseline.getP99Latency())
            / baseline.getP99Latency() * 100;
        double errorRateIncrease =
            during.getErrorRate() - baseline.getErrorRate();

        boolean steadyStateMaintained =
            latencyDegradation < 50 &&          // < 50% latency increase
            errorRateIncrease < 0.01;           // < 1% error rate increase

        return ExperimentResult.builder()
            .experimentName(experimentName)
            .steadyStateMaintained(steadyStateMaintained)
            .latencyDegradation(latencyDegradation)
            .errorRateIncrease(errorRateIncrease)
            .finding(steadyStateMaintained ? "PASS" : "WEAKNESS FOUND")
            .build();
    }
```

## 6. RTO and RPO Targets

### 6.1 Defining targets

```
Business inputs to RTO/RPO:
  Revenue per hour downtime:   directly sets RTO ceiling
  Cost of data loss per minute: directly sets RPO ceiling
  Contractual SLAs:            SLA defines maximum acceptable RTO
  Regulatory requirements:      some industries mandate specific RPO/RTO

Process:
  1. Business: "We lose $500k/hr. One hour downtime = $500k."
  2. Engineering: "Multi-AZ active-active costs $200k/yr, gives RTO < 1 min"
  3. Business: decision: pay $200k/yr to avoid $500k+ incidents → yes

Common tier classification:
  Tier 0 (Mission Critical): payment processing, authentication
    RTO: < 1 minute  RPO: 0 (zero data loss)
    Strategy: active-active multi-AZ, synchronous replication

  Tier 1 (Critical): order placement, user accounts
    RTO: < 15 minutes  RPO: < 1 minute
    Strategy: multi-AZ, async replication, automated failover

  Tier 2 (Important): product catalog, search, recommendations
    RTO: < 1 hour  RPO: < 15 minutes
    Strategy: single-region with read replicas, scheduled backups

  Tier 3 (Non-Critical): analytics, reporting, batch jobs
    RTO: < 4 hours  RPO: < 1 hour
    Strategy: nightly backups, manual restore acceptable
```

### 6.2 RTO/RPO to architecture mapping

```
RPO = 0 (zero data loss):
  → Synchronous replication: every write confirmed on ≥2 nodes before ACK
  → Tools: PostgreSQL sync_commit, Aurora Multi-AZ, Redis AOF + replica

RPO = seconds:
  → Asynchronous replication with sub-second lag
  → Continuous WAL archiving (pgBackRest) to durable storage
  → Tools: PostgreSQL async replica, Aurora Global Database (< 1s cross-region)

RPO = minutes:
  → Frequent WAL/binlog shipping (every 60s)
  → Managed: AWS RDS automated backups, Aurora PITR

RPO = hours:
  → Scheduled full/differential backups
  → Acceptable for non-critical data (reports, analytics)

RTO = seconds:
  → Active-Active: traffic rerouted by health check on LB (< 30s DNS TTL)
  → Kubernetes: failing pods replaced automatically (<30s)
  → Redis Sentinel: automatic master election (< 30s)

RTO = minutes:
  → Active-Passive warm standby: DNS flip + standby promotion
  → AWS RDS Multi-AZ: automatic failover (60-120s)
  → Patroni: leader election + promotion (15-30s)

RTO = tens of minutes:
  → Warm standby in DR region: spin up app tier, promote DB replica
  → Requires automation: runbook or Terraform to provision

RTO = hours:
  → Cold standby: provision infrastructure from scratch
  → Restore from backup: acceptable for non-critical systems
```

### 6.3 SLA and error budget

```java
// SLA definition and error budget tracking
// Error budget = time allowed to be down within SLA

// 99.9% SLA = 8.76 hours/year = 43.8 minutes/month downtime budget
// If you've used 30 minutes this month, 13.8 minutes remain

@Service
@RequiredArgsConstructor
public class ErrorBudgetTracker {
    private final MetricsClient metrics;

    public ErrorBudget calculateMonthlyBudget(String service, double slaTarget) {
        double uptimeFraction = metrics.getUptimeFraction(service, Period.ofMonths(1));
        double budgetFraction = 1.0 - slaTarget; // e.g. 0.001 for 99.9%

        // Minutes in a month
        double totalMinutes = 30.0 * 24 * 60;  // 43,200 minutes
        double budgetMinutes = totalMinutes * budgetFraction;
        double usedMinutes = totalMinutes * (1 - uptimeFraction);
        double remainingMinutes = budgetMinutes - usedMinutes;

        double burnRate = usedMinutes / budgetMinutes;

        return ErrorBudget.builder()
            .service(service)
            .slaTarget(slaTarget)
            .budgetMinutesPerMonth(budgetMinutes)
            .usedMinutes(usedMinutes)
            .remainingMinutes(remainingMinutes)
            .burnRate(burnRate)
            .status(burnRate > 1.0 ? "EXCEEDED" :
                    burnRate > 0.8 ? "AT_RISK" : "HEALTHY")
            .build();
    }

    // Alert when error budget is burning too fast (multi-window burn rate alert)
    // If 5% of monthly budget consumed in 1 hour → alert (2% budget/hr vs 0.14% normal)
    @Scheduled(fixedDelay = 60_000)
    public void checkBurnRate() {
        for (String service : criticalServices) {
            double hourlyBurnRate = metrics.getDowntimeMinutes(service, Duration.ofHours(1));
            double monthlyBudget = 43_200 * (1 - getSLATarget(service));

            if (hourlyBurnRate > monthlyBudget * 0.05) {  // 5% of monthly budget in 1 hour
                alertService.page(service + " error budget burning at " +
                    hourlyBurnRate + " min/hr — investigate immediately");
            }
        }
    }
}
```

## 7. Interview Questions

**Q: What is the difference between RTO and RPO, and how do they drive architecture decisions?**

A: RTO is the maximum acceptable time from failure to service restoration — it answers "how long can we be down?" RPO is the maximum acceptable data loss measured in time — it answers "how much data can we lose?" They drive architecture independently. RPO drives replication strategy: RPO = 0 requires synchronous replication where every write is confirmed on multiple nodes before acknowledging to the client, at the cost of write latency. RPO = 1 minute allows asynchronous replication or continuous WAL archiving, which is cheaper and adds no write latency. RTO drives failover strategy: RTO = 30 seconds demands active-active deployment where healthy nodes instantly absorb failed nodes' traffic; RTO = 15 minutes allows warm standby with DNS failover; RTO = hours allows cold standby with manual recovery. The cost increases steeply as either target tightens — active-active multi-region for RPO=0 and RTO=30s might cost 5-10x more than a warm standby. The business must decide how much that insurance is worth based on the cost of downtime.

---

**Q: What is chaos engineering, and how is it different from regular load testing?**

A: Load testing validates that a system handles expected (and peak) traffic volumes — it stresses the happy path under high volume. Chaos engineering deliberately breaks things to validate that the system handles failures gracefully. Load testing answers "can we handle 10x traffic?" Chaos engineering answers "what happens when the database fails at 2x traffic?" The methodology follows a scientific pattern: define the system's "steady state" (measurable metrics like p99 latency and error rate), form a hypothesis ("the system will maintain steady state when one AZ fails"), inject the failure, observe whether steady state is maintained, and fix any weakness found. The key insight is that if failures will happen in production regardless, it is far better to encounter them intentionally — during business hours, with engineers watching dashboards — than accidentally, at 3am, with no warning. Blast radius minimisation is critical: start with a single instance, not an entire region; start in staging; have a kill switch ready; ensure monitoring is in place before running experiments.

---

**Q: How do you design a system to achieve 99.99% availability?**

A: Four nines means only 52 minutes of downtime per year, which requires eliminating every single point of failure across the entire stack. I'd address each layer: DNS with multiple providers and low TTL (Route53 + Cloudflare, TTL=60s); CDN with a global provider (inherently redundant); load balancers as a managed HA pair or cloud-managed (AWS ALB); application servers as stateless instances across at minimum 2 AZs with auto-scaling and pod anti-affinity so no AZ failure takes more than half the capacity; cache with Redis Cluster (3+ shards, each with a replica); database as synchronous multi-AZ replica for RPO=0, with Patroni or RDS Multi-AZ for automatic failover in < 60 seconds. Cross-cutting: circuit breakers prevent cascade failures; health checks ensure traffic stops routing to unhealthy pods within seconds; PodDisruptionBudgets ensure rolling deploys never drop below minimum capacity; chaos engineering validates that the failover mechanisms actually work. Finally, 99.99% requires measuring it — error budget tracking with multi-window burn rate alerts catches problems before the SLA is breached.

---

**Q: Explain the transactional outbox pattern in the context of disaster recovery.**

A: In a distributed system, data loss during a failure can occur not just from database crashes but from the gap between writing state to the database and publishing the corresponding event to downstream services. If a service saves an order, crashes, and the Kafka publish never happens, the downstream inventory and payment services never process the order — silent data loss from an application perspective. The transactional outbox solves this by writing both the business record and the outbox event in a single database transaction. Since the database has durable storage (WAL, replicated across AZs), the event survives any application crash. The relay process (or Debezium CDC) reads the outbox and publishes to Kafka; it retries if Kafka is unavailable. From a DR perspective, the database backup and PITR capability covers the outbox table alongside business data, so a restore from backup recovers both state and unpublished events. This pattern treats the database as the source of truth for both data and the event stream, which is the safest approach to cross-system consistency.

---

**Q: What is the difference between active-passive and active-active multi-region deployments, and what are the trade-offs?**

A: In active-passive, one region serves all traffic while the other runs a standby replica receiving asynchronous replication. On primary region failure, DNS or traffic routing is switched to the DR region. The trade-off: the standby resources are idle or underutilised (wasted cost), failover takes time (DNS propagation, potential replica promotion), and the RPO equals the replication lag at failure time. Active-active serves live traffic from both regions simultaneously — users are routed to the nearest region by GeoDNS or anycast. On region failure, surviving regions absorb the traffic (they must have spare capacity). The trade-offs are write conflict complexity (if both regions accept writes to the same data, conflicts must be resolved — last-write-wins, CRDTs, or geographic partitioning of write responsibility), higher cost (full infrastructure in both regions), and the need for a conflict resolution strategy that suits the domain. For most stateless services (APIs, web servers), active-active is straightforward. For stateful data, geographic write partitioning (users in EU always write to EU, users in US to US) is the simplest conflict avoidance strategy, though it fails for globally shared data like inventory.

## 8. Key Takeaways

- **HA prevents downtime; DR recovers from it** — both are needed; excellent HA with no tested DR runbook leaves you exposed to catastrophic events
- **RTO drives failover strategy; RPO drives replication strategy** — define both from business cost of downtime/data loss, then choose the cheapest architecture that meets both
- **Eliminate every SPOF at every layer** — N+1 redundancy for critical components, spanning multiple AZs, with health checks and auto-replacement
- **Synchronous replication = RPO 0 but adds write latency** — use for financial and inventory data; accept async replication (RPO = seconds) for most other data
- **Backups are useless until tested** — weekly automated restore tests to staging, verify data freshness, alert on failures; treat backup verification as production-critical
- **Chaos engineering is the only way to know failover actually works** — automated failure injection during business hours beats accidental failures at 3am; define steady state first, inject failure second, fix findings third
- **Error budgets make SLAs operational** — 99.9% SLA = 43.8 minutes/month downtime budget; multi-window burn rate alerting catches budget exhaustion before it becomes a breach
- **Active-active costs more but gives near-zero RTO** — active-passive warm standby is the pragmatic choice for most systems (5–15 min RTO, < 1 min RPO) at far lower cost
- **DR runbooks must be written before the disaster** — tested, version-controlled, accessible without the system being down; include decision criteria, step-by-step commands, and validation checks
