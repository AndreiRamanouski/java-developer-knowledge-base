# Scalability Fundamentals

## Overview

Scalability is the ability of a system to handle growing load — more users, more data, more requests — without requiring a redesign. A system that works for 1,000 users may collapse at 1,000,000 not because the code is wrong, but because its architecture makes implicit assumptions about load that break at scale. Understanding scalability means understanding _where_ those assumptions live and _how_ to remove them before they become production incidents.

**The four axes of scale:**

```
Traffic:  requests per second (RPS), concurrent connections
Data:     storage volume, dataset size, query complexity
Geography: latency to globally distributed users
Complexity: team size, service count, deployment frequency

Most systems hit traffic limits first, then data limits.
Geographic and complexity scaling are usually addressed later.
```

**Back-of-envelope baselines (memorise these):**

```
Single server:              ~1,000–10,000 RPS (depends on work per request)
Single relational DB:       ~10,000–50,000 QPS (simple queries, indexed)
Single Redis node:          ~100,000–1,000,000 ops/sec
Single Kafka partition:     ~100 MB/s write throughput
CDN edge node:              millions of requests/sec
Network round trip (DC):    ~0.5 ms intra-region, ~100 ms cross-continent
Disk sequential read:       ~500 MB/s (SSD), ~150 MB/s (HDD)
Memory random access:       ~100 ns vs ~10 ms for SSD
```

## 1. Horizontal vs Vertical Scaling

### 1.1 Vertical Scaling (Scale Up)

Add more resources (CPU, RAM, faster disk) to a single machine.

```
Before:           After vertical scaling:
┌──────────┐      ┌──────────────────┐
│ 8 core   │  →   │ 64 core          │
│ 32 GB    │      │ 512 GB RAM       │
│ 1 TB SSD │      │ 8 TB NVMe RAID   │
└──────────┘      └──────────────────┘
Single server     Single larger server

Advantages:
  ✓ No application changes required
  ✓ No distributed system complexity (no network, no consistency issues)
  ✓ SQL transactions work unchanged
  ✓ Effective for stateful workloads (databases, in-memory caches)

Disadvantages:
  ✗ Hard ceiling — hardware limits exist (e.g. max ~24 TB RAM per server in 2024)
  ✗ Single point of failure — one machine, one fault domain
  ✗ Expensive — price per core increases non-linearly at top-end hardware
  ✗ Downtime required to upgrade most hardware components
  ✗ Does not improve geographic distribution
```

**When to use vertical scaling:**

```
Databases (first resort): PostgreSQL, MySQL — vertical is simpler than sharding.
  A 64-core server with 1 TB RAM can handle enormous read loads.
  Shard only after vertical scaling is insufficient.

In-memory caches: Redis — single large node avoids distributed cache complexity.
  Redis Cluster adds overhead; use only when single node is insufficient.

Message queue brokers: Kafka gains throughput from more CPU and disk IOPS.

Rule of thumb: scale vertically first, horizontally when you hit a wall.
  Most startups never need horizontal scaling of their primary database.
```

### 1.2 Horizontal Scaling (Scale Out)

Add more machines (nodes) of the same type and distribute load across them.

```
Before:           After horizontal scaling:
┌──────────┐      ┌──────────┐ ┌──────────┐ ┌──────────┐
│ Server 1 │  →   │ Server 1 │ │ Server 2 │ │ Server 3 │
│ (4 core) │      │ (4 core) │ │ (4 core) │ │ (4 core) │
└──────────┘      └──────────┘ └──────────┘ └──────────┘
                        └───── Load Balancer ─────┘

Advantages:
  ✓ Theoretically unlimited ceiling (add more machines)
  ✓ No single point of failure — survive individual node failures
  ✓ Cost-effective at scale — commodity hardware is cheap per unit
  ✓ Can scale specific bottleneck tiers independently
  ✓ Geographic distribution — put nodes close to users

Disadvantages:
  ✗ Application must be designed for distribution (stateless or shared state)
  ✗ Distributed system complexity: network partitions, consistency, coordination
  ✗ Session state must be externalised (Redis, database)
  ✗ Operational complexity: service discovery, health checks, deployments
```

**Key requirement for horizontal scaling: statelessness**

```java
// BAD: session stored in server memory — breaks with multiple servers
@RestController
public class CartController {
    private Map<String, Cart> sessions = new HashMap<>(); // local state!

    @GetMapping("/cart")
    public Cart getCart(HttpSession session) {
        return sessions.get(session.getId()); // only works on THIS server
    }
}

// GOOD: session stored externally — any server can handle any request
@RestController
@RequiredArgsConstructor
public class CartController {
    private final RedisTemplate<String, Cart> redis;

    @GetMapping("/cart")
    public Cart getCart(@RequestHeader("X-Session-Id") String sessionId) {
        return redis.opsForValue().get("cart:" + sessionId); // any server can serve this
    }
}
```


## 2. Load Balancing Strategies

A load balancer distributes incoming requests across multiple backend servers to maximise throughput, minimise latency, and avoid overloading any single server.

### 2.1 Load balancing algorithms

```
Round Robin:
  Requests distributed in sequence: server 1, 2, 3, 1, 2, 3...
  Pro: Simple, equal distribution
  Con: Ignores server capacity and current load
  Use: When servers are identical and requests are similar in cost

Weighted Round Robin:
  Server 1 (weight 3): gets 3 requests, Server 2 (weight 1): gets 1 request
  Pro: Handles heterogeneous server capacity
  Use: Mixed-size instances (e.g. some 8-core, some 32-core)

Least Connections:
  Route to server with fewest active connections
  Pro: Accounts for long-lived connections (WebSockets, slow requests)
  Con: Requires connection count state on the load balancer
  Use: Variable request processing time (API servers, real-time services)

Least Response Time:
  Route to server with lowest (active connections × response time)
  Pro: Directly optimises for latency
  Con: Requires active monitoring of backend response times
  Use: Latency-sensitive services

IP Hash / Sticky Sessions:
  hash(client_ip) % num_servers → always same server for same client
  Pro: Session affinity without shared state (simple legacy apps)
  Con: Breaks on server changes; uneven if many clients share NAT IP
  Use: Legacy stateful apps during migration to stateless

Random with Two Choices (Power of Two):
  Pick 2 servers randomly, route to the less loaded one
  Pro: Near-optimal load distribution with O(1) state — avoids herd effect
  Con: Needs connection count per server
  Use: Envoy, NGINX Plus, HAProxy — best practice for modern systems

Consistent Hashing:
  Map requests and servers onto a ring; route to nearest server clockwise
  Pro: Adding/removing servers only remaps ~1/n of requests (minimal churn)
  Con: Hot spots if keys are unevenly distributed (use virtual nodes)
  Use: Cache clusters, distributed databases, CDN routing
```

### 2.2 Layer 4 vs Layer 7 load balancing

```
Layer 4 (Transport — TCP/UDP):
  Sees: source IP, destination IP, TCP ports
  Does: NAT, connection routing without inspecting payload
  Performance: ~10M connections/sec (no payload parsing)
  Examples: AWS NLB, HAProxy (TCP mode), Linux IPVS
  Use when: Low latency required, TLS termination upstream

Layer 7 (Application — HTTP/HTTPS):
  Sees: HTTP headers, URL paths, cookies, request body
  Does: Content-based routing, SSL termination, header injection,
        authentication, rate limiting, request transformation
  Performance: ~1M req/sec (payload inspection overhead)
  Examples: AWS ALB, Nginx, Envoy, HAProxy (HTTP mode), Kong
  Use when: Need routing by path/host/header, A/B testing, canary deploys

Example L7 routing rules (Spring Cloud Gateway / Nginx):
  /api/v1/users/**   → user-service
  /api/v1/orders/**  → order-service
  /static/**         → CDN or object storage
  Host: admin.*      → admin-service (internal only)
```

```yaml
# Nginx upstream configuration
upstream app_servers {
    least_conn;                            # algorithm: least connections
    server app1.example.com:8080 weight=3; # higher weight = more traffic
    server app2.example.com:8080 weight=1;
    server app3.example.com:8080 backup;   # only used when others fail
    keepalive 64;                          # reuse connections to backends
}

server {
    listen 80;
    location /api/ {
        proxy_pass http://app_servers;
        proxy_connect_timeout 5s;
        proxy_read_timeout 30s;

        # Health check
        proxy_next_upstream error timeout http_500 http_502 http_503;
        proxy_next_upstream_tries 2;
    }
}
```

### 2.3 Health checks and failure handling

```
Active health checks: load balancer polls backend /health endpoints
  - Passive health check: mark server unhealthy after N consecutive failures
  - Active: periodic probes regardless of traffic

Circuit at LB level:
  Server healthy → in rotation
  3 consecutive failures → removed from pool
  Check every 10s → if healthy, return to pool after 2 successes

Health check endpoint best practices:
  /actuator/health: returns 200 if healthy, 503 if unhealthy
  Shallow: only check if the service process is alive
  Deep: check DB connectivity, cache connectivity, downstream dependencies
  → Use deep for readiness, shallow for liveness (Kubernetes pattern)
```

```java
// Spring Boot custom health indicator for load balancer
@Component
public class DatabaseHealthIndicator implements HealthIndicator {
    private final DataSource dataSource;

    @Override
    public Health health() {
        try (Connection conn = dataSource.getConnection()) {
            conn.createStatement().executeQuery("SELECT 1");
            return Health.up()
                .withDetail("database", "reachable")
                .withDetail("pool_active", getActiveConnections())
                .build();
        } catch (SQLException e) {
            return Health.down()
                .withDetail("error", e.getMessage())
                .build(); // triggers 503 → LB removes from pool
        }
    }
}
```

### 2.4 Global Load Balancing (GeoDNS / Anycast)

```
GeoDNS:
  DNS resolver returns different IP based on client location.
  Client in EU → eu-west datacenter IP
  Client in US → us-east datacenter IP

  Propagation delay: DNS TTL (typically 60-300 seconds) — slow failover
  Use: Route users to nearest datacenter, DR failover

Anycast:
  Same IP address announced from multiple datacenters via BGP.
  Router naturally selects nearest datacenter via shortest BGP path.
  Used by: Cloudflare, Google (8.8.8.8), AWS Global Accelerator
  Failover: seconds (BGP convergence) vs minutes (DNS TTL)
  Use: CDN edge, DDoS mitigation, ultra-low latency requirements
```

## 3. Caching Layers

Caching stores computed or fetched results so future requests can be served faster. It is the single highest-ROI scalability technique — a cache hit costs microseconds; a database query costs milliseconds to seconds.

### 3.1 Cache hierarchy

```
Layer             Location          Latency     Capacity    Cost
──────────────────────────────────────────────────────────────────
CPU L1 cache      On-chip           ~1 ns        ~32 KB      ████
CPU L3 cache      On-chip           ~10 ns       ~32 MB      ███
Application RAM   Process memory    ~100 ns      GBs         ██
Local cache       Same JVM          ~1 µs        bounded     ██
Distributed cache Remote server     ~1-5 ms      TBs         █
CDN edge          POP near user     ~10-50 ms    TBs         ░
Database          Remote server     ~10-100 ms   TBs         ░
Object storage    Remote cloud      ~100-500 ms  Unlimited   ░
```

### 3.2 CDN — Content Delivery Network

CDNs cache static and dynamic content at geographically distributed Points of Presence (PoPs), dramatically reducing latency for globally distributed users.

```
Without CDN:
  User in Tokyo → Origin server in Virginia → ~150ms round trip

With CDN:
  User in Tokyo → Tokyo PoP (cache hit) → ~10ms round trip
  User in Tokyo → Tokyo PoP (cache miss) → Origin → ~150ms + store in PoP

CDN caches:
  ✓ Static assets: images, JS bundles, CSS, fonts (long TTL, immutable)
  ✓ API responses with caching headers (short TTL, stale-while-revalidate)
  ✓ Video streaming (segment caching, range request support)
  ✗ Personalised responses (vary by user — cache by session key)
  ✗ POST/PUT/DELETE requests (side effects)
  ✗ Real-time data (price feeds, live scores)
```

```java
// Spring Boot: set caching headers for CDN
@RestController
public class ProductController {

    @GetMapping("/products/{id}")
    public ResponseEntity<Product> getProduct(@PathVariable Long id) {
        Product product = productService.findById(id);

        return ResponseEntity.ok()
            // Cache for 5 minutes publicly (CDN + browser)
            .cacheControl(CacheControl.maxAge(5, TimeUnit.MINUTES).cachePublic())
            // Serve stale content for 30s while revalidating in background
            .header("Cache-Control", "max-age=300, stale-while-revalidate=30")
            // ETag for conditional requests (304 Not Modified)
            .eTag(String.valueOf(product.getVersion()))
            .body(product);
    }

    @GetMapping("/users/{id}/profile")
    public ResponseEntity<UserProfile> getProfile(@PathVariable Long id) {
        return ResponseEntity.ok()
            // Private: browser can cache, CDN must not
            .cacheControl(CacheControl.maxAge(1, TimeUnit.MINUTES).cachePrivate())
            .body(userService.getProfile(id));
    }

    @GetMapping("/static/bundle.{hash}.js")
    public ResponseEntity<byte[]> getBundle(@PathVariable String hash) {
        return ResponseEntity.ok()
            // Immutable: content-addressed file never changes — cache forever
            .cacheControl(CacheControl.maxAge(365, TimeUnit.DAYS)
                .cachePublic().immutable())
            .body(assetService.getBundle(hash));
    }
}
```

### 3.3 Application-level cache

```java
// Spring Cache abstraction — works with Redis, Caffeine, EHCache
@Service
public class ProductService {

    // @Cacheable: return cached value if exists, else invoke method and cache result
    @Cacheable(value = "products", key = "#id",
               condition = "#id > 0",
               unless = "#result == null")
    public Product findById(Long id) {
        return productRepository.findById(id).orElse(null); // DB hit only on cache miss
    }

    // @CachePut: always invoke method AND update cache
    @CachePut(value = "products", key = "#result.id")
    public Product save(Product product) {
        return productRepository.save(product);
    }

    // @CacheEvict: remove from cache on update/delete
    @CacheEvict(value = "products", key = "#id")
    public void delete(Long id) {
        productRepository.deleteById(id);
    }

    // Evict entire cache namespace on bulk operation
    @CacheEvict(value = "products", allEntries = true)
    public void invalidateAll() {}
}

// application.yml — Caffeine (local) + Redis (distributed) dual cache
spring:
  cache:
    type: caffeine              # local tier (L1)
    caffeine:
      spec: maximumSize=1000,expireAfterWrite=60s

# For Redis (L2/distributed):
# spring.cache.type: redis
# spring.redis.host: redis-cluster
```

**Two-tier caching (L1 local + L2 Redis):**

```java
@Service
public class TwoTierCacheService {
    private final Cache<Long, Product> localCache = Caffeine.newBuilder()
        .maximumSize(1_000)
        .expireAfterWrite(30, TimeUnit.SECONDS)
        .recordStats()
        .build();

    private final RedisTemplate<String, Product> redis;
    private final ProductRepository repository;

    public Product get(Long id) {
        // L1: check local Caffeine cache (~1µs)
        Product cached = localCache.getIfPresent(id);
        if (cached != null) return cached;

        // L2: check Redis (~1-5ms)
        cached = redis.opsForValue().get("product:" + id);
        if (cached != null) {
            localCache.put(id, cached); // populate L1
            return cached;
        }

        // L3: database hit (~10-100ms)
        Product product = repository.findById(id).orElseThrow();
        redis.opsForValue().set("product:" + id, product, 5, TimeUnit.MINUTES);
        localCache.put(id, product);
        return product;
    }
}
```

### 3.4 Cache eviction policies

```
LRU (Least Recently Used):
  Evict item not accessed for longest time.
  Good for: temporal locality (recently accessed items likely accessed again)
  Weakness: one sequential scan evicts all hot items ("cache pollution")

LFU (Least Frequently Used):
  Evict item accessed fewest times.
  Good for: frequency-skewed workloads (hot items accessed millions of times)
  Weakness: recently popular items evicted if historically less accessed

LIRS / ARC (Adaptive Replacement Cache):
  Hybrid: adapts between LRU and LFU based on workload.
  Used by: ZFS, some database buffer managers.

TTL (Time To Live):
  Evict after fixed duration regardless of access pattern.
  Good for: data with known staleness bounds (pricing, inventory)
  Often combined with LRU: evict on TTL expiry OR when capacity exceeded.

Redis eviction policies:
  allkeys-lru:    LRU across all keys (most common for cache)
  volatile-lru:   LRU only among keys with TTL set
  allkeys-lfu:    LFU across all keys (Redis 4+, good for hot-key workloads)
  noeviction:     Return error when full (use for non-cache data)
```

### 3.5 Cache invalidation strategies

```
Cache-Aside (Lazy Loading):                    Application controls cache
  Read:  check cache → miss → load DB → write cache → return
  Write: update DB → invalidate cache (or update cache)
  Pro:   cache only holds what's needed; tolerates cache failure
  Con:   cache miss = 3 operations; thundering herd on cold start

Write-Through:                                 Cache updated synchronously with DB
  Write: update cache AND DB atomically
  Read:  check cache → always fresh → return
  Pro:   cache is always consistent with DB
  Con:   write latency increases; cache fills with data that may never be read

Write-Behind (Write-Back):                     DB updated asynchronously
  Write: update cache → return immediately → async write to DB
  Pro:   lowest write latency; batches DB writes
  Con:   data loss risk if cache fails before DB write; complex recovery

Read-Through:                                  Cache fetches from DB on miss
  Cache layer automatically loads from DB on miss (cache is the data source)
  Application only talks to cache, never directly to DB
  Used by: NCache, some ORM second-level caches

Refresh-Ahead:                                 Proactive cache warming
  Cache asynchronously refreshes entries before they expire.
  Pro:   no cache miss latency for hot data
  Con:   wastes resources refreshing data that might not be requested again
```

### 3.6 Cache pitfalls

```java
// Cache Stampede (Thundering Herd):
// Many requests arrive simultaneously for an expired or absent key.
// All miss the cache, all hit the database simultaneously → DB overload.

// Solution 1: Mutex/lock — only one request populates the cache
private final Map<String, Lock> locks = new ConcurrentHashMap<>();

public Product getWithLock(Long id) {
    String key = "product:" + id;
    Product cached = redis.opsForValue().get(key);
    if (cached != null) return cached;

    Lock lock = locks.computeIfAbsent(key, k -> new ReentrantLock());
    lock.lock();
    try {
        // Double-check after acquiring lock
        cached = redis.opsForValue().get(key);
        if (cached != null) return cached;

        Product product = repository.findById(id).orElseThrow();
        redis.opsForValue().set(key, product, 5, TimeUnit.MINUTES);
        return product;
    } finally {
        lock.unlock();
        locks.remove(key);
    }
}

// Solution 2: Probabilistic early expiration
// Re-compute cache before expiry with probability proportional to remaining TTL
// Prevents all processes from waiting for the same expiry moment

// Cache Penetration:
// Repeated queries for keys that don't exist → always miss cache → always hit DB.
// Solution: cache negative results ("null" sentinel with short TTL)
// Solution: Bloom filter — check before cache; definite NO means don't query DB

// Cache Avalanche:
// Many cache entries expire simultaneously → mass DB hits at the same time.
// Solution: randomise TTLs → cache.set(key, value, baseTTL + random(0, 60))
public void setWithJitter(String key, Object value, long baseTtlSeconds) {
    long jitter = ThreadLocalRandom.current().nextLong(0, 60);
    redis.opsForValue().set(key, value, baseTtlSeconds + jitter, TimeUnit.SECONDS);
}
```

## 4. Database Replication and Sharding

### 4.1 Replication

Replication maintains copies of a database on multiple servers to improve read throughput and availability.

```
Primary-Replica (Master-Slave):

┌──────────┐   Sync/Async   ┌──────────┐
│ Primary  │ ─── writes ──▶ │ Replica1 │  reads only
│ (writes) │                └──────────┘
└──────────┘
                             ┌──────────┐
         └── writes ───────▶ │ Replica2 │  reads only
                             └──────────┘

Writes:  go to primary only
Reads:   can go to any replica (eventual consistency window)
Failover: replica promoted to primary on primary failure

Synchronous replication:
  Primary waits for at least 1 replica to confirm write before ACK-ing client.
  Pro: zero data loss on primary failure (RPO = 0)
  Con: write latency increases by replica round-trip (~1-5ms same DC)
  PostgreSQL: synchronous_standby_names

Asynchronous replication (default):
  Primary ACK-s client immediately; replicas catch up in background.
  Pro: no write latency overhead
  Con: replica lag (typically ms-seconds); potential data loss on primary failure
  Replication lag: how far behind replica is from primary
```

```java
// Spring — route reads to replica, writes to primary
@Configuration
public class DataSourceConfig {
    @Bean @Primary
    public DataSource primaryDataSource() {
        return DataSourceBuilder.create()
            .url("jdbc:postgresql://primary-db:5432/mydb").build();
    }

    @Bean("replicaDataSource")
    public DataSource replicaDataSource() {
        return DataSourceBuilder.create()
            .url("jdbc:postgresql://replica-db:5432/mydb").build();
    }
}

@Service
public class ProductService {
    private final ProductRepository primaryRepo;  // writes
    private final ProductRepository replicaRepo;  // reads

    @Transactional(readOnly = true)    // Spring routes to replica datasource
    public Product findById(Long id) {
        return replicaRepo.findById(id).orElseThrow();
    }

    @Transactional                     // routes to primary
    public Product save(Product p) {
        return primaryRepo.save(p);
    }
}
```

**Read-Your-Writes consistency problem:**

```
User writes a profile update → goes to primary.
User immediately reads their profile → goes to replica.
Replica lag = 200ms → user sees old data → confusing!

Solutions:
1. Read from primary after write (for the same user's requests, briefly)
2. Sticky routing: route user to primary for 1-2 seconds after write
3. Version token: client holds a "min replica version" and waits for replica to catch up
4. Monotonic read: user always reads from same replica (consistent order)
```

### 4.2 Sharding (Horizontal Partitioning)

Sharding splits a dataset across multiple database servers (shards), each holding a subset of the data. Required when a single server (even the largest available) can no longer hold or serve the full dataset.

```
Without sharding (10B rows, 1 server):
  ┌────────────────────────────┐
  │ users 1 → 10,000,000,000  │  → too slow, too large
  └────────────────────────────┘

With range sharding (3 shards):
  ┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐
  │ Shard 1          │  │ Shard 2          │  │ Shard 3          │
  │ users 1 → 3.3B   │  │ users 3.3B → 6.6B│  │ users 6.6B → 10B │
  └──────────────────┘  └──────────────────┘  └──────────────────┘
```

**Sharding strategies:**

```
Range Sharding:
  Partition by range of shard key (user_id 1-1M → shard1, 1M-2M → shard2...)
  Pro: Range queries are efficient — only one shard queried
  Con: Hot spots if data is unevenly distributed or has recency bias
       (new users all go to last shard; time-series data always hits one shard)

Hash Sharding:
  shard = hash(shard_key) % num_shards
  Pro: Uniform distribution — no hot spots
  Con: Range queries require scatter-gather (query all shards)
       Resharding (adding shards) requires remapping most data

Consistent Hashing:
  Place shards and keys on a virtual ring; key routes to nearest shard.
  Pro: Adding/removing shards remaps only ~1/n of keys
  Con: Can still have hot spots; needs virtual nodes for balance
  Used by: Cassandra, DynamoDB, Redis Cluster

Directory-Based Sharding:
  Central lookup table: key → shard mapping.
  Pro: Flexible remapping, any distribution strategy
  Con: Lookup table is a bottleneck and single point of failure

Geo Sharding:
  Users in EU → EU shard, Users in US → US shard.
  Pro: Data locality for compliance (GDPR) and latency
  Con: Cross-region queries expensive; requires careful key design
```

**Shard key selection — the most critical decision:**

```
Good shard key properties:
  High cardinality:  enough unique values to distribute evenly
  Even distribution: avoid hot shards (celebrity user problem)
  Query alignment:   most queries filter on the shard key
                     (cross-shard queries = expensive scatter-gather)
  Immutable:         changing shard key = moving data = expensive

Examples:
  user_id:    ✓ high cardinality, ✓ most queries by user
  tenant_id:  ✓ for multi-tenant SaaS (all user data in same shard = no cross-shard joins)
  timestamp:  ✗ time-series hot spot (all writes to latest shard)
  country:    ✗ low cardinality (uneven: US >> Liechtenstein)

Compound key: user_id + timestamp — distribute user time-series evenly
```

**Sharding challenges:**

```
Cross-shard JOINs:
  SELECT u.name, o.total FROM users u JOIN orders o ON u.id = o.user_id
  If users and orders are on different shards → requires application-level join
  → Denormalise or co-locate related data on same shard (tenant_id pattern)

Distributed transactions:
  UPDATE accounts SET balance = balance - 100 WHERE id = 1 (shard 1)
  UPDATE accounts SET balance = balance + 100 WHERE id = 2 (shard 2)
  → Requires 2-Phase Commit or Saga pattern (see Distributed Transactions guide)

Resharding:
  Adding a new shard requires moving data. Downtime or complex live migration.
  → Use consistent hashing to minimise disruption
  → Pre-shard: create more shards than needed initially (MongoDB chunks)

Global sequences:
  Auto-increment IDs don't work across shards (collisions).
  → Use UUID, Snowflake ID, or shard-prefixed IDs:
     Snowflake: [41-bit timestamp | 10-bit machine_id | 12-bit sequence] = 64-bit unique ID
```

## 5. Stateless vs Stateful Services

### 5.1 Stateless services

A stateless service holds no mutable data between requests. Every request carries all the information needed to process it, and any server can handle any request.

```java
// Stateless REST API — any instance can handle any request
@RestController
@RequiredArgsConstructor
public class OrderController {
    private final OrderService orderService;
    private final JwtService jwtService;

    @GetMapping("/orders/{id}")
    public Order getOrder(
            @PathVariable Long id,
            @RequestHeader("Authorization") String token) {
        // All state is in the request:
        //   - 'id' identifies the resource
        //   - JWT token carries authenticated user identity
        //   - 'orderService' queries external DB (shared state, not local)
        Long userId = jwtService.extractUserId(token);
        return orderService.findByIdForUser(id, userId);
    }
    // No instance variables hold user data → fully horizontally scalable
}
```

**Making a stateful service stateless:**

```
State that must be externalised:

HTTP Sessions        → Redis session store
  HttpSession        → Spring Session + Redis
  JSESSIONID cookie  → sessionId → Redis lookup

In-memory counters   → Redis INCR / atomic counters
User shopping carts  → Redis hash per user
WebSocket state      → Redis pub/sub + connection registry
Circuit breaker state→ Shared (Resilience4j with Redis or in-LB)
Rate limit counters  → Redis sliding window
Distributed locks    → Redis Redlock or ZooKeeper
```

### 5.2 Stateful services

Some services inherently maintain state: databases, message brokers, coordination services, and long-running computations.

```
Stateful service characteristics:
  - Maintains data across requests (sessions, connections, ownership)
  - Not all instances are equivalent (primary vs replica, leader vs follower)
  - Instance identity matters — specific data lives on specific nodes
  - Kubernetes StatefulSet instead of Deployment (stable pod identity)

Examples of necessary statefulness:
  PostgreSQL primary   — holds the authoritative write log
  Kafka partition      — specific partition assigned to specific broker
  Zookeeper leader     — leader election state
  WebSocket server     — connection is tied to specific server process

Kubernetes StatefulSet for stateful workloads:
  - Stable pod names: kafka-0, kafka-1, kafka-2 (not random hashes)
  - Stable network identity: kafka-0.kafka-headless.namespace.svc.cluster.local
  - Ordered startup/shutdown (kafka-0 starts first)
  - Persistent volume claims survive pod restarts
```

```yaml

kind: StatefulSet

    - metadata:
```

## 6. Asynchronous Processing

Synchronous processing blocks the caller until work completes. Asynchronous processing decouples the caller from the worker — the caller returns immediately and the work happens later, in parallel, or in the background.

### 6.1 Why asynchronous processing matters for scale

```
Synchronous request lifecycle (blocking):
  Client → HTTP request → Server process:
    1. Validate input (1ms)
    2. Query database (20ms) ← blocked
    3. Call external payment API (500ms) ← blocked
    4. Send confirmation email (200ms) ← blocked
    5. Return response
  Total: 721ms — thread blocked entire time

With thread pool of 200 threads: max concurrent requests = 200
At 721ms average: throughput = 200 / 0.721 = ~277 RPS

Asynchronous (offload non-critical work):
  Client → HTTP request → Server process:
    1. Validate input (1ms)
    2. Queue payment job (2ms) ← immediate
    3. Return 202 Accepted (3ms total)
  Background worker (separate service):
    4. Process payment (500ms)
    5. Send email (200ms)
    6. Notify client via webhook or polling

With 200 threads at 3ms average: throughput = 200 / 0.003 = ~66,000 RPS
```

### 6.2 Message queues

```
Producer → [Queue / Topic] → Consumer(s)

Guarantees vary by system:
  At-most-once:  message may be lost, never duplicated
  At-least-once: message always delivered, may be duplicated → idempotent consumers
  Exactly-once:  delivered exactly once (Kafka transactions, expensive)

RabbitMQ (queue semantics):
  Message goes to one consumer (work queue / competing consumers)
  Use: task distribution, RPC, job queues
  Ordering: per-queue FIFO (not across queues)
  Retention: discarded after ACK

Kafka (log semantics):
  Message retained for configurable time (days/weeks)
  Multiple consumer groups each get all messages
  Partitions provide parallelism within a topic
  Use: event streaming, audit log, fan-out to multiple consumers, replay
```

```java
// Spring Boot + Kafka: async order processing
// Producer: HTTP request handler returns immediately
@RestController
@RequiredArgsConstructor
public class OrderController {
    private final KafkaTemplate<String, OrderEvent> kafka;
    private final OrderRepository orderRepo;

    @PostMapping("/orders")
    public ResponseEntity<OrderResponse> placeOrder(@RequestBody OrderRequest req) {
        // Save initial state to DB
        Order order = orderRepo.save(Order.builder()
            .userId(req.getUserId())
            .status(OrderStatus.PENDING)
            .items(req.getItems())
            .build());

        // Publish event — returns immediately
        kafka.send("order.placed", order.getId().toString(),
            new OrderEvent(order.getId(), req));

        // Return 202 Accepted with polling URL
        return ResponseEntity.accepted()
            .header("Location", "/orders/" + order.getId())
            .body(new OrderResponse(order.getId(), OrderStatus.PENDING));
    }
}

// Consumer: runs in separate thread pool, processes asynchronously
@Component
@RequiredArgsConstructor
public class OrderProcessor {
    private final PaymentService paymentService;
    private final InventoryService inventoryService;
    private final EmailService emailService;
    private final OrderRepository orderRepo;

    @KafkaListener(topics = "order.placed", groupId = "order-processor",
                   concurrency = "5") // 5 threads processing in parallel
    public void processOrder(OrderEvent event,
                              Acknowledgment ack) {
        try {
            Order order = orderRepo.findById(event.getOrderId()).orElseThrow();

            inventoryService.reserve(order.getItems());
            paymentService.charge(order);
            emailService.sendConfirmation(order);

            order.setStatus(OrderStatus.CONFIRMED);
            orderRepo.save(order);

            ack.acknowledge(); // commit offset — message processed
        } catch (Exception e) {
            // Don't ACK — Kafka will redeliver (at-least-once)
            // Handle idempotency: check if order already processed
            if (isRetryable(e)) throw e; // retry
            else sendToDLQ(event);       // dead letter queue
        }
    }
}
```

### 6.3 Async patterns in Spring

```java
// @Async — run method in background thread pool
@Service
public class EmailService {

    @Async("emailThreadPool")           // uses custom executor
    @Retryable(maxAttempts = 3, backoff = @Backoff(delay = 1000))
    public CompletableFuture<Void> sendWelcomeEmail(User user) {
        externalEmailProvider.send(user.getEmail(), "Welcome!");
        return CompletableFuture.completedFuture(null);
    }
}

@Configuration
@EnableAsync
public class AsyncConfig {
    @Bean("emailThreadPool")
    public Executor emailExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(10);
        executor.setMaxPoolSize(50);
        executor.setQueueCapacity(500);
        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
        executor.initialize();
        return executor;
    }
}

// CompletableFuture for parallel execution — fan-out then join
@Service
public class DashboardService {
    public DashboardData getDashboard(Long userId) {
        // Fire all three in parallel
        CompletableFuture<Orders> orders    = CompletableFuture.supplyAsync(
            () -> orderService.getRecent(userId));
        CompletableFuture<Balance> balance  = CompletableFuture.supplyAsync(
            () -> paymentService.getBalance(userId));
        CompletableFuture<Messages> messages = CompletableFuture.supplyAsync(
            () -> messageService.getUnread(userId));

        // Join results — wait for all
        return CompletableFuture.allOf(orders, balance, messages)
            .thenApply(v -> new DashboardData(
                orders.join(), balance.join(), messages.join()))
            .join();
        // Total time = max(orders, balance, messages) instead of sum
    }
}
```

### 6.4 Event-driven architecture

```java
// Domain events: services communicate via events, not direct calls
// Decouples producers from consumers — neither knows about the other

// Publisher (Order Service) — publishes domain events
@Service
@RequiredArgsConstructor
public class OrderService {
    private final ApplicationEventPublisher eventPublisher;  // Spring local events
    private final KafkaTemplate<String, Object> kafka;       // distributed events

    @Transactional
    public Order confirmOrder(Long orderId) {
        Order order = findAndConfirm(orderId);

        // Local event (synchronous, same JVM — for transactional consistency)
        eventPublisher.publishEvent(new OrderConfirmedEvent(order));

        // Distributed event (async, fan-out to all interested services)
        kafka.send("order.confirmed", orderId.toString(),
            new OrderConfirmedMessage(order));

        return order;
    }
}

// Listener 1: Inventory Service — listen to order events
@Component
public class InventoryEventHandler {
    @KafkaListener(topics = "order.confirmed", groupId = "inventory-service")
    public void onOrderConfirmed(OrderConfirmedMessage event) {
        inventoryService.deductStock(event.getItems());
    }
}

// Listener 2: Notification Service — listen to same event, different consumer group
@Component
public class NotificationEventHandler {
    @KafkaListener(topics = "order.confirmed", groupId = "notification-service")
    public void onOrderConfirmed(OrderConfirmedMessage event) {
        emailService.sendConfirmation(event.getUserId(), event.getOrderId());
    }
}
// Both groups receive EVERY message — Kafka fan-out
```

## 7. Putting It All Together — Scalability Checklist

```
When a system needs to scale, walk through each layer:

Traffic spikes:
  □ Load balancer in front of stateless application servers
  □ Auto-scaling group (AWS ASG, K8s HPA) scales on CPU / RPS
  □ CDN absorbs static traffic and offloads origin
  □ Rate limiting at LB / API gateway prevents overload

Read throughput:
  □ Read replicas for database read scaling
  □ Application cache (Redis) for hot data
  □ CDN for public API responses
  □ Query optimisation, indexes before scaling hardware

Write throughput:
  □ Async processing via Kafka/RabbitMQ decouples write path
  □ Database write throughput: vertical scaling first
  □ Sharding when vertical limit reached (rare)
  □ Event sourcing / CQRS for write-heavy with complex query patterns

Latency:
  □ CDN for geographic distribution
  □ Cache hit rate — reduce DB round trips
  □ Async for non-critical path operations
  □ Connection pooling (HikariCP) — no connection overhead per request
  □ Efficient queries — no N+1, proper indexing

Availability:
  □ No single point of failure at any layer
  □ Health checks + automatic removal of unhealthy nodes
  □ Circuit breakers prevent cascade failures
  □ Multi-AZ / multi-region deployment for geographic redundancy
  □ Graceful degradation: serve stale cache on DB failure
```

## 8. Interview Questions

**Q: When would you choose horizontal scaling over vertical scaling for a database?**

A: Vertical scaling should always be attempted first for databases because it avoids the massive complexity of distributed data: no cross-shard queries, no distributed transactions, no resharding operations. A modern server with 64 cores, 1 TB RAM, and NVMe SSDs can handle most applications' lifetimes. Horizontal scaling (sharding) becomes necessary when the dataset exceeds what a single server can hold (storage limit), when write throughput exceeds a single server's capacity (even on the largest available hardware), when network bandwidth to a single server is saturated, or when geographic data locality is required for compliance (GDPR data residency). Read replicas solve read scaling without sharding complexity and should be exhausted before sharding.

---

**Q: What is the thundering herd problem in caching, and how do you solve it?**

A: The thundering herd occurs when a popular cache entry expires and many concurrent requests all experience a cache miss simultaneously, all query the database at once, and overwhelm it. It's most severe for very hot keys — if a cached product page viewed 10,000 times per second expires, all 10,000 in-flight requests hit the database at once. Solutions include mutex-based cache population (only the first miss triggers a database query; others wait), probabilistic early expiration (start refreshing before the TTL expires, with probability inversely proportional to remaining TTL), and stale-while-revalidate (serve the stale cached value while a background refresh runs, so users never experience a miss). For extremely hot keys, a distributed lock (Redis SETNX) can gate cache population to a single process.

---

**Q: What are the tradeoffs between different sharding strategies, and how do you choose a shard key?**

A: Range sharding is simple and supports efficient range queries but creates hot spots when data has a natural ordering bias (recent data, trending users). Hash sharding distributes load uniformly but makes range queries expensive (scatter-gather across all shards). Consistent hashing minimises data movement when adding shards but still requires virtual nodes for balance. The shard key choice is more important than the strategy. A good key has high cardinality (many unique values), even distribution (no celebrities — a user who generates 1000x normal traffic shouldn't live on one shard), and query alignment (most queries filter by the shard key, avoiding cross-shard joins). The entity that owns the primary access pattern should be the shard key — for a multi-tenant SaaS, `tenant_id` is often optimal because all tenant data co-locates on one shard, eliminating cross-shard joins entirely.

---

**Q: How do you ensure statelessness in a service that needs to track user sessions?**

A: Statelessness means a server's local memory contains no user-specific data that would make one server non-interchangeable with another. For session tracking, this requires externalising session state to a shared store like Redis. With Spring Session and Redis, the `HttpSession` API is transparently backed by Redis — the session data is serialised and stored under a UUID key in Redis, and the client receives only the UUID as a cookie. Any server can reconstruct the full session from Redis on the next request. JWT tokens go further — they eliminate the server-side session entirely by embedding claims (user ID, roles, expiry) in the token itself, signed with the server's private key. The server validates the signature without any database or cache lookup, making the service truly stateless with zero shared state for authentication.

---

**Q: What is the difference between a message queue and an event stream, and when would you use each?**

A: A message queue (RabbitMQ, SQS) is designed for work distribution — each message is consumed by exactly one consumer and deleted after acknowledgement. It's appropriate for task queues: sending an email, processing a payment, resizing an image. An event stream (Kafka) is a durable, ordered log where messages are retained for a configurable period and multiple independent consumer groups each receive the full stream. It's appropriate for event-driven architectures where multiple downstream services need to react to the same event (order placed → inventory reserves, warehouse picks, email sends, analytics records), for event sourcing (the log is the source of truth), for replayability (re-process historical events for a new consumer), and for audit trails. Use queues when you want point-to-point delivery with clear ownership; use streams when you want publish-subscribe fan-out, temporal decoupling, and the ability to replay or process the same event with different logic in the future.

## 9. Key Takeaways

- **Scale vertically first**: databases especially — a 64-core 1TB-RAM server is simpler than sharding; shard only when vertical limits are reached
- **Horizontal scaling requires stateless application servers**: externalise sessions to Redis, use JWTs, never store user state in JVM heap
- **Load balancer algorithm matters**: least-connections for variable request cost; consistent hashing for cache clusters; power-of-two-choices for general-purpose APIs
- **Caching ROI order**: in-process Caffeine (µs) → Redis (ms) → CDN (network round trip) — hit the fastest tier first
- **Cache pitfalls**: stampede (mutex/probabilistic TTL), penetration (Bloom filter/negative cache), avalanche (TTL jitter)
- **Replication before sharding for reads**: read replicas solve read scaling without the nightmare of cross-shard queries and distributed transactions
- **Shard key is permanent and critical**: choose based on query patterns, cardinality, and distribution — changing it later requires a costly data migration
- **Async unlocks write throughput**: decouple non-critical path work (emails, notifications, analytics) from the request-response cycle using Kafka/RabbitMQ; return 202 Accepted immediately
- **Stateful services need special treatment**: Kubernetes StatefulSet, stable identity, persistent volumes — not Deployments
