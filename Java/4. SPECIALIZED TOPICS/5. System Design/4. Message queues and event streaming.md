# Message Queues and Event Streaming

## Overview

Message queues and event streaming systems decouple producers from consumers, transforming synchronous request-response architectures into asynchronous, resilient pipelines. A producer publishes a message without knowing when or by whom it will be processed; a consumer processes messages at its own pace without blocking the producer. This decoupling is the foundation of scalable, fault-tolerant distributed systems.

**Core distinction — queue vs stream:**

```
Message Queue (RabbitMQ, SQS):          Event Stream (Kafka):
  Message → consumed once → deleted       Event → retained in log → replayable
  Work distribution (one consumer wins)   Fan-out (multiple consumers, all receive)
  Short-lived messages (seconds-minutes)  Long-lived log (days-forever)
  Task execution semantics                Event history / audit log semantics

Choose queue when:  task needs to happen exactly once by one worker
Choose stream when: multiple systems react to the same event, or you need replay
```

**When async messaging solves real problems:**

```
Without messaging (synchronous):
  Order API → calls Inventory → calls Payment → calls Email → calls Analytics
  Latency: sum of all services (~2 seconds)
  Failure: any service down → order fails

With messaging (asynchronous):
  Order API → saves order → publishes OrderPlaced event → returns 202
  Inventory, Payment, Email, Analytics each consume independently
  Latency: time to publish only (~5ms)
  Failure: any consumer down → retries when it recovers; order already saved
```

## 1. Kafka vs RabbitMQ vs SQS

### 1.1 Apache Kafka

Kafka is a distributed, append-only log. Producers write records to topics; consumers read from topics by tracking their position (offset) in the log. Records are retained regardless of consumption, enabling replay and multiple independent consumer groups.

```
Kafka architecture:

  Producers                Topics (partitioned)              Consumers
  ┌─────────┐             ┌────────────────────┐            ┌────────────┐
  │ Order   │──┐          │ orders (3 parts.)  │    ┌──────▶│ Inventory  │
  │ Service │  │          │ Part 0: [0,1,2,3]  │    │       │ Consumer   │
  └─────────┘  │          │ Part 1: [0,1,2]    │◀───┤       └────────────┘
               ├─ write ─▶│ Part 2: [0,1,2,3,4]│    │       ┌────────────┐
  ┌─────────┐  │          └────────────────────┘    └──────▶│ Analytics  │
  │ Payment │──┘             Each partition on           CG2 │ Consumer   │
  │ Service │             a different broker                  └────────────┘
  └─────────┘

Key concepts:
  Topic:         named log — logically a category of events
  Partition:     ordered, immutable sequence within a topic — unit of parallelism
  Offset:        position of a record within a partition (0-based, monotonically increasing)
  Consumer Group: set of consumers sharing partition assignment — each partition
                  consumed by exactly one member of the group at a time
  Broker:        Kafka server — stores partitions, serves producers and consumers
  Replication:   each partition has N replicas across brokers for HA
  Retention:     records kept for configured time (days) or size limit regardless of consumption
```

```java
// Spring Kafka producer
@Service
@RequiredArgsConstructor
public class OrderEventPublisher {
    private final KafkaTemplate<String, OrderEvent> kafka;

    public void publishOrderPlaced(Order order) {
        // Key = orderId → ensures all events for same order go to same partition
        // (ordering guarantee within a partition)
        kafka.send("orders.placed",
                   order.getId().toString(),  // partition key
                   new OrderEvent(order))
             .addCallback(
                 result -> log.info("Published to partition {} offset {}",
                     result.getRecordMetadata().partition(),
                     result.getRecordMetadata().offset()),
                 ex -> log.error("Failed to publish order {}", order.getId(), ex)
             );
    }

    // Transactional producer — all or nothing across multiple sends
    @Transactional("kafkaTransactionManager")
    public void publishOrderAndPayment(Order order, Payment payment) {
        kafka.send("orders.placed",   order.getId().toString(),   toEvent(order));
        kafka.send("payments.created", payment.getId().toString(), toEvent(payment));
        // Both committed atomically, or neither — Kafka transactions
    }
}

// Spring Kafka consumer
@Component
@RequiredArgsConstructor
public class InventoryConsumer {

    @KafkaListener(
        topics = "orders.placed",
        groupId = "inventory-service",
        concurrency = "3"           // 3 consumer threads = up to 3 partitions in parallel
    )
    public void onOrderPlaced(
            @Payload OrderEvent event,
            @Header(KafkaHeaders.RECEIVED_PARTITION) int partition,
            @Header(KafkaHeaders.OFFSET) long offset,
            Acknowledgment ack) {
        try {
            inventoryService.reserve(event.getItems());
            ack.acknowledge(); // manually commit offset — at-least-once
        } catch (RetryableException e) {
            throw e; // don't ACK — Kafka will redeliver
        } catch (NonRetryableException e) {
            sendToDeadLetterTopic(event); // give up, prevent blocking
            ack.acknowledge();           // ACK to move past the poison pill
        }
    }
}
```

**Kafka configuration for production:**

```yaml
spring:
  kafka:
    bootstrap-servers: kafka-1:9092,kafka-2:9092,kafka-3:9092
    producer:
      acks: all              # wait for all ISR replicas to confirm write
      retries: 3             # retry on transient failure
      batch-size: 16384      # batch multiple records per request (throughput)
      linger-ms: 5           # wait up to 5ms to fill batch before sending
      compression-type: snappy  # compress batches (reduces network I/O ~4x)
      enable-idempotence: true  # exactly-once per partition (deduplicates retries)
    consumer:
      auto-offset-reset: earliest   # start from beginning if no committed offset
      enable-auto-commit: false     # manual offset commit (at-least-once control)
      max-poll-records: 500         # batch size per poll
      isolation-level: read_committed  # only read committed transactional messages
```

### 1.2 RabbitMQ

RabbitMQ is a message broker implementing AMQP (Advanced Message Queuing Protocol). Producers publish to **exchanges**; exchanges route messages to **queues** based on routing rules; consumers subscribe to queues.

```
RabbitMQ routing model:

  Producer                Exchange              Queue          Consumer
  ┌────────┐   publish   ┌──────────┐  route   ┌──────────┐  subscribe
  │        │────────────▶│  Direct  │─────────▶│ queue-A  │──────────▶│ Worker 1│
  │        │             └──────────┘          └──────────┘           └─────────┘
  │        │   publish   ┌──────────┐  route   ┌──────────┐
  │        │────────────▶│  Topic   │──────────│ queue-B  │──────────▶│ Worker 2│
  │        │             │ (*.order)│  ┌───────│ queue-C  │           └─────────┘
  └────────┘             └──────────┘  │       └──────────┘
                                       │
                         ┌──────────┐  │
                         │ Fanout   │──┘  → all bound queues receive copy
                         └──────────┘

Exchange types:
  Direct:  route by exact routing key match (orders.created → queue-orders)
  Topic:   route by pattern (#.order, order.* — wildcards)
  Fanout:  broadcast to ALL bound queues (copy to every subscriber)
  Headers: route by message header attributes

Acknowledgement model:
  Consumer receives message → processes → sends ACK
  If consumer dies before ACK → RabbitMQ requeues message (at-least-once)
  NACK with requeue=false → message sent to DLX (dead letter exchange)
```

```java
// Spring AMQP configuration
@Configuration
public class RabbitMQConfig {
    public static final String ORDER_EXCHANGE  = "orders.exchange";
    public static final String ORDER_QUEUE     = "orders.processing";
    public static final String DLX             = "orders.dlx";
    public static final String DLQ             = "orders.dlq";

    @Bean TopicExchange orderExchange() {
        return new TopicExchange(ORDER_EXCHANGE, true, false);
    }

    @Bean Queue orderQueue() {
        return QueueBuilder.durable(ORDER_QUEUE)
            .withArgument("x-dead-letter-exchange", DLX)    // failed → DLX
            .withArgument("x-message-ttl", 300_000)          // 5-min TTL
            .withArgument("x-max-length", 100_000)           // max 100k messages
            .build();
    }

    @Bean Binding orderBinding() {
        return BindingBuilder.bind(orderQueue())
            .to(orderExchange())
            .with("orders.#");  // topic pattern: any routing key starting with "orders."
    }

    // Dead letter infrastructure
    @Bean DirectExchange dlx()  { return new DirectExchange(DLX); }
    @Bean Queue          dlq()  { return QueueBuilder.durable(DLQ).build(); }
    @Bean Binding        dlqBinding() {
        return BindingBuilder.bind(dlq()).to(dlx()).with(ORDER_QUEUE);
    }
}

// Producer
@Service
@RequiredArgsConstructor
public class OrderPublisher {
    private final RabbitTemplate rabbit;

    public void publishOrderCreated(Order order) {
        rabbit.convertAndSend(
            RabbitMQConfig.ORDER_EXCHANGE,
            "orders.created",            // routing key
            order,
            msg -> {
                msg.getMessageProperties().setMessageId(order.getId().toString());
                msg.getMessageProperties().setDeliveryMode(MessageDeliveryMode.PERSISTENT);
                return msg;
            });
    }
}

// Consumer
@Component
@RequiredArgsConstructor
public class OrderConsumer {
    @RabbitListener(queues = RabbitMQConfig.ORDER_QUEUE)
    public void processOrder(Order order, Channel channel,
                              @Header(AmqpHeaders.DELIVERY_TAG) long tag)
            throws IOException {
        try {
            orderService.process(order);
            channel.basicAck(tag, false);  // ACK: remove from queue
        } catch (RetryableException e) {
            channel.basicNack(tag, false, true);  // NACK + requeue
        } catch (Exception e) {
            channel.basicNack(tag, false, false); // NACK no requeue → DLQ
        }
    }
}
```

### 1.3 Amazon SQS

SQS is a fully managed queue service with two modes: Standard (best-effort ordering, at-least-once) and FIFO (strict ordering, exactly-once within a group).

```
SQS Standard Queue:
  ✓ Nearly unlimited throughput
  ✓ At-least-once delivery (rare duplicates)
  ✗ Best-effort ordering (not guaranteed)
  Use: work queues where order doesn't matter, task processing

SQS FIFO Queue:
  ✓ Exactly-once processing (deduplication)
  ✓ FIFO ordering within a Message Group ID
  ✗ 300 TPS (3,000 with batching) per queue
  Use: ordered processing (financial transactions, ordered workflows)

Key SQS concepts:
  Visibility Timeout:  message invisible to other consumers while being processed
                       Default 30s; extend if processing takes longer
  Message Retention:   1 min to 14 days (default 4 days)
  Receive Count:       how many times a message has been received
  Dead Letter Queue:   messages that exceed maxReceiveCount go to DLQ
```

```java
// Spring Cloud AWS SQS consumer
@SqsListener(value = "orders-queue", deletionPolicy = SqsMessageDeletionPolicy.ON_SUCCESS)
public void processOrder(@Payload Order order,
                          @Headers Map<String, String> headers) {
    // ON_SUCCESS: message deleted only if method completes without exception
    // ON_FAILURE: message deleted even on exception (fire-and-forget)
    // NEVER: manual deletion control
    orderService.process(order);
    // Automatic deletion on success; on exception → visibility timeout expires → requeue
}

// SQS producer
@Service
@RequiredArgsConstructor
public class SQSOrderPublisher {
    private final QueueMessagingTemplate queueMessagingTemplate;
    private final SqsAsyncClient sqsClient;

    public void publish(Order order) {
        queueMessagingTemplate.convertAndSend("orders-queue", order);
    }

    // FIFO queue with deduplication
    public void publishFifo(Order order) {
        sqsClient.sendMessage(req -> req
            .queueUrl("https://sqs.us-east-1.amazonaws.com/123/orders.fifo")
            .messageBody(toJson(order))
            .messageGroupId(order.getCustomerId().toString()) // ordering group
            .messageDeduplicationId(order.getId().toString()) // idempotency key
        );
    }
}
```

### 1.4 Kafka vs RabbitMQ vs SQS comparison

|Dimension|Kafka|RabbitMQ|SQS|
|---|---|---|---|
|Model|Distributed log|Message broker (AMQP)|Managed queue|
|Retention|Configurable (days-forever)|Until consumed (or TTL)|Up to 14 days|
|Replay|✓ (seek to any offset)|✗ (consumed = gone)|✗|
|Ordering|Per-partition FIFO|Per-queue FIFO|FIFO queue only|
|Throughput|Very high (millions/sec)|High (100k+/sec)|High (Standard: unlimited)|
|Fan-out|Consumer groups (built-in)|Fanout exchange|SNS fan-out pattern|
|Delivery|At-least-once (exactly-once w/ transactions)|At-least-once|At-least-once (FIFO: exactly-once)|
|Routing|Topic + partition key|Exchange type + routing key|Queue name only|
|Operations|Complex (ZooKeeper/KRaft, brokers)|Moderate|None (fully managed)|
|Best for|Event streaming, audit log, replay|Complex routing, RPC, work queues|Simple work queues, AWS-native|

## 2. At-Least-Once vs Exactly-Once

### 2.1 Delivery semantics

```
At-Most-Once:   Message may be lost, never duplicated.
                Producer fires and forgets (no retry on failure).
                Consumer processes before ACKing (message deleted on receive, not ACK).
                Use: analytics, metrics, telemetry — occasional loss acceptable.

At-Least-Once:  Message always delivered, may be duplicated.
                Producer retries on failure (may send twice if first ACK is lost).
                Consumer ACKs after processing (redelivered if consumer crashes mid-process).
                Use: most business events — design consumers to be idempotent.

Exactly-Once:   Message delivered and processed exactly once. Never lost, never duplicated.
                Hardest guarantee — requires coordination between producer, broker, consumer.
                In Kafka: idempotent producer + transactional producer + read_committed.
                In SQS FIFO: MessageDeduplicationId.
                Use: financial transactions, inventory debits — where duplicates = real money.
```

### 2.2 Why exactly-once is hard

```
The "ACK lost" scenario creates unavoidable duplication with at-least-once:

  Producer → sends message → Broker receives and stores
  Broker → sends ACK → ACK lost in network
  Producer → timeout → retries → sends DUPLICATE message
  Broker → stores duplicate

  Consumer → processes message → sends ACK → consumer crashes BEFORE ACK arrives
  Broker → resends message (thinks consumer died)
  Consumer → processes DUPLICATE on restart

Exactly-once requires eliminating both scenarios:
  1. Idempotent producer: broker assigns producer ID + sequence number; 
     deduplicates retries at broker level (Kafka: enable.idempotence=true)
  2. Transactional reads: consumer only sees committed messages 
     (Kafka: isolation.level=read_committed)
  3. Transactional writes: if consumer also writes to Kafka, uses Kafka transactions
     to atomically mark offset as consumed AND publish result
```

### 2.3 Exactly-once in Kafka

```java
// Kafka Streams exactly-once: consume → process → produce atomically
@Configuration
public class ExactlyOnceConfig {
    @Bean
    public KafkaTransactionManager<String, Object> kafkaTransactionManager(
            ProducerFactory<String, Object> pf) {
        return new KafkaTransactionManager<>(pf);
    }
}

// Consume from "orders" and produce to "invoices" exactly once
// Uses Kafka transactions: offset commit + produce are atomic
@KafkaListener(topics = "orders.placed", groupId = "invoice-generator")
@Transactional("kafkaTransactionManager")  // Kafka transaction (not DB)
public void generateInvoice(OrderEvent event) {
    Invoice invoice = invoiceService.create(event);
    // Both operations committed atomically:
    kafkaTemplate.send("invoices.created", invoice.getId().toString(), invoice);
    // Offset committed when transaction commits — if this method throws,
    // offset NOT committed → message redelivered, no duplicate invoice sent
}

// Producer side: idempotent + transactional config
spring:
  kafka:
    producer:
      enable-idempotence: true          # deduplicates retries within a session
      transaction-id-prefix: invoice-   # enables Kafka transactions
      acks: all
    consumer:
      isolation-level: read_committed   # only read committed transactional messages
```

### 2.4 Idempotent consumers — the practical alternative

For most systems, exactly-once delivery is overkill. Instead, design **idempotent consumers**: processing the same message twice produces the same result as processing it once.

```java
@Component
@RequiredArgsConstructor
public class IdempotentOrderConsumer {
    private final OrderRepository orderRepo;
    private final ProcessedMessageRepository processedRepo;

    @KafkaListener(topics = "orders.placed", groupId = "order-processor")
    @Transactional
    public void processOrder(OrderEvent event, Acknowledgment ack) {
        String messageId = event.getMessageId(); // unique ID per event

        // Idempotency check: have we processed this message before?
        if (processedRepo.existsByMessageId(messageId)) {
            log.info("Duplicate message {}, skipping", messageId);
            ack.acknowledge();
            return;
        }

        // Process (wrapped in same transaction as idempotency record)
        Order order = orderService.createOrder(event);

        // Record as processed — atomic with the business operation
        processedRepo.save(new ProcessedMessage(messageId, Instant.now()));

        ack.acknowledge();
    }
}

// Alternative: upsert with unique constraint as natural idempotency
@Transactional
public void processPayment(PaymentEvent event) {
    // ON CONFLICT (order_id) DO NOTHING — duplicate events are no-ops
    paymentRepo.saveIfAbsent(Payment.builder()
        .orderId(event.getOrderId())         // unique constraint on order_id
        .amount(event.getAmount())
        .build());
}
```

## 3. Message Ordering Guarantees

### 3.1 Ordering in Kafka

```
Kafka ordering guarantees:
  WITHIN a partition:     strict FIFO — records are totally ordered by offset
  ACROSS partitions:      no ordering guarantee — different partitions = concurrent

Implications:
  If order matters → all related events must go to the SAME partition
  Partition key determines placement: hash(key) % num_partitions

Good partition keys (ensure related events co-locate):
  order_id:     all events for one order → same partition → ordered processing
  user_id:      all user actions → same partition → ordered per user
  account_id:   all account transactions → same partition → sequential balance

Bad partition keys:
  null key:     round-robin → events may scatter → no ordering
  timestamp:    all current traffic → same partition → hot spot
  random UUID:  defeats ordering purpose entirely
```

```java
// Ensure ordering: all events for same order go to same partition
kafka.send("order.events",
           order.getId().toString(),   // KEY = orderId → consistent partition
           new OrderStatusChanged(order.getId(), OrderStatus.CONFIRMED));

kafka.send("order.events",
           order.getId().toString(),   // SAME KEY → same partition → ordered after previous
           new OrderStatusChanged(order.getId(), OrderStatus.SHIPPED));

// Never interleave with null key:
kafka.send("order.events", null, event); // ← destroys ordering guarantee

// Consumer must NOT use multiple threads on same partition
// concurrency = "3" means 3 threads, each assigned DIFFERENT partitions
// All events for order:42 go to partition 2 → processed by thread 2 → ordered
@KafkaListener(topics = "order.events", groupId = "...", concurrency = "3")
public void onEvent(OrderEvent event) { ... }
```

### 3.2 Ordering with concurrent consumers

```java
// Problem: single partition, single consumer thread — sequential but slow
// Solution: application-level ordering within concurrent processing

@KafkaListener(topics = "order.events", groupId = "order-processor",
               concurrency = "6")   // 6 threads for 6 partitions
public void processOrderEvent(OrderEvent event) {
    // All events for the SAME orderId are on the SAME partition
    // → processed by the SAME thread → no concurrent processing of same order
    // Events for DIFFERENT orders may process in parallel → high throughput
    orderStateMachine.transition(event.getOrderId(), event.getStatus());
}

// For RabbitMQ (no partition concept): use consistent hashing exchange
// or route by orderId to specific queue → specific consumer
```

### 3.3 Ordering in SQS

```java
// SQS Standard: no ordering — messages may arrive out of order
// SQS FIFO: ordering within Message Group ID

// Same order's messages must use same MessageGroupId
sqsClient.sendMessage(req -> req
    .queueUrl(FIFO_QUEUE_URL)
    .messageBody(toJson(event))
    .messageGroupId(event.getOrderId().toString())  // FIFO within this group
    .messageDeduplicationId(event.getEventId())     // dedup ID for exactly-once
);

// Messages with DIFFERENT groupIds may process in parallel
// Messages with SAME groupId are strictly ordered (one at a time)
```

## 4. Dead Letter Queues

A dead letter queue (DLQ) is a holding area for messages that could not be successfully processed after a configured number of retries. Without a DLQ, a poison pill message blocks the entire queue indefinitely.

### 4.1 DLQ pattern

```
Normal flow:
  Queue → Consumer → success → ACK → message deleted

Failure flow:
  Queue → Consumer → failure → NACK/timeout → requeue → retry
  After maxRetries exceeded:
  Queue → DLQ → alerting → manual inspection → replay or discard
```

### 4.2 Kafka DLQ implementation

```java
@Configuration
public class KafkaDLQConfig {
    // Main topic: orders.placed
    // Retry topics: orders.placed.retry-1, orders.placed.retry-2, orders.placed.retry-3
    // DLT (dead letter topic): orders.placed.dlt

    @Bean
    public DefaultErrorHandler errorHandler(KafkaTemplate<Object, Object> template) {
        // Exponential backoff: 1s, 2s, 4s — then send to DLT
        ExponentialBackOffWithMaxRetries backOff =
            new ExponentialBackOffWithMaxRetries(3);
        backOff.setInitialInterval(1_000);
        backOff.setMultiplier(2.0);
        backOff.setMaxInterval(30_000);

        DeadLetterPublishingRecoverer recoverer =
            new DeadLetterPublishingRecoverer(template,
                (record, ex) -> {
                    // Route to DLT: originalTopic + ".dlt"
                    return new TopicPartition(
                        record.topic() + ".dlt",
                        record.partition()  // same partition as original for ordering
                    );
                });

        DefaultErrorHandler handler = new DefaultErrorHandler(recoverer, backOff);

        // Non-retryable exceptions: send directly to DLT without retrying
        handler.addNotRetryableExceptions(
            ValidationException.class,
            DeserializationException.class
        );

        return handler;
    }
}

// DLT consumer: inspect and replay failed messages
@Component
@Slf4j
public class DLTConsumer {

    @KafkaListener(topics = "orders.placed.dlt", groupId = "dlt-inspector")
    public void handleDeadLetter(
            ConsumerRecord<String, OrderEvent> record,
            @Header(KafkaHeaders.EXCEPTION_CAUSE_FQCN) String exceptionClass,
            @Header(KafkaHeaders.EXCEPTION_MESSAGE) String errorMessage,
            @Header(KafkaHeaders.ORIGINAL_TOPIC) String originalTopic,
            @Header(KafkaHeaders.ORIGINAL_OFFSET) long originalOffset) {

        log.error("Dead letter: topic={} offset={} exception={} msg={}",
            originalTopic, originalOffset, exceptionClass, errorMessage);

        // Alert ops team
        alertService.sendDLQAlert(record.key(), exceptionClass, errorMessage);

        // Store for analysis
        dlqRepository.save(DeadLetterRecord.builder()
            .key(record.key())
            .payload(record.value())
            .exception(exceptionClass)
            .errorMessage(errorMessage)
            .originalTopic(originalTopic)
            .originalOffset(originalOffset)
            .receivedAt(Instant.now())
            .build());
    }

    // Manual replay endpoint: reprocess DLQ messages after fixing the bug
    public void replayDeadLetters(String originalTopic) {
        List<DeadLetterRecord> dlqMessages = dlqRepository.findByTopic(originalTopic);
        dlqMessages.forEach(msg ->
            kafkaTemplate.send(originalTopic, msg.getKey(), msg.getPayload())
        );
    }
}
```

### 4.3 RabbitMQ DLQ configuration

```java
// On NACK without requeue → message routes to Dead Letter Exchange → DLQ
@Bean
Queue mainQueue() {
    return QueueBuilder.durable("orders.processing")
        .withArgument("x-dead-letter-exchange", "orders.dlx")
        .withArgument("x-dead-letter-routing-key", "orders.dlq")
        .withArgument("x-message-ttl", 300_000)    // also DL on TTL expiry
        .build();
}

// Consumer: decide retry vs DLQ based on exception type
@RabbitListener(queues = "orders.processing")
public void process(Order order, Channel channel,
                     @Header(AmqpHeaders.DELIVERY_TAG) long tag,
                     @Header(AmqpHeaders.REDELIVERED) boolean redelivered)
        throws IOException {
    try {
        orderService.process(order);
        channel.basicAck(tag, false);
    } catch (TransientException e) {
        if (!redelivered) {
            channel.basicNack(tag, false, true);  // requeue once
        } else {
            channel.basicNack(tag, false, false); // second failure → DLQ
        }
    } catch (PermanentException e) {
        channel.basicNack(tag, false, false); // immediately to DLQ
    }
}
```

## 5. Event-Driven Architecture

### 5.1 Core patterns

```
Event-Driven Architecture (EDA): services communicate by publishing and consuming
events rather than direct synchronous calls.

Three EDA sub-patterns:

1. Event Notification:
   Service A changes state → publishes lightweight event ("Order 42 placed")
   Other services react, fetch details they need from Service A if required.
   Loose coupling: consumer doesn't need Order Service up to process event.

2. Event-Carried State Transfer:
   Event carries ALL relevant data ("OrderPlaced {id, items, userId, total}")
   Consumer doesn't need to call back — processes event self-sufficiently.
   Higher event payload; consumer can be fully autonomous.

3. Event Sourcing:
   System state IS the event log — no separate state store.
   Current state = replay of all events from beginning.
   (Covered in detail in section 6)
```

### 5.2 Choreography vs Orchestration

```
Orchestration (central coordinator):          Choreography (events, no coordinator):
  ┌──────────────────┐                          ┌─────────────┐
  │ Order Orchestrator│                          │ Order       │ publishes OrderPlaced
  │  1. Reserve inv  │───▶ Inventory Service     │ Service     │──────────────────────▶
  │  2. Charge card  │───▶ Payment Service       └─────────────┘
  │  3. Send email   │───▶ Email Service                            ┌─────────────────┐
  └──────────────────┘                                              │ Inventory Svc   │
                                                   subscribes ─────▶│ reserves stock  │
  Pro:  visible, easy to trace                                      │ publishes Stk   │
  Con:  central coupling, single point                              │ Reserved        │
        of failure and knowledge                                    └─────────────────┘

  Use: complex workflows with rollback                              ┌─────────────────┐
       (Saga orchestration pattern)                subscribes ─────▶│ Payment Service │
                                                                    │ charges card    │
                                                                    └─────────────────┘

                                                  Pro:  loose coupling, autonomy
                                                  Con:  hard to trace, emergent behaviour
                                                  Use:  simple reactions, fan-out
```

### 5.3 Transactional outbox — guaranteed event publishing

```java
// Problem: save order AND publish event atomically
// Naive dual-write: save succeeds, Kafka publish fails → event lost forever

// Solution: Transactional Outbox
// Write event to DB table in SAME transaction as business data
// Background process reads outbox and publishes to Kafka (at-least-once)

@Entity
@Table(name = "outbox_events")
public class OutboxEvent {
    @Id String id;           // UUID
    String aggregateType;    // "Order"
    String aggregateId;      // "42"
    String eventType;        // "OrderPlaced"
    String payload;          // JSON
    Instant createdAt;
    Instant publishedAt;     // null until published
    int retryCount;
}

@Service
@RequiredArgsConstructor
public class OrderService {
    private final OrderRepository orders;
    private final OutboxRepository outbox;

    @Transactional  // single DB transaction
    public Order placeOrder(PlaceOrderRequest req) {
        Order order = orders.save(Order.from(req));

        // Event written in SAME transaction — atomic with order creation
        outbox.save(OutboxEvent.builder()
            .id(UUID.randomUUID().toString())
            .aggregateType("Order")
            .aggregateId(order.getId().toString())
            .eventType("OrderPlaced")
            .payload(toJson(new OrderPlacedEvent(order)))
            .createdAt(Instant.now())
            .build());

        return order; // DB commit: order + outbox event both committed or both rolled back
    }
}

// Outbox relay: polls for unpublished events, publishes to Kafka
// In production: use Debezium CDC on outbox table for sub-100ms latency
@Scheduled(fixedDelay = 100)  // poll every 100ms
@Transactional
public void relayOutboxEvents() {
    List<OutboxEvent> pending = outbox.findTop100ByPublishedAtIsNull();
    for (OutboxEvent event : pending) {
        try {
            kafkaTemplate.send(
                event.getAggregateType().toLowerCase() + "." +
                event.getEventType().toLowerCase(),
                event.getAggregateId(),
                event.getPayload()
            ).get(5, TimeUnit.SECONDS);   // wait for broker ACK

            event.setPublishedAt(Instant.now());
            outbox.save(event);
        } catch (Exception e) {
            event.setRetryCount(event.getRetryCount() + 1);
            outbox.save(event);
            if (event.getRetryCount() > 10) alertService.page("Outbox stuck: " + event.getId());
        }
    }
}
```

### 5.4 Saga pattern — distributed transactions via events

```java
// Saga: sequence of local transactions coordinated by events
// Each step publishes an event; on failure, compensating transactions undo previous steps

// Order Saga: Place order → Reserve Inventory → Charge Payment → Confirm Order
// Compensation:          ← Release Inventory ← Refund Payment ← Cancel Order

// Orchestration-based Saga using Spring State Machine
@StateMachine
public class OrderSaga {

    // Forward steps
    @StepDefinition
    public void reserveInventory(OrderSagaContext ctx) {
        inventoryService.reserve(ctx.getOrderId(), ctx.getItems());
        ctx.setState(SagaState.INVENTORY_RESERVED);
        kafkaTemplate.send("saga.inventory.reserved", ctx.toEvent());
    }

    @StepDefinition
    public void chargePayment(OrderSagaContext ctx) {
        Payment payment = paymentService.charge(ctx.getOrderId(), ctx.getAmount());
        ctx.setPaymentId(payment.getId());
        kafkaTemplate.send("saga.payment.charged", ctx.toEvent());
    }

    @StepDefinition
    public void confirmOrder(OrderSagaContext ctx) {
        orderRepository.updateStatus(ctx.getOrderId(), OrderStatus.CONFIRMED);
        kafkaTemplate.send("order.confirmed", ctx.toEvent());
    }

    // Compensating transactions (undo steps)
    @CompensationStep
    public void releaseInventory(OrderSagaContext ctx) {
        inventoryService.release(ctx.getOrderId(), ctx.getItems());
    }

    @CompensationStep
    public void refundPayment(OrderSagaContext ctx) {
        paymentService.refund(ctx.getPaymentId());
    }

    @CompensationStep
    public void cancelOrder(OrderSagaContext ctx) {
        orderRepository.updateStatus(ctx.getOrderId(), OrderStatus.CANCELLED);
        kafkaTemplate.send("order.cancelled", ctx.toEvent());
    }
}
```

## 6. CQRS and Event Sourcing

### 6.1 CQRS — Command Query Responsibility Segregation

CQRS separates the write model (commands that change state) from the read model (queries that return data). The two models can use different data stores, different schemas, and scale independently.

```
Traditional CRUD (single model):
  ┌─────────────────────────────────────────┐
  │              Order Service              │
  │  Read: findById, findByUser, search     │
  │  Write: create, update, cancel          │
  │                                         │
  │         Single PostgreSQL DB            │
  └─────────────────────────────────────────┘
  Problem: optimising for reads (indexes, joins) conflicts with write performance

CQRS (separate models):
  Commands (Write Side):                     Queries (Read Side):
  ┌────────────────────┐                    ┌────────────────────────┐
  │ CreateOrderCommand │                    │ GetOrderById           │
  │ CancelOrderCommand │                    │ GetOrdersByUser        │
  │                    │                    │ SearchOrders           │
  │  Validated,        │                    │                        │
  │  business logic    │  ── events ──────▶ │  Denormalised,         │
  │  PostgreSQL        │                    │  query-optimised       │
  └────────────────────┘                    │  Elasticsearch / Redis │
                                            └────────────────────────┘
  Benefits:
    Write model: normalised, domain-rich, ACID
    Read model:  denormalised, query-optimised, can be rebuilt from events
    Scale independently: read replicas, separate caches
    Read model schema can change freely without touching write model
```

```java
// CQRS Command side — rich domain model, validation, business rules
@Service
@RequiredArgsConstructor
public class OrderCommandService {

    @Transactional
    public OrderId handle(CreateOrderCommand cmd) {
        cmd.validate();  // throw if invalid

        Order order = Order.create(
            cmd.getCustomerId(),
            cmd.getItems(),
            pricingService.calculateTotal(cmd.getItems())
        );

        Order saved = orderRepository.save(order);

        // Publish event for read-side projection and other consumers
        eventPublisher.publish(new OrderCreatedEvent(
            saved.getId(), saved.getCustomerId(),
            saved.getItems(), saved.getTotal(), Instant.now()
        ));

        return saved.getId();
    }

    @Transactional
    public void handle(CancelOrderCommand cmd) {
        Order order = orderRepository.findById(cmd.getOrderId()).orElseThrow();
        order.cancel(cmd.getReason()); // domain logic — can only cancel if PENDING
        orderRepository.save(order);
        eventPublisher.publish(new OrderCancelledEvent(order.getId(), cmd.getReason()));
    }
}

// CQRS Query side — optimised for reads, denormalised
@Service
@RequiredArgsConstructor
public class OrderQueryService {
    private final ElasticsearchOperations es;
    private final RedisTemplate<String, OrderSummaryDTO> redis;

    public OrderDetailDTO findById(Long orderId) {
        // Check Redis first (hot data)
        OrderSummaryDTO cached = redis.opsForValue().get("order:" + orderId);
        if (cached != null) return cached.toDetail();

        // Elasticsearch for rich query
        return es.get(GetQuery.of(orderId.toString()), OrderDetailDTO.class);
    }

    public Page<OrderSummaryDTO> findByCustomer(Long customerId, Pageable page) {
        NativeSearchQuery query = new NativeSearchQueryBuilder()
            .withQuery(matchQuery("customerId", customerId))
            .withSort(SortBuilders.fieldSort("createdAt").order(DESC))
            .withPageable(page)
            .build();
        return es.searchForPage(query, OrderSummaryDTO.class);
    }
}

// Read-side projector: handles events and updates query model
@Component
@KafkaListener(topics = {"orders.created", "orders.cancelled"})
public class OrderProjector {

    public void on(OrderCreatedEvent event) {
        OrderDocument doc = OrderDocument.builder()
            .id(event.getOrderId().toString())
            .customerId(event.getCustomerId())
            .status("CREATED")
            .total(event.getTotal())
            .items(event.getItems())
            .createdAt(event.getOccurredAt())
            .build();

        es.index(doc);                            // update Elasticsearch
        redis.opsForValue().set(                  // update Redis cache
            "order:" + event.getOrderId(),
            doc.toSummary(),
            Duration.ofMinutes(10));
    }

    public void on(OrderCancelledEvent event) {
        // Partial update — only status changed
        UpdateQuery update = UpdateQuery.builder(event.getOrderId().toString())
            .withDocument(Map.of("status", "CANCELLED", "cancelReason", event.getReason()))
            .build();
        es.update(update, IndexCoordinates.of("orders"));
        redis.delete("order:" + event.getOrderId()); // invalidate cache
    }
}
```

### 6.2 Event Sourcing

In event sourcing, the application state is not stored as current values in a database — instead, it is derived by replaying a complete sequence of events. The event log IS the source of truth.

```
Traditional state storage:
  Save order:    INSERT INTO orders (status='PENDING', total=99.99)
  Update order:  UPDATE orders SET status='CONFIRMED' WHERE id=42
  Cancel order:  UPDATE orders SET status='CANCELLED' WHERE id=42
  Final state:   { id: 42, status: 'CANCELLED', total: 99.99 }
  History:       LOST — only current state visible

Event sourcing:
  All events appended to event store (immutable log):
  1. OrderPlaced     { id:42, items:[...], total:99.99,  at: T1 }
  2. PaymentReceived { id:42, paymentId:P1, amount:99.99, at: T2 }
  3. OrderConfirmed  { id:42,                             at: T3 }
  4. OrderCancelled  { id:42, reason:"out of stock",      at: T4 }

  Current state = replay events 1→4 through state machine
  History: COMPLETE — every change preserved, who changed it and when
```

```java
// Event sourcing: Order aggregate
public class Order {
    private Long id;
    private OrderStatus status;
    private BigDecimal total;
    private List<OrderItem> items;
    private Long version;  // for optimistic locking / snapshot versioning

    private final List<DomainEvent> uncommittedEvents = new ArrayList<>();

    // Factory method: create from command (produces first event)
    public static Order place(PlaceOrderCommand cmd) {
        Order order = new Order();
        order.apply(new OrderPlacedEvent(
            cmd.getOrderId(), cmd.getCustomerId(),
            cmd.getItems(), cmd.getTotal()
        ));
        return order;
    }

    // Business method: validate + raise event (no direct state mutation)
    public void confirm(ConfirmOrderCommand cmd) {
        if (status != OrderStatus.PAYMENT_RECEIVED)
            throw new IllegalStateException("Cannot confirm order in status " + status);
        apply(new OrderConfirmedEvent(id, Instant.now()));
    }

    public void cancel(CancelOrderCommand cmd) {
        if (status == OrderStatus.SHIPPED || status == OrderStatus.DELIVERED)
            throw new IllegalStateException("Cannot cancel shipped order");
        apply(new OrderCancelledEvent(id, cmd.getReason(), Instant.now()));
    }

    // Apply: mutate state based on event (also used during replay)
    private void apply(DomainEvent event) {
        uncommittedEvents.add(event); // new events tracked for persistence
        mutate(event);                // state transition
    }

    // State transitions — pure function of event
    private void mutate(DomainEvent event) {
        if (event instanceof OrderPlacedEvent e) {
            this.id = e.getOrderId();
            this.items = e.getItems();
            this.total = e.getTotal();
            this.status = OrderStatus.PENDING;
        } else if (event instanceof OrderConfirmedEvent) {
            this.status = OrderStatus.CONFIRMED;
        } else if (event instanceof OrderCancelledEvent) {
            this.status = OrderStatus.CANCELLED;
        }
        // More event types...
        this.version++;
    }

    // Reconstruct from event history (replay)
    public static Order reconstitute(List<DomainEvent> history) {
        Order order = new Order();
        history.forEach(order::mutate); // replay all events, no side effects
        return order;
    }
}

// Event store: append-only log of domain events
@Repository
@RequiredArgsConstructor
public class EventStore {
    private final JdbcTemplate jdbc;

    public void append(Long aggregateId, String aggregateType,
                        List<DomainEvent> events, Long expectedVersion) {
        // Optimistic concurrency: reject if someone else wrote since we read
        Long currentVersion = jdbc.queryForObject(
            "SELECT MAX(version) FROM events WHERE aggregate_id = ?",
            Long.class, aggregateId);
        if (!Objects.equals(currentVersion, expectedVersion))
            throw new OptimisticConcurrencyException(
                "Expected version " + expectedVersion + " but found " + currentVersion);

        for (int i = 0; i < events.size(); i++) {
            DomainEvent event = events.get(i);
            jdbc.update("""
                INSERT INTO events (aggregate_id, aggregate_type, event_type,
                                    payload, version, occurred_at)
                VALUES (?, ?, ?, ?, ?, ?)
                """,
                aggregateId, aggregateType,
                event.getClass().getSimpleName(),
                toJson(event),
                expectedVersion + i + 1,
                event.getOccurredAt());
        }
    }

    public List<DomainEvent> load(Long aggregateId) {
        return jdbc.query(
            "SELECT event_type, payload FROM events WHERE aggregate_id = ? ORDER BY version",
            (rs, n) -> deserialise(rs.getString("event_type"), rs.getString("payload")),
            aggregateId);
    }

    public List<DomainEvent> loadFromVersion(Long aggregateId, Long fromVersion) {
        return jdbc.query(
            "SELECT event_type, payload FROM events WHERE aggregate_id = ? " +
            "AND version > ? ORDER BY version",
            (rs, n) -> deserialise(rs.getString("event_type"), rs.getString("payload")),
            aggregateId, fromVersion);
    }
}
```

### 6.3 Snapshots — optimising event replay

```java
// Replaying 10,000 events per request is expensive
// Snapshot: periodically serialise current state; replay only events after snapshot

@Service
@RequiredArgsConstructor
public class OrderRepository {
    private final EventStore eventStore;
    private final SnapshotStore snapshotStore;

    public Order load(Long orderId) {
        // Try to load latest snapshot
        Optional<Snapshot<Order>> snapshot = snapshotStore.findLatest(orderId);

        Order order;
        if (snapshot.isPresent()) {
            // Start from snapshot state
            order = snapshot.get().getState();
            // Replay only events AFTER the snapshot version
            List<DomainEvent> recentEvents = eventStore.loadFromVersion(
                orderId, snapshot.get().getVersion());
            recentEvents.forEach(e -> applyEvent(order, e));
        } else {
            // Full replay from beginning
            List<DomainEvent> allEvents = eventStore.load(orderId);
            order = Order.reconstitute(allEvents);
        }
        return order;
    }

    public void save(Order order) {
        eventStore.append(order.getId(), "Order",
            order.getUncommittedEvents(), order.getVersion() - order.getUncommittedEvents().size());
        order.clearUncommittedEvents();

        // Take snapshot every 100 events
        if (order.getVersion() % 100 == 0) {
            snapshotStore.save(new Snapshot<>(order.getId(), order, order.getVersion()));
        }
    }
}
```

### 6.4 CQRS + Event Sourcing combined

```
Write side:             Event Store              Read side (projections):
  Commands               (Kafka / DB)              Query models
  ─────────              ────────────              ───────────────────
  PlaceOrder ────────▶  OrderPlaced    ──────────▶ OrderSummary (Redis)
  ConfirmOrder ─────────▶  OrderConfirmed ─────────▶ OrderDetail (ES)
  CancelOrder ──────────▶  OrderCancelled ─────────▶ CustomerOrders (Postgres)
                                           ─────────▶ Analytics (Clickhouse)
  Aggregate: Order       Source of truth          Projections: eventually consistent
  (event sourced)                                  Rebuilt: replay events any time
                                                   Multiple: different models, same events
```

**Advantages of CQRS + Event Sourcing:**

```
✓ Complete audit trail:     every state change recorded with who, what, when
✓ Temporal queries:         "what was the state of order 42 at 3pm yesterday?"
✓ Projections rebuild:      add new read model? replay all events to build it
✓ Event-driven integration: downstream services consume same event log
✓ Debugging:                reproduce bugs by replaying exact event sequence
✓ No update anomalies:      events are immutable — append-only log never corrupted

Disadvantages:
✗ Complexity:               two models, projections, event versioning, snapshots
✗ Eventual consistency:     read models lag behind write model
✗ Event schema evolution:   renaming event fields requires upcasting/migration
✗ Query limitations:        complex ad-hoc queries require careful projection design
✗ Learning curve:           mental model shift from CRUD to event-centric thinking

Use when:
  ✓ Audit requirements (financial, medical, legal)
  ✓ Complex domain with rich behaviour (not CRUD-heavy)
  ✓ Multiple read models needed from same data
  ✓ Event-driven integration between many services
Avoid when:
  ✗ Simple CRUD application
  ✗ Small team unfamiliar with the pattern
  ✗ Strong consistency required on every read (projections lag)
```

## 7. Interview Questions

**Q: When would you choose Kafka over RabbitMQ, and what are the key architectural differences?**

A: I'd choose Kafka when I need event replay, multiple independent consumers processing the same events, or high-throughput durable log semantics. Kafka retains messages for a configurable period regardless of consumption — a new consumer group can process all historical events from the beginning, which is invaluable for rebuilding read models, debugging, or adding new downstream services without re-generating events. RabbitMQ deletes messages after consumption, which is fine for work queues but breaks fan-out and replay requirements. The architectural difference is fundamental: Kafka is an append-only distributed log where consumers track their own position (offset); RabbitMQ is a broker where messages are removed after acknowledgement. For a new microservices system, I'd default to Kafka for domain events (durable, replayable, fan-out) and consider RabbitMQ only for task queues with complex routing logic (topic exchanges, priority queues, delayed delivery) that Kafka doesn't support natively.

---

**Q: Why is exactly-once delivery so difficult in distributed systems, and how does Kafka implement it?**

A: Exactly-once is difficult because of two unavoidable failure scenarios at opposite ends of the pipeline. On the producer side: the broker receives and stores a message, sends an ACK, but the ACK is lost in the network. The producer times out, assumes failure, and retries — creating a duplicate. On the consumer side: the consumer processes a message and crashes before committing the offset — the broker redelivers on restart, creating a duplicate processing. Kafka solves the producer side with idempotent producers: each producer gets a unique producer ID and attaches a monotonically increasing sequence number to every message. The broker deduplicates retries by rejecting any message with a sequence number it's already seen. For the consumer side, Kafka transactions make the offset commit and any downstream Kafka writes atomic: they either both commit or both roll back, so a crash before commit means the message is redelivered and the downstream write is not visible to other consumers. The combination (`enable.idempotence=true`, transactions, `isolation.level=read_committed`) provides end-to-end exactly-once semantics within the Kafka ecosystem.

---

**Q: What is the transactional outbox pattern, and why is it preferable to dual-writing?**

A: Dual-writing means updating the database and publishing a Kafka event as two separate operations in the same code path. If the database commit succeeds but the Kafka publish fails (network error, broker down), the event is permanently lost — the database has the state change but no downstream services know about it. The system becomes silently inconsistent with no automatic recovery. The transactional outbox solves this by writing both the business data and an "outbox event" record in the same database transaction. Since both writes share the same ACID transaction, they either both commit or both roll back — the event cannot be lost. A separate relay process (or Debezium CDC reading the database's write-ahead log) publishes the outbox events to Kafka with at-least-once semantics. The at-least-once delivery is acceptable because consumers are designed to be idempotent. In production, Debezium is preferred over polling because it reacts to WAL changes with sub-100ms latency instead of polling every 100ms, and it works even if the application crashes immediately after commit.

---

**Q: What is the difference between CQRS and Event Sourcing, and do they have to be used together?**

A: They address different problems and can be used independently. CQRS separates the write model from the read model — commands go through domain logic and update a normalised write store; queries are served from a denormalised read store optimised for the specific query patterns. The benefit is that each side can be designed, scaled, and deployed independently. Event Sourcing stores state as an immutable sequence of domain events rather than current values — the current state is derived by replaying events. The benefits are a complete audit trail, temporal queries, and the ability to build new projections by replaying history. CQRS without Event Sourcing is common: use a traditional database on the write side, publish change events to maintain denormalised read models. Event Sourcing without CQRS is theoretically possible but unusual — the event store naturally becomes the write side, making CQRS a natural companion. They are used together when you need both the query flexibility of CQRS and the auditability and temporal query capability of Event Sourcing — typically in financial systems, healthcare, or domains with complex regulatory requirements.

---

**Q: How do dead letter queues work, and what operational practices should surround them?**

A: A DLQ receives messages that cannot be successfully processed after a configured number of retries. Without a DLQ, a single poison pill message — one that always causes an exception, perhaps due to malformed data or a bug — would block the queue forever as the broker redelivers it endlessly. The DLQ breaks this cycle by giving up after N attempts and routing the message to a separate queue for inspection. In Kafka, the Spring `DefaultErrorHandler` with `DeadLetterPublishingRecoverer` routes failed messages to a `.dlt` suffixed topic after exponential backoff exhaustion. Operationally, three practices are essential: alerting on any DLQ message (a message reaching DLQ means something broke and needs human attention), structured storage of the failure reason alongside the original message (the original topic, offset, exception class, and error message should be preserved as headers or in a database), and a replay mechanism to reprocess DLQ messages after fixing the bug (routing them back to the original topic). DLQ depth should be a dashboard metric with a paging alert threshold — a growing DLQ is a warning sign of a systematic failure.

## 8. Key Takeaways

- **Queue vs stream**: queues distribute work (one consumer wins, message deleted); streams are replayable logs (all consumer groups receive all events) — choose based on whether you need fan-out and replay
- **Kafka default for new systems**: replayable, durable, fan-out to multiple consumer groups — use RabbitMQ only when complex routing (priority, delayed delivery, topic exchange) is genuinely needed
- **At-least-once + idempotent consumers** is the pragmatic production strategy: design all consumers with an idempotency key check rather than fighting for exactly-once infrastructure
- **Exactly-once in Kafka**: `enable.idempotence=true` + transactions + `read_committed` isolation — needed for financial double-entry, not for notifications
- **Partition key = ordering unit**: all events that must be ordered go to the same partition via the same key; null key destroys ordering
- **DLQ is mandatory**: every consumer that retries must have a DLQ; without it, one malformed message blocks forever; alert immediately on any DLQ message
- **Transactional outbox**: write event to same DB transaction as business data; use Debezium CDC for sub-100ms relay with no polling overhead
- **CQRS**: separate command (write, normalised, ACID) from query (read, denormalised, eventually consistent) — scale and optimise each independently
- **Event Sourcing**: events are the source of truth; current state is replay; append-only log gives complete audit trail, time travel, and projection rebuild capability — high complexity, use only when audit/temporal requirements justify it
- **Saga pattern**: distributed transactions without 2PC — choreography for simple fan-out, orchestration for multi-step rollback scenarios
