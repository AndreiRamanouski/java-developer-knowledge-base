# Caching Strategies

## Overview

Caching is the highest-ROI performance technique available to a system designer. A cache hit costs microseconds; a database query costs milliseconds — a 1,000x difference. But caching introduces a distributed state problem: multiple representations of the same data must be kept consistent. Every caching strategy is a different answer to the same fundamental question: **when does the cache learn about changes to the underlying data, and what happens in the gap?**

**The caching decision space:**

```
Strategy            Who populates?    When?              Consistency gap
─────────────────────────────────────────────────────────────────────────
Cache-Aside         Application       On miss (lazy)     After write until eviction
Write-Through       Application       On every write     None (sync write to both)
Write-Behind        Application       On write, async    Brief (async DB flush)
Read-Through        Cache layer       On miss             Like cache-aside, automated
Refresh-Ahead       Background job    Before expiry      Brief warming window
```

**Cache metrics to monitor always:**

```
Hit rate:     hits / (hits + misses)   → target > 90% for effectiveness
Miss rate:    misses / (hits + misses)
Eviction rate: items ejected before natural expiry → signals undersized cache
Latency p99:  tail latency reveals hot-key contention or network issues
Memory usage: headroom before eviction pressure begins
```

## 1. Cache-Aside (Lazy Loading)

Cache-aside is the most common pattern. The application manages the cache explicitly: check the cache first, on a miss fetch from the database, then populate the cache for subsequent requests.

### 1.1 Read path

```
Application                Cache               Database
    │                         │                    │
    │──── GET product:42 ─────▶│                    │
    │◀─── null (MISS) ─────────│                    │
    │──────────────────────────────── SELECT * ─────▶│
    │◀──────────────────────────────── product ──────│
    │──── SET product:42 (TTL) ▶│                    │
    │◀─── OK ──────────────────│                    │
    │ return product            │                    │
    │                          │                    │
    │ (next request)           │                    │
    │──── GET product:42 ─────▶│                    │
    │◀─── product (HIT) ───────│                    │ ← no DB call
```

### 1.2 Write path

```
Application                Cache               Database
    │                         │                    │
    │──────────────────────────────── UPDATE ───────▶│
    │◀──────────────────────────────── OK ───────────│
    │──── DEL product:42 ─────▶│                    │  ← invalidate, not update
    │◀─── OK ──────────────────│                    │
    │                          │                    │
    │ (next read will miss → repopulate from DB)    │
```

### 1.3 Implementation

```java
@Service
@RequiredArgsConstructor
public class ProductService {
    private final ProductRepository repository;
    private final RedisTemplate<String, Product> redis;
    private static final Duration TTL = Duration.ofMinutes(10);

    // READ — cache-aside pattern
    public Product findById(Long id) {
        String key = "product:" + id;

        // 1. Check cache
        Product cached = redis.opsForValue().get(key);
        if (cached != null) {
            metrics.increment("cache.hit");
            return cached;
        }

        metrics.increment("cache.miss");

        // 2. Miss — load from database
        Product product = repository.findById(id)
            .orElseThrow(() -> new NotFoundException("Product " + id));

        // 3. Populate cache for future reads
        redis.opsForValue().set(key, product, TTL);
        return product;
    }

    // WRITE — update DB, invalidate cache
    @Transactional
    public Product update(Long id, ProductUpdateRequest req) {
        Product product = repository.findById(id).orElseThrow();
        product.setName(req.getName());
        product.setPrice(req.getPrice());
        Product saved = repository.save(product);

        // Invalidate — next read will repopulate from DB with fresh data
        redis.delete("product:" + id);
        return saved;
    }

    // DELETE
    @Transactional
    public void delete(Long id) {
        repository.deleteById(id);
        redis.delete("product:" + id);
    }
}
```

### 1.4 Spring @Cacheable shorthand

```java
// Spring Cache abstraction automates cache-aside pattern
@Service
public class ProductServiceAnnotated {

    @Cacheable(value = "products", key = "#id",
               unless = "#result == null")   // don't cache nulls
    public Product findById(Long id) {
        return repository.findById(id).orElse(null);
        // Spring checks cache before invoking; populates on miss automatically
    }

    @CachePut(value = "products", key = "#result.id")  // update cache on write
    public Product save(Product product) {
        return repository.save(product);
    }

    @CacheEvict(value = "products", key = "#id")        // evict on delete
    public void delete(Long id) {
        repository.deleteById(id);
    }

    @CacheEvict(value = "products", allEntries = true)  // nuke entire namespace
    @Scheduled(cron = "0 0 3 * * *")                   // daily at 3am
    public void clearStaleCache() {}
}
```

### 1.5 Cache-aside tradeoffs

```
Advantages:
  ✓ Cache only holds data actually requested — no wasted memory
  ✓ Cache failures are non-fatal — fall through to DB (graceful degradation)
  ✓ Flexible — application controls TTL, serialisation, key structure
  ✓ Works with any cache technology (Redis, Memcached, Caffeine)
  ✓ Simple read model — easy to reason about

Disadvantages:
  ✗ Cache miss = 3 operations (cache GET + DB read + cache SET)
  ✗ Cold start — empty cache on deployment → thundering herd (all misses)
  ✗ Stale data window — TTL determines how long stale data is served after DB write
  ✗ Cache and DB can diverge if cache SET fails after DB update
  ✗ Race condition: two threads miss simultaneously → both hit DB → one overwrites
```

### 1.6 Cold start / thundering herd mitigation

```java
// Cache warming: pre-populate on startup before serving traffic
@Component
@RequiredArgsConstructor
public class CacheWarmer implements ApplicationListener<ApplicationReadyEvent> {
    private final ProductService productService;
    private final ProductRepository repository;

    @Override
    public void onApplicationEvent(ApplicationReadyEvent event) {
        log.info("Warming cache with top 1000 products");
        repository.findTopByAccessCount(1000)
            .forEach(p -> productService.findById(p.getId())); // triggers cache population
    }
}

// Mutex to prevent thundering herd on single hot key
// Only first thread fetches from DB; others wait for the cache to be populated
private final Map<String, ReentrantLock> keyLocks = new ConcurrentHashMap<>();

public Product findByIdWithMutex(Long id) {
    String key = "product:" + id;
    Product cached = redis.opsForValue().get(key);
    if (cached != null) return cached;

    ReentrantLock lock = keyLocks.computeIfAbsent(key, k -> new ReentrantLock());
    lock.lock();
    try {
        // Double-checked locking — another thread may have populated while we waited
        cached = redis.opsForValue().get(key);
        if (cached != null) return cached;

        Product product = repository.findById(id).orElseThrow();
        redis.opsForValue().set(key, product, TTL);
        return product;
    } finally {
        lock.unlock();
        keyLocks.remove(key, lock);
    }
}
```

## 2. Write-Through Cache

Write-through updates the cache and the database synchronously on every write. The cache always reflects the most recent write.

### 2.1 Flow

```
Application                Cache               Database
    │                         │                    │
    │──── PUT product:42 ─────▶│                    │
    │                         │────── UPDATE ───────▶│
    │                         │◀───── OK ────────────│
    │◀─── OK ──────────────────│                    │
    │ (cache and DB both updated atomically)        │
    │                         │                    │
    │──── GET product:42 ─────▶│                    │
    │◀─── product (HIT) ───────│                    │ ← always fresh
```

### 2.2 Implementation

```java
@Service
@RequiredArgsConstructor
public class WriteThrough ProductService {
    private final ProductRepository repository;
    private final RedisTemplate<String, Product> redis;

    // READ — standard cache-aside on read
    public Product findById(Long id) {
        String key = "product:" + id;
        Product cached = redis.opsForValue().get(key);
        if (cached != null) return cached;

        Product product = repository.findById(id).orElseThrow();
        redis.opsForValue().set(key, product, Duration.ofHours(1));
        return product;
    }

    // WRITE — synchronously update BOTH cache and DB
    @Transactional
    public Product update(Long id, ProductUpdateRequest req) {
        Product product = repository.findById(id).orElseThrow();
        product.setName(req.getName());
        product.setPrice(req.getPrice());

        // Write to DB first (authoritative)
        Product saved = repository.save(product);

        // Immediately update cache — next read will be a cache hit with fresh data
        redis.opsForValue().set("product:" + id, saved, Duration.ofHours(1));
        return saved;

        // Risk: DB commit succeeds, Redis SET fails → cache stale until TTL expires
        // Mitigation: use @CachePut or handle Redis failure explicitly
    }
}
```

### 2.3 Two-phase write with fallback

```java
// More robust write-through with explicit failure handling
@Transactional
public Product updateRobust(Long id, ProductUpdateRequest req) {
    Product saved = performDatabaseUpdate(id, req);

    try {
        redis.opsForValue().set("product:" + id, saved, Duration.ofHours(1));
    } catch (RedisException e) {
        // Cache write failed — invalidate instead of leaving potentially stale data
        // Next read will miss and repopulate from fresh DB data
        log.warn("Cache write failed for product {}, invalidating", id);
        try { redis.delete("product:" + id); } catch (RedisException ignored) {}
    }
    return saved;
}
```

### 2.4 Write-through tradeoffs

```
Advantages:
  ✓ Cache always consistent with DB — no stale data after writes
  ✓ Simple read path — cache hit guaranteed for recently written data
  ✓ No cache invalidation logic needed (cache is updated, not evicted)
  ✓ Excellent for read-heavy workloads with frequent writes to hot keys

Disadvantages:
  ✗ Write latency = DB write + cache write (two round trips)
  ✗ Cache filled with data that may never be read (write-heavy → wasted memory)
  ✗ Cache and DB write are two separate operations — not truly atomic
     (one can fail without the other rolling back)
  ✗ Cache churn on write-heavy workloads: items written but rarely read
```

## 3. Write-Behind Cache (Write-Back)

Write-behind acknowledges the write after updating the cache only, then asynchronously flushes changes to the database. This gives the lowest write latency of any caching strategy.

### 3.1 Flow

```
Application                Cache               Database
    │                         │                    │
    │──── PUT product:42 ─────▶│                    │
    │◀─── OK (immediate) ──────│                    │  ← ACK before DB write
    │                         │                    │
    │ (milliseconds later)    │                    │
    │                         │────── UPDATE ───────▶│  ← async flush
    │                         │◀───── OK ────────────│
    │                         │ (dirty flag cleared) │
```

### 3.2 Implementation pattern

```java
@Service
@RequiredArgsConstructor
public class WriteBehindProductService {
    private final RedisTemplate<String, Product> redis;
    private final ProductRepository repository;

    // "Dirty set" — keys with pending DB writes
    private static final String DIRTY_SET = "cache:dirty:products";

    // WRITE — update cache immediately, enqueue for DB write
    public Product update(Long id, ProductUpdateRequest req) {
        Product existing = findById(id); // load current state
        existing.setName(req.getName());
        existing.setPrice(req.getPrice());

        // Synchronously update cache
        redis.opsForValue().set("product:" + id, existing, Duration.ofHours(1));

        // Mark as dirty — needs DB flush
        redis.opsForSet().add(DIRTY_SET, id.toString());

        return existing; // return immediately — no DB wait
    }

    // Background flusher — runs every 500ms
    @Scheduled(fixedDelay = 500)
    public void flushDirtyEntries() {
        Set<String> dirtyKeys = redis.opsForSet().members(DIRTY_SET);
        if (dirtyKeys == null || dirtyKeys.isEmpty()) return;

        for (String idStr : dirtyKeys) {
            try {
                Long id = Long.parseLong(idStr);
                Product product = redis.opsForValue().get("product:" + id);
                if (product != null) {
                    repository.save(product);  // flush to DB
                }
                redis.opsForSet().remove(DIRTY_SET, idStr); // clear dirty flag
            } catch (Exception e) {
                log.error("Failed to flush product {} to DB", idStr, e);
                // Leave in dirty set — will retry on next flush cycle
            }
        }
    }
}
```

### 3.3 Write-behind with batching

```java
// Batch writes for efficiency — coalesce multiple updates to same key
@Scheduled(fixedDelay = 1000)
public void flushBatched() {
    Set<String> dirtyIds = redis.opsForSet().members(DIRTY_SET);
    if (dirtyIds == null || dirtyIds.isEmpty()) return;

    // Fetch all dirty products from cache in one pipeline (O(1) round trips)
    List<Long> ids = dirtyIds.stream().map(Long::parseLong).collect(toList());
    List<Product> products = redis.opsForValue()
        .multiGet(ids.stream().map(id -> "product:" + id).collect(toList()))
        .stream().filter(Objects::nonNull).collect(toList());

    if (!products.isEmpty()) {
        repository.saveAll(products);           // single batch INSERT/UPDATE
        redis.opsForSet().remove(DIRTY_SET,     // clear all dirty flags
            dirtyIds.toArray(new Object[0]));
    }
}
```

### 3.4 Write-behind tradeoffs

```
Advantages:
  ✓ Lowest write latency — client ACK before any DB I/O
  ✓ Write coalescing — multiple updates to same key = single DB write
  ✓ DB write batching — amortises DB overhead across many operations
  ✓ Absorbs write spikes — cache buffers bursts, DB sees smoothed load
  ✓ Ideal for high-frequency updates: counters, scores, real-time positions

Disadvantages:
  ✗ Data loss risk — cache failure before flush → lost writes
  ✗ DB temporarily inconsistent with cache — other services reading DB see stale
  ✗ Complex failure recovery — what if flush fails? Retry? Dead-letter?
  ✗ Ordering issues — flush order may not match write order without care
  ✗ Operational complexity — requires monitoring flush queue depth

Use for: view counters, game scores, position tracking, analytics counters
Avoid for: financial transactions, inventory (must be durable immediately)
```

## 4. Read-Through Cache

In read-through, the cache itself is responsible for loading data from the database on a miss. The application only talks to the cache — it never accesses the database directly.

```java
// Cache abstraction implementing read-through via CacheLoader
// Caffeine with a CacheLoader — populates automatically on miss
LoadingCache<Long, Product> readThroughCache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(Duration.ofMinutes(10))
    .recordStats()
    .build(id -> {
        // Invoked automatically on cache miss — application never calls DB directly
        log.debug("Cache miss for product {}, loading from DB", id);
        return repository.findById(id).orElse(null);
    });

// Application code — simple, no cache logic visible
public Product findById(Long id) {
    return readThroughCache.get(id); // misses handled transparently by CacheLoader
}

// AsyncLoadingCache — non-blocking read-through
AsyncLoadingCache<Long, Product> asyncCache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(Duration.ofMinutes(10))
    .buildAsync(id -> CompletableFuture.supplyAsync(
        () -> repository.findById(id).orElse(null)));

public CompletableFuture<Product> findByIdAsync(Long id) {
    return asyncCache.get(id); // returns future immediately; loads DB async on miss
}
```

**Read-through vs cache-aside:**

```
Cache-Aside:    Application checks cache, goes to DB on miss, populates cache.
                Application is cache-aware — must handle miss logic everywhere.

Read-Through:   Cache checks DB on miss automatically (via CacheLoader).
                Application is cache-unaware — just calls get(), never touches DB.
                Cleaner abstraction; harder to implement for distributed caches.
                Works naturally with Caffeine, Guava Cache, JCache (JSR-107).
```

## 5. Refresh-Ahead Cache

Refresh-ahead proactively refreshes cache entries before they expire, eliminating the miss latency for hot items that would otherwise expire under load.

### 5.1 Implementation

```java
// Caffeine refreshAfterWrite — refresh asynchronously when entry is accessed
// after refresh interval, but before expiry
LoadingCache<Long, Product> refreshAheadCache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .refreshAfterWrite(Duration.ofMinutes(5))  // refresh threshold
    .expireAfterWrite(Duration.ofMinutes(10))   // hard expiry (fallback)
    .build(id -> repository.findById(id).orElse(null));

// Behaviour:
//   0-5 min: return cached value (no DB call)
//   5-10 min: return STALE cached value immediately + trigger async refresh
//             → no latency hit for the request that triggers refresh
//   >10 min: entry expired → synchronous load (rare; only if refresh fails)

// Custom refresh-ahead with scheduled background refresh
@Component
@RequiredArgsConstructor
public class RefreshAheadCache {
    private final RedisTemplate<String, Product> redis;
    private final ProductRepository repository;
    private static final Duration TTL = Duration.ofMinutes(10);
    private static final Duration REFRESH_THRESHOLD = Duration.ofMinutes(2);

    // Read: check if approaching expiry and pre-emptively refresh
    public Product get(Long id) {
        String key = "product:" + id;
        Product cached = redis.opsForValue().get(key);
        Long ttlRemaining = redis.getExpire(key, TimeUnit.SECONDS);

        if (cached == null) {
            return loadAndCache(id, key); // hard miss
        }

        // Approaching expiry — refresh asynchronously, serve stale now
        if (ttlRemaining != null && ttlRemaining < REFRESH_THRESHOLD.getSeconds()) {
            triggerAsyncRefresh(id, key);
        }

        return cached; // always return immediately — no wait for refresh
    }

    @Async
    public void triggerAsyncRefresh(Long id, String key) {
        try {
            Product fresh = repository.findById(id).orElse(null);
            if (fresh != null) {
                redis.opsForValue().set(key, fresh, TTL);
            }
        } catch (Exception e) {
            log.warn("Background refresh failed for {}", key, e);
            // Not catastrophic — entry will eventually expire and be reloaded
        }
    }

    private Product loadAndCache(Long id, String key) {
        Product product = repository.findById(id).orElseThrow();
        redis.opsForValue().set(key, product, TTL);
        return product;
    }
}
```

### 5.2 Probabilistic early expiration

```java
// More sophisticated: decide stochastically whether to refresh
// Reduces the "stampede" when many instances check refresh simultaneously
// Based on XFetch algorithm (Vattani et al., 2015)
public Product getWithXFetch(Long id) {
    String key = "product:" + id;

    // Get value + remaining TTL atomically
    Product cached = redis.opsForValue().get(key);
    Long remainingTtl = redis.getExpire(key, TimeUnit.MILLISECONDS);

    if (cached == null || remainingTtl == null || remainingTtl <= 0) {
        return loadAndCache(id, key);
    }

    // XFetch formula: refresh with probability based on TTL remaining
    // early_recompute = (-beta * delta * log(random)) >= remaining_ttl
    double beta = 1.0;            // tuning parameter
    double delta = 50.0;          // estimated recomputation time in ms
    double random = Math.random();
    double score = -beta * delta * Math.log(random);

    if (score >= remainingTtl) {
        // Stochastically decided to refresh — do it async
        triggerAsyncRefresh(id, key);
    }

    return cached;
}
```

### 5.3 Refresh-ahead tradeoffs

```
Advantages:
  ✓ Eliminates latency spikes from cache misses for hot data
  ✓ Stale data window is tiny (only during refresh flight time)
  ✓ Smooths DB load — refreshes spread over time rather than all at TTL expiry
  ✓ Ideal for high-traffic, latency-sensitive endpoints (homepage, trending)

Disadvantages:
  ✗ Wastes resources refreshing data that may not be accessed again
  ✗ Memory pressure from keeping items alive longer than TTL suggests
  ✗ Complexity — probabilistic logic harder to debug than simple TTL
  ✗ Requires background thread pool or async infrastructure

Use for: homepage content, trending items, configuration, hot product listings
Avoid for: rarely accessed items (wasted refreshes), user-specific data (high cardinality)
```

## 6. Cache Invalidation Strategies

Cache invalidation is famously difficult — Phil Karlton's "two hard things" in computer science. The challenge: how do downstream caches learn about upstream data changes?

### 6.1 TTL-based expiry

```java
// Simplest form: let entries expire naturally after a fixed duration
redis.opsForValue().set("product:42", product, Duration.ofMinutes(5));
// After 5 minutes, next read misses → fresh data loaded from DB

// Sliding expiry: reset TTL on every access (keep hot data alive)
redis.opsForValue().set("product:42", product, Duration.ofMinutes(5));
redis.expire("product:42", Duration.ofMinutes(5)); // reset on each GET

// TTL selection guide:
//   Pricing data:          30-60 seconds (changes frequently, staleness costly)
//   Product descriptions:  10-30 minutes (changes rarely)
//   User profile:          5-15 minutes (changes occasionally)
//   Static reference data: hours to days (config, categories)
//   Immutable data:        indefinite (content-addressed, never changes)

// Jitter to prevent avalanche (simultaneous expiry of many keys)
private long ttlWithJitter(long baseTtlSeconds) {
    return baseTtlSeconds + ThreadLocalRandom.current().nextLong(0, baseTtlSeconds / 10);
}
redis.opsForValue().set(key, value,
    Duration.ofSeconds(ttlWithJitter(300))); // 300s ± 30s
```

### 6.2 Event-driven invalidation

```java
// Kafka-based cache invalidation: DB change → event → all cache nodes invalidate
// More precise than TTL — invalidate exactly when data changes, not on schedule

// Producer: publish invalidation event when data changes
@Transactional
public Product update(Long id, ProductUpdateRequest req) {
    Product saved = performUpdate(id, req);
    kafkaTemplate.send("cache.invalidation",
        new CacheInvalidationEvent("product", id.toString()));
    return saved;
}

// Consumer: every service instance listens and invalidates its local caches
@KafkaListener(topics = "cache.invalidation", groupId = "${spring.application.name}-${random.uuid}")
// Note: unique groupId → EVERY instance gets EVERY message (fan-out, not work-queue)
public void onInvalidation(CacheInvalidationEvent event) {
    if ("product".equals(event.getType())) {
        // Invalidate both L1 (local Caffeine) and L2 (Redis)
        localProductCache.invalidate(Long.parseLong(event.getId()));
        redis.delete("product:" + event.getId());
        log.debug("Invalidated cache for {} {}", event.getType(), event.getId());
    }
}

// Redis Pub/Sub alternative — lower latency than Kafka for simple invalidation
@Component
public class RedisCacheInvalidator implements MessageListener {
    @Override
    public void onMessage(Message message, byte[] pattern) {
        String key = new String(message.getBody());
        localCache.invalidate(key); // each instance invalidates its own L1 cache
    }
}
// Publisher side:
redis.convertAndSend("cache:invalidate", "product:42");
```

### 6.3 Version-based invalidation (cache versioning)

```java
// Embed a version number in the cache key — changing the version instantly
// invalidates all entries without scanning or deleting individual keys
// Especially useful for namespace-level invalidation

@Service
public class VersionedCacheService {
    // Version stored in Redis — increment to invalidate all entries of that type
    private static final String VERSION_KEY = "cache:version:products";

    public String buildKey(Long id) {
        Long version = redis.opsForValue().get(VERSION_KEY);
        if (version == null) {
            redis.opsForValue().set(VERSION_KEY, 1L);
            version = 1L;
        }
        return "product:v" + version + ":" + id;
    }

    public Product findById(Long id) {
        String key = buildKey(id);
        Product cached = redis.opsForValue().get(key);
        if (cached != null) return cached;

        Product product = repository.findById(id).orElseThrow();
        redis.opsForValue().set(key, product, Duration.ofMinutes(10));
        return product;
    }

    // Invalidate ALL products instantly — just increment the version
    // Old keys are orphaned and expire naturally (no scan-and-delete needed)
    public void invalidateAllProducts() {
        redis.opsForValue().increment(VERSION_KEY);
        // All existing "product:v{old}:*" keys are now unreachable via buildKey()
        // They'll be garbage collected by TTL — O(1) "bulk invalidation"
    }
}
```

### 6.4 Tag-based invalidation

```java
// Associate cache entries with tags; invalidate all entries sharing a tag
// Example: product belongs to "category:electronics" — invalidate all
// cached responses that include electronics when a product in that category updates

@Service
public class TaggedCacheService {
    // tag → set of cache keys that use data tagged with this tag
    private static final String TAG_PREFIX = "cache:tag:";

    public void setWithTags(String key, Object value,
                             Duration ttl, String... tags) {
        redis.opsForValue().set(key, value, ttl);
        for (String tag : tags) {
            redis.opsForSet().add(TAG_PREFIX + tag, key);
            redis.expire(TAG_PREFIX + tag, ttl.plus(Duration.ofMinutes(5)));
        }
    }

    public void invalidateByTag(String tag) {
        String tagKey = TAG_PREFIX + tag;
        Set<String> keys = redis.opsForSet().members(tagKey);
        if (keys != null && !keys.isEmpty()) {
            redis.delete(keys);   // delete all tagged cache entries
        }
        redis.delete(tagKey);     // delete the tag set itself
    }
}

// Usage:
taggedCache.setWithTags("product:42", product, Duration.ofMinutes(10),
    "product:42", "category:electronics", "brand:apple");

// When Apple updates brand info — invalidate all pages using Apple brand data
taggedCache.invalidateByTag("brand:apple");
// Automatically clears product pages, brand pages, search results for Apple
```

### 6.5 Cache stampede / dog-pile protection

```java
// Problem: popular key expires → concurrent requests all miss → all hit DB
// Solution: Redis SET NX (set if not exists) as a distributed mutex

public Product getWithDistributedLock(Long id) {
    String key = "product:" + id;
    String lockKey = "lock:product:" + id;

    Product cached = (Product) redis.opsForValue().get(key);
    if (cached != null) return cached;

    // Try to acquire lock — only one process wins
    Boolean acquired = redis.opsForValue()
        .setIfAbsent(lockKey, "locked", Duration.ofSeconds(5));

    if (Boolean.TRUE.equals(acquired)) {
        try {
            // This process won the lock — fetch from DB and populate cache
            Product product = repository.findById(id).orElseThrow();
            redis.opsForValue().set(key, product, Duration.ofMinutes(10));
            return product;
        } finally {
            redis.delete(lockKey);
        }
    } else {
        // Another process holds the lock — wait briefly and retry
        try { Thread.sleep(50); } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
        return getWithDistributedLock(id); // retry — will likely hit cache now
    }
}
```

## 7. Distributed Caching

A distributed cache runs across multiple nodes, sharing memory across the cluster. This enables cache sizes beyond a single server's RAM and provides high availability.

### 7.1 Redis Cluster

```
Redis Cluster: 16,384 hash slots distributed across N master nodes
  Each master owns a range of slots; replicas shadow each master.
  Key is hashed to a slot: hash_slot = CRC16(key) % 16384

Example 3-master cluster:
  Master 1: slots 0–5460     (shard 1)
  Master 2: slots 5461–10922 (shard 2)
  Master 3: slots 10923–16383 (shard 3)
  Each master has 1 replica for HA

Key routing:
  GET product:42
  → CRC16("product:42") % 16384 = slot 3274
  → slot 3274 owned by Master 1
  → client connects directly to Master 1

Hash tags for co-location:
  Keys with the same {tag} are guaranteed to hash to the same slot
  {user:123}:cart and {user:123}:profile → same slot → can use MULTI/EXEC together
  Without hash tags, multi-key operations requiring same node would fail
```

```java
// Lettuce (default Spring Boot Redis client) — cluster-aware
@Configuration
public class RedisClusterConfig {
    @Bean
    public RedisConnectionFactory redisConnectionFactory() {
        RedisClusterConfiguration clusterConfig = new RedisClusterConfiguration(
            List.of("redis-1:6379", "redis-2:6379", "redis-3:6379")
        );
        clusterConfig.setMaxRedirects(3); // MOVED redirect retries

        LettuceClientConfiguration clientConfig = LettuceClientConfiguration.builder()
            .readFrom(ReadFrom.REPLICA_PREFERRED) // read from replicas when available
            .build();

        return new LettuceConnectionFactory(clusterConfig, clientConfig);
    }
}

// Hash tag — ensure related keys land on same slot for atomic operations
String cartKey    = "{user:123}:cart";    // slot = CRC16("user:123") % 16384
String profileKey = "{user:123}:profile"; // same slot as cart
// Can now use MULTI/EXEC (transaction) on these two keys
```

### 7.2 Consistent hashing for cache clusters

```java
// Consistent hashing: distribute cache keys across nodes so that
// adding/removing a node only remaps ~1/N of keys (vs all keys with modulo hashing)

public class ConsistentHashCache {
    private final TreeMap<Long, String> ring = new TreeMap<>(); // hash → node
    private final int virtualNodes = 150; // virtual nodes per physical node

    public void addNode(String node) {
        for (int i = 0; i < virtualNodes; i++) {
            long hash = hash(node + ":" + i);
            ring.put(hash, node);
        }
    }

    public void removeNode(String node) {
        for (int i = 0; i < virtualNodes; i++)
            ring.remove(hash(node + ":" + i));
    }

    public String getNode(String key) {
        if (ring.isEmpty()) throw new IllegalStateException("No nodes in ring");
        long hash = hash(key);
        Map.Entry<Long, String> entry = ring.ceilingEntry(hash); // clockwise lookup
        return (entry != null ? entry : ring.firstEntry()).getValue(); // wrap around
    }

    private long hash(String key) {
        // Use consistent hashing function (MurmurHash or MD5 truncated)
        return Hashing.murmur3_128().hashString(key, StandardCharsets.UTF_8)
            .asLong();
    }
}

// When node B is removed:
//   Before: keys in [A..B] → B
//   After:  keys in [A..B] → C (next node on ring)
//   Only ~1/N keys remapped; all others unaffected
```

### 7.3 Hot key problem

```java
// A single key receiving disproportionate traffic → single Redis node bottleneck
// Example: Super Bowl start time stored in Redis → millions of reads/sec

// Solution 1: Local in-process cache for super-hot keys
// Offloads Redis with L1 cache on each app server
private final Cache<String, Object> localHotCache = Caffeine.newBuilder()
    .maximumSize(100)
    .expireAfterWrite(Duration.ofSeconds(5)) // short TTL — accept tiny staleness
    .build();

public Object getHotKey(String key) {
    return localHotCache.get(key, k -> redis.opsForValue().get(k));
}

// Solution 2: Key replication — spread hot key across N shards
// Write to N copies: "product:hot:1:42", "product:hot:2:42", ..., "product:hot:N:42"
// Read from random copy — distributes load across N nodes

public void setHotKey(String baseKey, Object value, Duration ttl) {
    int replicas = 10;
    for (int i = 0; i < replicas; i++) {
        redis.opsForValue().set(baseKey + ":" + i, value, ttl);
    }
}

public Object getHotKey(String baseKey) {
    int replica = ThreadLocalRandom.current().nextInt(10);
    return redis.opsForValue().get(baseKey + ":" + replica);
}

// Solution 3: Read-only replica routing
// Redis Cluster: configure client to read from replicas for specific keys
// Lettuce: ReadFrom.REPLICA_PREFERRED routes reads to replicas by default
```

// Leaderboard — Redis sorted set
public class Leaderboard {
    private static final String KEY = "leaderboard:weekly";

    public void updateScore(String userId, double score) {
        redis.opsForZSet().add(KEY, userId, score);           // O(log n)
        redis.expire(KEY, Duration.ofDays(7));
    }

    public List<String> getTopN(int n) {
        return new ArrayList<>(
            redis.opsForZSet().reverseRange(KEY, 0, n - 1)); // O(log n + n)
    }

    }
}

// Rate limiter — Redis sliding window counter
public class RateLimiter {
    public boolean isAllowed(String userId, int maxRequests, Duration window) {
        String key = "rate:" + userId;
        long now = System.currentTimeMillis();
        long windowStart = now - window.toMillis();

        // Atomic Lua script — ZADD + ZREMRANGEBYSCORE + ZCARD in one operation
        String script = """
            local key = KEYS[1]
            local now = tonumber(ARGV[1])
            local windowStart = tonumber(ARGV[2])
            local maxRequests = tonumber(ARGV[3])
            local ttl = tonumber(ARGV[4])
            redis.call('ZREMRANGEBYSCORE', key, 0, windowStart)
            local count = redis.call('ZCARD', key)
            if count < maxRequests then
                redis.call('ZADD', key, now, now)
                redis.call('EXPIRE', key, ttl)
                return 1
            end
            return 0
            """;

        Long result = redis.execute(RedisScript.of(script, Long.class),
            List.of(key),
            String.valueOf(now),
            String.valueOf(windowStart),
            String.valueOf(maxRequests),
            String.valueOf(window.toSeconds()));
        return Long.valueOf(1).equals(result);
    }
}

// Distributed lock — Redlock algorithm
public class DistributedLock {
    public boolean tryLock(String resource, String token, Duration ttl) {
        // SET key value NX PX milliseconds — atomic: only sets if absent
        Boolean acquired = redis.opsForValue()
            .setIfAbsent(resource, token, ttl);
        return Boolean.TRUE.equals(acquired);
    }

    public void unlock(String resource, String token) {
        // Lua script — only delete if token matches (prevents releasing another's lock)
        String script = """
            if redis.call('get', KEYS[1]) == ARGV[1] then
                return redis.call('del', KEYS[1])
            else
                return 0
            end
            """;
        redis.execute(RedisScript.of(script, Long.class),
            List.of(resource), token);
    }
}

## 8. Cache Coherence

Cache coherence is the problem of keeping multiple cache copies of the same data consistent when the underlying data changes. In a distributed system with multiple cache layers (L1 per server, shared L2 Redis), coherence is non-trivial.

### 8.1 The multi-layer coherence problem

```
                 Write to PostgreSQL
                        │
                        ▼
             product.price = $9.99 (DB)

But in caches:
  App Server 1: L1 Caffeine → product.price = $12.99 (stale, TTL: 45s remaining)
  App Server 2: L1 Caffeine → product.price = $12.99 (stale, TTL: 52s remaining)
  Redis L2:                 → product.price = $12.99 (stale, TTL: 4m remaining)

User on Server 1 sees $9.99 (got fresh from DB after cache miss)
User on Server 2 sees $12.99 (still in stale L1 cache)
→ Same user, different prices depending on which server handles the request
```

### 8.2 Write-invalidate protocol

```java
// Most common coherence strategy: on write, invalidate ALL cache tiers
// Next read on any server will miss → fetch fresh from DB

@Transactional
public Product updatePrice(Long id, BigDecimal newPrice) {
    // 1. Update database (authoritative source)
    Product product = repository.findById(id).orElseThrow();
    product.setPrice(newPrice);
    repository.save(product);

    // 2. Invalidate L2 Redis (shared — one call covers all servers' L2)
    redis.delete("product:" + id);

    // 3. Broadcast L1 invalidation to all app servers via Redis pub/sub
    redis.convertAndSend("cache:invalidate:product", id.toString());

    return product;
}

// Every app server subscribes and invalidates its own L1 on message
@Component
public class L1CacheInvalidator implements MessageListener {
    private final Cache<Long, Product> localCache;

    @Override
    public void onMessage(Message message, byte[] pattern) {
        Long id = Long.parseLong(new String(message.getBody()));
        localCache.invalidate(id);
        log.debug("L1 invalidated for product {}", id);
    }
}

// Redis subscription setup
@Bean
public RedisMessageListenerContainer invalidationListener(
        RedisConnectionFactory factory, L1CacheInvalidator handler) {
    RedisMessageListenerContainer container = new RedisMessageListenerContainer();
    container.setConnectionFactory(factory);
    container.addMessageListener(handler,
        new ChannelTopic("cache:invalidate:product"));
    return container;
}
```

### 8.3 Write-update protocol

```java
// Alternative: on write, update all caches (not just invalidate)
// Benefit: next read is always a cache hit (no miss after write)
// Risk: update to Redis may succeed while update to some L1s fails → inconsistent L1s

@Transactional
public Product updateProduct(Long id, ProductUpdateRequest req) {
    Product saved = performUpdate(id, req);

    // Update L2 Redis
    redis.opsForValue().set("product:" + id, saved, Duration.ofMinutes(10));

    // Publish updated value for L1 update (not just invalidation)
    redis.convertAndSend("cache:update:product",
        objectMapper.writeValueAsString(new CacheUpdateMessage(id, saved)));

    return saved;
}

// App servers receive update and populate L1 directly
@Component
public class L1CacheUpdater implements MessageListener {
    @Override
    public void onMessage(Message message, byte[] pattern) {
        CacheUpdateMessage msg = deserialise(message);
        localCache.put(msg.getId(), msg.getProduct()); // update, not just invalidate
    }
}
// Risk: if any L1 update fails silently, that server serves stale data
// until TTL expires. Write-invalidate is safer — it degrades to a miss, not stale.
```

### 8.4 Version vectors for coherence

```java
// Attach a version to every cached value
// On read, compare cached version to DB version — discard if stale

@Cacheable(value = "products", key = "#id")
public VersionedProduct findByIdVersioned(Long id) {
    Product product = repository.findById(id).orElseThrow();
    return new VersionedProduct(product, product.getVersion()); // version from @Version field
}

public VersionedProduct getWithVersionCheck(Long id) {
    VersionedProduct cached = redisCache.get("product:" + id);
    if (cached == null) return load(id);

    // Lightweight version check: only SELECT version column (no full row fetch)
    Long dbVersion = repository.findVersionById(id);
    if (dbVersion != null && !dbVersion.equals(cached.getVersion())) {
        // Cache stale — reload and update
        return load(id);
    }
    return cached;
}
// Cost: one extra SELECT per read to check version
// Benefit: no stale data served, no need for explicit invalidation
// Use when: correctness is critical but invalidation events are unreliable
```

### 8.5 Cache coherence strategies comparison

|Strategy|Staleness|Write Cost|Read Miss Rate|Complexity|
|---|---|---|---|---|
|TTL only|Up to TTL duration|Low|After expiry|Simple|
|Write-invalidate (push)|Near-zero|Medium (publish event)|After every write|Moderate|
|Write-update (push)|Near-zero|High (update all layers)|Low (hit after write)|Higher|
|Version check (pull)|Near-zero|Low|After version change|Moderate|
|Refresh-ahead|Very low|Background only|Very low|Higher|

## 9. Cache Anti-Patterns to Avoid

```
1. Caching everything
   Problem: Low hit rate + high memory pressure + complex invalidation
   Fix: Profile access patterns; cache hot paths with clear hit rate targets

2. No TTL (eternal cache)
   Problem: Stale data served indefinitely; memory exhaustion over time
   Fix: Always set TTL; use jitter to prevent avalanche

3. Caching mutable aggregates without invalidation
   Problem: "order count = 42" cached; new orders placed → count stays 42
   Fix: Invalidate or recalculate on mutation; prefer immutable snapshots

4. Cache as primary data store
   Problem: Redis OOM → all data lost; no persistence guarantee
   Fix: Redis is a cache, not a database; authoritative data lives in DB

5. Cache the N+1 query result
   Problem: Cache "user's orders" as list → any new order stales entire list
   Fix: Cache individual items by ID; compose in application

6. Ignoring serialization cost
   Problem: Java object serialization (JDK default) → large payloads → slow
   Fix: Use JSON (Jackson) or binary (Kryo, Protocol Buffers) with explicit sizing

7. Not handling cache failures gracefully
   Problem: Redis down → NullPointerException cascades → total outage
   Fix: Wrap cache calls in try/catch; fall through to DB on cache failure
```

## 10. Interview Questions

**Q: Walk me through how you would implement cache-aside and explain the consistency risks.**

A: In cache-aside, the application reads from the cache first; on a miss, it fetches from the database and populates the cache. For writes, it updates the database and then invalidates (or updates) the cache key. The consistency risks are: first, the invalidation race — a thread reads a stale value between the database write and the cache invalidation, then the cache is invalidated and another thread repopulates it correctly, but the first thread served stale data to the user. Second, the two-phase failure — the database write succeeds but the subsequent cache invalidation fails (network error), leaving stale data in the cache until TTL expiry. Third, the thundering herd — a popular key expires while under high load, causing concurrent misses that all hit the database simultaneously. Mitigation: use TTLs as a safety net for any failed invalidation, use distributed locks for stampede protection on very hot keys, and treat cache failures as non-fatal by falling through to the database.

---

**Q: What is the difference between write-through and write-behind, and when would you choose each?**

A: Write-through updates both the cache and the database synchronously on every write — the write completes only when both succeed. This ensures the cache is always consistent with the database at the cost of write latency (two round trips) and potential cache pollution (writing data that may never be read). Write-behind (write-back) updates only the cache synchronously and flushes to the database asynchronously in the background — giving the lowest possible write latency. The risk is data loss: if the cache node fails before the async flush, the written data is lost. I'd choose write-through for financial transactions, inventory, and any data where durability is non-negotiable and writes are not the bottleneck. I'd choose write-behind for high-frequency low-value updates: view counters, game scores, position tracking, analytics events — where the occasional loss of a single increment is acceptable and write throughput matters more than durability.

---

**Q: How do you maintain cache coherence across multiple application servers each with their own local (L1) caches?**

A: The core challenge is that each application server has its own JVM-local cache (Caffeine) that is invisible to other servers. When data changes, the local caches on other servers remain stale until their TTL expires. The standard solution is a pub/sub invalidation broadcast: when a write occurs, the writing server updates the database, invalidates the shared L2 Redis cache, and publishes an invalidation message to a Redis channel (or Kafka topic). Every application server subscribes to that channel and immediately invalidates the matching entry in its local L1 cache. This achieves near-zero coherence lag across all servers. The write-invalidate approach is safer than write-update: if an L1 update message is lost, the server serves a cache miss (falls back to Redis or DB) rather than indefinitely stale data. Each server subscribes with a unique consumer group so every server receives every invalidation event, not just one of them.

---

**Q: Explain the thundering herd problem in caching and give two different solutions.**

A: The thundering herd (or cache stampede) occurs when a popular cache entry expires — simultaneously, many requests experience a cache miss, all of them query the database at once, and the sudden burst of queries overwhelms the database. It's worst for very popular keys under sustained high traffic. Two solutions: first, mutex-based serialisation — only the first thread to detect the miss acquires a distributed lock (Redis `SETNX`); it fetches from the database and populates the cache. All other threads wait briefly and then re-read the now-populated cache, resulting in exactly one database query instead of hundreds. Second, probabilistic early expiration (XFetch algorithm) — instead of waiting until the TTL expires, the application begins probabilistically refreshing the cache entry before expiry, with refresh probability increasing as the TTL approaches zero. Because the refresh is triggered while the cache is still valid and returns stale data immediately, there is no latency spike and no stampede. The first approach is simpler and more precise; the second eliminates any miss window entirely but requires tuning the probability parameters.

---

**Q: What is cache penetration, and how is it different from a cache stampede?**

A: Cache penetration occurs when requests are made for keys that have never existed — like a non-existent product ID or a malicious query with random IDs. Since the key is absent from both the cache and the database, every request misses the cache, queries the database, finds nothing, and never populates the cache. Each subsequent request repeats this cycle. Unlike a stampede (which is a burst at the moment of expiry), penetration is a sustained drain — every request costs a full database round trip indefinitely. Two solutions: cache negative results (store a sentinel value like `null` or `{"exists":false}` with a short TTL so repeated requests for non-existent keys are served from cache), and Bloom filters (a probabilistic data structure that can definitively say "this key does NOT exist" — if the Bloom filter says a product ID doesn't exist, skip the cache and database entirely). Bloom filters are used in practice by DynamoDB, Cassandra, and HBase to avoid unnecessary disk lookups for keys that don't exist.

## 11. Key Takeaways

- **Cache-aside** is the default — lazy, application-controlled, cache failures are non-fatal; trade-off is stale data window and thundering herd risk on popular expiring keys
- **Write-through** keeps cache fresh on writes but doubles write latency and wastes memory with never-read entries; best for read-heavy with important write consistency
- **Write-behind** gives lowest write latency via async DB flush — excellent for counters and scores; never use for financial or inventory data where durability is required
- **Refresh-ahead** eliminates miss latency for hot keys by proactively refreshing before TTL expiry — at the cost of background load and complexity
- **Invalidation** is the hardest part: TTL is the safety net; event-driven (Kafka/pub-sub) is precise; version numbers are elegant for bulk invalidation; tag-based handles dependency graphs
- **Thundering herd**: mutex for single hot key; probabilistic early expiration (XFetch) for zero-miss-window; cache warming on startup for cold-start scenarios
- **Cache penetration**: always cache negative results with short TTL; Bloom filter for high-volume non-existent key attacks
- **Cache avalanche**: always add TTL jitter (`baseTtl + random(0, baseTtl/10)`)
- **Distributed caching coherence**: write-invalidate via pub/sub is safer than write-update — a missed invalidation causes a miss (correctable), not silently stale data (silent failure)
- **Hot keys**: local L1 Caffeine cache per server with 5-10 second TTL offloads Redis by orders of magnitude for truly hot data
