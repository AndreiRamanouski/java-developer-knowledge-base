# Database Design Decisions

## Overview

Database selection is one of the most consequential and hardest-to-reverse decisions in system design. Unlike application code, which can be refactored incrementally, migrating a live database storing billions of records requires months of careful dual-writing, data validation, and cutover coordination. The goal is to match the database's data model, consistency guarantees, and operational characteristics to the specific access patterns and reliability requirements of each problem — not to pick a favourite technology.

**The core question at every decision point:**

```
"What access patterns does this data have, and what guarantees does
 the application need when accessing it?"

Access pattern drives data model:
  Relational queries (joins, aggregates)   → SQL
  Key-value lookups                        → Redis, DynamoDB
  Document (nested, schema-flexible)       → MongoDB, Firestore
  Time-series (append, range scan by time) → InfluxDB, TimescaleDB
  Graph (multi-hop traversal)              → Neo4j, Neptune
  Full-text search                         → Elasticsearch, OpenSearch
  Blob/file storage                        → S3, GCS

Consistency requirement drives guarantee level:
  Financial transactions, inventory        → ACID (PostgreSQL, MySQL)
  User activity, analytics                 → BASE / eventual (Cassandra, DynamoDB)
  Session, rate limit, ephemeral           → In-memory (Redis)
```

## 1. SQL vs NoSQL Trade-offs

### 1.1 Relational Databases (SQL)

SQL databases organise data into tables with defined schemas and relationships enforced by foreign keys. They guarantee ACID transactions across multiple tables.

```
Core strengths:
  Relational model:    JOINs across tables avoid data duplication
  Schema enforcement:  Data integrity guaranteed at DB level
  ACID transactions:   Multi-row, multi-table atomic operations
  Ad-hoc queries:      SQL expresses arbitrary queries without pre-planning
  Mature ecosystem:    40+ years of tooling, expertise, optimisers

Core weaknesses:
  Horizontal scale:    Difficult to shard; designed for single powerful servers
  Schema rigidity:     ALTER TABLE on 10B-row table = hours of downtime
  Object-relational:   Application objects → rows = impedance mismatch
  Nested data:         Representing tree/graph structures = complex joins

Examples:
  PostgreSQL — open source, extensible (JSON, full-text, geospatial, arrays)
  MySQL       — ubiquitous, excellent replication, used by Facebook, Twitter
  Oracle      — enterprise, mature optimiser, expensive
  SQL Server  — Microsoft ecosystem
  CockroachDB — distributed SQL, horizontal scale with ACID guarantees
  PlanetScale — MySQL-compatible, serverless, horizontal sharding via Vitess
```

**When SQL is the right default:**

```
Start here unless you have a specific reason not to.

Use SQL when:
  ✓ Data has relationships (users → orders → items → products)
  ✓ Transactions span multiple entities (bank transfer: debit + credit)
  ✓ Query patterns are unpredictable (analytics, reporting, ad-hoc)
  ✓ Data integrity is critical (financial data, user accounts)
  ✓ Team is familiar with SQL — operational simplicity matters
  ✓ Dataset fits on a single powerful server (< ~5 TB, < ~100k QPS writes)
```

### 1.2 NoSQL Databases

NoSQL databases trade the relational model and strict ACID guarantees for specific performance, scalability, or data model benefits.

```
Four NoSQL data models:

1. Document (MongoDB, Firestore, CouchDB)
   Data: JSON/BSON documents with arbitrary nested structure
   Query: filter by any field, aggregation pipeline
   Use: product catalogs, user profiles, CMSs, when schema evolves frequently
   Strength: no JOINs needed for self-contained entities
   Weakness: poor for cross-document relations; transactions limited

2. Key-Value (Redis, DynamoDB, Riak)
   Data: opaque blob per key — no query capability on value contents
   Query: only by exact key lookup (DynamoDB adds sort key ranges)
   Use: sessions, caches, rate limits, shopping carts
   Strength: O(1) lookup, extreme throughput
   Weakness: no secondary indexes (in pure KV); no query flexibility

3. Wide-Column (Cassandra, HBase, Bigtable)
   Data: rows identified by partition key, columns grouped by clustering key
   Query: must specify partition key; efficient range scans on clustering key
   Use: time-series, activity feeds, IoT sensor data, audit logs
   Strength: horizontal scale, write-optimised (LSM tree), tunable consistency
   Weakness: must design tables for specific queries; no JOINs; limited flexibility

4. Graph (Neo4j, Amazon Neptune, TigerGraph)
   Data: nodes and edges with properties
   Query: Cypher / SPARQL — traverse relationships
   Use: social networks (friends of friends), fraud detection, recommendation engines
   Strength: multi-hop traversal is natural and fast
   Weakness: poor fit for non-graph queries; limited horizontal scale
```

### 1.3 SQL vs NoSQL decision matrix

|Dimension|SQL|NoSQL|
|---|---|---|
|Data model|Normalised tables, relations|Varies: document, KV, column, graph|
|Schema|Enforced, migrations required|Flexible (schema-on-read)|
|Transactions|Full ACID multi-table|Varies: document-level to none|
|Query flexibility|High (arbitrary SQL)|Low–medium (pre-planned access)|
|Horizontal scale|Hard (sharding complexity)|Designed for it (Cassandra, DynamoDB)|
|Consistency|Strong (default)|Tunable (eventual to strong)|
|Maturity|High|Varies — MongoDB mature, others less so|
|Best for|Relations, integrity, ad-hoc|Scale, specific access patterns, flexibility|

**Common misconceptions:**

```
"NoSQL is faster than SQL" — FALSE.
  PostgreSQL with good indexes and connection pooling handles 100k+ QPS.
  The bottleneck is rarely SQL vs NoSQL — it's schema design and indexing.

"NoSQL scales better" — PARTIALLY TRUE.
  NoSQL scales writes more easily. But most apps don't need that.
  A well-tuned PostgreSQL with read replicas handles most startups forever.

"NoSQL has no schema" — FALSE.
  NoSQL has schema-on-read (application enforces structure, not DB).
  This moves schema validation to application code — a trade-off, not a win.
```

## 2. Normalization vs Denormalization

### 2.1 Normalization

Normalization eliminates data redundancy by organising data into tables such that each piece of information is stored exactly once.

```
Unnormalized (single table — all data duplicated):
┌──────────────────────────────────────────────────────────────┐
│ order_id │ customer_name │ customer_email │ product_name │ qty│
│ 1        │ Alice         │ alice@ex.com   │ Widget A     │ 2  │
│ 1        │ Alice         │ alice@ex.com   │ Widget B     │ 1  │
│ 2        │ Bob           │ bob@ex.com     │ Widget A     │ 3  │
└──────────────────────────────────────────────────────────────┘
Problem: alice@ex.com repeated — update one row, others stale → anomaly

1NF (First Normal Form): atomic values, no repeating groups
2NF (Second Normal Form): no partial dependencies on composite key
3NF (Third Normal Form): no transitive dependencies (non-key depends only on key)

Normalised (3NF):
customers(id, name, email)
products(id, name, price, description)
orders(id, customer_id FK, created_at)
order_items(order_id FK, product_id FK, quantity, unit_price)

Benefits:
  ✓ No update anomalies — change email in one row, consistent everywhere
  ✓ Storage efficiency — no duplicate data
  ✓ Data integrity — foreign keys enforce referential integrity
  ✓ Write performance — single row update, no cascading writes

Cost:
  ✗ Read joins — "show order with customer info and all items" = 3-table JOIN
  ✗ Query complexity — aggregate queries require complex SQL
```

### 2.2 Denormalization

Denormalization intentionally introduces redundancy to optimise for read performance — trading write complexity for read simplicity.

```java
// Normalised: requires JOIN to read order with customer
SELECT o.id, c.name, c.email, oi.quantity, p.name AS product
FROM orders o
JOIN customers c    ON o.customer_id = c.id
JOIN order_items oi ON oi.order_id = o.id
JOIN products p     ON oi.product_id = p.id
WHERE o.id = 123;
// Efficient with indexes, but 3 JOINs at scale = latency

// Denormalised: store customer snapshot in order document
{
  "order_id": 123,
  "customer": {                  // snapshot at time of order
    "id": 456,
    "name": "Alice",
    "email": "alice@example.com"
  },
  "items": [
    { "product_id": 1, "name": "Widget A", "qty": 2, "price": 9.99 },
    { "product_id": 2, "name": "Widget B", "qty": 1, "price": 14.99 }
  ],
  "total": 34.97
}
// Single document fetch — no joins needed

// Denormalised SQL: add computed columns for common aggregates
ALTER TABLE orders ADD COLUMN total_amount DECIMAL(10,2);
ALTER TABLE orders ADD COLUMN item_count INT;
-- Update on every order_items change (trigger or application logic)
```

**When to denormalize:**

```
Read-heavy workloads:
  If reads >> writes (10:1 ratio or more), denormalize to speed up reads.
  Write overhead of maintaining redundant data is worth the read gain.

Pre-computed aggregates:
  "Total order value" calculated from 50 items every read → compute once on write.
  Event systems: maintain count columns updated incrementally.

Document self-containment:
  Data accessed together should be stored together (document model).
  Order history never needs current customer email — store snapshot.

NoSQL requirements:
  Cassandra table design: each query = one table.
  "Get user's last 20 posts ordered by time" → posts_by_user table
  "Get all posts liked by user" → posts_liked_by_user table
  Same post data stored in multiple tables (write to all, read from one).
```

### 2.3 Normalization vs Denormalization decision

```
Normalize when:
  ✓ Write-heavy workloads (orders, transactions, audit logs)
  ✓ Data changes frequently (user profiles, prices — single update needed)
  ✓ Ad-hoc query requirements (analytics — can't pre-compute all access patterns)
  ✓ Storage is constrained
  ✓ Data integrity is paramount

Denormalize when:
  ✓ Read-heavy workloads (product catalogs, news feeds)
  ✓ Query speed is critical and specific access patterns are known
  ✓ Data rarely changes (order history, archived records)
  ✓ Using NoSQL where JOINs aren't available
  ✓ Reporting/analytics pre-computation (summary tables, materialized views)

PostgreSQL materialized views — best of both worlds:
  Store normalized source data + pre-computed views for read performance
```

```sql
-- Normalized source tables remain clean
-- Materialized view pre-computes expensive aggregation
CREATE MATERIALIZED VIEW order_summary AS
SELECT
    c.id       AS customer_id,
    c.name     AS customer_name,
    COUNT(o.id) AS total_orders,
    SUM(o.total_amount) AS lifetime_value,
    MAX(o.created_at)   AS last_order_date
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id
GROUP BY c.id, c.name;

CREATE UNIQUE INDEX ON order_summary (customer_id); -- supports fast lookup

-- Refresh periodically or on demand
REFRESH MATERIALIZED VIEW CONCURRENTLY order_summary; -- non-blocking with CONCURRENTLY
```

## 3. ACID vs BASE

### 3.1 ACID

ACID (Atomicity, Consistency, Isolation, Durability) defines the guarantees that relational databases provide to ensure data validity even in the face of failures and concurrent access.

```
Atomicity:    Transaction is all-or-nothing.
              If any part fails, entire transaction rolls back.
              "Transfer $100: debit A AND credit B — never one without the other"

Consistency:  Transaction leaves DB in a valid state.
              All defined rules (constraints, cascades, triggers) are satisfied.
              "Account balance cannot go negative — constraint enforced"

Isolation:    Concurrent transactions appear to execute serially.
              One transaction doesn't see another's intermediate state.
              Levels: Read Uncommitted → Read Committed → Repeatable Read → Serializable

Durability:   Committed transactions survive crashes.
              Written to disk (WAL — Write-Ahead Log) before ACK-ing client.
              "Power failure after COMMIT → data still there on restart"
```

**Isolation levels in practice:**

```sql
-- Isolation levels control what concurrent transactions can see
-- Lower isolation = better concurrency, more anomalies possible
-- Higher isolation = fewer anomalies, more locking/blocking

-- Read Committed (PostgreSQL default):
--   Sees committed data from other transactions at time of each statement.
--   Prevents dirty reads. Allows non-repeatable reads and phantom reads.
BEGIN;
SELECT balance FROM accounts WHERE id = 1; -- reads 1000
-- Another transaction commits, changes balance to 800
SELECT balance FROM accounts WHERE id = 1; -- reads 800 (non-repeatable read)
COMMIT;

-- Repeatable Read:
--   Snapshot taken at start of transaction — same rows return same data.
--   Prevents dirty reads + non-repeatable reads. Allows phantom reads (MySQL).
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
BEGIN;
SELECT balance FROM accounts WHERE id = 1; -- reads 1000
-- Another transaction commits, changes to 800 — invisible to THIS transaction
SELECT balance FROM accounts WHERE id = 1; -- still reads 1000
COMMIT;

-- Serializable (strongest):
--   Transactions appear to execute one at a time, in serial order.
--   Prevents all anomalies but reduces concurrency (blocking or abort on conflict).
--   Use for: financial reconciliation, inventory reservations
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- PostgreSQL uses SSI (Serializable Snapshot Isolation) — detection not locking:
-- Multiple readers/writers proceed without blocking; conflict detected at commit.
-- Much better than lock-based serializable in most workloads.
```

```java
// Spring — specify isolation level per transaction
@Transactional(isolation = Isolation.SERIALIZABLE)
public void transferFunds(Long fromId, Long toId, BigDecimal amount) {
    Account from = accountRepo.findByIdWithLock(fromId); // SELECT FOR UPDATE
    Account to   = accountRepo.findByIdWithLock(toId);

    if (from.getBalance().compareTo(amount) < 0)
        throw new InsufficientFundsException();

    from.setBalance(from.getBalance().subtract(amount));
    to.setBalance(to.getBalance().add(amount));

    accountRepo.save(from);
    accountRepo.save(to);
    // Atomic: both updates commit or both roll back
}
```

### 3.2 BASE

BASE (Basically Available, Soft state, Eventually consistent) describes the alternative model chosen by distributed databases that prioritise availability and partition tolerance over strict consistency.

```
Basically Available:
  System guarantees availability — every request receives a response
  (not necessarily the most recent or consistent one).
  Read may return stale data; write is always accepted.

Soft State:
  System state may change over time without input (replicas converging).
  Data flowing through the system may be in transit or partially applied.

Eventually Consistent:
  Given no new updates, all replicas will converge to the same value.
  Convergence time: milliseconds to seconds in normal operation.
  Applications must tolerate reading stale data.

Example: Cassandra with replication factor 3, consistency QUORUM
  Write: sent to 3 nodes, waits for 2 ACKs (majority)
  Read:  reads from 2 nodes, returns latest version
  → Any node failure still allows writes and reads
  → Brief window after write where some reads return old value (stale replica)
```

**Eventual consistency challenges:**

```
Read-your-writes:
  User updates profile photo → writes to replica 1
  Next request routed to replica 2 (lagging) → sees old photo
  Solution: sticky routing, vector clocks, client-side version tracking

Lost updates (concurrent writes):
  User A and User B both read count=5 simultaneously
  Both increment locally → both write count=6
  One write overwrites the other → lost update
  Solution: last-write-wins (LWW) with timestamp, CRDTs, optimistic locking

Read anomalies:
  Item visible in one region, not yet replicated to another
  Order placed but inventory not yet decremented at all nodes
  Solution: design for eventual consistency (idempotent operations, reconciliation)
```

### 3.3 ACID vs BASE comparison

|Property|ACID|BASE|
|---|---|---|
|Consistency|Strong (immediate)|Eventual (convergent)|
|Availability on partition|Sacrificed (refuse operations)|Maintained (accept with staleness)|
|Concurrency model|Transactions with isolation|Optimistic / last-write-wins / CRDTs|
|Complexity|In database (simple app code)|In application (must handle conflicts)|
|Use case|Financial, inventory, user accounts|Social feeds, activity logs, analytics|
|Examples|PostgreSQL, MySQL, Oracle|Cassandra, DynamoDB, CouchDB|

## 4. CAP Theorem

### 4.1 The theorem

The CAP theorem states that a distributed system can provide at most **two** of three guarantees simultaneously:

```
C — Consistency:     Every read receives the most recent write or an error.
                     All nodes see the same data at the same time.

A — Availability:    Every request receives a (non-error) response —
                     but it might not contain the most recent write.

P — Partition        The system continues to operate even when network
    Tolerance:       partitions (lost messages) occur between nodes.

The partition trade-off:
  Network partitions WILL occur in any distributed system.
  During a partition, choose: refuse requests (CP) or serve possibly-stale data (AP).
  After partition heals, eventually consistent systems converge (AP → eventual C).

CA (no partition tolerance) = single-node system only.
  Multi-node systems MUST tolerate partitions → real choice is CP vs AP.
```

```
CP Systems (Consistent + Partition Tolerant):
  During partition: refuse writes or reads to avoid returning stale data.
  Returns error or blocks until partition heals.
  Examples: ZooKeeper, HBase, etcd, Redis (single master)
  Use: coordination, configuration, leader election, distributed locks

AP Systems (Available + Partition Tolerant):
  During partition: continue serving reads (possibly stale) and accepting writes.
  Conflicting writes resolved later (convergence).
  Examples: Cassandra, DynamoDB, CouchDB, DNS
  Use: user-facing services where availability > consistency

CA (theoretical only):
  Single-node relational databases (PostgreSQL on one server = no partition risk)
  "CA" only makes sense when there's no distribution — partitions can't occur.
```

### 4.2 PACELC — the extended model

CAP only describes behavior during a partition. PACELC extends it to also describe the latency-consistency tradeoff in normal operation (no partition):

```
PACELC: "If Partition: choose A or C. Else (normal): choose L or C"

During partition (P):   A (availability) vs C (consistency)  [same as CAP]
During normal ops (E):  L (latency)      vs C (consistency)

System              Partition    Normal ops    Tradeoff
──────────────────────────────────────────────────────
DynamoDB            AP           EL            Availability + Low latency
Cassandra           AP           EL            Availability + Low latency (tunable)
MongoDB (primary)   CP           EC            Consistency + Higher latency
PostgreSQL          CP           EC            Consistency + Higher latency
ZooKeeper/etcd      CP           EC            Strong consistency, may block
CockroachDB         CP           EC            Distributed SQL, serializability

Cassandra tunable consistency:
  Consistency level ONE:    fastest, least consistent (read/write 1 node)
  Consistency level QUORUM: balanced (read/write majority)
  Consistency level ALL:    slowest, fully consistent (all nodes must respond)
  → Application chooses per-operation based on data criticality
```

### 4.3 Practical implications

```
"I need both strong consistency AND high availability" — usually possible.
  CAP applies during a network partition — a relatively rare event.
  In normal operation, a well-designed system is both consistent and available.
  Design for the partition case specifically; don't let it drive all decisions.

Strong consistency without partition sacrifice:
  Google Spanner: uses TrueTime (atomic clocks + GPS) to provide external
  consistency globally with ~1-10ms latency. Technically still CP — during
  partition it blocks — but partitions last <1ms thanks to physical clocks.
  CockroachDB: similar approach, open source, slightly higher latency.

Choosing consistency level by operation:
  Reading a product description → stale OK → Cassandra ONE
  Reading inventory count before purchase → must be accurate → QUORUM or ALL
  Writing a payment record → durability critical → ALL replicas must confirm
```

## 5. Choosing Data Stores

### 5.1 Decision framework

```
Step 1: Define access patterns
  What are the top 5 query types?
  Are they key lookups, range scans, full-text searches, aggregations, or traversals?
  Read/write ratio? Latency requirements?

Step 2: Define consistency and durability requirements
  Can users tolerate reading stale data (eventual consistency OK)?
  What's the cost of data loss? (financial = zero tolerance; activity log = OK)
  Do operations need to span multiple entities atomically?

Step 3: Define scale requirements
  Data volume now and in 2 years?
  Writes per second? Reads per second?
  Will it fit on one well-provisioned server?

Step 4: Match to data store category
  Then pick a specific product based on operational familiarity, cloud provider,
  licensing cost, and team expertise.
```

### 5.2 Data store selection guide

```
Relational (PostgreSQL, MySQL):
  ✓ Structured data with well-defined relationships
  ✓ ACID transactions across entities
  ✓ Ad-hoc and analytical queries
  ✓ Strong consistency required
  ✓ Default choice for new services
  Data: user accounts, orders, financial records, inventory

Document (MongoDB, Firestore):
  ✓ Semi-structured / self-contained entities (no joins needed)
  ✓ Schema evolves frequently (product attributes differ by category)
  ✓ Nested data is natural (order with embedded items)
  ✗ Avoid when: strong cross-document transactions needed
  Data: product catalogs, user profiles, CMS content, game state

Key-Value (Redis, DynamoDB):
  ✓ O(1) lookup by a single key
  ✓ High throughput, low latency required
  ✓ Simple value types (strings, counters, hashes, sets)
  Redis: ephemeral data, caching, rate limiting, pub/sub
  DynamoDB: durable KV with optional sort key for range scans
  Data: sessions, caches, rate limiters, feature flags, leaderboards

Wide-Column (Cassandra, HBase):
  ✓ Write-heavy time-series data
  ✓ Massive horizontal scale (petabytes)
  ✓ Specific, pre-defined access patterns
  ✗ Avoid for: ad-hoc queries, complex aggregations, frequent schema changes
  Data: IoT sensor readings, activity feeds, audit logs, messaging history

Search (Elasticsearch, OpenSearch, Solr):
  ✓ Full-text search with relevance ranking
  ✓ Faceted filtering (filter by category AND price range AND brand)
  ✓ Aggregations / analytics dashboards
  ✗ Not a primary data store — sync from authoritative source
  Data: product search, log analytics (ELK), document search

Graph (Neo4j, Amazon Neptune):
  ✓ Multi-hop relationship traversal
  ✓ Data is fundamentally a network (social graph, knowledge graph)
  ✗ Avoid for: simple lookups, non-graph data
  Data: social connections, fraud detection, recommendation engines

Time-Series (InfluxDB, TimescaleDB, Prometheus):
  ✓ High-frequency write of timestamped measurements
  ✓ Time-range queries and downsampling
  ✓ Automatic retention policies (keep 1 year of data)
  Data: metrics, monitoring, IoT sensors, financial tick data

Object Storage (S3, GCS, Azure Blob):
  ✓ Binary large objects (images, videos, backups, ML datasets)
  ✓ Virtually unlimited storage at low cost
  ✓ CDN integration for global delivery
  ✗ Not queryable — metadata only in search layer
  Data: user uploads, static assets, data lake storage, backups
```

### 5.3 Concrete system design examples

```
E-commerce platform:
  Users, orders, products   → PostgreSQL (ACID for financial data)
  Product search            → Elasticsearch (full-text + faceted filtering)
  Sessions, carts           → Redis (fast KV, TTL-based eviction)
  User activity feed        → Cassandra (write-heavy, time-ordered, huge scale)
  Product images            → S3 + CloudFront CDN

Social media platform:
  User accounts, follows    → PostgreSQL (relations, graph-light)
  Posts, comments           → MongoDB (document, flexible schema)
  News feed (pre-computed)  → Redis sorted set (score = timestamp)
  Connections graph         → Neo4j or DynamoDB (if < 3 hops needed)
  Notifications             → Cassandra or DynamoDB
  Media files               → S3 + CDN

Real-time analytics:
  Raw events (write path)   → Kafka (durable streaming log)
  Event store (queryable)   → Cassandra or ClickHouse (write-optimised)
  Aggregates / dashboards   → Elasticsearch or ClickHouse
  ML feature store          → Redis (low-latency feature serving)
  Historical data lake      → S3 + Parquet (Athena, BigQuery, Snowflake)
```

## 6. Polyglot Persistence

Polyglot persistence is the practice of using multiple different database technologies within a single application, each chosen to suit the specific data and access patterns of the component it serves.

### 6.1 Architecture pattern

```
                     ┌─────────────────────────────────────┐
                     │         API Gateway / BFF            │
                     └─────────────┬───────────────────────┘
                                   │
           ┌───────────────────────┼──────────────────────────┐
           │                       │                          │
    ┌──────▼──────┐         ┌──────▼──────┐          ┌───────▼──────┐
    │  User       │         │  Product    │          │  Activity    │
    │  Service    │         │  Service    │          │  Service     │
    └──────┬──────┘         └──────┬──────┘          └───────┬──────┘
           │                       │                          │
    ┌──────▼──────┐         ┌──────▼──────┐          ┌───────▼──────┐
    │ PostgreSQL  │         │ MongoDB +   │          │  Cassandra   │
    │ (accounts,  │         │ Elastic     │          │  (events,    │
    │  billing)   │         │ (catalog,   │          │   timeseries)│
    └─────────────┘         │  search)    │          └──────────────┘
                            └─────────────┘
           ┌───────────────────────────────────────────────────┐
           │                    Redis                           │
           │    (sessions · rate limits · caches · pubsub)     │
           └───────────────────────────────────────────────────┘
                     ┌─────────────────────────────┐
                     │      S3 / Object Storage     │
                     │  (images, uploads, backups)  │
                     └─────────────────────────────┘
```

### 6.2 Data synchronisation between stores

When data needs to be in multiple stores (e.g. product data in PostgreSQL for write authority and Elasticsearch for search), synchronisation patterns become critical.

```java
// Pattern 1: Dual-write (simple but dangerous)
// Risk: one write succeeds, other fails → stores diverge
@Transactional
public Product saveProduct(Product product) {
    Product saved = productRepository.save(product);   // PostgreSQL
    searchIndexer.index(saved);                         // Elasticsearch
    // If Elasticsearch write fails after PostgreSQL commits → inconsistent!
    return saved;
}

// Pattern 2: Transactional Outbox (reliable — preferred)
// Write event to same DB transaction as the data change
// Background process reads outbox and publishes to downstream stores
@Transactional
public Product saveProduct(Product product) {
    Product saved = productRepository.save(product);
    // Write outbox event IN SAME TRANSACTION — atomic with the data change
    outboxRepository.save(OutboxEvent.builder()
        .aggregateType("Product")
        .aggregateId(saved.getId().toString())
        .eventType("ProductUpdated")
        .payload(toJson(saved))
        .build());
    return saved;
    // Single DB commit: product + outbox event together
}

// Outbox relay (polling or CDC)
@Scheduled(fixedDelay = 100)
public void relayOutboxEvents() {
    List<OutboxEvent> events = outboxRepository.findUnpublished();
    for (OutboxEvent event : events) {
        try {
            if ("ProductUpdated".equals(event.getEventType())) {
                Product product = fromJson(event.getPayload(), Product.class);
                searchIndexer.index(product);    // Elasticsearch
                kafkaTemplate.send("products", event.getPayload()); // Kafka
            }
            event.markPublished();
            outboxRepository.save(event);
        } catch (Exception e) {
            event.incrementRetries();
            outboxRepository.save(event);
        }
    }
}

// Pattern 3: Change Data Capture (CDC) — production preferred
// Debezium reads PostgreSQL WAL (write-ahead log) and publishes changes to Kafka
// Elasticsearch Kafka Connect sink consumes Kafka → updates search index
// Zero application code change; guaranteed at-least-once delivery
```

```yaml
  "config": {
    "connector.class": "io.debezium.connector.postgresql.PostgresConnector",
    "database.hostname": "postgres",
    "database.dbname": "mydb",
    "table.include.list": "public.products",
    "topic.prefix": "cdc",
    "plugin.name": "pgoutput"
    # → publishes to Kafka topic "cdc.public.products" on every INSERT/UPDATE/DELETE
  }
}
```

### 6.3 Consistency challenges in polyglot systems

```
Challenge: Two stores, temporarily out of sync
  PostgreSQL: product.price = $9.99 (committed 500ms ago)
  Elasticsearch: product.price = $12.99 (not yet replicated)
  User searches, sees $12.99, clicks, lands on product page showing $9.99.

Solutions:
  1. Accept eventual consistency (most common — small window, acceptable UX)
  2. Read from authoritative store for critical fields (price from PostgreSQL)
  3. Version numbers: reject ES updates older than current PG version
  4. Reduce replication lag: CDC via Debezium < 100ms typical lag

Challenge: Distributed queries across stores
  "Get users who purchased product X in the last 30 days"
  Orders in PostgreSQL, user profiles in MongoDB.

Solutions:
  1. Denormalise: copy relevant fields to one store at write time
  2. API composition: fetch from both stores in application, join in code
  3. Data warehouse: replicate both to a single analytics store (BigQuery, Snowflake)
     for cross-system queries
```

### 6.4 Operational complexity of polyglot persistence

```
More databases = more operational burden:

Per database, you need:
  ✓ Monitoring and alerting (different metrics per DB type)
  ✓ Backup and recovery procedures
  ✓ Security (authentication, network isolation, encryption at rest)
  ✓ Capacity planning
  ✓ Team expertise (who understands Cassandra data modelling?)
  ✓ Connection pooling and client configuration
  ✓ Schema/index management and migration tooling

Mitigation:
  Managed cloud services (RDS, DynamoDB, ElastiCache, OpenSearch) shift
  operational burden to cloud provider — strongly preferred over self-managed.

Rule: don't add a new database technology until the existing ones are
  clearly insufficient for the use case. Start with PostgreSQL.
  Add Redis when you need caching/sessions.
  Add Elasticsearch when you need full-text search.
  Add Cassandra when you need massive write-scale time-series.
  Each addition must justify its operational cost.
```

## 7. Interview Questions

**Q: When would you use a NoSQL database over PostgreSQL, and what are the risks of that decision?**

A: I'd choose NoSQL for specific, justified reasons rather than as a default. Document databases like MongoDB suit services with self-contained, schema-flexible entities — a product catalog where different product types have different attributes, or a CMS where content structures evolve. Wide-column stores like Cassandra suit massive write-heavy time-series workloads (IoT, activity logs) that need horizontal write scale beyond what a single PostgreSQL primary can provide. Key-value stores like Redis suit ephemeral or latency-critical data. The risks: schema flexibility becomes schema chaos without discipline (every "optional" field eventually exists in production data in inconsistent states); lack of JOINs forces application-level joins or denormalization that multiplies write complexity; most NoSQL systems offer weaker transactional guarantees, pushing conflict resolution into application code. I'd always ask whether a well-indexed PostgreSQL with read replicas would suffice before adding a new database technology.

---

**Q: Explain the CAP theorem and why "CA" systems don't exist in distributed computing.**

A: CAP says a distributed system can guarantee at most two of: Consistency (every read gets the latest write), Availability (every request gets a non-error response), and Partition Tolerance (the system works despite network partitions). Network partitions — nodes unable to communicate — are an inevitable reality in any multi-node system: cables fail, switches reboot, network congestion causes packet loss. Since partitions will occur, partition tolerance is not optional — it's the cost of distribution. During a partition, the real choice is: refuse requests until the partition heals and data is consistent (CP), or continue serving potentially stale data (AP). "CA" would require guaranteeing both consistency and availability during a partition, which is a logical contradiction — if two nodes can't communicate, they cannot guarantee they have the same data (consistency violation) without refusing requests (availability violation). A single-node PostgreSQL is effectively CA because with one node there are no partitions, but as soon as you add replication, you're in the CP/AP trade-off.

---

**Q: What is the difference between normalization and denormalization, and when do you choose each?**

A: Normalization organises data to eliminate redundancy — each fact is stored once, referenced via foreign keys. It optimises for write correctness (update one row, consistent everywhere), storage efficiency, and data integrity. Denormalization intentionally duplicates data to optimise for read performance — storing a product name inside an order record so the order can be fetched without joining to the products table. I normalize by default for write-heavy transactional data where correctness matters most: user accounts, orders, financial records. I denormalize for read-heavy workloads with known, fixed access patterns: a news feed that always shows the same fields, an order history page that shows a snapshot of what was ordered (not the current product details), or a Cassandra table designed around one specific query. The best of both worlds is PostgreSQL materialized views: keep normalized source tables for write integrity, create pre-computed views for read-heavy dashboards, and refresh them on a schedule or on demand.

---

**Q: How do you keep data synchronized between multiple databases in a polyglot persistence architecture?**

A: The naive approach is dual-writing from application code — write to PostgreSQL and Elasticsearch in the same request. This is dangerous: if the Elasticsearch write fails after PostgreSQL commits, the stores diverge with no automatic reconciliation. The transactional outbox pattern solves this: write the data change and an "outbox event" in the same database transaction. A background relay process reads uncommitted outbox events and publishes them to downstream stores, retrying on failure. This guarantees at-least-once delivery because the outbox event is durable in the primary database. For production at scale, Change Data Capture via Debezium reads the database's write-ahead log directly — zero application changes required, sub-100ms lag, and guaranteed capture of every committed change including those from migrations or direct SQL. The downstream stores (Elasticsearch, Kafka, read replicas) are driven from the CDC stream, treating PostgreSQL as the single source of truth.

---

**Q: What is the practical difference between ACID and BASE, and how does it affect application design?**

A: ACID pushes correctness enforcement into the database layer — the database guarantees that concurrent operations produce results as if they ran serially, and that committed data survives crashes. Application code can treat the database as a black box of correctness: write business logic, let the database handle isolation and atomicity. BASE systems push consistency management into the application layer. An eventually consistent database may return stale data — the application must decide whether stale data is acceptable for this operation, or if it needs a stronger read (QUORUM in Cassandra). Concurrent writes may conflict — the application must define conflict resolution (last-write-wins, CRDTs, versioned updates). The practical impact: ACID lets you write straightforward service logic for financial operations; BASE requires designing idempotent operations (so retries are safe), handling read-your-writes inconsistency (user writes then immediately reads), and accepting that some data anomalies will reach users. The right choice depends entirely on whether the application can tolerate eventual consistency — activity feeds can, bank balances cannot.

## 8. Key Takeaways

- **Default to PostgreSQL**: it handles most applications' lifetime without the operational complexity of NoSQL; add other databases only when PostgreSQL is genuinely insufficient for a specific use case
- **Normalization vs denormalization is a read/write tradeoff**: normalize for write-heavy, consistency-critical data; denormalize for read-heavy, known-access-pattern data; use materialized views for the middle ground
- **ACID** pushes consistency into the database (simple application code); **BASE** pushes it into the application (complex conflict resolution, idempotency requirements)
- **CAP is about partition behavior**: real choice is CP (refuse during partition) vs AP (serve stale during partition); "CA" is only a single node
- **PACELC extends CAP**: during normal operation, the tradeoff is latency (L) vs consistency (C) — tunable in systems like Cassandra
- **Polyglot persistence is powerful but costly**: each additional database type multiplies operational burden — use managed cloud services and justify every addition
- **Synchronise data across stores with CDC** (Debezium + Kafka) or the transactional outbox pattern — never dual-write directly from application code
- **Shard key is the most critical and hardest-to-reverse decision in NoSQL**: choose based on cardinality, distribution, and query alignment
