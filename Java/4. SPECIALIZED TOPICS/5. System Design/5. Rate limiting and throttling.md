# Rate Limiting and Throttling

## Overview

Rate limiting controls how many requests a client can make in a given time window. Throttling is the enforcement action — slowing down or rejecting requests that exceed limits. Together they protect services from overload, prevent abuse, enforce fair usage, and enable monetisation through API tiers.

**Why rate limiting is non-negotiable at scale:**

```
Without rate limiting:
  One misbehaving client → exhausts thread pool → all clients degraded
  DDoS or traffic spike → cascade failure across services
  Expensive compute per request → unlimited cost with no billing cap

With rate limiting:
  Each client gets guaranteed fair share of capacity
  Burst is absorbed (token bucket) or smoothed (leaky bucket)
  Overload converted to predictable 429 responses instead of timeouts

Where to enforce limits:
  API Gateway (Kong, AWS API GW):  global, before traffic reaches services
  Reverse proxy (Nginx, Envoy):    per-upstream, before application code
  Application (Spring, Filter):    fine-grained, per-user business logic
  Database (connection pools):     prevents DB saturation from above layers

Response codes:
  429 Too Many Requests:   rate limit exceeded — retry after delay
  503 Service Unavailable: overloaded, shed load — longer backoff required
  Retry-After header:      seconds until client may retry (mandatory on 429)
```

## 1. Token Bucket Algorithm

The token bucket is the most widely used rate limiting algorithm. A bucket holds up to `capacity` tokens. Tokens refill at a fixed rate (`refillRate` per second). Each request consumes one or more tokens. If the bucket has tokens, the request is allowed; if empty, the request is rejected or queued.

### 1.1 Concept

```
Token Bucket visualised:

  Capacity = 10 tokens
  Refill = 2 tokens/second

  Time 0s:  [●●●●●●●●●●]  10 tokens — full
  Request burst: 7 requests → consumes 7 tokens
  Time 0s:  [●●●        ]   3 tokens remaining

  Time 1s:  [●●●●●      ]   5 tokens (3 + 2 refilled)
  Time 2s:  [●●●●●●●    ]   7 tokens (5 + 2 refilled)

  Properties:
    Allows bursts up to capacity (absorbs traffic spikes)
    Sustained rate limited to refillRate tokens/second
    Unused capacity accumulates (up to capacity ceiling)
    Well-suited for: API rate limits, user quotas, burstable network I/O
```

### 1.2 In-memory implementation

```java
public class TokenBucket {
    private final long capacity;          // max tokens
    private final double refillRate;      // tokens per nanosecond
    private double tokens;
    private long lastRefillNanos;

    public TokenBucket(long capacity, long refillPerSecond) {
        this.capacity = capacity;
        this.refillRate = (double) refillPerSecond / 1_000_000_000L; // per nanosecond
        this.tokens = capacity;     // start full
        this.lastRefillNanos = System.nanoTime();
    }

    public synchronized boolean tryConsume(int requested) {
        refill();
        if (tokens >= requested) {
            tokens -= requested;
            return true;   // request allowed
        }
        return false;      // insufficient tokens — reject
    }

    private void refill() {
        long now = System.nanoTime();
        double elapsed = now - lastRefillNanos;
        double newTokens = elapsed * refillRate;
        tokens = Math.min(capacity, tokens + newTokens);  // cap at capacity
        lastRefillNanos = now;
    }

    // How long until N tokens are available (for Retry-After header)
    public long nanosUntilAvailable(int needed) {
        refill();
        if (tokens >= needed) return 0;
        double deficit = needed - tokens;
        return (long) (deficit / refillRate);
    }
}
```

### 1.3 Redis-based token bucket (distributed)

```java
// Atomic token bucket via Redis Lua script — no race conditions across nodes
@Service
@RequiredArgsConstructor
public class RedisTokenBucket {
    private final RedisTemplate<String, String> redis;

    // Lua script: refill + consume atomically
    private static final String CONSUME_SCRIPT = """
        local key        = KEYS[1]
        local capacity   = tonumber(ARGV[1])
        local refillRate = tonumber(ARGV[2])   -- tokens per second
        local requested  = tonumber(ARGV[3])
        local now        = tonumber(ARGV[4])   -- current time in seconds (float)

        local data = redis.call('HMGET', key, 'tokens', 'last_refill')
        local tokens     = tonumber(data[1]) or capacity
        local lastRefill = tonumber(data[2]) or now

        -- Refill based on elapsed time
        local elapsed = math.max(0, now - lastRefill)
        tokens = math.min(capacity, tokens + elapsed * refillRate)

        local allowed = 0
        if tokens >= requested then
            tokens = tokens - requested
            allowed = 1
        end

        redis.call('HMSET', key,
            'tokens',      tokens,
            'last_refill', now)
        redis.call('EXPIRE', key, math.ceil(capacity / refillRate) * 2)

        return { allowed, math.floor(tokens) }
        """;

    public RateLimitResult tryConsume(String clientId, long capacity,
                                       long refillPerSecond, int requested) {
        String key = "ratelimit:token:" + clientId;
        double now = System.currentTimeMillis() / 1000.0;

        List<Long> result = redis.execute(
            RedisScript.of(CONSUME_SCRIPT, List.class),
            List.of(key),
            String.valueOf(capacity),
            String.valueOf(refillPerSecond),
            String.valueOf(requested),
            String.format("%.3f", now)
        );

        boolean allowed = result.get(0) == 1L;
        long remaining = result.get(1);
        long retryAfterSeconds = allowed ? 0 :
            (long) Math.ceil((requested - remaining) / (double) refillPerSecond);

        return new RateLimitResult(allowed, remaining, retryAfterSeconds);
    }
}
```

### 1.4 Token bucket in Spring filter

```java
@Component
@Order(Ordered.HIGHEST_PRECEDENCE)
@RequiredArgsConstructor
public class RateLimitFilter extends OncePerRequestFilter {
    private final RedisTokenBucket tokenBucket;
    private final RateLimitConfig config;

    @Override
    protected void doFilterInternal(HttpServletRequest req,
                                     HttpServletResponse res,
                                     FilterChain chain) throws IOException, ServletException {
        String clientId = resolveClientId(req);
        RateLimitTier tier = config.getTierForClient(clientId);

        RateLimitResult result = tokenBucket.tryConsume(
            clientId, tier.getCapacity(), tier.getRefillRate(), 1);

        // Always add rate limit headers (even on success — RFC 6585)
        res.setHeader("X-RateLimit-Limit",     String.valueOf(tier.getCapacity()));
        res.setHeader("X-RateLimit-Remaining", String.valueOf(result.getRemaining()));
        res.setHeader("X-RateLimit-Reset",     String.valueOf(
            System.currentTimeMillis() / 1000 + result.getRetryAfterSeconds()));

        if (!result.isAllowed()) {
            res.setStatus(HttpStatus.TOO_MANY_REQUESTS.value());
            res.setHeader("Retry-After", String.valueOf(result.getRetryAfterSeconds()));
            res.setContentType(MediaType.APPLICATION_JSON_VALUE);
            res.getWriter().write("""
                {"error":"rate_limit_exceeded",
                 "message":"Too many requests. Retry after %d seconds.",
                 "retryAfter":%d}
                """.formatted(result.getRetryAfterSeconds(),
                               result.getRetryAfterSeconds()));
            return;
        }

        chain.doFilter(req, res);
    }

    private String resolveClientId(HttpServletRequest req) {
        // Priority: API key header → authenticated user ID → IP address
        String apiKey = req.getHeader("X-API-Key");
        if (apiKey != null) return "apikey:" + apiKey;

        Authentication auth = SecurityContextHolder.getContext().getAuthentication();
        if (auth != null && auth.isAuthenticated()) return "user:" + auth.getName();

        return "ip:" + getClientIp(req);
    }

    private String getClientIp(HttpServletRequest req) {
        String forwarded = req.getHeader("X-Forwarded-For");
        return forwarded != null ? forwarded.split(",")[0].trim()
                                 : req.getRemoteAddr();
    }
}
```

## 2. Leaky Bucket Algorithm

The leaky bucket processes requests at a constant output rate, regardless of input burst. Incoming requests fill a bucket (queue); the bucket drains at a fixed rate. If the bucket overflows (queue full), excess requests are dropped.

### 2.1 Concept

```
Leaky Bucket visualised:

  Bucket capacity = 10 (queue size)
  Leak rate       = 2 requests/second (processing rate)

  Input burst: 15 requests arrive instantly
    → 10 queued (bucket fills)
    → 5 dropped (bucket overflow)

  Processing: 2 requests/second dequeued → processed
  Output: perfectly smooth stream at 2 req/s regardless of input pattern

  vs Token Bucket:
    Token bucket:  allows burst up to capacity, then limits to refill rate
    Leaky bucket:  absorbs burst into queue, always outputs at constant rate
    → Token bucket: better for burstable APIs (user experience)
    → Leaky bucket: better for protecting downstream (database, external API)
```

### 2.2 Implementation

```java
public class LeakyBucket {
    private final int capacity;           // max queue size
    private final double leakRatePerMs;   // requests processed per millisecond
    private final Queue<Runnable> queue = new LinkedList<>();
    private final ScheduledExecutorService scheduler;
    private int currentSize = 0;

    public LeakyBucket(int capacity, int leakPerSecond) {
        this.capacity = capacity;
        this.leakRatePerMs = leakPerSecond / 1000.0;

        // Drain queue at fixed rate
        this.scheduler = Executors.newSingleThreadScheduledExecutor();
        long intervalMs = Math.max(1, (long)(1000.0 / leakPerSecond));
        scheduler.scheduleAtFixedRate(this::leak, 0, intervalMs, TimeUnit.MILLISECONDS);
    }

    // Attempt to enqueue a request — returns false if bucket is full (drop)
    public synchronized boolean tryEnqueue(Runnable request) {
        if (currentSize >= capacity) {
            return false;   // bucket full — drop request
        }
        queue.offer(request);
        currentSize++;
        return true;
    }

    // Drain one request from the bucket at fixed rate
    private synchronized void leak() {
        Runnable request = queue.poll();
        if (request != null) {
            currentSize--;
            request.run();   // process at controlled rate
        }
    }
}

// Leaky bucket as rate limiter (counter-based, no actual queue)
// Tracks "water level" — increments on request, decrements over time
public class LeakyBucketCounter {
    private double waterLevel = 0;
    private long lastLeakNanos = System.nanoTime();
    private final double capacity;
    private final double leakRatePerNano; // requests/nanosecond

    public LeakyBucketCounter(double capacity, double leakPerSecond) {
        this.capacity = capacity;
        this.leakRatePerNano = leakPerSecond / 1_000_000_000.0;
    }

    public synchronized boolean tryConsume() {
        leak();  // drain elapsed water first
        if (waterLevel < capacity) {
            waterLevel += 1.0;
            return true;
        }
        return false;
    }

    private void leak() {
        long now = System.nanoTime();
        double elapsed = now - lastLeakNanos;
        waterLevel = Math.max(0, waterLevel - elapsed * leakRatePerNano);
        lastLeakNanos = now;
    }
}
```

### 2.3 Token bucket vs leaky bucket

|Property|Token Bucket|Leaky Bucket|
|---|---|---|
|Burst handling|Allows burst up to capacity|Smooths burst into queue; drops overflow|
|Output rate|Variable (up to capacity burst, then refillRate)|Constant (always leakRate)|
|Latency|No queueing — immediate accept or reject|Queueing adds latency for bursting clients|
|Dropped requests|Immediate on empty bucket|Only on queue overflow|
|Use case|User-facing APIs (burst-friendly)|Protecting downstream services|
|Implementation complexity|Simpler|More complex (queue management)|

## 3. Fixed Window vs Sliding Window

### 3.1 Fixed window counter

```
Fixed window: count requests within discrete time windows (00:00-00:01, 00:01-00:02, ...)

  Window 1 (00:00 - 00:01):  requests: 95 / limit: 100 ← 95 allowed
  Window 2 (00:01 - 00:02):  requests:  0 / limit: 100

  Problem — boundary burst:
    00:00:59: 95 requests (window 1: 95 total)
    00:01:00: 95 requests (window 2: 95 total — NEW window, counter resets)
    → 190 requests in 2 seconds despite "100 per minute" limit!
    This is the "thunderclap" at window boundaries.
```

```java
// Fixed window with Redis INCR + EXPIRE
public class FixedWindowRateLimiter {
    private final RedisTemplate<String, String> redis;

    public boolean isAllowed(String clientId, int limit, Duration window) {
        // Key includes the window start time — automatically rotates
        long windowStart = System.currentTimeMillis() / window.toMillis();
        String key = "ratelimit:fixed:" + clientId + ":" + windowStart;

        // INCR is atomic — safe for concurrent access across nodes
        Long count = redis.opsForValue().increment(key);

        if (count == 1) {
            // First request in this window — set expiry
            redis.expire(key, window.plusSeconds(1));
        }

        return count <= limit;
    }

    // Atomic INCR + EXPIRE in one round trip via Lua
    private static final String INCR_SCRIPT = """
        local key   = KEYS[1]
        local limit = tonumber(ARGV[1])
        local ttl   = tonumber(ARGV[2])
        local count = redis.call('INCR', key)
        if count == 1 then
            redis.call('EXPIRE', key, ttl)
        end
        return count
        """;

    public boolean isAllowedAtomic(String clientId, int limit, Duration window) {
        long windowId = System.currentTimeMillis() / window.toMillis();
        String key = "ratelimit:fixed:" + clientId + ":" + windowId;

        Long count = redis.execute(
            RedisScript.of(INCR_SCRIPT, Long.class),
            List.of(key),
            String.valueOf(limit),
            String.valueOf(window.toSeconds() + 1)
        );
        return count != null && count <= limit;
    }
}
```

### 3.2 Sliding window log

```
Sliding window log: store timestamp of each request; count requests in [now - window, now]

  Limit: 100 requests per minute
  At time T:  requests in log from [T-60s, T] = 63 → allowed (< 100)
  At time T+1: remove expired entries, add new → 64 → allowed

  Exact: no boundary burst problem
  Cost:  O(requests in window) memory per client — impractical for high-rate clients
         (100 req/s limit, 1M clients = 6B entries in Redis)
  Use:   low-rate limits (10/day, 1000/hour) where log is small
```

```java
// Sliding window log with Redis sorted set (score = timestamp)
public class SlidingWindowLogLimiter {
    private final RedisTemplate<String, String> redis;

    public boolean isAllowed(String clientId, int limit, Duration window) {
        String key = "ratelimit:swlog:" + clientId;
        long now = System.currentTimeMillis();
        long windowStart = now - window.toMillis();

        // Atomic Lua: remove expired, count, conditionally add
        String script = """
            local key         = KEYS[1]
            local now         = tonumber(ARGV[1])
            local windowStart = tonumber(ARGV[2])
            local limit       = tonumber(ARGV[3])
            local ttl         = tonumber(ARGV[4])

            redis.call('ZREMRANGEBYSCORE', key, 0, windowStart)
            local count = redis.call('ZCARD', key)

            if count < limit then
                redis.call('ZADD', key, now, now .. '-' .. math.random(1000000))
                redis.call('EXPIRE', key, ttl)
                return 1
            end
            return 0
            """;

        Long result = redis.execute(
            RedisScript.of(script, Long.class),
            List.of(key),
            String.valueOf(now),
            String.valueOf(windowStart),
            String.valueOf(limit),
            String.valueOf(window.toSeconds() + 1)
        );
        return Long.valueOf(1L).equals(result);
    }
}
```

### 3.3 Sliding window counter

```
Sliding window counter: approximates sliding window using two fixed window counters.
Trades small accuracy loss for O(1) memory per client.

  Current window count + Previous window count × overlap fraction

  Example: limit=100/min, window=1min
    At T=00:01:45 (45 seconds into current window):
      Previous window (00:00 - 00:01): count = 70
      Current window  (00:01 - 00:02): count = 40

    Overlap = (60-45)/60 = 0.25 of previous window still in sliding range
    Estimated count = 40 + 70 × 0.25 = 40 + 17.5 = 57.5 → 57 < 100 → allowed

  Memory: O(1) per client (just two counters + timestamps)
  Accuracy: typically within 0.003% of exact sliding window (empirically)
  Used by: Cloudflare, Nginx
```

```java
public class SlidingWindowCounterLimiter {
    private final RedisTemplate<String, String> redis;

    public boolean isAllowed(String clientId, int limit, Duration window) {
        long windowMs = window.toMillis();
        long now = System.currentTimeMillis();
        long currentWindowStart = (now / windowMs) * windowMs;
        long prevWindowStart = currentWindowStart - windowMs;

        String currKey = "ratelimit:swc:" + clientId + ":" + currentWindowStart;
        String prevKey = "ratelimit:swc:" + clientId + ":" + prevWindowStart;

        String script = """
            local currKey = KEYS[1]
            local prevKey = KEYS[2]
            local limit   = tonumber(ARGV[1])
            local now     = tonumber(ARGV[2])
            local windowStart = tonumber(ARGV[3])
            local windowMs    = tonumber(ARGV[4])
            local ttl     = tonumber(ARGV[5])

            local currCount = tonumber(redis.call('GET', currKey) or '0')
            local prevCount = tonumber(redis.call('GET', prevKey) or '0')

            -- Fraction of previous window that falls within current sliding window
            local elapsed  = now - windowStart
            local overlap  = (windowMs - elapsed) / windowMs
            local estimate = currCount + prevCount * overlap

            if estimate < limit then
                redis.call('INCR', currKey)
                redis.call('EXPIRE', currKey, ttl)
                return 1
            end
            return 0
            """;

        Long result = redis.execute(
            RedisScript.of(script, Long.class),
            List.of(currKey, prevKey),
            String.valueOf(limit),
            String.valueOf(now),
            String.valueOf(currentWindowStart),
            String.valueOf(windowMs),
            String.valueOf(window.toSeconds() * 2 + 1)
        );
        return Long.valueOf(1L).equals(result);
    }
}
```

### 3.4 Algorithm comparison

|Algorithm|Memory|Accuracy|Burst handling|Complexity|
|---|---|---|---|---|
|Fixed window|O(1)|Boundary burst possible|Allows double-rate at window edge|Simplest|
|Sliding window log|O(requests in window)|Exact|Exact|Simple but memory-hungry|
|Sliding window counter|O(1)|~99.997% accurate|Smoothed|Moderate|
|Token bucket|O(1)|Exact|Explicit burst headroom|Moderate|
|Leaky bucket|O(queue size)|Exact|Smoothed (dropped or queued)|Complex|

## 4. Distributed Rate Limiting

Single-server rate limiting breaks when the service scales horizontally. A client making 10 requests/second split evenly across 10 servers sees only 1 request/server — all pass a local 5 req/s limit.

### 4.1 Centralised store (Redis)

```
All nodes share a single Redis counter. Every request hits Redis before being processed.

  App-1 ──▶ ─────────────────
  App-2 ──▶    Redis Cluster  ──── shared counters, atomic operations
  App-3 ──▶ ─────────────────

  Advantages:
    Exact accuracy — single counter, no drift between nodes
    Consistent limits regardless of traffic distribution across nodes
    Supports per-user, per-API-key, per-IP with no local state

  Disadvantages:
    Redis round trip on every request (adds ~1ms latency)
    Redis is a single point of failure (mitigate: cluster, sentinel)
    Network partition: Redis down → allow-all or deny-all policy needed
```

```java
// Resilience4j + Redis rate limiter with fallback
@Service
@RequiredArgsConstructor
public class ResilientRateLimiter {
    private final RedisTokenBucket redisBucket;
    private final LocalTokenBucket localFallback;

    public boolean isAllowed(String clientId, RateLimitTier tier) {
        try {
            return redisBucket.tryConsume(
                clientId, tier.getCapacity(), tier.getRefillRate(), 1).isAllowed();
        } catch (RedisConnectionFailureException e) {
            // Redis unavailable: fall back to local bucket
            // Accept slightly inaccurate limits rather than rejecting all traffic
            log.warn("Redis unavailable, using local rate limiter for {}", clientId);
            return localFallback.tryConsume(clientId, tier);
        }
    }
}
```

### 4.2 Local + central hybrid (two-layer limiting)

```
Reduce Redis calls by pre-allocating token batches to each node.
Each node holds a local bucket of N tokens; replenishes from Redis when exhausted.

  Redis:  user:alice → 1000 tokens/min  (global budget)
  Node 1: local bucket → 100 tokens  ← pre-allocated batch
  Node 2: local bucket → 100 tokens  ← pre-allocated batch
  Node 3: local bucket → 100 tokens  ← pre-allocated batch

  Request hits Node 1 → deduct from local bucket (no Redis call)
  Node 1 local bucket empty → request 100 more from Redis global bucket
  Redis deducts 100 from global → grants to Node 1

  Redis calls: 1 per 100 requests (100x reduction in Redis load)
  Trade-off: user may briefly exceed limit if multiple nodes replenish simultaneously
             (acceptable for most use cases)
```

```java
@Service
public class TwoLayerRateLimiter {
    private final RedisTemplate<String, String> redis;
    private final int localBatchSize = 100;

    // Per-client local buckets
    private final ConcurrentHashMap<String, AtomicLong> localBuckets = new ConcurrentHashMap<>();
    private final ConcurrentHashMap<String, ReentrantLock> refillLocks = new ConcurrentHashMap<>();

    public boolean isAllowed(String clientId, long globalLimitPerMin) {
        AtomicLong local = localBuckets.computeIfAbsent(clientId, k -> new AtomicLong(0));

        // Fast path: deduct from local bucket (no Redis)
        if (local.decrementAndGet() >= 0) return true;

        // Local bucket empty — try to replenish from Redis
        ReentrantLock lock = refillLocks.computeIfAbsent(clientId, k -> new ReentrantLock());
        if (lock.tryLock()) {
            try {
                long granted = replenishFromRedis(clientId, globalLimitPerMin, localBatchSize);
                if (granted > 0) {
                    local.addAndGet(granted);   // restore local bucket
                    local.decrementAndGet();    // consume current request
                    return true;
                }
                return false;  // global budget exhausted
            } finally {
                lock.unlock();
            }
        }
        // Another thread is replenishing — current request rejected
        // (conservative: avoids over-granting during concurrent replenishment)
        return false;
    }

    private long replenishFromRedis(String clientId, long globalLimit, int requested) {
        String script = """
            local key     = KEYS[1]
            local limit   = tonumber(ARGV[1])
            local batch   = tonumber(ARGV[2])
            local window  = tonumber(ARGV[3])

            local current = tonumber(redis.call('GET', key) or '0')
            local available = math.min(limit - current, batch)
            if available > 0 then
                redis.call('INCRBY', key, available)
                redis.call('EXPIRE', key, window)
                return available
            end
            return 0
            """;

        Long granted = redis.execute(
            RedisScript.of(script, Long.class),
            List.of("ratelimit:global:" + clientId),
            String.valueOf(globalLimit),
            String.valueOf(requested),
            "61" // 61-second window
        );
        return granted != null ? granted : 0;
    }
}
```

## 5. API Quotas

Quotas are long-horizon rate limits — typically daily, monthly, or per-billing-cycle caps tied to business tiers rather than per-second throughput limits.

### 5.1 Quota vs rate limit

```
Rate Limit:  short window, protects infrastructure
  "max 100 requests/second per user"
  Enforced: real-time, millisecond precision
  Purpose:  prevent overload, ensure fair share

Quota:       long window, enforces business tier
  "max 10,000 API calls/month for free plan"
  Enforced:  eventually (end-of-window billing)
  Purpose:   monetisation, capacity planning

Both are often combined:
  Free tier:    100 req/min (rate limit) + 50,000/month (quota)
  Pro tier:     1,000 req/min (rate limit) + 5M/month (quota)
  Enterprise:   10,000 req/min + unlimited (quota)
```

### 5.2 Quota enforcement implementation

```java
@Entity
public class ApiQuota {
    @Id Long id;
    String clientId;
    String planId;           // free, pro, enterprise
    long monthlyLimit;       // max calls this billing cycle
    long usedCount;          // current cycle usage
    YearMonth billingCycle;  // current period
    Instant lastResetAt;
}

@Service
@RequiredArgsConstructor
public class QuotaService {
    private final ApiQuotaRepository quotaRepo;
    private final RedisTemplate<String, Long> redis;

    // Fast path: check Redis counter (updated incrementally)
    // Slow path: verify against DB (reconcile on discrepancy)
    public QuotaCheckResult checkAndConsume(String clientId) {
        String redisKey = "quota:" + clientId + ":" + YearMonth.now();
        Long used = redis.opsForValue().increment(redisKey);

        if (used == 1) {
            // First request this month — set expiry to end of month + buffer
            redis.expireAt(redisKey, endOfMonth().plusDays(2));
        }

        ApiQuota quota = quotaRepo.findByClientIdAndBillingCycle(
            clientId, YearMonth.now()).orElseThrow();

        if (used > quota.getMonthlyLimit()) {
            redis.opsForValue().decrement(redisKey); // don't count rejected requests
            return QuotaCheckResult.exceeded(quota.getMonthlyLimit(), used - 1);
        }

        // Async DB sync every 100 requests (avoid DB write per request)
        if (used % 100 == 0) syncUsageToDb(clientId, used);

        return QuotaCheckResult.allowed(quota.getMonthlyLimit(), used);
    }

    // Reset quota at start of new billing cycle
    @Scheduled(cron = "0 0 0 1 * *")  // 1st of every month
    public void resetMonthlyQuotas() {
        quotaRepo.findAll().forEach(q -> {
            q.setUsedCount(0);
            q.setBillingCycle(YearMonth.now());
            quotaRepo.save(q);
        });
        log.info("Monthly quotas reset for {} clients", quotaRepo.count());
    }

    private Instant endOfMonth() {
        return YearMonth.now().atEndOfMonth()
            .atTime(LocalTime.MAX).toInstant(ZoneOffset.UTC);
    }
}

// Combined rate limit + quota check in one filter

@Order(1)

                                     HttpServletResponse res,

        if (apiKey == null) { res.sendError(401, "API key required"); return; }

        // 1. Check per-second rate limit
        RateLimitResult rateResult = rateLimiter.check(apiKey);
        if (!rateResult.isAllowed()) {
            sendRateLimitResponse(res, rateResult);
            return;
        }

        // 2. Check monthly quota
        QuotaCheckResult quotaResult = quotaService.checkAndConsume(apiKey);
        if (!quotaResult.isAllowed()) {
            sendQuotaExceededResponse(res, quotaResult);
            return;
        }

        // 3. Add quota headers
        res.setHeader("X-Quota-Limit",     String.valueOf(quotaResult.getLimit()));
        res.setHeader("X-Quota-Remaining", String.valueOf(quotaResult.getRemaining()));
        res.setHeader("X-Quota-Reset",     quotaResult.getResetDate().toString());

        chain.doFilter(req, res);
    }
}
```

## 6. Backpressure Handling

Backpressure is the mechanism by which a downstream system signals its capacity to upstream systems, preventing the downstream from being overwhelmed and causing cascade failures.

### 6.1 Why backpressure matters

```
Without backpressure:
  Producer generates 10,000 msg/s
  Consumer processes 1,000 msg/s
  Queue grows at 9,000 msg/s → exhausts memory → OutOfMemoryError → crash

With backpressure:
  Consumer signals capacity: "I can take 1,000 msg/s"
  Producer throttles to 1,000 msg/s
  OR: queue full → producer blocks / rejects / sheds excess load

Backpressure is the correct answer to: "what happens when your queue is full?"
```

### 6.2 Reactive backpressure (Project Reactor)

```java
// Reactor: consumer controls flow via request(n) — demand-driven backpressure
@Service
public class ReactiveOrderProcessor {

    public void processOrderStream() {
        Flux<Order> orderStream = orderSource.getOrderFlux();

        orderStream
            // Backpressure operator: buffer up to 100, then signal upstream to slow down
            .onBackpressureBuffer(100,
                dropped -> log.warn("Order dropped due to backpressure: {}", dropped.getId()),
                BufferOverflowStrategy.DROP_LATEST)

            // Alternatively: drop newest when overwhelmed
            // .onBackpressureDrop(order -> deadLetterService.send(order))

            // Alternatively: error immediately on overflow (fast-fail)
            // .onBackpressureError()

            // Process with controlled parallelism
            .flatMap(order -> processOrder(order),
                     4,      // max 4 concurrent in-flight
                     8)      // prefetch buffer per inner publisher

            // Retry with exponential backoff on transient error
            .retryWhen(Retry.backoff(3, Duration.ofMillis(500))
                .filter(ex -> ex instanceof TransientException))
            .subscribe(
                result -> metrics.increment("orders.processed"),
                error  -> log.error("Stream failed", error)
            );
    }

    // Explicit demand control: request one at a time (maximum backpressure)
    public void processSequentially(Publisher<Order> source) {
        source.subscribe(new BaseSubscriber<>() {
            @Override
            protected void hookOnSubscribe(Subscription subscription) {
                request(1);  // request first item only
            }

            @Override
            protected void hookOnNext(Order order) {
                processOrder(order)
                    .doOnSuccess(r -> request(1))  // request next after processing current
                    .subscribe();
            }
        });
    }
}
```

### 6.3 Thread pool backpressure

```java
// Thread pool rejection policies — forms of backpressure at the execution layer
@Configuration
public class ThreadPoolConfig {

    @Bean("orderProcessorPool")
    public ThreadPoolExecutor orderProcessorPool() {
        return new ThreadPoolExecutor(
            10,           // corePoolSize
            50,           // maximumPoolSize
            60L, TimeUnit.SECONDS,
            new ArrayBlockingQueue<>(500),     // bounded queue — provides backpressure
            new ThreadPoolExecutor.CallerRunsPolicy()  // backpressure: caller thread executes
            // CallerRunsPolicy: if pool full, caller thread runs the task
            //   → automatically slows the producer (HTTP thread pool)
            //   → natural backpressure to upstream clients
        );
    }
}

// RejectedExecutionHandler options:
//   AbortPolicy (default):   throw RejectedExecutionException → 503
//   CallerRunsPolicy:        run in calling thread → slows caller → backpressure
//   DiscardPolicy:           silently drop newest → data loss
//   DiscardOldestPolicy:     drop oldest queued task → drop queued, accept new

// Monitoring queue depth for proactive backpressure
@Scheduled(fixedDelay = 5_000)
public void monitorThreadPool() {
    ThreadPoolExecutor pool = (ThreadPoolExecutor) orderProcessorPool;
    int queueSize = pool.getQueue().size();
    double utilisation = (double) pool.getActiveCount() / pool.getMaximumPoolSize();

    metrics.gauge("threadpool.queue.size", queueSize);
    metrics.gauge("threadpool.utilisation", utilisation);

    if (utilisation > 0.9) {
        log.warn("Thread pool at {}% utilisation — shedding load", (int)(utilisation * 100));
    }
}
```

```java
// Kafka consumer backpressure: pause partitions when downstream is slow
@Component
@RequiredArgsConstructor
public class BackpressureAwareConsumer {

    @KafkaListener(topics = "orders.placed", groupId = "order-processor",
                   id = "orderConsumer")
    public void consume(ConsumerRecord<String, OrderEvent> record,
                         Consumer<String, OrderEvent> consumer) {

        // Check downstream health before processing
        if (isDownstreamSaturated()) {
            // Pause this partition — Kafka will not push more messages
            consumer.pause(consumer.assignment());
            log.warn("Downstream saturated — pausing Kafka partitions");
            scheduleResume(consumer);
            return;
        }

        processRecord(record);
    }

    private void scheduleResume(Consumer<String, OrderEvent> consumer) {
        scheduler.schedule(() -> {
            if (!isDownstreamSaturated()) {
                consumer.resume(consumer.assignment());
                log.info("Downstream recovered — resuming Kafka partitions");
            } else {
                scheduleResume(consumer); // check again in 5s
            }
        }, 5, TimeUnit.SECONDS);
    }

    private boolean isDownstreamSaturated() {
        // Check DB connection pool, downstream service health, queue depth
        return dbPool.getActiveConnections() > dbPool.getMaximumPoolSize() * 0.9
            || orderProcessorQueue.size() > 8000;
    }
}
```

### 6.5 Load shedding

```java
// Load shedding: proactively reject low-priority requests when overloaded
// Prioritise: critical paths over non-critical; paying customers over free tier

@Component
@RequiredArgsConstructor
public class LoadSheddingFilter extends OncePerRequestFilter {
    private final SystemHealthMonitor healthMonitor;

    @Override
    protected void doFilterInternal(HttpServletRequest req,
                                     HttpServletResponse res,
                                     FilterChain chain) throws IOException, ServletException {

        SystemLoad load = healthMonitor.getCurrentLoad();

        if (load.isCritical()) {     // CPU > 95%, queue > 90% full
            // Only allow health checks and critical payments
            if (!isCriticalEndpoint(req)) {
                res.setStatus(HttpStatus.SERVICE_UNAVAILABLE.value());
                res.setHeader("Retry-After", "30");
                res.getWriter().write("{\"error\":\"service_overloaded\"}");
                metrics.increment("requests.shed", Tags.of("reason", "critical_load"));
                return;
            }
        } else if (load.isHigh()) {  // CPU > 80%, queue > 70% full
            // Shed non-critical traffic: analytics, batch, background sync
            if (isLowPriorityEndpoint(req)) {
                res.setStatus(HttpStatus.TOO_MANY_REQUESTS.value());
                res.setHeader("Retry-After", "10");
                return;
            }
        }

        chain.doFilter(req, res);
    }

    private boolean isCriticalEndpoint(HttpServletRequest req) {
        String path = req.getRequestURI();
        return path.startsWith("/payments") ||
               path.startsWith("/actuator/health") ||
               path.startsWith("/orders");
    }

    private boolean isLowPriorityEndpoint(HttpServletRequest req) {
        String path = req.getRequestURI();
        return path.startsWith("/analytics") ||
               path.startsWith("/recommendations") ||
               path.startsWith("/batch");
    }
}
```

```java
@Service

    private final RateLimiterRegistry registry;

    // Annotation-based: throws RequestNotPermitted on limit exceeded
    @RateLimiter(name = "default", fallbackMethod = "getProductFallback")
    public Product getProduct(Long id) {
        return productRepository.findById(id).orElseThrow();
    }

    // Programmatic: check permit, define own behaviour
    public Product getProductControlled(Long id, String clientTier) {
        RateLimiter limiter = registry.rateLimiter(clientTier);

        if (!limiter.acquirePermission()) {
            throw new RateLimitExceededException(
```

## 8. Interview Questions

**Q: Explain the token bucket algorithm and how it differs from the leaky bucket. When would you use each?**

A: The token bucket maintains a bucket with a maximum capacity of N tokens. Tokens are added at a constant rate (e.g. 100 tokens/second). Each incoming request consumes one token. If tokens are available, the request is allowed immediately; if the bucket is empty, the request is rejected. The key property is that unused capacity accumulates — if a client makes no requests for 10 seconds, they accumulate 10 seconds' worth of tokens (up to the capacity ceiling), allowing a burst. The leaky bucket processes requests at a fixed output rate regardless of input pattern: incoming requests fill a queue; the queue drains at a constant rate; requests that would overflow the queue are dropped. Token bucket is better for user-facing APIs because it allows natural bursts (a user clicking several buttons in quick succession) while enforcing a sustained rate. Leaky bucket is better for protecting downstream services that cannot tolerate bursts — for example, a payment processor that charges per transaction and fails if overwhelmed. The output of the leaky bucket is always smooth and predictable.

---

**Q: What is the boundary burst problem with fixed window rate limiting, and how does the sliding window counter solve it?**

A: The boundary burst problem occurs because fixed windows reset at discrete intervals. A client can make requests at the end of window 1 (using almost the full limit) and then immediately again at the start of window 2 (the counter just reset), effectively making twice the allowed requests in a short period. For a 100 req/min limit, a client could make 99 requests at 00:00:59 and 99 more at 00:01:00 — 198 requests in two seconds. The sliding window counter approximates a true sliding window using just two counters: the current window count and the previous window count, weighted by how much of the previous window falls within the current sliding interval. If the window is 1 minute and the current time is 45 seconds into the new window, then 25% of the previous window is still within the 60-second lookback, so the estimate is `current + previous × 0.25`. This eliminates the boundary spike at the cost of a small approximation error (empirically less than 0.003% inaccuracy) while maintaining O(1) memory per client, making it practical for millions of clients.

---

**Q: How do you implement rate limiting in a horizontally scaled service with 20 application nodes?**

A: With 20 nodes, per-node in-memory rate limiters only see 1/20th of the traffic, so a 100 req/s limit becomes effectively 2,000 req/s globally. The solution is a shared Redis counter as the central source of truth. Every node consults Redis on each request — the atomic INCR or Lua script ensures no race conditions across nodes. The tradeoff is Redis round-trip latency on every request (typically 1-2ms). To reduce this overhead for high-throughput services, a two-layer approach pre-allocates token batches to each node: each node holds a local bucket of 50 tokens and replenishes from Redis only when exhausted, reducing Redis calls by 50x. The accuracy tradeoff is that a client may briefly exceed the global limit if multiple nodes replenish simultaneously, but this overage is bounded and typically acceptable. For resilience, if Redis becomes unavailable, nodes fall back to local limiting — accepting slightly inaccurate limits rather than rejecting all traffic. The fallback policy (allow vs deny on Redis failure) is a business decision based on whether over-limiting or under-limiting is more costly.

---

**Q: What is backpressure, and what mechanisms does Spring provide to implement it?**

A: Backpressure is the signal from a consumer to a producer to slow down — it prevents a fast producer from overwhelming a slow consumer by propagating capacity signals upstream rather than buffering indefinitely. Without backpressure, queues grow until memory exhaustion. Spring provides several backpressure mechanisms. At the reactive layer, Project Reactor's `Flux` implements Reactive Streams backpressure natively: operators like `onBackpressureBuffer(maxSize)` buffer up to N elements and signal overflow via drop or error; `flatMap(fn, concurrency, prefetch)` controls how many in-flight items exist simultaneously. At the thread pool layer, bounded `ArrayBlockingQueue` combined with `CallerRunsPolicy` creates natural backpressure: when the pool is full, the HTTP request thread that submitted the task runs it directly, blocking that thread from accepting new connections — automatically slowing inbound request acceptance. At the Kafka layer, consumer `pause()`/`resume()` on partitions lets a consumer signal to the broker to stop delivering messages until capacity recovers. The right choice depends on the processing model: reactive for streaming data pipelines, thread pool for request-response workloads, and Kafka pause/resume for event-driven microservices.

---

**Q: What HTTP status code should be returned for rate limiting, and what headers should accompany it?**

A: The correct status is `429 Too Many Requests` (RFC 6585), not 503. A `503 Service Unavailable` signals that the server itself is down or overloaded — it implies a problem with the service infrastructure. `429` specifically communicates "you've sent too many requests; slow down and retry." The mandatory header is `Retry-After`, which tells the client how many seconds to wait before retrying — without it, clients typically implement exponential backoff from their own logic, which may be too aggressive or too conservative. Best practice is to also include `X-RateLimit-Limit` (the cap), `X-RateLimit-Remaining` (tokens left in current window), and `X-RateLimit-Reset` (Unix timestamp when the window resets or the bucket refills). These headers should be sent on every response — not just on 429 — so well-behaved clients can proactively throttle themselves before hitting the limit. The response body should include a machine-readable error code (`"error": "rate_limit_exceeded"`) and a human-readable message for debugging.

## 9. Key Takeaways

- **Token bucket** allows bursts up to capacity, sustained rate limited to refill rate — best for user-facing APIs where natural bursts are acceptable
- **Leaky bucket** enforces smooth constant output rate — best for protecting downstream systems from burst traffic
- **Fixed window** has O(1) memory but is vulnerable to boundary burst (2x limit at window edge); **sliding window counter** eliminates this with ~99.997% accuracy at same O(1) memory cost
- **Distributed rate limiting** requires shared Redis counter for accuracy across nodes; two-layer local+Redis hybrid reduces Redis calls 50-100x at the cost of slight over-allowance
- **Always return 429 with `Retry-After` header** — not 503; include `X-RateLimit-Limit/Remaining/Reset` on every response so clients self-throttle proactively
- **Quotas** (monthly caps) and **rate limits** (per-second throttles) serve different purposes and should be enforced independently — quotas for business tiers, rate limits for infrastructure protection
- **Backpressure is the correct response to a full queue**: `CallerRunsPolicy` in thread pools, `onBackpressureBuffer` in Reactor, `pause()`/`resume()` in Kafka consumers — never grow unbounded queues
- **Load shedding** is the last line of defence: reject low-priority requests during critical overload rather than letting all requests time out — a 429/503 is better than a 30-second hang
- **Resilience4j** provides annotation-driven rate limiting with fallback methods — prefer for simple per-service limits; use Redis-backed custom implementation for distributed multi-tenant limits
