# Sorting Algorithms

## Overview

Sorting is one of the most studied problems in computer science — not because sorting itself is the end goal, but because the right algorithm depends on data characteristics that recur everywhere: size, distribution, partial order, key type, and memory constraints. Java's default sort (`Arrays.sort` for objects, `Collections.sort`) uses TimSort, a sophisticated hybrid that achieves O(n log n) worst-case while exploiting natural order already present in real-world data.

**Core concepts before diving in:**

```
In-place:     O(1) extra space (QuickSort, HeapSort)
Not in-place: O(n) extra space (MergeSort)
Stable:       Equal elements preserve their original relative order
              Critical when sorting objects by one key while preserving another
Adaptive:     Runs faster on partially sorted input (TimSort, InsertionSort)
Comparison-based: O(n log n) lower bound (proven by decision tree argument)
Non-comparison: Can beat O(n log n) by exploiting key structure (Radix, Counting)
```

## 1. QuickSort

QuickSort is the fastest comparison sort in practice for random data. It partitions the array around a **pivot** such that all elements less than the pivot are on the left, greater on the right, then recursively sorts both halves.

### 1.1 Lomuto Partition Scheme

```java
public class QuickSort {

    public void sort(int[] arr) {
        quickSort(arr, 0, arr.length - 1);
    }

    private void quickSort(int[] arr, int low, int high) {
        if (low >= high) return;

        int pivot = partition(arr, low, high);
        quickSort(arr, low, pivot - 1);
        quickSort(arr, pivot + 1, high);
    }

    // Lomuto: pivot = last element, scan left-to-right
    private int partition(int[] arr, int low, int high) {
        int pivot = arr[high];
        int i = low - 1; // boundary of "less than pivot" region

        for (int j = low; j < high; j++) {
            if (arr[j] <= pivot) {
                i++;
                swap(arr, i, j); // extend "less than" region
            }
        }
        swap(arr, i + 1, high); // place pivot at its final position
        return i + 1;
    }

    private void swap(int[] arr, int i, int j) {
        int tmp = arr[i]; arr[i] = arr[j]; arr[j] = tmp;
    }
}
```

**Trace on [3, 6, 8, 10, 1, 2, 1], pivot = 1:**

```
Initial: [3, 6, 8, 10, 1, 2, | 1]   pivot=1, i=-1
j=0: arr[0]=3 > pivot → skip
j=1: arr[1]=6 > pivot → skip
j=2: arr[2]=8 > pivot → skip
j=3: arr[3]=10 > pivot → skip
j=4: arr[4]=1 ≤ pivot → i=0, swap(0,4): [1, 6, 8, 10, 3, 2, | 1]
j=5: arr[5]=2 > pivot → skip
Swap pivot to i+1=1: [1, 1, 8, 10, 3, 2, 6]
Pivot at index 1. Recurse on [1] and [8,10,3,2,6]
```

### 1.2 Hoare Partition Scheme

More efficient in practice — makes fewer swaps than Lomuto:

```java
// Hoare: pivot = first element, two pointers move inward
private int hoarePartition(int[] arr, int low, int high) {
    int pivot = arr[low + (high - low) / 2]; // middle element as pivot
    int i = low - 1, j = high + 1;

    while (true) {
        do { i++; } while (arr[i] < pivot);  // advance left
        do { j--; } while (arr[j] > pivot);  // advance right
        if (i >= j) return j;                // crossing point
        swap(arr, i, j);
    }
}

private void quickSortHoare(int[] arr, int low, int high) {
    if (low >= high) return;
    int pivot = hoarePartition(arr, low, high);
    quickSortHoare(arr, low, pivot);       // note: pivot+1 for Lomuto, pivot for Hoare
    quickSortHoare(arr, pivot + 1, high);
}
```

### 1.3 3-Way Partition (Dutch National Flag) — handles duplicates

Standard QuickSort degrades to O(n²) with many equal elements. Dutch National Flag partitions into three regions: `< pivot`, `= pivot`, `> pivot`.

```java
// Dijkstra's 3-way partitioning — O(n) for all-equal arrays
private void quickSort3Way(int[] arr, int low, int high) {
    if (low >= high) return;

    int lt = low, gt = high, i = low;
    int pivot = arr[low];

    while (i <= gt) {
        if      (arr[i] < pivot) swap(arr, lt++, i++); // extend less region
        else if (arr[i] > pivot) swap(arr, i, gt--);   // extend greater region
        else                     i++;                   // equal to pivot — skip
    }
    // arr[low..lt-1] < pivot
    // arr[lt..gt]   = pivot (all in final position)
    // arr[gt+1..high] > pivot

    quickSort3Way(arr, low, lt - 1);
    quickSort3Way(arr, gt + 1, high);
}
```

### 1.4 Pivot selection strategies

```java
// Bad pivot choice → O(n²): sorted array + last element as pivot
// All other elements are larger → partition produces [n-1 | 1] split every time

// Strategy 1: Random pivot — reduces probability of O(n²) to negligible
int pivotIdx = low + random.nextInt(high - low + 1);
swap(arr, pivotIdx, high);
// then use Lomuto partition as before

// Strategy 2: Median-of-three — deterministic, good in practice
private int medianOfThree(int[] arr, int low, int high) {
    int mid = low + (high - low) / 2;
    // Sort arr[low], arr[mid], arr[high] relative to each other
    if (arr[low] > arr[mid])  swap(arr, low, mid);
    if (arr[low] > arr[high]) swap(arr, low, high);
    if (arr[mid] > arr[high]) swap(arr, mid, high);
    // arr[low] ≤ arr[mid] ≤ arr[high]
    // Place median at high-1 (Sedgewick's trick), use as pivot
    swap(arr, mid, high - 1);
    return arr[high - 1];
}
```

### 1.5 QuickSort complexity

|Case|Time|Cause|
|---|---|---|
|Best|O(n log n)|Pivot always splits evenly|
|Average|O(n log n)|Random data, random pivot|
|Worst|O(n²)|Sorted/reverse-sorted + bad pivot|
|Space|O(log n) avg|Recursion stack depth|
|Space worst|O(n)|Degenerate partition every time|

**QuickSort is NOT stable** — the partition swaps equal elements past each other.

## 2. MergeSort

MergeSort divides the array in half, recursively sorts both halves, then **merges** two sorted halves into one. It is the standard divide-and-conquer sort and is **stable**.

```java
public class MergeSort {

    public void sort(int[] arr) {
        if (arr.length <= 1) return;
        int[] temp = new int[arr.length]; // allocate once — avoids repeated GC pressure
        mergeSort(arr, temp, 0, arr.length - 1);
    }

    private void mergeSort(int[] arr, int[] temp, int left, int right) {
        if (left >= right) return;

        int mid = left + (right - left) / 2; // avoid overflow vs (left+right)/2
        mergeSort(arr, temp, left, mid);
        mergeSort(arr, temp, mid + 1, right);
        merge(arr, temp, left, mid, right);
    }

    private void merge(int[] arr, int[] temp, int left, int mid, int right) {
        // Copy to temp buffer
        System.arraycopy(arr, left, temp, left, right - left + 1);

        int i = left;      // pointer into left half
        int j = mid + 1;   // pointer into right half
        int k = left;      // pointer into merged output

        while (i <= mid && j <= right) {
            // <= for stability: prefer left half when equal
            if (temp[i] <= temp[j]) arr[k++] = temp[i++];
            else                    arr[k++] = temp[j++];
        }
        // Copy remaining elements (one side will already be exhausted)
        while (i <= mid)    arr[k++] = temp[i++];
        while (j <= right)  arr[k++] = temp[j++];
    }
}
```

**Merge trace on [3,1,2,5,4]:**

```
mergeSort([3,1,2,5,4])
  mergeSort([3,1])
    mergeSort([3]) → [3]
    mergeSort([1]) → [1]
    merge([3],[1]) → [1,3]
  mergeSort([2,5,4])
    mergeSort([2]) → [2]
    mergeSort([5,4])
      merge([5],[4]) → [4,5]
    merge([2],[4,5]) → [2,4,5]
  merge([1,3],[2,4,5]) → [1,2,3,4,5] ✓
```

### 2.1 Bottom-up MergeSort (iterative — no recursion)

```java
// Avoids recursion stack overhead — useful for very large datasets
public void mergeSortBottomUp(int[] arr) {
    int n = arr.length;
    int[] temp = new int[n];

    // Start with subarrays of size 1, double size each pass
    for (int size = 1; size < n; size *= 2) {
        for (int left = 0; left < n - size; left += 2 * size) {
            int mid   = left + size - 1;
            int right = Math.min(left + 2 * size - 1, n - 1);
            merge(arr, temp, left, mid, right);
        }
    }
}
```

### 2.2 MergeSort on Linked Lists

MergeSort is the preferred sort for linked lists — it requires no random access, and merging linked lists is O(n) with O(1) space (just pointer rewiring):

```java
public ListNode mergeSort(ListNode head) {
    if (head == null || head.next == null) return head;

    // Find middle using slow/fast pointers
    ListNode slow = head, fast = head.next;
    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
    }
    ListNode mid = slow.next;
    slow.next = null; // split the list

    ListNode left  = mergeSort(head);
    ListNode right = mergeSort(mid);
    return mergeLists(left, right);
}

private ListNode mergeLists(ListNode l1, ListNode l2) {
    ListNode dummy = new ListNode(0), curr = dummy;
    while (l1 != null && l2 != null) {
        if (l1.val <= l2.val) { curr.next = l1; l1 = l1.next; }
        else                  { curr.next = l2; l2 = l2.next; }
        curr = curr.next;
    }
    curr.next = (l1 != null) ? l1 : l2;
    return dummy.next;
}
```

### 2.3 MergeSort complexity

|Case|Time|Space|
|---|---|---|
|Best|O(n log n)|O(n) — temp buffer|
|Average|O(n log n)|O(n)|
|Worst|O(n log n)|O(n)|
|Linked list|O(n log n)|O(log n) — call stack|

**MergeSort IS stable.** External sort (data too large for RAM) uses MergeSort's merge step — read chunks, sort in-memory, merge sorted runs from disk.

## 3. HeapSort

HeapSort builds a **max-heap** from the array, then repeatedly extracts the maximum (swapping to end) to produce a sorted array. It is in-place and guarantees O(n log n) worst-case.

```java
public class HeapSort {

    public void sort(int[] arr) {
        int n = arr.length;

        // Phase 1: Build max-heap — O(n)
        // Start from last non-leaf node, sift down each
        for (int i = n / 2 - 1; i >= 0; i--)
            heapify(arr, n, i);

        // Phase 2: Extract max repeatedly — O(n log n)
        for (int i = n - 1; i > 0; i--) {
            swap(arr, 0, i);    // move current max to end
            heapify(arr, i, 0); // restore heap property for reduced heap
        }
    }

    // Sift down node at index i in heap of size n
    private void heapify(int[] arr, int n, int i) {
        int largest = i;
        int left    = 2 * i + 1;
        int right   = 2 * i + 2;

        if (left  < n && arr[left]  > arr[largest]) largest = left;
        if (right < n && arr[right] > arr[largest]) largest = right;

        if (largest != i) {
            swap(arr, i, largest);
            heapify(arr, n, largest); // recursively fix the affected subtree
        }
    }

    private void swap(int[] arr, int i, int j) {
        int tmp = arr[i]; arr[i] = arr[j]; arr[j] = tmp;
    }
}
```

**HeapSort trace on [4, 10, 3, 5, 1]:**

```
Build max-heap:
  Start at index 1 (n/2-1=1): heapify([4,10,3,5,1], n=5, i=1)
    left=3(arr[3]=5), right=4(arr[4]=1), largest=3(5)
    swap(1,3): [4,5,3,10,1] → heapify(i=3): leaf, done
  heapify at index 0: left=1(5), right=2(3), largest=1
    wait: left subtree has 5, but arr[3]=10... re-heapify properly:
  Final max-heap: [10, 5, 3, 4, 1]

Extract phase:
  swap(0,4): [1,5,3,4,10], heapify(size=4,i=0) → [5,4,3,1 | 10]
  swap(0,3): [1,4,3,5 | 10], heapify(size=3,i=0) → [4,1,3 | 5,10]
  swap(0,2): [3,1,4 | 5,10], heapify(size=2,i=0) → [3,1 | 4,5,10]
  swap(0,1): [1,3 | 4,5,10] → done
  Result: [1, 3, 4, 5, 10] ✓
```

### 3.1 HeapSort complexity

|Case|Time|Space|
|---|---|---|
|All cases|O(n log n)|O(1) in-place|
|Build heap|O(n)|—|
|Extract n times|O(n log n)|—|

**HeapSort is NOT stable** — the heap operations move equal elements relative to each other.

**Why HeapSort is rarely used in practice despite O(n log n) guaranteed:**

```
Poor cache performance: heapify jumps between distant memory locations
  (parent at i, children at 2i+1 and 2i+2 — distant in large arrays)
QuickSort: sequential memory access → cache-friendly → faster in practice
TimSort: beats HeapSort on real-world (partially sorted) data
HeapSort's advantage: guaranteed O(n log n) worst-case, O(1) space
Use HeapSort when: memory is extremely constrained AND worst-case guarantee needed
```

## 4. Stability in Sorting

**Stability**: a sort is stable if equal elements retain their original relative order in the output.

### 4.1 Why stability matters

```java
// Sort employees first by department, then by salary
// If salary sort is stable, employees within same salary stay in department order

List<Employee> employees = List.of(
    new Employee("Alice", "Engineering", 90000),
    new Employee("Bob",   "Marketing",   75000),
    new Employee("Carol", "Engineering", 75000),
    new Employee("Dave",  "Marketing",   90000)
);

// Step 1: Sort by department (stable)
employees.sort(Comparator.comparing(Employee::getDepartment));
// [Alice/Eng/90k, Carol/Eng/75k, Bob/Mkt/75k, Dave/Mkt/90k]

// Step 2: Sort by salary (STABLE sort preserves department order for equal salaries)
employees.sort(Comparator.comparingInt(Employee::getSalary));
// [Carol/Eng/75k, Bob/Mkt/75k, Alice/Eng/90k, Dave/Mkt/90k]
//  ↑ Carol before Bob (both 75k, but Eng < Mkt from previous sort)
//  ↑ Alice before Dave (both 90k, but Eng < Mkt from previous sort)

// With UNSTABLE sort: Carol/Bob and Alice/Dave order is arbitrary
```

### 4.2 Stability by algorithm

|Algorithm|Stable|Notes|
|---|---|---|
|MergeSort|✓|Prefers left half when equal: `if (left <= right)`|
|TimSort|✓|Hybrid of MergeSort + InsertionSort — Java's default|
|InsertionSort|✓|Only shifts when strictly less: `while (key < arr[j])`|
|BubbleSort|✓|Only swaps when strictly greater|
|QuickSort|✗|Partition swaps equal elements past each other|
|HeapSort|✗|Heap operations destroy relative order|
|SelectionSort|✗|Finds minimum ignoring relative position|
|CountingSort|✓|If implemented with backward scan|
|RadixSort|✓|Requires stable digit sort (counting sort)|

### 4.3 Making unstable sorts stable

```java
// Trick: add original index as tiebreaker
// Transforms any sort into a stable one

record IndexedValue<T>(T value, int index) {}

public <T> List<T> stableSort(List<T> list, Comparator<T> cmp) {
    List<IndexedValue<T>> indexed = new ArrayList<>();
    for (int i = 0; i < list.size(); i++)
        indexed.add(new IndexedValue<>(list.get(i), i));

    indexed.sort((a, b) -> {
        int result = cmp.compare(a.value(), b.value());
        return result != 0 ? result : Integer.compare(a.index(), b.index()); // tiebreak by original index
    });

    return indexed.stream().map(IndexedValue::value).collect(Collectors.toList());
}
```

## 5. TimSort — Java's Default Sort

TimSort is Java's sort for **object arrays** (`Arrays.sort(Object[])`, `Collections.sort()`, `List.sort()`). It is a **hybrid of MergeSort and InsertionSort** designed to exploit natural runs already present in real-world data.

### 5.1 Core ideas

```
1. Natural runs: scan array for already-sorted sequences (ascending or descending)
   If descending, reverse it.

2. Minimum run size (minRun): 32–64 elements.
   If a run is shorter than minRun, extend it with InsertionSort.
   InsertionSort is O(n²) generally but O(n) for nearly-sorted small arrays
   and has excellent cache performance on small inputs.

3. Merge runs: maintain a stack of runs, merge when stack top violates
   invariants:
     - len(run3) > len(run2) + len(run1)
     - len(run2) > len(run1)
   This keeps runs balanced and total merge work O(n log n).

4. Galloping mode: during merge, if one side "wins" many comparisons in a row,
   switch to binary search to find how far it dominates → O(log k) per gallop
   instead of O(k) comparisons. Excellent for partially sorted data.
```

### 5.2 Simplified TimSort simulation

```java
// Simplified TimSort showing the key concepts
public class TimSort {
    private static final int MIN_MERGE = 32;

    public void sort(int[] arr) {
        int n = arr.length;
        int minRun = computeMinRun(n);

        // Step 1: Sort individual runs with insertion sort
        for (int i = 0; i < n; i += minRun) {
            insertionSort(arr, i, Math.min(i + minRun - 1, n - 1));
        }

        // Step 2: Merge sorted runs
        for (int size = minRun; size < n; size *= 2) {
            for (int left = 0; left < n; left += 2 * size) {
                int mid   = Math.min(left + size - 1, n - 1);
                int right = Math.min(left + 2 * size - 1, n - 1);
                if (mid < right) merge(arr, left, mid, right);
            }
        }
    }

    // minRun: chosen so n/minRun is a power of 2 (or just below)
    // This ensures merges are balanced
    private int computeMinRun(int n) {
        int r = 0; // becomes 1 if any 1 bits are shifted off
        while (n >= MIN_MERGE) {
            r |= (n & 1);
            n >>= 1;
        }
        return n + r; // result in range [MIN_MERGE/2, MIN_MERGE]
    }

    // InsertionSort — O(n) on nearly-sorted, O(n²) worst
    private void insertionSort(int[] arr, int left, int right) {
        for (int i = left + 1; i <= right; i++) {
            int key = arr[i];
            int j = i - 1;
            while (j >= left && arr[j] > key) {
                arr[j + 1] = arr[j];
                j--;
            }
            arr[j + 1] = key;
        }
    }

    private void merge(int[] arr, int left, int mid, int right) {
        // Standard stable merge (same as MergeSort merge)
        int[] temp = Arrays.copyOfRange(arr, left, mid + 1);
        int i = 0, j = mid + 1, k = left;
        while (i < temp.length && j <= right) {
            if (temp[i] <= arr[j]) arr[k++] = temp[i++];
            else                   arr[k++] = arr[j++];
        }
        while (i < temp.length) arr[k++] = temp[i++];
    }
}
```

### 5.3 TimSort in Java

```java
// Java's Arrays.sort uses:
//   - Dual-Pivot QuickSort for PRIMITIVE arrays (int[], long[], etc.)
//   - TimSort for OBJECT arrays (Integer[], String[], custom objects)

// Reason for the distinction:
//   - Primitives: stability doesn't matter (no identity, just value)
//                 Dual-Pivot QuickSort is faster in practice for primitives
//   - Objects: stability is expected by Java spec (Comparable contract)
//              TimSort guarantees stability

// Dual-Pivot QuickSort (Java 7+ for primitives)
// Uses two pivots p1 < p2, partitioning into three regions:
//   [< p1] [p1 ≤ x ≤ p2] [> p2]
// Better cache behaviour than single-pivot for most inputs

int[] primitives = {5, 3, 1, 4, 2};
Arrays.sort(primitives); // Dual-Pivot QuickSort

Integer[] objects = {5, 3, 1, 4, 2};
Arrays.sort(objects);    // TimSort
```

|Worst|O(n log n)|O(n)|Guaranteed — unlike QuickSort|

## 6. Radix Sort for Integers

Radix Sort bypasses the O(n log n) comparison lower bound by sorting digit-by-digit from least significant to most significant (LSD radix sort), using a **stable** counting sort at each digit.

**Key insight**: for integers with d digits and base b, complexity is O(d × (n + b)). For 32-bit integers with base 256 (d=4 passes): O(4 × (n + 256)) = O(n).

```java
public class RadixSort {

    // LSD Radix Sort for non-negative integers — O(d × n)
    public void sort(int[] arr) {
        int max = Arrays.stream(arr).max().getAsInt();

        // Process one digit position at a time (least significant first)
        for (int exp = 1; max / exp > 0; exp *= 10) {
            countingSortByDigit(arr, exp);
        }
    }

    private void countingSortByDigit(int[] arr, int exp) {
        int n = arr.length;
        int[] output = new int[n];
        int[] count  = new int[10]; // digits 0-9

        // Count occurrences of each digit
        for (int val : arr) count[(val / exp) % 10]++;

        // Prefix sum — count[i] = number of elements with digit ≤ i
        for (int i = 1; i < 10; i++) count[i] += count[i - 1];

        // Build output array — iterate BACKWARDS for stability
        for (int i = n - 1; i >= 0; i--) {
            int digit = (arr[i] / exp) % 10;
            output[--count[digit]] = arr[i];
        }

        System.arraycopy(output, 0, arr, 0, n);
    }
}
```

```
  Digits: [0, 5, 5, 0, 2, 4, 2, 6]
  After:  [170, 90, 802, 2, 24, 45, 75, 66]

exp=10 (tens digit):
  Digits: [7, 9, 0, 0, 2, 4, 7, 6]
  After:  [802, 2, 24, 45, 66, 170, 75, 90]

exp=100 (hundreds digit):
  Digits: [8, 0, 0, 0, 0, 1, 0, 0]

```

### 6.1 Radix Sort with negative integers

```java
public void sortWithNegatives(int[] arr) {
    // Separate negatives and positives
    int[] negatives = Arrays.stream(arr).filter(x -> x < 0)
                           .map(x -> -x).toArray(); // negate for sorting
    int[] positives = Arrays.stream(arr).filter(x -> x >= 0).toArray();

    sort(negatives); // sort absolute values
    sort(positives);

    // Combine: negatives reversed (most negative first), then positives
    int k = 0;
    for (int i = negatives.length - 1; i >= 0; i--) arr[k++] = -negatives[i];
    for (int pos : positives) arr[k++] = pos;
}
```

### 6.2 Base-256 Radix Sort (faster for 32-bit integers)

```java
// 4 passes instead of 10 (for integers up to 10 digits)
// Each pass handles 8 bits at a time
public void radixSort256(int[] arr) {
    int n = arr.length;
    int[] temp = new int[n];

    for (int shift = 0; shift < 32; shift += 8) {
        int[] count = new int[256];

        for (int val : arr) count[(val >> shift) & 0xFF]++;
        for (int i = 1; i < 256; i++) count[i] += count[i - 1];
        for (int i = n - 1; i >= 0; i--)
            temp[--count[(arr[i] >> shift) & 0xFF]] = arr[i];

        System.arraycopy(temp, 0, arr, 0, n);
    }
}
```

### 6.3 Radix Sort complexity

||Radix Sort|Comparison Sorts|
|---|---|---|
|Time|O(d × (n + b))|O(n log n)|
|Space|O(n + b)|O(1) to O(n)|
|In-place|No|HeapSort: yes|
|Stability|Yes|MergeSort, TimSort: yes|
|Adaptivity|No|TimSort: yes|
|Integer keys only|Yes|No (general)|

**When Radix Sort wins**: large n, small d (e.g., sorting 10 million 32-bit integers — 4 passes of O(n) beats O(n log n) by roughly log₂(10M) / 4 ≈ 6x fewer operations).

**When Radix Sort loses**: small n (constant factors dominate), large d (string keys with many characters), or when data isn't integer-like.

## 7. Custom Comparators

### 7.1 Comparator API in Java

```java
// Comparator contract: compare(a, b)
//   returns negative → a comes before b
//   returns zero     → a and b are equal (stable sort preserves order)
//   returns positive → a comes after b

// Lambda comparator
Comparator<String> byLength = (a, b) -> Integer.compare(a.length(), b.length());

// Comparator.comparing — clean, readable
Comparator<Employee> bySalary     = Comparator.comparingInt(Employee::getSalary);
Comparator<Employee> byName       = Comparator.comparing(Employee::getName);
Comparator<Employee> byNameLength = Comparator.comparingInt(e -> e.getName().length());

// Chaining comparators
Comparator<Employee> byDeptThenSalary = Comparator
```

### 7.3 Comparator pitfalls

```java
// WRONG: integer overflow in subtraction comparator
// Works for small values, silently wrong for large/mixed-sign integers
Comparator<Integer> bad = (a, b) -> a - b; // OVERFLOW if a=MAX_VALUE, b=-1

// CORRECT: use Integer.compare
Comparator<Integer> good = (a, b) -> Integer.compare(a, b);
Comparator<Integer> also = Integer::compare;

// WRONG: comparator violates transitivity → undefined sort behavior
// Example: a > b, b > c, but a < c (rock-paper-scissors)
Comparator<String> broken = (a, b) -> {
    if (a.length() == b.length()) return 0;
    return a.length() > b.length() ? 1 : -1;
    // Missing: doesn't consistently order equal-length strings
};
// This can cause infinite loops or wrong results in Java's TimSort

// CORRECT: consistent total order
Comparator<String> fixed = Comparator
    .comparingInt(String::length)
    .thenComparing(Comparator.naturalOrder()); // tiebreak with lexicographic
```

### 7.4 Implementing Comparable for natural ordering

```java
// Implement Comparable for natural ordering (used by TreeSet, TreeMap, Collections.sort)
public class Version implements Comparable<Version> {
    private final int major, minor, patch;

    @Override
    public int compareTo(Version other) {
        if (this.major != other.major) return Integer.compare(this.major, other.major);
        if (this.minor != other.minor) return Integer.compare(this.minor, other.minor);
        return Integer.compare(this.patch, other.patch);
    }

    // Contract: if a.compareTo(b) == 0, then a.equals(b) should be true
    // (recommended but not strictly required)
    @Override
    public boolean equals(Object obj) {
        if (!(obj instanceof Version other)) return false;
        return major == other.major && minor == other.minor && patch == other.patch;
    }

    @Override
    public int hashCode() { return Objects.hash(major, minor, patch); }
}
```

## 8. Sorting Complexity Analysis

### 8.1 The O(n log n) lower bound for comparison sorts

```
Any comparison-based sort can be modelled as a decision tree:
  - Each internal node = comparison (is arr[i] < arr[j]?)
  - Each leaf = one possible sorted permutation
  - For n elements: n! possible permutations = n! leaves
  - A binary tree with n! leaves has height ≥ log₂(n!)

By Stirling's approximation:
  log₂(n!) ≈ n log₂(n) − n log₂(e) = Ω(n log n)

Therefore: any comparison-based sort requires Ω(n log n) comparisons
           in the worst case. MergeSort and HeapSort are optimal.
```

### 8.3 Practical performance characteristics

```
Random data (n = 10 million):
  QuickSort:   ~1.2 seconds  (best cache behaviour, few branch mispredicts)
  MergeSort:   ~2.1 seconds  (extra memory allocation and copying)
  HeapSort:    ~3.5 seconds  (terrible cache performance — random access pattern)
  TimSort:     ~1.8 seconds  (overhead of run detection pays off on real data)

Nearly sorted (95% in place):
  TimSort:     ~0.1 seconds  (detects long runs, minimal merging)
  InsertionSort: ~0.2 seconds (O(n) shifts instead of swaps)
  QuickSort:   ~0.8 seconds  (doesn't exploit order)

Many duplicates (only 10 distinct values):
  3-Way QuickSort: ~0.3 seconds (O(n) for all-equal, O(n log n) otherwise)
  Standard QuickSort: ~3.2 seconds (degrades toward O(n²))
  RadixSort:   ~0.4 seconds
```

### 8.4 Algorithm selection guide

```
When to use which sort:

General purpose (objects):
    → TimSort (Java default) — stable, adaptive, O(n log n) guaranteed

General purpose (primitives, no stability needed):
    → Dual-Pivot QuickSort (Java default) — fastest in practice

Need guaranteed O(n log n) worst-case + O(1) space:
    → HeapSort (rarely used — poor cache performance)

Need stable + O(n log n) guaranteed + external sorting (data on disk):
    → MergeSort — merge step works across storage media

Small array (n < 16):
    → InsertionSort — O(n) for nearly sorted, tiny constant factor

Partially sorted data:
    → TimSort or InsertionSort — O(n) adaptive behaviour

Integer keys, large n, bounded range:
    → RadixSort — beats O(n log n) by constant factors

Floating-point in [0,1], uniform distribution:
    → BucketSort — expected O(n)

Frequency/lexicographic multi-key sort:
    → RadixSort (LSD for fixed-length, MSD for variable-length)
```

```java
// Pattern 1: Sort + Two Pointers (Two Sum variants)
public boolean hasPairWithSum(int[] arr, int target) {
    Arrays.sort(arr);
    int lo = 0, hi = arr.length - 1;
    while (lo < hi) {
        int sum = arr[lo] + arr[hi];
        if (sum == target) return true;
        else if (sum < target) lo++;
        else hi--;
    }
    return false;
}

// Pattern 2: Custom sort for interval merging
public int[][] merge(int[][] intervals) {
    Arrays.sort(intervals, (a, b) -> Integer.compare(a[0], b[0]));
    List<int[]> result = new ArrayList<>();
    int[] current = intervals[0];

    for (int i = 1; i < intervals.length; i++) {
        if (intervals[i][0] <= current[1]) {
            current[1] = Math.max(current[1], intervals[i][1]); // extend
        } else {
            result.add(current);
            current = intervals[i];
        }
    }
    result.add(current);
    return result.toArray(new int[0][]);
}

// Pattern 3: Kth largest — partial sort with max-heap
public int findKthLargest(int[] nums, int k) {
    PriorityQueue<Integer> minHeap = new PriorityQueue<>();
    for (int num : nums) {
        minHeap.offer(num);
        if (minHeap.size() > k) minHeap.poll();
    }
    return minHeap.peek(); // O(n log k) — better than full sort O(n log n) when k << n
}

// Pattern 4: Sort by custom comparator for lexicographic number concatenation
// LeetCode 179: Largest Number
public String largestNumber(int[] nums) {
    String[] strs = Arrays.stream(nums).mapToObj(String::valueOf).toArray(String[]::new);
    Arrays.sort(strs, (a, b) -> (b + a).compareTo(a + b)); // compare concatenations
    if (strs[0].equals("0")) return "0"; // edge case: all zeros
    return String.join("", strs);
}
```

## 10. Interview Questions

**Q: Why does Java use different sorting algorithms for primitive arrays vs object arrays?**

A: Java uses Dual-Pivot QuickSort for primitive arrays (`int[]`, `long[]`, etc.) and TimSort for object arrays (`Integer[]`, `String[]`, custom objects). For primitives, stability is irrelevant — there's no concept of identity, only value — and Dual-Pivot QuickSort is faster in practice with better cache utilisation. For object arrays, the Java specification guarantees a stable sort (per `Arrays.sort` Javadoc). This matters because objects have identity and are frequently sorted by one attribute while preserving order from a previous sort by another attribute. TimSort provides both stability and O(n log n) worst-case, with the bonus of O(n) performance on already-sorted or nearly-sorted data.

---

**Q: What is the difference between QuickSort and MergeSort, and when would you choose each?**

A: QuickSort is in-place (O(log n) stack space), not stable, and has O(n²) worst-case but excellent average-case performance due to cache-friendly sequential memory access. MergeSort requires O(n) extra space, is stable, and guarantees O(n log n) in all cases. Choose QuickSort for sorting primitives in-memory where stability doesn't matter and average-case performance is prioritised — it's typically 2-3x faster than MergeSort in practice for random data. Choose MergeSort when stability is required, when sorting linked lists (O(1) extra space with pointer rewiring), or for external sorting where data must be merged from multiple files on disk.

---

**Q: How does TimSort achieve O(n) performance on nearly-sorted data?**

A: TimSort scans the input for "natural runs" — maximal ascending (or descending) sequences already present in the data. If the entire array is already sorted, TimSort detects a single run covering the whole array and returns immediately — O(n) for the scan. For nearly-sorted data with k runs, TimSort's total merge work is proportional to the merge entropy — roughly O(n log k) rather than O(n log n). Short runs below minRun (32-64 elements) are extended with InsertionSort, which is O(n) on nearly-sorted small arrays due to early termination. This adaptive behaviour is why Java chose TimSort for real-world object sorting: production data often has natural partial ordering.

---

**Q: Why can Radix Sort beat O(n log n) — doesn't that violate the sorting lower bound?**

A: The O(n log n) lower bound applies only to **comparison-based** sorts that derive ordering solely from pairwise comparisons. Radix Sort is not comparison-based — it exploits the internal structure of integer keys (individual digits/bits) to sort. The lower bound proof via decision trees assumes the only information extracted is "is a > b?" — which requires at least log₂(n!) comparisons. Radix Sort bypasses this by directly inspecting bit patterns, which encodes more information per operation than a single comparison. The trade-off is that Radix Sort only works on integer-like keys with a fixed representation, whereas comparison sorts are fully general.

---

**Q: What makes a comparator invalid, and what happens in Java if you provide one?**

A: A valid comparator must define a consistent total order: it must be reflexive (`compare(a,a) == 0`), antisymmetric (`compare(a,b) < 0` implies `compare(b,a) > 0`), and transitive (`compare(a,b) < 0` and `compare(b,c) < 0` implies `compare(a,c) < 0`). Violating transitivity is the most common mistake — for example, a comparator that says A > B, B > C, but C > A (rock-paper-scissors). Java's `TimSort` (and its merge operations) assumes transitivity to avoid unnecessary comparisons during galloping mode. An invalid comparator can cause incorrect output, `ArrayIndexOutOfBoundsException`, or infinite loops in TimSort — because TimSort's galloping mode skips comparisons that transitivity would make redundant, but with an intransitive comparator those skipped comparisons could reverse the ordering.

## 11. Key Takeaways

- **QuickSort**: fastest in practice for random data, in-place, not stable, O(n²) worst-case — use 3-way partition for duplicates, random pivot for adversarial inputs
- **MergeSort**: stable, O(n log n) guaranteed, O(n) space — preferred for linked lists and external sorting
- **HeapSort**: O(n log n) guaranteed, O(1) space, not stable — poor cache behaviour makes it slower in practice than QuickSort
- **TimSort**: Java's default for objects — hybrid MergeSort + InsertionSort, stable, adaptive (O(n) on sorted), O(n log n) guaranteed
- **Java distinction**: primitives use Dual-Pivot QuickSort (faster, stability irrelevant); objects use TimSort (stable, spec-required)
- **Stability matters** for multi-key sorting — always use a stable sort or add original index as tiebreaker
- **Radix Sort** beats O(n log n) for large integer arrays (4 passes for 32-bit ints) — bypasses comparison lower bound
- **Comparator pitfalls**: never use subtraction (overflow), always ensure transitivity or risk corrupting TimSort's galloping mode
- **O(n log n) lower bound** applies only to comparison sorts — counting sort, radix sort, and bucket sort escape it by exploiting key structure
