# Functional Error Handling

## Overview

Functional error handling treats errors as values, not control flow—instead of throwing exceptions that can be caught anywhere, return values that explicitly represent success or failure. This makes errors visible in the type system, forces callers to handle them, and enables composition. This guide covers six critical patterns: Optional for Nullable Values (wraps potentially absent value, eliminates NullPointerException at type level, map/flatMap chain operations safely, orElse provides defaults, better than returning null), Either for Success/Failure (Left contains error, Right contains success value, explicit in function signature, forces caller to handle both cases, typed errors better than exceptions), Try for Exception Handling (wraps operations that throw, Success or Failure container, converts exceptions to values, composes with map/flatMap, eliminates try-catch blocks), Result Types (custom containers for domain-specific errors, includes both value and validation errors, richer than Either, can include warnings and metadata), Railway-Oriented Programming (visualize two-track system—success track and failure track, functions switch tracks on error, composition keeps you on correct track, errors propagate automatically without manual checking), and Avoiding Checked Exceptions (checked exceptions break function composition, can't use in lambdas easily, forces defensive catching, functional approach makes errors explicit in return types instead). The key insight: exceptions are like goto statements—they break local reasoning by allowing control flow to jump anywhere. Functional error handling keeps errors local and explicit. Instead of `throw new UserNotFoundException()` that can be caught 10 levels up, return `Either<UserNotFound, User>` that forces the immediate caller to handle it. Common mistakes: overusing Optional where Either would be clearer (Optional says "might be absent" but not why), using Try for domain errors (Try is for unexpected exceptions, Either is for expected failures), not accumulating validation errors (fail on first vs collect all errors), mixing functional and exception-based error handling (pick one style and stick to it), creating Result types without clear error cases.


**The Philosophy:**

```
"Make illegal states unrepresentable."
— Yaron Minsky

Exceptions as control flow:
  - Invisible in type signature
  - Can be caught anywhere
  - Break local reasoning
  - Easy to ignore
  
Errors as values:
  - Visible in type signature
  - Must be handled by caller
  - Local reasoning works
  - Impossible to ignore

Traditional approach:
  public User getUser(String id) throws UserNotFoundException
  // Throws exception. Caller might forget to handle it.

Functional approach:
  public Either<UserNotFound, User> getUser(String id)
  // Returns either error or user. Type system forces handling.

The goal: Errors you can't ignore, handled where they occur,
composed like any other values.
```

## 1. Optional for Nullable Values

### The Anti-Null Pattern

```java
// ========================================
// WITHOUT OPTIONAL: Null everywhere
// ========================================

public class UserService {
    
    // ❌ Null return means "not found" - implicit contract
    public User findUser(String id) {
        User user = database.query(id);
        return user;  // Might be null!
    }
    
    // ❌ Caller must remember to check
    public String getUserCity(String userId) {
        User user = findUser(userId);
        if (user == null) return "Unknown";  // Easy to forget!
        
        Address address = user.getAddress();
        if (address == null) return "Unknown";  // Must check again!
        
        String city = address.getCity();
        return city != null ? city : "Unknown";  // And again!
    }
}

// Problems:
// - No indication in type that null is possible
// - Easy to forget null checks
// - NullPointerException waiting to happen
```

```java
// ========================================
// WITH OPTIONAL: Explicit in type system
// ========================================

public class UserService {
    
    // ✅ Optional makes absence explicit
    public Optional<User> findUser(String id) {
        return database.query(id);  // Returns Optional<User>
        // Type says: "Might not find a user"
    }
    
    // ✅ Chained operations handle absence automatically
    public String getUserCity(String userId) {
        return findUser(userId)                    // Optional<User>
            .flatMap(User::getAddress)             // Optional<Address>
            .map(Address::getCity)                 // Optional<String>
            .orElse("Unknown");                    // String
        // Zero null checks. Optional handles all absence.
    }
}
```

```java
public class OptionalGuidelines {

    public Optional<User> findUserByEmail(String email) {

        // Clearly communicates: might not find a user
    }

        return Optional.ofNullable(middleName);
        // Not everyone has a middle name
    }

    public Optional<Money> calculateRefund(Order order) {

        }
        return Optional.of(order.getTotal());
    }

    public class User {
        private Optional<String> middleName;  // NO! Just use String
        // Fields can be null. That's okay. Use Optional for return types.
    }
    
    // ❌ BAD: Don't use as parameter
    public void updateUser(Optional<String> newEmail) {  // NO!
        // Forces caller to wrap. Just use overloading or nullable parameter.
    }
    
    public void updateUser(String newEmail) { /* ... */ }  // Better
    
    // ❌ BAD: Don't use for collections (use empty collection instead)
    public Optional<List<User>> getUsers() {  // NO!
        return Optional.of(users);
    }
    
    public List<User> getUsers() {  // Better
        return users != null ? users : List.of();  // Empty list, never null
    }
}
```

```java
public class OptionalPatterns {
    
    // Pattern 1: Convert null to Optional
    public Optional<String> getConfig(String key) {
        String value = properties.get(key);  // Might be null
        return Optional.ofNullable(value);   // Wraps in Optional
    }

    public Optional<Order> getUserOrder(String userId) {

        // If user not found, returns empty without calling getCurrentOrder
    }
    
    // Pattern 3: Conditional execution
    public void processUserIfFound(String userId) {
        findUser(userId)
            .filter(User::isActive)              // Only active users
            .ifPresent(user -> {
                // Process user
                emailService.sendWelcome(user);
            });
        // No if statements. Runs only if user found AND active.
    }
    
    // Pattern 4: Fallback chain
    public User getUser(String userId) {
        return findUser(userId)
            .or(() -> findUserInCache(userId))    // Try cache if not in DB
            .or(() -> findUserInArchive(userId))  // Try archive if not in cache
            .orElseGet(() -> User.anonymous());   // Finally, anonymous user
    }
    
    // Pattern 5: Exception on absence
    public User getUserOrFail(String userId) {
        return findUser(userId)
            .orElseThrow(() -> new UserNotFoundException(userId));
        // Converts absence to exception (use sparingly!)
    }
    
    private Optional<User> findUser(String userId) { return Optional.empty(); }
    private Optional<User> findUserInCache(String userId) { return Optional.empty(); }
    private Optional<User> findUserInArchive(String userId) { return Optional.empty(); }
}
```

## 2. Either for Success/Failure

### Explicit Error Types

```java
// ========================================
// VAVR EITHER: Left = Error, Right = Success
// ========================================

import io.vavr.control.Either;

/**
 * Either<L, R>: Container for one of two values
 *   - Left<L>: Error case (by convention)
 *   - Right<R>: Success case
 * 
 * Operations (map, flatMap) only work on Right.
 * Left values pass through unchanged.
 */

public class EitherExamples {
    
    // ✅ Either makes errors explicit in signature
    public Either<String, User> createUser(UserRequest request) {
        if (request.getName() == null || request.getName().isBlank()) {
            return Either.left("Name is required");
        }
        if (request.getEmail() == null || !request.getEmail().contains("@")) {
            return Either.left("Invalid email");
        }
        if (request.getAge() < 18) {
            return Either.left("Must be 18 or older");
        }
        
        User user = new User(request.getName(), request.getEmail(), request.getAge());
        return Either.right(user);
    }
    
    // ✅ Typed errors (better than String)
    public Either<UserError, User> createUserTyped(UserRequest request) {
        if (request.getName() == null || request.getName().isBlank()) {
            return Either.left(UserError.INVALID_NAME);
        }
        if (request.getEmail() == null || !request.getEmail().contains("@")) {
            return Either.left(UserError.INVALID_EMAIL);
        }
        if (request.getAge() < 18) {
            return Either.left(UserError.UNDERAGE);
        }
        
        User user = new User(request.getName(), request.getEmail(), request.getAge());
        return Either.right(user);
    }
    
    enum UserError {
        INVALID_NAME,
        INVALID_EMAIL,
        UNDERAGE,
        EMAIL_ALREADY_EXISTS
    }
}
```

```java
// ========================================
// CHAINING WITH EITHER
// ========================================

public class EitherChaining {
    
    // ✅ Chain operations: first failure short-circuits
    public Either<String, Receipt> processOrder(OrderRequest request) {
        return validateRequest(request)              // Either<String, OrderRequest>
            .flatMap(this::createOrder)              // Either<String, Order>
            .flatMap(this::chargePayment)            // Either<String, PaymentResult>
            .map(this::generateReceipt);             // Either<String, Receipt>
        
        // If validation fails: returns Left, never calls createOrder
        // If createOrder fails: returns Left, never calls chargePayment
        // If all succeed: returns Right<Receipt>
    }
    
    private Either<String, OrderRequest> validateRequest(OrderRequest request) {
        if (request.getItems().isEmpty()) {
            return Either.left("Order must contain items");
        }
        return Either.right(request);
    }
    
    private Either<String, Order> createOrder(OrderRequest request) {
        try {
            Order order = orderService.create(request);
            return Either.right(order);
        } catch (InsufficientInventoryException e) {
            return Either.left("Insufficient inventory: " + e.getProduct());
        }
    }
    
    private Either<String, PaymentResult> chargePayment(Order order) {
        try {
            PaymentResult result = paymentService.charge(order);
            return Either.right(result);
        } catch (PaymentDeclinedException e) {
            return Either.left("Payment declined: " + e.getReason());
        }
    }
    
    private Receipt generateReceipt(PaymentResult result) {
        return new Receipt(result);
    }
}
```

```java
// ========================================
// HANDLING EITHER RESULTS
// ========================================

public class EitherHandling {
    
    // Pattern 1: fold - handle both cases
    public String describeResult(Either<String, User> result) {
        return result.fold(
            error -> "Error: " + error,          // Left case
            user -> "Success: " + user.getName() // Right case
        );
    }
    
    // Pattern 2: getOrElse - extract or provide default
    public User getUserOrAnonymous(String userId) {
        return findUser(userId)
            .getOrElse(User.anonymous());
    }
    
    // Pattern 3: orElse - try alternative on failure
    public Either<String, User> getUserWithFallback(String userId) {
        return findUser(userId)
            .orElse(() -> findUserInCache(userId));
    }
    
    // Pattern 4: map/flatMap on Right only
    public Either<String, String> getUserName(String userId) {
        return findUser(userId)                   // Either<String, User>
            .map(User::getName);                  // Either<String, String>
        // If Left: passes through unchanged
        // If Right: transforms user to name
    }
    
    // Pattern 5: mapLeft - transform error
    public Either<UserError, User> getUserWithTypedError(String userId) {
        return findUser(userId)                   // Either<String, User>
            .mapLeft(this::parseError);           // Either<UserError, User>
    }
    
    // Pattern 6: peek - side effect on Right only
    public Either<String, User> getUserAndLog(String userId) {
        return findUser(userId)
            .peek(user -> logger.info("Found user: {}", user.getId()));
        // Logs only if Right. Returns Either unchanged.
    }
    
    private Either<String, User> findUser(String userId) { 
        return Either.right(new User("Alice", "alice@example.com", 25)); 
    }
    
    private Either<String, User> findUserInCache(String userId) { 
        return Either.left("Not in cache"); 
    }
    
    private UserError parseError(String error) { 
        return UserError.INVALID_NAME; 
    }
    
    enum UserError { INVALID_NAME, INVALID_EMAIL, UNDERAGE }
}
```

## 3. Try for Exception Handling

### Capturing Exceptions as Values

```java
// ========================================
// VAVR TRY: Success or Failure
// ========================================

import io.vavr.control.Try;

/**
 * Try<T>: Result of operation that might throw
 *   - Success<T>: Contains the value
 *   - Failure<T>: Contains the exception
 * 
 * Operations work on Success, skip Failure.
 */

public class TryExamples {
    
    // ❌ Traditional exception handling
    public String traditionalParseInt(String input) {
        try {
            int value = Integer.parseInt(input);
            return "Parsed: " + value;
        } catch (NumberFormatException e) {
            return "Invalid number";
        }
    }
    
    // ✅ Functional exception handling
    public String functionalParseInt(String input) {
        return Try.of(() -> Integer.parseInt(input))  // Try<Integer>
            .map(value -> "Parsed: " + value)         // Try<String>
            .getOrElse("Invalid number");             // String
    }
    
    // ✅ Complex chain with multiple operations that throw
    public Try<Receipt> processPayment(String userId, String orderId) {
        return Try.of(() -> userRepository.findById(userId))        // Try<User>
            .mapTry(user -> orderRepository.findById(orderId))      // Try<Order>
            .mapTry(order -> paymentService.charge(order))          // Try<PaymentResult>
            .map(this::generateReceipt);                            // Try<Receipt>
        
        // Any step that throws: captured in Failure
        // All steps that succeed: Success<Receipt>
        // No try-catch blocks!
    }
    
    private Receipt generateReceipt(PaymentResult result) {
        return new Receipt(result);
    }
}
```

```java
// ========================================
// TRY OPERATIONS
// ========================================

public class TryOperations {
    
    // map: Transform Success value
    public Try<String> getUppercaseName(String userId) {
        return Try.of(() -> userRepository.findById(userId))  // Try<User>
            .map(User::getName)                                // Try<String>
            .map(String::toUpperCase);                         // Try<String>
    }
    
    // mapTry: Transform with operation that throws
    public Try<Double> parseAndCalculate(String input) {
        return Try.of(() -> Double.parseDouble(input))   // Try<Double>
            .mapTry(value -> {
                if (value < 0) throw new IllegalArgumentException("Negative");
                return Math.sqrt(value);
            });
    }
    
    // flatMap: Chain operations that return Try
    public Try<Order> getUserOrder(String userId) {
        return Try.of(() -> userRepository.findById(userId))  // Try<User>
            .flatMap(user -> 
                Try.of(() -> orderRepository.findById(user.getOrderId())));
    }
    
    // recover: Provide fallback on failure
    public String getUserNameSafe(String userId) {
        return Try.of(() -> userRepository.findById(userId))
            .map(User::getName)
            .recover(UserNotFoundException.class, "Anonymous")  // Handle specific exception
            .recover(Exception.class, ex -> "Error: " + ex.getMessage())
            .get();
    }
    
    // recoverWith: Alternative Try on failure
    public Try<User> getUserWithFallback(String userId) {
        return Try.of(() -> userRepository.findById(userId))
            .recoverWith(UserNotFoundException.class, 
                Try.of(() -> userRepository.findByEmail(userId + "@example.com")));
    }
    
    // onFailure: Execute action on failure (side effect)
    public Try<User> getUserAndLogError(String userId) {
        return Try.of(() -> userRepository.findById(userId))
            .onFailure(ex -> logger.error("Failed to fetch user", ex));
    }
    
    // toEither: Convert to Either
    public Either<String, User> getUserAsEither(String userId) {
        return Try.of(() -> userRepository.findById(userId))
            .toEither()                                       // Either<Throwable, User>
            .mapLeft(Throwable::getMessage);                  // Either<String, User>
    }
}
```

```java
    public Try<Document> parseXml(String xml) {

    }

    }

    }

    public Try<User> validateUser(UserRequest request) {  // NO!
        return Try.of(() -> {
            if (request.getAge() < 18) {
                throw new ValidationException("Too young");
            }
            return new User(request);
        });
        // This is a DOMAIN ERROR, not an unexpected exception
        // Use Either<ValidationError, User> instead
    }
    
    // ✅ GOOD: Either for domain errors
    public Either<ValidationError, User> validateUserCorrect(UserRequest request) {
        if (request.getAge() < 18) {
            return Either.left(ValidationError.UNDERAGE);
        }
        return Either.right(new User(request));
    }
```

## 4. Result Types

### Custom Error Containers

```java
// ========================================
// CUSTOM RESULT TYPE: Richer than Either
// ========================================

/**
 * Result<T> captures:
 *   - Success value
 *   - Error details
 *   - Warnings (optional)
 *   - Metadata (optional)
 */

public sealed interface Result<T> {
    
    record Success<T>(T value) implements Result<T> {
        // Contains the success value
    }
    
    record Failure<T>(String error, String code, Throwable cause) implements Result<T> {
        public Failure(String error, String code) {
            this(error, code, null);
        }
    }
    
    // Factory methods
    static <T> Result<T> success(T value) {
        return new Success<>(value);
    }
    
    static <T> Result<T> failure(String error, String code) {
        return new Failure<>(error, code);
    }
    
    static <T> Result<T> failure(String error, String code, Throwable cause) {
        return new Failure<>(error, code, cause);
    }
    
    // Query methods
    default boolean isSuccess() {
        return this instanceof Success;
    }
    
    default boolean isFailure() {
        return this instanceof Failure;
    }
    
    // Transform
    default <R> Result<R> map(Function<T, R> mapper) {
        return switch (this) {
            case Success<T>(var value) -> success(mapper.apply(value));
            case Failure<T> f -> (Result<R>) f;
        };
    }
    
    default <R> Result<R> flatMap(Function<T, Result<R>> mapper) {
        return switch (this) {
            case Success<T>(var value) -> mapper.apply(value);
            case Failure<T> f -> (Result<R>) f;
        };
    }
    
    // Extract
    default T getOrElse(T defaultValue) {
        return switch (this) {
            case Success<T>(var value) -> value;
            case Failure<T> f -> defaultValue;
        };
    }
    
    default T getOrThrow() {
        return switch (this) {
            case Success<T>(var value) -> value;
            case Failure<T> f -> throw new RuntimeException(f.error(), f.cause());
        };
    }
}
```

```java
        try {
            User user = new User(request.getName(), request.getEmail());
            userRepository.save(user);
            return Result.success(user);
        } catch (DatabaseException e) {
            return Result.failure(
                "Failed to save user",

            );

    public Result<Receipt> processOrder(String userId, OrderRequest request) {

            .flatMap(user -> validateOrder(request))         // Result<Order>
            .flatMap(order -> chargePayment(order))          // Result<PaymentResult>

    }

    public ResponseEntity<?> createUserEndpoint(UserRequest request) {
        Result<User> result = createUser(request);
        
        return switch (result) {
            case Result.Success<User>(var user) -> 
                ResponseEntity.ok(user);
                
            case Result.Failure<User> f -> {
                ErrorResponse error = new ErrorResponse(f.error(), f.code());

                    default -> HttpStatus.BAD_REQUEST;
                };
                yield ResponseEntity.status(status).body(error);
            }
        };
    }

        return Result.success(new User("Alice", "alice@example.com")); 
    }
    
    private Result<Order> validateOrder(OrderRequest request) { 
        return Result.success(new Order("order_123")); 
    }
    
    private Result<PaymentResult> chargePayment(Order order) { 
        return Result.success(new PaymentResult(order));

        return new Receipt(result); 
    }
```

```java
// RESULT WITH VALIDATION ERRORS

/**
 * ValidationResult: Accumulates multiple errors
 */
public sealed interface ValidationResult<T> {
    
    record Valid<T>(T value) implements ValidationResult<T> {}
    
    record Invalid<T>(List<ValidationError> errors) implements ValidationResult<T> {
        public Invalid {
            errors = List.copyOf(errors);  // Immutable
        }
    }
    
    record ValidationError(String field, String message, String code) {}
    
    // Factory
    static <T> ValidationResult<T> valid(T value) {
        return new Valid<>(value);
    }
    
    static <T> ValidationResult<T> invalid(List<ValidationError> errors) {
        return new Invalid<>(errors);
    }
    
    static <T> ValidationResult<T> invalid(ValidationError error) {
        return new Invalid<>(List.of(error));
    }
    
    // Combine validations
    static <T> ValidationResult<T> combine(
            ValidationResult<T> first,
            ValidationResult<T> second) {
        return switch (first) {
            case Valid<T>(var value) -> second;
            case Invalid<T> f1 -> switch (second) {
                case Valid<T> v -> first;
                case Invalid<T> f2 -> {
                    List<ValidationError> combined = new ArrayList<>(f1.errors());
                    combined.addAll(f2.errors());
                    yield invalid(combined);
                }
            };
        };
    }
}

public class ValidationExample {
    
    public ValidationResult<User> validateUser(UserRequest request) {
        List<ValidationError> errors = new ArrayList<>();
        
        // Validate all fields, collect all errors
        if (request.getName() == null || request.getName().isBlank()) {
            errors.add(new ValidationError("name", "Name is required", "REQUIRED"));
        } else if (request.getName().length() < 2) {
            errors.add(new ValidationError("name", "Name too short", "MIN_LENGTH"));
        }
        
        if (request.getEmail() == null || !request.getEmail().contains("@")) {
            errors.add(new ValidationError("email", "Invalid email", "INVALID_FORMAT"));
        }
        
        if (request.getAge() < 18) {
            errors.add(new ValidationError("age", "Must be 18 or older", "MIN_VALUE"));
        } else if (request.getAge() > 120) {
            errors.add(new ValidationError("age", "Invalid age", "MAX_VALUE"));
        }
        
        if (!errors.isEmpty()) {
            return ValidationResult.invalid(errors);
        }
        
        return ValidationResult.valid(
            new User(request.getName(), request.getEmail(), request.getAge())
        );
    }
}
```

## 5. Railway-Oriented Programming

### The Two-Track Model

```java
// ========================================
// RAILWAY METAPHOR: Two tracks (success/failure)
// ========================================

/**
 * Visualize error handling as a railway:
 * 
 *   Success Track:  ─────→─────→─────→─────→  (stays on track)
 *   Failure Track:  ─────→─────→─────→─────→  (once failed, stays failed)
 *                        ↓
 *                   [Error switches to failure track]
 * 
 * Functions are "switches" that can move from success to failure track.
 * Once on failure track, stay there (short-circuit).
 */

public class RailwayOriented {
    
    // ✅ Each step is a "switch" that might fail
    public Either<String, Receipt> processOrder(OrderRequest request) {
        return validateRequest(request)       // Switch 1: might fail
            .flatMap(this::checkInventory)    // Switch 2: might fail
            .flatMap(this::createOrder)       // Switch 3: might fail
            .flatMap(this::chargePayment)     // Switch 4: might fail
            .flatMap(this::updateInventory)   // Switch 5: might fail
            .map(this::generateReceipt);      // Final success
        
        // Execution:
        // - validateRequest succeeds? Continue on success track.
        // - validateRequest fails? Switch to failure track, skip rest.
        // - checkInventory succeeds? Stay on success track.
        // - checkInventory fails? Switch to failure track, skip rest.
        // ... and so on
    }
    
    // Each function is a switch: Right keeps you on success track
    private Either<String, OrderRequest> validateRequest(OrderRequest request) {
        if (request.getItems().isEmpty()) {
            return Either.left("Order must contain items");  // Switch to failure
        }
        return Either.right(request);  // Stay on success track
    }
    
    private Either<String, OrderRequest> checkInventory(OrderRequest request) {
        for (OrderItem item : request.getItems()) {
            if (inventoryService.getStock(item.getProductId()) < item.getQuantity()) {
                return Either.left("Insufficient stock: " + item.getProductId());
            }
        }
        return Either.right(request);
    }
    
    private Either<String, Order> createOrder(OrderRequest request) {
        try {
            Order order = orderService.create(request);
            return Either.right(order);
        } catch (Exception e) {
            return Either.left("Failed to create order: " + e.getMessage());
        }
    }
    
    private Either<String, PaymentResult> chargePayment(Order order) {
        try {
            PaymentResult result = paymentService.charge(order);
            return Either.right(result);
        } catch (PaymentDeclinedException e) {
            return Either.left("Payment declined: " + e.getReason());
        }
    }
    
    private Either<String, PaymentResult> updateInventory(PaymentResult result) {
        try {
            inventoryService.decrementStock(result.getOrder().getItems());
            return Either.right(result);
        } catch (Exception e) {
            // Payment already charged! Need to refund!
            return Either.left("Inventory update failed: " + e.getMessage());
        }
    }
    
    private Receipt generateReceipt(PaymentResult result) {
        return new Receipt(result);
    }
}
```

```java
public class RailwayComposition {
    
    // ✅ Two-track functions (Either<E, T>)
    interface Switch<E, T, R> extends Function<T, Either<E, R>> {}

    public static <E, A, B, C> Switch<E, A, C> compose(
            Switch<E, A, B> first,
            Switch<E, B, C> second) {
        return input -> first.apply(input).flatMap(second);
    }

            .flatMap(checkInventory)
```

## 6. Avoiding Checked Exceptions

### Why Checked Exceptions Break Composition

```java
// ========================================
// PROBLEM: Checked exceptions break lambdas
// ========================================

public class CheckedExceptionProblem {
    
    // ❌ Cannot use method that throws checked exception in Stream
    public List<User> loadUsers(List<String> userIds) {
        return userIds.stream()
            .map(id -> userRepository.findById(id))  // COMPILE ERROR!
            // findById() throws UserNotFoundException (checked)
            // Function<T, R> doesn't declare throws
            .toList();
    }
    
    // ❌ Ugly workaround: wrap in unchecked
    public List<User> loadUsersWorkaround(List<String> userIds) {
        return userIds.stream()
            .map(id -> {
                try {
                    return userRepository.findById(id);
                } catch (UserNotFoundException e) {
                    throw new RuntimeException(e);  // Wrap in unchecked
                }
            })
            .toList();
        // Works but ugly. Exception handling scattered.
    }
}
```

```java
// ========================================
// SOLUTION 1: Return Optional instead of throwing
// ========================================

public class OptionalApproach {
    
    // ✅ Return Optional instead of throwing checked exception
    public interface UserRepository {
        Optional<User> findById(String id);  // No throws clause!
    }
    
    // ✅ Works seamlessly in streams
    public List<User> loadUsers(List<String> userIds) {
        return userIds.stream()
            .map(userRepository::findById)      // Stream<Optional<User>>
            .flatMap(Optional::stream)          // Stream<User>
            .toList();
        // Clean! No try-catch. Missing users filtered out.
    }
    
    // ✅ Caller handles absence explicitly
    public User getUser(String userId) {
        return userRepository.findById(userId)
            .orElseThrow(() -> new UserNotFoundException(userId));
        // Throws only if caller wants to
    }
}
```

```java
// ========================================
// SOLUTION 2: Return Try to capture exceptions
// ========================================

public class TryApproach {
    
    // ✅ Methods return Try instead of throwing
    public interface UserRepository {
        Try<User> findById(String id);  // Captures exceptions as values
    }
    
    // ✅ Works in streams and composes nicely
    public List<User> loadUsers(List<String> userIds) {
        return userIds.stream()
            .map(userRepository::findById)           // Stream<Try<User>>
            .filter(Try::isSuccess)                   // Keep only successes
            .map(Try::get)                            // Stream<User>
            .toList();
        // Failures automatically filtered out
    }
    
    // ✅ Or collect both successes and failures
    public record LoadResult(List<User> users, List<String> errors) {}
    
    public LoadResult loadUsersWithErrors(List<String> userIds) {
        Map<Boolean, List<Try<User>>> partitioned = userIds.stream()
            .map(userRepository::findById)
            .collect(Collectors.partitioningBy(Try::isSuccess));
        
        List<User> users = partitioned.get(true).stream()
            .map(Try::get)
            .toList();
        
        List<String> errors = partitioned.get(false).stream()
            .map(Try::getCause)
            .map(Throwable::getMessage)
            .toList();
        
        return new LoadResult(users, errors);
    }
}
```

```java
// FUNCTIONAL EXCEPTION UTILITIES
// ========================================

public class ExceptionUtils {
    
    // ✅ Utility: Wrap checked exception in Function
    @FunctionalInterface
    public interface ThrowingFunction<T, R> {
        R apply(T t) throws Exception;
    }
    
    public static <T, R> Function<T, R> unchecked(ThrowingFunction<T, R> f) {
        return t -> {
            try {
                return f.apply(t);
            } catch (Exception e) {
                throw new RuntimeException(e);
            }
        };
    }
    
    // ✅ Utility: Wrap checked exception in Try
    public static <T, R> Function<T, Try<R>> toTry(ThrowingFunction<T, R> f) {
        return t -> Try.of(() -> f.apply(t));
    }
    
    // ✅ Utility: Wrap checked exception in Either
    public static <T, R> Function<T, Either<String, R>> toEither(
            ThrowingFunction<T, R> f) {
        return t -> {
            try {
                return Either.right(f.apply(t));
            } catch (Exception e) {
                return Either.left(e.getMessage());
            }
        };
    }
    
    // Usage
    public void example(List<String> paths) {
        // Convert checked exception to unchecked
        List<String> contents1 = paths.stream()
            .map(unchecked(Files::readString))
            .toList();
        
        // Convert to Try
        List<String> contents2 = paths.stream()
            .map(toTry(Files::readString))
            .filter(Try::isSuccess)
            .map(Try::get)
            .toList();
        
        // Convert to Either
        List<String> contents3 = paths.stream()
            .map(toEither(Files::readString))
            .filter(Either::isRight)
            .map(Either::get)
            .toList();
    }
}
```

## Summary

```
TECHNIQUE              WHEN TO USE                           BENEFIT
────────────────────────────────────────────────────────────────────────────
Optional               Value might be absent.                Eliminates NPE.
                       "Not found" is a valid case.          Explicit in type.
                       Simple absence, no error details.     Composes nicely.

Either                 Explicit success/failure.             Typed errors.
                       Need error details.                   Forces handling.
                       Domain validation errors.             Railway-oriented.

Try                    Working with code that throws.        Captures exceptions.
                       Unexpected failures (I/O, parsing).   Functional style.
                       Third-party libraries.                Composes with map/flatMap.

Result                 Need error codes + messages.          Rich error info.
                       Multiple error types.                 Metadata support.
                       Custom domain requirements.           HTTP status mapping.

Railway-Oriented       Multi-step workflows.                 Visual model.
                       Each step might fail.                 Short-circuits on failure.
                       Want to avoid nested ifs.             Clean composition.

────────────────────────────────────────────────────────────────────────────

DECISION FRAMEWORK:

Value might be absent?
  → Optional<T>

Expected domain failure?
  → Either<Error, T>

Unexpected exception (I/O, parsing)?
  → Try<T>

Need error codes, HTTP status mapping?
  → Result<T>

Multi-step process where any step fails?
  → Railway-oriented (Either or Result)

Working with checked exceptions?
  → Avoid them! Return Optional/Try/Either instead

────────────────────────────────────────────────────────────────────────────

THE TRANSFORMATION:

Before (exceptions):
  try {
    User user = findUser(id);
    Order order = findOrder(user.getOrderId());
    PaymentResult result = chargePayment(order);
    return generateReceipt(result);
  } catch (UserNotFoundException e) {
    return null;
  } catch (OrderNotFoundException e) {
    return null;
  } catch (PaymentException e) {
    return null;
  }

After (functional):
  return findUser(id)
    .flatMap(user -> findOrder(user.getOrderId()))
    .flatMap(order -> chargePayment(order))
    .map(result -> generateReceipt(result))
    .orElse(null);

Same logic. Zero try-catch. Errors explicit in types.
Composes naturally. Type system forces handling.
```

**Key Takeaway:**

```
Errors as values, not control flow.

Traditional (exceptions):
  - Invisible in type signature
  - Can be caught anywhere
  - Easy to ignore
  - Break composition

Functional (return types):
  - Visible in type signature
  - Must be handled by caller
  - Impossible to ignore
  - Compose naturally

The transformation:

try {
  User user = findUser(id);
  if (user == null) return null;
  Order order = findOrder(user.getOrderId());
  if (order == null) return null;
  return order.getTotal();
} catch (Exception e) {
  return null;
}

becomes:

findUser(id)
  .flatMap(user -> findOrder(user.getOrderId()))
  .map(Order::getTotal)
  .orElse(null);

Zero null checks. Zero try-catch.
The container handles all error propagation.

138 guides in — you now write code where
errors are impossible to ignore.
```
