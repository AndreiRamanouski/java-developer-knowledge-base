# Pure Functions and Immutability

## Overview

Functional programming principles bring predictability, testability, and thread safety to Java code through pure functions and immutable data. Pure functions are functions with no side effects—same inputs always produce same outputs, no hidden state changes, no external interactions, completely deterministic and cacheable. Immutability means objects cannot change after creation—no setters, all fields final, defensive copying on construction, structural sharing for efficiency. This guide covers six core concepts: Defining Pure Functions (mathematical functions translated to code—f(x) always returns same result, referential transparency means you can replace function call with its result, composition works reliably, testing trivial because no setup needed), Benefits of Immutability (thread safety without synchronization, simpler reasoning about code, safe to share references, enables caching and memoization, prevents temporal coupling, eliminates defensive copying), Immutable Collections (Collections.unmodifiableList wraps but doesn't copy, List.of creates truly immutable list, Guava ImmutableList with structural sharing, copy-on-write for rare writes), Persistent Data Structures (immutable data structures that share structure between versions, trees enable O(log n) updates, structural sharing means memory efficient, functional updates return new version without copying everything), Copy-on-Write Strategies (when reads dominate writes use COW, CopyOnWriteArrayList for rare modifications, snapshot entire structure on write, all readers see consistent snapshot), and Performance Implications (immutability adds allocation cost but removes synchronization cost, structural sharing reduces memory overhead, escape analysis can optimize away allocations, modern GCs handle short-lived objects well, thread safety is free with immutability). The key insight: immutability is not about "no changes"—it's about controlled change through new objects. Instead of mutating state in place (dangerous, hard to reason about), you create new versions (safe, predictable). This trades some memory/CPU for massive gains in correctness, maintainability, and concurrency. Common mistakes: using Collections.unmodifiableList thinking it's deep immutable (it's just a wrapper), creating new objects on every change without structural sharing (wasteful), making everything immutable including performance-critical tight loops (overkill), thinking immutability means no state changes (it means explicit new states), confusing immutable with constant (immutable objects can have different values, they just don't change after creation).


**The Philosophy:**

```
"Object-oriented programming makes code understandable
 by encapsulating moving parts.
 Functional programming makes code understandable
 by minimizing moving parts."
 — Michael Feathers

Mutable state is the root of most bugs:
  - Race conditions
  - Temporal coupling
  - Unexpected side effects
  - Difficult testing
  - Unreliable caching

Immutability eliminates an entire class of problems.
The cost: Creating new objects instead of mutating.
The benefit: Code that is predictable, testable, and thread-safe.

Modern Java gives you the tools:
  - Records (immutable by default)
  - List.of(), Set.of(), Map.of()
  - java.util.Optional
  - java.time.* (all immutable)
  - Stream API (pure transformations)
```

## 1. Defining Pure Functions

### Mathematical Functions in Code

```java
// ========================================
// PURE FUNCTION: Same inputs → Same outputs, always
// ========================================

/**
 * A pure function has TWO properties:
 * 
 * 1. DETERMINISTIC: Given the same inputs, always returns the same output
 * 2. NO SIDE EFFECTS: Does not modify any external state
 * 
 * This means:
 *   ✓ Can be cached (memoization)
 *   ✓ Can be called in any order
 *   ✓ Can be called multiple times safely
 *   ✓ Can be tested without mocks
 *   ✓ Can be parallelized safely
 */

// ✅ PURE: Only depends on inputs, no side effects
public class PureFunctions {
    
    public static int add(int a, int b) {
        return a + b;
    }
    // Calling add(2, 3) always returns 5
    // No side effects, no hidden dependencies
    
    public static String toUpperCase(String s) {
        return s.toUpperCase();
    }
    // Pure because String is immutable
    // toUpperCase creates a NEW string, doesn't modify the original
    
    public static List<String> filterLongWords(List<String> words, int minLength) {
        return words.stream()
            .filter(w -> w.length() >= minLength)
            .toList();
    }
    // Pure: Returns NEW list, doesn't modify input
    // Same input always produces same output
    
    public static Money calculateTax(Money amount, double taxRate) {
        return amount.multiply(taxRate);
    }
    // Pure: Depends only on inputs, returns new Money object
    // (Assuming Money is immutable)
}

// ❌ IMPURE: Has side effects or depends on external state
public class ImpureFunctions {
    
    private int counter = 0;  // Mutable state
    
    // ❌ IMPURE: Modifies external state (counter)
    public int getAndIncrement() {
        return counter++;  // Side effect: modifies 'counter'
    }
    // Calling this twice with no inputs returns different values!
    // Not deterministic. Not testable without setup.
    
    // ❌ IMPURE: Depends on external mutable state
    public int getCounter() {
        return counter;  // Result depends on mutable field
    }
    // Returns different values depending on when you call it
    
    // ❌ IMPURE: Modifies input (side effect)
    public void sortList(List<String> words) {
        Collections.sort(words);  // Mutates the input list!
    }
    // Caller's list is changed. Surprising. Dangerous.
    
    // ❌ IMPURE: I/O is a side effect
    public void logMessage(String message) {
        System.out.println(message);  // Side effect: console output
    }
    // Can't call this 100 times without consequences
    
    // ❌ IMPURE: Database access is a side effect
    public User findUser(String id) {
        return database.query("SELECT * FROM users WHERE id = ?", id);
    }
    // Result depends on external state (database)
    // Not deterministic (database could change)
    
    // ❌ IMPURE: Current time is not deterministic
    public boolean isExpired(LocalDateTime expiryDate) {
        return LocalDateTime.now().isAfter(expiryDate);
    }
    // Same input returns different output at different times!
}
```

```java
// ========================================
// REFERENTIAL TRANSPARENCY
// ========================================

/**
 * A function is referentially transparent if you can replace
 * the function call with its return value without changing
 * the program's behavior.
 * 
 * Pure functions are referentially transparent.
 */

// ✅ REFERENTIAL TRANSPARENCY: Can replace call with value
int result = add(2, 3);
// Can be replaced with:
int result = 5;
// Program behaves identically.

List<String> longWords = filterLongWords(words, 5);
// If words is ["hi", "hello", "hey"], can be replaced with:
List<String> longWords = List.of("hello");
// Program behaves identically.

// ❌ NOT REFERENTIALLY TRANSPARENT: Cannot replace with value
int count = getAndIncrement();
// Cannot be replaced with a constant value because the function
// has side effects. Each call returns a different value.

// The benefit: Pure functions are like algebra
//   x = 5;
//   y = x + 3;
//   z = y * 2;
// You can substitute values and reason mathematically.
```

```java
// ========================================
// CONVERTING IMPURE TO PURE
// ========================================

// ❌ IMPURE VERSION: Modifies input
public class OrderProcessor {
    public void applyDiscount(Order order, double discountPercent) {
        double discount = order.getTotal() * discountPercent;
        order.setTotal(order.getTotal() - discount);
        // Mutates the order object - dangerous!
    }
}

// ✅ PURE VERSION: Returns new object
public class OrderProcessor {
    public Order applyDiscount(Order order, double discountPercent) {
        double currentTotal = order.getTotal();
        double discount = currentTotal * discountPercent;
        double newTotal = currentTotal - discount;
        
        // Return NEW order with updated total, don't mutate input
        return new Order(
            order.getId(),
            order.getCustomer(),
            order.getItems(),
            newTotal
        );
    }
}
// Now you can call applyDiscount without worrying about
// side effects. Original order unchanged.

// ──────────────────────────────────────

// ❌ IMPURE VERSION: Depends on current time
public class SessionValidator {
    public boolean isValid(Session session) {
        return session.getExpiryTime().isAfter(LocalDateTime.now());
        // Depends on external state (clock)
    }
}

// ✅ PURE VERSION: Clock passed as parameter
public class SessionValidator {
    public boolean isValid(Session session, LocalDateTime currentTime) {
        return session.getExpiryTime().isAfter(currentTime);
        // Now deterministic: same inputs always produce same output
    }
}
// Caller provides the clock:
//   boolean valid = validator.isValid(session, LocalDateTime.now());
// 
// Benefits:
//   - Testable: Can test with any time you want
//   - Deterministic: Same session + time = same result
//   - Cacheable: Can cache result for given session + time

// ──────────────────────────────────────

// ❌ IMPURE VERSION: Logs (side effect)
public class PaymentService {
    public PaymentResult processPayment(Payment payment) {
        logger.info("Processing payment: {}", payment.getId());  // Side effect
        PaymentResult result = gateway.charge(payment);
        logger.info("Payment result: {}", result.getStatus());  // Side effect
        return result;
    }
}

// ✅ PURE VERSION: Logging pushed to boundary
public class PaymentService {
    public PaymentResult processPayment(Payment payment) {
        // Pure logic - no side effects
        return gateway.charge(payment);
    }
}

// Logging happens at the edge (controller/main):
public class PaymentController {
    public ResponseEntity<?> charge(@RequestBody PaymentRequest req) {
        logger.info("Processing payment: {}", req.getId());
        PaymentResult result = paymentService.processPayment(req.toPayment());
        logger.info("Payment result: {}", result.getStatus());
        return ResponseEntity.ok(result);
    }
}
// Pure functions stay pure. Side effects isolated at boundaries.
```

## 2. Benefits of Immutability

### Why Immutability Matters

```java
// ========================================
// BENEFIT 1: THREAD SAFETY (No synchronization needed)
// ========================================

// ❌ MUTABLE: Requires synchronization
public class MutableCounter {
    private int count = 0;  // Mutable state
    
    public synchronized void increment() {
        count++;  // Must synchronize!
    }
    
    public synchronized int getCount() {
        return count;  // Must synchronize!
    }
}
// Every access requires locking. Slow. Easy to get wrong.

// ✅ IMMUTABLE: Thread-safe by definition
public record ImmutableCounter(int count) {
    public ImmutableCounter increment() {
        return new ImmutableCounter(count + 1);
    }
}
// No synchronization needed. Can be shared freely across threads.
// Multiple threads can read simultaneously without coordination.

// Example: Shared state across threads
ImmutableCounter counter = new ImmutableCounter(0);

// Thread 1:
ImmutableCounter newCounter1 = counter.increment();  // Creates new object

// Thread 2:
ImmutableCounter newCounter2 = counter.increment();  // Creates different new object

// Original 'counter' unchanged. No race condition possible.
// Both threads see consistent state.
```

```java
// ========================================
// BENEFIT 2: EASIER REASONING (No temporal coupling)
// ========================================

// ❌ MUTABLE: Hard to reason about
public class MutableOrder {
    private List<OrderItem> items = new ArrayList<>();
    private double total;
    
    public void addItem(OrderItem item) {
        items.add(item);
        recalculateTotal();  // Must remember to call this!
    }
    
    public void removeItem(OrderItem item) {
        items.remove(item);
        // Oops! Forgot to recalculate total. Bug!
    }
    
    private void recalculateTotal() {
        total = items.stream().mapToDouble(OrderItem::getPrice).sum();
    }
    
    public double getTotal() {
        return total;  // Could be stale if someone forgot recalculate!
    }
}
// State can become inconsistent. Must maintain invariants manually.

// ✅ IMMUTABLE: Always consistent
public record ImmutableOrder(
    List<OrderItem> items,  // Immutable list
    double total) {
    
    // Factory method ensures consistency
    public static ImmutableOrder create(List<OrderItem> items) {
        double total = items.stream().mapToDouble(OrderItem::getPrice).sum();
        return new ImmutableOrder(List.copyOf(items), total);
    }
    
    public ImmutableOrder addItem(OrderItem item) {
        List<OrderItem> newItems = new ArrayList<>(items);
        newItems.add(item);
        return create(newItems);  // Total is recalculated automatically
    }
    
    public ImmutableOrder removeItem(OrderItem item) {
        List<OrderItem> newItems = new ArrayList<>(items);
        newItems.remove(item);
        return create(newItems);  // Total is recalculated automatically
    }
}
// Impossible to have inconsistent state. Total always matches items.
```

```java
// ========================================
// BENEFIT 3: SAFE TO SHARE (No defensive copying)
// ========================================

// ❌ MUTABLE: Must defensive copy
public class MutableOrderService {
    private final List<Order> orders = new ArrayList<>();
    
    public List<Order> getOrders() {
        // Must create defensive copy to prevent caller from modifying
        return new ArrayList<>(orders);  // Expensive!
    }
    
    public void addOrder(Order order) {
        // Must copy incoming list to prevent caller from changing it later
        orders.add(new Order(order));  // Expensive!
    }
}
// Every access requires copying. Slow. Error-prone.

// ✅ IMMUTABLE: Safe to share references
public class ImmutableOrderService {
    private final List<Order> orders;  // Immutable list
    
    public ImmutableOrderService(List<Order> orders) {
        this.orders = List.copyOf(orders);  // One-time copy
    }
    
    public List<Order> getOrders() {
        return orders;  // Safe to return reference - can't be modified!
    }
    
    public ImmutableOrderService addOrder(Order order) {
        List<Order> newOrders = new ArrayList<>(orders);
        newOrders.add(order);
        return new ImmutableOrderService(newOrders);
    }
}
// No defensive copying needed. References can be shared freely.
```

```java
// ========================================
// BENEFIT 4: CACHING AND MEMOIZATION
// ========================================

// ✅ IMMUTABLE: Safe to cache
public class TaxCalculator {
    private final Map<TaxRequest, Money> cache = new ConcurrentHashMap<>();
    
    public Money calculateTax(Order order, TaxRate rate) {
        TaxRequest request = new TaxRequest(order, rate);
        
        // Safe to cache because TaxRequest is immutable
        // Same request will always produce same result
        return cache.computeIfAbsent(request, this::doCalculateTax);
    }
    
    private Money doCalculateTax(TaxRequest request) {
        // Expensive calculation
        return request.order().getTotal().multiply(request.rate().value());
    }
}

// TaxRequest is immutable - safe as cache key
record TaxRequest(Order order, TaxRate rate) {
    // hashCode and equals based on immutable fields
    // Will never change, so safe to use as HashMap key
}

// ❌ Cannot cache if inputs are mutable:
// If Order could change after being used as a key,
// the HashMap would become corrupted (hashCode changes).
```

```java
// ========================================
// BENEFIT 5: PREVENTS TEMPORAL COUPLING
// ========================================

// ❌ MUTABLE: Order of calls matters
public class MutableAccount {
    private double balance;
    
    public void deposit(double amount) {
        balance += amount;
    }
    
    public void withdraw(double amount) {
        balance -= amount;
    }
    
    public double getBalance() {
        return balance;
    }
}

// Dangerous:
MutableAccount account = new MutableAccount();
account.deposit(100);
account.withdraw(50);
double balance1 = account.getBalance();  // 50

// But if you accidentally swap the order:
MutableAccount account2 = new MutableAccount();
account2.withdraw(50);  // Negative balance! Logic error.
account2.deposit(100);
double balance2 = account2.getBalance();  // 50, but wrong semantics

// ✅ IMMUTABLE: Order doesn't matter, explicit state flow
public record ImmutableAccount(double balance) {
    public ImmutableAccount deposit(double amount) {
        return new ImmutableAccount(balance + amount);
    }
    
    public ImmutableAccount withdraw(double amount) {
        if (amount > balance) {
            throw new InsufficientFundsException();
        }
        return new ImmutableAccount(balance - amount);
    }
}

// Explicit state flow:
ImmutableAccount account = new ImmutableAccount(0);
ImmutableAccount afterDeposit = account.deposit(100);
ImmutableAccount afterWithdrawal = afterDeposit.withdraw(50);
double balance = afterWithdrawal.balance();  // 50

// Cannot accidentally withdraw before deposit because you need
// the afterDeposit object to call withdraw on.
// State flow is explicit in the code.
```

```java
// ========================================
// BENEFIT 6: FEARLESS CONCURRENCY
// ========================================

/**
 * Real-world scenario: Processing orders from multiple threads
 */

// ❌ MUTABLE VERSION: Requires complex synchronization
public class MutableOrderProcessor {
    private final Map<String, Order> orders = new ConcurrentHashMap<>();
    private double totalRevenue = 0;  // Shared mutable state!
    
    public void processOrder(Order order) {
        // Multiple steps - hard to make atomic
        synchronized (this) {  // Must lock entire operation
            orders.put(order.getId(), order);
            totalRevenue += order.getTotal();
        }
        // Any other thread that reads totalRevenue during this operation
        // sees inconsistent state. Locking is required everywhere.
    }
    
    public double getTotalRevenue() {
        synchronized (this) {
            return totalRevenue;  // Must lock reads too!
        }
    }
}
// Complex locking. Easy to get wrong. Deadlocks possible.

// ✅ IMMUTABLE VERSION: No synchronization needed
public record OrderSummary(
    Map<String, Order> orders,
    double totalRevenue) {
    
    public static OrderSummary empty() {
        return new OrderSummary(Map.of(), 0);
    }
    
    public OrderSummary addOrder(Order order) {
        Map<String, Order> newOrders = new HashMap<>(orders);
        newOrders.put(order.getId(), order);
        return new OrderSummary(
            Map.copyOf(newOrders),
            totalRevenue + order.getTotal()
        );
    }
}

// Multiple threads can safely process orders:
AtomicReference<OrderSummary> summary = new AtomicReference<>(OrderSummary.empty());

// Thread 1:
summary.updateAndGet(s -> s.addOrder(order1));

// Thread 2:
summary.updateAndGet(s -> s.addOrder(order2));

// AtomicReference handles the atomicity.
// No locks needed in OrderSummary itself.
// Simple. Correct. Fast.
```

## 3. Immutable Collections

### Java's Built-in Immutable Collections

```java
// ========================================
// COLLECTIONS.UNMODIFIABLE* (Wrappers - Be careful!)
// ========================================

// ⚠️ SHALLOW IMMUTABILITY: Wrapper, not truly immutable
List<String> mutableList = new ArrayList<>(List.of("A", "B", "C"));
List<String> unmodifiableList = Collections.unmodifiableList(mutableList);

// This fails - wrapper prevents modification:
unmodifiableList.add("D");  // ❌ UnsupportedOperationException

// But this succeeds - original list can still be modified!
mutableList.add("D");  // ✅ Works!
System.out.println(unmodifiableList);  // [A, B, C, D] - Changed!

// Problem: The unmodifiable wrapper doesn't copy. It just wraps.
// If someone has a reference to the original, they can still modify it.

// Also not deeply immutable:
class MutablePerson {
    private String name;
    public void setName(String name) { this.name = name; }
    public String getName() { return name; }
}

List<MutablePerson> people = new ArrayList<>();
people.add(new MutablePerson());
List<MutablePerson> unmodifiable = Collections.unmodifiableList(people);

// Can't add/remove from list:
unmodifiable.add(new MutablePerson());  // ❌ Fails

// But can modify objects inside:
unmodifiable.get(0).setName("Changed");  // ✅ Works!
// Not deeply immutable.
```

```java
// ========================================
// LIST.OF(), SET.OF(), MAP.OF() (True immutability)
// ========================================

// ✅ TRULY IMMUTABLE: Creates defensive copy
List<String> immutableList = List.of("A", "B", "C");

// All modification attempts fail:
immutableList.add("D");        // ❌ UnsupportedOperationException
immutableList.remove(0);       // ❌ UnsupportedOperationException
immutableList.set(0, "X");     // ❌ UnsupportedOperationException
immutableList.clear();         // ❌ UnsupportedOperationException

// Even if original was mutable, List.of() copies:
List<String> original = new ArrayList<>(List.of("A", "B", "C"));
List<String> copy = List.of(original.toArray(new String[0]));
original.add("D");
System.out.println(copy);  // Still [A, B, C] - unchanged

// Null elements not allowed:
List.of("A", null, "C");  // ❌ NullPointerException
// Design choice: Nulls add complexity, encourage Optional instead

// Set.of() - order not guaranteed
Set<String> immutableSet = Set.of("A", "B", "C");
immutableSet.add("D");  // ❌ UnsupportedOperationException

// Map.of() - convenient for small maps
Map<String, Integer> immutableMap = Map.of(
    "Alice", 25,
    "Bob", 30,
    "Carol", 28
);
immutableMap.put("Dave", 35);  // ❌ UnsupportedOperationException

// For larger maps, use Map.ofEntries():
Map<String, Integer> largeMap = Map.ofEntries(
    Map.entry("Key1", 1),
    Map.entry("Key2", 2),
    // ... 100 more entries
    Map.entry("Key99", 99)
);
```

```java
// LIST.COPYOF() (Defensive copying)
// ========================================

List<String> mutableList = new ArrayList<>(List.of("A", "B", "C"));

// Creates immutable copy:
List<String> immutableCopy = List.copyOf(mutableList);
```

## 4. Persistent Data Structures

### Efficient Immutability Through Structural Sharing

```java
// ========================================
// NAIVE IMMUTABILITY: Full copy (Expensive!)
// ========================================

/**
 * Immutable list that copies entire array on every change.
 * O(n) time and space for every operation.
 */
public class NaiveImmutableList<T> {
    private final T[] elements;
    
    @SafeVarargs
    public NaiveImmutableList(T... elements) {
        this.elements = Arrays.copyOf(elements, elements.length);
    }
    
    public NaiveImmutableList<T> add(T element) {
        // Copy entire array!
        T[] newElements = Arrays.copyOf(elements, elements.length + 1);
        newElements[elements.length] = element;
        return new NaiveImmutableList<>(newElements);
    }
    
    public T get(int index) {
        return elements[index];
    }
    
    public int size() {
        return elements.length;
    }
}

// Problem: Adding 1000 elements requires copying 1000 arrays!
// 1 + 2 + 3 + ... + 1000 = 500,500 element copies
// O(n²) complexity. Very expensive.
```

```java
// ========================================
// STRUCTURAL SHARING: Tree-based (Efficient!)
// ========================================

/**
 * Persistent data structure using tree with structural sharing.
 * 
 * Example: Binary tree immutable list
 * 
 * Original tree:
 *        [A, B]
 *       /      \
 *     [A]      [B]
 * 
 * Add C to create new tree:
 *        [A, B, C]         <- New root
 *       /    |    \
 *     [A]  [B]   [C]       <- [A] and [B] are SHARED from old tree
 *                          <- Only [C] and new root are created
 * 
 * Memory saved: [A] and [B] nodes are reused.
 * Only 2 new nodes created instead of copying all 3.
 */

// Simplified persistent list (conceptual)
public abstract class PersistentList<T> {
    
    // Empty list (singleton)
    private static final PersistentList<?> EMPTY = new Empty<>();
    
    @SuppressWarnings("unchecked")
    public static <T> PersistentList<T> empty() {
        return (PersistentList<T>) EMPTY;
    }
    
    // Core operations
    public abstract PersistentList<T> add(T element);
    public abstract T get(int index);
    public abstract int size();
    
    // Empty list
    private static class Empty<T> extends PersistentList<T> {
        @Override
        public PersistentList<T> add(T element) {
            return new Node<>(element, this);  // Create new node pointing to empty
        }
        
        @Override
        public T get(int index) {
            throw new IndexOutOfBoundsException();
        }
        
        @Override
        public int size() {
            return 0;
        }
    }
    
    // Non-empty list node
    private static class Node<T> extends PersistentList<T> {
        private final T head;
        private final PersistentList<T> tail;
        private final int size;
        
        Node(T head, PersistentList<T> tail) {
            this.head = head;
            this.tail = tail;
            this.size = tail.size() + 1;
        }
        
        @Override
        public PersistentList<T> add(T element) {
            // STRUCTURAL SHARING: Reuse entire current list as tail!
            return new Node<>(element, this);
        }
        
        @Override
        public T get(int index) {
            if (index == 0) return head;
            return tail.get(index - 1);  // Recurse down the tail
        }
        
        @Override
        public int size() {
            return size;
        }
    }
}

// Usage:
PersistentList<String> list1 = PersistentList.empty();
PersistentList<String> list2 = list1.add("A");  // Creates 1 node
PersistentList<String> list3 = list2.add("B");  // Creates 1 node, shares node from list2
PersistentList<String> list4 = list3.add("C");  // Creates 1 node, shares nodes from list3

// Memory structure:
//   list1: []
//   list2: [A] -> []
//   list3: [B] -> [A] -> []
//   list4: [C] -> [B] -> [A] -> []
//
// All versions coexist! Each add creates only 1 new node.
// Previous versions are preserved and share structure.
```

```java
/**
 * Real persistent maps (like Clojure's) use Hash Array Mapped Trie (HAMT).
 * 
 * Key idea: Tree structure where paths share structure.
 * 
 * Original map: {A: 1, B: 2}
 * 
 *     Root
 *    /    \
 *   A:1   B:2
 * 
 * Add C:3:
 * 
 *     Root'        <- New root
 *    /  |  \
 *   A:1 B:2 C:3   <- A:1 and B:2 are SHARED from old tree
 * 
 * Only the new root and C:3 node are created.
 * A:1 and B:2 are reused from the previous version.
 * 
 * Operations: O(log n) instead of O(n)
 * Space overhead: O(log n) new nodes per operation
 */

// Simplified example (real HAMTs are more complex):
public class PersistentMap<K, V> {
    private final Node<K, V> root;
    
    public PersistentMap<K, V> put(K key, V value) {
        Node<K, V> newRoot = root.insert(key, value, key.hashCode(), 0);
        return new PersistentMap<>(newRoot);
    }
    
    public V get(K key) {
        return root.find(key, key.hashCode(), 0);
    }
    
    // Each node in the tree shares children with previous versions
    private static class Node<K, V> {
        Node<K, V> insert(K key, V value, int hash, int level) {
            // Creates new node, shares existing children
            // Only the path from root to new entry is copied
            // All other branches are shared
            // ...
        }
        
        V find(K key, int hash, int level) {
            // Navigates tree to find value
            // ...
        }
    }
}

// Result: O(log n) time and space instead of O(n)
```

## 5. Copy-on-Write Strategies

### When to Copy the Entire Structure

```java
// ========================================
// COPYONWRITEARRAYLIST: Reads dominate writes
// ========================================

/**
 * Copy-on-Write strategy: All writes copy the entire array.
 * All reads require no synchronization.
 * 
 * Use when:
 *   ✓ Reads are much more frequent than writes (10:1 or higher)
 *   ✓ Collection is small (hundreds of elements, not thousands)
 *   ✓ Iteration must be consistent (no ConcurrentModificationException)
 * 
 * Don't use when:
 *   ✗ Writes are frequent (copying is expensive)
 *   ✗ Collection is large (copying thousands of elements is slow)
 *   ✗ Write performance matters
 */

import java.util.concurrent.CopyOnWriteArrayList;

public class EventListenerRegistry {
    // Listeners are registered once, invoked many times
    private final CopyOnWriteArrayList<EventListener> listeners = 
        new CopyOnWriteArrayList<>();
    
    // Write operation (rare): Copies entire array
    public void addListener(EventListener listener) {
        listeners.add(listener);  // Internally copies the array
    }
    
    // Read operation (frequent): No synchronization needed
    public void fireEvent(Event event) {
        for (EventListener listener : listeners) {
            listener.onEvent(event);  // Safe iteration, no locks
        }
    }
}

// How it works internally:
// 1. listeners.add(listener) creates NEW array with old elements + new element
// 2. Atomically updates the reference to point to new array
// 3. Old array is no longer referenced (GC collects it)
// 4. Readers see either old snapshot or new snapshot (both consistent)
// 5. No reader ever sees partial state

// Example of consistency:
CopyOnWriteArrayList<String> list = new CopyOnWriteArrayList<>();
list.add("A");
list.add("B");

// Thread 1: Iterating
for (String s : list) {
    System.out.println(s);  // Sees snapshot at start of iteration
    Thread.sleep(100);      // Even with delay...
}

// Thread 2: Modifying during iteration
list.add("C");
list.add("D");

// Thread 1's iteration is unaffected!
// It sees [A, B] even though list now contains [A, B, C, D]
// No ConcurrentModificationException
```

```java
// COPYONWRITEARRAYSET: Same principle, Set semantics

import java.util.concurrent.CopyOnWriteArraySet;

/**

public class ActiveUsersTracker {

    // Read (frequent): Check if user is active
    public boolean isActive(String userId) {
        return activeUsers.contains(userId);  // No locks
    }
    
    // Read (frequent): Get count
    public int getActiveCount() {
        return activeUsers.size();  // No locks
    }
}

// Perfect use case: User set changes rarely (logins/logouts),
// but we check "is user active?" constantly.
```

```java
// ========================================
// CUSTOM COPY-ON-WRITE: Snapshot pattern
// ========================================

/**
 * Custom immutable snapshot for complex objects.
 */

public class ConfigurationManager {
    // Internal mutable state (only accessed by this class)
    private volatile Map<String, String> config = new HashMap<>();
    
    // Write operation: Replace entire map
    public synchronized void updateConfig(Map<String, String> newConfig) {
        // Create immutable copy
        this.config = Map.copyOf(newConfig);
        // All readers will now see new snapshot
    }
    
    public synchronized void setProperty(String key, String value) {
        // Copy-on-write for single property
        Map<String, String> newConfig = new HashMap<>(config);
        newConfig.put(key, value);
        this.config = Map.copyOf(newConfig);
    }
    
    // Read operation: Return current snapshot
    public Map<String, String> getConfig() {
        return config;  // Immutable, safe to share
    }
    
    public String getProperty(String key) {
        return config.get(key);  // No locks needed for reads
    }
}

// Benefits:
// - Readers never block writers or other readers
// - Readers always see consistent snapshot
// - No ConcurrentModificationException possible
// - Simpler than fine-grained locking

// Trade-off: Write operations are more expensive (full copy)
```

```java
// ========================================
// WHEN NOT TO USE COPY-ON-WRITE
// ========================================

// ❌ BAD: Frequent writes on large collection
public class BadUsage {
    // Don't use COW for collections modified frequently!
    private final CopyOnWriteArrayList<String> messages = 
        new CopyOnWriteArrayList<>();
    
    public void processMessages() {
        for (int i = 0; i < 10000; i++) {
            messages.add("Message " + i);  // Copies array 10,000 times!
            // Each copy is O(n), so total is O(n²)
            // This is catastrophically slow
        }
    }
}

// ✅ GOOD: Use ArrayList with synchronization instead
public class GoodUsage {
    private final List<String> messages = 
        Collections.synchronizedList(new ArrayList<>());
    
    public void processMessages() {
        for (int i = 0; i < 10000; i++) {
            messages.add("Message " + i);  // O(1) amortized
        }
    }
}

// Or better: Use concurrent collection designed for writes
public class BetterUsage {
    private final Queue<String> messages = new ConcurrentLinkedQueue<>();
    
    public void processMessages() {
        for (int i = 0; i < 10000; i++) {
            messages.offer("Message " + i);  // O(1), lock-free
        }
    }
}
```

## 6. Performance Implications

### Understanding the Trade-offs

```java
// ========================================
// COST 1: Object Allocation
// ========================================

/**
 * Immutability means more objects created.
 * But: Modern GCs handle short-lived objects very efficiently.
 */

// Mutable version: 1 object, modified in place
public class MutableCounter {
    private int value;
    
    public void increment() {
        value++;  // Modify in place - no allocation
    }
}

// Immutable version: New object on every change
public record ImmutableCounter(int value) {
    public ImmutableCounter increment() {
        return new ImmutableCounter(value + 1);  // Allocate new object
    }
}

// Cost measurement:
// Incrementing 1 million times:
//   Mutable:   ~1ms (no allocations)
//   Immutable: ~50ms (1 million allocations)
//
// BUT: In real programs, this is rarely the bottleneck.
// Most time is spent on I/O, database queries, network calls.
// And immutability eliminates synchronization cost (can be huge).

/**
 * Escape Analysis Optimization (JVM):
 * 
 * If the JVM can prove an object doesn't escape the method,
 * it can optimize away the allocation entirely (scalar replacement).
 */

public int sumOfSquares(int a, int b) {
    Point p1 = new Point(a, 0);  // Doesn't escape this method
    Point p2 = new Point(0, b);  // Doesn't escape this method
    return p1.distanceSquared() + p2.distanceSquared();
}
// JVM can eliminate Point allocations and inline the math directly.
// Zero allocation cost!

record Point(int x, int y) {
    int distanceSquared() { return x * x + y * y; }
}
```

```java
// ========================================
// BENEFIT 1: No Synchronization Cost
// ========================================

/**
 * Synchronization is expensive.
 * Immutability eliminates synchronization for reads.
 */

// Mutable with synchronization
public class MutableCache {
    private final Map<String, String> cache = new HashMap<>();
    
    public synchronized String get(String key) {
        return cache.get(key);  // Must lock for thread safety
    }
    
    public synchronized void put(String key, String value) {
        cache.put(key, value);  // Must lock
    }
}

// Benchmark: 1000 threads doing 10,000 reads each
//   Mutable synchronized: ~2000ms
//   (Threads contend for lock, serialize execution)

// Immutable with COW
public class ImmutableCache {
    private volatile Map<String, String> cache = Map.of();
    
    public String get(String key) {
        return cache.get(key);  // No lock needed!
    }
    
    public synchronized void put(String key, String value) {
        Map<String, String> newCache = new HashMap<>(cache);
        newCache.put(key, value);
        cache = Map.copyOf(newCache);
    }
}

// Benchmark: 1000 threads doing 10,000 reads each
//   Immutable: ~50ms
//   (Threads read concurrently, no contention)
//
// 40x faster for read-heavy workloads!
```

```java
// ========================================
// BENEFIT 2: Cache Locality and GC Efficiency
// ========================================

/**
 * Immutable objects can be allocated in young generation.
 * Most die quickly, collected very efficiently.
 */

// Typical object lifetime with immutability:
public Money calculateTotal(List<Money> amounts) {
    Money total = Money.ZERO;  // Immutable starting point
    for (Money amount : amounts) {
        total = total.add(amount);  // Create new Money object
        // Previous 'total' is now garbage
    }
    return total;  // Only final total survives
}

// All intermediate Money objects die in young generation.
// Young GC is very fast (<10ms typically).
// This is called "generational hypothesis": Most objects die young.

// GC cost breakdown:
//   Creating 1000 short-lived immutable objects: ~0.1ms GC overhead
//   Creating 1 long-lived mutable object: 0ms GC overhead
//   But synchronizing that mutable object 1000 times: ~10ms overhead
//
// Immutability wins in multi-threaded scenarios!
```

```java
// OPTIMIZATION: Structural Sharing

 * Avoid full copying by sharing structure.

// ❌ Naive immutability: Full copy
public record Order(List<OrderItem> items) {
    public Order addItem(OrderItem item) {
        List<OrderItem> newItems = new ArrayList<>(items);  // Copy all!
        newItems.add(item);
        return new Order(List.copyOf(newItems));
    }
}
// Adding 1000 items: O(n²) time and space

public class PersistentOrder {
    private final PersistentVector<OrderItem> items;

        return new PersistentOrder(items.append(item));
        // Only O(log n) new nodes created
        // Old nodes are shared
    }
}
// Adding 1000 items: O(n log n) time, O(n log n) space

// Real-world comparison:
//   Naive approach: Adding 10,000 items = 50 million element copies
//   Persistent approach: Adding 10,000 items = 130,000 node creations
//   385x fewer allocations!
```

```java
// WHEN TO USE MUTABLE (Performance-critical paths)

/**
 * Tight loops with high allocation pressure:
 * Use mutable locally, expose immutable publicly.

// ✅ Builder pattern: Mutable internally, immutable externally
public final class Customer {
    private final String id;
    private final String name;
    private final List<String> emailAddresses;
    
    private Customer(Builder builder) {
        this.id = builder.id;
        this.name = builder.name;
        this.emailAddresses = List.copyOf(builder.emailAddresses);
    }
    
    public static Builder builder() {
        return new Builder();
    }
    
    // Immutable accessors
    public String id() { return id; }
    public String name() { return name; }
    public List<String> emailAddresses() { return emailAddresses; }
    
    // Mutable builder (not exposed outside)
    public static class Builder {
        private String id;
        private String name;
        private final List<String> emailAddresses = new ArrayList<>();
        
        public Builder id(String id) {
            this.id = id;
            return this;
        }
        
        public Builder name(String name) {
            this.name = name;
            return this;
        }
        
        public Builder addEmail(String email) {
            this.emailAddresses.add(email);  // Mutable!
            return this;
        }
        
        public Customer build() {
            return new Customer(this);  // Immutable result
        }
    }
}

// Usage:

    .id("cust_123")
    .name("Alice")
    .addEmail("alice@example.com")
    .addEmail("alice2@example.com")
    .build();  // Final object is immutable

// - Efficient construction (mutable during building)
// - Immutable result (safe to share)
// - Best of both worlds
```

## Summary

```
CONCEPT                  KEY INSIGHT                         BENEFIT
────────────────────────────────────────────────────────────────────────────
Pure Functions           Same inputs → Same outputs.         Testable, cacheable,
                         No side effects.                    parallelizable.
                         Referentially transparent.          Reason mathematically.

Immutability Benefits    Thread-safe by definition.          No synchronization.
                         No temporal coupling.               Simpler reasoning.
                         Safe to share references.           No defensive copying.
                         Enables caching.                    Performance boost.

Immutable Collections    List.of(), Set.of(), Map.of()      True immutability.
                         are truly immutable.                Null-hostile.
                         Collections.unmodifiable*           Defensive copies.
                         are shallow wrappers.               Safe sharing.

Persistent Data          Structural sharing via trees.       O(log n) updates.
Structures               Versions coexist.                   Memory efficient.
                         Only path copied, rest shared.      Undo/redo trivial.

Copy-on-Write            Entire structure copied on write.   Reads never block.
                         Reads see consistent snapshot.      No locks for reads.
                         Use when reads >> writes.           Consistent iteration.

Performance              Allocation cost < Sync cost.        Profile first.
                         GC handles short-lived well.        Escape analysis helps.
                         Immutable by default.               Mutable when measured.

────────────────────────────────────────────────────────────────────────────

THE CORE INSIGHT:

Immutability is not about "no changes."
It's about controlled, explicit change through new objects.

Instead of:
  account.setBalance(newBalance);  // Hidden mutation
  
Do:
  Account newAccount = account.withBalance(newBalance);  // Explicit new state

This trades:
  - Some memory (new objects)
  - Some CPU (copying/allocation)

For:
  - Thread safety (no synchronization needed)
  - Simpler reasoning (no temporal coupling)
  - Reliable caching (objects never change)
  - Fearless concurrency (share freely)
  - Easier testing (no setup/teardown)

Modern Java makes this practical:
  - Records (immutable by default)
  - List.of(), Set.of(), Map.of()
  - java.time.* (all immutable)
  - Stream API (pure transformations)
  - Pattern matching (structural decomposition)

The result: Code that is predictable, testable, and thread-safe.
```

**Key Takeaway:**

```
Immutability is not a constraint.
It is a superpower.

Mutable state:
  account.setBalance(100);
  account.setBalance(200);
  // What happened between these lines?
  // Did someone else modify it?
  // Is it thread-safe?
  // Can I trust the balance?

  // Can share freely

Modern JVMs optimize away allocations (escape analysis).
The cost is negligible. The benefit is enormous.
```
