# Higher-Order Functions

## Overview

Higher-order functions are functions that operate on other functions—they take functions as parameters or return functions as results. This enables powerful abstraction patterns: instead of repeating similar code with slight variations, you pass the variation as a function parameter; instead of hardcoding behavior, you return a function that captures that behavior. This guide covers six critical aspects: Functions as Parameters, Functions as Return Values (factories that create customized behavior—return a validator configured for specific rules, return a formatter specialized for a context, closures capture environment), Function Composition (build complex operations from simple ones—f.andThen(g) pipes output of f into g, f.compose(g) applies g first then f, composing validators creates validation chains, composing transformations creates data pipelines), Currying in Java (transform multi-parameter function into chain of single-parameter functions—calculate(a, b, c) becomes calculate(a).apply(b).apply(c), enables partial application, simulated in Java using nested lambdas or method chaining), Partial Application (fix some arguments, create specialized function—createUser(name, email, role) becomes createAdmin(name, email) where role is pre-filled, reduces boilerplate, captures common patterns), and Real-World Examples (sorting with Comparator, stream operations with predicates, validation chains, middleware patterns, retry logic, caching decorators, authentication filters). The key insight: higher-order functions are about abstraction and composition—you separate "what to do" from "when to do it." Instead of writing ten methods that differ only in one calculation, write one method that takes the calculation as a parameter. Instead of duplicating retry logic everywhere, write a higher-order function that adds retry to any operation. This is the functional equivalent of dependency injection—behavior is injected as a function rather than an object. Common mistakes: overusing functions when a class would be clearer (functions are great for simple behavior, objects for complex state + behavior), nesting too deeply (callbacks within callbacks within callbacks becomes unreadable), ignoring type inference (verbose lambda types when compiler can infer), not using method references (::methodName is clearer than x -> obj.methodName(x)), thinking functional means no objects (Java is multi-paradigm, use both).


**The Philosophy:**

```
"Functions are first-class citizens."

This means:
  - Functions can be assigned to variables
  - Functions can be passed as arguments
  - Functions can be returned from functions
  - Functions can be stored in data structures

This enables:
  - Strategy pattern without creating classes
  - Template Method pattern with functions
  - Decorator pattern by composing functions
  - Dependency injection of behavior

Higher-order functions let you:
  - Abstract over behavior (not just data)
  - Compose complex operations from simple ones
  - Reduce code duplication dramatically
  - Create DSLs for specific domains

The result: Code that is more generic, reusable,
and composable.
```

```java
public class ListOperations {
    
    // ❌ Different method for each operation
    public List<Integer> filterEvens(List<Integer> numbers) {
        List<Integer> result = new ArrayList<>();
        for (Integer n : numbers) {
            if (n % 2 == 0) {
                result.add(n);
            }
        }
        return result;
    }
    
    public List<Integer> filterOdds(List<Integer> numbers) {
        List<Integer> result = new ArrayList<>();
        for (Integer n : numbers) {
            if (n % 2 != 0) {  // Only this line differs!
                result.add(n);
            }
        }
        return result;
    }
    
    public List<Integer> filterPositive(List<Integer> numbers) {
        List<Integer> result = new ArrayList<>();
        for (Integer n : numbers) {
            if (n > 0) {  // Only this line differs!
                result.add(n);
            }
        }
        return result;
    }
    
    // Need 10 more filters? Write 10 more methods!
}

// ========================================
// WITH HIGHER-ORDER FUNCTIONS: Generic abstraction
// ========================================

public class ListOperations {
    
    // ✅ ONE method that accepts behavior as parameter
    public <T> List<T> filter(List<T> items, Predicate<T> predicate) {
        List<T> result = new ArrayList<>();
        for (T item : items) {
            if (predicate.test(item)) {  // Behavior injected here
                result.add(item);
            }
        }
        return result;
    }
}

// Usage: Pass different predicates for different behavior
ListOperations ops = new ListOperations();
List<Integer> numbers = List.of(1, 2, 3, 4, 5, 6);

List<Integer> evens = ops.filter(numbers, n -> n % 2 == 0);
List<Integer> odds = ops.filter(numbers, n -> n % 2 != 0);
List<Integer> positive = ops.filter(numbers, n -> n > 0);
List<Integer> greaterThan3 = ops.filter(numbers, n -> n > 3);

// Infinite flexibility with ONE method!
```

```java
// ========================================
// JAVA'S FUNCTIONAL INTERFACES

/**
 * Java provides standard functional interfaces in java.util.function
 * 
 * Function<T, R>    : T -> R          (transformation)
 * Predicate<T>      : T -> boolean    (test/filter)
 * Consumer<T>       : T -> void       (side effect)
 * Supplier<T>       : () -> T         (factory)
 * BiFunction<T,U,R> : (T, U) -> R     (two inputs)
 * UnaryOperator<T>  : T -> T          (same input/output type)
 */

import java.util.function.*;

public class FunctionalInterfaceExamples {
    
    // Function<T, R>: Transform one type to another
    public <T, R> List<R> map(List<T> items, Function<T, R> mapper) {
        return items.stream()
            .map(mapper)
            .toList();
    }
    
    // Usage:
    List<String> names = List.of("Alice", "Bob", "Carol");
    List<Integer> lengths = map(names, String::length);  // [5, 3, 5]
    List<String> upper = map(names, String::toUpperCase);  // [ALICE, BOB, CAROL]
    
    // Predicate<T>: Test/filter
    public <T> boolean anyMatch(List<T> items, Predicate<T> predicate) {
        for (T item : items) {
            if (predicate.test(item)) {
                return true;
            }
        }
        return false;
    }
    
    // Usage:
    boolean hasLongName = anyMatch(names, name -> name.length() > 4);  // true (Alice, Carol)
    
    // Consumer<T>: Perform action (side effect)
    public <T> void forEach(List<T> items, Consumer<T> action) {
        for (T item : items) {
            action.accept(item);
        }
    }
    
    // Usage:
    forEach(names, System.out::println);  // Print each name
    forEach(names, name -> logger.info("Processing: {}", name));
    
    // Supplier<T>: Factory/generator
    public <T> List<T> generate(int count, Supplier<T> supplier) {
        List<T> result = new ArrayList<>();
        for (int i = 0; i < count; i++) {
            result.add(supplier.get());
        }
        return result;
    }
    
    // Usage:
    List<Double> randomNumbers = generate(10, Math::random);
    List<UUID> ids = generate(5, UUID::randomUUID);
    
    // BiFunction<T, U, R>: Two inputs -> one output
    public <T, U, R> R combine(T first, U second, BiFunction<T, U, R> combiner) {
        return combiner.apply(first, second);
    }
    
    // Usage:
    String fullName = combine("John", "Doe", (first, last) -> first + " " + last);
    Integer sum = combine(10, 20, (a, b) -> a + b);
}
```

```java
public class Validator<T> {
    
    // ✅ Generic validation logic
    public ValidationResult validate(T value, Predicate<T> rule, String errorMessage) {
        if (rule.test(value)) {
            return ValidationResult.valid();
        }
        return ValidationResult.invalid(errorMessage);
    }
    
    // ✅ Multiple rules combined
    public ValidationResult validateAll(T value, List<ValidationRule<T>> rules) {
        List<String> errors = new ArrayList<>();
        for (ValidationRule<T> rule : rules) {
            if (!rule.predicate().test(value)) {
                errors.add(rule.errorMessage());
            }
        }
        return errors.isEmpty() 
            ? ValidationResult.valid() 
            : ValidationResult.invalid(errors);
    }
}

record ValidationRule<T>(Predicate<T> predicate, String errorMessage) {}

// Usage: Different validation rules for different contexts
Validator<String> validator = new Validator<>();

// Email validation
ValidationResult emailResult = validator.validateAll(email, List.of(
    new ValidationRule<>(s -> s != null && !s.isBlank(), "Email is required"),
    new ValidationRule<>(s -> s.contains("@"), "Email must contain @"),
    new ValidationRule<>(s -> s.length() <= 255, "Email too long")
));

// Password validation
ValidationResult passwordResult = validator.validateAll(password, List.of(
    new ValidationRule<>(s -> s.length() >= 8, "Password must be at least 8 characters"),
    new ValidationRule<>(s -> s.matches(".*[A-Z].*"), "Password must contain uppercase"),
    new ValidationRule<>(s -> s.matches(".*[0-9].*"), "Password must contain digit"),
    new ValidationRule<>(s -> s.matches(".*[!@#$%^&*].*"), "Password must contain special character")
));
```

```java
public class PredicateFactory {
    
    // ✅ Returns a function configured for specific criteria
    public static <T extends Comparable<T>> Predicate<T> greaterThan(T threshold) {
        return value -> value.compareTo(threshold) > 0;
    }
    
    public static <T extends Comparable<T>> Predicate<T> between(T min, T max) {
        return value -> value.compareTo(min) >= 0 && value.compareTo(max) <= 0;
    }
    
    public static <T> Predicate<T> not(Predicate<T> predicate) {
        return value -> !predicate.test(value);
    }
}

// Usage: Create customized predicates
```

```java
public class DiscountCalculator {
    
    // ✅ Returns a function that remembers the discount rate
    public static Function<Double, Double> createDiscounter(double discountPercent) {
        return price -> price * (1 - discountPercent / 100);
        // The lambda captures 'discountPercent' from enclosing scope
    }
    
    public static Function<Double, Double> createBulkDiscounter(
            double discountPercent, 
            double minimumAmount) {
        return price -> {
            if (price >= minimumAmount) {
                return price * (1 - discountPercent / 100);
            }
            return price;
        };
        // Captures both discountPercent and minimumAmount
    }
}

// Usage: Create specialized discounters
Function<Double, Double> memberDiscount = DiscountCalculator.createDiscounter(10);
Function<Double, Double> vipDiscount = DiscountCalculator.createDiscounter(20);
Function<Double, Double> bulkDiscount = DiscountCalculator.createBulkDiscounter(15, 100.0);

// Apply to different prices
double price1 = 50.0;
System.out.println(memberDiscount.apply(price1));  // 45.0 (10% off)
System.out.println(vipDiscount.apply(price1));     // 40.0 (20% off)
System.out.println(bulkDiscount.apply(price1));    // 50.0 (no discount, < $100)

double price2 = 150.0;
System.out.println(bulkDiscount.apply(price2));    // 127.5 (15% off, >= $100)
```

```java
// REAL EXAMPLE: Retry decorator

public class RetryDecorator {
    
    // ✅ Returns a function that adds retry logic to any operation
    public static <T> Supplier<T> withRetry(
            Supplier<T> operation,
            int maxAttempts,
            Duration delayBetweenAttempts) {
        
        return () -> {
            int attempts = 0;
            while (true) {
                try {
                    return operation.get();
                } catch (Exception e) {
                    attempts++;
                    if (attempts >= maxAttempts) {
                        throw new RuntimeException("Failed after " + attempts + " attempts", e);
                    }
                    try {
                        Thread.sleep(delayBetweenAttempts.toMillis());
                    } catch (InterruptedException ie) {
                        Thread.currentThread().interrupt();
                        throw new RuntimeException("Interrupted during retry", ie);
                    }
                }
            }
        };
    }
}

// Usage: Add retry to any operation
Supplier<User> fetchUser = () -> userService.fetchFromDatabase("user_123");

// Wrap with retry logic
Supplier<User> fetchUserWithRetry = RetryDecorator.withRetry(
    fetchUser,
    3,  // max 3 attempts
    Duration.ofSeconds(1)  // 1 second between attempts
);

// Now fetchUserWithRetry automatically retries on failure
User user = fetchUserWithRetry.get();

// Apply to different operations:
Supplier<Order> fetchOrderWithRetry = RetryDecorator.withRetry(
    () -> orderService.fetch("order_456"),
    5,
    Duration.ofMillis(500)
);
```

```java
    // ✅ Returns a function that adds caching to any function
    public static <T, R> Function<T, R> withCache(Function<T, R> expensive) {
        Map<T, R> cache = new ConcurrentHashMap<>();
```

## 3. Function Composition

### Building Complex Functions from Simple Ones

```java
// ========================================
// FUNCTION.ANDTHEN: f then g
// ========================================

/**
 * andThen: Executes this function, then the next function
 * 
 * f.andThen(g) is equivalent to: x -> g(f(x))
 */

import java.util.function.Function;

public class FunctionComposition {
    
    // Simple functions
    Function<String, String> trim = String::trim;
    Function<String, String> lowercase = String::toLowerCase;
    Function<String, String> removeSpaces = s -> s.replaceAll("\\s+", "");
    
    // ✅ Compose into pipeline
    Function<String, String> normalize = trim
        .andThen(lowercase)
        .andThen(removeSpaces);
    
    // Usage:
    String input = "  Hello World  ";
    String result = normalize.apply(input);  // "helloworld"
    
    // Execution order: trim -> lowercase -> removeSpaces
    //   "  Hello World  " -> "Hello World" -> "hello world" -> "helloworld"
}

// ========================================
// FUNCTION.COMPOSE: g then f
// ========================================

/**
 * compose: Executes the argument function first, then this function
 * 
 * f.compose(g) is equivalent to: x -> f(g(x))
 */

public class ComposeExample {
    
    Function<Integer, Integer> multiplyBy2 = x -> x * 2;
    Function<Integer, Integer> add10 = x -> x + 10;
    
    // ✅ andThen: multiply THEN add
    Function<Integer, Integer> pipeline1 = multiplyBy2.andThen(add10);
    int result1 = pipeline1.apply(5);  // (5 * 2) + 10 = 20
    
    // ✅ compose: add THEN multiply
    Function<Integer, Integer> pipeline2 = multiplyBy2.compose(add10);
    int result2 = pipeline2.apply(5);  // (5 + 10) * 2 = 30
    
    // andThen vs compose:
    //   f.andThen(g)  =  x -> g(f(x))   "Do f, then g"
    //   f.compose(g)  =  x -> f(g(x))   "Do g, then f"
}
```

```java
/**
 * Middleware: Functions that wrap request handlers
 * Common in web frameworks
 */

interface RequestHandler extends Function<Request, Response> {}

public class MiddlewarePattern {
    
    // ✅ Authentication middleware
    static RequestHandler withAuth(RequestHandler handler) {
        return request -> {
            if (!request.hasValidToken()) {
                return Response.unauthorized();
            }
            return handler.apply(request);
        };
    }
    
    // ✅ Logging middleware
    static RequestHandler withLogging(RequestHandler handler) {
        return request -> {
            System.out.println("Request: " + request.getPath());
            Response response = handler.apply(request);
            System.out.println("Response: " + response.getStatus());
            return response;
        };
    }
    
    // ✅ Rate limiting middleware
    static RequestHandler withRateLimit(RequestHandler handler, int maxRequests) {
        Map<String, Integer> counts = new ConcurrentHashMap<>();
        return request -> {
            String clientId = request.getClientId();
            int count = counts.merge(clientId, 1, Integer::sum);
            if (count > maxRequests) {
                return Response.tooManyRequests();
            }
            return handler.apply(request);
        };
    }
    
    // ✅ Compose middleware
    public static RequestHandler createHandler() {
        RequestHandler baseHandler = request -> {
            // Actual business logic
            return Response.ok("Hello " + request.getUser());
        };
        
        // Wrap with middleware
        return withLogging(
            withAuth(
                withRateLimit(baseHandler, 100)
            )
        );
        // Execution order: logging -> auth -> rate limit -> business logic
    }
}
```

## 4. Currying in Java

### Transforming Multi-Parameter Functions

```java
// ========================================
// WHAT IS CURRYING?
// ========================================

/**
 * Currying: Transform a function with N parameters into
 * N functions with 1 parameter each.
 * 
 * Non-curried: f(a, b, c) -> result
 * Curried:     f(a) -> (b -> (c -> result))
 * 
 * Enables partial application: fix some arguments,
 * get a function that expects the rest.
 */

// ❌ Non-curried (standard Java)
public class NonCurried {
    public static int add(int a, int b, int c) {
        return a + b + c;
    }
    
    // Must provide all arguments at once
    int result = add(1, 2, 3);  // 6
}

// ✅ Curried (simulated in Java)
public class Curried {
    public static Function<Integer, Function<Integer, Function<Integer, Integer>>> add() {
        return a -> b -> c -> a + b + c;
    }
    
    // Can provide arguments one at a time
    var step1 = add().apply(1);           // Function<Integer, Function<Integer, Integer>>
    var step2 = step1.apply(2);           // Function<Integer, Integer>
    var result = step2.apply(3);          // 6
    
    // Or all at once
    var resultDirect = add().apply(1).apply(2).apply(3);  // 6
}
```

```java
// ========================================
// CURRYING: BiFunction to curried form
// ========================================

public class CurryingExamples {
    
    // ✅ Curry a BiFunction<T, U, R>
    public static <T, U, R> Function<T, Function<U, R>> curry(BiFunction<T, U, R> biFunction) {
        return t -> u -> biFunction.apply(t, u);
    }
    
    // ✅ Uncurry back to BiFunction
    public static <T, U, R> BiFunction<T, U, R> uncurry(Function<T, Function<U, R>> curried) {
        return (t, u) -> curried.apply(t).apply(u);
    }
    
    public static void main(String[] args) {
        // Original BiFunction
        BiFunction<Integer, Integer, Integer> multiply = (a, b) -> a * b;
        
        // Curry it
        Function<Integer, Function<Integer, Integer>> curriedMultiply = curry(multiply);
        
        // Use curried form (partial application)
        Function<Integer, Integer> multiplyBy5 = curriedMultiply.apply(5);
        System.out.println(multiplyBy5.apply(10));  // 50
        System.out.println(multiplyBy5.apply(20));  // 100
        
        // Uncurry back
        BiFunction<Integer, Integer, Integer> uncurriedMultiply = uncurry(curriedMultiply);
        System.out.println(uncurriedMultiply.apply(3, 4));  // 12
    }
}
```

## 5. Partial Application

### Fixing Arguments to Create Specialized Functions

```java
// ========================================
// PARTIAL APPLICATION: Fix some arguments
// ========================================

/**
 * Partial application: Take a function with N parameters,
 * fix M of them, get a function with N-M parameters.
 * 
 * Different from currying:
 *   Currying:  f(a,b,c) -> f(a)(b)(c)  (transforms signature)
 *   Partial:   f(a,b,c) + fix(a=1) -> g(b,c) = f(1,b,c)  (reduces parameters)
 */

public class PartialApplication {
    
    // ✅ Utility method for partial application
    public static <T, U, R> Function<U, R> partial(
            BiFunction<T, U, R> function,
            T fixedArg) {
        return u -> function.apply(fixedArg, u);
    }
    
    // ✅ Three-argument version
    public static <T, U, V, R> BiFunction<U, V, R> partial(
            TriFunction<T, U, V, R> function,
            T fixedArg) {
        return (u, v) -> function.apply(fixedArg, u, v);
    }
    
    interface TriFunction<T, U, V, R> {
        R apply(T t, U u, V v);
    }
    
    public static void main(String[] args) {
        // Original function
        BiFunction<String, Integer, String> repeat = (str, times) -> str.repeat(times);
        
        // Partial application: fix the string
        Function<Integer, String> repeatHello = partial(repeat, "Hello ");
        
        System.out.println(repeatHello.apply(3));  // "Hello Hello Hello "
        System.out.println(repeatHello.apply(5));  // "Hello Hello Hello Hello Hello "
        
        // Different partial application: fix the count
        Function<String, String> repeat3Times = partial(
            (BiFunction<Integer, String, String>) (times, str) -> str.repeat(times),
            3
        );
        
        System.out.println(repeat3Times.apply("Hi "));  // "Hi Hi Hi "
    }
}
```

```java
// EXAMPLE 3: Retry with exponential backoff

public class RetryUtils {
    
    public static <T> T retry(
            Supplier<T> operation,
            int maxAttempts,
            Predicate<Exception> shouldRetry,
            Function<Integer, Duration> backoffStrategy) {
        
        int attempts = 0;
        while (true) {
            try {
                return operation.get();
            } catch (Exception e) {
                attempts++;
                if (attempts >= maxAttempts || !shouldRetry.test(e)) {
                    throw new RuntimeException("Failed after " + attempts + " attempts", e);
                }
                Duration delay = backoffStrategy.apply(attempts);
                sleep(delay);
            }
        }
    }
    
    // ✅ Predefined backoff strategies
    public static Function<Integer, Duration> exponentialBackoff(Duration initial) {
        return attempt -> initial.multipliedBy((long) Math.pow(2, attempt - 1));
    }
    
    public static Function<Integer, Duration> linearBackoff(Duration increment) {
        return attempt -> increment.multipliedBy(attempt);
    }
    
    public static Function<Integer, Duration> fixedBackoff(Duration delay) {
        return attempt -> delay;
    }
    
    private static void sleep(Duration duration) {
        try {
            Thread.sleep(duration.toMillis());
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new RuntimeException(e);
        }
    }
}

// Usage: Different retry strategies for different operations
User user = RetryUtils.retry(
    () -> userService.fetch("user_123"),
    5,  // max 5 attempts
    e -> e instanceof NetworkException,  // Only retry network errors
    RetryUtils.exponentialBackoff(Duration.ofSeconds(1))  // 1s, 2s, 4s, 8s, 16s
);

Order order = RetryUtils.retry(
    () -> orderService.fetch("order_456"),
    3,  // max 3 attempts
    e -> !(e instanceof ValidationException),  // Don't retry validation errors
    RetryUtils.linearBackoff(Duration.ofMillis(500))  // 500ms, 1000ms, 1500ms
);
```

```java
    }

    static Predicate<User> canViewDashboard = isAuthenticated.and(isActive);

        .and(isActive)
        .and(isAdmin);

        .and(isActive)
        .and(isPremium);
    
    static Predicate<User> canEditResource(String resourceId) {
        return isAuthenticated
            .and(isActive)
            .and(isAdmin.or(ownsResource(resourceId)));
    }

    public static <T> Function<User, T> requireAuth(
            Predicate<User> authRule,
            Function<User, T> operation) {

            if (!authRule.test(user)) {
                throw new UnauthorizedException();
            }
            return operation.apply(user);
        };
    }

        // Wrap operations with authorization

            canViewDashboard,
```

## Summary

```
CONCEPT                  KEY INSIGHT                         ENABLES
────────────────────────────────────────────────────────────────────────────
Functions as            Pass behavior as parameter.         Strategy pattern without classes.
Parameters              Predicate, Function, Consumer,      Generic filter, map, reduce.
                        Supplier, Comparator.               Infinite flexibility, one method.

Functions as            Return configured behavior.         Factories of specialized functions.
Return Values           Closures capture environment.       Retry decorators, caching.
                        Create specialized functions.       Middleware patterns.

Function                Build complex from simple.          Validation chains.
Composition             f.andThen(g), f.compose(g)         Data pipelines.
                        Predicate.and, Predicate.or        Authorization rules.

Currying                f(a,b,c) -> f(a)(b)(c)            Enables partial application.
                        One parameter at a time.            Specialized functions.
                        Simulated with nested lambdas.      Fluent APIs.

Partial Application     Fix some arguments.                 Specialized loggers.
                        f(a,b,c) + fix(a) -> g(b,c)       Database query builders.
                        Reduce parameter count.             HTTP client builders.

Real-World Examples     Strategy: Functions in map.         Eliminate class hierarchies.
                        Template Method: Inject hooks.      Reduce boilerplate.
                        Retry: Configurable backoff.        Composable behaviors.

────────────────────────────────────────────────────────────────────────────

THE TRANSFORMATION:

Before (OOP):
  - Strategy pattern: Interface + 5 implementations
  - Template Method: Abstract class + 3 subclasses
  - Decorator: Wrapper classes for each behavior
  - Result: 10+ classes, complex hierarchy

After (Functional):
  - Strategy: Map<String, Function<>>
  - Template Method: Constructor(Consumer, Consumer)
  - Decorator: withRetry(operation), withCache(operation)
  - Result: Functions, zero classes

The insight: Higher-order functions are the functional equivalent
of dependency injection. Instead of injecting objects that contain
behavior, you inject the behavior directly as functions.

Benefits:
  ✓ Less boilerplate (no interface/class ceremony)
  ✓ More composable (functions compose naturally)
  ✓ More flexible (create on the fly)
  ✓ More testable (easy to create test doubles)

Trade-offs:
  ✗ Complex state + behavior → Use objects
  ✗ Deep nesting → Hard to read
  ✗ Need lifecycle → Objects better

Use functions for simple behavior.
Use objects for complex state + behavior.
Java is multi-paradigm. Use both.
```
