# Monads in Java

## Overview

A monad is a design pattern for composing operations that have context—Optional handles potential absence, Stream handles multiple values, CompletableFuture handles asynchrony, Try handles exceptions, Either handles success-or-failure. The unifying pattern: monads wrap values in a context, provide operations that work within that context, and let you chain transformations without escaping the context. This guide covers seven critical aspects: Optional Monad (wraps potentially absent value, map transforms the value if present, flatMap chains operations that return Optional, avoids null checks throughout code, orElse provides default value), Stream Monad (wraps zero-or-more values, map transforms each element, flatMap expands collections, filters remove elements, lazy evaluation until terminal operation), CompletableFuture Monad (wraps asynchronous computation, thenApply transforms result, thenCompose chains async operations, handles threading automatically, exception handling with exceptionally), Try Monad from Vavr (wraps operation that might throw exception, Success or Failure container, map/flatMap work on success path, recover provides fallback, eliminates try-catch blocks), Either Monad (wraps one of two possible values—Left for error, Right for success, map/flatMap only transform Right, provides railway-oriented programming pattern), Monad Laws (Left Identity: wrapping then transforming equals just transforming, Right Identity: transforming with identity does nothing, Associativity: order of nesting doesn't matter, these ensure predictable behavior), and Practical Applications (when to use each monad, chaining database queries with Optional, processing collections with Stream, async workflows with CompletableFuture, error handling with Try/Either, avoiding nested if statements). The key insight: monads are containers with a superpower—they let you chain operations while the container handles the messy parts (null safety, exception handling, asynchrony, multiple values). Instead of checking if value exists then transforming then checking again, you just chain: `optional.map(transform).flatMap(asyncOperation).orElse(default)`. The monad handles all the conditional logic. Common mistakes: using map when you need flatMap (creates nested monads), using flatMap when map suffices (unnecessary complexity), mixing monad types carelessly (Optional<Stream<Try<>>>), not understanding lazy evaluation (Stream operations don't run until terminal operation), overusing monads where simple code is clearer (if statement is sometimes better than Optional).


**The Philosophy:**

```
"A monad is just a monoid in the category of endofunctors."
— This is technically correct but useless for practitioners.

Better definition:
"A monad is a design pattern for chaining operations
 while the container handles the messy parts."

The messy parts:
  - Optional: Null checks
  - Stream: Iteration and collection
  - CompletableFuture: Threading and callbacks
  - Try: Exception handling
  - Either: Success/failure branching

Without monads:
  if (user != null) {
    Order order = user.getOrder();
    if (order != null) {
      Address address = order.getAddress();
      if (address != null) {
        return address.getCity();
      }
    }
  }
  return "Unknown";

With monads:
  return user
    .flatMap(User::getOrder)
    .flatMap(Order::getAddress)
    .map(Address::getCity)
    .orElse("Unknown");

The monad handles all null checks. You focus on transformations.
```

## 1. Optional Monad

### Handling Potential Absence

```java
// ========================================
// WITHOUT OPTIONAL: Null checks everywhere
// ========================================

public String getUserCity(String userId) {
    User user = userRepository.findById(userId);
    if (user == null) {
        return "Unknown";
    }
    
    Order order = user.getCurrentOrder();
    if (order == null) {
        return "Unknown";
    }
    
    Address address = order.getShippingAddress();
    if (address == null) {
        return "Unknown";
    }
    
    String city = address.getCity();
    if (city == null) {
        return "Unknown";
    }
    
    return city;
}

// Every step requires null check. Pyramid of doom. Error-prone.
```

```java
// ========================================
// WITH OPTIONAL: Chained transformations
// ========================================

public String getUserCity(String userId) {
    return userRepository.findById(userId)        // Optional<User>
        .flatMap(User::getCurrentOrder)           // Optional<Order>
        .flatMap(Order::getShippingAddress)       // Optional<Address>
        .map(Address::getCity)                    // Optional<String>
        .orElse("Unknown");                       // String
}

// Zero null checks. The Optional monad handles absence at every step.
```

```java
// ========================================
// OPTIONAL OPERATIONS
// ========================================

import java.util.Optional;

public class OptionalExamples {
    
    // ✅ map: Transform the value if present
    public Optional<String> getUserName(String userId) {
        return findUser(userId)              // Optional<User>
            .map(User::getName);             // Optional<String>
        // If user exists: returns Optional.of(name)
        // If user absent: returns Optional.empty()
    }
    
    // ✅ flatMap: Chain operations that return Optional
    public Optional<String> getUserEmail(String userId) {
        return findUser(userId)              // Optional<User>
            .flatMap(User::getEmail);        // Optional<String>
        // User::getEmail returns Optional<String>
        // flatMap unwraps one level, avoids Optional<Optional<String>>
    }
    
    // ✅ filter: Keep only if predicate matches
    public Optional<User> getActiveUser(String userId) {
        return findUser(userId)              // Optional<User>
            .filter(User::isActive);         // Optional<User>
        // If user exists AND is active: returns Optional.of(user)
        // Otherwise: returns Optional.empty()
    }
    
    // ✅ orElse: Provide default value
    public String getUserNameOrDefault(String userId) {
        return findUser(userId)
            .map(User::getName)
            .orElse("Anonymous");            // String
        // Returns name if present, "Anonymous" if absent
    }
    
    // ✅ orElseGet: Provide default via Supplier (lazy)
    public String getUserNameOrCompute(String userId) {
        return findUser(userId)
            .map(User::getName)
            .orElseGet(() -> generateDefaultName(userId));
        // Only calls generateDefaultName if Optional is empty
        // Better than orElse when default is expensive to compute
    }
    
    // ✅ orElseThrow: Throw exception if absent
    public User getUserOrFail(String userId) {
        return findUser(userId)
            .orElseThrow(() -> new UserNotFoundException(userId));
        // Returns user if present, throws exception if absent
    }
    
    // ✅ ifPresent: Execute action if present (side effect)
    public void logUserIfFound(String userId) {
        findUser(userId)
            .ifPresent(user -> log.info("Found user: {}", user.getName()));
        // Executes lambda only if user is present
    }
    
    // ✅ ifPresentOrElse: Execute one action if present, another if absent
    public void processUser(String userId) {
        findUser(userId)
            .ifPresentOrElse(
                user -> log.info("Processing: {}", user),
                () -> log.warn("User not found: {}", userId)
            );
    }
    
    private Optional<User> findUser(String userId) {
        // Simulated database lookup
        return Optional.empty();
    }
    
    private String generateDefaultName(String userId) {
        return "User_" + userId;
    }
}
```

```java
// ========================================
// CHAINING: Real-world example
// ========================================

public class OrderService {
    
    // ✅ Complex chain: Find user, get order, calculate total, apply discount
    public Optional<Money> calculateDiscountedTotal(String userId) {
        return userRepository.findById(userId)              // Optional<User>
            .flatMap(User::getCurrentOrder)                 // Optional<Order>
            .map(Order::getItems)                           // Optional<List<Item>>
            .map(this::calculateSubtotal)                   // Optional<Money>
            .flatMap(subtotal -> getDiscount(userId)        // Optional<Discount>
                .map(discount -> applyDiscount(subtotal, discount)))
            .filter(total -> total.isPositive());           // Optional<Money>
    }
    
    private Money calculateSubtotal(List<OrderItem> items) {
        return items.stream()
            .map(OrderItem::getPrice)
            .reduce(Money.ZERO, Money::add);
    }
    
    private Optional<Discount> getDiscount(String userId) {
        return discountRepository.findActiveDiscountForUser(userId);
    }
    
    private Money applyDiscount(Money subtotal, Discount discount) {
        return subtotal.multiply(1 - discount.getPercent() / 100.0);
    }
}

// Each step might fail (user not found, no order, no discount).
// Optional handles all the absence gracefully.
// No nested if statements. Clean pipeline.
```

```java
// ========================================
// ANTI-PATTERNS: When NOT to use Optional
// ========================================

// ❌ DON'T: Use Optional as field
public class User {
    private Optional<String> middleName;  // BAD!
    // Just use String and allow null
}

// ❌ DON'T: Use Optional as parameter
public void processUser(Optional<User> user) {  // BAD!
    // Caller must wrap in Optional. Unnecessary.
    // Just use User and check for null or overload
}

// ❌ DON'T: Call get() without checking
public String getUserName(String userId) {
    Optional<User> user = findUser(userId);
    return user.get().getName();  // BAD! Throws if empty
    // Use orElse, orElseThrow, or orElseGet instead
}

// ❌ DON'T: Use isPresent() + get()
if (optional.isPresent()) {
    String value = optional.get();  // BAD! Defeats the purpose
    // Use ifPresent(lambda) or map/flatMap instead
}

// ✅ DO: Use Optional for return types where absence is expected
public Optional<User> findUser(String userId) { /* ... */ }

// ✅ DO: Chain operations to handle absence
public String getNameOrDefault(String userId) {
    return findUser(userId)
        .map(User::getName)
        .orElse("Anonymous");
}
```

## 2. Stream Monad

### Handling Collections

```java
// ========================================
// STREAM AS MONAD: Multiple values in context
// ========================================

import java.util.stream.Stream;
import java.util.List;

public class StreamMonadExamples {
    
    // ✅ map: Transform each element
    public List<String> getUserNames(List<User> users) {
        return users.stream()              // Stream<User>
            .map(User::getName)            // Stream<String>
            .toList();                     // List<String>
    }
    
    // ✅ flatMap: Expand collections (unwrap nested streams)
    public List<OrderItem> getAllItems(List<Order> orders) {
        return orders.stream()             // Stream<Order>
            .flatMap(order -> order.getItems().stream())  // Stream<OrderItem>
            .toList();                     // List<OrderItem>
        // Without flatMap: Stream<Stream<OrderItem>> (nested!)
        // With flatMap: Stream<OrderItem> (flat)
    }
    
    // ✅ filter: Keep only matching elements
    public List<User> getActiveUsers(List<User> users) {
        return users.stream()              // Stream<User>
            .filter(User::isActive)        // Stream<User> (filtered)
            .toList();
    }
    
    // ✅ Chaining multiple operations
    public List<String> getActiveUserEmails(List<User> users) {
        return users.stream()
            .filter(User::isActive)                        // Keep active
            .map(User::getEmail)                           // Get email
            .flatMap(Optional::stream)                     // Unwrap Optional<String>
            .map(String::toLowerCase)                      // Normalize
            .distinct()                                    // Remove duplicates
            .sorted()                                      // Sort
            .toList();
    }
}
```

```java
// ========================================
// LAZY EVALUATION: Operations don't run until terminal operation
// ========================================

public class LazyEvaluation {
    
    public static void main(String[] args) {
        List<Integer> numbers = List.of(1, 2, 3, 4, 5);
        
        // ❌ This doesn't execute yet!
        Stream<Integer> stream = numbers.stream()
            .peek(n -> System.out.println("Processing: " + n))  // Side effect
            .map(n -> n * 2)
            .filter(n -> n > 5);
        
        // Nothing printed yet. Operations are lazy.
        
        // ✅ Terminal operation triggers execution
        List<Integer> result = stream.toList();
        // NOW the operations execute: Processing 1, 2, 3, 4, 5
        
        System.out.println(result);  // [6, 8, 10]
    }
}

// Benefits of laziness:
// - Short-circuits: findFirst stops after finding one
// - Efficiency: Only processes what's needed
// - Composability: Build complex pipelines cheaply
```

```java
public class OrderProcessor {
    
    public Map<String, Double> calculateRevenueByProduct(List<Order> orders) {
        return orders.stream()
            .flatMap(order -> order.getItems().stream())          // Stream<OrderItem>
            .collect(Collectors.groupingBy(
                item -> item.getProduct().getName(),              // Group by product name
                Collectors.summingDouble(item -> 
                    item.getPrice() * item.getQuantity())         // Sum revenue
            ));
    }
    
    public List<Order> findHighValueOrders(List<Order> orders, double threshold) {
        return orders.stream()
            .filter(order -> calculateTotal(order) > threshold)
            .sorted(Comparator.comparing(this::calculateTotal).reversed())
            .limit(10)                                            // Top 10
            .toList();
    }
    
    private double calculateTotal(Order order) {
        return order.getItems().stream()
            .mapToDouble(item -> item.getPrice() * item.getQuantity())
            .sum();
    }
}
```

```java
// NESTED STREAMS: When to use flatMap

public class FlatMapExample {
    
    // ❌ map creates nested structure
    public void wrongApproach(List<User> users) {
        Stream<Stream<Order>> nested = users.stream()
            .map(user -> user.getOrders().stream());  // Stream<Stream<Order>>
        // Can't easily work with this nested structure!
    }
    
    // ✅ flatMap flattens the structure
    public List<Order> correctApproach(List<User> users) {
        return users.stream()
            .flatMap(user -> user.getOrders().stream())  // Stream<Order>
            .toList();
    }
    
    // Rule: Use flatMap when the transformation returns a Stream/Optional/Collection
    //       Use map when the transformation returns a simple value
}
```

## 3. CompletableFuture Monad

### Handling Asynchrony

```java
// ========================================
// COMPLETABLEFUTURE: Async computations as monad
// ========================================

import java.util.concurrent.CompletableFuture;
import java.util.concurrent.ExecutorService;

public class AsyncMonadExamples {
    
    // ✅ thenApply: Transform result (like Optional.map)
    public CompletableFuture<String> getUserName(String userId) {
        return fetchUserAsync(userId)                    // CompletableFuture<User>
            .thenApply(User::getName);                   // CompletableFuture<String>
    }
    
    // ✅ thenCompose: Chain async operations (like Optional.flatMap)
    public CompletableFuture<Order> getUserOrder(String userId) {
        return fetchUserAsync(userId)                    // CompletableFuture<User>
            .thenCompose(user -> fetchOrderAsync(user.getOrderId()));
        // fetchOrderAsync returns CompletableFuture<Order>
        // thenCompose unwraps, avoids CompletableFuture<CompletableFuture<Order>>
    }
    
    // ✅ thenCombine: Combine two independent futures
    public CompletableFuture<OrderSummary> getOrderSummary(String userId, String orderId) {
        CompletableFuture<User> userFuture = fetchUserAsync(userId);
        CompletableFuture<Order> orderFuture = fetchOrderAsync(orderId);
        
        return userFuture.thenCombine(orderFuture, 
            (user, order) -> new OrderSummary(user, order));
        // Waits for both, then combines results
    }
    
    // ✅ exceptionally: Handle errors
    public CompletableFuture<User> getUserWithFallback(String userId) {
        return fetchUserAsync(userId)
            .exceptionally(ex -> {
                log.error("Failed to fetch user", ex);
                return User.anonymous();              // Fallback user
            });
    }
    
    // ✅ handle: Transform both success and failure
    public CompletableFuture<String> getUserNameSafe(String userId) {
        return fetchUserAsync(userId)
            .thenApply(User::getName)
            .handle((name, ex) -> {
                if (ex != null) {
                    return "Unknown";
                }
                return name;
            });
    }
    
    private CompletableFuture<User> fetchUserAsync(String userId) {
        return CompletableFuture.supplyAsync(() -> {
            // Simulate async database call
            return userRepository.findById(userId);
        });
    }
    
    private CompletableFuture<Order> fetchOrderAsync(String orderId) {
        return CompletableFuture.supplyAsync(() -> {
            return orderRepository.findById(orderId);
        });
    }
}
```

## 4. Try Monad (Vavr)

### Handling Exceptions Functionally

```java
// ========================================
// VAVR TRY: Exception handling as monad
// ========================================

import io.vavr.control.Try;

/**
 * Try<T> is either:
 *   - Success<T> containing the result
 *   - Failure<T> containing the exception
 * 
 * Operations on Success pass through.
 * Operations on Failure are skipped.
 */

public class TryMonadExamples {
    
    // ❌ Traditional try-catch
    public String traditionalApproach(String userId) {
        try {
            User user = userRepository.findById(userId);
            Order order = orderRepository.findById(user.getOrderId());
            return order.getTrackingNumber();
        } catch (UserNotFoundException e) {
            return "User not found";
        } catch (OrderNotFoundException e) {
            return "Order not found";
        } catch (Exception e) {
            return "Error: " + e.getMessage();
        }
    }
    
    // ✅ Try monad approach
    public String tryApproach(String userId) {
        return Try.of(() -> userRepository.findById(userId))     // Try<User>
            .mapTry(user -> orderRepository.findById(user.getOrderId()))  // Try<Order>
            .map(Order::getTrackingNumber)                        // Try<String>
            .getOrElse("Tracking unavailable");
    }
    
    // ✅ map: Transform success value
    public Try<String> getUserName(String userId) {
        return Try.of(() -> userRepository.findById(userId))     // Try<User>
            .map(User::getName);                                  // Try<String>
        // If Success: returns Try.success(name)
        // If Failure: returns Try.failure(exception)
    }
    
    // ✅ flatMap: Chain operations that return Try
    public Try<Order> getUserOrder(String userId) {
        return Try.of(() -> userRepository.findById(userId))     // Try<User>
            .flatMap(user -> 
                Try.of(() -> orderRepository.findById(user.getOrderId())));
        // Avoids Try<Try<Order>>
    }
    
    // ✅ recover: Provide fallback on failure
    public String getUserNameSafe(String userId) {
        return Try.of(() -> userRepository.findById(userId))
            .map(User::getName)
            .recover(UserNotFoundException.class, "Anonymous")   // Handle specific exception
            .recover(Exception.class, ex -> "Error: " + ex.getMessage())
            .get();
    }
    
    // ✅ orElse: Alternative Try on failure
    public Try<User> getUserWithFallback(String userId) {
        return Try.of(() -> userRepository.findById(userId))
            .orElse(Try.of(() -> userRepository.findByEmail(userId + "@example.com")));
        // If first fails, tries second approach
    }
    
    // ✅ toEither: Convert to Either for explicit error handling
    public Either<String, User> getUserAsEither(String userId) {
        return Try.of(() -> userRepository.findById(userId))
            .toEither()                                          // Either<Throwable, User>
            .mapLeft(Throwable::getMessage);                     // Either<String, User>
    }
}
```

```java
    public Try<Receipt> processPayment(String userId, String orderId) {
        return Try.of(() -> userRepository.findById(userId))

    }
```

## 5. Either Monad

### Railway-Oriented Programming

```java
// ========================================
// EITHER: Success (Right) or Failure (Left)
// ========================================

import io.vavr.control.Either;

/**
 * Either<L, R> holds one of two values:
 *   - Left<L>: Error/failure case
 *   - Right<R>: Success case
 * 
 * Convention: Right is correct, Left is error.
 * 
 * map/flatMap only operate on Right.
 * Left values are passed through unchanged.
 */

public class EitherMonadExamples {
    
    // ✅ Either for validation
    public Either<String, User> validateUser(UserRequest request) {
        if (request.getName() == null || request.getName().isBlank()) {
            return Either.left("Name is required");
        }
        if (request.getEmail() == null || !request.getEmail().contains("@")) {
            return Either.left("Valid email is required");
        }
        if (request.getAge() < 18) {
            return Either.left("Must be 18 or older");
        }
        return Either.right(new User(request.getName(), request.getEmail(), request.getAge()));
    }
    
    // ✅ map: Transform Right value
    public Either<String, String> getUserName(String userId) {
        return findUser(userId)                            // Either<String, User>
            .map(User::getName);                           // Either<String, String>
        // If Right: transforms the user to name
        // If Left: passes through unchanged
    }
    
    // ✅ flatMap: Chain operations that return Either
    public Either<String, Order> getUserOrder(String userId) {
        return findUser(userId)                            // Either<String, User>
            .flatMap(user -> findOrder(user.getOrderId()));  // Either<String, Order>
        // If findUser returns Left: short-circuits, never calls findOrder
        // If findUser returns Right: calls findOrder with the user
    }
    
    // ✅ fold: Handle both cases
    public String processUser(String userId) {
        return findUser(userId)
            .fold(
                error -> "Error: " + error,                // Left case
                user -> "Welcome, " + user.getName()       // Right case
            );
    }
    
    // ✅ getOrElse: Extract Right or provide default
    public User getUserOrDefault(String userId) {
        return findUser(userId)
            .getOrElse(User.anonymous());
    }
    
    private Either<String, User> findUser(String userId) {
        // Simulated lookup
        return Either.right(new User("Alice", "alice@example.com", 25));
    }
    
    private Either<String, Order> findOrder(String orderId) {
        return Either.right(new Order(orderId));
    }
}
```

```java
// ========================================
// RAILWAY-ORIENTED PROGRAMMING
// ========================================

/**
 * Think of Either as a railway with two tracks:
 *   - Success track (Right)
 *   - Failure track (Left)
 * 
 * Operations stay on the Right track until something fails,
 * then switch to Left track and stay there.
 */

public class ValidationPipeline {
    
    // ✅ Chain validations: first failure short-circuits
    public Either<String, User> createUser(UserRequest request) {
        return validateName(request.getName())
            .flatMap(name -> validateEmail(request.getEmail()))
            .flatMap(email -> validateAge(request.getAge()))
            .map(age -> new User(request.getName(), request.getEmail(), age));
        
        // If validateName fails: returns Left, never calls validateEmail
        // If validateEmail fails: returns Left, never calls validateAge
        // If all succeed: returns Right<User>
    }
    
    private Either<String, String> validateName(String name) {
        if (name == null || name.isBlank()) {
            return Either.left("Name is required");
        }
        if (name.length() < 2) {
            return Either.left("Name too short");
        }
        return Either.right(name);
    }
    
    private Either<String, String> validateEmail(String email) {
        if (email == null || !email.contains("@")) {
            return Either.left("Invalid email");
        }
        return Either.right(email);
    }
    
    private Either<String, Integer> validateAge(int age) {
        if (age < 18) {
            return Either.left("Must be 18 or older");
        }
        if (age > 120) {
            return Either.left("Invalid age");
        }
        return Either.right(age);
    }
}
```

## 6. Monad Laws

### Why They Matter Practically

```java
// ========================================
// THE THREE MONAD LAWS
// ========================================

/**
 * Monad laws ensure predictable, composable behavior.
 * They're not academic curiosities—they guarantee code works as expected.
 * 
 * 1. LEFT IDENTITY
 *    unit(a).flatMap(f) == f(a)
 *    Wrapping then transforming == just transforming
 * 
 * 2. RIGHT IDENTITY
 *    m.flatMap(unit) == m
 *    Transforming with identity does nothing
 * 
 * 3. ASSOCIATIVITY
 *    m.flatMap(f).flatMap(g) == m.flatMap(x -> f(x).flatMap(g))
 *    Order of nesting doesn't matter
 */

public class MonadLaws {
    
    // ========================================
    // LAW 1: LEFT IDENTITY
    // ========================================
    
    public void leftIdentityLaw() {
        String value = "hello";
        Function<String, Optional<String>> f = s -> Optional.of(s.toUpperCase());
        
        // Left side: wrap then flatMap
        Optional<String> left = Optional.of(value).flatMap(f);
        
        // Right side: just apply function
        Optional<String> right = f.apply(value);
        
        assert left.equals(right);  // Must be equal
        
        // Why it matters: You can refactor Optional.of(x).flatMap(f) to f(x)
        // without changing behavior. The wrapping doesn't add overhead.
    }
    
    // ========================================
    // LAW 2: RIGHT IDENTITY
    // ========================================
    
    public void rightIdentityLaw() {
        Optional<String> m = Optional.of("hello");
        
        // Left side: flatMap with identity (wrap in Optional)
        Optional<String> left = m.flatMap(Optional::of);
        
        // Right side: original monad
        Optional<String> right = m;
        
        assert left.equals(right);  // Must be equal
        
        // Why it matters: flatMap with Optional::of does nothing.
        // You can remove such operations without changing behavior.
    }
    
    // ========================================
    // LAW 3: ASSOCIATIVITY
    // ========================================
    
    public void associativityLaw() {
        Optional<String> m = Optional.of("hello");
        Function<String, Optional<String>> f = s -> Optional.of(s.toUpperCase());
        Function<String, Optional<String>> g = s -> Optional.of(s + "!");
        
        // Left side: nested flatMaps (left-to-right)
        Optional<String> left = m.flatMap(f).flatMap(g);
        
        // Right side: nested flatMaps (inside-out)
        Optional<String> right = m.flatMap(x -> f.apply(x).flatMap(g));
        
        assert left.equals(right);  // Must be equal
        
        // Why it matters: You can refactor nested flatMaps freely.
        // (a.flatMap(f)).flatMap(g) == a.flatMap(x -> f(x).flatMap(g))
    }
}
```

```java
// ========================================
// PRACTICAL IMPACT OF MONAD LAWS
// ========================================

public class MonadLawsInPractice {
    
    // ✅ Refactoring enabled by Left Identity
    public Optional<String> getUppercaseName(User user) {
        // Original:
        return Optional.of(user.getName()).flatMap(name -> Optional.of(name.toUpperCase()));
        
        // Simplified using left identity:
        // Optional.of(user.getName()).flatMap(f) == f(user.getName())
        return Optional.of(user.getName().toUpperCase());
    }
    
    // ✅ Refactoring enabled by Right Identity
    public Optional<User> fetchUser(String userId) {
        Optional<User> user = userRepository.findById(userId);
        
        // Original (pointless flatMap):
        return user.flatMap(Optional::of);
        
        // Simplified using right identity:
        // m.flatMap(Optional::of) == m
        return user;
    }
    
    // ✅ Refactoring enabled by Associativity
    public Optional<String> getOrderCity(String userId) {
        // Original (left-to-right nesting):
        return findUser(userId)
            .flatMap(User::getCurrentOrder)
            .flatMap(Order::getShippingAddress)
            .map(Address::getCity);
        
        // Equivalent (inside-out nesting) - less readable but equivalent:
        return findUser(userId)
            .flatMap(user -> user.getCurrentOrder()
                .flatMap(order -> order.getShippingAddress()
                    .map(Address::getCity)));
        
        // Both are valid because of associativity.
        // Choose the more readable left-to-right version.
    }
    
    private Optional<User> findUser(String userId) { return Optional.empty(); }
}

/**
 * Monad laws guarantee:
 *   - Predictable behavior when refactoring
 *   - Safe composition of operations
 *   - No hidden surprises
 * 
 * If a type doesn't obey monad laws, things break in subtle ways.
 * Example: A "monad" where flatMap executes operations twice
 * would violate associativity and cause bugs.
 */
```

```java
// ========================================
// DECISION FRAMEWORK: Which monad to use?
// ========================================

public class MonadDecisionGuide {
    
    // ✅ Use Optional when:
    //   - Value might be absent
    //   - Null is a valid state
    //   - Need to chain operations that might return null
    public Optional<String> getUserEmail(String userId) {
        return userRepository.findById(userId)
            .flatMap(User::getEmail);
    }
    
    // ✅ Use Stream when:
    //   - Working with collections
    //   - Need to transform/filter/aggregate multiple values
    //   - Want lazy evaluation
    public List<String> getActiveUserEmails() {
        return userRepository.findAll().stream()
            .filter(User::isActive)
            .flatMap(user -> user.getEmail().stream())
            .toList();
    }
    
    // ✅ Use CompletableFuture when:
    //   - Operation is asynchronous
    //   - Need to avoid blocking threads
    //   - Want to compose async operations
    public CompletableFuture<Order> processOrderAsync(OrderRequest request) {
        return validateAsync(request)
            .thenCompose(this::createOrderAsync)
            .thenCompose(this::chargePaymentAsync);
    }
    
    // ✅ Use Try when:
    //   - Operation might throw exception
    //   - Want functional exception handling
    //   - Don't want try-catch blocks everywhere
    public Try<Receipt> processPayment(Payment payment) {
        return Try.of(() -> paymentGateway.charge(payment))
            .map(this::generateReceipt)
            .recover(PaymentException.class, ex -> Receipt.declined());
    }
    
    // ✅ Use Either when:
    //   - Need explicit success/failure distinction
    //   - Want typed error messages
    //   - Implementing railway-oriented programming
    public Either<String, User> createUser(UserRequest request) {
        return validateName(request.getName())
            .flatMap(name -> validateEmail(request.getEmail()))
            .map(email -> new User(name, email));
    }

    private Receipt generateReceipt(PaymentResult result) { return null; }
```

## Summary

```
MONAD                    CONTEXT                           USE WHEN
────────────────────────────────────────────────────────────────────────────
Optional                 Value might be absent.            Avoiding null checks.
                         map/flatMap/orElse                Return types where absence
                                                          is a valid state.

Stream                   Multiple values (collection).     Processing collections.
                         map/flatMap/filter                Transforming/filtering data.
                         Lazy evaluation                   Building data pipelines.

CompletableFuture        Asynchronous computation.         Non-blocking operations.
                         thenApply/thenCompose             Composing async workflows.
                         Handles threading                 Parallel execution.

Try (Vavr)              Operation might throw.            Functional exception handling.
                         Success or Failure                Avoiding try-catch blocks.
                         map/flatMap/recover               Chaining fallible operations.

Either (Vavr)           Success (Right) or Error (Left).  Explicit error handling.
                         map/flatMap only on Right         Railway-oriented programming.
                         fold handles both                 Typed error messages.

────────────────────────────────────────────────────────────────────────────

THE PATTERN:

All monads follow the same structure:
  1. Wrap value in context (Optional.of, Stream.of, Try.of)
  2. Transform value: map (value -> value)
  3. Chain operations: flatMap (value -> Monad)
  4. Unwrap: get, orElse, toList, join

The monad handles the "messy part":
  - Optional: null checks
  - Stream: iteration
  - CompletableFuture: threading
  - Try: exception handling
  - Either: success/failure branching

WITHOUT MONADS:
  if (user != null) {
    Order order = user.getOrder();
    if (order != null) {
      return order.getTotal();
    }
  }
  return 0.0;

WITH MONADS:
  return user
    .flatMap(User::getOrder)
    .map(Order::getTotal)
    .orElse(0.0);

The insight: Monads let you focus on transformations.
The container handles the control flow.
```

**Key Takeaway:**

```
A monad is a design pattern for composition.

The pattern:
  1. Wrap value in context
  2. Transform: map (value -> value)
  3. Chain: flatMap (value -> Monad)
  4. Unwrap: get, orElse, etc.

The benefit: Focus on transformations.
The monad handles the messy parts.

Without Optional:
  String city = null;
  if (user != null) {
    Order order = user.getOrder();
    if (order != null) {
      Address addr = order.getAddress();
      if (addr != null) {
        city = addr.getCity();
      }
    }
  }
  return city != null ? city : "Unknown";

With Optional:
  return user
    .flatMap(User::getOrder)
    .flatMap(Order::getAddress)
    .map(Address::getCity)
    .orElse("Unknown");

Zero null checks. Zero nested ifs.
The Optional monad handles all of it.

```
