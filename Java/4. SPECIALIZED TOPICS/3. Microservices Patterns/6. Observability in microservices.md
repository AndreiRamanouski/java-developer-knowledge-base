# Observability in Microservices

## Overview

Observability is the ability to understand the internal state of a system from its external outputs. In a monolith, a single stack trace tells you everything. In microservices with 20+ services, a single user request might touch 8 services, span 3 databases, and produce logs scattered across 40 pod instances — making failures nearly invisible without dedicated tooling.

**The three pillars of observability:**

```
Logs     → What happened (discrete events with context)
Metrics  → How the system is behaving (aggregated numbers over time)
Traces   → Where time was spent (end-to-end request flow)

Logs answer: "What error occurred in order-service at 14:32:05?"
Metrics answer: "Is order-service's error rate rising?"
Traces answer: "Why did this specific request take 3 seconds?"
```

## 1. Distributed Tracing (Zipkin, Jaeger)

A distributed trace follows a single request as it travels across multiple services. Each service adds a **span** — a timed unit of work — and all spans for one request share a **trace ID**.

### 1.1 Core concepts

```
Trace: the complete journey of one request through the system
  └── Span: a single unit of work within one service
        ├── Span ID: unique ID for this span
        ├── Trace ID: shared across all spans in the request
        ├── Parent Span ID: which span triggered this one
        ├── Start time + Duration
        └── Tags/Annotations: metadata (HTTP status, DB query, etc.)

Example trace for "place order":
  TraceID: abc123
  │
  ├── Span: API Gateway (10ms)
  │     └── calls order-service
  ├── Span: Order Service (250ms)
  │     ├── calls inventory-service (parallel)
  │     ├── calls payment-service (parallel)
  │     └── writes to DB (5ms)
  ├── Span: Inventory Service (40ms)
  │     └── reads from DB (8ms)
  └── Span: Payment Service (200ms)
        └── calls external Stripe API (180ms)
```

### 1.2 Micrometer Tracing — Spring Boot 3 standard

Spring Boot 3 replaced Spring Cloud Sleuth with **Micrometer Tracing** — a vendor-neutral tracing facade that works with both Zipkin and Jaeger.

```xml
<!-- Micrometer Tracing + Brave bridge (for Zipkin) -->
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-tracing-bridge-brave</artifactId>
</dependency>
<dependency>
    <groupId>io.zipkin.reporter2</groupId>
    <artifactId>zipkin-reporter-brave</artifactId>
</dependency>

<!-- OR: Micrometer Tracing + OpenTelemetry bridge (for Jaeger, OTEL Collector) -->
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-tracing-bridge-otel</artifactId>
</dependency>
<dependency>
    <groupId>io.opentelemetry</groupId>
    <artifactId>opentelemetry-exporter-otlp</artifactId>
</dependency>
```

```yaml
# application.yml — tracing configuration
management:
  tracing:
    sampling:
      probability: 1.0    # 1.0 = 100% of requests traced (dev)
                          # 0.1 = 10% in production (adjust for volume)
  zipkin:
    tracing:
      endpoint: http://zipkin:9411/api/v2/spans

# For OpenTelemetry / Jaeger:
otel:
  exporter:
    otlp:
      endpoint: http://jaeger-collector:4318
  traces:
    exporter: otlp
```

### 1.3 Automatic instrumentation

Spring Boot auto-configures tracing for:

```
✓ Spring MVC / WebFlux HTTP requests — inbound and outbound
✓ WebClient / RestTemplate — propagates trace context in headers
✓ Spring Data (JDBC, R2DBC) — database span per query
✓ Spring Kafka — producer/consumer spans
✓ Scheduled tasks (@Scheduled)
✓ @Async methods
```

```java
// Nothing needed for most HTTP calls — auto-instrumented
@RestController
@RequiredArgsConstructor
public class OrderController {

    private final OrderService orderService;
    private final WebClient inventoryClient;

    @GetMapping("/api/orders/{id}")
    public Mono<OrderDto> getOrder(@PathVariable Long id) {
        // Micrometer Tracing automatically:
        // 1. Creates a span for this request
        // 2. Injects trace/span IDs into MDC for logging
        // 3. Propagates trace context in outbound WebClient headers
        return orderService.findById(id);
    }
}
```

### 1.4 Custom spans for business operations

```java
@Service
@RequiredArgsConstructor
@Slf4j
public class OrderService {

    private final Tracer tracer;
    private final OrderRepository orderRepository;

    public Order processOrder(CreateOrderRequest request) {
        // Create a custom span for a business-significant operation
        Span span = tracer.nextSpan()
            .name("order.process")
            .tag("order.userId", request.getUserId().toString())
            .tag("order.itemCount", String.valueOf(request.getItems().size()))
            .start();

        try (Tracer.SpanInScope scope = tracer.withSpan(span)) {
            // All work inside this scope is attributed to this span

            Order order = validateAndSave(request);

            span.tag("order.id", order.getId().toString());
            span.tag("order.total", order.getTotalAmount().toString());
            span.event("order.saved"); // point-in-time event within span

            return order;

        } catch (Exception ex) {
            span.tag("error", ex.getMessage());
            span.error(ex);
            throw ex;
        } finally {
            span.end();
        }
    }
}

// @NewSpan — annotation-based span creation (simpler)
@Service
public class InventoryService {

    @NewSpan("inventory.reserve-stock")
    public void reserveStock(
            @SpanTag("order.id") Long orderId,
            List<OrderItem> items) {
        // This method runs in its own span
        // orderId is added as a tag automatically
        doReserve(orderId, items);
    }
}
```


```
  - Click span to see tags, logs, errors
```

### 1.6 Jaeger

Jaeger is CNCF's distributed tracing platform, designed for cloud-native environments. It natively supports OpenTelemetry (OTEL) and is the standard in Kubernetes deployments.

```yaml
# Jaeger all-in-one (dev/testing)
services:
  jaeger:
    image: jaegertracing/all-in-one:latest
    ports:
      - "16686:16686"   # Jaeger UI
      - "4317:4317"     # OTLP gRPC
      - "4318:4318"     # OTLP HTTP
    environment:
      COLLECTOR_OTLP_ENABLED: "true"
      SPAN_STORAGE_TYPE: memory    # or elasticsearch, cassandra

# Production: separate components for scale
# jaeger-collector (receives spans) + jaeger-query (UI) + Elasticsearch (storage)
```

### 1.7 Zipkin vs Jaeger

||Zipkin|Jaeger|
|---|---|---|
|Protocol|Zipkin v2 (HTTP/JSON), Kafka|OpenTelemetry (OTLP), Zipkin|
|Storage|Elasticsearch, Cassandra, MySQL|Elasticsearch, Cassandra, Kafka|
|UI|Simple, focused|Feature-rich, adaptive sampling|
|Sampling|Client-side|Client-side + adaptive (auto-tunes sample rate)|
|Architecture|Simple|Microservice components (collector, query, agent)|
|CNCF project|No|Yes|
|Spring Boot 3|Brave bridge|OTEL bridge|
|Best for|Simple setups, Spring Boot native|K8s environments, large scale|

## 2. Centralized Logging (ELK, Splunk)

In microservices, logs are produced by dozens of services on hundreds of pods. Centralized logging aggregates all logs into one searchable store, correlated by trace ID.

### 2.1 Structured Logging — prerequisite

Before centralizing logs, make them machine-parseable:

```xml
<!-- Logback JSON encoder — outputs logs as JSON -->
<dependency>
    <groupId>net.logstash.logback</groupId>
    <artifactId>logstash-logback-encoder</artifactId>
    <version>7.4</version>
</dependency>
```

```xml
<!-- logback-spring.xml -->
<configuration>
    <springProperty name="APPLICATION_NAME" source="spring.application.name"/>

    <appender name="JSON_CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder class="net.logstash.logback.encoder.LogstashEncoder">
            <providers>
                <timestamp/>
                <logLevel/>
                <loggerName/>
                <message/>
                <mdc/>           <!-- includes traceId, spanId from Micrometer -->
                <arguments/>
                <stackTrace/>
            </providers>
            <customFields>{"service":"${APPLICATION_NAME}","env":"${SPRING_PROFILES_ACTIVE:-dev}"}</customFields>
        </encoder>
    </appender>

    <root level="INFO">
        <appender-ref ref="JSON_CONSOLE"/>
    </root>
</configuration>
```

**Resulting log output (JSON):**

```json
{
  "@timestamp": "2024-01-15T14:32:05.123Z",
  "level": "INFO",
  "logger": "com.example.OrderService",
  "message": "Order placed successfully",
  "service": "order-service",
  "env": "production",
  "traceId": "abc123def456",
  "spanId": "789xyz",
  "orderId": "1234",
  "userId": "5678",
  "amount": "99.99"
}
```

### 2.2 The ELK Stack

```
Filebeat (ships logs)
    │ tails container stdout
    ▼
Logstash (parses, transforms, enriches)
    │ applies grok patterns, adds metadata
    ▼
Elasticsearch (indexes and stores)
    │ full-text search, aggregations
    ▼
Kibana (UI — search, dashboards, alerts)
```

**Filebeat configuration:**

```yaml
# filebeat.yml — runs as DaemonSet in Kubernetes
filebeat.inputs:
  - type: container
    paths:
      - /var/log/containers/*.log      # tails all container logs
    processors:
      - add_kubernetes_metadata:       # enriches with K8s pod/namespace metadata
          host: ${NODE_NAME}
          matchers:
            - logs_path:
                logs_path: "/var/log/containers/"

processors:
  - decode_json_fields:
      fields: ["message"]             # parse JSON log lines
      target: ""
      overwrite_keys: true

output.logstash:
  hosts: ["logstash:5044"]

# OR: send directly to Elasticsearch
output.elasticsearch:
  hosts: ["elasticsearch:9200"]
  index: "logs-%{[service]}-%{+yyyy.MM.dd}"
```

**Logstash pipeline:**

```ruby
# logstash.conf
input {
  beats { port => 5044 }
}

filter {
  # Parse structured JSON logs
  if [message] =~ /^\{/ {
    json { source => "message" }
  }

  # Add geo-IP enrichment for user IP addresses
  if [client_ip] {
    geoip { source => "client_ip" }
  }

  # Drop noisy health check logs
  if [request_uri] == "/actuator/health" {
    drop {}
  }

  # Parse duration strings
  if [duration] {
    mutate { convert => { "duration" => "integer" } }
  }
}

output {
  elasticsearch {
    hosts => ["elasticsearch:9200"]
    index => "logs-%{[service]}-%{+YYYY.MM.dd}"
  }
}
```

**Kibana — key use cases:**

```
Discover:
  - Full-text search across all logs
  - Filter by traceId, service, level, userId
  - Query: traceId:"abc123" AND level:"ERROR"

Visualize:
  - Error rate by service over time (line chart)
  - Top 10 slowest endpoints (bar chart)
  - Error distribution by service (pie chart)

Dashboard:
  - Combine multiple visualizations
  - Real-time production health overview

Alerts:
  - "If error rate > 5% in last 5 minutes → page on-call"
  - "If any FATAL log → immediate alert"
```

### 2.4 Splunk

Splunk is the enterprise alternative to ELK — proprietary, expensive, but operationally simpler and with powerful query language (SPL).

```yaml
# Splunk Universal Forwarder (ships logs to Splunk)
# OR: Splunk HTTP Event Collector (HEC) — send directly from app

# Logback appender for Splunk HEC
# application.yml
splunk:
  hec:
    url: https://splunk:8088
    token: ${SPLUNK_HEC_TOKEN}
    index: application-logs
    source: order-service
```

```
Splunk Query Language (SPL):
  # Find all errors in order-service last 1 hour
  index=application-logs service=order-service level=ERROR earliest=-1h

  # Count errors by service
  index=application-logs level=ERROR | stats count by service

  # P99 response time per endpoint
  index=application-logs | stats perc99(duration) by endpoint

  # Join on traceId — trace a request across services
  index=application-logs traceId="abc123" | table _time service message
  | sort _time
```

### 2.5 ELK vs Splunk

||ELK Stack|Splunk|
|---|---|---|
|Cost|Open source (infra cost only)|Licensed (expensive at scale)|
|Setup complexity|High|Low (managed available)|

|ML / anomaly detection|X-Pack (paid)|Built-in|

## 3. Metrics Aggregation (Prometheus)

Metrics are aggregated numerical measurements over time: request rates, error rates, latency percentiles, JVM heap, connection pool usage. Prometheus is the de-facto standard in cloud-native environments.

### 3.1 Spring Boot Actuator + Micrometer

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-registry-prometheus</artifactId>
</dependency>
```

```yaml
# application.yml
management:
  endpoints:
    web:
      exposure:
        include: health, info, metrics, prometheus
  metrics:
    tags:
      application: ${spring.application.name}   # tag all metrics with service name
      environment: ${spring.profiles.active}
    distribution:
      percentiles-histogram:
        http.server.requests: true              # enable histogram for HTTP latency
      percentiles:
        http.server.requests: 0.5, 0.90, 0.95, 0.99
      sla:
        http.server.requests: 100ms, 500ms, 1s  # SLA buckets
```

**Auto-provided metrics by Micrometer:**

```
JVM:
  jvm.memory.used, jvm.memory.max
  jvm.gc.pause (GC pause durations)
  jvm.threads.live, jvm.threads.daemon

HTTP:
  http.server.requests (count, sum, max, histogram by uri, method, status)
  http.client.requests (WebClient outbound calls)

HikariCP (connection pool):
  hikaricp.connections.active
  hikaricp.connections.pending
  hikaricp.connections.timeout.total

Kafka:
  kafka.producer.record.send.rate
  kafka.consumer.records.lag (consumer lag per partition)

JVM Process:
  process.cpu.usage
  system.cpu.usage
  process.uptime
```

### 3.2 Custom metrics

```java
@Service
@RequiredArgsConstructor
public class OrderMetrics {

    private final MeterRegistry registry;

    // Counter — monotonically increasing
    private final Counter ordersPlaced = Counter.builder("orders.placed")
        .description("Total orders placed")
        .tag("service", "order")
        .register(registry);

    private final Counter ordersFailedCounter;

    @PostConstruct
    public void init() {
        // Counter with tags set at record time
        ordersFailedCounter = registry.counter("orders.failed");
    }

    // Gauge — current point-in-time value
    private final AtomicInteger activeOrders = registry.gauge(
        "orders.active",
        new AtomicInteger(0)
    );

    // Timer — measures duration
    private final Timer orderProcessingTimer = Timer.builder("orders.processing.time")
        .description("Time to process an order")
        .publishPercentiles(0.5, 0.95, 0.99)
        .publishPercentileHistogram()
        .sla(Duration.ofMillis(100), Duration.ofMillis(500), Duration.ofSeconds(1))
        .register(registry);

    // Distribution Summary — measures value distribution (e.g., order size)
    private final DistributionSummary orderSizeSummary =
        DistributionSummary.builder("orders.item.count")
            .description("Number of items per order")
            .publishPercentiles(0.5, 0.95, 0.99)
            .register(registry);

    // Usage
    public Order processOrder(CreateOrderRequest request) {
        return orderProcessingTimer.record(() -> {
            ordersPlaced.increment();
            activeOrders.incrementAndGet();

            try {
                Order order = doProcess(request);
                orderSizeSummary.record(order.getItems().size());

                registry.counter("orders.placed", "status", "success").increment();
                return order;

            } catch (Exception ex) {
                registry.counter("orders.placed",
                    "status", "failure",
                    "error", ex.getClass().getSimpleName()
                ).increment();
                throw ex;

            } finally {
                activeOrders.decrementAndGet();
            }
        });
    }
}
```

**PromQL — key queries:**

```promql
# Request rate per service (5-minute window)
rate(http_server_requests_seconds_count{application="order-service"}[5m])

# Error rate (percentage of 5xx responses)
sum(rate(http_server_requests_seconds_count{status=~"5.."}[5m]))
  / sum(rate(http_server_requests_seconds_count[5m])) * 100

  sum(rate(http_server_requests_seconds_bucket[5m]))
  by (le, application, uri)
)

# Kafka consumer lag
sum(kafka_consumer_records_lag) by (group, topic)

# JVM heap usage percentage
jvm_memory_used_bytes{area="heap"}
  / jvm_memory_max_bytes{area="heap"} * 100

```

### 3.5 Alerting with Prometheus Alertmanager

```yaml
# prometheus-rules.yml — alerting rules
groups:
  - name: microservice-alerts
    rules:
      - alert: HighErrorRate
        expr: |
          sum(rate(http_server_requests_seconds_count{status=~"5.."}[5m]))
            by (application)
          / sum(rate(http_server_requests_seconds_count[5m]))
            by (application) > 0.05
        for: 2m   # must be true for 2 minutes before firing
        labels:
          severity: critical
        annotations:
          summary: "High error rate on {{ $labels.application }}"
          description: "{{ $labels.application }} error rate is {{ $value | humanizePercentage }}"

      - alert: HighLatencyP99
        expr: |
          histogram_quantile(0.99,
            sum(rate(http_server_requests_seconds_bucket[5m]))
            by (le, application)
          ) > 2
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "High P99 latency on {{ $labels.application }}"

      - alert: KafkaConsumerLagHigh
        expr: sum(kafka_consumer_records_lag) by (group) > 10000
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "Kafka consumer lag high for {{ $labels.group }}"

      - alert: ServiceDown
        expr: up{job=~".*-service"} == 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "{{ $labels.job }} is down"
```

## 4. Service Mesh Observability

A service mesh like Istio or Linkerd provides **zero-configuration observability** — metrics, traces, and traffic visualization for all service-to-service communication without any code changes.

### 4.1 What the mesh provides automatically

```
For every service-to-service call, the sidecar proxy captures:

Metrics (Prometheus-compatible):
  istio_requests_total          — request count by source, dest, status code
  istio_request_duration_ms     — latency histogram
  istio_request_bytes           — request payload sizes
  istio_response_bytes          — response payload sizes
  istio_tcp_connections_opened  — TCP connection count

Traces (exported to Zipkin/Jaeger):
  Span per inbound request
  Span per outbound request
  Propagated trace context (W3C TraceContext headers)

Access logs:
  Every request logged by Envoy with full details
```

### 4.2 Istio metrics in Grafana

```yaml
# Istio ships pre-built Grafana dashboards:
#   Istio Service Dashboard  — per-service request rate, error rate, latency
#   Istio Workload Dashboard — per-pod metrics
#   Istio Mesh Dashboard     — cluster-wide overview

# PromQL for Istio metrics:

# Inbound success rate per service
sum(rate(istio_requests_total{
  destination_service_name="order-service",
  response_code!~"5.*"
}[5m]))
/ sum(rate(istio_requests_total{
  destination_service_name="order-service"
}[5m]))

# P99 latency from Istio (no app code needed)
histogram_quantile(0.99,
  sum(rate(istio_request_duration_milliseconds_bucket{
    destination_service_name="order-service"
  }[5m])) by (le)
)
```

### 4.5 Combining app-level and mesh-level observability

```
App-level (Micrometer + Prometheus):
  ✓ Business metrics (orders placed, payment success rate)
  ✓ Internal JVM metrics (GC, heap, threads)
  ✓ Internal component metrics (queue depth, cache hit rate)
  ✓ Custom spans for business logic

Mesh-level (Istio/Linkerd sidecar):
  ✓ Service-to-service latency and error rate
  ✓ Network-level metrics (retried requests, connection errors)
  ✓ mTLS verification status
  ✓ Cross-service traces without code changes

Together:
  App metrics answer: "Is the payment processing logic slow?"
  Mesh metrics answer: "Is payment-service slow to respond to order-service?"
  The difference tells you: is it a network issue or a code issue?
```

## 5. Correlation IDs

A correlation ID is a unique identifier attached to every request that flows through the entire system, allowing you to find all logs, spans, and events for a single user action across all services.

### 5.1 The problem without correlation IDs

```
User reports: "My order checkout was slow at 2:30 PM"

Without correlation ID:
  - Logs from 20 services, 40 pods, all timestamped at 2:30 PM
  - No way to find which log lines belong to this specific request
  - Grep for userId still returns thousands of logs from that minute

With correlation ID:
  - Filter ALL logs: traceId="abc123def456"
  - Instantly see the complete request journey across all services
```

### 5.2 Micrometer Tracing auto-propagates trace IDs

With Spring Boot 3 + Micrometer Tracing, the `traceId` and `spanId` are **automatically injected** into the MDC and propagated via HTTP headers:

```
Outbound HTTP header (W3C Trace Context):
  traceparent: 00-abc123def456-789xyz123-01
               │  └─trace ID─┘ └span ID┘ └flags
               └─version

Legacy B3 headers (Zipkin):
  X-B3-TraceId: abc123def456
  X-B3-SpanId: 789xyz123
  X-B3-Sampled: 1
```

```java
// No code needed — Micrometer Tracing auto-injects traceId into MDC

// logback-spring.xml — include in log output
<pattern>%d{ISO8601} [%X{traceId},%X{spanId}] %-5level %logger{36} - %msg%n</pattern>

// JSON output (with logstash encoder) includes traceId automatically
// {"timestamp":"...","traceId":"abc123","spanId":"789xyz","message":"..."}
```

### 5.3 Custom correlation ID alongside trace ID

Sometimes you need a business-level correlation ID (e.g., a user session ID or order ID) in addition to the technical trace ID:

```java
// Filter — extract or generate correlation ID from incoming requests
@Component
@Order(Ordered.HIGHEST_PRECEDENCE)
public class CorrelationIdFilter implements WebFilter {

    private static final String CORRELATION_ID_HEADER = "X-Correlation-ID";
    private static final String CORRELATION_ID_MDC_KEY = "correlationId";

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, WebFilterChain chain) {
        String correlationId = Optional
            .ofNullable(exchange.getRequest().getHeaders().getFirst(CORRELATION_ID_HEADER))
            .orElse(UUID.randomUUID().toString()); // generate if not provided

        // Add to response headers so client can correlate
        exchange.getResponse().getHeaders().add(CORRELATION_ID_HEADER, correlationId);

        // Add to Reactor Context for downstream access
        return chain.filter(exchange)
            .contextWrite(Context.of(CORRELATION_ID_MDC_KEY, correlationId))
            .doOnEach(signal -> {
                if (!signal.isOnComplete() && !signal.isOnError()) {
                    ContextView ctx = signal.getContextView();
                    ctx.getOrEmpty(CORRELATION_ID_MDC_KEY)
                        .ifPresent(id -> MDC.put(CORRELATION_ID_MDC_KEY, (String) id));
                }
            });
    }
}

// Propagate correlation ID in outbound WebClient calls
@Component
public class CorrelationIdWebClientFilter implements ExchangeFilterFunction {

    @Override
    public Mono<ClientResponse> filter(ClientRequest request, ExchangeFunction next) {
        return Mono.deferContextual(ctx -> {
            String correlationId = ctx.getOrDefault("correlationId",
                UUID.randomUUID().toString());

            ClientRequest enriched = ClientRequest.from(request)
                .header("X-Correlation-ID", correlationId)
                .build();

            return next.exchange(enriched);
        });
    }
}
```

### 5.4 Baggage — propagating context through the trace

Micrometer Tracing supports **baggage** — key-value pairs that are automatically propagated across all services in a trace, without explicit code in each service.

```java
@Configuration
public class BaggageConfig {

    @Bean
    public BaggageManager baggageManager(Tracer tracer) {
        return new MicrometerBaggageManager(tracer);
    }
}

@Component
@RequiredArgsConstructor
public class RequestEnrichmentFilter implements WebFilter {

    private final BaggagePropagation baggagePropagation;
    private final Tracer tracer;

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, WebFilterChain chain) {
        // Set baggage on the current span — auto-propagated to all downstream calls
        Span currentSpan = tracer.currentSpan();
        if (currentSpan != null) {
            currentSpan.tag("userId",
                exchange.getRequest().getHeaders().getFirst("X-User-Id"));
        }

        // Baggage fields — propagated in B3/W3C headers automatically
        BaggageField userId = BaggageField.create("userId");
        userId.updateValue(
            exchange.getRequest().getHeaders().getFirst("X-User-Id")
        );

        return chain.filter(exchange);
    }
}
```

### 5.5 End-to-end: tracing a request through all services

```
1. API Gateway receives request, generates traceId (or continues existing)
   Header: traceparent: 00-abc123-aaa-01

2. Order Service receives call
   MDC: { traceId: "abc123", spanId: "bbb", userId: "user-42" }
   Log: {"traceId":"abc123","message":"Processing order","userId":"user-42"}

3. Order Service calls Inventory Service
   Header injected: traceparent: 00-abc123-ccc-01 (same traceId, new spanId)
   Inventory MDC: { traceId: "abc123", spanId: "ccc" }

4. Order Service calls Payment Service
   Header injected: traceparent: 00-abc123-ddd-01
   Payment MDC: { traceId: "abc123", spanId: "ddd" }

5. All logs indexed in Kibana:
   Query: traceId:"abc123"
   Returns: logs from gateway + order + inventory + payment — all correlated

6. Trace visualized in Zipkin/Jaeger:
   Gantt chart shows: gateway(10ms) → order(250ms) = inventory(40ms) + payment(200ms)
   → Payment is the bottleneck
```

### 5.6 Correlation ID in Kafka messages

```java
// Producer — embed correlation metadata in Kafka message headers
@Component
@RequiredArgsConstructor
public class EventPublisher {

    private final KafkaTemplate<String, String> kafka;
    private final Tracer tracer;

    public void publish(String topic, String key, Object event) {
        ProducerRecord<String, String> record =
            new ProducerRecord<>(topic, key, serialize(event));

        // Inject trace context into Kafka headers
        Span currentSpan = tracer.currentSpan();
        if (currentSpan != null) {
            record.headers()
                .add("X-Trace-Id", currentSpan.context().traceId().getBytes())
                .add("X-Span-Id", currentSpan.context().spanId().getBytes());
        }

        // Add business correlation ID
        Optional.ofNullable(MDC.get("correlationId"))
            .ifPresent(id ->
                record.headers().add("X-Correlation-ID", id.getBytes()));

        kafka.send(record);
    }
}

// Consumer — extract and restore correlation context
@KafkaListener(topics = "order.placed", groupId = "inventory-service")
public void onOrderPlaced(
        ConsumerRecord<String, String> record,
        @Header(value = "X-Trace-Id", required = false) String traceId,
        @Header(value = "X-Correlation-ID", required = false) String correlationId) {

    // Restore MDC context from message headers
    if (traceId != null) MDC.put("traceId", traceId);
    if (correlationId != null) MDC.put("correlationId", correlationId);

    try {
        processEvent(record.value());
    } finally {
        MDC.remove("traceId");
        MDC.remove("correlationId");
    }
}
```

## 7. Interview Questions

**Q: What are the three pillars of observability and how do they complement each other?**

A: The three pillars are logs, metrics, and traces. Metrics (aggregated numbers over time) tell you _that_ something is wrong — error rate is rising, latency P99 is high. Logs tell you _what_ happened in detail — the specific error message, stack trace, and context for individual events. Traces tell you _where_ time is being spent — which service in a distributed call chain is causing the slowness. In practice: a Grafana alert fires on a metrics anomaly, you check the metrics dashboard to understand the scope, you open a trace to see which service is slow, and you search logs filtered by the trace ID to see the exact error. Each pillar answers a different question, and they're linked by the trace ID.

---

**Q: What is the difference between Zipkin and Jaeger and when would you choose each?**

A: Both are distributed tracing backends, but they differ in design goals. Zipkin is simpler and Spring Boot-native — the Brave bridge integrates directly and it has lower operational overhead. Jaeger is CNCF-graduated, OpenTelemetry-native, and designed for Kubernetes environments. Jaeger supports adaptive sampling (automatically adjusting the sample rate based on traffic volume), has richer UI features, and integrates more naturally with the OTEL ecosystem. Choose Zipkin for simpler Spring Boot stacks where you want minimal setup; choose Jaeger in Kubernetes environments with Istio or when you're committed to the OpenTelemetry standard for vendor portability.

---

**Q: Why is structured (JSON) logging important for centralized log aggregation?**

A: Plain text logs are parseable only with fragile regex patterns that break when log format changes. JSON logs are machine-parseable by definition — Elasticsearch can index every field automatically, Kibana can filter on `traceId`, `userId`, or `orderStatus` as structured fields rather than extracting them from strings. Structured logging also enables log correlation: because `traceId` is a proper field (not buried in a string), you can run `traceId:"abc123"` to instantly retrieve all logs for one request across all services. The logstash-logback-encoder library makes this trivial — one dependency and one logback config change produces JSON logs that include MDC fields (traceId, spanId) automatically.

---

**Q: How does Micrometer Tracing propagate trace context across service boundaries?**

A: Micrometer Tracing injects trace context into outgoing HTTP headers automatically for `WebClient` and `RestTemplate`. It uses either W3C TraceContext format (`traceparent` header) or Zipkin's B3 format (`X-B3-TraceId`, `X-B3-SpanId`). On the receiving side, Spring's inbound filter extracts these headers and creates a child span with the same trace ID — continuing the trace rather than starting a new one. For Kafka, Micrometer Tracing (via Spring Kafka instrumentation) injects trace context into Kafka message headers on produce and extracts it on consume. The trace ID is also injected into the MDC (Mapped Diagnostic Context), making it available in every log line produced during that request without explicit logging code.

---

**Q: What is the purpose of a correlation ID versus a trace ID?**

A: A trace ID is a technical identifier generated by the tracing system to correlate spans across services for a single request. It's automatically generated and propagated. A correlation ID is a business-level identifier — it might be a user session ID, an order ID, or a client-provided request ID that has meaning to the business and the client. The client can include `X-Correlation-ID: order-1234-checkout` in their request and later call support asking "what happened to correlation ID order-1234-checkout?" — something more meaningful than a UUID. In practice, you want both: the trace ID for technical debugging in Jaeger/Zipkin, and a correlation ID in logs and API responses so that clients and support teams can reference a specific operation in a human-readable way.

---

**Q: What does a service mesh add to observability that application-level instrumentation doesn't provide?**

A: Application-level instrumentation (Micrometer) gives you metrics and traces for operations your code knows about — HTTP handlers, database queries, Kafka messages. The service mesh sidecar captures metrics and traces for the _network layer_ between services without any code changes: connection counts, TCP errors, request timeouts at the proxy level, mTLS verification status, and retry attempts the proxy makes transparently. This is especially valuable for detecting network-level issues that your application never sees — a request that times out at the network layer before reaching your service won't produce an application span, but the Envoy sidecar records it. Kiali's service graph, built entirely from sidecar-captured data, shows you the real topology of your service mesh including services that have no Micrometer instrumentation.

---

## 8. Key Takeaways

- **Three pillars**: metrics (what's wrong), logs (what happened), traces (where time went) — all linked by trace ID
- **Micrometer Tracing** is the Spring Boot 3 standard — vendor-neutral facade supporting both Brave/Zipkin and OTEL/Jaeger bridges; auto-instruments HTTP, JDBC, Kafka
- **Structured JSON logging** is mandatory for centralized log aggregation — use logstash-logback-encoder with MDC for auto-included traceId
- **ELK** for engineering teams wanting open-source control; **Splunk** for enterprises needing managed operational simplicity
- **Prometheus + Grafana + Alertmanager** is the cloud-native metrics stack — use the RED method (Rate, Errors, Duration) for service dashboards
- **Service mesh observability** (Istio/Linkerd) gives network-layer metrics and traces for free — complements, never replaces, application-level instrumentation
- **Correlation IDs** link technical traces to business context — propagate in HTTP headers, Kafka message headers, and always include in log MDC
