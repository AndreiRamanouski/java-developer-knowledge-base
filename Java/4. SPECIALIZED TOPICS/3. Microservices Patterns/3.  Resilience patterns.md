# Resilience Patterns

## Overview

Resilience patterns protect a service from the inevitable failures of its dependencies. In a microservices system with 10 services, each with 99.9% availability, the probability that all 10 are up simultaneously is 99.0% — a 10x degradation from a single service. Without resilience patterns, one slow or failing dependency cascades into a system-wide outage.

**The core principle**: fail fast, fail gracefully, recover automatically.

**Resilience4j** is the modern standard for Spring Boot applications (replacing the deprecated Hystrix). It's modular, lightweight, and built for Java 8+ functional programming.

```xml
<!-- Resilience4j Spring Boot 3 starter — includes all modules -->
<dependency>
    <groupId>io.github.resilience4j</groupId>
    <artifactId>resilience4j-spring-boot3</artifactId>
    <version>2.2.0</version>
</dependency>
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-aop</artifactId>
</dependency>

<!-- Micrometer metrics integration -->
<dependency>
    <groupId>io.github.resilience4j</groupId>
    <artifactId>resilience4j-micrometer</artifactId>
</dependency>
```

## 1. Circuit Breaker — Resilience4j and Hystrix

### 1.1 Resilience4j Circuit Breaker — Core Configuration

```yaml
# application.yml
resilience4j:
  circuitbreaker:
    instances:
      payment-service:
        # ── Failure detection ───────────────────────────────────────────
        failure-rate-threshold: 50            # open if ≥50% calls fail
        slow-call-rate-threshold: 80          # also open if ≥80% are slow
        slow-call-duration-threshold: 3s      # "slow" = longer than 3s
        minimum-number-of-calls: 10           # ignore rate until ≥10 calls

        # ── Sliding window ──────────────────────────────────────────────
        sliding-window-type: COUNT_BASED      # COUNT_BASED or TIME_BASED
        sliding-window-size: 20               # last 20 calls evaluated
        # TIME_BASED alternative:
        # sliding-window-type: TIME_BASED
        # sliding-window-size: 10             # last 10 seconds evaluated

        # ── OPEN state ──────────────────────────────────────────────────
        wait-duration-in-open-state: 30s      # stay OPEN for 30s before probing

        # ── HALF-OPEN state ─────────────────────────────────────────────
        permitted-number-of-calls-in-half-open-state: 5
        automatic-transition-from-open-to-half-open-enabled: true

        # ── Exception handling ──────────────────────────────────────────
        record-exceptions:
          - java.io.IOException
          - java.util.concurrent.TimeoutException
          - feign.FeignException$ServiceUnavailable
        ignore-exceptions:
          - com.example.NotFoundException       # 404 is not a circuit failure
          - com.example.ValidationException     # client errors don't count
```

### 1.2 Annotation-based usage

```java
@Service
@RequiredArgsConstructor
@Slf4j
public class PaymentServiceClient {

    private final WebClient webClient;

    @CircuitBreaker(name = "payment-service", fallbackMethod = "chargeCardFallback")
    public Mono<PaymentResult> chargeCard(ChargeRequest request) {
        return webClient.post()
            .uri("http://payment-service/api/payments/charge")
            .bodyValue(request)
            .retrieve()
            .onStatus(HttpStatus::is5xxServerError,
                res -> Mono.error(new ServiceUnavailableException("Payment service error")))
            .bodyToMono(PaymentResult.class);
    }

    // Fallback for any exception
    private Mono<PaymentResult> chargeCardFallback(ChargeRequest request, Throwable ex) {
        log.warn("Payment service unavailable, using fallback. Cause: {}", ex.getMessage());
        return Mono.just(PaymentResult.pending(request.getOrderId()));
    }

    // More specific fallback for circuit OPEN (overrides generic)
    private Mono<PaymentResult> chargeCardFallback(ChargeRequest request,
                                                    CallNotPermittedException ex) {
        log.warn("Payment circuit is OPEN — fast failing order {}", request.getOrderId());
        return Mono.just(PaymentResult.circuitOpen(request.getOrderId()));
    }
}
```

### 1.3 Programmatic usage with Reactor

```java
@Service
@RequiredArgsConstructor
@Slf4j
public class PaymentServiceClient {

    private final CircuitBreakerRegistry registry;
    private final WebClient webClient;

    public Mono<PaymentResult> chargeCard(ChargeRequest request) {
        CircuitBreaker cb = registry.circuitBreaker("payment-service");

        return webClient.post()
            .uri("http://payment-service/api/payments/charge")
            .bodyValue(request)
            .retrieve()
            .bodyToMono(PaymentResult.class)
            .transformDeferred(CircuitBreakerOperator.of(cb))   // wrap with CB
            .doOnError(ex -> log.error("Payment failed: {}", ex.getMessage()))
            .onErrorResume(CallNotPermittedException.class,
                ex -> Mono.just(PaymentResult.circuitOpen(request.getOrderId())))
            .onErrorResume(ex ->
                Mono.just(PaymentResult.failed(request.getOrderId(), ex.getMessage())));
    }

    @PostConstruct
    public void attachEventListeners() {
        registry.circuitBreaker("payment-service")
            .getEventPublisher()
            .onStateTransition(e -> log.warn("CB transition: {} → {}",
                e.getStateTransition().getFromState(),
                e.getStateTransition().getToState()))
            .onFailureRateExceeded(e ->
                log.error("Failure rate {}% exceeded threshold", e.getFailureRate()))
            .onCallNotPermitted(e ->
                log.debug("Call rejected — circuit OPEN"));
    }
}
```

Hystrix was Netflix's original circuit breaker, now in maintenance mode. Understanding it matters for maintaining legacy Spring Cloud Netflix codebases.

```java
            @HystrixProperty(name = "coreSize", value = "10"),
            @HystrixProperty(name = "maxQueueSize", value = "20")
        }

```

### 1.5 Hystrix vs Resilience4j

||Hystrix|Resilience4j|
|---|---|---|
|Status|Maintenance only (2018)|Actively developed|
|Architecture|Thread pool isolation|Decorators / operators|
|Reactive|Partial (HystrixObservableCommand)|Native Reactor/RxJava|
|Configuration|Archaius properties|Spring Boot YAML|
|Metrics|Hystrix Dashboard, Turbine|Micrometer (Prometheus, Grafana)|
|Modularity|Monolithic|Modular (use only what you need)|
|Spring Cloud|spring-cloud-netflix (deprecated)|spring-cloud-circuitbreaker|
|Overhead|Higher (thread pools)|Lower (semaphore/decorators)|
|**Verdict**|Legacy — migrate to Resilience4j|Use this|

**Migration from Hystrix:**

```java
// BEFORE (Hystrix)
@HystrixCommand(fallbackMethod = "fallback")
public String call() { return restTemplate.getForObject(url, String.class); }

// AFTER (Resilience4j)
@CircuitBreaker(name = "my-service", fallbackMethod = "fallback")
@Retry(name = "my-service")
@TimeLimiter(name = "my-service")
public CompletableFuture<String> call() {
    return CompletableFuture.supplyAsync(() ->
        restTemplate.getForObject(url, String.class));
}
```

## 2. Retry with Backoff

Transient failures (network blips, brief overload) are recoverable if you wait and try again. The key is to retry intelligently — not immediately, not indefinitely, and not all error types.

### 2.1 Retry configuration

```yaml
resilience4j:
  retry:
    instances:
      inventory-service:
        max-attempts: 4                       # 1 original + 3 retries
        wait-duration: 500ms                  # wait 500ms between attempts

        # Exponential backoff
        enable-exponential-backoff: true
        exponential-backoff-multiplier: 2.0   # 500ms → 1s → 2s
        exponential-max-wait-duration: 10s    # cap at 10s

        # Jitter — randomize wait to prevent thundering herd
        enable-randomized-wait: true
        randomized-wait-factor: 0.5           # ±50% jitter: 250ms-750ms, 500ms-1500ms...

        # What to retry
        retry-exceptions:
          - java.io.IOException
          - java.net.SocketTimeoutException
          - org.springframework.web.reactive.function.client.WebClientRequestException

        # What NOT to retry — client errors and business exceptions
        ignore-exceptions:
          - com.example.NotFoundException
          - com.example.ValidationException
          - io.github.resilience4j.circuitbreaker.CallNotPermittedException
```

### 2.2 Annotation-based retry

```java
@Service
@RequiredArgsConstructor
@Slf4j
public class InventoryClient {

    private final WebClient webClient;

    @Retry(name = "inventory-service", fallbackMethod = "checkStockFallback")
    @CircuitBreaker(name = "inventory-service")   // CB wraps retry — CB sees all attempts
    public Mono<StockLevel> checkStock(Long productId) {
        return webClient.get()
            .uri("http://inventory-service/api/stock/{id}", productId)
            .retrieve()
            .bodyToMono(StockLevel.class);
    }

    // Fallback called only when all retries exhausted
    private Mono<StockLevel> checkStockFallback(Long productId, Throwable ex) {
        log.error("All retries exhausted for product {}: {}", productId, ex.getMessage());
        return Mono.just(StockLevel.unknown(productId));  // optimistic default
    }
}
```

### 2.3 Programmatic retry with Reactor

```java
public Mono<StockLevel> checkStock(Long productId) {
    RetryConfig config = RetryConfig.custom()
        .maxAttempts(4)
        .waitDuration(Duration.ofMillis(500))
        .intervalFunction(IntervalFunction.ofExponentialRandomBackoff(
            Duration.ofMillis(500),  // base
            2.0,                     // multiplier
            Duration.ofSeconds(10)   // max wait
        ))
        .retryOnException(ex ->
            ex instanceof IOException || ex instanceof SocketTimeoutException)
        .ignoreException(ex ->
            ex instanceof NotFoundException || ex instanceof ValidationException)
        .build();

    Retry retry = Retry.of("inventory-service", config);

    return webClient.get()
        .uri("http://inventory-service/api/stock/{id}", productId)
        .retrieve()
        .bodyToMono(StockLevel.class)
        .transformDeferred(RetryOperator.of(retry))
        .doOnError(ex -> log.error("All retries exhausted for product {}", productId, ex));
}
```

### 2.4 Exponential backoff with jitter — why jitter matters

```
Without jitter — thundering herd:
  T=0:    10 services fail simultaneously
  T=500ms: all 10 retry at the same time → same overload
  T=1000ms: all 10 retry again → still overwhelmed

With jitter — staggered retries:
  T=0:    10 services fail simultaneously
  T=300-700ms: retries spread randomly → downstream load 10x lower per interval
  T=600-1400ms: second retries spread further → downstream can recover
```

```java
// Spring Retry with exponential backoff + jitter (alternative library)
@Retryable(
    retryFor = {IOException.class, SocketTimeoutException.class},
    noRetryFor = {NotFoundException.class},
    maxAttempts = 4,
    backoff = @Backoff(
        delay = 500,
        multiplier = 2.0,
        maxDelay = 10000,
        random = true          // adds jitter
    )
)
public StockLevel checkStockBlocking(Long productId) {
    return inventoryRestTemplate.getForObject("/api/stock/{id}", StockLevel.class, productId);
}
```

### 2.5 Retry + Circuit Breaker interaction — ordering matters

```java
// CORRECT order — retry is INSIDE the circuit breaker
// CB sees only the final outcome (success or all-retries-exhausted failure)
// This protects the downstream service from N × retry-attempts calls when it's struggling

@Retry(name = "service")           // 2. inner — retries on failure
@CircuitBreaker(name = "service")  // 1. outer — wraps the entire retry sequence
public Mono<Result> call() { ... }

// Resilience4j decorator order in code:
Supplier<Mono<Result>> decorated = Decorators.ofSupplier(() -> makeCall())
    .withCircuitBreaker(circuitBreaker)   // outer
    .withRetry(retry)                     // inner
    .withTimeLimiter(timeLimiter)
    .withBulkhead(bulkhead)
    .decorate();
```

## 3. Timeout Strategies

Timeouts are the simplest and most important resilience mechanism. Without them, slow dependencies hold threads/resources indefinitely, eventually exhausting the thread pool.

### 3.1 Resilience4j TimeLimiter

```yaml
resilience4j:
  timelimiter:
    instances:
      payment-service:
        timeout-duration: 5s           # cancel after 5s
        cancel-running-future: true    # cancel the underlying CompletableFuture
```

```java
// TimeLimiter requires CompletableFuture return type
@TimeLimiter(name = "payment-service", fallbackMethod = "timeoutFallback")
@CircuitBreaker(name = "payment-service")
public CompletableFuture<PaymentResult> chargeCard(ChargeRequest request) {
    return CompletableFuture.supplyAsync(() ->
        paymentClient.charge(request), asyncExecutor);
}

private CompletableFuture<PaymentResult> timeoutFallback(ChargeRequest req,
                                                          TimeoutException ex) {
    log.warn("Payment timed out for order {}", req.getOrderId());
    return CompletableFuture.completedFuture(
        PaymentResult.timedOut(req.getOrderId()));
}
```

### 3.2 WebClient timeout (reactive)

```java
// Three layers of timeout in reactive WebClient calls
@Component
public class PaymentWebClient {

    private final WebClient webClient;

    public PaymentWebClient() {
        // 1. TCP connection timeout — how long to wait to establish the connection
        HttpClient httpClient = HttpClient.create()
            .option(ChannelOption.CONNECT_TIMEOUT_MILLIS, 2000)   // 2s to connect
            .responseTimeout(Duration.ofSeconds(5))                // 5s for response
            .doOnConnected(conn -> conn
                .addHandlerLast(new ReadTimeoutHandler(5))         // read timeout
                .addHandlerLast(new WriteTimeoutHandler(2)));      // write timeout

        this.webClient = WebClient.builder()
            .clientConnector(new ReactorClientHttpConnector(httpClient))
            .build();
    }

    public Mono<PaymentResult> charge(ChargeRequest request) {
        return webClient.post()
            .uri("http://payment-service/api/payments/charge")
            .bodyValue(request)
            .retrieve()
            .bodyToMono(PaymentResult.class)
            // 2. Reactor timeout — overall pipeline deadline
            .timeout(Duration.ofSeconds(8))
            // 3. Fallback on timeout
            .onErrorResume(TimeoutException.class,
                ex -> Mono.just(PaymentResult.timedOut(request.getOrderId())));
    }
}
```

### 3.3 RestTemplate timeout (imperative)

```java
@Bean
public RestTemplate restTemplate() {
    HttpComponentsClientHttpRequestFactory factory =
        new HttpComponentsClientHttpRequestFactory();
    factory.setConnectTimeout(2000);         // 2s to connect
    factory.setConnectionRequestTimeout(1000); // 1s to get connection from pool
    factory.setReadTimeout(5000);            // 5s for response

    return new RestTemplate(factory);
}
```

### 3.4 Database query timeout

```yaml
# application.yml — Spring Data / R2DBC timeout
spring:
  datasource:
    hikari:
      connection-timeout: 3000          # 3s to acquire connection from pool
      validation-timeout: 1000          # 1s for connection validation
      idle-timeout: 600000              # 10min before idle connection evicted
      max-lifetime: 1800000             # 30min max connection lifetime

  r2dbc:
    properties:
      lock-timeout: 5s
      statement-timeout: 10s
```

```java
// Query-level timeout in Spring Data JPA
@Repository
public interface OrderRepository extends JpaRepository<Order, Long> {

    @QueryHints(@QueryHint(name = "javax.persistence.query.timeout", value = "5000"))
    @Query("SELECT o FROM Order o WHERE o.userId = :userId")
    List<Order> findByUserId(@Param("userId") Long userId);
}

// Programmatic timeout for R2DBC
public Mono<Order> findById(Long id) {
    return orderRepository.findById(id)
        .timeout(Duration.ofSeconds(3))
        .onErrorResume(TimeoutException.class,
            ex -> Mono.error(new ServiceException("DB query timed out for order " + id)));
}
```

### 3.5 Timeout strategy guidelines

```
Connection timeout:   shortest (1-3s) — TCP handshake should be near-instant
Read timeout:         matches your SLA (3-10s typical for synchronous calls)
Circuit breaker:      slightly longer than read timeout (catches slow calls)
API gateway timeout:  longest — sum of all downstream timeouts + processing

                  User
                   │ 10s total timeout
                   ▼
            API Gateway
                   │ 8s
                   ▼
            Service A
           /        \
          │ 3s        │ 3s
          ▼           ▼
       Service B   Service C (parallel calls)

Rule: set timeouts at EVERY layer, not just the outermost.
A 10s gateway timeout does not protect Service A's thread pool
from a 30s connection to Service B.
```

## 4. Bulkhead Pattern

A bulkhead isolates a resource pool per dependency so that a slow or failing dependency can only consume a bounded slice of resources, leaving the rest available for healthy dependencies.

### 4.1 The problem without bulkheads

```
Without bulkhead:
  Service A has 200 threads (Tomcat default)
  Payment service starts being slow (20s response)
  Requests pile up → all 200 threads waiting on payment
  User service calls (fast, 10ms) also queue → entire service unresponsive

With bulkhead:
  Payment service: max 20 threads
  Inventory service: max 20 threads
  User service: max 20 threads
  Remaining 140 threads: general purpose
  → Payment slowdown affects only those 20 threads
```

### 4.2 Semaphore bulkhead — concurrent call limit

```yaml
resilience4j:
  bulkhead:
    instances:
      payment-service:
        max-concurrent-calls: 20       # max simultaneous in-flight calls
        max-wait-duration: 100ms       # wait for a slot; reject after 100ms
      inventory-service:
        max-concurrent-calls: 30
        max-wait-duration: 0           # reject immediately if no slot
      user-service:
        max-concurrent-calls: 50
        max-wait-duration: 200ms
```

```java
@Service
@RequiredArgsConstructor
public class PaymentClient {

    @Bulkhead(name = "payment-service",
              type = Bulkhead.Type.SEMAPHORE,
              fallbackMethod = "chargeFallback")
    @CircuitBreaker(name = "payment-service")
    public Mono<PaymentResult> charge(ChargeRequest request) {
        return webClient.post()
            .uri("http://payment-service/api/payments/charge")
            .bodyValue(request)
            .retrieve()
            .bodyToMono(PaymentResult.class);
    }

    private Mono<PaymentResult> chargeFallback(ChargeRequest request,
                                                BulkheadFullException ex) {
        log.warn("Bulkhead full for payment service — rejecting order {}",
            request.getOrderId());
        return Mono.just(PaymentResult.rejected("Payment service at capacity"));
    }
}
```

### 4.3 Thread pool bulkhead — dedicated thread pool per dependency

```yaml
resilience4j:
  thread-pool-bulkhead:
    instances:
      payment-service:
        max-thread-pool-size: 10     # dedicated thread pool
        core-thread-pool-size: 5
        queue-capacity: 20           # task queue before rejection
        keep-alive-duration: 20ms
        writeable-stack-trace-enabled: true
```

```java
// Thread pool bulkhead requires CompletableFuture
@Bulkhead(name = "payment-service",
          type = Bulkhead.Type.THREADPOOL,
          fallbackMethod = "chargeFallback")
public CompletableFuture<PaymentResult> charge(ChargeRequest request) {
    // Runs on the dedicated payment-service thread pool
    return CompletableFuture.supplyAsync(() -> paymentClient.charge(request));
}

private CompletableFuture<PaymentResult> chargeFallback(ChargeRequest request,
                                                          BulkheadFullException ex) {
    return CompletableFuture.completedFuture(
        PaymentResult.rejected("Thread pool full"));
}
```

### 4.4 Semaphore vs Thread Pool bulkhead

||Semaphore|Thread Pool|
|---|---|---|
|Resource isolated|Concurrent count|Dedicated thread pool|
|Works with|Reactive / sync|Blocking / CompletableFuture|
|Overhead|Very low|Thread pool management|
|Timeout support|No (use TimeLimiter)|Yes (via Future timeout)|
|Use case|Reactive, fast operations|Blocking, legacy integrations|
|Hystrix equivalent|Semaphore isolation|Thread isolation (default in Hystrix)|

### 4.5 Bulkhead with metrics

```java
@Component
@RequiredArgsConstructor
public class BulkheadMonitor {

    private final BulkheadRegistry bulkheadRegistry;
    private final MeterRegistry meterRegistry;

    @PostConstruct
    public void setupMetrics() {
        Bulkhead bulkhead = bulkheadRegistry.bulkhead("payment-service");

        // Current metrics
        Gauge.builder("bulkhead.available.slots", bulkhead,
            b -> b.getMetrics().getAvailableConcurrentCalls())
            .tag("name", "payment-service")
            .register(meterRegistry);

        // Event-based monitoring
        bulkhead.getEventPublisher()
            .onCallRejected(e ->
                log.warn("Bulkhead rejected call to payment-service"))
            .onCallFinished(e ->
                log.debug("Bulkhead slot released: {} remaining",
                    bulkhead.getMetrics().getAvailableConcurrentCalls()));
    }
}
```

## 5. Rate Limiting

Rate limiting controls how many requests a service processes per time window — protecting it from being overwhelmed by a single caller or a sudden traffic spike.

### 5.1 Resilience4j RateLimiter

```yaml
resilience4j:
  ratelimiter:
    instances:
      external-api:
        limit-for-period: 100          # 100 calls per period
        limit-refresh-period: 1s       # refresh window = 1 second = 100 RPS
        timeout-duration: 500ms        # wait up to 500ms for a permit; reject after
        subscribe-for-events: true
      payment-gateway:
        limit-for-period: 10           # external API has its own rate limit
        limit-refresh-period: 1s
        timeout-duration: 0            # fail immediately if no permit
```

```java
@Service
@RequiredArgsConstructor
public class ExternalApiClient {

    @RateLimiter(name = "external-api", fallbackMethod = "rateLimitedFallback")
    public Mono<ApiResponse> callExternalApi(ApiRequest request) {
        return webClient.post()
            .uri("https://external-api.com/endpoint")
            .bodyValue(request)
            .retrieve()
            .bodyToMono(ApiResponse.class);
    }

    private Mono<ApiResponse> rateLimitedFallback(ApiRequest request,
                                                   RequestNotPermitted ex) {
        log.warn("Rate limit exceeded for external API");
        return Mono.error(new TooManyRequestsException("Rate limit exceeded"));
    }
}
```

### 5.2 Spring Cloud Gateway rate limiting (Redis token bucket)

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-gateway</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-redis-reactive</artifactId>
</dependency>
```

```yaml
# application.yml — rate limiting at the gateway layer
spring:
  cloud:
    gateway:
      routes:
        - id: order-service
          uri: lb://order-service
          predicates:
            - Path=/api/orders/**
          filters:
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 100   # tokens added per second
                redis-rate-limiter.burstCapacity: 200   # max tokens (burst allowance)
                redis-rate-limiter.requestedTokens: 1   # tokens consumed per request
                key-resolver: "#{@userKeyResolver}"     # rate limit per user
```

```java
@Configuration
public class RateLimiterConfig {

    // Rate limit per authenticated user
    @Bean
    public KeyResolver userKeyResolver() {
        return exchange -> Mono.just(
            exchange.getRequest().getHeaders()
                .getFirst("X-User-Id")
        ).defaultIfEmpty("anonymous");
    }

    // Rate limit per IP
    @Bean
    public KeyResolver ipKeyResolver() {
        return exchange -> Mono.just(
            Objects.requireNonNull(exchange.getRequest()
                .getRemoteAddress()).getAddress().getHostAddress()
        );
    }

    // Rate limit per API key
    @Bean
    public KeyResolver apiKeyResolver() {
        return exchange -> Mono.just(
            Optional.ofNullable(
                exchange.getRequest().getHeaders().getFirst("X-API-Key")
            ).orElse("no-key")
        );
    }
}
```

### 5.3 Bucket4j — token bucket for service-level rate limiting

```xml
<dependency>
    <groupId>com.bucket4j</groupId>
    <artifactId>bucket4j-core</artifactId>
    <version>8.7.0</version>
</dependency>
<dependency>
    <groupId>com.bucket4j</groupId>
    <artifactId>bucket4j-redis</artifactId>  <!-- distributed rate limiting -->
</dependency>
```

```java
@Component
public class ApiRateLimiter {

    // Token bucket: 100 requests per second, burst of 200
    private final Bucket bucket = Bucket.builder()
        .addLimit(Bandwidth.classic(200, Refill.greedy(100, Duration.ofSeconds(1))))
        .build();

    // Distributed: per-user rate limiting with Redis backend
    private final ProxyManager<String> proxyManager;

    public boolean tryConsume(String userId) {
        BucketConfiguration config = BucketConfiguration.builder()
            .addLimit(Bandwidth.classic(100, Refill.greedy(100, Duration.ofMinutes(1))))
            .build();

        Bucket userBucket = proxyManager.builder()
            .build(userId, () -> config);

        return userBucket.tryConsume(1);
    }
}

// Rate limiting filter for Spring MVC
@Component
@Order(1)
public class RateLimitingFilter implements HandlerInterceptor {

    private final ApiRateLimiter rateLimiter;

    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response,
                             Object handler) throws Exception {
        String userId = (String) request.getAttribute("userId");

        if (!rateLimiter.tryConsume(userId)) {
            response.setStatus(HttpStatus.TOO_MANY_REQUESTS.value());
            response.setHeader("X-RateLimit-RetryAfter", "1");
            response.getWriter().write("{\"error\": \"Too many requests\"}");
            return false;
        }
        return true;
    }
}
```

### 5.4 Rate limiting response headers (best practice)

```java
// Always return rate limit headers so clients can self-throttle
@GetMapping("/api/resource")
public ResponseEntity<Resource> getResource(HttpServletRequest request) {
    RateLimitStatus status = rateLimiter.getStatus(getUserId(request));

    return ResponseEntity.ok()
        .header("X-RateLimit-Limit", String.valueOf(status.getLimit()))
        .header("X-RateLimit-Remaining", String.valueOf(status.getRemaining()))
        .header("X-RateLimit-Reset", String.valueOf(status.getResetEpochSeconds()))
        .body(resourceService.getResource());
}

// 429 response when limit exceeded
// Retry-After header tells client when to try again
return ResponseEntity.status(HttpStatus.TOO_MANY_REQUESTS)
    .header("Retry-After", "60")
    .body(new ErrorResponse("Rate limit exceeded. Try again in 60 seconds."));
```

## 6. Failover and Fallback

Failover routes traffic to a secondary system when the primary fails. Fallback returns degraded but acceptable data instead of an error.

### 6.1 Fallback strategies

```java
@Service
@RequiredArgsConstructor
@Slf4j
public class ProductService {

    private final ProductRepository primaryDb;
    private final RedisTemplate<String, Product> cache;
    private final ProductSearchService searchService;

    // Strategy 1: Cache fallback
    @CircuitBreaker(name = "product-db", fallbackMethod = "getFromCache")
    public Mono<Product> getProduct(Long id) {
        return primaryDb.findById(id)
            .switchIfEmpty(Mono.error(new NotFoundException("Product " + id)));
    }

    private Mono<Product> getFromCache(Long id, Throwable ex) {
        log.warn("DB unavailable, checking cache for product {}", id);
        return Mono.fromCallable(() -> cache.opsForValue().get("product:" + id))
            .filter(Objects::nonNull)
            .switchIfEmpty(Mono.error(new ServiceUnavailableException(
                "Product service unavailable and cache miss")));
    }

    // Strategy 2: Static/default data fallback
    public Mono<Product> getProductWithDefault(Long id) {
        return getProduct(id)
            .onErrorReturn(Product.placeholder(id)); // always returns something
    }

    // Strategy 3: Stale data with staleness indicator
    public Mono<ProductResponse> getProductStaleOk(Long id) {
        return getProduct(id)
            .map(p -> new ProductResponse(p, false, null))
            .onErrorResume(ex ->
                getCachedProduct(id)
                    .map(p -> new ProductResponse(p, true, "Data may be outdated"))
            );
    }

    // Strategy 4: Partial response — return what's available
    public Mono<ProductPageDto> getProductPage(Long id) {
        Mono<Product> product = getProduct(id)
            .onErrorReturn(Product.minimal(id)); // minimal product on error

        Mono<List<Review>> reviews = reviewService.getReviews(id)
            .collectList()
            .onErrorReturn(Collections.emptyList()); // empty reviews on error

        Mono<List<Product>> related = relatedService.getRelated(id)
            .collectList()
            .onErrorReturn(Collections.emptyList()); // no recommendations on error

        return Mono.zip(product, reviews, related)
            .map(t -> new ProductPageDto(t.getT1(), t.getT2(), t.getT3()));
        // Page always loads — worst case: minimal product, no reviews, no related
    }
}
```

### 6.2 Geographic failover

```java
// Primary region fails → failover to secondary region
@Component
@RequiredArgsConstructor
@Slf4j
public class GeoFailoverClient {

    private final WebClient primaryClient;   // us-east-1
    private final WebClient secondaryClient; // eu-west-1
    private final CircuitBreakerRegistry cbRegistry;

    public Mono<OrderDto> getOrder(Long id) {
        return primaryClient.get()
            .uri("/api/orders/{id}", id)
            .retrieve()
            .bodyToMono(OrderDto.class)
            .transformDeferred(CircuitBreakerOperator.of(
                cbRegistry.circuitBreaker("primary-region")))
            .onErrorResume(ex -> {
                log.warn("Primary region failed, failing over to secondary: {}", ex.getMessage());
                return secondaryClient.get()
                    .uri("/api/orders/{id}", id)
                    .retrieve()
                    .bodyToMono(OrderDto.class)
                    .timeout(Duration.ofSeconds(5));
            });
    }
}
```

```java
@Configuration
public class DataSourceConfig {

    @Primary

            @Qualifier("primaryDs") DataSource primary,

                    // Test connectivity before returning

                    return primary;  // fallback to primary for reads too

            }
        };
    }
```

### 6.4 Saga pattern — compensating transactions for failover

```java
// When a distributed operation partially fails, compensate to maintain consistency
@Service
@RequiredArgsConstructor
@Slf4j
public class OrderSagaService {

    public Mono<Order> placeOrder(CreateOrderRequest request) {
        return Mono.empty()
            // Step 1: reserve inventory
            .then(inventoryService.reserve(request.getItems()))
            .flatMap(reservation ->
                // Step 2: charge payment
                paymentService.charge(request.getPayment())
                    .onErrorResume(ex -> {
                        // Payment failed — compensate: release inventory
                        log.warn("Payment failed, releasing inventory reservation");
                        return inventoryService.release(reservation.getId())
                            .then(Mono.error(ex));   // re-throw after compensation
                    })
            )
            .flatMap(payment ->
                // Step 3: create order record
                orderRepository.save(request.toOrder(payment))
                    .onErrorResume(ex -> {
                        // Save failed — compensate: refund payment + release inventory
                        log.warn("Order save failed, initiating compensation");
                        return Mono.when(
                            paymentService.refund(payment.getId()),
                            inventoryService.release(payment.getReservationId())
                        ).then(Mono.error(ex));
                    })
            );
    }
}
```

### 6.5 Fallback decision matrix

```
Exception Type               → Fallback Strategy
──────────────────────────────────────────────────────────────────
NotFoundException (404)      → No fallback — propagate as-is
ValidationException (400)    → No fallback — propagate as-is
TimeoutException             → Cached data or empty/default response
ServiceUnavailableException  → Cached data, reduced feature set
CallNotPermittedException    → Cached data (circuit open — avoid retry)
BulkheadFullException        → Queue for later or HTTP 429
RateLimitException           → HTTP 429 with Retry-After header
Database unavailable         → Read from cache, writes to queue
Network partition            → Failover to secondary region
```

```java
// Stacking all patterns — annotation order matters
// Outer → Inner execution order:
// RateLimiter → Bulkhead → CircuitBreaker → Retry → TimeLimiter → actual call

@RateLimiter(name = "payment-service")
@Bulkhead(name = "payment-service", type = Bulkhead.Type.SEMAPHORE)
@CircuitBreaker(name = "payment-service", fallbackMethod = "fallback")
@Retry(name = "payment-service")
@TimeLimiter(name = "payment-service")
public CompletableFuture<PaymentResult> processPayment(PaymentRequest request) {
    return CompletableFuture.supplyAsync(() ->
        paymentClient.process(request)
    );
}

private CompletableFuture<PaymentResult> fallback(PaymentRequest request, Throwable ex) {
    return switch (ex) {
        case CallNotPermittedException e ->
            CompletableFuture.completedFuture(PaymentResult.circuitOpen());
        case BulkheadFullException e ->
            CompletableFuture.completedFuture(PaymentResult.rejected("At capacity"));
        case RequestNotPermitted e ->
            CompletableFuture.completedFuture(PaymentResult.rateLimited());
        case TimeoutException e ->
            CompletableFuture.completedFuture(PaymentResult.timedOut());
        default ->
            CompletableFuture.completedFuture(PaymentResult.failed(ex.getMessage()));
    };
}
```

## 8. Interview Questions

**Q: Why was Hystrix deprecated in favor of Resilience4j?**

A: Hystrix entered maintenance mode in 2018 because it was built for a different era of Java — it relies heavily on RxJava 1, uses thread pool isolation by default (expensive), and doesn't integrate cleanly with Project Reactor or Java 8+ functional patterns. Resilience4j is modular (you include only what you need), uses decorators rather than commands, supports reactive pipelines natively via `CircuitBreakerOperator.of()`, integrates with Micrometer for metrics, and is configured through standard Spring Boot YAML. The annotation model is also simpler — `@CircuitBreaker` vs `@HystrixCommand` with nested property annotations.

---

**Q: When should you use a semaphore bulkhead versus a thread pool bulkhead?**

A: Use a semaphore bulkhead for reactive or non-blocking code — it limits the number of concurrent calls without the overhead of a separate thread pool, which would be wasted since reactive code doesn't hold threads during I/O. Use a thread pool bulkhead when the call is blocking — it isolates the blocking operation on a dedicated pool so it doesn't starve the main thread pool. The thread pool bulkhead also inherently provides a timeout through `Future.get(timeout)`, while a semaphore bulkhead relies on `TimeLimiter` for timeout enforcement.

---

**Q: What is the thundering herd problem with retries and how does jitter solve it?**

A: If many clients fail simultaneously (e.g., a brief outage), and all retry with the same exponential backoff schedule, they all retry at the same moments in time — at T+500ms, T+1000ms, T+2000ms simultaneously. This synchronized burst can re-overload the recovering service, extending the outage. Jitter adds random variation to the wait duration (e.g., ±50% of the computed backoff). This spreads retries across a time window, turning a synchronized burst into a smooth trickle that a recovering service can handle. The `Retry.backoff` in Resilience4j and the `@Backoff(random = true)` in Spring Retry both implement jitter.

---

**Q: What is the correct order of Resilience4j decorators and why?**

A: The recommended order from outermost to innermost is: RateLimiter → Bulkhead → CircuitBreaker → Retry → TimeLimiter → actual call. This order makes semantic sense: first check if the caller is allowed to make calls at all (rate limiter); then check if there's capacity (bulkhead); then check if the circuit allows it (circuit breaker); then attempt with retries (retry); then enforce a deadline per attempt (time limiter). If the circuit breaker were inside the retry, each retry attempt would count against the CB's sliding window independently — which is usually correct. However, TimeLimiter must be inside Retry so that the timeout applies per-attempt, not for the entire retry sequence.

---

**Q: How would you implement a fallback strategy for a product detail page where some components are optional?**

A: Use `Mono.zip` with per-component fallbacks so partial failures return a degraded but functional page. Mandatory components (like the product itself) get `onErrorMap` to a domain exception that bubbles up and returns a proper error page. Optional components (reviews, recommendations, related items) get `onErrorReturn(emptyList())` so a failure returns an empty section rather than failing the whole page. For the product data specifically, layer cache fallback after the DB call: `productDb.findById(id).onErrorResume(ex -> cache.get(id)).onErrorResume(ex -> Mono.just(Product.placeholder()))`. This way the page degrades gracefully — worst case, the user sees a skeletal product page instead of an error.

---

**Q: What is the difference between a circuit breaker and a rate limiter?**

A: A circuit breaker is **reactive** — it responds to observed failure rates and prevents calls to an unhealthy downstream. It opens after detecting problems and recovers automatically when the downstream recovers. A rate limiter is **proactive** — it enforces a maximum throughput regardless of the downstream's health, protecting both the downstream (from overload) and your service (from quota exhaustion with paid external APIs). They solve different problems: the circuit breaker handles _downstream failures_; the rate limiter handles _throughput control_. In production, you typically apply the rate limiter first (don't send more than the downstream can handle) and the circuit breaker second (stop sending if what does get through starts failing).

---

## 9. Key Takeaways

- **Circuit breaker**: detect failure rate → open → fast fail → probe → recover. Use COUNT_BASED window for predictable traffic, TIME_BASED for variable.
- **Retry with backoff + jitter**: only retry transient errors, never client errors (4xx). Jitter is mandatory in distributed systems to prevent thundering herd.
- **Timeouts**: set at every network boundary — connection, read, and pipeline timeout. Outer timeouts alone don't protect inner thread pools.
- **Bulkhead**: semaphore for reactive code, thread pool for blocking code. Limits blast radius so one slow dependency can't consume all resources.
- **Rate limiter**: protects downstream from overload and enforces quotas. Always return `Retry-After` headers with 429 responses.
- **Fallback**: distinguish mandatory from optional components — return degraded partial responses rather than total failures where possible.
- **Decorator order**: RateLimiter → Bulkhead → CircuitBreaker → Retry → TimeLimiter. TimeLimiter per-attempt (inside Retry), not per-sequence (outside).
- **Hystrix is dead** — migrate to Resilience4j. If maintaining Hystrix code, understand the mapping: thread isolation → thread pool bulkhead, semaphore isolation → semaphore bulkhead.
