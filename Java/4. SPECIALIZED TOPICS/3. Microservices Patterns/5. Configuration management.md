# Configuration Management

## Overview

Configuration management in microservices solves a critical operational problem: how do you manage configuration for dozens of services across multiple environments (dev, staging, prod) without hardcoding values, duplicating config, or redeploying services for every change?

**The core goals:**

- Centralize configuration — one source of truth per environment
- Externalize secrets — no credentials in code or config files
- Enable dynamic updates — change config without restarting services
- Audit configuration changes — who changed what, when

**The 12-Factor App** mandates: store config in the environment, never in code.

## 1. Spring Cloud Config

Spring Cloud Config provides a **centralized HTTP-based configuration server** backed by a Git repository (or filesystem, Vault, JDBC). Services fetch their configuration at startup from the config server using their application name and active profile.

### 1.1 Config Server Setup

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-config-server</artifactId>
</dependency>
```

```java
@SpringBootApplication
@EnableConfigServer
public class ConfigServerApplication {
    public static void main(String[] args) {
        SpringApplication.run(ConfigServerApplication.class, args);
    }
}
```

```yaml
# application.yml — Config Server
server:
  port: 8888

spring:
  application:
    name: config-server
  cloud:
    config:
      server:
        git:
          uri: https://github.com/myorg/config-repo
          default-label: main             # Git branch
          search-paths:
            - '{application}'             # look in /{service-name}/ subdirectory
            - '{application}/{profile}'
          clone-on-start: true            # fail fast if repo unreachable
          force-pull: true                # always pull latest from remote
          timeout: 10                     # Git clone/fetch timeout (seconds)
          # For private repos:
          username: ${GIT_USERNAME}
          password: ${GIT_TOKEN}         # or use SSH key

        # Encryption of property values
        encrypt:
          enabled: true

management:
  endpoints:
    web:
      exposure:
        include: health, refresh, bus-refresh

# Secure the config server — clients must authenticate
spring:
  security:
    user:
      name: ${CONFIG_SERVER_USER}
      password: ${CONFIG_SERVER_PASSWORD}
```

### 1.2 Config Repository Structure

```
config-repo/                          ← Git repository root
├── application.yml                   ← shared by ALL services
├── application-production.yml        ← shared by ALL services in production
│
├── order-service/
│   ├── order-service.yml             ← order-service, all profiles
│   ├── order-service-dev.yml         ← order-service, dev profile
│   ├── order-service-staging.yml     ← order-service, staging profile
│   └── order-service-production.yml  ← order-service, production profile
│
├── payment-service/
│   ├── payment-service.yml
│   └── payment-service-production.yml
│
└── inventory-service/
    └── inventory-service.yml
```

**Property resolution order (highest to lowest priority):**

```
1. {application}-{profile}.yml       (order-service-production.yml)
2. {application}.yml                 (order-service.yml)
3. application-{profile}.yml         (application-production.yml)
4. application.yml                   (shared defaults)
```

### 1.3 Config Client Setup (any microservice)

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-config</artifactId>
</dependency>
```

```yaml
# bootstrap.yml (or application.yml with spring.config.import in Spring Boot 2.4+)
spring:
  application:
    name: order-service            # determines which config file to fetch
  profiles:
    active: ${SPRING_PROFILES_ACTIVE:dev}

    config:
      uri: http://config-server:8888

      password: ${CONFIG_SERVER_PASSWORD}
      label: main                  # Git branch
      fail-fast: true              # fail startup if config server unreachable
      retry:
        initial-interval: 1000     # 1s before first retry
        multiplier: 1.5
        max-interval: 5000         # cap at 5s
        max-attempts: 6
```

**Accessing configuration in code:**

```java
// Option 1: @Value — simple property injection
@RestController
public class OrderController {

    @Value("${order.max-items-per-order:50}")
    private int maxItemsPerOrder;

    @Value("${order.processing-timeout-seconds}")
    private int processingTimeoutSeconds;
}

// Option 2: @ConfigurationProperties — type-safe, structured (preferred)
@ConfigurationProperties(prefix = "order")
@Component
@Data
@Validated
public class OrderProperties {

    @Min(1) @Max(200)
    private int maxItemsPerOrder = 50;

    @NotNull
    @DurationUnit(ChronoUnit.SECONDS)
    private Duration processingTimeout = Duration.ofSeconds(30);

    private RetryConfig retry = new RetryConfig();

    @Data
    public static class RetryConfig {
        private int maxAttempts = 3;
        private Duration backoff = Duration.ofMillis(500);
    }
}

// Usage:
@Service
@RequiredArgsConstructor
public class OrderService {
    private final OrderProperties orderProperties;

    public void processOrder(Order order) {
        if (order.getItems().size() > orderProperties.getMaxItemsPerOrder()) {
            throw new OrderTooLargeException(orderProperties.getMaxItemsPerOrder());
        }
    }
}
```

## 2. Consul KV

HashiCorp Consul provides a **key-value store** built into the same tool as service discovery. Unlike Spring Cloud Config's Git-backed model, Consul KV stores config directly in Consul's distributed database — changes are immediately visible to all connected clients.

### 2.1 Consul KV Structure

```
Key structure: {prefix}/{application}/{profile}/key

Example:
  config/application/server.port              ← shared all services
  config/order-service/spring.datasource.url  ← order-service all profiles
  config/order-service,production/spring.datasource.password  ← prod only
```

**Loading values via Consul CLI:**

```bash
# Write config values
consul kv put config/order-service/order.max-items-per-order "50"
consul kv put config/order-service/order.processing-timeout-seconds "30"

# Production-specific
consul kv put config/order-service,production/spring.datasource.url \
  "jdbc:postgresql://prod-db:5432/orders"

# Shared config
consul kv put config/application/spring.jackson.default-property-inclusion "non_null"

# Read
consul kv get config/order-service/order.max-items-per-order

# Watch for changes (blocking query — returns when key changes)
consul watch -type keyprefix -prefix config/order-service/ \
  'cat | jq .'
```

### 2.2 Spring Cloud Consul Config Client

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-consul-config</artifactId>
</dependency>
```

```yaml
# bootstrap.yml
spring:
  application:
    name: order-service
  cloud:
    consul:
      host: ${CONSUL_HOST:localhost}
      port: ${CONSUL_PORT:8500}
      config:
        enabled: true
        prefix: config                  # root key prefix
        default-context: application    # shared config key name
        data-key: data                  # if storing as YAML blob under one key
        format: YAML                    # or PROPERTIES, KEY_VALUE
        watch:
          enabled: true                 # watch for changes (dynamic refresh)
          delay: 1000                   # check every 1s
          wait-time: 55                 # blocking query wait time (seconds)
      discovery:
        enabled: true
        register: true
```

### 2.3 Storing YAML blobs vs individual keys

```bash
# Option A: Individual keys (verbose but easy to manage via CLI)
consul kv put config/order-service/spring.datasource.url "jdbc:postgresql://..."
consul kv put config/order-service/spring.datasource.username "orderuser"
consul kv put config/order-service/order.max-items-per-order "50"

# Option B: YAML blob under a single key (easier for complex configs)
consul kv put config/order-service/data "
spring:
  datasource:
    url: jdbc:postgresql://db:5432/orders
    username: orderuser
order:
  max-items-per-order: 50
  processing-timeout-seconds: 30
"
```

```yaml
# application.yml — use format: YAML with data-key: data for blob approach
spring:
  cloud:
    consul:
      config:
        format: YAML
        data-key: data   # reads the YAML blob stored at config/order-service/data
```

### 2.4 Consul ACL — access control for config

```hcl
# Consul ACL policy — order-service can only read its own config
policy "order-service-config" {
  key_prefix "config/order-service" {
    policy = "read"
  }
  key_prefix "config/application" {
    policy = "read"
  }
  # Cannot read other services' config
}
```

### 2.5 Spring Cloud Config vs Consul KV

||Spring Cloud Config|Consul KV|
|---|---|---|
|Storage backend|Git (primary)|Consul distributed DB|
|Change propagation|Pull (client must refresh)|Push (watch/blocking query)|
|Version history|Full Git history|No built-in versioning|
|Secret handling|Limited (encrypt + Vault)|Separate (Vault)|
|UI|None (Git UI)|Consul UI + API|
|Latency to change|Seconds (refresh cycle)|Milliseconds (watch)|
|Existing Consul usage|Separate tool needed|Already integrated|
|Best for|Git-workflow teams, audit history|Already using Consul, fast propagation|

## 4. Dynamic Configuration Updates

The goal: change configuration values and have running services pick them up **without restarting or redeploying**.

### 4.1 @RefreshScope — the core mechanism

```java
// @RefreshScope re-instantiates the bean when /actuator/refresh is called
@Component
@RefreshScope   // ← magic annotation
@ConfigurationProperties(prefix = "order")
@Data
public class OrderProperties {
    private int maxItemsPerOrder;
    private Duration processingTimeout;
    private boolean newFeatureEnabled;
}

// @Value with @RefreshScope
@RestController
@RefreshScope
public class FeatureController {

    @Value("${feature.new-checkout:false}")
    private boolean newCheckoutEnabled;

    @GetMapping("/checkout")
    public ResponseEntity<?> checkout(@RequestBody CartDto cart) {
        if (newCheckoutEnabled) {
            return newCheckoutService.process(cart);
        }
        return legacyCheckoutService.process(cart);
    }
}
```

### 4.2 Triggering refresh — Spring Cloud Bus

Manually calling `/actuator/refresh` on each instance doesn't scale. **Spring Cloud Bus** broadcasts a refresh event via a message broker (Kafka or RabbitMQ) to all instances at once.

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-bus-kafka</artifactId>
</dependency>
```

```yaml
# application.yml — all microservices
spring:
  cloud:
    bus:
      enabled: true
    stream:
      kafka:
        binder:
          brokers: kafka:9092

management:
  endpoints:
    web:
      exposure:
        include: bus-refresh, refresh, health
```

```bash
# Push new config to Git repo, then trigger refresh on ONE instance
# Spring Cloud Bus broadcasts to ALL instances

# Refresh ALL services
POST http://any-service:8080/actuator/bus-refresh

# Refresh only order-service instances
POST http://any-service:8080/actuator/bus-refresh?destination=order-service:**

# Refresh a specific instance
POST http://any-service:8080/actuator/bus-refresh?destination=order-service:8080
```

**With Git webhook — fully automated refresh:**

```yaml
# Config server: expose /monitor endpoint (Spring Cloud Config Monitor)
spring:
  cloud:
    config:
      server:
        git:
          uri: https://github.com/myorg/config-repo

# GitHub webhook: POST /monitor on every push to config-repo
# Config server broadcasts Spring Cloud Bus refresh event to all services
# All @RefreshScope beans re-initialized — no manual trigger needed
```

### 4.3 Consul watch — native push-based refresh

With Consul KV, Spring automatically watches for key changes:

```yaml
spring:
  cloud:
    consul:
      config:
        watch:
          enabled: true      # Consul watch — no manual refresh needed
          delay: 1000        # polling interval for watch
```

```java
// When Consul KV changes, @RefreshScope beans are automatically refreshed
// No /actuator/refresh needed — Consul watch triggers it

// Listen for refresh events programmatically
@Component
@Slf4j
public class ConfigRefreshListener {

    @EventListener(RefreshScopeRefreshedEvent.class)
    public void onRefresh(RefreshScopeRefreshedEvent event) {
        log.info("Configuration refreshed: {}", event.getName());
    }

    @EventListener(EnvironmentChangeEvent.class)
    public void onEnvironmentChange(EnvironmentChangeEvent event) {
        log.info("Properties changed: {}", event.getKeys());
    }
}
```

### 4.4 What can and cannot be refreshed

```java
// ✓ CAN be refreshed with @RefreshScope:
@Component @RefreshScope
class FeatureFlags { @Value("${feature.x}") boolean enabled; }

// ✓ @ConfigurationProperties is refresh-aware by default in Spring Boot 2.4+
@ConfigurationProperties(prefix = "order")
class OrderProperties { int maxItems; }

// ✗ CANNOT be refreshed (created before ApplicationContext fully started):
//    - DataSource (connection pool) — changing DB URL requires restart
//    - KafkaListenerContainerFactory — broker address requires restart
//    - @Bean methods in @Configuration without @RefreshScope

// ✗ Dangerous to refresh — causes brief bean inconsistency:
//    - Singleton beans holding state (caches, counters)
//    - Beans used in constructor injection of non-refreshable beans

// Safe refresh candidates:
//    ✓ Feature flags
//    ✓ Timeout/threshold values
//    ✓ Rate limits
//    ✓ Message templates
//    ✓ Toggle third-party integrations
```

### 4.5 Feature flags with dynamic refresh

```java
// Feature flag service with dynamic refresh
@Component
@RefreshScope
@ConfigurationProperties(prefix = "feature")
@Data
public class FeatureFlags {
    private boolean newPaymentGateway = false;
    private boolean betaCheckout = false;
    private double betaCheckoutPercentage = 0.0;  // canary — 0-100% rollout
    private Set<String> betaUserIds = new HashSet<>();
}

@Service
@RequiredArgsConstructor
public class FeatureFlagService {

    private final FeatureFlags flags;

    public boolean isNewPaymentEnabled(String userId) {
        if (!flags.isNewPaymentGateway()) return false;

        // Percentage-based rollout
        if (flags.getNewPaymentPercentage() > 0) {
            double hash = Math.abs(userId.hashCode() % 100);
            return hash < flags.getNewPaymentPercentage();
        }

        // User whitelist
        return flags.getBetaUserIds().contains(userId);
    }
}
```

## 5. Secrets Management (Vault)

HashiCorp Vault is the standard for secrets management in microservices. It provides **dynamic secrets** (credentials generated on-demand, automatically expiring), **encryption as a service**, and fine-grained access policies.

### 5.1 Why Vault over environment variables

```
Environment Variables:
  ✗ Stored in plaintext in container env
  ✗ Visible in ps aux, /proc, container inspect
  ✗ No rotation without redeploy
  ✗ No audit log of who accessed what
  ✗ Static — once set, until next deploy

Vault:
  ✓ Encrypted at rest and in transit
  ✓ Dynamic secrets — rotate automatically
  ✓ Audit log — every read is logged
  ✓ Lease-based — secrets expire, forcing rotation
  ✓ Fine-grained policies — service A can't read service B's secrets
  ✓ Revocation — instantly revoke a compromised secret
```

### 5.2 Vault setup and Spring Boot integration

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-vault-config</artifactId>
</dependency>
```

```yaml
# bootstrap.yml — Vault config (must be in bootstrap, not application)
spring:
  application:
    name: order-service
  cloud:
    vault:
      host: vault.internal
      port: 8200
      scheme: https
      authentication: KUBERNETES       # K8s service account auth
      kubernetes:
        role: order-service            # Vault role bound to K8s service account
        kubernetes-path: auth/kubernetes
        service-account-token-file: /var/run/secrets/kubernetes.io/serviceaccount/token
      kv:
        enabled: true
        backend: secret                # KV secrets engine mount path
        default-context: order-service # reads secret/order-service
        application-name: order-service
      database:
        enabled: true                  # dynamic DB credentials
        role: order-service-role       # Vault role that generates credentials
        backend: database
```

### 5.3 Vault KV secrets engine — static secrets

```bash
# Store secrets in Vault KV
vault kv put secret/order-service \
  spring.datasource.username=orderuser \
  spring.datasource.password=supersecret \
  stripe.api-key=sk_live_... \
  jwt.secret=my-jwt-signing-key

# Read (for debugging — in production, services read automatically)
vault kv get secret/order-service

# Versioned KV (KV v2)
vault kv put secret/data/order-service/production \
  spring.datasource.password=newpassword

vault kv get -version=2 secret/data/order-service/production
```

**Vault KV values are injected as Spring properties automatically:**

```yaml
# After spring-cloud-vault reads secret/order-service,
# these become available as Spring properties:

spring:
  datasource:
    username: orderuser        # from Vault KV
    password: supersecret      # from Vault KV
stripe:
  api-key: sk_live_...         # from Vault KV
jwt:
  secret: my-jwt-signing-key   # from Vault KV
```

### 5.4 Dynamic database credentials — the killer feature

Vault can generate temporary database credentials on-demand. The service never stores a long-lived password.

```bash
# Configure Vault database secrets engine
vault secrets enable database

vault write database/config/orders-db \
    plugin_name=postgresql-database-plugin \
    connection_url="postgresql://{{username}}:{{password}}@postgres:5432/orders" \
    allowed_roles="order-service-role" \
    username="vault_root" \
    password="rootpassword"

# Define a role — SQL to create the temporary user
vault write database/roles/order-service-role \
    db_name=orders-db \
    creation_statements="
        CREATE ROLE \"{{name}}\" WITH LOGIN PASSWORD '{{password}}' VALID UNTIL '{{expiration}}';
        GRANT SELECT, INSERT, UPDATE ON ALL TABLES IN SCHEMA public TO \"{{name}}\";
    " \
    default_ttl="1h" \     # credential expires after 1 hour
    max_ttl="24h"
```

**Spring Boot with dynamic credentials:**

```yaml
# bootstrap.yml
spring:
  cloud:
    vault:
      database:
        enabled: true
        role: order-service-role
        backend: database
```

```java
// Spring Cloud Vault automatically:
// 1. Requests credentials from Vault on startup
// 2. Configures DataSource with the temporary username/password
// 3. Renews the lease before it expires
// 4. Re-requests new credentials if renewal fails

// Your code doesn't change — just use the DataSource normally
@Service
@RequiredArgsConstructor
public class OrderRepository {
    private final JdbcTemplate jdbcTemplate; // connected with dynamic credentials
}
```

### 5.5 Vault policies — least privilege

```hcl
# order-service can only read its own secrets
path "secret/data/order-service/*" {
  capabilities = ["read"]
}

# order-service can generate database credentials for its role only
path "database/creds/order-service-role" {
  capabilities = ["read"]
}

# order-service cannot read payment-service secrets
path "secret/data/payment-service/*" {
  capabilities = []    # denied
}
```

### 5.6 Kubernetes authentication with Vault

```yaml
# Kubernetes Vault auth — no hardcoded Vault token in pods
# Pod uses its Kubernetes service account JWT to authenticate with Vault

# Step 1: Enable K8s auth in Vault
vault auth enable kubernetes

vault write auth/kubernetes/config \
    kubernetes_host="https://kubernetes.default.svc" \
    kubernetes_ca_cert=@/var/run/secrets/kubernetes.io/serviceaccount/ca.crt

# Step 2: Bind Vault role to K8s service account
```

## 6. Configuration Best Practices

### 6.1 Configuration precedence in Spring Boot

```
(Highest priority)
1. Command-line arguments:          --server.port=9090
2. SPRING_APPLICATION_JSON env var
3. OS environment variables:        SERVER_PORT=9090
4. application-{profile}.properties/yml (external)
5. application.properties/yml (external)
6. @PropertySource annotations
7. application-{profile}.properties/yml (classpath — packaged)
8. application.properties/yml (classpath — packaged)
9. @SpringBootApplication default values
(Lowest priority)
```

### 6.2 Naming conventions

```yaml
# Spring Boot relaxed binding — all equivalent:
spring.datasource.maximum-pool-size: 20  # kebab-case (recommended)
spring.datasource.maximumPoolSize: 20    # camelCase
SPRING_DATASOURCE_MAXIMUM_POOL_SIZE: 20  # env var (uppercase + underscores)
spring.datasource.maximum_pool_size: 20  # underscore

# Use kebab-case in yml files — it's the Spring standard
# Use UPPER_SNAKE_CASE for environment variables and Kubernetes

# Group related properties under a prefix:
order:
  service:
    max-items-per-order: 50
    processing-timeout: 30s
    retry:
      max-attempts: 3
      backoff: 500ms
```

### 6.3 Never in config files

```yaml
# NEVER commit these to Git (even private repos):
# ✗ Passwords, API keys, tokens
# ✗ Database credentials
# ✗ Private keys, certificates

# Instead:
spring:
  datasource:
    password: ${DB_PASSWORD}           # environment variable placeholder
    # or: resolved by Vault at startup
    # or: injected by Kubernetes Secret
```

### 6.4 Config validation at startup

```java
// Validate configuration at startup — fail fast with a clear error
@ConfigurationProperties(prefix = "payment")
@Validated
@Data
public class PaymentProperties {

    @NotBlank(message = "payment.api-url must not be blank")
    private String apiUrl;

    @NotBlank
    @Pattern(regexp = "sk_.*", message = "payment.api-key must start with 'sk_'")
    private String apiKey;

    @Min(value = 1000, message = "payment.timeout must be at least 1000ms")
    @Max(value = 30000)
    private int timeoutMillis = 5000;
}

// Fail fast — application won't start with invalid config
// Error at startup is better than NPE at runtime
```

## 7. Interview Questions

**Q: How does Spring Cloud Config propagate configuration changes to running services without restarting them?**

A: Spring Cloud Config itself only stores configuration — it doesn't push changes. The propagation mechanism requires two components. First, `@RefreshScope` on beans marks them for re-instantiation when a refresh is triggered. Second, Spring Cloud Bus broadcasts a refresh event over a message broker (Kafka or RabbitMQ). When configuration changes in Git, a webhook hits the config server's `/monitor` endpoint, which publishes a `RefreshRemoteApplicationEvent` to the bus. All subscribed service instances receive it, call `/actuator/refresh` internally, and re-instantiate their `@RefreshScope` beans with the new values. Without Spring Cloud Bus, you'd need to manually POST to `/actuator/refresh` on every instance.

---

**Q: What is the difference between Spring Cloud Config and Consul KV for configuration management?**

A: Spring Cloud Config is backed by Git, giving you a full version history of every configuration change with commit messages and diffs — great for auditing and rollback. However, changes require a Git push followed by triggering a refresh. Consul KV stores config directly in Consul's distributed database, so changes are propagated to watching clients in milliseconds via blocking queries, with no refresh trigger needed. The trade-off is that Consul KV has no built-in version history. Choose Spring Cloud Config when you value audit trail and Git-based workflow; choose Consul KV when you're already using Consul for service discovery and need near-real-time configuration propagation.

---

**Q: What are dynamic database credentials in Vault and why are they better than static ones?**

A: Vault's dynamic credentials feature generates a unique username and password for each service instance on startup, with a configurable TTL (e.g., 1 hour). When the TTL expires, Vault automatically renews the lease (Spring Cloud Vault handles this). If a service is compromised, you revoke its specific credentials instantly — other services are unaffected. With static credentials, a leaked password requires rotating it everywhere simultaneously. Dynamic credentials also create a clear audit trail — every DB connection is tied to a specific Vault-issued credential, so you can trace exactly which service made which query.

---

**Q: What is the priority order for Spring Boot configuration sources and why does it matter?**

A: Spring Boot resolves properties from highest to lowest priority: command-line args, then OS environment variables, then external property files, then packaged application.yml, then `@PropertySource` annotations. This matters because it defines how you override config in different environments without changing the packaged artifact — a Kubernetes ConfigMap injects environment variables that override the packaged application.yml without repackaging the JAR. Vault-injected properties (via Spring Cloud Vault) are also high-priority, overriding anything in application.yml. Understanding this order is critical when debugging "why is this value being used instead of the one I set?"

---

**Q: What should you never store in a config file committed to Git, and what should you do instead?**

A: Never commit passwords, API keys, database credentials, private keys, JWT signing secrets, or any other credentials — even in private repositories. History is forever in Git, and access control on repos is rarely as strict as needed. Instead: use placeholder syntax (`${DB_PASSWORD}`) and inject via environment variables (in K8s: Secret + `envFrom`); use Spring Cloud Vault to fetch secrets at startup with fine-grained access policies per service; or use the Vault Agent sidecar to inject secrets as files. The distinction is: config files in Git hold structure and non-sensitive values; sensitive values come from a secrets manager at runtime.

---

**Q: What is @RefreshScope and what are its limitations?**

A: `@RefreshScope` marks a Spring bean to be re-instantiated when a refresh event is received (via `/actuator/refresh` or Spring Cloud Bus). On refresh, the bean proxy is cleared and rebuilt with the latest property values from the environment. Limitations: it cannot refresh infrastructure beans like `DataSource` or `KafkaListenerContainerFactory` that are created very early in the application lifecycle and are depended upon by other non-refreshable beans — changing the database URL or Kafka broker address requires a restart. Also, `@RefreshScope` creates a proxy, which adds overhead and can cause subtle issues with singleton beans that cache references to the refreshed bean. It should be used selectively on beans that genuinely need dynamic updates — typically feature flags, timeout values, and threshold settings.

---

## 8. Key Takeaways

- **Spring Cloud Config**: Git-backed, full audit history, requires Bus for broadcast refresh — ideal for teams with a Git-based ops workflow
- **Consul KV**: real-time propagation via blocking queries, integrated with Consul service discovery — ideal if you're already running Consul
- **Environment-specific configs**: use Spring profiles (`application-{profile}.yml`), activated via `SPRING_PROFILES_ACTIVE`; inject K8s-specific values via ConfigMap/Secret `envFrom`
- **@RefreshScope + Spring Cloud Bus**: change config in Git → webhook → Bus broadcast → all instances refresh without restart — applies to feature flags, timeouts, thresholds (not DataSource or Kafka)
- **Vault**: never store secrets in config files; Vault provides dynamic credentials (auto-rotating, auto-expiring), fine-grained policies, and full audit log; Kubernetes auth eliminates hardcoded Vault tokens
- **Fail fast**: validate `@ConfigurationProperties` with `@Validated` at startup — a startup failure with a clear message beats a runtime NPE
