# Service Communication

## Overview

In a microservices architecture, services are isolated processes that must communicate over a network. Every architectural decision about _how_ services talk to each other has cascading consequences on latency, coupling, resilience, and operational complexity. Getting service communication right is one of the most impactful decisions in microservices design.

**Core tension**: Synchronous calls are simple to reason about but create tight temporal coupling — if Service B is slow, Service A is slow. Asynchronous messaging decouples services but adds complexity in consistency, ordering, and error handling.

---

## 1. REST vs gRPC vs GraphQL

### 1.1 REST — Representational State Transfer

REST is the default choice for service-to-service HTTP communication. It's resource-oriented, uses standard HTTP verbs, and is universally understood.

**Characteristics:**

- Text-based (JSON/XML) — human-readable, easy to debug
- HTTP/1.1 or HTTP/2
- Stateless — each request contains all necessary information
- Cacheable via HTTP caching headers
- Widely supported by every language, framework, and tool

```java
// Spring Boot REST controller (producer side)
@RestController
@RequestMapping("/api/orders")
@RequiredArgsConstructor
public class OrderController {

    private final OrderService orderService;

    @GetMapping("/{id}")
    public ResponseEntity<OrderDto> getOrder(@PathVariable Long id) {
        return orderService.findById(id)
            .map(ResponseEntity::ok)
            .orElse(ResponseEntity.notFound().build());
    }

    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public OrderDto createOrder(@RequestBody @Valid CreateOrderRequest request) {
        return orderService.create(request);
    }

    @GetMapping("/users/{userId}")
    public List<OrderDto> getOrdersByUser(
            @PathVariable Long userId,
            @RequestParam(defaultValue = "0") int page,
            @RequestParam(defaultValue = "20") int size) {
        return orderService.findByUserId(userId, PageRequest.of(page, size));
    }
}

// Spring Boot REST client (consumer side)
@Component
@RequiredArgsConstructor
public class OrderClient {

    private final RestTemplate restTemplate; // or WebClient for reactive

    public OrderDto getOrder(Long id) {
        return restTemplate.getForObject(
            "http://order-service/api/orders/{id}",
            OrderDto.class, id
        );
    }

    // WebClient for non-blocking calls
    private final WebClient webClient;

    public Mono<OrderDto> getOrderReactive(Long id) {
        return webClient.get()
            .uri("/api/orders/{id}", id)
            .retrieve()
            .onStatus(HttpStatus::is4xxClientError,
                res -> Mono.error(new OrderNotFoundException(id)))
            .bodyToMono(OrderDto.class);
    }
}
```

**OpenAPI / Swagger — contract-first REST:**

```yaml

  title: Order Service API
  version: "1.0"
paths:
  /api/orders/{id}:

```

**REST strengths:**

- Universal client support (browsers, mobile, any language)
- Human-readable for debugging
- HTTP caching, CDN-friendly
- Simple to understand and document
- Great tooling (Swagger, Postman, curl)

**REST weaknesses:**

- Verbose (JSON overhead per request)
- No built-in streaming (need SSE/WebSocket separately)
- No compile-time contract enforcement
- HTTP/1.1 head-of-line blocking (mitigated by HTTP/2)
- Over-fetching / under-fetching data

### 1.2 gRPC — Google Remote Procedure Call

gRPC uses **Protocol Buffers** (protobuf) as its IDL and serialization format, and runs over **HTTP/2**. It generates strongly-typed client and server stubs in any language from a `.proto` contract file.

**Characteristics:**

- Binary serialization (protobuf) — ~5-10x smaller payload than JSON
- HTTP/2 — multiplexing, header compression, full-duplex streaming
- Strongly-typed contracts in `.proto` files
- Code generation in 10+ languages from one contract
- Four communication modes: unary, server streaming, client streaming, bidirectional streaming

**Protocol Buffer definition:**

```protobuf
// order.proto
syntax = "proto3";
package com.example.order;

option java_package = "com.example.order.grpc";
option java_multiple_files = true;

service OrderService {
  // Unary — request/response
  rpc GetOrder(GetOrderRequest) returns (OrderResponse);

  // Server streaming — one request, stream of responses
  rpc GetOrdersByUser(GetOrdersByUserRequest) returns (stream OrderResponse);

  // Client streaming — stream of requests, one response
  rpc CreateOrders(stream CreateOrderRequest) returns (BatchCreateResponse);

  // Bidirectional streaming — full duplex
  rpc OrderUpdates(stream OrderUpdateRequest) returns (stream OrderUpdateResponse);
}

message GetOrderRequest {
  int64 id = 1;
}

message OrderResponse {
  int64 id = 1;
  string status = 2;
  double total_amount = 3;
  repeated OrderItem items = 4;
  google.protobuf.Timestamp created_at = 5;
}

message OrderItem {
  int64 product_id = 1;
  int32 quantity = 2;
  double price = 3;
}
```

**Spring Boot gRPC server (using grpc-spring-boot-starter):**

```xml
<dependency>
    <groupId>net.devh</groupId>
    <artifactId>grpc-server-spring-boot-starter</artifactId>
    <version>2.15.0.RELEASE</version>
</dependency>
```

```java
@GrpcService
@RequiredArgsConstructor
public class OrderGrpcService extends OrderServiceGrpc.OrderServiceImplBase {

    private final OrderService orderService;

    @Override
    public void getOrder(GetOrderRequest request, StreamObserver<OrderResponse> observer) {
        try {
            Order order = orderService.findById(request.getId())
                .orElseThrow(() -> new OrderNotFoundException(request.getId()));

            observer.onNext(toProto(order));
            observer.onCompleted();
        } catch (OrderNotFoundException ex) {
            observer.onError(Status.NOT_FOUND
                .withDescription(ex.getMessage())
                .asRuntimeException());
        }
    }

    @Override
    public void getOrdersByUser(GetOrdersByUserRequest request,
                                StreamObserver<OrderResponse> observer) {
        // Server streaming — send multiple responses
        orderService.findByUserId(request.getUserId()).forEach(order -> {
            if (!observer.isReady()) return; // basic flow control
            observer.onNext(toProto(order));
        });
        observer.onCompleted();
    }
}
```

**Spring Boot gRPC client:**

```xml
<dependency>
    <groupId>net.devh</groupId>
    <artifactId>grpc-client-spring-boot-starter</artifactId>
    <version>2.15.0.RELEASE</version>
</dependency>
```

```java
@Component
public class OrderGrpcClient {

    @GrpcClient("order-service") // name matches application.yml config
    private OrderServiceGrpc.OrderServiceBlockingStub blockingStub;

    @GrpcClient("order-service")
    private OrderServiceGrpc.OrderServiceStub asyncStub;

    // Unary blocking call
    public OrderResponse getOrder(long id) {
        try {
            return blockingStub.getOrder(
                GetOrderRequest.newBuilder().setId(id).build()
            );
        } catch (StatusRuntimeException ex) {
            if (ex.getStatus().getCode() == Status.Code.NOT_FOUND) {
                throw new OrderNotFoundException(id);
            }
            throw ex;
        }
    }

    // Server streaming — reactive with Reactor
    public Flux<OrderResponse> getOrdersByUser(long userId) {
        return Flux.create(sink -> {
            asyncStub.getOrdersByUser(
                GetOrdersByUserRequest.newBuilder().setUserId(userId).build(),
                new StreamObserver<OrderResponse>() {
                    public void onNext(OrderResponse value) { sink.next(value); }
                    public void onError(Throwable t) { sink.error(t); }
                    public void onCompleted() { sink.complete(); }
                }
            );
        });
    }
}
```

```yaml
# application.yml — gRPC client config
grpc:
  client:
    order-service:
      address: discovery:///order-service  # with service discovery
      negotiation-type: plaintext           # or TLS
```

**gRPC strengths:**

- 5-10x smaller payload, lower latency for high-volume internal traffic
- Strongly-typed contracts — compile-time safety, auto-generated clients
- Native streaming support (4 modes)
- Excellent for polyglot service-to-service communication
- Built-in deadlines, cancellation, flow control

**gRPC weaknesses:**

- Not browser-native (needs gRPC-Web proxy for browser clients)
- Binary format — harder to debug (need grpcurl, BloomRPC)
- Schema evolution requires care (field numbering, deprecation)
- More setup than REST

### 1.3 GraphQL

GraphQL is a query language for APIs where the **client specifies exactly what data it needs**. Unlike REST's server-defined response shape, GraphQL lets clients ask for precisely the fields they want.

**Characteristics:**

- Single endpoint (`POST /graphql`)
- Client-controlled response shape — no over-fetching or under-fetching
- Strongly-typed schema
- Supports queries (read), mutations (write), and subscriptions (real-time)
- Schema introspection — clients can discover the API

**Spring Boot GraphQL (spring-boot-starter-graphql):**

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-graphql</artifactId>
</dependency>
```

```graphql
# schema.graphqls
type Query {
  order(id: ID!): Order
  ordersByUser(userId: ID!, limit: Int = 20): [Order!]!
}

type Mutation {
  createOrder(input: CreateOrderInput!): Order!
  cancelOrder(id: ID!): Order!
}

type Subscription {
  orderStatusChanged(orderId: ID!): Order!
}

type Order {
  id: ID!
  status: OrderStatus!
  totalAmount: Float!
  items: [OrderItem!]!
  user: User!          # nested type — resolved separately
  createdAt: String!
}

type OrderItem {
  productId: ID!
  quantity: Int!
  price: Float!
  product: Product!   # nested type from product service
}

enum OrderStatus { PENDING CONFIRMED SHIPPED DELIVERED CANCELLED }

input CreateOrderInput {
  items: [OrderItemInput!]!
}
```

```java
@Controller
@RequiredArgsConstructor
public class OrderGraphQLController {

    private final OrderService orderService;

    @QueryMapping
    public Order order(@Argument Long id) {
        return orderService.findById(id)
            .orElseThrow(() -> new OrderNotFoundException(id));
    }

    @QueryMapping
    public List<Order> ordersByUser(@Argument Long userId, @Argument int limit) {
        return orderService.findByUserId(userId, limit);
    }

    @MutationMapping
    public Order createOrder(@Argument CreateOrderInput input) {
        return orderService.create(input);
    }

    // DataLoader pattern — solves N+1 problem
    @SchemaMapping(typeName = "Order", field = "user")
    public CompletableFuture<User> user(Order order, DataLoader<Long, User> userLoader) {
        return userLoader.load(order.getUserId()); // batched — avoids N+1
    }

    @SubscriptionMapping
    public Flux<Order> orderStatusChanged(@Argument Long orderId) {
        return orderService.subscribeToOrderUpdates(orderId);
    }
}

// DataLoader configuration — batches N user lookups into 1 call
@Component
public class UserDataLoaderConfig {

    @Bean
    public BatchLoaderRegistry batchLoaderRegistry(UserService userService) {
        BatchLoaderRegistry registry = new DefaultBatchLoaderRegistry();
        registry.forTypePair(Long.class, User.class)
            .registerBatchLoader((ids, env) ->
                Mono.fromCallable(() -> userService.findAllById(ids)));
        return registry;
    }
}
```

**GraphQL client query:**

```graphql

query GetOrderPage($id: ID!) {
  order(id: $id) {
    id
    status
    totalAmount
    items {
      quantity
      price

        name
        imageUrl
      }

  }

```

**GraphQL strengths:**

- Eliminates over-fetching and under-fetching
- Single endpoint — simpler client integration
- Self-documenting schema with introspection
- Real-time subscriptions built in
- Ideal for BFF pattern and frontend-driven APIs

**GraphQL weaknesses:**

- Complex caching (queries are dynamic — no URL-based cache)
- N+1 query problem requires DataLoader
- Harder to secure (field-level authorization complexity)
- Overkill for simple CRUD APIs
- Performance analysis harder than REST

### 1.4 REST vs gRPC vs GraphQL — Decision Matrix

|Concern|REST|gRPC|GraphQL|
|---|---|---|---|
|Client type|Any (browser, mobile, service)|Service-to-service|Frontend-driven|
|Payload|JSON (verbose)|Protobuf (binary, compact)|JSON (client-defined)|
|Type safety|OpenAPI (optional)|Protobuf (mandatory)|Schema (mandatory)|
|Streaming|SSE / WebSocket (separate)|Native (4 modes)|Subscriptions|
|Caching|HTTP cache (URL-based)|Client-side only|Complex (query hash)|
|Browser support|Native|Needs proxy (gRPC-Web)|Native|
|Debugging|Easy (curl, Postman)|Harder (binary)|GraphiQL / Playground|
|Performance|Good|Excellent|Good|
|Learning curve|Low|Medium|Medium-High|
|Best for|Public APIs, CRUD|High-throughput internal|BFF, flexible clients|

## 2. Synchronous vs Asynchronous Communication

### 2.1 Synchronous — direct call, wait for response

```
Service A ──► Service B ──► response ──► Service A
             (A is blocked until B responds)
```

```java
// Synchronous REST call via WebClient (reactive but still synchronous in terms of dependency)
public Mono<OrderConfirmation> placeOrder(Order order) {
    return inventoryClient.checkStock(order.getItems())  // must complete first
        .flatMap(stockOk -> {
            if (!stockOk) return Mono.error(new OutOfStockException());
            return paymentClient.charge(order.getPayment());  // then this
        })
        .flatMap(payment -> orderRepository.save(order));    // then this
}
```

**Synchronous strengths:**

- Simple mental model — call and get result
- Immediate consistency — caller knows the operation succeeded
- Easy error propagation — exceptions flow back to caller
- Simpler debugging — linear trace

**Synchronous weaknesses:**

- **Temporal coupling** — if B is down, A fails
- **Cascading failures** — a slow B causes slow A, which can cascade
- **Higher latency** — sum of all downstream calls
- Requires circuit breakers, timeouts, retries to be resilient

---

### 2.2 Asynchronous — fire event, continue

```
Service A ──► [Message Broker] ──► Service B (processes later)
             (A continues immediately)
```

```java
// Asynchronous via Kafka — Service A publishes, Service B consumes independently
@Service
@RequiredArgsConstructor
public class OrderService {

    private final KafkaTemplate<String, OrderEvent> kafkaTemplate;
    private final OrderRepository orderRepository;

    public Order placeOrder(CreateOrderRequest request) {
        Order order = orderRepository.save(request.toOrder());

        // Publish event — don't wait for downstream processing
        kafkaTemplate.send("order.placed", order.getId().toString(),
            new OrderPlacedEvent(order.getId(), order.getUserId(), order.getItems()));

        return order; // return immediately — inventory/payment happen asynchronously
    }
}

// Service B (Inventory) reacts to the event
@Component
@RequiredArgsConstructor
public class InventoryEventHandler {

    private final InventoryService inventoryService;

    @KafkaListener(topics = "order.placed", groupId = "inventory-service")
    public void onOrderPlaced(OrderPlacedEvent event) {
        inventoryService.reserve(event.getOrderId(), event.getItems());
    }
}
```

**Asynchronous strengths:**

- **Temporal decoupling** — B can be down; messages queue until it recovers
- **Higher throughput** — no waiting for downstream
- **Natural load leveling** — broker absorbs spikes
- **Independent scaling** — producers and consumers scale separately

**Asynchronous weaknesses:**

- **Eventual consistency** — caller doesn't know immediately if it succeeded
- **Harder debugging** — distributed traces across time
- **Ordering complexity** — messages may arrive out of order
- **Additional infrastructure** — message broker required
- **Idempotency required** — messages may be delivered more than once

---

### 2.3 Choosing Synchronous vs Asynchronous

```
Does the caller need an immediate result?
├── Yes → Synchronous
│    └── Example: User creates order, needs order ID to show confirmation page
└── No → Asynchronous (if latency tolerance allows)
     └── Example: After order placed, send confirmation email, update inventory,
                  notify warehouse — none need to block the HTTP response

Is strict consistency required?
├── Yes → Synchronous (distributed transactions — avoid if possible)
└── No → Asynchronous (eventual consistency is usually fine)

Can the operation tolerate failure in downstream?
├── No → Synchronous with circuit breaker
└── Yes → Asynchronous — downstream retries independently
```

### 2.4 Hybrid — synchronous for user-facing, async for side effects

```java
// Best of both worlds: immediate response + async side effects
@PostMapping("/orders")
public ResponseEntity<OrderDto> createOrder(@RequestBody CreateOrderRequest req) {
    // Synchronous: create order, return ID immediately
    Order order = orderService.create(req); // saves to DB

    // Asynchronous: downstream effects don't block the response
    eventPublisher.publishAsync("order.placed", new OrderPlacedEvent(order));
    // ↑ triggers: inventory reservation, payment, email, analytics — all async

    return ResponseEntity.created(URI.create("/api/orders/" + order.getId()))
        .body(toDto(order));
}
```

## 3. Service Mesh (Istio, Linkerd)

A service mesh is an **infrastructure layer** that handles all service-to-service communication concerns — traffic management, security, observability — without changing application code.

### 3.1 The Problem it Solves

Without a service mesh, every service must implement:

- mTLS between services
- Circuit breaking
- Retries with backoff
- Distributed tracing
- Traffic splitting for canary deployments
- Rate limiting

This logic is duplicated across every service, in every language.

### 3.2 Architecture — Sidecar Proxy

```
┌─────────────────────────────────────────────────────────┐
│  Pod (Service A)                                         │
│  ┌──────────────────┐     ┌──────────────────────────┐  │
│  │  App Container   │◄───►│  Sidecar Proxy (Envoy)   │  │
│  │  (your code)     │     │  (Istio/Linkerd injects)  │  │
│  └──────────────────┘     └──────────────────────────┘  │
│                                     │                    │
└─────────────────────────────────────│────────────────────┘
                                      │ (all traffic routed through proxy)
┌─────────────────────────────────────│────────────────────┐
│  Pod (Service B)                    │                    │
│  ┌──────────────────┐     ┌─────────▼────────────────┐  │
│  │  App Container   │◄───►│  Sidecar Proxy (Envoy)   │  │
│  └──────────────────┘     └──────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
         ▲
         │
┌────────┴────────┐
│  Control Plane  │  (Istiod / Linkerd control plane)
│  - Config dist. │  configures all sidecars
│  - Cert mgmt    │
│  - Discovery    │
└─────────────────┘
```

### 3.3 Istio

Istio is the most feature-rich service mesh, using **Envoy** as its sidecar proxy and providing a comprehensive control plane.

**Key Istio features:**

```yaml
# Traffic management — canary deployment (10% to new version)
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: order-service
spec:
  http:
  - route:
    - destination:
        host: order-service
        subset: v1
      weight: 90
    - destination:
        host: order-service
        subset: v2           # new version gets 10% of traffic
      weight: 10
---
# Retry policy — automatic retries without code changes
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: payment-service
spec:
  http:
  - retries:
      attempts: 3
      perTryTimeout: 2s
      retryOn: 5xx,reset,connect-failure
    timeout: 10s
    route:
    - destination:
        host: payment-service
---
# Circuit breaker — via DestinationRule
apiVersion: networking.istio.io/v1alpha3
kind: DestinationRule
metadata:
  name: inventory-service
spec:
  host: inventory-service
  trafficPolicy:
    outlierDetection:
      consecutive5xxErrors: 5
      interval: 30s
      baseEjectionTime: 30s
      maxEjectionPercent: 50
```

```yaml
# mTLS — enforce mutual TLS between all services
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: production
spec:
  mtls:
    mode: STRICT  # all service-to-service traffic must use mTLS
---
# Authorization policy — service A can only call service B's /api/orders endpoint
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: order-service-policy
spec:
  selector:
    matchLabels:
      app: order-service
  rules:
  - from:
    - source:
        principals: ["cluster.local/ns/default/sa/api-gateway"]
  - to:
    - operation:
        methods: ["GET", "POST"]
        paths: ["/api/orders*"]
```

### 3.4 Linkerd

Linkerd is lighter and simpler than Istio, using a **Rust-based micro-proxy** instead of Envoy. Better suited for teams who want service mesh benefits without Istio's operational complexity.

```yaml
# Linkerd annotation — inject sidecar into a namespace
apiVersion: v1
kind: Namespace
metadata:
  name: production
  annotations:
    linkerd.io/inject: enabled  # all pods in this namespace get sidecar

# Linkerd ServiceProfile — define per-route metrics and retries
apiVersion: linkerd.io/v1alpha2
kind: ServiceProfile
metadata:
  name: order-service.production.svc.cluster.local
  namespace: production
spec:
  routes:
  - name: GET /api/orders/{id}
    condition:
      method: GET
      pathRegex: /api/orders/[0-9]+
    isRetryable: true        # enable retries for this route
    timeout: 5s
  - name: POST /api/orders
    condition:
      method: POST
      pathRegex: /api/orders
    isRetryable: false       # don't retry POST (not idempotent)
```

### 3.5 Istio vs Linkerd

||Istio|Linkerd|
|---|---|---|
|Proxy|Envoy (C++)|Linkerd2-proxy (Rust)|
|Complexity|High|Low|
|Features|Comprehensive|Core features|
|Resource usage|Higher|Lower|
|mTLS|Yes|Yes|
|Traffic splitting|Yes|Yes|
|Circuit breaking|Yes|Basic|
|Wasm extensions|Yes|No|
|Learning curve|Steep|Gentle|
|Best for|Large orgs, full control|Simpler setup, Kubernetes-native|

### 3.6 What your Spring Boot app needs to do differently with a service mesh

```java
// Almost nothing — the mesh handles retries, mTLS, tracing at the infrastructure level
// Your code stays clean

// BEFORE service mesh — you handled retries in code:
WebClient.builder()
    .filter(ExchangeFilterFunctions.retry(3))
    .build();

// AFTER service mesh — Istio retries at the proxy layer:
WebClient.builder().build(); // just call the service — mesh handles retries

// HOWEVER: still implement application-level timeouts
// Mesh retries don't know about business-level idempotency
@GetMapping("/orders/{id}")
public Mono<Order> getOrder(@PathVariable Long id) {
    return orderClient.getOrder(id)
        .timeout(Duration.ofSeconds(5)); // application timeout still important
}
```

## 4. API Gateway Pattern

An API Gateway is a **single entry point** for all client traffic into a microservices cluster. It handles cross-cutting concerns that would otherwise be duplicated across every service.

### 4.1 Responsibilities

```
Client ──► API Gateway ──► Service A
                      └──► Service B
                      └──► Service C

Gateway handles:
  ✓ Authentication / Authorization (JWT validation)
  ✓ Rate limiting
  ✓ SSL termination
  ✓ Request routing
  ✓ Load balancing
  ✓ Request/response transformation
  ✓ Logging and tracing
  ✓ Caching
  ✓ API versioning
  ✓ Protocol translation (REST → gRPC)
```

### 4.2 Spring Cloud Gateway

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-gateway</artifactId>
</dependency>
```

```yaml
# application.yml — declarative routing
spring:
  cloud:
    gateway:
      routes:
        - id: order-service
          uri: lb://order-service          # lb:// = load-balanced via service discovery
          predicates:
            - Path=/api/orders/**
          filters:
            - StripPrefix=1                # remove /api prefix before forwarding
            - name: CircuitBreaker
              args:
                name: orderServiceCB
                fallbackUri: forward:/fallback/orders
            - name: RateLimiter
              args:
                redis-rate-limiter.replenishRate: 100
                redis-rate-limiter.burstCapacity: 200
                redis-rate-limiter.requestedTokens: 1

        - id: user-service
          uri: lb://user-service
          predicates:
            - Path=/api/users/**
          filters:
            - AddRequestHeader=X-Gateway-Version, 1.0

        - id: product-service-v2
          uri: lb://product-service
          predicates:
            - Path=/api/products/**
            - Header=X-API-Version, 2      # route v2 requests to new service
```

```java
// Programmatic route configuration
@Configuration
public class GatewayConfig {

    @Bean
    public RouteLocator customRoutes(RouteLocatorBuilder builder) {
        return builder.routes()
            .route("order-service", r -> r
                .path("/api/orders/**")
                .filters(f -> f
                    .rewritePath("/api/(?<segment>.*)", "/${segment}")
                    .addRequestHeader("X-Service-Name", "order-service")
                    .retry(config -> config
                        .setRetries(3)
                        .setMethods(HttpMethod.GET)
                        .setBackoff(Duration.ofMillis(100),
                                    Duration.ofSeconds(1), 2, true))
                )
                .uri("lb://order-service"))
            .build();
    }
}

// JWT authentication filter — validates token before routing
@Component
public class JwtAuthenticationFilter implements GlobalFilter, Ordered {

    private final JwtTokenProvider tokenProvider;

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String token = extractToken(exchange.getRequest());

        if (token == null) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }

        try {
            Claims claims = tokenProvider.validateToken(token);
            // Add user info to request headers for downstream services
            ServerHttpRequest mutatedRequest = exchange.getRequest().mutate()
                .header("X-User-Id", claims.getSubject())
                .header("X-User-Roles", String.join(",", claims.get("roles", List.class)))
                .build();
            return chain.filter(exchange.mutate().request(mutatedRequest).build());
        } catch (JwtException ex) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }
    }

    @Override
    public int getOrder() { return -1; } // run before other filters
}
```

## 5. Backend for Frontend (BFF)

The BFF pattern creates **a dedicated API gateway for each type of client** (mobile, web, third-party). Rather than one generic API, each BFF is tailored to the exact needs of its frontend.

### 5.1 The Problem BFF Solves

```
Problem with one generic API:
  Mobile app: needs lean payloads (slow networks, small screens)
  Web app: needs rich data (fast connection, full UI)
  Third party: needs stable, versioned public API

Generic API → either bloated (over-fetches for mobile)
              or thin (under-fetches for web → extra round trips)
```

### 5.2 BFF Architecture

```
                    ┌────────────────┐
Mobile App   ──────►│  Mobile BFF    │
                    └───────┬────────┘
                            │
                    ┌────────────────┐
Web App      ──────►│   Web BFF      ├──────► Order Service
                    └───────┬────────┘        User Service
                            │                 Product Service
                    ┌────────────────┐        Inventory Service
3rd Party    ──────►│  Public API GW │
                    └────────────────┘
```

### 5.3 Mobile BFF — lean, battery-optimized responses

```java
@RestController
@RequestMapping("/mobile/v1")
@RequiredArgsConstructor
public class MobileBffController {

    private final OrderService orderService;
    private final UserService userService;

    // Lean order list — only fields the mobile app needs
    @GetMapping("/orders")
    public Mono<MobileOrderListResponse> getOrders(
            @RequestHeader("X-User-Id") Long userId,
            @RequestParam(defaultValue = "10") int limit) {

        return orderService.findByUserId(userId, limit)
            .map(orders -> MobileOrderListResponse.builder()
                .orders(orders.stream()
                    .map(o -> MobileOrderSummary.builder()
                        .id(o.getId())
                        .status(o.getStatus())
                        .totalAmount(o.getTotalAmount())
                        // omit: detailed items, user info, timestamps (mobile doesn't need them)
                        .build())
                    .collect(Collectors.toList()))
                .totalCount(orders.size())
                .build());
    }

    // Aggregated single endpoint — mobile can't afford multiple round trips
    @GetMapping("/home")
    public Mono<MobileHomeResponse> getHomeScreen(@RequestHeader("X-User-Id") Long userId) {
        return Mono.zip(
            orderService.findRecentByUserId(userId, 3).collectList(),
            userService.findById(userId),
            promotionService.getActivePromotions(userId, 5)
        ).map(tuple -> MobileHomeResponse.builder()
            .recentOrders(tuple.getT1().stream().map(this::toMobileSummary).toList())
            .userName(tuple.getT2().getFirstName()) // just first name
            .promotions(tuple.getT3())
            .build());
    }
}
```

## 6. Interview Questions

**Q: When would you choose gRPC over REST for service-to-service communication?**

A: gRPC is the better choice when performance is critical for high-volume internal service calls — its binary protobuf payload is 5-10x smaller than JSON, and HTTP/2 multiplexing eliminates head-of-line blocking. It's also the right choice when you need native streaming (server push, client streaming, or bidirectional), or when you're building polyglot microservices where auto-generated clients in multiple languages from one `.proto` contract reduce boilerplate and ensure consistency. REST remains better for public APIs, browser clients (gRPC-Web adds complexity), and teams without protobuf tooling experience.

---

**Q: What is the key difference between a service mesh and an API gateway?**

A: An API gateway sits at the **edge** of your system — it handles north-south traffic (external clients to internal services), providing authentication, rate limiting, and routing for external consumers. A service mesh handles **east-west** traffic — communication between services inside your cluster. It provides mTLS, retries, circuit breaking, and observability for internal service-to-service calls via sidecar proxies without application code changes. In practice, you typically have both: a gateway at the perimeter and a mesh for internal communication.

---

**Q: What is the N+1 problem in GraphQL and how do you solve it?**

A: The N+1 problem occurs when resolving nested types. For example, fetching 10 orders and then resolving the `user` field on each order triggers 10 separate database calls (1 for orders + N for users). The standard solution is DataLoader — it batches all user ID lookups that occur within a single request into one call (`SELECT * FROM users WHERE id IN (1, 2, 3, ...)`). Spring Boot's `@SchemaMapping` with `DataLoader<Long, User>` parameter handles this automatically, grouping all field resolutions within the same request tick into a single batch call.

---

**Q: When should you use asynchronous communication over synchronous?**

A: Asynchronous communication is the better choice when the caller doesn't need an immediate result, when temporal decoupling is valuable (upstream can proceed even if downstream is temporarily unavailable), when you want natural load leveling (a message broker absorbs traffic spikes), or when operations are long-running. Use synchronous communication when the caller genuinely needs the result to continue (a user creating an order needs the order ID to return in the HTTP response), when you need immediate consistency guarantees, or when the operation is simple and fast and the added infrastructure of a broker isn't justified.

---

**Q: What is the BFF pattern and why would you use it?**

A: Backend for Frontend is an API layer purpose-built for a specific client type — typically separate BFFs for mobile, web, and public/third-party consumers. Rather than forcing all clients to use one generic API that under-serves some and over-serves others, each BFF aggregates exactly the data its client needs in the shape it needs it. Mobile BFFs return lean payloads (small data, fewer round trips for battery/bandwidth efficiency); web BFFs return rich aggregated responses (web apps can handle large payloads and benefit from reducing the number of calls). The BFF is owned by the team building the frontend — it's a thin orchestration layer, not a place for business logic.

---

**Q: What does a service mesh handle that you would otherwise implement in every Spring Boot service?**

A: Without a service mesh, each service must implement: mTLS (mutual TLS between services), retry logic with exponential backoff, circuit breaking, distributed tracing header propagation, connection pool tuning, and traffic management for canary deployments. With a service mesh like Istio, all of this moves to the sidecar proxy layer and is configured via Kubernetes manifests — your Spring Boot service makes a plain HTTP call and the Envoy sidecar handles encryption, retries, and tracing transparently. This eliminates duplicated resilience code across every service and ensures consistent behavior regardless of language or framework.

---

## 7. Key Takeaways

- **REST** for public APIs and browser-facing endpoints; **gRPC** for high-throughput internal service calls; **GraphQL** for flexible client-driven APIs and BFF patterns
- **Synchronous** when the caller needs an immediate result; **asynchronous** when temporal decoupling and throughput matter more than immediate consistency
- A **service mesh** handles east-west (service-to-service) cross-cutting concerns at the infrastructure layer — your code stays clean
- An **API Gateway** handles north-south (external client to cluster) concerns: auth, rate limiting, routing, SSL termination
- **BFF** creates purpose-built API layers per client type — mobile gets lean, web gets rich, third-party gets stable and versioned
- API Gateway and service mesh are complementary, not alternatives — most production systems have both
