# Distributed Transactions

## Overview

A distributed transaction spans multiple services or databases that must all succeed or all fail together. In a monolith, a single ACID transaction handles this trivially. In microservices, each service owns its own database — you cannot span a transaction across service boundaries using traditional mechanisms without paying severe costs in availability, latency, and coupling.

**The core problem**: Microservices cannot share a database transaction. If Service A updates its DB and Service B's update fails, you have inconsistent state. How do you recover?

**The brutal truth**: there is no distributed equivalent of ACID that is practical at scale. The solutions below are pragmatic trade-offs, not perfect replacements.

---

## 1. Two-Phase Commit (2PC) Problems

Understanding why 2PC fails in microservices is the foundation for understanding why Saga exists.

### 1.1 How 2PC Works

```
Coordinator (e.g., transaction manager)
    │
    ├── Phase 1: PREPARE ────────────────────────────────────────
    │       │
    │       ├──► Service A DB: "Prepare to commit — can you?"
    │       │◄── "YES, prepared (locks acquired)"
    │       │
    │       ├──► Service B DB: "Prepare to commit — can you?"
    │       │◄── "YES, prepared (locks acquired)"
    │       │
    │       └──► Service C DB: "Prepare to commit — can you?"
    │           ◄── "YES, prepared (locks acquired)"
    │
    ├── Phase 2: COMMIT (all said YES) ──────────────────────────
    │       ├──► Service A DB: "COMMIT"
    │       ├──► Service B DB: "COMMIT"
    │       └──► Service C DB: "COMMIT"
    │
    └── Phase 2: ROLLBACK (any said NO) ────────────────────────
            ├──► Service A DB: "ROLLBACK"
            ├──► Service B DB: "ROLLBACK"
            └──► Service C DB: "ROLLBACK"
```

### 1.2 The Problems with 2PC

**Problem 1: Blocking locks — availability killer**

```
During Phase 1 → Phase 2, all participants hold locks.
If the coordinator crashes after PREPARE but before COMMIT:
  - All participants are blocked, holding locks
  - Cannot commit (coordinator gone)
  - Cannot rollback (coordinator gone — might have sent COMMIT to some)
  - System stuck until coordinator recovers
  - Duration: seconds to minutes of complete unavailability
```

**Problem 2: Coordinator is a single point of failure**

```
If coordinator crashes mid-transaction:
  - Participants in PREPARED state don't know whether to commit or rollback
  - Must wait for coordinator recovery
  - In high-traffic systems: cascading lock buildup
```

**Problem 3: Synchronous coupling**

```
All participants must be available simultaneously during both phases.
Service B being slow → entire transaction is slow.
Service C being down → entire transaction cannot proceed.
CAP theorem: during a network partition, 2PC sacrifices availability.
```

**Problem 4: Doesn't work across polyglot databases**

```
MySQL + MongoDB + Elasticsearch cannot participate in a distributed 2PC.
XA transactions only work with supporting databases.
Most modern NoSQL, messaging, and search systems don't support XA.
```

**Problem 5: Performance**

```
2PC requires 4+ network round trips (Prepare x N + Commit x N).
Each round trip adds latency.
For 3 participants: 6 network calls minimum, all synchronous.
This doesn't scale.
```

### 1.3 When 2PC is acceptable

```
2PC is acceptable ONLY when:
  ✓ All resources support XA (JTA-compatible JDBC, JMS)
  ✓ Operation count is very low (not high-traffic)
  ✓ All participants are in the same datacenter (low latency)
  ✓ Strong consistency is non-negotiable (financial ledger entries)
  ✓ Operational complexity of Saga is unjustifiable for the use case

In Spring Boot (Spring Data + JTA):
```

```java
// 2PC via JTA (only practical for same-datacenter, XA-capable resources)
@Configuration
public class JtaConfig {
    @Bean
    public JtaTransactionManager transactionManager() {
        return new JtaTransactionManager(); // delegates to JTA provider (Atomikos/Bitronix)
    }
}

@Service
public class TransferService {

    @Transactional  // JTA — spans two XA-capable datasources
    public void transfer(Long fromAccount, Long toAccount, BigDecimal amount) {
        accountDb1.debit(fromAccount, amount);   // XA datasource 1
        accountDb2.credit(toAccount, amount);    // XA datasource 2
        // If either fails: both roll back atomically
    }
}
```

## 2. Saga Pattern — Choreography vs Orchestration

A Saga replaces a single distributed transaction with a **sequence of local transactions**, each publishing an event or message that triggers the next step. If any step fails, compensating transactions undo the preceding steps.

```
Local TX 1 → event → Local TX 2 → event → Local TX 3
    ↓ fail                ↓ fail
compensate ←── compensate ←── (rollback chain)
```

### 2.1 Saga: Choreography

In choreography, **there is no central coordinator**. Each service listens for events from the previous step and reacts by performing its own local transaction and publishing the next event. Services communicate indirectly through events.

```
OrderService                InventoryService           PaymentService
     │                            │                          │
     │── OrderPlaced ────────────►│                          │
     │                            │── StockReserved ────────►│
     │                            │                          │── PaymentCharged ──►
     │                            │                          │
     │◄─ OrderConfirmed ──────────┴──────────────────────────┘
     │
     │  (on failure)
     │◄─ PaymentFailed ─────── PaymentService
     │◄─ StockReleased ─────── InventoryService (compensates on PaymentFailed)
     │── OrderCancelled ──────► (notify customer)
```

**Implementation with Spring Kafka:**

```java
// Step 1: Order Service — places order and publishes event
@Service
@RequiredArgsConstructor
@Slf4j
public class OrderService {

    private final OrderRepository orderRepository;
    private final KafkaTemplate<String, Object> kafka;

    @Transactional
    public Order placeOrder(CreateOrderRequest request) {
        Order order = orderRepository.save(
            Order.builder()
                .userId(request.getUserId())
                .items(request.getItems())
                .status(OrderStatus.PENDING)
                .build()
        );

        // Publish within same transaction (outbox pattern — see section 5)
        kafka.send("order.placed", order.getId().toString(),
            new OrderPlacedEvent(order.getId(), order.getUserId(), order.getItems()));

        return order;
    }

    // Compensation: cancel order when payment fails
    @KafkaListener(topics = "payment.failed", groupId = "order-service")
    @Transactional
    public void onPaymentFailed(PaymentFailedEvent event) {
        orderRepository.findById(event.getOrderId()).ifPresent(order -> {
            order.setStatus(OrderStatus.CANCELLED);
            order.setCancelReason("Payment failed: " + event.getReason());
            orderRepository.save(order);
            kafka.send("order.cancelled", order.getId().toString(),
                new OrderCancelledEvent(order.getId(), event.getReason()));
        });
    }
}

// Step 2: Inventory Service — reacts to order.placed
@Service
@RequiredArgsConstructor
@Slf4j
public class InventoryService {

    private final StockRepository stockRepository;
    private final KafkaTemplate<String, Object> kafka;

    @KafkaListener(topics = "order.placed", groupId = "inventory-service")
    @Transactional
    public void onOrderPlaced(OrderPlacedEvent event) {
        try {
            reserveStock(event.getOrderId(), event.getItems()); // local TX
            kafka.send("stock.reserved", event.getOrderId().toString(),
                new StockReservedEvent(event.getOrderId(), event.getUserId()));
        } catch (InsufficientStockException ex) {
            kafka.send("stock.reservation.failed", event.getOrderId().toString(),
                new StockReservationFailedEvent(event.getOrderId(), ex.getMessage()));
        }
    }

    // Compensation: release stock when payment fails
    @KafkaListener(topics = "payment.failed", groupId = "inventory-service")
    @Transactional
    public void onPaymentFailed(PaymentFailedEvent event) {
        releaseStock(event.getOrderId()); // undo the reservation
        kafka.send("stock.released", event.getOrderId().toString(),
            new StockReleasedEvent(event.getOrderId()));
    }
}

// Step 3: Payment Service — reacts to stock.reserved
@Service
@RequiredArgsConstructor
@Slf4j
public class PaymentService {

    private final PaymentRepository paymentRepository;
    private final KafkaTemplate<String, Object> kafka;

    @KafkaListener(topics = "stock.reserved", groupId = "payment-service")
    @Transactional
    public void onStockReserved(StockReservedEvent event) {
        try {
            Payment payment = chargeCard(event.getOrderId(), event.getUserId());
            kafka.send("payment.charged", event.getOrderId().toString(),
                new PaymentChargedEvent(event.getOrderId(), payment.getId()));
        } catch (PaymentDeclinedException ex) {
            kafka.send("payment.failed", event.getOrderId().toString(),
                new PaymentFailedEvent(event.getOrderId(), ex.getMessage()));
        }
    }
}
```

**Choreography strengths:**

- No central coordinator — services are fully autonomous
- Services are loosely coupled — only through event contracts
- Easy to add new steps (subscribe to existing events, publish new ones)
- Scales well — each service processes independently

**Choreography weaknesses:**

- Hard to understand the overall flow — it's implicit in event chains
- Difficult to track saga state — no single place to query "where is this order?"
- Cyclic dependencies can emerge as the event graph grows
- Testing requires spinning up multiple services or a complex event bus mock

### 2.2 Saga: Orchestration

In orchestration, a **central orchestrator** (also called a saga manager) explicitly tells each service what to do and handles the coordination logic. It owns the saga state machine.

```
                    ┌──────────────────────────────┐
                    │      OrderSagaOrchestrator    │
                    │                               │
                    │  PENDING → RESERVING_STOCK    │
                    │         → CHARGING_PAYMENT    │
                    │         → CONFIRMING_ORDER    │
                    │         → COMPLETED           │
                    │  (or)   → COMPENSATING        │
                    │         → CANCELLED           │
                    └──────────────────────────────┘
                           │          ▲
               commands    │          │ replies
                           ▼          │
               ┌─────────────────────────────────┐
               │ InventoryService │ PaymentService│
               └─────────────────────────────────┘
```

**Implementation with Spring State Machine + Kafka:**

```java
// Saga state machine — tracks the saga's current state
public enum OrderSagaState {
    PENDING,
    RESERVING_STOCK,
    STOCK_RESERVED,
    CHARGING_PAYMENT,
    PAYMENT_CHARGED,
    CONFIRMING_ORDER,
    COMPLETED,
    // Compensation states
    RELEASING_STOCK,
    REFUNDING_PAYMENT,
    CANCELLED
}

public enum OrderSagaEvent {
    ORDER_CREATED,
    STOCK_RESERVED, STOCK_RESERVATION_FAILED,
    PAYMENT_CHARGED, PAYMENT_FAILED,
    ORDER_CONFIRMED,
    COMPENSATION_COMPLETE
}

// Saga entity — persisted state
@Entity
@Table(name = "order_sagas")
@Data
public class OrderSaga {
    @Id
    private UUID sagaId;
    private Long orderId;
    private Long userId;

    @Enumerated(EnumType.STRING)
    private OrderSagaState state;

    private String failureReason;
    private LocalDateTime createdAt;
    private LocalDateTime updatedAt;

    @Version
    private Long version; // optimistic locking — prevents concurrent state transitions
}

// Orchestrator — owns the saga state and sends commands
@Service
@RequiredArgsConstructor
@Slf4j
public class OrderSagaOrchestrator {

    private final OrderSagaRepository sagaRepository;
    private final KafkaTemplate<String, Object> kafka;

    // Start the saga
    @Transactional
    public void startOrderSaga(Order order) {
        OrderSaga saga = OrderSaga.builder()
            .sagaId(UUID.randomUUID())
            .orderId(order.getId())
            .userId(order.getUserId())
            .state(OrderSagaState.RESERVING_STOCK)
            .createdAt(LocalDateTime.now())
            .build();
        sagaRepository.save(saga);

        // Send command to inventory service
        kafka.send("commands.inventory.reserve-stock",
            order.getId().toString(),
            new ReserveStockCommand(saga.getSagaId(), order.getId(), order.getItems()));
    }

    // React to inventory reply
    @KafkaListener(topics = "replies.inventory.stock-reserved", groupId = "saga-orchestrator")
    @Transactional
    public void onStockReserved(StockReservedReply reply) {
        OrderSaga saga = sagaRepository.findBySagaId(reply.getSagaId())
            .orElseThrow(() -> new SagaNotFoundException(reply.getSagaId()));

        saga.setState(OrderSagaState.CHARGING_PAYMENT);
        sagaRepository.save(saga);

        // Send next command
        kafka.send("commands.payment.charge",
            saga.getOrderId().toString(),
            new ChargePaymentCommand(saga.getSagaId(), saga.getOrderId(), saga.getUserId()));
    }

    @KafkaListener(topics = "replies.inventory.stock-reservation-failed",
                   groupId = "saga-orchestrator")
    @Transactional
    public void onStockReservationFailed(StockReservationFailedReply reply) {
        OrderSaga saga = sagaRepository.findBySagaId(reply.getSagaId())
            .orElseThrow(() -> new SagaNotFoundException(reply.getSagaId()));

        // No compensation needed — nothing was changed yet
        saga.setState(OrderSagaState.CANCELLED);
        saga.setFailureReason("Insufficient stock: " + reply.getReason());
        sagaRepository.save(saga);

        kafka.send("commands.order.cancel",
            saga.getOrderId().toString(),
            new CancelOrderCommand(saga.getSagaId(), saga.getOrderId(), reply.getReason()));
    }

    @KafkaListener(topics = "replies.payment.charged", groupId = "saga-orchestrator")
    @Transactional
    public void onPaymentCharged(PaymentChargedReply reply) {
        OrderSaga saga = sagaRepository.findBySagaId(reply.getSagaId())
            .orElseThrow(() -> new SagaNotFoundException(reply.getSagaId()));

        saga.setState(OrderSagaState.COMPLETED);
        sagaRepository.save(saga);

        kafka.send("commands.order.confirm",
            saga.getOrderId().toString(),
            new ConfirmOrderCommand(saga.getSagaId(), saga.getOrderId()));
    }

    @KafkaListener(topics = "replies.payment.failed", groupId = "saga-orchestrator")
    @Transactional
    public void onPaymentFailed(PaymentFailedReply reply) {
        OrderSaga saga = sagaRepository.findBySagaId(reply.getSagaId())
            .orElseThrow(() -> new SagaNotFoundException(reply.getSagaId()));

        // Must compensate: release the reserved stock
        saga.setState(OrderSagaState.RELEASING_STOCK);
        saga.setFailureReason("Payment failed: " + reply.getReason());
        sagaRepository.save(saga);

        kafka.send("commands.inventory.release-stock",
            saga.getOrderId().toString(),
            new ReleaseStockCommand(saga.getSagaId(), saga.getOrderId()));
    }

    @KafkaListener(topics = "replies.inventory.stock-released", groupId = "saga-orchestrator")
    @Transactional
    public void onStockReleased(StockReleasedReply reply) {
        OrderSaga saga = sagaRepository.findBySagaId(reply.getSagaId())
            .orElseThrow(() -> new SagaNotFoundException(reply.getSagaId()));

        saga.setState(OrderSagaState.CANCELLED);
        sagaRepository.save(saga);

        kafka.send("commands.order.cancel",
            saga.getOrderId().toString(),
            new CancelOrderCommand(saga.getSagaId(), saga.getOrderId(),
                saga.getFailureReason()));
    }
}
```

**Orchestration strengths:**

- Single place to understand the entire flow — easy to reason about
- Saga state is explicitly persisted and queryable
- Easier to handle complex conditional logic and branches
- Simpler to debug — check saga state table to know exactly where it stopped

**Orchestration weaknesses:**

- Central orchestrator can become a God object with business logic leaking in
- More infrastructure — orchestrator is a service that must be scaled/monitored
- Slightly more coupled — services know about the orchestrator's command topics

---

### 2.3 Choreography vs Orchestration

||Choreography|Orchestration|
|---|---|---|
|Coordination|Decentralized (events)|Centralized (orchestrator)|
|Coupling|Loose — services unaware of each other|Tighter — services know command topics|
|Flow visibility|Implicit — distributed across services|Explicit — in orchestrator state machine|
|Saga state|No central state|Persisted in orchestrator|
|Business logic|Distributed|Centralized in orchestrator|
|Debugging|Harder — trace events across services|Easier — check saga table|
|Adding steps|Easy — subscribe to event|Must modify orchestrator|
|Best for|Simple linear flows, autonomous services|Complex flows, many branches, needs auditability|

## 3. Eventual Consistency

Eventual consistency is the guarantee that, in the absence of new updates, **all replicas will eventually converge to the same value**. It is the trade-off accepted when you abandon 2PC.

### 3.1 What eventual consistency means in practice

```
At T=0: User places order
  Order Service DB:     status = PENDING       (immediately consistent)
  Inventory Service DB: reservation = null     (not yet processed)
  Payment Service DB:   payment = null         (not yet processed)

At T=100ms: Kafka delivers order.placed event
  Inventory Service DB: reservation = RESERVED  (eventually consistent)
  Payment Service DB:   payment = null          (still not processed)

At T=200ms: Kafka delivers stock.reserved event
  Payment Service DB: payment = CHARGED          (eventually consistent)

At T=250ms: All services consistent
  Order Service:     status = CONFIRMED
  Inventory:         reservation = RESERVED
  Payment:           payment = CHARGED
```

### 3.2 Handling the consistency window in your API

```java
// Option 1: Return 202 Accepted — acknowledge, but not guaranteed yet
@PostMapping("/orders")
public ResponseEntity<OrderAcceptedResponse> placeOrder(@RequestBody CreateOrderRequest req) {
    Order order = orderService.placeOrder(req);
    return ResponseEntity
        .accepted()                        // 202 — not 201 Created
        .header("Location", "/api/orders/" + order.getId())
        .body(new OrderAcceptedResponse(
            order.getId(),
            "Order accepted. Processing asynchronously.",
            "/api/orders/" + order.getId() + "/status"  // polling URL
        ));
}

// Option 2: Polling endpoint for order status
@GetMapping("/orders/{id}/status")
public OrderStatusDto getOrderStatus(@PathVariable Long id) {
    Order order = orderRepository.findById(id)
        .orElseThrow(() -> new NotFoundException("Order " + id));
    return new OrderStatusDto(
        order.getId(),
        order.getStatus(),          // PENDING, CONFIRMED, FAILED
        order.getStatusMessage(),   // "Processing payment..."
        order.getUpdatedAt()
    );
}

// Option 3: WebSocket / SSE push when status changes
@GetMapping(value = "/orders/{id}/status/stream",
            produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<OrderStatusDto> streamOrderStatus(@PathVariable Long id) {
    return orderStatusService.subscribeToStatus(id)
        .takeUntil(status ->
            status.getStatus() == OrderStatus.CONFIRMED ||
            status.getStatus() == OrderStatus.CANCELLED
        );
}
```

### 3.3 Read-your-writes consistency — a common problem

```java
// Problem: User creates a post, immediately queries their feed — post not there yet
// (write went to primary, read went to replica — replication lag)

// Solution 1: Read from primary after write
@Transactional(readOnly = false)
public Post createPost(CreatePostRequest req) {
    Post post = postRepository.save(req.toPost());
    // Force read from primary for this session for next N seconds
    readRoutingContext.setForcePrimary(Duration.ofSeconds(5));
    return post;
}

// Solution 2: Return the created entity in the write response
// Client doesn't need to re-read — they already have it
@PostMapping("/posts")
public ResponseEntity<PostDto> createPost(@RequestBody CreatePostRequest req) {
    Post post = postService.create(req);
    return ResponseEntity.created(URI.create("/api/posts/" + post.getId()))
        .body(toDto(post)); // return the created post directly
}

// Solution 3: Version token — client verifies they're seeing post-write state
@GetMapping("/feed")
public FeedResponse getFeed(@RequestHeader(value = "X-Min-Version",
                                            required = false) Long minVersion) {
    if (minVersion != null && feedVersion.getCurrent() < minVersion) {
        // Client is ahead of read replica — wait briefly or redirect to primary
        return ResponseEntity.status(HttpStatus.SERVICE_UNAVAILABLE)
            .header("Retry-After", "1")
            .build();
    }
    return feedService.getFeed();
}
```

### 3.4 Idempotency — essential for eventual consistency

When messages may be delivered more than once (Kafka at-least-once), consumers must be idempotent.

```java
@Service
@RequiredArgsConstructor
@Slf4j
public class PaymentEventHandler {

    private final PaymentRepository paymentRepository;
    private final ProcessedEventRepository processedEvents;

    @KafkaListener(topics = "stock.reserved", groupId = "payment-service")
    @Transactional
    public void onStockReserved(StockReservedEvent event,
                                 @Header(KafkaHeaders.RECEIVED_MESSAGE_KEY) String key) {
        String idempotencyKey = "stock.reserved:" + event.getOrderId();

        // Check if already processed — idempotency guard
        if (processedEvents.existsByKey(idempotencyKey)) {
            log.info("Duplicate event ignored: {}", idempotencyKey);
            return;
        }

        processPayment(event);

        // Mark as processed — within same transaction
        processedEvents.save(new ProcessedEvent(idempotencyKey, LocalDateTime.now()));
    }
}

// ProcessedEvent table schema
@Entity
@Table(name = "processed_events",
       uniqueConstraints = @UniqueConstraint(columnNames = "idempotency_key"))
public class ProcessedEvent {
    @Id @GeneratedValue
    private Long id;

    @Column(name = "idempotency_key", nullable = false, unique = true)
    private String key;

    private LocalDateTime processedAt;
}
```

## 4. Compensating Transactions

A compensating transaction **semantically undoes** the effects of a previously committed local transaction. It is not a rollback — the original transaction has already committed; the compensating transaction creates a new transaction that reverses the business effect.

### 4.1 Compensation vs Rollback

```
Rollback (ACID):        Database-level, instant, as if nothing happened.
                        Only possible if transaction hasn't committed.

Compensating TX:        Business-level, creates audit trail.
                        Applied AFTER original transaction has committed.
                        Example: issue a refund (not reverse a payment).
```

### 4.2 Designing compensatable operations

```java
// Every forward operation must have a corresponding compensating operation
// Document them together

public interface InventoryPort {
    // Forward:      reserve stock for an order
    ReservationId reserveStock(Long orderId, List<OrderItem> items);

    // Compensating: release the reservation (called on saga failure)
    void releaseStock(ReservationId reservationId);
}

public interface PaymentPort {
    // Forward:      charge the customer
    PaymentId chargeCard(Long orderId, Long userId, BigDecimal amount);

    // Compensating: refund the payment
    void refundPayment(PaymentId paymentId);
}

public interface OrderPort {
    // Forward:      confirm the order
    void confirmOrder(Long orderId);

    // Compensating: cancel the order
    void cancelOrder(Long orderId, String reason);
}
```

### 4.3 Compensation chain implementation

```java
// Saga orchestrator implementing compensation chain
@Service
@RequiredArgsConstructor
@Slf4j
public class OrderSagaCompensator {

    private final InventoryPort inventory;
    private final PaymentPort payment;
    private final OrderPort order;

    /**
     * Compensate a partially completed saga.
     * Must be idempotent — may be called multiple times on retry.
     *
     * @param saga The saga to compensate, with its last successful step recorded
     */
    public Mono<Void> compensate(OrderSaga saga) {
        log.warn("Compensating saga {} from state {}", saga.getSagaId(), saga.getState());

        return switch (saga.getState()) {
            case PAYMENT_CHARGED -> // payment was charged — refund it, then release stock
                payment.refundPayment(saga.getPaymentId())
                    .then(inventory.releaseStock(saga.getReservationId()))
                    .then(order.cancelOrder(saga.getOrderId(), saga.getFailureReason()));

            case STOCK_RESERVED -> // stock was reserved but payment not charged — just release
                inventory.releaseStock(saga.getReservationId())
                    .then(order.cancelOrder(saga.getOrderId(), saga.getFailureReason()));

            case PENDING, RESERVING_STOCK -> // nothing committed yet — just cancel
                order.cancelOrder(saga.getOrderId(), saga.getFailureReason());

            default -> {
                log.error("Unknown state for compensation: {}", saga.getState());
                yield Mono.empty();
            }
        };
    }
}
```

### 4.4 Making compensations idempotent

Compensations may be retried. They must be safe to execute multiple times:

```java
@Service
@RequiredArgsConstructor
public class InventoryCompensationService {

    private final ReservationRepository reservationRepo;

    public Mono<Void> releaseStock(ReservationId reservationId) {
        return reservationRepo.findById(reservationId.getValue())
            .flatMap(reservation -> {
                // Idempotency: only release if currently RESERVED
                if (reservation.getStatus() == ReservationStatus.RELEASED) {
                    log.info("Stock already released for reservation {} — skipping",
                        reservationId);
                    return Mono.empty(); // safe to call multiple times
                }
                reservation.setStatus(ReservationStatus.RELEASED);
                reservation.setReleasedAt(Instant.now());
                return reservationRepo.save(reservation).then();
            })
            .switchIfEmpty(Mono.error(
                new ReservationNotFoundException(reservationId.getValue())));
    }
}

@Service
@RequiredArgsConstructor
public class PaymentCompensationService {

    private final PaymentRepository paymentRepo;
    private final PaymentGateway gateway;

    public Mono<Void> refundPayment(PaymentId paymentId) {
        return paymentRepo.findById(paymentId.getValue())
            .flatMap(payment -> {
                // Idempotency: only refund if currently CHARGED
                if (payment.getStatus() == PaymentStatus.REFUNDED) {
                    log.info("Payment {} already refunded — skipping", paymentId);
                    return Mono.empty();
                }
                return gateway.issueRefund(payment.getGatewayId(), payment.getAmount())
                    .flatMap(refundRef -> {
                        payment.setStatus(PaymentStatus.REFUNDED);
                        payment.setRefundReference(refundRef);
                        return paymentRepo.save(payment).then();
                    });
            });
    }
}
```

### 4.5 The pivot transaction — point of no return

```
In a Saga, the "pivot transaction" is the last step that can still trigger compensation.
After it, subsequent steps either succeed or are retried — but not compensated.

Example: Order Saga
  reserveStock()   ← compensatable (release stock)
  chargePayment()  ← PIVOT — after this, we're committed
  confirmOrder()   ← retried until success, never compensated
  sendEmail()      ← retried, not compensated (email can't be unsent anyway)

Design insight: put your most irreversible action as late as possible in the saga,
so that if it fails, there's less to compensate.
```

## 5. Outbox Pattern

The Outbox pattern solves a fundamental atomicity problem: **how do you atomically update your database AND publish an event to a message broker?**

### 5.1 The Problem

```java
// This is NOT atomic — two separate I/O operations
@Transactional
public Order placeOrder(CreateOrderRequest req) {
    Order order = orderRepository.save(req.toOrder()); // DB write

    // What if this fails after the DB commit? Event is lost.
    // What if the app crashes between these two lines?
    kafka.send("order.placed", new OrderPlacedEvent(order)); // Kafka write

    return order;
}

// Problems:
//   1. DB commits, Kafka send fails → order exists, no event → inventory never notified
//   2. Kafka sends, DB commits, then crash → event sent for uncommitted data (phantom event)
//   3. No way to guarantee exactly-once across DB + Kafka in a single operation
```

### 5.2 Outbox Pattern — solution

Instead of writing directly to Kafka, **write the event to an outbox table in the same DB transaction** as the business data. A separate relay process reads the outbox and publishes to Kafka.

```
Step 1: DB Transaction (atomic)
  ┌─────────────────────────────────────────────┐
  │  INSERT INTO orders (...)     ← business    │
  │  INSERT INTO outbox_events    ← event       │
  │  (same transaction)                         │
  └─────────────────────────────────────────────┘

Step 2: Outbox Relay (separate process)
  ┌─────────────────────────────────────────────┐
  │  SELECT * FROM outbox_events                │
  │    WHERE published = false                  │
  │  FOR UPDATE SKIP LOCKED                     │
  │                                             │
  │  kafka.send(event)  ← publish to Kafka      │
  │  UPDATE outbox_events SET published = true  │
  └─────────────────────────────────────────────┘
```

### 5.3 Outbox table schema

```sql
CREATE TABLE outbox_events (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    aggregate_type  VARCHAR(255) NOT NULL,  -- e.g., 'Order'
    aggregate_id    VARCHAR(255) NOT NULL,  -- e.g., '1234'
    event_type      VARCHAR(255) NOT NULL,  -- e.g., 'OrderPlaced'
    payload         JSONB NOT NULL,         -- event body
    topic           VARCHAR(255) NOT NULL,  -- target Kafka topic
    created_at      TIMESTAMP NOT NULL DEFAULT now(),
    published_at    TIMESTAMP,              -- null = not yet published
    published       BOOLEAN NOT NULL DEFAULT false,
    retry_count     INT NOT NULL DEFAULT 0
);

CREATE INDEX idx_outbox_unpublished ON outbox_events(created_at)
    WHERE published = false;
```

### 5.4 Writing to the outbox in the same transaction

```java
@Entity
@Table(name = "outbox_events")
@Data
@Builder
public class OutboxEvent {
    @Id
    private UUID id;
    private String aggregateType;
    private String aggregateId;
    private String eventType;

    @Column(columnDefinition = "jsonb")
    private String payload;

    private String topic;
    private LocalDateTime createdAt;
    private LocalDateTime publishedAt;
    private boolean published;
    private int retryCount;
}

@Service
@RequiredArgsConstructor
@Slf4j
public class OrderService {

    private final OrderRepository orderRepository;
    private final OutboxEventRepository outboxRepository;
    private final ObjectMapper objectMapper;

    @Transactional  // single local transaction — DB + outbox atomically
    public Order placeOrder(CreateOrderRequest request) {
        // 1. Business write
        Order order = orderRepository.save(
            Order.builder()
                .userId(request.getUserId())
                .items(request.getItems())
                .totalAmount(request.getTotal())
                .status(OrderStatus.PENDING)
                .build()
        );

        // 2. Outbox write — same transaction
        outboxRepository.save(OutboxEvent.builder()
            .id(UUID.randomUUID())
            .aggregateType("Order")
            .aggregateId(order.getId().toString())
            .eventType("OrderPlaced")
            .payload(serialize(new OrderPlacedEvent(order)))
            .topic("order.placed")
            .createdAt(LocalDateTime.now())
            .published(false)
            .build()
        );

        // NO Kafka send here — relay handles it asynchronously
        return order;
    }

    private String serialize(Object event) {
        try {
            return objectMapper.writeValueAsString(event);
        } catch (JsonProcessingException ex) {
            throw new RuntimeException("Failed to serialize event", ex);
        }
    }
}
```

### 5.5 Outbox relay — polling approach

```java
@Component
@RequiredArgsConstructor
@Slf4j
public class OutboxRelayService {

    private final OutboxEventRepository outboxRepository;
    private final KafkaTemplate<String, String> kafka;
    private final TransactionTemplate txTemplate;

    @Scheduled(fixedDelay = 100)  // poll every 100ms
    public void publishPendingEvents() {
        txTemplate.execute(status -> {
            // Select unpublished events, lock to prevent concurrent relay instances
            List<OutboxEvent> pending = outboxRepository
                .findTop100ByPublishedFalseOrderByCreatedAtAsc();

            for (OutboxEvent event : pending) {
                try {
                    kafka.send(event.getTopic(),
                        event.getAggregateId(),
                        event.getPayload()).get(); // wait for ack

                    event.setPublished(true);
                    event.setPublishedAt(LocalDateTime.now());
                    outboxRepository.save(event);

                } catch (Exception ex) {
                    log.error("Failed to publish outbox event {}: {}",
                        event.getId(), ex.getMessage());
                    event.setRetryCount(event.getRetryCount() + 1);
                    outboxRepository.save(event);
                    // Continue — don't block other events
                }
            }
            return null;
        });
    }
}
```

### 5.6 Outbox relay — CDC approach (Debezium)

The polling approach adds latency (up to the poll interval) and load on the DB. **Debezium** solves both problems by reading the PostgreSQL WAL (write-ahead log) — capturing DB changes as they happen in real time, with zero polling overhead.

```
With Debezium:

       │
```

## 6. Pattern Comparison and When to Use Each

```
Scenario                              → Pattern
──────────────────────────────────────────────────────────────────────
< 5 XA-capable resources,            → 2PC (XA/JTA)
  same datacenter, low traffic,
  strict consistency required

Simple linear workflow,               → Saga (Choreography)
  < 4 steps, autonomous services,
  team comfortable with event-driven

Complex workflow with branches,       → Saga (Orchestration)
  needs audit log of saga state,
  multiple failure paths

Any Saga that publishes events        → Outbox Pattern
  to Kafka/broker

Intermediate state visible to users   → Eventual consistency
  is acceptable (order status)        + polling/SSE

Irreversible steps that failed        → Compensating Transactions
  must be undone (payment refund)
```

## 7. Interview Questions

**Q: Why is 2PC considered impractical for microservices?**

A: 2PC has three fundamental problems in a microservices context. First, it requires all participants to hold locks during both phases — if the coordinator crashes between PREPARE and COMMIT, all participants are blocked until recovery, causing cascading unavailability that can last minutes. Second, it requires all resources to be available simultaneously, which violates the independence goal of microservices — if any participant is down, the entire transaction is blocked. Third, it only works with XA-compatible resources. Most modern datastores (MongoDB, Cassandra, Elasticsearch, DynamoDB) and messaging systems don't support XA, making 2PC practically impossible in polyglot architectures.

---

**Q: What is the difference between Saga choreography and orchestration? When would you choose each?**

A: In choreography, services communicate solely through events — each service reacts to an event from the previous step and publishes the next, with no central coordinator. In orchestration, a dedicated saga manager explicitly tells each service what to do and tracks the saga's state centrally. Choose choreography for simple, linear workflows where services are truly autonomous and the team is comfortable reasoning about distributed event chains. Choose orchestration when the flow has complex conditional logic or multiple failure branches, when you need a queryable audit trail of saga state, or when the implicit event chain of choreography would be difficult for new developers to understand. Most production systems with 5+ saga steps benefit from orchestration's clarity.

---

**Q: What is the outbox pattern and what specific problem does it solve?**

A: The outbox pattern solves the dual-write problem: you cannot atomically write to a database and publish to a message broker in one operation — if you write to both separately, a crash between the two operations leaves them inconsistent (event published but DB rolled back, or DB committed but event never published). The outbox pattern writes the event to an outbox table in the same local database transaction as the business data, guaranteeing atomicity. A separate relay process reads unpublished outbox rows and publishes them to Kafka. With Debezium, the relay uses CDC (change data capture) on the database WAL instead of polling, giving near real-time event delivery with zero polling overhead.

---

**Q: What makes a compensating transaction different from a database rollback?**

A: A database rollback is an ACID operation that undoes changes as if the transaction never happened — it happens before commit and leaves no trace. A compensating transaction is a new, separate business transaction that semantically reverses the effects of a previously committed transaction. It leaves an audit trail (refund records, cancellation events), may have its own business rules (a refund might take 3 business days), and cannot always perfectly undo the original action (you cannot un-send an email). For Sagas, compensation must be designed explicitly for each forward operation, must be idempotent (safe to call multiple times on retry), and must account for partial states.

---

**Q: How do you handle idempotency in a Saga where messages may be delivered more than once?**

A: At-least-once delivery in Kafka means consumers must be idempotent — processing the same message twice must produce the same result as processing it once. The standard approach is to store a processed event ID (the Kafka message key or a business idempotency key) in a `processed_events` table within the same database transaction as the business operation. Before processing, check if the key already exists — if so, skip. The check-and-insert happens in the same transaction as the business logic, making it atomic. Additionally, design compensating transactions with status checks — a `releaseStock` operation should check if the reservation is already RELEASED before executing, making it naturally idempotent.

---

**Q: What is eventual consistency and how do you communicate it to API clients?**

A: Eventual consistency means that after a write, all replicas or downstream systems will converge to the same state eventually, but not immediately. In a microservices Saga, a client that places an order may see status PENDING for several hundred milliseconds while inventory reservation and payment processing complete asynchronously. Communicate this to clients by returning HTTP 202 Accepted (not 201 Created) for async operations, providing a status polling URL in the Location header, and optionally streaming status updates via SSE or WebSocket. Clients must be designed to tolerate the consistency window — the UI should show a "processing" state rather than assuming immediate completion.

## 8. Key Takeaways

- **2PC is impractical** for microservices: blocking locks, coordinator SPOF, polyglot incompatibility — avoid it except for rare same-datacenter XA scenarios
- **Saga replaces 2PC** with local transactions + compensation: choreography for simple autonomous flows, orchestration for complex conditional flows needing audit state
- **Compensating transactions are not rollbacks** — they're new business transactions that semantically reverse committed effects, must be idempotent, and leave an audit trail
- **Eventual consistency** is the trade-off you accept — design APIs to communicate it (202 Accepted, polling, SSE) and make consumers idempotent
- **Outbox pattern** is mandatory when publishing events in Sagas — atomically write event + business data in one DB transaction, relay to Kafka separately
- **Debezium CDC** is the production-grade outbox relay: near-realtime, no polling overhead, ordering preserved via WAL
- **Idempotency** is not optional in eventually consistent systems — every Saga step must be safe to execute multiple times
