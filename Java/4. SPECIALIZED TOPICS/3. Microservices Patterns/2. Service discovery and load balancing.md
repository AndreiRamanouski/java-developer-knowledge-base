# Service Discovery and Load Balancing

## Overview

In a microservices system, services are ephemeral — they start, stop, scale, and move across hosts constantly. Service discovery solves the fundamental question: **how does Service A find the current network address of Service B?** Load balancing answers the follow-up: **which of Service B's instances should handle this request?**

Hardcoding IP addresses is not an option. Service discovery replaces static configuration with a dynamic registry, and load balancing distributes requests intelligently across healthy instances.

---

## 1. Client-Side Discovery (Eureka, Consul)

In client-side discovery, **the calling service is responsible** for looking up available instances and choosing which one to call. The service registry holds the list of available instances; the client queries it directly.

```
Service A                  Registry (Eureka/Consul)
    │                             │
    ├── "where is order-service?" ─►│
    │◄── [10.0.0.1:8080,          │
    │     10.0.0.2:8080,          │
    │     10.0.0.3:8080]          │
    │                             │
    ├── picks 10.0.0.2 (LB logic) │
    │                             │
    ▼
10.0.0.2:8080 (Order Service instance)
```

### 1.1 Spring Cloud Eureka

Eureka is Netflix's service registry, integrated into Spring Cloud. It follows a peer-to-peer replication model — Eureka servers replicate registrations among themselves.

**Eureka Server:**

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-netflix-eureka-server</artifactId>
</dependency>
```

```java
@SpringBootApplication
@EnableEurekaServer
public class ServiceRegistryApplication {
    public static void main(String[] args) {
        SpringApplication.run(ServiceRegistryApplication.class, args);
    }
}
```

```yaml
# application.yml — Eureka Server
server:
  port: 8761

eureka:
  instance:
    hostname: localhost
  client:
    register-with-eureka: false   # server doesn't register itself
    fetch-registry: false
  server:
    enable-self-preservation: true  # don't evict instances during network partition
    eviction-interval-timer-in-ms: 5000
```

**Eureka Client (any microservice):**

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-netflix-eureka-client</artifactId>
</dependency>
```

```java
@SpringBootApplication
// @EnableEurekaClient — optional in newer Spring Cloud, auto-configured
public class OrderServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderServiceApplication.class, args);
    }
}
```

```yaml
# application.yml — Eureka Client (Order Service)
spring:
  application:
    name: order-service    # this is the service name used for discovery

eureka:
  client:
    service-url:
      defaultZone: http://localhost:8761/eureka/
    fetch-registry: true
    register-with-eureka: true
    registry-fetch-interval-seconds: 30   # how often to refresh the local cache
  instance:
    prefer-ip-address: true               # register with IP, not hostname
    lease-renewal-interval-in-seconds: 10 # heartbeat interval
    lease-expiration-duration-in-seconds: 30 # evict if no heartbeat for this long
    instance-id: ${spring.application.name}:${server.port}:${random.uuid}
    metadata-map:
      version: "2.1.0"      # custom metadata for routing decisions
      zone: "us-east-1a"
```

**Using discovery with Spring Cloud LoadBalancer:**

```java
// @LoadBalanced RestTemplate — resolves service names via Eureka
@Configuration
public class RestConfig {
    @Bean
    @LoadBalanced
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }

    @Bean
    @LoadBalanced
    public WebClient.Builder webClientBuilder() {
        return WebClient.builder();
    }
}

@Service
@RequiredArgsConstructor
public class OrderClient {

    private final RestTemplate restTemplate;
    private final WebClient.Builder webClientBuilder;

    public OrderDto getOrder(Long id) {
        // "order-service" resolved to an actual IP:port via Eureka + load balancing
        return restTemplate.getForObject(
            "http://order-service/api/orders/{id}",
            OrderDto.class, id
        );
    }

    public Mono<OrderDto> getOrderReactive(Long id) {
        return webClientBuilder.build()
            .get()
            .uri("http://order-service/api/orders/{id}", id)
            .retrieve()
            .bodyToMono(OrderDto.class);
    }
}
```

**Eureka self-preservation mode — important behaviour:**

```
Normal operation: Eureka evicts instances that miss heartbeats.

Self-preservation (default: on): If Eureka detects it's receiving fewer
heartbeats than expected (e.g., a network partition), it stops evicting
instances — it assumes the registry itself has a connectivity problem,
not that services are actually down.

Effect: During a network partition, stale (potentially unreachable) entries
stay in the registry. Clients may get addresses that don't respond.

In production: Always combine discovery with health checks + circuit breakers
to handle stale entries gracefully.
```

### 1.2 Consul

Consul is HashiCorp's service mesh and service discovery tool. Unlike Eureka (which is a dedicated registry), Consul provides service discovery, health checking, key-value storage, and multi-datacenter support in one tool.

**Key differences from Eureka:**

- Built-in health checking (not just heartbeats — can check HTTP, TCP, script)
- Native DNS interface (services are discoverable as `order-service.service.consul`)
- Key-value store (config storage alongside discovery)
- First-class multi-datacenter support
- Written in Go — single binary, easy to run

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-consul-discovery</artifactId>
</dependency>
```

```yaml
# application.yml — Consul client
spring:
  application:
    name: order-service
  cloud:
    consul:
      host: localhost
      port: 8500
      discovery:
        register: true
        instance-id: ${spring.application.name}-${random.uuid}
        health-check-path: /actuator/health    # Consul polls this endpoint
        health-check-interval: 15s
        health-check-timeout: 5s
        health-check-critical-timeout: 30s     # deregister if critical for 30s
        tags:
          - version=2.1.0
          - zone=us-east-1a
        prefer-ip-address: true
      config:
        enabled: true                           # also use Consul for config
        prefix: config
        default-context: application
```

**Consul agent configuration (consul.hcl):**

```hcl
# Service registration via Consul config file (alternative to Spring auto-reg)

  name = "order-service"
  port = 8080
  tags = ["v2", "primary"]

  check {
    http     = "http://localhost:8080/actuator/health"
    interval = "10s"
    timeout  = "3s"
    deregister_critical_service_after = "60s"
  }
}
```

### 1.3 Eureka vs Consul

||Eureka|Consul|
|---|---|---|
|Consistency model|AP (availability)|CP (consistency)|
|Health checking|Client heartbeat|Active polling (HTTP/TCP/script)|
|DNS interface|No|Yes|
|Key-value store|No|Yes|
|Multi-datacenter|Limited|First-class|
|Service mesh|No|Yes (Connect)|
|Language|Java|Go|
|Spring Cloud support|Native|Good|
|Best for|Pure Java/Spring stacks|Polyglot, multi-DC|

**CAP theorem implication**: Eureka is AP — during a network partition, it keeps returning (possibly stale) data. Consul (with Raft consensus) is CP — it may reject reads/writes during a partition to preserve consistency. For service discovery, AP is usually preferable (stale data + circuit breaker beats total unavailability).

## 2. Server-Side Discovery

In server-side discovery, the calling service **does not query the registry directly**. It makes a request to a fixed address (load balancer or router), and the load balancer handles instance lookup and selection.

```
Service A                  Load Balancer / Router
    │                             │          Registry
    ├── request ─────────────────►│◄─────────── │
    │                             │  "where     │
    │                             │   is B?"    │
    │                             │◄─────────── │
    │                             │ picks instance
    │                             ├──► Service B instance 1
    │◄── response ────────────────┤    or B instance 2
    │                             │    or B instance 3
```

### 2.1 Kubernetes Service — the canonical server-side discovery example

In Kubernetes, a `Service` is a stable virtual IP (ClusterIP) in front of a set of pods. `kube-proxy` (or eBPF in newer setups) handles instance selection transparently.

```yaml
# Kubernetes Service — stable address for order-service pods
apiVersion: v1
kind: Service
metadata:
  name: order-service
  namespace: production
spec:
  selector:
    app: order-service        # selects pods with this label
  ports:
    - protocol: TCP
      port: 80                # service port (what callers use)
      targetPort: 8080        # container port (what pods listen on)
  type: ClusterIP             # internal only

---
# Deployment — multiple replicas behind the service
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service
spec:
  replicas: 3
  selector:
    matchLabels:
      app: order-service
  template:
    metadata:
      labels:
        app: order-service
    spec:
      containers:
        - name: order-service
          image: myrepo/order-service:2.1.0
          ports:
            - containerPort: 8080
          readinessProbe:
            httpGet:
              path: /actuator/health/readiness
              port: 8080
            initialDelaySeconds: 10
            periodSeconds: 5
          livenessProbe:
            httpGet:
              path: /actuator/health/liveness
              port: 8080
            initialDelaySeconds: 30
            periodSeconds: 10
```

```java
// Spring Boot service calling another via Kubernetes DNS
// No Eureka, no Spring Cloud LoadBalancer — just plain HTTP to the service name
@Component
public class OrderClient {

    private final WebClient webClient;

    public OrderClient(WebClient.Builder builder) {
        // "order-service" resolves to Kubernetes ClusterIP via CoreDNS
        this.webClient = builder
            .baseUrl("http://order-service")
            .build();
    }

    public Mono<OrderDto> getOrder(Long id) {
        return webClient.get()
            .uri("/api/orders/{id}", id)
            .retrieve()
            .bodyToMono(OrderDto.class);
    }
}
```

### 2.3 Client-Side vs Server-Side — trade-offs

||Client-Side|Server-Side|
|---|---|---|
|Load balancing logic|In the client|In the load balancer|
|Registry coupling|Client queries registry|Client unaware of registry|
|Infrastructure|Registry only|Registry + load balancer|
|Language support|Library needed per language|Language-agnostic|
|Advanced LB logic|In client code (flexible)|Configured in LB (limited)|
|Latency|One less hop|Extra hop through LB|
|Examples|Eureka + Ribbon/Spring LB|Kubernetes Service, AWS ALB|

## 3. Load Balancing Algorithms

### 3.1 Round Robin — default, simple

```
Requests:    1   2   3   4   5   6
Instances:   A   B   C   A   B   C
```

Distributes requests sequentially across all instances. Each instance gets an equal share.

```java
// Spring Cloud LoadBalancer — Round Robin (default)
@Configuration
public class LoadBalancerConfig {

    @Bean
    @Primary
    public ReactorLoadBalancer<ServiceInstance> roundRobinLoadBalancer(
            Environment env,
            LoadBalancerClientFactory factory) {
        String name = env.getProperty(LoadBalancerClientFactory.PROPERTY_NAME);
        return new RoundRobinLoadBalancer(
            factory.getLazyProvider(name, ServiceInstanceListSupplier.class),
            name
        );
    }
}
```

**Best for**: Homogeneous instances with similar request processing times.

---

### 3.2 Random

Selects an instance randomly on each request. In practice, converges to equal distribution over time (law of large numbers), but with more variance than round robin for short periods.

```java
@Bean
public ReactorLoadBalancer<ServiceInstance> randomLoadBalancer(
        Environment env, LoadBalancerClientFactory factory) {
    String name = env.getProperty(LoadBalancerClientFactory.PROPERTY_NAME);
    return new RandomLoadBalancer(
        factory.getLazyProvider(name, ServiceInstanceListSupplier.class),
        name
    );
}
```

**Best for**: Simple distribution when instances are homogeneous. Avoids the "thundering herd" on instance 0 that strict round-robin can cause after restarts.

---

### 3.3 Least Connections

Routes each request to the instance with the **fewest active connections**. Better than round robin when requests have variable processing times.

```
Instance A: 10 active connections
Instance B: 3 active connections  ← next request goes here
Instance C: 7 active connections
```

```java
// Custom least-connections implementation with Spring Cloud LoadBalancer
@Bean
public ReactorLoadBalancer<ServiceInstance> leastConnectionsLoadBalancer(
        Environment env, LoadBalancerClientFactory factory) {
    String name = env.getProperty(LoadBalancerClientFactory.PROPERTY_NAME);
    return new LeastConnectionsLoadBalancer(
        factory.getLazyProvider(name, ServiceInstanceListSupplier.class),
        name
    );
}

// Track active connections with AtomicInteger per instance
public class LeastConnectionsLoadBalancer implements ReactorServiceInstanceLoadBalancer {
    private final ConcurrentHashMap<String, AtomicInteger> activeConnections =
        new ConcurrentHashMap<>();

    @Override
    public Mono<Response<ServiceInstance>> choose(Request request) {
        return serviceInstanceListSupplierProvider.getIfAvailable()
            .get()
            .next()
            .map(instances -> instances.stream()
                .min(Comparator.comparingInt(i ->
                    activeConnections.computeIfAbsent(
                        i.getInstanceId(), k -> new AtomicInteger(0)).get()
                ))
                .map(DefaultResponse::new)
                .orElse(new EmptyResponse())
            );
    }
}
```

**Best for**: Services with variable request processing times (some requests are expensive, some cheap). Common in database connection pools and gRPC connection pools.

---

### 3.4 Weighted Round Robin

Assigns more traffic to instances with higher capacity (more CPU, more memory).

```
Instance A (weight 1): 10% traffic  (smaller machine)
Instance B (weight 3): 30% traffic
Instance C (weight 6): 60% traffic  (larger machine)
```

```yaml
# Weighted routing in Istio VirtualService
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: order-service
spec:
  http:
  - route:
    - destination:
        host: order-service
        subset: standard
      weight: 30
    - destination:
        host: order-service
        subset: large        # larger instances get more traffic
      weight: 70
```

**Best for**: Heterogeneous infrastructure where instances have different capacities.

---

### 3.5 IP Hash / Sticky Sessions

Routes requests from the same client IP (or session ID) to the same instance. Ensures session affinity.

```
Client 192.168.1.1 → always routes to Instance B
Client 192.168.1.2 → always routes to Instance A
Client 192.168.1.3 → always routes to Instance C
```

```yaml
# Nginx upstream — IP hash
upstream order_service {
    ip_hash;
    server 10.0.0.1:8080;
    server 10.0.0.2:8080;
    server 10.0.0.3:8080;
}

# AWS ALB — sticky sessions via cookie
# Target Group attribute: stickiness.enabled = true
# stickiness.type = lb_cookie
# stickiness.lb_cookie.duration_seconds = 86400
```

**Warning**: Sticky sessions undermine horizontal scalability and make rolling deployments harder. Prefer stateless services and distribute sessions to a shared store (Redis) instead.

---

### 3.6 Zone-Aware Load Balancing

Routes preferentially to instances in the **same availability zone** to reduce cross-zone latency and data transfer costs.

```java
// Spring Cloud LoadBalancer — zone-aware (prefers same AZ)
@Configuration
public class ZoneAwareLoadBalancerConfig {

    @Bean
    public ServiceInstanceListSupplier serviceInstanceListSupplier(
            ConfigurableApplicationContext context) {
        return ServiceInstanceListSupplier.builder()
            .withDiscoveryClient()
            .withZonePreference()    // prefer same zone
            .withHealthChecks()      // filter unhealthy
            .build(context);
    }
}
```

```yaml
# application.yml — declare this instance's zone
eureka:
  instance:
    metadata-map:
      zone: us-east-1a   # matches cloud provider AZ
spring:
  cloud:
    loadbalancer:
      zone: us-east-1a
```

### 3.7 Algorithm Comparison

|Algorithm|Even Distribution|Handles Variable Load|State Required|Best For|
|---|---|---|---|---|
|Round Robin|✓|✗|Counter|Homogeneous instances, equal request cost|
|Random|~✓|✗|None|Simple cases|
|Least Connections|~✓|✓|Connection count|Variable request duration|
|Weighted RR|Weighted|✗|Weights|Heterogeneous instances|
|IP Hash|No|✗|Hash table|Legacy stateful apps|
|Zone-Aware|✓|✗|Zone metadata|Multi-AZ deployments|

## 4. Health Checks

Health checks are how the discovery system and load balancer know which instances are ready to receive traffic. Without them, requests are routed to starting, stopping, or unhealthy instances.

### 4.1 Spring Boot Actuator Health Endpoints

```xml
</dependency>
```

```yaml
# application.yml — configure health endpoint
```

### 4.2 Liveness vs Readiness — the critical distinction

```
Liveness probe  → "Is the application alive?"
                   If fails: Kubernetes RESTARTS the pod
                   Use for: detecting deadlocks, fatal errors

Readiness probe → "Is the application ready to serve traffic?"
                   If fails: Kubernetes REMOVES pod from Service endpoints
                   (no restarts, just stops sending traffic)
                   Use for: startup, DB connection lost, circuit open
```

```java
// Custom readiness indicator — signal not ready when dependencies are down
@Component
public class DatabaseReadinessIndicator implements HealthIndicator {

    private final DataSource dataSource;

    @Override
    public Health health() {
        try (Connection conn = dataSource.getConnection()) {
            if (conn.isValid(1)) {
                return Health.up()
                    .withDetail("database", "reachable")
                    .build();
            }
        } catch (SQLException ex) {
            return Health.down()
                .withDetail("database", "unreachable")
                .withException(ex)
                .build();
        }
        return Health.down().withDetail("database", "connection invalid").build();
    }
}

// Custom liveness indicator — detect permanent failure states
@Component
public class CriticalThreadPoolLivenessIndicator implements HealthIndicator {

    private final ThreadPoolTaskExecutor executor;

    @Override
    public Health health() {
        if (executor.getThreadPoolExecutor().isTerminated()) {
            return Health.down()
                .withDetail("threadPool", "executor terminated unexpectedly")
                .build();
        }
        return Health.up().build();
    }
}
```

```yaml
# Kubernetes pod spec — liveness and readiness probes
spec:
  containers:
    - name: order-service
      image: myrepo/order-service:2.1.0
      readinessProbe:
        httpGet:
          path: /actuator/health/readiness
          port: 8080
        initialDelaySeconds: 10     # wait before first check
        periodSeconds: 5            # check every 5s
        failureThreshold: 3         # fail 3 times before marking unready
        successThreshold: 1         # one success to mark ready
      livenessProbe:
        httpGet:
          path: /actuator/health/liveness
          port: 8080
        initialDelaySeconds: 30     # give app time to start fully
        periodSeconds: 10
        failureThreshold: 3         # restart after 3 consecutive failures
      startupProbe:                 # for slow-starting apps — disables liveness until ready
        httpGet:
          path: /actuator/health/liveness
          port: 8080
        failureThreshold: 30        # 30 × 10s = 5 minute startup budget
        periodSeconds: 10
```

### 4.4 Graceful shutdown — don't fail health checks during deployment

```yaml
# application.yml
server:
  shutdown: graceful              # wait for in-flight requests to complete

spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s  # max wait time
```

```java
// Spring automatically sets readiness to REFUSING_TRAFFIC on shutdown signal
// Kubernetes sees readiness fail → removes pod from Service endpoints
// Existing connections drain (up to 30s) → pod terminates cleanly
// Result: zero-downtime rolling deployments

// Mark readiness programmatically (e.g., when circuit breaker opens)
@Component
@RequiredArgsConstructor
public class CircuitBreakerReadinessAdapter {

    private final ApplicationEventPublisher eventPublisher;

    public void onCircuitOpen(String serviceName) {
        // Signal Kubernetes to stop routing traffic while circuit is open
        eventPublisher.publishEvent(
            new AvailabilityChangeEvent<>(this, ReadinessState.REFUSING_TRAFFIC)
        );
    }

    public void onCircuitClose(String serviceName) {
        eventPublisher.publishEvent(
            new AvailabilityChangeEvent<>(this, ReadinessState.ACCEPTING_TRAFFIC)
        );
    }
}
```

## 5. Circuit Breaker Pattern

A circuit breaker prevents **cascading failures** by detecting when a downstream service is unhealthy and failing fast — returning an error or fallback immediately instead of waiting for timeouts.

### 5.1 State Machine

```
                  failure threshold exceeded
         ┌─────────────────────────────────────────┐
         │                                         ▼
    ┌────┴────┐    success threshold          ┌──────────┐
    │  CLOSED │ ──── exceeded ─────────────── │  OPEN    │
    │(normal) │ ◄────────────────────────────  │(fail fast│
    └────┬────┘   half-open probe succeeds    └────┬─────┘
         │                                         │
         └─────────────────────────────────────────┘
                    half-open probe                │
                    wait timeout ─────────────────►│
                                               ┌───▼──────┐
                                               │HALF-OPEN │
                                               │(testing) │
                                               └──────────┘

CLOSED:    Normal operation. Tracks failure rate.
OPEN:      Fast-fail all requests. No calls to downstream.
HALF-OPEN: Allow limited probe requests. If they succeed → CLOSED.
                                           If they fail  → OPEN again.
```

### 5.2 Resilience4j — Spring Boot integration

```xml
<dependency>
    <groupId>io.github.resilience4j</groupId>
    <artifactId>resilience4j-spring-boot3</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-aop</artifactId>  <!-- required for annotations -->
</dependency>
```

```yaml
# application.yml — circuit breaker configuration
resilience4j:
  circuitbreaker:
    instances:
      order-service:
        # Failure detection
        failure-rate-threshold: 50             # open if 50%+ of calls fail
        slow-call-rate-threshold: 80           # also track slow calls
        slow-call-duration-threshold: 3s       # calls slower than 3s = slow call
        minimum-number-of-calls: 10            # min calls before evaluating rate

        # Sliding window
        sliding-window-type: COUNT_BASED       # or TIME_BASED
        sliding-window-size: 20                # last 20 calls evaluated

        # OPEN state
        wait-duration-in-open-state: 30s       # stay OPEN for 30s

        # HALF-OPEN state
        permitted-number-of-calls-in-half-open-state: 5  # allow 5 probe calls
        automatic-transition-from-open-to-half-open-enabled: true

        # What counts as a failure
        record-exceptions:
          - java.io.IOException
          - java.util.concurrent.TimeoutException
          - feign.FeignException
        ignore-exceptions:
          - com.example.OrderNotFoundException  # 404s don't count as failures

  retry:
    instances:
      order-service:
        max-attempts: 3
        wait-duration: 500ms
        enable-exponential-backoff: true
        exponential-backoff-multiplier: 2
        retry-exceptions:
          - java.io.IOException
        ignore-exceptions:
          - com.example.OrderNotFoundException

  bulkhead:
    instances:
      order-service:
        max-concurrent-calls: 20       # max simultaneous calls
        max-wait-duration: 100ms       # wait for a slot before rejecting

  timelimiter:
    instances:
      order-service:
        timeout-duration: 5s
        cancel-running-future: true
```

### 5.3 Annotation-based usage

```java
@Service
@RequiredArgsConstructor
public class OrderServiceClient {

    private final WebClient webClient;

    // Circuit breaker wraps the method
    @CircuitBreaker(name = "order-service", fallbackMethod = "getOrderFallback")
    @Retry(name = "order-service")
    @TimeLimiter(name = "order-service")
    @Bulkhead(name = "order-service")
    public CompletableFuture<OrderDto> getOrder(Long id) {
        return webClient.get()
            .uri("http://order-service/api/orders/{id}", id)
            .retrieve()
            .bodyToMono(OrderDto.class)
            .toFuture();
    }

    // Fallback — same signature + Throwable parameter
    private CompletableFuture<OrderDto> getOrderFallback(Long id, Throwable ex) {
        log.warn("Circuit open for order {}, using fallback. Cause: {}", id, ex.getMessage());
        return CompletableFuture.completedFuture(OrderDto.unavailable(id));
    }

    // Fallback specifically for CallNotPermittedException (circuit is OPEN)
    private CompletableFuture<OrderDto> getOrderFallback(Long id, CallNotPermittedException ex) {
        log.warn("Order service circuit OPEN — fast failing for order {}", id);
        return CompletableFuture.completedFuture(OrderDto.unavailable(id));
    }
}
```

### 5.4 Programmatic usage — more control

```java
@Service
@RequiredArgsConstructor
public class OrderServiceClient {

    private final CircuitBreakerRegistry circuitBreakerRegistry;
    private final WebClient webClient;

    public Mono<OrderDto> getOrder(Long id) {
        CircuitBreaker cb = circuitBreakerRegistry.circuitBreaker("order-service");

        return webClient.get()
            .uri("http://order-service/api/orders/{id}", id)
            .retrieve()
            .bodyToMono(OrderDto.class)
            .transformDeferred(CircuitBreakerOperator.of(cb))
            .onErrorResume(CallNotPermittedException.class,
                ex -> Mono.just(OrderDto.unavailable(id)))
            .onErrorResume(ex -> {
                log.error("Order service call failed for {}", id, ex);
                return Mono.just(OrderDto.unavailable(id));
            });
    }

    @PostConstruct
    public void registerEventListeners() {
        CircuitBreaker cb = circuitBreakerRegistry.circuitBreaker("order-service");

        // React to state transitions — e.g., update readiness, alert
        cb.getEventPublisher()
            .onStateTransition(event -> {
                log.warn("Circuit breaker state: {} → {}",
                    event.getStateTransition().getFromState(),
                    event.getStateTransition().getToState());

                if (event.getStateTransition().getToState() == CircuitBreaker.State.OPEN) {
                    // Signal readiness to Kubernetes — stop routing traffic
                    readinessAdapter.onCircuitOpen("order-service");
                } else if (event.getStateTransition().getToState() == CircuitBreaker.State.CLOSED) {
                    readinessAdapter.onCircuitClose("order-service");
                }
            })
            .onFailureRateExceeded(event ->
                log.error("Failure rate exceeded: {}%", event.getFailureRate()))
            .onCallNotPermitted(event ->
                log.debug("Call rejected — circuit is OPEN"));
    }
}
```

### 5.5 Bulkhead — limit concurrent calls

The bulkhead pattern prevents a slow downstream service from consuming all available threads/connections.

```java
// Semaphore bulkhead — limits concurrent calls
@Bulkhead(name = "inventory-service", type = Bulkhead.Type.SEMAPHORE,
          fallbackMethod = "inventoryFallback")
public InventoryDto checkInventory(Long productId) {
    return inventoryClient.check(productId);
}

// Thread pool bulkhead — isolates on separate thread pool
@Bulkhead(name = "payment-service", type = Bulkhead.Type.THREADPOOL,
          fallbackMethod = "paymentFallback")

    return paymentClient.processAsync(request);
}
```

## 6. Interview Questions

**Q: What is the difference between client-side and server-side service discovery?**

A: In client-side discovery, the calling service queries the registry directly (e.g., Eureka), gets a list of available instances, applies a load balancing algorithm, and makes the call itself. In server-side discovery, the client makes a request to a stable address (Kubernetes Service ClusterIP, AWS ALB) and the load balancer/proxy handles instance lookup and selection transparently. Client-side gives more flexibility (custom LB logic, zone awareness) but couples every client to the registry. Server-side is simpler for clients and language-agnostic but requires additional infrastructure and offers less control over routing logic.

---

**Q: What is the difference between a liveness probe and a readiness probe in Kubernetes, and what happens when each fails?**

A: A liveness probe answers "is this container alive?" — if it fails repeatedly, Kubernetes restarts the pod. It's for detecting unrecoverable states like deadlocks or OOM situations. A readiness probe answers "is this container ready to serve traffic?" — if it fails, Kubernetes removes the pod from the Service endpoints so no new traffic is routed to it, but the pod is not restarted. Readiness is used for startup time, temporary dependency outages (DB connection lost), or when the circuit breaker opens. Confusing the two is a common mistake: using liveness for dependency checks causes unnecessary pod restarts when a downstream service is temporarily unavailable.

---

**Q: Walk through the circuit breaker state transitions and what triggers each.**

A: The circuit starts CLOSED — normal operation, tracking a rolling window of call outcomes. When the failure rate (or slow call rate) exceeds the configured threshold (e.g., 50% of the last 20 calls failed), the circuit transitions to OPEN — all subsequent calls immediately fail with `CallNotPermittedException` without reaching the downstream service. After the configured `waitDurationInOpenState` (e.g., 30 seconds), it transitions to HALF-OPEN and allows a limited number of probe calls through. If those probes succeed at or above the success threshold, the circuit returns to CLOSED. If they fail, it returns to OPEN and the wait period resets.

---

**Q: Why is round robin load balancing suboptimal when request processing times vary significantly?**

A: Round robin distributes requests evenly by count, not by load. If some requests take 1ms and others take 500ms, a slow request on instance A means that instance accumulates a backlog while B and C continue receiving new requests at the same rate. Least connections is better for variable workloads — it sends the next request to whichever instance has the fewest active connections, naturally routing to instances that are processing faster. This is why database connection pools and gRPC load balancers typically use least connections rather than round robin.

---

**Q: What is Eureka's self-preservation mode and when can it cause problems?**

A: Self-preservation activates when Eureka detects that it's receiving fewer heartbeats than expected — which could mean services are down, but could also mean a network partition between the Eureka server and its clients. Rather than evicting all instances (which would be catastrophic if the registry itself has a connectivity problem), it stops evictions and keeps the existing registry. The problem occurs when instances genuinely are down but self-preservation prevents their eviction — clients get stale addresses and waste time on connections that fail. The mitigation is to combine service discovery with health checks, short timeouts, and circuit breakers so that even if a stale address is returned, the circuit breaker catches repeated failures quickly.

---

**Q: What is the bulkhead pattern and how does it complement the circuit breaker?**

A: The circuit breaker detects failure rates and stops calls to an unhealthy downstream service. The bulkhead limits the number of concurrent calls to a downstream service to prevent it from consuming all available threads or connections in the calling service. Without a bulkhead, a slow downstream service could saturate the thread pool — all threads blocked waiting for responses, causing the calling service to stop responding to all requests, not just the ones involving the slow dependency. Together, the bulkhead limits blast radius (slow dependency can only use N threads), and the circuit breaker stops calls once failures accumulate, allowing the thread pool to recover.

---

## 7. Key Takeaways

- **Client-side discovery** (Eureka, Consul) gives clients direct registry access and flexible LB logic; **server-side** (Kubernetes Service, ALB) is simpler for clients and language-agnostic
- **Least connections** outperforms round robin when requests have variable processing times; **zone-aware** reduces latency and cross-AZ costs
- **Liveness** failures restart pods (use sparingly); **readiness** failures remove pods from load balancing (use for temporary conditions) — confusing them causes unnecessary restarts
- **Circuit breaker** prevents cascading failures: CLOSED → OPEN on threshold breach → HALF-OPEN after wait → CLOSED on probe success
- **Bulkhead** limits a slow dependency's blast radius by capping concurrent calls — complements the circuit breaker, which responds to failure rate
- In Kubernetes, **Istio handles infrastructure-level resilience** (outlier detection, connection limits) and **Resilience4j handles application-level fallbacks** — both layers are appropriate
