# Project Reactor Fundamentals

## Overview

Project Reactor is the **reactive library at the core of Spring WebFlux**. It implements the Reactive Streams specification and provides two primary types — `Mono` and `Flux` — along with a rich operator API for building non-blocking, backpressure-aware pipelines.

**Key dependency:**

```xml
<dependency>
    <groupId>io.projectreactor</groupId>
    <artifactId>reactor-core</artifactId>
    <!-- Managed by Spring Boot BOM — no version needed in Spring projects -->
</dependency>
```

**Mental model**: Think of `Mono` and `Flux` as lazy, composable descriptions of an asynchronous computation. Nothing executes until something subscribes.

---

## 1. Mono vs Flux

### 1.1 Mono — 0 or 1 item

```
Mono: ──●──|     (emits one item, then completes)
      ──|        (empty — completes without item)
      ──✕        (error)
```

```java
public interface Mono<T> // Publisher<T> that emits at most 1 item
```

**When to use Mono:**

- HTTP response (you get one response)
- Finding a single record by ID
- Save/update operations (return saved entity or void)
- Any computation with a single result

```java
Mono<User> user = userRepository.findById(id);   // 0 or 1 user
Mono<Void> delete = userRepository.deleteById(id); // just signals completion
Mono<String> value = Mono.just("hello");
Mono<String> empty = Mono.empty();
```

### 1.2 Flux — 0 to N items

```
Flux: ──●──●──●──●──|     (emits multiple items, then completes)
      ──|              (empty)
      ──●──●──✕         (error after some items)
```

```java
public interface Flux<T> // Publisher<T> that emits 0..N items
```

**When to use Flux:**

- Collections / lists of results
- Streaming data (events, messages)
- Paginated results
- Any multi-value source

```java
Flux<User> users = userRepository.findAll();         // all users
Flux<String> words = Flux.just("a", "b", "c");
Flux<Integer> range = Flux.range(1, 10);             // 1..10
Flux<Long> ticks = Flux.interval(Duration.ofSeconds(1)); // 0, 1, 2, 3... every second
```

### 1.3 Mono vs Flux — Key Differences

||MonoT|FluxT|
|---|---|---|
|Items|0 or 1|0 to N|
|Use case|Single value / void|Collection / stream|
|Backpressure|Not meaningful (≤1 item)|Full backpressure support|
|`collectList()`|N/A|`Flux<T>` → `Mono<List<T>>`|
|`flatMapMany()`|`Mono<T>` → `Flux<R>`|N/A|

### 1.4 Converting Between Them

```java
// Flux → Mono
Mono<List<String>> list = Flux.just("a","b","c").collectList();
Mono<String> first = Flux.just("a","b","c").next();
Mono<Long> count = Flux.range(1,100).count();

// Mono → Flux
Flux<String> flux = Mono.just("hello").flux();
Flux<String> expanded = Mono.just("hello").flatMapMany(s -> Flux.just(s.split("")));
```

## 2. Creating Publishers

### 2.1 Factory Methods

```java
// ── Mono ──────────────────────────────────────────────────────────────────

// From a known value
Mono<String> m1 = Mono.just("hello");

// Empty — completes without emitting
Mono<String> m2 = Mono.empty();

// Error — immediately signals onError
Mono<String> m3 = Mono.error(new RuntimeException("oops"));

// From a supplier (lazy — evaluated on subscribe)
Mono<String> m4 = Mono.fromSupplier(() -> expensiveComputation());

// From a Callable
Mono<String> m5 = Mono.fromCallable(() -> blockingCall()); // use with subscribeOn!

// From a CompletableFuture
Mono<String> m6 = Mono.fromFuture(CompletableFuture.supplyAsync(() -> "async"));

// From a Runnable (emits nothing, just signals completion)
Mono<Void> m7 = Mono.fromRunnable(() -> System.out.println("done"));

// Deferred — factory runs per subscriber (ensures cold behavior)
Mono<Long> m8 = Mono.defer(() -> Mono.just(System.currentTimeMillis()));


// ── Flux ──────────────────────────────────────────────────────────────────

// From varargs
Flux<String> f1 = Flux.just("a", "b", "c");

// From iterable
Flux<String> f2 = Flux.fromIterable(List.of("x", "y", "z"));

// From array
Flux<Integer> f3 = Flux.fromArray(new Integer[]{1, 2, 3});

// From stream (consumed once — use defer for repeatability)
Flux<String> f4 = Flux.fromStream(Stream.of("p", "q"));

// Integer range
Flux<Integer> f5 = Flux.range(1, 5); // 1, 2, 3, 4, 5

// Periodic ticks
Flux<Long> f6 = Flux.interval(Duration.ofMillis(500)); // 0, 1, 2... every 500ms

// Empty and error
Flux<String> f7 = Flux.empty();
Flux<String> f8 = Flux.error(new RuntimeException("fail"));

// Deferred
Flux<String> f9 = Flux.defer(() -> Flux.fromIterable(fetchFromDB()));
```

### 2.2 Programmatic Creation

```java
// Flux.create — full control, multi-threaded safe sink
Flux<String> created = Flux.create(sink -> {
    // Can call sink.next() from any thread
    eventBus.register(event -> {
        if (event.isEnd()) sink.complete();
        else sink.next(event.getData());
    });
}, FluxSink.OverflowStrategy.BUFFER);

// Flux.generate — synchronous, one item at a time, with state
Flux<String> generated = Flux.generate(
    () -> 0,                          // initial state
    (state, sink) -> {
        sink.next("item-" + state);
        if (state == 9) sink.complete();
        return state + 1;             // next state
    }
);

// Mono.create — async, single item
Mono<String> monoCreated = Mono.create(sink -> {
    asyncOperation(result -> sink.success(result),
                   error -> sink.error(error));
});
```

### 2.3 Using Sinks (Hot publishers, Reactor 3.4+)

```java
// Unicast — one subscriber only
Sinks.One<String> one = Sinks.one();
one.tryEmitValue("result");
Mono<String> mono = one.asMono();

// Many — multicast to multiple subscribers
Sinks.Many<String> many = Sinks.many().multicast().onBackpressureBuffer();
many.tryEmitNext("event-1");
many.tryEmitNext("event-2");
Flux<String> flux = many.asFlux();
```

## 3. Transforming Operators

### 3.1 map — synchronous 1:1 transformation

```java
// Transforms each item synchronously
Flux<String> upper = Flux.just("hello", "world")
    .map(String::toUpperCase); // HELLO, WORLD

Mono<Integer> length = Mono.just("reactor")
    .map(String::length); // 7
```

```
Input:  ──●──────●──────●──|
           "a"   "b"   "c"
           map(s → s.toUpperCase())
Output: ──●──────●──────●──|
           "A"   "B"   "C"
```

**Rule**: `map` must be synchronous and non-blocking. For blocking or async work, use `flatMap`.

---

### 3.2 flatMap — async 1:many transformation

```java
// Maps each item to a Publisher, subscribes concurrently, merges results
Flux<User> users = Flux.just(1L, 2L, 3L)
    .flatMap(id -> userRepository.findById(id)); // each returns Mono<User>

// Spring WebFlux HTTP call per item
Flux<String> results = Flux.just("url1", "url2", "url3")
    .flatMap(url -> webClient.get().uri(url).retrieve().bodyToMono(String.class));
```

```
Input:  ──●──────────●──────────●──|
           1          2          3
           flatMap(id → findById(id))

Sub1:   ────────────────●──|           (id=1, responds late)
Sub2:   ────────●──|                   (id=2, responds fast)
Sub3:   ──────────────────────●──|     (id=3, responds latest)

Output: ────────●──────────●──●──|
                 2          1   3     (order NOT guaranteed)
```

**Key behaviors:**

- Subscribes to all inner publishers **concurrently** (default: 256 concurrent)
- Results arrive in **completion order**, not input order
- Use `flatMap(mapper, concurrency)` to limit concurrent subscriptions

```java
// Limit concurrency to 4 simultaneous requests
Flux.range(1, 100)
    .flatMap(id -> fetchUser(id), 4); // max 4 concurrent
```

---

### 3.3 concatMap — sequential flatMap

```java
// Like flatMap but subscribes to inner publishers ONE AT A TIME, in order
Flux<User> users = Flux.just(1L, 2L, 3L)
    .concatMap(id -> userRepository.findById(id));
```

```
Input:  ──●──────────●──────────●──|
           1          2          3

Output: ───────●──────────●──────────●──|
               1           2          3   (order guaranteed)
```

**flatMap vs concatMap:**

||flatMap|concatMap|
|---|---|---|
|Concurrency|Concurrent (up to 256)|Sequential (1 at a time)|
|Order|Not guaranteed|Guaranteed|
|Performance|Higher throughput|Lower throughput|
|Use when|Order doesn't matter|Order matters|

---

### 3.4 filter — keep matching items

```java
Flux<Integer> evens = Flux.range(1, 10)
    .filter(n -> n % 2 == 0); // 2, 4, 6, 8, 10

// Filter with transformation (filterWhen for async predicate)
Flux<User> activeUsers = userFlux
    .filter(user -> user.isActive());

// Async predicate
Flux<User> verified = userFlux
    .filterWhen(user -> isVerified(user)); // isVerified returns Mono<Boolean>
```

---

### 3.5 Other Essential Transforming Operators

```java
// flatMapMany — Mono<T> to Flux<R>
Flux<String> words = Mono.just("hello world")
    .flatMapMany(s -> Flux.fromArray(s.split(" ")));

// switchMap — cancels previous inner publisher on new item (latest wins)
// Ideal for search-as-you-type
Flux<SearchResult> results = searchInput
    .switchMap(query -> searchService.search(query));

// take / takeLast
Flux<Integer> first3 = Flux.range(1, 100).take(3);       // 1, 2, 3
Flux<Integer> last3 = Flux.range(1, 100).takeLast(3);    // 98, 99, 100

// takeUntil / takeWhile
Flux<Integer> untilTen = Flux.range(1, 100).takeWhile(n -> n < 10);

// skip / skipLast
Flux<Integer> skip3 = Flux.range(1, 10).skip(3);         // 4..10

// distinct
Flux<String> unique = Flux.just("a","b","a","c","b").distinct(); // a, b, c

// sort (buffers all items — careful with infinite streams)
Flux<Integer> sorted = Flux.just(3,1,4,1,5).sort();

// reduce — aggregate to single value
Mono<Integer> sum = Flux.range(1, 5).reduce(0, Integer::sum); // 15

// scan — like reduce but emits intermediate results
Flux<Integer> running = Flux.range(1, 5).scan(0, Integer::sum); // 0,1,3,6,10,15

// collectList, collectMap
Mono<List<Integer>> list = Flux.range(1, 5).collectList();
Mono<Map<Integer,String>> map = userFlux.collectMap(User::getId, User::getName);

// defaultIfEmpty — emit fallback if stream is empty
Mono<User> user = userRepository.findById(id)
    .defaultIfEmpty(User.guest());

// switchIfEmpty — switch to another publisher if empty
Mono<User> user2 = userRepository.findById(id)
    .switchIfEmpty(Mono.error(new UserNotFoundException(id)));
```

## 4. Combining Operators

### 4.1 zip — combine items pairwise

Waits for all sources to emit, then combines the nth item from each.

```java
// Combine two Monos
Mono<String> name = Mono.just("John");
Mono<Integer> age = Mono.just(30);

Mono<String> combined = Mono.zip(name, age)
    .map(tuple -> tuple.getT1() + " is " + tuple.getT2()); // "John is 30"

// Zip up to 8 sources
Mono<Tuple3<A, B, C>> triple = Mono.zip(monoA, monoB, monoC);

// Flux zip — pairs items by index
Flux<String> zipped = Flux.zip(
    Flux.just("A", "B", "C"),
    Flux.just(1, 2, 3),
    (letter, number) -> letter + number  // A1, B2, C3
);
```

```
Stream1: ──A──────B──────C──|
Stream2: ──1──────2──────3──|
Zipped:  ──A1─────B2─────C3─|
```

**Key point**: zip completes when the **shortest** source completes.

---

### 4.2 merge — interleave items as they arrive

```java
Flux<String> stream1 = Flux.just("A", "B", "C").delayElements(Duration.ofMillis(100));
Flux<String> stream2 = Flux.just("1", "2", "3").delayElements(Duration.ofMillis(150));

Flux<String> merged = Flux.merge(stream1, stream2);
// Output order depends on timing: A, 1, B, 2, C, 3 (approximately)
```

```
Stream1: ──A────B────C────|
Stream2: ──────1────2────3────|
Merged:  ──A───1─B──2─C──3────|   (interleaved as they arrive)
```

**merge vs mergeWith:**

```java
// Same result
Flux<String> r1 = Flux.merge(stream1, stream2);
Flux<String> r2 = stream1.mergeWith(stream2);
```

---

### 4.3 concat — sequential, in order

```java
Flux<String> first = Flux.just("A", "B", "C");
Flux<String> second = Flux.just("1", "2", "3");

Flux<String> concatenated = Flux.concat(first, second);
// A, B, C, 1, 2, 3 — always in order
```

```
Stream1: ──A──B──C──|
Stream2:              ──1──2──3──|
Concat:  ──A──B──C────1──2──3──|   (stream2 starts only after stream1 completes)
```

---

### 4.4 combineLatest — re-emit when any source emits

```java
// Emits a new combined value whenever ANY source emits
// Uses the latest value from each source
Flux<String> combined = Flux.combineLatest(
    stockPriceA,
    stockPriceB,
    (a, b) -> "A=" + a + " B=" + b
);
// Useful for dashboards, form validation, any "latest state" scenario
```

```
Stream1: ──1──────────3──────────|
Stream2: ──────2───────────4─────|
Combined:──────12────32────34────|   (latest from each when either emits)
```

---

### 4.5 zipWhen / zipWith

```java
// zipWith — inline zip with another publisher
Mono<String> result = Mono.just("userId")
    .zipWith(Mono.just("roleId"))
    .map(tuple -> tuple.getT1() + ":" + tuple.getT2());

// zipWhen — zip with a publisher derived from current item
Mono<UserDetails> details = userRepository.findById(id)
    .zipWhen(user -> roleRepository.findByUserId(user.getId()))
    .map(tuple -> new UserDetails(tuple.getT1(), tuple.getT2()));
```

---

### 4.6 merge vs concat vs zip — Quick Reference

|Operator|Concurrency|Order|Completes when|
|---|---|---|---|
|`merge`|Concurrent|Arrival order|All sources complete|
|`concat`|Sequential|Source order|All sources complete|
|`zip`|Concurrent|Paired by index|Shortest source completes|
|`combineLatest`|Concurrent|Latest on any emit|All sources complete|

## 5. Error Handling Operators

### 5.1 onErrorReturn — fallback value

```java
Mono<User> user = userRepository.findById(id)
    .onErrorReturn(User.empty()); // return empty user on any error

// Only handle specific exception
Mono<User> user2 = userRepository.findById(id)
    .onErrorReturn(NotFoundException.class, User.empty());
```

---

### 5.2 onErrorResume — fallback publisher

```java
// Switch to another publisher on error
Mono<User> user = userRepository.findById(id)
    .onErrorResume(NotFoundException.class, ex ->
        cacheRepository.findById(id)); // try cache on not found

// Rethrow as different exception
Mono<User> user2 = userRepository.findById(id)
    .onErrorResume(DataAccessException.class, ex ->
        Mono.error(new ServiceException("DB unavailable", ex)));
```

---

### 5.3 onErrorMap — transform the error

```java
// Translate infrastructure exceptions to domain exceptions
Mono<User> user = userRepository.findById(id)
    .onErrorMap(DataAccessException.class,
        ex -> new ServiceException("Database error", ex));
```

---

### 5.4 doOnError — side effects on error (doesn't handle it)

```java
Mono<User> user = userRepository.findById(id)
    .doOnError(ex -> log.error("Failed to find user {}: {}", id, ex.getMessage()));
    // Error still propagates — use onErrorResume/Return to handle it
```

---

### 5.5 retry and retryWhen

```java
// Retry immediately up to 3 times
Mono<String> result = webClient.get().uri("/api")
    .retrieve()
    .bodyToMono(String.class)
    .retry(3);

// Retry with exponential backoff
Mono<String> result2 = webClient.get().uri("/api")
    .retrieve()
    .bodyToMono(String.class)
    .retryWhen(Retry.backoff(3, Duration.ofSeconds(1))
        .maxBackoff(Duration.ofSeconds(10))
        .filter(ex -> ex instanceof WebClientResponseException));
```

---

### 5.6 timeout

```java
Mono<User> user = userRepository.findById(id)
    .timeout(Duration.ofSeconds(5))
    .onErrorReturn(TimeoutException.class, User.empty());

// Fallback publisher on timeout
Mono<User> user2 = userRepository.findById(id)
    .timeout(Duration.ofSeconds(5), Mono.just(User.cached()));
```

---

### 5.7 Error Handling Chain Order

```java
// Full error handling pipeline
userRepository.findById(id)
    .timeout(Duration.ofSeconds(5))         // 1. Timeout guard
    .retryWhen(Retry.backoff(2, Duration.ofMillis(500))) // 2. Retry on failure
    .onErrorMap(TimeoutException.class,
        ex -> new ServiceUnavailableException()) // 3. Map remaining errors
    .onErrorResume(NotFoundException.class,
        ex -> Mono.just(User.guest()))            // 4. Fallback for specific error
    .doOnError(ex -> log.error("Unhandled: ", ex)); // 5. Log any surviving error
```

## 6. Schedulers and Threading

By default, Reactor is thread-agnostic — operators run on whatever thread subscribes or emits. Schedulers let you control **which thread pool** work runs on.

### 6.1 Built-in Schedulers

```java
// Schedulers.immediate() — current thread (default, no switch)
Schedulers.immediate();

// Schedulers.single() — single reusable thread (serial tasks)
Schedulers.single();

// Schedulers.boundedElastic() — elastic thread pool for blocking I/O
// Grows on demand, idles threads are reclaimed, max = 10 × CPU cores
Schedulers.boundedElastic();

// Schedulers.parallel() — fixed thread pool for CPU-bound work
// Size = number of CPU cores
Schedulers.parallel();

// Schedulers.fromExecutorService(exec) — wrap your own executor
Schedulers.fromExecutorService(Executors.newFixedThreadPool(4));
```

### 6.2 subscribeOn — where subscription and source run

```java
// Move the source (and everything upstream) to another thread
Mono<String> result = Mono.fromCallable(() -> blockingDatabaseCall())
    .subscribeOn(Schedulers.boundedElastic()); // DB call runs on boundedElastic

// subscribeOn affects the source — only the FIRST subscribeOn matters
Flux.range(1, 10)
    .subscribeOn(Schedulers.parallel())    // this one takes effect
    .map(n -> n * 2)
    .subscribeOn(Schedulers.single())      // this is IGNORED
    .subscribe(System.out::println);
```

```
subscribe() call
    │
    ▼ subscribeOn(boundedElastic)
[boundedElastic thread]
    │
    ▼ source.subscribe() — DB call runs here
    ▼ onNext emissions travel back downstream
```

---

### 6.3 publishOn — where downstream operators run

```java
// Switch thread for operators BELOW this point
Flux.range(1, 10)
    .map(n -> n * 2)                        // runs on subscribe thread
    .publishOn(Schedulers.parallel())       // switch here
    .map(n -> "item-" + n)                  // runs on parallel thread
    .publishOn(Schedulers.boundedElastic()) // switch again
    .subscribe(System.out::println);        // runs on boundedElastic
```

```
map(n → n*2) ─── [subscribe thread]
publishOn(parallel) ─── SWITCH
map(n → "item-"+n) ─── [parallel thread]
publishOn(boundedElastic) ─── SWITCH
subscribe ─── [boundedElastic thread]
```

---

### 6.4 subscribeOn vs publishOn

||subscribeOn|publishOn|
|---|---|---|
|Affects|Source + all upstream|All operators downstream|
|Position matters|No — only first one applies|Yes — affects everything below|
|Use for|Lifting blocking source off calling thread|Switching CPU vs I/O threads mid-pipeline|
|Typical use|Wrapping blocking calls|CPU-intensive transformation after I/O|

---

### 6.5 Common Threading Patterns

**Pattern 1: Blocking I/O wrapped in reactive**

```java
// WRONG — blocks a reactive thread
Mono<User> bad = Mono.just(jdbcTemplate.queryForObject(...));

// CORRECT — offload to boundedElastic
Mono<User> good = Mono.fromCallable(() -> jdbcTemplate.queryForObject(...))
    .subscribeOn(Schedulers.boundedElastic());
```

**Pattern 2: CPU-bound work on parallel scheduler**

```java
Flux.fromIterable(largeList)
    .publishOn(Schedulers.parallel())
    .map(item -> cpuIntensiveTransform(item)) // spread across CPU cores
    .subscribe(result -> saveResult(result));
```

**Pattern 3: Spring WebFlux — don't block on Netty threads**

```java
@GetMapping("/users/{id}")
public Mono<User> getUser(@PathVariable Long id) {
    // Good — stays reactive all the way
    return userRepository.findById(id); // R2DBC reactive repo

    // If you MUST use blocking code:
    return Mono.fromCallable(() -> jdbcUserRepository.findById(id))
               .subscribeOn(Schedulers.boundedElastic());
}
```

**Pattern 4: parallel() for data parallelism**

```java
Flux.range(1, 10)
    .parallel()                           // split into rails
    .runOn(Schedulers.parallel())         // each rail runs on parallel thread
    .map(n -> heavyComputation(n))
    .sequential()                         // merge back to single Flux
    .subscribe(System.out::println);
```

---

### 6.6 doOn* Hooks (side effects without modifying the stream)

```java
Flux.range(1, 5)
    .doOnSubscribe(sub -> log.info("Subscribed"))
    .doOnNext(n -> log.info("Item: {}", n))
    .doOnError(ex -> log.error("Error: ", ex))
    .doOnComplete(() -> log.info("Completed"))
    .doOnTerminate(() -> log.info("Terminated (complete or error)"))
    .doFinally(signalType -> log.info("Finally: {}", signalType)) // always runs
    .subscribe();
```

## 7. Interview Questions

**Q: What is the difference between `map` and `flatMap` in Reactor?**

A: `map` is a synchronous, 1:1 transformation — it takes an item and returns a transformed item directly. `flatMap` is for async 1:N transformations — it takes an item, returns a `Publisher<R>`, subscribes to all inner publishers concurrently, and merges their outputs. Use `map` for pure in-memory transformations; use `flatMap` when the transformation involves I/O or returns a reactive type.

---

**Q: What's the difference between `subscribeOn` and `publishOn`?**

A: `subscribeOn` affects where the subscription and source emission happens — it lifts the entire upstream onto a scheduler, and only the first `subscribeOn` in a chain matters. `publishOn` affects where operators downstream of it execute — it inserts an asynchronous boundary and can appear multiple times in a chain. Use `subscribeOn` to offload a blocking source; use `publishOn` to switch between thread pools mid-pipeline.

---

**Q: When would you use `boundedElastic` vs `parallel` scheduler?**

A: `boundedElastic` is for blocking I/O — wrapping JDBC calls, legacy blocking APIs, or file I/O. It uses an elastic thread pool that grows as needed. `parallel` is for CPU-bound, non-blocking work — it has a fixed thread pool sized to the number of CPU cores. Blocking a `parallel` thread starves the CPU pool and is a common bug in Spring WebFlux.

---

**Q: Why does `Flux.fromStream()` cause issues if subscribed multiple times?**

A: Java `Stream` is a one-shot data structure — once consumed, it cannot be re-read. `Flux.fromStream(stream)` is cold in that it starts on subscribe, but uses the same underlying stream. A second subscriber will get an `IllegalStateException` because the stream is already consumed. Use `Flux.defer(() -> Flux.fromStream(stream))` to create a fresh stream per subscriber.

---

**Q: What is the difference between `merge`, `concat`, and `zip`?**

A: `merge` subscribes to all sources concurrently and emits items in arrival order — no ordering guarantee. `concat` subscribes to sources sequentially, waiting for each to complete before starting the next — preserves source order. `zip` pairs the nth item from each source and emits a combined value — stops when the shortest source completes. Use `merge` for throughput, `concat` for ordering, `zip` for combining parallel computations.

---

**Q: What happens if you don't call `subscribe()` on a Flux or Mono?**

A: Nothing executes. Reactor pipelines are **lazy** — they are a description of computation, not the computation itself. Without a subscriber (or a blocking call like `block()`), no data flows, no operators run, no side effects occur. This is a common source of confusion for developers coming from imperative Spring MVC — they return a `Mono` from a service call expecting it to execute, but it only executes when Spring WebFlux (or a test) subscribes to it.

---

**Q: What is `switchIfEmpty` and when would you use it over `defaultIfEmpty`?**

A: `defaultIfEmpty(value)` emits a fallback value when the source is empty, but the value must already exist. `switchIfEmpty(publisher)` switches to an entirely different publisher when the source is empty — the fallback can itself be async, do a database lookup, or throw a specific exception. Use `defaultIfEmpty` for static fallback values; use `switchIfEmpty` when the fallback involves computation or a specific error signal (e.g., `Mono.error(new NotFoundException())`).

---

## 8. Key Takeaways

- `Mono` = 0..1 items; `Flux` = 0..N items — pick based on expected cardinality
- Reactor is **lazy** — nothing runs until subscribe
- `map` is synchronous; `flatMap` is async and concurrent; `concatMap` is async and sequential
- `merge` is concurrent/unordered; `concat` is sequential/ordered; `zip` is paired/concurrent
- `subscribeOn` moves the source; `publishOn` moves the downstream
- Use `boundedElastic` for blocking I/O, `parallel` for CPU work — never block on `parallel`
- Always handle errors — unhandled errors in WebFlux result in dropped signals
