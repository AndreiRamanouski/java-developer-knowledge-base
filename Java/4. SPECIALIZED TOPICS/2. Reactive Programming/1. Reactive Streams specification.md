# Reactive Streams Specification

## Overview

Reactive Streams is a specification (not a library) that defines a standard for **asynchronous stream processing with non-blocking backpressure**. It was created to solve a fundamental problem: how do you handle a fast producer and a slow consumer without losing data or running out of memory?

**Key insight**: Reactive Streams flips the traditional push model вЂ” instead of a producer pushing data as fast as possible, the consumer _requests_ how much it can handle.

**Official spec**: [reactive-streams.org](https://www.reactive-streams.org/)  
**Java package**: `org.reactivestreams`  
**Java 9+**: Adopted as `java.util.concurrent.Flow` (identical interfaces)

---

## 1. The Four Core Interfaces

The entire spec is just **4 interfaces**. Everything else (Project Reactor, RxJava, Akka Streams) is an implementation of these contracts.

```

              (via Subscription)

                    в”‚ request(n) / cancel()

```

---

### 1.1 Publisher

```java
public interface Publisher<T> {
    void subscribe(Subscriber<? super T> subscriber);
}
```

**Responsibilities:**

- Accepts subscribers
- Emits items to subscribers **only when requested**
- Must not emit more items than requested (`request(n)`)
- Must signal `onComplete()` or `onError()` when done

**Rules (from spec):**

- `onSubscribe` must be called exactly once per `subscribe()`
- Total `onNext` calls must be в‰¤ total requested via `request(n)`
- `onError` and `onComplete` are terminal вЂ” nothing comes after
- If subscriber cancels, publisher _should_ stop (best effort)

**Real implementations:**

- `Flux<T>` in Project Reactor (0..N elements)
- `Observable<T>` / `Flowable<T>` in RxJava 2/3
- `java.util.concurrent.Flow.Publisher`

---

### 1.2 Subscriber

```java
public interface Subscriber<T> {
    void onSubscribe(Subscription s);   // always called first
    void onNext(T t);                   // called for each item
    void onError(Throwable t);          // terminal: stream failed
    void onComplete();                  // terminal: stream finished
}
```

**Lifecycle:**

```
subscribe()
    в”‚
    в–ј
onSubscribe(subscription)    в†ђ Subscriber receives handle to control flow
    в”‚
    в–ј
subscription.request(n)      в†ђ Subscriber requests n items
    в”‚
    в–ј
onNext(item) Г— n             в†ђ Publisher sends up to n items
    в”‚
    в–ј
subscription.request(m)      в†ђ Subscriber can request more
    в”‚
    в–ј
onComplete() or onError()    в†ђ Stream ends
```

**Example вЂ” manual subscriber:**

```java
publisher.subscribe(new Subscriber<String>() {
    private Subscription subscription;

    @Override
    public void onSubscribe(Subscription s) {
        this.subscription = s;
        s.request(1); // request first item
    }

    @Override
    public void onNext(String item) {
        System.out.println("Received: " + item);
        subscription.request(1); // request next item
    }

    @Override
    public void onError(Throwable t) {
        System.err.println("Error: " + t.getMessage());
    }

    @Override
    public void onComplete() {
        System.out.println("Done!");
    }
});
```

### 1.3 Subscription

```java
public interface Subscription {
    void request(long n);   // demand signal вЂ” request n more items
    void cancel();          // tell publisher to stop
}
```

**The backpressure mechanism lives here.**

- `request(n)` is additive: calling `request(3)` then `request(5)` means 8 total items can be sent
- `request(Long.MAX_VALUE)` вЂ” effectively "unbounded" (disables backpressure)
- `cancel()` вЂ” subscriber is no longer interested; publisher should stop ASAP

**Rules:**

- `request(0)` or negative `n` в†’ publisher must signal `onError(IllegalArgumentException)`
- After `cancel()`, `onNext` may still arrive (in-flight items)
- Subscription is **1:1** between a Publisher and Subscriber pair

---

### 1.4 Processor

```java
public interface Processor<T, R> extends Subscriber<T>, Publisher<R> {
}
```

A Processor is both a Subscriber _and_ a Publisher вЂ” it sits in the middle of a pipeline, consuming items of type `T` and emitting items of type `R`.

**Use cases:**

- Transforming items (map, filter)
- Buffering between a fast producer and slow consumer
- Splitting or merging streams

**Warning from spec authors**: Processors are hard to implement correctly. Prefer using operator chains (`map`, `flatMap`, etc.) over creating custom Processors. In Project Reactor, use `Sinks` instead.

```java
// Reactor's Sinks replaced direct Processor usage
Sinks.Many<String> sink = Sinks.many().multicast().onBackpressureBuffer();
Flux<String> flux = sink.asFlux();

sink.tryEmitNext("hello");
sink.tryEmitNext("world");
sink.tryEmitComplete();
```

## 2. Backpressure Handling

Backpressure is the mechanism that allows a **slow consumer to signal a fast producer to slow down**.

### 2.1 The Problem Without Backpressure

```
Producer: 1,000,000 items/sec
Consumer:         10 items/sec
в†’ Buffer fills в†’ OutOfMemoryError
```

Traditional event listeners, callbacks, and iterators have no standard way to handle this.

### 2.2 How Reactive Streams Solves It

```
Consumer says "give me 10"
Producer sends 10
Consumer processes 10, says "give me 10 more"
Producer sends 10
...
```

The producer **never sends more than requested** вЂ” the consumer controls the pace.

### 2.3 Backpressure Strategies (Project Reactor)

When you can't slow down the upstream (e.g., it's a hot source like a UI event stream), you need a strategy for overflow:

```java
// DROP вЂ” discard items when consumer can't keep up
Flux.create(sink -> { /* emit fast */ }, FluxSink.OverflowStrategy.DROP)

// BUFFER вЂ” buffer all items (careful: can OOM)
Flux.create(sink -> { /* emit fast */ }, FluxSink.OverflowStrategy.BUFFER)

// LATEST вЂ” keep only the most recent item
Flux.create(sink -> { /* emit fast */ }, FluxSink.OverflowStrategy.LATEST)

// ERROR вЂ” throw exception when buffer overflows
Flux.create(sink -> { /* emit fast */ }, FluxSink.OverflowStrategy.ERROR)
```

**Using operators:**

```java
Flux<Integer> fast = Flux.range(1, 1_000_000);

// Sample: only take latest value in a time window
fast.sample(Duration.ofMillis(100))
    .subscribe(System.out::println);

// onBackpressureBuffer with max size
fast.onBackpressureBuffer(100)
    .subscribe(System.out::println);

// onBackpressureDrop with dropped item handler
fast.onBackpressureDrop(dropped -> log.warn("Dropped: {}", dropped))
    .subscribe(System.out::println);
```

## 3. Cold vs Hot Publishers

### 3.1 Cold Publishers

A cold publisher **starts fresh for each subscriber**. Data generation begins at subscription time, and each subscriber gets the **full sequence independently**.

```
Subscriber A subscribes в†’ gets items 1, 2, 3, 4, 5
Subscriber B subscribes в†’ gets items 1, 2, 3, 4, 5 (independent stream)
```

**Analogy**: A Netflix movie вЂ” each viewer starts from the beginning.

**Examples:**

- HTTP calls (`WebClient.get()...`)
- Database queries
- `Flux.fromIterable(list)`
- `Flux.range(1, 10)`

```java
// Cold вЂ” each subscriber triggers a separate HTTP call
Flux<String> coldFlux = Flux.defer(() ->
    webClient.get().uri("/api/data").retrieve().bodyToFlux(String.class)
);

coldFlux.subscribe(s -> System.out.println("Sub1: " + s));
coldFlux.subscribe(s -> System.out.println("Sub2: " + s)); // separate HTTP call
```

### 3.2 Hot Publishers

A hot publisher **emits regardless of subscribers**. Subscribers receive only items emitted **after they subscribe** вЂ” they miss earlier items.

```
Publisher emits: 1, 2, 3
                          в†ђ Subscriber A joins here
Publisher emits: 4, 5, 6
Subscriber A receives:    4, 5, 6
```

**Analogy**: A live TV broadcast вЂ” you only see what's airing now.

**Examples:**

- UI click events
- Stock price feeds
- WebSocket messages
- Kafka topic consumption

```java
Sinks.Many<String> hotSink = Sinks.many().multicast().onBackpressureBuffer();
Flux<String> hotFlux = hotSink.asFlux();

hotFlux.subscribe(s -> System.out.println("Sub1: " + s));

hotSink.tryEmitNext("event-1"); // Sub1 sees this
hotSink.tryEmitNext("event-2"); // Sub1 sees this

// Sub2 joins late вЂ” misses event-1 and event-2
hotFlux.subscribe(s -> System.out.println("Sub2: " + s));

hotSink.tryEmitNext("event-3"); // Both see this
```

### 3.3 Converting Cold to Hot

```java
Flux<String> cold = Flux.interval(Duration.ofSeconds(1))
    .map(i -> "item-" + i);

// publish().refCount() вЂ” connects on first subscriber, disconnects when none left
Flux<String> hot = cold.publish().refCount();
// shortcut: cold.share()

// publish().autoConnect(n) вЂ” connects after n subscribers, stays connected
Flux<String> hot2 = cold.publish().autoConnect(2);
```

### 3.4 Quick Comparison

||Cold|Hot|
|---|---|---|
|Data generation|Starts on subscribe|Independent of subscribers|
|Late subscribers|Get full sequence|Miss past items|
|Resource usage|Per-subscriber|Shared|
|Examples|DB query, HTTP call|Events, WebSocket, Kafka|
|Reactor types|`Flux.range()`, `Mono.fromCallable()`|`Sinks`, `publish()`|

## 5. TCK (Technology Compatibility Kit)

### 5.1 What is the TCK?

The TCK is a **test suite** that verifies an implementation correctly follows the Reactive Streams specification. If you build a custom `Publisher`, `Subscriber`, or `Processor`, run the TCK to validate it.

**Why it matters**: The spec has subtle rules (ordering guarantees, threading requirements) that are easy to get wrong. The TCK catches violations automatically.

### 5.2 Dependency

```xml
<dependency>
    <groupId>org.reactivestreams</groupId>
    <artifactId>reactive-streams-tck</artifactId>
    <version>1.0.4</version>
    <scope>test</scope>
</dependency>
<dependency>
    <groupId>org.testng</groupId>
    <artifactId>testng</artifactId>
    <version>7.5</version>
    <scope>test</scope>
</dependency>
```

> **Note**: TCK tests use **TestNG**, not JUnit.

### 5.3 Testing a Custom Publisher

```java
import org.reactivestreams.tck.PublisherVerification;
import org.reactivestreams.tck.TestEnvironment;

public class MyPublisherTest extends PublisherVerification<Integer> {

    public MyPublisherTest() {
        super(new TestEnvironment());
    }

    @Override
    public Publisher<Integer> createPublisher(long elements) {
        // Return your publisher that will emit exactly `elements` items
        return new MyCustomPublisher(elements);
    }

    @Override
    public Publisher<Integer> createFailedPublisher() {
        // Return a publisher that immediately signals onError
        return subscriber -> {
            subscriber.onSubscribe(new Subscription() {
                public void request(long n) {}
                public void cancel() {}
            });
            subscriber.onError(new RuntimeException("intentional failure"));
        };
    }
}
```

Running this gives you ~30+ spec-compliance tests automatically covering request handling, cancellation, error signaling, and threading rules.

### 5.4 Testing a Custom Subscriber

```java
import org.reactivestreams.tck.SubscriberBlackboxVerification;
import org.reactivestreams.tck.TestEnvironment;

public class MySubscriberTest extends SubscriberBlackboxVerification<Integer> {

        super(new TestEnvironment());
    }

    @Override

    }

    @Override

    }

```

## 6. Java 9+ Flow API

Java 9 adopted the Reactive Streams spec directly into the JDK as `java.util.concurrent.Flow`:

```java
// Identical to Reactive Streams, just different package
java.util.concurrent.Flow.Publisher<T>
java.util.concurrent.Flow.Subscriber<T>
java.util.concurrent.Flow.Subscription

```

**Bridge adapter (Reactor):**

```java
// Flow в†’ Reactor
Flow.Publisher<String> jdkPublisher = ...;

// Reactor в†’ Flow
Flow.Publisher<String> flowPublisher = JdkFlowAdapter.publisherToFlowPublisher(reactorFlux);
```

## 7. Interview Questions

**Q: What problem does Reactive Streams solve that regular callbacks don't?**

A: Callbacks have no backpressure mechanism. If a fast producer calls a slow consumer, the consumer either drops events or buffers them unboundedly leading to `OutOfMemoryError`. Reactive Streams standardizes `request(n)` so the consumer controls exactly how much data it receives at any time, preventing overflow.

---

**Q: What's the difference between `request(Long.MAX_VALUE)` and normal backpressure?**

A: `request(Long.MAX_VALUE)` effectively disables backpressure вЂ” it tells the publisher "send everything as fast as you can." This is fine for bounded or naturally slow sources but dangerous for fast producers. Most high-level `subscribe()` overloads (e.g., `flux.subscribe(item -> ...)`) use `Long.MAX_VALUE` internally, which is why you need explicit backpressure operators like `onBackpressureBuffer()` when consuming fast streams.

---

**Q: Why is `Processor` considered problematic?**

A: `Processor` combines `Subscriber` and `Publisher`, which creates tricky situations around threading. For example, you must never call `onNext` concurrently, but a Processor receiving items on one thread and being subscribed to on another makes this easy to violate. The spec authors themselves recommend avoiding custom Processors in favor of operator chains or `Sinks` (in Reactor).

---

**Q: What's the difference between cold and hot publishers? Give a Spring example of each.**

A: A cold publisher starts fresh per subscriber вЂ” e.g., `WebClient.get()...retrieve()` where each subscriber triggers a new HTTP call. A hot publisher emits regardless of subscribers вЂ” e.g., a Kafka consumer wrapped in a reactive `Flux` where messages arrive whether anyone is subscribed or not. In Spring, publishing events to a `Sinks.Many` from a `@EventListener` is a common hot pattern.

---

**Q: What is `flatMap` vs `concatMap` vs `switchMap`?**

A: All three map each item to an inner publisher, but differ in how they handle them. `flatMap` subscribes to all inner publishers concurrently and merges results (no ordering guarantee). `concatMap` subscribes sequentially вЂ” waits for each inner publisher to complete before starting the next (preserves order, lower concurrency). `switchMap` cancels the previous inner publisher when a new item arrives вЂ” useful for "latest wins" scenarios like search-as-you-type.

---

**Q: Can a Subscriber call `request(n)` from inside `onNext()`?**

A: Yes, and it's common. Requesting 1 item in `onSubscribe`, then requesting 1 more in each `onNext`, creates a serial pull pattern. This is safe because the spec guarantees `onNext` calls are sequential (never concurrent), so calling `request(1)` at the end of `onNext` simply queues the next delivery.

---

**Q: What is the TCK and when would you use it?**

A: The TCK (Technology Compatibility Kit) is a TestNG-based test suite that validates a Reactive Streams implementation against the specification. You'd use it when writing a custom `Publisher`, `Subscriber`, or `Processor` to ensure compliance вЂ” particularly for subtle rules around ordering, threading, and error signaling that are easy to get wrong.

---

## 8. Key Takeaways

- The Reactive Streams spec is **4 interfaces** вЂ” everything else is implementation detail
- **Backpressure** is the core innovation: consumer controls flow via `request(n)`
- **Cold publishers** restart per subscriber; **hot publishers** emit regardless of subscribers
- **Marble diagrams** are the visual language for reactive operators вЂ” learn to read them
- **Processor** is part of the spec but avoid implementing it; use `Sinks` in Reactor instead
- **Java 9 Flow API** is the same spec, different package вЂ” bridge adapters available
- **TCK** validates compliance when building custom implementations
