# RxJava Patterns

## Overview

RxJava is the Java implementation of **ReactiveX** (Reactive Extensions) — a library for composing asynchronous and event-based programs using observable sequences. It predates Project Reactor and the Reactive Streams spec, which is why its type system is richer but more complex.

**Key distinction**: RxJava 2 introduced the Reactive Streams-compliant `Flowable` alongside the legacy `Observable`. RxJava 3 (current) is a refinement of RxJava 2 with Java 8+ improvements.

**Dependency (RxJava 3):**

```xml
<dependency>
    <groupId>io.reactivex.rxjava3</groupId>
    <artifactId>rxjava</artifactId>
    <version>3.1.8</version>
</dependency>
```

**Package**: `io.reactivex.rxjava3.core`

---

## 1. The Five Core Types

RxJava has **5 base classes**, each designed for a specific cardinality. Choosing the right type makes your API self-documenting.

```
Type          Items       Backpressure    Reactive Streams
──────────────────────────────────────────────────────────
Observable    0..N        ✗ No            ✗ No
Flowable      0..N        ✓ Yes           ✓ Yes
Single        exactly 1   ✗ No            ✗ No
Maybe         0 or 1      ✗ No            ✗ No
Completable   0 (void)    ✗ No            ✗ No
```

### 1.1 Observable — 0..N items, no backpressure

```java
import io.reactivex.rxjava3.core.Observable;

// The original RxJava type — no backpressure support
Observable<String> obs = Observable.just("A", "B", "C");

// From iterable
Observable<User> users = Observable.fromIterable(userList);

// Timer / interval (always on computation scheduler)
Observable<Long> ticker = Observable.interval(1, TimeUnit.SECONDS);

// Create with emitter
Observable<String> custom = Observable.create(emitter -> {
    emitter.onNext("hello");
    emitter.onNext("world");
    emitter.onComplete();
});

// Subscribe
obs.subscribe(
    item    -> System.out.println("Item: " + item),   // onNext
    error   -> System.err.println("Error: " + error), // onError
    ()      -> System.out.println("Done")              // onComplete
);
```

**When to use Observable:**

- UI events (button clicks, user input) — items are naturally slow
- In-memory collections where backpressure is irrelevant
- When the source emits at a manageable rate

**When NOT to use Observable:**

- Database result sets with millions of rows
- File I/O with large data
- Any scenario where the consumer can't keep up

---

### 1.2 Flowable — 0..N items, WITH backpressure

```java
import io.reactivex.rxjava3.core.Flowable;

// Reactive Streams compliant — implements Publisher<T>
Flowable<String> flow = Flowable.just("A", "B", "C");

// Range
Flowable<Integer> range = Flowable.range(1, 1_000_000);

// From Publisher (Reactor, any RS implementation)
Flowable<String> fromReactor = Flowable.fromPublisher(reactorFlux);

// Create with backpressure strategy
Flowable<String> custom = Flowable.create(emitter -> {
    for (int i = 0; i < 1000; i++) {
        if (emitter.isCancelled()) return;
        emitter.onNext("item-" + i);
    }
    emitter.onComplete();
}, BackpressureStrategy.BUFFER); // see section 4

// Subscribe with explicit request (manual backpressure)
flow.subscribe(new DisposableSubscriber<String>() {
    @Override
    public void onStart() {
        request(10); // request first 10 items
    }

    @Override
    public void onNext(String s) {
        System.out.println(s);
        request(1); // request one more
    }

    @Override
    public void onError(Throwable t) { t.printStackTrace(); }

    @Override
    public void onComplete() { System.out.println("Done"); }
});
```

**Flowable vs Observable — rule of thumb:**

- Use `Flowable` when source emits **10k+ items** OR you're integrating with other Reactive Streams implementations
- Use `Observable` for UI events, in-memory collections, short sequences

---

### 1.3 Single — exactly 1 item or error

```java
import io.reactivex.rxjava3.core.Single;

// Emits exactly one item or an error — never empty, never multiple
Single<User> user = Single.just(new User("Alice"));

// From callable (lazy)
Single<User> fromDB = Single.fromCallable(() -> userRepo.findById(id));

// Subscribe
user.subscribe(
    u     -> System.out.println("Got: " + u),
    error -> System.err.println("Error: " + error)
);

// Convert to/from Observable
Observable<User> obs = user.toObservable();
Single<User> back = Observable.just(new User("Bob")).singleOrError();
```

**When to use Single:**

- HTTP GET that always returns a response body
- `findById()` that throws `NotFoundException` if not found
- Any operation guaranteed to produce exactly one result

**Reactor equivalent**: `Mono<T>` where the stream is never empty

---

### 1.4 Maybe — 0 or 1 item, or error

```java
import io.reactivex.rxjava3.core.Maybe;

// Emits 0 or 1 items — like Optional but reactive
Maybe<User> user = Maybe.just(new User("Alice"));
Maybe<User> empty = Maybe.empty();
Maybe<User> error = Maybe.error(new RuntimeException("not found"));

// From callable that might return null
Maybe<User> fromDB = Maybe.fromCallable(() -> userRepo.findByIdOrNull(id));

// Subscribe
user.subscribe(
    u     -> System.out.println("Got: " + u),   // onSuccess
    error -> System.err.println("Error"),        // onError
    ()    -> System.out.println("Empty")         // onComplete (nothing emitted)
);
```

**When to use Maybe:**

- `findById()` that returns null/empty when not found
- Optional-like reactive operations
- Cache lookups that might miss

**Reactor equivalent**: `Mono<T>` (Mono handles all three cases)

---

### 1.5 Completable — no items, just completion or error

```java
import io.reactivex.rxjava3.core.Completable;

// Signals only completion or error — no value
Completable save = Completable.fromAction(() -> userRepo.save(user));
Completable delete = Completable.fromRunnable(() -> cache.invalidate(key));

// Chain completables
Completable pipeline = validateUser(user)
    .andThen(saveUser(user))
    .andThen(sendWelcomeEmail(user));

// Subscribe
pipeline.subscribe(
    ()    -> System.out.println("All done"),
    error -> System.err.println("Failed: " + error)
);

// Convert from other types
Completable fromSingle = Single.just("result").ignoreElement();
Completable fromObservable = Observable.just(1, 2, 3).ignoreElements();
```

**When to use Completable:**

- Save/update/delete operations where the result value doesn't matter
- Fire-and-forget side effects
- Chaining sequential operations with `andThen()`

**Reactor equivalent**: `Mono<Void>`

---

### 1.6 Type Conversion Summary

```java
// Observable ↔ Flowable
Flowable<T> f = observable.toFlowable(BackpressureStrategy.BUFFER);
Observable<T> o = flowable.toObservable();

// → Single
Single<T> s = observable.firstOrError();
Single<T> s2 = maybe.toSingle(defaultValue);

// → Maybe
Maybe<T> m = observable.firstElement();
Maybe<T> m2 = single.toMaybe();

// → Completable
Completable c = observable.ignoreElements();
Completable c2 = single.ignoreElement();

// Completable → others
Single<T> s3 = completable.andThen(Single.just(value));
Observable<T> o2 = completable.andThen(Observable.just(value));
```

## 2. Schedulers

RxJava Schedulers control which thread pool an operation executes on. They work the same way as Reactor's `subscribeOn`/`publishOn` but with different named pools.

### 2.1 Built-in Schedulers

```java
import io.reactivex.rxjava3.schedulers.Schedulers;
import io.reactivex.rxjava3.android.schedulers.AndroidSchedulers; // Android only

// io() — unbounded thread pool, cached, for blocking I/O
// Threads idle for 60s then are reclaimed
Schedulers.io();

// computation() — fixed pool (CPU cores), for non-blocking CPU work
// Size = Runtime.getRuntime().availableProcessors()
Schedulers.computation();

// newThread() — creates a NEW thread for each unit of work
// Expensive — prefer io() for most I/O tasks
Schedulers.newThread();

// single() — single background thread, serial execution
Schedulers.single();

// trampoline() — queues work on current thread (serial, no new thread)
// Useful for testing and preventing stack overflow in recursive chains
Schedulers.trampoline();

// from() — wrap your own Executor
Schedulers.from(Executors.newFixedThreadPool(4));
Schedulers.from(myExecutor, /* interruptible= */ true);
```

### 2.2 subscribeOn — where source runs

```java
// Move the source emission to io() thread pool
Single.fromCallable(() -> database.findUser(id))
    .subscribeOn(Schedulers.io())        // DB call runs on io thread
    .observeOn(Schedulers.computation()) // switch for downstream
    .map(user -> transform(user))        // runs on computation
    .subscribe(result -> handleResult(result));

// Only FIRST subscribeOn matters (same as Reactor)
Observable.range(1, 10)
    .subscribeOn(Schedulers.io())          // applies
    .subscribeOn(Schedulers.computation()) // IGNORED
    .subscribe(System.out::println);
```

### 2.3 observeOn — where downstream runs

```java
// Switch thread for operators below this point (same as Reactor's publishOn)
Observable.fromCallable(() -> loadFile())
    .subscribeOn(Schedulers.io())          // file I/O on io thread
    .observeOn(Schedulers.computation())   // CPU parsing on computation
    .map(data -> parseJson(data))
    .observeOn(Schedulers.single())        // results handled serially
    .subscribe(result -> updateState(result));
```

### 2.4 Scheduler Comparison

|Scheduler|Thread Pool|Best For|Notes|
|---|---|---|---|
|`io()`|Unbounded, cached|Blocking I/O (DB, HTTP, files)|Threads reclaimed after 60s idle|
|`computation()`|Fixed (CPU cores)|CPU-bound non-blocking work|Never block these threads|
|`newThread()`|1 new thread per task|Rare — mostly legacy|Expensive, prefer `io()`|
|`single()`|1 shared thread|Serial background tasks|All tasks queue on one thread|
|`trampoline()`|Current thread|Testing, recursion|Queues work, no new threads|

**Reactor → RxJava Scheduler mapping:**

|Reactor|RxJava|
|---|---|
|`Schedulers.boundedElastic()`|`Schedulers.io()`|
|`Schedulers.parallel()`|`Schedulers.computation()`|
|`Schedulers.single()`|`Schedulers.single()`|

## 3. Operators Comparison with Reactor

Most Reactor operators have direct RxJava equivalents. The main differences are naming and method signatures.

### 3.1 Transformation Operators

|Reactor|RxJava|Behavior|
|---|---|---|
|`map()`|`map()`|Sync 1:1 transform|
|`flatMap()`|`flatMap()`|Async, concurrent, unordered|
|`concatMap()`|`concatMap()`|Async, sequential, ordered|
|`switchMap()`|`switchMap()`|Latest wins, cancels previous|
|`filter()`|`filter()`|Keep matching items|
|`take(n)`|`take(n)`|First n items|
|`skip(n)`|`skip(n)`|Skip first n items|
|`distinct()`|`distinct()`|Unique items|
|`scan()`|`scan()`|Running aggregate|
|`reduce()`|`reduce()`|Final aggregate → Single|
|`collectList()`|`toList()`|All items → List|
|`defaultIfEmpty()`|`defaultIfEmpty()`|Fallback value if empty|
|`switchIfEmpty()`|`switchIfEmpty()`|Fallback publisher if empty|

```java
// Reactor
Flux.range(1, 10)
    .filter(n -> n % 2 == 0)
    .map(n -> n * 3)
    .collectList();

// RxJava — identical structure
Observable.range(1, 10)
    .filter(n -> n % 2 == 0)
    .map(n -> n * 3)
    .toList();
```

---

### 3.2 Combining Operators

|Reactor|RxJava|Behavior|
|---|---|---|
|`Flux.merge()`|`Observable.merge()`|Concurrent, arrival order|
|`Flux.concat()`|`Observable.concat()`|Sequential, source order|
|`Mono.zip()` / `Flux.zip()`|`Observable.zip()`|Combine by index|
|`Flux.combineLatest()`|`Observable.combineLatest()`|Emit on any source update|
|`.mergeWith()`|`.mergeWith()`|Inline merge|
|`.concatWith()`|`.concatWith()`|Inline concat|
|`.zipWith()`|`.zipWith()`|Inline zip|
|`.startWith()`|`.startWith()`|Prepend items|
|`Flux.amb()`|`Observable.amb()`|First source to emit wins|

```java
// amb — only the fastest source continues, others are disposed
Observable<String> source1 = Observable.timer(100, TimeUnit.MILLISECONDS).map(t -> "source1");
Observable<String> source2 = Observable.timer(50, TimeUnit.MILLISECONDS).map(t -> "source2");

Observable.amb(Arrays.asList(source1, source2))
    .subscribe(System.out::println); // "source2" wins
```

---

### 3.3 Error Handling Operators

|Reactor|RxJava|Behavior|
|---|---|---|
|`onErrorReturn()`|`onErrorReturnItem()`|Fallback value|
|`onErrorResume()`|`onErrorResumeWith()`|Fallback publisher|
|`onErrorMap()`|`onErrorResumeWith(e -> Observable.error(...))`|Transform error|
|`retry(n)`|`retry(n)`|Retry n times|
|`retryWhen()`|`retryWhen()`|Conditional retry|
|`timeout()`|`timeout()`|Timeout signal|
|`doOnError()`|`doOnError()`|Side effect on error|

```java
// RxJava error handling chain
Single.fromCallable(() -> database.findUser(id))
    .subscribeOn(Schedulers.io())
    .timeout(5, TimeUnit.SECONDS)
    .retry(3)
    .onErrorReturnItem(User.empty())
    .subscribe(
        user  -> handleUser(user),
        error -> log.error("Failed", error)
    );
```

---

### 3.4 Lifecycle / Side Effect Operators

|Reactor|RxJava|When called|
|---|---|---|
|`doOnSubscribe()`|`doOnSubscribe()`|On subscribe|
|`doOnNext()`|`doOnNext()`|On each item|
|`doOnError()`|`doOnError()`|On error|
|`doOnComplete()`|`doOnComplete()`|On completion|
|`doOnTerminate()`|`doOnTerminate()`|On complete or error|
|`doFinally()`|`doFinally()`|Always — like finally block|
|`doOnDispose()`|`doOnDispose()`|When disposed/cancelled|

### 3.5 Key Naming Differences

```java
// Reactor → RxJava naming gotchas

// Reactor: onErrorReturn     → RxJava: onErrorReturnItem (for value)
//                                      onErrorReturn (for function)
flux.onErrorReturn(User.empty());
observable.onErrorReturnItem(User.empty());

// Reactor: collectList()     → RxJava: toList() returns Single<List<T>>
flux.collectList();   // Mono<List<T>>
obs.toList();         // Single<List<T>>

// Reactor: subscribe(Consumer) → RxJava same but returns Disposable
Disposable d = observable.subscribe(item -> handle(item));
d.dispose(); // cancel subscription

// Reactor: Subscription.cancel() → RxJava: Disposable.dispose()
```

## 4. Backpressure Strategies

Only `Flowable` supports backpressure in RxJava. `Observable` has no backpressure mechanism.

### 4.1 BackpressureStrategy (for Flowable.create)

```java
// BUFFER — buffer all items in memory until consumer is ready
// Risk: OutOfMemoryError if consumer is far too slow
Flowable.create(emitter -> {
    for (int i = 0; i < 1_000_000; i++) emitter.onNext(i);
    emitter.onComplete();
}, BackpressureStrategy.BUFFER);

// DROP — drop new items when the downstream can't keep up
Flowable.create(emitter -> {
    // emit fast
}, BackpressureStrategy.DROP);

// LATEST — keep only the most recent item when overwhelmed
Flowable.create(emitter -> {
    // emit fast
}, BackpressureStrategy.LATEST);

// ERROR — throw MissingBackpressureException when buffer overflows
Flowable.create(emitter -> {
    // emit fast
}, BackpressureStrategy.ERROR);

// MISSING — no backpressure strategy applied
// Use when downstream handles it via onBackpressureXxx operators
Flowable.create(emitter -> {
    // emit fast
}, BackpressureStrategy.MISSING);
```

### 4.2 Backpressure Operators (on existing Flowables)

```java
Flowable<Integer> fast = Flowable.range(1, 1_000_000);

// onBackpressureBuffer — buffer with optional capacity and overflow action
fast.onBackpressureBuffer(1000,
    () -> log.warn("Buffer full!"),
    BackpressureOverflowStrategy.DROP_OLDEST);

// onBackpressureDrop — drop with optional observer
fast.onBackpressureDrop(dropped -> log.warn("Dropped: {}", dropped));

// onBackpressureLatest — only keep latest
fast.onBackpressureLatest();
```

### 4.3 Converting Observable to Flowable with strategy

```java
Observable<String> hotObservable = getEventStream(); // fast, no backpressure

// Convert and apply strategy
Flowable<String> flowable = hotObservable
    .toFlowable(BackpressureStrategy.BUFFER);

// Then add downstream operators
flowable
    .observeOn(Schedulers.io(), false, 128) // prefetch = 128
    .subscribe(item -> processSlowly(item));
```

### 4.4 Controlling Demand with request(n)

```java
// Manual backpressure via DisposableSubscriber
Flowable.range(1, 1000)
    .subscribe(new DisposableSubscriber<Integer>() {
        @Override
        public void onStart() {
            request(1); // start by requesting 1 item
        }

        @Override
        public void onNext(Integer item) {
            process(item);
            request(1); // request next item only after processing current
        }

        @Override
        public void onError(Throwable t) { t.printStackTrace(); }

        @Override
        public void onComplete() { System.out.println("Done"); }
    });
```

### 4.5 MissingBackpressureException

The most common RxJava error for developers new to reactive:

```java
// This will throw MissingBackpressureException
Flowable.interval(1, TimeUnit.MILLISECONDS)   // emits fast
    .observeOn(Schedulers.computation())       // internal buffer = 128 by default
    .doOnNext(n -> Thread.sleep(10))           // consumer is 10x slower
    .subscribe(System.out::println);
    // Buffer fills (128 items), then MissingBackpressureException

// Fix: add a backpressure strategy
Flowable.interval(1, TimeUnit.MILLISECONDS)
    .onBackpressureDrop()                      // drop when overwhelmed
    .observeOn(Schedulers.computation())
    .doOnNext(n -> Thread.sleep(10))
    .subscribe(System.out::println);
```

## 5. Testing with TestScheduler

`TestScheduler` gives you **virtual time control** — you can advance time manually without waiting for real wall-clock time. Essential for testing time-based operators like `interval`, `delay`, `timeout`, and `debounce`.

### 5.1 Basic TestScheduler Usage

```java
import io.reactivex.rxjava3.schedulers.TestScheduler;
import java.util.concurrent.TimeUnit;

TestScheduler scheduler = new TestScheduler();

// Create an observable that emits every 1 second (on test scheduler)
Observable<Long> ticker = Observable.interval(1, TimeUnit.SECONDS, scheduler);

// Collect emissions for assertion
List<Long> results = new ArrayList<>();
ticker.subscribe(results::add);

// Nothing has happened yet — time hasn't advanced
assertThat(results).isEmpty();

// Advance virtual time by 3 seconds
scheduler.advanceTimeBy(3, TimeUnit.SECONDS);
assertThat(results).containsExactly(0L, 1L, 2L);

// Advance to a specific point in time
scheduler.advanceTimeTo(10, TimeUnit.SECONDS);
assertThat(results).hasSize(10);
```

### 5.2 Testing delay

```java
@Test
void testDelay() {
    TestScheduler scheduler = new TestScheduler();

    Single<String> delayed = Single.just("result")
        .delay(5, TimeUnit.SECONDS, scheduler);

    TestObserver<String> observer = delayed.test();

    // Before delay completes
    scheduler.advanceTimeBy(4, TimeUnit.SECONDS);
    observer.assertNoValues();

    // After delay completes
    scheduler.advanceTimeBy(1, TimeUnit.SECONDS);
    observer.assertValue("result");
    observer.assertComplete();
}
```

### 5.3 Testing timeout

```java
@Test
void testTimeout() {
    TestScheduler scheduler = new TestScheduler();

    // Source that never emits
    Observable<String> neverEmits = Observable.never();

    TestObserver<String> observer = neverEmits
        .timeout(3, TimeUnit.SECONDS, scheduler)
        .test();

    scheduler.advanceTimeBy(2, TimeUnit.SECONDS);
    observer.assertNoErrors(); // still waiting

    scheduler.advanceTimeBy(1, TimeUnit.SECONDS);
    observer.assertError(TimeoutException.class); // timeout fired
}
```

### 5.4 TestObserver and TestSubscriber

```java
// TestObserver — for Observable, Single, Maybe, Completable
TestObserver<String> observer = observable.test();

// TestSubscriber — for Flowable
TestSubscriber<String> subscriber = flowable.test();

// Assertions
observer.assertNoErrors();
observer.assertComplete();
observer.assertValueCount(3);
observer.assertValues("A", "B", "C");
observer.assertValue(s -> s.startsWith("A"));
observer.assertError(RuntimeException.class);
observer.assertError(ex -> ex.getMessage().equals("oops"));
observer.assertNotComplete();
observer.assertEmpty(); // no items, no error, no completion

// Await for async operations
observer.awaitDone(5, TimeUnit.SECONDS);
observer.assertComplete();
```

### 5.5 Testing async with awaitDone

```java
@Test
void testAsyncOperation() throws InterruptedException {
    TestObserver<User> observer = userService
        .findById(1L)           // runs on Schedulers.io()
        .test();                // subscribes immediately

    // Wait up to 5 seconds for the observable to complete
    observer.awaitDone(5, TimeUnit.SECONDS);

    observer.assertNoErrors();
    observer.assertValueCount(1);
    observer.assertValue(u -> u.getId() == 1L);
}
```

### 5.6 Testing debounce (common interview question)

```java
@Test
void testDebounce() {
    TestScheduler scheduler = new TestScheduler();
    PublishSubject<String> subject = PublishSubject.create();

    TestObserver<String> observer = subject
        .debounce(300, TimeUnit.MILLISECONDS, scheduler)
        .test();

    subject.onNext("a");
    scheduler.advanceTimeBy(100, TimeUnit.MILLISECONDS);
    subject.onNext("ab");         // resets the 300ms window
    scheduler.advanceTimeBy(100, TimeUnit.MILLISECONDS);
    subject.onNext("abc");        // resets again
    scheduler.advanceTimeBy(299, TimeUnit.MILLISECONDS);
    observer.assertNoValues();    // debounce window not expired

    scheduler.advanceTimeBy(1, TimeUnit.MILLISECONDS); // window expires
    observer.assertValue("abc"); // only last value emitted
}
```

### 5.7 Subject — hot observable bridge

Subjects are both an `Observer` and `Observable` — useful for bridging imperative and reactive code, and for testing.

```java
// PublishSubject — hot, subscribers only see items after subscribe
PublishSubject<String> publish = PublishSubject.create();
publish.subscribe(s -> System.out.println("A: " + s));
publish.onNext("1");   // A sees this
publish.subscribe(s -> System.out.println("B: " + s));
publish.onNext("2");   // both A and B see this

// BehaviorSubject — replays last item to new subscribers
BehaviorSubject<String> behavior = BehaviorSubject.createDefault("initial");
behavior.subscribe(s -> System.out.println("Late: " + s)); // gets "initial"
behavior.onNext("updated");

// ReplaySubject — replays ALL items (or configurable window)
ReplaySubject<String> replay = ReplaySubject.create();
replay.onNext("old-1");
replay.onNext("old-2");
replay.subscribe(s -> System.out.println(s)); // gets old-1, old-2 immediately

// AsyncSubject — only emits the LAST item, on completion
AsyncSubject<String> async = AsyncSubject.create();
async.subscribe(s -> System.out.println(s));
async.onNext("1");
async.onNext("2");
async.onNext("3");
async.onComplete(); // only "3" is emitted to subscriber
```

## 6. Disposable and Resource Management

```java
// All subscriptions return Disposable
Disposable d = Observable.interval(1, TimeUnit.SECONDS)
    .subscribe(System.out::println);

// Cancel when done
d.dispose();
System.out.println(d.isDisposed()); // true

// CompositeDisposable — manage multiple subscriptions
CompositeDisposable composite = new CompositeDisposable();

composite.add(
    userStream.subscribe(this::onUser)
);
composite.add(
    errorStream.subscribe(this::onError)
);

// Dispose all at once (e.g., in onDestroy or @PreDestroy)
composite.dispose();
```

## 7. Interview Questions

**Q: Why does RxJava have both `Observable` and `Flowable`? Why not just one type?**

A: `Observable` was the original type from RxJava 1, which had no backpressure support. RxJava 2 introduced `Flowable` as the Reactive Streams-compliant type with backpressure. `Observable` was kept because many use cases (UI events, small in-memory collections) don't need backpressure, and the overhead of tracking `request(n)` for every emission is wasteful. Separating them allows developers to choose the right tool: `Observable` for fast/bounded sources, `Flowable` for potentially slow consumers or large datasets.

---

**Q: When would you use `Single` over `Maybe`?**

A: Use `Single` when you're certain the operation always produces a value — if it fails, it errors. For example, an HTTP POST that always returns the created entity, or a `findById` that throws `NotFoundException` when missing. Use `Maybe` when the operation might legitimately produce nothing — like `findById` returning empty when the record doesn't exist without that being an error condition. `Maybe` is reactive `Optional`.

---

**Q: What is `MissingBackpressureException` and how do you fix it?**

A: It's thrown when a `Flowable` emits more items than the downstream has requested (via `request(n)`), and there's no strategy to handle the overflow. It typically occurs when an `Observable` is converted to `Flowable` with `BackpressureStrategy.MISSING`, or when using `observeOn` whose internal buffer fills up. Fix it by applying a backpressure operator (`onBackpressureBuffer()`, `onBackpressureDrop()`, `onBackpressureLatest()`) before the problematic operator, or by choosing the right `BackpressureStrategy` in `Flowable.create()`.

---

**Q: What is the difference between `subscribeOn` and `observeOn` in RxJava?**

A: `subscribeOn` specifies the scheduler on which the source emission and subscription happen — it affects the entire upstream chain and only the first one matters. `observeOn` switches the scheduler for all operators downstream of it — it can appear multiple times in a chain to switch between thread pools. The common pattern is `subscribeOn(Schedulers.io())` to move blocking I/O off the main thread, and `observeOn(AndroidSchedulers.mainThread())` (in Android) to deliver results to the UI thread.

---

**Q: How does `TestScheduler` work, and why do you need it?**

A: `TestScheduler` replaces real wall-clock time with virtual time that you control manually. Without it, testing a 5-minute timeout would require actually waiting 5 minutes. With `TestScheduler`, you call `advanceTimeBy(5, TimeUnit.MINUTES)` and all time-based operators (interval, delay, debounce, timeout) respond as if 5 minutes elapsed instantly. You inject it into operators via the scheduler overload — e.g., `Observable.interval(1, TimeUnit.SECONDS, testScheduler)`.

---

**Q: What is the difference between `PublishSubject` and `BehaviorSubject`?**

A: `PublishSubject` is a hot observable that emits only items arriving after a subscriber subscribes — late subscribers miss earlier items. `BehaviorSubject` replays the most recent item (or a seed value) to new subscribers — they immediately get the current state on subscription. `BehaviorSubject` is useful for modeling "current state" like the latest value of a form field or a user's login status, where a new subscriber should always start with the current value.

---

**Q: How would you convert a RxJava `Flowable` to a Project Reactor `Flux`?**

A: Since both implement Reactive Streams `Publisher`, the conversion is straightforward using Reactor's `from()` factory: `Flux<T> flux = Flux.from(flowable)`. Going the other way: `Flowable<T> flowable = Flowable.fromPublisher(flux)`. Note that only `Flowable` (not `Observable`) interoperates with Reactor this way, since `Observable` doesn't implement `Publisher`.

## 8. Key Takeaways

- RxJava has **5 types**: `Observable` (no backpressure), `Flowable` (Reactive Streams), `Single` (exactly 1), `Maybe` (0 or 1), `Completable` (void)
- Use `Flowable` for potentially large/fast data sources; `Observable` for UI events and in-memory collections
- `subscribeOn` moves the source; `observeOn` switches threads for downstream operators
- `io()` for blocking I/O, `computation()` for CPU-bound work — never block `computation()` threads
- Backpressure in RxJava is opt-in via `Flowable` — `MissingBackpressureException` means your source is faster than your consumer
- `TestScheduler` gives you virtual time for deterministic testing of time-based operators
- `Subjects` bridge imperative and reactive code — `PublishSubject` for events, `BehaviorSubject` for state
- `Disposable`/`CompositeDisposable` is how you cancel subscriptions — always clean up in lifecycle methods
