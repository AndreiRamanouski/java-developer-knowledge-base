# Reactive vs Imperative

## Overview

Reactive programming is not a universal upgrade over imperative code — it's a different tool for a different class of problems. Understanding when reactive genuinely helps, when it hurts, and how to migrate or debug it separates developers who use reactive effectively from those who fight it constantly.

**The central tension**: Reactive shines under high concurrency with I/O-bound workloads. It struggles with CPU-bound work, blocking integrations, complex debugging, and teams without reactive experience.

---

## 1. When to Use Reactive

### 1.1 The Core Metric: Thread Utilization Under I/O

The fundamental question is: **what is your thread doing while it waits for I/O?**

**Imperative (thread-per-request):**

```
Thread-1: ──[DB query: 50ms wait]────────────[process: 2ms]──► response
           └── thread is BLOCKED for 50ms, doing nothing
```

**Reactive (event-loop):**

```
Thread-1: ──[subscribe DB]──[other requests...]──[DB result arrives]──[process]──► response
           └── thread is FREE during the 50ms wait, handles other requests
```

With 500 concurrent users and 50ms DB latency:

- Imperative needs ~500 threads (each blocked 50ms) → ~500MB thread stack memory
- Reactive needs ~4-8 threads (event loop cores) → handles the same load

### 1.2 Use Reactive When

**High concurrency + I/O-bound workload:**

```java
// This is where reactive excels
// 10,000 concurrent WebSocket connections, each waiting for events
@GetMapping(value = "/feed", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<FeedItem> liveStream(@RequestParam String userId) {
    return feedService.subscribeToFeed(userId); // 10k streams, 8 threads
}
```

**Service aggregation (fan-out):**

```java
// Calling 5 downstream services concurrently — reactive is ideal
public Mono<ProductPage> getProductPage(Long productId) {
    return Mono.zip(
        productService.getDetails(productId),      // 30ms
        reviewService.getReviews(productId),       // 40ms
        pricingService.getPrice(productId),        // 20ms
        inventoryService.getStock(productId),      // 25ms
        recommendationService.getSimilar(productId) // 35ms
    ).map(t -> assembleProductPage(t));
    // Total: ~40ms (max of parallel calls), not 150ms (sum of sequential)
}
```

**Streaming data pipelines:**

```java
// Processing 1M Kafka events/minute — reactive stream processing is natural
kafkaReceiver.receive()
    .flatMap(record -> enrich(record), 64)
    .filter(this::isRelevant)
    .groupBy(Event::getType)
    .flatMap(group -> group.buffer(Duration.ofSeconds(5)).flatMap(this::batchInsert))
    .subscribe();
```

**Real-time data push:**

- WebSocket servers with many simultaneous connections
- Server-Sent Events for live dashboards
- Chat applications, notification systems

**Microservice API gateways:**

- A gateway aggregating many downstream calls benefits enormously from reactive — the gateway thread is always waiting on downstream I/O

### 1.3 Do NOT Use Reactive When

**CPU-bound workloads:**

```java
// Reactive adds overhead with no benefit for pure computation
// Imperative is simpler and faster here
List<Result> results = items.parallelStream()
    .map(item -> cpuIntensiveTransform(item)) // Java parallel streams are fine
    .collect(Collectors.toList());

// Reactive version is needlessly complex:
Flux.fromIterable(items)
    .parallel()
    .runOn(Schedulers.parallel())
    .map(item -> cpuIntensiveTransform(item))
    .sequential()
    .collectList()
    .block(); // and you still block at the end
```

**Low-concurrency applications:**

- A back-office tool used by 10 employees simultaneously doesn't need reactive
- A batch job running once a day — just use Spring Batch
- Internal tools, admin panels, simple CRUD APIs with low traffic

**Heavy blocking integrations you can't change:**

```java
// If ALL your I/O is blocking (JDBC, legacy SOAP, old SDKs),
// reactive just adds complexity without the concurrency benefit

// Wrapping every blocking call:
Mono.fromCallable(() -> jdbcRepo.findById(id))  // you're still blocking a thread
    .subscribeOn(Schedulers.boundedElastic())    // just on a different pool
// At high concurrency, boundedElastic grows unboundedly — same as thread-per-request
```

**Teams without reactive experience:**

- The productivity cost of learning curve + debugging time often outweighs throughput gains for moderate traffic

### 1.4 Decision Framework

```
Is your bottleneck I/O? (DB, HTTP, messaging)
    ├── No  → Use imperative. Reactive won't help.
    └── Yes → Do you have high concurrency? (100+ req/sec or sustained connections)
                  ├── No  → Use imperative. Thread-per-request is fine.
                  └── Yes → Is your team experienced with reactive?
                                ├── No  → Start imperative, plan migration
                                └── Yes → Is there a reactive driver? (R2DBC, WebClient, reactive Kafka)
                                              ├── No  → Wrap blocking in boundedElastic,
                                              │         consider if benefit justifies complexity
                                              └── Yes → Use reactive
```

## 2. Performance Implications

### 2.1 Where Reactive Wins

**Thread count under high concurrency:**

|Scenario|Imperative (threads)|Reactive (threads)|
|---|---|---|
|1,000 concurrent requests, 50ms I/O|~1,000|~8|
|10,000 WebSocket connections|~10,000|~8|
|500 req/sec, 100ms avg latency|~500|~8|

**Memory:**

- Each Java thread uses ~256KB-1MB of stack memory by default
- 1,000 threads = ~256MB-1GB just for stacks
- Reactive event loop: ~8 threads + queue structures — far less memory pressure

**Latency under load:**

```
Imperative at 80% thread pool capacity:
  New requests wait in queue → latency spikes

Reactive at 80% CPU:
  Event loop processes at full speed → latency stays flat until CPU saturation
```

### 2.2 Where Reactive Has Overhead

**Per-item overhead:** Reactive adds overhead for each emission due to operator chaining, scheduler interactions, and Reactive Streams signaling. For a simple `map()` over an in-memory list:

```
Imperative loop:   ~2ns per item
Reactive Flux:     ~50-200ns per item (operator overhead)
```

For large in-memory datasets without I/O, a simple loop or parallel stream is faster.

**Object allocation:** Each operator in a chain allocates internal objects (lambdas, subscription objects). Under heavy load this increases GC pressure. Reactor mitigates this with fusion optimizations, but it's still higher than imperative loops.

**Startup and first-request latency:** Reactive pipelines have slightly higher assembly cost (building the operator chain) vs imperative code. Negligible at scale but relevant for short-lived Lambda/serverless functions.

### 2.3 The Blocking Tax — the worst of both worlds

The most dangerous performance antipattern: blocking inside a reactive pipeline.

```java
// WRONG — blocks event loop thread → 1 block = 1 thread unusable
@GetMapping("/bad")
public Mono<User> badEndpoint(Long id) {
    User user = userRepository.findById(id).get(); // BLOCKING JDBC call
    return Mono.just(user); // reactive wrapper on blocking result — pointless
}

// Under 10 concurrent requests with 50ms DB time:
// → All 8 event loop threads blocked → request queue → cascading failure

// RIGHT — offload blocking to separate pool
@GetMapping("/good")
public Mono<User> goodEndpoint(Long id) {
    return Mono.fromCallable(() -> userRepository.findById(id))
               .subscribeOn(Schedulers.boundedElastic());
}
```

**BlockHound** — detect accidental blocking in reactive code:

```xml
<dependency>
    <groupId>io.projectreactor.tools</groupId>
    <artifactId>blockhound</artifactId>
    <version>1.0.8.RELEASE</version>
</dependency>
```

```java
// In tests or development — throws error if blocking call detected on reactive thread
@SpringBootApplication
public class Application {
    public static void main(String[] args) {
        BlockHound.install(); // enable blocking detection
        SpringApplication.run(Application.class, args);
    }
}
```

### 2.4 Realistic Throughput Comparison

```
Scenario: REST API, 50ms DB query per request, 4-core server

Imperative (Spring MVC, Tomcat):
  Max thread pool: 200 threads
  Throughput: ~200 req / 0.05s = ~4,000 req/sec (before queue pressure)
  Memory: 200 × ~500KB = ~100MB thread stacks

Reactive (Spring WebFlux, Netty):
  Threads: 8 (Netty event loop)
  Throughput: limited by DB connection pool and CPU
  Memory: ~8 × 500KB = ~4MB thread stacks
  Better latency curve as load increases toward DB limit
```

**Important caveat**: if the DB connection pool is the bottleneck (both have 20 connections), throughput ceiling is the same — reactive just gets there with less memory and more graceful degradation.

## 3. Learning Curve

### 3.1 Mental Model Shift

Imperative code describes **what to do**:

```java
// Read like a recipe: do this, then that
User user = userRepository.findById(id);     // 1. fetch user
List<Role> roles = roleRepo.findByUser(user); // 2. fetch roles
String report = buildReport(user, roles);    // 3. build report
return report;
```

Reactive code describes **a pipeline to assemble**:

```java
// Read like a data flow: declare transformations, subscribe triggers execution
return userRepository.findById(id)           // declare: fetch user
    .zipWith(                                // declare: fetch roles in parallel
        roleRepo.findByUser(id).collectList()
    )
    .map(tuple -> buildReport(              // declare: transform when both arrive
        tuple.getT1(), tuple.getT2()
    ));                                     // nothing runs yet — caller subscribes
```

### 3.2 Common Learning Stages

**Stage 1 — Confusion (week 1-2):**

- "Why doesn't my code run?" → forgetting to subscribe
- "Where did my data go?" → returning `Mono<Mono<T>>` instead of using `flatMap`
- "Why is this null?" → trying to extract values imperatively from a `Mono`

```java
// Classic Stage 1 mistake
String result = null;
Mono.just("hello").subscribe(s -> result = s); // async — result is still null!
System.out.println(result); // null
```

**Stage 2 — Fighting the types (week 2-4):**

- Wrapping everything in `Mono.just()` to "get back to normal"
- Using `.block()` everywhere (defeats the purpose)
- Confused about `flatMap` vs `map` — using `map` and getting `Mono<Mono<T>>`

```java
// Stage 2 mistake — map returns Mono<Mono<User>>
Mono<Mono<User>> wrong = Mono.just(1L).map(id -> userRepo.findById(id));

// Correct — flatMap subscribes to inner Mono
Mono<User> right = Mono.just(1L).flatMap(id -> userRepo.findById(id));
```

**Stage 3 — Cargo-culting (month 1-2):**

- Code works but the structure is wrong: unnecessary `Mono.fromCallable()` wrappers everywhere
- `subscribeOn` and `publishOn` used randomly "just in case"
- Error handling missing on fire-and-forget operations

**Stage 4 — Fluency (month 2-6):**

- Thinking in pipelines naturally
- Knowing when `flatMap`/`concatMap`/`switchMap` applies
- Understanding threading model well enough to debug scheduler issues
- Recognizing when NOT to use reactive

### 3.3 Steepest Learning Challenges

**Debugging stack traces:**

```
// Reactive stack trace — hard to follow
java.lang.NullPointerException: null
    at com.example.UserService.lambda$findUser$3(UserService.java:45)
    at reactor.core.publisher.FluxMapFuseable$MapFuseableSubscriber.onNext(...)
    at reactor.core.publisher.FluxFilterFuseable$FilterFuseableSubscriber.onNext(...)
    at reactor.core.publisher.MonoZip$ZipCoordinator.signal(...)
    at reactor.core.publisher.FluxSubscribeOn$SubscribeOnSubscriber.onNext(...)
    ... 30 more reactor internal frames
```

The actual problem (line 45) is buried. See section 4 for solutions.

**Testing async pipelines:**

- Unit tests require `StepVerifier` or `.block()` — `@Test` methods return `void`, not `Mono`
- Time-based operators need `VirtualTimeScheduler`

**Backpressure surprises:**

- `MissingBackpressureException` appearing under load but not in tests
- Unbounded `flatMap` creating more connections than the DB pool allows

## 4. Debugging Challenges

### 4.1 The Stack Trace Problem

Reactive stack traces are deep and full of framework internals. The actual error location is often several layers removed from where the exception appears.

**Solution 1: Reactor's debug mode (development only)**

```java
// Enable globally — HEAVY performance cost, development only
Hooks.onOperatorDebug();

// Now stack traces include assembly points:
// Error has been observed at the following site(s):
//   *__checkpoint ⇢ UserService.findUser(UserService.java:45)
//   *__checkpoint ⇢ Handler com.example.UserController#getUser [DispatcherHandler]
```

**Solution 2: ReactorDebugAgent (low-overhead alternative)**

```xml
<dependency>
    <groupId>io.projectreactor</groupId>
    <artifactId>reactor-tools</artifactId>
</dependency>
```

```java
// In main() or test setup — uses Java agent, much lower overhead than Hooks.onOperatorDebug()
ReactorDebugAgent.init();
```

**Solution 3: checkpoint() — surgical debugging**

```java
// Add named checkpoints at critical points in your pipeline
userRepository.findById(id)
    .checkpoint("after-user-fetch")
    .flatMap(user -> roleRepository.findByUserId(user.getId()))
    .checkpoint("after-roles-fetch")
    .map(roles -> buildResponse(roles))
    .checkpoint("after-response-build");

// Stack trace will include checkpoint name — pinpoints exactly which stage failed
```

### 4.2 Logging in Reactive Pipelines

```java
// log() operator — logs all signals (subscribe, request, onNext, onError, onComplete)
userRepository.findById(id)
    .log("user-fetch")          // logs with category "user-fetch"
    .flatMap(this::enrichUser)
    .log("user-enrich", Level.DEBUG)  // logs at DEBUG level
    .subscribe();

// Output:
// [user-fetch] | onSubscribe([MonoFlatMap])
// [user-fetch] | request(unbounded)
// [user-fetch] | onNext(User{id=1})
// [user-fetch] | onComplete()
```

__Structured logging with doOn_ hooks:_*

```java
return userRepository.findById(id)
    .doOnSubscribe(sub -> log.debug("Fetching user {}", id))
    .doOnNext(user -> log.debug("Found user: {}", user.getId()))
    .doOnError(ex -> log.error("Failed to fetch user {}: {}", id, ex.getMessage()))
    .doOnSuccess(user -> log.debug("Pipeline complete for user {}", id));
```

### 4.3 Context Propagation (MDC / tracing)

A classic problem: MDC (Mapped Diagnostic Context) for logging correlation IDs doesn't work across reactive thread switches.

```java
// WRONG — MDC lost after scheduler switch
MDC.put("requestId", requestId);
return userRepository.findById(id)
    .subscribeOn(Schedulers.boundedElastic()); // MDC not available here
```

**Solution: Reactor Context**

```java
// Store in Reactor Context — propagated automatically across operators
return userRepository.findById(id)
    .contextWrite(ctx -> ctx.put("requestId", requestId));

// Read in operators
.flatMap(user -> Mono.deferContextual(ctx -> {
    String requestId = ctx.get("requestId");
    log.debug("[{}] Processing user {}", requestId, user.getId());
    return enrichUser(user);
}))
```

**Spring Boot auto-configures Micrometer tracing (Sleuth replacement) for this:**

```yaml
# application.yml
management:
  tracing:
    sampling:
      probability: 1.0
```

```java
// Automatically propagates trace context across reactive boundaries
// No manual MDC needed when using Micrometer + Spring WebFlux
```

### 4.4 Testing Reactive Code

```java
// StepVerifier — the correct way to test reactive pipelines
@Test
void testUserFetch() {
    StepVerifier.create(userService.findById(1L))
        .assertNext(user -> {
            assertThat(user.getId()).isEqualTo(1L);
            assertThat(user.getName()).isNotBlank();
        })
        .verifyComplete();
}

// Test error scenarios
@Test
void testUserNotFound() {
    StepVerifier.create(userService.findById(999L))
        .expectError(UserNotFoundException.class)
        .verify();
}

// Test time-based operators with virtual time
@Test
void testWithDelay() {
    StepVerifier.withVirtualTime(() ->
        Flux.interval(Duration.ofHours(1)).take(3)
    )
    .expectSubscription()
    .expectNoEvent(Duration.ofHours(1))
    .expectNext(0L)
    .thenAwait(Duration.ofHours(2))
    .expectNext(1L, 2L)
    .verifyComplete();
}
```

### 4.5 Common Debugging Checklist

```
□ Pipeline not executing?
  → Check: is someone subscribing? Is it a Mono<Mono<T>> due to wrong map/flatMap?

□ NullPointerException with no clear cause?
  → Add .checkpoint() or enable ReactorDebugAgent

□ Data arriving in wrong order?
  → Check: flatMap (unordered) vs concatMap (ordered)

□ Memory leak / unbounded queue?
  → Check: flatMap without concurrency limit, missing backpressure strategy

□ Blocking exception on event loop thread?
  → Install BlockHound to detect, move to boundedElastic

□ MDC / tracing context lost?
  → Use Reactor Context + Micrometer tracing instead of ThreadLocal MDC

□ Test always passing even with wrong data?
  → Ensure StepVerifier.verify() or verifyComplete() is called — silently passes if not
```

## 5. Migration Strategies

### 5.1 Migration Approaches

**Approach A: Big Bang — full rewrite (not recommended)**

```
Risk: high | Duration: months | Team disruption: severe
Only viable for: greenfield rewrites or complete service replacement
```

**Approach B: Strangler Fig — incremental boundary replacement (recommended)**

```
Replace at the edges first (HTTP layer), work inward over time
Each iteration delivers value and builds team familiarity
```

**Approach C: Hybrid — coexistence (practical for brownfield)**

```
Keep imperative core, add reactive for new high-concurrency features
Use Spring MVC + WebFlux in the same application (supported)
```

### 5.2 The Strangler Fig Pattern Applied

```
Phase 1 — HTTP layer only:
  Controller: Mono/Flux return types (WebFlux)
  Service: still imperative
  Repository: still JDBC

Controller wraps service calls:
  return Mono.fromCallable(() -> userService.findById(id))
             .subscribeOn(Schedulers.boundedElastic());
```

```
Phase 2 — Service layer:
  Service returns Mono/Flux
  Calls downstream reactively
  Wraps remaining blocking calls

Phase 3 — Repository layer:
  Migrate JDBC → R2DBC
  Remove Schedulers.boundedElastic() wrappers
  Fully non-blocking end-to-end
```

### 5.4 Migrating a Specific Service — Step by Step

```java
// BEFORE — imperative Spring MVC service
@Service
public class OrderService {
    public OrderDto getOrder(Long id) {
        Order order = orderRepository.findById(id)
            .orElseThrow(() -> new OrderNotFoundException(id));
        User user = userRepository.findById(order.getUserId())
            .orElseThrow(() -> new UserNotFoundException(order.getUserId()));
        List<OrderItem> items = itemRepository.findByOrderId(id);
        return OrderDto.from(order, user, items);
    }
}
```

```java
// STEP 1 — Return Mono, wrap blocking calls
@Service
public class OrderService {
    public Mono<OrderDto> getOrder(Long id) {
        return Mono.fromCallable(() -> {
            Order order = orderRepository.findById(id)
                .orElseThrow(() -> new OrderNotFoundException(id));
            User user = userRepository.findById(order.getUserId())
                .orElseThrow(() -> new UserNotFoundException(order.getUserId()));
            List<OrderItem> items = itemRepository.findByOrderId(id);
            return OrderDto.from(order, user, items);
        }).subscribeOn(Schedulers.boundedElastic());
    }
}
```

```java
// STEP 2 — Decompose into reactive chain, still using blocking repos
@Service
public class OrderService {
    public Mono<OrderDto> getOrder(Long id) {
        Mono<Order> orderMono = Mono.fromCallable(() ->
            orderRepository.findById(id).orElseThrow(() -> new OrderNotFoundException(id))
        ).subscribeOn(Schedulers.boundedElastic());

        return orderMono.flatMap(order ->
            Mono.zip(
                Mono.fromCallable(() -> userRepository.findById(order.getUserId())
                    .orElseThrow(() -> new UserNotFoundException(order.getUserId())))
                    .subscribeOn(Schedulers.boundedElastic()),
                Mono.fromCallable(() -> itemRepository.findByOrderId(id))
                    .subscribeOn(Schedulers.boundedElastic())
            ).map(tuple -> OrderDto.from(order, tuple.getT1(), tuple.getT2()))
        );
    }
}
```

```java
// STEP 3 — Replace blocking repos with R2DBC reactive repos (fully non-blocking)
@Service
public class OrderService {
    public Mono<OrderDto> getOrder(Long id) {
        return orderRepository.findById(id)                          // R2DBC
            .switchIfEmpty(Mono.error(new OrderNotFoundException(id)))
            .flatMap(order -> Mono.zip(
                userRepository.findById(order.getUserId())           // R2DBC
                    .switchIfEmpty(Mono.error(new UserNotFoundException(order.getUserId()))),
                itemRepository.findByOrderId(id).collectList()       // R2DBC
            ).map(tuple -> OrderDto.from(order, tuple.getT1(), tuple.getT2()))
        );
    }
}
```

### 5.5 Handling Legacy Blocking Dependencies

Some dependencies will never get reactive drivers (Oracle XA, LDAP, legacy SOAP):

```java
// Strategy: isolate behind a dedicated thread pool with circuit breaker
@Component
public class LegacyLdapClient {

    private final Scheduler ldapScheduler =
        Schedulers.newBoundedElastic(20, 100, "ldap-pool"); // dedicated pool

    public Mono<UserInfo> lookupUser(String username) {
        return Mono.fromCallable(() -> legacyLdapTemplate.lookup(username))
            .subscribeOn(ldapScheduler)
            .timeout(Duration.ofSeconds(3))
            .retryWhen(Retry.backoff(2, Duration.ofMillis(500)));
    }
}
```

## 6. Side-by-Side Comparison

### 6.1 Code Style Comparison

```java
// Same feature — get user with roles and recent orders

// ── IMPERATIVE ─────────────────────────────────────────────────────────

public UserPageDto getUserPage(Long userId) {
    User user = userRepository.findById(userId)         // blocks 20ms
        .orElseThrow(() -> new UserNotFoundException(userId));

    List<Role> roles = roleRepository.findByUserId(userId); // blocks 15ms

    List<Order> orders = orderRepository                    // blocks 25ms
        .findRecentByUserId(userId, 10);

    return UserPageDto.from(user, roles, orders);           // total: ~60ms sequential
}

// ── REACTIVE ────────────────────────────────────────────────────────────

public Mono<UserPageDto> getUserPage(Long userId) {
    return Mono.zip(
        userRepository.findById(userId)                     // 20ms
            .switchIfEmpty(Mono.error(new UserNotFoundException(userId))),
        roleRepository.findByUserId(userId).collectList(),  // 15ms
        orderRepository.findRecentByUserId(userId, 10)
            .collectList()                                  // 25ms
    )
    .map(t -> UserPageDto.from(t.getT1(), t.getT2(), t.getT3()));
    // Total: ~25ms (max of parallel calls, not sum)
}
```

|Concern|Imperative|Reactive|
|---|---|---|
|Readability|High (sequential flow)|Lower (pipeline thinking)|
|Debuggability|High (simple stack traces)|Low (deep reactor frames)|
|Concurrency model|Thread-per-request|Event loop|
|I/O concurrency|Sequential by default|Parallel by default|
|Memory under load|High (thread stacks)|Low|

|Blocking I/O|Native|Requires `subscribeOn` workaround|

## 7. Interview Questions

**Q: What is the primary advantage of reactive over imperative Spring MVC for I/O-bound workloads?**

A: Thread efficiency. In Spring MVC, each request occupies a thread from Tomcat's pool (default 200) for its entire duration, including the time spent waiting on I/O. At high concurrency, the thread pool exhausts and requests queue. In Spring WebFlux, a small event loop (typically 8 threads) handles all requests — a thread is only used when there's CPU work to do. During I/O waits, the thread is released to process other requests. This allows WebFlux to handle thousands of concurrent connections with the same hardware that MVC would struggle with at a few hundred.

---

**Q: When would you recommend staying with Spring MVC instead of migrating to WebFlux?**

A: Several scenarios favor MVC: when the workload is CPU-bound rather than I/O-bound (reactive doesn't help); when concurrency is low (a team of 50 users doesn't need reactive); when the team lacks reactive experience and deadlines are tight; when integrations rely entirely on blocking drivers (JDBC, legacy SDKs) with no reactive alternatives — in that case you get the complexity of reactive with few of the benefits; and when the application is primarily a batch processor or back-office tool. Reactive is a performance optimization for a specific problem, not a universal best practice.

---

**Q: What is the blocking tax in a reactive application and how do you detect it?**

A: The blocking tax occurs when a blocking call (JDBC query, `Thread.sleep`, synchronous HTTP) executes on an event loop thread. Since the event loop has only a handful of threads, even one blocked thread meaningfully reduces throughput — at high concurrency, all threads can be blocked simultaneously, causing a complete stall. Detect it with BlockHound (`BlockHound.install()`), which throws an exception whenever a blocking call is detected on a non-blocking thread. The fix is to move blocking calls to `Schedulers.boundedElastic()` via `subscribeOn`.

---

**Q: What are the main debugging challenges with reactive code and how do you address them?**

A: Three main challenges: first, stack traces are deep and filled with Reactor internals — the actual error location is hard to find. Address with `ReactorDebugAgent.init()` (low overhead) or `Hooks.onOperatorDebug()` (high overhead, dev only), and use `.checkpoint("name")` to mark specific pipeline stages. Second, context like MDC correlation IDs is lost when operators switch threads, because MDC uses `ThreadLocal`. Address by using Reactor's `Context` API and Micrometer's reactive tracing support. Third, tests that don't call `StepVerifier.verify()` silently pass even if the pipeline errors. Always use `StepVerifier` and always terminate with `verify()`, `verifyComplete()`, or `verifyError()`.

**Q: Describe a safe migration strategy from Spring MVC to WebFlux.**

A: The strangler fig approach: migrate incrementally at the edges, working inward. Phase 1 — change controller return types to `Mono<T>` and `Flux<T>` while wrapping existing service calls in `Mono.fromCallable(...).subscribeOn(Schedulers.boundedElastic())`. The application is now reactive at the HTTP boundary but still uses blocking I/O internally. Phase 2 — refactor services to return reactive types and decompose sequential blocking calls into parallel reactive pipelines. Phase 3 — migrate repositories to R2DBC, eliminating the `subscribeOn` wrappers and achieving fully non-blocking I/O. This approach lets the team learn reactive incrementally and delivers some throughput improvement even before the full migration.

---

**Q: If reactive Flux adds 50-200ns overhead per item compared to a plain loop, why would you use it for stream processing?**

A: The overhead matters only when the pipeline is purely in-memory and CPU-bound. In real stream processing, each item involves I/O — a Kafka record requires deserialization, a DB write, maybe an HTTP call. At 50ms per item, the 150ns operator overhead is negligible (0.0003%). What reactive buys you is backpressure (the consumer controls how many items arrive), built-in concurrency control (`flatMap(f, maxConcurrency)`), composable error handling, and natural integration with reactive databases and messaging. For pure in-memory computation on a list, a `parallelStream()` is faster and simpler.

## 8. Key Takeaways

- Reactive wins when **concurrency is high** and the work is **I/O-bound** — this is the only scenario where the thread model advantage materializes
- Reactive loses to imperative for CPU-bound work, low concurrency, blocking-only dependencies, and inexperienced teams
- The **blocking tax** — any blocking call on an event loop thread — is the most dangerous reactive antipattern; detect with BlockHound
- Debugging requires deliberate effort: `ReactorDebugAgent`, `.checkpoint()`, `.log()`, and Micrometer tracing for context propagation
- The **strangler fig migration** — HTTP layer first, service layer second, repository layer last — is the safest path
- Running MVC and WebFlux side by side in one Spring Boot app is supported and practical for brownfield migrations
- Always measure before migrating: reactive's value must be demonstrated in metrics, not assumed
