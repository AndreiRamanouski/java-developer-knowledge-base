# Reactive Design Patterns

## Overview

Reactive design patterns are architectural blueprints for building systems that are **responsive, resilient, elastic, and message-driven** — the four pillars of the Reactive Manifesto. These patterns go beyond operator usage; they define how you structure entire features and services around reactive principles.

**Why patterns matter**: Reactive code is easy to write incorrectly — blocking calls hidden inside pipelines, fire-and-forget operations that silently fail, event streams with no backpressure. These patterns encode the correct way to solve recurring problems.

---

## 1. Request-Response Pattern

The most familiar pattern — a client sends a request and waits for a single response. In reactive systems, "waits" means subscribing to a `Mono`, not blocking a thread.

### 1.1 Basic Structure

```
Client ──── request ────► Service ──── Mono<Response> ────► Client
```

The key insight: the response is represented as a `Mono<T>` — a lazy, composable future. Nothing executes until subscribed, and the calling thread is never blocked.

### 1.2 Spring WebFlux Controller (HTTP request-response)

```java
@RestController
@RequestMapping("/api/users")
@RequiredArgsConstructor
public class UserController {

    private final UserService userService;

    // GET — single item response
    @GetMapping("/{id}")
    public Mono<ResponseEntity<UserDto>> getUser(@PathVariable Long id) {
        return userService.findById(id)
            .map(user -> ResponseEntity.ok(toDto(user)))
            .defaultIfEmpty(ResponseEntity.notFound().build());
    }

    // POST — create and return created resource
    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public Mono<UserDto> createUser(@RequestBody @Valid Mono<CreateUserRequest> request) {
        return request
            .flatMap(userService::create)
            .map(this::toDto);
    }
}
```

### 1.3 Service Layer — composing Monos

```java
@Service
@RequiredArgsConstructor
public class UserService {

    private final UserRepository userRepository;
    private final RoleRepository roleRepository;
    private final AuditService auditService;

    public Mono<UserDetails> findById(Long id) {
        // Parallel fetch — user and roles simultaneously
        return Mono.zip(
            userRepository.findById(id)
                .switchIfEmpty(Mono.error(new UserNotFoundException(id))),
            roleRepository.findByUserId(id).collectList()
        )
        .map(tuple -> new UserDetails(tuple.getT1(), tuple.getT2()))
        .doOnSuccess(user -> auditService.logAccess(id).subscribe()) // fire-and-forget audit
        .doOnError(ex -> log.error("Failed to find user {}", id, ex));
    }
}
```

### 1.4 WebClient — reactive HTTP client (service-to-service)

```java
@Component
@RequiredArgsConstructor
public class PaymentClient {

    private final WebClient webClient;

    public Mono<PaymentResult> charge(ChargeRequest request) {
        return webClient
            .post()
            .uri("/payments/charge")
            .contentType(MediaType.APPLICATION_JSON)
            .bodyValue(request)
            .retrieve()
            .onStatus(HttpStatus::is4xxClientError,
                response -> response.bodyToMono(ErrorResponse.class)
                    .flatMap(err -> Mono.error(new PaymentException(err.getMessage()))))
            .onStatus(HttpStatus::is5xxServerError,
                response -> Mono.error(new ServiceUnavailableException("Payment service down")))
            .bodyToMono(PaymentResult.class)
            .timeout(Duration.ofSeconds(10))
            .retryWhen(Retry.backoff(3, Duration.ofMillis(500))
                .filter(ex -> ex instanceof ServiceUnavailableException));
    }
}
```

### 1.5 Parallel Request-Response (scatter-gather)

```java
// Fetch data from 3 services simultaneously, combine into one response
public Mono<DashboardDto> getDashboard(Long userId) {
    Mono<UserProfile> profile = profileService.getProfile(userId);
    Mono<List<Order>> orders  = orderService.getRecentOrders(userId, 10);
    Mono<AccountBalance> balance = billingService.getBalance(userId);

    return Mono.zip(profile, orders, balance)
        .map(tuple -> DashboardDto.builder()
            .profile(tuple.getT1())
            .orders(tuple.getT2())
            .balance(tuple.getT3())
            .build())
        .timeout(Duration.ofSeconds(5));
}
```

### 1.6 Pattern Rules

- Always return `Mono<T>` from service methods — never `T` (blocking)
- Use `switchIfEmpty(Mono.error(...))` to signal "not found" as an error when appropriate
- Use `Mono.zip()` for parallel calls — don't chain with `flatMap` when calls are independent
- Apply `timeout()` on all outbound calls
- Never call `.block()` inside a reactive pipeline

## 2. Fire-and-Forget Pattern

Send a message or trigger an operation **without waiting for the result**. The caller doesn't need confirmation — it just needs the operation submitted.

### 2.1 Basic Structure

```
Caller ──── trigger ────► Operation (runs independently)
  └── continues immediately
```

### 2.2 The Wrong Way (common mistake)

```java
// WRONG — subscription is never made, nothing happens
public Mono<User> createUser(CreateUserRequest request) {
    return userRepository.save(request.toUser())
        .doOnSuccess(user -> {
            emailService.sendWelcomeEmail(user); // returns Mono, never subscribed!
            auditService.logCreation(user);      // same problem
        });
}
```

### 2.3 The Right Way — subscribing to side effects

```java
// CORRECT — explicitly subscribe to the fire-and-forget operation
public Mono<User> createUser(CreateUserRequest request) {
    return userRepository.save(request.toUser())
        .doOnSuccess(user -> {
            // Subscribe explicitly — this is intentional fire-and-forget
            emailService.sendWelcomeEmail(user)
                .doOnError(ex -> log.error("Email failed for {}: {}", user.getId(), ex.getMessage()))
                .onErrorComplete()   // swallow error — we don't want it to affect the caller
                .subscribe();        // explicit subscribe — intentional

            auditService.logCreation(user)
                .onErrorComplete()
                .subscribe();
        });
}
```

### 2.4 Better approach — using Schedulers for isolation

```java
// Fire-and-forget with thread isolation
public Mono<User> createUser(CreateUserRequest request) {
    return userRepository.save(request.toUser())
        .doOnSuccess(user ->
            emailService.sendWelcomeEmail(user)
                .subscribeOn(Schedulers.boundedElastic()) // run on separate thread pool
                .onErrorResume(ex -> {
                    log.error("Welcome email failed: {}", ex.getMessage());
                    return Mono.empty(); // silently recover
                })
                .subscribe()
        );
}
```

### 2.5 Event publishing as fire-and-forget

```java
@Component
@RequiredArgsConstructor
public class OrderService {

    private final OrderRepository orderRepository;
    private final Sinks.Many<OrderEvent> eventSink;

    public Mono<Order> placeOrder(PlaceOrderRequest request) {
        return orderRepository.save(request.toOrder())
            .doOnSuccess(order -> {
                // Publish event — subscribers handle it asynchronously
                Sinks.EmitResult result = eventSink.tryEmitNext(
                    new OrderPlacedEvent(order.getId(), order.getUserId())
                );
                if (result.isFailure()) {
                    log.warn("Failed to publish OrderPlacedEvent: {}", result);
                }
            });
    }
}

// Separate handler processes events
@Component
public class OrderEventHandler {

    @Autowired
    public void setup(Sinks.Many<OrderEvent> eventSink,
                      InventoryService inventoryService,
                      NotificationService notificationService) {

        eventSink.asFlux()
            .flatMap(event -> Mono.when(
                inventoryService.reserve(event.getOrderId()),
                notificationService.notifyUser(event.getUserId())
            ))
            .onErrorContinue((ex, event) ->
                log.error("Failed processing event {}: {}", event, ex.getMessage()))
            .subscribeOn(Schedulers.boundedElastic())
            .subscribe();
    }
}
```

### 2.6 Kafka fire-and-forget (Spring Kafka reactive)

```java
@Component
@RequiredArgsConstructor
public class EventPublisher {

    private final KafkaSender<String, String> kafkaSender;
    private final ObjectMapper objectMapper;

    public <T> void publishAsync(String topic, String key, T event) {
        String payload = serialize(event);
        SenderRecord<String, String, String> record = SenderRecord.create(
            new ProducerRecord<>(topic, key, payload), key
        );

        // Fire and forget — don't wait for ack
        kafkaSender.send(Mono.just(record))
            .doOnNext(result -> log.debug("Published to {}: {}", topic, key))
            .doOnError(ex -> log.error("Failed to publish to {}: {}", topic, ex.getMessage()))
            .onErrorComplete()
            .subscribe();
    }
}
```

### 2.7 Pattern Rules

- Always `.subscribe()` explicitly — a `Mono` that isn't subscribed does nothing
- Always attach `.onErrorResume()` or `.onErrorComplete()` before subscribing — unhandled errors in fire-and-forget are silently dropped or crash threads
- Log failures — fire-and-forget doesn't mean "doesn't matter if it fails"
- Use `subscribeOn(Schedulers.boundedElastic())` to isolate fire-and-forget work from the main pipeline
- Consider `Sinks` or a message broker for durable fire-and-forget

## 3. Stream Processing Pattern

Continuously process a potentially infinite stream of data — transforming, filtering, aggregating, and routing events as they arrive.

### 3.1 Basic Structure

```
Source ──► [Transform] ──► [Filter] ──► [Aggregate] ──► Sink
  (Kafka)    (map)          (filter)     (window)       (DB/WebSocket)
```

### 3.2 Basic Spring WebFlux SSE (Server-Sent Events)

```java
// Server pushes a stream to browser — infinite Flux as HTTP response
@GetMapping(value = "/events", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<ServerSentEvent<String>> streamEvents() {
    return Flux.interval(Duration.ofSeconds(1))
        .map(tick -> ServerSentEvent.<String>builder()
            .id(String.valueOf(tick))
            .event("tick")
            .data("Server time: " + Instant.now())
            .build());
}

// Domain event stream
@GetMapping(value = "/orders/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<OrderDto> streamOrders(@RequestParam Long userId) {
    return orderEventService.getOrderStream()
        .filter(event -> event.getUserId().equals(userId))
        .map(this::toDto)
        .doOnCancel(() -> log.info("Client disconnected: {}", userId));
}
```

### 3.3 Windowed aggregation

### 3.4 Stream branching and routing

```java
// Route different event types to different processors
Flux<OrderEvent> eventStream = kafkaReceiver.receive()
    .map(this::deserialize);

// Branch 1: fraud detection for high-value orders
eventStream
    .filter(e -> e instanceof OrderPlacedEvent)
    .cast(OrderPlacedEvent.class)
    .filter(e -> e.getAmount().compareTo(HIGH_VALUE_THRESHOLD) > 0)
    .flatMap(fraudDetectionService::analyze)
    .subscribe(this::handleFraudResult);

// Branch 2: inventory updates for all orders
eventStream
    .filter(e -> e instanceof OrderPlacedEvent || e instanceof OrderCancelledEvent)
    .flatMap(inventoryService::update)
    .subscribe(this::handleInventoryUpdate);

// Branch 3: analytics for everything
eventStream
    .flatMap(analyticsService::record)
    .onErrorContinue((ex, event) -> log.warn("Analytics failed for {}", event))
    .subscribe();
```

### 3.5 Reactive Kafka stream processing

```java
@Component
@RequiredArgsConstructor
public class OrderStreamProcessor {

    private final ReceiverOptions<String, String> receiverOptions;
    private final OrderRepository orderRepository;

    @PostConstruct
    public void startProcessing() {
        KafkaReceiver.create(receiverOptions)
            .receive()
            .flatMap(record -> processRecord(record)
                .doOnSuccess(v -> record.receiverOffset().acknowledge())
                .doOnError(ex -> log.error("Failed: {}", record.value(), ex))
                .onErrorComplete()
            , 4) // max 4 concurrent processing
            .retryWhen(Retry.backoff(Long.MAX_VALUE, Duration.ofSeconds(1))
                .maxBackoff(Duration.ofSeconds(30)))
            .subscribeOn(Schedulers.boundedElastic())
            .subscribe();
    }

    private Mono<Void> processRecord(ReceiverRecord<String, String> record) {
        return Mono.fromCallable(() -> objectMapper.readValue(record.value(), OrderEvent.class))
            .flatMap(event -> orderRepository.save(event.toOrder()))
            .then();
    }
}
```

### 3.6 Backpressure in stream processing

```java
// Control the rate at which Kafka messages are consumed
KafkaReceiver.create(receiverOptions)
    .receive()
    .onBackpressureBuffer(1000,
        record -> log.warn("Buffer full, dropping: {}", record.key()),
        BackpressureOverflowStrategy.DROP_OLDEST)
    .flatMap(record -> processSlowly(record), 8) // max 8 concurrent
    .subscribe();

// Rate limiting — process at most N items per second
Flux.interval(Duration.ofMillis(100)) // tick every 100ms
    .zipWith(itemFlux)                 // pair tick with item — natural rate limit
    .map(Tuple2::getT2)                // unwrap item
    .flatMap(this::processItem)
    .subscribe();
```

### 3.7 Pattern Rules

- Use `onErrorContinue` on the main processing loop so one bad message doesn't kill the stream
- Limit `flatMap` concurrency explicitly — unbounded concurrency can overwhelm downstream systems
- Always acknowledge/commit offsets only after successful processing
- Use `retryWhen` with exponential backoff on the outer loop for transient infrastructure failures
- Apply backpressure before CPU-intensive operators to avoid memory buildup

## 4. Event Sourcing with Reactive

Event sourcing stores the **sequence of events that led to the current state**, rather than just the current state. Reactive makes this natural — an entity's state is a `reduce()` over its event stream.

### 4.1 Core Concept

```
Events stored:  [AccountOpened, MoneyDeposited(100), MoneyWithdrawn(30)]
Current state:  Account { balance: 70 }   ← derived by replaying events
```

### 4.2 Event and Aggregate definitions

```java
// Domain events
public sealed interface AccountEvent permits
    AccountOpened, MoneyDeposited, MoneyWithdrawn, AccountClosed {
    UUID accountId();
    Instant occurredAt();
    long sequenceNumber();
}

public record AccountOpened(UUID accountId, String ownerId,
                             Instant occurredAt, long sequenceNumber) implements AccountEvent {}
public record MoneyDeposited(UUID accountId, BigDecimal amount,
                              Instant occurredAt, long sequenceNumber) implements AccountEvent {}
public record MoneyWithdrawn(UUID accountId, BigDecimal amount,
                              Instant occurredAt, long sequenceNumber) implements AccountEvent {}

// Aggregate — reconstructed from events
public class Account {
    private UUID id;
    private String ownerId;
    private BigDecimal balance = BigDecimal.ZERO;
    private AccountStatus status;

    // Apply events to build state
    public Account apply(AccountEvent event) {
        return switch (event) {
            case AccountOpened e -> {
                this.id = e.accountId();
                this.ownerId = e.ownerId();
                this.status = AccountStatus.ACTIVE;
                yield this;
            }
            case MoneyDeposited e -> {
                this.balance = this.balance.add(e.amount());
                yield this;
            }
            case MoneyWithdrawn e -> {
                this.balance = this.balance.subtract(e.amount());
                yield this;
            }
            case AccountClosed e -> {
                this.status = AccountStatus.CLOSED;
                yield this;
            }
        };
    }
}
```

### 4.3 Event Store Repository

```java
public interface EventStoreRepository {
    // Append new events (optimistic concurrency via expectedVersion)
    Mono<Void> append(UUID aggregateId, List<AccountEvent> events, long expectedVersion);

    // Load all events for an aggregate
    Flux<AccountEvent> loadEvents(UUID aggregateId);

    // Load events after a sequence number (for incremental replay)
    Flux<AccountEvent> loadEvents(UUID aggregateId, long afterSequence);

    // Subscribe to new events (for projections)
    Flux<AccountEvent> subscribeToEvents(UUID aggregateId);
}

@Repository
@RequiredArgsConstructor
public class R2dbcEventStoreRepository implements EventStoreRepository {

    private final DatabaseClient databaseClient;
    private final ObjectMapper objectMapper;

    @Override
    public Flux<AccountEvent> loadEvents(UUID aggregateId) {
        return databaseClient.sql("""
                SELECT event_type, payload, sequence_number, occurred_at
                FROM account_events
                WHERE aggregate_id = :aggregateId
                ORDER BY sequence_number ASC
                """)
            .bind("aggregateId", aggregateId)
            .map(row -> deserialize(row))
            .all();
    }

    @Override
    public Mono<Void> append(UUID aggregateId, List<AccountEvent> events, long expectedVersion) {
        // Check current version, then insert — transactional
        return databaseClient.sql(
                "SELECT MAX(sequence_number) FROM account_events WHERE aggregate_id = :id")
            .bind("id", aggregateId)
            .mapValue(Long.class)
            .one()
            .defaultIfEmpty(-1L)
            .flatMap(currentVersion -> {
                if (!currentVersion.equals(expectedVersion)) {
                    return Mono.error(new OptimisticLockException(
                        "Expected version " + expectedVersion + " but was " + currentVersion));
                }
                return insertEvents(aggregateId, events);
            });
    }
}
```

### 4.4 Aggregate Service — load, decide, append

```java
@Service
@RequiredArgsConstructor
public class AccountCommandService {

    private final EventStoreRepository eventStore;

    // Reconstruct current state by replaying all events
    private Mono<Account> loadAccount(UUID id) {
        return eventStore.loadEvents(id)
            .reduce(new Account(), Account::apply)  // fold events into state
            .switchIfEmpty(Mono.error(new AccountNotFoundException(id)));
    }

    public Mono<Void> deposit(UUID accountId, BigDecimal amount) {
        return loadAccount(accountId)
            .flatMap(account -> {
                // Validate business rules
                if (account.getStatus() != AccountStatus.ACTIVE) {
                    return Mono.error(new AccountNotActiveException(accountId));
                }

                // Produce new event
                MoneyDeposited event = new MoneyDeposited(
                    accountId, amount, Instant.now(),
                    account.getSequenceNumber() + 1
                );

                // Append to store (optimistic lock on current sequence number)
                return eventStore.append(accountId, List.of(event), account.getSequenceNumber());
            });
    }

    public Mono<Void> withdraw(UUID accountId, BigDecimal amount) {
        return loadAccount(accountId)
            .flatMap(account -> {
                if (account.getBalance().compareTo(amount) < 0) {
                    return Mono.error(new InsufficientFundsException(accountId));
                }

                MoneyWithdrawn event = new MoneyWithdrawn(
                    accountId, amount, Instant.now(),
                    account.getSequenceNumber() + 1
                );

                return eventStore.append(accountId, List.of(event), account.getSequenceNumber());
            });
    }
}
```

### 4.5 Projections — building read models from event streams

```java
// A projection subscribes to the event stream and builds a query-optimized view
@Component
@RequiredArgsConstructor
public class AccountBalanceProjection {

    private final EventStoreRepository eventStore;
    private final BalanceViewRepository balanceViewRepository;

    // Subscribe to all new events and update read model
    @PostConstruct
    public void start() {
        eventStore.subscribeToAllEvents()
            .ofType(AccountEvent.class)
            .flatMap(this::applyToReadModel)
            .onErrorContinue((ex, event) ->
                log.error("Projection failed for event {}", event, ex))
            .subscribeOn(Schedulers.boundedElastic())
            .subscribe();
    }

    private Mono<Void> applyToReadModel(AccountEvent event) {
        return switch (event) {
            case AccountOpened e ->
                balanceViewRepository.insert(new BalanceView(e.accountId(), BigDecimal.ZERO));
            case MoneyDeposited e ->
                balanceViewRepository.incrementBalance(e.accountId(), e.amount());
            case MoneyWithdrawn e ->
                balanceViewRepository.decrementBalance(e.accountId(), e.amount());
            default -> Mono.empty();
        };
    }
}
```

### 4.6 Snapshot optimization

```java
// For aggregates with many events, periodically snapshot state to avoid full replay
@Service
@RequiredArgsConstructor
public class SnapshotService {

    private final SnapshotRepository snapshotRepository;
    private final EventStoreRepository eventStore;
    private static final int SNAPSHOT_EVERY = 100;

    public Mono<Account> loadWithSnapshot(UUID accountId) {
        return snapshotRepository.findLatest(accountId)
            .flatMap(snapshot ->
                // Replay only events AFTER the snapshot
                eventStore.loadEvents(accountId, snapshot.getSequenceNumber())
                    .reduce(snapshot.getState(), Account::apply)
            )
            .switchIfEmpty(
                // No snapshot — full replay
                eventStore.loadEvents(accountId)
                    .reduce(new Account(), Account::apply)
            );
    }

    public Mono<Void> snapshotIfNeeded(UUID accountId, Account state) {
        if (state.getSequenceNumber() % SNAPSHOT_EVERY == 0) {
            return snapshotRepository.save(
                new AccountSnapshot(accountId, state, state.getSequenceNumber())
            );
        }
        return Mono.empty();
    }
}
```

## 5. CQRS with Reactive

**Command Query Responsibility Segregation** separates write operations (commands) from read operations (queries). Reactive makes this natural — commands return `Mono<Void>`, queries return `Mono<T>` or `Flux<T>`, and the two sides communicate via events.

### 5.1 Architecture

```
                    ┌─────────────────────────────────────────┐
Client              │  Command Side          Query Side        │
  │                 │                                          │
  ├── Command ─────►│  CommandHandler ──► EventStore           │
  │                 │       │                    │             │
  │                 │       └── Events ──────────►─────────┐   │
  │                 │                        Projection    │   │
  └── Query ───────►│  QueryHandler ◄──── ReadModel ◄──────┘   │
                    └─────────────────────────────────────────┘
```

### 5.2 Command model

```java
// Commands — intent to change state
public sealed interface AccountCommand permits
    OpenAccount, DepositMoney, WithdrawMoney, CloseAccount {}

public record OpenAccount(UUID accountId, String ownerId, BigDecimal initialDeposit)
    implements AccountCommand {}
public record DepositMoney(UUID accountId, BigDecimal amount, String reference)
    implements AccountCommand {}
public record WithdrawMoney(UUID accountId, BigDecimal amount, String reference)
    implements AccountCommand {}

// Command handler — validates and produces events
@Component
@RequiredArgsConstructor
public class AccountCommandHandler {

    private final EventStoreRepository eventStore;

    public Mono<Void> handle(OpenAccount command) {
        // Idempotency check — don't open twice
        return eventStore.exists(command.accountId())
            .flatMap(exists -> {
                if (exists) return Mono.empty(); // already processed
                return eventStore.append(command.accountId(), List.of(
                    new AccountOpened(command.accountId(), command.ownerId(),
                        Instant.now(), 0),
                    new MoneyDeposited(command.accountId(), command.initialDeposit(),
                        Instant.now(), 1)
                ), -1L);
            });
    }

    public Mono<Void> handle(DepositMoney command) {
        return loadAccount(command.accountId())
            .flatMap(account -> eventStore.append(command.accountId(), List.of(
                new MoneyDeposited(command.accountId(), command.amount(),
                    Instant.now(), account.getSequenceNumber() + 1)
            ), account.getSequenceNumber()));
    }
}
```

### 5.5 Eventual consistency between write and read sides

```java
// The gap between command execution and read model update is the eventual consistency window
// Handle this in the API layer

@PostMapping("/{accountId}/deposits")
public Mono<ResponseEntity<Void>> deposit(@PathVariable UUID accountId,
                                           @RequestBody DepositRequest request) {
    return commandHandler.handle(new DepositMoney(accountId, request.getAmount(), request.getReference()))
        .then(Mono.just(ResponseEntity.accepted().<Void>build()))
        // HTTP 202 Accepted signals: "command received, read model may not reflect it yet"
        // Client should poll or use event stream for confirmation
        .onErrorReturn(InsufficientFundsException.class,
            ResponseEntity.unprocessableEntity().build());
}
```

### 5.6 CQRS + Event Sourcing combined

```java
// Full pipeline: Command → Events → Event Store → Projection → Read Model
//
// Write side:
//   1. Load aggregate by replaying events
//   2. Validate command against current state
//   3. Produce new events
//   4. Append events to event store
//
// Read side (async):
//   5. Event store publishes new events
//   6. Projections consume events and update read models
//   7. Queries read from denormalized view tables

@Service
@RequiredArgsConstructor
public class CqrsAccountService {

    private final AccountCommandHandler commandHandler;
    private final AccountQueryHandler queryHandler;
    private final EventStoreRepository eventStore;

    // Command — fire and confirm eventually
    public Mono<UUID> openAccount(OpenAccountRequest request) {
        UUID accountId = UUID.randomUUID();
        return commandHandler.handle(new OpenAccount(accountId, request.getOwnerId(),
                request.getInitialDeposit()))
            .thenReturn(accountId);
    }

    // Query — always reads from read model, never event store directly
    public Mono<AccountSummaryView> getAccount(UUID id) {
        return queryHandler.getAccountSummary(id);
    }
}
```

## 6. Pattern Comparison Summary

|Pattern|Returns|Caller blocks?|Best For|
|---|---|---|---|
|Request-Response|`Mono<T>`|No (reactive)|HTTP APIs, service calls|
|Fire-and-Forget|`void` / `Mono<Void>`|No|Notifications, audit, analytics|
|Stream Processing|`Flux<T>`|No|Event pipelines, Kafka, SSE|
|Event Sourcing|`Mono<Void>` (write) / `Flux<Event>` (read)|No|Audit trail, temporal queries|
|CQRS|`Mono<Void>` (command) / `Mono<T>` or `Flux<T>` (query)|No|High read/write ratio, scalability|

## 7. Interview Questions

**Q: What is the core difference between fire-and-forget and request-response in a reactive system?**

A: In request-response, the caller subscribes to a `Mono<T>` and waits (reactively) for the result before continuing — the pipeline is connected end to end. In fire-and-forget, the caller submits the operation and immediately returns, without subscribing to any result. The key implementation detail is that fire-and-forget requires an explicit `.subscribe()` on the inner operation, with error handling attached, because an unsubscribed `Mono` in `doOnSuccess` silently does nothing.

---

**Q: How does event sourcing differ from traditional state-based persistence, and why is reactive a good fit?**

A: Traditional persistence stores only current state — you overwrite the row with each update, losing history. Event sourcing stores the append-only log of events; current state is derived by replaying them. Reactive is a natural fit because the event log is a `Flux<Event>`, and deriving state is just `reduce(new State(), State::apply)`. The event stream is also naturally composable with projections, subscriptions, and stream processing pipelines.

---

**Q: What is eventual consistency in CQRS and how do you handle it in a Spring WebFlux API?**

A: In CQRS, the command side writes to the event store while the query side reads from a denormalized view built by projections. There is a window between a command completing and the projection updating the read model — this is eventual consistency. In a WebFlux API, you handle it by returning HTTP 202 Accepted instead of 200 OK for commands, signaling that the operation is acknowledged but the read model may not yet reflect it. Clients should use event streams (SSE or WebSocket) or poll with a version number to detect when the read model catches up.

---

**Q: What is the danger of calling `.subscribe()` inside a reactive pipeline, and when is it acceptable?**

A: Calling `.subscribe()` inside a pipeline (e.g., inside `flatMap` or `doOnNext`) creates a detached subscription — errors from it won't propagate to the outer pipeline, it runs on an uncontrolled thread, and it's easy to introduce resource leaks. It's acceptable _only_ for fire-and-forget operations where you genuinely don't need the result and have attached explicit error handling before the subscribe. In all other cases, use `flatMap` to chain the inner publisher into the main pipeline so errors propagate correctly.

---

**Q: How do you handle backpressure in a Kafka stream processing pipeline in Spring?**

A: `KafkaReceiver` is a `Flux` of records. Backpressure is controlled by limiting concurrency in `flatMap(record -> process(record), maxConcurrent)` — this prevents subscribing to more processing tasks than the system can handle. For overflow, `onBackpressureBuffer` with a bounded capacity and a drop strategy prevents unbounded memory growth. Offset acknowledgment must happen only after successful processing to ensure at-least-once delivery, and `retryWhen` with exponential backoff handles transient failures without stopping the stream.

---

**Q: Why do projections in event sourcing use `onErrorContinue` rather than `onErrorResume`?**

A: A projection is a long-running infinite stream processing all events for a read model. If one event fails to apply (e.g., a deserialization error or a transient DB write failure), you don't want the entire projection to stop — that would halt all read model updates. `onErrorContinue` lets you log the error and skip the problematic event, keeping the stream running. `onErrorResume` is better for finite pipelines where you want to switch to a fallback when an error occurs.

## 8. Key Takeaways

- **Request-response**: Always return `Mono<T>` from service methods, use `Mono.zip()` for parallel calls, apply `timeout()` on all outbound requests
- **Fire-and-forget**: Always `.subscribe()` explicitly, always attach error handling first, use `subscribeOn(Schedulers.boundedElastic())` for isolation
- **Stream processing**: Use `onErrorContinue` on infinite loops, limit `flatMap` concurrency, acknowledge only after success
- **Event sourcing**: State = `reduce()` over events; append-only store; use snapshots for large aggregates; projections build read models from event streams
- **CQRS**: Commands return `Mono<Void>`, queries return `Mono<T>` or `Flux<T>`; return HTTP 202 for commands; never read from the event store on the query path
- All five patterns compose naturally — CQRS and Event Sourcing are frequently combined, with reactive stream processing connecting the two sides
