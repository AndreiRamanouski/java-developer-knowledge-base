# Top K Elements

## Overview

Top K problems ask for the k largest, k smallest, k most frequent, or k closest elements from a collection. Three algorithms solve the full family: a min-heap of size k (O(n log k)), Quickselect (O(n) average), and full sort (O(n log n)). Choosing correctly depends on whether k is much smaller than n, whether the data is a stream, and whether an exact or expected-case guarantee is required.

**Algorithm selection guide:**

```
Full sort (O(n log n)):
  Use when: k is close to n, or you need the k elements in sorted order.
  Code: Arrays.sort(), return last k elements. Simple but rarely optimal.

Min-heap of size k (O(n log k)):
  Use when: streaming data (elements arrive one at a time), k << n,
            or the problem asks for k largest/most frequent.
  Always gives correct answer. Preferred in interviews for clarity.
  Space: O(k).

Quickselect (O(n) average, O(n²) worst):
  Use when: need kth element (not all k), array fits in memory,
            and average-case O(n) is acceptable.
  Not suitable for streams. Worst case avoidable with random pivot.
  Space: O(1) in-place (or O(log n) with recursion stack).

Counting sort / bucket sort (O(n)):
  Use when: values are in a bounded integer range.
  Top K Frequent: bucket by frequency (max frequency ≤ n) → O(n).
```

## 1. Heap Fundamentals

```java
// Java PriorityQueue: min-heap by default (smallest element at top).
PriorityQueue<Integer> minHeap = new PriorityQueue<>();
PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Comparator.reverseOrder());

// Min-heap of size k → tracks k LARGEST elements seen so far.
// Invariant: heap.peek() = smallest of the k largest = the kth largest overall.
// When a new element arrives:
//   If > heap.peek(): it belongs in top k → add it, remove the smallest.
//   If ≤ heap.peek(): not in top k → discard.

// Pattern:
for (int num : nums) {
    minHeap.offer(num);
    if (minHeap.size() > k) minHeap.poll(); // evict smallest
}
// After processing all n elements: heap contains the k largest.
// heap.peek() = kth largest.

// Similarly, max-heap of size k → tracks k SMALLEST elements.
```

## 2. Kth Largest Element in an Array — LeetCode 215

### 2.1 Min-heap solution

```java
// Time: O(n log k)  Space: O(k)
// Maintain a min-heap of size k. After all elements processed,
// the heap contains the k largest; heap.peek() is the kth largest.
public int findKthLargest(int[] nums, int k) {
    PriorityQueue<Integer> minHeap = new PriorityQueue<>(k);

    for (int num : nums) {
        minHeap.offer(num);
        if (minHeap.size() > k) {
            minHeap.poll(); // remove smallest: maintains only k largest
        }
    }
    return minHeap.peek(); // smallest of the k largest = kth largest
}
```

**Trace on `nums=[3,2,1,5,6,4]`, `k=2`:**

```
Process 3: heap=[3]
Process 2: heap=[2,3]
Process 1: heap=[1,2,3] → size>2 → poll 1 → heap=[2,3]
Process 5: heap=[2,3,5] → size>2 → poll 2 → heap=[3,5]
Process 6: heap=[3,5,6] → size>2 → poll 3 → heap=[5,6]
Process 4: heap=[4,5,6] → size>2 → poll 4 → heap=[5,6]
heap.peek() = 5 ✓ (5 is the 2nd largest)
```

### 2.2 Quickselect — O(n) average

```java
// Partition-based selection. Finds kth largest without full sort.
// Time: O(n) average, O(n²) worst  Space: O(1) in-place (O(log n) stack)
//
// Key: we want kth largest = (n - k)th smallest (0-indexed).
// Partition around pivot: pivot goes to its final sorted position.
//   If pivot index == target: done.
//   If pivot index < target:  kth largest is in right partition.
//   If pivot index > target:  kth largest is in left partition.
public int findKthLargestQuickSelect(int[] nums, int k) {
    int target = nums.length - k; // kth largest = (n-k)th element when sorted ascending
    return quickSelect(nums, 0, nums.length - 1, target);
}

private int quickSelect(int[] nums, int lo, int hi, int target) {
    if (lo == hi) return nums[lo]; // single element

    // Random pivot to avoid O(n²) worst case on sorted/reverse-sorted input
    int pivotIdx = lo + (int)(Math.random() * (hi - lo + 1));
    swap(nums, pivotIdx, hi); // move pivot to end

    int partitionIdx = partition(nums, lo, hi);

    if      (partitionIdx == target) return nums[partitionIdx];
    else if (partitionIdx < target)  return quickSelect(nums, partitionIdx + 1, hi, target);
    else                             return quickSelect(nums, lo, partitionIdx - 1, target);
}

// Lomuto partition: pivot = nums[hi], place all ≤ pivot to the left.
// Returns final index of pivot.
private int partition(int[] nums, int lo, int hi) {
    int pivot = nums[hi];
    int i = lo; // i = boundary: everything left of i is ≤ pivot

    for (int j = lo; j < hi; j++) {
        if (nums[j] <= pivot) {
            swap(nums, i, j);
            i++;
        }
    }
    swap(nums, i, hi); // place pivot at its final position
    return i;
}

private void swap(int[] nums, int i, int j) {
    int t = nums[i]; nums[i] = nums[j]; nums[j] = t;
}
```

**Quickselect trace on `[3,2,1,5,6,4]`, k=2, target=4:**

```
quickSelect(lo=0, hi=5, target=4)
  pivot=4 (say), partition around 4:
  [3,2,1,4,6,5]? Let's say partition returns index 3 (4 is 4th element, 0-indexed)
  partitionIdx=3 < target=4 → recurse right: quickSelect(lo=4, hi=5, target=4)
    pivot=6, partition:
    [5,6] → partitionIdx=5 (6 at end), 5 < target? no. 5 > 4? yes → recurse left:
    quickSelect(lo=4, hi=4, target=4)
      lo==hi → return nums[4] = 5 ✓
```

### 2.3 Three-way partition (Dutch National Flag) for duplicates

```java
// If array has many duplicates, three-way partition is more efficient.
// Avoids O(n²) when many elements equal the pivot.
private int[] threeWayPartition(int[] nums, int lo, int hi) {
    int pivot = nums[hi];
    int lt = lo, gt = hi, i = lo; // [lo,lt)=<pivot, [lt,i]=pivot, (gt,hi]=>pivot

    while (i <= gt) {
        if      (nums[i] < pivot) swap(nums, lt++, i++);
        else if (nums[i] > pivot) swap(nums, i,   gt--);
        else                      i++;
    }
    return new int[]{lt, gt}; // range [lt,gt] contains all elements == pivot
}
```

## 3. Top K Frequent Elements — LeetCode 347

### 3.1 Min-heap by frequency

```java
// Time: O(n log k)  Space: O(n) for frequency map + O(k) for heap
public int[] topKFrequent(int[] nums, int k) {
    // Step 1: count frequencies
    Map<Integer, Integer> freq = new HashMap<>();
    for (int num : nums) freq.merge(num, 1, Integer::sum);

    // Step 2: min-heap ordered by frequency (smallest frequency at top)
    // When heap exceeds k: evict the element with lowest frequency
    PriorityQueue<Map.Entry<Integer, Integer>> minHeap =
        new PriorityQueue<>(Comparator.comparingInt(Map.Entry::getValue));

    for (Map.Entry<Integer, Integer> entry : freq.entrySet()) {
        minHeap.offer(entry);
        if (minHeap.size() > k) minHeap.poll(); // remove least frequent
    }

    // Step 3: extract results (k most frequent elements)
    int[] result = new int[k];
    for (int i = k - 1; i >= 0; i--) {
        result[i] = minHeap.poll().getKey();
    }
    return result;
}
```

**Trace on `nums=[1,1,1,2,2,3]`, `k=2`:**

```
freq = {1:3, 2:2, 3:1}

Process (1,3): heap=[(1,3)]
Process (2,2): heap=[(2,2),(1,3)]
Process (3,1): heap=[(3,1),(1,3),(2,2)] → size>2 → poll (3,1) (smallest freq) → heap=[(2,2),(1,3)]

Result: [1,2] ✓ (top 2 most frequent)
```

### 3.2 Bucket sort — O(n) solution

```java
// Frequency can be at most n (if all elements are the same).
// Create buckets indexed by frequency; collect from highest bucket down.
// Time: O(n)  Space: O(n)
public int[] topKFrequentBucket(int[] nums, int k) {
    Map<Integer, Integer> freq = new HashMap<>();
    for (int num : nums) freq.merge(num, 1, Integer::sum);

    // Bucket[i] = list of elements with frequency i
    List<Integer>[] buckets = new List[nums.length + 1];
    for (Map.Entry<Integer, Integer> e : freq.entrySet()) {
        int f = e.getValue();
        if (buckets[f] == null) buckets[f] = new ArrayList<>();
        buckets[f].add(e.getKey());
    }

    // Collect from highest frequency bucket down to lowest
    int[] result = new int[k];
    int idx = 0;
    for (int f = buckets.length - 1; f >= 1 && idx < k; f--) {
        if (buckets[f] == null) continue;
        for (int num : buckets[f]) {
            result[idx++] = num;
            if (idx == k) return result;
        }
    }
    return result;
}
```

**Trace on `[1,1,1,2,2,3]`, `k=2`:**

```
freq = {1:3, 2:2, 3:1}
buckets[3]=[1], buckets[2]=[2], buckets[1]=[3]

Scan from f=6 down:
  f=6: null
  f=5: null
  f=4: null
  f=3: [1] → result[0]=1, idx=1
  f=2: [2] → result[1]=2, idx=2 == k → return [1,2] ✓
```

## 4. Kth Largest in a Stream — LeetCode 703

```java
// Stream of integers; find kth largest after each add().
// Min-heap of size k: heap.peek() = kth largest at all times.
// Time: O(log k) per add()  Space: O(k)
class KthLargest {
    private final PriorityQueue<Integer> minHeap;
    private final int k;

    public KthLargest(int k, int[] nums) {
        this.k = k;
        this.minHeap = new PriorityQueue<>(k);
        for (int num : nums) add(num);
    }

    public int add(int val) {
        minHeap.offer(val);
        if (minHeap.size() > k) minHeap.poll();
        return minHeap.peek(); // kth largest
    }
}
// Why min-heap (not max-heap):
//   max-heap of size k: peek = largest of k largest = 1st largest (wrong!)
//   min-heap of size k: peek = smallest of k largest = kth largest ✓
```

## 5. K Closest Points to Origin — LeetCode 973

```java
// Find k points closest to origin (0,0). Distance² = x² + y² (no sqrt needed).
// Time: O(n log k)  Space: O(k)
//
// Max-heap of size k: stores k SMALLEST distances (largest at top for eviction).
// When new point arrives: if its distance < heap.peek(), replace top.
public int[][] kClosest(int[][] points, int k) {
    // Max-heap by distance: largest distance at top → evict it when heap exceeds k
    PriorityQueue<int[]> maxHeap = new PriorityQueue<>(
        (a, b) -> (b[0]*b[0] + b[1]*b[1]) - (a[0]*a[0] + a[1]*a[1])
    );

    for (int[] point : points) {
        maxHeap.offer(point);
        if (maxHeap.size() > k) maxHeap.poll(); // remove farthest point
    }

    return maxHeap.toArray(new int[0][]);
}

// Quickselect variant: partition by distance — O(n) average
public int[][] kClosestQuickSelect(int[][] points, int k) {
    quickSelectPoints(points, 0, points.length - 1, k);
    return Arrays.copyOfRange(points, 0, k);
}

private void quickSelectPoints(int[][] pts, int lo, int hi, int k) {
    if (lo >= hi) return;
    int pivotIdx = partition(pts, lo, hi);
    if      (pivotIdx + 1 == k) return;              // exactly k points on left
    else if (pivotIdx + 1 < k)  quickSelectPoints(pts, pivotIdx + 1, hi, k);
    else                        quickSelectPoints(pts, lo, pivotIdx - 1, k);
}

private int partition(int[][] pts, int lo, int hi) {
    long pivotDist = dist(pts[hi]);
    int i = lo;
    for (int j = lo; j < hi; j++) {
        if (dist(pts[j]) <= pivotDist) {
            int[] t = pts[i]; pts[i] = pts[j]; pts[j] = t;
            i++;
        }
    }
    int[] t = pts[i]; pts[i] = pts[hi]; pts[hi] = t;
    return i;
}

private long dist(int[] p) { return (long)p[0]*p[0] + (long)p[1]*p[1]; }
```

## 6. Top K Frequent Words — LeetCode 692

```java
// Return k most frequent words. Tie-break: alphabetical order (lower = higher priority).
// Time: O(n log k)  Space: O(n)
public List<String> topKFrequent(String[] words, int k) {
    Map<String, Integer> freq = new HashMap<>();
    for (String w : words) freq.merge(w, 1, Integer::sum);

    // Min-heap: evict element with lowest priority (lowest freq, or highest alpha on tie)
    // Comparator: REVERSE of desired order → least desirable at top (gets evicted)
    PriorityQueue<String> minHeap = new PriorityQueue<>(
        (a, b) -> freq.get(a).equals(freq.get(b))
            ? b.compareTo(a)          // same freq: higher alphabetical → evict first
            : freq.get(a) - freq.get(b) // lower freq → evict first
    );

    for (String word : freq.keySet()) {
        minHeap.offer(word);
        if (minHeap.size() > k) minHeap.poll();
    }

    // Extract in reverse order (heap gives us least-desirable first)
    LinkedList<String> result = new LinkedList<>();
    while (!minHeap.isEmpty()) result.addFirst(minHeap.poll());
    return result;
}
// ["i","love","leetcode","i","love","coding"], k=2:
// freq={i:2, love:2, leetcode:1, coding:1}
// After processing: heap contains "i" and "love"
// heap.peek() = "love" (alphabetically after "i", evicted on tie)
// Result: ["i","love"] ✓
```

## 7. Find K Pairs with Smallest Sums — LeetCode 373

```java
// Given two sorted arrays, find k pairs (u,v) with smallest u+v.
// Use min-heap seeded with (nums1[i], nums2[0]) for all i.
// On extracting (nums1[i], nums2[j]), push (nums1[i], nums2[j+1]).
// Time: O(k log k)  Space: O(k)
public List<List<Integer>> kSmallestPairs(int[] nums1, int[] nums2, int k) {
    List<List<Integer>> result = new ArrayList<>();
    if (nums1.length == 0 || nums2.length == 0) return result;

    // Min-heap stores [sum, i (index into nums1), j (index into nums2)]
    PriorityQueue<int[]> minHeap = new PriorityQueue<>(
        Comparator.comparingInt(a -> a[0])
    );

    // Seed: pair each nums1[i] with nums2[0]
    for (int i = 0; i < Math.min(nums1.length, k); i++) {
        minHeap.offer(new int[]{nums1[i] + nums2[0], i, 0});
    }

    while (!minHeap.isEmpty() && result.size() < k) {
        int[] curr = minHeap.poll();
        int i = curr[1], j = curr[2];
        result.add(List.of(nums1[i], nums2[j]));

        if (j + 1 < nums2.length) {
            minHeap.offer(new int[]{nums1[i] + nums2[j + 1], i, j + 1});
        }
    }
    return result;
}
// nums1=[1,7,11], nums2=[2,4,6], k=3:
// Seed: heap=[(3,0,0),(9,1,0),(13,2,0)]
// Extract (3,0,0)=[1,2], push (5,0,1): heap=[(5,0,1),(9,1,0),(13,2,0)]
// Extract (5,0,1)=[1,4], push (7,0,2): heap=[(7,0,2),(9,1,0),(13,2,0)]
// Extract (7,0,2)=[1,6], push (no j+1): heap=[(9,1,0),(13,2,0)]
// Result: [[1,2],[1,4],[1,6]] ✓
```

## 8. Reorganise String — LeetCode 767

```java
// Arrange string so no two adjacent characters are the same.
// Greedy with max-heap: always place the most frequent available character.
// Time: O(n log n)  Space: O(n)
public String reorganizeString(String s) {
    int[] freq = new int[26];
    for (char c : s.toCharArray()) freq[c - 'a']++;

    // Max-heap by frequency
    PriorityQueue<int[]> maxHeap = new PriorityQueue<>(
        (a, b) -> b[1] - a[1] // higher frequency first
    );
    for (int i = 0; i < 26; i++) {
        if (freq[i] > 0) maxHeap.offer(new int[]{i, freq[i]});
    }

    // If any character frequency > ceil(n/2): impossible
    if (maxHeap.peek()[1] > (s.length() + 1) / 2) return "";

    StringBuilder sb = new StringBuilder();
    while (maxHeap.size() >= 2) {
        int[] first  = maxHeap.poll(); // most frequent
        int[] second = maxHeap.poll(); // second most frequent

        sb.append((char)('a' + first[0]));
        sb.append((char)('a' + second[0]));

        if (--first[1]  > 0) maxHeap.offer(first);
        if (--second[1] > 0) maxHeap.offer(second);
    }
    if (!maxHeap.isEmpty()) sb.append((char)('a' + maxHeap.poll()[0]));
    return sb.toString();
}
// "aab": freq={a:2,b:1}, maxHeap=[(a,2),(b,1)]
// Extract (a,2),(b,1): sb="ab", push (a,1)
// One element left: sb="aba" ✓
```

## 9. Quickselect — Deep Dive

### 9.1 Why random pivot avoids O(n²)

```
Without randomisation:
  Sorted array [1,2,3,4,5], find kth smallest, always pick last element as pivot:
    Partition: [1,2,3,4] | 5 → pivot at index 4
    Partition: [1,2,3] | 4   → pivot at index 3
    ...each partition reduces by 1 → n + (n-1) + ... + 1 = O(n²)

With random pivot:
  Expected partition: roughly splits array in half.
  Recurrence: T(n) = T(n/2) + O(n) → O(n) by master theorem.
  (Same as why Quicksort is O(n log n) expected)

Median-of-three pivot (deterministic O(n) guarantee):
  Choose median of first, middle, last elements as pivot.
  Avoids worst case on sorted/reverse-sorted inputs.
  Not true O(n) worst case (adversary can still construct bad input).

True O(n) worst case: Median-of-Medians (rarely needed in interviews).
```

### 9.2 Iterative Quickselect (avoids stack overflow on large inputs)

```java
public int quickSelectIterative(int[] nums, int k) {
    int target = nums.length - k;
    int lo = 0, hi = nums.length - 1;

    while (lo < hi) {
        // Random pivot
        int pivotIdx = lo + (int)(Math.random() * (hi - lo + 1));
        swap(nums, pivotIdx, hi);
        int p = partition(nums, lo, hi);

        if      (p == target) return nums[p];
        else if (p < target)  lo = p + 1;
        else                  hi = p - 1;
    }
    return nums[lo];
}
```

### 9.3 Heap vs Quickselect decision matrix

```
                    Heap O(n log k)              Quickselect O(n) avg
Streaming data:     ✓ YES — add one at a time    ✗ NO — needs full array
Exact worst case:   ✓ O(n log k) always          ✗ O(n²) without randomisation
Returns all k:      ✓ YES — all k in heap        Partial — need extra step
In-place:           ✗ O(k) extra                 ✓ O(1) extra (iterative)
Best avg complexity:✗ O(n log k)                 ✓ O(n)
Code simplicity:    ✓ Simpler                    ✗ More complex

Rule of thumb:
  k << n AND streaming → heap
  k << n AND batch AND avg-case O(n) acceptable → Quickselect
  Need all k in sorted order → sort O(n log n)
  Frequency-based → bucket sort O(n) when range bounded
```

## 11. Interview Questions

**Q: Why use a min-heap of size k to find the k largest elements, rather than a max-heap?**

A: A min-heap of size k maintains the k largest elements seen so far, with the smallest of those k elements sitting at the top — that element is exactly the kth largest. When a new element arrives, we compare it against `heap.peek()`: if it's larger, it belongs in the top k, so we add it and evict the current minimum (maintaining size k); if it's smaller or equal, it can't be in the top k and is discarded. The heap always represents a current answer, and `heap.peek()` is that answer after all elements are processed. A max-heap of size k would keep the k smallest elements, with the largest of those at the top — `heap.peek()` would then be the kth smallest, not the kth largest. The counterintuitive pairing — min-heap for k largest, max-heap for k smallest — is because the heap's root is what gets evicted, so we want to evict the boundary element (smallest of the k largest, or largest of the k smallest) when the heap exceeds size k.

---

**Q: Explain Quickselect and its expected O(n) time complexity.**

A: Quickselect applies the partition step from Quicksort repeatedly, but only recurses into the half that contains the target element. After partitioning around a pivot, the pivot lands at its final sorted position. If that position equals k, we're done. If it's less than k, the kth element is in the right partition; if greater, it's in the left. Each partition step takes O(n) for the current subarray, and with a random pivot, each partition is expected to reduce the problem size by roughly half. The recurrence T(n) = T(n/2) + O(n) resolves to O(n) by the master theorem — the first partition does O(n) work, the second O(n/2), the third O(n/4), summing to O(n)(1 + 1/2 + 1/4 + ...) = O(2n) = O(n). Without a random pivot, an adversary can construct input where every partition is maximally unbalanced — always picking the smallest or largest element as pivot on a sorted array, reducing the subarray by only 1 each time, giving O(n²). Randomisation makes this astronomically unlikely.

---

**Q: When would you choose bucket sort over a heap for Top K Frequent Elements?**

A: Bucket sort applies when element frequencies are bounded and we need O(n) guaranteed time rather than O(n log k). Since element frequencies range from 1 to n (you can't appear more than n times in an n-element array), we create n+1 buckets where bucket[f] holds all elements with frequency f. After counting frequencies and filling buckets, we scan from bucket[n] down to bucket[1], collecting elements until we have k. This is O(n) end-to-end — frequency counting is O(n), bucket filling is O(distinct elements) ≤ O(n), and collection is O(n). The heap approach is O(n log k), which for small k is better in the constant sense but asymptotically worse. The bucket approach uses more memory (O(n) buckets vs O(k) heap), and only works when the value domain is bounded integers — it doesn't apply to Top K Frequent Words where frequencies are bounded but we also need to handle tie-breaking by alphabetical order.

---

**Q: What is the time complexity of finding k pairs with the smallest sums from two sorted arrays, and how does the heap approach exploit sorting?**

A: The heap approach runs in O(k log k). The key insight is that the pairs are generated in sorted order of their sums by exploiting the sorted structure of both arrays. Seed the min-heap with pairs (nums1[i], nums2[0]) for i in [0, min(n1, k)] — these are the k smallest possible "starting" pairs since nums2[0] is the minimum of nums2. Extract the minimum-sum pair (nums1[i], nums2[j]); the next candidate pair from the same nums1[i] is (nums1[i], nums2[j+1]) — push it. This lazily generates the next-best candidate from each row of the conceptual n1×n2 sum matrix. Each of the k extractions takes O(log k) heap operations, and at most k pairs are pushed, giving O(k log k) total. This is optimal for the output alone — just listing k pairs takes O(k) — and the log k factor comes from heap maintenance. We never enumerate all n1×n2 pairs, making this far better than sorting all sums at O(n1·n2·log(n1·n2)).

---

**Q: How do you handle tie-breaking in a custom comparator for a heap, and what goes wrong if the comparator is inconsistent?**

A: Java's `PriorityQueue` requires a comparator satisfying the contract: `compare(a, b) < 0` iff a has higher priority than b, consistently. For Top K Frequent Words, two words with the same frequency should be ordered alphabetically — lower alphabetical order = higher priority (should appear in result). Since we use a min-heap that evicts the lowest-priority element, the comparator must put the worst element at the top: lower frequency is worse, and for equal frequency, higher alphabetical order is worse. The comparator `(a, b) -> freq.get(a).equals(freq.get(b)) ? b.compareTo(a) : freq.get(a) - freq.get(b)` achieves this — it returns positive when b has lower or equal frequency and is alphabetically before a, meaning b is "less than" a in heap priority and sits closer to the top for eviction. An inconsistent comparator — one where `compare(a,b) < 0` and `compare(b,a) < 0` simultaneously, or where transitivity breaks — causes undefined heap behaviour in Java, potentially putting higher-priority elements below lower-priority ones and returning wrong results silently.

## 12. Key Takeaways

- **Min-heap of size k tracks k largest**: `heap.peek()` = kth largest; evict when `size > k`; counter-intuitive but correct
- **Max-heap of size k tracks k smallest**: `heap.peek()` = kth smallest; evict when `size > k`
- **Heap time: O(n log k)** — each of n elements does at most one offer + one poll, each O(log k)
- **Quickselect time: O(n) average** — random pivot prevents O(n²) worst case; iterative version avoids stack overflow
- **Heap beats Quickselect for streams**: Quickselect requires the full array in memory; heap processes one element at a time
- **Bucket sort for Top K Frequent when frequencies are bounded**: O(n) vs O(n log k) for heap — always O(n) since max frequency ≤ n
- **K pairs smallest sum**: seed heap with (nums1[i], nums2[0]), push (nums1[i], nums2[j+1]) on each extraction — O(k log k)
- **Custom comparator in min-heap = "worst element at top"**: comparator defines eviction order, not desired result order — design to put the least desirable of the k elements at the top
- **Quickselect modifies the array** — acceptable for batch problems, disqualifying for read-only or streaming scenarios
- **Reorganise String**: max-heap greedy — always place the two most frequent characters alternately; return empty if any frequency > ⌈n/2⌉
