# Merge Intervals

## Overview

Interval problems involve ranges defined by a start and end point. The core challenge is determining how intervals relate to each other — do they overlap, can they be merged, do they conflict? Sorting by start time is the universal preprocessing step that converts a chaotic set of intervals into a linear scan problem: once sorted, each interval only needs to be compared with its immediate neighbours.

**The four ways two intervals can relate:**

```
Interval A: [─────]
Interval B:

1. No overlap (B before A):   [──B──] [──A──]
2. No overlap (B after A):    [──A──] [──B──]
3. Partial overlap:           [──A──]
                                   [──B──]
4. Full containment:          [────A────]
                                 [─B─]

After sorting by start:
  Only cases 2, 3, 4 need checking (case 1 cannot occur between adjacent intervals).
  Overlap test: B.start <= A.end  (B starts before A ends)
  Merged result: [min(A.start, B.start), max(A.end, B.end)]
  Since sorted: min(A.start, B.start) = A.start always
```

**Sorting as preprocessing — why it works:**

```
Unsorted: must compare every pair → O(n²)
Sorted by start: each interval only overlaps with its immediate predecessor (in the
  output list) because any interval that could overlap A must start before A ends,
  and we process in start-time order.

Intervals [a,b] and [c,d] overlap iff c ≤ b (given a ≤ c after sorting).
If they DON'T overlap (c > b): all subsequent intervals also don't overlap with [a,b]
  (they all start ≥ c > b) → can safely close [a,b] and move on.
```

## 1. Merge Intervals — LeetCode 56

### 1.1 Core algorithm

```java
// Merge all overlapping intervals. Return sorted, non-overlapping list.
// Time: O(n log n)  Space: O(n) for output
public int[][] merge(int[][] intervals) {
    if (intervals.length <= 1) return intervals;

    // Step 1: Sort by start time — enables linear scan
    Arrays.sort(intervals, (a, b) -> Integer.compare(a[0], b[0]));

    List<int[]> merged = new ArrayList<>();
    int[] current = intervals[0]; // current interval being extended

    for (int i = 1; i < intervals.length; i++) {
        int[] next = intervals[i];

        if (next[0] <= current[1]) {
            // Overlapping: extend current interval's end if needed
            current[1] = Math.max(current[1], next[1]);
        } else {
            // No overlap: commit current, start tracking next
            merged.add(current);
            current = next;
        }
    }
    merged.add(current); // add the last interval
    return merged.toArray(new int[0][]);
}
```

**Trace on `[[1,3],[2,6],[8,10],[15,18]]`:**

```
After sort: [[1,3],[2,6],[8,10],[15,18]] (already sorted)
current = [1,3]

i=1: next=[2,6], 2 ≤ 3 → overlap → current=[1, max(3,6)]=[1,6]
i=2: next=[8,10], 8 > 6 → no overlap → add [1,6], current=[8,10]
i=3: next=[15,18], 15 > 10 → no overlap → add [8,10], current=[15,18]
Final add: [15,18]

Output: [[1,6],[8,10],[15,18]] ✓
```

**Trace on `[[1,4],[4,5]]` (touching intervals):**

```
current = [1,4]
i=1: next=[4,5], 4 ≤ 4 → overlap (touching counts!) → current=[1,5]
Output: [[1,5]] ✓

Key: use ≤ not < in overlap check — intervals that share an endpoint are merged.
```

**Trace on `[[1,4],[2,3]]` (fully contained):**

```
current = [1,4]
i=1: next=[2,3], 2 ≤ 4 → overlap → current=[1, max(4,3)]=[1,4] (unchanged)
Output: [[1,4]] ✓

Key: max(current.end, next.end) handles the containment case.
```

### 1.2 Common variants

```java
// Variant: return total covered length after merging
public int totalCoveredLength(int[][] intervals) {
    if (intervals.length == 0) return 0;
    Arrays.sort(intervals, (a, b) -> a[0] - b[0]);

    int total = 0;
    int[] current = intervals[0];
    for (int i = 1; i < intervals.length; i++) {
        if (intervals[i][0] <= current[1]) {
            current[1] = Math.max(current[1], intervals[i][1]);
        } else {
            total += current[1] - current[0];
            current = intervals[i];
        }
    }
    total += current[1] - current[0];
    return total;
}

// Variant: count number of merges performed
public int countMerges(int[][] intervals) {
    Arrays.sort(intervals, (a, b) -> a[0] - b[0]);
    int merges = 0;
    int[] current = intervals[0];

    for (int i = 1; i < intervals.length; i++) {
        if (intervals[i][0] <= current[1]) {
            current[1] = Math.max(current[1], intervals[i][1]);
            merges++;
        } else {
            current = intervals[i];
        }
    }
    return merges;
}
```

## 2. Insert Interval — LeetCode 57

### 2.1 Algorithm

```java
// Insert newInterval into a sorted, non-overlapping list of intervals.
// Merge any overlapping intervals that result.
// Time: O(n)  Space: O(n) — list is already sorted, no re-sort needed
//
// Three-phase scan:
//   Phase 1: Add all intervals that END before newInterval starts (no overlap)
//   Phase 2: Merge all intervals that OVERLAP with newInterval
//   Phase 3: Add all intervals that START after newInterval ends (no overlap)
public int[][] insert(int[][] intervals, int[] newInterval) {
    List<int[]> result = new ArrayList<>();
    int i = 0, n = intervals.length;

    // Phase 1: intervals ending before newInterval starts → add unchanged
    while (i < n && intervals[i][1] < newInterval[0]) {
        result.add(intervals[i++]);
    }

    // Phase 2: intervals overlapping with newInterval → merge into newInterval
    while (i < n && intervals[i][0] <= newInterval[1]) {
        newInterval[0] = Math.min(newInterval[0], intervals[i][0]);
        newInterval[1] = Math.max(newInterval[1], intervals[i][1]);
        i++;
    }
    result.add(newInterval); // add merged interval

    // Phase 3: intervals starting after newInterval ends → add unchanged
    while (i < n) {
        result.add(intervals[i++]);
    }

    return result.toArray(new int[0][]);
}
```

**Trace on `intervals=[[1,2],[3,5],[6,7],[8,10],[12,16]]`, `newInterval=[4,8]`:**

```
Phase 1: intervals[i][1] < 4
  i=0: [1,2], 2 < 4 → add [1,2], i=1
  i=1: [3,5], 5 < 4? NO → stop

Phase 2: intervals[i][0] <= 8
  i=1: [3,5], 3 ≤ 8 → merge: new=[min(4,3),max(8,5)]=[3,8], i=2
  i=2: [6,7], 6 ≤ 8 → merge: new=[min(3,6),max(8,7)]=[3,8], i=3
  i=3: [8,10], 8 ≤ 8 → merge: new=[min(3,8),max(8,10)]=[3,10], i=4
  i=4: [12,16], 12 ≤ 8? NO → stop
  Add [3,10]

Phase 3: add remaining
  i=4: add [12,16]

Output: [[1,2],[3,10],[12,16]] ✓
```

**Trace on `intervals=[[1,5]]`, `newInterval=[2,3]` (insertion inside existing):**

```

Phase 2: [1,5].start=1 ≤ 3 → merge: new=[min(2,1),max(3,5)]=[1,5]

Output: [[1,5]] ✓ (newInterval absorbed by existing)
```

**Trace on `intervals=[[1,5]]`, `newInterval=[6,8]` (no overlap, insert after):**

```

```

## 3. Meeting Rooms I — LeetCode 252

### 3.1 Can one person attend all meetings?

```java
// Given meeting intervals, determine if a person can attend ALL meetings.
// No two meetings can overlap.
// Time: O(n log n)  Space: O(1)
public boolean canAttendMeetings(int[][] intervals) {
    Arrays.sort(intervals, (a, b) -> Integer.compare(a[0], b[0]));

    for (int i = 1; i < intervals.length; i++) {
        // If next meeting starts before previous ends → conflict
        if (intervals[i][0] < intervals[i - 1][1]) return false;
    }
    return true;
}
// [[0,30],[5,10],[15,20]]:
//   After sort: [[0,30],[5,10],[15,20]]
//   i=1: 5 < 30 → return false ✓
//
// [[7,10],[2,4]]:
//   After sort: [[2,4],[7,10]]
//   i=1: 7 < 4? NO → continue
//   return true ✓

// Note: use < not ≤ for end==start is NOT a conflict (back-to-back meetings are fine)
```

## 4. Meeting Rooms II — LeetCode 253

### 4.1 Minimum conference rooms required

```java
// Minimum number of rooms needed to schedule all meetings.
// At any point in time, the room count = number of meetings overlapping at that instant.
// The answer = maximum simultaneous overlap at any point.
// Time: O(n log n)  Space: O(n)

// Approach 1: Min-heap tracking room end times
// Process meetings in order of start time.
// If earliest-ending room is free (its end ≤ next start), reuse it.
// Otherwise, allocate a new room.
public int minMeetingRooms(int[][] intervals) {
    if (intervals.length == 0) return 0;
    Arrays.sort(intervals, (a, b) -> Integer.compare(a[0], b[0]));

    // Min-heap: tracks end times of ongoing meetings (earliest end at top)
    PriorityQueue<Integer> heap = new PriorityQueue<>();

    for (int[] meeting : intervals) {
        // If earliest-ending room finishes before this meeting starts → reuse it
        if (!heap.isEmpty() && heap.peek() <= meeting[0]) {
            heap.poll(); // free the room
        }
        heap.offer(meeting[1]); // assign room: record end time
    }
    return heap.size(); // rooms still occupied = minimum rooms needed
}
```

**Trace on `[[0,30],[5,10],[15,20]]`:**

```

heap = []

meeting=[0,30]: heap empty → add 30.   heap=[30]          rooms=1
meeting=[5,10]: heap.peek()=30 > 5 → no free room → add 10. heap=[10,30]  rooms=2
meeting=[15,20]: heap.peek()=10 ≤ 15 → free room → remove 10 → add 20. heap=[20,30] rooms=2

return heap.size() = 2 ✓
```

**Trace on `[[1,5],[2,6],[3,7],[4,8],[4,10]]`:**

### 4.2 Approach 2: Separate start/end arrays

```java
// Chronological event sweep: treat each start as +1 and each end as -1.
// Track running count; maximum is the answer.
// Time: O(n log n)  Space: O(n)
public int minMeetingRoomsEvents(int[][] intervals) {
    int n = intervals.length;
    int[] starts = new int[n];
    int[] ends   = new int[n];

    for (int i = 0; i < n; i++) {
        starts[i] = intervals[i][0];
        ends[i]   = intervals[i][1];
    }
    Arrays.sort(starts);
    Arrays.sort(ends);

    int rooms = 0, maxRooms = 0;
    int j = 0; // pointer into ends array

    for (int i = 0; i < n; i++) {
        // A meeting started: need a room
        rooms++;
        // If the earliest-ending meeting ended before this one starts: free a room
        if (starts[i] >= ends[j]) {
            rooms--;
            j++;
        }
        maxRooms = Math.max(maxRooms, rooms);
    }
    return maxRooms;
}
// Why correct: starts[i] >= ends[j] means the earliest-ending meeting has
// ended by the time starts[i] arrives → its room can be reused.
// We increment j to track which end time we've "consumed".
```

**Trace on `[[0,30],[5,10],[15,20]]`:**

```
starts=[0,5,15]  ends=[10,20,30]  j=0

i=0: rooms=1; starts[0]=0 ≥ ends[0]=10? NO  → maxRooms=1
i=1: rooms=2; starts[1]=5 ≥ ends[0]=10? NO  → maxRooms=2
i=2: rooms=3; starts[2]=15 ≥ ends[0]=10? YES → rooms=2, j=1 → maxRooms=2

return 2 ✓
```

## 5. Non-Overlapping Intervals — LeetCode 435

```java
// Minimum intervals to REMOVE to make the rest non-overlapping.
// Greedy: always keep the interval with the earliest end time (maximises future space).
// This is equivalent to: find maximum number of non-overlapping intervals,
// then answer = total - maximum non-overlapping.
// Time: O(n log n)  Space: O(1)
public int eraseOverlapIntervals(int[][] intervals) {
    if (intervals.length == 0) return 0;

    // Sort by END time — greedy: prefer intervals that end soonest
    Arrays.sort(intervals, (a, b) -> Integer.compare(a[1], b[1]));

    int count = 0;       // intervals removed
    int end = intervals[0][1]; // end time of last KEPT interval

    for (int i = 1; i < intervals.length; i++) {
        if (intervals[i][0] < end) {
            // Overlap with last kept interval → remove current (it ends later by sort order)
            count++;
        } else {
            // No overlap → keep this interval, update end
            end = intervals[i][1];
        }
    }
    return count;
}
// [[1,2],[2,3],[3,4],[1,3]]:
//   Sort by end: [[1,2],[1,3],[2,3],[3,4]]
//   end=2, count=0
//   [1,3]: start=1 < end=2 → overlap → count=1
//   [2,3]: start=2 ≥ end=2 → keep → end=3
//   [3,4]: start=3 ≥ end=3 → keep → end=4
//   return 1 ✓ (remove [1,3])
```

**Why sort by end (not start) for this problem:**

```
Sort by start and keep earliest-start on overlap → may keep a long interval
  that blocks many future intervals.

Sort by end and keep earliest-end on overlap → interval finishes soonest,
  leaving maximum room for subsequent intervals (Activity Selection Problem).

Example: [[1,100],[2,3],[3,4]]
  Sort by end: [[2,3],[3,4],[1,100]]
  Keep [2,3], keep [3,4], [1,100] overlaps with [3,4] → remove [1,100]
  Remove 1 interval ✓

  Sort by start (wrong): keep [1,100], remove [2,3], remove [3,4]
  → removes 2 intervals ✗
```

## 6. Interval List Intersections — LeetCode 986

```java
// Find all intersections between two lists of non-overlapping sorted intervals.
// Two-pointer: advance the pointer whose interval ends first.
// Time: O(m+n)  Space: O(1) ignoring output
public int[][] intervalIntersection(int[][] A, int[][] B) {
    List<int[]> result = new ArrayList<>();
    int i = 0, j = 0;

    while (i < A.length && j < B.length) {
        // Intersection of A[i] and B[j] if they overlap
        int lo = Math.max(A[i][0], B[j][0]); // overlap start
        int hi = Math.min(A[i][1], B[j][1]); // overlap end

        if (lo <= hi) result.add(new int[]{lo, hi}); // valid intersection

        // Advance the pointer whose interval ends sooner
        // (the one that ends sooner cannot intersect with the NEXT interval of the other)
        if (A[i][1] < B[j][1]) i++;
        else                    j++;
    }
    return result.toArray(new int[0][]);
}
// A=[[0,2],[5,10],[13,23],[24,25]]
// B=[[1,5],[8,12],[15,24],[25,26]]
//
// i=0,j=0: lo=max(0,1)=1, hi=min(2,5)=2 → [1,2] ✓; A ends first → i=1
// i=1,j=0: lo=max(5,1)=5, hi=min(10,5)=5 → [5,5] ✓; B ends first → j=1
// i=1,j=1: lo=max(5,8)=8, hi=min(10,12)=10 → [8,10] ✓; A ends first → i=2
// i=2,j=1: lo=max(13,8)=13, hi=min(23,12)=12 → 13>12, no intersection; B ends first → j=2
// i=2,j=2: lo=max(13,15)=15, hi=min(23,24)=23 → [15,23] ✓; A ends first → i=3
// i=3,j=2: lo=max(24,15)=24, hi=min(25,24)=24 → [24,24] ✓; B ends first → j=3
// i=3,j=3: lo=max(24,25)=25, hi=min(25,26)=25 → [25,25] ✓; A ends first → i=4
// i=4: exit loop
// Output: [[1,2],[5,5],[8,10],[15,23],[24,24],[25,25]] ✓
```

## 7. Employee Free Time — LeetCode 759

```java
// Given schedules of multiple employees, find time intervals when ALL are free.
// Flatten all intervals, sort, merge, then find gaps between merged intervals.
// Time: O(n log n)  Space: O(n)
public List<Interval> employeeFreeTime(List<List<Interval>> schedule) {
    // Flatten all employee intervals into one list
    List<int[]> all = new ArrayList<>();
    for (List<Interval> employee : schedule)
        for (Interval interval : employee)
            all.add(new int[]{interval.start, interval.end});

    // Sort and merge
    all.sort((a, b) -> Integer.compare(a[0], b[0]));

    List<int[]> merged = new ArrayList<>();
    int[] current = all.get(0);
    for (int i = 1; i < all.size(); i++) {
        if (all.get(i)[0] <= current[1]) {
            current[1] = Math.max(current[1], all.get(i)[1]);
        } else {
            merged.add(current);
            current = all.get(i);
        }
    }
    merged.add(current);

    // Gaps between consecutive merged intervals = free time
    List<Interval> free = new ArrayList<>();
    for (int i = 1; i < merged.size(); i++) {
        free.add(new Interval(merged.get(i - 1)[1], merged.get(i)[0]));
    }
    return free;
}
// schedule = [[[1,3],[6,7]], [[2,4]], [[2,5],[9,12]]]
// Flattened: [[1,3],[6,7],[2,4],[2,5],[9,12]]
// Sorted:    [[1,3],[2,4],[2,5],[6,7],[9,12]]
// Merged:    [[1,5],[6,7],[9,12]]
// Free time: [5,6], [7,9] ✓
```

## 8. Sorting as Preprocessing — Deeper Analysis

### 8.1 Which field to sort by

```
Sort by START when:
  Processing intervals sequentially, extending/merging as you go
  → Merge Intervals, Insert Interval, Meeting Rooms I and II, Interval Intersections

Sort by END when:
  Greedy keep/skip decisions where "frees up the most future space" matters
  → Non-Overlapping Intervals (Activity Selection)
  → Interval Scheduling Maximisation

Sort by START then END (tie-break) when:
  Multiple intervals with same start but different spans
  → Critical for correct overlap detection in merge
  → Example: [[1,5],[1,3],[1,7]] sorted as [[1,3],[1,5],[1,7]]
     current=[1,3], 1≤3 → current=[1,5], 1≤5 → current=[1,7] ✓

Sort by negative end (descending end) when:
  You want to process the longest intervals first (e.g. calendar blocking problems)
```

### 8.2 Sorting with custom comparators

```java
// Standard: sort by start ascending
Arrays.sort(intervals, (a, b) -> Integer.compare(a[0], b[0]));

// Sort by end ascending (Non-Overlapping Intervals, Activity Selection)
Arrays.sort(intervals, (a, b) -> Integer.compare(a[1], b[1]));

// Sort by start, break ties by end descending
// (useful for "remove covered intervals" — longer intervals first)
Arrays.sort(intervals, (a, b) -> {
    if (a[0] != b[0]) return Integer.compare(a[0], b[0]);
    return Integer.compare(b[1], a[1]); // descending end on tie
});

// Sort Interval objects (not arrays)
Collections.sort(intervalList, Comparator.comparingInt(i -> i.start));

// Integer overflow guard: use Integer.compare, not a[0] - b[0]
// a[0] - b[0] overflows when values are large negatives or large positives
// WRONG: (a, b) -> a[0] - b[0]
// RIGHT: (a, b) -> Integer.compare(a[0], b[0])
```

### 8.3 Remove covered intervals — LeetCode 1288

```java
// Remove all intervals that are covered by another interval.
// [a,b] covered by [c,d] if c ≤ a and b ≤ d.
// Sort by start ascending, break ties by end DESCENDING.
// Then scan: any interval with end ≤ current max end is covered.
// Time: O(n log n)  Space: O(1)
public int removeCoveredIntervals(int[][] intervals) {
    Arrays.sort(intervals, (a, b) ->
        a[0] != b[0] ? Integer.compare(a[0], b[0])
                     : Integer.compare(b[1], a[1])); // descending end for ties

    int count = 0, maxEnd = 0;
    for (int[] interval : intervals) {
        if (interval[1] > maxEnd) {
            count++;           // not covered by any previous interval
            maxEnd = interval[1];
        }
        // else: interval[1] ≤ maxEnd → covered by the previous max-end interval
    }
    return count;
}
// [[1,4],[3,6],[2,8]]:
//   Sort: [[1,8],[1,4],[2,6]] wait — sort by start asc, end desc:
//   [[1,8],[1,4],[3,6]]
//   Wait recalculate: [1,4],[3,6],[2,8] sorted:
//   start=1: [1,4] and no other start=1
//   start=2: [2,8]
//   start=3: [3,6]
//   Sorted: [[1,4],[2,8],[3,6]]
//   maxEnd=0
//   [1,4]: 4>0 → count=1, maxEnd=4
//   [2,8]: 8>4 → count=2, maxEnd=8
//   [3,6]: 6>8? NO → covered → skip
//   return 2 ✓
```

## 9. Pattern Recognition and Decision Tree

```
Is the problem about intervals? →

Do intervals need to be combined?
  YES → Merge Intervals pattern
  "merge all overlapping" → sort by start, linear scan with extend-or-commit

Does a new interval need to be inserted?
  YES → Insert Interval pattern
  "already sorted, non-overlapping" → three-phase scan (before, overlap, after)

Is there a resource constraint (rooms, machines, workers)?
  YES → Meeting Rooms II / Resource Allocation
  "minimum rooms" → sort by start + min-heap of end times
  "maximum meetings in one room" → sort by end + greedy keep earliest-end

Do you want maximum non-overlapping intervals?
  YES → Activity Selection / Non-Overlapping Intervals
  Sort by END, greedily keep earliest-ending non-conflicting interval

Do you need intersections between two lists?
  YES → Interval Intersection
  Two pointers one per list, advance the one ending sooner

Is one interval contained in another?
  YES → Remove Covered / Count Covered
  Sort by start asc, end desc; track running max end

Are there gaps to find?
  YES → Employee Free Time / Find Gaps
  Merge all intervals; gaps between consecutive merged intervals = free time
```

## 11. Interview Questions

**Q: Why do we sort by start time for merging intervals, and what invariant does it establish?**

A: Sorting by start time establishes the invariant that for any two adjacent intervals in the sorted order, the left interval starts no later than the right. This means two adjacent intervals can only relate in three ways after sorting: the right interval starts after the left ends (no overlap), the right interval starts during the left interval (overlap), or the right is fully contained within the left. Crucially, it eliminates the fourth case — right ending before left begins — which can't happen once sorted. This invariant allows a single linear scan: for each interval, we only need to check whether it overlaps with the most recently accumulated interval (the current merge candidate). If there's no overlap, all subsequent intervals also can't overlap with the current one from the left (they all start even later), so we safely commit it and begin a new candidate. Without sorting, every interval could potentially overlap with every other, requiring O(n²) comparisons.

**Q: For Meeting Rooms II, why does the min-heap of end times give the optimal room count?**

A: The min-heap tracks the end times of all currently ongoing meetings, with the earliest-ending meeting at the top. When a new meeting arrives, we check whether any room has been freed by comparing the new meeting's start time against the earliest end time. If the earliest-ending room finishes at or before the new meeting starts, that room is available and we reuse it (pop the old end time, push the new one). If not, we must open a new room. This is optimal because: first, we process meetings in order of start time, so all decisions are made in chronological order. Second, by always reusing the earliest-ending room when available, we never unnecessarily open a new room. The room count at any point equals the number of entries in the heap, which equals the number of meetings still in progress. The heap size at the end is the maximum concurrent meetings, which is exactly the minimum rooms needed.

**Q: What is the difference between sorting by start and sorting by end, and when does each apply?**

A: Sorting by start is correct when you process intervals in temporal order and want to know which ones overlap with the current "active" interval — the standard merge, insert, and room-allocation problems. The key operation is comparing a new interval's start against the current interval's end. Sorting by end is correct for greedy selection problems where you want to maximise the number of intervals you can keep without conflicts — Activity Selection and Non-Overlapping Intervals. The intuition is that an interval ending earlier "frees up time" for more future intervals. By always keeping the interval that ends soonest among overlapping candidates, you leave the most room for subsequent intervals. Choosing to keep a later-ending interval might block two or more future intervals that would otherwise fit. This is the classic Activity Selection greedy proof: the locally optimal choice (earliest end) leads to the globally optimal solution.

**Q: How does Insert Interval work in O(n) without sorting?**

A: The key insight is that the input is already sorted and non-overlapping — no re-sort is needed, so we get O(n) instead of O(n log n). The algorithm makes a single left-to-right scan in three phases. Phase one adds all intervals that end strictly before the new interval starts — these cannot possibly overlap with it and are correct in their original position. Phase two merges all intervals that overlap with the new interval by continuously expanding the new interval's boundaries (taking the minimum start and maximum end of each overlapping pair). Phase three appends all remaining intervals unchanged — they start after the new interval ends and cannot overlap. The three phases together guarantee every interval is processed exactly once. The critical boundary conditions are: phase one ends when `current.end >= newInterval.start` (touching or overlapping); phase two ends when `current.start > newInterval.end` (new interval is strictly before current).

**Q: How would you find the minimum number of arrows to burst all balloons (LeetCode 452)?**

A: This is the Activity Selection / Non-Overlapping Intervals problem in disguise. Each balloon's x-extent is an interval, and one arrow at position x bursts all balloons whose interval contains x. The minimum arrows equals the minimum number of groups of non-overlapping intervals that together cover all balloons, which equals the total count minus the maximum number of non-overlapping intervals. Sort by end time. Greedily select the earliest-ending interval; any arrow shot at its end point also bursts all overlapping balloons (since they all contain a point ≤ that end). Advance past all balloons burst by this arrow, then repeat. The count of arrows equals the number of groups selected. The difference from Non-Overlapping Intervals is the boundary condition: touching intervals `[1,6]` and `[6,8]` are burst by one arrow at x=6 (they share the point), so the overlap condition is `start <= end` (not strict less-than), meaning we use `<=` in the sort-by-end greedy check.

## 12. Key Takeaways

- **Sort by start** for merge, insert, and room-allocation problems — establishes left-to-right processing order where each interval only needs to be compared with its immediate predecessor
- **Sort by end** for maximisation/minimisation problems (Activity Selection, Non-Overlapping Intervals) — earliest-ending non-overlapping interval greedily leaves the most room for future intervals
- **Overlap condition after sort-by-start**: `next.start <= current.end` — touching intervals (`next.start == current.end`) ARE considered overlapping for merging; use `<` (not `<=`) only for Meeting Rooms I where back-to-back meetings don't conflict
- **Merge**: extend `current.end = max(current.end, next.end)` — the `max` handles full containment where next is shorter than current
- **Insert Interval**: three-phase scan — before (add unchanged), overlap (merge into new), after (add unchanged) — O(n) because no sorting needed
- **Meeting Rooms II**: min-heap of end times — heap size = rooms in use at any moment; reuse room when `heap.peek() <= next.start`
- **Non-Overlapping Intervals**: sort by end, keep earliest-ending non-conflicting interval — greedy proof via Activity Selection theorem
- **Interval Intersections**: two pointers on two sorted lists — intersection is `[max(a.start,b.start), min(a.end,b.end)]`; advance the pointer whose interval ends sooner
- **Never use `a[0] - b[0]` in comparators** — integer overflow on large values; always use `Integer.compare(a[0], b[0])`
