# Fast and Slow Pointers

## Overview

The fast and slow pointer technique (also called Floyd's Tortoise and Hare) uses two pointers that traverse a sequence at different speeds. The slow pointer advances one step at a time; the fast pointer advances two. This speed difference creates predictable geometric relationships — if a cycle exists, the pointers must eventually meet; if not, the fast pointer exits first. The same speed ratio naturally positions the slow pointer at structural midpoints.

**Why the speed difference works:**

```
In a cyclic sequence of length λ starting at position μ:

  Slow travels:  t steps in t time units
  Fast travels: 2t steps in t time units

  When they meet (if cycle exists):
    slow = t mod λ  (position in cycle)
    fast = 2t mod λ
    They meet when t ≡ 2t (mod λ) → t ≡ 0 (mod λ)
    → They meet after at most λ steps inside the cycle

  No cycle: fast reaches null → terminate immediately

Key properties that drive all problems:
  1. If cycle exists, fast and slow MUST meet (fast gains 1 step/iteration)
  2. When fast reaches end (null), slow is at the MIDDLE
  3. After meeting inside cycle, distance from meeting point to entry
     equals distance from head to entry (Floyd's proof)
```

**All fast/slow pointer problems reduce to three questions:**

```
1. Does a cycle exist?                → detect meeting point
2. Where does the cycle start?        → Floyd's two-phase algorithm
3. Where is the structural midpoint?  → slow is mid when fast exits
```

## 1. Floyd's Cycle Detection Algorithm

### 1.1 Core algorithm — does a cycle exist?

```java
// Time: O(n)  Space: O(1)
// Alternative HashMap approach: O(n) time but O(n) space — Floyd wins on space
public boolean hasCycle(ListNode head) {
    ListNode slow = head;
    ListNode fast = head;

    while (fast != null && fast.next != null) {
        slow = slow.next;        // advance 1
        fast = fast.next.next;   // advance 2

        if (slow == fast) return true; // pointers met inside cycle
    }
    // fast hit null → sequence is finite, no cycle
    return false;
}

// Why fast.next != null check is needed:
//   If list has odd length: fast lands on last node (not null)
//   fast.next would then throw NullPointerException without the guard
//   Both conditions needed: fast != null (even length) AND fast.next != null (odd length)
```

**Trace on cyclic list `1→2→3→4→5→3` (cycle at node 3):**

```
Nodes:  1 → 2 → 3 → 4 → 5
                ↑_________↑

Step 0: slow=1, fast=1
Step 1: slow=2, fast=3
Step 2: slow=3, fast=5
Step 3: slow=4, fast=4  ← MEET (both at node 4)
return true ✓

Why node 4?
  Cycle starts at node 3 (index 2), cycle length λ=3 (3→4→5→3)
  Pre-cycle length μ=2
  Meeting point = μ mod λ steps into cycle from start
  = 2 mod 3 = 2 steps from node 3 = node 5... let's recount:
  Fast gains 1 step per iteration. After μ=2 iterations both are in cycle.
  Slow entered at index 0 of cycle, fast at index 2μ mod λ = 4 mod 3 = 1.
  Gap = λ - 1 = 2. Fast closes gap at 1 step/iteration → meets after 2 more steps.
  Total iterations = 4 → slow at index 2 of cycle = node 5... 
  (Exact meeting point varies by cycle structure; what matters: they ALWAYS meet)
```

### 1.2 Finding the cycle entry point — Phase 2

```java
// After detecting the meeting point, find where the cycle BEGINS.
// Time: O(n)  Space: O(1)
//
// Floyd's proof:
//   Let μ = steps from head to cycle entry
//   Let λ = cycle length
//   Let k = steps from cycle entry to meeting point
//
//   At meeting: slow traveled (μ + k), fast traveled (μ + k + nλ) for some n≥1
//   Fast travels 2× slow: 2(μ + k) = μ + k + nλ
//   Simplify: μ + k = nλ → μ = nλ - k
//
//   Reset one pointer to head. Advance both one step at a time.
//   After μ steps:
//     Reset pointer: at position μ from head = cycle entry
//     Meeting pointer: at position k + μ = k + nλ - k = nλ from meeting
//                      = exactly at cycle entry (nλ steps in cycle = full rotations)
//   → Both pointers arrive at cycle entry simultaneously ✓

public ListNode detectCycleEntry(ListNode head) {
    ListNode slow = head, fast = head;

    // Phase 1: detect meeting point
    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
        if (slow == fast) break;
    }

    // No cycle
    if (fast == null || fast.next == null) return null;

    // Phase 2: find entry point
    ListNode entry = head; // reset one pointer to head
    while (entry != slow) {
        entry = entry.next;
        slow  = slow.next;  // both advance one step at a time
    }
    return entry; // they meet exactly at the cycle entry node
}
```

**Trace phase 2 on `1→2→3→4→5→3`, meeting at node 4:**

```
Phase 2: entry=1(head), slow=4(meeting point)
  Step 1: entry=2, slow=5
  Step 2: entry=3, slow=3  ← MEET at node 3 = cycle entry ✓

Verify: μ=2 (1→2→3, so 2 steps to entry 3)
        Meeting at node 4 = 1 step into cycle from entry 3
        k=1, λ=3 (3→4→5→3)
        μ = nλ - k = 1×3 - 1 = 2 ✓
```

### 1.3 Cycle length

```java
// Once meeting point is found, measure cycle length by keeping one pointer
// fixed and counting steps until they meet again.
public int cycleLength(ListNode head) {
    ListNode slow = head, fast = head;

    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
        if (slow == fast) {
            // Count cycle length from meeting point
            int length = 1;
            ListNode current = slow.next;
            while (current != slow) {
                current = current.next;
                length++;
            }
            return length;
        }
    }
    return 0; // no cycle
}
```

## 2. Middle of Linked List

### 2.1 Finding the middle — LeetCode 876

```java
// When fast reaches the end, slow is at the middle.
// For even-length lists: returns the SECOND middle node (standard).
// Time: O(n)  Space: O(1)
public ListNode middleNode(ListNode head) {
    ListNode slow = head, fast = head;

    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
    }
    return slow;
}

// Why slow is at middle when fast exits:
//   After k iterations: slow at position k, fast at position 2k
//   Loop exits when fast is at last node (odd) or past last (even)
//   Odd length n: fast at n-1 (0-indexed), slow at (n-1)/2 = middle ✓
//   Even length n: fast at n (null), slow at n/2 = second middle ✓

// Trace — odd length [1,2,3,4,5]:
//   Step 0: slow=1, fast=1
//   Step 1: slow=2, fast=3
//   Step 2: slow=3, fast=5  → fast.next=null → exit
//   Return slow=3 ✓ (middle of 5 elements)

// Trace — even length [1,2,3,4]:
//   Step 0: slow=1, fast=1
//   Step 1: slow=2, fast=3
//   Step 2: slow=3, fast=null → exit (fast=5 doesn't exist)
//   Return slow=3 (second middle) ✓

// To get FIRST middle of even-length list: stop when fast.next.next == null
public ListNode firstMiddle(ListNode head) {
    ListNode slow = head, fast = head;
    while (fast.next != null && fast.next.next != null) {
        slow = slow.next;
        fast = fast.next.next;
    }
    return slow;
    // [1,2,3,4]: returns 2 (first middle) instead of 3 (second middle)
}
```

### 2.2 Middle as building block

The middle-finding algorithm is rarely the final answer — it is a sub-step in more complex algorithms. The two most important uses:

```java
// 1. Merge Sort on linked list — split at middle, recurse, merge
public ListNode sortList(ListNode head) {
    if (head == null || head.next == null) return head;

    // Find middle and split
    ListNode mid  = findMiddleAndSplit(head); // returns second half's head
    ListNode left = sortList(head);
    ListNode right = sortList(mid);
    return mergeSorted(left, right);
}

private ListNode findMiddleAndSplit(ListNode head) {
    ListNode slow = head, fast = head, prev = null;

    while (fast != null && fast.next != null) {
        prev = slow;
        slow = slow.next;
        fast = fast.next.next;
    }
    if (prev != null) prev.next = null; // sever the list at midpoint
    return slow; // head of second half
}

// 2. Check palindrome — find middle, reverse second half, compare (see section 3)
```

## 3. Palindrome Linked List

### 3.1 Full algorithm — LeetCode 234

```java
// Check if linked list is a palindrome.
// Strategy: find middle → reverse second half → compare both halves → restore
// Time: O(n)  Space: O(1)
//
// Why O(1) space: we cannot use extra array/stack (that would be O(n)).
// Reversing in-place avoids any extra allocation.
public boolean isPalindrome(ListNode head) {
    if (head == null || head.next == null) return true;

    // Step 1: Find middle of list
    ListNode slow = head, fast = head;
    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
    }
    // slow is now at: middle (odd length) or second middle (even length)

    // Step 2: Reverse second half starting from slow
    ListNode secondHalf = reverse(slow);
    ListNode secondCopy = secondHalf; // save for restoration

    // Step 3: Compare first half and reversed second half
    ListNode first = head;
    boolean isPalin = true;
    while (secondHalf != null) {
        if (first.val != secondHalf.val) {
            isPalin = false;
            break;
        }
        first      = first.next;
        secondHalf = secondHalf.next;
    }

    // Step 4: Restore original list (good practice; required if list must not be mutated)
    reverse(secondCopy);

    return isPalin;
}

private ListNode reverse(ListNode head) {
    ListNode prev = null, curr = head;
    while (curr != null) {
        ListNode next = curr.next;
        curr.next = prev;
        prev = curr;
        curr = next;
    }
    return prev; // new head (old tail)
}
```

**Trace on `[1, 2, 2, 1]`:**

```
Original:  1 → 2 → 2 → 1 → null

Step 1 — find middle:
  slow=1, fast=1
  slow=2, fast=2→  (fast.next.next skips to 1... wait let's index)

  Nodes: [0:1] [1:2] [2:2] [3:1]
  Iteration 1: slow=node[1]=2, fast=node[2]=2
  Iteration 2: fast=node[2].next.next=null → exit
  slow = node[2] = value 2 ✓ (second middle of even list)

Step 2 — reverse second half [2,1]:
  Before: 2 → 1 → null
  After:  1 → 2 → null
  secondHalf head = node with value 1

Step 3 — compare:
  first=1, secondHalf=1 → match
  first=2, secondHalf=2 → match
  secondHalf=null → exit loop
  isPalin = true ✓

Step 4 — restore:
  Reverse [1,2] back to [2,1]; original list restored
```

**Trace on `[1, 2, 3, 2, 1]` (odd length):**

```
Nodes: [0:1] [1:2] [2:3] [3:2] [4:1]

Step 1 — find middle:
  Iteration 1: slow=node[1]=2, fast=node[2]=3
  Iteration 2: slow=node[2]=3, fast=node[4]=1
  fast.next=null → exit. slow=node[2]=3 (true middle) ✓

Step 2 — reverse [3,2,1]:
  After: 1 → 2 → 3 → null

Step 3 — compare:
  first half:    1 → 2 → 3 (stops when secondHalf exhausted)
  second half:   1 → 2 → 3
  All match → isPalin=true ✓

Note: first half has 3 nodes but second half also has 3 (including middle).
      Comparison terminates when secondHalf == null — correct because
      middle node matches itself, and any extra nodes in first half are ignored.
```

### 3.2 Alternative: stack-based (O(n) space, simpler code)

```java
// Useful if modifying the list is not allowed and space is not constrained
public boolean isPalindromeStack(ListNode head) {
    Deque<Integer> stack = new ArrayDeque<>();
    ListNode slow = head, fast = head;

    // Push first half onto stack using fast/slow to find midpoint
    while (fast != null && fast.next != null) {
        stack.push(slow.val);
        slow = slow.next;
        fast = fast.next.next;
    }

    // Skip middle element for odd-length lists
    if (fast != null) slow = slow.next;

    // Compare second half with stack (reversed first half)
    while (slow != null) {
        if (slow.val != stack.pop()) return false;
        slow = slow.next;
    }
    return true;
}
// No list mutation — at cost of O(n/2) = O(n) stack space
```

## 4. Happy Number — Floyd on a Value Sequence

```java
// A happy number eventually reaches 1 when summing digit squares repeatedly.
// Unhappy numbers enter an infinite cycle that never reaches 1.
// Apply Floyd's cycle detection on the sequence f(n), f(f(n)), ...
// Time: O(log n) per step × O(cycle length) steps ≈ O(log n)  Space: O(1)
public boolean isHappy(int n) {
    int slow = n;
    int fast = digitSquareSum(n);

    while (fast != 1 && slow != fast) {
        slow = digitSquareSum(slow);           // one step
        fast = digitSquareSum(digitSquareSum(fast)); // two steps
    }
    return fast == 1; // true if terminated at 1, false if cycle detected
}

private int digitSquareSum(int n) {
    int sum = 0;
    while (n > 0) {
        int d = n % 10;
        sum += d * d;
        n /= 10;
    }
    return sum;
}

// Trace for n=19 (happy):
//   Sequence: 19 → 82 → 68 → 100 → 1 ✓
//   slow: 19 → 82 → 68 → 100 → 1
//   fast: 82 → 100 → 1
//   fast==1 before slow==fast → return true ✓

// Trace for n=2 (unhappy):
//   Sequence: 2 → 4 → 16 → 37 → 58 → 89 → 145 → 42 → 20 → 4 (cycle!)
//   slow and fast will eventually meet inside the cycle → return false ✓
```

## 5. Find Duplicate Number — LeetCode 287

```java
// Array of n+1 integers, each in [1,n]. Exactly one duplicate. Find it.
// Constraint: cannot modify array, O(1) extra space.
//
// Key insight: treat array as linked list — index i points to nums[i].
// Duplicate value = two indices pointing to same "next node" = cycle entry.
// Array has n+1 elements with values in [1,n] → pigeonhole guarantees cycle.
//
// Apply Floyd's to find cycle entry = duplicate number.
// Time: O(n)  Space: O(1)
public int findDuplicate(int[] nums) {
    // Phase 1: find meeting point inside cycle
    int slow = nums[0];
    int fast = nums[0];

    do {
        slow = nums[slow];             // follow one link
        fast = nums[nums[fast]];       // follow two links
    } while (slow != fast);

    // Phase 2: find cycle entry (= duplicate number)
    int entry = nums[0];
    while (entry != slow) {
        entry = nums[entry];
        slow  = nums[slow];
    }
    return entry;
}

// Why this works:
//   nums = [1, 3, 4, 2, 2]  (n=4, duplicate=2)
//   Implicit linked list:
//     index: 0 → 1 → 3 → 2 → 4 → 2 (cycle!)
//     value: 1   3   2   4   2
//   0→nums[0]=1→nums[1]=3→nums[3]=2→nums[2]=4→nums[4]=2→nums[2]=4...
//   Cycle: 2→4→2 (length 2), entry at 2
//
// Trace:
//   Phase 1: slow=1,fast=1 (start at nums[0])
//     slow=nums[1]=3, fast=nums[nums[1]]=nums[3]=2
//     slow=nums[3]=2, fast=nums[nums[2]]=nums[4]=2
//     slow==fast==2 → meeting point found
//   Phase 2: entry=nums[0]=1, slow=2
//     entry=nums[1]=3, slow=nums[2]=4
//     entry=nums[3]=2, slow=nums[4]=2
//     entry==slow==2 → duplicate = 2 ✓
```

## 6. Reorder List — LeetCode 143

```java
// Reorder: L0→L1→...→Ln-1→Ln  to  L0→Ln→L1→Ln-1→L2→Ln-2→...
// Three steps: find middle → reverse second half → merge alternating
// Time: O(n)  Space: O(1)
public void reorderList(ListNode head) {
    if (head == null || head.next == null) return;

    // Step 1: Find middle (split into two halves)
    ListNode slow = head, fast = head, prev = null;
    while (fast != null && fast.next != null) {
        prev = slow;
        slow = slow.next;
        fast = fast.next.next;
    }
    prev.next = null; // sever: first half ends at prev

    // Step 2: Reverse second half
    ListNode second = reverse(slow);

    // Step 3: Merge two halves alternately
    ListNode first = head;
    while (second != null) {
        ListNode tmp1 = first.next;
        ListNode tmp2 = second.next;
        first.next  = second;
        second.next = tmp1;
        first  = tmp1;
        second = tmp2;
    }
}

// Trace [1,2,3,4,5]:
//   Split:   [1,2] and [3,4,5]
//   Reverse: [1,2] and [5,4,3]
//   Merge:   1→5→2→4→3 ✓
//   (second runs out before first; last connection 2→3 already set by tmp assignment)
```

## 7. Rotate List — LeetCode 61

```java
// Rotate list to the right by k places.
// Step 1: find length and connect tail to head (make circular)
// Step 2: find new tail at position (n - k % n - 1); new head follows it
// Time: O(n)  Space: O(1)
public ListNode rotateRight(ListNode head, int k) {
    if (head == null || head.next == null || k == 0) return head;

    // Find length and tail
    int length = 1;
    ListNode tail = head;
    while (tail.next != null) {
        tail = tail.next;
        length++;
    }

    k = k % length; // rotations > length are redundant
    if (k == 0) return head;

    // Connect tail to head (circular)
    tail.next = head;

    // Find new tail: (length - k - 1) steps from head
    int stepsToNewTail = length - k - 1;
    ListNode newTail = head;
    for (int i = 0; i < stepsToNewTail; i++) newTail = newTail.next;

    ListNode newHead = newTail.next;
    newTail.next = null; // break circle
    return newHead;
}
// [1,2,3,4,5], k=2: length=5, k=2, stepsToNewTail=2
//   newTail = node[2]=3, newHead=node[3]=4
//   Result: [4,5,1,2,3] ✓
```

## 8. Linked List Intersection — LeetCode 160

```java
// Find the node where two linked lists intersect (same node reference, not just value).
// Two-pointer approach: advance both; on reaching null, redirect to other list's head.
// Both pointers traverse same total distance → meet at intersection or null.
// Time: O(m+n)  Space: O(1)
public ListNode getIntersectionNode(ListNode headA, ListNode headB) {
    ListNode a = headA, b = headB;

    while (a != b) {
        a = (a == null) ? headB : a.next; // redirect to other list on null
        b = (b == null) ? headA : b.next;
    }
    return a; // intersection node, or null if no intersection

    // Why it works:
    //   a travels: lenA + (lenB - intersection) = lenA + lenB - common
    //   b travels: lenB + (lenA - intersection) = lenA + lenB - common
    //   Same total distance → meet at intersection node
    //   If no intersection: both reach null simultaneously after lenA+lenB steps
}
```

## 9. Remove Nth Node From End — LeetCode 19

```java
// Remove the nth node from the end in one pass.
// Advance fast n steps ahead; then advance both until fast reaches null.
// When fast exits, slow.next is the node to remove.
// Time: O(n)  Space: O(1)
public ListNode removeNthFromEnd(ListNode head, int n) {
    ListNode dummy = new ListNode(0);
    dummy.next = head;

    ListNode slow = dummy, fast = dummy;

    // Advance fast n+1 steps (so slow stops one before the target)
    for (int i = 0; i <= n; i++) fast = fast.next;

    // Advance both until fast exits
    while (fast != null) {
        slow = slow.next;
        fast = fast.next;
    }

    slow.next = slow.next.next; // remove nth from end
    return dummy.next;
}

// Trace [1,2,3,4,5], n=2:
//   dummy→1→2→3→4→5
//   fast advances 3 steps (n+1=3): fast=node[3]=4
//   slow=dummy, fast=4:
//     slow=1, fast=5
//     slow=2, fast=null → exit
//   slow.next=3 is 2nd from end → slow.next=slow.next.next=4
//   Result: [1,2,4,5] ✓
```

## 10. Summary: Fast and Slow Pointer Patterns

```
Problem                       Fast step   Slow step   Key insight
────────────────────────────────────────────────────────────────────────────────
Cycle detection               ×2          ×1          If cycle: must meet
                                                       If no cycle: fast exits

Cycle entry point             ×2 → ×1     ×1 → ×1     Reset one to head; advance
  (Floyd phase 2)                                      together → meet at entry

Cycle length                  Fixed       Count        Keep slow fixed; count
                                                       steps until return

List middle                   ×2          ×1          Fast exits → slow at mid
                                                       Odd: exact middle
                                                       Even: second middle

Palindrome list               ×2          ×1 + reverse Find mid → reverse second
                                                       half → compare → restore

Find duplicate (as cycle)     ×2          ×1          Array as implicit list;
  (LeetCode 287)                                       duplicate = cycle entry

Happy number                  ×2 (seq)    ×1 (seq)    Unhappy → cycle; happy → 1

Remove nth from end           lead by n   ×1          n-step lead → slow at pivot

Intersection of two lists     redirect    redirect    Equal total distance
                              on null                 → meet at intersection

Reorder list                  ×2 (find mid) ×1        Find mid + reverse + merge

Middle as building block      ×2          ×1          Sub-step for merge sort,
                                                       reorder, palindrome check
```

## 11. Interview Questions

**Q: Why is Floyd's cycle detection algorithm preferred over using a HashSet to track visited nodes?**

A: Both detect cycles correctly and run in O(n) time, but Floyd's uses O(1) space while the HashSet approach uses O(n) space. For a linked list with millions of nodes, O(n) space is significant. Floyd's achieves O(1) by exploiting the mathematical property that two pointers moving at different speeds in a cyclic sequence must eventually coincide — no bookkeeping required. The HashSet approach is simpler to code and is perfectly valid when space is not a concern, but in interviews the O(1) space solution is almost always the expected answer. Floyd's also extends naturally to the entry-point problem via phase 2, which doesn't have an obvious analogue in the HashSet approach without additional tracking.

---

**Q: Walk through Floyd's phase 2 proof — why do the two pointers meet at the cycle entry?**

A: Define μ as the number of nodes before the cycle starts, λ as the cycle length, and k as the number of steps from the cycle entry to the meeting point. At the meeting point, slow has traveled μ + k steps, and fast has traveled twice as far: 2(μ + k). Fast's extra distance must be a whole number of cycle lengths: 2(μ + k) − (μ + k) = μ + k = nλ for some integer n. This gives us μ = nλ − k. Now reset one pointer to the head and keep the other at the meeting point. After μ more steps, the reset pointer is at the cycle entry. The meeting-point pointer has traveled an additional μ = nλ − k steps from its current position k steps into the cycle, landing at position k + nλ − k = nλ within the cycle. Since nλ is a whole number of complete loops, it is exactly at the cycle entry. The two pointers arrive at the cycle entry simultaneously after exactly μ more steps.

---

**Q: How do you find the middle of a linked list, and what happens differently for odd vs even lengths?**

A: Advance slow by one step and fast by two steps simultaneously. When fast cannot advance two steps — either fast is null (even length) or fast.next is null (odd length) — slow is at the middle. For odd length n, fast lands on the final node (not null) at exactly the moment slow reaches position (n−1)/2, which is the true unique middle. For even length n, fast falls off the end (becomes null) when slow reaches position n/2, which is the second of the two middle nodes. Depending on the problem, you may want the first middle instead — change the loop condition to `fast.next != null && fast.next.next != null`, which stops one step earlier and leaves slow at the first middle node. For palindrome checking, the second middle (standard condition) is what you want because the reversed second half will include the middle element, and comparing it with itself always matches.

---

**Q: Explain how LeetCode 287 (Find Duplicate Number) reduces to a cycle detection problem.**

A: Interpret the array as an implicit linked list where each index i points to the node at position nums[i]. Starting from index 0, following the "links" creates a path: 0 → nums[0] → nums[nums[0]] → ... Since there are n+1 values all in the range [1,n], by the pigeonhole principle at least one value is repeated. That repeated value means two different indices point to the same next position — exactly the condition that creates a cycle in the implicit list. The duplicate value is the entry point of the cycle, because it is the node that two different predecessors both point to. Applying Floyd's two-phase algorithm on this implicit list (using array indexing instead of pointer following) finds the cycle entry, which is the duplicate number. This runs in O(n) time and O(1) space — far better than sorting (O(n log n)) or a HashSet (O(n) space). The constraint that the array cannot be modified rules out marking visited cells.

---

**Q: How does the palindrome linked list algorithm work, and why is restoring the list important?**

A: The algorithm has four steps: find the middle with fast/slow pointers, reverse the second half in-place, compare the first and reversed-second half node by node, then restore the list by reversing the second half again. The fast/slow middle-finding ensures O(1) space by avoiding copying values into an array or stack. The in-place reversal also uses O(1) space. Comparison terminates when the second-half pointer becomes null — for odd-length lists, the middle node is in the second half and matches itself, so it doesn't affect the result. Restoring the list matters for correctness when the calling code continues to use the list after the palindrome check — an unreversed second half leaves the list in an inconsistent state. In an interview, explicitly mentioning restoration demonstrates awareness of side effects and is often what separates a good answer from an excellent one. If the problem guarantees the list will not be used afterward, restoration can be skipped, but it is better practice to always restore.

---

## 12. Key Takeaways

- **Fast (×2) + Slow (×1)**: if a cycle exists they must meet; if no cycle, fast exits first — every fast/slow problem reduces to one of these two outcomes
- **Floyd's phase 2**: after meeting, reset one pointer to head and advance both at ×1 — they converge at the cycle entry after exactly μ more steps; the proof uses μ = nλ − k
- **Slow at middle when fast exits**: odd length → exact middle; even length → second middle (standard); use `fast.next.next != null` guard for first middle
- **Palindrome linked list** = find middle + reverse second half + compare + restore; the restore step prevents side effects and should always be mentioned in interviews
- **Find duplicate (LeetCode 287)** = array as implicit linked list; duplicate value is cycle entry; same Floyd's algorithm, same O(n)/O(1) complexity
- **Happy number** = Floyd's on a numeric digit-square sequence; cycle → unhappy; reaches 1 → happy
- **Remove nth from end**: advance fast n+1 steps first; then advance both; slow stops one node before the target — use dummy head to handle edge case of removing the first node
- **List intersection**: redirect each pointer to the other list's head on null; equal total distance guarantees meeting at intersection (or both reach null together)
- **Middle node as sub-step**: reorder list, merge sort on linked list, and palindrome check all use middle-finding as their first step — master it as a primitive
