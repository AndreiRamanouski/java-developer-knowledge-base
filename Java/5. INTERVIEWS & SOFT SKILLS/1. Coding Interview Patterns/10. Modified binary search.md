# Modified Binary Search

## Overview

Binary search finds a target in a sorted array in O(log n) by eliminating half the search space each step. Modified binary search extends this to arrays that are not perfectly sorted but retain enough structural regularity — rotated arrays, bitonic arrays, infinite arrays, answer-space searches — to still eliminate half the candidates per step.

**The universal binary search invariant:**

```
At every step, the target (or answer) is guaranteed to lie within [lo, hi].
The mid comparison must eliminate at least one half with certainty.
Never eliminate a half that might contain the answer.

Template:
  lo = 0, hi = n - 1 (or the answer range)
  while lo <= hi:
    mid = lo + (hi - lo) / 2    ← avoids integer overflow vs (lo+hi)/2
    if found: return mid
    if target in right half: lo = mid + 1
    if target in left half:  hi = mid - 1
  return -1 (or lo, depending on problem)

The critical design question for every variant:
  "Given the value at mid, which half is guaranteed NOT to contain the answer?"
  Eliminate that half. Keep the other.
```

**Why `mid = lo + (hi - lo) / 2` not `(lo + hi) / 2`:**

```
lo=2_000_000_000, hi=2_000_000_001:
  (lo + hi) / 2 = 4_000_000_001 / 2 → INTEGER OVERFLOW in Java (int range ~2.1B)
  lo + (hi - lo) / 2 = 2_000_000_000 + 0 = 2_000_000_000 ✓

Always use the safe form. No exceptions.
```

## 1. Classic Binary Search — LeetCode 704

```java
// Standard sorted array: O(log n) time, O(1) space.
public int search(int[] nums, int target) {
    int lo = 0, hi = nums.length - 1;

    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;

        if      (nums[mid] == target) return mid;
        else if (nums[mid] < target)  lo = mid + 1; // target in right half
        else                          hi = mid - 1; // target in left half
    }
    return -1; // not found
}

// Variants: lo <= hi (exact match) vs lo < hi (find boundary)
// lo <= hi:  terminates with lo > hi; returns -1 if not found
// lo < hi:   terminates with lo == hi; lo (or hi) is the answer boundary
```

## 2. Search in Rotated Sorted Array

### 2.1 Core insight — one half is always sorted

```
A rotated array like [4,5,6,7,0,1,2] was once [0,1,2,4,5,6,7].
After rotation, it has two sorted sub-segments joined at the rotation point.

Key observation: for any mid, AT LEAST ONE of the two halves [lo,mid] or [mid,hi]
is guaranteed to be fully sorted. Why?
  If nums[lo] <= nums[mid]: left half [lo,mid] is sorted (no rotation cut within it).
  Otherwise:                right half [mid,hi] is sorted.

Once we identify the sorted half, we check if the target falls within it:
  If yes → search that half (standard binary search in sorted range).
  If no  → search the other half (must contain the target).

This eliminates one half per step → O(log n) total.
```

### 2.2 No duplicates — LeetCode 33

```java
// Time: O(log n)  Space: O(1)
public int search(int[] nums, int target) {
    int lo = 0, hi = nums.length - 1;

    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;

        if (nums[mid] == target) return mid;

        if (nums[lo] <= nums[mid]) {
            // Left half [lo, mid] is sorted
            if (nums[lo] <= target && target < nums[mid]) {
                hi = mid - 1; // target in sorted left half
            } else {
                lo = mid + 1; // target in right half
            }
        } else {
            // Right half [mid, hi] is sorted
            if (nums[mid] < target && target <= nums[hi]) {
                lo = mid + 1; // target in sorted right half
            } else {
                hi = mid - 1; // target in left half
            }
        }
    }
    return -1;
}
```

**Trace on `[4,5,6,7,0,1,2]`, target=0:**

```
lo=0(4), hi=6(2), mid=3(7)
  nums[lo=4] <= nums[mid=7] → left [4,5,6,7] is sorted
  Is 0 in [4,7]? 4 <= 0? NO → go right: lo=4

lo=4(0), hi=6(2), mid=5(1)
  nums[lo=0] <= nums[mid=1] → left [0,1] is sorted
  Is 0 in [0,1)? 0 <= 0 < 1? YES → go left: hi=4

lo=4(0), hi=4(0), mid=4(0)
  nums[mid]=0 == target=0 → return 4 ✓
```

**Trace on `[4,5,6,7,0,1,2]`, target=3 (not present):**

```
lo=0(4), hi=6(2), mid=3(7)
  Left sorted. Is 3 in [4,7)? NO → lo=4

lo=4(0), hi=6(2), mid=5(1)
  Left [0,1] sorted. Is 3 in [0,1)? NO → lo=6

lo=6(2), hi=6(2), mid=6(2)
  nums[mid]=2 ≠ 3
  nums[lo=2] <= nums[mid=2] → left "sorted" (single element).
  Is 3 in [2,2)? NO → lo=7

lo=7 > hi=6 → return -1 ✓
```

### 2.3 With duplicates — LeetCode 81

```java
// Duplicates prevent determining which half is sorted when nums[lo]==nums[mid].
// Worst case: O(n) (e.g. [1,1,1,1,1] searching for 0).
// Time: O(log n) average, O(n) worst  Space: O(1)
public boolean searchWithDuplicates(int[] nums, int target) {
    int lo = 0, hi = nums.length - 1;

    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;
        if (nums[mid] == target) return true;

        // Can't determine sorted half when lo, mid, hi all equal
        if (nums[lo] == nums[mid] && nums[mid] == nums[hi]) {
            lo++; hi--; // shrink both boundaries by 1 — O(1) elimination
        } else if (nums[lo] <= nums[mid]) {
            // Left half sorted
            if (nums[lo] <= target && target < nums[mid]) hi = mid - 1;
            else lo = mid + 1;
        } else {
            // Right half sorted
            if (nums[mid] < target && target <= nums[hi]) lo = mid + 1;
            else hi = mid - 1;
        }
    }
    return false;
}
// [1,1,1,0,1], target=0:
//   lo=0(1), hi=4(1), mid=2(1): nums[lo]=nums[mid]=nums[hi]=1 → lo=1, hi=3
//   lo=1(1), hi=3(0), mid=2(1): nums[lo=1]<=nums[mid=1] → left sorted
//     1 <= 0 < 1? NO → lo=3
//   lo=3(0), hi=3(0), mid=3(0): nums[mid]=0==target → return true ✓
```

### 2.4 Find rotation index (minimum element) — LeetCode 153

```java
// Minimum element = the point where the rotation occurred.
// In a rotated sorted array (no duplicates), min is the only element
// that is less than its predecessor.
// Time: O(log n)  Space: O(1)
public int findMin(int[] nums) {
    int lo = 0, hi = nums.length - 1;

    while (lo < hi) { // lo < hi: terminate when lo == hi (single element = min)
        int mid = lo + (hi - lo) / 2;

        if (nums[mid] > nums[hi]) {
            // Mid is in the left (higher-value) segment → min is to the right
            lo = mid + 1;
        } else {
            // Mid is in the right (lower-value) segment → min is at mid or left
            hi = mid; // don't exclude mid — it might be the minimum
        }
    }
    return nums[lo]; // lo == hi == index of minimum
}
```

**Trace on `[4,5,6,7,0,1,2]`:**

```
lo=0(4), hi=6(2), mid=3(7)
  7 > 2 (nums[hi]) → min is right of mid → lo=4

lo=4(0), hi=6(2), mid=5(1)
  1 > 2? NO → min at mid or left → hi=5

lo=4(0), hi=5(1), mid=4(0)
  0 > 1? NO → hi=4

lo=4, hi=4 → exit. nums[4]=0 ✓ (minimum is 0)
```

**Trace on `[1,2,3,4,5]` (no rotation):**

```
lo=0(1), hi=4(5), mid=2(3)
  3 > 5? NO → hi=2
lo=0(1), hi=2(3), mid=1(2)
  2 > 3? NO → hi=1
lo=0(1), hi=1(2), mid=0(1)
  1 > 2? NO → hi=0
lo=0, hi=0 → nums[0]=1 ✓
```

## 3. Search in Infinite Array

### 3.1 Exponential search — find the range first

```java
// ArrayReader interface: get(i) returns value at index i (Integer.MAX_VALUE if out of bounds)
// Strategy: double the search window until target is in range, then binary search.
// Time: O(log n) where n = index of target  Space: O(1)
public int search(ArrayReader reader, int target) {
    // Step 1: Find bounds [lo, hi] that contain target
    int lo = 0, hi = 1;

    while (reader.get(hi) < target) {
        lo = hi;
        hi *= 2; // exponential growth: 1, 2, 4, 8, 16...
    }
    // target is in [lo, hi]

    // Step 2: Standard binary search within [lo, hi]
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;
        int val = reader.get(mid);

        if      (val == target)                 return mid;
        else if (val == Integer.MAX_VALUE || val > target) hi = mid - 1;
        else                                    lo = mid + 1;
    }
    return -1;
}
```

**Trace on infinite sorted array starting [1,2,4,6,8,10,...], target=10:**

```
Bound-finding phase:
  lo=0, hi=1: reader.get(1)=2 < 10 → lo=1, hi=2
  lo=1, hi=2: reader.get(2)=4 < 10 → lo=2, hi=4
  lo=2, hi=4: reader.get(4)=8 < 10 → lo=4, hi=8
  lo=4, hi=8: reader.get(8) >= 10 → stop. target in [4,8]

Binary search [4,8]:
  mid=6: reader.get(6)=10 == target → return 6 ✓

Total steps: 4 (doubling) + 3 (binary search) = 7 = O(log 6) = O(log n)
```

**Why exponential growth gives O(log n):**

```
If target is at index n, we double until hi >= n → takes ceil(log₂ n) doublings.
Each doubling is O(1). After doubling, hi <= 2n, so binary search range <= 2n.
Binary search on range 2n → O(log n) steps.
Total: O(log n) + O(log n) = O(log n).
```

## 4. Bitonic Array Search

### 4.1 Find peak in bitonic array — LeetCode 162

```java
// Bitonic (mountain) array: strictly increases then strictly decreases.
// [1,3,8,12,4,2]: peak is 12 at index 3.
// LeetCode 162 (Find Peak Element): find ANY peak where nums[i] > neighbours.
// Time: O(log n)  Space: O(1)
//
// Key: if nums[mid] < nums[mid+1], peak is to the RIGHT (we're on upslope).
//      if nums[mid] > nums[mid+1], peak is at mid or LEFT (we're on downslope or AT peak).
public int findPeakElement(int[] nums) {
    int lo = 0, hi = nums.length - 1;

    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;

        if (nums[mid] < nums[mid + 1]) {
            lo = mid + 1; // ascending slope → peak is to the right
        } else {
            hi = mid;     // descending slope or peak → peak at mid or left
        }
    }
    return lo; // lo == hi == peak index
}
```

**Trace on `[1,3,8,12,4,2]`:**

```
lo=0(1), hi=5(2), mid=2(8)
  nums[2]=8 < nums[3]=12 → ascending → lo=3

lo=3(12), hi=5(2), mid=4(4)
  nums[4]=4 > nums[5]=2 → descending → hi=4

lo=3(12), hi=4(4), mid=3(12)
  nums[3]=12 > nums[4]=4 → descending → hi=3

lo=3, hi=3 → return 3 ✓ (peak at index 3, value 12)
```

**Why this works:**

```
If nums[mid] < nums[mid+1]:
  The sequence is increasing at mid. Peak must be at mid+1 or beyond.
  Eliminating everything ≤ mid is safe — peak cannot be there.

If nums[mid] > nums[mid+1]:
  The sequence is decreasing at mid. Either mid is the peak,
  or there's a higher peak to the left. Cannot eliminate mid.
  → hi = mid (include mid in remaining search space).

Boundary: nums[-1] and nums[n] are defined as -∞,
  so any local maximum qualifies as a "peak."
```

### 4.2 Search in bitonic array — two binary searches

```java
// Search for target in a bitonic array.
// Strategy: find peak, binary search ascending half, binary search descending half.
// Time: O(log n)  Space: O(1)
public int searchBitonic(int[] arr, int target) {
    // Step 1: find peak index
    int peak = findPeak(arr);

    // Step 2: binary search in ascending half [0, peak]
    int idx = binarySearch(arr, target, 0, peak, true);
    if (idx != -1) return idx;

    // Step 3: binary search in descending half [peak+1, n-1]
    return binarySearch(arr, target, peak + 1, arr.length - 1, false);
}

private int findPeak(int[] arr) {
    int lo = 0, hi = arr.length - 1;
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        if (arr[mid] < arr[mid + 1]) lo = mid + 1;
        else                         hi = mid;
    }
    return lo;
}

private int binarySearch(int[] arr, int target, int lo, int hi, boolean ascending) {
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;
        if (arr[mid] == target) return mid;

        if (ascending) {
            if (arr[mid] < target) lo = mid + 1;
            else                   hi = mid - 1;
        } else {
            if (arr[mid] > target) lo = mid + 1; // descending: larger on left
            else                   hi = mid - 1;
        }
    }
    return -1;
}
// [1,3,8,12,4,2], target=4:
//   peak=3 (value 12)
//   Binary search ascending [0,3] for 4: not found
//   Binary search descending [4,5] for 4:
//     mid=4(4) == target → return 4 ✓
```

## 5. Binary Search on Answer Space

### 5.1 The paradigm — search the answer, not the array

```
Many optimisation problems have a monotonic structure:
  "Is answer X feasible?" → if X is feasible, all larger X are also feasible (or smaller).
  Binary search on X over the range of possible answers.

Recognition pattern:
  "Minimise the maximum..."
  "Maximise the minimum..."
  "Find the smallest X such that condition(X) is true"
  "Find the largest X such that condition(X) is true"

Template:
  lo = minimum_possible_answer
  hi = maximum_possible_answer
  while lo < hi:
    mid = lo + (hi - lo) / 2
    if canAchieve(mid):
      hi = mid      (or lo = mid + 1 if searching for first false)
    else:
      lo = mid + 1  (or hi = mid - 1)
  return lo
```

### 5.2 Koko eating bananas — LeetCode 875

```java
// Koko eats at speed k bananas/hour. Find minimum k to eat all piles within h hours.
// Monotone: if speed k works, speed k+1 also works.
// Answer space: [1, max(piles)]
// Time: O(n log(max))  Space: O(1)
public int minEatingSpeed(int[] piles, int h) {
    int lo = 1, hi = Arrays.stream(piles).max().getAsInt();

    while (lo < hi) {
        int mid = lo + (hi - lo) / 2; // candidate speed
        if (canFinish(piles, mid, h)) {
            hi = mid;     // mid works → try smaller speed
        } else {
            lo = mid + 1; // mid too slow → need faster
        }
    }
    return lo; // smallest speed that works
}

private boolean canFinish(int[] piles, int speed, int h) {
    int hours = 0;
    for (int pile : piles) {
        hours += (pile + speed - 1) / speed; // ceiling division
    }
    return hours <= h;
}
// piles=[3,6,7,11], h=8:
//   lo=1, hi=11
//   mid=6: canFinish? ceil(3/6)+ceil(6/6)+ceil(7/6)+ceil(11/6)=1+1+2+2=6<=8 → hi=6
//   mid=3: 1+2+3+4=10 > 8 → lo=4
//   mid=5: 1+2+2+3=8 <= 8 → hi=5
//   mid=4: 1+2+2+3=8 <= 8 → hi=4
//   lo=4==hi → return 4 ✓
```

### 5.3 Minimum days to make bouquets — LeetCode 1482

```java
// Can make m bouquets of k consecutive bloomed flowers.
// Monotone: if day d works, day d+1 also works.
// Time: O(n log(max_day))  Space: O(1)
public int minDays(int[] bloomDay, int m, int k) {
    if ((long) m * k > bloomDay.length) return -1; // impossible

    int lo = 1, hi = Arrays.stream(bloomDay).max().getAsInt();

    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        if (canMake(bloomDay, m, k, mid)) hi = mid;
        else                              lo = mid + 1;
    }
    return lo;
}

private boolean canMake(int[] bloomDay, int m, int k, int day) {
    int bouquets = 0, consecutive = 0;
    for (int bd : bloomDay) {
        if (bd <= day) {
            consecutive++;
            if (consecutive == k) { bouquets++; consecutive = 0; }
        } else {
            consecutive = 0; // reset: non-bloomed flower breaks streak
        }
    }
    return bouquets >= m;
}
```

### 5.4 Split array largest sum — LeetCode 410

```java
// Split array into m non-empty subarrays, minimise the largest subarray sum.
// Monotone: if max sum X is achievable with m splits, X+1 is also achievable.
// Binary search on X in [max(nums), sum(nums)].
// Time: O(n log(sum))  Space: O(1)
public int splitArray(int[] nums, int m) {
    int lo = 0, hi = 0;
    for (int num : nums) { lo = Math.max(lo, num); hi += num; }
    // lo = max element (minimum possible max sum — single element subarrays)
    // hi = total sum (maximum possible max sum — one subarray)

    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        if (canSplit(nums, m, mid)) hi = mid;
        else                        lo = mid + 1;
    }
    return lo;
}

private boolean canSplit(int[] nums, int m, int maxSum) {
    int parts = 1, currentSum = 0;
    for (int num : nums) {
        if (currentSum + num > maxSum) {
            parts++;           // start new subarray
            currentSum = num;
            if (parts > m) return false; // too many splits needed
        } else {
            currentSum += num;
        }
    }
    return true;
}
// [7,2,5,10,8], m=2:
//   lo=10(max), hi=32(sum)
//   mid=21: canSplit([7,2,5,10,8],2,21)? [7,2,5],[10,8]=2 parts ≤ 2 → hi=21
//   mid=15: [7,2,5],[10],[8]=3 parts > 2 → lo=16
//   mid=18: [7,2,5],[10,8]=2 ≤ 2 → hi=18
//   mid=17: [7,2,5],[10],[8]=3 > 2 → lo=18
//   lo=18==hi → return 18 ✓
```

// Find index where target should be inserted to keep array sorted.
// Equivalent to: first index where nums[i] >= target.

public int searchInsert(int[] nums, int target) {

### 6.2 First and last position of element — LeetCode 34

```java
// Find [first, last] occurrence of target in sorted array.
// Two binary searches: one for left boundary, one for right.
// Time: O(log n)  Space: O(1)
public int[] searchRange(int[] nums, int target) {
    return new int[]{findFirst(nums, target), findLast(nums, target)};
}

// Find first index where nums[i] == target (left boundary)
private int findFirst(int[] nums, int target) {
    int lo = 0, hi = nums.length - 1, result = -1;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;
        if      (nums[mid] == target) { result = mid; hi = mid - 1; } // record, go left
        else if (nums[mid] < target)  lo = mid + 1;
        else                          hi = mid - 1;
    }
    return result;
}

// Find last index where nums[i] == target (right boundary)
private int findLast(int[] nums, int target) {
    int lo = 0, hi = nums.length - 1, result = -1;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;
        if      (nums[mid] == target) { result = mid; lo = mid + 1; } // record, go right
        else if (nums[mid] < target)  lo = mid + 1;
        else                          hi = mid - 1;
    }
    return result;
}
// [5,7,7,8,8,10], target=8:
//   findFirst: lo=0,hi=5 → mid=2(7)<8→lo=3 → mid=4(8)→result=4,hi=3 → mid=3(8)→result=3,hi=2 → exit. return 3 ✓
//   findLast:  lo=0,hi=5 → mid=2(7)<8→lo=3 → mid=4(8)→result=4,lo=5 → mid=5(10)>8→hi=4 → exit. return 4 ✓
```

### 6.3 Sqrt(x) — LeetCode 69

```java
// Find floor(sqrt(x)) using binary search on answer space [0, x].
// Time: O(log x)  Space: O(1)
public int mySqrt(int x) {
    if (x < 2) return x;
    int lo = 1, hi = x / 2; // sqrt(x) <= x/2 for x >= 4

    while (lo <= hi) {
        long mid = lo + (hi - lo) / 2; // long to prevent mid*mid overflow
        long sq  = mid * mid;
        if      (sq == x) return (int) mid;
        else if (sq < x)  lo = (int) mid + 1; // try larger
        else              hi = (int) mid - 1; // try smaller
    }
    return hi; // hi = floor(sqrt(x)) when loop exits
}
// x=8: lo=1,hi=4
//   mid=2: 4<8 → lo=3
//   mid=3: 9>8 → hi=2
//   lo>hi → return hi=2 ✓ (floor(sqrt(8))=2)
```

### 6.4 Median of two sorted arrays — LeetCode 4

```java
// Find median of two sorted arrays combined. O(log(min(m,n))).
// Binary search on partition point in the smaller array.
// Time: O(log(min(m,n)))  Space: O(1)
public double findMedianSortedArrays(int[] nums1, int[] nums2) {
    // Ensure nums1 is the smaller array
    if (nums1.length > nums2.length) return findMedianSortedArrays(nums2, nums1);

    int m = nums1.length, n = nums2.length;
    int lo = 0, hi = m;
    int half = (m + n + 1) / 2; // left half of the combined sorted array

    while (lo <= hi) {
        int i = lo + (hi - lo) / 2; // partition nums1: i elements on left
        int j = half - i;            // partition nums2: j elements on left

        int maxLeft1  = (i == 0) ? Integer.MIN_VALUE : nums1[i - 1];
        int minRight1 = (i == m) ? Integer.MAX_VALUE : nums1[i];
        int maxLeft2  = (j == 0) ? Integer.MIN_VALUE : nums2[j - 1];
        int minRight2 = (j == n) ? Integer.MAX_VALUE : nums2[j];

        if (maxLeft1 <= minRight2 && maxLeft2 <= minRight1) {
            // Found correct partition
            if ((m + n) % 2 == 1) return Math.max(maxLeft1, maxLeft2);
            return (Math.max(maxLeft1, maxLeft2) + Math.min(minRight1, minRight2)) / 2.0;
        } else if (maxLeft1 > minRight2) {
            hi = i - 1; // too many elements from nums1 on left
        } else {
            lo = i + 1; // too few elements from nums1 on left
        }
    }
    return 0.0; // unreachable for valid input
}
```

### 6.5 Count of smaller numbers after self — binary search on sorted result

```java
// For each element, count elements to its right that are smaller.
// Process right to left, maintain sorted list, binary search for insertion point.
// Time: O(n log n)  Space: O(n)
public List<Integer> countSmaller(int[] nums) {
    List<Integer> sorted = new ArrayList<>();
    Integer[] counts = new Integer[nums.length];

    for (int i = nums.length - 1; i >= 0; i--) {
        // Find insertion position = count of smaller elements to the right
        int lo = 0, hi = sorted.size();
        while (lo < hi) {
            int mid = lo + (hi - lo) / 2;
            if (sorted.get(mid) < nums[i]) lo = mid + 1;
            else                           hi = mid;
        }
        counts[i] = lo; // lo = number of elements in sorted that are < nums[i]
        sorted.add(lo, nums[i]); // insert at correct position (O(n) but acceptable here)
    }
    return Arrays.asList(counts);
}
```

## 7. Binary Search Boundary Patterns

```java
// Pattern A: find FIRST position where condition is TRUE
// [F,F,F,T,T,T] → find first T
// Use: lo < hi, hi=mid when true, lo=mid+1 when false, return lo
int findFirst(int[] arr, Predicate<Integer> condition) {
    int lo = 0, hi = arr.length - 1;
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        if (condition.test(arr[mid])) hi = mid;   // true: might be first, go left
        else                          lo = mid + 1; // false: not first, go right
    }
    return condition.test(arr[lo]) ? lo : -1;
}

// Pattern B: find LAST position where condition is TRUE
// [T,T,T,F,F,F] → find last T
// Use: lo < hi, lo=mid+1 when false, hi=mid when true, but advance carefully
int findLast(int[] arr, Predicate<Integer> condition) {
    int lo = 0, hi = arr.length - 1;
    while (lo < hi) {
        int mid = lo + (hi - lo + 1) / 2; // CEILING division — prevents infinite loop
        if (condition.test(arr[mid])) lo = mid;   // true: might be last, go right
        else                          hi = mid - 1; // false: not last, go left
    }
    return condition.test(arr[lo]) ? lo : -1;
}

// WHY ceiling division for Pattern B:
//   lo=3, hi=4: floor mid = 3. If condition(arr[3]) is true: lo = 3. Infinite loop!
//   ceil  mid = 4. If condition(arr[3]) is true: lo = 4. Loop terminates. ✓
```

## 9. Interview Questions

**Q: How do you identify which half of a rotated sorted array is sorted, and why is this the key insight?**

A: Compare `nums[lo]` with `nums[mid]`. If `nums[lo] <= nums[mid]`, the left half `[lo, mid]` is sorted — no rotation cut can fall within it, because if it did, `nums[lo]` would be larger than `nums[mid]`. Otherwise, `nums[lo] > nums[mid]`, meaning the rotation point falls somewhere in the left half, which guarantees the right half `[mid, hi]` is sorted. Once we know which half is sorted, we check whether the target falls within the sorted range using two comparisons. If it does, we binary search that half; if not, the target must be in the other half. This eliminates one half per step, preserving the O(log n) guarantee. The invariant "at least one half is always sorted" is the structural property that makes binary search possible despite the rotation.

---

**Q: Why does binary search on the answer space work for problems like "Koko Eating Bananas," and how do you verify the feasibility function is monotone?**

A: Binary search on the answer space applies when the feasibility function is monotone — if speed k is sufficient to finish within h hours, then speed k+1 is also sufficient (eating faster can only help). This creates the boolean pattern [F,F,...,F,T,T,...,T] over the answer range. Binary search finds the boundary between F and T — the minimum speed that works. To verify monotonicity, ask: "if X is feasible, is X+1 necessarily feasible?" For Koko: higher speed means fewer hours needed — yes, monotone. For Split Array: higher max sum allows fewer splits needed — yes, monotone. The feasibility check (canFinish) must run in O(n) for the total complexity to be O(n log(answer range)). The answer range is [lo, hi] where lo is the theoretically minimum answer and hi is the trivially achievable maximum — for Koko, `[1, max(piles)]`.

---

**Q: What is the difference between `lo < hi` and `lo <= hi` as the binary search loop condition, and when do you use each?**

A: `lo <= hi` is used when you are searching for an exact target and the loop should continue until every candidate has been checked. The loop exits with `lo > hi`, and you return a result found during the loop or -1. `lo < hi` is used when you are converging on a boundary — the first or last position satisfying a condition. The loop exits with `lo == hi`, and that position is the answer. With `lo < hi`, you must set `hi = mid` (not `mid - 1`) when the condition is true, to avoid skipping the actual boundary. A related subtlety: when advancing `lo = mid` (Pattern B, finding last true), use ceiling division `mid = lo + (hi - lo + 1) / 2` — if `lo = 3, hi = 4`, floor division gives `mid = 3`, setting `lo = 3` creates an infinite loop. Ceiling gives `mid = 4`, advancing `lo` and terminating. This is one of the most common binary search bugs.

---

**Q: How does exponential search handle an infinite (or unknown-length) sorted array, and what is its time complexity?**

A: Standard binary search requires knowing the array bounds upfront. For an infinite array, we first find bounds that contain the target using exponential doubling: start with `hi = 1`, and double `hi` until the element at that index is greater than or equal to the target. The target is then guaranteed to be in `[lo, hi]` where `lo` is the previous hi. This probing phase takes O(log n) steps where n is the target's index. The binary search within the found range `[lo, 2n]` takes another O(log n) steps. Total: O(log n). The out-of-bounds access returns `Integer.MAX_VALUE` (by convention), which is treated as "greater than target" — meaning the hi boundary is past the actual array end but harmlessly skips those positions during binary search. The exponential probing is also useful for sorted arrays where the length is known but very large and the target is expected near the beginning.

---

**Q: For the bitonic array search, why is finding the peak necessary rather than just binary searching directly?**

A: A bitonic array is not monotone — it increases to a peak then decreases — so there's no single comparison at mid that tells you whether to go left or right for an arbitrary target. A target value less than the peak could exist on both the ascending and descending sides, so a naive binary search would incorrectly eliminate valid halves. By first finding the peak in O(log n), we split the problem into two monotone sub-problems: an ascending half and a descending half. Each half supports standard binary search. For the descending half, the comparison is reversed: if `arr[mid] > target`, the target is to the right (larger values are to the left in a descending sequence). The total complexity is O(log n) for peak finding plus O(log n) for each of two binary searches — still O(log n) overall. An alternative approach binary searches once with `mid+1` comparison to locate the peak implicitly, but the two-search approach is clearer to implement and explain in interviews.

## 10. Key Takeaways

- **Safe mid calculation**: always `lo + (hi - lo) / 2` — integer overflow with `(lo + hi) / 2` when both are near `Integer.MAX_VALUE`
- **Rotated array**: compare `nums[lo]` vs `nums[mid]` to identify the sorted half; check if target is in that sorted range; only two extra comparisons vs standard binary search
- **Find minimum in rotated array**: compare `nums[mid]` vs `nums[hi]`; if `mid > hi`, min is right; if `mid < hi`, min is at mid or left — use `lo < hi` loop, return `nums[lo]`
- **Infinite array**: exponential probe to find `[lo, hi]`, then standard binary search — O(log n) total
- **Bitonic peak**: compare `nums[mid]` vs `nums[mid+1]`; ascending → `lo = mid+1`; descending → `hi = mid`; use `lo < hi`, return `lo`
- **Answer space binary search**: verify feasibility function is monotone; set `lo = min_answer`, `hi = max_answer`; `canAchieve(mid)` → `hi = mid`, else `lo = mid + 1`; return `lo`
- **`lo < hi` for boundary search**: loop exits with `lo == hi` = answer; use `hi = mid` when condition true, `lo = mid + 1` when false
- **Ceiling mid for "find last true"**: `mid = lo + (hi - lo + 1) / 2` prevents infinite loop when `lo = mid` is the update
- **Duplicates in rotated array**: when `nums[lo] == nums[mid] == nums[hi]`, shrink both boundaries by 1 — degrades to O(n) worst case
- **Binary search on answer space covers**: Koko bananas, split array, minimum days, capacity to ship packages — all "minimise the maximum / maximise the minimum" problems
