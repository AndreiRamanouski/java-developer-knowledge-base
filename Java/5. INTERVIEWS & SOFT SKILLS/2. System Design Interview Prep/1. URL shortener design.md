# URL Shortener Design

## Overview

A URL shortener converts long URLs into short aliases (e.g. `https://very-long-domain.com/path?params=many` → `https://short.ly/aB3xY9`) and redirects users who visit the short URL to the original. It is one of the most common system design interview questions because it touches every major design concern: capacity estimation, key generation, database selection, caching, and analytics — all in a bounded scope that can be fully covered in 45 minutes.

**Real-world examples:** bit.ly, TinyURL, t.co (Twitter), goo.gl (Google, now retired)

**Why interviewers ask this:**

```
Tests whether you can:
  1. Drive requirements gathering before jumping to solutions
  2. Translate business requirements into concrete numbers (capacity estimation)
  3. Design APIs that are clean, RESTful, and versioned
  4. Choose appropriate data stores for different access patterns
  5. Reason about key generation trade-offs (uniqueness, collisions, predictability)
  6. Build analytics pipelines without impacting the read hot path
  7. Scale a read-heavy, globally distributed system
```

## 1. Requirements Gathering

### 1.1 Functional requirements — clarify before designing

```
Questions to ask the interviewer:

1. "Who are the users — anonymous public, registered users, or API clients?"
   → Affects auth, rate limiting, and quota design.

2. "Should short URLs be random or can users choose custom aliases?"
   → Custom aliases require collision checking and a different key space.

3. "Should short URLs expire? If so, what is the default TTL?"
   → Drives storage retention policy and cleanup jobs.

4. "Do we need analytics — click counts, referrers, geography, device type?"
   → Significant write amplification; usually handled asynchronously.

5. "Should redirects be 301 (permanent) or 302 (temporary)?"
   → 301: browser caches redirect, fewer hits to our servers but no analytics.
     302: every click hits our servers, enabling analytics. Almost always 302.

6. "Do we need link preview / safety check (phishing detection)?"
   → Nice-to-have; out of scope for most interviews unless asked.

7. "Multi-region? What is the SLA for redirect latency?"
   → Drives CDN/edge caching and multi-region replication strategy.
```

### 1.2 Confirmed functional requirements

```
Core:
  ✓ Given a long URL, generate a unique short URL
  ✓ Given a short URL, redirect to the original long URL
  ✓ Users can optionally choose a custom alias
  ✓ Short URLs expire after a configurable TTL (default: 1 year)
  ✓ Each user account has a creation quota (e.g. 1,000 URLs/day)

Analytics:
  ✓ Track total click count per URL
  ✓ Track unique visitors (approximate), referrer, country, device type
  ✓ Analytics visible to URL owner via dashboard

Out of scope:
  ✗ Link preview / safety scanning
  ✗ A/B testing of redirect targets
  ✗ QR code generation (can add later)
```

### 1.3 Non-functional requirements

```
Availability:     99.99% uptime for redirect service (< 52 min/year downtime)
                  99.9% for URL creation (< 8.7 hr/year)
Latency:          Redirect: p99 < 50ms globally
                  Creation: p99 < 500ms
Durability:       URLs must never be silently lost once created
Consistency:      Eventual consistency for analytics is acceptable
                  Strong consistency for URL creation (no duplicate short codes)
Scalability:      Handle 10x traffic spikes without manual intervention
Security:         Short codes must not be guessable/enumerable
```

## 2. Capacity Estimation

### 2.1 Traffic estimates

```
Assumptions (state these explicitly in the interview):
  Daily active users: 100 million
  URL creation rate: 1% of DAU create a URL per day
    → 100M × 0.01 = 1M URL creations/day
  Redirect rate: each URL clicked 100 times on average
    → 1M × 100 = 100M redirects/day

Write QPS (creations):
  1M / 86,400 seconds = ~12 writes/second
  Peak (10x): ~120 writes/second

Read QPS (redirects):
  100M / 86,400 = ~1,160 reads/second
  Peak (10x): ~11,600 reads/second

Read:write ratio = 100:1 → heavily read-dominant system
→ Design choices: aggressive caching, read replicas, CDN for redirects
```

### 2.2 Storage estimates

```
URL record size:
  short_code:    8 bytes    (7-char Base62 code)
  long_url:      200 bytes  (average URL length)
  user_id:       8 bytes
  created_at:    8 bytes
  expires_at:    8 bytes
  click_count:   8 bytes
  Total:         ~240 bytes per URL

Storage over 5 years:
  1M URLs/day × 365 × 5 = 1.825 billion URLs
  1.825B × 240 bytes = 438 GB

  With 3× replication: ~1.3 TB
  With analytics events (100 bytes/click): 100M clicks/day × 365 × 5 × 100 = 18.25 TB
  → Separate analytics store (columnar, cost-optimised)

Key space:
  7-character Base62 code: 62^7 = 3.52 trillion unique codes
  At 1M URLs/day × 5 years = 1.825B total → 0.05% of key space used
  → 7 characters is sufficient; 6 gives 62^6 = 56.8B (still fine for 5 years)
```

### 2.3 Bandwidth estimates

```
Inbound (creation):
  12 writes/s × 200 bytes average URL = 2.4 KB/s ≈ negligible

Outbound (redirects — 302 response with Location header):
  1,160 reads/s × 200 bytes (redirect response) = 232 KB/s ≈ 2 Mbps
  Peak: ~20 Mbps → trivial for modern infrastructure

Memory (caching):
  Cache hot URLs: 20% of URLs receive 80% of traffic (Pareto)
  Cache 20% of daily URLs: 1M × 0.20 × 240 bytes = 48 MB/day
  Keep 1 week of hot URLs: 48 MB × 7 = 336 MB → fits in a single Redis instance
  With growth buffer: 4 GB Redis cluster is conservative
```

## 3. API Design

### 3.1 REST API

```
POST   /api/v1/urls          Create a short URL
GET    /api/v1/urls/{code}   Get URL metadata (NOT the redirect — see below)
DELETE /api/v1/urls/{code}   Delete a URL
GET    /{code}               Redirect to long URL (on redirect domain)
GET    /api/v1/urls/{code}/analytics   Get click analytics

Separate domains:
  API:      api.short.ly    (creation, management)
  Redirect: r.short.ly      (redirect hot path — can use CDN/edge)
  
Why separate?
  Redirect domain can be cached aggressively at CDN layer.
  API domain has auth, rate limiting, slower — don't mix hot/cold paths.
```

### 3.2 Create short URL

```
POST /api/v1/urls
Authorization: Bearer <token>

Request:
{
  "long_url":    "https://very-long-domain.com/path?param=value",
  "custom_alias": "my-link",           // optional
  "expires_in":   86400,               // optional, seconds (default: 365 days)
  "tags":         ["campaign", "q4"]   // optional, for analytics grouping
}

Response 201 Created:
{
  "short_url":    "https://r.short.ly/aB3xY9",
  "short_code":   "aB3xY9",
  "long_url":     "https://very-long-domain.com/path?param=value",
  "created_at":   "2024-03-15T10:30:00Z",
  "expires_at":   "2025-03-15T10:30:00Z",
  "click_count":  0
}

Error responses:
  400 Bad Request:   invalid URL, alias contains invalid characters
  409 Conflict:      custom alias already taken
  422 Unprocessable: URL blocked (spam/phishing blacklist)
  429 Too Many Req:  rate limit exceeded
```

### 3.3 Redirect endpoint

```
GET /aB3xY9   (on r.short.ly)

Response 302 Found:
  Location: https://very-long-domain.com/path?param=value
  Cache-Control: no-store          // prevent browser caching (need analytics)
  X-Request-Id: <uuid>             // for distributed tracing

Error responses:
  404 Not Found:  short code does not exist
  410 Gone:       short URL has expired
  
Notes:
  301 vs 302:
    301 Permanent: browser caches, subsequent visits skip our servers
        → analytics counts only first click per browser. NOT suitable.
    302 Temporary: every visit hits our servers (or CDN, which logs)
        → full analytics. USE THIS.
        
  410 Gone vs 404:
    410 signals "this existed and was intentionally removed or expired"
    SEO-friendly: tells crawlers not to retry.
```

## 4. Database Schema

### 4.1 URL store — relational (PostgreSQL)

```sql
-- Core URL table
CREATE TABLE urls (
    id           BIGSERIAL       PRIMARY KEY,
    short_code   VARCHAR(10)     NOT NULL UNIQUE,
    long_url     TEXT            NOT NULL,
    user_id      BIGINT          NOT NULL REFERENCES users(id),
    created_at   TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    expires_at   TIMESTAMPTZ,                      -- NULL = never expires
    is_active    BOOLEAN         NOT NULL DEFAULT TRUE,
    custom       BOOLEAN         NOT NULL DEFAULT FALSE,
    tags         TEXT[]                            -- optional grouping tags
);

-- Indices for access patterns
CREATE UNIQUE INDEX idx_urls_short_code ON urls(short_code);   -- redirect lookup: O(log n)
CREATE        INDEX idx_urls_user_id   ON urls(user_id, created_at DESC); -- user's URLs
CREATE        INDEX idx_urls_expires   ON urls(expires_at) WHERE expires_at IS NOT NULL;
-- Partial index on expires_at: only non-null values — used by expiry cleanup job

-- Click counter: separate table to avoid hot-row lock contention on urls
CREATE TABLE url_click_counts (
    short_code   VARCHAR(10)     PRIMARY KEY REFERENCES urls(short_code),
    total_clicks BIGINT          NOT NULL DEFAULT 0
);
-- Updated by async aggregator from analytics stream, not on every click
```

### 4.2 Why separate click_count table?

```
Anti-pattern: UPDATE urls SET click_count = click_count + 1 WHERE short_code = 'aB3xY9'

Problem: viral URL with 10,000 clicks/second → 10,000 concurrent updates to
         same row → exclusive row locks → serialised writes → throughput collapse.

Solutions:
  Option A: Separate click_count table, updated by async aggregator from Kafka events
            → main url row never updated on redirect; no lock contention.
  Option B: Redis INCR for real-time count, periodic flush to DB
            → O(1) atomic increment, no DB writes on hot path.
  Option C: Approximate counts using HyperLogLog for unique visitors.

Design choice: Redis INCR for real-time display, Kafka stream for durable analytics.
```

### 4.3 Analytics store — event-based (Cassandra or ClickHouse)

```sql
-- Cassandra schema (wide-column, append-only analytics events)
-- Partition by short_code + bucket date (prevents unbounded partition growth)
CREATE TABLE click_events (
    short_code   TEXT,
    bucket_date  DATE,                    -- partition by day
    event_id     TIMEUUID,               -- time-ordered, globally unique
    timestamp    TIMESTAMP,
    user_agent   TEXT,
    ip_address   TEXT,                    -- hashed for privacy
    referrer     TEXT,
    country_code TEXT,
    device_type  TEXT,                    -- 'mobile'|'desktop'|'tablet'
    PRIMARY KEY ((short_code, bucket_date), event_id)
) WITH CLUSTERING ORDER BY (event_id DESC)
  AND default_time_to_live = 7776000;     -- 90-day TTL (analytics retention policy)

-- Why Cassandra for analytics?
--   Append-only workload (events never updated)
--   High write throughput (log-structured merge tree)
--   Time-series natural partition key
--   TTL support for data retention
--   Wide-column allows adding dimensions without schema migration

-- Alternative: ClickHouse (columnar OLAP)
--   Better for ad-hoc analytical queries (GROUP BY country, GROUP BY day)
--   Columnar compression: 10:1 vs row stores
--   MergeTree engine with TTL — same concept, faster analytical reads
--   Write via Kafka consumer → ClickHouse Kafka engine table
```

## 5. Key Generation Strategies

### 5.1 Strategy comparison

```
Strategy          Pros                          Cons
──────────────────────────────────────────────────────────────────────────────
Random Base62     Simple, unpredictable         Collision probability grows
                  No coordination needed         Must check DB on every generation
                  O(1) generation               Birthday problem: ~50% collision
                                                 at sqrt(62^7) ≈ 1.8M codes

Pre-generated     Zero collision risk            Requires coordination service
  (Key DB)        O(1) lookup, no DB check       KDB is SPOF (must replicate)
                  Predictable                    Complex operational setup

MD5/SHA hash      Deterministic (same URL        First 7 chars of hash → collision
  of URL          → same short code)             Same URL ≠ same owner
                  No coordination needed         Must handle hash collisions

Snowflake ID      Globally unique, time-sorted   64-bit → longer code than needed
  + Base62        No coordination needed         Timestamp leaks creation time
                  Monotonically increasing        Sequential codes guessable

Counter           Simple, no collisions          Counter is SPOF, needs coordination
  (Auto-          + Base62 conversion            Sequential: guessable
   increment)     Predictable code lengths       Range-partitioning needed at scale

Recommended: Pre-generated keys in Key Generation Service (KGS)
```

### 5.2 Recommended: Key Generation Service (KGS)

```
Architecture:
  ┌───────────────────────────────────────────────────┐
  │              Key Generation Service (KGS)          │
  │                                                   │
  │  Key DB (used_keys table, unused_keys table)       │
  │                                                   │
  │  On startup: load N keys from unused → used_keys  │
  │  (in-memory buffer of pre-fetched keys)            │
  │                                                   │
  │  On request: pop key from in-memory buffer,        │
  │              return immediately                    │
  └───────────────────────────────────────────────────┘

Key DB schema:
  CREATE TABLE unused_keys (key CHAR(7) PRIMARY KEY);
  CREATE TABLE used_keys   (key CHAR(7) PRIMARY KEY);

  Pre-populate with all (or a large batch of) 7-char Base62 codes.
  Keys move from unused_keys to used_keys when assigned.

KGS operation:
  1. On boot: SELECT key FROM unused_keys LIMIT 1000 FOR UPDATE SKIP LOCKED
     → load 1000 keys into memory, mark them used in a transaction.
  2. On key request: return next key from in-memory buffer (O(1), no DB call).
  3. When buffer depletes: refill from DB (background thread, not on request path).

Benefits:
  No collision possible (each key used exactly once).
  No DB round trip on redirect creation.
  KGS can be replicated; each instance takes a disjoint key batch.

Failure handling:
  KGS crashes: in-memory keys lost → acceptable (small gap in key space, not a bug).
  Key DB fails: KGS falls back to random generation with DB collision check.
```

### 5.3 Snowflake ID approach (alternative)

```java
// Twitter Snowflake: 64-bit ID composed of:
//   1 bit sign (always 0)
//   41 bits timestamp (milliseconds since epoch) → ~69 years
//   10 bits machine/datacenter ID                → 1024 machines
//   12 bits sequence number                       → 4096 IDs/ms/machine

// Convert to Base62 for the short code
public String generateShortCode(long snowflakeId) {
    return toBase62(snowflakeId); // 64-bit → ~11 Base62 chars (too long)
    // Use lower 42 bits for shorter codes: 62^7 = 3.5T, 2^42 = 4.4T
}

private static final String BASE62 = "0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz";

public String toBase62(long num) {
    StringBuilder sb = new StringBuilder();
    while (num > 0) {
        sb.append(BASE62.charAt((int)(num % 62)));
        num /= 62;
    }
    return sb.reverse().toString();
}

// Downside: codes grow monotonically — attacker can enumerate recent URLs by
// incrementing the decoded timestamp. Use shuffled Base62 alphabet or XOR mask
// to obfuscate while maintaining uniqueness.
```

### 5.4 Hash-based approach with collision handling

```java
@Service
public class HashKeyGenerator {
    private final UrlRepository repo;

    public String generate(String longUrl, String userId) {
        // MD5 hash → take first 7 Base62 chars
        byte[] md5 = MessageDigest.getInstance("MD5").digest(
            (longUrl + userId + System.nanoTime()).getBytes()
        );
        String candidate = toBase62(md5).substring(0, 7);

        // Collision retry loop (rare — expected <0.001% at our scale)
        int attempts = 0;
        while (repo.existsByShortCode(candidate)) {
            candidate = toBase62((longUrl + attempts++).getBytes()).substring(0, 7);
            if (attempts > 5) throw new KeyGenerationException("Too many collisions");
        }
        return candidate;
    }
}
// Collision probability: birthday problem with 62^7 = 3.5T slots, 1.8B URLs at 5y
// P(collision) ≈ 1 - e^(-n²/2m) = 1 - e^(-1.8B²/(2×3.5T)) ≈ 32%
// → At scale, collisions are common → retry loop is required → KGS is better
```

### 6.1 Redirect service — the hot path

```java
@RestController
public class RedirectController {

    @GetMapping("/{code}")
    public ResponseEntity<Void> redirect(@PathVariable String code,
                                          HttpServletRequest request) {
        // 1. Cache lookup (< 1ms)
        String longUrl = cache.get("url:" + code);

        if (longUrl == null) {
            // 2. DB lookup on cache miss
            Url url = urlRepository.findByShortCode(code)
                .orElseThrow(() -> new UrlNotFoundException(code));

            if (url.isExpired()) throw new UrlExpiredException(code);

            longUrl = url.getLongUrl();
            cache.set("url:" + code, longUrl, Duration.ofHours(24));
        }

        // 3. Async: publish click event to Kafka (non-blocking — fire and forget)
        eventPublisher.publishAsync(ClickEvent.of(code, request));

        // 4. Redirect
        return ResponseEntity.status(HttpStatus.FOUND)
                             .header("Location", longUrl)
                             .header("Cache-Control", "no-store")
                             .build();
    }
}
```

## 7. Analytics Tracking

### 7.1 Architecture — async pipeline, never block redirect

```
Redirect happens synchronously.
Analytics happens asynchronously via event stream.
NEVER write analytics synchronously on the redirect path — it would add
database latency to every click, destroying redirect performance.

Pipeline:
  1. Redirect service publishes ClickEvent to Kafka topic 'click-events'
     (fire-and-forget: at-most-once delivery acceptable for analytics)

  2. Flink / Kafka Streams consumer:
       Enriches event: IP → country (MaxMind GeoIP), User-Agent → device type
       Aggregates: per short_code per hour counts
       Writes to ClickHouse (columnar OLAP) for ad-hoc analytics queries
       Updates Redis counter: INCR url:{code}:clicks (real-time display)

  3. Batch job (nightly):
       Sync Redis click counters to PostgreSQL url_click_counts table
       Prune expired URLs (soft-delete is_active = false)
       Generate analytics summaries for dashboard
```

### 7.3 Real-time click counter with Redis

```java
@Service
public class ClickCounterService {
    private final StringRedisTemplate redis;
    private final ClickCountRepository dbRepo;

    // On every redirect: fire-and-forget increment
    public void recordClick(String shortCode) {
        String key = "clicks:" + shortCode;
        redis.opsForValue().increment(key);
        redis.expire(key, Duration.ofDays(7)); // rolling TTL
    }

    // API: get current count (fast path via Redis)
    public long getClickCount(String shortCode) {
        String val = redis.opsForValue().get("clicks:" + shortCode);
        if (val != null) return Long.parseLong(val);

        // Cache miss: fall back to DB (for older or cold URLs)
        return dbRepo.findByShortCode(shortCode)
                     .map(UrlClickCount::getTotalClicks)
                     .orElse(0L);
    }

    // Scheduled: flush Redis counters to PostgreSQL
    @Scheduled(cron = "0 */10 * * * *") // every 10 minutes
    public void flushCountsToDB() {
        Set<String> keys = redis.keys("clicks:*");
        for (String key : keys) {
            String code = key.replace("clicks:", "");
            Long count  = redis.opsForValue().getAndSet(key, "0");
            if (count != null && count > 0) {
                dbRepo.incrementBy(code, count);
            }
        }
    }
}
```

### 7.4 Unique visitor estimation with HyperLogLog

```java
// HyperLogLog: probabilistic counting with ~0.81% standard error, O(1) space per key
// Redis PFADD / PFCOUNT: native HyperLogLog support

public void trackUniqueVisitor(String shortCode, String sessionId) {
    redis.opsForHyperLogLog().add("unique:" + shortCode, sessionId);
    // PFADD short_code session_id — O(1), uses ~12KB regardless of cardinality
}

public long getUniqueVisitors(String shortCode) {
    return redis.opsForHyperLogLog().size("unique:" + shortCode);
    // PFCOUNT — returns approximate cardinality
}
// vs exact counting: SET of session IDs grows unboundedly, O(n) space
// HyperLogLog gives 0.81% error at constant 12KB — acceptable for analytics
```

## 8. Scaling Considerations

### 8.1 Caching strategy

```
Cache what: short_code → long_url mappings (hot read path)
Cache where: Redis Cluster (in-memory, O(1) lookup)
Cache TTL: 24 hours (shorter for soon-to-expire URLs)
Cache aside pattern: check cache → on miss, load from DB → write to cache

Eviction policy: allkeys-lru (evict least recently used when memory full)
  Alternative: allkeys-lfu for long-tail access patterns

Cache hit rate target: >99% for recently created URLs
  20% of URLs get 80% of traffic → cache those 20% (Pareto principle)
  1M daily URLs × 0.20 × 240 bytes = 48 MB/day — trivially cacheable
```

### 8.2 Database scaling

```
Read scaling:
  Add read replicas (PostgreSQL streaming replication)
  Route all SELECT to replicas; only writes go to primary
  Acceptable replication lag for redirect: < 100ms

Write scaling:
  Horizontal sharding by short_code (hash-based)
  Shard key: first 2 chars of short_code → 62^2 = 3,844 shards (logical)
  Physical shards: 10 initially, expand as needed

  Alternative: Use managed DB with auto-sharding
    PlanetScale (MySQL-compatible), CockroachDB, DynamoDB

For URL creation at 120 writes/s peak: single PostgreSQL primary handles easily
  PostgreSQL: ~10,000 simple writes/s → 120 writes/s is trivial
  Future scaling: shard only when approaching 1,000+ writes/s
```

### 8.3 Global multi-region deployment

```
Active-Active setup:
  Region US-EAST: serves US traffic (primary write region)
  Region EU-WEST: serves EU traffic (read + write with CRDT conflict resolution)
  Region AP-EAST: serves Asia traffic

DNS-based routing:
  GeoDNS routes users to nearest region
  Short URL resolution: served from local region's cache/DB
  URL creation: can write to local region, replicate globally

Data consistency:
  Eventual consistency for reads (acceptable: stale data for 1-2 seconds)
  Strong consistency for creation: use a global transaction or accept
    "last writer wins" with short_code uniqueness enforced per-region
    (two regions rarely generate the same random code simultaneously)

CDN for redirects:
  Cache redirect at CDN edge (BUT: 302 + Cache-Control: no-store prevents CDN caching)
  Workaround: log click at CDN edge (Lambda@Edge / Cloudflare Workers)
              then redirect — analytics without hitting origin servers
```

## 9. URL Expiry and Cleanup

```java
// Approach 1: Lazy expiry (check on access)
// Check expires_at on every redirect — if expired, return 410 Gone.
// Pro: simple, no background job needed.
// Con: expired entries accumulate in DB, slow index scans over time.

// Approach 2: Active cleanup job (recommended)
@Scheduled(cron = "0 0 2 * * *") // 2am daily, low-traffic window
@Transactional
public void cleanupExpiredUrls() {
    // Soft delete first (is_active = false), hard delete after 30-day grace period
    int softDeleted = urlRepository.softDeleteExpired(Instant.now());
    int hardDeleted = urlRepository.hardDeleteOldSoftDeleted(
        Instant.now().minus(Duration.ofDays(30))
    );
    log.info("Expired URLs: soft={}, hard={}", softDeleted, hardDeleted);
    // Optionally return freed short codes to KGS unused_keys pool
}

// Approach 3: TTL via Redis (for recently created URLs)
// On creation: SET url:{code} {long_url} EX {ttl_seconds}
// Redis auto-deletes on expiry — no cleanup job needed for cache layer
// DB expiry still handled by periodic job
```

## 10. Security Considerations

```
1. Short code unpredictability:
   Base62 random 7 chars → 62^7 = 3.5T possibilities
   Brute-force scanning: even 10M/second = 350,000 seconds = 4 days to enumerate 0.001%
   Acceptable for most use cases; add captcha on bulk redirect if needed

2. Rate limiting:
   Creation: 100 URLs/minute per API key (sliding window in Redis)
   Redirects: 1000/minute per IP (prevent DoS scraping)

3. URL validation and blacklisting:
   Validate URL format on creation (reject javascript:, data: protocols)
   Check against Google Safe Browsing API or internal blacklist
   Domain allowlist for enterprise plans

4. Custom alias abuse:
   Reject aliases containing profanity, brand names, reserved words
   Keep list of reserved codes (api, www, admin, health, metrics, static...)

5. Analytics privacy:
   Hash IP addresses before storage (SHA-256 + salt) — GDPR compliance
   Store geolocation (country/city) but NOT raw IP in analytics DB
   Honor Do Not Track (DNT) header for analytics recording
```

## 11. Interview Questions

**Q: Walk through what happens end-to-end when a user clicks a short URL.**

A: The user's browser sends a GET request to `r.short.ly/aB3xY9`. DNS resolves the domain to our load balancer (or CDN edge node). The redirect service first checks Redis for `url:aB3xY9` — a cache hit returns the long URL in under a millisecond. On a cache miss, it queries the PostgreSQL read replica using the unique index on short_code. If found and not expired, it writes the result to Redis with a 24-hour TTL. Simultaneously, a ClickEvent is published to a Kafka topic — this is fire-and-forget, not blocking the redirect. The service returns an HTTP 302 response with a `Location` header pointing to the long URL and `Cache-Control: no-store` to prevent browser caching (ensuring analytics works). The browser follows the redirect, and the user lands on the destination. Total p99 latency target: 50ms globally, achievable since the hot path involves only one Redis lookup and one Kafka async publish.

---

**Q: How do you ensure short code uniqueness at scale across multiple service instances?**

A: There are two viable approaches. The Key Generation Service pre-generates all possible Base62 codes offline, stores them in a `unused_keys` table, and issues batches to each application instance using `SELECT ... FOR UPDATE SKIP LOCKED`. Each instance holds a buffer of 1,000 keys in memory. Since keys are atomically moved from unused to used during the batch acquisition, no two instances ever receive the same key — there are zero collisions at runtime, no DB check per creation. The alternative is a hash-based approach (hash the long URL + user ID + timestamp, take first 7 Base62 characters) which requires a uniqueness check in the DB and a retry loop on collision. The birthday problem makes collisions frequent at scale — with 1.8 billion URLs, collision probability for 7-character codes is ~32%. KGS eliminates this entirely at the cost of operational complexity and the need to replicate the key service. For interview purposes, explaining both and recommending KGS with its failure modes (instance crash loses in-memory keys — acceptable, key space gap is harmless) demonstrates depth.

---

**Q: Why should analytics tracking be asynchronous, and what are the trade-offs?**

A: Redirects happen hundreds of millions of times per day. Synchronous analytics — writing a click record to the database on every redirect — would add a database write to the critical path. At 1,160 redirects/second, this means 1,160 database writes/second just for analytics, adding 5-50ms to every redirect response and creating a write bottleneck that could take down the entire redirect service if the analytics DB is slow. The asynchronous approach publishes a Kafka event (microseconds) and returns the 302 immediately. The Kafka consumer pipeline enriches events with GeoIP and user agent parsing, aggregates them, and writes to ClickHouse in micro-batches. The trade-off: analytics are eventually consistent — click counts may lag by 5-30 seconds. For a URL shortener, this is entirely acceptable. A second trade-off: at-most-once delivery (fire-and-forget Kafka publish) means a small percentage of clicks may not be recorded during service restarts. Switching to at-least-once with synchronous Kafka confirmation costs ~5ms per redirect — worth it for paying customers who need exact counts.

---

**Q: How would you handle the thundering herd problem if a short URL suddenly goes viral?**

A: A URL going viral means one short code receiving sudden massive traffic — potentially millions of requests per second. First line of defence: Redis cache. If the URL is cached, all requests hit Redis, which handles millions of operations per second. The DB is insulated. Second problem: cache stampede — if the cache entry expires while the URL is viral, thousands of simultaneous requests all miss the cache and hit the DB simultaneously. Mitigation: use a mutex lock or probabilistic early expiration. With a mutex, only one request fetches from DB while others wait and then read from the freshly populated cache. Probabilistic early expiration recomputes the cache slightly before expiry (based on "how expensive is the recompute" × "how early to refresh"), effectively preventing expiry during high traffic. Third problem: Kafka backpressure — millions of ClickEvents published per second. Use Kafka partitioning on short_code so the analytics pipeline can scale horizontally. The click counter uses Redis INCR (atomic, millions/second) rather than Kafka for real-time count, falling back to Kafka for detailed event storage. Finally, rate limiting per IP prevents a single origin from flooding the redirect service, and circuit breakers on the DB connection pool ensure a slow DB doesn't cascade to service failure.

---

**Q: How would you support custom analytics — e.g. "show me all clicks from mobile devices in Germany last week"?**

A: This is an ad-hoc analytical query that belongs in a columnar store, not the OLTP PostgreSQL database. Every click event is published to Kafka and consumed by ClickHouse via its Kafka engine table. ClickHouse stores events in a MergeTree with partition by event date and sorting by short_code — enabling efficient range scans by date and exact matches by code. The query "clicks from mobile, Germany, last 7 days" becomes: `SELECT count() FROM click_events WHERE short_code='aB3xY9' AND event_date BETWEEN today()-7 AND today() AND country_code='DE' AND device_type='mobile'`. ClickHouse's columnar storage reads only the three relevant columns, not the full row, making this orders of magnitude faster than PostgreSQL on hundreds of millions of rows. Pre-aggregated materialised views (daily/hourly rollups per short_code per country per device) further accelerate dashboard queries — the rollup table is tiny compared to raw events, and most dashboard queries target the rollup rather than raw events. TTL policies on raw events (90-day retention) prevent unbounded growth while rollups are kept forever.

## 12. Key Takeaways

- **Read:write is 100:1** — optimise the redirect path aggressively; creation can be slower and more complex
- **Redirect path = cache lookup + async Kafka publish + 302 response**: never block on DB write or analytics on the redirect hot path
- **302 not 301**: 301 caches in browser, breaking analytics; 302 sends every click through your servers (or CDN with logging)
- **Key generation**: KGS with pre-generated keys in `unused_keys` table eliminates collisions entirely; hash-based with retry loop is simpler but fragile at scale
- **Click counting**: Redis INCR for real-time display (atomic, millions/second); flush to PostgreSQL periodically; never UPDATE the URL row on every click (row lock contention)
- **Analytics store**: separate from OLTP — ClickHouse (columnar) or Cassandra (time-series) for click events; PostgreSQL for URL metadata
- **Unique visitor estimation**: HyperLogLog (Redis PFADD/PFCOUNT) gives ~0.81% error at constant 12KB memory vs O(n) exact set
- **Expiry**: check lazily on access (simple) + active cleanup job at low-traffic window (required for index health); soft-delete before hard-delete
- **URL validation**: reject javascript:/data: protocols, check against phishing blacklist, reserve system aliases (api, admin, health)
- **Scale levers**: Redis cache (hits prevent DB load), read replicas (read scaling), KGS batching (write scaling), ClickHouse (analytics scaling), CDN + Lambda@Edge (global redirect latency)
