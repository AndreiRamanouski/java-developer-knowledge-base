# Social Media Feed Design (Twitter/X)

## Overview

A social media feed is one of the most demanding system design problems: it combines high write throughput (tweets), extremely high read throughput (feed loads), complex fan-out logic, personalisation, and real-time delivery — all at global scale. Twitter processes ~6,000 tweets per second at peak, serves ~800 million timeline requests per day, and must deliver feeds in under 200ms.

**The core tension:**

```
Every tweet must appear in the feeds of all followers.
A user with 100M followers posting one tweet = 100M feed updates.
A user loading their feed follows 500 people = 500 feeds to merge.

Fan-out on write:  push tweet to all follower feeds at write time
                   → fast reads, expensive writes (celebrities are catastrophic)

Fan-out on read:   compute feed at read time by merging followees' tweets
                   → cheap writes, expensive reads (active users pay on every load)

Real systems use a HYBRID of both.
```

## 1. Requirements Gathering

### 1.1 Clarifying questions

```
1. "What is the primary feed — home timeline (followed users) or discovery feed?"
   → Home timeline: followed users' tweets. Discovery: algorithmic recommendations.
   → Start with home timeline; note algorithmic feed as extension.

2. "Do we need real-time delivery or is eventual consistency acceptable?"
   → Tweets should appear in follower feeds within a few seconds (soft real-time).
   → Not strict real-time; a 5-second lag is acceptable.

3. "What media types? Text only, images, video?"
   → Text + images in scope. Video storage/CDN is a separate sub-problem.

4. "What is the follower scale? Do celebrities (millions of followers) exist?"
   → Yes — the celebrity problem is a central design challenge.

5. "Do we need ranked/algorithmic feeds or just reverse chronological?"
   → Both: default is reverse-chronological; algorithmic is an extension.

6. "What engagement interactions exist? Likes, retweets, replies?"
   → In scope for data model; feed ranking uses engagement signals.

7. "Notifications? Push to mobile?"
   → Out of scope for this design; separate notification service.
```

### 1.2 Functional requirements

```
Core:
  ✓ Post a tweet (text ≤ 280 chars, optional media)
  ✓ Follow / unfollow users
  ✓ Home timeline: ordered feed of tweets from followed users
  ✓ User timeline: all tweets by a specific user
  ✓ Like, retweet, reply to tweets
  ✓ Delete a tweet

Feed:
  ✓ New tweets appear in follower feeds within ~5 seconds
  ✓ Reverse-chronological default ordering
  ✓ Paginated (cursor-based, not offset)
  ✓ Algorithmic ranking as an optional mode

Out of scope:
  ✗ Direct messages
  ✗ Video upload pipeline
  ✗ Push notifications
  ✗ Trending topics / search
```

### 1.3 Non-functional requirements

```
Scale:
  300 million monthly active users, 100M daily active users
  6,000 tweets/second (peak ~12,000)
  800M timeline reads/day (~9,300 reads/second average, peak ~50,000)

Latency:
  Timeline load: p99 < 200ms
  Tweet post → appears in follower feeds: < 5 seconds (soft)

Availability:
  Read path: 99.99% (timeline must always load something)
  Write path: 99.9% (brief tweet post failures acceptable)

Consistency:
  Eventual consistency for feed delivery (acceptable)
  Strong consistency for tweet creation (tweet must be durable once posted)

Storage:
  Tweets: ~100 bytes average
  6,000 tweets/s × 86,400 s/day × 365 × 5 years × 100 bytes = ~9.5 TB (text only)
  Follower graph: ~500M edges (each user follows ~500 on average)
```

### 2.1 Core tables

```sql
-- Tweets
CREATE TABLE tweets (
    tweet_id     BIGINT          PRIMARY KEY,  -- Snowflake ID (time-sortable)
    user_id      BIGINT          NOT NULL,

    retweet_count BIGINT         NOT NULL DEFAULT 0,
    reply_count  BIGINT          NOT NULL DEFAULT 0

CREATE INDEX idx_tweets_user_created ON tweets(user_id, tweet_id DESC);

-- Users
CREATE TABLE users (
    user_id      BIGINT          PRIMARY KEY,
    username     VARCHAR(50)     NOT NULL UNIQUE,
    display_name VARCHAR(100),
    follower_count  BIGINT       NOT NULL DEFAULT 0,
    following_count BIGINT       NOT NULL DEFAULT 0,
    is_celebrity BOOLEAN         NOT NULL DEFAULT FALSE,
    -- is_celebrity: follower_count > CELEBRITY_THRESHOLD (1M)
    -- Set by background job, not real-time
    created_at   TIMESTAMPTZ     NOT NULL
);

-- Social graph (follow relationships)
CREATE TABLE follows (
    follower_id  BIGINT          NOT NULL,  -- who is following
    followee_id  BIGINT          NOT NULL,  -- who is being followed
    created_at   TIMESTAMPTZ     NOT NULL,
    PRIMARY KEY (follower_id, followee_id)
);
CREATE INDEX idx_follows_followee ON follows(followee_id);  -- "who follows me?"
CREATE INDEX idx_follows_follower ON follows(follower_id);  -- "who do I follow?"

-- Timeline cache (materialised feed per user — fan-out on write result)
-- Stored in Redis, not PostgreSQL, but schema shown for clarity
-- KEY: timeline:{user_id}
-- VALUE: sorted set of tweet_ids ordered by timestamp score
```

### 2.2 Snowflake ID for tweet_id

```
Snowflake guarantees tweet_ids are:
  1. Globally unique across all tweet service instances
  2. Time-sortable (higher ID = more recent tweet)
  3. Monotonically increasing within a single millisecond per machine

This means:
  Fetching "most recent tweets by user" = range scan on tweet_id DESC
  No ORDER BY timestamp needed — tweet_id ordering IS timestamp ordering
  Timeline construction = merge sorted lists by tweet_id

Snowflake structure:
  [1 bit sign][41 bits timestamp ms][10 bits machine][12 bits sequence]
  → 64-bit integer → fits in BIGINT → O(1) index lookups
```

## 3. Fan-out on Write vs Fan-out on Read

### 3.1 Fan-out on write (push model)

```
When user A posts a tweet:
  1. Store tweet in tweets table
  2. Look up all of A's followers
  3. For each follower F: append tweet_id to F's timeline cache
  
Result: F's timeline is pre-computed and ready to read instantly.

┌────────┐    post tweet     ┌───────────────┐
│ User A │──────────────────▶│ Tweet Service │
└────────┘                   └───────┬───────┘
                                     │ 1. store tweet
                                     │ 2. load followers (500K)
                                     ▼
                            ┌───────────────────┐     append to
                            │  Fan-out Service  │────────────────▶ [Feed F1]
                            │  (async workers)  │────────────────▶ [Feed F2]
                            └───────────────────┘     ......
                                                  ────────────────▶ [Feed F500K]

Read path: trivial
  GET timeline:user_123 → return pre-built sorted set from Redis

Pros:
  ✓ Feed reads are O(1): just fetch the pre-built list from Redis
  ✓ No computation on read — very fast, predictable latency
  ✓ Easy to paginate

Cons:
  ✗ Celebrity problem: 100M followers × 1 tweet = 100M Redis writes
  ✗ Fan-out latency: followers see tweet after minutes, not seconds
  ✗ Wasted work: inactive users' feeds are updated but never read
  ✗ Storage: each tweet_id stored in every follower's feed
    → 1 tweet × 100M followers × 8 bytes = 800 MB for one celebrity tweet
```

### 3.2 Fan-out on read (pull model)

```
When user F loads their timeline:
  1. Look up all users F follows
  2. For each followee: fetch their N most recent tweets
  3. Merge all lists by timestamp → return top K

┌────────┐   load timeline    ┌───────────────────┐
│ User F │───────────────────▶│  Timeline Service  │
└────────┘                   └────────┬────────────┘
                                       │ 1. get followees (500 users)
                                       │ 2. fetch recent tweets for each
                                       │ 3. merge 500 sorted lists → top 20
                                       ▼
                              ┌─────────────────┐
                              │  tweets table   │  × 500 queries or
                              │  + user caches  │  1 scatter-gather
                              └─────────────────┘

Pros:
  ✓ No fan-out cost on write: tweet stored once
  ✓ No celebrity problem: celebrity's tweets just sit in their cache
  ✓ No wasted storage for inactive users

Cons:
  ✗ Slow reads: merge 500 sources on every timeline load
  ✗ N+1 query problem: 500 DB lookups per timeline request
    → At 50,000 timeline reads/second: 25 million DB queries/second
  ✗ Hard to rank: sorting 500 sources is expensive
  ✗ Cold start per request: can't precompute across diverse followings
```

### 3.3 Hybrid model — what Twitter actually does

```
Fan-out on write for NORMAL users (< 1M followers):
  On tweet post: push tweet_id to all follower timelines in Redis.
  Fast reads for the vast majority of the user base.

Fan-out on read for CELEBRITIES (> 1M followers):
  Do NOT push celebrity tweets to followers' Redis timelines.
  Instead, at read time: inject celebrity tweets into the timeline.

Algorithm at read time:
  1. Load user's pre-built timeline from Redis (fan-out-on-write portion)
     → contains tweets from normal users the person follows
  2. Load IDs of celebrities the user follows (small set: usually < 10)
  3. For each celebrity: fetch their recent tweets from celebrity tweet cache
  4. Merge the two lists by timestamp → return top K

Result:
  Normal user tweet → fan-out to ~500 followers (cheap)
  Celebrity tweet   → zero fan-out; injected lazily at read time
  Timeline read     → 1 Redis fetch + small merge (fast)
```

### 4.2 Tweet post flow

```java
// Tweet creation: synchronous store + async fan-out
@Service
public class TweetService {

    @Transactional
    public TweetResponse postTweet(long userId, CreateTweetRequest req) {
        // 1. Generate Snowflake ID
        long tweetId = snowflakeIdGenerator.nextId();

        // 2. Store tweet durably
        Tweet tweet = Tweet.builder()
            .tweetId(tweetId)
            .userId(userId)
            .content(req.getContent())
            .mediaIds(req.getMediaIds())
            .createdAt(Instant.now())
            .build();
        tweetRepository.save(tweet);

        // 3. Cache this user's tweet for fan-out-on-read (celebrity path)
        tweetCache.cacheTweet(userId, tweet); // ZADD user_tweets:{userId} tweetId score=timestamp

        // 4. Async: publish to Kafka for fan-out workers
        fanOutEventPublisher.publish(new TweetCreatedEvent(tweetId, userId));

        return TweetResponse.from(tweet);
    }
}

@KafkaListener(topics = "tweet-created", concurrency = "32")
public class FanOutWorker {

    public void onTweetCreated(TweetCreatedEvent event) {
        User author = userService.getUser(event.getUserId());

        if (author.isCelebrity()) {
            // Celebrity: no fan-out. Tweet sits in author's cache.
            // Will be injected lazily at timeline read time.
            return;
        }

        // Normal user: fan-out to all followers
        List<Long> followerIds = socialGraphService.getFollowers(event.getUserId());

        // Batch Redis writes for efficiency
        List<List<Long>> batches = Lists.partition(followerIds, 500);
        for (List<Long> batch : batches) {
            redisPipeline.execute(pipeline -> {
                for (long followerId : batch) {
                    String key = "timeline:" + followerId;
                    pipeline.zadd(key, event.getTweetId(), // score=tweetId (sortable by time)
                                  String.valueOf(event.getTweetId()));
                    pipeline.zremrangeByRank(key, 0, -(TIMELINE_CACHE_SIZE + 1)); // keep top 800
                }
            });
        }
    }
}
```

### 4.3 Timeline read flow

```java
@Service
public class TimelineService {
    private static final int PAGE_SIZE = 20;
    private static final int TIMELINE_CACHE_SIZE = 800;

    public TimelinePage getHomeTimeline(long userId, String cursor) {
        // 1. Load pre-built timeline from Redis (fan-out-on-write portion)
        List<Long> cachedTweetIds = loadFromCache(userId, cursor, PAGE_SIZE * 2);

        // 2. Identify celebrities this user follows
        List<Long> celebFollowees = socialGraphService.getCelebrityFollowees(userId);

        // 3. Fetch celebrity tweets lazily (fan-out-on-read portion)
        List<Long> celebTweetIds = celebFollowees.stream()
            .flatMap(celebId -> tweetCache.getRecentTweetIds(celebId, PAGE_SIZE).stream())
            .collect(Collectors.toList());

        // 4. Merge and sort by tweet_id (descending = most recent first)
        List<Long> mergedIds = mergeSorted(cachedTweetIds, celebTweetIds);
        List<Long> pageIds   = applyPagination(mergedIds, cursor, PAGE_SIZE);

        // 5. Hydrate: fetch full tweet objects for the page's tweet IDs
        //    (from tweet cache or DB, batch lookup)
        List<Tweet> tweets = tweetService.hydrate(pageIds);

        // 6. If ranking enabled: re-score and re-sort
        if (userPreferences.isRankedFeed(userId)) {
            tweets = rankingService.rank(tweets, userId);
        }

        return TimelinePage.of(tweets, nextCursor(tweets));
    }

    private List<Long> loadFromCache(long userId, String cursor, int limit) {
        long maxScore = cursor == null ? Long.MAX_VALUE : Long.parseLong(cursor);
        return redis.zrevrangeByScore("timeline:" + userId, maxScore, 0, limit);
        // ZREVRANGEBYSCORE: O(log N + M) where M = results returned
    }
}
```

### 4.4 Cache rebuild on cache miss

```java
// If Redis timeline for a user is empty (new user, cache eviction, restart):
// Rebuild by querying the DB for recent tweets from all followees.
private List<Long> rebuildTimeline(long userId) {
    List<Long> followeeIds = socialGraphService.getFollowees(userId);

    // Parallel fetch: recent tweets from each followee
    List<CompletableFuture<List<Long>>> futures = followeeIds.stream()
        .map(fid -> CompletableFuture.supplyAsync(
            () -> tweetRepository.findRecentByUser(fid, TIMELINE_CACHE_SIZE),
            fanOutExecutor))
        .collect(Collectors.toList());

    List<Long> allTweetIds = futures.stream()
        .map(CompletableFuture::join)
        .flatMap(Collection::stream)
        .sorted(Comparator.reverseOrder()) // Snowflake IDs are time-sortable
        .limit(TIMELINE_CACHE_SIZE)
        .collect(Collectors.toList());

    // Populate cache
    redis.zadd("timeline:" + userId,
        allTweetIds.stream().collect(
            Collectors.toMap(id -> (double) id, id -> String.valueOf(id))));

    return allTweetIds;
}
```

## 5. Caching Strategies

### 5.1 Cache layers

```
Layer 1 — Timeline cache (Redis Sorted Set)
  Key:   timeline:{user_id}
  Value: sorted set of tweet_ids, score = tweet_id (Snowflake = time-sortable)
  Size:  top 800 tweet_ids per user (covers ~40 pages of 20)
  TTL:   7 days (evict inactive users' timelines)
  Usage: home timeline reads — primary hot path

Layer 2 — Tweet content cache (Redis Hash / String)
  Key:   tweet:{tweet_id}
  Value: serialised Tweet object (content, counts, author info)
  TTL:   24 hours (viral tweets stay hot longer; use EXPIRE refresh on access)
  Usage: tweet hydration — avoids DB read per tweet on every timeline load

Layer 3 — User profile cache (Redis Hash)
  Key:   user:{user_id}
  Value: username, display_name, avatar_url, follower_count, is_celebrity
  TTL:   1 hour
  Usage: timeline rendering — each tweet needs author info displayed

Layer 4 — Celebrity tweet cache (Redis Sorted Set)
  Key:   user_tweets:{user_id}  (for celebrities only)
  Value: sorted set of recent tweet_ids
  Size:  200 most recent tweets (covers fan-out-on-read merging)
  TTL:   48 hours
  Usage: injected into timelines of celebrity followers at read time

Layer 5 — Social graph cache (Redis Set)
  Key:   followees:{user_id}     (who does this user follow?)
         celebrity_followees:{user_id}  (celebrity subset — used at read time)
  TTL:   5 minutes (follow/unfollow changes propagate quickly)
  Usage: fan-out routing + timeline merge
```

### 5.2 Cache sizing

```
Timeline cache (top 800 tweet_ids per active user):
  100M DAU × 800 tweet_ids × 8 bytes = 640 GB
  → Redis Cluster with 10 nodes × 128 GB = 1.28 TB capacity → sufficient

Tweet content cache (recently active tweets):
  6,000 tweets/s × 86,400 s × 7 days hot = 3.6B tweets in cache window
  Cache only the top 10% most-accessed = 360M tweets × 200 bytes = 72 GB
  → Fits in a Redis cluster, LRU eviction handles the rest

Celebrity tweet cache:
  10,000 celebrity users × 200 tweet_ids × 8 bytes = 16 MB → trivial

Total estimated Redis memory: ~800 GB across the cluster
Deployment: Redis Cluster with consistent hashing, 16 shards × 64 GB
```

### 5.3 Cache invalidation for counts

```
Problem: tweet.like_count, tweet.retweet_count change constantly.
         Rewriting the full tweet object to cache on every like is expensive.

Solution: store engagement counts separately from tweet content.

  tweet:{tweet_id}:content → static fields (author, text, media) — long TTL
  tweet:{tweet_id}:counts  → {likes, retweets, replies} — short TTL (5 min)
                           → or use Redis counter: INCR tweet:{id}:likes

On cache read: fetch both and merge.
On like: INCR tweet:{tweet_id}:likes (atomic, O(1), no lock)
         Async flush to DB every 5 minutes.

This is the same pattern as URL shortener click counting.
```

### 5.4 Handling cache misses gracefully

```
Timeline cache miss (empty sorted set):
  → Trigger async rebuild (fan-out on read from DB)
  → Return partial results or empty feed with "loading" state
  → Do NOT block the response waiting for full rebuild

Tweet content cache miss:
  → Batch fetch missing tweet_ids from tweets DB (single multi-get)
  → Cache results before returning
  → Use read-through cache pattern

Social graph cache miss:
  → Query follows table (indexed on follower_id)
  → Cache result with 5-minute TTL
  → At scale: social graph is in a dedicated graph store (TAO at Meta)
```

## 6. The Celebrity Problem

### 6.1 Why it breaks fan-out on write

```
Scenario: Barack Obama (100M followers) posts a tweet.

Fan-out on write:
  Need 100M Redis ZADD operations
  At 100,000 Redis ops/second: takes 1,000 seconds = ~17 minutes
  By the time fan-out completes, tweet is stale and users have refreshed

  Memory: 100M followers × 8 bytes (tweet_id) = 800 MB for ONE tweet

  Each follower timeline update requires:
    1. ZADD timeline:{follower_id} tweetId
    2. ZREMRANGEBYRANK (trim to 800 items)
  = 200M Redis operations for one celebrity tweet
  = catastrophic for a system handling thousands of tweets/second
```

### 6.2 The hybrid solution in detail

```
Celebrity threshold: users with > 1M followers (configurable)
  Maintained by background job updating users.is_celebrity flag
  Why not real-time? Following count changes are eventual anyway

For celebrity tweets:
  Step 1: Store tweet in tweets table (same as normal user)
  Step 2: Cache tweet in user_tweets:{celeb_id} sorted set (celebrity tweet cache)
  Step 3: NO fan-out to follower timelines

For timeline reads when user follows celebrities:
  Step 1: Load pre-built timeline from timeline:{user_id} (normal user tweets)
  Step 2: SMEMBERS celebrity_followees:{user_id} → list of celebrity followee IDs
  Step 3: For each celebrity: ZREVRANGE user_tweets:{celeb_id} 0 49
  Step 4: Merge celebrity tweets into timeline by tweet_id (Snowflake sort)
  Step 5: Return merged page

Performance:
  Timeline read for user following 2 celebrities:
    1 ZREVRANGEBYSCORE (timeline cache)  = O(log N + K)
    2 ZREVRANGE calls (celebrity caches) = O(log M + 50) each
    1 merge of 3 sorted lists            = O(K log 3) ≈ O(K)
    Total: sub-millisecond Redis operations → well within 200ms SLA
```

### 6.3 Edge cases

```
User unfollows a celebrity:
  Remove from celebrity_followees:{user_id} set immediately.
  Their tweets naturally disappear from future reads.
  No need to remove from the pre-built timeline (they weren't added to it).

Celebrity posts storm (rapidly posting many tweets):
  Rate limit tweet creation per user (max 2,400/day, ~17/minute for top users).
  Celebrity tweet cache capped at 200 entries (ZREMRANGEBYRANK trim).

User BECOMES a celebrity (follower count crosses threshold):
  Background job detects crossing and flips is_celebrity = true.
  Existing follower timelines still contain their old tweets.
  Future tweets: no fan-out (hybrid handles it).
  Old fan-out entries remain in follower timelines — no cleanup needed.

User DE-lists as celebrity (follower count drops below threshold):
  Flip is_celebrity = false.
  Resume fan-out on future tweets.
  Slight inconsistency during transition window — acceptable.

Mutual celebrities (celebrity follows another celebrity):
  Both sides handle the other as a celebrity.
  No fan-out either direction; both inject each other's tweets at read time.
```

## 7. Ranking Algorithms

### 7.1 Reverse chronological vs ranked feed

```
Reverse chronological:
  Sort by tweet_id (Snowflake = time order) descending.
  Simple, predictable, no ML needed.
  Problem: if you follow 500 users, you see whoever posted most recently,
           not necessarily what's most relevant or engaging.

Ranked feed:
  Score each candidate tweet by a relevance function.
  Surface tweets with highest predicted engagement to this specific user.
  Used by Twitter, Instagram, TikTok for higher user engagement.
  Cost: significant compute per user per timeline load.
```

### 7.2 Candidate generation → ranking pipeline

```
Ranked feed is a two-stage pipeline:

Stage 1: Candidate Generation
  Pull a superset of candidate tweets (e.g. top 500 recent tweets
  from all followees, including celebrities).
  Goal: recall — don't miss good candidates.
  Must be fast: use pre-built timeline cache for normal users.

Stage 2: Ranking / Scoring
  Score each of the 500 candidates for this specific user.
  Goal: precision — surface the best 20 from the 500.
  Uses ML model; can afford more latency here.

Stage 3: Filtering
  Remove tweets the user has seen (dedup by session).
  Remove blocked users' tweets.
  Apply content policies (NSFW filtering, etc.)

Stage 4: Return top K
  Return top 20 scored, filtered candidates.
```

### 7.3 Ranking signal taxonomy

```
Static signals (tweet-level, same for all users):
  recency:         newer tweets score higher (time decay)
  like_count:      raw engagement signal
  retweet_count:   amplification signal
  reply_count:     conversation signal (can be negative — controversy)
  media_present:   tweets with images/video get higher engagement
  author_verified: slight boost for verified accounts

Personalised signals (user × tweet, different per viewer):
  follows_author:          higher weight for followed accounts
  interacted_before:       user has liked/replied to this author previously
  topic_affinity:          user frequently engages with this tweet's hashtags/topics
  author_interaction_rate: how often viewer engages with this author's content
  social_proof:            people viewer follows have liked/retweeted this tweet

Contextual signals:
  time_of_day:     tweets performing well right now (trending)
  session_context: what has the user engaged with in this session?
  device:          mobile users prefer shorter content

Negative signals (downranking):
  low_quality_indicator: high reply rate relative to likes (controversy)
  spam_signals:          mass posting, low-quality accounts
  already_seen:          dedup within session
```

### 7.4 Simplified scoring function

```java
@Service
public class TweetRankingService {

    public List<Tweet> rank(List<Tweet> candidates, long userId) {
        UserProfile profile = userProfileService.get(userId);
        List<ScoredTweet> scored = candidates.stream()
            .map(t -> new ScoredTweet(t, score(t, profile)))
            .sorted(Comparator.comparingDouble(ScoredTweet::getScore).reversed())
            .collect(Collectors.toList());

        return scored.stream().map(ScoredTweet::getTweet).collect(Collectors.toList());
    }

    private double score(Tweet tweet, UserProfile viewer) {
        double score = 0.0;

        // Recency: exponential decay with 4-hour half-life
        long ageSeconds = Instant.now().getEpochSecond() - tweet.getCreatedAt().getEpochSecond();
        double recency = Math.exp(-0.693 * ageSeconds / (4 * 3600.0)); // λ = ln2/halflife
        score += 0.3 * recency;

        // Engagement (normalised logarithmically — avoid domination by viral tweets)
        double engagementScore = Math.log1p(tweet.getLikeCount() + 2 * tweet.getRetweetCount());
        score += 0.2 * normalise(engagementScore, 0, 15); // 0-15 mapped to 0-1

        // Author affinity: prior interaction rate
        double affinity = viewer.getAuthorAffinity(tweet.getUserId()); // [0,1]
        score += 0.25 * affinity;

        // Social proof: fraction of viewer's followees who liked this tweet
        double socialProof = socialProofService.getScore(tweet.getTweetId(), viewer.getUserId());
        score += 0.15 * socialProof;

        // Media bonus
        if (tweet.hasMedia()) score += 0.05;

        // Diversity penalty: reduce score if author already has 2+ tweets in current page
        int authorCountInPage = viewer.getCurrentPageAuthorCount(tweet.getUserId());
        score -= 0.1 * Math.max(0, authorCountInPage - 1);

        return score;
    }
}

// In production: replace hand-tuned weights with a learning-to-rank model
// (LambdaMART, XGBoost, or neural ranking) trained on engagement labels.
// Features are the same signals; weights learned from data.
```

### 7.5 Ranking at scale — avoiding per-request ML inference

```
Problem: scoring 500 candidates × 9,300 requests/second = 4.65M model inferences/second
         Full ML inference per request: too slow for 200ms SLA.

Solutions:

Option A: Lightweight scoring model (< 5ms per 500 candidates)
  Use LightGBM / XGBoost: ~1-2ms for 500 predictions.
  Acceptable if model is pre-loaded in memory (not a remote call).
  Feature fetching is the bottleneck, not model inference.

Option B: Pre-scored tweet cache
  When tweet is created, compute a base score (static signals only).
  Cache static_score:{tweet_id}.
  At read time: fetch cached static score, add personalised signals (fast).
  Only personalised signals computed at read time (small delta computation).

Option C: Candidate score precomputation per user
  Background job: every 30 minutes, precompute ranked timeline for each active user.
  At read time: serve the pre-ranked list.
  Cons: stale rankings; not responsive to new tweets in the last 30 minutes.
  Used for less active users as a cost-saving measure.

Production approach: Option A for active users, Option C for low-activity users.
```

## 8. Unfollow and Delete

### 8.1 Unfollow

```
When user A unfollows user B:
  1. Delete from follows table: DELETE WHERE follower_id=A AND followee_id=B
  2. Invalidate social graph caches: DEL followees:A, celebrity_followees:A
  3. Remove B's tweets from A's timeline cache:
     → Option 1: Eager — scan timeline:A and remove B's tweet_ids
       → Expensive: O(n) scan of up to 800 items, must know which are from B
     → Option 2: Lazy — keep B's tweets in cache but filter at read time
       → Check follows at hydration time; skip tweets from unfollowed users
       → Timeline becomes stale but self-heals over time as new tweets push out old
  
  Recommended: lazy removal with tombstone filter at hydration.
  Stale tweets from unfollowed users appear briefly then are pushed out — acceptable.
```

### 8.2 Tweet deletion

```
When user deletes a tweet:
  1. Soft delete: SET tweets.deleted_at = NOW() (keep for audit/legal)
  2. Invalidate tweet cache: DEL tweet:{tweet_id}
  3. Remove from user's celebrity cache if celebrity: ZREM user_tweets:{userId} tweetId
  4. Follower timeline caches: DO NOT eagerly clean
     → Too expensive: N followers × 1 ZREM each
     → Lazy: at hydration time, if tweet is soft-deleted, skip it
     → Hard-delete from follower timelines via async cleanup job (low priority)

The hydration step is the natural filter:
  for tweetId in pageIds:
      tweet = tweetCache.get(tweetId)  // returns null if deleted
      if tweet != null and not tweet.isDeleted():
          results.add(tweet)
```

## 9. Scale and Partitioning

### 9.1 Tweets table sharding

```
Shard by user_id (hash-based):
  All tweets for a user on the same shard → efficient user timeline queries
  Shard key: user_id % NUM_SHARDS

Pros:
  "Get all tweets by user X" = single shard query
  Even distribution (user_ids are random Snowflake IDs)

Cons:
  Hot shards for celebrity users (solved by caching — DB rarely hit directly)
  Cross-shard queries for home timeline (solved by feed cache — also rarely hits DB)

Shard count: start with 16, expand to 256 as data grows
  Use consistent hashing via middleware (Vitess for MySQL, Citus for PostgreSQL)
```

### 9.2 Social graph storage at scale

```
Follows table at scale:
  500M users × 500 follows average = 250B edges
  Relational: viable for moderate scale (< 1B edges)
  Beyond: dedicated graph stores

Graph store options:
  TAO (Meta): distributed key-value store for social graph, custom
  Neo4j:      general-purpose graph DB, handles complex traversals
  Amazon Neptune: managed graph DB (Gremlin/SPARQL)
  Redis Sets: for hot-path lookups (followees:{user_id} in memory)

For this design:
  Follows stored in MySQL/PostgreSQL sharded by follower_id
  Hot graph (active users) mirrored in Redis Sets
  Social graph service abstracts the storage behind a clean API
```

### 9.3 Fan-out service scaling

```
Fan-out is the most compute-intensive part of the write path.

Architecture: Kafka → Fan-out Workers (horizontally scalable)

Kafka topic: tweet-created
  Partitioned by user_id (ensures tweets from one user fan-out in order)
  32 partitions → 32 parallel fan-out workers initially

Each worker:
  Consumes tweet-created events
  Fetches follower list (from DB or cache)
  Writes to Redis timelines in batches (Redis pipeline: 500 ZADDs per round-trip)

Throughput:
  12,000 tweets/second × 500 average followers = 6M Redis ops/second
  Redis: 1M simple ops/second per instance → need 6 Redis instances minimum
  With clustering (10 shards): 10M ops/second capacity → comfortable headroom

Priority queuing:
  High-priority topic: tweets from users with < 10K followers (fast fan-out)
  Low-priority topic: tweets from users with 10K-1M followers (slower fan-out ok)
  Celebrity topic: routed to celebrity-tweet-cache service (no fan-out)
```

## 10. Summary Architecture

```
Write path:
  Client → API Gateway → Tweet Service → (save to Cassandra/MySQL)
                                       → (publish to Kafka tweet-created)
                      Kafka → Fan-out Workers → Redis timeline sorted sets
                             → Celebrity Cache Service → user_tweets:{celeb_id}

Read path:
  Client → API Gateway → Timeline Service
    → Redis: ZREVRANGEBYSCORE timeline:{user_id} (fan-out-on-write results)
    → Redis: ZREVRANGE user_tweets:{celeb_id} for each celebrity followed
    → Merge sorted lists (by Snowflake ID)
    → Hydrate: batch fetch tweet objects (Redis tweet cache → Cassandra fallback)
    → Rank (optional): LightGBM scoring with personalised signals
    → Filter: deleted, blocked, already-seen
    → Return paginated response

Fan-out decision:
  Normal user (< 1M followers):  fan-out on write → Redis timelines updated
  Celebrity (≥ 1M followers):    fan-out on read → injected at timeline read time
```

## 11. Interview Questions

**Q: Explain the fan-out on write vs fan-out on read trade-off, and why neither alone works at Twitter's scale.**

A: Fan-out on write pushes tweet_ids to every follower's pre-built timeline cache at write time. Reads are then O(1) — just fetch the pre-built list. This works beautifully for normal users with a few hundred followers, but fails for celebrities: one Obama tweet requires 100 million Redis writes, takes 17 minutes to complete at realistic throughput, and consumes 800 MB of memory for tweet_id copies alone. Fan-out on read computes the timeline at read time by merging recent tweets from all followees. This eliminates the celebrity write explosion but creates a different problem: a user following 500 people requires merging 500 sources on every timeline load, and at 50,000 timeline reads per second, that's 25 million database queries per second — equally unworkable. The hybrid model resolves this by fan-outing on write for normal users (cheap) and deferring celebrity tweet injection to read time (where it's a small, bounded merge of a few celebrity caches). The vast majority of followed accounts are normal users, so fan-out covers most tweets, while celebrities — a tiny fraction of accounts — are handled specially. This gives O(1) reads for the common case and bounded read-time work for the celebrity injection.

---

**Q: How would you handle a user with 100 million followers posting a tweet, step by step?**

A: When the tweet is posted, the Tweet Service stores it in the tweets table and publishes a `TweetCreatedEvent` to Kafka. The fan-out worker consumes the event, looks up the user's profile, sees `is_celebrity = true`, and immediately returns — no fan-out to follower timelines occurs. Instead, the tweet is written to `user_tweets:{celeb_id}` — a Redis sorted set capped at the 200 most recent tweet IDs. This is one write regardless of follower count. When any follower loads their home timeline, the Timeline Service loads their pre-built timeline from `timeline:{user_id}`, then fetches the IDs of celebrities they follow from `celebrity_followees:{user_id}`. For each celebrity, it fetches the top 50 recent tweet IDs from `user_tweets:{celeb_id}`. These two sorted lists are merged by Snowflake ID (time order). The merge is cheap — it's a small fixed-size operation regardless of total follower count. The follower sees the celebrity's tweet within milliseconds of posting because the tweet is immediately in `user_tweets:{celeb_id}` and any subsequent timeline load injects it.

---

**Q: How does cursor-based pagination differ from offset pagination for timelines, and why does it matter?**

A: Offset pagination (`LIMIT 20 OFFSET 40`) asks for rows 40-60 by counting from the start every time. It has two serious problems for feeds: first, it's O(offset) at the database — scanning and discarding 40 rows to get to position 40. At page 50 that's 1,000 rows scanned per request. Second, it produces inconsistent results: if new tweets are posted between page 1 and page 2 loads, all tweets shift positions, and the user sees duplicates (first tweet of page 2 was last tweet of page 1) or misses tweets. Cursor-based pagination uses a stable identifier — the tweet_id of the last seen tweet — as the cursor. The query is "give me tweet_ids less than cursor X, top 20." In Redis: `ZREVRANGEBYSCORE timeline:{user_id} cursor -inf LIMIT 0 20` — this is O(log N + 20) regardless of how deep in the timeline the user is. New tweets never shift old tweets' positions because tweet_ids are monotonically decreasing going back in time. The cursor encodes position in the logical timeline, not a row count.

---

**Q: What ranking signals would you use for an algorithmic feed, and how do you balance freshness against relevance?**

A: The most important signals split into static (same for all viewers) and personalised (per viewer-tweet pair). Static signals include recency with exponential decay (a 4-hour half-life works well for Twitter's pace), raw engagement counts (likes, retweets — log-normalised to prevent viral tweets from dominating), and media presence. Personalised signals are more powerful: author affinity (has this viewer liked or replied to this author before?), topic affinity (does this tweet's topics match the viewer's engagement history?), and social proof (have people the viewer follows engaged with this tweet?). Balancing freshness against relevance is the core tension: a heavily downweighted recency signal produces a feed full of month-old viral content; a heavily downweighted relevance signal produces a feed of recent tweets from low-engagement accounts the user rarely reads. The exponential decay function with a tunable half-life parameter lets you adjust this balance. A 4-hour half-life means a tweet that's 4 hours old scores half as high on recency as a new tweet, but a tweet with 10x more engagement can still outrank a newer tweet. In production, these weights are not hand-tuned but learned from data using a learning-to-rank model trained on engagement labels — which tweets users actually clicked, liked, or spent time reading.

---

**Q: How would you ensure the timeline is consistent after a user unfollows someone or deletes a tweet?**

A: Both cases use the same philosophy: lazy invalidation rather than eager cleanup, because eager cleanup of fan-out data is too expensive. For unfollow: when user A unfollows B, we immediately delete the follow edge from the database and invalidate the social graph cache entries. B's tweets may still exist in A's Redis timeline sorted set, but on the next timeline load, during the hydration step — where we fetch full tweet objects for each tweet_id in the page — we check whether the tweet's author is still followed. If not, we skip that tweet. Over time, B's tweets age out of the top of the timeline naturally. For tweet deletion: we soft-delete the tweet in the database and invalidate `tweet:{tweet_id}` in the tweet cache. The tweet_id may still be in many followers' timeline sorted sets, but the next hydration for any of those followers returns null for that tweet_id and it's silently skipped. An asynchronous cleanup job runs periodically to remove soft-deleted tweet_ids from timeline caches, but this is a best-effort background task, not on the critical path. The key insight is that the hydration step — where tweet_ids are resolved to tweet objects — is the natural filter point for all stale or deleted entries.

## 12. Key Takeaways

- **Fan-out hybrid**: fan-out on write for normal users (fast reads), fan-out on read for celebrities (tweet injected at read time) — the threshold is typically ~1M followers
- **Celebrity tweet cache**: `user_tweets:{celeb_id}` Redis sorted set — one write per tweet regardless of follower count; injected into follower timelines at read time
- **Timeline = Redis sorted set**: `ZADD timeline:{user_id} tweetId score=tweetId` — Snowflake IDs are both timestamps and sort keys; `ZREVRANGEBYSCORE` with cursor for O(log N) pagination
- **Cache layers**: timeline cache (800 tweet_ids/user), tweet content cache (24h TTL, LRU), celebrity tweet cache, social graph cache (5 min TTL), user profile cache (1h TTL)
- **Engagement counts in Redis**: `INCR tweet:{id}:likes` for hot-path like counts — atomic, millions/second; async flush to DB; same pattern as URL shortener click counting
- **Ranking pipeline**: candidate generation (pull ~500 from timeline cache) → scoring (recency decay + engagement + author affinity + social proof) → filtering (deleted, blocked, seen) → return top 20
- **Snowflake IDs are sort keys**: tweet_id ordering = time ordering — no `ORDER BY timestamp` needed; merge sorted lists by tweet_id to merge timelines
- **Lazy invalidation for unfollow/delete**: cheaper than eager cleanup of fan-out; hydration step is the natural filter point
- **Fan-out throughput**: Kafka partitioned by user_id → parallel workers → Redis pipeline batching (500 ZADDs per round-trip) — handles 6M Redis ops/second across cluster
- **Cursor pagination over offset pagination**: stable identity-based cursor vs position-based offset — prevents duplicates/gaps as new tweets arrive, O(log N) vs O(offset) at DB
