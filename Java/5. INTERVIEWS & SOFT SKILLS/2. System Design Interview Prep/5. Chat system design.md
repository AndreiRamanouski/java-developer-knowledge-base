# Chat System Design

## Overview

A chat system delivers messages between users in near-real-time. At scale it combines several hard problems simultaneously: maintaining millions of persistent connections, guaranteeing message delivery across unreliable networks, tracking online presence for millions of users, storing billions of messages efficiently, and routing notifications to offline devices. WhatsApp serves 100 billion messages per day; Slack handles 1.5 billion actions per week.

**The core challenge:**

```
HTTP is request-response: the client must ask before the server can answer.
Chat is server-push: the server must deliver a message the moment it arrives,
to a client that did not ask.

Three approaches — worst to best:
  Short polling:   client asks "any messages?" every N seconds
                   → N-second latency; massive wasted bandwidth; scales poorly
  Long polling:    client asks and server holds the connection open until a message arrives
                   → better latency; half-duplex; still one round trip per message
  WebSocket:       persistent full-duplex TCP connection; server pushes at any time
                   → best latency; efficient; standard for modern chat

This design uses WebSocket for real-time delivery with long polling as fallback.
```

## 1. Requirements Gathering

### 1.1 Clarifying questions

```
1. "1-on-1 chat only, or group chat too? What is the maximum group size?"
   → Both: 1-on-1 and group chat up to 500 members.

2. "Does the system need to support multiple devices per user?"
   → Yes: messages must sync across all of a user's logged-in devices.

3. "What media types are supported — text only, or images/files/video?"
   → Text + images + file attachments. Video calling is out of scope.

4. "What are the message delivery guarantees — at-most-once, at-least-once, exactly-once?"
   → At-least-once delivery; deduplication on the client prevents duplicates showing.

5. "Do we need message history / persistence? For how long?"
   → Indefinite: users scroll back to any historical message.

6. "Do we need read receipts and online presence (last seen, typing indicators)?"
   → Yes to all three.

7. "What scale: DAU, messages per day?"
   → 50M DAU; 100 billion messages per day.

8. "What is the acceptable end-to-end latency for message delivery?"
   → < 500ms for online recipients; push notification within 5 seconds for offline.
```

### 1.2 Confirmed requirements

```
Functional:
  ✓ 1-on-1 messaging
  ✓ Group messaging (up to 500 members)
  ✓ Message persistence (indefinite history)
  ✓ Multi-device sync (message appears on all logged-in devices)
  ✓ Read receipts (delivered / read status)
  ✓ Online presence (online/offline, last seen)
  ✓ Typing indicators
  ✓ Push notifications for offline users
  ✓ Media attachments (images, files)

Non-functional:
  ✓ 50M DAU; 100B messages/day (~1.16M messages/second)
  ✓ End-to-end delivery < 500ms (online recipients)
  ✓ Push notification within 5 seconds (offline)
  ✓ Message ordering guaranteed within a conversation
  ✓ Availability: 99.99% for message delivery path
  ✓ Durability: zero message loss once acknowledged by server

Out of scope:
  ✗ End-to-end encryption (architectural consideration, not in scope today)
  ✗ Video/audio calling
  ✗ Message search full-text indexing
```

### 1.3 Scale estimation

```
Messages:
  100B messages/day = 100B / 86,400 ≈ 1.16M messages/second average
  Peak (3× average): ~3.5M messages/second

Storage per message:
  message_id:   8 bytes  (Snowflake)
  sender_id:    8 bytes
  conversation: 8 bytes
  content:      100 bytes (average text message)
  timestamp:    8 bytes
  type/status:  2 bytes
  Total:        ~134 bytes/message

Storage growth:
  100B × 134 bytes = 13.4 TB/day
  Compressed (~5:1): ~2.7 TB/day
  5-year retention: ~4.9 PB (compressed) → tiered storage (hot/warm/cold)

WebSocket connections:
  50M DAU; assume 20% concurrent = 10M active connections
  10M connections distributed across chat servers
  Each chat server handles ~50,000 connections (10M / 200 servers)
  Memory per connection: ~50 KB (kernel + app state) → 50K × 50KB = 2.5 GB/server

Presence heartbeats:
  10M online users × 1 heartbeat/30s = 333,333 heartbeat events/second
```

## 2. Real-Time Transport: WebSocket vs Long Polling

### 2.1 WebSocket — the right choice for chat

```
WebSocket protocol:
  1. Client sends HTTP Upgrade request to ws:// or wss:// endpoint
  2. Server responds 101 Switching Protocols
  3. Connection upgrades to WebSocket: persistent, full-duplex TCP tunnel
  4. Either side can send frames at any time without the other asking

Why WebSocket wins for chat:
  Full-duplex:  server pushes messages to client; client sends messages to server
               on the same connection simultaneously
  Low overhead: after handshake, each WebSocket frame has only 2-10 byte header
               vs HTTP which repeats headers (200-800 bytes) on every request
  Low latency:  server pushes the moment a message arrives; no polling interval
  Connection state: server knows which users are connected without separate tracking

WebSocket connection lifecycle:
  CONNECT → OPEN → [send/receive frames] → CLOSE
  Frames: text, binary, ping, pong, close
  Ping/pong: keep-alive and connection health check (every 25-30 seconds)
```

```java
// Spring WebSocket endpoint
@Configuration
@EnableWebSocket
public class WebSocketConfig implements WebSocketConfigurer {

    @Override
    public void registerWebSocketHandlers(WebSocketHandlerRegistry registry) {
        registry.addHandler(chatWebSocketHandler(), "/ws/chat")
                .addInterceptors(new JwtHandshakeInterceptor()) // auth at upgrade time
                .setAllowedOrigins("*");
    }
}

@Component
public class ChatWebSocketHandler extends TextWebSocketHandler {
    // Map: userId → Set<WebSocketSession> (user may have multiple devices)
    private final ConcurrentHashMap<Long, CopyOnWriteArraySet<WebSocketSession>> sessions
        = new ConcurrentHashMap<>();

    @Override
    public void afterConnectionEstablished(WebSocketSession session) {
        long userId = getUserId(session); // extracted from JWT during handshake
        sessions.computeIfAbsent(userId, k -> new CopyOnWriteArraySet<>()).add(session);
        presenceService.setOnline(userId, session.getId());
        log.info("User {} connected. Session: {}", userId, session.getId());
    }

    @Override
    public void afterConnectionClosed(WebSocketSession session, CloseStatus status) {
        long userId = getUserId(session);
        Set<WebSocketSession> userSessions = sessions.get(userId);
        if (userSessions != null) {
            userSessions.remove(session);
            if (userSessions.isEmpty()) {
                presenceService.setOffline(userId);
            }
        }
    }

    @Override
    protected void handleTextMessage(WebSocketSession session, TextMessage message) {
        InboundMessage msg = objectMapper.readValue(message.getPayload(), InboundMessage.class);
        messageService.send(msg, getUserId(session));
    }

    // Called by message delivery service to push to recipient
    public boolean pushToUser(long userId, OutboundMessage message) {
        Set<WebSocketSession> userSessions = sessions.get(userId);
        if (userSessions == null || userSessions.isEmpty()) return false; // offline

        String payload = objectMapper.writeValueAsString(message);
        for (WebSocketSession s : userSessions) {
            if (s.isOpen()) {
                s.sendMessage(new TextMessage(payload)); // push to each device
            }
        }
        return true;
    }
}
```

### 2.2 Long polling — fallback for restricted environments

```
Long polling flow:
  1. Client sends GET /messages?after=lastSeenId
  2. Server holds request open (up to 30 seconds)
  3. When a message arrives: server responds with the message, closes connection
  4. Client immediately opens a new long-poll request (repeat)

Pros:
  Works everywhere (no WebSocket support needed — older browsers, firewalls)
  Simple to implement with standard HTTP
  Compatible with HTTP/1.1 (no upgrade needed)

Cons:
  Half-duplex: client sends messages via separate POST, receives via long poll
  One round-trip latency per message (connection setup overhead)
  Server holds open connections (similar memory cost to WebSocket)
  TCP connection torn down and re-established after each message

When to use:
  Corporate environments with proxies/firewalls blocking WebSocket upgrades
  IoT devices with limited HTTP libraries
  Graceful degradation fallback when WebSocket fails to establish
```

### 2.3 Server-Sent Events (SSE) — half-duplex middle ground

```
SSE: server pushes events to client over a single HTTP connection.
  Client receives: EventSource API (W3C standard)
  Client sends: separate HTTP POST requests (not on SSE connection)

  GET /events → persistent text/event-stream response
  Server sends: data: {"type":"message","payload":...}\n\n

Pros: simpler than WebSocket; automatic reconnection built in; HTTP/2 multiplexing
Cons: unidirectional (server → client only); client sends via separate connection

Suitable for: notification streams, live feeds (not ideal for chat's bidirectionality)
```

### 2.4 Comparison table

```
                WebSocket         Long Polling      SSE
Direction:      Full-duplex       Half-duplex       Server→Client only
Latency:        ~5-10ms           ~50-200ms         ~10-30ms
Protocol:       WS/WSS            HTTP/1.1+         HTTP/1.1+ (text/event-stream)
Reconnection:   Manual            Automatic (loop)  Automatic (built-in)
Binary support: Yes               Base64            No (text only)
Firewall risk:  Medium (port 80)  None              None
Connection cost: Persistent TCP   Per-message TCP   Persistent TCP (recv only)

Choice: WebSocket primary, long polling fallback.
```

## 3. Message Flow Architecture

### 3.1 High-level components

```
┌───────────────────────────────────────────────────────────────────────────┐
│                         Client (Mobile / Web)                              │
└──────────────────────────┬─────────────────────────────────────────────────┘
                            │ WebSocket (wss://)
                ┌───────────▼────────────┐
                │      API Gateway        │
                │  (TLS, Auth, Routing)   │
                └───────────┬────────────┘
                            │
           ┌────────────────▼───────────────────────┐
           │            Chat Service Cluster          │
           │  chat-server-1  chat-server-2  ...N      │
           │  (WebSocket sessions for active users)   │
           └────────────────┬───────────────────────┘
                            │
             ┌──────────────┼──────────────────────┐
             │              │                      │
   ┌─────────▼──────┐  ┌────▼────────┐  ┌─────────▼──────┐
   │  Message Store  │  │   Kafka     │  │  Presence Store │
   │  (Cassandra)    │  │  (routing)  │  │  (Redis)        │
   └─────────────────┘  └────┬────────┘  └────────────────┘
                             │
              ┌──────────────┼───────────────────┐
              │              │                   │
   ┌──────────▼──────┐  ┌────▼────────┐  ┌──────▼────────┐
   │  Delivery       │  │  Push Notif │  │  Notification  │
   │  Service        │  │  Service    │  │  Store         │
   │  (WebSocket     │  │  (APNs/FCM) │  │  (Redis)       │
   │   push)         │  └─────────────┘  └────────────────┘
   └─────────────────┘
```

### 3.2 Message send flow (sender → recipient)

```
Step 1: Sender sends message via WebSocket
  Client sends: {type:"message", conversationId:123, content:"Hello", clientMsgId:"uuid-abc"}
  clientMsgId: client-generated UUID for idempotency / dedup

Step 2: Chat server receives and persists
  a. Generate server-side message_id (Snowflake — globally unique, time-sortable)
  b. Write to Cassandra message store (durable)
  c. Ack to sender: {type:"ack", clientMsgId:"uuid-abc", serverMsgId:12345, status:"sent"}
     → sender knows message is durable; UI shows single tick

Step 3: Route message to recipients
  Chat server publishes to Kafka topic "messages" with conversation_id as partition key
  (ensures ordering within a conversation)

Step 4: Delivery service consumes from Kafka
  For each recipient in the conversation:
    a. Look up which chat server holds recipient's WebSocket session
       (from Redis routing table: userId → chatServerId)
    b. If recipient is connected: push directly to their chat server via gRPC
    c. Chat server pushes to recipient's WebSocket session(s)
    d. Recipient's device acks receipt → update delivery status

Step 5: Offline delivery
  If recipient has no active WebSocket session:
    a. Store message in notification queue
    b. Push notification via APNs (iOS) or FCM (Android)
    c. When user comes online: fetch undelivered messages from Cassandra
```

### 3.3 Cross-server message routing

```
Problem: sender connected to server-A; recipient connected to server-B.
Server-A cannot directly push to server-B's WebSocket sessions.

Solution: routing table in Redis
  On connect:    SET routing:{userId} serverA  EX 300 (5-min TTL, refreshed by heartbeat)
  On disconnect: DEL routing:{userId}

  Server-A receives message for user Bob:
    1. GET routing:Bob → "server-B"
    2. gRPC call to server-B: DeliverMessage(userId=Bob, message=...)
    3. Server-B finds Bob's WebSocket sessions → push

Alternative: Kafka fan-out
  Every chat server subscribes to Kafka.
  Chat server only processes messages where routing:{userId} == self.
  Simpler but all servers see all messages (bandwidth overhead).

Production (Slack/Discord approach):
  Kafka for persistence and audit log.
  gRPC for direct server-to-server delivery (lower latency than Kafka consumer lag).
```

## 4. 1-on-1 vs Group Chat

### 4.1 1-on-1 messaging

```
Conversation between two users: user A and user B.
conversation_id = deterministic hash of sorted(user_A_id, user_B_id)
  → same pair always maps to same conversation_id
  → no need to look up if conversation exists; can compute it

Delivery:
  Simple: recipient is exactly one user → one routing lookup → push.
  If online: WebSocket push.
  If offline: push notification + store in undelivered queue.

Read receipts:
  When B reads the message: send receipt event to server
  Server updates delivery_status = READ for that message
  Server pushes receipt event to A: {type:"read", messageId:123, readBy:B, readAt:...}
  A's UI updates: single tick → double tick → blue double tick (WhatsApp style)
```

### 4.2 Group chat — fan-out at scale

```
Group with N members: one sent message must reach N-1 recipients.

For small groups (N ≤ 100):
  Fan-out on delivery: for each member, look up routing and push.
  At N=100 and 1M messages/group/day:
    1M × 100 = 100M routing lookups/day per group → manageable.

For large groups (N up to 500):
  Fan-out per message = 499 push operations.
  At 1,000 active large groups, each with 100 msg/hour:
    1,000 × 100 × 499 = 49.9M push ops/hour → significant but feasible with batching.

Fan-out strategies:

Option A: Fan-out on write (push to each member's queue at send time)
  Server generates N copies (or references) in delivery queues.
  Pro: O(1) read for each recipient.
  Con: expensive writes for large groups; amplification on viral groups.

Option B: Fan-out on read (each recipient fetches from shared conversation log)
  One write per message to the conversation log.
  Each recipient reads from the log independently.
  Pro: O(1) write; scales to massive groups.
  Con: each recipient must track their own read position; slightly higher read latency.

Hybrid (used by WhatsApp, Signal for large groups):
  Small groups (≤ 100):  fan-out on write per recipient queue.
  Large groups (> 100):  shared conversation log; each recipient reads independently.
                         Push notification still sent individually (N pushes) but payload
                         is just a wakeup signal, not the full message.
```

## 5. Message Storage

### 5.1 Why Cassandra for messages

```
Requirements for message storage:
  Write-heavy:     100B writes/day = 1.16M writes/second
  Read pattern:    range scan by conversation + descending time (scroll back)
  Immutable:       messages never updated (only delivery status)
  Scale:           petabytes; horizontal scaling required
  Latency:         write < 1ms; read < 5ms

Cassandra is ideal:
  Log-structured merge tree (LSM): optimised for high write throughput
  Partition key = conversation_id: all messages in a conversation on same partition
  Clustering key = message_id DESC: messages stored in reverse time order
  Linear horizontal scaling: add nodes to increase capacity
  Tunable consistency: writes at QUORUM; reads at LOCAL_ONE for timeline
  No single point of failure: multi-node, multi-datacenter replication

Why NOT PostgreSQL for primary message store:
  MVCC write amplification at 1M writes/second
  B-tree index updates on every write
  Single primary bottleneck (without complex sharding)
  PostgreSQL CAN work for smaller scale (< 10M msg/day)
```

### 5.2 Message schema

```sql
-- Cassandra message table
CREATE TABLE messages (
    conversation_id  BIGINT,
    message_id       BIGINT,         -- Snowflake: time-sortable, globally unique
    sender_id        BIGINT,
    message_type     TEXT,           -- 'text' | 'image' | 'file' | 'system'
    content          TEXT,           -- text content or media metadata JSON
    reply_to_id      BIGINT,         -- NULL if not a reply
    created_at       TIMESTAMP,
    PRIMARY KEY (conversation_id, message_id)
) WITH CLUSTERING ORDER BY (message_id DESC)   -- most recent first
  AND compaction = {'class': 'TimeWindowCompactionStrategy',
                    'compaction_window_unit': 'DAYS',
                    'compaction_window_size': 1}  -- compact daily windows
  AND default_time_to_live = 0;  -- indefinite (TTL set by tier for old messages)

-- Delivery status (separate table — high update rate, don't collide with message reads)
CREATE TABLE message_delivery (
    message_id       BIGINT,
    recipient_id     BIGINT,
    status           TEXT,      -- 'sent' | 'delivered' | 'read'
    updated_at       TIMESTAMP,
    PRIMARY KEY (message_id, recipient_id)
);

-- User's last read position per conversation (for unread counts)
CREATE TABLE user_conversation_state (
    user_id          BIGINT,
    conversation_id  BIGINT,
    last_read_msg_id BIGINT,
    last_read_at     TIMESTAMP,
    PRIMARY KEY (user_id, conversation_id)
);
```

### 5.3 Message ID design (Snowflake)

```
Why message ordering matters:
  Messages in a conversation must be ordered consistently across all devices.
  "Server receives message B before A due to network jitter" must not reorder them.

Snowflake ID properties:
  Globally unique:      no collisions across all servers
  Time-sortable:        higher ID = later message (within ~1ms precision)
  Per-server sequential: messages from one server never duplicate or skip
  64-bit integer:       fits in BIGINT; efficient index key

Structure:
  [1 sign bit][41 bits epoch ms][10 bits server ID][12 bits sequence]

  41 bits epoch: 2^41 ms from custom epoch ≈ 69 years
  10 bits server: 1024 chat servers
  12 bits seq: 4096 messages per millisecond per server

Result: at 1.16M messages/second, max concurrent servers needed:
  1.16M / 4096 = ~283 servers → well within 1024 server ID capacity.

Cassandra clustering order on message_id DESC:
  Since Snowflake IDs are time-sortable, ORDER BY message_id DESC = ORDER BY time DESC.
  No separate timestamp column needed for ordering.
```

### 5.4 Tiered storage for message history

```
Hot tier (0-30 days): Cassandra — in-memory or SSD
  Accessed frequently (users scrolling recent history)
  Full query capability, sub-millisecond reads

Warm tier (30-365 days): Cassandra with compaction + compression
  Accessed occasionally (searching old conversations)
  TimeWindowCompactionStrategy compacts by day → efficient range reads
  Snappy compression: ~3:1 ratio for message text

Cold tier (1+ years): object store (S3 + Parquet / columnar format)
  Accessed rarely (user exports, compliance, legal hold)
  Background migration job moves old partitions to S3
  Query via Athena/Spark for analytics; direct S3 download for user exports

Migration trigger:
  @Scheduled: nightly job moves messages older than 30 days from Cassandra
  to compressed files in S3, then deletes from Cassandra
  On retrieval of old messages: fetch from S3, cache temporarily in Redis
```

## 6. Online Presence

### 6.1 Presence architecture

```
Presence states:
  Online:       user has active WebSocket connection
  Away:         connected but idle > 5 minutes
  Offline:      no active connection
  Last seen:    timestamp of last known presence event

Scale challenge:
  50M DAU × 1 heartbeat/30s = ~1.67M events/second
  Each user's contacts may want to know their status
  At 50 contacts per user × 50M users: 2.5B presence subscriptions

Solution: fan-out presence updates lazily
  Do NOT push presence updates to all contacts of a user immediately.
  Instead: contacts pull presence status when they open a conversation.
  Exception: push to contacts who are currently viewing a conversation with the user.
```

### 6.2 Presence storage and heartbeat

```java
@Service
public class PresenceService {
    // Redis key: presence:{userId}
    // Value: JSON {"status":"online","lastSeen":timestamp,"serverId":"chat-7"}
    // TTL: 35 seconds (heartbeat every 30s + 5s grace)

    public void setOnline(long userId, String serverId) {
        PresenceState state = PresenceState.online(userId, serverId, Instant.now());
        redis.set("presence:" + userId,
                  objectMapper.writeValueAsString(state),
                  35, TimeUnit.SECONDS); // auto-expires if heartbeat stops

        // Notify active conversation partners (push presence event via Kafka)
        presenceEventPublisher.publish(PresenceEvent.online(userId));
    }

    public void setOffline(long userId) {
        Instant lastSeen = Instant.now();
        PresenceState state = PresenceState.offline(userId, lastSeen);
        // Keep last-seen for 7 days (display "last seen 2 days ago")
        redis.set("presence:" + userId,
                  objectMapper.writeValueAsString(state),
                  7, TimeUnit.DAYS);

        presenceEventPublisher.publish(PresenceEvent.offline(userId, lastSeen));
    }

    // Heartbeat: called by WebSocket ping handler every 30 seconds
    public void heartbeat(long userId, String serverId) {
        redis.expire("presence:" + userId, 35); // refresh TTL
        // If TTL had nearly expired (user was briefly disconnected): re-announce online
    }

    public PresenceState getPresence(long userId) {
        String json = redis.get("presence:" + userId);
        if (json == null) return PresenceState.neverSeen(userId);
        return objectMapper.readValue(json, PresenceState.class);
    }

    // Bulk presence for a list of users (e.g. all members of a group chat)
    public Map<Long, PresenceState> getBulkPresence(List<Long> userIds) {
        List<String> keys = userIds.stream()
            .map(id -> "presence:" + id).collect(Collectors.toList());
        List<String> values = redis.mget(keys); // O(n) single round-trip
        // ... map back to PresenceState objects
    }
}
```

### 6.3 Heartbeat on the WebSocket connection

```java
// WebSocket ping/pong: detect dead connections AND refresh presence TTL
@Component
public class HeartbeatManager {
    private static final Duration PING_INTERVAL = Duration.ofSeconds(25);
    private static final Duration PONG_TIMEOUT  = Duration.ofSeconds(10);

    @Scheduled(fixedDelay = 25_000)
    public void pingAllSessions() {
        sessionRegistry.getAllSessions().forEach(session -> {
            if (!session.isOpen()) {
                sessionRegistry.remove(session);
                return;
            }
            // Send WebSocket ping frame
            session.sendMessage(new PingMessage());
            // Schedule timeout: if pong not received within 10s, close connection
            scheduler.schedule(
                () -> checkPongReceived(session),
                PONG_TIMEOUT
            );
        });
    }

    public void onPongReceived(WebSocketSession session) {
        long userId = getUserId(session);
        presenceService.heartbeat(userId, serverId);
        session.setAttribute("lastPong", Instant.now());
    }

    private void checkPongReceived(WebSocketSession session) {
        Instant lastPong = (Instant) session.getAttribute("lastPong");
        if (lastPong == null || lastPong.isBefore(Instant.now().minus(PONG_TIMEOUT))) {
            session.close(CloseStatus.SESSION_NOT_RELIABLE);
            // afterConnectionClosed will trigger presence.setOffline()
        }
    }
}
```

### 6.4 Typing indicators

```
Typing indicators are ephemeral — they do not need to be persisted.
Delivery: via WebSocket only (skip if recipient is offline).

Protocol:
  User A starts typing in conversation with B:
    Client sends: {type:"typing_start", conversationId:123}
    Server pushes to B: {type:"typing", conversationId:123, userId:A}

  User A stops typing (or sends message):
    Client sends: {type:"typing_stop", conversationId:123}
    Server pushes to B: {type:"typing_stop", conversationId:123, userId:A}

  Auto-stop: server auto-sends typing_stop if no typing_start received
             for > 5 seconds (client may have crashed without sending stop).

Redis TTL approach:
  On typing_start: SET typing:{conversationId}:{userId} 1 EX 5
  Background: if key expires → synthesise typing_stop event
  On typing_stop: DEL typing:{conversationId}:{userId}

For group chat (N members):
  Fan-out typing events to all online members of the group.
  At 500-member group: push to up to 499 WebSocket sessions.
  Throttle: only forward typing_start at most once every 2 seconds per user per conversation.
```

## 7. Message Delivery Guarantees

### 7.1 The delivery guarantee problem

```
The network is unreliable. Any of these can happen:

  Client sends message → server never receives it (network drop before ack)
  Server persists message → Kafka publish fails (message lost before delivery)
  Kafka delivers to consumer → consumer crashes before acking (reprocessed)
  Server pushes to recipient → recipient's WebSocket buffer drops it (TCP issue)
  Server pushes to recipient → recipient receives but ACK lost (server re-sends)

Guarantee levels:
  At-most-once:   message sent once; if lost, not retried. Simple but lossy.
  At-least-once:  message retried until ACK received. Possible duplicates.
  Exactly-once:   message delivered exactly once. Complex; requires coordination.

Chat systems use at-least-once + client-side deduplication:
  Server retries until ACK; client deduplicates by message_id.
  "Exactly once" behaviour from the user's perspective at low complexity cost.
```

### 7.2 Sender-side delivery guarantee

```java
@Service
public class MessageService {

    public SendResult send(SendMessageRequest req, long senderId) {
        // 1. Idempotency check: has this clientMsgId been processed before?
        String idempKey = "idempotent:msg:" + req.getClientMsgId();
        String existingMsgId = redis.get(idempKey);
        if (existingMsgId != null) {
            // Already processed: return existing server message_id (idempotent)
            return SendResult.alreadySent(Long.parseLong(existingMsgId));
        }

        // 2. Generate Snowflake message_id
        long serverMsgId = snowflakeGenerator.nextId();

        // 3. Persist to Cassandra (durable BEFORE acking sender)
        Message message = Message.builder()
            .messageId(serverMsgId)
            .conversationId(req.getConversationId())
            .senderId(senderId)
            .content(req.getContent())
            .createdAt(Instant.now())
            .build();
        messageRepository.save(message); // write to Cassandra at QUORUM

        // 4. Record idempotency key (prevent duplicate processing on retry)
        redis.set(idempKey, String.valueOf(serverMsgId), 24, TimeUnit.HOURS);

        // 5. Ack sender IMMEDIATELY (message is durable)
        // Ack sent via WebSocket: {type:"ack", clientMsgId:..., serverMsgId:...}

        // 6. Publish to Kafka for async delivery to recipients
        kafkaTemplate.send("messages",
                           String.valueOf(req.getConversationId()),
                           MessageEvent.from(message));

        return SendResult.sent(serverMsgId);
    }
}
```

### 7.3 Recipient-side delivery guarantee

```java
@KafkaListener(topics = "messages", concurrency = "16")
public class MessageDeliveryConsumer {

    public void onMessage(MessageEvent event, Acknowledgment ack) {
        List<Long> recipients = getRecipients(event);

        for (long recipientId : recipients) {
            boolean delivered = attemptDelivery(recipientId, event);

            if (!delivered) {
                // Recipient offline: enqueue for notification + future delivery
                offlineDeliveryQueue.enqueue(recipientId, event);
                notificationService.sendPush(recipientId, event);
            }
        }

        ack.acknowledge(); // commit Kafka offset AFTER all delivery attempts
        // at-least-once: if this crashes before ack, message is re-delivered
    }

    private boolean attemptDelivery(long recipientId, MessageEvent event) {
        // Check routing table: which chat server holds this user's session?
        String serverId = redis.get("routing:" + recipientId);
        if (serverId == null) return false; // offline

        // gRPC push to the correct chat server
        try {
            DeliverResponse resp = chatServerClient.deliver(serverId,
                DeliverRequest.of(recipientId, event));
            if (resp.isSuccess()) {
                updateDeliveryStatus(event.getMessageId(), recipientId, "delivered");
                return true;
            }
        } catch (StatusRuntimeException e) {
            // gRPC failed: server may have crashed since routing entry was written
            redis.delete("routing:" + recipientId); // invalidate stale entry
        }
        return false;
    }
}
```

## 8. Push Notifications

### 8.1 When to send push notifications

```
Push notification = wake up an offline device to alert about a new message.

Decision tree:
  Is recipient online (active WebSocket)?   → WebSocket push (no notification needed)
  Is recipient connected but app in background? → Push notification + WebSocket if possible
  Is recipient offline (no connection)?     → Push notification only

Avoid duplicate: do NOT send push if WebSocket delivery succeeded.
  Only send push after delivery attempt fails (routing lookup returns null).

Notification content:
  iOS (APNs): "New message from Alice: Hey, are you free tonight?"
  Android (FCM): same
  
  Privacy consideration: show "New message" without content for private conversations.
  Let user configure: full preview / sender only / no preview.
```

### 8.2 Push notification architecture

```java
@Service
public class PushNotificationService {

    public void sendPush(long recipientId, MessageEvent message) {
        // 1. Look up user's registered device tokens
        List<DeviceToken> tokens = deviceTokenRepository.findByUserId(recipientId);
        if (tokens.isEmpty()) return; // no registered devices

        // 2. Build notification payload
        PushPayload payload = PushPayload.builder()
            .title(getSenderName(message.getSenderId()))
            .body(truncate(message.getContent(), 100)) // APNs limit
            .badge(getUnreadCount(recipientId))
            .data(Map.of(
                "conversationId", message.getConversationId(),
                "messageId",      message.getMessageId(),
                "type",           "new_message"
            ))
            .build();

        // 3. Fan out to all devices (user may have phone + tablet + web)
        for (DeviceToken token : tokens) {
            switch (token.getPlatform()) {
                case IOS     -> apnsClient.send(token.getToken(), payload);
                case ANDROID -> fcmClient.send(token.getToken(), payload);
                case WEB     -> webPushClient.send(token.getEndpoint(), payload);
            }
        }
    }

    // Handle delivery receipts from APNs/FCM (invalid tokens → remove)
    public void handleDeliveryFeedback(String token, boolean invalid) {
        if (invalid) {
            deviceTokenRepository.deleteByToken(token); // app uninstalled
        }
    }
}
```

### 8.3 Notification batching and throttling

```
Problem: user receives 50 messages in 30 seconds while offline.
  Sending 50 push notifications: annoying + burns device battery + APNs rate limit.

Solution: notification coalescing
  Group messages within a time window (5 seconds) into one notification:
  "Alice sent 3 messages" rather than 3 separate notifications.

  Redis: when offline delivery is queued:
    INCR notify:{userId}:{conversationId}  → count
    EXPIRE notify:{userId}:{conversationId} 5  → 5-second window
    If count == 1: schedule notification delivery in 5 seconds
    If count > 1:  update payload on the scheduled notification (collapse)

  APNs collapse key: set apns-collapse-id to conversationId
    → iOS replaces previous notification from same conversation with new one.

  FCM collapse key: set collapse_key to conversationId
    → Android replaces previous notification similarly.

Rate limiting pushes:
  Max 1 notification per user per conversation per 5-second window.
  Max 10 notifications per user per minute across all conversations.
  APNs enforces its own limits; FCM has 1,000 messages/second per app.
```

## 9. Multi-Device Sync

### 9.1 The multi-device problem

```
User Alice has:
  - iPhone (session A on chat-server-3)
  - MacBook (session B on chat-server-7)
  - iPad (offline)

When Bob sends a message to Alice:
  1. Both iPhone and MacBook should receive the message in real-time.
  2. When iPad comes online, it should sync missed messages.
  3. When Alice reads on iPhone: MacBook should show "read" status too.
  4. Messages Alice sends from MacBook must appear on iPhone (and vice versa).

Key insight: the routing table maps userId → SET of sessions, not one session.
  routing:{userId} → {sessionA:chat-server-3, sessionB:chat-server-7}
  Delivery service pushes to ALL sessions for a user, not just one.
```

### 9.2 Device registration and session tracking

```java
// Redis hash: routing:{userId} → {sessionId: serverId, ...}
// Allows multiple sessions per user

public void registerSession(long userId, String sessionId, String serverId) {
    redis.hset("routing:" + userId, sessionId, serverId);
    redis.expire("routing:" + userId, 300); // 5-min TTL refreshed by heartbeat
}

public void unregisterSession(long userId, String sessionId) {
    redis.hdel("routing:" + userId, sessionId);
    // If hash is now empty: user fully offline
    if (redis.hlen("routing:" + userId) == 0) {
        presenceService.setOffline(userId);
    }
}

public Map<String, String> getActiveSessions(long userId) {
    return redis.hgetAll("routing:" + userId); // {sessionId → serverId}
}

// Delivery: push to every session
public void deliverToUser(long userId, OutboundMessage message) {
    Map<String, String> sessions = getActiveSessions(userId);
    for (Map.Entry<String, String> entry : sessions.entrySet()) {
        String sessionId = entry.getKey();
        String serverId  = entry.getValue();
        chatServerClient.deliver(serverId,
            DeliverRequest.ofSession(sessionId, message));
    }
}
```

### 9.3 Read receipt sync across devices

```
When Alice reads a message on her iPhone:
  1. iPhone sends: {type:"read", conversationId:123, messageId:9876}
  2. Server updates message_delivery status = READ
  3. Server pushes read receipt to Bob (sender knows Alice read it)
  4. Server also pushes read sync to Alice's OTHER devices:
```

## 11. Scaling and Fault Tolerance

### 11.1 Chat server horizontal scaling

```
Adding chat servers:
  New server registers itself in service discovery (Consul / Kubernetes).
  Load balancer routes new WebSocket connections to it.
  Existing connections stay on their server (sticky sessions by design).

Chat server failure:
  All WebSocket sessions on the failed server are dropped.
  Clients detect disconnect (ping/pong timeout) → reconnect to any available server.
  On reconnect: client sends sync request → receives missed messages from Cassandra.
  Routing table entries for failed server expire (TTL 300s) → cleaned up automatically.
  No message loss: messages are in Cassandra; delivery state recoverable.

Session migration:
  NOT supported: WebSocket sessions cannot be migrated between servers.
  Reconnect-on-failure is the recovery mechanism (clients handle this).
  Reconnect time: typically < 2 seconds (client implements immediate retry).
```

### 11.2 Kafka partitioning for message ordering

```
Message ordering invariant: within a conversation, messages must be delivered
in the order they were sent (or rather, in server_message_id order).

Kafka guarantee: within a partition, messages are ordered.
Partition key: conversation_id
  → All messages for conversation 123 go to the same partition.
  → Delivery consumer processes them in order.
  → Message_id (Snowflake, time-sortable) enforces fine-grained ordering.

Partition count: start with 128 (128 parallel consumers)
  1.16M messages/second / 128 partitions = 9,062 messages/partition/second
  Each partition consumer handles one conversation at a time → ordered delivery.
```

### 11.3 Cassandra replication

```
Keyspace replication factor: 3 (tolerate 2 node failures)
Write consistency: QUORUM (2 of 3 nodes confirm → durable before ack)
Read consistency: LOCAL_ONE (fastest; acceptable for chat history — eventual is ok)

Data centre setup:
  3 DCs (US-East, EU-West, AP-Southeast)
  Replication factor 3 per DC (total 9 copies)
  NetworkTopologyStrategy with LOCAL_QUORUM writes

Compaction: TimeWindowCompactionStrategy (TWCS)
  Compacts within daily time windows → minimal read amplification for time-range queries
  Old windows (>30 days) are candidates for cold storage migration
```

## 12. Interview Questions

**Q: Why is WebSocket better than long polling for chat, and in what scenarios would you still choose long polling?**

A: WebSocket establishes a persistent full-duplex TCP connection after a single HTTP upgrade handshake. The server can push a message to the client the instant it arrives, with only a 2-10 byte frame header overhead. Long polling requires the client to issue a new HTTP request after every message or timeout, adding round-trip latency and repeating HTTP header overhead (200-800 bytes) on every cycle. WebSocket is also full-duplex — the client can send messages on the same connection while simultaneously receiving them, which is essential for chat where both directions are active. Long polling is half-duplex: a separate HTTP POST is needed to send messages while the GET holds open for receives. That said, long polling remains the right choice when WebSocket connections are blocked — corporate firewalls and HTTP proxies sometimes inspect or block WebSocket upgrades, or only support HTTP/1.1 without the Upgrade mechanism. In those environments, long polling works over any standard HTTP proxy. A robust chat client implements WebSocket with automatic fallback to long polling.

---

**Q: How do you guarantee message ordering within a conversation across multiple chat servers?**

A: Two mechanisms work together. First, Snowflake IDs for message_id ensure a globally unique, time-sortable identifier: higher ID means later message. Since Snowflake generation uses a central or per-server epoch, messages generated within the same millisecond on different servers will have different server IDs embedded in the Snowflake, making them distinguishable and orderable. Second, Kafka partitioning by conversation_id ensures all messages within a conversation flow through the same Kafka partition, which Kafka delivers in order. The delivery consumer processes messages from one partition sequentially, so recipients receive them in message_id order. The Cassandra schema uses conversation_id as the partition key and message_id as the clustering key with DESC ordering, meaning range queries naturally return messages in reverse chronological order. The combination — Snowflake for unique ordering + Kafka for in-order delivery + Cassandra clustering key — guarantees that all clients see messages in a consistent order regardless of which chat server processed the send.

---

**Q: How does the hybrid fan-out strategy work for group chats, and where is the threshold between fan-out-on-write and fan-out-on-read?**

A: For small groups, fan-out on write delivers a message copy (or reference) to each member's delivery queue at send time. When a member comes online, their messages are ready and reads are O(1). This approach costs O(N) work per message — fine for N=10 but expensive for N=500 where one message generates 499 push operations. For large groups, fan-out on read stores one copy of the message in the shared conversation log. Each member independently reads from the log, tracking their own position via `last_read_msg_id` in `user_conversation_state`. Reads cost a Cassandra range scan starting from the last read position, but writes are O(1). The threshold in practice is typically 100-200 members: below this, per-recipient fan-out is manageable and gives each member a personal "inbox" that is fast to read; above this, the write amplification and delivery latency of fan-out-on-write become unacceptable. Push notifications are sent individually regardless of strategy — a wakeup signal to each member — but the notification payload is minimal (a badge update), not the full message content.

**Q: Walk through the complete lifecycle of a message from Alice sending it to Bob receiving it, including all failure scenarios.**

A: Alice types "Hello" and taps send. Her client generates a UUID (clientMsgId) and sends `{type:message, conversationId:123, content:"Hello", clientMsgId:uuid-1}` over her WebSocket connection to chat-server-3. The server checks the idempotency store (Redis) for uuid-1 — not seen — generates Snowflake message_id 999, writes the message to Cassandra at QUORUM (waits for 2 of 3 replicas to confirm), stores uuid-1 → 999 in Redis for 24 hours, and immediately acks Alice: `{type:ack, clientMsgId:uuid-1, serverMsgId:999}`. Alice's UI shows a single tick. The server publishes a MessageEvent to Kafka partition for conversation 123. The delivery consumer reads the event, looks up Bob's routing: `routing:Bob = {session-xyz: chat-server-7}`. It makes a gRPC call to chat-server-7, which pushes `{type:message, messageId:999, content:"Hello", ...}` to Bob's WebSocket. Bob's client acks receipt, delivery consumer updates `message_delivery(999, Bob) = delivered`, pushes a delivery receipt to Alice — her UI shows double tick. Bob reads the message, his client sends a read receipt, Alice's UI shows blue double tick. **Failure paths**: if Alice's ack is lost, her client retries with the same uuid-1 — idempotency check prevents double storage. If chat-server-7 is down when delivery is attempted, gRPC fails, routing entry is invalidated, Bob is treated as offline — push notification sent via FCM, message queued for sync. When Bob reconnects, his client sends a sync request with lastMsgId=998, server returns message 999 from Cassandra.

**Q: How does presence tracking work at scale, and how do you avoid the thundering herd when a popular user comes online?**

A: Each user's online status is stored in Redis as a key `presence:{userId}` with a 35-second TTL. The WebSocket server refreshes this TTL every 30 seconds via a heartbeat (WebSocket ping/pong cycle). When the TTL expires without renewal, the user is considered offline — no explicit offline signal needed for most disconnects. `last_seen` is set to the current timestamp when the key is explicitly deleted on clean disconnect or when the TTL expires. The thundering herd problem arises when a user with millions of followers comes online: naively broadcasting a presence update to all contacts simultaneously would generate millions of push events. The solution is lazy pull rather than eager push. Presence updates are only pushed to users who currently have a conversation open with the online user — a small, bounded set (contacts currently viewing their chat with that user). All other contacts see the updated presence the next time they open a conversation, which triggers a single `GET presence:{userId}` Redis lookup. For popular users, we additionally rate-limit presence event fan-out: coalesce status changes within a 30-second window before broadcasting, and cap fan-out to 10,000 active watchers per user.

## 13. Key Takeaways

- **WebSocket for chat**: full-duplex persistent connection; server pushes without client polling; 2-10 byte frame overhead vs 200-800 byte HTTP headers; long polling as fallback for restricted environments
- **Multi-device routing**: `routing:{userId}` → hash of `{sessionId → serverId}` in Redis; delivery fans out to ALL sessions per user; on disconnect, remove session; when hash empty → user offline
- **Cassandra for messages**: partition key = conversation_id (all messages in a conversation collocated), clustering key = message_id DESC (Snowflake = time-sortable), TimeWindowCompactionStrategy for time-series compaction
- **Snowflake message IDs**: globally unique + time-sortable = sort key IS the timestamp; message ordering within conversation guaranteed without separate timestamp index
- **At-least-once + client dedup**: server retries until ACK; clientMsgId in Redis idempotency store (24h TTL) prevents duplicate processing; client deduplicates by serverMsgId
- **Kafka partition key = conversation_id**: all messages for one conversation in one partition → in-order delivery to consumers → ordered fan-out to recipients
- **Presence via Redis TTL**: key expires in 35s if heartbeat stops; heartbeat = WebSocket ping/pong every 30s; lazy pull (not eager push) for most contacts; push only to active conversation watchers
- **Fan-out hybrid for groups**: N ≤ 100 → fan-out on write per recipient queue; N > 100 → shared conversation log, fan-out on read; push notification always individual regardless
- **Push notification coalescing**: APNs collapse-id / FCM collapse_key per conversation; batch messages within 5-second window; max 1 notification per conversation per 5 seconds
- **Tiered storage**: Cassandra hot (0-30 days) + Cassandra compressed warm (30-365 days) + S3/Parquet cold (1+ years); nightly migration job; cold access via S3 fetch + Redis cache
