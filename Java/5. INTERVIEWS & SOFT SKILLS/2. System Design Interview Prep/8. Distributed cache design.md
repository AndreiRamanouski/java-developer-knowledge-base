# Distributed Cache Design

## Overview

A distributed cache sits between application servers and the database, serving frequently accessed data from memory in sub-millisecond latency instead of the 1-50ms a database query costs. At scale, a well-designed cache absorbs 95-99% of read traffic, allowing a database sized for writes to also handle peak read load without collapsing.

**Why caching is architecturally necessary:**

```
Without cache:
  100K req/sec → 100K DB queries/sec → DB saturates at ~50K queries/sec
  Result: 503 errors, cascading failures, angry users

With cache (95% hit rate):
  100K req/sec → 5K DB queries/sec → DB handles with headroom
  Result: sub-millisecond reads, stable DB, happy users

The cache is not an optimisation — at internet scale it is structural.
Remove it and the system collapses. This is why cache design is a first-class
system design topic.
```

**Cache taxonomy:**

```
Local (in-process):  Caffeine, Guava, EHCache
  Pros: zero network latency, no serialisation
  Cons: not shared across servers; small (JVM heap bounded); lost on restart

Distributed (shared): Redis, Memcached, Apache Ignite
  Pros: shared across all servers; survives individual server restarts; large
  Cons: network round-trip (< 1ms local network); serialisation overhead

Multi-tier: local L1 + distributed L2 + database L3 (used by Facebook, LinkedIn)
  Pros: near-zero latency for hottest data + shared correctness
  Cons: L1-L2 divergence; complex invalidation

This design: distributed cache (Redis Cluster) with optional local L1.
```

## 1. Requirements Gathering

### 1.1 Clarifying questions

```
1. "What is the read/write ratio? What data is being cached?"
   → 90:10 read-heavy. Caching: user profiles, product data, session tokens,
     computed feed results, rate limit counters.

2. "What consistency guarantees are required?"
   → Strong consistency for session/auth tokens (stale = security risk).
     Eventual consistency acceptable for product data (stale by seconds OK).

3. "What is the expected cache size and request throughput?"
   → 10 TB total data; 1M cache operations/second peak.

4. "What should happen on cache node failure — fast recovery or never lose data?"
   → High availability: automatic failover within seconds. Some cache misses
     on failover acceptable; no data loss for write-through data.

5. "Do we need TTL-based expiry, manual invalidation, or both?"
   → Both: TTL for naturally expiring data (sessions, rate limits);
     explicit invalidation when source data changes (user profile updates).

6. "Single region or multi-region?"
   → Multi-region; regional caches with cross-region replication for
     read-heavy shared data.
```

### 1.2 Confirmed requirements

```
Functional:
  ✓ GET / SET / DELETE with optional TTL
  ✓ Atomic operations: INCR, DECR, SETNX (set if not exists)
  ✓ Data structures: String, Hash, List, Set, Sorted Set
  ✓ Cache-aside, write-through, and write-behind patterns
  ✓ TTL-based expiry (lazy + active)
  ✓ Explicit invalidation (single key, prefix, tag-based)
  ✓ Cache warming on startup

Non-functional:
  ✓ Latency: p99 < 1ms for GET/SET
  ✓ Throughput: 1M ops/second
  ✓ Availability: 99.99% (< 52 min/year downtime)
  ✓ Capacity: 10 TB across cluster
  ✓ Consistency: eventual for most data; strong for auth tokens
  ✓ Auto-scaling: add/remove nodes without full restart

Out of scope:
  ✗ Persistent storage (Redis AOF/RDB details)
  ✗ Pub/sub messaging
  ✗ Lua scripting (noted as available but not designed here)
```

### 1.3 Scale estimation

```
Memory:
  10 TB data × 2 (replication factor) = 20 TB total memory required
  Each cache node: 128 GB RAM → 20 TB / 128 GB ≈ 160 nodes

Throughput:
  1M ops/sec; Redis single-threaded I/O: ~100K simple ops/sec/instance
  → Minimum 10 instances for throughput; 160 for memory → memory-bound

Throughput per node: 1M / 160 = 6,250 ops/sec/node → well within limits

Network:
  Average value size: 1 KB (after compression)
  1M ops/sec × 1 KB = 1 GB/s aggregate; per node: 1 GB/s / 160 = 6.25 MB/s
  → Each node needs 1 Gbps NIC (standard); 10 Gbps for burst headroom
```

## 2. Consistent Hashing

### 2.1 Why consistent hashing

```
Naive sharding: key → node = hash(key) % NUM_NODES
  Add 1 node: NUM_NODES changes → almost every key maps to a different node
  Result: entire cache invalidated on every scale event → thundering herd on DB

Consistent hashing: keys and nodes both placed on a ring [0, 2^32)
  Each key maps to the nearest node clockwise on the ring.
  Add 1 node: only keys between the new node and its predecessor are remapped.
  At N nodes: adding/removing 1 node remaps only 1/N of all keys (not all).

Impact:
  Without consistent hashing: add node → 100% cache miss rate momentarily
  With consistent hashing:    add node → 1/N cache miss rate (e.g. 1% for 100 nodes)
```

### 2.2 Ring construction and virtual nodes

```
Basic ring:
  Hash each node name to a position on the ring.
  hash("node-1") → position 12,341
  hash("node-2") → position 67,891
  hash("node-3") → position 89,234

  Key "user:123" → hash → position 45,000 → nearest clockwise node = node-2

Problem: non-uniform distribution.
  Three nodes may end up clustered on one half of the ring, leaving the other
  half to one node which handles disproportionately more keys.

Solution: virtual nodes (vnodes).
  Each physical node owns M virtual positions on the ring (M = 150-200 typically).
  hash("node-1#0") → pos A
  hash("node-1#1") → pos B
  ...
  hash("node-1#150") → pos Z
  Virtual nodes interleave uniformly → load spreads evenly.

Adding a node:
  New node claims some virtual nodes from its neighbours.
  Keys belonging to claimed vnodes migrate to the new node.
  Only 1/N of keys migrate (by design); DB load spike proportional to migration.

Removing/failing a node:
  Its virtual nodes are absorbed by the next clockwise neighbour.
  Data on the failed node is served from replicas (see Section 4).
```

```java
public class ConsistentHashRing<T> {
    private final TreeMap<Long, T> ring = new TreeMap<>();
    private final int virtualNodes;
    private final HashFunction hasher;

    public ConsistentHashRing(int virtualNodes) {
        this.virtualNodes = virtualNodes;
        this.hasher = Hashing.murmur3_128();
    }

    public void addNode(T node) {
        for (int i = 0; i < virtualNodes; i++) {
            long hash = hash(node.toString() + "#" + i);
            ring.put(hash, node);
        }
    }

    public void removeNode(T node) {
        for (int i = 0; i < virtualNodes; i++) {
            long hash = hash(node.toString() + "#" + i);
            ring.remove(hash);
        }
    }

    public T getNode(String key) {
        if (ring.isEmpty()) throw new IllegalStateException("No nodes in ring");
        long hash = hash(key);
        // Find nearest clockwise node
        Map.Entry<Long, T> entry = ring.ceilingEntry(hash);
        if (entry == null) entry = ring.firstEntry(); // wrap around ring
        return entry.getValue();
    }

    // Get N distinct nodes for a key (for replication)
    public List<T> getReplicaNodes(String key, int replicationFactor) {
        List<T> nodes = new ArrayList<>();
        Set<T> seen  = new HashSet<>();
        long hash = hash(key);

        // Walk clockwise, collecting distinct physical nodes
        NavigableMap<Long, T> tail = ring.tailMap(hash, true);
        for (T node : Iterables.concat(tail.values(), ring.values())) {
            if (seen.add(node)) nodes.add(node);
            if (nodes.size() == replicationFactor) break;
        }
        return nodes;
    }

    private long hash(String key) {
        return hasher.hashString(key, StandardCharsets.UTF_8).asLong();
    }
}
```

### 2.3 Key routing in Redis Cluster

```
Redis Cluster uses a simplified consistent hashing variant called hash slots:
  16,384 hash slots (0-16383)
  Each key → CRC16(key) % 16384 → slot number
  Each node owns a contiguous range of slots (e.g. node-1: 0-5460)

Hash tags: {user:123}:profile
  Only the part inside {} is hashed → all keys sharing the same tag map to
  the same slot → enables multi-key operations (MGET, transactions)
  on keys that must co-locate.

Slot migration on scale:
  Adding node: take a subset of slots from each existing node.
  Removing node: redistribute that node's slots to remaining nodes.
  Migration is live: during migration, requests for migrating slots are
  forwarded via MOVED/ASK redirects; clients cache the new mapping.
```

## 3. Cache Eviction Policies

### 3.1 Eviction policy overview

```
When memory is full and a new key arrives: which existing key to remove?

Policy              Algorithm              Best for
──────────────────────────────────────────────────────────────────────────
LRU (Least Recently  Track access time;    General-purpose; most common
  Used)              evict oldest access   Works well for temporal locality

LFU (Least Freq.     Track access count;   Viral content; items with stable
  Used)              evict lowest count    frequency patterns

ARC (Adaptive        Balances LRU+LFU      Mixed workloads; self-tuning
  Replacement Cache) dynamically

FIFO                 Evict oldest insert   Simple; poor hit rate in practice

Random               Evict random key      Surprisingly competitive; O(1)

TTL-based            Evict expired keys    Time-sensitive data (sessions)
  (volatile-ttl)     first

noeviction           Reject writes when    Critical data that must not
                     full                  be lost (use with care)

Redis maxmemory-policy options:
  allkeys-lru:       apply LRU to all keys (no TTL required)
  volatile-lru:      apply LRU only to keys with TTL set
  allkeys-lfu:       apply LFU to all keys (Redis 4.0+)
  volatile-lfu:      apply LFU to keys with TTL
  allkeys-random:    random eviction from all keys
  volatile-ttl:      evict key with shortest remaining TTL first
  noeviction:        return error on write when memory full
```

### 3.2 LRU implementation

```
True LRU requires O(1) get and O(1) eviction.
Data structure: HashMap + doubly-linked list.
  HashMap: key → list node (O(1) lookup)
  Doubly-linked list: ordered by recency (head = most recent, tail = LRU)

On GET: move accessed node to head.
On SET: add to head; if over capacity, remove tail.
Both operations: O(1).
```

```java
public class LRUCache<K, V> {
    private final int capacity;
    private final LinkedHashMap<K, V> cache;

    public LRUCache(int capacity) {
        this.capacity = capacity;
        // LinkedHashMap with accessOrder=true: iteration order = access order (LRU)
        this.cache = new LinkedHashMap<>(capacity, 0.75f, true) {
            @Override
            protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
                return size() > capacity; // evict LRU when over capacity
            }
        };
    }

    public synchronized V get(K key) {
        return cache.getOrDefault(key, null); // moves to end (most recent)
    }

    public synchronized void put(K key, V value) {
        cache.put(key, value); // auto-evicts eldest if over capacity
    }

    public synchronized void remove(K key) {
        cache.remove(key);
    }
}
```

### 3.3 LFU implementation

```
LFU requires O(1) get, set, and eviction.
Challenge: tracking access counts efficiently.

Data structure (O(1) LFU — Ketan Shah's algorithm):
  keyToCount:  HashMap<K, int>         — key → access count
  keyToVal:    HashMap<K, V>           — key → value
  countToKeys: HashMap<int, LinkedHashSet<K>> — count → set of keys with that count
  minCount:    int                     — current minimum count (O(1) to find LFU key)

On GET(key):
  increment count; move from countToKeys[count] to countToKeys[count+1]
  if countToKeys[minCount] empty: minCount++

On SET(key, val) — new key:
  if at capacity: evict head of countToKeys[minCount] (LFU; tie-break LRU)
  add key to countToKeys[1]; set minCount = 1

O(1) for all operations.
```

```java
public class LFUCache<K, V> {
    private final int capacity;
    private int minCount = 0;
    private final Map<K, V>                    keyToVal    = new HashMap<>();
    private final Map<K, Integer>              keyToCount  = new HashMap<>();
    private final Map<Integer, LinkedHashSet<K>> countToKeys = new HashMap<>();

    public LFUCache(int capacity) { this.capacity = capacity; }

    public V get(K key) {
        if (!keyToVal.containsKey(key)) return null;
        incrementCount(key);
        return keyToVal.get(key);
    }

    public void put(K key, V value) {
        if (capacity <= 0) return;
        if (keyToVal.containsKey(key)) {
            keyToVal.put(key, value);
            incrementCount(key);
            return;
        }
        if (keyToVal.size() >= capacity) evict();
        keyToVal.put(key, value);
        keyToCount.put(key, 1);
        countToKeys.computeIfAbsent(1, k -> new LinkedHashSet<>()).add(key);
        minCount = 1;
    }

    private void incrementCount(K key) {
        int count = keyToCount.get(key);
        keyToCount.put(key, count + 1);
        countToKeys.get(count).remove(key);
        if (countToKeys.get(count).isEmpty()) {
            countToKeys.remove(count);
            if (minCount == count) minCount++;
        }
        countToKeys.computeIfAbsent(count + 1, k -> new LinkedHashSet<>()).add(key);
    }

    private void evict() {
        LinkedHashSet<K> minKeys = countToKeys.get(minCount);
        K evictKey = minKeys.iterator().next(); // LRU among LFU (insertion order)
        minKeys.remove(evictKey);
        if (minKeys.isEmpty()) countToKeys.remove(minCount);
        keyToVal.remove(evictKey);
        keyToCount.remove(evictKey);
    }
}
```

### 3.4 ARC — Adaptive Replacement Cache

```
ARC adapts between LRU and LFU automatically based on workload.

Maintains four lists:
  T1: recently accessed once (LRU candidates)
  T2: recently accessed more than once (frequently used — LFU candidates)
  B1: ghost entries evicted from T1 (metadata only, no data)
  B2: ghost entries evicted from T2 (metadata only, no data)

Adaptation rule:
  Cache miss, key in B1: LRU was better → shift balance toward T2
  Cache miss, key in B2: LFU was better → shift balance toward T1
  Target p = size(T1) adjusts dynamically

ARC is used by: ZFS (filesystem), some database buffer pools, storage systems.
Redis uses approximate LRU (samples 5-10 random keys, evicts the LRU among them).
Approximate LRU: within 1-2% of true LRU in practice; avoids per-key access timestamp overhead.
```

## 4. Replication Strategy

### 4.1 Why replicate the cache

```
Without replication:
  Node failure → all keys on that node become cache misses
  → sudden flood of requests to DB → potential cascading failure

With replication (primary-replica):
  Node failure → replica promoted → cache serves from replica
  Brief miss window (election time): seconds, not minutes
  DB load spike: proportional to missed keys on failed node (1/N of total)
```

### 4.2 Primary-replica replication

```
Architecture: each primary node has R replicas (typically R=1 or R=2).

Replication flow:
  Client writes to primary only.
  Primary synchronously or asynchronously replicates to replicas.
  Reads can be served by primary or replicas (load balancing).

Sync vs async replication:
  Synchronous: write ack only after replica confirms.
    Pros: zero data loss on primary failure.
    Cons: write latency = max(primary, replica) RTT; replica latency adds to write path.

  Asynchronous (Redis default): write ack after primary confirms; replica updates eventually.
    Pros: write latency = primary only; fast.
    Cons: primary failure before replica sync → small data loss window.
    Acceptable for cache: losing a cache entry means a DB read, not a corruption.

Redis replication:
  REPLICAOF primary-host primary-port  (replica side)
  Primary streams a replication backlog (RDB snapshot + command stream)
  Partial resync on reconnect (PSYNC2) — no full dump needed after brief disconnect
```

### 4.3 Redis Sentinel vs Redis Cluster

```
Redis Sentinel (HA for standalone Redis):
  3+ Sentinel processes monitor primary and replicas.
  On primary failure: Sentinels vote, elect new primary, notify clients.
  Failover time: 5-30 seconds (configurable with down-after-milliseconds).
  Max data loss: async replication lag at time of failure.
  Suitable for: single-shard, high-availability setups.

Redis Cluster (HA + sharding):
  16,384 hash slots distributed across N primary nodes.
  Each primary has 1-2 replicas.
  Automatic failover: primary failure → replica promoted within seconds.
  Gossip protocol: nodes monitor each other (no dedicated sentinel needed).
  Client redirects: MOVED (permanent) / ASK (temporary during migration).
  Suitable for: large datasets that require both HA and horizontal sharding.

┌──────────────────────────────────────────────────────────────────┐
│                      Redis Cluster (6 nodes)                      │
│  Primary-1 (slots 0-5460)     Primary-2 (slots 5461-10921)       │
│  Replica-1a                   Replica-2a                          │
│                                                                  │
│  Primary-3 (slots 10922-16383)                                   │
│  Replica-3a                                                       │
└──────────────────────────────────────────────────────────────────┘

Failure scenario:
  Primary-1 fails → cluster detects via gossip (FAIL flag after timeout)
  → Replica-1a promoted to Primary-1 (replica election)
  → Clients receive MOVED redirect to new Primary-1 address
  → Cache serves normally within ~2-5 seconds of failure
```

### 4.4 Failover and split-brain prevention

```
Split-brain: primary and replica both think they are the primary.
  Dangerous for databases (both accept writes → divergence).
  For cache: less dangerous (stale data is temporary; no durability requirement).
  But still avoid: double-write to different nodes causes inconsistency.

Redis Cluster prevention:
  Majority quorum required for failover election.
  Replica needs votes from > half of cluster masters.
  If a network partition isolates a minority: they cannot elect a new primary.
  Majority side continues serving; minority side rejects writes (cluster_state=fail).

Fencing with version tokens:
  Each primary has a generation number (epoch).
  Writes accepted only from the highest-epoch primary.
  Old primary receiving writes from stale client: rejected (stale epoch).
```

## 5. Cache Invalidation

### 5.1 The hardest problem in computer science

```
"There are only two hard things in computer science: cache invalidation and
naming things." — Phil Karlton

Why invalidation is hard:
  1. Timing: invalidate too early → stale reads during the window
             invalidate too late  → stale reads for longer
  2. Atomicity: update DB + invalidate cache in two steps → race conditions
  3. Cascading: one entity update may invalidate many dependent cached objects
  4. Multi-level: local L1 + distributed L2 both need invalidation
```

### 5.2 Cache invalidation patterns

```
Pattern 1: TTL expiry (simplest, eventually consistent)
  Set cache with TTL matching data's acceptable staleness.
  user:{id}:profile → TTL 5 minutes (user rarely changes profile)
  rate:limit:{id}   → TTL 60 seconds (must be fresh)
  session:{token}   → TTL = session lifetime

  Pros: no explicit invalidation needed; dead simple.
  Cons: stale reads up to TTL; must choose TTL carefully per data type.

Pattern 2: Cache-aside with on-write invalidation (most common)
  On write: DELETE cache key immediately after DB write.
  On read: cache miss → read from DB → store in cache.
  
  Pros: cache always fresh after write; simple to implement.
  Cons: one request after invalidation always misses; thundering herd possible.

Pattern 3: Write-through
  On write: update DB AND update cache atomically (within the write path).
  Cache always contains latest value; no cold cache after write.
  
  Pros: no stale reads; no thundering herd after invalidation.
  Cons: write latency includes cache update; cache stores data never read (wasteful).

Pattern 4: Write-behind (write-back)
  On write: update cache immediately; update DB asynchronously (batch).
  
  Pros: write latency = cache only (very fast); DB batching reduces write load.
  Cons: data loss if cache node fails before async DB write; complex recovery.
  Use when: write-heavy counters (view counts), session data, shopping carts.
```

### 5.3 Race conditions in cache invalidation

```
Race 1: Read-aside stale write
  T1: Writer updates DB (user name = "Bob")
  T2: Reader cache miss; reads DB (gets "Bob")
  T3: Writer deletes cache key (key gone)
  T4: Reader stores "Bob" in cache (correct — no race here)
  Result: correct ✓

  But:
  T1: Writer reads old value to increment (user score = 100)
  T2: Writer updates DB (score = 101)
  T3: Reader cache miss; reads DB (score = 101)
  T4: Writer deletes cache key
  T5: Reader stores 101 in cache
  Result: correct ✓ (if writer invalidates AFTER DB write)

Race 2: Classic cache invalidation race (invalidate-before-write)
  T1: Writer starts update; deletes cache key FIRST
  T2: Reader: cache miss → reads DB (gets old value 100) → stores in cache
  T3: Writer: commits DB update (101)
  Cache now has: 100 (stale!) until TTL expires.
  Fix: ALWAYS invalidate AFTER DB write, not before.

Race 3: Double-delete race (Cache-aside with concurrent writers)
  Thread A: read miss → DB query → about to SET cache
  Thread B: update DB → DELETE cache key
  Thread A: SET cache (now holds stale data!)
  Fix 1: use short TTL to bound staleness window
  Fix 2: "lease" pattern (Facebook's solution):
    On cache miss: server issues a lease token to the requesting client.
    Other clients missing the same key wait for the lease holder to populate it.
    Lease expires in 10ms; writer invalidates lease on DB update.
    Requesting client can only SET cache if its lease is still valid.
```

```java
@Service
public class CacheInvalidationService {

    // Pattern: invalidate after DB write (not before)
    @Transactional
    public void updateUser(long userId, UserUpdateRequest req) {
        // 1. Write to DB (within transaction)
        userRepository.update(userId, req);
        // 2. Invalidate cache AFTER commit (via TransactionSynchronization)
        TransactionSynchronizationManager.registerSynchronization(
            new TransactionSynchronizationAdapter() {
                @Override
                public void afterCommit() {
                    // Only invalidate after DB transaction commits
                    // If TX rolls back: cache remains valid (no stale write committed)
                    cache.delete("user:" + userId);
                    cache.delete("user:" + userId + ":profile");
                    // Invalidate any computed caches that depend on this user
                    cache.delete("user:" + userId + ":feed:page:1");
                }
            }
        );
    }

    // Tag-based invalidation: invalidate all keys sharing a tag
    // Useful for: "invalidate all cached results that include product 123"
    public void taggedInvalidation(String tag) {
        // Store tag → set of keys in Redis
        Set<String> keys = redis.smembers("tag:" + tag);
        if (!keys.isEmpty()) {
            redis.del(keys.toArray(new String[0]));
            redis.del("tag:" + tag);
        }
    }

    // On write: register key with its tags
    public void setWithTags(String key, Object value, Duration ttl, String... tags) {
        cache.set(key, value, ttl);
        for (String tag : tags) {
            redis.sadd("tag:" + tag, key);
            redis.expire("tag:" + tag, (long) ttl.getSeconds() + 60);
        }
    }
}
```

### 5.4 Cache invalidation via CDC (Change Data Capture)

```
Problem: application code must remember to invalidate caches on every write path.
  Easy to forget; especially in complex microservice architectures.

Solution: CDC-based invalidation (used by LinkedIn, Airbnb)
  Database change events → Kafka → cache invalidation service

Architecture:
  PostgreSQL → Debezium (CDC connector) → Kafka topic "db-changes"
  Cache invalidation service consumes "db-changes"
  Maps DB table + row key → cache key(s) to invalidate
  Issues DEL commands to Redis

Benefits:
  Decoupled: application writes to DB; cache is automatically kept in sync
  Consistent: all writes (even direct DB writes) trigger invalidation
  Auditable: Kafka log shows every invalidation event
  Recoverable: on cache invalidation service restart, replay Kafka from offset

Implementation:
  Debezium: open-source CDC for PostgreSQL, MySQL, MongoDB
  Publishes: {operation: "UPDATE", table: "users", key: {id: 123}, before: {...}, after: {...}}
  Invalidation service maps: table "users" + id 123 → cache keys:
    ["user:123", "user:123:profile", "user:123:feed:*"]
  Issues DEL / pattern delete to Redis
```

## 6. Cache Warming

### 6.1 The cold start problem

```
Cold cache: freshly started cache node has no data.
  Every request is a cache miss → all traffic hits DB → DB overloaded.
  After a major incident requiring full cluster restart: catastrophic if
  cache warms up slowly while production traffic floods in.

Cache warming strategies:

1. Proactive warming (before going live)
   Before routing traffic: pre-populate cache with expected hot keys.
   Run a "warm-up job" that reads top-N keys from DB and loads them into cache.
   Traffic starts only after cache hit rate > threshold (e.g. 80%).

2. Shadow traffic replay
   Replay production traffic logs against new cache node.
   New node mirrors requests from existing node for warmup.

3. Backup/snapshot restore
   Redis RDB snapshot: periodic dump of in-memory data to disk.
   On restart: restore from latest RDB snapshot → cache near-warm instantly.
   Acceptable staleness: keys loaded from snapshot; TTL-expired keys evicted.

4. Gradual traffic shifting
   Route 5% of traffic to new cluster → allow it to warm.
   Gradually increase to 100% as hit rate improves.
```

```java
@Service
public class CacheWarmingService {

    @PostConstruct
    public void warmCacheOnStartup() {
        // Async warming: do not block application startup
        CompletableFuture.runAsync(() -> {
            log.info("Starting cache warm-up");

            // Load top-10K most-accessed users (from analytics table)
            List<Long> hotUserIds = analyticsRepository.getTopAccessedUserIds(10_000);
            for (List<Long> batch : Iterables.partition(hotUserIds, 500)) {
                List<User> users = userRepository.findAllById(batch);
                for (User user : users) {
                    cache.set("user:" + user.getId(),
                              user,
                              Duration.ofMinutes(30));
                }
            }

            // Load top-1K products
            List<Product> hotProducts = productRepository.findTopViewed(1_000);
            hotProducts.forEach(p ->
                cache.set("product:" + p.getId(), p, Duration.ofHours(1)));

            log.info("Cache warm-up complete. Hit rate: {}%",
                     cacheMetrics.getHitRate() * 100);
        }, warmupExecutor);
    }
}
```

## 7. Monitoring and Alerting

### 7.1 Key metrics

```
Performance metrics:
  hit_rate:              cache hits / (hits + misses)
                         Alert: < 80% (down from baseline) → investigate eviction or TTL
  latency_p99:           99th percentile GET/SET latency
                         Alert: > 2ms → network or Redis CPU issue
  ops_per_second:        total commands/second
                         Alert: > 80% of max capacity → scale out
  evictions_per_second:  keys evicted due to memory pressure
                         Alert: > 0 sustained → cache undersized or TTL too long

Memory metrics:
```

## 8. Cache Stampede and Thundering Herd Protection

### 8.1 The cache stampede problem

```
Scenario: popular key expires simultaneously.
  1,000 requests/second hit this key.
  Key expires → all 1,000 requests get cache miss simultaneously.
  All 1,000 make DB queries → DB overloaded.

Three solutions:

1. Mutex / request coalescing (most common):
   First thread to miss acquires a lock.
   Lock holder queries DB, populates cache.
   Other threads wait, then read from cache (no duplicate DB queries).

2. Probabilistic early expiration (XFetch algorithm):
   Before TTL expires, proactively re-compute with probability proportional to
   how close to expiry the key is and how expensive the recompute is.
   Avoids synchronised expiry: different clients regen at slightly different times.

3. Stale-while-revalidate:
   Return stale cached value immediately (fast response).
   Trigger async background refresh of cache.
   Next request gets fresh value; no request waits on DB.
```

```java
@Service
public class CacheStampedeProtector<K, V> {
    private final ConcurrentHashMap<K, CompletableFuture<V>> inFlight = new ConcurrentHashMap<>();
    private final Cache<K, V> cache;
    private final Function<K, V> loader; // DB query

    // Request coalescing: only one DB query per key per miss window
    public V get(K key) {
        V cached = cache.get(key);
        if (cached != null) return cached;

        // Use computeIfAbsent to atomically get-or-create the in-flight future
        CompletableFuture<V> future = inFlight.computeIfAbsent(key, k -> {
            // Only one thread creates the future; all others share it
            CompletableFuture<V> f = CompletableFuture.supplyAsync(() -> {
                V value = loader.apply(k);           // DB query (once per miss burst)
                cache.set(k, value, Duration.ofMinutes(5));
                return value;
            });
            // Remove from inFlight when done (success or failure)
            f.whenComplete((v, ex) -> inFlight.remove(k));
            return f;
        });

        try {
            return future.get(500, TimeUnit.MILLISECONDS); // wait max 500ms
        } catch (Exception e) {
            inFlight.remove(key);
            return loader.apply(key); // fallback: direct DB query
        }
    }
}

// XFetch probabilistic early expiration
public boolean shouldRefresh(String key, double beta, double delta) {
    // beta: recompute cost factor (higher = refresh earlier)
    // delta: time to recompute in seconds
    Long ttl = redis.pttl(key); // remaining TTL in milliseconds
    if (ttl == null || ttl < 0) return true; // expired

    double remainingSeconds = ttl / 1000.0;
    // Refresh if: currentTime + beta * delta * log(random()) > expiry_time
    return -beta * delta * Math.log(Math.random()) > remainingSeconds;
}
```

## 9. Cache Patterns Summary

### 9.1 Pattern selection guide

```
Cache-aside (Lazy loading):
  Application checks cache; on miss, loads from DB; stores in cache.
  When to use: read-heavy; unpredictable access patterns; large datasets.
  Pros: only cache what's needed; resilient to cache failure.
  Cons: cold start; stale reads possible; thundering herd on popular key miss.

Write-through:
  Every write goes to cache AND DB synchronously.
  When to use: read-after-write consistency required; not too write-heavy.
  Pros: always fresh; no cache misses after first write.
  Cons: write latency; cache stores data never read.

Write-behind (write-back):
  Write to cache immediately; async batch write to DB.
  When to use: write-heavy counters, sessions, carts.
  Pros: very low write latency; DB write batching.
  Cons: data loss risk if cache fails before DB flush; complex recovery.

Read-through:
  Cache library fetches from DB on miss (not application code).
  Application always reads from cache only; never talks to DB directly.
  When to use: cleaner application code; read-heavy; homogeneous data.
  Pros: clean separation; cache always up to date on first access.
  Cons: cache is SPOF if no fallback; initial cold queries are synchronous.

Refresh-ahead:
  Cache proactively refreshes keys before they expire (based on access pattern).
  When to use: predictable access patterns; cannot tolerate miss latency.
  Pros: no cold reads; low-latency even for frequently changing data.
  Cons: predictability required; wasted refreshes if pattern changes.
```

### 9.2 Multi-level cache architecture

```java
@Service
public class MultiLevelCache<K, V> {
    // L1: local Caffeine cache (per-JVM, sub-microsecond)
    private final Cache<K, V> l1 = Caffeine.newBuilder()
        .maximumSize(10_000)
        .expireAfterWrite(30, SECONDS)
        .build();

    // L2: distributed Redis cache (shared, < 1ms)
    private final RedisCache<K, V> l2;

    // L3: database (source of truth, 1-50ms)
    private final Repository<K, V> db;

    public V get(K key) {
        // L1 hit: sub-microsecond
        V l1Hit = l1.getIfPresent(key);
        if (l1Hit != null) return l1Hit;

        // L2 hit: < 1ms
        V l2Hit = l2.get(key);
        if (l2Hit != null) {
            l1.put(key, l2Hit); // promote to L1
            return l2Hit;
        }

        // L3 read: 1-50ms
        V dbValue = db.findById(key);
        if (dbValue != null) {
            l2.set(key, dbValue, Duration.ofMinutes(5)); // populate L2
            l1.put(key, dbValue);                         // populate L1
        }
        return dbValue;
    }

    public void invalidate(K key) {
        l1.invalidate(key);   // immediate local invalidation
        l2.delete(key);        // distributed invalidation
        // Other servers' L1 caches will expire naturally (within 30s TTL)
        // For strict consistency: publish invalidation event via Redis pub/sub
        invalidationPublisher.publish(key); // other servers evict L1 on receive
    }
}
```

## 10. Interview Questions

**Q: Why does consistent hashing with virtual nodes solve the hotspot problem that simple modulo sharding cannot?**

A: Simple modulo sharding (`hash(key) % N`) distributes keys evenly in theory, but two problems arise in practice. First, when N changes (node added or removed), nearly every key remaps to a different node. Adding one node to a 10-node cluster causes 90% of keys to remap — the entire cache becomes cold simultaneously, and the database faces a thundering herd. Second, without virtual nodes, physical node positions on the consistent hash ring are determined by their own hashes, which may cluster nodes together on one arc, leaving another arc underrepresented and one node responsible for a disproportionate share of keys. Virtual nodes solve both problems. With V=150 virtual nodes per physical node, adding a new physical node inserts 150 positions on the ring, each taking a slice from a different neighbour. Only 1/N of all keys remap (the target theoretical minimum). The 150 small slices come from all existing nodes proportionally, ensuring the new node receives a balanced share. The distribution of virtual nodes across the ring approaches uniform as V increases, eliminating hotspots from unlucky hash positioning.

---

**Q: Explain the cache invalidation race condition and how the "invalidate after write" rule and lease pattern solve it.**

A: The classic race occurs when invalidation and read happen concurrently. Thread A reads a cache miss, queries the database (getting value 100), and is about to store it in the cache. Thread B updates the database to 101 and deletes the cache key. Thread A then stores 100 in the cache — a stale value that will persist until the TTL expires. The "invalidate after DB write" rule partially addresses this: if Thread B only deletes the cache key after the DB write commits, and Thread A queries the DB after Thread B's delete, Thread A gets 101. But if Thread A's DB query happens before Thread B's DB write, the race still produces a stale value. The lease pattern (Facebook's solution) closes this race completely: on a cache miss, the cache server issues a fencing token (lease) to the first client that misses. The lease-holding client is the only one allowed to populate the cache; other clients miss and must wait or retry. When Thread B writes to the DB, it invalidates the lease for that key. If Thread A tries to SET the cache with a revoked lease, the SET is rejected. Thread A retries, gets another miss, queries the fresh DB value (101), and stores it correctly. Leases bound the window of staleness to the lease duration (typically 10 milliseconds).

---

**Q: Compare LRU and LFU eviction policies. When does each fail, and what does ARC do differently?**

A: LRU evicts the least recently accessed key. It excels for workloads with temporal locality — recently used data is likely to be used again soon. It fails for scan patterns: a one-time sequential scan of 10 million keys (a nightly report) evicts the entire working set of frequently used data. The scan keys are accessed once, fill the cache, and push out genuinely hot keys. LFU evicts the least frequently accessed key. It protects frequently used data from scan eviction but has a stale frequency problem: a key accessed 10,000 times six months ago has a high count and is never evicted, even though it is now rarely accessed. New, genuinely hot keys arrive with a count of 1 and are immediately evicted. LFU essentially has no mechanism to forget old frequency. ARC maintains two LRU lists: T1 for recently accessed once, T2 for accessed multiple times. Two ghost lists (B1, B2) track recently evicted entries from each. When a cache miss hits a ghost entry in B1, the workload shows LRU is too small — ARC shifts capacity toward recency (T1). A miss in B2 means LFU is too small — ARC shifts toward frequency (T2). This dynamic rebalancing adapts to changing workload patterns without manual tuning, making ARC consistently outperform both pure LRU and pure LFU across mixed workloads.

**Q: Walk through the complete flow when a cache node fails, from failure detection to client transparency.**

A: In a Redis Cluster setup, each node sends PING messages to peer nodes every second. If a node does not respond within `cluster-node-timeout` (default 15 seconds), the detecting node marks it as PFAIL (possibly failed). When a majority of master nodes also mark it PFAIL within a timeout window, the cluster collectively flags it FAIL. The replica with the smallest replication lag for the failed primary initiates an election: it increments its configuration epoch and requests votes from other masters. Once it receives votes from a majority, it promotes itself to primary, takes over the dead node's hash slots, and broadcasts a CLUSTER UPDATE message. This entire process takes roughly 2-5 seconds after the node is declared FAIL. During this window, clients trying to access keys on the failed node receive CLUSTERDOWN errors. Well-behaved clients implement retry logic: on CLUSTERDOWN or connection failure, retry the operation after a brief backoff, allowing time for the election to complete. After promotion, clients receive MOVED redirects pointing to the new primary for those hash slots and update their local slot-to-node map. From the application perspective, operations on the affected keys experience a brief 2-5 second window of errors, after which service resumes transparently. The routing table update propagates to all clients within one subsequent request.

**Q: How does CDC-based cache invalidation differ from application-level invalidation, and when would you choose each?**

A: Application-level invalidation requires every write path in the application code to explicitly call a cache invalidation function after the database write. This is simple to understand and trace — the developer writes to the DB and immediately deletes the cache key. It fails in practice as systems grow: multiple code paths may modify the same entity (direct API writes, admin tools, background jobs, migrations, direct SQL fixes during incidents). Any path that forgets to invalidate leaves the cache stale indefinitely. In a microservices architecture, multiple services may write to the same data, and coordinating cache invalidation across service boundaries requires tight coupling. CDC-based invalidation uses Debezium or a similar connector to tail the database's write-ahead log (WAL) and emit every change as a Kafka event. A cache invalidation service consumes these events and issues Redis DEL commands. This approach is database-aware, not application-aware — it catches every write regardless of which code path initiated it, including emergency SQL updates, migrations, and direct DB changes. It decouples the cache from the application: services write to the DB without thinking about the cache; the invalidation service handles it. The trade-off: CDC adds infrastructure complexity (Debezium, Kafka, invalidation service), introduces a small lag (milliseconds to seconds) between DB write and cache invalidation, and requires mapping every DB table/column to its corresponding cache keys. Application-level invalidation is appropriate for simple systems with few write paths; CDC is preferred for complex systems with many writers or strict eventual consistency requirements.

## 11. Key Takeaways

- **Consistent hashing + virtual nodes**: adding 1 node remaps 1/N keys (not all); 150 virtual nodes per physical node ensures uniform distribution; eliminates thundering herd on scale events
- **LRU vs LFU**: LRU for temporal locality; LFU for stable frequency patterns; ARC adapts dynamically between both; Redis uses approximate LRU (sample 5-10 keys) — within 1-2% of true LRU
- **Invalidate AFTER DB write**: never before — pre-invalidation allows readers to cache stale pre-write values; register invalidation in `TransactionSynchronization.afterCommit()`
- **Redis Cluster failover**: PFAIL → FAIL (majority quorum) → replica election → promotion in 2-5 seconds; clients retry with backoff; slot map updates via MOVED redirect
- **Cache stampede protection**: request coalescing (mutex per key, share the single in-flight DB query among concurrent misses); XFetch probabilistic early expiration; stale-while-revalidate
- **Write patterns**: cache-aside (lazy, resilient); write-through (fresh, higher write latency); write-behind (fast writes, data loss risk); refresh-ahead (predictable patterns)
- **Hit rate is the north star metric**: alert below 80%; sustained evictions = cache undersized; check with `redis-cli INFO stats` → `keyspace_hits` / `keyspace_misses`
- **CDC invalidation for complex systems**: Debezium tails DB WAL → Kafka → invalidation service → Redis DEL; catches all writes including direct DB changes; decouples cache from application
- **TTL is your safety net**: even with invalidation, always set a TTL; cache entries without TTL can persist indefinitely if invalidation is missed
- **Multi-level cache**: Caffeine L1 (per-JVM, 30s TTL, 10K keys) + Redis L2 (shared, 5-min TTL) + DB L3; propagate L1 invalidation via Redis pub/sub for cross-server consistency
