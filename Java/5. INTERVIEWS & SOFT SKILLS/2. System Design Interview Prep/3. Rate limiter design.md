# Rate Limiter Design

## Overview

A rate limiter controls how many requests a client can make within a time window. Without rate limiting, a single misbehaving client can consume all server resources, degrade service for everyone, or run up unbounded infrastructure costs. Rate limiting is a first-class concern in any public API, internal microservice mesh, or user-facing platform.

**Why rate limiting matters:**

```
Denial of Service (DoS) protection:   prevent one client from starving others
Cost control:                          LLM APIs, SMS, payment processors charge per call
Quality of service:                    guarantee p99 latency for paying customers
Abuse prevention:                      credential stuffing, scraping, spam
Compliance:                            enforce contract SLAs per customer tier
```

**Rate limiting dimensions:**

```
By what?          user_id, api_key, IP address, endpoint, tenant_id
What threshold?   requests per second, per minute, per day, per month
What action?      reject (429), throttle (slow down), degrade (return cached response)
Hard vs soft:     hard limit = reject all over-quota; soft = allow burst, then throttle
```

## 1. Requirements Gathering

### 1.1 Clarifying questions

```
1. "What granularity of limiting? Per user, per IP, per API key, or per endpoint?"
   → Multiple dimensions simultaneously (per user AND per endpoint).

2. "Should limits be configurable per customer tier?"
   → Free: 100 req/min. Pro: 1,000 req/min. Enterprise: custom.

3. "What is the response for rate-limited requests — reject or queue?"
   → Reject with 429 Too Many Requests + Retry-After header.

4. "Must limits be exact or is approximate acceptable?"
   → Approximate is acceptable (within 0.1% error at high throughput).

5. "Single region or distributed/multi-region?"
   → Distributed: multiple rate limiter nodes must share state.

6. "What is the latency budget for the rate limiting check itself?"
   → < 5ms per check (added to every API call's critical path).

7. "Do we need analytics — which clients are being throttled and why?"
   → Yes: dashboards for operations team + customer-facing quota usage APIs.
```

### 1.2 Confirmed requirements

```
Functional:
  ✓ Limit requests per user per endpoint per time window
  ✓ Configurable limits per customer tier (free / pro / enterprise)
  ✓ Return 429 with Retry-After header when limit exceeded
  ✓ Expose remaining quota in response headers (X-RateLimit-Remaining)
  ✓ Allow temporary burst above base rate (configurable)
  ✓ Analytics: per-client quota usage visible via API

Non-functional:
  ✓ Latency: < 5ms added to every request (p99)
  ✓ Accuracy: at most 0.1% over-counting (slight leniency acceptable, not hardness)
  ✓ Availability: rate limiter failure must FAIL OPEN (allow traffic rather than block all)
  ✓ Throughput: handle 500,000 req/second across the fleet
  ✓ Distributed: consistent limits across 50+ API server nodes
```

## 2. Rate Limiting Algorithms

### 2.1 Algorithm comparison

```
Algorithm           Burst handling    Memory     Accuracy    Complexity
──────────────────────────────────────────────────────────────────────────
Fixed Window        Yes (at boundary) O(1)       Moderate    Low
Sliding Window Log  None (smooth)     O(req/win) High        High
Sliding Window Ctr  Limited           O(1)       Good (~1%)  Medium
Token Bucket        Yes (burst cap)   O(1)       High        Medium
Leaky Bucket        No (smoothed)     O(queue)   High        Medium

Recommended for most APIs: Token Bucket (natural burst, O(1) state)
Recommended for strict smoothing: Leaky Bucket
Recommended for simplicity: Sliding Window Counter
```

### 2.2 Fixed window counter

```
Divide time into fixed windows (e.g. 1-minute buckets).
Count requests per window; reject when count exceeds limit.

Window:  |---minute 0---|---minute 1---|---minute 2---|
Limit:   100 req/min

Problem: boundary burst
  At 00:59: 100 requests → allowed (minute 0 is full, minute 1 starts next)
  At 01:00: 100 requests → allowed (minute 1 is fresh, counter reset)
  Result:   200 requests in 2 seconds — 2× the intended rate!

Implementation:
  key = "rl:{user_id}:{endpoint}:{minute_bucket}"
  INCR key → if count > limit: reject
  EXPIRE key 60 (auto-cleanup)

Pros:  O(1) space, extremely simple
Cons:  boundary burst allows 2× rate at window edges
```

```java
public boolean allowFixedWindow(String userId, String endpoint, int limitPerMinute) {
    long bucket = System.currentTimeMillis() / 60_000; // current minute
    String key = "rl:" + userId + ":" + endpoint + ":" + bucket;

    long count = redis.incr(key);
    if (count == 1) redis.expire(key, 120); // 2-minute TTL: covers current + next window

    return count <= limitPerMinute;
}
```

### 2.3 Sliding window log

```
Store timestamp of every request in a sorted set.
On each request: remove timestamps older than window_size, count remaining.

Window: 60 seconds rolling (not fixed bucket)
Limit:  100 req/60s

On request at t=75:
  Remove all entries with timestamp < 75-60 = 15
  Count remaining entries
  If count < 100: add timestamp 75, allow
  Else: reject

Pros:  Perfectly accurate (no boundary burst)
Cons:  O(req/window) memory per user — 100 req/min × 1M users = 100M timestamps
       → 800 MB just for timestamps at 8 bytes each
       → infeasible at scale

Implementation: Redis ZSET with score = timestamp
  ZREMRANGEBYSCORE key 0 (now-windowMs)
  ZCARD key → count
  ZADD key now "now"
  EXPIRE key windowMs/1000
```

### 2.4 Sliding window counter (hybrid — recommended for most cases)

```
Divide time into fixed windows (e.g. 1-minute buckets).
Estimate current window count as weighted sum of current + previous window.

Formula:
  estimated_count = prev_window_count × (1 - elapsed_fraction) + curr_window_count

Example: limit=100/min, at 45 seconds into current minute (75% elapsed):
  elapsed_fraction = 0.75
  prev_window_count = 80
  curr_window_count = 30
  estimated = 80 × (1 - 0.75) + 30 = 20 + 30 = 50 → ALLOW

Why this works:
  We estimate "how many of the last 60 seconds' requests came from the previous window"
  as (1 - elapsed_fraction) × prev_count.
  Error bound: < 1% in practice for uniformly distributed traffic.

Pros:  O(1) space (just two counters), good accuracy, no boundary burst
Cons:  Assumes uniform traffic within windows (slight approximation)
```

```java
public boolean allowSlidingWindowCounter(String userId, int limit, int windowSeconds) {
    long now = System.currentTimeMillis();
    long windowMs = windowSeconds * 1000L;
    long currBucket = now / windowMs;
    long prevBucket = currBucket - 1;

    String currKey = "rl:" + userId + ":curr:" + currBucket;
    String prevKey = "rl:" + userId + ":prev:" + prevBucket;

    long currCount = Optional.ofNullable(redis.get(currKey))
                             .map(Long::parseLong).orElse(0L);
    long prevCount = Optional.ofNullable(redis.get(prevKey))
                             .map(Long::parseLong).orElse(0L);

    double elapsedFraction = (double)(now % windowMs) / windowMs;
    double estimatedCount = prevCount * (1.0 - elapsedFraction) + currCount;

    if (estimatedCount >= limit) return false; // reject

    redis.incr(currKey);
    redis.expire(currKey, windowSeconds * 2L);
    return true;
}
```

### 2.5 Token bucket

```
Conceptual model:
  Bucket holds up to 'capacity' tokens.
  Tokens refill at 'refill_rate' tokens/second.
  Each request consumes 1 token (or N tokens for weighted requests).
  If bucket has tokens: allow and consume. Else: reject.

Parameters:
  capacity:     maximum burst size (e.g. 200 tokens)
  refill_rate:  steady-state rate (e.g. 100 tokens/second)

Behaviour:
  Burst: if bucket is full, client can send 200 requests instantly
         then settles to 100/second sustained rate
  Idle period: tokens accumulate up to capacity — client earns "burst credit"
  Heavy use: bucket empties, rate limited to exactly refill_rate

Why it matches real API usage:
  Real users burst at login/app start, then settle to steady state.
  Token bucket naturally accommodates short bursts without penalising normal users.

State per client: {tokens: float, last_refill_time: long}
  → O(1) memory
```

```java
@Service
public class TokenBucketRateLimiter {
    // Lua script ensures atomicity: read-modify-write in one Redis round-trip
    private static final String TOKEN_BUCKET_LUA = """
        local key = KEYS[1]
        local capacity = tonumber(ARGV[1])
        local refill_rate = tonumber(ARGV[2])   -- tokens per second
        local now = tonumber(ARGV[3])            -- current time in milliseconds
        local requested = tonumber(ARGV[4])      -- tokens requested (usually 1)
        
        local bucket = redis.call('HMGET', key, 'tokens', 'last_refill')
        local tokens = tonumber(bucket[1]) or capacity
        local last_refill = tonumber(bucket[2]) or now
        
        -- Refill tokens based on elapsed time
        local elapsed = (now - last_refill) / 1000.0  -- convert ms to seconds
        tokens = math.min(capacity, tokens + elapsed * refill_rate)
        
        local allowed = 0
        if tokens >= requested then
            tokens = tokens - requested
            allowed = 1
        end
        
        redis.call('HMSET', key, 'tokens', tokens, 'last_refill', now)
        redis.call('EXPIRE', key, 3600)  -- 1-hour inactivity TTL
        
        return { allowed, math.floor(tokens), math.floor(capacity - tokens) }
        """;

    public RateLimitResult allow(String clientId, String endpoint) {
        String key = "tb:" + clientId + ":" + endpoint;
        RateLimitConfig config = configService.getConfig(clientId, endpoint);

        List<Object> result = redis.execute(
            tokenBucketScript, List.of(key),
            String.valueOf(config.getCapacity()),
            String.valueOf(config.getRefillRate()),
            String.valueOf(System.currentTimeMillis()),
            "1"
        );

        boolean allowed = ((Long) result.get(0)) == 1;
        long remaining = (Long) result.get(1);
        long retryAfterMs = allowed ? 0 :
            (long)((1.0 / config.getRefillRate()) * 1000); // time to earn 1 token

        return new RateLimitResult(allowed, remaining, retryAfterMs);
    }
}
```

**Token bucket trace:**

```
Config: capacity=10, refill_rate=2 tokens/sec

t=0:    bucket=10 (full). Request → tokens=9. ALLOW.
t=0:    8 more rapid requests → tokens=1. ALLOW all.
t=0:    2 more requests → tokens=0, then -1. REJECT second.
t=0.5s: refill: +1 token → tokens=1. Request → tokens=0. ALLOW.
t=1.0s: refill: +2 tokens → tokens=2. Two requests → tokens=0. ALLOW both.
t=5.0s: idle for 4s → refill: +8 → tokens=min(10,8)=8. Burst credit earned.
```

### 2.6 Leaky bucket

```
Conceptual model:
  Requests enter a queue (the "bucket") of fixed size.
  Requests are processed (leak) at a fixed rate regardless of input rate.
  If queue is full when new request arrives: reject.

Parameters:
  queue_size:    maximum queued requests (burst buffer)
  leak_rate:     processing rate (requests/second)

Behaviour:
  Bursty input:  requests queue up; processed smoothly at leak_rate
  Empty queue:   new request processed immediately (no waiting)
  Full queue:    new requests rejected (not queued)

Vs token bucket:
  Token bucket: allows instantaneous burst (all stored tokens consumed at once)
  Leaky bucket: smooths all traffic to constant rate (no instantaneous burst)

Use case: outgoing requests to downstream services (protect them from your bursts)
  Not great for API-facing rate limiting (users dislike artificial queuing delays)
```

```java
// Leaky bucket as a request queue with fixed-rate drain
// Implemented as a bounded queue + scheduled thread pool
@Service
public class LeakyBucketLimiter {
    private final BlockingQueue<Runnable> requestQueue;
    private final ScheduledExecutorService drainer;

    public LeakyBucketLimiter(int queueSize, int leakRatePerSecond) {
        this.requestQueue = new ArrayBlockingQueue<>(queueSize);

        long intervalMs = 1000L / leakRatePerSecond;
        this.drainer = Executors.newSingleThreadScheduledExecutor();
        drainer.scheduleAtFixedRate(() -> {
            Runnable task = requestQueue.poll();
            if (task != null) task.run(); // process one request at fixed interval
        }, 0, intervalMs, TimeUnit.MILLISECONDS);
    }

    public boolean submit(Runnable request) {
        return requestQueue.offer(request); // false = bucket full = reject
    }
}
// Note: leaky bucket is more useful as a client-side rate limiter
// (to avoid flooding a downstream service) than as a server-side API gate.
```

## 3. Redis Implementation

### 3.1 Why Redis for distributed rate limiting

```
Requirements for distributed rate limiting:
  Shared state:   all N API servers must see the same counter
  Atomic ops:     read-modify-write must be atomic (no race conditions)
  Low latency:    < 1ms for counter ops (rate limit check adds < 5ms total)
  TTL support:    counters auto-expire when window closes
  High throughput: 500,000 API requests/sec → 500K Redis ops/sec

Redis delivers all of these:
  In-memory:      sub-millisecond ops
  INCR is atomic: single-threaded execution model, no race on increment
  EXPIRE:         built-in TTL for automatic cleanup
  Lua scripts:    multi-step operations (token bucket) execute atomically
  Cluster mode:   horizontal scale to millions of ops/second
```

### 3.2 Atomic Lua scripts — why they are necessary

```
Race condition without Lua:
  Server A: GET counter → 99
  Server B: GET counter → 99
  Server A: 99 < 100 → INCR → 100
  Server B: 99 < 100 → INCR → 101 ← BOTH allowed! Limit bypassed.

With Lua script (executes atomically, no interleaving):
  Script A: GET 99 → 99 < 100 → INCR → 100 → ALLOW
  Script B: (waits for A to finish) GET 100 → 100 >= 100 → REJECT

Redis guarantees: Lua scripts are atomic — no other command executes
while the script runs. This is the standard pattern for rate limiting.
```

### 3.3 Sliding window counter in Lua (production-grade)

```lua
-- sliding_window.lua
-- KEYS[1] = base key prefix (e.g. "rl:user123:POST:/api/tweets")
-- ARGV[1] = limit (max requests per window)
-- ARGV[2] = window_ms (window size in milliseconds)
-- ARGV[3] = now_ms (current timestamp in milliseconds)
-- Returns: {allowed (0/1), remaining, retry_after_ms}

local key_curr = KEYS[1] .. ':' .. math.floor(tonumber(ARGV[3]) / tonumber(ARGV[2]))
local key_prev = KEYS[1] .. ':' .. (math.floor(tonumber(ARGV[3]) / tonumber(ARGV[2])) - 1)
local limit = tonumber(ARGV[1])
local window_ms = tonumber(ARGV[2])
local now_ms = tonumber(ARGV[3])

local curr_count = tonumber(redis.call('GET', key_curr) or '0')
local prev_count = tonumber(redis.call('GET', key_prev) or '0')

local elapsed_fraction = (now_ms % window_ms) / window_ms
local estimated = prev_count * (1.0 - elapsed_fraction) + curr_count

if estimated >= limit then
    -- Calculate when next token available
    local retry_after = math.ceil((1.0 / limit) * window_ms)
    return {0, 0, retry_after}
end

redis.call('INCR', key_curr)
redis.call('PEXPIRE', key_curr, window_ms * 2)

local remaining = math.floor(limit - estimated - 1)
return {1, remaining, 0}
```

### 3.4 Rate limiter as middleware

```java
@Component
@Order(Ordered.HIGHEST_PRECEDENCE)
public class RateLimitFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest req,
                                     HttpServletResponse res,
                                     FilterChain chain) throws IOException, ServletException {
        String clientId = resolveClientId(req);   // API key or user_id from JWT
        String endpoint = req.getMethod() + ":" + req.getRequestURI();

        RateLimitResult result = rateLimiter.allow(clientId, endpoint);

        // Always set quota headers (even on success — clients use these for backoff)
        res.setHeader("X-RateLimit-Limit",     String.valueOf(result.getLimit()));
        res.setHeader("X-RateLimit-Remaining", String.valueOf(result.getRemaining()));
        res.setHeader("X-RateLimit-Reset",     String.valueOf(result.getResetEpochSeconds()));

        if (!result.isAllowed()) {
            res.setHeader("Retry-After", String.valueOf(result.getRetryAfterSeconds()));
            res.setStatus(429);
            res.setContentType("application/json");
            res.getWriter().write("""
                {"error":"rate_limit_exceeded",
                 "message":"Too many requests. Retry after %d seconds.",
                 "retry_after":%d}
                """.formatted(result.getRetryAfterSeconds(), result.getRetryAfterSeconds()));
            return;
        }

        chain.doFilter(req, res);
    }

    private String resolveClientId(HttpServletRequest req) {
        // Priority: API key > authenticated user > IP address
        String apiKey = req.getHeader("X-API-Key");
        if (apiKey != null) return "apikey:" + apiKey;

        String userId = extractUserIdFromJwt(req);
        if (userId != null) return "user:" + userId;

        return "ip:" + req.getRemoteAddr(); // fallback: IP-based limiting
    }
}
```

### 3.5 Key design and namespacing

```
Rate limit key structure:
  rl:{dimension}:{entity_id}:{endpoint}:{window_bucket}

Examples:
  rl:user:usr_12345:POST:/tweets:minute-27460      (per-user per-endpoint per-minute)
  rl:apikey:key_abc123:global:hour-458              (per-API-key global hourly)
  rl:ip:192.168.1.1:GET:/search:minute-27460        (per-IP per-endpoint)
  rl:tenant:acme_corp:global:day-189                (per-tenant daily)

Multiple limits simultaneously (enforce ALL):
  A single request may check multiple keys:
    1. user-level per-minute limit
    2. user-level per-day limit
    3. API-key-level per-hour limit
    4. endpoint-level global rate (protect a specific expensive endpoint)

  All must pass for the request to proceed.
  Each check is one Lua script execution (one Redis round-trip per key).
  Total: 4 Redis calls in parallel → ~2-3ms total.

TTL convention:
  Window TTL = 2 × window_size (previous + current window both needed for sliding counter)
  Inactivity TTL = max(2 × window_size, 3600) — garbage collect idle clients
```

## 4. Distributed Rate Limiting

### 4.1 The distributed problem

```
Single-node: trivial. One counter, one server, atomic INCR.

Multi-node: 50 API servers, each getting requests for the same user.
  Requirement: user's total requests across ALL 50 servers ≤ limit.

Architecture options:

Option A: Centralised Redis (Recommended)
  All 50 servers call the same Redis cluster for every request.
  Redis is the single source of truth for counters.

  ┌────────┐ ┌────────┐ ┌────────┐   All servers
  │API Srv1│ │API Srv2│ │API Srv3│   share one
  └───┬────┘ └───┬────┘ └───┬────┘   Redis counter
      └──────────┼──────────┘
                 │
         ┌───────▼────────┐
         │  Redis Cluster  │  counter: rl:user123:tweets
         └────────────────┘

  Pros: exact accuracy, simple
  Cons: Redis is on critical path of every request (< 1ms per call acceptable)

Option B: Local counter + async sync
  Each server maintains a local counter.
  Periodically (every 100ms) sync local counts to Redis.
  Allowed locally = (limit / num_servers) per local node.

  Pros: no Redis on request hot path
  Cons: approximate (burst window = sync interval),
        uneven traffic distribution causes under/over-counting

Option C: Sticky routing (consistent hashing of client_id to server)
  Route all requests from client X to the same API server.
  That server owns X's counter locally.
  No distributed coordination needed.

  Pros: exact, no Redis overhead for rate limiting
  Cons: uneven load distribution, requires stateful routing layer,
        server failure requires counter migration
        
Production recommendation: Option A (centralised Redis) for accuracy.
  Redis cluster handles 1M+ simple ops/second — well within budget.
  Use connection pooling + pipelining to minimise latency.
```

### 4.2 Multi-region distributed rate limiting

```
Problem: user in US and EU simultaneously — their requests hit different regions.
         Each region has its own Redis cluster.
         US Redis sees 80 req/min; EU Redis sees 30 req/min.
         Total: 110 — over the 100 req/min limit, but neither region alone knows.

Solutions:

Option A: Single global Redis (cross-region)
  All regions call one Redis cluster (in, say, US-East).
  Pros: exact global rate limiting
  Cons: cross-region latency (50-150ms to Redis from EU/AP) → violates 5ms budget

Option B: Regional limits with global coordination (Recommended)
  Assign regional quotas: total_limit / num_active_regions
  (e.g. 100/min global → US: 70/min, EU: 20/min, AP: 10/min, proportional to traffic)
  Background job redistributes quota every 60 seconds based on actual usage.
  Each region enforces its own quota locally.

  Pros: low latency (local Redis), approximate global accuracy
  Cons: cross-region burst possible during redistribution window

Option C: Gossip-based synchronisation
  Regions gossip counter updates every few seconds.
  Each region maintains an approximate global view.
  Allow = local_remaining + (sum of received gossip increments) < global_limit

  Pros: no single point of coordination
  Cons: complex, eventual consistency means burst windows
  Used by: Cloudflare's distributed rate limiting

Option D: Synchronous cross-region write (for strict limits)
  For highly sensitive limits (payment API: max 3 charges/second):
    Write to all regions synchronously before allowing request.
    Accept higher latency for correctness.
  Impractical for general high-throughput APIs.
```

### 4.3 Race conditions and edge cases

```
Race: two requests arrive simultaneously at the exact rate limit boundary.
  Both read count=99 (limit=100), both increment → count=101.
  Lua script prevents this: only one can execute at a time in Redis.

Clock skew between servers:
  If Server A thinks it's 12:00:59.999 and Server B thinks 12:01:00.001:
    Server A increments minute-0 bucket.
    Server B increments minute-1 bucket (new window!).
    Result: both increments allowed — boundary burst possible.
  Mitigation: NTP sync with < 1ms drift; use millisecond sliding window
    rather than minute buckets to shrink the boundary window.

Redis failure (node down):
  FAIL OPEN: if Redis is unreachable, allow the request.
  Rationale: it is better to serve traffic than to deny all users due to
             a rate limiter outage. Rate limiter is a protection mechanism,
             not a correctness requirement.
  Implementation:
    try {
        return rateLimiter.check(clientId);
    } catch (RedisException e) {
        log.error("Rate limiter unavailable, failing open", e);
        metrics.increment("rate_limiter.fail_open");
        return RateLimitResult.ALLOWED; // fail open
    }

Redis slow (latency spike):
  Set a timeout on Redis calls: 2ms timeout with fail-open fallback.
  Circuit breaker: after 5 consecutive timeouts, bypass Redis for 30 seconds.
```

## 5. Handling Burst Traffic

### 5.1 What "burst" means in rate limiting context

```
Burst types:

1. Legitimate user burst (good):
   Mobile app starts up → batch of API calls (session init, user profile, feed load).
   Should be allowed: short, predictable, then settles to normal rate.
   → Token bucket handles this naturally (accumulated tokens = burst credit).

2. Retry storm (bad — self-inflicted):
   Server returns 503 → all clients retry simultaneously → thundering herd.
   → Exponential backoff with jitter required on client side.

3. Malicious burst (bad):
   DDoS attempt: single IP floods with 10,000 req/second.
   → IP-level rate limiting with much lower threshold (1,000 req/min).

4. Traffic spike (neutral — manage carefully):
   Marketing campaign launches → 10× normal traffic.
   → System should auto-scale application servers.
   → Rate limiter should smooth this, not crash under it.
```

### 5.2 Token bucket burst configuration

```
Strategy: separate burst_capacity from sustained_rate

Example: Pro plan API
  sustained_rate: 100 req/min  (1.67 req/sec)
  burst_capacity: 200 tokens   (allows 200 consecutive requests before throttling)

Behaviour:
  After 1 idle minute: bucket full at 200 tokens.
  User makes 200 rapid requests: all allowed (burst credit spent).
  Next 60 seconds: limited to 100 req/min until bucket refills.

Token bucket parameters tuning:
  burst_capacity = sustained_rate × burst_multiple
  burst_multiple: 2–5× for typical APIs; 10–20× for batch/bulk endpoints

  Free tier:       capacity=50,  rate=1/sec    (small burst, tight rate)
  Pro tier:        capacity=200, rate=50/sec   (generous burst, fast recovery)
  Enterprise:      capacity=1000,rate=500/sec  (large burst, fast recovery)

How to set burst_multiple:
  Interview answer: "I'd look at actual client behavior in logs — p99 burst
  size for legitimate users sets the floor. Marketing says campaigns drive
  3× spikes, so 5× gives headroom."
```

### 5.3 Retry-After and client-side backoff

```
Server side: always return Retry-After header with 429.
  Retry-After: 12        (12 seconds until next request allowed)
  X-RateLimit-Reset: 1710000060  (Unix timestamp when window resets)

Client side: MUST implement exponential backoff with jitter.
  Without jitter (naive):
    All throttled clients retry at t=12 → synchronised burst → everyone throttled again.
    Self-reinforcing thundering herd.

  With full jitter (AWS recommendation):
    sleep = random(0, min(cap, base × 2^attempt))
    cap = 60s, base = 1s:
      attempt 0: sleep = random(0, 1)   = 0.4s
      attempt 1: sleep = random(0, 2)   = 1.3s
      attempt 2: sleep = random(0, 4)   = 2.8s
      attempt 3: sleep = random(0, 8)   = 5.1s
      ...capped at 60s

  Clients spread their retries randomly → no synchronised burst.
```

```java
// Client-side retry with exponential backoff and jitter
public <T> T executeWithRateLimit(Supplier<T> apiCall, int maxRetries) {
    int attempt = 0;
    while (attempt <= maxRetries) {
        try {
            return apiCall.get();
        } catch (RateLimitException e) {
            if (attempt == maxRetries) throw e;

            long retryAfterMs = e.getRetryAfterMs();
            if (retryAfterMs > 0) {
                // Server told us when to retry — use it with small jitter
                long sleep = retryAfterMs + ThreadLocalRandom.current().nextLong(0, 1000);
                Thread.sleep(sleep);
            } else {
                // Exponential backoff with full jitter
                long cap = 60_000L;
                long base = 1_000L;
                long maxSleep = Math.min(cap, base * (1L << attempt));
                long sleep = ThreadLocalRandom.current().nextLong(0, maxSleep);
                Thread.sleep(sleep);
            }
            attempt++;
        }
    }
    throw new MaxRetriesExceededException();
}
```

### 5.4 Adaptive rate limiting

```
Problem: static limits don't account for system load.
  At 10% CPU: 1,000 req/sec feels fine.
  At 90% CPU: 1,000 req/sec causes cascading failures.

Solution: adaptive limits — tighten limits under system stress.

  Base limit: 1,000 req/sec
  At CPU > 70%: limit = 800 req/sec  (80% of base)
  At CPU > 85%: limit = 500 req/sec  (50% of base)
  At CPU > 95%: limit = 200 req/sec  (20% of base) + alert

Implementation:
  1. Health check service polls CPU, memory, DB connection pool utilisation.
  2. Publishes "load factor" [0.0, 1.0] to Redis key: "system:load_factor"
  3. Rate limiter reads load_factor (cached locally for 1 second).
  4. effective_limit = configured_limit × (1.0 - load_factor × 0.8)

This is similar to Netflix's Concurrency Limit (gradient2 algorithm):
  Observe: RTT of downstream calls
  Adjust: if RTT increasing → reduce limit; if RTT decreasing → increase limit
  Probes: periodically increase limit to discover new headroom
```

### 6.2 Layered rate limiting strategy

```
Layer 1: Infrastructure layer (nginx/load balancer)
  IP-level: 10,000 req/min per IP (DDoS protection, coarse grain)
  Handled before requests reach application servers.
  Ultra-fast (no Redis): connection-count based limiting.

Layer 2: API Gateway
  API-key level: per-tier limits (Redis token bucket)
  Global endpoint limits: protect expensive endpoints across all users
  Returns 429 before request reaches microservices.

Layer 3: Service level (within microservices)
  Service-to-service: prevent one service from overwhelming another
  Example: notification service → email provider: max 100 emails/sec
  Implemented as leaky bucket (smooth outgoing rate)

Layer 4: User/tenant level (application logic)
  Business rules: max 3 payment attempts/hour (fraud prevention)
  Implemented in application code, stored in PostgreSQL (durable)

Diagram:
  Client → [Layer 1: nginx IP limit]
         → [Layer 2: API Gateway API-key limit]
         → [Layer 3: Service mesh service limit]
         → [Layer 4: Application business logic limit]
         → Actual business logic
```

## 9. Interview Questions

**Q: Walk through how you would implement a token bucket rate limiter in Redis, and why a Lua script is required.**

A: A token bucket for a client stores two values: the current token count and the last refill timestamp. On each request, we calculate elapsed time since last refill, compute new tokens to add at the refill rate, cap at capacity, then check if at least one token is available. If yes, decrement and allow; if no, reject with a Retry-After header indicating when the next token will be available. The naive implementation — GET the hash, compute in application code, SET back — has a race condition: two simultaneous requests both read count=1, both see "sufficient tokens," both decrement, resulting in count=-1 and both requests allowed. A Lua script in Redis solves this because Redis executes Lua atomically — no other command runs while the script is executing. The entire read-compute-write cycle is one atomic operation. This is not a Redis transaction (MULTI/EXEC), which buffers commands but doesn't prevent other commands from interleaving between buffered commands. Lua is strictly atomic. The only performance consideration is that Lua blocks Redis for its duration, so the script must be short — typically under 100 microseconds.

---

**Q: How does the sliding window counter algorithm approximate the sliding window log without the memory cost?**

A: The sliding window log stores a timestamp for every request, giving exact counts but O(requests-per-window) memory — catastrophic at scale. The sliding window counter uses only two integers: the previous window's count and the current window's count. It estimates the count for the rolling window using the formula: `prev_count × (1 - elapsed_fraction) + curr_count`, where `elapsed_fraction` is how far through the current window we are. The insight is that if traffic is roughly uniform within each window, the fraction of the previous window's requests that fall within the current rolling window is proportional to how much of the current window has elapsed. If we're 75% through the current window, approximately 25% of last window's requests are within the last 60 seconds. This is an approximation — bursty traffic within a window can cause up to ~1% error — but for rate limiting this is entirely acceptable. The memory cost is exactly two integers per client regardless of request volume, making it practical for millions of concurrent clients.

---

**Q: What happens to your rate limiter when Redis goes down, and how do you handle it?**

A: The correct answer is to fail open — allow requests through rather than blocking all traffic. Rate limiting is a protection mechanism, not a correctness requirement. Blocking all traffic because the rate limiter is unavailable would mean a Redis outage takes down the entire API — which is far worse than briefly allowing some over-quota traffic. The implementation wraps the Redis call in a try-catch: if Redis throws a connection exception or times out (use a short timeout, 2ms), log the failure, increment a `rate_limiter.fail_open` metric (alerting the on-call team immediately), and return ALLOWED. A circuit breaker pattern adds resilience: after 5 consecutive Redis failures, open the circuit and bypass Redis entirely for 30 seconds, then probe with a single request to check recovery. Two additional mitigations help: first, use a Redis Cluster with automatic failover — primary failure triggers promotion of a replica within ~1 second, minimising exposure. Second, each API server can maintain a local in-memory token bucket as a secondary limiter — it's less accurate (no coordination between servers) but prevents unlimited traffic during Redis outage.

---

**Q: How does your rate limiter handle a distributed denial-of-service attack from thousands of different IP addresses?**

A: A volumetric DDoS from diverse IPs bypasses per-IP rate limits because each individual IP stays under the threshold. Defence requires multiple layers. First, at the infrastructure layer (nginx/load balancer), implement global connection-rate limiting per second — if total incoming connections spike beyond a threshold (say, 10× normal), engage connection queue management or return 503 before hitting application servers. Second, the API gateway can implement a global endpoint rate limit independent of client identity — for example, `POST /api/tweets` accepts at most 50,000 requests/second globally. A single Redis counter tracks this; when exceeded, all further requests to that endpoint are rejected regardless of who sends them. Third, anomaly detection: track request rates per IP in a sliding window; IPs showing unusual patterns (e.g. suddenly sending 1,000 req/sec when normal is 10 req/min) get flagged for challenge (CAPTCHA) or temporary blocking via a deny-list. Fourth, for sustained attacks, CDN providers like Cloudflare offer DDoS mitigation at the edge, absorbing attack traffic before it reaches origin servers. The rate limiter's role is one layer in a defence-in-depth strategy, not the sole protection.

---

**Q: How would you design rate limiting for a multi-tenant SaaS where tenants have different limits and some tenants have millions of end users?**

A: Multi-tenant rate limiting requires a hierarchy: tenant-level limits, user-within-tenant limits, and endpoint-specific overrides. Store tier configurations in a config service backed by PostgreSQL, cached in Redis with a 60-second TTL. The rate limit key includes the tenant ID: `rl:tenant:{tenant_id}:global:{bucket}` for tenant-level, `rl:user:{tenant_id}:{user_id}:{endpoint}:{bucket}` for user-level. Both checks run on every request — tenant quota protects against one tenant consuming all infrastructure; user quota protects the tenant's other users from a single rogue actor within the tenant. For enterprise tenants with millions of users, the per-user limits are generous (their contract covers it) but the tenant-level limit is what constrains total spend. To avoid the Redis key explosion problem — millions of users each getting their own counters — use sliding window counters with a 2-minute TTL, which auto-expires after inactivity. At 1 million active users, each with two counter keys at 50 bytes each, that's 100 MB per tenant — manageable with Redis Cluster sharding by tenant_id prefix. Configuration changes (a tenant upgrading their plan) invalidate the cached config and take effect within 60 seconds.

## 10. Key Takeaways

- **Algorithm selection**: token bucket for most APIs (natural burst support, O(1) state); sliding window counter for simplicity and bounded approximation; avoid sliding window log at scale (O(n) memory)
- **Lua scripts are mandatory for atomicity**: GET-compute-SET in application code has a race condition; Lua executes atomically in Redis, preventing two simultaneous requests from both passing at the limit boundary
- **Fail open on Redis failure**: rate limiter unavailability must not take down the API; allow traffic with monitoring/alerting rather than rejecting all requests
- **Token bucket parameters**: `burst_capacity` sets instantaneous burst tolerance; `refill_rate` sets sustained rate; tune burst_multiple to 2-5× for typical APIs
- **Response headers are part of the contract**: always return `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`, and `Retry-After` on 429 — clients use these to self-throttle
- **Retry-After + jitter prevents thundering herd**: without jitter, all throttled clients retry simultaneously and overwhelm the server again; full jitter distributes retries randomly
- **Multi-layer architecture**: infrastructure (IP/connection), API gateway (API-key/tier), service mesh (service-to-service), application (business rules) — each layer has different granularity and enforcement mechanism
- **Multi-region limits are approximate**: exact global limits require cross-region Redis (unacceptable latency); use regional quota allocation with background rebalancing for acceptable approximation
- **Key design**: include tier, entity type, entity ID, endpoint, and time bucket in the key; use 2× window TTL so previous window data is available for sliding window calculation
- **Adaptive limits**: under system stress, dynamically reduce effective limits (load_factor × configured_limit) to protect downstream services — similar to Netflix's concurrency limit algorithm
