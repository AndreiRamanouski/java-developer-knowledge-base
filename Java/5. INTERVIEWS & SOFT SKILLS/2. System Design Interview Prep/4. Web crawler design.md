# Web Crawler Design

## Overview

A web crawler systematically downloads web pages, extracts links, and recursively follows them to discover and index the web. At scale, a crawler must process billions of URLs while respecting server politeness rules, avoiding duplicate content, handling failures gracefully, and distributing work across hundreds of machines. Google's search index crawls ~15 billion pages; even a focused domain crawler may process hundreds of millions of URLs.

**Core crawl loop:**

```
Seed URLs → URL Frontier → Fetch → Parse → Extract Links
               ▲                                  │
               └──────────── New URLs ────────────┘
                              (deduped)

The challenge is not the loop itself — it is managing this loop at
billions of URLs with politeness, deduplication, prioritisation,
fault tolerance, and freshness, all simultaneously.
```

**Crawler taxonomy:**

```
General crawler:   indexes the entire web (Google, Bing, CommonCrawl)
Focused crawler:   targets a specific domain or topic (e-commerce price tracker)
Incremental:       re-crawls pages to detect changes (news aggregator)
Deep web:          handles login forms, JavaScript SPAs (specialised)

This design: general-purpose, distributed, politeness-aware
```

## 1. Requirements Gathering

### 1.1 Clarifying questions

```
1. "What is the target scale — how many pages, how quickly?"
   → 1 billion pages over 30 days = ~386 pages/second average.

2. "Is this a general web crawler or focused on specific domains/content?"
   → General web crawler; seed set of 1M popular URLs to start.

3. "Do we need to re-crawl pages to detect updates?"
   → Yes: high-priority pages (news) re-crawled daily; others weekly/monthly.

4. "Do we need to handle JavaScript-rendered pages (SPAs)?"
   → Nice to have; start with static HTML; mark as extension.

5. "How should we prioritise which pages to crawl first?"
   → By estimated importance (PageRank-like score, domain authority).

6. "What is the output — raw HTML, parsed text, extracted links?"
   → All three: raw HTML in object store, parsed content + metadata in index,
     links fed back into the URL frontier.

7. "What politeness constraints must we follow?"
   → Respect robots.txt, limit requests per domain, honour Crawl-delay.

8. "Are there legal/ethical constraints?"
   → Honour robots.txt strictly; add User-Agent identifying our crawler;
     respect noindex meta tags; no login-walled content.
```

### 1.2 Confirmed requirements

```
Functional:
  ✓ Crawl 1 billion URLs in 30 days (~400 pages/second)
  ✓ Respect robots.txt (disallow rules, Crawl-delay directive)
  ✓ Avoid re-crawling the same URL in the same crawl cycle
  ✓ Extract and follow links (BFS traversal of the web graph)
  ✓ Store raw HTML, parsed text, and extracted metadata
  ✓ Support re-crawl scheduling based on content change frequency
  ✓ Prioritise crawl by page importance score

Non-functional:
  ✓ Distributed: 100s of crawler nodes
  ✓ Fault tolerant: node failure must not lose URL queue entries
  ✓ Politeness: max 1 request per domain per second by default
  ✓ Extensible: plug in new content parsers (PDF, images, JS rendering)
  ✓ Idempotent: same URL processed by two nodes → stored once, no corruption

Out of scope:
  ✗ JavaScript rendering (noted as extension)
  ✗ Content indexing / full-text search (downstream consumer)
  ✗ Link graph analysis / PageRank computation
```

### 1.3 Scale estimation

```
Target: 1 billion pages / 30 days
  = 1B / (30 × 86,400) ≈ 386 pages/second → round to 400/second

Average HTML page size: 100 KB (compressed: ~30 KB)
Raw storage: 1B × 100 KB = 100 TB over 30 days
Compressed storage: ~30 TB

Links per page: average 50 outbound links
Total URLs discovered: 1B pages × 50 links = 50B candidate URLs
After deduplication (90% already seen): ~5B new URLs enqueued per cycle

URL size: ~100 bytes average
URL frontier storage: 5B × 100 bytes = 500 GB

DNS cache:
  At 400 pages/sec, each page requires 1 DNS lookup (if not cached)
  Unique domains: ~200M (rough estimate)
  Cache TTL: 24 hours; active cache: 5M most frequent domains
  5M × 256 bytes (hostname + IP) = 1.28 GB → fits in memory

Bandwidth:
  400 pages/sec × 30 KB (compressed) = 12 MB/s inbound
  × 100 crawler nodes = 1.2 GB/s aggregate → ~10 Gbps capacity needed
```

## 3. URL Frontier

### 3.1 What the frontier does

```
The URL frontier is the data structure that manages which URLs to crawl next.
It must support:
  1. Priority:    high-value pages before low-value ones
  2. Politeness:  don't send two requests to the same domain simultaneously
  3. Dedup:       don't enqueue already-crawled or already-queued URLs
  4. Scale:       hold billions of URLs durably (not just in memory)
  5. Freshness:   re-enqueue URLs for re-crawl based on change frequency
```

### 3.2 Two-tier queue design

```
The URL frontier separates PRIORITY from POLITENESS using two tiers of queues.

Tier 1: Priority Queues (one per priority level)
  Rank URLs by estimated importance before they reach crawler nodes.
  Priority = f(PageRank estimate, domain authority, freshness need, discovery depth)

  High-priority queue:   news sites, popular domains (crawl within minutes)
  Medium-priority queue: established sites (crawl within hours)
  Low-priority queue:    obscure pages, deep links (crawl within days)

  Input:  newly extracted links + re-crawl scheduled URLs
  Output: URLs fed into Tier 2 queues

Tier 2: Politeness Queues (one per domain)
  Each domain has its own FIFO queue.
  A selector ensures minimum delay between consecutive requests to same domain.

  domain:nytimes.com → [url1, url2, url3, ...]    (wait >= 1s between each)
  domain:bbc.com     → [url4, url5, ...]
  domain:reddit.com  → [url6, ...]

  Crawler nodes pull from domain queues only when the domain's delay has elapsed.

Flow:
  New URL → Dedup check → Priority assignment → Priority Queue
  Priority Queue → Domain Queue Selector → Domain Queue (per domain)
  Domain Queue → (wait for politeness delay) → Crawler Node
```

```
Visualisation:

Tier 1 Priority Queues:
  [High]   -- [cnn.com/article1, bbc.com/news2, nytimes.com/story3 ...]
  [Medium] -- [example.com/page1, blog.com/post2 ...]
  [Low]    -- [obscure.net/deep/link/123 ...]
               │
               │ Prioritiser distributes to domain queues
               ▼
Tier 2 Domain Queues:
  cnn.com    → [article1, article7, article9]  ← last fetch: 0.8s ago
  bbc.com    → [news2, news5]                  ← last fetch: 1.2s ago → READY
  nytimes.com→ [story3]                        ← last fetch: 0.3s ago

Queue Selector: pick bbc.com (politeness delay satisfied) → dispatch to crawler
```

### 3.3 Storage implementation

```
URL frontier storage options:

In-memory (Redis): fast, limited capacity
  Only the "hot" portion (next N URLs to crawl) in Redis sorted set
  Sorted set: score = priority score, member = URL
  Capacity: ~10M URLs at 100 bytes each = 1 GB — reasonable for hot tier

Persistent (Kafka or database): full frontier durability
  Kafka topic per priority level: "frontier-high", "frontier-medium", "frontier-low"
  Consumer group: domain-partitioned workers (one per domain partition)
  Kafka provides durability, replayability, and backpressure naturally

  Alternatively: dedicated URL frontier DB (PostgreSQL or Cassandra)
  Table: (url_hash, url, priority, domain, enqueued_at, next_crawl_at, status)

Combined architecture:
  Kafka: durable queue of all pending URLs (persistent, survives restarts)
  Redis sorted set: hot tier (next 1M URLs), populated by Kafka consumer
  Crawlers pull from Redis; Redis refilled from Kafka as slots open
```

```java
@Service
public class UrlFrontierService {

    public void enqueue(String url, double priorityScore) {
        String urlHash = sha256(url);

        // 1. Dedup check against Bloom filter (fast, probabilistic)
        if (bloomFilter.mightContain(urlHash)) {
            return; // probably already seen — skip
        }

        // 2. Confirm against persistent seen-url store (eliminates false positives)
        if (seenUrlStore.exists(urlHash)) {
            return; // definitively already crawled
        }

        // 3. Determine domain queue
        String domain = extractDomain(url);

        // 4. Store in Kafka (durable)
        CrawlTask task = CrawlTask.builder()
            .url(url)
            .urlHash(urlHash)
            .domain(domain)
            .priorityScore(priorityScore)
            .enqueuedAt(Instant.now())
            .build();
        kafkaTemplate.send("frontier-" + priorityBucket(priorityScore),
                           domain,  // partition key: same domain → same partition
                           task);

        // 5. Add to Bloom filter to prevent future re-enqueueing
        bloomFilter.put(urlHash);
    }
}
```

### 3.4 Priority scoring

```java
public double computePriority(String url, CrawlMetadata metadata) {
    double score = 0.0;

    // Domain authority (pre-computed, updated weekly)
    score += 0.30 * normalize(domainAuthorityService.getScore(extractDomain(url)), 0, 100);

    // Crawl depth (pages closer to seed = higher priority)
    score += 0.20 * Math.max(0, 1.0 - (metadata.getDepth() / 10.0));

    // Content freshness need
    double freshnessUrgency = computeFreshnessUrgency(metadata);
    score += 0.25 * freshnessUrgency;

    // Inlink count (how many discovered pages link to this URL)
    score += 0.15 * normalize(Math.log1p(metadata.getInlinkCount()), 0, 15);

    // Content type hint (HTML > PDF > images for text indexing)
    score += 0.10 * contentTypeScore(url);

    return score; // [0.0, 1.0]
}
```

## 4. Duplicate Detection

### 4.1 The scale of the dedup problem

```
50 billion candidate URLs discovered across a 30-day crawl.
Must determine for each: "have we crawled or queued this before?"

Naive approach: HashSet<String> of all URL hashes
  50B URLs × 32 bytes (SHA-256 hash) = 1.6 TB in memory
  → Completely infeasible

Better: store URL hashes in database (PostgreSQL/Cassandra)
  50B rows × 32 bytes = 1.6 TB on disk — feasible but slow
  Single DB lookup per URL: 50B × ~1ms = 50B ms = ~580 days → too slow

Best: Bloom filter for fast probabilistic check, DB for definitive check
```

### 4.2 Bloom filter

```
A Bloom filter answers: "Is this element in the set?"
with possible FALSE POSITIVES but NO FALSE NEGATIVES.

Properties:
  False positive (FP):  "yes" when actually not in set → re-check with DB
  False negative (FN):  IMPOSSIBLE — if element is in set, always returns "yes"

For URL dedup:
  FP consequence: briefly re-examine a URL in DB that was already seen → minor overhead
  FN consequence: crawl same URL twice → acceptable in small numbers

Space efficiency at 1% false positive rate:
  m = -n × ln(p) / (ln2)²
  m = -50B × ln(0.01) / (0.693)² = -50B × (-4.605) / 0.480 ≈ 479B bits = 60 GB
  vs 1.6 TB for HashSet → 26× more space-efficient

Optimal hash functions:
  k = (m/n) × ln(2) ≈ 7 hash functions for 1% FP rate
  Use MurmurHash3 or xxHash (fast, good distribution, non-cryptographic)
```

```java
@Service
public class DistributedBloomFilter {
    private static final long BIT_ARRAY_SIZE = 479_000_000_000L; // 60 GB in bits
    private static final int  HASH_COUNT = 7;

    // Kirsch-Mitzenmacher optimisation: two base hashes → k virtual hashes
    // h_i(x) = h1(x) + i * h2(x)  for i in [0, k)
    private long[] hash(String item) {
        long[] hashes = new long[HASH_COUNT];
        byte[] data = item.getBytes(StandardCharsets.UTF_8);
        long h1 = MurmurHash3.hash128(data, 0, data.length, 0)[0];
        long h2 = MurmurHash3.hash128(data, 0, data.length, 1)[0];
        for (int i = 0; i < HASH_COUNT; i++) {
            hashes[i] = Math.abs(h1 + i * h2) % BIT_ARRAY_SIZE;
        }
        return hashes;
    }

    public void add(String url) {
        long[] positions = hash(url);
        redis.executePipelined(conn -> {
            for (long pos : positions) {
                conn.bitCommands().setBit("bloom:filter", pos, true);
            }
            return null;
        });
    }

    public boolean mightContain(String url) {
        long[] positions = hash(url);
        List<Boolean> bits = redis.executePipelined(conn -> {
            for (long pos : positions) {
                conn.bitCommands().getBit("bloom:filter", pos);
            }
            return null;
        }).stream().map(b -> (Boolean) b).collect(Collectors.toList());

        return bits.stream().allMatch(b -> b);
    }
}
// Redis bit array sharding: a single Redis string key holds max 512 MB.
// For 60 GB: shard across 120 keys using position/shard_size to select shard key.
```

### 4.3 URL normalisation before dedup

```java
// Two URLs may refer to the same page but look different.
// Normalise before hashing to maximise dedup effectiveness.
public String normalise(String rawUrl) throws MalformedURLException {
    URL url = new URL(rawUrl);

    // 1. Lowercase scheme and host
    String scheme = url.getProtocol().toLowerCase();
    String host   = url.getHost().toLowerCase();

    // 2. Remove default port (http:80, https:443)
    int port = url.getPort();
    if ((scheme.equals("http")  && port == 80)  ||
        (scheme.equals("https") && port == 443)) port = -1;

    // 3. Decode percent-encoding then re-encode consistently
    String path = normalisePath(url.getPath());

    // 4. Sort query parameters alphabetically
    String query = sortQueryParams(url.getQuery());

    // 5. Remove tracking/analytics parameters
    query = removeParams(query, Set.of("utm_source", "utm_medium", "utm_campaign",
                                        "fbclid", "gclid", "ref", "source"));

    // 6. Remove fragment (#anchor — not sent to server, irrelevant for crawl)

    // 7. Remove trailing slash: /path/ → /path
    if (path.endsWith("/") && path.length() > 1)
        path = path.substring(0, path.length() - 1);

    return buildUrl(scheme, host, port, path, query);
}
// Before: HTTP://Example.COM:80/page?b=2&a=1&utm_source=tw#section
// After:  http://example.com/page?a=1&b=2
```

### 4.4 Content-level dedup with SimHash

```
Problem: different URLs may serve identical content (mirrors, syndicated articles).
  URL dedup passes both, but they generate duplicate indexed content.

Solution: SimHash fingerprints document content.
  Similar documents → similar (Hamming-close) fingerprints.
  Exact duplicates → identical fingerprint.
  Near-duplicates → fingerprint differs by ≤ 3 bits.

SimHash algorithm (64-bit):
  1. Tokenise document into words (with tf-idf weights)
  2. Hash each token to 64-bit value
  3. For each bit position i: accumulate +weight if hash bit i = 1, else -weight
  4. Final fingerprint: bit i = 1 if sum[i] > 0, else 0

Near-duplicate lookup at scale (pigeonhole principle):
  Split 64-bit fingerprint into 4 × 16-bit segments.
  Any two fingerprints with Hamming distance ≤ 3 must match exactly
  in at least one of the 4 segments.
  Maintain 4 hash tables, one per segment → 4 O(1) lookups per document.
  Storage: 50B documents × 8 bytes fingerprint = 400 GB across cluster.
```

## 5. Politeness Policy

### 5.1 Why politeness matters

```
Aggressive crawling harms target websites:
  - Overloads servers with unexpected traffic spikes
  - Consumes their bandwidth (CDN egress costs)
  - May trigger DDoS protection, blocking the crawler permanently
  - Legal exposure: Terms of Service violations

Politeness rules:
  1. Respect robots.txt (disallow rules, Crawl-delay directive)
  2. Default minimum inter-request delay per host (1 second)
  3. Back off exponentially on 429 (Too Many Requests) or 503
  4. Identify via User-Agent header (e.g. "MyCrawler/1.0 (+https://example.com/bot)")
  5. Respect Retry-After header on 429/503
  6. Do not crawl login-walled content
```

### 5.2 robots.txt parsing and caching

```java
@Service
public class RobotsTxtService {
    // Cache per domain — robots.txt rarely changes
    private final Cache<String, RobotRules> cache = Caffeine.newBuilder()
        .maximumSize(5_000_000)           // 5M domains in memory
        .expireAfterWrite(24, HOURS)
        .build();

    public RobotRules getRules(String domain) {
        return cache.get(domain, this::fetchAndParse);
    }

    private RobotRules fetchAndParse(String domain) {
        try {
            String robotsUrl = "https://" + domain + "/robots.txt";
            HttpResponse<String> resp = httpClient.get(robotsUrl, 5_000); // 5s timeout

            if (resp.statusCode() == 404) return RobotRules.ALLOW_ALL;
            if (resp.statusCode() != 200)  return RobotRules.ALLOW_ALL; // liberal on error

            return RobotsTxtParser.parse(resp.body(), OUR_USER_AGENT);
        } catch (Exception e) {
            return RobotRules.ALLOW_ALL; // network error: proceed (be liberal)
        }
    }

    public boolean isAllowed(String url) {
        return getRules(extractDomain(url)).isAllowed(url);
    }

    public long getCrawlDelayMs(String domain) {
        long directiveMs = getRules(domain).getCrawlDelayMs();
        return Math.max(directiveMs, DEFAULT_MIN_DELAY_MS); // enforce our minimum
    }

    // robots.txt example handled:
    // User-agent: *
    // Disallow: /private/
    // Crawl-delay: 10         ← honoured: wait 10s between requests
    // Allow: /public/         ← takes precedence over Disallow for matching paths
}
```

### 5.3 Politeness scheduler

```java
@Service
public class PolitenessScheduler {
    // Redis key: politeness:{domain} → timestamp of last successful fetch
    // TTL: 5 minutes (auto-cleanup for inactive domains)

    public boolean canFetch(String domain) {
        String lastFetchStr = redis.get("politeness:" + domain);
        if (lastFetchStr == null) return true;

        long elapsed = System.currentTimeMillis() - Long.parseLong(lastFetchStr);
        return elapsed >= robotsTxtService.getCrawlDelayMs(domain);
    }

    public void recordFetch(String domain) {
        redis.set("politeness:" + domain,
                  String.valueOf(System.currentTimeMillis()),
                  300, TimeUnit.SECONDS);
    }

    // Exponential backoff on error
    public long computeBackoffMs(int httpStatus, int consecutiveErrors) {
        if (httpStatus == 429 || httpStatus == 503) {
            return Math.min(
                300_000L,                                        // max 5-minute backoff
                1_000L * (1L << consecutiveErrors)               // 1s, 2s, 4s, 8s...
            );
        }
        return DEFAULT_MIN_DELAY_MS;
    }
}
```

### 5.4 Domain queue selector

```java
@Component
public class DomainQueueSelector {

    public Optional<CrawlTask> selectNext() {
        // Redis sorted set "ready_domains": score = priority of top URL per domain
        // Only domains whose politeness delay has elapsed are in this set
        Set<String> candidates = redis.zrevrangeByScore(
            "ready_domains", Double.MAX_VALUE, 0, 0, 10
        );

        for (String domain : candidates) {
            if (politenessScheduler.canFetch(domain)) {
                CrawlTask task = domainQueue.pop(domain);
                if (task != null) {
                    politenessScheduler.recordFetch(domain);
                    return Optional.of(task);
                }
            }
        }
        return Optional.empty(); // no ready domain — sleep briefly
    }
}
```

## 6. Distributed Crawling

### 6.1 Partitioning work across crawler nodes

```
Challenge: 100 crawler nodes must not crawl the same URLs simultaneously.
Also: all URLs for a given domain should go to the same node for accurate
per-domain politeness tracking without cross-node coordination.

Solution: consistent hashing of domain → crawler node
  hash(domain) % NUM_NODES → assigned crawler node

  "cnn.com"    → hash → node 7   → node 7 handles ALL cnn.com URLs
  "bbc.com"    → hash → node 23  → node 23 handles ALL bbc.com URLs

Benefits:
  Politeness state (last fetch time) is LOCAL to one node → no distributed lock
  Adding/removing nodes: consistent hashing minimises URL reassignment
  Natural domain isolation: one slow domain (crawl-delay=10) doesn't block others

Drawbacks:
  Hot domains (many URLs) create hot nodes.
  Mitigation: split very large domains across multiple virtual nodes.

Implementation: Kafka topics partitioned by domain hash.
  Producer key = domain → deterministic partition assignment.
  Each crawler node consumes its assigned partitions.
```

### 6.2 Crawler node architecture

```java
@Component
public class CrawlerNode {

    @Async
    public void crawlLoop() {
        while (running) {
            Optional<CrawlTask> taskOpt = frontier.pollMyDomains(nodeId);
            if (taskOpt.isEmpty()) { sleep(100); continue; }

            CrawlTask task = taskOpt.get();

            // 1. Check robots.txt
            if (!robotsTxtService.isAllowed(task.getUrl())) {
                seenUrlStore.markDisallowed(task.getUrlHash());
                continue;
            }

            // 2. Politeness check
            if (!politenessScheduler.canFetch(task.getDomain())) {
                frontier.requeue(task);
                continue;
            }

            // 3. Fetch
            FetchResult result = fetcher.fetch(task.getUrl());
            politenessScheduler.recordFetch(task.getDomain());

            // 4. Route on response status
            switch (result.getStatus()) {
                case OK_200          -> processSuccess(task, result);
                case REDIRECT_3XX    -> handleRedirect(task, result);
                case TOO_MANY_429,
                     UNAVAILABLE_503 -> requeueWithBackoff(task, result);
                case NOT_FOUND_404,
                     GONE_410        -> markDead(task);
                default              -> requeueWithBackoff(task, result);
            }
        }
    }

    private void processSuccess(CrawlTask task, FetchResult result) {
        // a. Store raw HTML (content-addressed: sha256(url) as key)
        objectStore.put(task.getUrlHash(), result.getBody());

        // b. Parse content
        ParsedPage page = parser.parse(result.getBody(), task.getUrl());

        // c. Content dedup via SimHash
        long fingerprint = simHashService.fingerprint(page.getTextContent());
        if (simHashStore.isDuplicate(fingerprint)) {
            seenUrlStore.markDuplicate(task.getUrlHash(), fingerprint);
            return;
        }
        simHashStore.add(fingerprint, task.getUrlHash());

        // d. Persist parsed content
        contentStore.save(page);

        // e. Enqueue extracted links
        for (String link : page.getExtractedLinks()) {
            String norm = urlNormaliser.normalise(link);
            frontier.enqueue(norm, priorityScorer.score(norm, task));
        }

        // f. Update crawl record
        seenUrlStore.markCrawled(task.getUrlHash(), Instant.now());
    }
}
```

### 6.3 Fault tolerance

```
Failure scenarios and mitigations:

1. Crawler node crashes mid-fetch:
   Mitigation: Kafka consumer manual offset commit.
   Offset committed ONLY after successful processing.
   On restart: consumer resumes from last committed offset → at-least-once.

2. Partial pipeline failure (HTML stored, link extraction failed):
   Raw HTML in S3 is durable; link extraction re-runnable from S3 content.
   All stages are idempotent: same url_hash → same S3 key (overwrite safe),
   content DB upsert on url_hash primary key.

3. DNS failure:
   Cache negative results (NXDOMAIN) for 5 minutes.
   Mark domain temporarily unavailable; requeue with backoff.

4. Kafka broker failure:
   Replication factor = 3; tolerate 2 broker failures.
   Consumer group rebalances automatically → other nodes take over partitions.

5. Bloom filter Redis failure:
   Fall back to DB-only dedup (slower but correct).
   Bloom filter reconstructible from seenUrlStore (rebuild job: ~1 hour).

6. Politeness state lost (Redis restart):
   Crawlers temporarily over-fetch for ~60 seconds until state rebuilds.
   Acceptable: each domain gets at most one extra request during recovery.
```

## 7. Content Parsing

### 7.1 HTML parser pipeline

```java
@Service
public class ContentParser {

    public ParsedPage parse(byte[] rawContent, String url) {
        // 1. Detect encoding (charset from HTTP header or HTML meta tag)
        Charset charset = charsetDetector.detect(rawContent);
        String html = new String(rawContent, charset);

        // 2. Parse HTML (Jsoup: lenient, handles malformed HTML)
        Document doc = Jsoup.parse(html, url);

        // 3. Check meta robots tag
        String metaRobots = doc.select("meta[name=robots]").attr("content").toLowerCase();
        boolean indexable  = !metaRobots.contains("noindex");
        boolean followable = !metaRobots.contains("nofollow");

        // 4. Extract canonical URL (de-duplication hint from site owner)
        String canonical = doc.select("link[rel=canonical]").attr("href");

        // 5. Extract structured metadata
        String title    = doc.title();
        String desc     = doc.select("meta[name=description]").attr("content");
        String lang     = doc.select("html").attr("lang");

        // 6. Extract clean text (strip non-content elements)
        doc.select("script, style, nav, footer, header, aside, .ad, .advertisement").remove();
        String textContent = doc.body() != null ? doc.body().text() : "";

        // 7. Extract outbound links
        List<String> links = Collections.emptyList();
        if (followable) {
            links = doc.select("a[href]").stream()
                .map(el -> el.absUrl("href"))
                .filter(href -> !href.isEmpty() && isValidUrl(href))
                .filter(href -> !isBlacklisted(href))
                .distinct()
                .collect(Collectors.toList());
        }

        return ParsedPage.builder()
            .url(url)
            .canonicalUrl(canonical.isEmpty() ? url : canonical)
            .title(title).description(desc).language(lang)
            .textContent(textContent)
            .outboundLinks(links)
            .indexable(indexable)
            .build();
    }
}
```

### 7.2 URL filtering

```java
private boolean isValidUrl(String url) {
    try {
        URI uri = new URI(url);
        if (!Set.of("http", "https").contains(uri.getScheme())) return false;
        if (uri.getHost() == null) return false;
        if (url.length() > 2048) return false; // likely spam / tracking
        return true;
    } catch (URISyntaxException e) {
        return false;
    }
}

private boolean isBlacklisted(String url) {
    if (domainBlacklist.contains(extractDomain(url))) return true;
    // Skip binary/media file extensions — not useful for text indexing
    return url.toLowerCase().matches(
        ".*\\.(jpg|jpeg|png|gif|ico|css|js|woff|woff2|mp4|avi|zip|exe)$"
    );
}
```

### 7.3 Redirect handling

```java
private void handleRedirect(CrawlTask task, FetchResult result) {
    String location = result.getLocationHeader();
    if (location == null) { markDead(task); return; }

    String target = urlNormaliser.normalise(resolveRelative(task.getUrl(), location));

    // Detect redirect loops and limit chain depth
    List<String> chain = task.getRedirectChain();
    if (chain.size() >= 5 || chain.contains(target)) {
        log.warn("Redirect loop or depth exceeded: {}", chain);
        markDead(task);
        return;
    }

    // Store canonical mapping: original → final
    seenUrlStore.storeRedirect(task.getUrlHash(), target);

    frontier.enqueue(
        task.withUrl(target).appendToChain(target)
    );
}
```

## 8. Re-crawl Scheduling

### 8.1 Adaptive re-crawl frequency

```
Not all pages change at the same rate:
  News articles:  change frequently in first hour, then static
  Product pages:  price/stock changes daily
  Wikipedia:      updated occasionally
  About Us pages: changes yearly

Strategy: exponential moving average of observed change intervals.
  change_interval[n] = 0.7 × change_interval[n-1] + 0.3 × observed_interval[n]
  next_crawl = last_crawl + change_interval[n] × 0.9 (crawl slightly early)

No change observed → back off: interval × 1.2 (page is more stable than estimated)
Change observed    → decrease interval (page is more volatile than estimated)
```

```java
@Service
public class RecrawlScheduler {

    @Scheduled(fixedRate = 60_000) // every minute
    public void scheduleRecrawls() {
        List<CrawlRecord> due = crawlRepository.findDueForRecrawl(
            Instant.now(), PageRequest.of(0, 10_000)
        );
        for (CrawlRecord record : due) {
            frontier.enqueue(record.getUrl(), computeRecrawlPriority(record));
        }
    }

    public void updateChangeEstimate(String urlHash, boolean changed, Instant crawledAt) {
        CrawlRecord r = crawlRepository.findByHash(urlHash);
        long observedHours = ChronoUnit.HOURS.between(r.getLastCrawledAt(), crawledAt);

        double updated = changed
            ? 0.7 * r.getEstimatedChangeIntervalHours() + 0.3 * observedHours
            : r.getEstimatedChangeIntervalHours() * 1.2; // stable: back off

        r.setEstimatedChangeIntervalHours(updated);
        r.setLastCrawledAt(crawledAt);
        r.setNextCrawlAt(crawledAt.plus((long) (updated * 0.9), ChronoUnit.HOURS));
        crawlRepository.save(r);
    }
}
```

## 10. Spider Trap Detection

```
Spider traps generate infinite distinct URLs pointing to the same/similar content.

Common examples:
  Calendar navigation:  /2024/01, /2024/02, ... /2099/12 (infinite)
  Session IDs:          ?session=<random-uuid> (unique per visit)
  Infinite pagination:  /page/1 → /page/2 → /page/3 → ...
  Faceted search:       /products?color=red&size=M&brand=X (exponential combinations)

Mitigations:

1. URL normalisation:
   Strip known tracking/session params (utm_*, session_id, PHPSESSID).
   Collapse UUID-like path segments.

2. Depth limiting:
   Max crawl depth from seed: 10-15 hops.
   Pages at depth > 15 are extremely unlikely to contain valuable content.

3. Per-domain URL count cap:
   If a single domain contributes > 1M URLs to the frontier: cap it.
   Log for manual review (may be a legitimate large site or a trap).

4. Path signature detection:
   Extract URL path template: /items/12345 → /items/{id}
   If template matches > 10,000 URLs from same domain: sample fractionally.

5. Content similarity threshold:
   If a domain's pages consistently have SimHash distance = 0 (all identical):
     reduce crawl priority to minimum; flag for review.
```

## 11. Summary Architecture

```
Write path (crawl pipeline):
  Seeds → Frontier.enqueue() → Bloom filter + DB dedup → Kafka frontier topics
  → Domain-consistent partition → Crawler node:
      robots.txt check → politeness check → HTTP fetch
      → SimHash dedup → S3 (raw HTML) + Cassandra (parsed content)
      → Extract links → Frontier.enqueue() [loop]

Re-crawl path:
  PostgreSQL scheduler (next_crawl_at due) → Frontier.enqueue() with priority

Failure recovery:
  Kafka: at-least-once; all stages idempotent (S3 overwrite, DB upsert)
  Bloom filter lost: reconstructed from Cassandra seen-url store (~1 hour)
  Politeness state lost: self-heals in ~60 seconds of activity
  Node failure: Kafka rebalance → surviving nodes take over partitions

Downstream consumers:
  Search indexer → reads Cassandra content store → inverted index
  Link graph builder → reads extracted links → PageRank computation
  Analytics → reads crawl metadata → dashboard / freshness monitoring
```

## 12. Interview Questions

**Q: Why is a Bloom filter the right data structure for URL deduplication, and how do you handle its false positives?**

A: At 50 billion URLs, a HashSet requires roughly 1.6 TB of memory — impractical even across a cluster. A Bloom filter achieves the same deduplication with ~60 GB at a 1% false positive rate using a fixed-size bit array and k independent hash functions. The critical property for this use case is that Bloom filters have no false negatives: if a URL has been seen, the filter always says so. A false positive means we occasionally think a new URL has been seen when it hasn't, causing us to skip crawling it. The consequence is a small number of missed URLs — acceptable for a web crawler where completeness is best-effort. To handle false positives, we use the Bloom filter as a first-pass fast filter. On a "mightContain" result, we confirm against a persistent Cassandra store before definitively skipping the URL. This two-phase check means false positives add one extra DB lookup per affected URL but never cause genuinely new URLs to be permanently skipped. The DB is the ground truth; the Bloom filter is the speed optimisation sitting in front of it.

---

**Q: Explain the two-tier queue design and why a single priority queue is insufficient.**

A: A single priority queue fails on two fronts simultaneously. Even with priority ordering, nothing prevents the queue from containing thousands of URLs for the same domain in sequence — a crawler would hammer that domain with thousands of requests before visiting any other, violating politeness and likely getting the crawler IP blocked. Conversely, a single per-domain FIFO queue provides no prioritisation: a newly discovered link to a major news site sits behind a deep link discovered earlier, even though the news site should be crawled first. The two-tier design separates these concerns cleanly. The first tier — priority queues — ensures pages are ordered by estimated importance. The second tier — per-domain FIFO queues — meters requests to each domain with the required delay. The selector dispatches from domain queues in descending priority order, but only when a domain's politeness delay has elapsed. Consistent hashing of domain to crawler node means each node's politeness tracking is entirely local, eliminating distributed lock contention. The invariant — a domain's URLs are always assigned to the same node — is what makes local politeness state coherent.

---

**Q: How does consistent hashing of domains to crawler nodes handle node failures and additions?**

A: Consistent hashing maps each domain to a position on a virtual ring by hashing the domain name. Each physical node owns a contiguous arc of the ring. When a node fails, its arc is absorbed by its neighbour — only the domains previously assigned to the failed node are remapped, roughly 1/N of all domains for an N-node cluster. Kafka consumer groups handle this automatically: the failed node's partitions are rebalanced to surviving nodes within seconds. The politeness state for remapped domains (stored in Redis as last-fetch timestamps) may be temporarily unavailable, causing brief over-fetching until those domains' timestamps re-register — at worst, one extra request per domain during the rebalance window, an acceptable trade-off. When a new node joins, consistent hashing assigns it a fraction of the ring, gradually redistributing load as Kafka partitions are reassigned. The key invariant — one node owns all URLs for a given domain at any moment — holds except during the brief rebalance window, which is a distributed systems reality accepted in exchange for the elimination of cross-node politeness locks.

---

**Q: How would you detect and prevent spider traps from consuming your entire crawl budget?**

A: Spider traps exploit the fact that the crawler follows all links recursively. A well-designed trap can generate an infinite URL space — calendar pages, session-parameterised URLs, deeply nested faceted search. Defence requires multiple layers. URL normalisation is the first and most powerful: stripping known tracking and session parameters (utm_source, PHPSESSID, UUID-like path segments) before hashing collapses many "distinct" trap URLs to the same canonical form. Depth limiting caps recursion at 10-15 hops from seed URLs; content of genuine value is almost never found that deep. Per-domain URL count caps detect sites contributing disproportionately to the frontier — a domain generating over 1 million frontier entries likely has structural issues. Path signature detection extracts URL templates (replacing numeric IDs and UUIDs with placeholders) and identifies when a domain produces thousands of pages matching the same template; we then sample fractionally rather than crawling all of them. Content similarity monitoring via SimHash catches traps where all pages are near-identical despite different URLs. These mitigations work in combination because different traps require different countermeasures — no single technique catches all forms.

---

**Q: How does SimHash enable near-duplicate content detection at web scale, and why is Hamming distance lookup fast?**

A: Exact deduplication via SHA-256 hash catches byte-for-byte identical pages but misses the common case: syndicated articles with different ads inserted, mirrored sites with different navigation, or content with minor updates. SimHash reduces a document to a 64-bit fingerprint where the Hamming distance between two fingerprints correlates with the semantic similarity of their documents — similar documents have fingerprints differing by only a few bits. After computing the fingerprint of a new document, we need to check whether any of the 50 billion previously seen fingerprints have Hamming distance ≤ 3. Naively comparing against all stored fingerprints is O(n) — infeasible. The pigeonhole principle provides the solution: divide the 64-bit fingerprint into 4 segments of 16 bits each. Any two fingerprints with Hamming distance ≤ 3 must have at least one segment that matches exactly (since only 3 bits can differ, at least one of the 4 segments must be unmodified). We maintain 4 hash tables, one per segment, where the key is the 16-bit segment value and values are sets of fingerprints sharing that segment. A lookup requires 4 hash table queries — effectively O(1) — instead of O(n) linear scan, making near-duplicate detection practical at billions of documents.

## 13. Key Takeaways

- **Two-tier frontier = priority + politeness separated**: tier 1 sorts by importance; tier 2 meters per-domain with FIFO — cannot combine these into one queue
- **Bloom filter properties**: no false negatives (critical); false positives handled by DB confirmation pass; 60 GB for 50B URLs at 1% FP rate vs 1.6 TB HashSet
- **URL normalisation before hashing**: lowercasing, default port removal, query param sorting, tracking param stripping, fragment removal — collapses 20-30% of "new" URLs to already-seen ones
- **Consistent hashing of domain → node**: local politeness state (no distributed lock); 1/N domain remapping on failure; Kafka consumer groups handle rebalancing automatically
- **robots.txt**: cached per domain (Caffeine, 24h TTL); 404 → ALLOW_ALL; network error → ALLOW_ALL (be liberal); honour both Disallow and Crawl-delay
- **SimHash near-duplicate detection**: 64-bit fingerprint; Hamming distance ≤ 3 = near-duplicate; 4-table pigeonhole lookup = O(1) search across 50B fingerprints
- **Idempotent pipeline for fault tolerance**: S3 overwrite on same key is safe; Cassandra upsert on url_hash; Bloom filter add is idempotent; Kafka at-least-once acceptable
- **Spider trap mitigations**: URL normalisation (stripping session params), depth limiting (10-15 hops), per-domain URL caps, path signature sampling, SimHash content similarity monitoring
- **Adaptive re-crawl**: EMA of observed change intervals; no change → back off 20%; change observed → decrease interval; high-importance pages checked more frequently
- **Fail-open on robots.txt fetch errors**: network errors → ALLOW_ALL; only explicit Disallow rules in a successfully fetched robots.txt restrict crawling
