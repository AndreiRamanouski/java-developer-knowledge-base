# Video Streaming Platform Design (YouTube / Netflix)

## Overview

A video streaming platform ingests raw video uploads, transcodes them into multiple formats and resolutions, distributes the content globally via CDN, streams it adaptively based on viewer bandwidth, and personalises recommendations. YouTube processes 500 hours of video uploaded per minute; Netflix serves 15% of global internet traffic; both require deeply specialised infrastructure at every layer.

**The core pipeline:**

```
Upload (raw video) → Transcoding (multiple formats/resolutions)
→ CDN distribution (global edge nodes)
→ Adaptive streaming (client picks quality in real-time)
→ View tracking → Recommendation engine → Personalised home screen

Each stage has distinct scaling challenges:
  Upload:         large binary blobs; resumable; metadata extraction
  Transcoding:    CPU-intensive; fan-out to parallel workers; hours per video
  CDN:            cache hit rate; cache eviction; origin shield
  Adaptive:       HLS/DASH segment generation; bandwidth estimation; ABR algorithm
  View count:     high write rate; eventual consistency acceptable; fraud detection
  Recommendations: ML inference at query time; pre-computation for popular users
```

## 1. Requirements Gathering

### 1.1 Clarifying questions

```
1. "Is this user-generated content (YouTube) or licensed content (Netflix)?"
   → User-generated; anyone can upload; moderation required.

2. "What video qualities should we support?"
   → 360p through 4K. Mobile-aware adaptive streaming.

3. "What is the expected scale — uploads per day, concurrent viewers?"
   → 1M video uploads/day; 1B daily active viewers; 5M concurrent streams peak.

4. "Do we need live streaming or video-on-demand only?"
   → Video-on-demand first. Live streaming as an extension.

5. "What are the monetisation requirements — ads, subscriptions, pay-per-view?"
   → Ad-supported free tier + paid ad-free subscription. Ads out of scope for design.

6. "What content moderation is needed?"
   → Automated content ID (copyright detection) + abuse detection. Separate pipeline.

7. "Do we need video chapters, subtitles, multi-audio tracks?"
   → Subtitles/captions yes (accessibility). Chapters yes (metadata). Multiple audio tracks optional.

8. "What view count guarantees? Real-time accurate or eventually consistent?"
   → Eventual consistency with < 1-minute lag is acceptable for display.
      Exact counts required for monetisation (settled within 24 hours).
```

### 1.2 Confirmed requirements

```
Functional:
  ✓ Upload video (up to 128 GB per file)
  ✓ Transcode to multiple formats (360p, 480p, 720p, 1080p, 1440p, 2160p/4K)
  ✓ Adaptive bitrate streaming (HLS / DASH)
  ✓ Video search by title/description/tags
  ✓ Like, comment, subscribe
  ✓ View count (eventually consistent display; exact for billing)
  ✓ Recommendations (personalised home feed + related videos sidebar)
  ✓ Subtitles / closed captions
  ✓ Version history / re-upload (replace video content)

Non-functional:
  ✓ 1M video uploads/day (~12 uploads/second)
  ✓ 1B DAV (daily active viewers); 5M concurrent streams peak
  ✓ Video start latency (time to first frame): < 2 seconds globally
  ✓ Buffering: < 1% of playback time
  ✓ Upload → available for viewing: < 10 minutes for 1080p (priority queue)
  ✓ 99.99% availability for video playback
  ✓ Durability: 12 nines for stored video content

Out of scope:
  ✗ Live streaming
  ✗ Ad insertion / SSAI (server-side ad insertion)
  ✗ Content ID / copyright matching pipeline
```

### 1.3 Scale estimation

```
Storage:
  1M uploads/day × 1 GB average raw video = 1 PB/day raw
  Transcoded (6 resolutions × ~0.3× compression) = ~1.8 PB/day transcoded
  5-year retention: 1.8 PB/day × 365 × 5 = ~3.3 EB total (use tiered storage)

Bandwidth for streaming:
  5M concurrent streams × 5 Mbps average = 25 Tbps peak outbound
  CDN absorbs >99% of this; origin serves < 1% (cache misses + new content)

Transcoding compute:
  1M uploads/day; average 10-minute video at 1080p 60fps
  Transcoding ratio: real-time factor ~10-50× (1 min video → 10-50 min CPU)
  Per video: 6 resolutions × average 20-min CPU = 120 CPU-minutes/video
  1M videos × 120 CPU-min = 120M CPU-minutes/day = 83,333 CPU-hours/day
  → ~3,500 transcode workers needed at 24/7 utilisation (use spot instances)

View events:
  1B DAV × 20 views/day average = 20B view events/day
  = 231,000 view events/second average; peak ~700K/second
```

## 2. Video Upload Pipeline

### 2.1 Resumable chunked upload

```
Video files are large (up to 128 GB). Network interruption mid-upload must not
require restarting from zero. Use the same chunked upload pattern as file storage,
adapted for video specifics.

Chunk size: 10 MB (larger than file sync — video chunks benefit from larger sizes
because we do not need byte-level delta sync; each upload is a fresh video).

Upload flow:
  1. Client: POST /api/v1/videos/upload/init
     Body: {title, description, fileSize, mimeType, fileSha256, tags}
     Response: {videoId, uploadId, chunkCount, presignedUrls: [...]}

  2. Client: PUT presignedUrl[i] for each chunk (direct to S3)
     Can parallelise up to 8 concurrent chunk uploads.
     Each chunk upload is idempotent: same chunk re-uploaded = same S3 key.

  3. Client: POST /api/v1/videos/upload/{uploadId}/complete
     Server: triggers transcoding pipeline, returns {videoId, status:"processing"}

  4. Status polling / WebSocket:
     GET /api/v1/videos/{videoId}/status
     Returns: {status:"transcoding", progress:45, availableResolutions:["360p","480p"]}
     Lower resolutions published first; 1080p+ takes longer.
```

### 2.2 Raw video storage

```
After upload: raw video stored in S3 "raw" bucket.
Content-addressed: key = sha256(video_content) (or videoId for simplicity).

Raw video is:
  - Input to transcoding pipeline (read once, never again after transcode)
  - Backup for re-transcoding if codecs/formats change
  - Moved to Glacier after 90 days (re-transcoding is rare)

Transcoded segments stored in separate S3 "transcoded" bucket:
  s3://videos-transcoded/{videoId}/{resolution}/{segmentNumber}.ts  (HLS)
  s3://videos-transcoded/{videoId}/{videoId}.m3u8                   (HLS master playlist)
  s3://videos-transcoded/{videoId}/{resolution}/{segmentNumber}.m4s (DASH)
  s3://videos-transcoded/{videoId}/{videoId}.mpd                    (DASH manifest)
```

### 2.3 Metadata extraction on upload

```java
@Service
public class VideoMetadataExtractor {

    // Called after upload completes, before transcoding starts
    public VideoMetadata extract(String rawVideoPath) throws IOException {
        // Use FFprobe to extract technical metadata without decoding
        ProcessBuilder pb = new ProcessBuilder(
            "ffprobe", "-v", "quiet",
            "-print_format", "json",
            "-show_format", "-show_streams",
            rawVideoPath
        );
        Process p = pb.start();
        JsonNode root = objectMapper.readTree(p.getInputStream());

        JsonNode videoStream = findVideoStream(root);
        JsonNode audioStream = findAudioStream(root);
        JsonNode format      = root.path("format");

        return VideoMetadata.builder()
            .durationSeconds(format.path("duration").asDouble())
            .fileSizeBytes(format.path("size").asLong())
            .codec(videoStream.path("codec_name").asText())        // h264, hevc, vp9...
            .width(videoStream.path("width").asInt())
            .height(videoStream.path("height").asInt())
            .frameRate(parseFrameRate(videoStream.path("r_frame_rate").asText())) // "30/1"
            .bitrate(format.path("bit_rate").asLong())
            .hasAudio(audioStream != null)
            .audioCodec(audioStream != null ? audioStream.path("codec_name").asText() : null)
            .build();
    }
}

## 3. Transcoding Service

### 3.1 Why transcoding is necessary

```
Raw uploads arrive in arbitrary formats:
  Codec:      H.264, H.265/HEVC, AV1, VP9, ProRes, DNxHD, MOV, AVI...
  Resolution: anything from 240p phone video to 8K professional
  Container:  MP4, MOV, MKV, AVI, WebM...
  Bitrate:    10 Mbps consumer to 400+ Mbps professional footage
  Frame rate: 24, 30, 60, 120 fps

Transcoding normalises all of this into:
  Standard codecs:    H.264 (universal compat), H.265/HEVC (high efficiency),
                      AV1 (open, efficient — Chrome, Firefox, Android)
  Standard containers: MPEG-TS (.ts) for HLS; fMP4 (.m4s) for DASH
  Standard resolutions: 360p, 480p, 720p, 1080p, 1440p, 2160p
  Fixed GOP (Group of Pictures) size: keyframe every 2 seconds → segment boundaries
  Standard bitrates:  per ladder below
```

### 3.2 Encoding ladder (bitrate per resolution)

```
Resolution   Codec      Video bitrate   Audio bitrate   Total bitrate
──────────────────────────────────────────────────────────────────────
360p         H.264      0.5 Mbps        128 kbps        ~0.63 Mbps
480p         H.264      1.0 Mbps        128 kbps        ~1.13 Mbps
720p 30fps   H.264      2.5 Mbps        192 kbps        ~2.69 Mbps
720p 60fps   H.264      3.5 Mbps        192 kbps        ~3.69 Mbps
1080p 30fps  H.264      5.0 Mbps        192 kbps        ~5.19 Mbps
1080p 60fps  H.264/HEVC 8.0 Mbps        192 kbps        ~8.19 Mbps
1440p        HEVC       12.0 Mbps       320 kbps        ~12.3 Mbps
2160p/4K     HEVC/AV1   25.0 Mbps       320 kbps        ~25.3 Mbps

Per-title encoding (Netflix approach):
  Instead of fixed bitrates, analyse complexity of each video.
  Simple animation at 1080p may need only 1 Mbps (H.264 encodes flat areas cheaply).
  Complex action film at 1080p may need 8 Mbps.
  Per-title saves 20-50% bandwidth vs fixed ladder.
```

### 3.3 Transcoding pipeline architecture

```
Transcoding is the most compute-intensive stage. Design as an async,
distributed job pipeline with parallel processing of each rendition.

### 3.4 Job scheduler and parallel processing

```java
@Service
public class TranscodeJobScheduler {

    public void scheduleTranscoding(VideoUploadedEvent event) {
        VideoMetadata meta = metadataExtractor.extract(event.getRawS3Path());
        videoRepository.updateMetadata(event.getVideoId(), meta);

        // Determine which renditions to produce based on source resolution
        List<TranscodeJob> jobs = buildRenderLadder(meta);

        // Publish all jobs to Kafka — workers pick them up in parallel
        for (TranscodeJob job : jobs) {
            kafkaTemplate.send("transcode-jobs",
                               event.getVideoId(),  // partition key: same video, same worker group
                               job);
        }

        // Track overall video processing status
        videoStatusService.initProcessing(event.getVideoId(), jobs.size());
    }

    private List<TranscodeJob> buildRenderLadder(VideoMetadata meta) {
        List<TranscodeJob> jobs = new ArrayList<>();
        // Always produce 360p (lowest quality, available first)
        jobs.add(TranscodeJob.of(meta.getVideoId(), Resolution.P360, Priority.HIGH));
        jobs.add(TranscodeJob.of(meta.getVideoId(), Resolution.P480, Priority.HIGH));
        if (meta.getHeight() >= 720) {
            jobs.add(TranscodeJob.of(meta.getVideoId(), Resolution.P720, Priority.HIGH));
        }
        if (meta.getHeight() >= 1080) {
            jobs.add(TranscodeJob.of(meta.getVideoId(), Resolution.P1080, Priority.NORMAL));
        }
        if (meta.getHeight() >= 1440) {
            jobs.add(TranscodeJob.of(meta.getVideoId(), Resolution.P1440, Priority.LOW));
        }
        if (meta.getHeight() >= 2160) {
            jobs.add(TranscodeJob.of(meta.getVideoId(), Resolution.P2160, Priority.LOW));
        }
        return jobs;
    }
}

@KafkaListener(topics = "transcode-jobs")
public class TranscodeWorker {

    public void process(TranscodeJob job) {
        String inputPath = s3Service.downloadToLocal(job.getRawS3Path()); // stream to disk

        // Run FFmpeg for this specific rendition
        FFmpegResult result = ffmpegExecutor.transcode(
            FFmpegCommand.builder()
                .input(inputPath)
                .videoCodec("libx264")           // H.264 for up to 1080p
                .videoCodec(job.getResolution().isHigh() ? "libx265" : "libx264")
                .resolution(job.getResolution().getDimensions())
                .videoBitrate(job.getResolution().getBitrate())
                .audioBitrate("192k")
                .hlsSegmentDuration(2)            // 2-second segments (HLS keyframe interval)
                .hlsSegmentFilename(buildSegmentPath(job))
                .output(buildOutputPath(job))
                .build()
        );

        // Upload transcoded segments to S3
        s3Service.uploadDirectory(result.getOutputDir(),
            "s3://videos-transcoded/" + job.getVideoId() + "/" + job.getResolution());

        // Notify completion
        videoStatusService.markRenditionComplete(job.getVideoId(), job.getResolution());

        // If this is the first rendition (360p) → make video publicly available immediately
        if (job.getResolution() == Resolution.P360) {
            videoPublisher.makeAvailable(job.getVideoId());
        }
    }
}
```

### 3.5 HLS segment generation and manifest creation

```
HLS (HTTP Live Streaming) — Apple standard, universally supported:

Output per resolution (e.g. 720p):
  segment_0000.ts   (2-second MPEG-TS segment, 0:00 - 0:02)
  segment_0001.ts   (0:02 - 0:04)
  segment_0002.ts   (0:04 - 0:06)
  ...
  720p.m3u8         (rendition playlist — lists all segments)

Master playlist (videoId.m3u8):
  #EXTM3U
  #EXT-X-STREAM-INF:BANDWIDTH=630000,RESOLUTION=640x360
  360p/360p.m3u8
  #EXT-X-STREAM-INF:BANDWIDTH=1130000,RESOLUTION=854x480
  480p/480p.m3u8
  #EXT-X-STREAM-INF:BANDWIDTH=2690000,RESOLUTION=1280x720
  720p/720p.m3u8
  #EXT-X-STREAM-INF:BANDWIDTH=5190000,RESOLUTION=1920x1080
  1080p/1080p.m3u8

Segment size (2 seconds at 5 Mbps for 1080p):
  5 Mbps × 2s = 10 Mb = 1.25 MB per segment

DASH (Dynamic Adaptive Streaming over HTTP) — MPEG standard, used by Netflix:
  Similar concept; uses .mpd manifest and .m4s (fragmented MP4) segments.
  Interchangeable from architecture perspective; HLS is more universally supported.
```

## 4. CDN Distribution

### 4.1 Why CDN is non-negotiable for video

```
Without CDN: all 5M concurrent streams → origin S3 + servers
  5M × 5 Mbps = 25 Tbps from one origin region
  Cross-continent latency: 150-300ms → unacceptable for video start time
  Origin S3 egress cost: $0.09/GB × 25 Tbps × 3600s/hr = $40.5M/hour!

With CDN: >99% of bytes served from edge nodes
  User in Tokyo served from Tokyo PoP (point of presence)
  Latency to edge: 5-20ms vs 150ms to US origin
  CDN caches popular segments permanently; only cache misses hit origin

CDN providers: Akamai (Netflix uses), CloudFront (YouTube-scale), Fastly, Cloudflare
Netflix: runs their own CDN called Open Connect — custom hardware appliances
         placed inside ISP networks. No CDN fees; optimised for their use case.
```

### 4.2 CDN cache hierarchy

```
Three-tier cache hierarchy:

Tier 1: Edge PoP (city-level)
  ~2,000 global PoPs (Akamai has 4,000+)
  Each PoP: 10-100 TB SSD storage
  Caches: popular segments (top 20% of videos = 80% of views)
  TTL: 24 hours for segments (immutable content — versioned by videoId+resolution)
  Cache miss: forward to mid-tier

Tier 2: Regional shield (origin shield)
  ~50 regional clusters (1 per continent/region)
  Each: 1-10 PB storage
  Purpose: consolidate cache misses from hundreds of edge PoPs
  → One region's misses hit the shield once, not origin hundreds of times
  TTL: 7 days for segments

Tier 3: Origin (S3)
  S3 serves only what regional shields miss
  Target: < 1% of requests reach origin
  S3 standard transfer: $0.09/GB but 99% avoided via CDN caching

Cache key design:
  Key: /videos/{videoId}/{resolution}/{segmentNumber}.ts
  Immutable: videoId+resolution+segment uniquely identifies unchanging content.
  When video is re-transcoded: new videoId issued → old CDN cache irrelevant (no purge needed).
  Manifests (.m3u8): short TTL (60 seconds) because they change during live events
                     and as new renditions become available post-upload.
```

// Video playback URL generation: direct client to nearest CDN edge
@GetMapping("/api/v1/videos/{videoId}/play")
public PlaybackResponse getPlaybackUrl(
        @PathVariable String videoId,
        @RequestHeader("CF-IPCountry") String country,   // Cloudflare header
        @RequestHeader("User-Agent") String userAgent) {

    Video video = videoService.getVideo(videoId);
    if (!video.isPublished()) throw new VideoNotFoundException(videoId);

    // Select CDN edge based on user geography
    String cdnBaseUrl = cdnRouter.selectEdge(country, userAgent);

    // Generate signed URL (prevents unauthorised access / hotlinking)
    String signedToken = cdnSigner.sign(videoId, Duration.ofHours(4));

    // Return master playlist URL — client fetches this, then individual segments
    String masterPlaylistUrl = cdnBaseUrl
        + "/videos/" + videoId + "/master.m3u8"
        + "?token=" + signedToken;

    return PlaybackResponse.builder()
        .videoId(videoId)
        .masterPlaylistUrl(masterPlaylistUrl)
        .thumbnailUrl(cdnBaseUrl + "/thumbs/" + videoId + "/thumb.jpg")
        .duration(video.getDurationSeconds())
        .build();
}

// CDN signed URL prevents:
//   - Hotlinking (embedding our video on other sites without auth)
//   - Unauthorised access to premium content
//   - URL sharing (token has viewer's session fingerprint)

## 5. Adaptive Bitrate Streaming

### 5.1 How ABR works

```
Problem: viewer's bandwidth fluctuates during playback.
  Starting at 1080p on slow connection → immediate buffering → bad experience.
  Staying at 360p on fast connection → poor quality unnecessarily.

Adaptive Bitrate (ABR) streaming:
  Player monitors download speed in real-time.
  Switches to higher quality when bandwidth is sufficient.
  Drops to lower quality when bandwidth degrades.
  All without interrupting playback (seamlessly, segment by segment).

HLS ABR flow:
  1. Player fetches master.m3u8 → sees all available resolutions and their bitrates.
  2. Player estimates current bandwidth (from segment download time).
  3. Player selects highest rendition below available bandwidth × safety factor.
  4. Fetches next segment at chosen quality.
  5. Monitors download speed of that segment; adjusts choice for next segment.
  6. Result: quality changes every 2 seconds, invisible to the viewer.
```

### 5.2 Buffer management

```
Pre-buffer (startup): fill 10-15 seconds of video in buffer before starting playback.
  Trade-off: longer pre-buffer = lower chance of rebuffering
             but higher startup latency.
  Target: 1-2 second startup time (player starts at 360p, upgrades while buffering).

Steady-state buffer target: 30-60 seconds ahead of playback position.
  Buffer > 60s: player stops downloading (bandwidth conserved).
  Buffer < 10s: player drops quality aggressively (rebuffering imminent).
  Buffer < 2s:  player pauses and shows spinner (rebuffering occurring).

Buffer state machine:
  LOADING → PRE_BUFFERING → PLAYING → BUFFERING (rebuffer) → PLAYING
                                ↓
                           QUALITY_UP (more bandwidth)
                                ↓
                           QUALITY_DOWN (less bandwidth)
```

```java
// Client-side ABR algorithm (simplified BOLA-like)
public class ABRController {
    private static final double SAFETY_MARGIN = 0.85; // use 85% of estimated bandwidth
    private static final int    BUFFER_TARGET_SEC = 30;
    private static final int    BUFFER_PANIC_SEC  = 5;

    private final BandwidthEstimator bandwidthEstimator;
    private final PlaybackBuffer     buffer;
    private final List<Rendition>    renditions; // sorted ascending by bitrate

    public Rendition selectNextSegment() {
        double availableBandwidthBps = bandwidthEstimator.estimate() * SAFETY_MARGIN;
        double bufferLevelSeconds    = buffer.getLevelSeconds();

        // Panic mode: buffer critically low → drop to lowest quality immediately
        if (bufferLevelSeconds < BUFFER_PANIC_SEC) {
            return renditions.get(0); // 360p
        }

        // Buffer-based selection: higher buffer = more aggressive quality upgrade
        double bufferFactor = Math.min(1.0, bufferLevelSeconds / BUFFER_TARGET_SEC);

        // Select highest rendition whose bitrate ≤ available bandwidth × buffer factor
        Rendition selected = renditions.get(0);
        for (Rendition r : renditions) {
            if (r.getBitrateKbps() * 1000 <= availableBandwidthBps * bufferFactor) {
                selected = r;
            }
        }
        return selected;
    }
}

// Bandwidth estimator: EWMA (exponential weighted moving average) of recent downloads
public class BandwidthEstimator {
    private double estimatedBps = 1_000_000; // 1 Mbps initial estimate

    public void recordDownload(long bytes, long durationMs) {
        double measuredBps = (bytes * 8.0 * 1000) / durationMs;
        // EWMA with α=0.3 (new observations weighted 30%)
        estimatedBps = 0.7 * estimatedBps + 0.3 * measuredBps;
    }

    public double estimate() { return estimatedBps; }
}
```

### 5.3 Low-latency streaming optimisations

```
Standard HLS: segment duration 2 seconds, pre-buffer 10s → 12+ second latency
Low-latency HLS (LL-HLS): chunk-based sub-segments within each 2s segment
  → 1-3 second latency; standard for live; increasingly used for VoD

Startup optimisation:
  1. Begin playback at 360p (smallest segments, fastest initial download)
  2. While playing the first 360p segment, download a 720p segment
  3. Switch to 720p after first 2 seconds (user barely notices the brief low quality)
  4. Continue upgrading to 1080p+ as buffer grows

Preloading on hover (YouTube):
  When user hovers over a video thumbnail: prefetch the first segment at 360p
  By the time they click: video starts instantly (segment already in browser cache)
  Implementation: HTML <link rel="prefetch"> or JavaScript fetch() on mouseenter
```

## 7. View Count Consistency

### 7.1 The view count problem

```
Scale challenge:
  1B DAV × 20 views/day = 20B view events/day = 231K events/second
  Peak: ~700K view events/second

Naive approach: UPDATE videos SET view_count = view_count + 1 WHERE video_id = ?
  At 700K updates/second to one row (for a viral video): catastrophic lock contention.
  Same hot-row problem as URL shortener click counts and social media likes.

Requirements:
  Display count: approximate OK (< 1-minute lag acceptable)
  Billing / monetisation: exact count required (settled within 24 hours)
  Fraud detection: duplicate views (same user, same minute) should not count

Solution: multi-tier count pipeline
  Tier 1: Redis INCR (atomic, ~1M ops/second per instance)  → display count
  Tier 2: Kafka stream                                        → durable event log
  Tier 3: Flink / Spark streaming                            → deduplication + aggregation
  Tier 4: OLAP DB (ClickHouse)                               → exact counts, analytics
  Tier 5: Periodic flush to PostgreSQL                       → canonical count for billing
```

### 7.2 View event pipeline

```java
// On video playback start (not just page load): fire view event
@PostMapping("/api/v1/videos/{videoId}/view")
public void recordView(@PathVariable String videoId,
                        @RequestBody ViewEvent event,
                        HttpServletRequest request) {

    // Dedup: same user watching same video within 30 seconds = 1 view
    String dedupKey = "view:dedup:" + event.getUserId() + ":" + videoId;
    Boolean isNew = redis.setIfAbsent(dedupKey, "1", 30, TimeUnit.SECONDS);
    if (!Boolean.TRUE.equals(isNew)) return; // duplicate view

    // Tier 1: Increment display counter in Redis (immediate)
    redis.incr("view:count:" + videoId);

    // Tier 2: Publish to Kafka (durable, for exact accounting)
    kafkaTemplate.send("view-events", videoId, ViewEventMessage.builder()
        .videoId(videoId)
        .userId(event.getUserId())
        .sessionId(event.getSessionId())
        .ipAddressHash(hashIp(request.getRemoteAddr()))
        .watchedSeconds(event.getWatchedSeconds())
        .timestamp(Instant.now())
        .build());
}

// Periodic flush: Redis counts → PostgreSQL (every 5 minutes)
@Scheduled(fixedRate = 300_000)
public void flushViewCounts() {
    Set<String> keys = redis.keys("view:count:*");
    for (String key : keys) {
        String videoId = key.replace("view:count:", "");
        Long delta = redis.getAndSet(key, "0"); // atomic swap to 0
        if (delta != null && delta > 0) {
            videoRepository.incrementViewCount(videoId, delta);
        }
    }
}
```

public class ViewCountAggregator {

    public DataStream<VideoViewCount> buildPipeline(StreamExecutionEnvironment env) {
        DataStream<ViewEvent> events = env
            .addSource(new FlinkKafkaConsumer<>("view-events", schema, kafkaProps));

        return events
            // Deduplicate: same userId+videoId within 30-second window
            .keyBy(e -> e.getUserId() + ":" + e.getVideoId())
            .window(TumblingProcessingTimeWindows.of(Time.seconds(30)))
            .apply(new DeduplicateViewFunction())   // emit max 1 per user per video per 30s

            // Aggregate: sum deduplicated views per video per minute
            .keyBy(ViewEvent::getVideoId)
            .window(TumblingProcessingTimeWindows.of(Time.minutes(1)))
            .aggregate(new ViewCountAggregateFunction())

            // Write to ClickHouse for analytics + exact billing
            .addSink(new ClickHouseSink("video_view_counts_1min"));
    }
}

### 7.4 View count fraud detection

```
Common fraud patterns:
  Same IP: 1,000 views from one IP in 1 minute → bot traffic
  Same user: repeated views within 30 seconds → accidental replay or bot
  Sudden spike: 10M views in 10 minutes → bought views (e.g. view farms)
  No engagement: views with 0 seconds watched → page load, not real view

Detection rules (applied in Flink stream):
  1. Rate limit: > 100 views from same IP in 1 minute → flag all as suspicious
  2. Watch time filter: view with < 5 seconds watched → do not count
  3. Session dedup: same sessionId, same videoId, within 5 minutes → 1 view
  4. Spike detection: video view rate > 10× its 7-day average → review queue
  5. User agent filter: headless browser / known bot agents → discard

Flagged views: stored in separate "suspect_views" table.
  Human review queue for channels with > 100K flagged views.
  Automatic removal if fraud confirmed (update view_count accordingly).
```

## 8. Recommendation System

### 8.1 Architecture overview

```
Recommendation pipeline:

Offline (batch):        runs daily/hourly; pre-computes candidate pools
  Collaborative filtering: "users like you also watched X"
  Content-based:           "you watched X; similar videos: Y, Z"
  Item-item similarity:    "viewers of X also watch Y" (co-watch matrix)
  Feature extraction:      video embeddings (title, transcript, visual)

Near-realtime (minutes): reacts to recent behaviour
  Session context:         watched 3 tech videos today → boost tech recommendations
  Trending:                videos spiking in views in the last hour
  Freshness:               new uploads from subscribed channels

Online (per-request):   fast scoring and ranking for individual user
  Candidate retrieval:     ANN lookup of nearest video embeddings
  Scoring:                 lightweight ML model ranks candidates
  Filter:                  remove watched, deleted, age-restricted for user
  Diversity:               avoid 10 identical thumbnails
```

### 8.2 Two-stage ranking

```
Stage 1: Candidate Generation (recall phase)
  Goal: retrieve ~500 candidate videos from billions
  Speed: must complete in < 20ms (index lookups)

  Methods:
    Collaborative filtering: matrix factorisation (user/video embeddings)
      → Approximate nearest neighbour search (Faiss, ScaNN, Annoy)
    Content similarity: video embedding lookup for videos like "recently watched"
    Subscriptions: latest videos from channels user follows
    Trending: top-N videos from user's regions/categories in last 24h
    Exploration: random sample from long tail (prevents filter bubble)

  Output: ~500 candidate videoIds with raw scores

Stage 2: Ranking (precision phase)
  Goal: rank 500 candidates to top 20 for display
  Speed: must complete in < 100ms

  Features per candidate:
    User features:    watch history, liked categories, session context, device
    Video features:   CTR (click-through rate), avg watch time %, like rate,
                      freshness, embedding similarity to watched videos
    Context features: time of day, day of week, query if from search

  Model: LambdaMART, Deep & Wide, or two-tower neural network
  Training: labels = user engagement (watch, like, share, subscribe after watch)
  Inference: batched for all 500 candidates → rank → return top 20
```

### 8.3 Video embedding and ANN search

```java
// Video embedding: represent each video as a 128-dimensional vector
// capturing semantic content (topics, style, audience)

// Offline: compute embeddings from video metadata
@Service
public class VideoEmbeddingService {

    // Pre-trained sentence transformer for title+description → 128-dim vector
    public float[] computeEmbedding(Video video) {
        String text = video.getTitle() + " " + video.getDescription()
                    + " " + String.join(" ", video.getTags());
        return embeddingModel.encode(text); // runs locally (model loaded at startup)
    }

    // Store embeddings in Faiss index for ANN search
    public void indexVideo(long videoId, float[] embedding) {
        faissIndex.add(videoId, embedding);
        // Also store in Redis for online lookup:
        redis.set("embed:" + videoId, serialize(embedding), Duration.ofDays(7));
    }
}

// Online: ANN search — "find 100 videos similar to this one"
@Service
public class CandidateRetriever {

    public List<Long> getSimilarVideos(long seedVideoId, int topK) {
        float[] queryEmbedding = getEmbedding(seedVideoId);

        // Approximate nearest neighbour: returns top-K similar videoIds
        // Faiss HNSW index: O(log N) search, handles billions of vectors
        long[] nearestIds = faissIndex.search(queryEmbedding, topK);

        return Arrays.stream(nearestIds)
            .boxed()
            .filter(id -> id != seedVideoId) // exclude self
            .collect(Collectors.toList());
    }
}
```

### 8.4 Click-through rate and watch time optimisation

```
Two key engagement signals (often in tension):

Click-Through Rate (CTR):
  CTR = clicks / impressions
  Optimising CTR alone → clickbait (misleading titles/thumbnails that get clicked
  but result in immediate abandonment)

Average Watch Percentage (AWP):
  AWP = seconds_watched / video_duration
  Optimising AWP alone → favours short videos (easier to watch completely)

Combined signal: "satisfaction score"
  satisfaction = CTR × AWP × completion_rate
  Plus: explicit positive signals (likes, shares, comments, subscribe_after_watch)
  Minus: explicit negative signals (dislike, "not interested", skip in first 5 seconds)

YouTube's "watch time" model (described in their 2016 paper):
  Train to predict expected watch time, not just probability of click.
  This naturally deprioritises clickbait: high CTR + low watch time = low score.
  Weighted ranking: expected_watch_time = P(click) × E(watch_time | click)
```

### 8.5 Recommendation serving infrastructure

```
Pre-computation for known users:
  Nightly batch job: pre-compute top-50 recommended videoIds per active user.
  Store in Redis: rec:{userId} → [videoId1, videoId2, ...] with 12-hour TTL.
  On page load: serve pre-computed list instantly (< 1ms Redis GET).
  Refresh if user watches a video (invalidate and recompute async).

Real-time scoring for new users / cold start:
  New user: no history → serve popularity-based recommendations by region + time.
  First 10 interactions: rapidly update user's preference vector.
  After ~20 interactions: full personalised model kicks in.

Cold start for new videos:
  New video: no CTR/watch-time data yet.
  Bootstrap with: content similarity to known popular videos,
                  channel's historical performance, metadata category.
  After first 1,000 views: real engagement data replaces bootstrap estimates.

A/B testing:
  Recommendation algorithms are A/B tested continuously.
  Treatment group: 5% of users see new algorithm.
  Metric: 7-day retention, total watch time, subscriber growth.
  Winner promoted after statistical significance (typically 2 weeks).
```

## 10. System Scalability

### 10.1 Transcoding at scale

```
Spot instance strategy for transcoding:
  Transcoding is:
    - Compute-intensive (CPU/GPU bound)
    - Stateless (workers pull jobs from queue; no sticky state)
    - Fault-tolerant (failed job re-queued to another worker)
    - Batch (not latency-sensitive — 10-minute SLA, not 10ms)
  → Perfect for EC2 Spot Instances (70% cheaper than on-demand)

Auto-scaling:
  Kafka consumer lag metric → CloudWatch alarm → ASG scales workers up/down.
  Peak upload period (evenings): scale to 10K workers.
  Off-peak (3 AM): scale down to 500 workers.

GPU vs CPU transcoding:
  CPU (x264/x265): quality codec, high CPU time, no GPU cost
  GPU (NVENC H.264): 10-50× faster than CPU, costs GPU instance
  For 4K/HEVC: GPU transcoding is cost-effective despite higher instance cost.
  Netflix: VMAF (Video Multi-method Assessment Fusion) quality scoring
           to determine optimal codec parameters per video segment.
```

## 11. Interview Questions

**Q: Why is adaptive bitrate streaming necessary, and how does the ABR algorithm decide which quality segment to download next?**

A: Without ABR, a player would need to commit to a single quality level for the entire video. Network conditions vary dramatically — mobile users move between WiFi and cellular, home internet speeds fluctuate with household usage, and corporate networks have variable congestion. A committed 1080p stream would buffer constantly on a slow connection; a committed 360p stream would look terrible on a fast connection. ABR solves this by splitting the video into 2-second segments at multiple quality levels and letting the player select the quality for each upcoming segment independently. The algorithm must balance two competing objectives: maximise quality (prefer high bitrate) and minimise rebuffering (keep buffer full). A common approach is buffer-based ABR: when the playback buffer is large (30+ seconds ahead), the player can afford to request the highest quality available within bandwidth; when the buffer is small (under 10 seconds), it aggressively drops to the lowest quality to refill the buffer quickly. Bandwidth estimation uses an exponential weighted moving average of recent segment download speeds, discounting older measurements to react to current conditions. A safety margin (typically 85% of estimated bandwidth) prevents overestimating available capacity.

---

**Q: How does the transcoding pipeline handle 1 million video uploads per day without accumulating a backlog?**

A: The pipeline parallelises at two levels. First, each video is split into independent jobs per rendition — a 10-minute video targeting six resolutions generates six independent transcoding jobs. These are published to a Kafka topic and consumed by a pool of stateless worker instances. Since workers have no shared state, adding more workers linearly increases throughput. Second, spot instances provide elastic scaling: CloudWatch monitors Kafka consumer lag, and an Auto Scaling Group spins up spot instances when lag grows and terminates them when it shrinks. At 1M uploads/day and approximately 120 CPU-minutes per video (six renditions averaged), the cluster needs roughly 83,333 CPU-hours per day — provided by ~3,500 continuously running workers or proportionally more with spot instance interruptions factored in. Priority queuing ensures the 360p rendition (smallest, fastest) is processed first so videos become available for viewing quickly. Higher resolutions (1440p, 4K) are lower priority and may take 30-60 minutes, which is acceptable since viewers tend not to seek 4K on newly uploaded content in the first hour.

---

**Q: How does your CDN design achieve 99%+ cache hit rates, and what happens on a cache miss?**

A: High cache hit rates require careful cache key design. Video segments use the key `/videos/{videoId}/{resolution}/{segmentNumber}.ts` — since videoId and segmentNumber uniquely identify immutable content, segments never need invalidation. A segment uploaded once is valid forever (re-transcoding issues a new videoId). This means segments can have very long TTLs — 7 days at regional shields, 24 hours at edge PoPs — without risk of serving stale data. Popular content (top 20% of videos representing 80% of views) remains warm in edge caches. The origin shield is the key architectural element for managing misses: rather than hundreds of edge PoPs independently missing and hammering the S3 origin, all misses within a region funnel to one regional shield cluster. This request coalescing means a cache miss at the edge PoP level almost always hits the shield (not origin), and the shield only requests from origin when its own cache is cold. In practice, origin receives less than 1% of all requests — primarily newly uploaded content and long-tail videos accessed rarely. On a genuine origin miss, the shield fetches from S3, caches the segment, and serves the response; the edge PoP caches it too; subsequent requests for that segment in that region hit the edge.

---

**Q: How do you ensure view counts are eventually consistent for display but exactly correct for monetisation?**

A: The two requirements have different consistency and latency needs, so they use different pipelines. Display count: when a valid view is recorded, the server performs a Redis INCR on `view:count:{videoId}` — an atomic O(1) operation handling millions of increments per second with no lock contention. Every 5 minutes, a batch job reads these Redis counters, atomically swaps them to zero, and adds the delta to the PostgreSQL view_count column. Display latency is at most 5 minutes behind reality — acceptable for viewers who don't notice a count being off by a few thousand. Monetisation count: every view event is published to a Kafka topic. A Flink streaming job consumes this topic, applies deduplication (same user + video within 30 seconds = 1 view, using a tumbling window with state), filters out fraud signals (watch time < 5 seconds, known bot user agents, IP rate limit violations), and aggregates into per-minute counts written to ClickHouse. The ClickHouse SummingMergeTree engine handles concurrent writes efficiently. Billing settlement runs a daily Spark job that sums ClickHouse counts into the canonical billing record. This count is exact — the Kafka log retains all events for 7 days, making the Flink pipeline fully replayable if a bug is discovered. The display and billing counts may briefly diverge, but settle within 24 hours.

---

**Q: How does your recommendation system handle the cold start problem for new users and new videos?**

A: Cold start affects two entities independently, each with different solutions. For new users, there is no interaction history to drive personalised recommendations. The system falls back to a multi-signal popularity model: trending videos in the user's detected region and language, top-performing content in the most common categories for that region, and content from channels the user explicitly subscribes to at signup. As the user watches their first 5-10 videos, a rapid-update mechanism continually refines their preference vector using lightweight collaborative filtering on their growing micro-history. After approximately 20 interactions, there is sufficient signal for the full two-tower model to run effectively. For new videos, there is no CTR or watch-time data to feed the ranking model. The bootstrap strategy uses content-based signals: the video's title, description, and tags are encoded into an embedding, and approximate nearest neighbour search in the Faiss index finds semantically similar established videos. The new video's predicted performance is derived from its semantic peers' historical CTR and watch time, adjusted for the channel's own historical performance baseline. Once a video accumulates its first 1,000 views, real engagement data progressively replaces the bootstrap estimates. The weighting between bootstrap prior and observed evidence follows a simple Bayesian update: `effective_CTR = (prior_clicks + observed_clicks) / (prior_impressions + observed_impressions)`, where prior weight diminishes as observed data grows.

## 12. Key Takeaways

- **Transcoding parallelism**: one video → N rendition jobs published to Kafka → consumed by N parallel spot-instance workers; lower resolutions (360p) prioritised so video becomes available quickly
- **HLS/DASH segment design**: 2-second segments at each resolution; master playlist lists all renditions; client selects per-segment quality based on current bandwidth
- **ABR algorithm**: buffer-based + bandwidth EWMA; panic mode at < 5s buffer (drop to lowest quality); safety margin 85% of estimated bandwidth; quality changes every 2 seconds invisibly
- **CDN cache key immutability**: `/videos/{videoId}/{resolution}/{segment}.ts` never changes → infinite TTL; re-transcoding = new videoId; manifests have short TTL (60s) as they change
- **Origin shield**: regional aggregation layer between edge PoPs and S3 origin; coalesces cache misses; keeps origin traffic < 1% of total; critical for cost and latency
- **View count pipeline**: Redis INCR → display (5-min lag); Kafka → Flink dedup+aggregate → ClickHouse exact counts → daily billing settlement; dedup key = userId+videoId per 30s window
- **Two-stage recommendation**: candidate generation (ANN on embeddings, ~500 candidates, < 20ms) → ranking (lightweight ML model on 500 candidates, < 100ms) → filter + diversity → top 20
- **Per-title encoding**: analyse each video's complexity; simple animation at 1080p needs 1 Mbps vs 5 Mbps for action film; saves 20-50% bandwidth vs fixed encoding ladder
- **Cold start for videos**: content embedding → ANN to semantic peers → inherit peer CTR/watch-time as prior; Bayesian update as real data accumulates
- **Spot instances for transcoding**: stateless, batch, fault-tolerant = perfect for 70%-cheaper spot; Kafka re-delivers interrupted jobs; auto-scale on consumer lag metric
