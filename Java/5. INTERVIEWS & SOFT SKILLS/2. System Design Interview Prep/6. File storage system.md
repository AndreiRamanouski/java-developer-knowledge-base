# File Storage System Design (Dropbox / Google Drive)

## Overview

A cloud file storage system lets users upload, download, and sync files across multiple devices, with sharing and collaboration. The hard problems are not file I/O — they are chunking for efficient delta sync, deduplication to control storage costs, conflict resolution when two devices edit simultaneously, and a sync algorithm that converges all clients to the same state with minimal bandwidth. Dropbox serves 700 million registered users; Google Drive handles 2 billion files uploaded per day.

**The core insight:**

```
Naive approach: upload the entire file on every save.
  User edits one line of a 1 GB file → re-upload 1 GB. Catastrophic.

Chunked approach: split file into fixed-size blocks.
  Only upload blocks that changed. For a 1 GB file with one edited paragraph:
  only 1-3 chunks (4 MB) re-uploaded instead of 1 GB → 250× bandwidth reduction.

Content-addressed storage: each chunk stored by hash(chunk_content).
  Two users upload identical files → stored once. Zero extra storage.
  User renames file → metadata changes; chunk content unchanged → zero re-upload.
```

## 3. Chunking

### 3.1 Why chunk files

```
Benefits of chunking:
  Delta sync:     only re-upload changed chunks (saves 90-99% bandwidth on re-saves)
  Parallel upload: multiple chunks uploaded concurrently → faster for large files
  Deduplication:  identical chunks stored once regardless of which file contains them
  Resumable:      interrupted upload resumes from last successfully uploaded chunk
  CDN efficiency: frequently accessed popular chunks cached at CDN edge nodes

Chunk size selection:
  Too small (4 KB): too many chunks → metadata overhead dominates
  Too large (64 MB): small edits still re-upload large chunk; slow progress feedback
  Sweet spot: 4-8 MB for general files

  Dropbox uses variable-size content-defined chunking (CDC).
  This design: fixed 4 MB blocks for simplicity, with variable final block.
```

### 3.2 Fixed-size chunking

```java
@Service
public class FileChunker {
    public static final int CHUNK_SIZE = 4 * 1024 * 1024; // 4 MB

    // Split file into chunks; compute SHA-256 hash per chunk
    public List<ChunkInfo> chunk(Path filePath) throws IOException {
        List<ChunkInfo> chunks = new ArrayList<>();
        try (FileInputStream fis = new FileInputStream(filePath.toFile())) {
            byte[] buffer = new byte[CHUNK_SIZE];
            int chunkIndex = 0;
            int bytesRead;

            while ((bytesRead = fis.read(buffer)) != -1) {
                byte[] chunkData = bytesRead < CHUNK_SIZE
                    ? Arrays.copyOf(buffer, bytesRead) // last chunk may be smaller
                    : buffer.clone();

                String hash = sha256Hex(chunkData);
                chunks.add(ChunkInfo.of(chunkIndex++, hash, bytesRead, chunkData));
            }
        }
        return chunks;
    }

    // Compute file-level fingerprint: SHA-256 of the concatenation of chunk hashes
    public String fileFingerprint(List<ChunkInfo> chunks) {
        MessageDigest digest = MessageDigest.getInstance("SHA-256");
        chunks.forEach(c -> digest.update(hexToBytes(c.getHash())));
        return bytesToHex(digest.digest());
    }

    private String sha256Hex(byte[] data) {
        return bytesToHex(
            MessageDigest.getInstance("SHA-256").digest(data)
        );
    }
}
```

### 3.3 Content-defined chunking (CDC) — Rabin fingerprinting

```
Fixed chunking fails on insertion: insert 1 byte at start of file →
ALL chunks shift by 1 byte → all chunk hashes change → full re-upload.

Example:
  Before: [AAAA][BBBB][CCCC]  (chunks by position)
  After:  [xAAA][ABBB][BCCC]  (x inserted at start — all chunks changed!)

Content-defined chunking (CDC) uses a rolling hash to find natural
chunk boundaries based on content, not position.

Rabin fingerprint algorithm:
  Maintain a sliding window of W bytes.
  At each byte position, compute polynomial rolling hash of window.
  If hash & MASK == PATTERN: this is a chunk boundary.
  MASK controls average chunk size: MASK=0xFFF → avg 4 KB; MASK=0x1FFF → avg 8 KB.

After insertion:
  Before: [AAA|BBB|CCC]   (| = natural content boundary)
  After:  [xAA|ABBB|CCC]  (first chunk absorbs x; rest unchanged!)
  → Only 1 chunk re-uploaded instead of 3.

Dropbox: uses CDC with min 512 KB, avg 4 MB, max 8 MB chunk size.
rsync: uses a simpler rolling hash for delta sync of fixed-position files.

For interview simplicity: use fixed-size chunking and note CDC as optimisation.
```

## 4. Upload and Download

### 4.1 Resumable chunked upload flow

```
Step 1: Client initiates upload
  POST /api/v1/files/upload/init
  Body: {filename, fileSize, fileFingerprint, chunkCount, parentFolderId}
  Response: {uploadId, chunksNeeded: [0,1,2,3,...], presignedUrls: {...}}

  Server checks: is fileFingerprint already in block store?
    YES: zero-byte upload (dedup at file level) → skip to Step 4
    NO:  proceed with chunk upload

Step 2: Client checks which chunks the server already has
  POST /api/v1/files/upload/{uploadId}/check
  Body: {chunkHashes: ["abc123", "def456", ...]}
  Response: {missingChunks: [0, 2]}  ← only chunks 0 and 2 are not in store

  Client only uploads missing chunks (delta upload).

Step 3: Client uploads missing chunks directly to S3 (pre-signed URLs)
  PUT https://s3.region.amazonaws.com/chunks/{sha256}?X-Amz-Signature=...
  Body: raw chunk bytes
  → Direct client-to-S3 upload: bypasses application servers (no server bandwidth cost)
  → Pre-signed URL expires in 1 hour; chunk upload is resumable at S3 level

Step 4: Client finalises upload
  POST /api/v1/files/upload/{uploadId}/finalise
  Body: {chunkHashes: [...in order...]}
  Server:
    a. Verifies all chunks present in S3
    b. Creates file_version record in metadata DB
    c. Updates folder's modified_at
    d. Publishes sync event to Kafka → notifies user's other devices
    e. Returns: {fileId, versionId, url}
```

### 4.2 Upload sequence diagram

```
Client          API Server       Metadata DB      S3 / Block Store      Kafka
  │                │                 │                  │                 │
  │─ Init Upload ─▶│                 │                  │                 │
  │                │─ Check ─────────▶                  │                 │
  │                │◀─ fileId,exists─│                  │                 │

  │◀─ missingChunks│                 │                  │                 │
```

### 4.3 Download flow

```
Step 1: Client requests file metadata
  GET /api/v1/files/{fileId}?version=latest
  Response: {fileId, filename, chunkHashes: ["abc123", "def456", ...],
             totalSize, mimeType, createdAt, modifiedAt}

Step 2: Client identifies chunks it already has locally (from previous sync)
  Client's local chunk cache: Set of hashes for locally present chunks.
  Needed = chunkHashes - localCache → only fetch missing chunks.

Step 3: Download missing chunks
  GET /api/v1/chunks/{hash} → 302 redirect to pre-signed S3 URL
    OR: pre-signed URLs returned in batch during Step 1 response
  Client fetches directly from S3 (or CloudFront CDN for hot/popular chunks).

Step 4: Reconstruct file
  Concatenate chunks in order (by chunk index, not hash).
  Verify assembled file hash matches fileFingerprint.
  Write to local filesystem.
```

### 4.4 Pre-signed URL architecture

```
Why pre-signed URLs (not proxying through API servers):
  Without pre-signed: Client → API Server → S3 → API Server → Client
    API server handles all file bytes: 185 GB/s × 2 = 370 GB/s through servers.
    Requires enormous server fleet just for byte passing.

  With pre-signed: Client → S3 directly
    API server issues a time-limited signed URL (expires in 1 hour).
    Client talks directly to S3 / CloudFront.
    API servers handle only metadata and auth (lightweight JSON).

  Pre-signed URL generation:
    String presignedUrl = s3Presigner.presignPutObject(req ->
        req.signatureDuration(Duration.ofHours(1))
           .putObjectRequest(put ->
               put.bucket("chunks").key(chunkHash)));
```

## 5. Deduplication

### 5.1 Content-addressed storage

```
Every chunk stored by its SHA-256 hash as the key:
  s3://chunks-bucket/{sha256-of-chunk-content}

Consequence:
  Two users upload identical files → same chunk hashes → stored once.
  User renames file → metadata changes; chunk hashes unchanged → no re-upload.
  User moves file to different folder → same chunks; only metadata updated.
  User creates copy of file → shares chunks with original; storage = 0 extra bytes.

Deduplication impact:
  Industry data: 30-50% of uploaded data is identical to existing chunks.
  Real savings: Dropbox reported 50% storage reduction from cross-user dedup.
```

### 5.2 Chunk existence check — the delta upload protocol

```java
@PostMapping("/upload/{uploadId}/check")
public ChunkCheckResponse checkChunks(
        @PathVariable String uploadId,
        @RequestBody ChunkCheckRequest req) {

    List<String> hashes = req.getChunkHashes();

    // Batch check: which hashes already exist in block store?
    // Use Redis SET for hot chunk existence index (recently uploaded chunks)
    // Fall back to S3 HeadObject for cold chunks

    List<String> missing = new ArrayList<>();
    List<String> existing = new ArrayList<>();

    // Step 1: batch Redis lookup (O(1) per hash, one round-trip for all)
    List<Boolean> inCache = redis.executePipelined(conn -> {
        hashes.forEach(h -> conn.setCommands().sIsMember("chunk:exists", h));
    }).stream().map(r -> (Boolean) r).collect(Collectors.toList());

    // Step 2: DB confirmation for cache misses
    for (int i = 0; i < hashes.size(); i++) {
        if (inCache.get(i)) {
            existing.add(hashes.get(i));
        } else {
            // Confirm via metadata DB (chunk_store table)
            if (chunkRepository.exists(hashes.get(i))) {
                existing.add(hashes.get(i));
                redis.sadd("chunk:exists", hashes.get(i)); // warm cache
            } else {
                missing.add(hashes.get(i));
            }
        }
    }

    // Issue pre-signed upload URLs only for missing chunks
    Map<String, String> presignedUrls = missing.stream()
        .collect(Collectors.toMap(
            hash -> hash,
            hash -> s3Presigner.presignPutObject(hash)
        ));

    return new ChunkCheckResponse(missing, existing, presignedUrls);
}
```

### 5.3 Reference counting for chunks

```sql
-- Track how many file versions reference each chunk
-- When count reaches 0: chunk is eligible for garbage collection

CREATE TABLE chunk_references (
    chunk_hash      CHAR(64)       PRIMARY KEY,  -- SHA-256 hex
    reference_count INT            NOT NULL DEFAULT 0,
    size_bytes      BIGINT         NOT NULL,
    first_uploaded  TIMESTAMPTZ    NOT NULL,
    last_accessed   TIMESTAMPTZ    NOT NULL
);

-- On file version creation: increment reference counts for all its chunks
-- On file version deletion / expiry: decrement reference counts
-- GC job: DELETE from S3 WHERE reference_count = 0 AND last_accessed < 30 days ago

-- Atomic increment to avoid race on concurrent uploads of same chunk:
UPDATE chunk_references SET reference_count = reference_count + 1
WHERE chunk_hash = ?
-- If no rows updated: this is a new chunk → INSERT
```

## 6. Metadata Storage

### 6.1 What metadata stores

```
Metadata service is the source of truth for everything EXCEPT chunk bytes:
  File hierarchy:    folders, parent-child relationships
  File identity:     fileId, name, MIME type, size, owner, created/modified times
  Chunk manifest:    ordered list of chunk hashes for each file version
  Version history:   previous versions of each file, with their chunk manifests
  Sharing:           who has access to what, with what permission
  Sync cursors:      per-device cursor pointing to latest synced event
  Conflict records:  which files have unresolved conflicts
```

### 6.2 Schema

```sql
-- PostgreSQL for metadata (sharded by user_id for scale)

-- Folder hierarchy (adjacency list — simple and efficient for typical depths)
CREATE TABLE folders (
    folder_id       BIGINT          PRIMARY KEY,
    owner_id        BIGINT          NOT NULL,
    parent_folder_id BIGINT,        -- NULL for root folder
    name            VARCHAR(255)    NOT NULL,
    created_at      TIMESTAMPTZ     NOT NULL,
    modified_at     TIMESTAMPTZ     NOT NULL,
    is_deleted      BOOLEAN         NOT NULL DEFAULT FALSE
);
CREATE INDEX idx_folders_parent ON folders(parent_folder_id);
CREATE INDEX idx_folders_owner  ON folders(owner_id);

-- File identity (one row per logical file)
CREATE TABLE files (
    file_id         BIGINT          PRIMARY KEY,
    owner_id        BIGINT          NOT NULL,
    folder_id       BIGINT          NOT NULL REFERENCES folders(folder_id),
    name            VARCHAR(1024)   NOT NULL,
    mime_type       VARCHAR(255),
    current_version BIGINT,         -- FK to file_versions
    is_deleted      BOOLEAN         NOT NULL DEFAULT FALSE,
    created_at      TIMESTAMPTZ     NOT NULL,
    modified_at     TIMESTAMPTZ     NOT NULL
);
CREATE INDEX idx_files_folder    ON files(folder_id);
CREATE INDEX idx_files_owner     ON files(owner_id, modified_at DESC);

-- File versions (one row per version)
CREATE TABLE file_versions (
    version_id      BIGINT          PRIMARY KEY,
    file_id         BIGINT          NOT NULL REFERENCES files(file_id),
    version_number  INT             NOT NULL,
    uploader_id     BIGINT          NOT NULL,
    size_bytes      BIGINT          NOT NULL,
    file_fingerprint CHAR(64)       NOT NULL,  -- hash of all chunk hashes
    created_at      TIMESTAMPTZ     NOT NULL,
    is_conflict_copy BOOLEAN        NOT NULL DEFAULT FALSE,
    UNIQUE (file_id, version_number)
);

-- Chunk manifest: ordered list of chunks per file version
CREATE TABLE version_chunks (
    version_id      BIGINT          NOT NULL REFERENCES file_versions(version_id),
    chunk_index     INT             NOT NULL,  -- position in file
    chunk_hash      CHAR(64)        NOT NULL,  -- content-addressed key in S3
    PRIMARY KEY (version_id, chunk_index)
);

-- Sync event log: append-only log of all changes (for device sync)
CREATE TABLE sync_events (
    event_id        BIGINT          PRIMARY KEY,  -- Snowflake (monotonically increasing)
    user_id         BIGINT          NOT NULL,
    event_type      VARCHAR(30)     NOT NULL,     -- 'file_created','file_updated','file_deleted','file_moved'
    file_id         BIGINT,
    folder_id       BIGINT,
    payload         JSONB           NOT NULL,     -- event-specific data
    created_at      TIMESTAMPTZ     NOT NULL
);
CREATE INDEX idx_sync_events_user ON sync_events(user_id, event_id);

-- Device cursors: each device tracks how far it has synced
CREATE TABLE device_sync_cursors (
    device_id       UUID            PRIMARY KEY,
    user_id         BIGINT          NOT NULL,
    last_event_id   BIGINT          NOT NULL DEFAULT 0,  -- last processed event_id
    last_synced_at  TIMESTAMPTZ     NOT NULL
);
```

### 6.3 Sharding strategy

```
Shard by owner_id (hash-based):
  All of a user's files, folders, and versions on the same shard.
  Queries like "list all files in folder X" = single shard query.
  Shard key: owner_id % NUM_SHARDS

Cross-shard queries (shared files):
  When file shared from user A to user B:
    Option 1: Duplicate file metadata into B's shard (denormalisation)
              → B's queries fast; keep in sync on updates
    Option 2: Store sharing record in sharded sharing table; B's shard
              has pointer to A's file metadata → cross-shard lookup on access
  
  Dropbox approach: sharing metadata stored separately; file metadata
  owned by original owner; share resolver service handles cross-shard joins.

Folder hierarchy challenge:
  "List all files in /Documents and all subfolders" = recursive query.
  PostgreSQL recursive CTE: feasible for moderate depth (< 10 levels).
  At scale: materialise the full path as a path string:
    /Documents/Work/Q4/report.pdf
    Index on path prefix: WHERE path LIKE '/Documents/%' → range scan.
    Update path on move: UPDATE files SET path = replace(path, old, new).
```

## 7. Sync Algorithm

### 7.1 The sync problem

```
Three devices: phone, laptop, desktop. All may be offline simultaneously.
Each makes changes independently. When coming online: must converge to
the same state without losing any changes.

Sync properties required:
  Convergence:   all devices eventually reach the same state
  Completeness:  no change is silently lost
  Conflict handling: simultaneous edits are detected and surfaced to user

The sync algorithm:
  Each device maintains a cursor (event_id of last processed sync event).
  Changes flow through an append-only sync event log on the server.
  On connecting: device sends its cursor, receives all events since that cursor.
  On change: device sends change to server; server appends event; broadcasts to
             all other devices of same user via WebSocket / push notification.
```

### 7.2 Event-sourced sync log

```
The sync_events table is an append-only log of every change made to a user's
file space. It is the single source of truth for sync.

Event types:
  file_created:  {fileId, folderId, name, versionId, chunkHashes}
  file_updated:  {fileId, newVersionId, oldVersionId, chunkHashes}
  file_deleted:  {fileId}
  file_moved:    {fileId, oldFolderId, newFolderId, newName}
  folder_created:{folderId, parentId, name}
  folder_deleted:{folderId}

Device sync algorithm:
  1. Device connects and sends: GET /sync?cursor=12345
  2. Server returns all events with event_id > 12345 for this user
  3. Device applies events locally (in order — event_id is Snowflake, monotone)
  4. Device updates its cursor to the highest event_id received
  5. Device subscribes to WebSocket for future events (real-time updates)
  6. When device makes a local change: POST to API → server creates event,
     pushes to device's OTHER devices via WebSocket

Event log retention:
  Keep events for 30 days (allows devices offline for up to 30 days to sync)
  Devices offline > 30 days: perform a full state sync instead of delta sync
  (fetch full file tree from metadata API; not the event log)
```

### 7.3 Sync client state machine

```java
public class SyncStateMachine {

    enum State { IDLE, SCANNING, UPLOADING, DOWNLOADING, SYNCING, ERROR }

    // Local change detected (filesystem watcher)
    public void onLocalChange(Path path, WatchEvent.Kind<?> kind) {
        SyncItem item = analyseChange(path, kind);

        if (item.isDirectory()) {
            metadataService.syncFolder(item);
            return;
        }

        // Chunk the file; compute which chunks are new
        List<ChunkInfo> chunks = fileChunker.chunk(path);
        String fingerprint = fileChunker.fileFingerprint(chunks);

        // Check if file content actually changed (avoid redundant uploads)
        if (fingerprint.equals(localState.getLastKnownFingerprint(path))) {
            return; // content unchanged (e.g. only access time changed)
        }

        // Upload delta: only upload chunks not already on server
        uploadService.uploadDelta(path, chunks, fingerprint);
    }

    // Remote change received (from WebSocket sync event)
    public void onRemoteEvent(SyncEvent event) {
        switch (event.getType()) {
            case FILE_CREATED, FILE_UPDATED -> downloadFile(event);
            case FILE_DELETED               -> deleteLocal(event);
            case FILE_MOVED                 -> moveLocal(event);
            case FOLDER_CREATED             -> createLocalFolder(event);
            case FOLDER_DELETED             -> deleteLocalFolder(event);
        }
    }

    private void downloadFile(SyncEvent event) {
        // Check for local conflict before downloading
        Optional<SyncConflict> conflict = conflictDetector.detect(event);
        if (conflict.isPresent()) {
            conflictResolver.resolve(conflict.get(), event);
            return;
        }
        downloadService.downloadDelta(event.getFileId(), event.getVersionId());
    }
}
```

### 7.4 Filesystem watcher

```java
@Service
public class FilesystemWatcher {
    private WatchService watchService;

    @PostConstruct
    public void startWatching() {
        watchService = FileSystems.getDefault().newWatchService();
        registerRecursive(syncRootPath);

        Thread watchThread = new Thread(this::watchLoop, "fs-watcher");
        watchThread.setDaemon(true);
        watchThread.start();
    }

    private void watchLoop() {
        while (true) {
            WatchKey key = watchService.take(); // blocks until event
            for (WatchEvent<?> event : key.pollEvents()) {
                Path path = ((Path) key.watchable()).resolve(
                    (Path) event.context()
                );
                // Debounce: wait 500ms for rapid successive events (autosave storms)
                changeDebouncer.schedule(path, event.kind(), 500, MILLISECONDS);
            }
            key.reset();
        }
    }

    // Debouncer coalesces rapid events: 10 autosave events in 500ms → 1 sync trigger
    // Without debounce: every keystroke in autosaving app triggers chunk computation
}
```

## 8. Conflict Resolution

### 8.1 When conflicts occur

```
Conflict: two devices modify the same file while at least one is offline.

Scenario:
  T=0:  File state = [v1] on all devices
  T=1:  Laptop goes offline
  T=2:  Phone edits file → uploads v2 (server has v2)
  T=3:  Laptop edits file (from v1 base) → comes online → tries to upload
  T=4:  Laptop's upload is based on v1, but server has v2 → CONFLICT

No conflict:
  Laptop comes online, downloads v2 first, then edits v2 → no conflict
  (Linear history: v1 → v2 → v3)

Conflict:
  Laptop uploads while based on v1 when server already has v2
  (Forked history: v1 → v2 (phone's) and v1 → v3 (laptop's))
```

### 8.2 Conflict detection

```java
@Service
public class ConflictDetector {

    // Detect conflict during file upload finalisation
    public Optional<ConflictInfo> detect(FinaliseUploadRequest req) {
        FileVersion serverVersion = fileVersionRepo.findLatest(req.getFileId());

        // Client sends the version it was editing from (baseVersionId)
        if (serverVersion == null) return Optional.empty(); // new file, no conflict

        if (serverVersion.getVersionId().equals(req.getBaseVersionId())) {
            return Optional.empty(); // client was editing the latest — no conflict
        }

        // Server has a newer version than what client based its edit on → CONFLICT
        return Optional.of(ConflictInfo.of(
            req.getFileId(),
            req.getBaseVersionId(),       // client's base
            serverVersion.getVersionId(), // server's current (conflicting)
            req.getUploaderId()
        ));
    }

    // Client-side: detect conflict when receiving remote update
    public Optional<ConflictInfo> detectLocal(SyncEvent remoteEvent, Path localPath) {
        String localFingerprint = localState.getFingerprint(localPath);
        String remoteBaseFingerprint = remoteEvent.getBaseFingerprint();

        // Local file has been modified since the remote event's base version
        boolean localModified = !localFingerprint.equals(remoteBaseFingerprint);
        // Remote file is a different version
        boolean remoteModified = !remoteEvent.getNewFingerprint().equals(localFingerprint);

        if (localModified && remoteModified) {
            return Optional.of(ConflictInfo.local(localPath, remoteEvent));
        }
        return Optional.empty();
    }
}
```

### 8.3 Conflict resolution strategy

```java
@Service
public class ConflictResolver {

    // Strategy: preserve both versions — never silently discard user data
    public void resolve(ConflictInfo conflict) {

        // 1. Keep the server version as the canonical version
        //    (first-to-arrive wins, which is the device that uploaded first)

        // 2. Save the conflicting version as a "conflict copy" alongside the original
        //    Name: "report (Laptop's conflicted copy 2024-03-15).docx"
        String conflictName = buildConflictName(
            conflict.getOriginalFilename(),
            conflict.getUploaderDevice(),
            Instant.now()
        );

        // 3. Create a new file entry for the conflict copy in the same folder
        //    with its own version history starting from the conflict version
        fileService.createConflictCopy(
            conflict.getFileId(),
            conflict.getFolderId(),
            conflictName,
            conflict.getConflictingChunks()
        );

        // 4. Notify user on all devices
        notificationService.notifyConflict(
            conflict.getOwnerId(),
            conflict.getOriginalFilename(),
            conflictName
        );

        // User sees:
        //   report.docx                                    ← server version (kept)
        //   report (Laptop's conflicted copy 2024-03-15).docx  ← conflict copy
        // User can manually merge or discard the conflict copy
    }

    private String buildConflictName(String original, String device, Instant time) {
        String ext = getExtension(original);          // ".docx"
        String base = removeExtension(original);      // "report"
        String date = DateTimeFormatter.ISO_LOCAL_DATE.format(
            time.atZone(ZoneId.systemDefault())
        );
        return String.format("%s (%s's conflicted copy %s)%s",
            base, device, date, ext);
    }
}
```

### 8.4 Last-write-wins vs conflict copy

```
Two common strategies:

Last-write-wins (LWW):
  Whoever uploads last silently overwrites.
  Simple; no user notification needed.
  Dangerous: user who uploaded first silently loses their work.
  Used by: Google Drive (with version history as safety net)

Conflict copy (Dropbox approach):
  Both versions preserved as separate files.
  User explicitly sees and resolves the conflict.
  Safer: no data loss.
  Noisier: users see extra files in their folder.

This design: conflict copy (safer for user data).
With version history: either way, user can recover via version history
  (previous versions retained for 30 days regardless).
```

### 10.2 Access control check

```java
@Service
public class AccessControlService {

    public boolean canAccess(long userId, long fileId, Permission required) {
        // 1. Is the user the owner?
        File file = fileRepository.findById(fileId);
        if (file.getOwnerId() == userId) return true; // owner has full access

        // 2. Is there a direct share for this file?
        Optional<FileShare> directShare = shareRepository
            .findByResourceAndUser("file", fileId, userId);
        if (directShare.isPresent() && satisfies(directShare.get().getPermission(), required)) {
            return true;
        }

        // 3. Is there an inherited share (ancestor folder is shared)?
        List<Long> ancestorFolderIds = folderService.getAncestors(file.getFolderId());
        for (long folderId : ancestorFolderIds) {
            Optional<FileShare> folderShare = shareRepository
                .findByResourceAndUser("folder", folderId, userId);
            if (folderShare.isPresent() && satisfies(folderShare.get().getPermission(), required)) {
                return true; // inherited permission from shared ancestor
            }
        }

        return false;
    }

    // Cache access decisions (TTL 60s) — called on every file operation
    @Cacheable(value = "acl", key = "#userId + ':' + #fileId + ':' + #required")
    public boolean canAccessCached(long userId, long fileId, Permission required) {
        return canAccess(userId, fileId, required);
    }
}
```

### 11.2 Metadata DB caching

```java
@Service
public class MetadataCacheService {
    // Three-level cache for file metadata

    // L1: Local process cache (Caffeine) — sub-millisecond, bounded size
    private final Cache<Long, FileMetadata> l1 = Caffeine.newBuilder()
        .maximumSize(100_000)
        .expireAfterWrite(30, SECONDS)
        .build();

    // L2: Redis cluster — shared across all API server instances
    // Key: file:{fileId} → serialised FileMetadata
    // TTL: 5 minutes; invalidated on file update

    // L3: PostgreSQL (source of truth)

    public FileMetadata getFile(long fileId) {
        // L1 lookup
        FileMetadata cached = l1.getIfPresent(fileId);
        if (cached != null) return cached;

        // L2 lookup
        String json = redis.get("file:" + fileId);
        if (json != null) {
            FileMetadata meta = deserialise(json);
            l1.put(fileId, meta);
            return meta;
        }

        // L3 fallback
        FileMetadata meta = fileRepository.findById(fileId);
        redis.set("file:" + fileId, serialise(meta), 300, SECONDS);
        l1.put(fileId, meta);
        return meta;
    }

    public void invalidate(long fileId) {
        l1.invalidate(fileId);
        redis.delete("file:" + fileId);
        // DB is updated by the write path before invalidation
    }
}
```

### 11.3 Cold storage migration

```java
// Move old file versions to cheaper cold storage (S3 Glacier / Coldline)
// Triggered when version age > 30 days AND it is not the current version

@Scheduled(cron = "0 0 3 * * *") // 3 AM daily
public void migrateColdChunks() {
    // Find chunks referenced only by old versions (not current)
    List<String> coldChunkHashes = chunkRepository.findChunksOnlyInOldVersions(
        Instant.now().minus(30, DAYS)
    );

    for (String hash : coldChunkHashes) {
        // Copy to S3 Glacier (lower cost, higher retrieval latency)
        s3Client.copyObject(CopyObjectRequest.builder()
            .sourceBucket("chunks-hot")
            .destinationBucket("chunks-cold")
            .sourceKey(hash)
            .destinationKey(hash)
            .storageClass(StorageClass.GLACIER)
            .build());

        // Delete from hot bucket
        s3Client.deleteObject("chunks-hot", hash);

        // Update chunk record: location = cold
        chunkRepository.markCold(hash);
    }
}

// On access of cold chunk (user views old version):
//   Initiate S3 Glacier restore (3-5 hours for standard, 1-5 min for expedited)
//   Return 202 Accepted: "Retrieval in progress, check back in X hours"
//   Or: proactively restore if user browses version history
```

## 12. Interview Questions

**Q: Why is chunking essential for a file sync system, and how does the chunk size choice affect performance?**

A: Without chunking, any change to a file — even editing one sentence of a 500 MB document — requires re-uploading the entire file. For a user who auto-saves every 30 seconds, this is catastrophic in terms of bandwidth and time. Chunking splits the file into fixed or content-defined blocks, computes a hash for each, and only re-uploads blocks whose hash changed. For typical document edits, 1-3 of perhaps 125 chunks change in a 500 MB file, reducing upload size from 500 MB to 4-12 MB — a 40-100× improvement. Chunk size is a direct tradeoff: smaller chunks (512 KB) detect changes at finer granularity (less redundant upload per edit) but create more metadata overhead and more S3 PUT requests; larger chunks (8 MB) mean fewer API calls and less metadata but more wasted bandwidth when large chunks must be re-uploaded due to small edits. Four megabytes is a common sweet spot. Content-defined chunking (CDC with Rabin fingerprinting) is a further optimisation: it places chunk boundaries based on content patterns rather than fixed positions, so inserting text at the start of a file only changes the first chunk, whereas fixed-size chunking shifts every chunk boundary and forces re-uploading the entire file.

---

**Q: How does content-addressed chunk storage achieve deduplication, and what is the reference counting problem?**

A: Content-addressed storage uses the SHA-256 hash of a chunk's content as its storage key. Two users uploading identical files — say, a popular PDF — produce identical chunk hashes and store to identical S3 keys. Since S3 key collisions result in overwriting with the same content, the second upload is effectively a no-op at the storage layer, and both users' file metadata points to the same chunk. This gives cross-user deduplication without any explicit dedup service. The reference counting problem arises when we want to garbage-collect chunks that are no longer needed: if chunk hash ABC is shared by three file versions and one is deleted, we must not delete the S3 object until all references are gone. A chunk_references table tracks the reference count per hash. Deleting a file version decrements reference counts for all its chunks atomically; a background garbage collection job deletes S3 objects whose count has reached zero. The concurrency hazard is a race between a reference count reaching zero and a new upload of the same content: we must not delete the S3 object between the reference count hitting zero and a concurrent upload's PUT reaching S3. The solution is a grace period: chunks with count=0 are marked eligible for GC but only deleted after 24 hours, giving concurrent uploads time to increment the count and write the S3 object.

---

**Q: Walk through the complete conflict resolution flow when two devices edit the same file simultaneously.**

A: User edits `report.docx` on their laptop (device A, based on version v2) while offline. Simultaneously on their phone (device B), they also edit `report.docx` and it uploads, creating server version v3. When the laptop reconnects, the sync client sends a finalise request with baseVersionId=v2. The server's conflict detector compares this to the current server version v3: since v2 ≠ v3, and both devices started from v2 (a fork in history), this is a conflict. The server accepts device A's upload as a new version (v4) but marks it as a conflict copy. It creates a new file entry in the same folder named `report (Laptop's conflicted copy 2024-03-15).docx` pointing to device A's chunks. The canonical `report.docx` remains at v3 (phone's version, which arrived first). A sync event is pushed to all the user's devices: `report.docx` is at v3, and a new file `report (Laptop's conflicted copy...).docx` exists. The user sees both files in their folder and must manually decide which version to keep or how to merge them. Version history remains available for both, so no data is ever lost. The alternative, last-write-wins, would silently overwrite v3 with v4, discarding the phone's changes — unacceptable for a user-facing sync tool.

---

**Q: How does the sync cursor mechanism work, and what happens when a device has been offline for more than 30 days?**

A: Each device maintains a cursor: the Snowflake event_id of the last sync event it has successfully processed. When the device connects, it sends `GET /sync?cursor=12345`. The server queries `SELECT * FROM sync_events WHERE user_id=? AND event_id > 12345 ORDER BY event_id ASC LIMIT 1000`. The device applies each event in order (since Snowflake IDs are monotonically increasing, ordering is guaranteed), updates its local filesystem, and advances its cursor to the highest event_id received. This is fundamentally event-sourcing applied to filesystem state — the event log is the authoritative record of all changes. If a device is offline for more than 30 days, the event log has been partially archived to cold storage. Rather than restoring archived events, the server triggers a full state sync: it returns the current complete file tree (all files, folders, versions for the user) as a snapshot, and the client diffs this against its local state — downloading missing files, deleting locally absent files, and resolving version mismatches. The device then receives a fresh cursor pointing to the current latest event_id, resuming incremental sync from that point forward. Full state sync is more expensive (bandwidth and compute) but rare; incremental cursor sync handles the 99.9% case of regularly connected devices.

---

**Q: How would you design the chunk upload to be resumable after a network interruption mid-upload?**

A: Resumability is handled at two levels. At the chunk level: each chunk is an independent S3 PUT request. If it fails, only that chunk must be retried — completed chunks are already durably stored in S3 and verified by their SHA-256 hash. The client tracks which chunks have been confirmed (server returned 200 for their pre-signed PUT) and retries only unconfirmed chunks on reconnect. At the session level: the uploadId returned at init serves as the session identifier. On reconnect, the client calls `POST /upload/{uploadId}/check` with all chunk hashes; the server returns which are already in S3 and issues new pre-signed URLs only for missing ones. This means interrupting a 5 GB upload after 4 GB completes and reconnecting 3 days later requires uploading only the remaining 1 GB. The uploadId is valid for 7 days. After that, the client re-initiates; the chunk check still skips already-stored chunks (content-addressed storage means previously uploaded chunks are found regardless of uploadId). For very large single chunks (a 4 MB chunk on a mobile connection), S3 itself supports multipart upload for objects over 5 MB, but since our chunk size is 4 MB, a single S3 PUT is sufficient and the retry granularity is the chunk itself.

## 13. Key Takeaways

- **Chunking purpose**: delta sync (only re-upload changed blocks), parallel upload, dedup across users, resumable transfer — 4 MB fixed blocks is the practical default
- **Content-defined chunking (CDC)**: Rabin fingerprinting finds content-based boundaries; insert at start → only first chunk changes (vs fixed chunking where all chunks shift)
- **Content-addressed storage**: SHA-256(chunk) as S3 key → identical chunks stored once across all users; dedup is automatic and zero-cost; rename/move = zero upload
- **Pre-signed URLs**: client uploads directly to S3 (bypasses app servers); saves 185 GB/s inbound on API fleet; time-limited, permission-scoped
- **Sync cursor = event log position**: `device_sync_cursors.last_event_id` advances as events applied; send cursor → get delta; offline > 30 days → full state snapshot sync
- **Conflict copy not LWW**: both conflicting versions preserved as separate files; user resolves manually; version history is the safety net either way
- **Reference counting for GC**: chunk_references table; decrement on version delete; GC only when count=0 AND 24-hour grace period elapsed (prevents race with concurrent upload)
- **Folder hierarchy as materialised path**: `/Documents/Work/Q4/report.pdf` stored as string; range scan on prefix; bulk rename/move = UPDATE with REPLACE; more efficient than recursive CTE at scale
- **Three-tier metadata cache**: Caffeine L1 (per-process, 30s TTL) → Redis L2 (shared, 5-min TTL) → PostgreSQL L3 (source of truth); invalidate on write
- **Chunk cold storage migration**: TimeWindow or nightly job moves chunks from S3 Standard to S3 Glacier after 30 days of being in old versions only; S3 Glacier restore on demand for version history access
