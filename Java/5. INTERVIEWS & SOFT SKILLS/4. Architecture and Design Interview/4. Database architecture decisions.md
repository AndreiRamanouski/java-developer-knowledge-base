# Database Architecture Decisions

## Overview

Database architecture decisions are among the longest-lived choices in a system. A framework selection can be swapped in a sprint; a database migration affecting petabytes of production data takes months and significant risk. Understanding how to choose, scale, replicate, and recover databases — and how to explain these trade-offs to an interviewer — is one of the most reliable signals of senior engineering judgment.

**The hierarchy of database decisions:**

```
1. What type of database? (relational, document, wide-column, graph, time-series)
   → Most consequential; hardest to reverse

2. What topology? (single node, primary-replica, multi-primary, sharded)
   → Determined by scale, consistency, and availability requirements

3. How to partition data? (sharding strategy)
   → Determines query routing, load distribution, and operational complexity

4. How to handle multiple tenants? (multi-tenancy model)
   → Determines isolation, cost, and operational overhead

5. How to evolve schema? (migration strategy)
   → Determines deployment risk and rollback capability

6. How to recover from failure? (backup and DR strategy)
   → Determines RTO, RPO, and the operational cost of those guarantees
```

## 1. Choosing Database Type

### 1.1 The decision framework

```
There is no universally correct database. Every database makes trade-offs.
The question is: which trade-offs best match your access patterns, consistency
requirements, and operational capabilities?

Start with three questions:
  1. What is the primary access pattern? (primary key lookup, range query,
     graph traversal, time-series aggregation, full-text search)
  2. What consistency is required? (strong ACID, eventual, flexible)
  3. What scale is anticipated? (writes/second, read:write ratio, data volume,
     geographic distribution)

The answers determine the candidate database types.
```

### 1.2 Relational databases (PostgreSQL, MySQL)

```
Best for:
  Complex queries with joins across multiple entities
  Transactions that span multiple tables (ACID required)
  Structured data with well-understood, relatively stable schema
  Reporting, analytics on operational data
  Referential integrity is a requirement (FK constraints)

Postgres-specific strengths:
  JSONB for semi-structured data alongside structured columns
  Full-text search (GIN index on tsvector)
  Partial indexes, expression indexes, covering indexes
  LISTEN/NOTIFY for lightweight pub/sub
  Logical replication for CDC (Debezium)
  PostGIS for geospatial queries
  Excellent support for complex queries (CTEs, window functions, arrays)

Limits:
  Vertical scaling only for writes (one primary)
  Schema migrations can be disruptive on large tables
  Complex to shard (requires middleware like Citus, Vitess, or application-level)
  Not suitable for: graph traversal, time-series at very high ingestion, full-text at scale

When NOT to use:
  High-cardinality time-series (use TimescaleDB or ClickHouse)
  Graph traversal (use Neo4j or Amazon Neptune)
  Simple key-value at millions of ops/sec (use Redis or DynamoDB)
  Document storage where schema is highly variable (use MongoDB)
```

### 1.3 Document databases (MongoDB, Firestore)

```
Best for:
  Semi-structured or variable schema data
  Nested document models (order + orderItems in one document)
  High write throughput with flexible schema
  Applications where schema changes frequently (early-stage products)
  Content management systems, user profiles, catalogs

Strengths:
  Schema flexibility: no migration required to add a field
  Locality: related data in one document → one read for complete object
  Horizontal scaling: sharding is a first-class feature
  Development speed: schema changes are additive

Limits:
  No joins across collections (application-level joins = N+1 problem)
  Limited transaction support across multiple documents (MongoDB 4+ has multi-doc transactions, but limited vs PostgreSQL)
  Document size limits (MongoDB: 16 MB)
  Consistency trade-offs in distributed mode
  Poor for relational queries (reporting across many documents requires aggregation pipeline)

When NOT to use:
  Data that requires complex joins between entities
  Strong ACID transactions across multiple entities
  Regulatory compliance requiring referential integrity
  Complex analytical queries
```

### 1.4 Wide-column stores (Cassandra, HBase, Bigtable)

```
Best for:
  Very high write throughput (millions of writes/second)
  Time-series data with known access patterns
  Append-heavy workloads (event logs, audit trails, IoT sensor data)
  Geographic distribution (multi-datacenter replication is first-class)
  Known, specific read patterns (no ad-hoc queries)

Cassandra strengths:
  Masterless architecture: no single primary; all nodes equal
  Tunable consistency: adjust per operation (ONE, QUORUM, ALL)
  Built-in multi-datacenter replication
  Linear horizontal scaling: double nodes → double throughput
  LSM tree: extremely fast writes (append to memtable)
  Compaction: background data organisation for read efficiency

Limits:
  Data model is query-driven: schema must be designed around known queries
  No joins; denormalisation is required
  Limited support for ad-hoc queries
  Updates are appends (tombstones accumulate; compaction required)
  Read performance degrades with too many tombstones or wide partitions

When NOT to use:
  Varied, ad-hoc query patterns
  Strong consistency requirements (banking, financial ledgers)
  Complex relationships between entities
  Small data (<1TB) where operational complexity is not justified
```

### 1.5 Key-value stores (Redis, DynamoDB)

```
Redis:
  Primary use: caching, session storage, rate limiting counters,
               pub/sub, queues, leaderboards
  Data structures: String, Hash, List, Set, Sorted Set, Stream, HyperLogLog
  Not for: primary persistent storage (ephemeral by default; persistence is add-on)
  Strengths: sub-millisecond latency, atomic operations, Lua scripting

DynamoDB:
  Primary use: high-scale applications requiring predictable single-digit ms latency
  Strengths: fully managed, serverless pricing, auto-scaling, global tables
  Limits: query patterns must be designed upfront; limited ad-hoc querying;
          strong consistency on reads costs 2× RCUs
  Best for: session stores, shopping carts, game leaderboards, IoT state
```

### 1.6 Decision matrix

```
Requirement                       Best candidates
──────────────────────────────────────────────────────────────────────────────
Complex relational queries        PostgreSQL, MySQL
ACID multi-entity transactions    PostgreSQL, MySQL (CockroachDB for distributed)
High write throughput (M/sec)     Cassandra, DynamoDB, MongoDB
Time-series at scale              TimescaleDB, InfluxDB, ClickHouse (OLAP)
Graph traversal                   Neo4j, Amazon Neptune, JanusGraph
Full-text search                  Elasticsearch, OpenSearch, PostgreSQL (small scale)
Sub-millisecond cache             Redis, Memcached
Document / flexible schema        MongoDB, Firestore, Couchbase
Analytics / OLAP                  ClickHouse, Redshift, BigQuery, Snowflake
Geospatial                        PostgreSQL + PostGIS, MongoDB (geospatial indexes)
Multi-model (one DB, many uses)   PostgreSQL (JSONB + GIN + PostGIS + FTS)
```

## 2. Sharding Strategies

### 2.1 Why sharding and when

```
Vertical scaling (bigger machine) has a ceiling. A single PostgreSQL primary
handles ~10K-50K simple queries per second depending on hardware. Beyond that,
horizontal scaling (more machines) is required.

When to shard:
  Write throughput exceeds what a single primary can handle
  Dataset exceeds what a single machine can store cost-effectively
  Operational requirements: all data in specific regions (GDPR, data residency)

When NOT to shard prematurely:
  Read replicas can absorb most read load without sharding
  Caching absorbs 90-99% of reads
  Sharding adds complexity: cross-shard queries, transactions, and rebalancing
  Most applications never need sharding; premature sharding is a common mistake

Alternatives to sharding (try these first):
  Read replicas for read scaling
  Connection pooling (PgBouncer) for connection overhead
  Caching (Redis) for repeated reads
  Table partitioning (native PostgreSQL) for large tables
  Vertical scaling (bigger machine) — cheaper than you think until truly large scale
```

### 2.2 Hash-based sharding

```
Algorithm: shard = hash(shard_key) % num_shards

Example: user data sharded by user_id
  hash("user_12345") % 4 = shard 2
  All data for user_12345 lives on shard 2.

Pros:
  Even distribution of data across shards (uniform hash)
  Simple routing: one hash function determines the shard
  No hotspot (assuming well-distributed hash values)

Cons:
  Rebalancing on shard count change: adding 1 shard → nearly all keys rehash
    → mass data migration required (use consistent hashing to mitigate)
  Range queries cross shards: "all users registered in January" → fan-out to all shards
  No control over data co-location

Consistent hashing mitigates rebalancing:
  Place shards on a virtual ring.
  Adding a shard: only keys between new shard and its predecessor migrate (~1/N of keys).
  Adding virtual nodes (150+ per physical shard): ensures uniform distribution.
  Used by Cassandra, DynamoDB, Redis Cluster (variant with 16,384 slots).
```

### 2.3 Range-based sharding

```
Algorithm: shard = range that contains shard_key value

Example: orders sharded by order_id ranges
  Shard 1: order_id 0 - 999,999
  Shard 2: order_id 1,000,000 - 1,999,999
  Shard 3: order_id 2,000,000+

Or by time: shard by month/year
  Shard 2024-01: all orders from January 2024
  Shard 2024-02: all orders from February 2024

Pros:
  Range queries are efficient: "all orders from Q1 2024" → one or two shards
  Data locality: recent data on the latest shard (useful for hot/cold tiering)
  Predictable: you know exactly which shard holds which data

Cons:
  Hotspots: if all new data goes to the latest shard, one shard absorbs all writes
    → "hot shard" problem; common with sequential IDs or time-based keys
  Uneven distribution: some ranges may have far more data than others
  Rebalancing: when a range is too large, it must be split manually

Mitigating hotspots in range sharding:
  Add a random salt prefix to distribute within the range shard
  Use Snowflake IDs with machine ID prefix to spread writes across shards
  Pre-split expected growth ranges before they become hot
```

### 2.4 Directory-based sharding

```
Algorithm: a lookup service maps shard_key → shard_id

Example:
  ShardDirectory:
    "tenant_acme"  → shard 3
    "tenant_beta"  → shard 1
    "tenant_gamma" → shard 3

Pros:
  Maximum flexibility: move any entity to any shard without changing the algorithm
  Supports tenant isolation: put large tenants on dedicated shards
  No data rehashing when adding shards

Cons:
  Lookup service is a critical SPOF (must be replicated and cached)
  Extra network hop for every request (mitigated by client-side caching of the directory)
  Operational complexity of maintaining the directory

Best for: multi-tenant SaaS where large tenants need isolated shards;
          systems where manual shard assignment is a business requirement.
```

### 2.5 Cross-shard queries

```
Cross-shard queries (scatter-gather) are the primary operational cost of sharding.

When a query cannot be routed to a single shard:
  Fan-out: send the query to all N shards in parallel.
  Collect and merge results at the application layer.
  Sort and paginate the merged results.

Performance implications:
  Latency = max(individual shard latency), not sum — parallel helps.
  But: N shards × tail latency → higher p99 than single-shard query.
  Network overhead × N.
  Application must handle partial failures (one shard down ≠ query fails, but results incomplete).

Strategies to minimise cross-shard queries:
  Choose a shard key that co-locates the data most commonly queried together.
  Denormalise: store summary data in a global index shard.
  Materialise cross-shard results in a reporting database.
  Use a CQRS read model that aggregates across shards asynchronously.

The right shard key question: "What query do we run most often, and what key
    would allow that query to hit exactly one shard?"
```

## 3. Replication Topology

### 3.1 Single primary / multiple replicas (most common)

```
Architecture:
  One primary: all writes go here.
  N replicas: read from here; replicate from primary asynchronously.

┌──────────┐     writes     ┌─────────────────────────────┐
│  Client  │───────────────▶│         Primary              │
└──────────┘                └────────┬──────────┬──────────┘
                                     │          │ async replication
                            ┌────────▼───┐  ┌───▼────────┐
                            │  Replica 1 │  │  Replica 2 │
                            └────────────┘  └────────────┘
                                  ▲                ▲
                                  └── reads ────────┘

Replication lag: async replication introduces lag (typically milliseconds to seconds).
  If primary fails before replica syncs: data loss equal to the lag (RPO > 0).
  Synchronous replication: zero lag but write latency = network RTT to replica.
  Most systems use async for performance and accept the small data loss window.

PostgreSQL replication modes:
  Synchronous:  primary waits for replica ACK before confirming write.
                Zero data loss; higher write latency; replica failure blocks writes.
  Asynchronous: primary does not wait; replica applies changes later.
                Fast writes; small data loss window on primary failure.
  Logical:      replicate specific tables or schemas (for CDC, cross-version migration).
  Physical:     byte-for-byte copy of the data files (for standby, HA).
```

### 3.2 Multi-primary replication

```
Architecture:
  Multiple primary nodes: each can accept writes.
  Replication between all primaries.

Use cases:
  Geographic distribution: each region has a primary; local writes, global sync.
  High write availability: if one primary fails, others continue accepting writes.

Challenges:
  Write conflicts: two primaries accept conflicting writes to the same row simultaneously.
  Resolution strategies:
    Last Write Wins (LWW): based on timestamp. Simple; may lose changes if clocks skew.
    Conflict-free Replicated Data Types (CRDTs): data structures designed to merge
    without conflicts (counters, sets, maps with specific semantics).
    Application-level resolution: write custom merge logic.

  Avoiding conflicts is better than resolving them:
    Route all writes for one entity to the same primary (consistent hashing of entity key to region).
    Accept writes only in the user's home region; replicate to other regions read-only.

Products: CockroachDB, YugabyteDB, MySQL Group Replication, PostgreSQL BDR.
Cost: significantly more complex than single-primary; justify only for geographic latency or write HA.
```

### 3.3 Galera Cluster / Synchronous multi-primary

```
Architecture: N nodes, all primaries, synchronous replication via group communication.
  Every write is committed to all nodes before the transaction completes.

Pros: no data loss on any single node failure; any node can accept writes.
Cons: write latency = max(node network RTT); sensitive to network partitions;
      poor for geo-distributed deployments (high cross-region RTT).

Best for: regional HA where nodes are in the same datacenter or region.
Not suitable for: global multi-region deployments (latency is prohibitive).
```

### 3.4 Read replicas: routing and staleness

```java
// Route writes to primary, reads to replicas
// AbstractRoutingDataSource in Spring: dynamically selects DataSource per request
public class ReplicationRoutingDataSource extends AbstractRoutingDataSource {

    @Override
    protected Object determineCurrentLookupKey() {
        return TransactionSynchronizationManager.isCurrentTransactionReadOnly()
            ? "replica"
            : "primary";
    }
}

// Service layer annotation
@Transactional(readOnly = true)  // routes to replica
public List<Order> findOrders(Long userId) { ... }

@Transactional           // routes to primary
public Order createOrder(CreateOrderRequest req) { ... }

// Staleness handling for read-your-writes consistency:
// After a write, the session should read from primary for N seconds
// to avoid reading stale data from a lagging replica.
@Component
public class ReplicationLagRouter {
    private final Map<String, Instant> recentWriters = new ConcurrentHashMap<>();
    private static final Duration LAG_WINDOW = Duration.ofSeconds(5);

    public boolean shouldUsePrimary(String userId) {
        Instant lastWrite = recentWriters.get(userId);
        return lastWrite != null && Duration.between(lastWrite, Instant.now()).compareTo(LAG_WINDOW) < 0;
    }

    public void recordWrite(String userId) {
        recentWriters.put(userId, Instant.now());
    }
}
```

## 4. Multi-Tenancy Approaches

### 4.1 The three models

```
Multi-tenancy: serving multiple customers (tenants) from the same infrastructure.
Three models, ordered from highest isolation to lowest isolation.

Model 1: Database per tenant
  Each tenant has their own database instance.
  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐
  │  Tenant A   │  │  Tenant B   │  │  Tenant C   │
  │  Database   │  │  Database   │  │  Database   │
  └─────────────┘  └─────────────┘  └─────────────┘

  Pros:
    Maximum isolation (a bug in tenant A never affects tenant B)
    Easy to customise schema per tenant (enterprise customisation)
    Compliance: data is physically isolated (GDPR, HIPAA, SOC2)
    Easy to migrate a single tenant (copy their database)
    Backup and restore per tenant

  Cons:
    Operational overhead: N databases to manage, monitor, backup, upgrade
    Resource inefficiency: idle tenants consume resources
    Schema migrations: run against all N databases (automation required)
    Connection overhead: N databases × connection pools

  Best for: enterprise B2B SaaS where isolation is a contractual requirement;
            regulated industries (healthcare, finance); large tenants.

Model 2: Schema per tenant (within one database instance)
  One database instance, separate schemas per tenant.
  ┌──────────────────────────────────────────────┐
  │              PostgreSQL Instance              │
  │  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
  │  │ tenant_a │  │ tenant_b │  │ tenant_c │   │
  │  │ .orders  │  │ .orders  │  │ .orders  │   │
  │  └──────────┘  └──────────┘  └──────────┘   │
  └──────────────────────────────────────────────┘

  Pros:
    Better isolation than shared tables (no query cross-contamination)
    Schema customisation per tenant possible
    Easier compliance than shared tables
    Moderate operational overhead (one database to manage)

  Cons:
    Schema migrations multiply by N (every migration runs N times)
    Performance isolation: a slow query in tenant A schema can still affect shared DB resources
    Most databases have soft limits on number of schemas (PostgreSQL: practical limit ~thousands)
    Not suitable for very large numbers of tenants (10K+)

  Best for: mid-market SaaS with moderate tenant counts (hundreds to low thousands);
            compliance requirements but not maximum isolation.

Model 3: Shared table with tenant_id column
  All tenants share the same tables; rows are discriminated by tenant_id.
  ┌──────────────────────────────────┐
  │           orders table           │
  │  tenant_id │ order_id │ amount   │
  │  tenant_a  │ 1001     │ 99.99    │
  │  tenant_b  │ 1002     │ 149.00   │
  │  tenant_a  │ 1003     │ 49.50    │
  └──────────────────────────────────┘

  Pros:
    Lowest operational overhead (one schema to manage)
    Resource efficiency (tenants share connection pools, cache)
    Schema migrations: one migration for all tenants
    Supports very large tenant counts (millions of small tenants)

  Cons:
    No physical data isolation (a bug or query error could expose cross-tenant data)
    Noisy neighbour: one tenant's heavy query affects everyone
    Compliance difficulty (auditor sees all tenants in one place)
    Every query must include WHERE tenant_id = ? (application-level enforcement; easy to forget)
    Row-Level Security (PostgreSQL RLS) enforces this at DB level but adds complexity

  Best for: B2C SaaS with millions of small tenants; applications where
            tenants have low data volumes and low isolation requirements.
```

### 4.2 Row-Level Security for shared table model

```sql
-- PostgreSQL Row-Level Security: enforces tenant isolation at the database level
-- Prevents application bugs from accidentally returning cross-tenant data

-- Enable RLS on the orders table
ALTER TABLE orders ENABLE ROW LEVEL SECURITY;
ALTER TABLE orders FORCE ROW LEVEL SECURITY;  -- applies to table owner too

-- Policy: users can only see rows where tenant_id = their app.current_tenant setting
CREATE POLICY tenant_isolation ON orders
    USING (tenant_id = current_setting('app.current_tenant')::uuid);

-- Application: set the tenant context on each connection before querying
@Before each query:
    conn.execute("SET app.current_tenant = '" + tenantId + "'");

-- Now any query on this connection automatically filters to the current tenant.
-- Forgetting WHERE tenant_id = ? in application code no longer causes cross-tenant exposure.
-- The database enforces it transparently.

-- Index: always include tenant_id as the leading column
CREATE INDEX idx_orders_tenant_id ON orders(tenant_id, created_at DESC);
-- Without this index, every query scans the full table then filters by tenant_id.
```

### 4.3 Hybrid multi-tenancy

```
Production SaaS systems often use a hybrid:
  Small tenants (< $1K MRR): shared table model (efficient)
  Mid tenants ($1K-$10K MRR): schema per tenant (moderate isolation)
  Enterprise tenants (> $10K MRR or compliance required): database per tenant

Routing:
  Tenant registry service maps tenant_id → {model: "shared|schema|database", connection_info}
  Application uses the registry to route connections at startup.
  Migrations run differently per tier.

This is the model used by Shopify, Salesforce, and most large SaaS platforms.
The key is the routing layer — tenants can be migrated between tiers as they grow
without changing application code.
```

## 5. Schema Migration Strategies

### 5.1 Why schema migrations are high-risk

```
A schema migration on a production database is one of the highest-risk
operations in software engineering. The combination of:
  Large tables (millions to billions of rows)
  Live traffic (queries reading/writing during migration)
  Irreversibility (an ALTER TABLE on a live database cannot be easily undone)
  Locking (many DDL operations acquire table locks that block all queries)

...means a poorly planned migration can cause minutes to hours of downtime.

Types of schema changes and their risk profiles:
  Low risk (no lock, fast):
    Adding a nullable column with no default
    Dropping an index
    Adding a non-unique index (in most databases, CONCURRENT creation)

  Medium risk (brief lock or sequential scan):
    Adding a NOT NULL column with a non-trivial default (must populate all rows)
    Adding a UNIQUE constraint (requires scanning for duplicates)
    Adding a foreign key constraint (validates entire table)

  High risk (long exclusive lock):
    DROP COLUMN on a large table (rewrites the table in some databases)
    RENAME COLUMN (must update all dependent queries simultaneously)
    Changing column type (may require full table rewrite)
    Adding a non-null column without default (must fill all existing rows)
    Rebuilding a table (VACUUM FULL, CLUSTER)
```

### 5.2 Zero-downtime migration patterns

```
Pattern 1: Expand-contract (most reliable for schema changes)
  Phase 1 (Expand): Add new column/table alongside old one. Both old and new
                    code run simultaneously. New column is nullable; old remains.
  Phase 2 (Migrate): Backfill new column from old data (batch, low priority).
                     Application reads from new, writes to both old and new.
  Phase 3 (Contract): Once migration is confirmed complete, drop old column/table.
                      New code only reads from new.

  Example: renaming column "user_email" to "email_address"
    Phase 1: ADD COLUMN email_address VARCHAR(255); (nullable — safe, instant)
    Phase 2: UPDATE users SET email_address = user_email WHERE email_address IS NULL LIMIT 1000;
             (batched, low-priority; runs overnight; new code writes to both columns)
    Phase 3: DROP COLUMN user_email; (after all code deployed and all rows backfilled)
             Then ALTER COLUMN email_address SET NOT NULL; (fast since all rows filled)

Pattern 2: Online schema change tools
  pg_repack (PostgreSQL): rewrites table in background, uses triggers to capture
                          live changes, swaps table when complete.
  gh-ost (GitHub, MySQL): GitHub's tool for online schema migrations; zero-lock table rewrites.
  pt-online-schema-change (Percona, MySQL): uses triggers for live capture.

  These tools allow high-risk operations (column type changes, index rebuilds)
  without downtime by doing the work in the background while the table is live.

Pattern 3: Create-then-populate for new tables
  CREATE TABLE new_table (...); — fast, no lock
  INSERT INTO new_table SELECT ... FROM old_table WHERE ...; — batch in background
  Atomic swap: rename or redirect reads when population is complete
  Never: CREATE TABLE ... AS SELECT ... — this holds locks for the entire duration
```

### 5.3 Migration tooling (Flyway and Liquibase)

```java
// Flyway: SQL-first migration tool
// Naming convention: V{version}__{description}.sql
// Migrations are applied in version order; each applied exactly once

// V1__create_orders_table.sql
CREATE TABLE orders (
    id          BIGSERIAL       PRIMARY KEY,
    user_id     BIGINT          NOT NULL,
    status      VARCHAR(20)     NOT NULL DEFAULT 'PLACED',
    total       DECIMAL(10,2)   NOT NULL,
    created_at  TIMESTAMPTZ     NOT NULL DEFAULT NOW()
);
CREATE INDEX idx_orders_user ON orders(user_id, created_at DESC);

// V2__add_currency_to_orders.sql
ALTER TABLE orders ADD COLUMN currency VARCHAR(3);  -- nullable first (safe)

// V3__backfill_currency_and_add_not_null.sql
-- Run only after V2 is deployed and all rows backfilled by background job
UPDATE orders SET currency = 'USD' WHERE currency IS NULL;
ALTER TABLE orders ALTER COLUMN currency SET NOT NULL;

// Spring Boot configuration
spring:
  flyway:
    enabled: true
    locations: classpath:db/migration
    validate-on-migrate: true     // fail fast if applied migrations changed
    out-of-order: false           // reject migrations applied out of sequence
    baseline-on-migrate: false    // never skip applied migrations in production

// CI/CD integration:
// Run flyway:migrate in CI against a production-schema database
// Any migration that fails means the deployment fails before touching production
```

### 5.4 Migration safety checklist

```
Before any migration on a large production table (>1M rows):

Technical checks:
  ✓ Tested on a production-sized replica (not staging with 10K rows)
  ✓ Execution time measured on the replica
  ✓ Rollback plan documented: what exact SQL undoes this migration?
  ✓ Used pg_repack or CONCURRENTLY for index operations
  ✓ Used expand-contract for column renames or type changes
  ✓ Batch backfill for data population (not single giant UPDATE)
  ✓ Verified no long-running transactions will hold lock on table
  ✓ Lock timeout set: SET lock_timeout = '5s' (fail fast if lock not acquired)
  ✓ Statement timeout set: SET statement_timeout = '30min' (prevent runaway migrations)

Operational checks:
  ✓ Maintenance window communicated to stakeholders
  ✓ On-call engineers available during migration window
  ✓ Database backup verified recent (restore tested, not just created)
  ✓ Monitoring dashboards open during migration
  ✓ Rollback communication plan in place
  ✓ Metrics baseline captured before migration for comparison
```

## 6. Backup and Disaster Recovery

### 6.1 RTO and RPO — the contractual basis of DR design

```
Recovery Time Objective (RTO):
  Maximum acceptable duration of downtime.
  "Our SLA requires 99.9% uptime, which allows 8.7 hours/year of downtime."
  Lower RTO = more expensive (hot standby vs warm vs cold).

Recovery Point Objective (RPO):
  Maximum acceptable data loss measured in time.
  "We can tolerate losing at most 1 hour of data."
  Lower RPO = more frequent backups or synchronous replication.

RTO/RPO to architecture mapping:
  RTO < 1 minute, RPO = 0:      Active-active multi-region with synchronous replication
                                  → Very expensive; rarely justified except financial systems
  RTO < 5 minutes, RPO < 1 min: Hot standby with async replication + Patroni/PgBouncer
                                  → Most common for Tier 1 services
  RTO < 1 hour, RPO < 15 min:  Warm standby with WAL shipping + manual failover
  RTO < 4 hours, RPO < 1 hour:  Regular backup + restore (cloud snapshot)
  RTO < 24 hours, RPO < 24 hr:  Daily backup; acceptable for low-criticality data

Business translation:
  The cost of your DR architecture must be less than the cost of outages × probability.
  RTO=0 may cost $1M/year in infrastructure; if downtime costs $10K/hour,
  you need > 100 hours/year of outages to justify it. Usually 5-10 minutes
  of RTO is the right balance for most applications.
```

### 6.2 Backup types and implementation

```
1. Logical backups (pg_dump, mysqldump)
  Exports data as SQL statements.
  Portable: can restore to a different database version.
  Selective: back up specific tables or schemas.
  Slow: serialises all data; for large databases, can take hours.
  Not suitable as the only backup for very large databases.

2. Physical backups (pg_basebackup, XtraBackup)
  Copies the raw data files.
  Fast: filesystem-level copy.
  Not portable: must restore to same or compatible database version.
  Requires consistent state: database must be paused or use backup mode.

3. Continuous WAL archiving (PostgreSQL WAL / MySQL binlog)
  Archive every write-ahead log file to object storage (S3).
  Enables point-in-time recovery (PITR): restore to any second in history.
  Combined with a base backup: base + WAL replay = any point in time.
  This is the production standard for PostgreSQL databases.

4. Snapshot backups (cloud provider)
  EBS snapshot (AWS), Persistent Disk snapshot (GCP).
  Near-instant capture (copy-on-write).
  Restores to a point in time (snapshot + subsequent WAL for PITR).
  Works in conjunction with WAL archiving, not instead of it.

Recommended strategy for production PostgreSQL:
  Daily pg_basebackup or EBS snapshot (base backup)
  Continuous WAL archiving to S3 (pgBackRest or WAL-G)
  Weekly logical backup (pg_dump) for portability and data verification
  Test restores monthly in a non-production environment
```

### 6.5 Disaster recovery runbook template

```
INCIDENT SEVERITY: P0 (database unavailable)
ESTIMATED RTO: 15 minutes
ESTIMATED RPO: 5 minutes (last WAL archive)

STEP 1: Confirm the failure (2 minutes)
  □ Verify primary is unreachable: psql -h primary-host -U postgres -c "SELECT 1"
  □ Check Patroni status: patronictl -c patroni.yml list
  □ Check application error logs: confirm DB connection errors, not application bugs
  □ Decision: primary failure confirmed → proceed to Step 2

STEP 2: Failover (5 minutes)
  □ If Patroni auto-failover has not occurred: patronictl -c patroni.yml failover postgres-ha
  □ Verify new primary: patronictl -c patroni.yml list | grep Leader
  □ Verify PgBouncer connection: psql -h pgbouncer-host -p 5432 -U postgres -c "SELECT 1"
  □ Application health check: curl -f https://app.example.com/health

STEP 3: Monitor (8 minutes)
  □ Monitor application error rates (target: back to baseline within 2 minutes of failover)
  □ Monitor replication lag on any remaining replicas
  □ Check WAL archiving resumed on new primary: SELECT pg_current_wal_lsn()

STEP 4: Root cause and recovery (after service restoration)
  □ Investigate why primary failed (instance failure, disk, network)
  □ Provision replacement replica from existing backup
  □ Rejoin replacement as replica: patronictl reinit postgres-ha node1
  □ Verify replication is caught up: SELECT pg_wal_lsn_diff(...)
  □ Write post-mortem within 24 hours

ESCALATION:
  If failover does not complete within 10 minutes: page database team lead
  If data loss suspected: page VP Engineering + DBA lead immediately
```

### 6.6 Monitoring database health

```sql
-- Replication lag (critical alert: > 60 seconds)
SELECT
    client_addr,
    state,
    pg_wal_lsn_diff(pg_current_wal_lsn(), sent_lsn) AS send_lag_bytes,
    pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn) AS replay_lag_bytes,
    write_lag, flush_lag, replay_lag
FROM pg_stat_replication;

-- Long-running queries (alert: > 5 minutes)
SELECT pid, now() - pg_stat_activity.query_start AS duration, query
FROM pg_stat_activity
WHERE state = 'active'
  AND query_start < now() - interval '5 minutes'
ORDER BY duration DESC;

-- Blocking locks (alert: any lock held > 30 seconds)
SELECT
    blocked.pid     AS blocked_pid,
    blocked.query   AS blocked_query,
    blocking.pid    AS blocking_pid,
    blocking.query  AS blocking_query
FROM pg_catalog.pg_locks blocked_locks
JOIN pg_catalog.pg_stat_activity blocked ON blocked.pid = blocked_locks.pid
JOIN pg_catalog.pg_locks blocking_locks ON blocking_locks.locktype = blocked_locks.locktype
    AND blocking_locks.relation = blocked_locks.relation
    AND blocking_locks.pid != blocked_locks.pid
JOIN pg_catalog.pg_stat_activity blocking ON blocking.pid = blocking_locks.pid
WHERE NOT blocked_locks.granted;

-- Table bloat (alert: bloat > 50% of table size)
SELECT schemaname, tablename,
    pg_size_pretty(pg_total_relation_size(quote_ident(schemaname) || '.' || quote_ident(tablename))) AS total_size,
    dead_tup_ratio
FROM pg_stat_user_tables
WHERE dead_tup_ratio > 0.5
ORDER BY dead_tup_ratio DESC;
```

## 7. Interview Questions

**Q: How do you choose between a relational database and a NoSQL solution for a new service?**

A: I start with the access patterns rather than the data shape, because the database must serve queries efficiently, not just store data conveniently. Three questions drive the decision. First, does the service require ACID transactions across multiple entities? Order creation that atomically creates an order, reserves inventory, and charges payment is a canonical ACID requirement — that points strongly to PostgreSQL. If the service is append-only, like an event log or sensor readings, a transaction requirement is less likely. Second, what is the primary read pattern? A high-volume key-value lookup by a primary key — user session state, product price by SKU — is served equally well by PostgreSQL with an index or DynamoDB, with DynamoDB having lower operational overhead at scale. Ad-hoc multi-dimensional queries — "show me all orders from Germany in Q4 over $500" — are served by PostgreSQL far better than DynamoDB. Third, what is the anticipated scale for writes? PostgreSQL with a single primary handles approximately 10-50K simple writes per second depending on the hardware. Beyond that, Cassandra or DynamoDB scale linearly where PostgreSQL needs sharding. I also consider operational capability: a team that has never operated Cassandra should not adopt it for a service that could work on PostgreSQL, because the operational risk of an unfamiliar system under production pressure is real. The best database is the one your team can operate correctly at 2 AM.

---

**Q: Explain the trade-offs of the three multi-tenancy models. When would you choose each?**

A: The three models exist on a spectrum from maximum isolation to maximum efficiency. Database per tenant gives each customer their own isolated instance — a bug in tenant A's query cannot affect tenant B's data, schema changes can be per-tenant, and compliance auditors find it easy to verify isolation. The cost is operational multiplication: every management operation (backup, migration, monitoring, upgrade) must run N times, typically automated. This is the right choice when tenants have high data volumes, regulatory isolation requirements like HIPAA or GDPR data residency, or enterprise contracts that demand it. Schema per tenant uses one database instance with separate schemas. Isolation is good — queries are contained to one schema — but they share database resources, so a heavy query in tenant A's schema still consumes CPU and I/O that tenant B needs. Migrations must run N times but are easier to parallelize. This fits mid-market SaaS with hundreds of tenants and moderate isolation needs. Shared tables with a tenant_id column are maximally efficient — one migration, shared connection pools, best resource utilisation — but have the weakest isolation. A missing WHERE tenant_id = ? clause in a query could expose all tenants' data; row-level security in PostgreSQL enforces this at the database layer and is strongly recommended. This is the right model for B2C or SMB SaaS with many small tenants where isolation is not a contractual requirement. Most large SaaS companies end up with a hybrid: small tenants on shared tables, enterprise tenants on dedicated databases, with a routing layer that handles the indirection.

---

**Q: How do you perform a zero-downtime schema migration on a large production table?**

A: The expand-contract pattern handles nearly every schema change safely. For a column rename — say "user_email" to "email_address" — a naive RENAME COLUMN takes a brief table lock and forces an atomic switchover of all reading and writing code. Zero-downtime requires doing this in three phases. The expansion phase adds the new column as nullable, which is an instant metadata operation with no lock: `ALTER TABLE users ADD COLUMN email_address VARCHAR(255)`. Application code is deployed that writes to both columns but reads from the old one. The migration phase backfills the new column from old data in small batches, running as a background job with intentional delays to avoid lock contention: `UPDATE users SET email_address = user_email WHERE email_address IS NULL AND id BETWEEN X AND Y`. When all rows are backfilled, a second deployment switches reads to the new column. The contraction phase drops the old column once all code no longer references it. For index creation, `CREATE INDEX CONCURRENTLY` in PostgreSQL builds the index in the background without holding a table lock — the trade-off is that it takes longer and cannot be run inside a transaction. For table rewrites — changing a column type, adding a non-null column without a default — I use pg_repack, which creates the new table structure in the background, captures live changes via triggers, and atomically swaps the tables with only a brief metadata lock. The most important practice: always test migrations on a production-sized replica first. A migration that takes 30 seconds on a 50K-row staging database may take 45 minutes on a 500M-row production table.

---

**Q: What is the difference between RPO and RTO, and how do they drive database architecture decisions?**

A: RTO (Recovery Time Objective) is how long the business can tolerate the system being unavailable — the maximum downtime window. RPO (Recovery Point Objective) is how much data the business can tolerate losing — the maximum lookback window for recovery. They are independent dimensions, and each has a cost. A hot standby with synchronous replication achieves near-zero RPO (zero data loss) and low RTO (< 1 minute failover) but costs 2× the infrastructure and adds write latency for every transaction because the primary waits for replica confirmation. An asynchronous replica achieves low RTO but non-zero RPO — the RPO equals the replication lag at the moment of failure, typically seconds to minutes. A warm standby that receives WAL archives every 5 minutes achieves RPO = 5 minutes and RTO = several minutes for promotion and reconnection. A daily backup achieves RPO = 24 hours and RTO = however long it takes to restore, potentially hours. The practical question is: what does one hour of downtime cost the business, and what does one hour of lost data cost? If downtime costs $100K/hour and a hot standby costs $50K/year in additional infrastructure, the break-even is less than 30 minutes of prevented downtime per year. For most businesses, a 5-minute RTO and 1-minute RPO using Patroni with asynchronous replication and WAL archiving is the right balance — it handles the common failure modes (hardware failure, primary crash) with a brief, automatable failover, at a cost proportional to the benefit.

**Q: Describe your approach to designing a sharding strategy for a rapidly growing application.**

A: I approach sharding as a last resort after exhausting vertical scaling and read replicas, because it introduces irreversible complexity. When it becomes necessary, the shard key decision is the most consequential choice in the design. The shard key must satisfy three properties: it must be present on most queries (queries that do not include the shard key become scatter-gather operations across all shards); it must have high cardinality (few distinct values create hot shards); and it must distribute write load evenly (avoid sequential keys like auto-increment IDs that funnel all new writes to one shard). For a social platform, user_id is usually the right shard key — all of a user's data co-locates on one shard, and most queries are in the context of a specific user. For a marketplace, tenant_id or seller_id might be better than order_id, because order history is always queried in the context of a user or seller. The queries that cross shard boundaries — "most popular products globally" — are handled through a denormalised reporting aggregate that is updated asynchronously, rather than scatter-gather queries. I also plan for rebalancing before it is needed. Consistent hashing with virtual nodes means adding a shard only migrates 1/N of the data. I provision more virtual nodes than physical shards initially so that physical rebalancing is just reassigning existing virtual nodes to new hardware, not rehashing keys. Directory-based sharding, where a routing service maps tenant to shard, gives maximum flexibility for migrating large tenants to dedicated shards as they grow, at the cost of the routing service becoming a critical dependency.

## 8. Key Takeaways

- **Choose the database for the access pattern, not the data shape**: the question is "what queries must be fast?" not "what does the data look like?" — relational for complex joins, Cassandra for high-write time-series, Redis for sub-millisecond key-value
- **Try read replicas and caching before sharding**: 90-99% of read load disappears with aggressive caching; sharding adds irreversible operational complexity that should only be introduced when genuinely necessary
- **Shard key selection is the most consequential decision in a sharded design**: must be present in most queries, high cardinality, and evenly distributed — a wrong shard key produces hot shards that negate all the scaling benefit
- **Expand-contract for zero-downtime migrations**: add new structure alongside old; backfill in batches; switch reads; drop old — never rename a column or change a type without this pattern on large production tables
- **Always test migrations on a production-sized replica first**: a 30-second migration on 50K rows can be 45 minutes on 500M rows — the production database should never be the first place you discover this
- **RPO drives replication strategy; RTO drives standby topology**: synchronous replication for zero data loss (costs write latency); async + Patroni for 5-minute RTO (practical for most); WAL archiving for point-in-time recovery
- **Multi-tenancy model follows data volume and isolation requirements**: shared tables for millions of small tenants; schema per tenant for hundreds with moderate isolation needs; database per tenant for enterprise with compliance requirements
- **Row-Level Security enforces tenant isolation at the database layer**: application-level WHERE tenant_id = ? is error-prone; RLS makes the wrong query structurally impossible
- **WAL archiving + base backups = point-in-time recovery**: enables restoring to any second in the retention window; pgBackRest or WAL-G implement this on S3; test restores monthly
- **Patroni for PostgreSQL HA achieves < 30-second automatic failover**: etcd/Consul for leader election; PgBouncer sits in front for transparent connection handling; the runbook is just as important as the technology
