# Performance Architecture

## Overview

Performance architecture is the discipline of designing systems that remain fast, efficient, and stable under the load they are built to serve — and that degrade gracefully when load exceeds expectations. The distinguishing mark of a senior engineer in this domain is not knowing which cache to add next; it is knowing how to identify the actual bottleneck, understanding the trade-offs of each optimisation, and building measurable feedback loops that validate improvements.

**The performance optimisation hierarchy:**

```
Measure first, optimise second. Always.
  "We optimised the database queries and improved performance by 40%."
  → How do you know the database was the bottleneck?
  → 40% improvement in what? p50? p99? Throughput? Error rate?

The optimisation hierarchy (try in order; stop when the target is met):
  1. Architecture:  eliminate unnecessary work entirely
                    (cache, pre-compute, push offline, do less)
  2. Algorithm:     reduce computational complexity (O(n²) → O(n log n))
  3. Data structure: choose the structure that matches the access pattern
  4. Concurrency:   parallelise independent work
  5. Resource:      scale hardware (last resort, never without measuring)

The most common mistake: jumping to step 5 (add more machines) before
trying steps 1-4. Horizontal scaling hides waste; it rarely eliminates it.
```

## 1. Identifying Bottlenecks

### 1.1 The universal bottleneck identification process

```
Step 1: Define what "slow" means in measurable terms.
  "The application is slow" is not actionable.
  "p99 latency on GET /products is 2,400ms; target is 500ms" is actionable.
  Agree on: which metric, which percentile, what target, measured how.

Step 2: Identify where time is being spent (profiling).
  A request that takes 2,400ms spends that time somewhere specific.
  Distributed trace: which service is the bottleneck?
  APM span: which function or query within that service?
  CPU profiler: which code path consumes the most CPU?
  DB slow log: which query is executing for too long?

Step 3: Formulate a hypothesis.
  "I believe the bottleneck is a full-table scan on the products search query
  because the slow query log shows it running for 1,800ms consistently."
  Hypothesis = specific, falsifiable claim about the cause.

Step 4: Test the hypothesis with a controlled change.
  Add the index. Measure again.
  Did p99 improve? By how much? Did something else get worse?
  One change at a time — never change two things and try to attribute the result.

Step 5: Verify the bottleneck moved.
  After fixing the original bottleneck: another will emerge.
  The next bottleneck is often not where you expect.
  Re-profile to find it before assuming what to fix next.
```

### 1.2 The four resource bottlenecks

```
Every system bottleneck is one of four resources: CPU, memory, I/O, or network.

CPU-bound:
  Symptoms: CPU > 80% sustained, high CPU-wait, low I/O wait
  Causes: expensive computations (cryptography, compression, image processing,
          regex on large strings), inefficient algorithms, deserialisation overhead
  Fixes: offload to background workers, use faster algorithms, cache results,
         GPU acceleration for appropriate workloads

Memory-bound:
  Symptoms: high memory usage, GC pauses (JVM), OOM kills, swapping to disk
  Causes: large object graphs, memory leaks, cache without eviction policy,
          uncontrolled collection growth
  Fixes: fix leaks, add eviction policies, reduce object size, tune GC settings,
         stream large datasets instead of loading into memory

I/O-bound (disk):
  Symptoms: high iowait, slow disk throughput, sequential reads of large files
  Causes: random I/O instead of sequential, missing indexes (full table scans),
          large unindexed queries, excessive logging
  Fixes: add indexes, switch to sequential access patterns, use SSD if on HDD,
         move hot data to memory (cache), reduce write amplification

Network-bound:
  Symptoms: high network latency, bandwidth saturation, many small round trips
  Causes: chatty APIs (N+1 calls), large uncompressed payloads, cross-region calls,
          synchronous chains of service calls
  Fixes: batch requests, compress payloads, move computation closer to data,
         async where synchronous is not required, CDN for static assets
```

### 1.3 Tools for bottleneck identification

```
Application Performance Monitoring (APM):
  Datadog APM, New Relic, Dynatrace, Elastic APM
  Provides: distributed traces, service dependency maps, error rates,
            latency histograms, slow transaction identification
  Use: identify which service in a distributed system is slow

Distributed Tracing (OpenTelemetry + Jaeger/Zipkin):
  Instrument every service call with trace/span context.
  A trace shows the full call tree for a single request.
  Immediately identifies: which hop is slow (DB? External API? CPU-bound service?)
  Essential for microservices; without it, you are guessing.

Database profiling:
  PostgreSQL: pg_stat_statements (aggregate stats), auto_explain (plan per query),
              slow query log (pg_log with log_min_duration_statement = 100)
  MySQL: Performance Schema, slow_query_log, EXPLAIN ANALYZE

JVM profiling:
  async-profiler: CPU and allocation profiling, flame graphs, low overhead
  JFR (Java Flight Recorder): continuous profiling with negligible overhead
  VisualVM / JConsole: heap and thread analysis

Infrastructure metrics:
  Prometheus + Grafana: CPU, memory, I/O, network per instance
  USE method: Utilisation, Saturation, Errors for every resource
    Utilisation: how busy is the resource (0-100%)?
    Saturation: is there a queue waiting for the resource?
    Errors: is the resource producing errors?
  RED method: Rate, Errors, Duration for every service
    These are the user-visible metrics that matter most.

Profiling in production:
  async-profiler: attaches to a running JVM with negligible overhead (< 3%)
  Continuous profiling (Pyroscope, Parca): always-on profiling aggregated over time
  Sample at 99Hz: captures enough signal without overwhelming the system
```

### 1.4 Flame graphs

```
A flame graph visualises where CPU time is spent across a call tree.
Width of a frame = proportion of total CPU time.
Reading a flame graph:
  Wide frames at the top: these functions are directly consuming the most CPU.
  Wide frames at the bottom: these are called frequently from many paths.
  Plateau (flat top): a function is on the CPU a lot; investigate this.

Interpreting a Java flame graph:
  Wide "GC" frames: garbage collection is a bottleneck → reduce allocations or tune GC
  Wide serialisation frames: Jackson serialising large objects on every request
    → cache serialised form, reduce response size, use more efficient format
  Wide "connection wait" frames: thread pool exhaustion → increase pool or reduce latency upstream

Generating a flame graph with async-profiler:
  ./profiler.sh -d 30 -f profile.html -e cpu <pid>
  → 30-second CPU profile, interactive HTML flame graph
  → Share with the team; visible exactly where time goes
```

## 2. Multi-Level Caching Strategies

### 2.1 The caching layer hierarchy

```
Level 1 (L1): In-process cache (per JVM instance)
  Implementation: Caffeine, Guava Cache
  Latency: < 1 microsecond (in-process memory access)
  Size: limited by JVM heap (typically 10K-100K entries)
  Scope: per-instance (different instances may have different cached values)
  TTL: short (30 seconds to 5 minutes — instances diverge over time)
  Best for: hottest data (config, reference data, frequently read but rarely changed)

Level 2 (L2): Distributed cache (shared across instances)
  Implementation: Redis Cluster, Memcached
  Latency: < 1 millisecond (network round-trip to co-located Redis)
  Size: limited by Redis memory (GBs to TBs with clustering)
  Scope: shared across all application instances
  TTL: minutes to hours
  Best for: session data, API responses, computed aggregations

Level 3 (L3): HTTP cache / CDN edge
  Implementation: CloudFront, Fastly, Varnish, browser cache
  Latency: < 5ms from edge node (close to user)
  Size: large (CDN has terabytes of edge storage globally)
  Scope: global (same cached response served to users worldwide)
  TTL: hours to days for immutable content; minutes for dynamic
  Best for: public API responses, static assets, computed pages

Level 4 (L4): Database query cache
  Implementation: materialised views, denormalised summary tables
  Latency: 1-10ms (database round-trip but O(1) query instead of O(n) join)
  Size: limited by database storage
  Scope: shared
  TTL: managed by refresh job or trigger
  Best for: expensive aggregations queried frequently (leaderboards, dashboards)
```

### 2.2 Cache patterns

```java
// Pattern 1: Cache-aside (most flexible, most common)
// Application checks cache; on miss, loads from DB, stores in cache.
// On write: UPDATE DB, then DELETE cache (not update — avoids stale write race).
public User getUser(long userId) {
    String key = "user:" + userId;

    // L1 check
    User l1 = l1Cache.getIfPresent(key);
    if (l1 != null) return l1;

    // L2 check
    String json = redis.get(key);
    if (json != null) {
        User user = deserialise(json, User.class);
        l1Cache.put(key, user);      // populate L1
        return user;
    }

    // L3: database
    User user = userRepository.findById(userId).orElseThrow();
    redis.setex(key, 300, serialise(user));  // populate L2, 5-min TTL
    l1Cache.put(key, user);                  // populate L1
    return user;
}

public void updateUser(long userId, UserUpdateRequest req) {
    userRepository.update(userId, req);
    // Invalidate AFTER commit via TransactionSynchronization.afterCommit()
    redis.del("user:" + userId);
    l1Cache.invalidate("user:" + userId);
}

// Pattern 2: Read-through (cache library fetches on miss; app never talks to DB directly)
// Cleaner code; library handles the miss loading.
// Harder to implement with complex loading logic.

// Pattern 3: Write-through (write to cache and DB simultaneously)
// Cache always has latest value; no cold miss after write.
// Risk: cache has data that is never read (wasteful).

// Pattern 4: Write-behind / write-back (write to cache, async flush to DB)
// Very fast writes; risk of data loss if cache fails before flush.
// Use for: counters (view counts, like counts), shopping carts, session data.
@Scheduled(fixedRate = 30_000)
public void flushDirtyEntries() {
    Set<String> dirty = dirtyKeys.getAll();
    for (String key : dirty) {
        String value = redis.get(key);
        if (value != null) db.update(parseKey(key), value);
        dirtyKeys.remove(key);
    }
}
```

### 2.3 Cache sizing and eviction

```
Cache sizing formula:
  Identify the working set: the data that is actually accessed repeatedly.
  For a Pareto (80/20) distribution: 20% of data receives 80% of requests.
  Cache the top 20% → hit rate ~80%.
  Cache the top 50% → hit rate ~95% (diminishing returns).

  Working set estimate:
    Measure: distinct cache keys accessed in a 24-hour window.
    Size: distinct_keys × average_value_size × 1.2 (20% headroom).
    For 1M daily active users × 200 bytes per user record × 1.2 = 240 MB.

Eviction policies:
  LRU (least recently used):  evict the key not accessed for the longest time.
    Best for: general-purpose; workloads with temporal locality.
    Risk: scan pollution (a one-time large scan evicts hot working set).

  LFU (least frequently used): evict the key accessed fewest times.
    Best for: stable frequency distributions (popular items stay; stale drop).
    Risk: frequency counts become stale; old popular items are never evicted.

  ARC (adaptive replacement cache): dynamically balances LRU and LFU.
    Best for: mixed workloads; self-tuning without manual policy selection.
    Available in: ZFS, some Redis configurations.

  Redis maxmemory-policy:
    allkeys-lru:   LRU across all keys (recommended general-purpose)
    allkeys-lfu:   LFU across all keys (recommended for stable frequency)
    volatile-ttl:  evict key with shortest remaining TTL first
    noeviction:    return error when memory full (only if cache must not lose data)
```

### 2.4 Cache stampede prevention

```
Cache stampede (thundering herd): popular key expires; hundreds of requests
simultaneously miss and flood the database.

Prevention techniques:

1. Mutex / request coalescing:
  First thread to miss acquires a Redis lock.
  Other threads wait on the lock.
  Lock holder queries DB, populates cache, releases lock.
  All waiting threads read from populated cache.

  String lockKey = "lock:" + cacheKey;
  if (redis.setnx(lockKey, "locked", 5 SECONDS)) {
      // I hold the lock: fetch from DB and populate cache
      User user = db.findUser(userId);
      redis.setex(cacheKey, 300, serialise(user));
      redis.del(lockKey);
      return user;
  } else {
      // Wait briefly and retry (another thread is populating)
      Thread.sleep(50);
      return getUser(userId); // recursive retry
  }

2. Probabilistic early expiration (XFetch):
  Don't wait for expiry; proactively refresh before it expires.
  Each access has a small probability of triggering a refresh proportional
  to how close to expiry the key is.

  boolean shouldRefresh(String key, double beta) {
      long ttl = redis.pttl(key);  // ms remaining
      if (ttl <= 0) return true;   // expired
      double recomputeTimeMs = measureRecomputeTime(); // from recent history
      // Refresh if random probability exceeds threshold
      return -beta * recomputeTimeMs * Math.log(Math.random()) > ttl;
  }

3. Stale-while-revalidate:
  Serve the stale cached value immediately.
  Trigger an async background refresh.
  Next request (milliseconds later) gets the fresh value.
  Cost: one request sees stale data; all subsequent get fresh.
  Best for: data where brief staleness is acceptable (news feeds, product prices).

  Cache-Control: max-age=60, stale-while-revalidate=30
  → Serve cached response for 60 seconds.
  → In seconds 60-90: serve stale but trigger background refresh.
  → After 90: block until revalidated.
```

## 3. Database Optimisation

### 3.1 Index design for performance

```
The golden rule: every query's WHERE clause, JOIN condition, and ORDER BY
should be covered by an index or explicitly accepted as a full scan.

Index types and uses:
  B-tree (default):  equality (=), range (<, >, BETWEEN), ORDER BY, prefix LIKE
  Hash:              equality only; no range queries (rarely used explicitly)
  GIN:               array contains, JSONB key existence, full-text search
  GiST:              geometric data, range types, full-text
  BRIN:              large tables with naturally ordered data (timestamps, log files)
  Partial index:     indexes a subset of rows (WHERE active = true); smaller, faster

The covering index (index-only scan):
  An index that includes all columns needed by the query avoids touching the table.
  CREATE INDEX idx_orders_user_status_total ON orders(user_id, status)
      INCLUDE (total_amount, created_at);
  Query: SELECT total_amount, created_at FROM orders WHERE user_id=? AND status='PLACED'
  → Index-only scan: reads only the index, never touches the table.
  → Significant improvement for read-heavy queries on large tables.

Index column ordering:
  Equality conditions before range conditions.
  (user_id, status): supports WHERE user_id=? AND status=?
                     and WHERE user_id=? (partial match)
  (status, user_id): supports WHERE status=? only efficiently
                     (not WHERE user_id=? alone — must scan all statuses)

When NOT to index:
  Low-cardinality columns (boolean, enum with 3 values): full scan may be faster
  Tables with very high write volume: each index adds overhead to every INSERT/UPDATE
  Never-queried columns: index maintenance cost with no benefit
  Columns already leading in a composite index covering the same queries
```

### 3.2 Query optimisation

```sql
-- Anti-pattern 1: N+1 queries
-- WRONG: one query for orders, then one query per order for customer name
for order in orders:
    customer = db.findById(order.customerId)  -- N queries
-- RIGHT: JOIN or batch load
SELECT o.*, u.name AS customer_name
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.user_id = ?;

-- Anti-pattern 2: SELECT * (fetches unnecessary columns)
-- WRONG:
SELECT * FROM products WHERE category_id = ?;
-- RIGHT (only fetch what the query needs):
SELECT id, name, price, thumbnail_url FROM products WHERE category_id = ?;

-- Anti-pattern 3: Functions on indexed columns break index usage
-- WRONG (cannot use index on email column):
SELECT * FROM users WHERE LOWER(email) = 'user@example.com';
-- RIGHT (use a functional index or store lowercase):
CREATE INDEX idx_users_email_lower ON users(LOWER(email));

-- Anti-pattern 4: LIKE with leading wildcard
-- WRONG: full table scan regardless of index
SELECT * FROM products WHERE name LIKE '%headphone%';
-- RIGHT: full-text search
SELECT * FROM products
WHERE to_tsvector('english', name) @@ to_tsquery('headphone:*');
CREATE INDEX idx_products_fts ON products USING GIN(to_tsvector('english', name));

-- Anti-pattern 5: Unbounded queries without LIMIT
-- WRONG: returns all matching rows (could be millions)
SELECT * FROM events WHERE user_id = ? ORDER BY created_at DESC;
-- RIGHT:
SELECT * FROM events WHERE user_id = ? ORDER BY created_at DESC LIMIT 20;

-- EXPLAIN ANALYZE: understand what PostgreSQL actually does
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT id, name, price FROM products
WHERE category_id = 5 AND price < 100
ORDER BY price ASC LIMIT 20;
-- Look for: "Seq Scan" (missing index), large "rows" estimate vs actual,
--           "Sort" on large datasets (missing covering index for ORDER BY)
```

### 3.3 Denormalisation for read performance

```
Normalisation eliminates redundancy; denormalisation trades redundancy for speed.
When a normalised query is too slow and indexing cannot fix it: denormalise.

Scenario: user dashboard showing each order with product name and seller name.
Normalised:
  SELECT o.id, p.name, u.name
  FROM orders o
  JOIN order_items oi ON o.id = oi.order_id
  JOIN products p ON oi.product_id = p.id
  JOIN users u ON p.seller_id = u.id
  WHERE o.buyer_id = ?
  → 3 joins; expensive for large tables

Denormalised (snapshot at order time):
  orders table includes: product_name, seller_name (columns added at order creation time)
  → Single table query; no joins at read time
  Trade-off: product_name or seller_name changes do not update historical orders
             (often correct: you want the name at time of purchase, not today's name)

Forms of denormalisation:
  Snapshot columns: embed frequently read data at write time (order includes product name)
  Materialised views: pre-computed JOIN results refreshed on schedule or on trigger
  Summary tables: aggregated counts/sums maintained by triggers or background jobs
  Separate read model: CQRS pattern — dedicated table optimised for a specific query

Materialised view example:
  CREATE MATERIALIZED VIEW product_stats AS
  SELECT
      p.id,
      p.name,
      COUNT(oi.id) AS total_sold,
      SUM(oi.price * oi.quantity) AS total_revenue,
      AVG(r.rating) AS avg_rating
  FROM products p
  LEFT JOIN order_items oi ON p.id = oi.product_id
  LEFT JOIN reviews r ON p.id = r.product_id
  GROUP BY p.id, p.name;

  CREATE UNIQUE INDEX ON product_stats(id);

  -- Refresh concurrently (does not block reads):
  REFRESH MATERIALIZED VIEW CONCURRENTLY product_stats;
  -- Schedule refresh every 15 minutes via pg_cron or external job
```

### 3.4 Connection pooling

```
Why connection pooling is critical:
  Creating a PostgreSQL connection: ~50ms (TCP handshake, auth, startup)
  Connection limit per PostgreSQL instance: typically 100-500
  Each application thread holding a connection while idle: wasted resource
  100 application servers × 20 threads = 2,000 simultaneous connections
  → PostgreSQL OOM; connection creation serialised

PgBouncer: the standard connection pooler for PostgreSQL
  Sits between application servers and PostgreSQL
  Maintains a small pool of DB connections; applications connect to PgBouncer
  Connection modes:
    Transaction pooling: connection returned to pool after each transaction
      → Best for most web applications (connection held only during transaction)
      → Allows many more application threads than DB connections
    Session pooling: connection held for the full client session
      → Required for session-scoped prepared statements
    Statement pooling: most restrictive; rarely used

Typical configuration:
  Application servers: 100 servers × 20 threads = 2,000 connections to PgBouncer
  PgBouncer → PostgreSQL: 100 connections (10× reduction)
  PostgreSQL comfortably handles 100 connections without pressure

pgbouncer.ini:
  [databases]
  mydb = host=postgres port=5432 dbname=mydb

  [pgbouncer]
  pool_mode = transaction
  max_client_conn = 10000
  default_pool_size = 100
  min_pool_size = 10
  reserve_pool_size = 10
  server_idle_timeout = 600
```

## 4. Async Processing

### 4.1 When to use asynchronous processing

```
Synchronous processing: client waits for the result.
  Appropriate for: read queries, validation, simple computations
  Inappropriate for: operations that take > 500ms, side effects that may fail
                     and need retry, operations independent of the user's workflow

Asynchronous processing: work is queued; client gets an immediate response.
  Appropriate for: sending emails, processing images/videos, generating reports,
                   updating analytics, propagating changes to downstream systems,
                   any operation where the result is not needed for the immediate response

The decision test:
  "Does the user need this result to continue their workflow RIGHT NOW?"
  No → make it asynchronous.

Examples:
  User uploads profile photo → resize to 5 resolutions → sync? NO
    → Respond immediately: "photo uploaded, processing..."
    → Resize in background worker
    → Push notification when complete

  User places order → charge payment → sync? YES
    → User needs confirmation that payment succeeded to complete checkout
    → But: sending confirmation email = async (not needed for checkout completion)

  User changes email → update all downstream systems → sync? NO
    → Change the email immediately (sync)
    → Propagate to CRM, email lists, audit logs = async
```

### 4.2 Message queue architecture for async processing

```
┌──────────────┐    publish    ┌─────────────┐    consume    ┌──────────────┐
│  API Server  │──────────────▶│   Kafka /   │──────────────▶│  Background  │
│              │               │   RabbitMQ  │               │  Workers     │
└──────────────┘               └─────────────┘               └──────────────┘
                                                                      │
                                                              ┌───────▼───────┐
                                                              │  Dead Letter  │
                                                              │  Queue        │
                                                              └───────────────┘

Queue choice:
  Kafka: high throughput, durable log, replay, multiple consumers, ordering
         Best for: event streams, audit logs, multiple downstream consumers
  RabbitMQ: flexible routing, immediate delivery, good for low-latency tasks
         Best for: task queues, RPC, routing by message type
  SQS/SNS: managed, serverless, deep AWS integration
         Best for: AWS-native architectures, auto-scaling workers

Work queue properties:
  At-least-once delivery: messages may be delivered more than once on failure
                           Workers must be idempotent (check if work already done)
  Dead letter queue: messages that fail repeatedly go here for manual review
  Visibility timeout: message hidden from other workers while being processed
                      If worker crashes before ack: message reappears after timeout
  Retry policy: exponential backoff (1s, 2s, 4s, 8s, max 32s) before DLQ
```

### 4.3 Async processing patterns

```java
// Pattern 1: Fire and forget
// Publish event; do not wait for processing; no result needed by caller.
@PostMapping("/users/{id}/photo")
public ResponseEntity<Void> uploadPhoto(@PathVariable Long id, @RequestPart MultipartFile file) {
    String objectKey = objectStore.store(id, file.getBytes()); // fast: S3 upload
    kafkaTemplate.send("photo-processing", new PhotoUploadedEvent(id, objectKey));
    return ResponseEntity.accepted().build(); // 202: queued, not yet done
}

// Pattern 2: Polling (client polls for completion)
// Appropriate for: operations taking minutes; user needs result eventually
@PostMapping("/reports")
public ResponseEntity<ReportJobResponse> generateReport(@RequestBody ReportRequest req) {
    String jobId = UUID.randomUUID().toString();
    jobRepository.save(new ReportJob(jobId, req, "QUEUED"));
    kafkaTemplate.send("report-jobs", new GenerateReportEvent(jobId, req));
    return ResponseEntity.accepted()
        .header("Location", "/reports/jobs/" + jobId)
        .body(new ReportJobResponse(jobId, "QUEUED"));
}

@GetMapping("/reports/jobs/{jobId}")
public ReportJobResponse getJobStatus(@PathVariable String jobId) {
    return jobRepository.findById(jobId); // returns QUEUED → PROCESSING → DONE
}

// Pattern 3: Webhook / push notification
// Server calls client URL when processing completes.
// Appropriate for: long-running jobs where the client is a server itself
@Component
public class ReportCompletionHandler {
    @KafkaListener(topics = "report-completed")
    public void onComplete(ReportCompletedEvent event) {
        ReportJob job = jobRepository.findById(event.getJobId());
        if (job.getWebhookUrl() != null) {
            webhookClient.post(job.getWebhookUrl(),
                new ReportReadyPayload(event.getJobId(), event.getReportUrl()));
        }
    }
}

// Pattern 4: Inbox pattern (idempotent worker)
// Prevents duplicate processing if message delivered twice.
@KafkaListener(topics = "photo-processing")
@Transactional
public void processPhoto(PhotoUploadedEvent event) {
    if (processedEvents.exists(event.getEventId())) return; // already done

    imageProcessor.resize(event.getObjectKey()); // the actual work
    processedEvents.record(event.getEventId()); // mark as done
}
```

## 5. CDN Usage

### 5.1 What to cache at the CDN layer

```
CDN-cacheable (static or semi-static):
  ✓ Static assets: JS, CSS, images, fonts, videos
    → Indefinite TTL with content-hash versioning (main.a3b4c5.js)
    → Cache-Control: public, max-age=31536000, immutable
    → When content changes: filename changes → automatic cache invalidation

  ✓ API responses that are public and cacheable:
    → Product catalog, pricing, news articles
    → Cache-Control: public, max-age=60, s-maxage=300 (CDN caches longer than browser)
    → s-maxage: TTL for shared caches (CDN); max-age: TTL for browser cache

  ✓ Pre-rendered HTML pages (SSG/ISR):
    → Gatsby, Next.js static generation
    → Cache-Control: public, max-age=3600, stale-while-revalidate=86400

NOT CDN-cacheable (user-specific or sensitive):
  ✗ API responses containing user-specific data (always add Vary: Authorization)
  ✗ Endpoints with POST/PUT/DELETE (never cached by CDN)
  ✗ Responses with Set-Cookie (browser prevents caching; mark private)
  ✗ Payment, authentication, session endpoints (always private)
```

### 5.2 Cache-Control header design

```
Key Cache-Control directives:
  public:          response may be cached by CDN and browser
  private:         response may be cached by browser only (not CDN)
  no-store:        never cache (analytics, sensitive data)
  no-cache:        cache but must revalidate before serving (ETag/If-None-Match)
  max-age=N:       browser cache TTL (seconds)
  s-maxage=N:      CDN cache TTL (overrides max-age for shared caches)
  stale-while-revalidate=N: serve stale while refreshing in background (0 perceived latency)
  stale-if-error=N: serve stale if origin returns 5xx (resilience)
  immutable:       content will never change; do not revalidate (for hashed assets)

Practical examples:
  Hashed static assets:
    Cache-Control: public, max-age=31536000, immutable
    → Browser caches forever; CDN caches forever; new filename on change

  Product listing (60s fresh, 5m stale-while-revalidate):
    Cache-Control: public, max-age=60, s-maxage=300, stale-while-revalidate=300
    → User sees data ≤ 60s old from browser cache
    → CDN refreshes from origin every 5 minutes
    → Background refresh serves stale during refresh; no latency spike

  User profile (private, not CDN-cacheable):
    Cache-Control: private, no-store
    → Browser does not cache; CDN does not cache; every request hits origin

  Authenticated API response:
    Cache-Control: no-store
    Vary: Authorization
    → Forces separate cache entries per Authorization header value
    → Prevents one user's response from being served to another
```

### 5.3 CDN edge functions for dynamic caching

```
Edge functions (Cloudflare Workers, AWS Lambda@Edge) enable:
  Personalised caching: strip PII, cache the sanitised response by partial key
  A/B testing at the edge: split traffic without origin involvement
  Authentication at edge: verify JWT before forwarding to origin
  Geolocation routing: direct traffic based on country/city
  Request transformation: add headers, modify URLs, canonical redirects

Use case: cache product pages with personalised sections
  Origin returns: product data (cacheable) + user recommendations (not cacheable)
  Edge function:
    1. Check cache for product data (cache key = productId)
    2. Forward to origin if cache miss
    3. Cache product data; strip the user-specific recommendations section
    4. Merge cached product + real-time user recommendations from API
    5. Return merged response (product from cache, recs fresh)
  Result: 90% of product page served from CDN; only the personalised section hits origin

// Cloudflare Worker example: cache product data by ID
addEventListener('fetch', event => {
  event.respondWith(handleRequest(event.request));
});

async function handleRequest(request) {
  const url = new URL(request.url);
  const productId = url.pathname.split('/')[2]; // /products/{id}

  const cacheKey = new Request(`https://cache.example.com/product/${productId}`);
  const cache = caches.default;

  let response = await cache.match(cacheKey);
  if (!response) {
    response = await fetch(request);
    response = new Response(response.body, response);
    response.headers.set('Cache-Control', 'public, max-age=300');
    event.waitUntil(cache.put(cacheKey, response.clone()));
  }
  return response;
}
```

## 6. Load Testing Strategies

### 6.1 Types of load tests

```
Load test types (run in this order):

1. Baseline test (smoke test):
   What: minimal load (1-5 users) to verify the system works at all.
   Goal: catch configuration errors, broken endpoints before ramping up.
   Duration: 1-2 minutes.

2. Load test (target load):
   What: expected peak production load.
   Goal: verify the system meets performance targets (p99 latency, error rate).
   Duration: 30-60 minutes (enough for JVM warmup, cache warming, stabilisation).

3. Stress test (above target):
   What: 2-5× expected peak load.
   Goal: find where the system breaks; what the failure mode is.
   Does it: degrade gracefully (slower responses) or fail catastrophically (5xx storm)?
   Duration: 15-30 minutes at each load level.

4. Spike test:
   What: sudden 10× traffic increase for 1-2 minutes, then return to normal.
   Goal: does auto-scaling respond fast enough? Does the system recover?
   Simulates: marketing campaign launch, viral event, flash sale.

5. Soak test (endurance test):
   What: sustained load at 70% of target for 24-72 hours.
   Goal: detect memory leaks, connection pool exhaustion, slow GC degradation.
   Pattern: issues that only appear after hours — memory leak causes OOM at hour 8.
```

### 6.3 Analysing load test results

```
Key metrics to review after a load test:

Latency distribution:
  p50 (median): typical user experience
  p95: most users' worst-case experience
  p99: the slowest 1% of requests (often the one causing timeouts / retries)
  p99.9 (max): outliers; may indicate GC pauses, cold starts, or occasional DB query

  The shape of the latency distribution matters:
  Bimodal (two peaks): one peak = cached responses; one = cache misses
  Long tail (wide spread): GC pauses or occasional expensive queries
  Linear increase under load: resource contention; getting closer to saturation

Error analysis:
  Error rate by status code: distinguish 4xx (client/config) from 5xx (server)
  Error rate over time: errors spike at specific load → saturation point found
  Error rate at specific endpoints: which endpoint breaks first?

Throughput vs latency:
  Plot both on the same graph over time as VUs increase.
  The point where latency starts increasing faster than linearly
  = the "knee of the curve" = the system's maximum efficient throughput.
  Beyond this point: adding more load makes things worse for everyone.

Resource correlation:
  During load test, simultaneously monitor:
  CPU, memory, I/O utilisation → which resource hits saturation first?
  DB connection pool → is the pool exhausted?
  Kafka consumer lag → is async processing falling behind?
  External API rate limits → are we hitting third-party limits?
```

## 7. Capacity Planning

### 7.1 The capacity planning mindset

```
Capacity planning answers: "How much infrastructure do we need, and when
do we need to add more?"

The goal is not to prevent scaling — it is to:
  1. Know before it becomes an emergency when you need to scale
  2. Scale to the right size (not 10× overprovisioned "just in case")
  3. Have a scaling plan validated by load tests before traffic arrives

Common failure modes:
  No capacity planning: system crashes on traffic spike; post-mortem says "we need more servers"
  Over-provisioning: 10× headroom costs 10× the money; most of it never used
  Under-provisioning: "we'll scale when we need to" → 3 AM emergency capacity increase
  Wrong metric: planned for request rate but bottleneck was DB connections
```

### 7.2 Bottom-up capacity estimation

```
Step 1: Measure baseline resource consumption
  For each component: CPU%, memory (GB), I/O (MB/s), network (Mbps) per RPS.
  Example: API server at 1,000 RPS:
    CPU:    40% of a 4-core instance (= 1.6 CPU)
    Memory: 4 GB JVM heap (stable; not growing with load)
    Network: 10 MB/s inbound, 25 MB/s outbound
    DB connections: 50 active connections

Step 2: Determine the limiting resource per component
  At what RPS does each resource hit 80% utilisation?
  80% is the target ceiling — leave 20% headroom for spikes.
  CPU hits 80% at ~2,000 RPS (linear extrapolation from baseline).
  DB connections hit 80% of pool (100 connections) at ~1,600 RPS.
  → DB connections are the bottleneck; add PgBouncer or increase pool first.

Step 3: Model load under anticipated growth
  Current: 500 RPS average, 1,500 RPS peak (3× average)
  Growth: 20% month-over-month
  Month 3 peak: 1,500 × 1.2^3 = 2,592 RPS
  Month 6 peak: 1,500 × 1.2^6 = 4,478 RPS

  At Month 3 peak (2,592 RPS):
    DB connections: 2,592 / 1,600 × 100 = 162 connections needed
    → Add connection pooling or another read replica before Month 3.

Step 4: Verify with load test
  Run a load test at Month 3 expected peak (2,592 RPS).
  Confirm system handles it within SLAs.
  Identify the next bottleneck that emerges.
  Plan capacity increase for that bottleneck.

Step 5: Define scaling triggers and thresholds
  Scaling trigger: 70% CPU utilisation sustained for 5 minutes → add instances
  Alert: 80% of DB connection pool in use → investigate or expand pool
  Alert: Kafka consumer lag > 60 seconds → add consumer instances
```

### 7.3 Auto-scaling design

```
Kubernetes HPA (Horizontal Pod Autoscaler):
  Scales pods based on metrics (CPU, memory, or custom metrics like queue depth).
  Target: keep CPU utilisation at 60% (leaves 40% headroom for traffic spikes).

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-server-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-server
  minReplicas: 3         # minimum for high availability
  maxReplicas: 50        # maximum: cost and infra ceiling
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 60    # scale when CPU > 60%
    - type: External
      external:
        metric:
          name: kafka_consumer_lag  # custom metric from Prometheus
          selector:
            matchLabels:
              topic: order-events
        target:
          type: AverageValue
          averageValue: "1000"      # scale when lag > 1000 messages per replica
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60   # avoid flapping; wait 60s before scaling up again
      policies:
        - type: Pods
          value: 4           # add max 4 pods per scaling event
          periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300  # conservative scale-down (5 minutes)
```

### 7.4 Capacity planning for databases

```
Databases are harder to auto-scale than stateless services.
Planning horizons: 3 months, 6 months, 12 months.

Storage growth:
  Measure: current data growth rate (GB/day or GB/week)
  Project: current size + daily growth × days
  Threshold: provision more storage when 80% full
  Buffer: always have 30% free (VACUUM needs space; unexpected growth)

IOPS (I/O Operations Per Second):
  AWS EBS gp3: 3,000 IOPS baseline, scalable to 16,000
  Measure: current IOPS utilisation from CloudWatch/Prometheus
  If sustained > 70% of provisioned IOPS: increase provisioned IOPS or add read replicas

Connection limits:
  PostgreSQL default: 100 connections (memory constrained: each ~5MB of RAM)
  Planning: (connections used at peak) / (limit × 0.8) = headroom ratio
  When ratio > 0.8: add PgBouncer, increase max_connections, or add read replicas

Read replica trigger:
  Primary CPU > 60% sustained (reads consuming write capacity)
  Replication lag > 10 seconds on existing replica (replica under load)
  Query latency increasing despite good indexes (DB under read pressure)

Shard trigger:
  Write throughput approaches primary's maximum (typically 10-50K TPS)
  Single table > 500M rows and growing fast (despite partitioning)
  Storage > 10 TB on a single instance (operational overhead)
```

## 8. Performance Architecture Decision Tree

```
User reports slowness → Measure: which metric, which endpoint, what value

        ↓
P99 > target threshold?
  → YES: identify bottleneck layer
    ├── APM trace shows service A is slow?
    │   ├── CPU > 80%? → CPU-bound: profile (flame graph), optimise hot path, add instances
    │   ├── DB query > 500ms? → query/index problem: EXPLAIN ANALYZE, add index, cache
    │   ├── External API slow? → timeout, circuit breaker, cache, async
    │   └── Memory > 80%? → heap dump, fix leak, tune GC, increase heap
    │
    ├── Multiple services slow (cascading)? → network latency, service mesh, batching
    └── Only under load? → resource saturation: identify which resource, scale or optimise

Error rate > threshold?
  → YES: are errors 4xx or 5xx?
    ├── 4xx (client errors)? → client bug or API contract mismatch
    ├── 5xx (server errors)?
    │   ├── 502/504 (upstream timeout)? → upstream service latency, circuit breaker
    │   ├── 503 (unavailable)?  → overloaded; rate limiting insufficient; scale up
    │   └── 500 (server error)? → bug; check logs; structured error with requestId

Throughput not meeting target?
  → YES: check resource utilisation at target RPS
    ├── CPU saturated? → horizontal scale or optimise
    ├── DB connections exhausted? → connection pooling, read replicas, optimise queries
    ├── Network saturated? → compress payloads, CDN for static, reduce round trips
    └── Queue lag growing? → add consumers, optimise consumer, partition Kafka topic
```

## 9. Interview Questions

**Q: You are asked to improve the performance of a slow API endpoint. Walk through your approach.**

A: I start by getting precise measurements rather than assumptions. The first step is to define "slow" in numbers: is the p50 slow (everyone experiences it), p99 only (an occasional outlier), or the error rate is rising? These have different root causes. Next I use distributed tracing — if we have it — to see where in the service call graph the time is spent. A trace showing 1,800ms in a single database query is immediately actionable; a trace showing 200ms spread across ten service calls is a different problem requiring batching or parallelism. If the trace points to a specific service, I use EXPLAIN ANALYZE on the slow query. Most single-query performance problems have one of three causes: a missing or poorly chosen index, a function applied to an indexed column defeating the index, or an N+1 query pattern where one visible query triggers many invisible ones. I add the appropriate index or rewrite the query, then re-measure with the same load pattern. If the trace shows a cache miss driving DB load, I look at whether the data is cacheable, what the right TTL is, and whether a stampede risk exists. Throughout this process I change one thing at a time — changing multiple things simultaneously means you cannot attribute the improvement and you may cancel out optimisations. The final step is verifying that the bottleneck actually moved: after fixing the original problem, I re-profile because the second bottleneck is often not where I expected.

---

**Q: How would you design a multi-level caching strategy for a high-traffic product catalog?**

A: A product catalog is read-heavy and tolerant of brief staleness, making it a good candidate for aggressive multi-level caching. The design has three levels. At the CDN edge (L3), product list pages and individual product pages — when they are publicly accessible — are cached with a short max-age (60 seconds) and a longer s-maxage (5 minutes), so the browser refreshes frequently but the CDN absorbs the bulk of traffic. Using stale-while-revalidate allows the CDN to serve a cached response instantly while refreshing in the background, so users never experience cache-miss latency for popular products. At the distributed cache layer (L2), product data by product ID is stored in Redis with a 15-minute TTL using cache-aside. On a cache miss, the Redis entry is populated from the database and subsequent requests for the same product go to Redis. At the in-process layer (L1), a Caffeine cache with a 30-second TTL and a maximum of 50,000 entries holds the hottest products in JVM memory for sub-microsecond access with zero network latency. For invalidation: when a product is updated, the application deletes both the L1 and L2 entries immediately after the database write commits; the CDN entry expires naturally within its s-maxage window. For flash sales where product price changes simultaneously across thousands of products, I would use a Kafka event to trigger a cache invalidation fan-out. The stampede risk exists when a popular product's cache expires: I would use a Redis lock pattern to ensure only one thread populates the cache on a miss, with all other threads waiting briefly.

---

**Q: What are the trade-offs of synchronous vs asynchronous processing, and how do you decide which to use?**

A: The decision hinges on one question: does the caller need the result of this operation to complete their immediate task? If yes, synchronous is required — a payment that must succeed before the order is confirmed cannot be deferred. If no, asynchronous is almost always better for operations that take significant time, can fail transiently, or have side effects that must be retried. Asynchronous processing has three concrete advantages: it keeps user-facing response times predictable regardless of downstream slowness (publishing to Kafka takes microseconds; the upstream service's processing time is irrelevant to the response latency); it naturally builds in retry capability since message queues re-deliver on processing failure; and it allows the processing rate to be smoothed over time rather than spiking with user activity patterns. The trade-offs are real though. Async adds eventual consistency: the side effect (sending email, updating analytics) happens after the response, creating a window where the state is incomplete. Error visibility degrades — a synchronous failure produces an immediate error response; an asynchronous failure may be silent until a monitoring alert fires or a user notices a missing confirmation email. Debugging distributed async flows is more complex than a synchronous call stack. For my decision: any operation that takes more than 100ms at p99 and is not required for the immediate response should be async. This includes: sending emails or push notifications, updating audit logs, processing images or files, propagating changes to reporting systems, and any write to a non-critical downstream service.

---

**Q: How do you approach capacity planning for a service expecting 10× growth over the next year?**

A: Capacity planning for 10× growth over a year means planning for approximately 20% month-over-month growth in peak load. I approach this in three stages. First, I establish a baseline: I instrument the current system comprehensively and measure resource consumption per RPS — CPU, memory, DB connections, network — to find the current limiting resource. Under current load, one of these will saturate before the others at some higher RPS; that is the first bottleneck I need to address. Second, I extrapolate the growth curve and identify when each resource hits 80% utilisation. With 20% MoM growth, month six has 2.5× current load and month twelve has 7× current load. I project each resource against those load levels and build a roadmap: "month 2: add read replicas for DB read load; month 4: add connection pooling; month 7: shard the orders table." Third, I validate with load tests before each scaling event rather than during a traffic crisis. Running a load test to 2× current peak before the month-six growth arrives confirms the capacity plan and surfaces unexpected bottlenecks. One important caution: extrapolating growth linearly is usually wrong. Traffic is bursty; marketing campaigns spike load 10× above the daily average; the 10× annual growth may arrive in two spikes of 5× rather than steady 20% monthly growth. My capacity plan includes both the steady-state growth model and a spike scenario test at each phase.

---

**Q: How do you distinguish between a bottleneck that needs more hardware and one that needs better code?**

A: Hardware is the right answer only after you have verified that the code is doing the minimum necessary work. I look for three signals. First, resource utilisation: if CPU is at 90% and the profiler shows the time is spent in business logic or I/O rather than framework overhead, that suggests the work is genuine and hardware helps. But if the profiler shows 40% of CPU in JSON serialisation of an unnecessarily large response, that is code waste — more hardware just wastes more. Second, efficiency ratios: I compare the current system's throughput per unit of resource against what we would expect from a well-implemented version. If our API server handles 500 RPS at 80% CPU but comparable systems handle 5,000 RPS at similar CPU, there is a 10× efficiency gap that hardware cannot close. Third, the shape of the scaling curve: if adding a second server doubles throughput, the problem is genuinely compute-bound and the work is necessary. If adding a second server improves throughput by only 30%, there is a shared contention point (the database, a lock, a single-threaded component) that hardware cannot fix. The most reliable test: profile under load to find what the CPU is actually doing. If it is computing necessary results, scale. If it is re-serialising the same objects, re-querying cached data, or spending 60% of time in GC collecting temporary objects, that is a code problem that will continue to cost money at 2×, 4×, or 8× scale.

## 10. Key Takeaways

- **Measure before optimising**: "the database is slow" is a hypothesis, not a diagnosis — EXPLAIN ANALYZE, distributed traces, and flame graphs convert hypotheses into evidence
- **The optimisation hierarchy**: eliminate work (caching, pre-computation) > improve algorithm complexity > improve data structure > add concurrency > add hardware — stop at the first level that meets the target
- **L1/L2/L3 cache hierarchy**: in-process (Caffeine) for sub-microsecond access to the hottest 50K entries; Redis for shared distributed cache; CDN edge for publicly cacheable responses; each layer has different TTL, scope, and invalidation strategy
- **Cache stampede prevention**: mutex/request coalescing prevents thundering herd on miss; probabilistic early expiration prevents synchronised expiry; stale-while-revalidate eliminates miss latency for most users
- **Index design is not optional**: every WHERE clause and JOIN condition should be covered by an index; covering indexes (INCLUDE columns) eliminate table access entirely; EXPLAIN ANALYZE before and after every query change
- **Async everything that the user does not immediately need**: email sending, analytics, downstream propagation, image processing — keep user-facing p99 predictable regardless of downstream slowness
- **CDN cache key design determines hit rate**: immutable assets (content-hashed filenames) get infinite TTL; public API responses get s-maxage/stale-while-revalidate; user-specific responses get Cache-Control: private or no-store
- **Load test before the traffic arrives, not after**: run baseline, load, stress, spike, and soak tests; the soak test (24-72 hours) is the only way to find memory leaks and slow connection exhaustion
- **Capacity planning is about knowing before it becomes an emergency**: measure resource consumption per RPS; extrapolate the growth curve; define scaling triggers; validate with load tests at each planned scaling event
- **The knee of the curve is your true capacity limit**: the point where latency increases faster than linearly as load increases is the system's maximum efficient throughput — operating consistently beyond this degrades everyone
