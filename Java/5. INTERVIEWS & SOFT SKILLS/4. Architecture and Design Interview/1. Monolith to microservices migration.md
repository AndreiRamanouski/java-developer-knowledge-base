# Monolith to Microservices Migration

## Overview

Migrating from a monolith to microservices is one of the most consequential architectural decisions an engineering organisation can make. Done well, it unlocks independent deployment, team autonomy, and targeted scalability. Done poorly, it produces a distributed monolith — all the operational complexity of microservices with none of the benefits — or causes extended business disruption. This guide covers the full decision framework, migration patterns, and the specific challenges (database decomposition, distributed transactions, backward compatibility) that distinguish a successful migration from a failed one.

**The honest starting position:**

```
Microservices are not inherently better than monoliths.
They are the right answer for specific organisational and technical problems.

Problems microservices solve well:
  Independent deployment:  teams block each other deploying a monolith together
  Targeted scaling:        one component needs 10× more resources than others
  Technology diversity:    different components benefit from different runtimes
  Team autonomy:           multiple teams own different business capabilities
  Fault isolation:         one component's failure should not take down everything

Problems microservices do NOT solve:
  Poorly understood domains: splitting a domain you do not understand compounds confusion
  Small teams:               < 10 engineers rarely benefit; operational cost too high
  Tightly coupled data:      shared data between services creates distributed transactions
  Lack of CI/CD maturity:    microservices require robust deployment infrastructure

The rule of thumb: start with a monolith, migrate when specific, measurable
problems demand it. "Microservices are modern" is not a measurable problem.
```

## 1. Identifying Service Boundaries

### 1.1 The cost of wrong service boundaries

```
Defining service boundaries incorrectly is the root cause of most failed
microservice migrations. Wrong boundaries produce:

  Chatty services:   service A calls service B 50 times per request
                     → network overhead; tight coupling disguised as separate services
  Distributed joins: service A needs data from B and C to serve one request
                     → the monolith's data model is now spread across network calls
  Shared databases:  multiple services read/write the same tables
                     → no isolation; shared deployment risk; distributed monolith

The test for correct boundaries: a service should be able to serve its primary
use cases without synchronous calls to other services for data it does not own.
If a service makes 5+ synchronous calls to other services per user request,
the boundary is likely in the wrong place.
```

### 1.2 Domain-Driven Design (DDD) for service boundaries

```
DDD provides the vocabulary and tools for finding natural service boundaries.

Bounded Context:
  A boundary within which a particular domain model applies consistently.
  The word "customer" means something different in billing (a paying entity with
  a contract), shipping (a delivery address), and support (a person with a problem).
  These are three different bounded contexts, each with a separate data model.

  Each bounded context is a strong candidate for a microservice.

Aggregate:
  A cluster of domain objects treated as a unit for data changes.
  An Order aggregate contains OrderLines, Addresses, and Payment references.
  Aggregates should not span services — if two services need to share an aggregate,
  the boundary is wrong.

Context Map:
  Documents the relationships between bounded contexts.
  Types of relationships:
    Shared Kernel:    two contexts share a common subset of the domain model
    Customer/Supplier: upstream context provides data; downstream consumes it
    Anti-Corruption Layer: downstream translates upstream's model to protect itself

Identifying bounded contexts in practice:
  1. Draw the major business capabilities (not technical layers) of your system.
  2. For each capability, identify: what data does it own? What workflows does it
     execute? What are its read and write operations?
  3. Find where data is shared: these are the hard boundary cases.
  4. Draw the context map: which contexts need to communicate, and how?
  5. The boundaries with the least inter-context data sharing are the best candidates
     for early extraction.
```

### 1.3 Identifying service boundaries — practical heuristics

```
Heuristic 1: Team structure (Conway's Law)
  "Organisations design systems that mirror their own communication structure."
  If your teams are structured by business capability (payments team, catalog team,
  orders team), structure your services the same way. A service boundary that crosses
  a team boundary creates coordination overhead.

Heuristic 2: Independent deployability
  Can this component be deployed independently without testing anything else?
  If yes: it is a good service boundary candidate.
  If no: it is too tightly coupled to extract yet.

Heuristic 3: Data ownership clarity
  Ask for each candidate service: "What database tables does this service own?"
  If the answer involves tables also owned by another service: wrong boundary.
  Each table should have exactly one service as its owner.

Heuristic 4: Change rate
  Components that change together should be deployed together (possibly same service).
  Components that change independently should be deployed independently (separate services).
  Frequent forced co-deployment is a signal that two components should be merged.

Heuristic 5: Scaling requirements
  Components with dramatically different scaling requirements belong in separate services.
  Product search (high read, cacheable) and order processing (high write, transactional)
  should not scale together.

Heuristic 6: Technology requirements
  A machine learning inference component that benefits from GPU acceleration
  should not be deployed with a CRUD service that does not need it.
```

### 1.4 Common boundary mistakes

```
Mistake 1: Splitting by technical layer (not business capability)
  WRONG: "front-end service", "API service", "database service"
  RIGHT: "payments service", "catalog service", "orders service"
  Why: technical layer splits follow the architecture, not the business.
  Every feature change touches all three layers = deployment coupling persists.

Mistake 2: Too fine-grained too early (nanoservices)
  "User profile service" + "user address service" + "user preferences service"
  → three services with a 1-to-1-to-1 relationship; always deployed together;
  any request needs all three
  → merge into "user service"

Mistake 3: Shared database with claimed service independence
  "Payments service" and "orders service" both write to the orders table.
  → Not independent; shared database is shared deployment risk.

Mistake 4: Synchronous dependency chains
  Request: API → Service A → Service B → Service C → Service D
  → Four synchronous hops; latency compounds; A-D failure cascade;
  this is a monolith connected by HTTP instead of function calls
  → Flatten the chain: API calls A and B directly; use events for C and D.
```

## 2. Strangler Fig Pattern

### 2.1 The pattern explained

```
Named after the strangler fig tree: a vine that grows around an existing tree,
gradually taking over until the original tree is hollowed out.

Applied to migration:
  1. New service functionality is built in the new microservice.
  2. A routing layer (proxy, API gateway, or facade) sits in front of both.
  3. Traffic for specific endpoints is incrementally redirected to the new service.
  4. The monolith continues to handle all other traffic.
  5. Gradually, the monolith shrinks as more functionality migrates.
  6. When the monolith serves no traffic: it is removed.

Key properties:
  The monolith remains the fallback at all times.
  Migration is incremental: redirect one endpoint at a time.
  Rollback is trivial: redirect traffic back to monolith.
  No big-bang cutover: the monolith never needs to be completely rewritten at once.

When to use: almost always. The alternative (big-bang rewrite) has a terrible
historical track record. Netscape, Basecamp, and many others have lost years
to big-bang rewrites. Incremental migration is safer and reversible.
```

### 2.2 Strangler fig implementation

```
Step 1: Add a routing layer
  Place an API gateway, reverse proxy (nginx), or facade in front of the monolith.
  Initially: all traffic routes to the monolith (no change).

  ┌──────────┐    ┌──────────────┐    ┌────────────┐
  │  Client  │───▶│ API Gateway  │───▶│  Monolith  │
  └──────────┘    └──────────────┘    └────────────┘

Step 2: Extract the first service
  Choose the least-coupled, least-critical service for the first extraction.
  Build it independently; deploy it; verify it behaves correctly in isolation.

  ┌──────────┐    ┌──────────────┐    ┌────────────┐
  │  Client  │───▶│ API Gateway  │───▶│  Monolith  │
  └──────────┘    └──────┬───────┘    └────────────┘
                         │            ┌────────────┐
                         └───────────▶│ Payment Svc│ (new)
                                      └────────────┘

Step 3: Route traffic incrementally
  Redirect 5% of traffic for the new service's endpoints to it.
  Monitor error rates, latency, data consistency.
  Increase to 25%, 50%, 100% as confidence grows.

  ┌──────────┐    ┌──────────────┐    ┌────────────┐
  │  Client  │───▶│ API Gateway  │───▶│  Monolith  │ (other traffic)
  └──────────┘    └──────┬───────┘    └────────────┘
                         │ 100% of    ┌────────────┐
                         └──/payments▶│ Payment Svc│
                                      └────────────┘

Step 4: Remove the feature from the monolith
  Once 100% of traffic for that capability routes to the new service:
  delete the code from the monolith. Do not leave dead code; it confuses future readers.

Step 5: Repeat for each capability
  Extract the next least-coupled service. Repeat until the monolith is gone.
```

### 2.3 Branch by abstraction

```
Branch by abstraction is the in-code equivalent of the strangler fig.
Used when you cannot put a routing proxy in front of the monolith
(the coupling is inside a library or a deeply embedded component).

Steps:
  1. Create an abstraction (interface) over the component to be replaced.
  2. Make the existing code implement that interface.
  3. Build the new implementation behind the same interface.
  4. Use a feature flag to switch between old and new implementations.
  5. Once the new implementation is verified: delete the old one; remove the flag.

Example:
  Old code: PaymentProcessor (concrete class)
  Step 1: Create interface IPaymentProcessor
  Step 2: Rename PaymentProcessor → LegacyPaymentProcessor implements IPaymentProcessor
  Step 3: Build StripePaymentProcessor implements IPaymentProcessor
  Step 4: Use feature flag to route new users to Stripe, existing users to legacy
  Step 5: After all users on Stripe: delete LegacyPaymentProcessor
```

### 2.4 Strangler fig with dual writes for data migration

```
Challenge: the new service needs its own data, but the monolith still owns the data.
During transition: both must stay in sync.

Dual-write pattern:
  Phase 1: Monolith writes to both its own DB and the new service's DB.
           New service reads from its own DB (may be slightly behind).
           New service does NOT write (reads only).

  Phase 2: Validate data consistency between both stores.
           Run comparison jobs to find discrepancies.
           Fix any sync bugs found.

  Phase 3: New service becomes write owner.
           Monolith switches to reading from new service's API for this data.
           (Monolith no longer writes to the shared data directly.)

  Phase 4: Remove legacy data from monolith DB.
           New service is the sole owner.

Risk: dual writes can diverge. Always have a reconciliation job that compares
both data stores and alerts on discrepancies above a threshold.
```

## 3. Database Decomposition

### 3.1 Why database decomposition is the hardest part

```
Most monoliths share a single database. This database is the coupling point that
microservices decomposition must address. You cannot have true service independence
if two services share a database: a schema migration by one service can break the other;
a transaction in one service locks rows the other needs.

"The database is the monolith's immune system — it defends the monolith
against decomposition."

Strategies for database decomposition (in order of increasing difficulty):
  1. Separate schemas within the same database server
  2. Separate database instances per service
  3. Polyglot persistence (different database engines per service)

Most teams should start with (1) and progress to (2) over time.
(3) is justified only when specific technical requirements demand it.
```

### 3.2 Schema-first decomposition

```
Before splitting services: split schemas.
All tables for service X move into schema X; all tables for service Y into schema Y.

Why this helps:
  It forces you to identify all the cross-schema queries before you have
  a network boundary to cross — in SQL they are visible; across services they
  are not until something breaks.
  It enforces ownership at the database level.
  Cross-schema foreign keys become explicit (and can be removed to test decoupling).

Example:
  Before: public.users, public.orders, public.payments (all in one schema)
  After:  users.users, orders.orders, orders.order_items, payments.payments

Transition:
  1. Create new schemas.
  2. Move tables (rename schema, not data copy).
  3. Grant appropriate permissions: payments schema grants read on users.user_id
     for validation — but this is temporary.
  4. Find all cross-schema queries: these represent coupling to resolve before
     splitting services.
  5. Resolve cross-schema queries either by duplicating data (read replica pattern)
     or by service-to-service API calls.
```

### 3.3 Resolving cross-schema data dependencies

```
Three patterns for handling data a service needs but does not own:

Pattern 1: API call to the owning service
  Payments service needs the user's billing address.
  Solution: payments service calls users service API synchronously.
  Pro: authoritative, always fresh.
  Con: adds network latency; payments service unavailable if users service is down.
  Best for: data needed for correctness that changes frequently.

Pattern 2: Replicated reference data (read replica via events)
  Payments service maintains a local copy of user billing information.
  When user updates billing address: users service publishes an event;
  payments service consumes it and updates its local copy.
  Pro: payments service can function when users service is down.
  Con: eventual consistency; payments may briefly have stale billing address.
  Best for: data needed frequently, changes slowly, small consistency window acceptable.

Pattern 3: Embed data at write time
  When an order is created: embed the user's current email and address
  into the order record itself (denormalise).
  Pro: order history is correct even if user later changes their address.
  Con: the embedded data is historical, not current — intentional for audit,
       wrong if you need current user state.
  Best for: audit records, immutable transaction history, order confirmations.

Decision matrix:
  Data changes frequently + must be current:     Pattern 1 (API call)
  Data changes infrequently + availability important: Pattern 2 (replicated)
  Data must reflect state at a point in time:    Pattern 3 (embedded at write)
```

### 3.4 Shared tables and the exclusion zone

```
The most dangerous migration pattern: two services sharing a single table.
Even with separate application code, shared tables mean:

  Schema coupling:   migration by Service A can break Service B's queries
  Transaction coupling: Service A's write holds a lock Service B is waiting on
  Implicit coupling: changes to one service's data requirements affect the table
                     that the other service also uses

The exclusion zone rule:
  Each database table has exactly ONE service that may write to it.
  Other services that need data from that table:
    Option A: call the owning service's API (synchronous)
    Option B: consume events published by the owning service (async)
  They may NOT query the table directly.

  Enforce with database permissions: only the owning service's DB user
  has WRITE permission on its tables. Other services' DB users have READ ONLY
  (transitional) or NO ACCESS (target state).

The shared-table problem in practice:
  If the orders and inventory services both write to the products table
  to update stock levels: this is a design problem, not a permission problem.
  Resolution: one service must own stock; the other calls or sends an event.
  Decide which service is the owner; give that service the write permission;
  the other service sends an "order placed, quantity N" event and trusts the
  inventory service to handle the stock update.
```

## 4. Handling Distributed Transactions

### 4.1 Why distributed transactions are difficult

```
In a monolith with one database: ACID transactions span all operations.
  begin transaction
    update inventory (decrement stock)
    create order record
    charge payment
    send confirmation email trigger
  commit
  → All succeed or all roll back. Guaranteed.

In microservices: there is no single transaction coordinator.
  Inventory Service: decrement stock ✓
  Order Service: create order ✓
  Payment Service: charge card → FAILS
  → Stock is decremented, order exists, payment failed. System is inconsistent.

Two approaches to handle this:
  1. Two-phase commit (2PC): distributed transaction protocol
  2. Saga pattern: compensating transactions for rollback
```

### 4.2 Two-phase commit (2PC)

```
Protocol:
  Phase 1 (Prepare): Coordinator asks all participants to prepare to commit.
    Each participant writes to a transaction log and locks resources.
    Each participant responds: "ready to commit" or "cannot commit."
  Phase 2 (Commit/Abort): If all responded "ready": coordinator tells all to commit.
    If any responded "cannot": coordinator tells all to abort and release locks.

Problems with 2PC:
  Blocking: if the coordinator crashes after Phase 1, all participants are locked
    forever waiting for the commit/abort signal. Entire system stalls.
  Latency: two network round trips required; both must complete before any
    participant can release locks.
  Scalability: all participants must be available simultaneously; coordinator is a SPOF.

When to use 2PC:
  Within a single database system (PostgreSQL XA transactions).
  Within a single cloud provider's managed services that support it.
  NOT recommended across independently operated microservices.

Recommendation: avoid 2PC for microservice transactions. Use Saga instead.
```

### 4.3 Saga pattern

```
A saga is a sequence of local transactions, each publishing an event or message
that triggers the next step. If any step fails: compensating transactions undo
the preceding steps.

Two coordination styles:

Choreography (event-driven):
  Each service listens for events and reacts.
  No central coordinator.

  Order Service: OrderCreated →
  Inventory Service: (consumes OrderCreated) StockReserved →
  Payment Service: (consumes StockReserved) PaymentProcessed →
  Notification Service: (consumes PaymentProcessed) ConfirmationSent

  Failure: Payment fails → PaymentFailed →
  Inventory Service: (consumes PaymentFailed) StockReleased
  Order Service: (consumes StockReleased) OrderCancelled

  Pros: loose coupling; no SPOF; simple services.
  Cons: hard to visualise the full transaction flow; hard to test end-to-end;
        services must know which events require compensation.

Orchestration (central coordinator):
  A saga orchestrator sends commands to services and handles responses.
  Orchestrator controls the sequence; services do not know about each other.

  CreateOrderSaga orchestrator:
    Step 1: Command → InventoryService.ReserveStock
      Success: → Step 2
      Failure: → End (nothing to compensate yet)
    Step 2: Command → PaymentService.ChargeCard
      Success: → Step 3
      Failure: → Compensate Step 1: InventoryService.ReleaseStock
    Step 3: Command → OrderService.ConfirmOrder
      Success: → Step 4
      Failure: → Compensate Step 2: PaymentService.RefundCard
                  Compensate Step 1: InventoryService.ReleaseStock
    Step 4: Command → NotificationService.SendConfirmation

  Pros: explicit flow; easy to test; one place to understand the full transaction.
  Cons: orchestrator knows about all services; potential SPOF; more complex to build.

When to use:
  Simple sagas (< 4 steps, linear): choreography
  Complex sagas (> 4 steps, conditional branches, parallel steps): orchestration
```

### 4.4 Saga implementation in Spring Boot

```java
// Orchestration-based saga using Axon Framework (or Eventuate Tram)
@Saga
@Slf4j
public class CreateOrderSaga {

    @Autowired
    private transient CommandGateway commandGateway;

    private String orderId;
    private String customerId;
    private BigDecimal amount;
    private String inventoryReservationId;

    @StartSaga
    @SagaEventHandler(associationProperty = "orderId")
    public void on(OrderCreatedEvent event) {
        this.orderId = event.getOrderId();
        this.customerId = event.getCustomerId();
        this.amount = event.getAmount();

        // Step 1: Reserve inventory
        commandGateway.send(new ReserveInventoryCommand(
            event.getOrderId(), event.getItems()
        ));
    }

    @SagaEventHandler(associationProperty = "orderId")
    public void on(InventoryReservedEvent event) {
        this.inventoryReservationId = event.getReservationId();

        // Step 2: Charge payment
        commandGateway.send(new ChargePaymentCommand(
            event.getOrderId(), this.customerId, this.amount
        ));
    }

    @SagaEventHandler(associationProperty = "orderId")
    public void on(PaymentFailedEvent event) {
        // Compensate: release inventory reservation
        commandGateway.send(new ReleaseInventoryCommand(
            this.inventoryReservationId
        ));
        commandGateway.send(new CancelOrderCommand(this.orderId, "Payment failed"));
        // Saga ends here
        SagaLifecycle.end();
    }

    @SagaEventHandler(associationProperty = "orderId")
    public void on(PaymentSucceededEvent event) {
        // Step 3: Confirm order
        commandGateway.send(new ConfirmOrderCommand(this.orderId));
    }

    @EndSaga
    @SagaEventHandler(associationProperty = "orderId")
    public void on(OrderConfirmedEvent event) {
        // Saga complete — no cleanup needed
        log.info("Order {} saga completed successfully", this.orderId);
    }
}
```

### 4.5 Saga failure handling — idempotency

```
Critical requirement: every step in a saga must be idempotent.
If a network timeout causes the orchestrator to retry a command:
the same command may be processed twice. The service must handle duplicates.

Idempotency key pattern:
  Include a unique idempotency key in every command.
  Before processing: check if this key has been processed before.
  If yes: return the previous result without re-executing.
  If no: execute and store the result with the key.

Example: ChargePaymentCommand includes idempotency_key = orderId + attempt_number.
If the payment network times out and we retry: same idempotency key prevents
double charge. The payment service returns the cached "success" response.

Implementation:
CREATE TABLE idempotency_keys (
    key     VARCHAR(255) PRIMARY KEY,
    result  JSONB        NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW()
);

On each command:
  SELECT result FROM idempotency_keys WHERE key = ?
  If found: return result (idempotent response)
  If not found: execute command, insert key+result, return result
```

## 5. Maintaining Backward Compatibility

### 5.1 Why backward compatibility is critical during migration

```
During a migration, multiple versions of the system coexist:
  - The monolith (old)
  - New microservices (new)
  - Clients consuming the API (may lag behind deployments)
  - Consumers of events published by migrated services

Breaking backward compatibility during migration:
  - Causes client failures before they have migrated
  - Forces coordinated releases (destroys independent deployability)
  - Creates a coordination tax that grows with the number of consumers

Two principles:
  Postel's Law: be conservative in what you send; liberal in what you accept.
  Expand-then-contract: add new fields/endpoints before removing old ones.
```

### 5.2 API versioning strategies

```
Strategy 1: URI versioning (most common)
  /api/v1/orders     → old contract
  /api/v2/orders     → new contract
  Both served simultaneously during migration.
  Clients migrate to v2 at their own pace.
  v1 deprecated with notice period (90 days minimum).

Strategy 2: Header versioning
  Accept: application/vnd.myapp.orders.v2+json
  Cleaner URIs; harder to test in browser; less visible to consumers.

Strategy 3: Query parameter
  GET /api/orders?version=2
  Simple; sometimes pollutes query parameter space.

Strategy 4: Content negotiation
  Server returns the version it supports based on client's Accept header.
  Most REST-correct; least common in practice.

Recommendation: URI versioning for external APIs (most visible, most compatible);
header versioning for internal service-to-service APIs.
```

### 5.3 Consumer-driven contract testing

```
Consumer-Driven Contract Testing (CDCT): before making a breaking API change,
verify that all known consumers can handle the new response format.

Tool: Pact (pact.io) — the industry standard for CDCT

How it works:
  1. Consumer (client) writes a "pact": a specification of what API response
     it expects from the provider.
  2. Pact stores the contract in a Pact Broker.
  3. Provider CI/CD pipeline verifies it can satisfy all registered pacts.
  4. If a provider change would break a consumer's pact: the build fails.

Integration in CI/CD:
  Consumer pipeline: run consumer tests → publish pact to broker
  Provider pipeline: run provider verification → check can-i-deploy?
    If all consumer pacts pass: deploy is safe.
    If any pact fails: deploy blocked until consumer is updated or pact revised.

Benefits:
  Broken API contracts are caught before deployment, not in production.
  Removes the need for a shared integration environment where everything
  must be deployed simultaneously to test together.
  Documents the real contract between services, not just the intended one.
```

```java
// Consumer-side Pact test (JUnit 5)
@ExtendWith(PactConsumerTestExt.class)
@PactTestFor(providerName = "orders-service")
public class OrdersServiceContractTest {

    @Pact(consumer = "payments-service")
    public RequestResponsePact createOrderLookupPact(PactDslWithProvider builder) {
        return builder
            .given("order 123 exists and is pending payment")
            .uponReceiving("a request for order 123 status")

                .headers(Map.of("Content-Type", "application/json"))

        // Use the mock server URL (Pact spins up a mock of the orders service)
        OrderStatus status = ordersClient.getOrderStatus("123");

}
```

### 5.4 Event schema evolution

```
Kafka and other event-driven systems require durable event schema compatibility.
Events may be consumed days or weeks after publication; consumers may be
temporarily offline during a producer schema change.

Schema Registry (Confluent Schema Registry or AWS Glue):
  Stores Avro, Protobuf, or JSON Schema definitions for each event topic.
  Enforces compatibility rules before publishing a new schema version.

Compatibility modes:
  BACKWARD:  new schema can read data written with old schema
             (consumers can upgrade before producers)
  FORWARD:   old schema can read data written with new schema
             (producers can upgrade before consumers)
  FULL:      both BACKWARD and FORWARD (safest; most restrictive)

Safe changes (always backward/forward compatible):
  Adding a new optional field with a default value ✓
  Adding a new optional field that can be null ✓
  Removing a field that had a default value (backward compatible only) ✓

Unsafe changes (break compatibility):
  Removing a required field ✗
  Changing a field's data type ✗
  Renaming a field ✗ (equivalent to remove + add)
  Adding a required field without a default ✗

Avro schema evolution example:
  Version 1: {"type":"record","name":"OrderPlaced","fields":[
    {"name":"orderId","type":"string"},
    {"name":"amount","type":"double"}
  ]}

  Version 2 (BACKWARD compatible — added optional field):
  {"type":"record","name":"OrderPlaced","fields":[
    {"name":"orderId","type":"string"},
    {"name":"amount","type":"double"},
    {"name":"currency","type":["null","string"],"default":null}  ← nullable with default
  ]}

  Old consumers reading v2 events: currency field ignored (unknown field).
  New consumers reading v1 events: currency is null (default applied).
  Both are correct — FULL compatibility achieved.
```

## 6. Migration Risks and Mitigation

### 6.1 Risk taxonomy for microservices migration

```
Technical risks:
  Data consistency gaps:     distributed transactions produce eventual consistency;
                             business rules that assumed ACID may silently violate constraints
  Latency regression:        monolith function call (1μs) → service HTTP call (5ms)
                             500× slower; multiplied by call depth
  Operational complexity:    each new service needs: CI/CD pipeline, health checks,
                             logging, monitoring, alerting, on-call rotation
  Testing complexity:        integration testing a distributed system is harder than
                             testing a monolith; contract tests mitigate but do not eliminate
  Network reliability:       the network is not reliable; every service call can fail,
                             time out, or return a partial response

Organisational risks:
  Skills gap:                teams may lack the operational expertise for distributed systems
  Coordination overhead:     service boundaries that do not match team boundaries create
                             excessive cross-team coordination
  Migration fatigue:         migrations can drag on for years; teams stop caring
  Over-engineering:          extracting services that were not actually creating the
                             business problem being solved
```

### 6.2 The distributed monolith — the most common failure mode

```
A distributed monolith has all the downsides of microservices and none of the benefits.

Symptoms:
  Services must be deployed together (no independent deployability).
  Failure in one service cascades to all others synchronously.
  Every feature change requires coordinating changes in 3+ services.
  Services share a database or share mutable state.

Causes:
  Wrong service boundaries (chatty services compensated with synchronous calls).
  Shared database between services.
  Synchronous dependency chains (A → B → C → D for every request).
  Logic that should be in one service split across multiple for superficial reasons.

Prevention:
  Define bounded contexts rigorously before splitting.
  Test that services can operate (degraded, not failed) when their dependencies are down.
  Track the "number of services touched per deploy" metric — it should decrease over time.
  Enforce the data ownership rule (one service owns each table).
```

### 6.3 Mitigation strategies for specific risks

```
Risk: Latency regression from synchronous calls
Mitigation:
  Map all synchronous call chains before migrating.
  Eliminate chains > 3 hops; use async events for non-critical paths.
  Add caching at the service boundary for frequently read, slowly changing data.
  Set latency budgets per service and alert when exceeded.

Risk: Data consistency gaps
Mitigation:
  Identify all workflows that currently use ACID transactions.
  For each: determine whether the business actually requires strong consistency,
  or whether eventual consistency is acceptable with a short convergence window.
  Implement sagas for multi-service workflows that require consistency.
  Add reconciliation jobs that detect and alert on inconsistencies.
  Do NOT migrate services that share a transaction boundary until
  the data ownership model is resolved.

Risk: Operational complexity
Mitigation:
  Do not extract a service until you have the infrastructure to run it:
  CI/CD pipeline, observability (logs + traces + metrics), runbook.
  Extract services in order of team readiness, not in order of complexity.
  Use service mesh (Istio, Linkerd) to handle cross-cutting concerns
  (mTLS, retries, circuit breakers) without duplicating them in each service.

Risk: Migration fatigue
Mitigation:
  Ship visible value at each phase. Every extraction should remove a deployment
  bottleneck the team has been complaining about.
  Set a completion criterion at the start: "migration is complete when
  the monolith is serving < 5% of requests."
  Track and celebrate milestones (first service extracted, monolith < 50% of traffic).

Risk: Cascading failures
Mitigation:
  Implement circuit breakers for all inter-service calls (Resilience4j).
  Define degraded-mode behaviour: what does the service return when its
  dependencies are unavailable? Often: cached data or a graceful error.
  Use bulkheads: separate thread pools per external dependency so one
  slow dependency does not exhaust all threads.
  Never let a downstream failure propagate synchronously if the downstream
  dependency is non-critical to the current operation.
```

### 6.4 Migration sequencing strategy

```
Extract services in this order (least risk to most risk):

Phase 1: Extract peripheral services (low coupling, low criticality)
  Good candidates: email notification, PDF generation, analytics event emission,
  scheduled report generation.
  These services have few inbound calls, no transaction coupling, and their
  failure does not break core user workflows.
  Goal: build operational confidence; establish CI/CD patterns for microservices.

Phase 2: Extract well-bounded domain services (medium coupling, medium criticality)
  Good candidates: user authentication, product catalog (read-heavy, no write coupling),
  search, recommendation engine.
  These are clearly bounded, have predictable data ownership, and moderate traffic.
  Goal: establish the data ownership model; prove the strangler fig pattern.

Phase 3: Extract core transactional services (high coupling, high criticality)
  Candidates: orders, payments, inventory.
  These require saga patterns for distributed transactions.
  Do not start this phase until Phase 2 services are stable and the team has
  operational confidence running microservices in production.
  Goal: eliminate the deployment coupling that motivated the migration.

What not to touch until last:
  Services with shared mutable state between multiple business capabilities.
  Legacy code with undocumented behaviour and no tests.
  Core persistence layer if it is tightly integrated with business logic.
```

## 7. Observability for Distributed Systems

```
Observability is not optional for microservices — it is a prerequisite.
A monolith can be debugged with a debugger or a single log file.
A microservice system requires distributed tracing, structured logging with
correlation IDs, and service-level dashboards.

Three pillars of observability:

Metrics (Prometheus + Grafana):
  Per-service: request rate, error rate, latency p50/p95/p99 (RED method)
  Per-service: resource utilisation, connection pool usage
  Per-saga: saga completion rate, compensation rate (indicates failure frequency)

Logs (structured JSON, centralised in Elasticsearch or Loki):
  Every log line includes: service name, trace_id, span_id, user_id, request_id.
  Correlation IDs allow reconstructing a single request's path through 10 services.
  Never log raw PII; log pseudonymised identifiers.

Distributed traces (Jaeger or Zipkin via OpenTelemetry):
  Trace spans across service boundaries.
  Every service call adds a child span to the trace tree.
  A latency problem in service A → B → C shows exactly which hop is slow.
  Essential for debugging latency regressions post-migration.

Service health:
  Each service exposes /health and /readiness endpoints.
  /health: is this instance alive?
  /readiness: is this instance able to serve traffic (DB connected, warmup complete)?
  Service mesh reads these to route traffic away from unhealthy instances.
```

## 8. Interview Questions

**Q: How do you decide where to draw service boundaries in a monolith?**

A: Service boundaries should follow bounded contexts from domain-driven design — the places where the meaning of a domain concept shifts. I start by identifying the major business capabilities of the system and mapping the data each capability owns. The places where data is cleanly owned by one capability become natural service boundaries. The places where data is shared become the hard problems to solve. I then apply Conway's Law as a practical sanity check: service boundaries that align with team structures require less coordination to maintain than those that cut across teams. The test for a correct boundary is whether a service can serve its primary use cases without synchronous calls to other services for data it does not own. If a service makes five or more synchronous calls to other services per user request, the boundary is probably wrong. Finally, I check change coupling: components that always change together should be deployed together. If two candidate services would need to be deployed as a pair for every feature, they probably belong in one service.

---

**Q: Explain the Strangler Fig pattern and when you would use it instead of a big-bang rewrite.**

A: The strangler fig pattern migrates incrementally by placing a routing layer in front of the monolith and redirecting traffic for specific endpoints to new services one at a time. The monolith continues to handle all other traffic and remains a safe fallback throughout. You choose the strangler fig over a big-bang rewrite in almost every case, because big-bang rewrites have a historically terrible track record — they take years, the new system never quite matches the behaviour of the old one, and the business is still running on the old system while the rewrite drags on. The strangler fig allows you to ship value at each extraction, maintain a functioning system at all times, and roll back a specific migration if it goes wrong by redirecting traffic back to the monolith. The only time a big-bang approach is justified is when the existing system is so broken that running them in parallel is technically impossible — which is rare and usually means the migration needs to start with making the system understandable, not rewriting it.

---

**Q: How do you handle a business transaction that spans multiple services without a distributed transaction?**

A: The Saga pattern replaces distributed transactions with a sequence of local transactions and compensating actions. There are two coordination styles. Choreography is event-driven: each service listens for events from other services and reacts — when the inventory service publishes "stock reserved," the payment service consumes that event and charges the card. If the payment fails, it publishes a "payment failed" event and the inventory service compensates by releasing the reservation. Orchestration uses a central saga coordinator that explicitly sends commands to each service and handles success/failure responses. Choreography is simpler for linear flows; orchestration is clearer for complex flows with branches and parallel steps. Both require every step to be idempotent — if the coordinator retries a failed command, the service must handle receiving the same command twice without double-executing. I implement idempotency using a stored idempotency key: before executing a command, check whether this key has already been processed; if yes, return the cached result. The fundamental trade-off compared to ACID transactions is that sagas are eventually consistent — during execution, the system is in an intermediate state that may violate invariants that a single transaction would have made atomic. This is usually acceptable for business transactions, where the compensation path (refund, release stock) is a normal business operation, not an error.

---

**Q: How do you ensure backward compatibility when evolving APIs and events during a migration?**

A: For APIs, I follow the expand-then-contract pattern: add new fields or endpoints before removing old ones, and version breaking changes explicitly using URI versioning. During migration, both the old and new API versions are served simultaneously; clients migrate at their own pace within a deprecation window. I enforce this with consumer-driven contract testing using Pact — every consumer registers a contract specifying exactly what API response it depends on, and the provider's CI/CD pipeline verifies it can satisfy all registered contracts before any deployment. This catches breaking changes before they reach production rather than after. For event schemas, I use a Schema Registry with Avro or Protobuf and enforce FULL compatibility mode — both new schema can read old events and old schema can read new events. This means only additive changes are allowed: new optional fields with defaults are safe; removing or renaming fields is not. The practical implication is that I design event schemas to be append-only by default, treating each field as permanent once published, and using versioned event types for breaking changes rather than modifying existing event schemas in place.

---

**Q: What is a distributed monolith and how do you avoid creating one?**

A: A distributed monolith has all the operational costs of microservices — separate deployments, network calls, operational overhead — without any of the benefits — independent deployability, team autonomy, fault isolation. You can recognise it by three symptoms: services that must be deployed together because they are runtime-coupled, services that share a database meaning a schema change in one breaks another, and synchronous dependency chains where every request traverses five services in sequence. The root cause is almost always incorrect service boundaries — services were split along technical rather than domain lines, or the data ownership model was never resolved, or the original coupling was just replaced with network calls rather than being structurally eliminated. To avoid it, I enforce three rules before splitting any service: define the data ownership clearly and enforce it with database permissions (one writer per table), verify that the candidate service can operate in a degraded mode when its dependencies are unavailable, and check that the service would not need to be deployed with other services for routine feature work. If any of these conditions cannot be met, the service is not ready to be extracted — the coupling must be resolved in the monolith first.

## 9. Key Takeaways

- **Microservices solve specific problems; confirm you have those problems first**: independent deployability blocked by team size, targeted scaling requirements, or fault isolation needs are valid motivators; "modern architecture" is not
- **Service boundaries follow bounded contexts, not technical layers**: each service owns a set of business capabilities with clear data ownership; splitting by "front-end/back-end/database" preserves all coupling, just distributed
- **Strangler fig over big-bang always**: route traffic incrementally through a proxy; the monolith is always a safe fallback; rollback is redirect, not redeployment
- **Database decomposition is the hardest and most important part**: shared tables between services = shared deployment risk; one service owns each table; other services call the API or consume events
- **Sagas replace distributed transactions with local transactions + compensating actions**: orchestration for complex flows, choreography for simple linear flows; every saga step must be idempotent
- **Expand-then-contract for API evolution**: add before removing; version breaking changes; consumer-driven contract tests catch regressions before production
- **Schema Registry with FULL compatibility enforces safe event evolution**: additive changes only; new optional fields with defaults; never remove or rename fields on a live topic
- **Distributed monolith is worse than the original monolith**: test for it by checking whether services can operate independently, share no database tables, and do not require coordinated deployment
- **Extract in order of operational risk**: peripheral services first (build confidence), well-bounded domains second (establish data ownership), core transactional services last (only after saga infrastructure is proven)
- **Observability is a prerequisite, not an afterthought**: correlation IDs across all logs, distributed tracing (OpenTelemetry/Jaeger), and per-service RED metrics before the first service is extracted to production
