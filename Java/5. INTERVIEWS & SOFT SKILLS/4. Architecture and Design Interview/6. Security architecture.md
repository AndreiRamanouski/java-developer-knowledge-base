# Security Architecture

## Overview

Security architecture is the discipline of building systems where the cost of a successful attack exceeds the value of what is protected, and where the blast radius of a breach is bounded by design rather than luck. Good security architecture is not a checklist appended after the system is built — it is a set of constraints and patterns woven into the design that make insecure states structurally difficult to reach.

**The attacker's economics mindset:**

```
Security is not about achieving perfect safety — it is about making attacks
more expensive than they are worth.

Defensive principles:
  Defence in depth:   multiple independent layers; one layer's failure is not fatal
  Least privilege:    every component holds only the permissions it needs
  Fail secure:        when a component fails, it defaults to a more restrictive state
  Zero trust:         do not assume that being inside the network means being trustworthy
  Assume breach:      design as if the attacker is already inside; limit blast radius
  Minimal attack surface: the less you expose, the less can be attacked

The layered model:
  Layer 1: Perimeter (network, firewall, WAF)         — do not rely on this alone
  Layer 2: Identity (authentication, authorisation)   — the most critical layer
  Layer 3: Application (input validation, output encoding)
  Layer 4: Data (encryption, access controls)
  Layer 5: Detection (logging, alerting, anomaly detection)

A breach at any layer should be detectable and bounded by the layers below it.
```

## 1. Authentication Flows

### 1.1 The authentication taxonomy

```
Authentication: proving who you are.
Authorisation:  determining what you are allowed to do.
These are separate concerns; conflating them produces broken access control.

Authentication factors:
  Something you know:  password, PIN, security question
  Something you have:  TOTP (authenticator app), hardware key (YubiKey), SMS OTP
  Something you are:   biometric (fingerprint, face)

Multi-factor authentication (MFA):
  Requiring two factors from different categories dramatically increases
  the difficulty of account takeover.
  Even if a password is stolen (something you know), the attacker needs
  the physical device (something you have) for the second factor.

Password storage (never store plaintext):
  WRONG: store password or MD5/SHA-1 hash (MD5 is broken; SHA-1 is fast → brute-forceable)
  RIGHT: bcrypt, Argon2id, or scrypt with per-user random salt
    Argon2id: recommended by OWASP; configurable memory, time, parallelism
    bcrypt: legacy but acceptable (work factor 10-12)
    PBKDF2: acceptable, widely supported (NIST-approved)
    Configuration: choose parameters so hash takes 100-300ms on your hardware
      (fast enough for users; slow enough to make brute-force impractical)
```

### 1.2 Session management

```java
// Session token requirements:
//   Minimum 128 bits of entropy (use SecureRandom, not Random)
//   Stored as HttpOnly, Secure, SameSite=Strict cookie
//   Rotated on privilege change (login, permission grant)
//   Invalidated on logout (server-side session table)
//   Short absolute TTL (24 hours) + shorter idle TTL (30 minutes)

@Service
public class SessionService {
    private static final int TOKEN_BYTES = 32; // 256 bits of entropy
    private final SessionRepository sessionRepo;
    private final SecureRandom secureRandom = new SecureRandom();

    public String createSession(long userId, HttpServletResponse response) {
        byte[] tokenBytes = new byte[TOKEN_BYTES];
        secureRandom.nextBytes(tokenBytes);
        String token = Base64.getUrlEncoder().withoutPadding().encodeToString(tokenBytes);

        Session session = Session.builder()
            .tokenHash(sha256(token))   // store hash, not plaintext
            .userId(userId)
            .createdAt(Instant.now())
            .lastAccessAt(Instant.now())
            .expiresAt(Instant.now().plus(Duration.ofHours(24)))
            .ipAddress(currentRequest.getRemoteAddr())
            .userAgent(currentRequest.getHeader("User-Agent"))
            .build();
        sessionRepo.save(session);

        ResponseCookie cookie = ResponseCookie.from("session", token)
            .httpOnly(true)              // inaccessible to JavaScript (XSS mitigation)
            .secure(true)                // HTTPS only
            .sameSite("Strict")          // no cross-site submission (CSRF mitigation)
            .maxAge(Duration.ofHours(24))
            .path("/")
            .build();
        response.addHeader(HttpHeaders.SET_COOKIE, cookie.toString());
        return token;
    }

    public Optional<Session> validateSession(String token) {
        String tokenHash = sha256(token);
        return sessionRepo.findByTokenHash(tokenHash)
            .filter(s -> s.getExpiresAt().isAfter(Instant.now()))
            .filter(s -> Duration.between(s.getLastAccessAt(), Instant.now())
                             .compareTo(Duration.ofMinutes(30)) < 0) // idle timeout
            .map(s -> {
                s.setLastAccessAt(Instant.now());
                sessionRepo.save(s);
                return s;
            });
    }

    public void invalidateSession(String token) {
        sessionRepo.deleteByTokenHash(sha256(token));
        // Also: invalidate all sessions for the user on password change
    }
}
```

### 1.3 OAuth2 and OpenID Connect

```
OAuth2: authorisation framework for delegated access.
OpenID Connect (OIDC): authentication layer on top of OAuth2.
  OAuth2 answers: "Can app X access resource Y on behalf of user Z?"
  OIDC answers:   "Who is the user?" (issues an ID token with identity claims)

OIDC ID Token (JWT):
{
  "iss": "https://auth.example.com",
  "sub": "user_12345",           // subject: stable user identifier
  "aud": "client_id_abc",        // audience: which app this token is for
  "exp": 1710000060,
  "iat": 1709996460,
  "nonce": "abc123xyz",          // prevents replay attacks
  "email": "user@example.com",
  "email_verified": true,
  "name": "Alice Smith",
  "auth_time": 1709996400,       // when authentication actually happened
  "amr": ["pwd", "mfa"]          // authentication method references (how they authenticated)
}

PKCE (Proof Key for Code Exchange) — required for all public clients:
  Prevents authorisation code interception attacks.
  1. Client generates code_verifier (random 32+ bytes) and
     code_challenge = BASE64URL(SHA256(code_verifier))
  2. Sends code_challenge in authorisation request.
  3. Server stores code_challenge.
  4. Client sends code + code_verifier in token request.
  5. Server verifies SHA256(code_verifier) == stored code_challenge.
  Result: the code alone (intercepted) is useless without code_verifier.

Token storage (browser):
  Access tokens: memory only (JavaScript variable; cleared on page reload)
    → Not in localStorage (XSS readable) or sessionStorage (same)
    → Not in cookies unless HttpOnly + SameSite=Strict
  Refresh tokens: HttpOnly Secure cookie (not accessible to JavaScript)
  ID tokens: memory (display purposes only; do not use for authorisation)
```

### 1.4 Service-to-service authentication

```
Services calling other services must authenticate, not just trust the internal network.
Zero trust principle: network location is not a credential.

Options:

1. mTLS (mutual TLS):
   Both client and server present certificates.
   Certificate authority issues short-lived certificates per service identity.
   Service A presents its cert when calling Service B.
   Service B verifies the cert is valid and issued by the trusted CA.
   Implemented by service mesh (Istio, Linkerd) transparently.
   Best for: Kubernetes-native environments; automatic certificate rotation.

2. JWT service tokens (workload identity):
   Service authenticates to an identity provider with its credentials.
   Receives a short-lived JWT (15 minutes) identifying the service.
   Uses the JWT as a Bearer token for downstream service calls.
   JWT contains: service name, allowed scopes, expiry.
   Best for: environments without a service mesh.

3. API keys (simplest but weakest):
   Pre-shared secret per service relationship.
   Stored in Vault or cloud secret manager.
   No automatic rotation (unless automated).
   Acceptable for: external third-party integrations.

Recommendation: mTLS via service mesh for internal service-to-service;
  short-lived JWTs for cross-cluster or cross-cloud calls.
```

## 2. Authorisation Models

### 2.1 RBAC — Role-Based Access Control

```
RBAC: permissions assigned to roles; roles assigned to users.
  User → Roles → Permissions

Advantages:
  Simple to understand and audit
  Roles map naturally to job functions
  Easy to add/remove a user's access by changing their roles

Disadvantages:
  Role explosion: fine-grained access requires many roles (N users × M resources)
  Static: roles are pre-defined; cannot easily express context-sensitive rules
  No temporal constraints: cannot express "only between 9-5"

Database schema:
CREATE TABLE roles (
    id   BIGINT PRIMARY KEY,
    name VARCHAR(50) UNIQUE NOT NULL  -- 'admin', 'editor', 'viewer', 'billing_admin'
);

CREATE TABLE permissions (
    id       BIGINT PRIMARY KEY,
    resource VARCHAR(100) NOT NULL,   -- 'order', 'product', 'report'
    action   VARCHAR(50)  NOT NULL,   -- 'read', 'write', 'delete', 'export'
    UNIQUE (resource, action)
);

CREATE TABLE role_permissions (
    role_id       BIGINT REFERENCES roles(id),
    permission_id BIGINT REFERENCES permissions(id),
    PRIMARY KEY (role_id, permission_id)
);

CREATE TABLE user_roles (
    user_id BIGINT,
    role_id BIGINT REFERENCES roles(id),
    PRIMARY KEY (user_id, role_id)
);

-- Check permission:
SELECT EXISTS (
    SELECT 1
    FROM user_roles ur
    JOIN role_permissions rp ON ur.role_id = rp.role_id
    JOIN permissions p ON rp.permission_id = p.id
    WHERE ur.user_id = :userId
      AND p.resource = :resource
      AND p.action   = :action
);
```

### 2.2 ABAC — Attribute-Based Access Control

```
ABAC: access decisions based on attributes of the subject, resource, action, and environment.
  Decision = f(subject attributes, resource attributes, action, environment context)

Example policy:
  "A user may approve a purchase order if:
    - their role is 'manager' (subject attribute)
    - the order belongs to their department (resource attribute)
    - the amount is ≤ their approval limit (subject attribute vs resource attribute)
    - it is during business hours in their timezone (environment attribute)"

Advantages:
  Expressive: can encode complex, context-sensitive rules
  Fine-grained: no role explosion; policies describe natural business rules
  Dynamic: environment attributes (time, location, risk score) enable adaptive access

Disadvantages:
  Complex: harder to reason about and audit than RBAC
  Performance: policy evaluation can be expensive without caching
  Debugging: "why was this denied?" is harder to answer

Standards and tools:
  XACML: XML-based policy standard (enterprise, complex)
  OPA (Open Policy Agent): modern, Go-based; Rego policy language; widely adopted
  Cedar: Amazon's policy language; used by AWS Verified Permissions

OPA example policy (Rego):
package orders

default allow = false

# Allow manager to approve orders in their department below their limit
allow {
    input.action == "approve"
    input.user.role == "manager"
    input.user.department == input.order.department
    input.order.amount <= input.user.approval_limit
    is_business_hours(input.environment.time, input.user.timezone)
}

# Helper rule
is_business_hours(time, timezone) {
    hour := time.hour_in_timezone(timezone)
    hour >= 9
    hour < 17
}
```

### 2.3 Spring Security implementation

```java
// Method-level security with Spring Security
@Configuration
@EnableMethodSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/v1/auth/**").permitAll()
                .requestMatchers(HttpMethod.GET, "/api/v1/products/**").hasAnyRole("USER", "ADMIN")
                .requestMatchers("/api/v1/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated()
            )
            .oauth2ResourceServer(oauth2 -> oauth2
                .jwt(jwt -> jwt.jwtAuthenticationConverter(jwtConverter()))
            )
            .sessionManagement(session ->
                session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .csrf(csrf -> csrf.disable()); // disabled for stateless JWT APIs
        return http.build();
    }
}

// Domain-specific permission check (beyond role-based)
@Service
public class OrderSecurityService {

    // Custom permission evaluator: can user modify this specific order?
    public boolean canModifyOrder(Authentication auth, Order order) {
        UserPrincipal user = (UserPrincipal) auth.getPrincipal();

        // Admin can modify any order
        if (user.hasRole("ADMIN")) return true;

        // User can modify their own orders in PLACED status only
        return user.getUserId().equals(order.getUserId())
            && order.getStatus() == OrderStatus.PLACED;
    }
}

// Method-level security using SpEL expression
@PatchMapping("/orders/{id}")
@PreAuthorize("@orderSecurityService.canModifyOrder(authentication, @orderRepository.findById(#id).orElse(null))")
public OrderDto updateOrder(@PathVariable Long id, @RequestBody UpdateOrderRequest req) {
    return orderService.update(id, req);
}
```

### 2.4 ReBAC — Relationship-Based Access Control

```
ReBAC: access determined by the relationship between subjects and objects.
  "User A can view Document D if A is a member of the group that owns D."

Popularised by Google Zanzibar (powers Google Drive, Docs, Calendar).
Open implementations: OpenFGA (Auth0), SpiceDB (Authzed), Warrant.

Model:
  Users have relationships to objects: owner, editor, viewer, member.
  Relationships are hierarchical: "editor" implies "viewer".
  Permission checks traverse the relationship graph.

Zanzibar tuple format:
  document:roadmap#owner@user:alice
  document:roadmap#editor@group:engineering#member
  group:engineering#member@user:bob

  Check: can bob view document:roadmap?
  → Is bob a viewer? No direct tuple.
  → Is bob an editor? No direct tuple.
  → Is bob a member of a group that has editor access?
  → group:engineering#member@user:bob exists.
  → document:roadmap#editor@group:engineering#member exists.
  → Editor implies viewer. → YES, bob can view.

Best for: Google Docs-style sharing where permissions are relationship-driven,
  not role-driven; multi-tenant SaaS where each tenant manages their own permissions.
```

## 3. Zero Trust Architecture

### 3.1 Zero trust principles

```
Traditional "castle and moat" security:
  Trust everything inside the network perimeter.
  Mistrust everything outside.
  Problem: once inside (VPN, compromised internal host), attacker moves freely.

Zero trust: "never trust, always verify"
  Network location is not a credential.
  Every request, from any network, must be authenticated and authorised.
  Every service must prove its identity before calling another service.
  Assume breach: limit blast radius with micro-segmentation.

The five zero trust tenets:
  1. Verify explicitly:     authenticate and authorise every request (not just at the edge)
  2. Least privilege:       minimal permissions; just-in-time access; time-bound
  3. Assume breach:         segment the network; limit lateral movement;
                            detect and respond; reduce blast radius
  4. Inspect all traffic:   encrypt even internal traffic (mTLS between services)
  5. Continuous validation: re-verify periodically; treat device health as a signal
```

### 3.2 Zero trust implementation layers

```
Identity layer:
  Strong authentication for all users (MFA required)
  Workload identity for services (SPIFFE/SPIRE, mTLS via Istio)
  Short-lived credentials (tokens expire in 15 minutes; certificates expire in 24 hours)
  Continuous re-evaluation: high-risk actions require step-up authentication

Device layer:
  Managed device posture check before access (MDM integration)
  Unmanaged devices: limited access only (read-only, no sensitive data)
  Certificate-based device authentication (not just password)

Network layer:
  Micro-segmentation: each service only reachable by permitted services
  No "internal network" trust: service A calling service B requires mTLS even if same cluster
  Network policies in Kubernetes:
    → By default: all pods can reach all pods (insecure)
    → With NetworkPolicy: each pod only receives traffic from explicitly allowed sources

  # Example: only allow payments-service to call inventory-service
  apiVersion: networking.k8s.io/v1
  kind: NetworkPolicy
  metadata:
    name: inventory-ingress
    namespace: production
  spec:
    podSelector:
      matchLabels:
        app: inventory-service
    ingress:
      - from:
        - podSelector:
            matchLabels:
              app: payments-service
        ports:
          - port: 8080

Application layer:
  Input validation and output encoding at every service boundary
  API gateway validates authentication and authorisation before forwarding
  Services re-validate authorisation on every request (not just the first)

Data layer:
  Encryption at rest for all sensitive data
  Encryption in transit (TLS 1.2+) for all service-to-service communication
  Field-level encryption for highly sensitive fields (SSN, payment card)
  Data access audit logging: who accessed what and when
```

### 3.3 Service mesh for zero trust

```
A service mesh (Istio, Linkerd) implements zero trust between microservices
without requiring each service to implement security itself.

What a service mesh provides:
  mTLS between all services:       every pod gets a certificate; all traffic encrypted
  Traffic authorisation policies:  service A may call service B on port 443 for /payments only
  Observability:                   all inter-service traffic logged with latency and status
  Certificate rotation:            automatic rotation every 24 hours (no manual cert management)

Istio AuthorizationPolicy example:
  # Only allow frontend to call product-service; only allow GET
  apiVersion: security.istio.io/v1beta1
  kind: AuthorizationPolicy
  metadata:
    name: product-service-policy
    namespace: production
  spec:
    selector:
      matchLabels:
        app: product-service
    action: ALLOW
    rules:
      - from:
        - source:
            principals: ["cluster.local/ns/production/sa/frontend-service"]
        to:
        - operation:
            methods: ["GET"]
            paths: ["/products/*"]

Without a service mesh: each service must implement mTLS, certificate management,
and traffic policy — duplicated across every service.
With a service mesh: consistent policy across all services; enforcement in the sidecar proxy.
```

## 4. Secret Management

### 4.1 What is a secret and what is not

```
Secrets: values that grant access to a resource.
  Database credentials, API keys, OAuth client secrets,
  JWT signing keys, TLS private keys, SSH keys,
  encryption keys, third-party service tokens.

Not secrets (but still sensitive):
  PII: protected but not a credential
  Configuration: environment-specific but not security-sensitive
  Feature flags: not secret

Where secrets must NOT live:
  ✗ Source code (committed to Git — permanent in history even if deleted)
  ✗ Environment variables in Dockerfiles or docker-compose.yml
  ✗ Application logs (logged accidentally)
  ✗ Error messages returned to clients
  ✗ Configuration files checked into source control

The Git history problem:
  A secret committed to Git is permanently compromised, even if deleted.
  The commit remains in history; anyone who ever cloned the repo may have it.
  Required action: rotate the credential immediately; treat it as breached.
  Prevention: pre-commit hooks (git-secrets, detect-secrets, truffleHog)
```

### 4.2 HashiCorp Vault

```
Vault: the industry-standard secret management platform.
  Central store for secrets: no secrets in code, config, or environment.
  Dynamic secrets: Vault generates short-lived credentials on demand.
  Audit log: every secret access logged with identity and timestamp.
  Lease management: secrets expire automatically; Vault revokes on expiry.

Secret types:
  Static secrets:  manually stored (API keys, third-party tokens)
  Dynamic secrets: Vault generates on-demand (database credentials, AWS IAM creds)

Dynamic database credentials (most powerful feature):
  Instead of: one password for the database shared across all instances (leaked = catastrophic)
  Vault does:  generates a unique username/password for each application instance,
               valid for 1 hour, automatically revoked on expiry.
  Breach of one instance: credentials expire in 1 hour; rotation is automatic.
  Vault creates a PostgreSQL role on demand:
    CREATE ROLE v-app-abc123 WITH LOGIN PASSWORD '...' VALID UNTIL '2024-03-15 11:00:00';
    GRANT SELECT, INSERT, UPDATE ON orders TO v-app-abc123;
  On expiry or revocation: DROP ROLE v-app-abc123.

Application integration (Spring Boot + Vault):
```

```java
// Spring Cloud Vault: reads secrets from Vault at startup
// application.yml
spring:
  cloud:
    vault:
      uri: https://vault.example.com:8200
      authentication: kubernetes      # uses Kubernetes service account token
      kubernetes:

#   database.url: jdbc:postgresql://db.internal:5432/mydb

```

### 4.3 Secret rotation strategy

```
Rotation reduces the window of opportunity if a secret is compromised.
Rotation without downtime requires the application to accept both old and new credentials
during the rotation window.

Zero-downtime rotation steps (database credentials):
  1. Add new credentials alongside old (CREATE USER new_user; GRANT same permissions)
  2. Update Vault/secret manager with new credentials
  3. Application instances gradually pick up new credentials via lease renewal
  4. Wait for all instances to have migrated (verify via audit log)
  5. Revoke old credentials (DROP USER old_user)

Rotation frequencies (NIST guidelines):
  Database passwords:    60-90 days (or on-demand via Vault dynamic secrets)
  API keys (external):   90-180 days
  JWT signing keys:      30-90 days (use JWK rotation with kid field)
  TLS certificates:      1 year (or 90 days with Let's Encrypt/auto-rotation)
  Encryption keys (KMS): annually (with key wrapping for old data)

Automated rotation in AWS:
  AWS Secrets Manager: built-in rotation via Lambda functions
  Rotation Lambda: creates new credential, tests it, updates the secret, deprecates old
  Application: fetches from Secrets Manager on each connection (cached, refreshed)
```

## 5. Encryption

### 5.1 Encryption in transit

```
Requirement: all data transmitted over any network must be encrypted.
  External traffic:  TLS 1.2 minimum; TLS 1.3 preferred
  Internal traffic:  mTLS between services (zero trust requirement)
  At rest:           see Section 5.2

TLS configuration (Java/Spring):
  TLS 1.3 features: 0-RTT (zero round-trip resumption), fewer handshake round trips,
                    no legacy cipher suites, forward secrecy by default.

  // application.yml: enforce TLS 1.2+ and strong cipher suites
  server:
    ssl:
      enabled: true
      protocol: TLSv1.3             // prefer 1.3
      enabled-protocols: TLSv1.2,TLSv1.3   // accept 1.2 for compatibility
      ciphers:
        - TLS_AES_256_GCM_SHA384    // TLS 1.3 (selected automatically by JVM)
        - TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384   // TLS 1.2
        - TLS_ECDHE_ECDSA_WITH_AES_256_GCM_SHA384

HTTP security headers:
  HSTS (HTTP Strict Transport Security):
    Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
    → Forces HTTPS for 1 year; protects against protocol downgrade attacks.
    → preload: submit domain to browser preload list (HTTPS even on first visit).

  CSP (Content Security Policy):
    Content-Security-Policy: default-src 'self'; script-src 'self' 'nonce-{nonce}'
    → Prevents XSS by whitelisting allowed sources for scripts, styles, etc.

  CORS: configure to allow only known origins.
    Do NOT set Access-Control-Allow-Origin: * for authenticated APIs.
```

### 5.2 Encryption at rest

```
What to encrypt:
  Database: column-level encryption for sensitive fields (PII, payment data)
            database-level encryption for breach protection
  Object storage: S3 server-side encryption (SSE-KMS, SSE-S3)
  Block storage: EBS encryption using KMS
  Application secrets: stored encrypted in Vault or cloud secret manager
  Backups: encrypted before transmission; encrypted in storage

Column-level encryption for PII:
  Use case: store user PII encrypted; only decrypt when needed.
  Key hierarchy: data encryption key (DEK) wraps the field value;
                 key encryption key (KEK) wraps the DEK;
                 KEK managed by HSM/KMS (never leaves HSM boundary).

  // Java: AES-256-GCM (authenticated encryption — detects tampering)
  public class FieldEncryption {
      private static final String ALGORITHM = "AES/GCM/NoPadding";
      private static final int IV_LENGTH   = 12;  // 96 bits
      private static final int TAG_LENGTH  = 128; // bits

      // Encrypt: generates a random IV; returns IV + ciphertext (base64)
      public String encrypt(String plaintext, SecretKey key) throws Exception {
          byte[] iv = new byte[IV_LENGTH];
          new SecureRandom().nextBytes(iv);

          Cipher cipher = Cipher.getInstance(ALGORITHM);
          cipher.init(Cipher.ENCRYPT_MODE, key,
              new GCMParameterSpec(TAG_LENGTH, iv));
          byte[] ciphertext = cipher.doFinal(plaintext.getBytes(UTF_8));

          // Prepend IV to ciphertext for storage
          byte[] result = new byte[IV_LENGTH + ciphertext.length];
          System.arraycopy(iv, 0, result, 0, IV_LENGTH);
          System.arraycopy(ciphertext, 0, result, IV_LENGTH, ciphertext.length);
          return Base64.getEncoder().encodeToString(result);
      }

      // Decrypt: extract IV from prefix; decrypt remainder
      public String decrypt(String encoded, SecretKey key) throws Exception {
          byte[] data = Base64.getDecoder().decode(encoded);
          byte[] iv         = Arrays.copyOfRange(data, 0, IV_LENGTH);
          byte[] ciphertext = Arrays.copyOfRange(data, IV_LENGTH, data.length);

          Cipher cipher = Cipher.getInstance(ALGORITHM);
          cipher.init(Cipher.DECRYPT_MODE, key, new GCMParameterSpec(TAG_LENGTH, iv));
          return new String(cipher.doFinal(ciphertext), UTF_8);
      }
  }
```

### 5.3 Key management

```
Key management system (KMS):
  AWS KMS, Google Cloud KMS, Azure Key Vault, HashiCorp Vault Transit
  The KMS holds master keys; application never sees the master key.
  Application sends plaintext → KMS encrypts with master key → returns ciphertext (envelope encryption).
  Or: application generates a data key (DEK) → KMS encrypts DEK with master key (KEK).
      Application uses DEK to encrypt data locally; stores encrypted DEK with data.

Envelope encryption (the standard pattern):
  1. Generate a data encryption key (DEK) — random 256-bit AES key.
  2. Encrypt data with DEK locally (fast, no KMS call per record).
  3. Call KMS to encrypt the DEK with the customer master key (CMK).
  4. Store: encrypted data + encrypted DEK alongside the data.
  5. To decrypt: call KMS to decrypt DEK; use DEK to decrypt data.

Benefits:
  One KMS call per data object (not per byte).
  KMS never sees the data; only the DEK.
  Key rotation: re-encrypt the DEK with new CMK; no data migration.
  Customer-managed keys (CMK): customer controls their data; can revoke access by deleting CMK.

Key rotation without data migration:
  Step 1: Generate new CMK version.
  Step 2: New writes use new CMK version to encrypt new DEKs.
  Step 3: Old data: DEKs encrypted with old CMK remain readable (KMS supports old versions).
  Step 4: Background job: re-encrypt old DEKs with new CMK at leisure (no urgency).
  Step 5: Old CMK version deactivated when all DEKs migrated.
  Data never needs to be decrypted and re-encrypted — only the DEK is re-encrypted.
```

## 6. Security Monitoring

### 6.1 What to log for security

```
Security-relevant events that must be logged:

Authentication events:
  Successful login (user, IP, device, timestamp)
  Failed login attempts (user, IP, count, timestamp)
  MFA challenges and outcomes
  Password changes and resets
  Account lockouts
  Session creation and invalidation

Authorisation events:
  Access denied (user, resource, action, reason)
  Privilege escalation attempts
  Admin actions (any action by admin role)

Data access events:
  Access to PII or sensitive data (who, what, when)
  Bulk data exports (>N records)
  Data deletion

System events:
  Secret/credential access (Vault audit log)
  Configuration changes
  Certificate issuance and rotation
  Infrastructure changes (Terraform apply, Kubernetes deployment)

Log format requirements:
  Structured (JSON) — machine-parseable, not freeform text
  Include: timestamp (ISO 8601 UTC), service, event_type, user_id, ip_address,
           request_id, outcome (success/failure), affected_resource
  Do NOT log: passwords, tokens, API keys, credit card numbers, SSNs
  Log masking: "user logged in with token [REDACTED]" not the token itself
```

### 6.2 SIEM and threat detection

```
SIEM (Security Information and Event Management):
  Aggregates logs from all systems; correlates events; generates alerts.
  Examples: Splunk, Datadog Security, Elastic Security, AWS Security Hub.

Detection rules:

1. Brute force detection:
   Alert: > 10 failed login attempts from the same IP in 5 minutes.
   Action: temporary IP block; notify security team.

2. Credential stuffing:
   Alert: > 5 failed logins for different accounts from the same IP.
   Action: require CAPTCHA; throttle IP; investigate.

3. Impossible travel:
   Alert: login from New York, then login from London 10 minutes later
          (physically impossible without time travel).
   Action: flag session; require step-up authentication; notify user.

4. Privilege escalation:
   Alert: user assigned admin role, then immediately accesses sensitive endpoint.
   Action: alert security; require approval for role changes.

5. Data exfiltration:
   Alert: user downloads > 10,000 records in a single session.
   Alert: bulk export at unusual hours (2 AM).
   Action: alert security; temporarily require approval for bulk exports.

6. Secrets access outside working hours:
   Alert: Vault secret accessed at 3 AM from unfamiliar IP.
   Action: alert security; potentially revoke and rotate.
```

### 6.3 Security headers and WAF

```
Web Application Firewall (WAF):
  Sits in front of the application; inspects HTTP traffic.
  Blocks common attacks: SQL injection, XSS, path traversal, rate abuse.
  AWS WAF, Cloudflare WAF, ModSecurity.
  Managed rule sets: OWASP core rule set blocks the OWASP Top 10.

  WAF is a detection and mitigation layer, not a replacement for
  secure application code. Attackers can bypass WAFs; secure code cannot be bypassed.

Security response headers:
  X-Content-Type-Options: nosniff
    → Prevents MIME-type sniffing (browser must use declared Content-Type)

  X-Frame-Options: DENY
    → Prevents clickjacking (page cannot be embedded in an iframe)
    → Or: Content-Security-Policy: frame-ancestors 'none' (newer, more flexible)

  Referrer-Policy: strict-origin-when-cross-origin
    → Limits referrer information sent in cross-origin requests

  Permissions-Policy: camera=(), microphone=(), geolocation=()
    → Disables browser features not needed by the application

// Spring Security: add security headers to all responses
@Configuration
public class SecurityHeadersConfig {
    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        http.headers(headers -> headers
            .contentSecurityPolicy(csp ->
                csp.policyDirectives("default-src 'self'; script-src 'self'; style-src 'self'"))
            .frameOptions(frame -> frame.deny())
            .contentTypeOptions(Customizer.withDefaults())
            .httpStrictTransportSecurity(hsts ->
                hsts.maxAgeInSeconds(31536000).includeSubDomains(true))
            .permissionsPolicyHeader(pp ->
                pp.policy("camera=(), microphone=(), geolocation=()"))
        );
        return http.build();
    }
}
```

## 7. Compliance Requirements

### 7.1 GDPR (General Data Protection Regulation)

```
Who it applies to: any organisation processing personal data of EU residents.
Personal data: any information that can identify a natural person.
  Direct identifiers: name, email, phone, national ID
  Indirect identifiers: IP address, cookie ID, device fingerprint, location data

Key principles:
  Lawful basis:    must have a legal reason to process data (consent, contract, legitimate interest)
  Data minimisation: collect only what is necessary for the stated purpose
  Storage limitation: do not retain longer than necessary; define retention periods
  Data portability: user can request export of their data in machine-readable format
  Right to erasure: "right to be forgotten" — user can request deletion of their data
  Privacy by design: privacy built into systems, not added later

Technical requirements for GDPR:

1. Data inventory and classification:
   Document: what personal data is collected, where it is stored, who can access it,
   how long it is retained, and what is the legal basis for processing.

2. Consent management:
   Explicit, informed, freely given consent before collecting data.
   Consent records stored (who consented, when, to what, via which mechanism).
   Easy withdrawal of consent (one click, not buried in settings).

3. Right to erasure implementation:
   When a user requests deletion: all their PII must be removed from:
     Primary database (hard delete or pseudonymisation)
     Backups (complex: backups are by definition historical)
     Analytics systems, data warehouses
     Third-party processors (must notify them)
   Solution: pseudonymisation in primary DB + backup retention policy.
   Backups containing old PII: acceptable if PII is pseudonymised before backup.

4. Data breach notification:
   Must notify supervisory authority within 72 hours of discovering a breach.
   Must notify affected individuals if the breach is likely to result in high risk.
   Requires: incident response plan, breach detection capability, contact procedures.

5. Data Processing Agreements (DPAs):
   Any third party that processes your users' data (cloud provider, analytics tool,
   email service) must have a DPA in place.
```

### 7.2 Right to erasure — engineering implementation

```java
// Right to erasure: delete or anonymise all PII for a user
@Service
@Transactional
public class UserErasureService {

    public void eraseUser(Long userId) {
        log.info("Starting GDPR erasure for user {}", userId);

        // 1. Anonymise primary user record (preserve aggregate data, remove PII)
        User user = userRepository.findById(userId).orElseThrow();
        user.setEmail("deleted_" + userId + "@anonymised.invalid");
        user.setFirstName("Deleted");
        user.setLastName("User");
        user.setPhoneNumber(null);
        user.setAddressLine1(null);
        user.setDeletedAt(Instant.now());
        user.setIsAnonymised(true);
        userRepository.save(user);

        // 2. Delete or anonymise related PII tables
        paymentMethodRepository.deleteByUserId(userId);
        addressRepository.deleteByUserId(userId);
        sessionRepository.deleteByUserId(userId);

        // 3. Anonymise order history (preserve financial records for legal obligation)
        // Legal basis: financial records must be kept 7 years (tax law)
        orderRepository.findByUserId(userId).forEach(order -> {
            order.setShippingAddressLine1("REDACTED");
            order.setShippingCity("REDACTED");
            order.setCustomerEmail("REDACTED");
            // Keep: orderId, amount, date, product IDs (not PII)
            orderRepository.save(order);
        });

        // 4. Delete from search indices
        searchIndexService.deleteUser(userId);

        // 5. Publish erasure event for downstream systems (analytics, CRM, etc.)
        eventPublisher.publish(new UserErasureRequestedEvent(userId, Instant.now()));

        // 6. Audit log the erasure (GDPR requires proof of erasure)
        auditLogService.record(AuditEvent.of("USER_ERASED", userId, "GDPR request"));

        log.info("GDPR erasure completed for user {}", userId);
    }
}
```


## 8. Common Vulnerabilities and Mitigations

### 8.1 OWASP Top 10 mitigations

```
A01: Broken Access Control
  → Enforce authorisation on every endpoint (not just the UI layer)
  → DENY by default; only permit explicitly
  → Validate user can access the specific resource (not just the resource type)
  → Use row-level security for multi-tenant data

A02: Cryptographic Failures
  → Encrypt all sensitive data at rest and in transit
  → Use strong algorithms (AES-256-GCM, bcrypt/Argon2id, RSA-2048+)
  → Never roll your own cryptography
  → Use KMS for key management

A03: Injection (SQL, NoSQL, OS Command)
  → Use parameterised queries / prepared statements (never string concatenation)
  → Use ORM (JPA/Hibernate) which parameterises by default
  → Validate and sanitise input; whitelist, not blacklist
  → Least privilege DB user: cannot DROP or execute admin functions

A05: Security Misconfiguration
  → Infrastructure as code with security policy enforcement (Terraform Sentinel)
  → Default credentials changed; default features disabled
  → Security headers on all responses
  → Error responses never expose stack traces or internals

A07: Identification and Authentication Failures
  → MFA required for all user accounts and all admin access
  → Secure session management (HttpOnly, Secure, SameSite cookies)
  → Account lockout after failed attempts (with exponential backoff)
  → Secure password reset (time-limited, single-use tokens)

A09: Security Logging and Monitoring Failures
  → Log all authentication events, access denials, and data access
  → Alert on anomalous patterns (brute force, impossible travel, bulk exports)
  → Centralised log aggregation with tamper protection
  → Incident response plan tested annually
```

## 9. Interview Questions

**Q: How would you design authentication and authorisation for a multi-tenant SaaS platform?**

A: The design separates authentication (who are you?) from authorisation (what can you do?) across three scopes. For authentication, I implement OAuth2 with OIDC: users authenticate against an identity provider (Auth0, Cognito, or a self-hosted Keycloak instance) which issues JWT access tokens and OIDC ID tokens. The access token contains the user's identity, their tenant ID, and their roles within that tenant. Short expiry (15 minutes) with refresh token rotation limits the blast radius of a stolen token. MFA is mandatory for admin accounts and configurable per tenant for their users — enterprise tenants often require it contractually. For authorisation, I use a hybrid model. Coarse-grained access (can this user type access this API endpoint?) is handled by RBAC implemented in the API gateway based on the JWT's `roles` claim. Fine-grained access (can this specific user see this specific record?) is handled at the application layer using a relationship-based model: resources belong to tenants, and users have roles within tenants. Row-level security in PostgreSQL enforces tenant data isolation at the database layer, preventing application bugs from causing cross-tenant data leakage. For the service layer, every service re-validates the JWT on each request — no implicit trust based on the request coming from an internal service. Service-to-service calls use mTLS via the service mesh, with short-lived certificates issued by SPIFFE/SPIRE.

---

**Q: What is zero trust architecture and how does it differ from traditional perimeter security?**

A: Traditional perimeter security treats the internal network as trusted and the external network as untrusted — once you are inside the corporate VPN or data centre network, you can reach most internal services. This model fails in two ways: insider threats (a malicious employee is already inside), and lateral movement after a breach (one compromised service can reach all others). Zero trust eliminates the concept of a trusted network location. Every request — internal or external — must be authenticated, and every service must verify that the caller is authorised for the specific action being requested. In practice this means four things. First, workload identity: every service gets a cryptographic identity (SPIFFE SVID certificate), and every service-to-service call uses mTLS where both parties present and verify certificates. Second, least privilege network policy: Kubernetes NetworkPolicy and Istio AuthorizationPolicy allow each service to receive traffic only from explicitly permitted callers, not from the entire cluster. Third, short-lived credentials: tokens and certificates expire in 15 minutes to 24 hours; the window of opportunity from a compromised credential is bounded. Fourth, assume breach: even if an attacker compromises one service, micro-segmentation limits what they can reach. The most visible operational change is that "works from the office" is no longer a security boundary — a developer connecting from a coffee shop with MFA and a managed device gets the same access as from the office, and an insider threat from the office gets the same scrutiny as an external actor.

---

**Q: How do you manage secrets in a microservices environment at scale?**

A: The goal is that no secret ever appears in source code, configuration files, or environment variables in a Dockerfile. I use HashiCorp Vault as the central secret management platform, with application authentication via Kubernetes service account tokens (the Vault Kubernetes auth method). On startup, each service authenticates to Vault using its Kubernetes service account token and receives a Vault token with access scoped to exactly the secrets it needs. For database credentials, I use Vault's dynamic secrets: Vault generates a unique username and password for each service instance with a 1-hour lease. If the service crashes or is compromised, those credentials expire automatically — there is no shared long-lived password to rotate across all instances. For static secrets (third-party API keys), I store them in Vault and the Spring Cloud Vault integration fetches them at startup. The application never stores the secret in memory beyond what is needed for the immediate operation; database connection pools refresh credentials before lease expiry. Secret rotation for JWT signing keys uses the `kid` (key ID) field in the JWT header — the identity provider publishes a JWKS (JSON Web Key Set) endpoint, services cache the public keys keyed by `kid`, and key rotation is transparent to all services. The audit log in Vault records every secret access with the caller's identity, timestamp, and which secret was accessed — essential for incident response when we need to know which services had access to a compromised credential.

---

**Q: Walk through your approach to GDPR compliance for a system storing user PII.**

A: GDPR compliance is primarily an engineering and process problem, not a legal one. I approach it in three phases. First, data inventory: I map every place user PII flows — what is collected, where it is stored, who can access it, how long it is retained, and the legal basis for processing. This inventory drives technical decisions. Second, technical controls: PII fields in the database are encrypted at rest using AES-256-GCM with keys managed by KMS; backups are encrypted before transmission. Access to PII tables is restricted by database user permissions and row-level security. Every access to PII is logged with user ID, timestamp, and what was accessed — audit logs are tamper-evident and retained for the compliance window. Third, the right to erasure: when a user requests deletion, I do not hard-delete records because related data (financial records, fraud signals) may need retention for other legal obligations. Instead, I pseudonymise the user record — replacing email, name, and contact information with anonymised placeholders — and publish a deletion event to downstream systems (analytics, CRM) which must also purge or anonymise their copies. The challenge is backups: I implement a backup retention policy that matches the GDPR retention periods and exclude anonymised users from restores where possible. Consent management is implemented as a first-class feature: consent is explicit, versioned, recorded with a timestamp and the exact consent text, and can be withdrawn in two clicks — the withdrawal is processed synchronously and the consent record is updated immediately.

---

**Q: How do you design for defence in depth when a perimeter control fails?**

A: Defence in depth means assuming any single control will eventually fail and designing so that failure of one layer does not result in a full breach. I design with five layers. The first is the network layer: WAF to block common attack patterns, DDoS protection, and network policies that prevent direct database access from untrusted sources. But I do not rely on this — the second layer is identity: every request is authenticated regardless of where it originates, using short-lived tokens. If the network layer is bypassed, an unauthenticated request is still rejected by the identity layer. The third layer is authorisation at the application layer: every operation checks that the authenticated identity is permitted for the specific resource. A compromised service can only perform actions its own identity is permitted to perform — not arbitrary actions across the system. The fourth layer is data: sensitive fields are encrypted with application-managed keys such that a database dump without the decryption keys is useless to an attacker. The fifth layer is detection and response: all authentication events, authorisation denials, and data access events are logged to a tamper-evident SIEM. Anomalous patterns — impossible travel, bulk exports, access outside business hours — trigger automated alerts. The assumption is that the attacker is already inside. The question is: how much time do they have before we detect them, and what can they access in that time? Defence in depth limits the answer to both questions.

## 10. Key Takeaways

- **Authentication and authorisation are separate concerns**: authenticate first (is this a valid identity?); then authorise each action (is this identity permitted for this specific resource?); never conflate them
- **RBAC for coarse-grained access; ABAC or ReBAC for fine-grained**: RBAC is simple and auditable; ABAC handles context-sensitive policies; ReBAC handles relationship-driven sharing (Google Drive model)
- **Zero trust: network location is not a credential**: mTLS between all services; short-lived credentials; least-privilege network policy; assume the attacker is already inside
- **Secrets must never live in code, config files, or environment variables**: use Vault or cloud secret manager; dynamic secrets with automatic rotation are far more secure than long-lived shared credentials
- **AES-256-GCM for symmetric encryption**: authenticated encryption detects tampering; always generate a fresh random IV per operation; envelope encryption (DEK + KEK) enables key rotation without data migration
- **GDPR right to erasure is an engineering problem**: pseudonymise rather than hard-delete (preserves referential integrity for financial records); publish deletion events to all downstream systems; backup retention policy must be consistent with erasure obligations
- **Log security events in structured format with requestId**: every authentication event, access denial, and PII access logged; SIEM correlates events for anomaly detection; audit logs are the investigation tool when an incident occurs
- **Session cookies must be HttpOnly + Secure + SameSite=Strict**: HttpOnly prevents JavaScript access (XSS mitigation); Secure enforces HTTPS; SameSite prevents CSRF; store session token hash in database, not plaintext
- **OWASP Top 10 is a checklist, not a goal**: parameterised queries prevent injection; denying by default prevents broken access control; short-lived tokens limit authentication failures — build these in from day one, not as afterthoughts
- **Compliance (GDPR, PCI, SOC 2) is evidence of existing controls**: the compliance requirement should not be creating the security practice; it should be formalising and auditing what good security engineering already does
