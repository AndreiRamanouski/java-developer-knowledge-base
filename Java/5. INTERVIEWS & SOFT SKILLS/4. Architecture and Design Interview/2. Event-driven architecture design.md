# Event-Driven Architecture Design

## Overview

Event-driven architecture (EDA) treats state changes as first-class citizens — every significant change in a system produces an event that can be consumed by any interested party, now or in the future. EDA enables loose coupling, temporal decoupling (producers and consumers do not need to be available simultaneously), and audit by default. It is the natural architectural style for systems that must scale independently, support multiple downstream consumers, and maintain a reliable history of what happened and when.

**The core shift in thinking:**

```
Request-response (RPC style):
  "Do this thing and give me the result right now."
  Caller waits; callee must be available; tight temporal coupling.

Event-driven style:
  "Something happened; here is the record of it."
  Publisher does not wait; subscriber processes when ready; loose coupling.

Not either/or: most production systems combine both.
  Synchronous RPC:  user-facing read paths where latency matters
  Event-driven:     write paths, cross-service workflows, async processing

The decision: "Does the publisher need to know what happens next,
and does it need to know right now?"
  Yes: use RPC.
  No:  use events.
```

**Event vocabulary:**

```
Event:          An immutable record of something that happened in the past.
                "OrderPlaced", "PaymentProcessed", "UserRegistered"
                Past tense; describes a fact, not a command.

Command:        A request to perform an action. May be rejected.
                "PlaceOrder", "ProcessPayment", "RegisterUser"
                Imperative; describes intent.

Query:          A request for information. No side effects.

Event stream:   An ordered, durable sequence of events (Kafka topic, Kinesis stream).

Event log:      The source of truth in event sourcing — a complete history of events.

Domain event:   An event meaningful to the business domain.
Integration event: An event shared across service boundaries.
```

## 1. Event Storming

### 1.1 What event storming is

```
Event Storming is a collaborative workshop technique for exploring a complex
domain by mapping domain events, commands, aggregates, policies, and read models
on a large surface (a wall covered with sticky notes) with all stakeholders present —
domain experts, developers, product managers, and architects.

Invented by Alberto Brandolini. Goal: build a shared understanding of the domain
before building software, surfacing assumptions and disagreements that would
otherwise be discovered late and expensively.

Unlike traditional requirements gathering:
  No UML diagrams (abstract)
  No data models (implementation detail)
  No process flowcharts (too sequential for complex domains)

Event storming uses: coloured sticky notes and a timeline. Everyone participates.
The output: a visual map of the domain that becomes the basis for service
boundaries, event schemas, and aggregate design.
```

### 1.2 Event storming vocabulary (sticky note colours)

```
Orange:  Domain Events
         Something that happened in the business domain.
         "Order Placed", "Payment Failed", "Item Shipped"
         Always past tense. These are the starting point.

Blue:    Commands
         Something an actor or system does to cause an event.
         "Place Order", "Cancel Order", "Submit Payment"
         Precede domain events.

Yellow:  Aggregates
         The domain entity that handles the command and produces the event.
         "Order", "Payment", "Inventory"
         Between command and event.

Purple:  Policies / Reactions
         Business rules that react to events and issue new commands.
         "When order placed → reserve inventory"
         "When payment fails → notify customer and cancel reservation"
         These are the glue between events and the commands they trigger.

Pink:    Read Models / Projections
         The data a user sees to make a decision.
         "Order History List", "Available Inventory View"
         What users look at before issuing commands.

Red:     Hotspots / Problems
         Disagreements, unclear areas, missing knowledge.
         These are as valuable as the agreed elements — they surface unknowns.

Actors: stick figures representing users or external systems that issue commands.
```

### 1.3 Event storming process

```
Phase 1: Chaotic exploration (30-60 minutes)
  Everyone writes domain events they can think of on orange stickies.
  No discussion, no organisation. Volume and diversity over precision.
  Place all events on the wall in rough time order (left = earlier, right = later).
  Goal: surface the full scope of the domain as perceived by the room.

Phase 2: Enforce the timeline (30-45 minutes)
  Walk the timeline left-to-right together.
  Identify duplicates, contradictions, and missing events.
  Arrange events into causal sequences: "A happens, then B, then C."
  Add red stickies for every disagreement or unknown.

Phase 3: Identify commands and actors (30 minutes)
  For each event: what command caused it? Who or what issued the command?
  Add blue stickies (commands) and actor annotations.
  Some events are caused by policies reacting to other events, not direct user commands.

Phase 4: Identify aggregates (30 minutes)
  Group commands and events by the aggregate that handles them.
  "Place Order" + "Order Placed" + "Order Cancelled" → Order aggregate.
  Add yellow stickies for each aggregate.
  Aggregates that handle too many commands/events are over-sized and may need splitting.

Phase 5: Add policies and read models (30 minutes)
  For each event: does it trigger another command automatically?
  Add purple stickies for policies.
  For each command: what read model does the user consult before issuing it?
  Add pink stickies for read models.

Phase 6: Define bounded contexts
  Look for natural clusters of aggregates, events, and policies.
  Clusters with few dependencies between them = candidate bounded contexts = candidate services.
  Red stickies at cluster boundaries indicate coupling that needs design attention.

Output: a wall map that becomes the shared domain model. Photograph it. It is the
starting point for your event schema design, service decomposition, and aggregate implementation.
```

### 1.4 Event storming anti-patterns

```
Anti-pattern 1: Developers run it without domain experts
  Result: a technical model of the existing system, not the business domain.
  Fix: non-negotiable — the domain experts (product, operations, business) must be present.

Anti-pattern 2: Too much process too early
  Result: team spends the session debating sticky note colours and placement rules
  before exploring the domain.
  Fix: start with chaos. Organisation comes later.

Anti-pattern 3: Events named as technical operations
  WRONG: "Database Record Updated", "API Called", "Queue Message Sent"
  RIGHT: "Order Confirmed", "Payment Received", "Delivery Scheduled"
  Events should be meaningful to someone who does not know the implementation.

Anti-pattern 4: Stopping after one session
  Result: the map captures one perspective; edge cases and error paths are missing.
  Fix: hold two or three sessions; invite different stakeholders to each.
  Error paths (payment failures, inventory shortfalls) are often as important as the happy path.
```

## 2. Event Sourcing Implementation

### 2.1 What event sourcing is

```
In a traditional (state-based) system:
  Store the CURRENT state of an entity.
  Update it in place when something changes.
  History is lost unless explicitly logged.

In event sourcing:
  Store the SEQUENCE OF EVENTS that produced the current state.
  Current state is derived by replaying events.
  History is the primary record; current state is a projection.

Traditional: bank account record = { accountId, balance: $1,500 }
Event sourced: event log = [
  AccountOpened { accountId, initialBalance: $1,000 },
  MoneyDeposited { accountId, amount: $500, timestamp },
  ... any future events
]
Current balance = replay all events: $1,000 + $500 = $1,500.

What you gain:
  Complete audit trail:       every change is recorded with who, what, when
  Temporal queries:           "what was the account balance on March 15?"
  Debugging:                  replay events to reproduce any historical state
  Event-driven integration:   events are the natural integration point for other services
  Undo/redo:                  append a compensating event (do not delete past events)

What it costs:
  Query complexity:           reading current state requires replay or a projection
  Storage growth:             event log grows indefinitely (manage with snapshots)
  Schema evolution:           old events must be deserialised by new code (careful design)
  Mental model shift:         team must think in terms of events, not CRUD
```

### 2.2 Event store design

```
The event store is the core infrastructure of event sourcing.
It must support:
  Append-only writes:     events are never updated or deleted
  Optimistic concurrency: detect concurrent writes to the same aggregate
  Stream reads:           read events for a specific aggregate by ID
  Global sequence:        read all events across all aggregates in order

Schema:
CREATE TABLE events (
    event_id         BIGSERIAL       PRIMARY KEY,        -- global sequence
    stream_id        UUID            NOT NULL,            -- aggregate ID
    stream_type      VARCHAR(100)    NOT NULL,            -- aggregate type
    event_type       VARCHAR(100)    NOT NULL,            -- event class name
    sequence_number  BIGINT          NOT NULL,            -- position within stream
    payload          JSONB           NOT NULL,            -- event data
    metadata         JSONB,                               -- correlation_id, user_id, timestamp
    created_at       TIMESTAMPTZ     NOT NULL DEFAULT NOW(),
    UNIQUE (stream_id, sequence_number)                   -- optimistic concurrency check
);

CREATE INDEX idx_events_stream ON events(stream_id, sequence_number);
CREATE INDEX idx_events_global  ON events(event_id);     -- for global event reading
```

### 2.3 Aggregate with event sourcing

```java
// Base class for event-sourced aggregates
public abstract class AggregateRoot {
    private final List<DomainEvent> uncommittedEvents = new ArrayList<>();
    protected UUID id;
    protected int version = 0;

    // Apply event to update state (called during both new events and replay)
    protected abstract void apply(DomainEvent event);

    // Record a new event: apply it immediately + mark as uncommitted
    protected void raiseEvent(DomainEvent event) {
        apply(event);              // update in-memory state
        uncommittedEvents.add(event); // queue for persistence
        version++;
    }

    // Reconstitute from event history (replay)
    public void loadFromHistory(List<DomainEvent> history) {
        for (DomainEvent event : history) {
            apply(event);
            version++;
        }
    }

    public List<DomainEvent> getUncommittedEvents() {
        return Collections.unmodifiableList(uncommittedEvents);
    }

    public void markEventsAsCommitted() {
        uncommittedEvents.clear();
    }
}

// Concrete aggregate: Order
public class Order extends AggregateRoot {
    private OrderStatus status;
    private BigDecimal totalAmount;
    private List<OrderItem> items = new ArrayList<>();
    private String customerId;

    // Command handler: Place new order
    public static Order place(String orderId, String customerId, List<OrderItem> items) {
        Order order = new Order();
        BigDecimal total = items.stream()
            .map(i -> i.getPrice().multiply(BigDecimal.valueOf(i.getQuantity())))
            .reduce(BigDecimal.ZERO, BigDecimal::add);

        order.raiseEvent(new OrderPlacedEvent(orderId, customerId, items, total, Instant.now()));
        return order;
    }

    // Command handler: Confirm order
    public void confirm() {
        if (status != OrderStatus.PLACED) {
            throw new IllegalStateException("Only PLACED orders can be confirmed");
        }
        raiseEvent(new OrderConfirmedEvent(id.toString(), Instant.now()));
    }

    // Command handler: Cancel order
    public void cancel(String reason) {
        if (status == OrderStatus.SHIPPED) {
            throw new IllegalStateException("Cannot cancel a shipped order");
        }
        raiseEvent(new OrderCancelledEvent(id.toString(), reason, Instant.now()));
    }

    // State mutations: only via apply() — called by both raiseEvent() and loadFromHistory()
    @Override
    protected void apply(DomainEvent event) {
        switch (event) {
            case OrderPlacedEvent e -> {
                this.id = UUID.fromString(e.getOrderId());
                this.customerId = e.getCustomerId();
                this.items = new ArrayList<>(e.getItems());
                this.totalAmount = e.getTotal();
                this.status = OrderStatus.PLACED;
            }
            case OrderConfirmedEvent e -> this.status = OrderStatus.CONFIRMED;
            case OrderCancelledEvent e -> this.status = OrderStatus.CANCELLED;
            default -> throw new IllegalArgumentException("Unknown event: " + event.getClass());
        }
    }
}
```

### 2.4 Event store repository

```java
@Repository
public class EventSourcedOrderRepository {
    private final EventStore eventStore;
    private final SnapshotStore snapshotStore;

    // Load aggregate: try snapshot first, then replay events since snapshot
    public Order findById(String orderId) {
        Order order = new Order();

        // Try to load snapshot (avoids replaying entire history for old aggregates)
        Optional<Snapshot> snapshot = snapshotStore.findLatest(orderId);
        int startingVersion = 0;

        if (snapshot.isPresent()) {
            order = (Order) snapshot.get().getState();
            startingVersion = snapshot.get().getVersion();
        }

        // Load events since the snapshot (or from beginning if no snapshot)
        List<DomainEvent> events = eventStore.loadStream(orderId, startingVersion);
        order.loadFromHistory(events);
        return order;
    }

    // Save aggregate: append new events with optimistic concurrency
    @Transactional
    public void save(Order order) {
        List<DomainEvent> newEvents = order.getUncommittedEvents();
        if (newEvents.isEmpty()) return;

        int expectedVersion = order.getVersion() - newEvents.size();

        // Optimistic concurrency: fail if another writer has already written at this version
        eventStore.appendToStream(
            order.getId().toString(),
            expectedVersion,   // expected current version in DB
            newEvents
        );

        order.markEventsAsCommitted();

        // Create snapshot every 50 events to avoid long replay chains
        if (order.getVersion() % 50 == 0) {
            snapshotStore.save(new Snapshot(order.getId().toString(), order.getVersion(), order));
        }
    }
}
```

### 2.5 Snapshots

```
Problem: an order with 500 events (placed, updated, cancelled, re-opened, shipped...)
         must replay all 500 events to reconstitute current state.
         At scale, this is O(n) per load where n = event count.

Solution: periodic snapshots.
  After every N events (N=50 is a common choice), persist the current aggregate state
  as a snapshot. On next load: start from the latest snapshot, replay only
  events since the snapshot.

Snapshot table:
CREATE TABLE snapshots (
    aggregate_id    UUID         NOT NULL,
    version         INT          NOT NULL,
    state           JSONB        NOT NULL,   -- serialised aggregate state
    created_at      TIMESTAMPTZ  NOT NULL,
    PRIMARY KEY (aggregate_id, version)
);

Trade-off:
  More frequent snapshots: faster reads, more storage writes.
  Less frequent snapshots: slower reads (more replay), fewer writes.
  50-event interval is a practical default; tune based on aggregate change rate.

Important: snapshots do not replace the event log.
  The event log is still the source of truth.
  Snapshots are a performance optimisation.
  If a snapshot is corrupted: delete it and replay from the full event log.
```

## 3. CQRS Pattern

### 3.1 What CQRS is and when to use it

```
Command Query Responsibility Segregation (CQRS): separate the model for
reading data (queries) from the model for writing data (commands).

Why separation helps:
  Read patterns differ from write patterns:
    Writes: validate business rules → update domain model → produce events
    Reads: join data from many sources → format for display → serve fast

  Scaling differs:
    Writes: typically 10-100× fewer than reads; benefit from strong consistency
    Reads: high volume; benefit from caching, denormalisation, eventual consistency

  Model complexity differs:
    Command model (aggregate): rich domain logic, business rules, invariants
    Query model (projection): simple, flat, optimised for the specific read use case

CQRS does NOT require:
  Event sourcing (though they work naturally together)
  Separate databases (though separate read stores are common)
  Microservices (though useful at any scale)

CQRS requires:
  Discipline: the read path must NEVER execute a command;
              the write path must NEVER return data for display
  Eventual consistency on the read side (projections may lag slightly behind writes)
```

### 3.2 CQRS architecture

```
┌─────────────────────────────────────────────────────────────────────┐
│                           Client                                     │
└───────────────┬─────────────────────────────────────────────────────┘
                │
      ┌─────────▼──────────────────────────────────────────────┐
      │                    API Layer                            │
      │   Commands routed to Command Bus                        │
      │   Queries routed to Query Bus                           │
      └──────────┬────────────────────────────┬────────────────┘
                 │ Commands                    │ Queries
    ┌────────────▼───────────┐    ┌────────────▼────────────────┐
    │   Command Side          │    │   Query Side                │
    │                         │    │                             │
    │  Command Handler        │    │  Query Handler              │
    │  → Load Aggregate       │    │  → Read from Read Store     │
    │  → Execute Business     │    │  → Return DTO directly      │
    │    Logic                │    │    (no domain objects)      │
    │  → Raise Domain Events  │    │                             │
    │  → Save to Event Store  │    │  Read Store:                │
    └────────────┬────────────┘    │  (PostgreSQL read tables,   │
                 │ Domain Events   │   Elasticsearch, Redis,     │
                 ▼                 │   MongoDB views)            │
    ┌────────────────────────┐     └─────────────────────────────┘
    │   Event Bus / Kafka    │──────────────────────────▲
    └────────────────────────┘     Projections consume events
                                   and update read store
```

### 3.3 Projections — building the read model

```java
// Projection: listens for domain events, maintains a denormalised read model
// This is what powers the query side of CQRS

@Component
public class OrderSummaryProjection {
    private final OrderSummaryRepository readRepository;

    // React to OrderPlacedEvent: create initial read model entry
    @EventHandler
    public void on(OrderPlacedEvent event) {
        OrderSummary summary = OrderSummary.builder()
            .orderId(event.getOrderId())
            .customerId(event.getCustomerId())
            .status("PLACED")
            .totalAmount(event.getTotal())
            .itemCount(event.getItems().size())
            .placedAt(event.getTimestamp())
            .build();
        readRepository.save(summary);
    }

    // React to OrderConfirmedEvent: update status
    @EventHandler
    public void on(OrderConfirmedEvent event) {
        readRepository.findById(event.getOrderId()).ifPresent(summary -> {
            summary.setStatus("CONFIRMED");
            summary.setConfirmedAt(event.getTimestamp());
            readRepository.save(summary);
        });
    }

    // React to OrderShippedEvent: update with tracking info
    @EventHandler
    public void on(OrderShippedEvent event) {
        readRepository.findById(event.getOrderId()).ifPresent(summary -> {
            summary.setStatus("SHIPPED");
            summary.setTrackingNumber(event.getTrackingNumber());
            summary.setShippedAt(event.getTimestamp());
            readRepository.save(summary);
        });
    }
}

// Read model: optimised for the specific query, not for the domain
@Entity
@Table(name = "order_summaries")
public class OrderSummary {
    @Id private String orderId;
    private String customerId;
    private String status;
    private BigDecimal totalAmount;
    private int itemCount;
    private Instant placedAt;
    private Instant confirmedAt;
    private Instant shippedAt;
    private String trackingNumber;
    // All data the "order list" screen needs — in one flat row, no joins
}

// Query handler: simple read, no business logic
@QueryHandler
public List<OrderSummary> handle(GetOrdersByCustomerQuery query) {
    return orderSummaryRepository.findByCustomerId(
        query.getCustomerId(),
        PageRequest.of(query.getPage(), query.getPageSize())
    );
```

### 3.4 Multiple read models for the same domain

```
The power of CQRS + projections: multiple read models from the same event stream,
each optimised for a different use case.

Example: the Order aggregate produces the same events, consumed by:
  OrderSummaryProjection    → flat table for "My Orders" screen
  OrderAnalyticsProjection  → ClickHouse table for revenue analytics
  OrderSearchProjection     → Elasticsearch index for admin search
  OrderNotificationProjection → triggers email/push when status changes
  InventoryProjection       → updates available stock in inventory service

All four are eventually consistent; all four consume the same Kafka stream.
Adding a new read model does not affect the write path at all.
This is the key extensibility benefit of CQRS + event sourcing.
```

### 3.5 Projection rebuild

```
When do you need to rebuild a projection?
  Bug fix in projection logic (incorrect state was computed)
  New projection added (needs historical data)
  Read model schema change requiring backfill

Rebuilding is safe because the event log is the source of truth.

Rebuild process:
  1. Pause the projection consumer (or use a separate consumer group for the rebuild).
  2. Truncate the read model table (or create a new table alongside the old one).
  3. Replay the event log from the beginning (or from a known checkpoint).
  4. While replay is in progress: old read model remains live for queries.
  5. Once replay completes and the new model is caught up: flip the read query to
     the new model; decommission the old one.

Zero-downtime rebuild:
  Run old and new projections in parallel during replay.
  Route 5% of queries to the new model to verify correctness.
  Route 100% once verified; stop the old projection.

This is analogous to blue-green deployment, applied to read models.
```

## 4. Saga Orchestration (Deep Dive)

### 4.1 Saga orchestration vs choreography — the full comparison

```
Both implement distributed workflows across services.
The question is where the coordination logic lives.

Choreography: each service reacts to events from others.
  Service A publishes event → Service B reacts → publishes event → Service C reacts

  Pros:
    Loose coupling: services do not know about each other
    Simple individual services: each service just handles its own domain
    Resilient: no single coordinator to fail

  Cons:
    Workflow is implicit: to understand the full transaction, you must trace
    events across all services — no single place to look
    Hard to test: requires all services running to test end-to-end flow
    Difficult compensations: each service must know when to compensate based
    on which events it receives — complex for multi-path workflows

Orchestration: a central saga coordinator directs the workflow.
  Orchestrator → Command to A → A responds → Orchestrator → Command to B...

  Pros:
    Explicit workflow: the full transaction is visible in one place
    Easy to test: test the orchestrator with mocked services
    Clear compensation: orchestrator explicitly directs rollback steps
    Conditional logic: orchestrator can handle branches and parallel steps

  Cons:
    Orchestrator knows about all services (more coupling)
    Orchestrator can become a bottleneck if not designed carefully
    More complex to build initially

Rule of thumb:
  Linear flow, 3 or fewer steps:   choreography
  Complex flow, branches, > 3 steps: orchestration
  Teams that value explicit workflows: orchestration (recommended for most production systems)
```

### 4.2 Saga state machine design

```java
// A saga is a durable state machine. It must survive restarts.
// State is persisted to a database; the saga resumes from its last state.

public enum CreateOrderSagaState {
    STARTED,
    INVENTORY_RESERVED,
    PAYMENT_CHARGING,
    PAYMENT_CHARGED,
    ORDER_CONFIRMED,
    // Compensation states
    PAYMENT_CHARGE_FAILED,
    INVENTORY_RELEASING,
    CANCELLED
}

@Entity
public class CreateOrderSagaData {
    @Id private String sagaId;
    @Enumerated(EnumType.STRING)
    private CreateOrderSagaState state;
    private String orderId;
    private String customerId;
    private BigDecimal amount;
    private String inventoryReservationId; // needed for compensation
    private String paymentTransactionId;   // needed for compensation
    private String failureReason;
    private int compensationRetryCount;
    private Instant createdAt;
    private Instant updatedAt;
}

// Saga coordinator — persists state transitions
@Service
@Transactional
public class CreateOrderSagaCoordinator {

    public void start(CreateOrderCommand cmd) {
        CreateOrderSagaData saga = new CreateOrderSagaData();
        saga.setSagaId(cmd.getIdempotencyKey());
        saga.setState(CreateOrderSagaState.STARTED);
        saga.setOrderId(cmd.getOrderId());
        saga.setCustomerId(cmd.getCustomerId());
        saga.setAmount(cmd.getAmount());
        sagaRepository.save(saga);

        // Send first command: reserve inventory
        inventoryClient.reserveInventory(ReserveInventoryCommand.builder()
            .sagaId(saga.getSagaId())
            .orderId(saga.getOrderId())
            .items(cmd.getItems())
            .idempotencyKey(saga.getSagaId() + ":inventory")
            .build());
    }

    public void onInventoryReserved(InventoryReservedEvent event) {
        CreateOrderSagaData saga = sagaRepository.findById(event.getSagaId())
            .orElseThrow(() -> new SagaNotFoundException(event.getSagaId()));

        // Guard: only proceed if in the expected state (idempotency)
        if (saga.getState() != CreateOrderSagaState.STARTED) return;

        saga.setInventoryReservationId(event.getReservationId());
        saga.setState(CreateOrderSagaState.INVENTORY_RESERVED);
        sagaRepository.save(saga);

        paymentClient.chargeCard(ChargeCardCommand.builder()
            .sagaId(saga.getSagaId())
            .customerId(saga.getCustomerId())
            .amount(saga.getAmount())
            .idempotencyKey(saga.getSagaId() + ":payment")
            .build());
    }

    public void onPaymentFailed(PaymentFailedEvent event) {
        CreateOrderSagaData saga = sagaRepository.findById(event.getSagaId())
            .orElseThrow(() -> new SagaNotFoundException(event.getSagaId()));

        saga.setFailureReason(event.getFailureReason());
        saga.setState(CreateOrderSagaState.PAYMENT_CHARGE_FAILED);
        sagaRepository.save(saga);

        // Compensate: release the inventory reservation
        inventoryClient.releaseInventory(ReleaseInventoryCommand.builder()
            .sagaId(saga.getSagaId())
            .reservationId(saga.getInventoryReservationId())
            .idempotencyKey(saga.getSagaId() + ":inventory:release")
            .build());
    }

    public void onInventoryReleased(InventoryReleasedEvent event) {
        CreateOrderSagaData saga = sagaRepository.findById(event.getSagaId())
            .orElseThrow(() -> new SagaNotFoundException(event.getSagaId()));

        if (saga.getState() == CreateOrderSagaState.PAYMENT_CHARGE_FAILED) {
            saga.setState(CreateOrderSagaState.CANCELLED);
            sagaRepository.save(saga);
            // Publish order cancelled event
            eventPublisher.publish(new OrderCancelledEvent(
                saga.getOrderId(), saga.getFailureReason()));
        }
    }
}
```

### 4.3 Saga timeout handling

```
What if a service never responds?
  The saga waits forever (blocked saga; resource leak).

Solution: timeouts with scheduled retry or escalation.

// Schedule a timeout check after N minutes
@Scheduled(fixedRate = 60_000)
public void checkSagaTimeouts() {
    Instant timeout = Instant.now().minus(Duration.ofMinutes(5));
    List<CreateOrderSagaData> stuckSagas = sagaRepository
        .findByStateAndUpdatedAtBefore(CreateOrderSagaState.INVENTORY_RESERVED, timeout);

    for (CreateOrderSagaData saga : stuckSagas) {
        if (saga.getCompensationRetryCount() < 3) {
            // Retry the stuck step
            retryCurrentStep(saga);
            saga.setCompensationRetryCount(saga.getCompensationRetryCount() + 1);
        } else {
            // Give up: compensate and cancel
            startCompensation(saga, "Timeout after 3 retries");
        }
        sagaRepository.save(saga);
    }
}

Timeout design principles:
  Each saga step should have a maximum expected duration.
  After timeout: retry (if idempotent) or compensate.
  Set a maximum total saga duration: after N minutes, force cancellation.
  Alert on sagas stuck in intermediate states — they indicate a service problem.
```

## 5. Event Schema Evolution

### 5.1 Why event schema evolution is hard

```
Events in an event-sourced system are permanent — they are never deleted
or modified after being written. This creates a challenge: code that was
written years later must be able to deserialise events written years earlier.

Two directions of compatibility:
  Backward compatibility:  new code can read old events
  Forward compatibility:   old code can read new events

Both are required in production:
  Backward: when you deploy new code, it replays old events during
            aggregate reconstitution and projection rebuild.
  Forward:  during rolling deployments, old code instances may receive
            events published by already-updated new instances.

The fundamental rule: once an event is published, treat its schema as immutable.
Adding fields is safe with defaults. Removing or renaming is not.
```

### 5.2 Upcasting — upgrading old event formats

```
Upcaster: a component that transforms an old event into a newer version
before the aggregate or projection processes it.

Allows: changing event schema while preserving backward compatibility with
the event log (which still contains old events that cannot be changed).

Example: OrderPlacedEvent v1 had a single "address" field.
         OrderPlacedEvent v2 split it into "shippingAddress" and "billingAddress".

// V1 event (stored in database, cannot be changed)
{ "type": "OrderPlacedEvent", "version": 1,
  "orderId": "123", "address": "123 Main St, NYC" }

// Upcaster: transforms V1 to V2 transparently
@Component
public class OrderPlacedEventUpcaster implements EventUpcaster {

    @Override
    public boolean canUpcast(EventRepresentation event) {
        return event.getType().equals("OrderPlacedEvent")
            && event.getVersion() == 1;
    }

    @Override
    public EventRepresentation upcast(EventRepresentation event) {
        JsonNode payload = objectMapper.readTree(event.getPayload());
        String address = payload.get("address").asText();

        // Split the single address field into two
        ObjectNode upgraded = objectMapper.createObjectNode();
        upgraded.put("orderId", payload.get("orderId").asText());
        upgraded.put("shippingAddress", address); // assume same address for historical data
        upgraded.put("billingAddress", address);
        upgraded.put("version", 2);

        return new EventRepresentation("OrderPlacedEvent", 2, upgraded.toString());
    }
}

// The aggregate and projection only see V2 events.
// V1 events are transparently upgraded by the upcaster chain.
// The event log is unchanged — V1 events remain as originally stored.
```

### 5.3 Event versioning strategy

```
Strategy 1: Versioned event types (most common)
  OrderPlacedEventV1, OrderPlacedEventV2 as separate classes.
  New publishers produce V2; old events remain V1 in the store.
  Pros: explicit; easy to see what version code expects.
  Cons: class proliferation over time.

Strategy 2: Version field in payload
  { "eventType": "OrderPlacedEvent", "version": 2, "payload": { ... } }
  Route to appropriate handler based on version field.
  Pros: single class; flexible routing.
  Cons: requires version-aware deserialisation.

Strategy 3: Avro / Protobuf schema versioning
  Schema Registry enforces compatibility rules.
  Producers and consumers registered against specific schema versions.
  Best for: high-volume Kafka-based event streams with many consumers.

Strategy 4: Event migration scripts
  Offline job rewrites old events in the event store to the new schema.
  WARNING: violates the immutability principle. Only acceptable for:
    (a) non-production environments, or (b) genuine data errors in events.
  Never do this in production without extreme care and a rollback plan.

Recommendation: use versioned event types (Strategy 1) for event-sourced
aggregates; use Avro/Schema Registry (Strategy 3) for integration events
published to Kafka consumed by multiple services.
```

## 6. Eventual Consistency Handling

### 6.1 Understanding eventual consistency

```
Eventual consistency: a consistency model in which, given no new updates,
all replicas of a data item will converge to the same value over time.

In event-driven systems:
  Command executes → event published → projection updated → read model reflects change
  This chain takes time: typically milliseconds to seconds.
  During that window: a read may return stale data.

Is this acceptable?
  User places order → immediately loads "My Orders" page → order not visible yet.
  This is the consistency gap. Whether it is acceptable depends on the use case.

  Acceptable:  "My Orders" list may take 1 second to reflect a new order
  Not acceptable: User pays for item; inventory immediately shows old stock level
                  to another user who then also purchases the same (now sold) item

The question is not "can we avoid eventual consistency?" but "what is the
acceptable consistency window for this specific use case, and have we
designed the system to meet it?"
```

### 6.2 Patterns for handling eventual consistency

```
Pattern 1: Optimistic UI update
  Client updates the UI immediately based on what the server should do,
  without waiting for the read model to catch up.
  If the command succeeds: the projection will eventually confirm.
  If the command fails: revert the optimistic update.
  Used by: Google Docs, Twitter likes, Facebook reactions.
  Best for: low-stakes updates where immediate visual feedback matters more
  than guaranteed accuracy.

Pattern 2: Read-your-writes consistency
  After a command, the response includes the event_id or version of the change.
  The next read includes this token, and the read model waits until it has
  processed at least up to that event_id before returning.
  Guarantees the user sees their own recent writes immediately.
  Cost: read may be slower if the projection is behind.

  Implementation:
    POST /orders → response: { orderId: "123", eventId: 9876 }
    GET /orders?minEventId=9876 → projection waits until processed event 9876

Pattern 3: Polling with timeout
  Client polls the read model at short intervals after a command,
  waiting for the expected state to appear.
  Simple; degrades gracefully if the projection falls behind.
  Used for: asynchronous job status, processing confirmations.

Pattern 4: Server-sent events / WebSocket push
  After a command, the server subscribes the client to updates.
  When the projection processes the event, it pushes the updated state
  to the client via WebSocket.
  Best for: real-time feedback on long-running operations.

Pattern 5: Accept the gap and communicate it
  Simply tell the user: "Your order has been placed. It may take a moment
  to appear in your order history."
  Underrated. Users accept small delays when they are communicated clearly.
  Works better than broken optimistic updates or complex consistency protocols.
```

### 6.3 Detecting and handling consistency violations

```
Some consistency violations are not acceptable even with short windows.
Design systems to detect and alert on these.

Invariant checker:
  A background job or Flink stream that reads from multiple projections
  and checks business invariants.
  "No order should have both status=SHIPPED and inventory_reservation=ACTIVE"
  Alert or compensate when invariants are violated.

Saga completion monitoring:
  Track saga states. Alert on sagas stuck in intermediate states.
  A saga that has been in INVENTORY_RESERVED for > 5 minutes without
  proceeding to payment is a consistency problem.

Reconciliation jobs:
  Nightly or hourly jobs that compare two eventually consistent views
  and produce a report of discrepancies.
  "Orders table says 1,432 confirmed orders; billing table shows 1,428 charged orders.
   4 orders may have been confirmed without payment."
  Alert triggers a human review process.

Outbox pattern for guaranteed event publishing:
  The source of many consistency violations is a command that succeeds
  but the event fails to publish (e.g. Kafka is unavailable).
  Solution: write the event to an outbox table within the same transaction
  as the state change; a separate publisher process reads the outbox and
  publishes to Kafka, retrying until success.

CREATE TABLE outbox (
    id          UUID            PRIMARY KEY,
    topic       VARCHAR(255)    NOT NULL,
    payload     JSONB           NOT NULL,
    created_at  TIMESTAMPTZ     NOT NULL,
    published_at TIMESTAMPTZ,                    -- NULL = not yet published
    retry_count INT             NOT NULL DEFAULT 0
);

// In the command handler transaction:
@Transactional
public void placeOrder(PlaceOrderCommand cmd) {
    Order order = Order.place(cmd.getOrderId(), cmd.getCustomerId(), cmd.getItems());
    orderRepository.save(order);

    // Write event to outbox in same transaction
    for (DomainEvent event : order.getUncommittedEvents()) {
        outboxRepository.save(Outbox.of("orders-events", event));
    }
    order.markEventsAsCommitted();
}

// Separate publisher (runs every 100ms):
@Scheduled(fixedDelay = 100)
public void publishOutboxEvents() {
    List<Outbox> pending = outboxRepository.findUnpublished(100);
    for (Outbox entry : pending) {
        kafkaTemplate.send(entry.getTopic(), entry.getPayload())
            .addCallback(
                success -> outboxRepository.markPublished(entry.getId()),
                failure -> outboxRepository.incrementRetry(entry.getId())
            );
    }
}
```

### 6.4 The Outbox pattern — why it matters

```
The two-generals problem in event publishing:
  Write to database → SUCCESS
  Publish to Kafka  → FAILURE (Kafka unavailable, network timeout, etc.)

Without Outbox:
  Database has the new state; Kafka has no event.
  Projections never see the change; downstream systems never react.
  Manual intervention required to replay.

With Outbox:
  Database write + outbox write in one ACID transaction.
  If the transaction commits: the event is in the outbox, guaranteed.
  The Kafka publish is a separate step that retries until success.
  At-least-once delivery guaranteed. Kafka consumer must be idempotent.

This is the foundational reliability pattern for event-driven systems.
Without it, every network hiccup between your database and Kafka creates
a silent consistency hole.
```

## 7. Putting It Together — End-to-End Event-Driven Order Flow

```
Complete flow: user places order → payment → inventory → shipping notification

1. API receives PlaceOrderCommand
   ↓
2. CreateOrderSaga starts
   ↓ (same transaction)
3. Order aggregate raises OrderPlacedEvent
   → Stored in events table
   → Written to outbox table
   ↓
4. Outbox publisher sends OrderPlacedEvent to Kafka "orders" topic
   ↓
5. Parallel consumers of "orders" topic:
   a. OrderSummaryProjection   → updates order_summaries read table
   b. CreateOrderSagaOrchestrator → issues ReserveInventoryCommand to inventory service
   c. AnalyticsProjection      → updates ClickHouse revenue data
   ↓
6. Inventory service processes ReserveInventoryCommand
   → InventoryReservedEvent published to "inventory" topic
   ↓
7. CreateOrderSagaOrchestrator consumes InventoryReservedEvent
   → Issues ChargeCardCommand to payment service
   ↓
8. Payment service processes ChargeCardCommand
   → PaymentProcessedEvent published to "payments" topic
   ↓
9. CreateOrderSagaOrchestrator consumes PaymentProcessedEvent
   → Issues ConfirmOrderCommand to order service
   ↓
10. Order aggregate processes ConfirmOrderCommand
    → Raises OrderConfirmedEvent
    → Written to events table + outbox
    ↓
11. OrderConfirmedEvent consumers:
    a. OrderSummaryProjection  → updates status to CONFIRMED
    b. NotificationService     → sends order confirmation email
    c. ShippingService         → schedules shipment

Failure at step 7 (payment fails):
    → PaymentFailedEvent published
    → Orchestrator issues ReleaseInventoryCommand (compensation)
    → InventoryReleasedEvent published
    → Orchestrator issues CancelOrderCommand
    → OrderCancelledEvent published
    → Notification service sends cancellation email
```

## 8. Interview Questions

**Q: What is the difference between event sourcing and event-driven architecture? Can you have one without the other?**

A: Event-driven architecture is an integration style where components communicate by publishing and subscribing to events. Services are decoupled because a producer does not know who will consume its events. Event sourcing is a data persistence pattern where an aggregate's state is stored as a sequence of events rather than as a current-state snapshot. They are independent concepts. You can have event-driven architecture without event sourcing — services publish integration events after writing to a traditional database, and other services consume those events. This is the most common deployment. You can also have event sourcing without event-driven architecture — a single service that stores its state as events internally but does not publish those events to other services. The two patterns work naturally together because event sourcing produces domain events as a natural output of the write path, which can then be published to an event bus for consumption by other services. However, the additional complexity of event sourcing — schema evolution, snapshot management, projection rebuilds — is only justified when its specific benefits (complete audit trail, temporal queries, event replay) are actually needed by the system. EDA's benefits (loose coupling, independent scaling) can be achieved without event sourcing.

---

**Q: How do you handle an event that a projection has processed incorrectly due to a bug?**

A: An incorrect projection is one of the most common operational issues in CQRS systems, and the standard recovery process exploits the immutability of the event log. First, fix the bug in the projection code. Second, decide on the rebuild strategy based on urgency. For non-urgent rebuilds: stop the projection consumer, truncate the read model table, create a new consumer group starting from offset zero on the Kafka topic or from event_id zero in the event store, replay all events through the fixed code, and switch queries to the new model once caught up. For zero-downtime rebuilds: run the old and new projections in parallel using different table names, route a small percentage of read traffic to the new projection to validate correctness, then flip all read traffic to the new projection when validation passes. The rebuild can take time for large event logs — this is where snapshots or Kafka topic compaction reduce the time to rebuild by allowing projection replay from a recent checkpoint rather than the beginning of history. The key architectural insight is that the correctness of the read model is temporary — it can always be rebuilt from the event log — while the event log itself is permanent and must be treated accordingly.

---

**Q: Explain the outbox pattern and why it is necessary for reliable event publishing.**

A: The outbox pattern solves the fundamental two-phase commit problem between a database write and a message broker publish. Without it, a service updates its database and then tries to publish an event to Kafka. If Kafka is unavailable, or the network times out between the database commit and the Kafka send, the database has the new state but no event was published. Downstream systems — projections, sagas, other services — never see the change. This creates a permanent consistency hole that requires manual intervention. The outbox pattern eliminates this by writing the event to an outbox table within the same database transaction as the state change. Since both the state change and the outbox entry are in the same ACID transaction, they either both commit or both roll back — no partial state is possible. A separate publisher process then reads unpublished entries from the outbox and sends them to Kafka, retrying with backoff until the publish succeeds and the entry is marked published. This guarantees at-least-once delivery to Kafka — the Kafka consumer must handle duplicates, which it should anyway. Change Data Capture tools like Debezium offer an alternative by tailing the database WAL to detect outbox entries without requiring a separate publisher process, but the principle is the same: treat event publishing as a durable, retryable operation rather than a best-effort fire-and-forget.

---

**Q: How do you design for eventual consistency in a user-facing feature where the user expects to see their changes immediately?**

A: The first design question is whether strong consistency is actually required for correctness, or just for user experience. These require different solutions. For user experience only — the user wants to see the order they just placed — the cleanest solution is the read-your-writes pattern combined with an optimistic UI update. The API response for a command includes the event ID or version number that represents the change. The client optimistically updates the UI immediately without waiting for the projection to catch up. If the command succeeded, the projection will catch up within milliseconds and confirm the optimistic state. If the command failed, the UI reverts to the pre-command state based on the error response. The event ID returned in the command response can be used to implement server-side read-your-writes: the read endpoint accepts a minimum event ID parameter and waits, with a short timeout, for the projection to process at least up to that event before returning. For correctness requirements — where two users must not both be able to purchase the last item in stock — eventual consistency is genuinely not acceptable, and the design must use either pessimistic locking, optimistic concurrency control at the command side, or a reservation pattern where inventory is reserved atomically as part of accepting the order command, before the event is published.

---

**Q: When would you choose choreography over orchestration for a saga, and vice versa?**

A: Choreography and orchestration are both valid coordination styles, and the choice primarily depends on the complexity of the flow and the importance of having the workflow logic in one visible place. I choose choreography when the flow is short and linear — three or fewer steps, no conditional branches, each step has a clear success or failure response — and when decoupling between services is the primary concern. With choreography, each service only knows about its own events and the events it needs to react to; adding a new step means adding a new consumer of an existing event, which does not require changing any existing service. The main operational risk is that the full workflow is implicit — it only exists as the collective behaviour of all the consumers, making it hard to debug or reason about from a single vantage point. I choose orchestration when the flow has more than three steps, conditional logic (different paths depending on payment method, customer type, or item availability), parallel steps that need to complete before proceeding, or complex compensation logic where different failures require different rollback sequences. Orchestration makes all of this explicit in one place — the saga coordinator — which is also the place where timeout handling, retry logic, and compensation state transitions are managed. The trade-off is that the orchestrator becomes aware of all the services it coordinates, creating more coupling than choreography. In practice, most production systems I have worked on use orchestration for business-critical sagas because the explicit visibility into saga state is worth the coupling cost — especially when debugging a stuck saga at 2 AM.

## 9. Key Takeaways

- **Events are facts, commands are requests**: events use past tense and cannot be rejected; commands are imperative and may fail — this distinction shapes your API design and schema choices
- **Event storming produces bounded contexts**: the natural clusters of aggregates, events, and policies that emerge from event storming are the best candidates for service boundaries — more reliable than technical layer analysis
- **Event sourcing gives you time travel at the cost of query complexity**: the event log is the audit trail, the history, and the truth; current state is a derived view — worth the complexity only when audit, temporal queries, or event replay are genuinely needed
- **CQRS separates scaling and model complexity**: read models can be denormalised, cached, and rebuilt without touching the command side; multiple projections from the same event stream serve different query needs without coupling
- **Outbox pattern is the foundation of reliable event publishing**: database write + event write in one ACID transaction; separate publisher retries until Kafka confirms; without it, every network hiccup creates a silent consistency hole
- **Sagas replace ACID with eventual consistency + compensating transactions**: choreography for simple linear flows; orchestration for complex flows with branches and timeouts; every step must be idempotent because retries are guaranteed
- **Events are immutable; upcasters upgrade their interpretation**: never modify stored events; write upcasters to translate old schemas to new ones transparently on read
- **Schema Registry with FULL compatibility enforces safe evolution**: additive changes only; nullable new fields with defaults; renaming is a remove + add = breaking change
- **Eventual consistency windows must be designed for each use case**: optimistic UI update for UX-only consistency; read-your-writes for user's own changes; reconciliation jobs for detecting business invariant violations
- **Projection rebuilds are the recovery mechanism**: the event log is the source of truth; projections are disposable views that can always be rebuilt — this makes event-sourced systems uniquely resilient to bugs in read-side logic
