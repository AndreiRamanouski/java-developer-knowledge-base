# API Design Principles

## Overview

An API is a contract between a producer and all its consumers. Bad API design is expensive in a specific way: every consumer builds around the API as designed, meaning a breaking change requires coordinating changes across every consumer simultaneously. Good API design is therefore less about elegance and more about designing contracts that are stable, explicit about failure modes, consistent in their patterns, and evolvable without breaking consumers.

**The cost of API design decisions:**

```
An API decision made in week one affects every consumer forever.

Cheap to change:   internal services with one consumer (refactor freely)
Medium cost:       internal API with a handful of services (coordinate deploys)
Expensive:         external API used by partners (months of deprecation)
Very expensive:    public API used by millions of developers (essentially permanent)

Design to the permanence of your audience.
A public REST API should be designed as if every field name and URL structure
is a commitment you will honour for five years. That discipline produces
better designs even for internal APIs.
```

## 1. RESTful Principles

### 1.1 REST constraints

```
REST (Representational State Transfer) is not a protocol — it is an architectural
style defined by six constraints. Understanding them explains why REST APIs
work the way they do.

1. Client-Server:
   Separate concerns. The client handles UI; the server handles data and logic.
   Neither depends on the other's implementation details.

2. Stateless:
   Each request must contain all information needed to process it.
   The server stores no session state between requests.
   Authentication state travels with the request (Authorization header).
   Implication: scale horizontally trivially (any server can handle any request).

3. Cacheable:
   Responses must declare whether they are cacheable.
   GET responses with Cache-Control headers enable CDN and client caching.
   Non-cacheable responses (POST, PUT) must say so explicitly.

4. Uniform Interface:
   The unifying constraint. Four sub-constraints:
   (a) Resource identification in requests (URIs identify resources)
   (b) Resource manipulation through representations (JSON/XML)
   (c) Self-descriptive messages (Content-Type, Accept headers)
   (d) HATEOAS (links in responses to navigate to related resources)
       In practice: HATEOAS is rarely implemented fully; most APIs skip it.

5. Layered System:
   Client cannot tell if it is connected to the origin server or an intermediary.
   Enables: CDN, API gateways, load balancers — all transparent to clients.

6. Code on Demand (optional):
   Server can extend client functionality by sending executable code.
   Rarely used in practice (JavaScript delivery was the original use case).
```

### 1.2 Resource design

```
Resources are nouns, not verbs. URLs identify resources; HTTP methods express actions.

WRONG (RPC-style URLs):
  POST /createOrder
  GET  /getOrderById?id=123
  POST /cancelOrder
  POST /getUserOrders

RIGHT (resource-oriented URLs):
  POST   /orders              Create an order
  GET    /orders/123          Get order 123
  PATCH  /orders/123          Update order 123 partially
  DELETE /orders/123          Cancel / delete order 123
  GET    /users/456/orders    Get orders for user 456

Hierarchy rules:
  Use hierarchical paths for genuinely nested resources.
  /users/{userId}/orders        → orders owned by a user (correct hierarchy)
  /orders/{orderId}/items       → items within an order (correct)
  /orders/{orderId}/shipments   → shipments for an order (correct)

  Do not create deep nesting just because there is a relationship.
  /users/{userId}/orders/{orderId}/items/{itemId}/reviews
  → flatten: /reviews?orderId=X&itemId=Y is often clearer

Singleton vs collection:
  /orders           → collection (GET returns list, POST creates new)
  /orders/{id}      → singleton (GET one, PUT replace, PATCH update, DELETE remove)
  /settings         → singleton without ID (user's settings; there is only one)
```

### 1.3 HTTP method semantics

```
Method     Safe?  Idempotent?  Body?  Typical use
──────────────────────────────────────────────────────────────────────
GET        Yes    Yes          No     Read a resource or collection
HEAD       Yes    Yes          No     Get headers only (existence check)
OPTIONS    Yes    Yes          No     CORS preflight, capability query
POST       No     No           Yes    Create a resource or trigger action
PUT        No     Yes          Yes    Replace a resource entirely
PATCH      No     No*          Yes    Update a resource partially
DELETE     No     Yes          No     Remove a resource

Safe: does not change server state (safe to retry, cache, prefetch)
Idempotent: calling N times = calling once (safe to retry on network failure)

* PATCH idempotency: depends on the patch format.
  JSON Patch (RFC 6902) with explicit operations: idempotent.
  Partial JSON body (common shorthand): usually not idempotent if updates are
  position-sensitive (e.g. appending to an array).

Idempotency matters for retry logic:
  GET, PUT, DELETE: client can retry safely on timeout.
  POST: retry may create duplicates → use Idempotency-Key header.
  PATCH: depends on the operation — treat as non-idempotent by default.
```

### 1.4 HTTP status codes — the right ones to use

```
2xx Success:
  200 OK:           Standard success. Use for GET, PUT, PATCH responses with body.
  201 Created:      Resource created. Include Location header with new resource URL.
  202 Accepted:     Request accepted for async processing (job queued; not yet done).
  204 No Content:   Success with no response body. Use for DELETE, some PATCHes.

3xx Redirection:
  301 Moved Permanently: resource URL has changed; update your bookmark.
  302 Found:             temporary redirect (common for OAuth flows).
  304 Not Modified:      cached version is still valid (ETag/If-None-Match).

4xx Client Errors:
  400 Bad Request:       Malformed request syntax, invalid parameters.
  401 Unauthorized:      Authentication required or credentials invalid.
  403 Forbidden:         Authenticated but not permitted (different from 401).
  404 Not Found:         Resource does not exist (or client should not know it exists).
  405 Method Not Allowed: HTTP method not supported for this endpoint.
  409 Conflict:          Conflict with current state (duplicate key, optimistic lock).
  410 Gone:              Resource existed but was permanently deleted.
  422 Unprocessable:     Syntactically valid but semantically invalid (validation errors).
  429 Too Many Requests: Rate limit exceeded.

5xx Server Errors:
  500 Internal Server Error: Unexpected server failure.
  502 Bad Gateway:           Upstream service returned invalid response.
  503 Service Unavailable:   Server temporarily unavailable (overloaded, maintenance).
  504 Gateway Timeout:       Upstream service timed out.

Common mistakes:
  Using 200 for errors (body contains "error: true") → breaks HTTP tooling
  Using 401 for "you don't have permission" (should be 403)
  Using 500 for validation errors (should be 400/422)
  Using 404 for "you're not authorised to see this" (leaks existence; use 403 or 404)
```

### 1.5 Naming conventions

```
URL naming:
  Lowercase, hyphenated: /order-items not /OrderItems or /order_items
  Plural nouns for collections: /orders not /order
  Singular for singleton resources: /users/123/profile (one profile per user)
  No file extensions: /orders/123 not /orders/123.json

Field naming:
  camelCase for JSON: { "orderId", "createdAt", "totalAmount" }
  ISO 8601 for dates: "2024-03-15T10:30:00Z" not Unix timestamps in public APIs
  null vs absent: use null for "exists but has no value"; omit for "does not apply"
  Booleans: is_active, has_discount, can_cancel — verb prefix helps readability

Consistency rules (more important than which convention):
  Pick one naming style and apply it everywhere.
  "createdAt" and "created_at" in the same API is an inexcusable inconsistency.
  Establish a style guide and enforce it in API review.
```

## 2. API Versioning Strategies

### 2.1 Why versioning is necessary

```
APIs must evolve. Business requirements change. Fields are renamed. New authentication
mechanisms replace old ones. Data types change.

Without versioning: any change that breaks existing consumers is forbidden.
The API calcifies. Technical debt accumulates. Consumers never update.

With versioning: breaking changes are possible with advance notice and a migration period.
Consumers can upgrade at their own pace. Old versions are deprecated and eventually removed.

Types of changes:
  Non-breaking (backward compatible) — safe to make any time:
    Adding new optional fields to responses
    Adding new optional query parameters
    Adding new endpoints
    Adding new enum values (if consumers handle unknown values gracefully)
    Relaxing validation rules (accepting more input)

  Breaking — require a new version:
    Removing or renaming fields
    Changing field types (string → integer)
    Changing URL structure
    Adding required fields to requests
    Changing authentication mechanism
    Removing endpoints
    Changing error response format
```

### 2.2 Versioning strategies compared

```
Strategy 1: URI path versioning (most common)
  /api/v1/orders
  /api/v2/orders

  Pros: visible in URL (easy to test, bookmark, document, log)
  Cons: "v2 of orders" vs "v2 of everything" — version scope ambiguity
  Best for: external/public APIs where discoverability matters

Strategy 2: HTTP header versioning
  GET /api/orders
  Headers: API-Version: 2 (custom) or Accept: application/vnd.myapp.v2+json

  Pros: clean URLs; REST-correct (version is metadata, not resource identity)
  Cons: invisible in browser; harder to test; caches must vary on the header
  Best for: internal APIs where all consumers control their headers

Strategy 3: Query parameter
  GET /api/orders?version=2

  Pros: easy to test in browser
  Cons: pollutes query parameter space; uncommon; may conflict with caching
  Best for: rarely used; avoid unless URL versioning is not possible

Strategy 4: Content negotiation (most REST-correct)
  Accept: application/vnd.myapp.orders.v2+json
  Server varies response format based on Accept header.

  Pros: truly REST-correct; fine-grained per-resource versioning
  Cons: verbose; difficult for clients to implement; caching complexity
  Best for: rarely used in practice despite being most correct

Recommendation:
  External/public APIs: URI path versioning (/v1, /v2)
  Internal service-to-service: header versioning
  Never mix strategies within the same API
```

### 2.3 Version lifecycle management

```
Version lifecycle:
  Active:      fully supported; new features may be added
  Deprecated:  sunset date announced; no new features; security fixes only
  Sunset:      no longer functional; requests return 410 Gone with migration guidance

Deprecation policy for external APIs:
  Minimum 12 months notice for breaking changes
  Deprecation notice in response headers from day one: Sunset: Sat, 01 Jan 2026 00:00:00 GMT
  Communication: email to registered developers, changelog, documentation
  Monitoring: track which API keys are still calling deprecated versions

Response headers for deprecation:
  Deprecation: true
  Sunset: Sat, 01 Jan 2026 00:00:00 GMT
  Link: <https://docs.api.com/migration/v1-to-v2>; rel="deprecation"

// Spring Boot: add deprecation headers via filter
@Component
public class DeprecationHeaderFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(HttpServletRequest req,
                                     HttpServletResponse res,
                                     FilterChain chain) throws ... {
        if (req.getRequestURI().startsWith("/api/v1/")) {
            res.addHeader("Deprecation", "true");
            res.addHeader("Sunset", "Sat, 01 Jan 2026 00:00:00 GMT");
            res.addHeader("Link",
                "<https://docs.api.com/migration/v1-to-v2>; rel=\"deprecation\"");
        }
        chain.doFilter(req, res);
    }
}
```

## 3. Pagination and Filtering

### 3.1 Pagination strategies

```
Problem: returning all records for a large collection in one response is infeasible.

Strategy 1: Offset pagination (most common, not recommended for large datasets)
  GET /orders?page=3&pageSize=20
  SELECT * FROM orders ORDER BY created_at DESC LIMIT 20 OFFSET 60

  Pros: random access (jump to page 5); simple to implement
  Cons:
    Page drift: records inserted while browsing shift subsequent pages
      User reads page 1 (records 1-20); a new record is inserted; reading page 2
      (records 21-40) skips the record that was pushed to position 21.
    Expensive for large offsets: SELECT ... OFFSET 100000 scans 100K rows to discard them
    Not suitable for real-time data or large datasets

Strategy 2: Cursor pagination (recommended for feeds and large datasets)
  GET /orders?cursor=eyJpZCI6MTIzfQ&pageSize=20
  cursor = opaque encoded reference to a position in the result set

  Implementation with Snowflake-ordered IDs:
    cursor = base64(lastSeenId)
    SELECT * FROM orders WHERE id < :cursor ORDER BY id DESC LIMIT 20

  Pros:
    Stable: new records inserted never shift existing pages
    Efficient: O(log n) index lookup, no full scan
    Works for infinite scroll and real-time data
  Cons:
    No random access (cannot jump to page 5)
    cursor is opaque to clients (cannot manually construct)

Strategy 3: Keyset pagination (explicit form of cursor pagination)
  GET /orders?after_id=12345&pageSize=20
  Exposes the key directly instead of an opaque cursor.
  Pros: transparent; debuggable
  Cons: leaks internal IDs; ID must be globally stable (not reassignable)

Standard pagination response shape:
{
  "data": [...],
  "pagination": {
    "nextCursor": "eyJpZCI6MTIzfQ",
    "hasNextPage": true,
    "pageSize": 20,
    "totalCount": 847  // optional — expensive to compute; include only if needed
  }
}
```

### 3.2 Cursor pagination implementation

```java
@GetMapping("/orders")
public PagedResponse<OrderDto> getOrders(
        @RequestParam(required = false) String cursor,
        @RequestParam(defaultValue = "20") @Max(100) int pageSize,
        @RequestParam(required = false) String status,
        Authentication auth) {

    Long cursorId = cursor != null ? decodeCursor(cursor) : Long.MAX_VALUE;

    List<Order> orders = orderRepository.findPage(
        auth.getUserId(), cursorId, pageSize + 1, status  // fetch N+1 to detect hasNext
    );

    boolean hasNextPage = orders.size() > pageSize;
    if (hasNextPage) orders = orders.subList(0, pageSize); // trim to page size

    String nextCursor = hasNextPage
        ? encodeCursor(orders.get(orders.size() - 1).getId())
        : null;

    return PagedResponse.<OrderDto>builder()
        .data(orders.stream().map(orderMapper::toDto).collect(toList()))
        .nextCursor(nextCursor)
        .hasNextPage(hasNextPage)
        .pageSize(pageSize)
        .build();
}

// Efficient keyset query: O(log n) using the ID index
// No OFFSET — the cursor IS the position in the result set
@Query("SELECT o FROM Order o WHERE o.userId = :userId AND o.id < :cursorId " +
       "AND (:status IS NULL OR o.status = :status) " +
       "ORDER BY o.id DESC")
List<Order> findPage(@Param("userId") Long userId,
                     @Param("cursorId") Long cursorId,
                     Pageable pageable,
                     @Param("status") String status);
```

### 3.3 Filtering, sorting, and field selection

```
Filtering:
  Simple: GET /orders?status=PLACED&currency=USD
  Range: GET /orders?createdAfter=2024-01-01&createdBefore=2024-03-31
  Multi-value: GET /orders?status=PLACED&status=CONFIRMED  (repeated param)
               or: GET /orders?status[]=PLACED&status[]=CONFIRMED
  Do NOT support arbitrary query expressions (security risk; SQL injection surface)

Sorting:
  GET /orders?sort=createdAt&direction=desc
  Multiple sort fields: GET /orders?sort=status,createdAt&direction=asc,desc
  Whitelist sortable fields — do not allow sorting on any column (leaks schema)

Sparse fieldsets (field selection):
  GET /orders?fields=orderId,status,totalAmount
  Returns only requested fields — reduces response size; useful for mobile clients.
  Implementation: project only requested fields in the query.
  Particularly valuable when a collection returns 50 objects with 30 fields each
  and the client only needs 3 fields.

Search:
  Simple: GET /products?q=wireless+headphones
  Structured: GET /products?name=headphones&minPrice=50&maxPrice=200
  Full-text search belongs in a dedicated search service (Elasticsearch),
  not as a database LIKE query on the primary store.
```

## 4. Error Response Design

### 4.1 Why error design matters

```
An API's error responses are part of its contract.
Consumers write code that handles errors. Inconsistent error formats mean
every error case requires special-casing.

A good error response tells the consumer:
  (a) What went wrong (machine-readable code)
  (b) Why it went wrong (human-readable message)
  (c) Where it went wrong (which field, which resource)
  (d) What to do next (how to fix it, or where to read more)

The most common mistake: returning only a human-readable message.
"Invalid email address" requires string matching to handle programmatically.
"INVALID_EMAIL_FORMAT" is a stable code the client can branch on.
```

### 4.2 Error response format

```json
// Single error
{
  "error": {
    "code": "INVALID_EMAIL_FORMAT",
    "message": "The provided email address is not a valid format.",
    "field": "email",
    "requestId": "req_abc123",
    "documentationUrl": "https://docs.api.com/errors/INVALID_EMAIL_FORMAT"
  }
}

// Multiple validation errors (422)
{
  "error": {
    "code": "VALIDATION_FAILED",
    "message": "One or more fields failed validation.",
    "requestId": "req_abc123",
    "details": [
      {
        "code": "REQUIRED_FIELD_MISSING",
        "field": "email",
        "message": "email is required"
      },
      {
        "code": "VALUE_OUT_OF_RANGE",
        "field": "quantity",
        "message": "quantity must be between 1 and 1000",
        "received": 0,
        "min": 1,
        "max": 1000
      }
    ]
  }
}

// Structured error fields:
//   code:             machine-readable string constant (stable across versions)
//   message:          human-readable description (may change; do not parse)
//   field:            which field caused the error (for validation errors)
//   requestId:        for correlation with server logs (essential for support)
//   documentationUrl: link to error reference (encourages self-service)
//   details:          list of sub-errors (for validation failures)
```

### 4.3 Error code taxonomy

```
Error codes should be in a consistent namespace and format.

Format options:
  SCREAMING_SNAKE_CASE string: "INVALID_EMAIL_FORMAT" (recommended)
  Dot-notation:                "auth.token.expired"
  Integer codes:               40001 (like HTTP status extended codes)

Recommended taxonomy:
  Authentication errors:    AUTH_TOKEN_MISSING, AUTH_TOKEN_EXPIRED,
                            AUTH_TOKEN_INVALID, AUTH_CREDENTIALS_INVALID
  Authorisation errors:     PERMISSION_DENIED, RESOURCE_ACCESS_FORBIDDEN
  Validation errors:        REQUIRED_FIELD_MISSING, INVALID_FORMAT,
                            VALUE_OUT_OF_RANGE, DUPLICATE_VALUE
  Resource errors:          RESOURCE_NOT_FOUND, RESOURCE_ALREADY_EXISTS,
                            RESOURCE_DELETED, RESOURCE_LOCKED
  Rate limit errors:        RATE_LIMIT_EXCEEDED, QUOTA_EXCEEDED
  Server errors:            INTERNAL_ERROR, SERVICE_UNAVAILABLE,
                            DEPENDENCY_TIMEOUT, DATABASE_ERROR (avoid leaking details)

DO NOT include in error responses:
  Stack traces (security risk; implementation leakage)
  Database error messages (SQL syntax errors reveal schema)
  Internal service names (leaks topology)
  Raw exception messages (inconsistent; leaks implementation)
```

### 4.4 Problem Details (RFC 7807)

```
RFC 7807 defines a standard JSON error format for HTTP APIs.
Using it enables tooling support and sets client expectations.

Content-Type: application/problem+json

{
  "type": "https://docs.api.com/problems/out-of-credit",
  "title": "You do not have enough credit.",
  "status": 403,
  "detail": "Your current balance is 30, but that costs 50.",
  "instance": "/account/12345/transactions/abc123",
  "balance": 30,      // extension fields: domain-specific context
  "required": 50
}

Fields:
  type:     URI identifying the error type (links to documentation)
  title:    short human-readable summary (stable; same for same error type)
  status:   HTTP status code (redundant but helpful for logging)
  detail:   human-readable explanation of this specific occurrence
  instance: URI identifying this specific error occurrence

Spring Boot implementation:
// Spring 6 / Boot 3: supports RFC 7807 ProblemDetail natively
@ExceptionHandler(InsufficientCreditException.class)
public ResponseEntity<ProblemDetail> handleInsufficientCredit(
        InsufficientCreditException ex) {
    ProblemDetail problem = ProblemDetail.forStatusAndDetail(
        HttpStatus.FORBIDDEN,
        "Your current balance is " + ex.getBalance() + ", but that costs " + ex.getRequired()
    );
    problem.setType(URI.create("https://docs.api.com/problems/out-of-credit"));
    problem.setTitle("Insufficient Credit");
    problem.setProperty("balance",  ex.getBalance());
    problem.setProperty("required", ex.getRequired());
    return ResponseEntity.status(HttpStatus.FORBIDDEN).body(problem);
}
```

## 5. Rate Limiting (API Design Perspective)

### 5.1 Rate limit response headers

```
Rate limiting must be communicated to clients so they can self-throttle.
Three standard headers (widely adopted though not formally standardised):

X-RateLimit-Limit:     The maximum number of requests allowed in the window.
X-RateLimit-Remaining: The number of requests remaining in the current window.
X-RateLimit-Reset:     Unix timestamp when the window resets (remaining = limit).

// Always include these headers, even for non-throttled responses
// Clients use them to implement proactive backoff before hitting the limit

When limit is exceeded: return 429 Too Many Requests
{
  "error": {
    "code": "RATE_LIMIT_EXCEEDED",
    "message": "API rate limit exceeded. Retry after 47 seconds.",
    "retryAfter": 47,
    "limit": 1000,
    "window": "1 minute",
    "documentationUrl": "https://docs.api.com/rate-limits"
  }
}

Headers on 429:
  Retry-After: 47              (seconds until client may retry)
  X-RateLimit-Limit: 1000
  X-RateLimit-Remaining: 0
  X-RateLimit-Reset: 1710000060

// Always include Retry-After — clients that respect it reduce retry storms
```

### 5.2 Idempotency keys for safe retries

```
Problem: POST requests are not idempotent by default.
Client sends POST /orders → network timeout → client retries → duplicate order created.

Solution: Idempotency-Key header (Stripe pioneered this pattern).
  Client generates a unique key (UUID) per request intention.
  Sends the same key on retries.
  Server stores the result keyed by idempotency key.
  On retry: returns the stored result without re-executing.

Request:
  POST /orders
  Idempotency-Key: 7a6f8b9c-1234-5678-abcd-ef1234567890
  Content-Type: application/json
  { "items": [...], "total": 99.99 }

Server behaviour:
  1. Check cache: has this Idempotency-Key been seen before?
  2. Yes: return cached response (same status, same body)
  3. No: execute the request, cache the result, return it

Cache TTL: 24 hours (client must use same key within this window for retry benefit)
Key scope: per user per endpoint (key "abc" for user A ≠ key "abc" for user B)

Response header: Idempotency-Key: 7a6f8b9c-1234-5678-abcd-ef1234567890 (echoed back)
Replayed response header: Idempotent-Replayed: true (signals this was a cached response)
```

## 6. API Security: OAuth2 and JWT

### 6.1 OAuth2 flow overview

```
OAuth2: authorisation framework (not authentication).
  Allows a user to grant an application access to their resources
  on another service, without sharing their credentials.

Four grant types:
  1. Authorisation Code (with PKCE for public clients) — for web/mobile apps
     Most secure; server-to-server token exchange.
  2. Client Credentials — for machine-to-machine (service accounts)
     No user involved; app authenticates with client_id + client_secret.
  3. Implicit — deprecated (insecure; replaced by Auth Code + PKCE)
  4. Device Code — for smart TVs, CLI tools with no browser

For user-facing APIs: Authorisation Code + PKCE
For service-to-service APIs: Client Credentials

Authorisation Code + PKCE flow:
  1. Client generates code_verifier (random string) + code_challenge (hash of verifier)
  2. Client redirects user to: /authorize?
       response_type=code&client_id=X&redirect_uri=Y&
       code_challenge=Z&code_challenge_method=S256&scope=read:orders
  3. User authenticates and approves scopes at authorisation server
  4. Authorisation server redirects to redirect_uri?code=ABC
  5. Client exchanges code for tokens:
       POST /token { code: ABC, code_verifier: V, client_id: X, redirect_uri: Y }
  6. Auth server validates code_verifier against stored code_challenge
  7. Returns: access_token, refresh_token, expires_in

Why PKCE: prevents authorisation code interception attacks.
  The code alone is useless without the code_verifier, which never travels over the network.
```

### 6.2 JWT structure and validation

```
JWT (JSON Web Token): a compact, self-contained token format.
Three parts: header.payload.signature (Base64Url encoded, dot-separated)

Header:
  { "alg": "RS256", "typ": "JWT", "kid": "key-id-for-rotation" }
  alg: signing algorithm (RS256 = RSA with SHA-256; HS256 = HMAC — avoid for APIs)
  kid: key ID — enables key rotation without breaking existing tokens

Payload (claims):
  {
    "sub": "user_12345",         // subject: the user or entity
    "iss": "https://auth.api.com", // issuer: the auth server
    "aud": ["https://api.com"],  // audience: who should accept this token
    "exp": 1710000060,           // expiry: Unix timestamp
    "iat": 1709996460,           // issued at: Unix timestamp
    "jti": "unique-token-id",   // JWT ID: for revocation tracking
    "scope": "read:orders write:orders",
    "roles": ["customer"]
  }

Signature: RSASHA256(base64(header) + "." + base64(payload), privateKey)

JWT validation steps (all must pass):
  1. Verify signature using the public key (keyed by kid from header)
  2. Verify exp > now (token not expired)
  3. Verify iss matches expected issuer
  4. Verify aud contains this API's identifier
  5. Verify required scopes are present for the requested operation
  6. (Optional) check jti against revocation list

Why RS256 over HS256:
  HS256: symmetric key; the same key signs and verifies.
         Every service that verifies tokens must know the secret.
         Compromise of any service = compromise of all tokens.
  RS256: asymmetric; private key signs, public key verifies.
         Services only need the public key to verify.
         Public key is safe to distribute; private key never leaves the auth server.
```

### 6.3 Scope design

```
Scopes define what an access token is permitted to do.
Good scope design is granular enough for least-privilege but not so granular
it becomes a burden on developers.

Format conventions:
  resource:action    →  orders:read, orders:write, payments:process
  Avoid:             →  full_access, admin (too broad)

Scope taxonomy example:
  orders:read         Read order history and status
  orders:write        Create and modify orders
  orders:cancel       Cancel an order (separate from write — more sensitive)
  payments:read       View payment history
  payments:process    Initiate payments
  users:profile:read  Read own profile
  users:profile:write Update own profile
  admin:*             Administrative access (restricted to internal services)

Scope validation in Spring Security:
@GetMapping("/orders/{id}")
@PreAuthorize("hasAuthority('SCOPE_orders:read')")
public OrderDto getOrder(@PathVariable String id) { ... }

@PostMapping("/orders")
@PreAuthorize("hasAuthority('SCOPE_orders:write')")
public OrderDto createOrder(@RequestBody CreateOrderRequest req) { ... }

@PostMapping("/orders/{id}/cancel")
@PreAuthorize("hasAnyAuthority('SCOPE_orders:cancel', 'SCOPE_admin:orders')")
public void cancelOrder(@PathVariable String id) { ... }
```

### 6.4 API key management (for service-to-service)

```
API keys: simpler than OAuth2 for server-to-server integration.
Not appropriate for user-facing apps (no delegation, no expiry).

Best practices:
  Never log API keys (even partially) — use request_id for log correlation
  Hash API keys in the database (SHA-256; same as password hashing principle)
  Include a prefix for easy identification: sk_live_abc123, pk_test_xyz789
  Separate live and test key spaces (prefix or separate namespace)
  Associate keys with a service account, not a person (keys survive personnel changes)
  Rotate keys with zero downtime: allow new + old key during rotation window
  Rate limit per key; revoke suspicious keys immediately

Key lookup optimisation:
  Full SHA-256 hash in DB: O(log n) lookup on indexed hash
  Prefix + hash: prefix (e.g. first 8 chars) narrows candidates before hash comparison
  Redis cache: cache key → service_account for hot path; TTL = 5 minutes
```

## 7. OpenAPI Documentation

### 7.1 OpenAPI specification overview

```
OpenAPI (formerly Swagger): the standard for describing REST APIs in machine-readable format.
  Generates documentation, client SDKs, server stubs, and validation.

File format: YAML or JSON (YAML is more human-readable).

Key sections:
  info:         API title, version, description, contact
  servers:      base URLs (production, staging, local)
  paths:        endpoint definitions (method, parameters, request body, responses)
  components:   reusable schemas, parameters, responses, security schemes
  security:     global security requirements

OpenAPI first vs code first:
  OpenAPI first: write the spec, generate stubs, implement.
    Pros: spec is the single source of truth; consumer agreement before implementation.
    Cons: spec drift (implementation may deviate); extra step in development.
    Best for: public APIs, team collaboration, consumer-first design.

  Code first: write the code, generate the spec from annotations.
    Pros: spec always reflects implementation; no drift.
    Cons: spec quality depends on annotation quality; less consumer focus upfront.
    Best for: internal APIs, rapidly iterating systems.

  Recommendation: OpenAPI first for external APIs; code first for internal.
```

## 8. API Design Checklist

```
Resource design:
  ✓ Resources are nouns, not verbs (/orders not /createOrder)
  ✓ Collections are plural (/orders), singletons are singular (/profile)
  ✓ Hierarchy reflects ownership, not joins (/users/123/orders)
  ✓ Maximum 3 levels of nesting (deeper = flatten with query params)

HTTP methods:
  ✓ GET is safe and idempotent; no side effects
  ✓ POST used for creation and non-idempotent actions only
  ✓ PUT replaces entirely; PATCH updates partially
  ✓ DELETE is idempotent (404 on repeat is acceptable)
  ✓ Idempotency-Key header offered for POST endpoints

Status codes:
  ✓ 201 + Location header on successful POST creation
  ✓ 204 on DELETE / side-effect-free PATCH
  ✓ 401 vs 403 used correctly
  ✓ 422 for semantic validation errors (not 400)
  ✓ 429 with Retry-After for rate limits

Responses:
  ✓ Consistent camelCase field naming throughout
  ✓ ISO 8601 timestamps everywhere
  ✓ Errors have machine-readable code + request_id + optional field
  ✓ Pagination uses cursor-based approach for large collections
  ✓ Rate limit headers on every response

Security:
  ✓ All endpoints require authentication (explicit exception for public)
  ✓ Scopes are least-privilege and documented
  ✓ JWT validated: signature, exp, iss, aud, scope
  ✓ API keys hashed in storage; never logged
  ✓ HTTPS only; HSTS header set

Versioning:
  ✓ Version strategy chosen and applied consistently
  ✓ Deprecation headers on deprecated versions
  ✓ Non-breaking changes catalogued and added without version bump

Documentation:
  ✓ OpenAPI spec kept in sync with implementation
  ✓ Every operation has a description and example
  ✓ Error codes documented with remediation guidance
  ✓ Changelog maintained for each version
```

## 9. Interview Questions

**Q: What is the difference between 401 and 403, and why does it matter?**

A: HTTP 401 Unauthorized means authentication is required or the provided credentials are invalid — the server is saying "I don't know who you are." HTTP 403 Forbidden means the request is authenticated but the caller does not have permission to access the resource — the server knows who you are but you cannot access this. Using the wrong code creates two problems. Returning 401 when you mean 403 misleads the client into thinking their authentication token is wrong and prompting a re-login when the real issue is a permission problem. More critically, returning 404 "Not Found" when you mean 403 is a deliberate choice in security-sensitive contexts: it tells potential attackers neither that the resource exists nor that they lack permission — useful when the existence of the resource is itself sensitive. A user trying to access another user's private data should receive 404 rather than 403, because 403 would confirm the resource exists. Both decisions — using the correct code and the strategic use of 404 — have security and UX implications.

---

**Q: When would you choose cursor-based pagination over offset pagination, and why?**

A: Cursor-based pagination is the right choice whenever the data changes while users are browsing — which is nearly always in production systems. Offset pagination has a fundamental instability problem: if records are inserted or deleted between page requests, items on subsequent pages shift, causing users to see items twice or miss items entirely. At offset 60, the database scans and discards 60 rows — expensive for large datasets. Cursor-based pagination encodes the position as the ID or a stable attribute of the last seen record: "give me all orders with ID less than 12345" uses an index seek, not a scan, and is O(log n) regardless of how deep into the result set you are. The trade-off is that cursor pagination does not support random access — you cannot jump to page 7 — but "jump to page 7" is not a user behaviour pattern for any data I have seen. For search results with total count and page jumping, offset pagination is still appropriate because the dataset is usually small and the query is re-run each time with filters that limit the result size. For feeds, history lists, and anything involving real-time data, cursor pagination is correct.

---

**Q: How do you design an API that can evolve over time without breaking existing clients?**

A: I design for backward compatibility from day one using three practices. First, I follow the expand-then-contract rule: I only add new fields, never remove them in the same version. New optional fields with sensible defaults are safe because existing clients ignore unknown fields (assuming they deserialise permissively, which they should). Removing or renaming fields requires a new major version. Second, I use additive versioning rather than replacement — when a breaking change is necessary, I create a new version that co-exists with the old one and maintain the old version for a defined deprecation period, communicated via Sunset response headers. Third, I design response consumers to be tolerant: I document that clients should ignore unknown fields, and I use explicit enums with an "UNKNOWN" fallback so new enum values added to the server do not cause client deserialisation failures. For testing these properties, consumer-driven contract testing with Pact verifies that any API change satisfies all registered consumer contracts before deployment — catching breaking changes in CI before they reach production.

---

**Q: What are the security considerations in JWT-based API authentication?**

A: There are several layered considerations. Algorithm security: always use RS256 or ES256 (asymmetric), never HS256 for APIs. HS256 requires every service to share the signing secret — a compromised service compromises all tokens. RS256 uses a private key to sign at the auth server and public keys to verify at APIs; public keys are safe to distribute. Claim validation: every service must validate the full set of claims — not just the signature. exp prevents using expired tokens; iss prevents a token issued for a different auth server from being accepted; aud prevents a token intended for service A from being accepted at service B. All three checks are required. Token lifetime: short expiry (15 minutes for access tokens) limits the blast radius of a stolen token. Refresh tokens should be long-lived (30 days) but rotated on use (one-time use, with the new token returned). The kid header supports key rotation without token invalidation — the auth server publishes new keys and marks old ones as deprecated; APIs look up the verification key by kid. Finally, revocation: JWTs are stateless so cannot be revoked by default. For high-security scenarios, maintain a revocation list indexed by jti and check it on each request, at the cost of a Redis lookup per API call.

---

**Q: Walk me through the design of a consistent error response schema for a public API.**

A: A consistent error schema serves two consumers simultaneously: the developer reading documentation and writing code, and the code itself that must branch on error conditions. I structure errors with three layers. The HTTP status code is the coarse-grained signal — 4xx for client errors, 5xx for server errors, with each code mapping to a specific class of problem. Within that, a machine-readable `code` field provides the stable identifier that code branches on — "RATE_LIMIT_EXCEEDED", "INSUFFICIENT_INVENTORY", "INVALID_EMAIL_FORMAT". This code is versioned and stable across API versions; the human-readable `message` is allowed to improve over time without breaking clients. Third, for validation errors a `details` array provides field-level errors so a form UI can show the exact fields that failed. Every error includes a `requestId` that maps to server logs — this is essential for support: the developer can file a ticket with the request ID and the support team can retrieve the full context. For error documentation, every error code links to a documentation page explaining the cause and the exact steps to resolve it. This serves as both developer documentation and as an implicit SLA — documenting an error code creates a commitment to keep it stable.

## 10. Key Takeaways

- **Resources are nouns, HTTP methods are verbs**: `/orders` + `POST` creates; `/orders/123` + `DELETE` removes — never `POST /createOrder`
- **401 vs 403 is a security decision**: 401 = unauthenticated; 403 = authenticated but not permitted; use 404 strategically to avoid confirming that a sensitive resource exists
- **Cursor pagination for feeds; offset only for small bounded searches**: cursor is O(log n) via index; offset is O(n) scan; cursor is stable when data changes between pages
- **Error codes are part of the API contract**: machine-readable `code` field is stable; human-readable `message` may improve; `requestId` is essential for support; field-level `details` for validation errors
- **Expand-then-contract for safe evolution**: add before removing; new optional fields with defaults are always non-breaking; removing or renaming requires a new version
- **RS256 over HS256 for JWTs**: asymmetric signing means only the auth server needs the private key; every API only needs the public key; compromise of one service does not compromise all tokens
- **OAuth2 scopes implement least privilege**: `orders:read` not `full_access`; validate required scopes on every protected endpoint; separate user-facing scopes from service-to-service scopes
- **Idempotency-Key enables safe POST retries**: client-generated UUID per operation; server caches result for 24 hours; essential for payment, order creation, and any non-idempotent operation
- **OpenAPI is a contract, not documentation**: write it first for external APIs; validate implementation against it in CI; every operation needs operationId, description, and error responses documented
- **Rate limit headers on every response**: `X-RateLimit-Remaining` lets clients self-throttle before hitting the limit; `Retry-After` on 429 prevents retry storms; both are essential for a usable public API
