# Virtual Threads (Project Loom - Java 19-21)

## Overview

Virtual threads, introduced as a preview in Java 19 and finalized in Java 21 as part of Project Loom, revolutionize Java concurrency by making threads lightweight and plentiful. Unlike traditional platform threads that are expensive OS resources limited to thousands, virtual threads are managed by the JVM and can number in the millions. A virtual thread still represents a thread of execution but is decoupled from OS threads, allowing massive concurrency without the overhead. The JVM schedules virtual threads onto a small pool of platform threads (carrier threads), mounting and unmounting them as needed. This enables a simple thread-per-request programming model even for highly concurrent applications, eliminating the need for complex asynchronous frameworks. Structured concurrency provides a way to manage the lifecycle of related threads, ensuring they complete or fail together. Scoped values offer a better alternative to ThreadLocal for virtual threads. This guide covers the fundamental differences between virtual and platform threads, multiple ways to create virtual threads, structured concurrency patterns, scoped values usage, performance characteristics showing why blocking is now cheap, when to use virtual threads, and practical migration strategies for existing applications.

```java
 * VIRTUAL VS PLATFORM THREADS

public class VirtualVsPlatformThreads {
    
    /**
     * Platform threads (traditional)
     */
    public static void platformThreads() {
        System.out.println("=== Platform Threads ===");
        
        // Traditional thread - wraps OS thread
        Thread platformThread = new Thread(() -> {
            System.out.println("Platform thread: " + 
                Thread.currentThread());
        });
        platformThread.start();
        
        try {
            platformThread.join();
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
        
        /**
         * PLATFORM THREADS:
         * 
         * What they are:
         * - Thin wrapper around OS threads
         * - 1:1 mapping to OS threads
         * - Heavy, expensive resources
         * 
         * Characteristics:
         * - Stack: ~1-2 MB per thread
         * - Creation: ~1 ms
         * - Context switch: ~10-100 μs
         * - Limit: Thousands (~10,000-50,000)
         * - Scheduling: OS scheduler
         * 
         * Problems:
         * ✗ Limited scalability
         * ✗ Expensive to create
         * ✗ Large memory footprint
         * ✗ Costly context switches
         * ✗ Thread pool complexity
         * 
         * Traditional solution:
         * Use thread pools to limit threads
         * Results in complex async code
         */
    }
    
    /**
     * Virtual threads (Java 19+)
     */
    public static void virtualThreads() {
        System.out.println("\n=== Virtual Threads ===");
        
        // Virtual thread - lightweight, JVM-managed
        Thread virtualThread = Thread.ofVirtual().start(() -> {
            System.out.println("Virtual thread: " + 
                Thread.currentThread());
        });
        
        try {
            virtualThread.join();
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
        
        /**
         * VIRTUAL THREADS:
         * 
         * What they are:
         * - Lightweight threads managed by JVM
         * - M:N mapping to platform threads
         * - Cheap, plentiful resources
         * 
         * Characteristics:
         * - Stack: Dynamic, ~few KB to start
         * - Creation: ~1 μs (1000x faster!)
         * - Mount/unmount: ~1 μs
         * - Limit: Millions (~1-10 million)
         * - Scheduling: JVM scheduler
         * 
         * Benefits:
         * ✓ Massive scalability
         * ✓ Very cheap to create
         * ✓ Tiny memory footprint
         * ✓ Simple synchronous code
         * ✓ No pool needed
         * 
         * Revolutionary change:
         * Thread-per-request is now viable!
         * No need for async/reactive
         */
    }

     * Comparison demonstration
     */

        int taskCount = 10_000;
        
        // Platform threads - limited by resources
        System.out.println("Platform threads:");

        try (var executor = java.util.concurrent.Executors

                    try {
                        Thread.sleep(100);

        long virtualTime = System.currentTimeMillis() - start;
        System.out.println("Time: " + virtualTime + "ms");
        
        System.out.println("\nSpeedup: " +

        /**
         * RESULTS:

     * Scheduling and mounting
     */
    public static void schedulingAndMounting() {
        /**
         * HOW VIRTUAL THREADS WORK:
         * 
         * 1. Creation:
         * Thread vt = Thread.ofVirtual().start(() -> {...});
         * - Virtual thread created
         * - Very cheap (~1 μs)
         * - Small memory footprint
         * 
         * 2. Execution:
         * - JVM picks carrier thread (platform thread)
         * - Mounts virtual thread onto carrier
         * - Code executes on carrier
         * 
         * 3. Blocking (e.g., I/O, sleep):
         * - Virtual thread unmounts from carrier
         * - Carrier thread freed for other virtual threads
         * - Original virtual thread parks
         * 
         * 4. Resumption:
         * - I/O completes or timer expires
         * - Virtual thread ready to run
         * - JVM picks available carrier
         * - Mounts and continues execution
         * 
         * 
         * CARRIER THREAD POOL:
         * 
         * Size: Typically number of CPU cores
         * Default: Runtime.availableProcessors()
         * 
         * Example: 8-core machine
         * - 8 carrier threads
         * - Can run 1 million virtual threads
         * - Only 8 running at a time
         * - Others parked, minimal overhead
         * 
         * 
         * BLOCKING IS CHEAP:
         * 
         * Platform thread blocking:
         * Thread.sleep(1000);
         * → Entire OS thread blocked
         * → Waste of expensive resource
         * 
         * Virtual thread blocking:
         * Thread.sleep(1000);
         * → Virtual thread unmounts
         * → Carrier freed immediately
         * → No resource waste!
         * 
         * Result: Can have millions of
         * sleeping/waiting virtual threads
         */
    }

         * │ Scheduling          │ OS           │ JVM          │
         * │ Context switch      │ ~10-100 μs   │ ~1 μs        │
         * │ Blocking cost       │ EXPENSIVE    │ CHEAP        │
         * │ Use with pools      │ YES          │ NO           │
         * │ Pinning risk        │ None         │ Possible     │
         * │ Thread-local cost   │ Low          │ Consider     │

         * 
         * WHEN TO USE:
         * 
         * Platform threads:

    }

}
```

## 2. Creating Virtual Threads

### Multiple Ways to Create and Use Virtual Threads

```java
/**
 * CREATING VIRTUAL THREADS
 * 
 * Different methods to create virtual threads
 */

import java.util.concurrent.*;

public class CreatingVirtualThreads {
    
    /**
     * Method 1: Thread.ofVirtual()
     */
    public static void threadOfVirtual() {
        System.out.println("=== Thread.ofVirtual() ===");
        
        // Create and start immediately
        Thread vt1 = Thread.ofVirtual().start(() -> {
            System.out.println("Virtual thread 1: " + 
                Thread.currentThread());
        });
        
        // Create but don't start yet
        Thread vt2 = Thread.ofVirtual().unstarted(() -> {
            System.out.println("Virtual thread 2: " + 
                Thread.currentThread());
        });
        vt2.start();  // Start when ready
        
        // With name
        Thread vt3 = Thread.ofVirtual()
            .name("my-virtual-thread")
            .start(() -> {
                System.out.println("Named: " + 
                    Thread.currentThread().getName());
            });
        
        try {
            vt1.join();
            vt2.join();
            vt3.join();
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
        
        /**
         * THREAD.OFVIRTUAL():
         * 
         * Thread.ofVirtual()
         *   .start(Runnable)     // Create and start
         *   .unstarted(Runnable) // Create, don't start
         *   .name(String)        // Set name
         *   .inheritInheritableThreadLocals(boolean)
         *   .uncaughtExceptionHandler(UncaughtExceptionHandler)
         * 
         * Returns: Thread.Builder.OfVirtual
         * Then: .start() or .unstarted()
         * 
         * Use for:
         * - Single virtual thread creation
         * - Custom configuration
         * - Direct thread management
         */
    }
    
    /**
     * Method 2: Thread.startVirtualThread()
     */
    public static void startVirtualThread() {
        System.out.println("\n=== Thread.startVirtualThread() ===");
        
        // Shortcut: create and start
        Thread vt = Thread.startVirtualThread(() -> {
            System.out.println("Started: " + 
                Thread.currentThread());
        });
        
        try {
            vt.join();
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
        
        /**
         * THREAD.STARTVIRTUALTHREAD():
         * 
         * Thread.startVirtualThread(Runnable)
         * 
         * Equivalent to:
         * Thread.ofVirtual().start(Runnable)
         * 
         * Convenient shortcut for:
         * - Quick thread creation
         * - Fire-and-forget tasks
         * - No configuration needed
         */
    }
    
    /**
     * Method 3: Executors.newVirtualThreadPerTaskExecutor()
     */
    public static void virtualThreadExecutor() {
        System.out.println("\n=== Virtual Thread Executor ===");
        
        // Executor that creates virtual thread per task
        try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
            
            for (int i = 0; i < 10; i++) {
                final int taskId = i;
                executor.submit(() -> {
                    System.out.println("Task " + taskId + 
                        " on " + Thread.currentThread());
                });
            }
            
        }  // Auto-closes, waits for all tasks
        
        System.out.println("All tasks complete");
        
        /**
         * VIRTUALTLHREADPERTASKEXECUTOR:
         * 
         * Executors.newVirtualThreadPerTaskExecutor()
         * 
         * Creates:
         * - New virtual thread for each task
         * - No thread pool
         * - No queueing
         * - Unbounded concurrency
         * 
         * AutoCloseable:
         * - try-with-resources support
         * - Waits for all tasks on close
         * - Clean shutdown
         * 
         * Use for:
         * - Many concurrent tasks
         * - I/O-bound operations
         * - Simple parallelism
         * - Replacing cached thread pools
         */
    }
    
    /**
     * Method 4: Custom ThreadFactory
     */
    public static void customThreadFactory() {
        System.out.println("\n=== Custom ThreadFactory ===");
        
        // Virtual thread factory
        ThreadFactory factory = Thread.ofVirtual()
            .name("worker-", 0)
            .factory();
        
        // Use with existing ExecutorService APIs
        ExecutorService executor = Executors.newThreadPerTaskExecutor(factory);
        
        for (int i = 0; i < 5; i++) {
            executor.submit(() -> {
                System.out.println("Factory thread: " + 
                    Thread.currentThread().getName());
            });
        }
        
        executor.close();
        
        /**
         * THREAD FACTORY:
         * 
         * Thread.ofVirtual().factory()
         * 
         * Returns: ThreadFactory
         * Creates: Virtual threads
         * 
         * Use with:
         * - Executors.newThreadPerTaskExecutor()
         * - CompletableFuture.supplyAsync()
         * - Custom executor services
         * 
         * Benefits:
         * - Compatible with existing APIs
         * - Drop-in replacement
         * - Maintains thread naming
         */
    }
    
    /**
     * Comparison of creation methods
     */
    public static void methodComparison() {
        /**
         * CREATION METHOD COMPARISON:
         * 
         * 1. Thread.ofVirtual().start():
         *    - Full control
         *    - Single thread
         *    - Custom configuration
         *    Use: One-off threads
         * 
         * 2. Thread.startVirtualThread():
         *    - Convenient shortcut
         *    - Single thread
         *    - No configuration
         *    Use: Quick fire-and-forget
         * 
         * 3. Executors.newVirtualThreadPerTaskExecutor():
         *    - Multiple tasks
         *    - ExecutorService API
         *    - AutoCloseable
         *    Use: Many concurrent tasks
         * 
         * 4. Thread.ofVirtual().factory():
         *    - Thread factory
         *    - Existing API integration
         *    - Custom naming
         *    Use: Compatibility with old code
         * 
         * 
         * CHOOSING THE RIGHT METHOD:
         * 
         * Single task:
         * Thread.startVirtualThread(() -> {...})
         * 
         * Multiple tasks:
         * try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
         *   executor.submit(() -> {...});
         * }
         * 
         * Custom config:
         * Thread.ofVirtual().name("...").start(() -> {...})
         * 
         * Legacy integration:
         * ThreadFactory factory = Thread.ofVirtual().factory();
         * executor = new ThreadPoolExecutor(..., factory);
         */
    }
    
    /**
     * Real-world example: Web server
     */
    public static void webServerExample() {
        System.out.println("\n=== Web Server Example ===");
        
        // Simulate handling HTTP requests
        try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
            
            for (int i = 0; i < 1000; i++) {
                final int requestId = i;
                executor.submit(() -> handleRequest(requestId));
            }
            
        }
        
        System.out.println("Server handled all requests");
        
        /**
         * WEB SERVER PATTERN:
         * 
         * Old way (platform threads):
         * ExecutorService pool = Executors.newFixedThreadPool(200);
         * - Limited to 200 concurrent requests
         * - Extra requests queue
         * - Need tuning
         * 
         * New way (virtual threads):
         * ExecutorService executor = 
         *   Executors.newVirtualThreadPerTaskExecutor();
         * - Unlimited concurrent requests
         * - No queueing
         * - No tuning needed
         * 
         * For each request:
         * executor.submit(() -> {
         *   readRequest();      // I/O - unmounts
         *   processRequest();   // CPU
         *   queryDatabase();    // I/O - unmounts
         *   writeResponse();    // I/O - unmounts
         * });
         * 
         * Result:
         * - Simple thread-per-request
         * - Scales to millions
         * - No complex async code
         */
    }
    
    private static void handleRequest(int id) {
        try {
            // Simulate I/O operations
            Thread.sleep(100);  // Database query
            Thread.sleep(50);   // External API call
            System.out.println("Handled request " + id);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
    
    public static void main(String[] args) {
        threadOfVirtual();
        startVirtualThread();
        virtualThreadExecutor();
        customThreadFactory();
        webServerExample();
    }
}
```

## 3. Structured Concurrency (Preview)

### Managing Thread Lifecycles Together

```java
/**
 * STRUCTURED CONCURRENCY
 * 
 * Preview feature in Java 19-21
 */

import java.util.concurrent.*;
import java.time.Duration;

public class StructuredConcurrencyExample {
    
    /**
     * The problem with unstructured concurrency
     */
    public static void unstructuredProblem() {
        /**
         * UNSTRUCTURED CONCURRENCY PROBLEMS:
         * 
         * ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor();
         * 
         * Future<String> future1 = executor.submit(() -> fetchData1());
         * Future<String> future2 = executor.submit(() -> fetchData2());
         * 
         * String result1 = future1.get();
         * String result2 = future2.get();
         * 
         * Problems:
         * ✗ If future1.get() fails, future2 still running
         * ✗ If future2.get() hangs, future1 result wasted
         * ✗ Manual cleanup required
         * ✗ Error handling complex
         * ✗ Resource leaks possible
         * ✗ No parent-child relationship
         * 
         * Need structured way to:
         * - Launch related tasks together
         * - Ensure all complete or all fail
         * - Automatic cleanup
         * - Clear lifecycle
         */
    }
    
    /**
     * Structured concurrency basics
     */
    public static void structuredBasics() throws Exception {
        System.out.println("=== Structured Concurrency Basics ===");
        
        // Note: Requires --enable-preview flag
        // This is preview API syntax (Java 21)
        
        /**
         * BASIC PATTERN:
         * 
         * try (var scope = new StructuredTaskScope<String>()) {
         *   
         *   Future<String> task1 = scope.fork(() -> fetchData1());
         *   Future<String> task2 = scope.fork(() -> fetchData2());
         *   
         *   scope.join();  // Wait for all
         *   
         *   String result1 = task1.resultNow();
         *   String result2 = task2.resultNow();
         *   
         * }  // Auto-cleanup
         * 
         * Benefits:
         * ✓ All tasks complete or fail together
         * ✓ Automatic cleanup
         * ✓ Structured lifecycle
         * ✓ Clear ownership
         */
        
        System.out.println("Structured concurrency example (conceptual)");
    }
    
    /**
     * ShutdownOnFailure policy
     */
    public static void shutdownOnFailure() {
        /**
         * SHUTDOWNONFAILURE:
         * 
         * Cancels all tasks if any fails
         * 
         * try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
         *   
         *   Future<String> user = scope.fork(() -> fetchUser(id));
         *   Future<String> order = scope.fork(() -> fetchOrder(id));
         *   
         *   scope.join();           // Wait for all
         *   scope.throwIfFailed();  // Throw if any failed
         *   
         *   // All succeeded
         *   String userData = user.resultNow();
         *   String orderData = order.resultNow();
         *   
         * }  // Cleanup
         * 
         * Behavior:
         * - If fetchUser() fails:
         *   * scope cancels fetchOrder()
         *   * throwIfFailed() throws exception
         * - If both succeed:
         *   * Results available
         * 
         * Use for:
         * - All-or-nothing operations
         * - Dependent tasks
         * - Fail-fast behavior
         */
    }
    
    /**
     * ShutdownOnSuccess policy
     */
    public static void shutdownOnSuccess() {
        /**
         * SHUTDOWNONSUCCESS:
         * 
         * Cancels all tasks when first succeeds
         * 
         * try (var scope = new StructuredTaskScope.ShutdownOnSuccess<String>()) {
         *   
         *   scope.fork(() -> fetchFromServer1());
         *   scope.fork(() -> fetchFromServer2());
         *   scope.fork(() -> fetchFromServer3());
         *   
         *   scope.join();  // Wait for first success
         *   
         *   String result = scope.result();  // Get the winner
         *   
         * }  // Cleanup
         * 
         * Behavior:
         * - First task to succeed:
         *   * scope cancels other tasks
         *   * result() returns that value
         * - All tasks fail:
         *   * result() throws exception
         * 
         * Use for:
         * - Fastest response wins
         * - Multiple redundant sources
         * - Load balancing
         * - Timeouts
         */
    }
    
    /**
     * Real-world example: Gathering data
     */
    public static void gatheringExample() {
        /**
         * REAL-WORLD: GATHER DATA FROM MULTIPLE SOURCES
         * 
         * record UserData(String profile, String orders, String preferences) {}
         * 
         * UserData fetchUserData(String userId) throws Exception {
         *   try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
         *     
         *     Future<String> profile = 
         *       scope.fork(() -> fetchProfile(userId));
         *     Future<String> orders = 
         *       scope.fork(() -> fetchOrders(userId));
         *     Future<String> preferences = 
         *       scope.fork(() -> fetchPreferences(userId));
         *     
         *     scope.join()
         *          .throwIfFailed();  // All must succeed
         *     
         *     return new UserData(
         *       profile.resultNow(),
         *       orders.resultNow(),
         *       preferences.resultNow()
         *     );
         *   }
         * }
         * 
         * Benefits:
         * - Three concurrent requests
         * - If any fails, all cancel
         * - Automatic cleanup
         * - Simple, readable code
         */
    }
    
    /**
     * Custom policies
     */
    public static void customPolicies() {
        /**
         * CUSTOM POLICIES:
         * 
         * Extend StructuredTaskScope for custom behavior:
         * 
         * class FirstNSuccesses<T> extends StructuredTaskScope<T> {
         *   private final int n;
         *   private final List<T> results = new CopyOnWriteArrayList<>();
         *   
         *   FirstNSuccesses(int n) {
         *     this.n = n;
         *   }
         *   
         *   @Override
         *   protected void handleComplete(Future<T> future) {
         *     if (future.state() == Future.State.SUCCESS) {
         *       results.add(future.resultNow());
         *       if (results.size() >= n) {
         *         shutdown();  // Cancel rest
         *       }
         *     }
         *   }
         *   
         *   public List<T> results() {
         *     return List.copyOf(results);
         *   }
         * }
         * 
         * Use:
         * try (var scope = new FirstNSuccesses<String>(3)) {
         *   for (int i = 0; i < 10; i++) {
         *     scope.fork(() -> fetchFromServer(i));
         *   }
         *   scope.join();
         *   List<String> first3 = scope.results();
         * }
         * 
         * Custom policies for:
         * - Majority voting
         * - Quorum consensus
         * - Best-effort collection
         * - Custom success criteria
         */
    }
    
    /**
     * Benefits summary
     */
    public static void benefits() {
        /**
         * STRUCTURED CONCURRENCY BENEFITS:
         * 
         * 1. Clear ownership:
         *    - Parent scope owns child tasks
         *    - Tasks can't outlive scope
         *    - No orphaned threads
         * 
         * 2. Automatic cleanup:
         *    - try-with-resources
         *    - Guaranteed shutdown
         *    - No resource leaks
         * 
         * 3. Error propagation:
         *    - Failures bubble up
         *    - All tasks cancelled on failure
         *    - Clear error handling
         * 
         * 4. Simplified code:
         *    - No manual Future tracking
         *    - No complex error handling
         *    - Readable, maintainable
         * 
         * 5. Composable:
         *    - Scopes can nest
         *    - Tasks can create subtasks
         *    - Clear hierarchy
         * 
         * 
         * COMPARISON:
         * 
         * Unstructured:
         * ExecutorService executor = ...;
         * Future<A> f1 = executor.submit(...);
         * Future<B> f2 = executor.submit(...);
         * try {
         *   A a = f1.get();
         *   B b = f2.get();
         *   return combine(a, b);
         * } finally {
         *   executor.shutdown();
         *   // But f1, f2 might still run!
         * }
         * 
         * Structured:
         * try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
         *   Future<A> f1 = scope.fork(...);
         *   Future<B> f2 = scope.fork(...);
         *   scope.join().throwIfFailed();
         *   return combine(f1.resultNow(), f2.resultNow());
         * }  // Guaranteed cleanup
         */
    }
    
    public static void main(String[] args) throws Exception {
        structuredBasics();
    }
```

```java
        System.out.println("Platform threads: " + platformTime + " ns");
        System.out.println("Virtual threads: " + virtualTime + " ns");
        System.out.println("Speedup: " + 
            (platformTime / (double) virtualTime) + "x");

         * - Stack allocation (~1-2 MB)

    public static void blockingOperations() {
        System.out.println("\n=== Blocking Operations ===");
        
        int threadCount = 10000;
        int sleepTime = 1000;  // 1 second

        System.out.println("Platform threads blocking:");

        long platformTime = System.currentTimeMillis() - start;
        System.out.println("Time: " + platformTime + "ms");
        
        // Virtual threads - cheap blocking
        System.out.println("\nVirtual threads blocking:");
        start = System.currentTimeMillis();
        try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
            for (int i = 0; i < threadCount; i++) {
                executor.submit(() -> {
                    try {
                        Thread.sleep(sleepTime);

         * PINNING:
         * 
         * Normally:
         * Virtual thread blocks → unmounts from carrier
         * 
         * Pinning:
         * Virtual thread blocks → stays mounted
         * → Carrier thread blocked
         * → Performance hit!
         * 
         * 
         * WHEN PINNING OCCURS:
         * 
         * 1. synchronized blocks:
         * synchronized (lock) {
         *   Thread.sleep(1000);  // PINNED!
         * }
         * 
         * 2. Native methods:
         * someNativeCall();  // May pin
         * 
         * 
         * WHY IT MATTERS:
         * 
         * Pinned virtual thread:
         * - Cannot unmount
         * - Holds carrier thread
         * - Reduces parallelism
         * 
         * Example:
         * 8 carrier threads
         * 1000 virtual threads
         * 100 pinned threads
         * → Only 8 can run at once
         * → Bottleneck!
         * 
         * 
         * SOLUTIONS:
         * 
         * 1. Replace synchronized with ReentrantLock:
         * 
         * Before (pins):
         * synchronized (lock) {
         *   Thread.sleep(1000);
         * }
         * 
         * After (no pinning):
         * lock.lock();
         * try {
         *   Thread.sleep(1000);  // Can unmount
         * } finally {
         *   lock.unlock();
         * }
         * 
         * 2. Keep synchronized sections short:
         * synchronized (lock) {
         *   // Quick operation only
         *   // No I/O, no blocking
         * }
         * 
         * 3. Monitor pinning:
         * -Djdk.tracePinnedThreads=full
         * 
         * 
         * DETECTING PINNING:
         * 
         * Run with:
         * java -Djdk.tracePinnedThreads=full MyApp
         * 
         * Output when pinning occurs:
         * Thread[#123,ForkJoinPool-1-worker-1,5,CarrierThreads]

         *   at MyClass.method(MyClass.java:42) <== monitors:1

    }
```

```java
public class WhenToUseVirtualThreads {

     * Perfect use cases
     */
    public static void perfectUseCases() {
        /**
         * PERFECT FOR VIRTUAL THREADS:
         * 
         * 1. Web servers:
         *    - Handle concurrent requests
         *    - Each request = virtual thread
         *    - I/O: database, APIs, files
         *    - Simple thread-per-request
         * 
         * Example:
         * void handleRequest(HttpRequest request) {
         *   Thread.startVirtualThread(() -> {
         *     User user = database.findUser(request.userId());
         *     Data data = externalAPI.fetch(user.id());
         *     Response response = buildResponse(data);
         *     writeResponse(response);
         *   });
         * }
         * 
         * 
         * 2. Microservices:
         *    - Many concurrent API calls
         *    - Network I/O dominant
         *    - Simple orchestration
         * 
         * Example:
         * try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
         *   Future<User> user = executor.submit(() -> userService.get(id));
         *   Future<Orders> orders = executor.submit(() -> orderService.get(id));
         *   Future<Prefs> prefs = executor.submit(() -> prefService.get(id));
         *   
         *   return new UserData(user.get(), orders.get(), prefs.get());
         * }
         * 
         * 
         * 3. Database applications:
         *    - Many concurrent queries
         *    - Connection pooling
         *    - JDBC operations
         * 
         * Example:
         * try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
         *   List<Future<Result>> futures = new ArrayList<>();
         *   
         *   for (int i = 0; i < 1000; i++) {
         *     futures.add(executor.submit(() -> 
         *       database.query("SELECT * FROM users WHERE ...")));
         *   }
         *   
         *   // Wait for all
         *   List<Result> results = futures.stream()
         *     .map(this::getFuture)
         *     .toList();
         * }
         * 
         * 
         * 4. File I/O:
         *    - Reading many files
         *    - Processing in parallel
         *    - Simple code
         * 
         * Example:
         * try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
         *   for (Path file : files) {
         *     executor.submit(() -> processFile(file));
         *   }
         * }
         * 
         * 
         * 5. WebSocket connections:
         *    - One thread per connection
         *    - Mostly idle/waiting
         *    - Millions of connections possible
         * 
         * Example:
         * void handleWebSocket(WebSocket ws) {
         *   Thread.startVirtualThread(() -> {
         *     while (ws.isOpen()) {
         *       String message = ws.receiveMessage();  // Blocks
         *       processMessage(message);
         *       ws.sendMessage(response);
         *     }
         *   });
         * }

         *    - Pure computation
         *    - No I/O
         *    - Limited by CPU cores
         * 
         * Example (DON'T):

         *   return fibonacci(50);

         *    - Already optimized
         *    - Uses ForkJoinPool
```

## Summary

### Quick Reference

**Creating Virtual Threads:**

```java
// Method 1: Direct
Thread.startVirtualThread(() -> task());

// Method 2: Builder
Thread.ofVirtual().start(() -> task());

// Method 3: Executor
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    executor.submit(() -> task());
}
```

**Key Differences:**

```
Platform Thread: 1 thread = 1 OS thread (~1-2 MB each)
Virtual Thread:  1M threads on ~8 platform threads (~1 KB each)

Creation: 1000x faster
Memory: 1000x less
Blocking: Free (unmounts)
Limit: Millions possible
```

**When to Use:**

```
✓ I/O-bound operations
✓ High concurrency needs
✓ Simple synchronous code
✓ Web servers, microservices

✗ CPU-bound computation
✗ Heavy synchronized blocks
✗ Parallel stream processing
```

- **1000x faster creation**: 1μs vs 1ms
- **1000x less memory**: 1KB vs 1-2MB
- **Millions of threads**: 1-10M vs 10-50K
- **Blocking is cheap**: Unmounts from carrier
- **Thread-per-request**: Now viable!
