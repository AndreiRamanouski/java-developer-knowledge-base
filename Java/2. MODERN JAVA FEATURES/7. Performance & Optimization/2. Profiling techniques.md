# Java Profiling Techniques

## Overview

Performance problems in production Java applications are often invisible without proper profiling—a CPU bottleneck might hide in an unexpected method, memory leaks can be subtle, and lock contention only appears under load. Java profiling tools provide visibility into application behavior by collecting runtime data: which methods consume CPU time, where memory is allocated, why threads are blocked. This guide covers the fundamental approaches to profiling (sampling vs instrumentation—lightweight observation vs detailed tracing), modern profiling tools (async-profiler for production-safe CPU profiling, Java Flight Recorder for comprehensive diagnostics, VisualVM and Mission Control for visual analysis), and specific profiling techniques (CPU profiling to find hot methods, memory profiling to track allocations and leaks, lock contention analysis to identify threading bottlenecks). Understanding profiling is critical for optimization—without it, you're guessing; with it, you're measuring and making data-driven decisions. Whether you're hunting down performance issues in production or optimizing during development, the right profiling approach reveals what's actually happening inside your application.

## 1. Sampling vs Instrumentation

### Two Fundamental Approaches

```java
/**
 * PROFILING APPROACHES EXPLAINED
 */

public class ProfilingApproaches {
    
    /**
     * Sampling profiling
     */
    public static void samplingProfiling() {
        /**
         * SAMPLING:
         * 
         * How it works:
         * 1. Periodically (e.g., every 10ms)
         * 2. Stop all threads
         * 3. Take stack trace snapshot
         * 4. Resume threads
         * 5. Aggregate snapshots
         * 
         * Example:
         * Time 0ms: methodA() -> methodB() -> methodC()
         * Time 10ms: methodA() -> methodB() -> methodD()
         * Time 20ms: methodA() -> methodE()
         * 
         * Result:
         * methodA: 3 samples (100%)
         * methodB: 2 samples (67%)
         * methodC: 1 sample (33%)
         * methodD: 1 sample (33%)
         * methodE: 1 sample (33%)
         * 
         * 
         * CHARACTERISTICS:
         * 
         * Overhead: Very low (1-3%)
         * Accuracy: Statistical approximation
         * Detail: Method-level
         * Production: Safe to use
         * 
         * Pros:
         * ✓ Low overhead
         * ✓ Can run in production
         * ✓ Minimal performance impact
         * ✓ Good for hot spot detection
         * 
         * Cons:
         * ✗ May miss short methods
         * ✗ Not exact counts
         * ✗ Statistical noise
         * ✗ Can't see every call
         */
    }
    
    /**
     * Instrumentation profiling
     */
    public static void instrumentationProfiling() {
        /**
         * INSTRUMENTATION:
         * 
         * How it works:
         * 1. Modify bytecode
         * 2. Add timing code to every method entry/exit
         * 3. Record every single call
         * 4. Aggregate exact counts and times
         * 
         * Example (modified code):
         * void methodA() {
         *   profiler.enter("methodA");
         *   try {
         *     // original code
         *     methodB();
         *   } finally {
         *     profiler.exit("methodA");
         *   }
         * }
         * 
         * Result:
         * methodA: called 1000 times, 5000ms total
         * methodB: called 1000 times, 3000ms total
         * methodC: called 500 times, 1000ms total
         * 
         * Exact counts and times!
         * 
         * 
         * CHARACTERISTICS:
         * 
         * Overhead: High (10-100%+)
         * Accuracy: Exact
         * Detail: Every method call
         * Production: Usually too slow
         * 
         * Pros:
         * ✓ Exact call counts
         * ✓ Precise timings
         * ✓ See every invocation
         * ✓ Detailed call trees
         * 
         * Cons:
         * ✗ High overhead
         * ✗ Changes behavior (observer effect)
         * ✗ Can distort results
         * ✗ Not production-safe
         */
    }
    
    /**
     * When to use each
     */
    public static void whenToUse() {
        /**
         * USE SAMPLING WHEN:
         * 
         * 1. Running in production
         *    - Low overhead critical
         *    - Can't slow down app
         * 
         * 2. Finding hot spots
         *    - Which methods use most CPU?
         *    - Where to focus optimization?
         * 
         * 3. Long-running applications
         *    - Servers, services
         *    - Continuous profiling
         * 
         * 4. Initial investigation
         *    - Quick overview
         *    - Low cost exploration
         * 
         * 
         * USE INSTRUMENTATION WHEN:
         * 
         * 1. Development/testing
         *    - Performance impact OK
         *    - Need exact data
         * 
         * 2. Exact call counts needed
         *    - How many times called?
         *    - Call relationships?
         * 
         * 3. Rare method calls
         *    - Sampling might miss
         *    - Need guaranteed capture
         * 
         * 4. Detailed investigation
         *    - After sampling identifies area
         *    - Deep dive into specific code
         * 
         * 
         * TYPICAL WORKFLOW:
         * 
         * 1. Start with sampling (low overhead)
         * 2. Identify hot spots
         * 3. Switch to instrumentation for details
         * 4. Fix issue
         * 5. Verify with sampling again
         */
    }
    
    /**
     * Hybrid approach
     */
    public static void hybridApproach() {
        /**
         * MODERN PROFILERS:
         * 
         * Many use hybrid approaches:
         * 
         * 1. Adaptive instrumentation:
         *    - Sample to find hot spots
         *    - Instrument only hot methods
         *    - Best of both worlds
         * 
         * 2. Selective instrumentation:
         *    - User selects methods to instrument
         *    - Rest use sampling
         *    - Minimize overhead
         * 
         * 3. Event-based profiling:
         *    - Use JVM events (allocation, GC, etc.)
         *    - Low overhead like sampling
         *    - More detail than sampling
         */
    }
}
```

## 2. Async-profiler

### Modern Low-Overhead Profiler

```bash
# ASYNC-PROFILER USAGE

# Download and setup
wget https://github.com/async-profiler/async-profiler/releases/download/v2.9/async-profiler-2.9-linux-x64.tar.gz
tar xzf async-profiler-2.9-linux-x64.tar.gz

# Basic CPU profiling
./profiler.sh -d 30 -f cpu-profile.html <PID>

# Arguments:
# -d 30: Profile for 30 seconds
# -f: Output file (HTML flame graph)
# <PID>: Java process ID

# Memory allocation profiling
./profiler.sh -d 30 -e alloc -f alloc-profile.html <PID>

# Lock profiling
./profiler.sh -d 30 -e lock -f lock-profile.html <PID>

# Wall-clock profiling (includes I/O wait)
./profiler.sh -d 30 -e wall -f wall-profile.html <PID>
```

```java
         * 
         * Wide boxes = Hot spots
         * - Most time spent here
         * - Focus optimization here
         * 
         * Tall stacks = Deep call chains
         * - Complex code paths
         * - May have optimization opportunities
         * 
         * Flat/wide plateaus = CPU intensive

         * FLAME GRAPH PATTERNS:
         * 
         * 1. Tower (tall, narrow):
         *    - Deep call stack
         *    - Little CPU time

         * 
         * 2. Plateau (wide, flat):
         *    - CPU-intensive method
         *    - Hot spot!
         *    - Optimization target

         *    - Multiple hot spots
         *    - Distributed work
```

## 3. Java Flight Recorder (JFR)

### Built-in Comprehensive Profiler

```bash
# JAVA FLIGHT RECORDER COMMANDS

# Start recording at JVM startup
java -XX:StartFlightRecording=duration=60s,filename=recording.jfr MyApp

# Or start recording on running JVM
jcmd <PID> JFR.start duration=60s filename=recording.jfr

# Check recording status
jcmd <PID> JFR.check

# Dump current recording
jcmd <PID> JFR.dump filename=dump.jfr

# Stop recording
jcmd <PID> JFR.stop
```

## 5. CPU Profiling

### Finding Performance Bottlenecks

```java
/**
 * CPU PROFILING TECHNIQUES
 */

public class CpuProfilingTechniques {
    
    /**
     * What to look for
     */
    public static void whatToLookFor() {
        /**
         * HOT SPOTS:
         * 
         * Methods consuming most CPU time
         * 
         * Indicators:
         * - High self time %
         * - Frequent in stack traces
         * - Wide in flame graph
         * 
         * 
         * COMMON PATTERNS:
         * 
         * 1. Inefficient algorithms:
         *    - O(n²) instead of O(n log n)
         *    - Nested loops
         *    - Redundant calculations
         * 
         * 2. Excessive object creation:
         *    - Allocating in tight loops
         *    - String concatenation
         *    - Autoboxing
         * 
         * 3. Reflection overuse:
         *    - Dynamic method calls
         *    - Class.forName() in loops
         *    - Field access via reflection
         * 
         * 4. Synchronization contention:
         *    - Hot locks
         *    - Thread waiting
         *    - Lock convoy
         * 
         * 5. I/O in compute:
         *    - File operations in loop
         *    - Network calls
         *    - Database queries
         */
    }
    
    /**
     * Example: Finding CPU bottleneck
     */
    @SuppressWarnings("unused")
    public static void exampleBottleneck() {
        /**
         * SCENARIO:
         * 
         * Application slow, CPU at 100%
         * 
         * STEP 1: Profile with async-profiler
         * 
         * ./profiler.sh -d 30 -f cpu.html <PID>
         * 
         * 
         * STEP 2: Analyze flame graph
         * 
         * See large plateau:
         * processData() -> validateItem() -> checkDuplicate()
         * 
         * checkDuplicate() is 60% of CPU time!
         * 
         * 
         * STEP 3: Examine code
         * 
         * boolean checkDuplicate(Item item, List<Item> items) {
         *   for (Item existing : items) {  // O(n)
         *     if (existing.equals(item)) {
         *       return true;
         *     }
         *   }
         *   return false;
         * }
         * 
         * Called for each item: O(n²) total!
         * 
         * 
         * STEP 4: Fix
         * 
         * Use HashSet instead:
         * 
         * Set<Item> seenItems = new HashSet<>();
         * 
         * boolean checkDuplicate(Item item) {
         *   return !seenItems.add(item);  // O(1)
         * }
         * 
         * Total: O(n) instead of O(n²)
         * 
         * 
         * STEP 5: Verify
         * 
         * Profile again
         * checkDuplicate() now < 5% CPU
         * 12x speedup!
         */
    }
    
    /**
     * Self time vs total time
     */
    public static void selfVsTotalTime() {
        /**
         * UNDERSTANDING TIME METRICS:
         * 
         * SELF TIME:
         * - Time in method itself
         * - Not including callees
         * - Direct CPU usage
         * 
         * Example:
         * void methodA() {
         *   // 10ms of work here
         *   methodB();  // 20ms
         * }
         * 
         * methodA self time: 10ms
         * 
         * 
         * TOTAL TIME:
         * - Time in method + callees
         * - Everything below in stack
         * 
         * methodA total time: 30ms
         * 
         *
```

```java
/**
 * LOCK CONTENTION ANALYSIS

public class LockContentionAnalysis {
    
    /**
     * What is lock contention?
     */
    public static void whatIsLockContention() {
        /**
         * LOCK CONTENTION:
         * 
         * Multiple threads competing for same lock
         * 
         * Example:
         * Thread 1: synchronized(lock) { work(); }
         * Thread 2: synchronized(lock) { work(); }  ← Blocked!
         * Thread 3: synchronized(lock) { work(); }  ← Blocked!
         * Thread 4: synchronized(lock) { work(); }  ← Blocked!
         * 
         * Problems:
         * - Threads waiting, not working
         * - CPU underutilized
         * - Poor scalability
         * - Latency spikes
         * 
         * 
         * SYMPTOMS:
         * 
         * 1. Low CPU usage:
         *    - Many threads
         *    - But CPU idle
         *    - Threads blocked
         * 
         * 2. Poor scaling:
         *    - Add threads
         *    - No improvement
         *    - May get worse
         * 
         * 3. Long latencies:
         *    - Requests slow
         *    - Waiting for locks
         *    - Unpredictable timing
         */
    }
    
    /**
     * Profiling lock contention
     */
    public static void profilingLockContention() {
        /**
         * ASYNC-PROFILER LOCK MODE:
         * 
         * ./profiler.sh -e lock -d 30 -f lock.html <PID>
         * 
         * Shows:
         * - Which locks have contention
         * - Where threads block
         * - Stack traces of blocked threads
         * 
         * 
         * JFR LOCK EVENTS:
         * 
         * jcmd <PID> JFR.start settings=profile
         * 
         * Events recorded:
         * - Monitor enter
         * - Monitor wait
         * - Park/unpark
         * - Duration of blocking
         * 
         * Analyze in JMC:
         * - Lock Instances view
         * - Shows contended locks
         * - Duration statistics
         */
    }
    
    /**
     * Thread dumps
     */
    public static void threadDumps() {
        /**
         * TAKING THREAD DUMPS:
         * 
         * 1. jstack:
         *    jstack <PID> > thread-dump.txt
         * 
         * 2. jcmd:
         *    jcmd <PID> Thread.print
         * 
         * 3. kill signal:
         *    kill -3 <PID>  (Unix)
         *    Ctrl+Break (Windows)
         * 
         * 4. VisualVM:
         *    "Thread Dump" button
         * 
         * 
         * READING THREAD DUMPS:
         * 
         * Thread state:
         * - RUNNABLE: Executing
         * - BLOCKED: Waiting for lock
         * - WAITING: Waiting for notification
         * - TIMED_WAITING: Waiting with timeout
         * 
         * Example:
         * "Thread-1" #12 prio=5 BLOCKED
         *   waiting to lock <0x00007f8c4c000000> (a java.lang.Object)
         *   at com.example.Service.process(Service.java:42)
         * 
         * Thread-1 is blocked waiting for lock!
         * 
         * 
         * FINDING DEADLOCKS:
         * 
         * Thread dumps show deadlocks:
         * 
         * Found one Java-level deadlock:
         * "Thread-1":
         *   waiting for lock <A>
         *   which is held by "Thread-2"
         * "Thread-2":
         *   waiting for lock <B>
         *   which is held by "Thread-1"
         */
    }
    
    /**
     * Example analysis
     */
    public static void exampleAnalysis() {
        /**
         * SCENARIO:
         * 
         * Application slow under load
         * CPU usage low (20%)
         * Many threads (100)
         * 
         * 
         * STEP 1: Take thread dump
         * 
         * jstack <PID> > threads.txt
         * 
         * See many threads:
         * "worker-42" #142 BLOCKED
         *   waiting to lock <0x00007f8c4c123456>
         *   at DataCache.get(DataCache.java:25)
         * 
         * "worker-87" #187 BLOCKED
         *   waiting to lock <0x00007f8c4c123456>
         *   at DataCache.get(DataCache.java:25)
         * 
         * 90 threads blocked on same lock!
         * 
         * 
         * STEP 2: Examine code
         * 
         * class DataCache {
         *   private Map<String, Data> cache = new HashMap<>();
         *   
         *   public synchronized Data get(String key) {  // Hot lock!
         *     return cache.get(key);
         *   }
         *   
         *   public synchronized void put(String key, Data data) {
         *     cache.put(key, data);
         *   }
         * }
         * 
         * Problem: Synchronized on entire object!
         * Only one thread can read at a time!
         * 
         * 
         * STEP 3: Fix
         * 
         * Use ConcurrentHashMap:
         * 
         * class DataCache {
         *   private Map<String, Data> cache = new ConcurrentHashMap<>();
         *   
         *   public Data get(String key) {  // No lock needed!
         *     return cache.get(key);
         *   }
         *   
         *   public void put(String key, Data data) {
         *     cache.put(key, data);
         *   }
         * }
         * 
         * 
         * STEP 4: Verify
         * 
         * - CPU usage: 80% (was 20%)
         * - Latency: 10ms (was 200ms)
         * - Throughput: 8x improvement!
         */
    }
    
    /**
     * Common contention patterns
     */
    public static void commonPatterns() {
        /**
         * CONTENTION PATTERNS:
         * 
         * 1. Coarse-grained locking:
         *    synchronized(bigObject) {
         *      // Lots of work
         *    }
         *    
         *    Fix: Fine-grained locks
         * 
         * 2. Synchronized collection:
         *    List<T> list = Collections.synchronizedList(new ArrayList<>());
         *    
         *    Fix: Use concurrent collections
         * 
         * 3. Lazy initialization:
         *    synchronized Data getData() {
         *      if (data == null) {
         *        data = load();  // Expensive!
         *      }
         *      return data;
         *    }
         *    
         *    Fix: Double-checked locking or holder pattern
         * 
         * 4. Synchronized getter:
         *    synchronized int getValue() {
         *      return value;  // Read only!
         *    }
         *    
         *    Fix: Use volatile or AtomicInteger
         * 
         * 5. Lock ordering:
         *    synchronized(lockA) {
         *      synchronized(lockB) { }
         *    }
         *    
         *    Fix: Consistent lock ordering
         */
    }
}
```

## Summary

### Quick Reference

**Profiling Approaches:**

```
Sampling:
• Low overhead (1-3%)
• Statistical approximation
• Production-safe
• Good for hot spots

Instrumentation:
• High overhead (10-100%+)
• Exact counts
• Development/testing only
• Good for detailed analysis
```

**Tools:**

```bash
# Async-profiler (CPU)
./profiler.sh -d 30 -f cpu.html <PID>

# Async-profiler (Allocation)
./profiler.sh -e alloc -f alloc.html <PID>

# JFR Recording
jcmd <PID> JFR.start duration=60s filename=recording.jfr

# Thread Dump
jstack <PID> > threads.txt

# Heap Dump
jcmd <PID> GC.heap_dump filename=heap.hprof
```

**Analysis:**

```
CPU Profiling:
→ Find hot methods
→ Check self time %
→ Optimize algorithms

Memory Profiling:
→ Track allocations
→ Find memory leaks
→ Analyze heap dumps

Lock Contention:
→ Identify hot locks
→ Check thread states
→ Use concurrent collections
```

---


**Key insights:**

- **Sampling = production-safe**: Low overhead (<1%)
- **Instrumentation = development only**: High overhead (10-100%+)
- **Async-profiler = best for CPU**: No safepoint bias
- **JFR = comprehensive**: All events, low overhead
- **Flame graphs = visual hot spots**: Wide = expensive
- **Heap dumps = memory forensics**: Find leaks
- **Thread dumps = lock analysis**: See contention

**The profiling workflow:**

```
1. Observe symptoms:
   → Slow response
   → High CPU/memory
   → Low throughput

2. Profile (sampling first):
   → Async-profiler for CPU
   → JFR for comprehensive
   → Low overhead, safe

3. Identify hot spots:
   → Flame graph plateaus
   → High self time %
   → Allocation hot spots

4. Deep dive (if needed):
   → Instrumentation
   → Heap dumps
   → Thread dumps

5. Fix and verify:
   → Make changes
   → Profile again
   → Measure improvement

Result: Data-driven optimization!
```

**Common findings:**

```
CPU Issues:
→ O(n²) algorithms
→ Synchronization contention
→ Excessive allocations

Memory Issues:
→ Growing caches
→ Listener leaks
→ ThreadLocal leaks

Threading Issues:
→ Coarse-grained locks
→ Synchronized collections
→ Lock ordering problems
```
