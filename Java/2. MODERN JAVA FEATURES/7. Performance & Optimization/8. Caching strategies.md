# Caching Strategies

## Overview

Caching is one of the most effective performance optimizations—storing expensive computation results or frequently accessed data in memory can transform an application from unusable to responsive. A cache hit that takes microseconds replaces a database query that takes milliseconds (1000x speedup) or a computation that takes seconds (1,000,000x speedup). However, caching introduces complexity: stale data, memory pressure, cache invalidation ("one of the two hard problems in computer science"), and distributed consistency challenges. This guide covers caching strategies: memoization patterns (simple function result caching), Guava Cache (Google's mature caching library with automatic loading and eviction), Caffeine (high-performance successor with better algorithms), cache eviction policies (LRU for recency, LFU for frequency, TTL for freshness), cache warming (preloading before traffic), distributed caching considerations (Redis, Hazelcast, consistency trade-offs), and cache coherence (keeping multiple caches synchronized). Understanding these patterns enables you to choose the right caching approach—from simple in-memory maps to sophisticated distributed systems—and avoid common pitfalls like cache stampede, memory leaks from unbounded caches, and inconsistent state across nodes.

## 1. Memoization Pattern

### Basic Function Result Caching

```java
/**
 * MEMOIZATION PATTERN
 */

import java.util.*;
import java.util.concurrent.*;
import java.util.function.Function;

public class MemoizationPattern {
    
    /**
     * What is memoization?
     */
    public static void whatIsMemoization() {
        /**
         * MEMOIZATION:
         * 
         * Cache function results
         * Input → cached output
         * 
         * Benefits:
         * ✓ Avoid recomputation
         * ✓ Trade memory for speed
         * ✓ Dramatic speedup for expensive functions
         * 
         * 
         * WHEN TO USE:
         * 
         * ✓ Pure functions (same input → same output)
         * ✓ Expensive computation
         * ✓ Called repeatedly with same inputs
         * ✓ Limited input space
         * 
         * ✗ Impure functions (side effects, time-dependent)
         * ✗ Unlimited input space (memory leak!)
         * ✗ Cheap computation (caching overhead > benefit)
         */
    }
    
    /**
     * Simple memoization
     */
    public static class SimpleMemoizer<T, R> {
        private final Map<T, R> cache = new ConcurrentHashMap<>();
        private final Function<T, R> function;
        
        public SimpleMemoizer(Function<T, R> function) {
            this.function = function;
        }
        
        public R compute(T input) {
            return cache.computeIfAbsent(input, function);
        }
        
        /**
         * USAGE:
         * 
         * Function<Integer, Integer> expensive = n -> {
         *   // Expensive computation
         *   Thread.sleep(1000);
         *   return n * n;
         * };
         * 
         * SimpleMemoizer<Integer, Integer> memoized = 
         *   new SimpleMemoizer<>(expensive);
         * 
         * memoized.compute(5);  // Takes 1000ms (computes)
         * memoized.compute(5);  // Takes ~0.001ms (cached)
         * 
         * 1,000,000x faster!
         */
    }
    
    /**
     * Example: Fibonacci with memoization
     */
    public static class FibonacciMemoized {
        private final Map<Integer, Long> cache = new HashMap<>();
        
        public long fibonacci(int n) {
            if (n <= 1) return n;
            
            return cache.computeIfAbsent(n, k -> 
                fibonacci(k - 1) + fibonacci(k - 2)
            );
        }
        
        /**
         * COMPARISON:
         * 
         * Without memoization:
         * fibonacci(40) → 331,160,281 recursive calls!
         * Time: ~1 second
         * 
         * With memoization:
         * fibonacci(40) → 40 cache misses, rest are hits
         * Time: ~0.001 ms
         * 
         * 1,000,000x faster!
         * 
         * 
         * WHY IT WORKS:
         * 
         * fibonacci(5) = fibonacci(4) + fibonacci(3)
         * fibonacci(4) = fibonacci(3) + fibonacci(2)
         * 
         * Without cache: fibonacci(3) computed multiple times
         * With cache: fibonacci(3) computed once, reused
         */
    }
    
    /**
     * Thread-safe memoization with Future
     */
    public static class FutureMemoizer<T, R> {
        private final Map<T, Future<R>> cache = new ConcurrentHashMap<>();
        private final Function<T, R> function;
        
        public FutureMemoizer(Function<T, R> function) {
            this.function = function;
        }
        
        public R compute(T input) throws InterruptedException, ExecutionException {
            Future<R> future = cache.get(input);
            if (future == null) {
                FutureTask<R> task = new FutureTask<>(() -> function.apply(input));
                future = cache.putIfAbsent(input, task);
                if (future == null) {
                    future = task;
                    task.run();  // Compute
                }
            }
            return future.get();  // Wait if computing
        }
        
        /**
         * THREAD SAFETY:
         * 
         * Multiple threads requesting same input:
         * - Only one computes
         * - Others wait for result
         * - No duplicate computation!
         * 
         * 
         * BENEFIT:
         * 
         * Cache stampede prevention:
         * Without: 100 threads compute same expensive value
         * With: 1 thread computes, 99 wait
         * 
         * 100x reduction in work!
         */
    }
    
    /**
     * Size-bounded memoization
     */
    public static class BoundedMemoizer<T, R> {
        private final Map<T, R> cache = new LinkedHashMap<T, R>(16, 0.75f, true) {
            @Override
            protected boolean removeEldestEntry(Map.Entry<T, R> eldest) {
                return size() > maxSize;
            }
        };
        private final Function<T, R> function;
        private final int maxSize;
        
        public BoundedMemoizer(Function<T, R> function, int maxSize) {
            this.function = function;
            this.maxSize = maxSize;
        }
        
        public synchronized R compute(T input) {
            return cache.computeIfAbsent(input, function);
        }
        
        /**
         * LRU EVICTION:
         * 
         * LinkedHashMap with accessOrder=true
         * Maintains access order
         * removeEldestEntry() evicts oldest
         * 
         * Prevents unbounded growth!
         * 
         * 
         * TRADE-OFF:
         * 
         * Bounded cache:
         * + No memory leak
         * + Predictable memory
         * - May evict useful entries
         * - Cache miss rate increases
         */
    }
    
    /**
     * Real-world example: Database query cache
     */
    public static class DatabaseQueryCache {
        private final Map<String, List<User>> cache = new ConcurrentHashMap<>();
        
        public List<User> getUsersByRole(String role) {
            return cache.computeIfAbsent(role, this::queryDatabase);
        }
        
        private List<User> queryDatabase(String role) {
            // Expensive database query
            // SELECT * FROM users WHERE role = ?
            System.out.println("Querying database for role: " + role);
            
            // Simulate query time
            try {
                Thread.sleep(100);  // 100ms query
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
            
            return new ArrayList<>();  // Return results
        }
        
        public void invalidate(String role) {
            cache.remove(role);
        }
        
        public void invalidateAll() {
            cache.clear();
        }
        
        /**
         * PERFORMANCE:
         * 
         * First call: 100ms (database query)
         * Subsequent calls: 0.001ms (cache hit)
         * 
         * 100,000x faster!
         * 
         * 
         * INVALIDATION:
         * 
         * When data changes:
         * - Call invalidate(role)
         * - Next call reloads from database
         * 
         * "There are only two hard things in Computer Science:
         *  cache invalidation and naming things."
         *  - Phil Karlton
         */
    }
    
    static class User {}
}
```

```java
/**

 */

import com.google.common.cache.*;
import java.util.concurrent.TimeUnit;
import java.util.concurrent.ExecutionException;

public class GuavaCacheExamples {
    
    /**
     * Basic Guava cache
     */
    public static void basicGuavaCache() {
        LoadingCache<String, String> cache = CacheBuilder.newBuilder()
            .maximumSize(1000)
            .expireAfterWrite(10, TimeUnit.MINUTES)
            .build(new CacheLoader<String, String>() {
                @Override
                public String load(String key) throws Exception {
                    return expensiveComputation(key);
                }
            });
        
        try {
            String value = cache.get("key");  // Auto-loads on miss
        } catch (ExecutionException e) {
            // Handle exception
        }
        
        /**
         * FEATURES:
         * 
         * ✓ Automatic loading
         * ✓ Size-based eviction
         * ✓ Time-based expiration
         * ✓ Refresh
         * ✓ Statistics
         * ✓ Weak/Soft references
         */
    }
    
    /**
     * Size-based eviction
     */
    public static void sizeBasedEviction() {
        LoadingCache<Integer, String> cache = CacheBuilder.newBuilder()
            .maximumSize(100)  // Max 100 entries
            .build(new CacheLoader<Integer, String>() {
                @Override
                public String load(Integer key) {
                    return "value-" + key;
                }
            });
        
        /**
         * BEHAVIOR:
         * 
         * When cache reaches 100 entries:
         * - New entry added
         * - Least recently used entry evicted
         * - LRU eviction policy
         * 
         * Prevents unbounded growth!
         */
    }
    
    /**
     * Time-based expiration
     */
    public static void timeBasedExpiration() {
        // Expire after write
        LoadingCache<String, Data> writeCache = CacheBuilder.newBuilder()
            .expireAfterWrite(10, TimeUnit.MINUTES)
            .build(new CacheLoader<String, Data>() {
                @Override
                public Data load(String key) {
                    return loadData(key);
                }
            });
        
        // Expire after access
        LoadingCache<String, Data> accessCache = CacheBuilder.newBuilder()
            .expireAfterAccess(10, TimeUnit.MINUTES)
            .build(new CacheLoader<String, Data>() {
                @Override
                public Data load(String key) {
                    return loadData(key);
                }
            });
        
        /**
         * DIFFERENCE:
         * 
         * expireAfterWrite:
         * - Entry expires 10 minutes after creation
         * - Regardless of access
         * - For time-sensitive data
         * 
         * expireAfterAccess:
         * - Entry expires 10 minutes after last access
         * - Frequently accessed stays
         * - For keeping hot data
         */
    }
    
    /**
     * Refresh
     */
    public static void refreshCache() {
        LoadingCache<String, Data> cache = CacheBuilder.newBuilder()
            .refreshAfterWrite(1, TimeUnit.MINUTES)
            .build(new CacheLoader<String, Data>() {
                @Override
                public Data load(String key) {
                    return loadData(key);
                }
            });
        
        /**
         * REFRESH VS EXPIRE:
         * 
         * Expire:
         * - Entry removed
         * - Next access blocks on load
         * - Stale data not served
         * 
         * Refresh:
         * - Entry marked stale
         * - Next access returns stale (fast!)
         * - Background refresh started
         * - Subsequent access gets fresh
         * 
         * Better user experience!
         */
    }
    
    /**
     * Statistics
     */
    public static void cacheStatistics() {
        LoadingCache<String, String> cache = CacheBuilder.newBuilder()
            .recordStats()  // Enable statistics
            .build(new CacheLoader<String, String>() {
                @Override
                public String load(String key) {
                    return key.toUpperCase();
                }
            });
        
        // Use cache
        cache.getUnchecked("hello");
        cache.getUnchecked("world");
        cache.getUnchecked("hello");  // Hit!
        
        CacheStats stats = cache.stats();
        System.out.println("Hit rate: " + stats.hitRate());
        System.out.println("Hits: " + stats.hitCount());
        System.out.println("Misses: " + stats.missCount());
        System.out.println("Load time: " + stats.averageLoadPenalty());
        
        /**
         * STATISTICS:
         * 
         * hitRate: hits / (hits + misses)
         * hitCount: Number of cache hits
         * missCount: Number of cache misses
         * loadCount: Number of loads
         * averageLoadPenalty: Average load time
         * evictionCount: Number of evictions
         * 
         * 
         * MONITORING:
         * 
         * Track hit rate over time
         * - High hit rate (> 90%): Cache working well
         * - Low hit rate (< 50%): Review cache strategy
         * 
         * Track eviction count
         * - High evictions: Increase cache size
         */
    }
    
    /**
     * Removal listener
     */
    public static void removalListener() {
        LoadingCache<String, Resource> cache = CacheBuilder.newBuilder()
            .maximumSize(100)
            .removalListener(new RemovalListener<String, Resource>() {
                @Override
                public void onRemoval(RemovalNotification<String, Resource> notification) {
                    System.out.println("Evicted: " + notification.getKey());
                    System.out.println("Cause: " + notification.getCause());
                    
                    // Cleanup resource
                    notification.getValue().close();
                }
            })
            .build(new CacheLoader<String, Resource>() {
                @Override
                public Resource load(String key) {
                    return new Resource(key);
                }
            });
        
        /**
         * REMOVAL CAUSES:
         * 
         * EXPLICIT: Manually removed (invalidate)
         * REPLACED: Value replaced
         * COLLECTED: GC'd (weak/soft reference)
         * EXPIRED: Time-based expiration
         * SIZE: Size-based eviction
         * 
         * 
         * USE CASES:
         * 
         * - Cleanup resources (close connections)
         * - Logging/metrics
         * - Cascade invalidation
         */
    }
    
    /**
     * Real-world example: API response cache
     */
    public static class ApiResponseCache {
        private final LoadingCache<String, ApiResponse> cache;
        
        public ApiResponseCache() {
            this.cache = CacheBuilder.newBuilder()
                .maximumSize(1000)
                .expireAfterWrite(5, TimeUnit.MINUTES)
                .refreshAfterWrite(1, TimeUnit.MINUTES)
                .recordStats()
                .build(new CacheLoader<String, ApiResponse>() {
                    @Override
                    public ApiResponse load(String url) throws Exception {
                        return fetchFromApi(url);
                    }
                });
        }
        
        public ApiResponse get(String url) {
            try {
                return cache.get(url);
            } catch (ExecutionException e) {
                throw new RuntimeException("Failed to load from API", e);
            }
        }
        
        public void invalidate(String url) {
            cache.invalidate(url);
        }
        
        public CacheStats getStats() {
            return cache.stats();
        }
        
        private ApiResponse fetchFromApi(String url) {
            // HTTP call
            return new ApiResponse();
        }
        
        /**
         * BENEFITS:
         * 
         * 1. Reduce API calls (cost savings)
         * 2. Faster response (cache hit ~1ms vs API ~100ms)
         * 3. Resilience (stale data during outages)
         * 4. Rate limit protection
         * 
         * 
         * CONFIGURATION:
         * 
         * - 5 min expire: Data fresh enough
         * - 1 min refresh: Background updates
         * - 1000 size: Reasonable memory
         * - Stats: Monitor effectiveness
         */
    }
    
    private static String expensiveComputation(String key) { return key; }
    private static Data loadData(String key) { return new Data(); }
    
    static class Data {}
    static class Resource {
        Resource(String key) {}
        void close() {}
    }
```

```java
        /**
         * CAFFEINE'S SECRET: TinyLFU
         * 
         * Traditional LRU:
         * - Evicts least recently used
         * - Problem: One-time spike evicts useful entries
         * 
         * Example:
         * Cache: A(100 hits), B(100 hits), C(100 hits)
         * Burst: X, Y, Z accessed once each
         * LRU evicts: A, B, C (most valuable!)
         * 
         * 
         * TinyLFU:
         * - Tracks frequency
         * - Evicts least frequently used
         * - Handles bursts better
         * 
         * Same scenario:
         * TinyLFU evicts: X, Y, Z (one-time use)
         * Keeps: A, B, C (frequently used)
         * 
         * 
         * RESULT:
         * 
         * Better hit rate!
         * Especially with varying access patterns
         * 
         * 
         * COMPARISON:
         * 
         * Guava LRU:     75% hit rate
         * Caffeine W-TinyLFU: 85% hit rate
         * 
         * 10% improvement!
         */
    }
```

## 4. Cache Eviction Policies

### Choosing the Right Strategy

```java
/**
 * CACHE EVICTION POLICIES
 */

import java.util.*;

public class CacheEvictionPolicies {
    
    /**
     * Understanding eviction policies
     */
    public static void evictionPolicies() {
        /**
         * WHY EVICTION:
         * 
         * Cache can't grow forever
         * Must remove entries when full
         * Which entries to remove?
         * 
         * 
         * COMMON POLICIES:
         * 
         * 1. LRU (Least Recently Used):
         *    - Evict oldest access
         *    - Good for temporal locality
         *    - Most common
         * 
         * 2. LFU (Least Frequently Used):
         *    - Evict lowest access count
         *    - Good for frequency patterns
         *    - Handles hot data well
         * 
         * 3. FIFO (First In First Out):
         *    - Evict oldest insertion
         *    - Simple, predictable
         *    - Rarely optimal
         * 
         * 4. TTL (Time To Live):
         *    - Evict after time expires
         *    - Good for freshness
         *    - Not size-based
         * 
         * 5. Random:
         *    - Evict random entry
         *    - Fast, simple
         *    - Surprisingly effective
         */
    }
    
    /**
     * LRU implementation
     */
    public static class LRUCache<K, V> {
        private final int capacity;
        private final Map<K, V> cache;
        
        public LRUCache(int capacity) {
            this.capacity = capacity;
            this.cache = new LinkedHashMap<K, V>(capacity, 0.75f, true) {
                @Override
                protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
                    return size() > capacity;
                }
            };
        }
        
        public synchronized V get(K key) {
            return cache.get(key);  // Moves to end
        }
        
        public synchronized void put(K key, V value) {
            cache.put(key, value);  // Moves to end
        }
        
        /**
         * LRU CHARACTERISTICS:
         * 
         * Pros:
         * ✓ Good temporal locality
         * ✓ Recent data likely reused
         * ✓ Simple to implement
         * 
         * Cons:
         * ✗ Vulnerable to scans
         * ✗ One-time burst evicts hot data
         * 
         * 
         * EXAMPLE:
         * 
         * Cache: A, B, C (capacity 3)
         * Access: A, B, C, A, B, C (working set)
         * 
         * Scan: X, Y, Z (one-time)
         * After scan: X, Y, Z in cache
         * A, B, C evicted!
         * 
         * Next access to A: Cache miss
         * LRU fails for scan pattern!
         */
    }
    
    /**
     * LFU implementation
     */
    public static class LFUCache<K, V> {
        private final int capacity;
        private final Map<K, V> values;
        private final Map<K, Integer> frequencies;
        private final Map<Integer, LinkedHashSet<K>> frequencyLists;
        private int minFrequency;
        
        public LFUCache(int capacity) {
            this.capacity = capacity;
            this.values = new HashMap<>();
            this.frequencies = new HashMap<>();
            this.frequencyLists = new HashMap<>();
            this.minFrequency = 0;
        }
        
        public V get(K key) {
            if (!values.containsKey(key)) {
                return null;
            }
            
            // Update frequency
            int freq = frequencies.get(key);
            frequencies.put(key, freq + 1);
            
            // Move to next frequency list
            frequencyLists.get(freq).remove(key);
            if (frequencyLists.get(freq).isEmpty()) {
                frequencyLists.remove(freq);
                if (minFrequency == freq) {
                    minFrequency++;
                }
            }
            
            frequencyLists.computeIfAbsent(freq + 1, k -> new LinkedHashSet<>()).add(key);
            
            return values.get(key);
        }
        
        public void put(K key, V value) {
            if (capacity <= 0) return;
            
            if (values.containsKey(key)) {
                values.put(key, value);
                get(key);  // Update frequency
                return;
            }
            
            // Evict if full
            if (values.size() >= capacity) {
                K evict = frequencyLists.get(minFrequency).iterator().next();
                frequencyLists.get(minFrequency).remove(evict);
                values.remove(evict);
                frequencies.remove(evict);
            }
            
            // Add new entry
            values.put(key, value);
            frequencies.put(key, 1);
            frequencyLists.computeIfAbsent(1, k -> new LinkedHashSet<>()).add(key);
            minFrequency = 1;
        }
        
        /**
         * LFU CHARACTERISTICS:
         * 
         * Pros:
         * ✓ Handles frequency well
         * ✓ Resistant to scans
         * ✓ Keeps hot data
         * 
         * Cons:
         * ✗ Complex implementation
         * ✗ Old frequent items hard to evict
         * ✗ Doesn't adapt to pattern changes
         * 
         * 
         * COMPARISON TO LRU:
         * 
         * Workload: Frequent access to A, B
         *          Scan X, Y, Z once
         * 
         * LRU: Evicts A, B (bad!)
         * LFU: Evicts X, Y, Z (good!)
         */
    }
    
    /**
     * TTL-based cache
     */
    public static class TTLCache<K, V> {
        private final Map<K, Entry<V>> cache = new ConcurrentHashMap<>();
        private final long ttlMillis;
        
        private static class Entry<V> {
            final V value;
            final long expireTime;
            
            Entry(V value, long expireTime) {
                this.value = value;
                this.expireTime = expireTime;
            }
            
            boolean isExpired() {
                return System.currentTimeMillis() > expireTime;
            }
        }
        
        public TTLCache(long ttlMillis) {
            this.ttlMillis = ttlMillis;
        }
        
        public V get(K key) {
            Entry<V> entry = cache.get(key);
            if (entry == null) {
                return null;
            }
            
            if (entry.isExpired()) {
                cache.remove(key);
                return null;
            }
            
            return entry.value;
        }
        
        public void put(K key, V value) {
            long expireTime = System.currentTimeMillis() + ttlMillis;
            cache.put(key, new Entry<>(value, expireTime));
        }
        
        /**
         * TTL CHARACTERISTICS:
         * 
         * Pros:
         * ✓ Time-based freshness
         * ✓ Predictable expiration
         * ✓ Good for stale-sensitive data
         * 
         * Cons:
         * ✗ Not size-bounded
         * ✗ Passive eviction (check on access)
         * ✗ Memory leak if keys not accessed
         * 
         * 
         * SOLUTION: Background cleanup
         * 
         * ScheduledExecutorService cleaner = ...;
         * cleaner.scheduleAtFixedRate(() -> {
         *   cache.entrySet().removeIf(e -> e.getValue().isExpired());
         * }, 1, 1, TimeUnit.MINUTES);
         */
    }
    
    /**
     * Policy comparison
     */
    public static void policyComparison() {
        /**
         * BENCHMARK (10,000 operations, 1000 cache size):
         * 
         * Workload: 80% repeated, 20% unique
         * 
         * LRU:     85% hit rate
         * LFU:     87% hit rate
         * FIFO:    75% hit rate
         * Random:  80% hit rate
         * 
         * 
         * Workload: Scan-heavy (bursts of unique)
         * 
         * LRU:     60% hit rate
         * LFU:     80% hit rate  (winner!)
         * FIFO:    55% hit rate
         * Random:  70% hit rate
         * 
         * 
         * Workload: Sequential access
         * 
         * LRU:     90% hit rate
         * LFU:     85% hit rate
         * FIFO:    80% hit rate
         * Random:  75% hit rate
         * 
         * 
         * CONCLUSION:
         * 
         * No silver bullet!
         * Depends on access pattern
         * 
         * General purpose: LRU or W-TinyLFU (Caffeine)
         * Scan-heavy: LFU
         * Sequential: LRU
         * Time-sensitive: TTL
         */
    }
```

```java
public class DistributedCaching {
    
    /**
     * Why distributed caching?
     */
    public static void whyDistributed() {
        /**
         * LOCAL CACHE LIMITATIONS:
         * 
         * ✗ Not shared across instances
         * ✗ Duplicated data
         * ✗ Inconsistent state
         * ✗ Limited to single JVM memory
         * 
         * 
         * DISTRIBUTED CACHE BENEFITS:
         * 
         * ✓ Shared across instances
         * ✓ Single source of truth
         * ✓ Consistent state
         * ✓ Scalable capacity
         * ✓ Survives restarts
         * 
         * 
         * TRADE-OFFS:
         * 
         * Local cache:
         * + Fast (nanoseconds)
         * + Simple
         * - Not shared
         * - Inconsistent
         * 
         * Distributed cache:
         * + Shared
         * + Consistent
         * - Slower (network roundtrip)
         * - Complex
         */
    }
    
    /**
     * Two-tier caching strategy
     */
    public static class TwoTierCache {
        private final LoadingCache<String, Data> localCache;  // L1
        private final RedisCache remoteCache;  // L2
        
        public TwoTierCache() {
            // L1: Fast, small, local
            this.localCache = Caffeine.newBuilder()
                .maximumSize(1_000)  // Small
                .expireAfterWrite(1, TimeUnit.MINUTES)  // Short TTL
                .build(this::loadFromRemote);
            
            // L2: Larger, shared, remote
            this.remoteCache = new RedisCache();
        }
        
        public Data get(String key) {
            // Try L1 (local)
            Data data = localCache.get(key);
            if (data != null) {
                return data;  // Hit in local cache
            }
            
            // Try L2 (remote)
            data = remoteCache.get(key);
            if (data != null) {
                localCache.put(key, data);  // Populate L1
                return data;
            }
            
            // Load from database
            data = loadFromDatabase(key);
            remoteCache.put(key, data);  // Populate L2
            localCache.put(key, data);   // Populate L1
            
            return data;
        }
        
        private Data loadFromRemote(String key) {
            return remoteCache.get(key);
        }
        
        private Data loadFromDatabase(String key) {
            return new Data();
        }
        
        /**
         * TWO-TIER PERFORMANCE:
         * 
         * L1 hit:  ~1 μs (local cache)
         * L2 hit:  ~1 ms (Redis)
         * Miss:    ~10 ms (database)
         * 
         * 
         * HIT RATE EXAMPLE:
         * 
         * L1: 80% hit rate
         * L2: 15% hit rate (of L1 misses)
         * Combined: 95% hit rate
         * 
         * Average latency:
         * 0.8 * 1μs + 0.15 * 1ms + 0.05 * 10ms
         * = 0.8μs + 150μs + 500μs
         * = 650.8μs
         * 
         * vs database only: 10ms
         * 
         * 15x faster!
         */
    }
    
    /**
     * Cache aside pattern
     */
    public static class CacheAsidePattern {
        private final RedisCache cache;
        
        public Data get(String key) {
            // Try cache
            Data data = cache.get(key);
            if (data != null) {
                return data;
            }
            
            // Load from database
            data = loadFromDatabase(key);
            
            // Update cache
            cache.put(key, data);
            
            return data;
        }
        
        public void update(String key, Data data) {
            // Update database
            saveToDatabase(key, data);
            
            // Invalidate cache
            cache.remove(key);
        }
        
        private Data loadFromDatabase(String key) {
            return new Data();
        }
        
        private void saveToDatabase(String key, Data data) {}
        
        /**
         * CACHE ASIDE:
         * 
         * Also called "lazy loading"
         * 
         * Read:
         * 1. Check cache
         * 2. If miss, load from database
         * 3. Update cache
         * 
         * Write:
         * 1. Write to database
         * 2. Invalidate cache
         * 
         * 
         * PROS:
         * ✓ Only caches accessed data
         * ✓ Resilient to cache failures
         * ✓ Simple
         * 
         * CONS:
         * ✗ Cache miss penalty
         * ✗ Potential stale data
         */
    }

     * Write-through pattern
     */
    public static class WriteThroughPattern {
        private final RedisCache cache;

                return data;
            }

         * Write:
         * 1. Write to database

         * 
         * 
         * PROS:
         * ✓ Cache always consistent
         * ✓ Read hits after write
         * 
         * CONS:
         * ✗ Write latency (cache + database)
         * ✗ Caches data that may not be read
```

```java
public class CacheCoherence {
    
    /**
     * The coherence problem
     */
    public static void coherenceProblem() {
        /**
         * DISTRIBUTED CACHE CHALLENGE:
         * 
         * Multiple cache instances
         * Data updated in one instance
         * Other instances have stale data!
         * 
         * 
         * EXAMPLE:
         * 
         * Server 1: local cache has User(id=1, name="Alice")
         * Server 2: local cache has User(id=1, name="Alice")
         * 
         * Request to Server 1: Update name to "Alicia"
         * - Server 1 updates database
         * - Server 1 invalidates local cache
         * 
         * Server 2 still has old data!
         * Next read from Server 2: Returns "Alice" (stale!)
         * 
         * 
         * SOLUTIONS:
         * 
         * 1. TTL (Time To Live)
         * 2. Pub/Sub invalidation
         * 3. Distributed cache (single source)
         * 4. Event sourcing
         */
    }
    
    /**
     * TTL-based coherence
     */
    public static class TTLCoherence {

        public TTLCoherence() {

                .build(this::loadFromDatabase);

         * TTL APPROACH:
         * 
         * Accept eventual consistency
         * Data stale for max TTL duration
         * 
         * Pros:
         * ✓ Simple
         * ✓ No coordination
         * 
         * Cons:
         * ✗ Bounded staleness
         * ✗ Not immediate consistency
         * 
         * 
         * WHEN TO USE:
         * 
         * ✓ Staleness acceptable (30 seconds OK)
         * ✓ Read-heavy workload
         * ✓ Simplicity important
         * 
         * Example: Product catalog, blog posts
         */
        
        private Data loadFromDatabase(String key) {
            return new Data();
        }
    }
    
    /**
     * Pub/Sub invalidation
     */
    public static class PubSubInvalidation {
        private final LoadingCache<String, Data> cache;
        private final MessageBus messageBus;
        
        public PubSubInvalidation(MessageBus messageBus) {
            this.cache = Caffeine.newBuilder()
                .maximumSize(10_000)
                .build(this::loadFromDatabase);
            
            this.messageBus = messageBus;
            
            // Subscribe to invalidation messages
            messageBus.subscribe("cache.invalidation", this::handleInvalidation);
        }
        
        public Data get(String key) {
            return cache.get(key);
        }
        
        public void update(String key, Data data) {
            // Update database
            saveToDatabase(key, data);
            
            // Invalidate local cache
            cache.invalidate(key);
            
            // Notify other instances
            messageBus.publish("cache.invalidation", new InvalidationMessage(key));
        }
        
        private void handleInvalidation(InvalidationMessage message) {
            cache.invalidate(message.key);
        }
        
        private Data loadFromDatabase(String key) {
            return new Data();
        }
        
        private void saveToDatabase(String key, Data data) {}
        
        /**
         * PUB/SUB APPROACH:
         * 
         * On update:
         * 1. Update database
         * 2. Publish invalidation message
         * 3. All instances receive message
         * 4. All instances invalidate cache
         * 
         * Pros:
         * ✓ Near-immediate consistency
         * ✓ Coordinates all instances
         * 
         * Cons:
         * ✗ Requires message bus (Redis Pub/Sub, RabbitMQ)
         * ✗ More complex
         * ✗ Network dependency
         * 
         * 
         * WHEN TO USE:
         * 
         * ✓ Staleness not acceptable
         * ✓ Multiple instances
         * ✓ Worth complexity
         * 
         * Example: User sessions, permissions
         */
    }
    
    /**
     * Version-based coherence
     */
    public static class VersionBasedCoherence {
        private final LoadingCache<String, VersionedData> cache;
        
        public VersionBasedCoherence() {
            this.cache = Caffeine.newBuilder()
                .maximumSize(10_000)
                .build(this::loadFromDatabase);
        }
        
        public Data get(String key) {
            VersionedData cached = cache.get(key);
            VersionedData current = loadFromDatabase(key);
            
            if (cached.version < current.version) {
                // Stale, update cache
                cache.put(key, current);
                return current.data;
            }
            
            return cached.data;
        }
        
        private VersionedData loadFromDatabase(String key) {
            return new VersionedData(new Data(), 1);
        }
        
        /**
         * VERSION APPROACH:
         * 
         * Each entry has version number
         * On read:
         * 1. Check cached version
         * 2. Compare with database version
         * 3. Update if stale
         * 
         * Pros:
         * ✓ Detects staleness
         * ✓ Self-healing
         * 
         * Cons:
         * ✗ Extra database query
         * ✗ Defeats cache purpose
         * 
         * 
         * OPTIMIZATION:
         * 
         * Only check version periodically
         * Or use conditional requests (ETags)
         */
    }
    
    /**
     * Cache stampede prevention
     */
    public static class StampedePrevention {
        private final LoadingCache<String, Data> cache;
        private final Map<String, CompletableFuture<Data>> loading = new ConcurrentHashMap<>();
        
        public StampedePrevention() {
            this.cache = Caffeine.newBuilder()
                .maximumSize(10_000)
                .build(this::loadFromDatabase);
        }
        
        public Data get(String key) {
            Data cached = cache.getIfPresent(key);
            if (cached != null) {
                return cached;
            }
            
            // Coordinate loading
            CompletableFuture<Data> future = loading.computeIfAbsent(key, k -> 
                CompletableFuture.supplyAsync(() -> {
                    Data data = loadFromDatabase(k);
                    cache.put(k, data);
                    loading.remove(k);
                    return data;
                })
            );
            
            return future.join();
        }
        
        private Data loadFromDatabase(String key) {
            return new Data();
        }
        
        /**
         * CACHE STAMPEDE:
         * 
         * Problem:
         * - Popular item expires
         * - 1000 requests arrive
         * - All miss cache
         * - All query database
         * - Database overwhelmed!
         * 
         * 
         * SOLUTION:
         * 
         * First request:
         * - Starts loading
         * - Others wait on same Future
         * - Only 1 database query
         * 
         * "Thundering herd" prevented!
         */
    }
```

## Summary

### Quick Reference

**Caching Strategies:**

```
1. Memoization: Simple function caching
   Map<Input, Output> cache = new ConcurrentHashMap<>();
   Result: 1,000,000x speedup for expensive functions

2. Guava Cache: Mature, feature-rich
   - Size/time-based eviction
   - Automatic loading
   - Statistics tracking
   - Refresh support

3. Caffeine: High-performance (2-3x faster)
   - W-TinyLFU algorithm (better hit rates)
   - Async support
   - Modern Java 8+ API
   → Use Caffeine for new projects!

4. Eviction Policies:
   LRU: Least recently used (most common)
   LFU: Least frequently used (scan-resistant)
   TTL: Time-based (freshness guarantee)
   W-TinyLFU: Caffeine's hybrid (best of both)

5. Cache Warming: Preload hot data
   - Avoid cold start penalty
   - Parallel warming: 10x faster startup
   - Smart warming: Use analytics

6. Distributed Caching:
   - Two-tier: Local (μs) + Remote (ms)
   - Redis, Hazelcast, Memcached
   - Cache aside vs write-through patterns

7. Cache Coherence:
   - TTL: Simple, eventual consistency
   - Pub/Sub: Near-immediate invalidation
   - Version: Self-healing
   - Stampede prevention: Coordinate loading
```

**Performance Numbers:**

```
Without cache:
  Database query: 10ms per request
  1000 req/sec: 10,000ms database time

With 90% cache hit rate:
  Cache hit: 0.001ms (10,000x faster!)
  Cache miss: 10ms
  Average: 0.9 * 0.001 + 0.1 * 10 = 1.0009ms
  1000 req/sec: 1,000ms database time

Result: 10x reduction in database load!
```

**Key insights:**

- **Caching = 10-10,000x speedup**: Database 10ms → Cache 0.001ms
- **90% hit rate = 10x reduction in load**: From 10,000ms → 1,000ms
- **Use Caffeine**: 2-3x faster than Guava, better algorithms
- **Two-tier caching wins**: Local (μs) + Redis (ms) = best of both
- **Cache warming matters**: Avoid cold start, 10x faster startup
- **Coherence is hard**: "Cache invalidation is one of two hard problems"
- **W-TinyLFU is clever**: Handles scans better than pure LRU

**The caching hierarchy:**

```
1. Local cache (fastest)
   - Nanoseconds latency
   - Not shared
   - Use for hot data

2. Distributed cache (shared)
   - Milliseconds latency
   - Shared across instances
   - Use for consistency

3. Two-tier (best of both)
   - L1: Local (1 μs)
   - L2: Redis (1 ms)
   - Result: 95% hit rate, 650 μs average

4. Database (slowest)
   - 10+ milliseconds
   - Only on cache miss
   - Protected by cache
```

**Common patterns:**

```java
// PATTERN 1: Simple memoization
Map<Input, Result> cache = new ConcurrentHashMap<>();
return cache.computeIfAbsent(input, this::expensiveComputation);

// PATTERN 2: Caffeine cache
LoadingCache<String, Data> cache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(10, TimeUnit.MINUTES)
    .recordStats()
    .build(key -> loadFromDatabase(key));

// PATTERN 3: Two-tier cache
Data data = localCache.get(key);  // Try L1
if (data == null) {
    data = redisCache.get(key);   // Try L2
    if (data != null) {
        localCache.put(key, data);  // Populate L1
    }
}

// PATTERN 4: Pub/Sub invalidation
void update(String key, Data data) {
    saveToDatabase(key, data);
    cache.invalidate(key);
    messageBus.publish("invalidate", key);  // Notify others
}

// PATTERN 5: Cache warming
void warmCache() {
    List<String> hotKeys = getHotKeys();  // Top 20% of keys
    hotKeys.parallelStream()
        .forEach(cache::get);  // Preload
}
```

**Cache sizing rules:**

```
Working set: Data actively used
Cache size: 20-50% of working set

Example:
- Working set: 100,000 items
- Cache: 20,000 items (20%)
- Result: 80-90% hit rate (80/20 rule)

If hit rate < 70%:
  → Increase cache size
  → Or improve eviction policy
  → Or analyze access patterns

If hit rate > 95%:
  → Cache too large?
  → Or perfect workload
```

**The cache invalidation problem:**

```
"There are only two hard things in Computer Science:
 cache invalidation and naming things."
 - Phil Karlton

Solutions:
1. TTL: Simple, eventual consistency (30s staleness OK)
2. Pub/Sub: Complex, near-immediate (staleness not OK)
3. Version: Self-healing, extra query
4. Event sourcing: Replay events, most consistent

No perfect solution!
Choose based on requirements.
```
