# JMH (Java Microbenchmark Harness)

## Overview

Writing accurate performance benchmarks in Java is deceptively difficult—the JVM's aggressive optimizations (JIT compilation, dead code elimination, constant folding, escape analysis) can make naive benchmarks meaningless or misleading. JMH (Java Microbenchmark Harness) is OpenJDK's official benchmarking framework that handles these complexities, providing reliable micro-benchmarks that resist JVM optimizations. This guide covers writing proper JMH benchmarks with correct annotations, understanding warmup vs measurement phases (why the JVM needs time to optimize), managing state scopes (thread-local, shared, per-iteration), preventing dead code elimination (the JVM removing your benchmark code!), using Blackhole and return values to force computation, avoiding constant folding (compile-time evaluation), interpreting benchmark results (throughput, average time, percentiles), and common pitfalls (measuring the wrong thing, insufficient warmup, improper state sharing). Whether you're optimizing algorithms, comparing implementations, or making architectural decisions, JMH provides scientifically rigorous performance data—but only if you use it correctly. This guide teaches you how.

## 2. Warmup and Measurement Iterations

### Why Warmup Matters

```java
/**
 * WARMUP VS MEASUREMENT
 */

import org.openjdk.jmh.annotations.*;
import java.util.concurrent.TimeUnit;

public class WarmupExplanation {
    
    /**
     * Understanding JIT compilation
     */
    public static void jitCompilation() {
        /**
         * JVM EXECUTION PHASES:
         * 
         * 1. INTERPRETATION (First ~1000 invocations):
         *    - Bytecode interpreted
         *    - Slow execution
         *    - Collecting profiling data
         * 
         * 2. C1 COMPILATION (Tier 1-3):
         *    - Client compiler
         *    - Quick compilation
         *    - Basic optimizations
         *    - Still profiling
         * 
         * 3. C2 COMPILATION (Tier 4):
         *    - Server compiler
         *    - Heavy optimizations
         *    - Inlining, escape analysis, etc.
         *    - Peak performance
         * 
         * 
         * WARMUP PURPOSE:
         * 
         * Get code to C2 compiled state
         * before measuring!
         * 
         * Without warmup: Measure interpretation + compilation
         * With warmup: Measure optimized code
         */
    }
    
    /**
     * No warmup - WRONG
     */
    @Benchmark
    @Warmup(iterations = 0)  // BAD!
    @Measurement(iterations = 5)
    public int noWarmup() {
        return calculateFibonacci(20);
    }
    
    /**
     * Insufficient warmup - STILL WRONG
     */
    @Benchmark
    @Warmup(iterations = 1, time = 100, timeUnit = TimeUnit.MILLISECONDS)  // BAD!
    @Measurement(iterations = 5)
    public int insufficientWarmup() {
        return calculateFibonacci(20);
    }
    
    /**
     * Proper warmup - CORRECT
     */
    @Benchmark
    @Warmup(iterations = 5, time = 1, timeUnit = TimeUnit.SECONDS)  // GOOD!
    @Measurement(iterations = 5, time = 1, timeUnit = TimeUnit.SECONDS)
    public int properWarmup() {
        return calculateFibonacci(20);
    }
    
    /**
     * Conservative warmup - VERY SAFE
     */
    @Benchmark
    @Warmup(iterations = 10, time = 1, timeUnit = TimeUnit.SECONDS)
    @Measurement(iterations = 10, time = 1, timeUnit = TimeUnit.SECONDS)
    public int conservativeWarmup() {
        return calculateFibonacci(20);
    }
    
    private int calculateFibonacci(int n) {
        if (n <= 1) return n;
        return calculateFibonacci(n - 1) + calculateFibonacci(n - 2);
    }
    
    /**
     * WARMUP GUIDELINES:
     * 
     * Minimum:
     * - 5 iterations of 1 second each
     * - Total: 5 seconds warmup
     * 
     * Recommended:
     * - 5-10 iterations of 1 second
     * - More for complex code
     * 
     * For simple operations:
     * - Can use shorter iterations
     * - But more of them
     * 
     * For complex operations:
     * - Use longer iterations
     * - Ensure full JIT optimization
     */
}
```

## 3. State Management

### Benchmark State Scopes

```java
/**
 * STATE MANAGEMENT IN JMH
 */

import org.openjdk.jmh.annotations.*;

public class StateManagement {
    
    /**
     * Thread-scoped state
     */
    @State(Scope.Thread)
    public static class ThreadState {
        int counter = 0;
        
        @Setup
        public void setup() {
            System.out.println("Thread setup: " + Thread.currentThread().getName());
            counter = 0;
        }
        
        @TearDown
        public void teardown() {
            System.out.println("Thread teardown: " + counter);
        }
    }
    
    @Benchmark
    @Threads(4)
    public int threadScopedState(ThreadState state) {
        return state.counter++;
        /**
         * THREAD SCOPE:
         * 
         * Each thread gets its own instance
         * No sharing between threads
         * No synchronization needed
         * 
         * Use for:
         * - Thread-local data
         * - No contention benchmarks
         */
    }
    
    /**
     * Benchmark-scoped state
     */
    @State(Scope.Benchmark)
    public static class BenchmarkState {
        int sharedCounter = 0;
        
        @Setup
        public void setup() {
            System.out.println("Benchmark setup");
            sharedCounter = 0;
        }
    }
    
    @Benchmark
    @Threads(4)
    public int benchmarkScopedState(BenchmarkState state) {
        return state.sharedCounter++;
        /**
         * BENCHMARK SCOPE:
         * 
         * Single instance shared by all threads
         * Measures contention
         * No synchronization by default
         * 
         * Use for:
         * - Testing contention
         * - Shared data structures
         * 
         * WARNING: Race conditions possible!
         */
    }
    
    /**
     * Group-scoped state
     */
    @State(Scope.Group)
    public static class GroupState {
        int counter = 0;
    }
    
    @Benchmark
    @Group("myGroup")
    public int groupScopedState(GroupState state) {
        return state.counter++;
        /**
         * GROUP SCOPE:
         * 
         * Shared within thread group
         * Different groups have different instances
         * 
         * Use for:
         * - Producer-consumer patterns
         * - Thread group interactions
         */
    }
}
```

```java
     * - Once per iteration (warmup or measurement)
     * - Use for: State reset between measurements
     * 
     * Invocation:
     * - Before/after EACH benchmark call
     * - Use for: Testing with fresh state every time
     * - WARNING: Very expensive!
     * 
     * 
     * RECOMMENDATION:
     * 
     * Default to Level.Trial
     * Use Level.Iteration if state must be reset
```

## 4. Avoiding Dead Code Elimination

### The JVM Will Outsmart You

```java
/**
 * DEAD CODE ELIMINATION PROBLEM
 */

import org.openjdk.jmh.annotations.*;
import org.openjdk.jmh.infra.Blackhole;

public class DeadCodeElimination {
    
    /**
     * BAD: Dead code elimination
     */
    @Benchmark
    public void deadCode() {
        int result = 1 + 1;
        // JVM sees: result is never used
        // Optimizes to: no-op
        // Benchmark measures nothing!
    }
    
    /**
     * SOLUTION 1: Return the value
     */
    @Benchmark
    public int returnValue() {
        int result = 1 + 1;
        return result;  // Forces computation
    }
    
    /**
     * SOLUTION 2: Use Blackhole
     */
    @Benchmark
    public void useBlackhole(Blackhole bh) {
        int result = 1 + 1;
        bh.consume(result);  // Prevents elimination
    }
    
    /**
     * BAD: Loop optimization
     */
    @Benchmark
    public void loopElimination() {
        int sum = 0;
        for (int i = 0; i < 1000; i++) {
            sum += i;
        }
        // JVM sees: sum never used
        // Removes entire loop!
    }
    
    /**
     * GOOD: Return loop result
     */
    @Benchmark
    public int loopWithReturn() {
        int sum = 0;
        for (int i = 0; i < 1000; i++) {
            sum += i;
        }
        return sum;
    }
    
    /**
     * Multiple calculations
     */
    @Benchmark
    public void multipleCalculationsBad() {
        int a = 1 + 1;
        int b = 2 + 2;
        int c = 3 + 3;
        // All eliminated!
    }
    
    /**
     * Multiple calculations - FIXED
     */
    @Benchmark
    public void multipleCalculationsGood(Blackhole bh) {
        int a = 1 + 1;
        int b = 2 + 2;
        int c = 3 + 3;
        bh.consume(a);
        bh.consume(b);
        bh.consume(c);
    }
    
    /**
     * DEAD CODE ELIMINATION:
     * 
     * JVM removes code that:
     * - Has no side effects
     * - Results are not used
     * - Doesn't affect program state
     * 
     * Your benchmark code might be:
     * - Completely eliminated
     * - Partially optimized away
     * - Replaced with constant
     * 
     * ALWAYS:
     * - Return results
     * - Or use Blackhole.consume()
     * - Never ignore return values
     */
}
```

## 5. Blackhole and Constant Folding

### Fighting Compiler Optimizations

```java
/**
 * BLACKHOLE EXPLAINED
 */

import org.openjdk.jmh.annotations.*;
import org.openjdk.jmh.infra.Blackhole;

public class BlackholeExplained {
    
    /**
     * What is Blackhole?
     */
    public static void blackholeDefinition() {
        /**
         * BLACKHOLE:
         * 
         * A special JMH construct that:
         * - Consumes values
         * - Prevents dead code elimination
         * - Has no measurable cost
         * - Cannot be optimized away
         * 
         * Think of it as:
         * "A sink that consumes values
         *  but the JVM can't optimize"
         * 
         * 
         * IMPLEMENTATION:
         * 
         * JMH generates special code
         * that looks like it uses the value
         * but actually does nothing
         * 
         * The JVM cannot prove
         * the value is unused
         */
    }
    
    /**
     * Using Blackhole for single value
     */
    @Benchmark
    public void singleValue(Blackhole bh) {
        int result = expensiveCalculation();
        bh.consume(result);  // Consume to prevent elimination
    }
    
    /**
     * Using Blackhole for multiple values
     */
    @Benchmark
    public void multipleValues(Blackhole bh) {
        int a = calculation1();
        int b = calculation2();
        int c = calculation3();
        
        bh.consume(a);
        bh.consume(b);
        bh.consume(c);
    }
    
    /**
     * Blackhole for objects
     */
    @Benchmark
    public void objectConsumption(Blackhole bh) {
        Object obj = new Object();
        bh.consume(obj);
    }
    
    /**
     * Return vs Blackhole
     */
    @Benchmark
    public int usingReturn() {
        return expensiveCalculation();
        // Simpler, use when possible
    }
    
    @Benchmark
    public void usingBlackhole(Blackhole bh) {
        bh.consume(expensiveCalculation());
        // Use when can't return
        // Or need to consume multiple values
    }
    
    /**
     * When to use Blackhole vs return
     */
    public static void whenToUse() {
        /**
         * USE RETURN WHEN:
         * - Single value to benchmark
         * - Simplest solution
         * - Most common case
         * 
         * USE BLACKHOLE WHEN:
         * - Multiple values to consume
         * - Side effects matter
         * - Can't return value
         * - Testing void methods
         * 
         * BOTH PREVENT:
         * - Dead code elimination
         * - Constant folding
         * - Loop optimization
         */
    }
    
    private int expensiveCalculation() { return 42; }
    private int calculation1() { return 1; }
    private int calculation2() { return 2; }
    private int calculation3() { return 3; }
}
```

```java
/**
 * CONSTANT FOLDING PROBLEM
 */

import org.openjdk.jmh.annotations.*;

public class ConstantFoldingProblem {
    
    /**
     * BAD: Constant folding
     */
    @Benchmark
    public int constantFolding() {
        return 2 + 2;
        /**
         * JVM sees: 2 + 2
         * Compiles to: return 4;
         * 
         * Benchmark measures:
         * returning a constant!
         * Not addition!
         */
    }
    
    /**
     * BAD: Computed constant
     */
    @Benchmark
    public int computedConstant() {
        int result = 0;
        for (int i = 0; i < 100; i++) {
            result += i;
        }
        return result;
        /**
         * JVM can:
         * - Compute loop at compile time
         * - Replace with constant 4950
         * - Benchmark measures nothing!
         */
    }
    
    /**
     * State to prevent constant folding
     */
    @State(Scope.Thread)
    public static class MyState {
        @Param({"10", "100", "1000"})
        int size;
        
        int[] data;
        
        @Setup
        public void setup() {
            data = new int[size];
            for (int i = 0; i < size; i++) {
                data[i] = i;
            }
        }
    }
    
    /**
     * GOOD: Using state
     */
    @Benchmark
    public int withState(MyState state) {
        int sum = 0;
        for (int i = 0; i < state.size; i++) {
            sum += i;
        }
        return sum;
        /**
         * state.size not known at compile time
         * JVM cannot constant-fold
         * Measures actual loop
         */
    }
    
    /**
     * GOOD: Using @Param
     */
    @State(Scope.Thread)
    public static class ParamState {
        @Param({"2", "4", "8"})
        int value;
    }
    
    @Benchmark
    public int withParam(ParamState state) {
        return state.value + state.value;
        /**
         * @Param values injected at runtime
         * Cannot be constant-folded
         * Benchmark runs for each parameter value
         */
    }
    
    /**
     * CONSTANT FOLDING PREVENTION:
     * 
     * 1. Use @State with @Param
     *    - Values from outside
     *    - Not compile-time constants
     * 
     * 2. Read from state fields
     *    - JVM can't know values
     *    - Forces actual computation
     * 
     * 3. Use @Setup to initialize
     *    - Data prepared at runtime
     *    - Not compile-time
     * 
     * NEVER:
     * - Use literal constants
     * - Compute from literals
     * - Use static final with literals
     */
}
```

## 6. Interpreting Results

### Understanding Benchmark Output

```java
/**
 * READING JMH OUTPUT
 */

public class InterpretingResults {
    
    /**
     * Example output
     */
    public static void exampleOutput() {
        /**
         * SAMPLE OUTPUT:
         * 
         * Benchmark                           Mode  Cnt     Score     Error  Units
         * MyBenchmark.testMethod             thrpt    5  1234.567 ±  12.345  ops/s
         * 
         * 
         * COLUMNS EXPLAINED:
         * 
         * Benchmark: Method name
         * Mode: Measurement mode (thrpt = throughput)
         * Cnt: Number of measurements
         * Score: Average result
         * Error: Confidence interval (95%)
         * Units: Result unit
         * 
         * 
         * INTERPRETATION:
         * 
         * Score: 1234.567 ops/s
         * Error: ± 12.345 ops/s
         * 
         * Means:
         * 95% confident true value is
         * between 1222.222 and 1246.912 ops/s
         */
    }
    
    /**
     * Comparing results
     */
    public static void comparingResults() {
        /**
         * COMPARING BENCHMARKS:
         * 
         * Benchmark A: 1000 ± 10 ops/s
         * Benchmark B: 1100 ± 10 ops/s
         * 
         * Difference: 100 ops/s (10%)
         * 
         * Is it significant?
         * 
         * Check if confidence intervals overlap:
         * A: [990, 1010]
         * B: [1090, 1110]
         * 
         * No overlap → Significant difference!
         * 
         * 
         * OVERLAP CASE:
         * 
         * Benchmark A: 1000 ± 50 ops/s [950, 1050]
         * Benchmark B: 1040 ± 50 ops/s [990, 1090]
         * 
         * Overlap exists → Not confident!
         * Need more measurements or bigger difference
         * 
         * 
         * RULE OF THUMB:
         * 
         * Difference > 2 × (errorA + errorB)
         * → Likely significant
         */
    }
    
    /**
     * Score interpretation by mode
     */
    public static void scoreInterpretation() {
        /**
         * THROUGHPUT (ops/time):
         * 1000 ops/s → 1000 operations per second
         * Higher is better
         * Common for comparing implementations
         * 
         * AVERAGE TIME (time/op):
         * 100 ns/op → 100 nanoseconds per operation
         * Lower is better
         * Good for understanding cost
         * 
         * SAMPLE TIME (percentiles):
         * p50: 100 ns → 50% of ops take ≤ 100ns
         * p95: 150 ns → 95% of ops take ≤ 150ns
         * p99: 200 ns → 99% of ops take ≤ 200ns
         * Shows latency distribution
         * 
         * SINGLE SHOT:
         * One-time execution
         * No warmup
         * Tests cold start
         */
    }
    
    /**
     * Statistical significance
     */
    public static void statisticalSignificance() {
        /**
         * ERROR MARGINS:
         * 
         * Small error (< 5%): Very confident
         * Medium error (5-10%): Acceptable
         * Large error (> 10%): Need more iterations
         * 
         * 
         * REDUCING ERROR:
         * 
         * 1. More iterations
         * 2. More forks
         * 3. Longer measurement time
         * 4. Reduce environmental noise
         * 
         * 
         * VARIANCE:
         * 
         * High variance means:
         * - Inconsistent performance
         * - External factors
         * - GC interference
         * - OS scheduling
         * 
         * Solutions:
         * - More warmup
         * - Longer iterations
         * - Check GC logs
         * - Reduce system load
         */
    }
```

## 7. Common Pitfalls

### What Can Go Wrong

```java
/**
 * COMMON JMH PITFALLS
 */

import org.openjdk.jmh.annotations.*;
import org.openjdk.jmh.infra.Blackhole;

public class CommonPitfalls {
    
    /**
     * PITFALL 1: Measuring the wrong thing
     */
    @Benchmark
    public void measuringCreation() {
        List<Integer> list = new ArrayList<>();
        // Benchmarks list creation, not usage!
    }
    
    @Benchmark
    public int measuringWrongThing() {
        List<Integer> list = new ArrayList<>();
        list.add(1);
        list.add(2);
        return list.size();
        // Benchmarks: creation + add + size
        // But wanted: just add performance
    }
    
    // CORRECT: Separate setup from measurement
    @State(Scope.Thread)
    public static class ListState {
        List<Integer> list;
        
        @Setup(Level.Iteration)
        public void setup() {
            list = new ArrayList<>();  // Not measured
        }
    }
    
    @Benchmark
    public void correctMeasurement(ListState state) {
        state.list.add(1);  // Only this is measured
    }
    
    /**
     * PITFALL 2: Shared mutable state
     */
    @State(Scope.Benchmark)  // WRONG for concurrent test
    public static class SharedState {
        int counter = 0;
    }
    
    @Benchmark
    @Threads(4)
    public void racyBenchmark(SharedState state) {
        state.counter++;  // Race condition!
        // Results unreliable
        // Lost updates
    }
    
    // CORRECT: Use Thread scope or synchronization
    @State(Scope.Thread)  // Each thread has own instance
    public static class ThreadLocalState {
        int counter = 0;
    }
    
    @Benchmark
    @Threads(4)
    public void correctConcurrency(ThreadLocalState state) {
        state.counter++;  // No race
    }
    
    /**
     * PITFALL 3: Not consuming results
     */
    @Benchmark
    public void forgotToConsume() {
        int result = expensiveComputation();
        // Oops! Result unused
        // Code eliminated
        // Measures nothing
    }
    
    @Benchmark
    public int correctConsumption() {
        return expensiveComputation();  // Fixed
    }
    
    /**
     * PITFALL 4: Loop in benchmark
     */
    @Benchmark
    public int loopInBenchmark() {
        int sum = 0;
        for (int i = 0; i < 1000; i++) {
            sum += i;
        }
        return sum;
        /**
         * PROBLEM:
         * 
         * Benchmarks: entire loop
         * Can't measure single iteration
         * 
         * If loop is optimized away:
         * Measures wrong thing!
         */
    }
    
    // CORRECT: Let JMH handle iterations
    @State(Scope.Thread)
    public static class IterationState {
        @Param({"0", "1", "2", "999"})
        int i;
    }
    
    @Benchmark
    public int singleIteration(IterationState state) {
        return state.i;  // JMH iterates
    }
    
    /**
     * PITFALL 5: Too short warmup
     */
    @Benchmark
    @Warmup(iterations = 1, time = 100, timeUnit = TimeUnit.MILLISECONDS)
    public int insufficientWarmup() {
        return fibonacci(30);
        // Not enough time for JIT optimization
    }
    
    @Benchmark
    @Warmup(iterations = 5, time = 1, timeUnit = TimeUnit.SECONDS)
    public int properWarmup() {
        return fibonacci(30);  // Much better
    }
    
    /**
     * PITFALL 6: Environmental noise
     */
    public static void environmentalNoise() {
        /**
         * SOURCES OF NOISE:
         * 
         * 1. Other processes:
         *    - Background apps
         *    - OS tasks
         *    - Antivirus
         * 
         * 2. Power management:
         *    - CPU throttling
         *    - Dynamic frequency
         * 
         * 3. Thermal throttling:
         *    - CPU overheating
         *    - Slowing down
         * 
         * 4. Network activity:
         *    - Interrupts
         *    - Context switches
         * 
         * 
         * MITIGATION:
         * 
         * 1. Close other apps
         * 2. Disable CPU frequency scaling
         * 3. Run multiple times
         * 4. Use dedicated machine
         * 5. Check consistency
         */
    }
    
    /**
     * PITFALL 7: Comparing incomparable
     */
    public static void comparingIncomparable() {
        /**
         * DON'T COMPARE:
         * 
         * 1. Different machines:
         *    - Different CPUs
         *    - Different RAM
         *    - Results meaningless
         * 
         * 2. Different JVMs:
         *    - HotSpot vs GraalVM
         *    - Different optimizations
         * 
         * 3. Different configurations:
         *    - Different heap sizes
         *    - Different GC
         * 
         * 
         * ALWAYS:
         * - Same machine
         * - Same JVM
         * - Same configuration
         * - Run all benchmarks together
         */
    }
    
    /**
     * PITFALL 8: Ignoring GC impact
     */
    @Benchmark
    public String ignoringGC() {
        return new String("test");  // Allocates
        /**
         * PROBLEM:
         * 
         * Allocations trigger GC
         * GC pauses affect results
         * Results include GC time
         * 
         * SOLUTION:
         * 
         * 1. Use -gc profiler
         * 2. Monitor GC frequency
         * 3. Increase heap if needed
         * 4. Use -XX:+UseG1GC for predictability
         */
    }
    
    /**
     * PITFALL 9: Method inlining issues
     */
    @Benchmark
    public int methodTooLarge() {
        // Very large method
        // May not be inlined
        // Different optimization than production
        return hugeMethod();
    }
    
    private int hugeMethod() {
        // 100+ lines of code
        // JVM won't inline
        return 42;
    }
    
    /**
     * PITFALL 10: Not using @Param
     */
    @Benchmark
    public int hardcodedSize() {
        int[] array = new int[1000];  // Constant!
        return array.length;
        // Only tests one size
    }
    
    @State(Scope.Thread)
    public static class ParamState {
        @Param({"100", "1000", "10000"})
        int size;
        
        int[] array;
        
        @Setup
        public void setup() {
            array = new int[size];
        }
    }
    
    @Benchmark
    public int withParam(ParamState state) {
        return state.array.length;
        // Tests multiple sizes!
    }
    
    private int expensiveComputation() { return 42; }
    private int fibonacci(int n) {
        if (n <= 1) return n;
        return fibonacci(n-1) + fibonacci(n-2);
    }
}
```

### Quick Reference

**Basic Benchmark:**


**With State:**

```java
    public void setup() {
        // Initialize
    }

}
```

**Avoiding Pitfalls:**

```java
// ✓ Return result or use Blackhole
@Benchmark
public int correct() {
    return compute();  // Prevents DCE
}

@Benchmark
public void alsoCorrect(Blackhole bh) {
    bh.consume(compute());  // Prevents DCE
}

// ✗ Don't ignore result
@Benchmark
public void wrong() {
    compute();  // Dead code!
}
```

---


**Key insights:**

- **JIT warmup critical**: Without it, you measure interpretation!
- **Dead code elimination**: JVM removes unused code
- **Return or consume**: Always handle benchmark results
- **State prevents constants**: Use @Param for runtime values
- **Multiple forks**: Reduces variance, increases confidence
- **Error margins matter**: Overlapping intervals = not significant

**The benchmarking revelation:**

```
Naive Benchmark:
public void test() {
    int x = 2 + 2;
}
→ JVM: "x unused, 2+2 is constant"
→ Result: Measures nothing!

Proper Benchmark:
@Benchmark
public int test() {
    return 2 + 2;
}
→ JVM: "Must compute and return"
→ Still problematic (constant folding)

Correct Benchmark:
@State(Scope.Thread)
public static class MyState {
    @Param({"2"}) int value;
}

@Benchmark
public int test(MyState s) {
    return s.value + s.value;
}
→ JVM: "Runtime value, must compute"
→ Measures actual addition!

Result: Accurate performance data!
```

**The JMH mantra:**

```
1. Warm up (5+ iterations)
2. Measure (5+ iterations)
3. Fork (2+ times)
4. Consume results (return or Blackhole)
5. Use state (@Param prevents constants)
6. Check errors (< 10% is good)
7. Compare confidence intervals

Follow these → Reliable benchmarks!
Ignore these → Meaningless numbers!
```
