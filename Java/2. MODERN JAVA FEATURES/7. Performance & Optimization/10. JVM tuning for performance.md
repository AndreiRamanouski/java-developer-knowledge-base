# JVM Tuning for Performance

## Overview

JVM tuning is the art of configuring Java Virtual Machine parameters to optimize application performance—proper tuning can transform an application from struggling with long GC pauses (seconds) to smooth sub-millisecond pauses, from high memory usage (gigabytes wasted) to efficient allocation, and from slow startup (minutes) to fast warmup (seconds). The JVM's adaptive nature means it performs well by default for many workloads, but production systems often benefit from tuning: heap sizing prevents OutOfMemoryErrors while minimizing GC overhead, GC selection matches workload characteristics (low latency vs high throughput), JIT compiler optimizations enable native-speed execution through inlining and escape analysis, and proper monitoring reveals bottlenecks invisible in code. This guide covers JVM performance tuning: heap sizing strategies (initial/max heap, generation ratios, survivor spaces using formulas and measurement), GC selection and tuning (G1GC for general use, ZGC for low latency, Parallel GC for throughput, tuning flags), JIT compiler optimization (tiered compilation, C1/C2 compilers, CompileThreshold, inlining thresholds), compressed oops (memory savings on 64-bit JVMs), escape analysis (stack allocation and lock elision), inline optimization (monomorphic vs megamorphic call sites), and monitoring/measuring impact (Java Flight Recorder, GC logs, metrics). Whether running microservices, data processing pipelines, or high-frequency trading systems, understanding JVM tuning enables you to extract maximum performance from your hardware.

### Finding the Right Balance

```java
/**
 * HEAP SIZING STRATEGIES
 */

public class HeapSizing {
    
    /**
     * Understanding heap structure
     */
    public static void heapStructure() {
        /**
         * JVM HEAP LAYOUT:
         * 
         * ┌─────────────────────────────────────┐
         * │         Java Heap                   │
         * ├─────────────────┬───────────────────┤
         * │  Young Gen      │  Old Gen          │
         * │  (1/3 of heap)  │  (2/3 of heap)    │
         * ├────────┬────────┤                   │
         * │ Eden   │Survivor│                   │
         * │ (80%)  │ (20%)  │                   │
         * │        ├────┬───┤                   │
         * │        │ S0 │S1 │                   │
         * │        │(10)│(10)│                  │
         * └────────┴────┴───┴───────────────────┘
         * 
         * 
         * OBJECT LIFECYCLE:
         * 
         * 1. Object allocated in Eden
         * 2. Eden fills → Minor GC
         * 3. Live objects → Survivor (S0)
         * 4. Next Minor GC → Move to S1
         * 5. After N GCs → Promoted to Old Gen
         * 6. Old Gen fills → Major/Full GC
         * 
         * 
         * KEY METRICS:
         * 
         * - Allocation rate: Objects/second allocated
         * - Promotion rate: Objects/second promoted to Old
         * - GC frequency: How often GC runs
         * - GC duration: How long GC takes
         */
    }
    
    /**
     * Basic heap sizing flags
     */
    public static void basicFlags() {
        /**
         * ESSENTIAL FLAGS:
         * 
         * -Xms<size>: Initial heap size
         * -Xmx<size>: Maximum heap size
         * 
         * 
         * RECOMMENDATION: SET EQUAL!
         * 
         * -Xms4g -Xmx4g
         * 
         * Why?
         * - Avoids heap resizing overhead
         * - Predictable memory footprint
         * - Better startup performance
         * - OS reserves memory upfront
         * 
         * 
         * BAD:
         * -Xms512m -Xmx4g
         * 
         * Problems:
         * - JVM grows heap dynamically
         * - Resizing causes pauses
         * - Unpredictable memory usage
         * - OS may not have memory when needed
         */
    }
    
    /**
     * Sizing young generation
     */
    public static void youngGenSizing() {
        /**
         * YOUNG GENERATION FLAGS:
         * 
         * -XX:NewRatio=<ratio>
         * - Ratio of Young:Old
         * - Default: 2 (Young = 1/3, Old = 2/3)
         * - Lower = Larger young gen
         * 
         * -XX:NewSize=<size>
         * - Initial young generation size
         * 
         * -XX:MaxNewSize=<size>
         * - Maximum young generation size
         * 
         * -Xmn<size>
         * - Sets both NewSize and MaxNewSize
         * - Recommended for fixed size
         * 
         * 
         * SIZING GUIDELINES:
         * 
         * High allocation rate:
         * → Larger young gen
         * → Reduces Minor GC frequency
         * → Example: -Xmn2g for -Xmx6g
         * 
         * Low allocation rate:
         * → Smaller young gen
         * → More heap for Old gen
         * → Example: -Xmn512m for -Xmx4g
         * 
         * 
         * MEASUREMENT:
         * 
         * 1. Monitor allocation rate
         * 2. Monitor Minor GC frequency
         * 3. Target: Minor GC every 1-5 seconds
         * 4. Adjust young gen size accordingly
         * 
         * 
         * EXAMPLE CALCULATION:
         * 
         * Allocation rate: 1 GB/second
         * Target Minor GC interval: 2 seconds
         * Young gen size: 1 GB/s * 2s = 2 GB
         * 
         * Plus buffer: 2 GB * 1.5 = 3 GB
         */
    }
    
    /**
     * Survivor space sizing
     */
    public static void survivorSizing() {
        /**
         * SURVIVOR FLAGS:
         * 
         * -XX:SurvivorRatio=<ratio>
         * - Ratio of Eden:Survivor
         * - Default: 8 (Eden = 80%, Survivor = 20%)
         * - Each Survivor = 10%
         * 
         * 
         * SIZING PROBLEM:
         * 
         * Too small:
         * - Survivors overflow
         * - Premature promotion to Old
         * - Frequent Full GCs
         * 
         * Too large:
         * - Wastes Eden space
         * - More frequent Minor GCs
         * 
         * 
         * OPTIMAL SIZING:
         * 
         * Survivor should hold:
         * - Objects that survive one GC
         * - But die before promotion
         * 
         * Rule of thumb:
         * - Survivor = 10-15% of Young Gen
         * - Default ratio (8) usually good
         * 
         * 
         * MONITORING:
         * 
         * Watch for:
         * - "promotion failure" in logs
         * - High promotion rate
         * - Objects promoted too young
         * 
         * Solution:
         * - Increase survivor space
         * - Or increase tenuring threshold
         */
    }
    
    /**
     * Heap sizing formula
     */
    public static void heapSizingFormula() {
        /**
         * SIZING FORMULA:
         * 
         * heap_size = live_data * (1 + allocation_rate * gc_interval) * safety_factor
         * 
         * Where:
         * - live_data: Long-lived objects in memory
         * - allocation_rate: New objects/second
         * - gc_interval: Target time between GCs
         * - safety_factor: Buffer (1.5-2.0)
         * 
         * 
         * EXAMPLE:
         * 
         * Live data: 500 MB
         * Allocation rate: 100 MB/s
         * Target GC interval: 5 seconds
         * Safety factor: 2.0
         * 
         * heap_size = 500 * (1 + 100/1000 * 5) * 2
         *          = 500 * (1 + 0.5) * 2
         *          = 500 * 1.5 * 2
         *          = 1500 MB
         * 
         * Round up: 2 GB
         * 
         * 
         * VERIFICATION:
         * 
         * 1. Set heap to calculated size
         * 2. Run application under load
         * 3. Monitor:
         *    - GC frequency (should match target)
         *    - GC duration (should be acceptable)
         *    - Heap usage after GC (should be ~50%)
         * 4. Adjust if needed
         */
    }
    
    /**
     * Real-world sizing example
     */
    public static class RealWorldSizing {
        /**
         * SCENARIO:
         * 
         * Application: E-commerce backend
         * Traffic: 1000 req/sec
         * Request processing: 100ms average
         * Object allocation per request: 1 MB
         * Live data: 1 GB (caches, connections, etc.)
         * 
         * 
         * CALCULATION:
         * 
         * Allocation rate:
         * - 1000 req/sec * 1 MB = 1000 MB/sec = 1 GB/sec
         * 
         * Target Minor GC interval: 5 seconds
         * 
         * Young gen size:
         * - 1 GB/sec * 5 sec = 5 GB
         * - With buffer: 5 GB * 1.5 = 7.5 GB
         * 
         * Old gen size:
         * - Live data + buffer
         * - 1 GB * 2 = 2 GB
         * 
         * Total heap:
         * - Young: 7.5 GB
         * - Old: 2 GB
         * - Total: 10 GB
         * 
         * 
         * FLAGS:
         * 
         * -Xms10g -Xmx10g -Xmn7680m
         * 
         * 
         * EXPECTED BEHAVIOR:
         * 
         * - Minor GC every ~5 seconds
         * - Minor GC duration: 10-50ms
         * - Full GC: Rare (hours/days)
         * - Heap usage after GC: ~1 GB (10%)
         */
    }
    
    /**
     * Container-aware sizing
     */
    public static void containerSizing() {
        /**
         * CONTAINER MEMORY LIMITS:
         * 
         * Docker/Kubernetes:
         * - Container has memory limit
         * - JVM must stay within limit
         * - OOM kill if exceeded
         * 
         * 
         * SIZING FORMULA:
         * 
         * container_memory = heap + metaspace + threads + native + buffer
         * 
         * Example: 2 GB container
         * - Heap: 1.5 GB (75%)
         * - Metaspace: 256 MB (13%)
         * - Threads: 100 * 1 MB = 100 MB (5%)
         * - Native: 50 MB (2%)
         * - Buffer: 100 MB (5%)
         * - Total: 2 GB
         * 
         * 
         * CONSERVATIVE APPROACH:
         * 
         * Heap = container_memory * 0.75
         * 
         * 2 GB container → -Xmx1536m
         * 4 GB container → -Xmx3g
         * 8 GB container → -Xmx6g
         * 
         * 
         * IMPORTANT FLAGS:
         * 
         * -XX:+UseContainerSupport (default since Java 10)
         * - Auto-detects container limits
         * - Sets heap based on container memory
         * 
         * -XX:MaxRAMPercentage=75.0
         * - Heap = 75% of container memory
         * - Safer than -Xmx
         */
    }
}
```

## 2. GC Selection and Tuning

### Choosing the Right Garbage Collector

```java
/**
 * GC SELECTION AND TUNING
 */

public class GCSelection {
    
    /**
     * GC comparison
     */
    public static void gcComparison() {
        /**
         * GARBAGE COLLECTORS:
         * 
         * ┌─────────────┬─────────────┬─────────────┬──────────────┐
         * │ GC          │ Pause Time  │ Throughput  │ Memory       │
         * ├─────────────┼─────────────┼─────────────┼──────────────┤
         * │ Serial      │ High (s)    │ Low         │ Low          │
         * │ Parallel    │ High (s)    │ High        │ Low          │
         * │ G1GC        │ Medium (ms) │ Medium      │ Medium       │
         * │ ZGC         │ Very Low(μs)│ Medium      │ High         │
         * │ Shenandoah  │ Very Low(μs)│ Medium      │ High         │
         * └─────────────┴─────────────┴─────────────┴──────────────┘
         * 
         * 
         * WHEN TO USE:
         * 
         * Serial (-XX:+UseSerialGC):
         * - Single core
         * - Small heaps (< 100 MB)
         * - Client applications
         * - Not for production servers
         * 
         * Parallel (-XX:+UseParallelGC):
         * - High throughput priority
         * - Batch processing
         * - Can tolerate pauses (seconds)
         * - Multi-core machines
         * 
         * G1GC (-XX:+UseG1GC) [DEFAULT]:
         * - Balanced latency/throughput
         * - Large heaps (> 4 GB)
         * - Predictable pause times
         * - General purpose
         * 
         * ZGC (-XX:+UseZGC):
         * - Ultra-low latency (< 1ms)
         * - Huge heaps (TB+)
         * - Java 15+
         * - Requires tuning
         * 
         * Shenandoah (-XX:+UseShenandoahGC):
         * - Low latency
         * - Large heaps
         * - Similar to ZGC
         * - Red Hat developed
         */
    }
    
    /**
     * G1GC tuning
     */
    public static void g1gcTuning() {
        /**
         * G1GC OVERVIEW:
         * 
         * - Region-based collector
         * - Heap divided into 2048 regions
         * - Collects regions with most garbage first
         * - Predictable pause times
         * 
         * 
         * KEY FLAGS:
         * 
         * -XX:MaxGCPauseMillis=<millis>
         * - Target pause time
         * - Default: 200ms
         * - G1 tries to meet this goal
         * - Trade-off: Lower target = less throughput
         * 
         * -XX:G1HeapRegionSize=<size>
         * - Size of each region
         * - Must be power of 2 (1, 2, 4, 8, 16, 32 MB)
         * - Auto-calculated by default
         * - Formula: heap_size / 2048
         * 
         * -XX:InitiatingHeapOccupancyPercent=<percent>
         * - Heap occupancy that triggers marking cycle
         * - Default: 45%
         * - Lower: More frequent GC, less pause time
         * - Higher: Less frequent GC, longer pause time
         * 
         * 
         * TUNING EXAMPLE:
         * 
         * Low latency:
         * -XX:+UseG1GC
         * -XX:MaxGCPauseMillis=50
         * -XX:G1HeapRegionSize=16m
         * -XX:InitiatingHeapOccupancyPercent=40
         * 
         * High throughput:
         * -XX:+UseG1GC
         * -XX:MaxGCPauseMillis=500
         * -XX:G1HeapRegionSize=32m
         * -XX:InitiatingHeapOccupancyPercent=60
         * 
         * 
         * MONITORING G1:
         * 
         * Watch for:
         * - Actual pause times vs target
         * - Mixed GC frequency
         * - Evacuation failures
         * - Humongous object allocations
         * 
         * 
         * HUMONGOUS OBJECTS:
         * 
         * Objects > 50% of region size
         * - Allocated directly in Old Gen
         * - Can cause performance issues
         * 
         * Solution:
         * - Increase region size
         * - Reduce object size
         * - Use object pooling
         */
    }
    
    /**
     * ZGC tuning
     */
    public static void zgcTuning() {
        /**
         * ZGC OVERVIEW:
         * 
         * - Scalable low-latency GC
         * - Pause times < 1ms (even for TB heaps!)
         * - Concurrent marking and compaction
         * - Java 15+ production-ready
         * 
         * 
         * KEY FLAGS:
         * 
         * -XX:+UseZGC
         * - Enable ZGC
         * 
         * -XX:ConcGCThreads=<threads>
         * - Concurrent GC threads
         * - Default: 25% of cores
         * - Increase for faster GC
         * 
         * -XX:ZAllocationSpikeTolerance=<factor>
         * - Allocation spike tolerance
         * - Default: 2.0
         * - Higher: More resilient to spikes
         * 
         * 
         * EXAMPLE CONFIGURATION:
         * 
         * -XX:+UseZGC
         * -Xms16g -Xmx16g
         * -XX:ConcGCThreads=4
         * 
         * 
         * WHEN TO USE ZGC:
         * 
         * ✓ Large heaps (> 16 GB)
         * ✓ Ultra-low latency required (< 1ms)
         * ✓ Can spare extra memory (10-20% overhead)
         * ✓ Java 15+
         * 
         * ✗ Small heaps (< 4 GB)
         * ✗ Memory constrained
         * ✗ Throughput priority
         * 
         * 
         * ZGC BENEFITS:
         * 
         * Traditional GC:
         * - 4 GB heap: 100ms pause
         * - 16 GB heap: 500ms pause
         * - 64 GB heap: 2000ms pause (2 seconds!)
         * 
         * ZGC:
         * - 4 GB heap: < 1ms pause
         * - 16 GB heap: < 1ms pause
         * - 64 GB heap: < 1ms pause
         * - 1 TB heap: < 1ms pause!
         * 
         * Consistent sub-millisecond pauses!
         */
    }
    
    /**
     * Parallel GC tuning
     */
    public static void parallelGCTuning() {
        /**
         * PARALLEL GC:
         * 
         * - Throughput-oriented
         * - Multi-threaded Minor GC
         * - Multi-threaded Major GC
         * - Can tolerate longer pauses
         * 
         * 
         * KEY FLAGS:
         * 
         * -XX:+UseParallelGC
         * - Enable Parallel GC
         * 
         * -XX:ParallelGCThreads=<threads>
         * - Number of GC threads
         * - Default: Number of cores
         * - More threads = faster GC = longer pauses
         * 
         * -XX:MaxGCPauseMillis=<millis>
         * - Target pause time
         * - GC tries to meet (best effort)
         * 
         * -XX:GCTimeRatio=<ratio>
         * - Throughput goal
         * - Default: 99 (1% time in GC)
         * - Higher: More throughput, longer pauses
         * 
         * 
         * EXAMPLE:
         * 
         * -XX:+UseParallelGC
         * -XX:ParallelGCThreads=8
         * -XX:GCTimeRatio=99
         * 
         * 
         * WHEN TO USE:
         * 
         * Batch processing:
         * - High throughput needed
         * - Pauses acceptable
         * - No real-time requirements
         * 
         * 
         * PERFORMANCE:
         * 
         * Parallel GC:
         * - Throughput: 99%+ (< 1% in GC)
         * - Pause time: 100-5000ms
         * - Best for: Batch jobs
         * 
         * G1GC:
         * - Throughput: 90-95%
         * - Pause time: 10-200ms
         * - Best for: Interactive apps
         * 
         * ZGC:
         * - Throughput: 85-90%
         * - Pause time: < 1ms
         * - Best for: Low latency apps
         */
    }
    
    /**
     * GC logging
     */
    public static void gcLogging() {
        /**
         * ESSENTIAL GC LOGS:
         * 
         * Java 8:
         * -XX:+PrintGCDetails
         * -XX:+PrintGCDateStamps
         * -Xloggc:/path/to/gc.log
         * 
         * Java 9+:
         * -Xlog:gc*:file=/path/to/gc.log:time,uptime,level,tags
         * 
         * 
         * ANALYZING GC LOGS:
         * 
         * Key metrics:
         * - GC frequency
         * - GC duration
         * - Heap usage before/after
         * - Promotion rate
         * - Allocation rate
         * 
         * 
         * TOOLS:
         * 
         * - GCViewer
         * - GCEasy (gceasy.io)
         * - GC log analyzers
         * 
         * 
         * RED FLAGS:
         * 
         * - Frequent Full GCs
         * - Long pause times
         * - Heap usage not dropping after GC
         * - High promotion rate
         * - Allocation failures
         */
    }
}
```

## 3. JIT Compiler Optimization

### Making Code Run at Native Speed

```java
/**
 * JIT COMPILER OPTIMIZATION
 */

public class JITOptimization {
    
    /**
     * Understanding JIT
     */
    public static void understandingJIT() {
        /**
         * JIT COMPILER:
         * 
         * Java bytecode → JIT → Native machine code
         * 
         * 
         * TWO COMPILERS:
         * 
         * C1 (Client Compiler):
         * - Fast compilation
         * - Simple optimizations
         * - Low overhead
         * - For quick startup
         * 
         * C2 (Server Compiler):
         * - Slow compilation
         * - Aggressive optimizations
         * - High overhead
         * - For peak performance
         * 
         * 
         * TIERED COMPILATION (Default):
         * 
         * Level 0: Interpreter
         * Level 1: C1 (no profiling)
         * Level 2: C1 (invocation counting)
         * Level 3: C1 (full profiling)
         * Level 4: C2 (aggressive optimization)
         * 
         * 
         * WORKFLOW:
         * 
         * 1. Method called → Interpreter (Level 0)
         * 2. Call count reaches threshold → C1 (Level 3)
         * 3. More calls + profiling data → C2 (Level 4)
         * 
         * Result: Fast startup + peak performance
         */
    }
    
    /**
     * JIT thresholds
     */
    public static void jitThresholds() {
        /**
         * COMPILATION THRESHOLDS:
         * 
         * -XX:CompileThreshold=<count>
         * - Invocations before compilation
         * - Default: 10,000 (C2)
         * - Lower: Faster warmup, more CPU
         * - Higher: Slower warmup, less CPU
         * 
         * 
         * TIERED THRESHOLDS:
         * 
         * -XX:Tier3InvocationThreshold=<count>
         * - Invocations for C1 compilation
         * - Default: 200
         * 
         * -XX:Tier4InvocationThreshold=<count>
         * - Invocations for C2 compilation
         * - Default: 5,000
         * 
         * 
         * EXAMPLE: Faster warmup
         * 
         * -XX:CompileThreshold=1000
         * -XX:Tier3InvocationThreshold=100
         * -XX:Tier4InvocationThreshold=2000
         * 
         * Result:
         * - Methods compile faster
         * - Reach peak performance sooner
         * - More CPU during warmup
         * 
         * 
         * WHEN TO LOWER THRESHOLDS:
         * 
         * ✓ Microservices (short-lived)
         * ✓ Function-as-a-Service
         * ✓ Frequent deployments
         * ✓ Startup time critical
         * 
         * ✗ Long-running applications
         * ✗ CPU-constrained
         */
    }
    
    /**
     * Method inlining
     */
    public static void methodInlining() {
        /**
         * WHAT IS INLINING:
         * 
         * Before:
         * public int compute(int x) {
         *   return square(x) + 1;
         * }
         * private int square(int x) {
         *   return x * x;
         * }
         * 
         * After inlining:
         * public int compute(int x) {
         *   return x * x + 1;  // square() inlined!
         * }
         * 
         * Benefits:
         * - No method call overhead
         * - Better optimization opportunities
         * - 2-10x faster
         * 
         * 
         * INLINING FLAGS:
         * 
         * -XX:MaxInlineSize=<bytes>
         * - Max method size to inline
         * - Default: 35 bytes
         * - Increase for larger methods
         * 
         * -XX:FreqInlineSize=<bytes>
         * - Max size for frequently called methods
         * - Default: 325 bytes
         * 
         * -XX:InlineSmallCode=<bytes>
         * - Max size of compiled code to inline
         * - Default: 2000 bytes
         * 
         * 
         * EXAMPLE: Aggressive inlining
         * 
         * -XX:MaxInlineSize=50
         * -XX:FreqInlineSize=500
         * 
         * 
         * CHECKING INLINING:
         * 
         * -XX:+PrintInlining
         * - Shows what was inlined
         * - Use during development
         * 
         * Look for:
         * "@ <line> <method> (size bytes) inline (hot)"
         * 
         * Or:
         * "@ <line> <method> too big" - not inlined!
         * 
         * 
         * OPTIMIZATION TIP:
         * 
         * Keep hot methods small (< 35 bytes)
         * Extract cold code to separate methods
         * JIT inlines hot path only
         */
    }
    
    /**
     * Escape analysis
     */
    public static void escapeAnalysis() {
        /**
         * WHAT IS ESCAPE ANALYSIS:
         * 
         * JIT analyzes object lifetime
         * If object doesn't "escape" method:
         * → Allocate on stack (not heap!)
         * → No GC overhead
         * → Much faster
         * 
         * 
         * EXAMPLE:
         * 
         * public int compute() {
         *   Point p = new Point(1, 2);  // Heap allocation
         *   return p.x + p.y;           // Only used locally
         * }
         * 
         * With escape analysis:
         * public int compute() {
         *   // Point allocated on stack!
         *   int x = 1, y = 2;
         *   return x + y;
         * }
         * 
         * Benefits:
         * - No heap allocation
         * - No GC pressure
         * - 10x faster allocation
         * 
         * 
         * FLAGS:
         * 
         * -XX:+DoEscapeAnalysis (default: enabled)
         * - Enable escape analysis
         * 
         * -XX:+EliminateAllocations
         * - Allow scalar replacement
         * 
         * -XX:+EliminateLocks
         * - Remove unnecessary locks
         * 
         * 
         * LOCK ELISION:
         * 
         * public void process() {
         *   StringBuilder sb = new StringBuilder();  // Local
         *   sb.append("hello");  // synchronized internally!
         * }
         * 
         * Escape analysis detects:
         * - StringBuilder never escapes
         * - Lock not needed
         * → Lock eliminated!
         * 
         * Result: 2-5x faster
         * 
         * 
         * WHEN OBJECTS ESCAPE:
         * 
         * ✗ Returned from method
         * ✗ Passed to another method
         * ✗ Stored in field
         * ✗ Stored in array
         * ✗ Captured by lambda
         * 
         * ✓ Only used locally
         * ✓ Not stored anywhere
         * ✓ Not passed out
         */
    }
    
    /**
     * Deoptimization
     */
    public static void deoptimization() {
        /**
         * SPECULATIVE OPTIMIZATION:
         * 
         * JIT makes assumptions:
         * - Class hierarchy stable
         * - Type never changes
         * - Method never overridden
         * 
         * If assumption violated:
         * → Deoptimization!
         * → Back to interpreter
         * → Re-compile with new info
         * 
         * 
         * CAUSES:
         * 
         * 1. Class loading:
         *    - New subclass loaded
         *    - Breaks monomorphic assumption
         * 
         * 2. Type changes:
         *    - Variable changes type
         *    - Megamorphic call site
         * 
         * 3. Uncommon traps:
         *    - Array bounds check fails
         *    - Null check fails
         *    - Type check fails
         * 
         * 
         * MONITORING:
         *

    public static void callSitePolymorphism() {

         * - Fast (can inline!)
         *
```

### Quick Reference

**JVM Tuning Essentials:**

```
1. Heap Sizing:
   -Xms = -Xmx (avoid resizing)

   G1GC: Default, balanced (< 200ms pauses)
   ZGC: Ultra-low latency (< 1ms pauses)
   Parallel: High throughput (can tolerate pauses)

3. JIT Optimization:
   Tiered compilation: Enabled by default
   Lower thresholds for faster warmup
   Keep hot methods small (< 35 bytes)
   Prefer monomorphic call sites

4. Compressed Oops:
   Enabled if heap < 32 GB
   37% memory savings!
   Avoid 32-40 GB heap range

5. Monitoring:
   JFR for profiling (< 1% overhead)
   Track GC pause times, frequency
   Monitor latency percentiles
   Alert on regressions
```

**Key JVM insights:**

- **Heap sizing matters**: Wrong size → OOM or excessive GC
- **GC selection depends on workload**: G1 (general), ZGC (low latency), Parallel (throughput)
- **JIT optimization is free**: Just write good code, JIT handles rest
- **Compressed oops saves 37%**: But only works if heap < 32 GB
- **Escape analysis eliminates allocation**: Local objects → stack → no GC
- **Inlining is critical**: Methods < 35 bytes → inlined → 2-10x faster
- **Monitoring validates tuning**: JFR shows real impact

**The JVM tuning hierarchy:**

```
1. Heap sizing (biggest impact)
   - Set Xms = Xmx
   - Young gen = 1/3 of heap
   - Use MaxRAMPercentage in containers

2. GC selection (latency vs throughput)
   - G1GC for balanced (default)
   - ZGC for ultra-low latency
   - Parallel for batch processing

3. JIT optimization (mostly automatic)
   - Lower thresholds for fast warmup
   - Keep hot methods small
   - Let escape analysis work

4. Monitoring (validate everything)
   - Use JFR (< 1% overhead)
   - Track GC metrics
   - Alert on regressions
```

**Common tuning mistakes:**

```java
// MISTAKE 1: Different Xms and Xmx
-Xms512m -Xmx4g  // ❌ Heap resizes!
// FIX: 
-Xms4g -Xmx4g    // ✅ Fixed size

// MISTAKE 2: Heap too small
-Xmx512m  // ❌ Constant GC
// FIX: Size based on live data
-Xmx4g    // ✅ Proper headroom

// MISTAKE 3: Wrong GC for workload
-XX:+UseParallelGC  // ❌ For low-latency app
// FIX: Match GC to requirements
-XX:+UseZGC         // ✅ < 1ms pauses

// MISTAKE 4: Ignoring compressed oops threshold
-Xmx40g  // ❌ No compressed oops, wasted memory
// FIX: Stay under 32 GB or go much higher
-Xmx30g  // ✅ Compressed oops enabled

// MISTAKE 5: No monitoring
// ❌ Tune blindly, hope for best
// FIX: Monitor and measure
-XX:StartFlightRecording=...  // ✅ JFR enabled
```

```
40 GB heap (compressed oops OFF):
  - Pointers: 8 bytes

  - Usable: ~28 GB
```
