# Memory Optimization Strategies

## Overview

Memory optimization is crucial for Java applications—excessive allocations trigger frequent garbage collection pauses, memory overhead increases costs in cloud environments, and poor memory usage limits scalability. While modern garbage collectors are efficient, reducing allocations and memory footprint remains valuable. This guide covers memory optimization strategies: object pooling (reusing expensive objects, but beware premature optimization), primitive specialization (avoiding wrapper objects that waste memory), preventing autoboxing (hidden allocations that kill performance), string interning (sharing string instances, but with caveats), weak references for caches (preventing memory leaks while maintaining performance), off-heap memory (bypassing GC for large data structures), and memory-mapped files (efficient file I/O without copying). Each technique has appropriate use cases and potential pitfalls—object pooling often makes code worse in modern JVMs, primitive arrays are dramatically more efficient than object arrays, and off-heap memory requires manual management. Understanding when and how to apply these strategies separates amateur optimization from professional performance engineering.

## 1. Object Pooling

### Reusing Objects (When Appropriate)

```java
/**
 * OBJECT POOLING BASICS
 */

import java.util.concurrent.*;
import java.util.ArrayList;
import java.util.List;

public class ObjectPoolingBasics {
    
    /**
     * What is object pooling?
     */
    public static void whatIsObjectPooling() {
        /**
         * OBJECT POOLING:
         * 
         * Reuse objects instead of creating new ones
         * 
         * Pattern:
         * 1. Create pool of objects
         * 2. Borrow object when needed
         * 3. Use object
         * 4. Return object to pool
         * 5. Reuse for next request
         * 
         * 
         * BENEFITS:
         * 
         * ✓ Reduce allocations
         * ✓ Reduce GC pressure
         * ✓ Predictable memory usage
         * ✓ Reuse expensive initialization
         * 
         * 
         * COSTS:
         * 
         * ✗ Complexity (borrow/return)
         * ✗ Memory overhead (pool size)
         * ✗ State management (reset objects)
         * ✗ Thread safety concerns
         * ✗ May hurt performance!
         */
    }
    
    /**
     * Simple object pool implementation
     */
    public static class SimpleObjectPool<T> {
        private final BlockingQueue<T> pool;
        private final Factory<T> factory;
        private final int maxSize;
        
        public interface Factory<T> {
            T create();
            void reset(T obj);
        }
        
        public SimpleObjectPool(Factory<T> factory, int size) {
            this.factory = factory;
            this.maxSize = size;
            this.pool = new LinkedBlockingQueue<>(size);
            
            // Pre-populate pool
            for (int i = 0; i < size; i++) {
                pool.offer(factory.create());
            }
        }
        
        public T borrow() throws InterruptedException {
            T obj = pool.poll();
            if (obj == null) {
                // Pool exhausted, create new (or wait)
                obj = factory.create();
            }
            return obj;
        }
        
        public void returnToPool(T obj) {
            factory.reset(obj);
            pool.offer(obj);
        }
    }
    
    /**
     * Example: StringBuilder pool
     */
    public static class StringBuilderPool {
        private static final int INITIAL_CAPACITY = 256;
        private static final int MAX_POOL_SIZE = 10;
        
        private static final ThreadLocal<List<StringBuilder>> POOL = 
            ThreadLocal.withInitial(() -> new ArrayList<>(MAX_POOL_SIZE));
        
        public static StringBuilder acquire() {
            List<StringBuilder> pool = POOL.get();
            if (!pool.isEmpty()) {
                return pool.remove(pool.size() - 1);
            }
            return new StringBuilder(INITIAL_CAPACITY);
        }
        
        public static void release(StringBuilder sb) {
            List<StringBuilder> pool = POOL.get();
            if (pool.size() < MAX_POOL_SIZE) {
                // Reset and return to pool
                sb.setLength(0);
                pool.add(sb);
            }
            // Otherwise let it be GC'd
        }
        
        /**
         * Usage:
         * StringBuilder sb = StringBuilderPool.acquire();
         * try {
         *   sb.append("data");
         *   return sb.toString();
         * } finally {
         *   StringBuilderPool.release(sb);
         * }
         */
    }
    
    /**
     * When to use object pooling
     */
    public static void whenToUse() {
        /**
         * GOOD USE CASES:
         * 
         * 1. Expensive object creation:
         *    - Database connections
         *    - Thread pools
         *    - SSL/TLS sessions
         *    - Parser instances
         * 
         * 2. Large objects:
         *    - Buffers (byte[], ByteBuffer)
         *    - Large arrays
         *    - Complex data structures
         * 
         * 3. Short-lived but frequent:
         *    - Protocol buffers
         *    - Message objects
         *    - Temporary buffers
         * 
         * 4. GC-sensitive applications:
         *    - Low-latency systems
         *    - Real-time applications
         *    - High-throughput servers
         */
    }
    
    /**
     * When NOT to use object pooling
     */
    public static void whenNotToUse() {
        /**
         * DON'T POOL:
         * 
         * 1. Cheap objects:
         *    - Small objects (<100 bytes)
         *    - Simple POJOs
         *    - Modern JVMs optimize these
         * 
         * 2. Short-lived objects:
         *    - Eden space handles well
         *    - Young GC is fast
         *    - Pooling adds overhead
         * 
         * 3. Complex state:
         *    - Hard to reset properly
         *    - Risk of state leaks
         *    - Bugs hard to track
         * 
         * 4. Low allocation rate:
         *    - GC not a problem
         *    - Pooling overhead > benefit
         * 
         * 
         * MODERN JVM REALITY:
         * 
         * - Allocation is FAST (bump-the-pointer)
         * - Young GC is FAST (eden collection)
         * - Object pooling often SLOWER!
         * 
         * Profile before pooling!
         * 
         * 
         * FAMOUS QUOTE:
         * "Object pooling is a premature optimization"
         * - Most cases
         * 
         * Exception: Truly expensive objects
         */
    }
}
```

## 2. Primitive Specialization

### Avoiding Wrapper Objects

```java
/**
 * PRIMITIVE SPECIALIZATION
 */

import java.util.*;
import java.util.stream.*;

public class PrimitiveSpecialization {
    
    /**
     * The wrapper object problem
     */
    public static void wrapperObjectProblem() {
        /**
         * WRAPPER OBJECTS:
         * 
         * int → Integer (16 bytes)
         * long → Long (24 bytes)
         * double → Double (24 bytes)
         * 
         * 
         * MEMORY COMPARISON:
         * 
         * int[1000]:
         * - Array object: 16 bytes
         * - Data: 4000 bytes (4 * 1000)
         * Total: 4016 bytes
         * 
         * Integer[1000]:
         * - Array object: 16 bytes
         * - Array references: 8000 bytes (8 * 1000)
         * - Integer objects: 16000 bytes (16 * 1000)
         * Total: 24016 bytes
         * 
         * 6x MORE MEMORY!
         * 
         * 
         * PERFORMANCE IMPACT:
         * 
         * Access int[i]:
         * - Single memory access
         * - Cache-friendly
         * - Fast
         * 
         * Access Integer[i]:
         * - Access array (cache miss)
         * - Follow reference (cache miss)
         * - Unbox value (extra operation)
         * - Slow
         * 
         * 2-5x SLOWER!
         */
    }
    
    /**
     * Using primitive arrays
     */
    public static void usingPrimitiveArrays() {
        // BAD: Wrapper array
        Integer[] numbers = new Integer[1000];
        for (int i = 0; i < 1000; i++) {
            numbers[i] = i;  // Boxing!
        }
        int sum = 0;
        for (Integer n : numbers) {
            sum += n;  // Unboxing!
        }
        
        // GOOD: Primitive array
        int[] primitives = new int[1000];
        for (int i = 0; i < 1000; i++) {
            primitives[i] = i;  // No boxing
        }
        int primitiveSum = 0;
        for (int n : primitives) {
            primitiveSum += n;  // No unboxing
        }
        
        /**
         * BENCHMARK RESULTS:
         * 
         * Integer[]: 1000 ops/ms, 24 KB memory
         * int[]:     5000 ops/ms, 4 KB memory
         * 
         * 5x faster, 6x less memory!
         */
    }
    
    /**
     * Primitive specialized collections
     */
    public static void primitiveCollections() {
        /**
         * PROBLEM WITH COLLECTIONS:
         * 
         * List<Integer> list = new ArrayList<>();
         * list.add(42);  // Boxing: new Integer(42)
         * 
         * Every primitive stored as wrapper object!
         * 
         * 
         * SOLUTIONS:
         * 
         * 1. Eclipse Collections:
         *    IntList list = new IntArrayList();
         *    list.add(42);  // No boxing!
         * 
         * 2. Trove:
         *    TIntArrayList list = new TIntArrayList();
         *    list.add(42);  // No boxing!
         * 
         * 3. FastUtil:
         *    IntArrayList list = new IntArrayList();
         *    list.add(42);  // No boxing!
         * 
         * 4. HPPC (High Performance Primitive Collections):
         *    IntArrayList list = new IntArrayList();
         *    list.add(42);  // No boxing!
         */
    }
    
    /**
     * Primitive streams
     */
    public static void primitiveStreams() {
        // BAD: Boxing
        List<Integer> numbers = Arrays.asList(1, 2, 3, 4, 5);
        int sum = numbers.stream()
            .mapToInt(Integer::intValue)  // Unboxing
            .sum();
        
        // GOOD: Primitive stream
        IntStream primitiveStream = IntStream.range(1, 6);
        int primitiveSum = primitiveStream.sum();  // No boxing!
        
        /**
         * PRIMITIVE STREAM TYPES:
         * 
         * IntStream:
         * - range(1, 10)
         * - of(1, 2, 3)
         * - Operations: map, filter, sum, average
         * 
         * LongStream:
         * - rangeClosed(1, 100)
         * - Operations: same as IntStream
         * 
         * DoubleStream:
         * - of(1.0, 2.0, 3.0)
         * - Operations: sum, average, min, max
         * 
         * 
         * BENEFITS:
         * 
         * ✓ No boxing/unboxing
         * ✓ Specialized operations
         * ✓ Better performance
         * ✓ Less GC pressure
         */
    }
    
    /**
     * Custom primitive collections
     */
    public static class IntArrayList {
        private int[] data;
        private int size;
        
        public IntArrayList() {
            this.data = new int[10];
            this.size = 0;
        }
        
        public void add(int value) {
            if (size == data.length) {
                resize();
            }
            data[size++] = value;
        }
        
        public int get(int index) {
            if (index >= size) {
                throw new IndexOutOfBoundsException();
            }
            return data[index];
        }
        
        public int size() {
            return size;
        }
        
        private void resize() {
            int[] newData = new int[data.length * 2];
            System.arraycopy(data, 0, newData, 0, size);
            data = newData;
        }
        
        /**
         * COMPARISON:
         * 
         * ArrayList<Integer>:
         * - add(42): Boxing overhead
         * - get(0): Unboxing overhead
         * - Memory: 16 bytes per Integer
         * 
         * IntArrayList:
         * - add(42): Direct storage
         * - get(0): Direct access
         * - Memory: 4 bytes per int
         * 
         * 4x less memory, 2-3x faster!
         */
    }
}
```

## 3. Avoiding Autoboxing

### Hidden Allocations

```java
/**
 * AUTOBOXING PITFALLS
 */

import java.util.*;

public class AutoboxingPitfalls {
    
    /**
     * What is autoboxing?
     */
    public static void whatIsAutoboxing() {
        /**
         * AUTOBOXING:
         * 
         * Automatic conversion between primitives and wrappers
         * 
         * int → Integer (boxing)
         * Integer → int (unboxing)
         * 
         * 
         * CONVENIENCE:
         * List<Integer> list = new ArrayList<>();
         * list.add(42);  // Auto-boxes to Integer
         * int x = list.get(0);  // Auto-unboxes to int
         * 
         * 
         * HIDDEN COST:
         * 
         * Each boxing creates new object!
         * Millions of hidden allocations!
         */
    }
    
    /**
     * Common autoboxing traps
     */
    public static void commonTraps() {
        /**
         * TRAP 1: Collections with primitives
         */
        List<Integer> numbers = new ArrayList<>();
        for (int i = 0; i < 1000; i++) {
            numbers.add(i);  // 1000 Integer objects created!
        }
        
        /**
         * TRAP 2: Comparing wrappers
         */
        Integer a = 200;
        Integer b = 200;
        if (a == b) {  // FALSE! Comparing references!
            // This doesn't execute
        }
        if (a.equals(b)) {  // TRUE
            // This executes
        }
        
        /**
         * TRAP 3: Null unboxing
         */
        Integer value = null;
        try {
            int x = value;  // NullPointerException!
        } catch (NullPointerException e) {
            // Unboxing null throws NPE
        }
        
        /**
         * TRAP 4: Mixed arithmetic
         */
        Integer count = 0;
        for (int i = 0; i < 1000; i++) {
            count += i;  // Unbox, add, box: 2000 operations!
        }
        // Should use: int count = 0;
        
        /**
         * TRAP 5: Map keys
         */
        Map<Integer, String> map = new HashMap<>();
        for (int i = 0; i < 1000; i++) {
            map.put(i, "value");  // 1000 Integer keys!
        }
        // Consider: Map<int, String> with primitive library
    }
    
    /**
     * Integer cache
     */
    public static void integerCache() {
        /**
         * INTEGER CACHE:
         * 
         * JVM caches Integer objects for -128 to 127
         * 
         * Integer a = 100;
         * Integer b = 100;
         * a == b  // TRUE! Same cached object
         * 
         * Integer x = 200;
         * Integer y = 200;
         * x == y  // FALSE! Different objects
         * 
         * 
         * CACHE SIZE:
         * 
         * Default: -128 to 127
         * Configurable: -XX:AutoBoxCacheMax=<size>
         * 
         * But don't rely on it!
         * Always use equals() for comparison
         */
    }
    
    /**
     * Detecting autoboxing
     */
    public static void detectingAutoboxing() {
        /**
         * TOOLS TO DETECT:
         * 
         * 1. IDE Warnings:
         *    - IntelliJ: "Boxing of primitive value"
         *    - Eclipse: "Boxing/unboxing conversion"
         * 
         * 2. Static Analysis:
         *    - SpotBugs
         *    - PMD
         *    - Error Prone
         * 
         * 3. Profiling:
         *    - Allocation profiler
         *    - Look for Integer.valueOf() calls
         *    - High allocation rate
         * 
         * 4. Bytecode inspection:
         *    javap -c MyClass.class
         *    Look for: invokestatic Integer.valueOf
         */
    }
    
    /**
     * Avoiding autoboxing
     */
    public static void avoidingAutoboxing() {
        /**
         * STRATEGIES:
         * 
         * 1. Use primitives where possible:
         *    int x = 42;  // Not Integer
         * 
         * 2. Use primitive arrays:
         *    int[] arr = new int[100];  // Not Integer[]
         * 
         * 3. Use primitive streams:
         *    IntStream.range(0, 100)  // Not Stream<Integer>
         * 
         * 4. Use primitive collections:
         *    IntArrayList  // Not ArrayList<Integer>
         * 
         * 5. Avoid mixing types:
         *    int sum = 0;
         *    for (int i = 0; i < 100; i++) {
         *      sum += i;  // All primitives
         *    }
         * 
         * 6. Check null before unboxing:
         *    if (value != null) {
         *      int x = value;
         *    }
         */
    }
}
```

## 4. String Interning

### Sharing String Instances

```java
/**
 * STRING INTERNING
 */

import java.util.*;

public class StringInterning {
    
    /**
     * What is string interning?
     */
    public static void whatIsStringInterning() {
        /**
         * STRING INTERNING:
         * 
         * Store only one copy of each distinct string
         * 
         * String pool in memory
         * Strings with same value share instance
         * 
         * 
         * LITERAL STRINGS:
         * 
         * String a = "hello";
         * String b = "hello";
         * a == b  // TRUE! Same object
         * 
         * Both point to same interned string
         * 
         * 
         * NEW STRINGS:
         * 
         * String x = new String("hello");
         * String y = new String("hello");
         * x == y  // FALSE! Different objects
         * 
         * Created on heap, not interned
         */
    }
    
    /**
     * Manual interning
     */
    public static void manualInterning() {
        String s1 = new String("hello");
        String s2 = new String("hello");
        
        System.out.println(s1 == s2);  // false
        
        // Intern strings
        String i1 = s1.intern();
        String i2 = s2.intern();
        
        System.out.println(i1 == i2);  // true
        
        /**
         * intern() METHOD:
         * 
         * If string in pool:
         * - Return pool reference
         * 
         * If string not in pool:
         * - Add to pool
         * - Return pool reference
         * 
         * 
         * MEMORY SAVINGS:
         * 
         * 1000 strings "hello":
         * Without interning: 1000 * 40 bytes = 40 KB
         * With interning: 1 * 40 bytes = 40 bytes
         * 
         * 1000x reduction!
         */
    }
    
    /**
     * When to use interning
     */
    public static void whenToUse() {
        /**
         * GOOD USE CASES:
         * 
         * 1. Many duplicate strings:
         *    - Configuration keys
         *    - Enum-like values
         *    - Status codes
         *    - Country codes
         * 
         * 2. Long-lived strings:
         *    - Held in caches
         *    - Stored in collections
         *    - Application lifetime
         * 
         * 3. Frequent equality checks:
         *    - Can use == instead of equals()
         *    - Much faster
         * 
         * 
         * EXAMPLE:
         * 
         * // Reading config file
         * Map<String, String> config = new HashMap<>();
         * for (String line : lines) {
         *   String[] parts = line.split("=");
         *   String key = parts[0].intern();  // Intern keys
         *   config.put(key, parts[1]);
         * }
         * 
         * Many keys repeat: database.host, database.port, etc.
         * Interning saves memory
         */
    }
    
    /**
     * When NOT to use interning
     */
    public static void whenNotToUse() {
        /**
         * DON'T INTERN:
         * 
         * 1. Unique strings:
         *    - User input
         *    - Generated UUIDs
         *    - Random data
         *    - No benefit, only overhead
         * 
         * 2. Large strings:
         *    - File contents
         *    - Large messages
         *    - Pool becomes huge
         *    - GC pressure increases
         * 
         * 3. Short-lived strings:
         *    - Temporary variables
         *    - Local processing
         *    - Will be GC'd anyway
         * 
         * 4. High churn rate:
         *    - Constantly changing
         *    - Pool grows unbounded
         *    - Memory leak risk
         * 
         * 
         * DANGER:
         * 
         * String pool never GC'd (before Java 7)
         * Java 7+: Pool in heap, can be GC'd
         * But still risk of memory leak if abused
         */
    }
    
    /**
     * Custom string pool
     */
    public static class CustomStringPool {
        private final Map<String, String> pool = new HashMap<>();
        
        public synchronized String intern(String s) {
            return pool.computeIfAbsent(s, k -> k);
        }
        
        public int size() {
            return pool.size();
        }
        
        public void clear() {
            pool.clear();
        }
        
        /**
         * ADVANTAGES:
         * 
         * ✓ Controlled lifetime
         * ✓ Can be cleared
         * ✓ Isolated from global pool
         * ✓ Application-specific
         * 
         * 
         * USE CASE:
         * 
         * Request-scoped interning
         * Parse request, intern strings, process, clear pool
         */
    }
    
    /**
     * Performance considerations
     */
    public static void performanceConsiderations() {
        /**
         * PERFORMANCE:
         * 
         * intern() is SLOW:
         * - HashMap lookup
         * - Synchronized
         * - String copy if new
         * 
         * But == is FAST:
         * - Reference comparison
         * - No method call
         * 
         * 
         * TRADE-OFF:
         * 
         * Cost of interning vs benefit of ==
         * 
         * If equals() called many times:
         * → Interning worth it
         * 
         * If equals() called rarely:
         * → Interning overhead > benefit
         * 
         * 
         * BENCHMARK:
         * 
         * 1000 strings, 1M equality checks:
         * 
         * Without interning:
         * - equals(): 500ms
         * 
         * With interning:
         * - intern(): 50ms
         * - ==: 10ms
         * Total: 60ms
         * 
         * 8x faster overall!
         */
    }
}
```

## 5. Weak References for Caches

### Preventing Memory Leaks

```java
/**
 * WEAK REFERENCES FOR CACHES
 */

import java.lang.ref.*;
import java.util.*;

public class WeakReferenceCaches {
    
    /**
     * Reference types explained
     */
    public static void referenceTypes() {
        /**
         * JAVA REFERENCE TYPES:
         * 
         * 1. STRONG REFERENCE (normal):
         *    Object obj = new Object();
         *    - Never GC'd while reachable
         *    - Most common type
         * 
         * 2. SOFT REFERENCE:
         *    SoftReference<Object> soft = new SoftReference<>(obj);
         *    - GC'd when memory needed
         *    - Good for caches
         * 
         * 3. WEAK REFERENCE:
         *    WeakReference<Object> weak = new WeakReference<>(obj);
         *    - GC'd at next GC
         *    - Good for metadata
         * 
         * 4. PHANTOM REFERENCE:
         *    PhantomReference<Object> phantom = new PhantomReference<>(obj, queue);
         *    - For cleanup actions
         *    - Advanced use case
         * 
         * 
         * REACHABILITY:
         * 
         * Strong → Soft → Weak → Phantom → Unreachable
         * 
         * GC collects from weakest to strongest
         */
    }
    
    /**
     * Problem with strong references
     */
    public static void strongReferenceProblem() {
        /**
         * MEMORY LEAK WITH CACHE:
         */
        Map<String, byte[]> cache = new HashMap<>();
        
        void loadData(String key) {
            byte[] data = cache.get(key);
            if (data == null) {
                data = expensiveLoad(key);
                cache.put(key, data);  // Never removed!
            }
            // Use data
        }
        
        /**
         * PROBLEM:
         * 
         * Cache grows forever
         * Old entries never removed
         * Eventually OutOfMemoryError
         * 
         * SOLUTIONS:
         * 
         * 1. Manual eviction (complex)
         * 2. Size-bounded cache (may evict needed data)
         * 3. Time-based eviction (may miss hot data)
         * 4. Weak/Soft references (GC handles it!)
         */
    }
    
    /**
     * Soft reference cache
     */
    public static class SoftReferenceCache<K, V> {
        private final Map<K, SoftReference<V>> cache = new HashMap<>();
        
        public synchronized V get(K key) {
            SoftReference<V> ref = cache.get(key);
            if (ref == null) {
                return null;
            }
            
            V value = ref.get();
            if (value == null) {
                // Was GC'd, remove entry
                cache.remove(key);
            }
            return value;
        }
        
        public synchronized void put(K key, V value) {
            cache.put(key, new SoftReference<>(value));
        }
        
        /**
         * SOFT REFERENCE BEHAVIOR:
         * 
         * JVM keeps soft references as long as possible
         * Only clears when memory needed
         * 
         * Perfect for caches:
         * - Hot data stays in memory
         * - Cold data cleared when needed
         * - Automatic memory management
         * 
         * 
         * MEMORY PRESSURE:
         * 
         * Low memory: Cache stays full
         * High memory pressure: Cache cleared
         * Automatic balancing!
         */
    }
    
    /**
     * Weak reference cache
     */
    public static class WeakReferenceCache<K, V> {
        private final Map<K, WeakReference<V>> cache = new HashMap<>();
        
        public synchronized V get(K key) {
            WeakReference<V> ref = cache.get(key);
            if (ref == null) {
                return null;
            }
            return ref.get();  // May return null if GC'd
        }
        
        public synchronized void put(K key, V value) {
            cache.put(key, new WeakReference<>(value));
        }
        
        /**
         * WEAK REFERENCE BEHAVIOR:
         * 
         * Cleared at next GC if no strong references
         * 
         * Use for:
         * - Canonical mappings
         * - Metadata caches
         * - Short-lived data
         * 
         * Not for:
         * - Long-term caches (use Soft)
         * - Critical data (may disappear)
         */
    }
    
    /**
     * WeakHashMap
     */
    public static void weakHashMap() {
        /**
         * WEAKHASHMAP:
         * 
         * Built-in map with weak keys
         * 
         * Map<User, Metadata> metadata = new WeakHashMap<>();
         * 
         * When User object GC'd:
         * - Entry automatically removed
         * - No memory leak
         * 
         * 
         * PERFECT FOR:
         * 
         * 1. Metadata maps:
         *    Map<Object, Metadata> metadata = new WeakHashMap<>();
         *    - Associate metadata with objects
         *    - Cleaned up when object GC'd
         *
```

```java
            for (int i = 0; i < data.length; i++) {
                unsafe.putByte(address + i, data[i]);

            Long oldAddress = addressMap.put(key, address);

         *

         *
```

```java
         * 
         * 1. Large file processing:
         *    - Database files
         *    - Log analysis
         *    - Binary data files
         * 
         * 2. Random access:

         *    - Memory access fast
         *

         *    - Shared memory
         *    - Fast data exchange

     * Basic memory-mapped file

             FileChannel channel = file.getChannel()) {

                0,  // Start position
```

## Summary

### Quick Reference

**Memory Optimization Techniques:**

```
Object Pooling:
✓ Database connections
✓ Thread pools
✓ Large buffers
✗ Small objects (JVM optimizes)

Primitives:
✓ Use int[] not Integer[]
✓ Use IntStream not Stream<Integer>
✓ Use primitive collections
→ 4-6x less memory, 2-5x faster

Autoboxing:
✗ Avoid in loops
✗ Avoid in collections
✗ Use primitives where possible
→ Each box creates object!

String Interning:
✓ Duplicate strings
✓ Long-lived strings
✗ Unique strings
✗ Large strings

Weak References:
✓ Caches (use SoftReference)
✓ Metadata (use WeakReference)
✓ WeakHashMap for canonical maps
→ Automatic memory management

Off-Heap:
✓ Large data (> heap limits)
✓ Long-lived data
✗ Complex to manage
→ No GC overhead

Memory-Mapped Files:
✓ Large file access
✓ Random access
✓ Inter-process communication
→ 2-10x faster than traditional I/O
```

---


**Key insights:**

- **Object pooling often hurts**: Modern JVMs allocate fast!
- **Primitives dramatically better**: int[] vs Integer[] = 6x memory
- **Autoboxing is expensive**: Each box creates object
- **Interning has trade-offs**: Good for duplicates, bad for unique
- **Soft references for caches**: GC clears when memory needed
- **Off-heap bypasses GC**: But requires manual management
- **Memory-mapped files fast**: Direct memory access to files

**The memory hierarchy:**

```
ALLOCATION SPEED:
Primitive → Stack allocation (fastest)
Small object → Eden allocation (fast)
Large object → Old gen direct (slower)
Off-heap → Native malloc (varies)

MEMORY EFFICIENCY:
int: 4 bytes
Integer: 16 bytes (4x overhead!)

int[1000]: 4 KB
Integer[1000]: 24 KB (6x overhead!)

GC IMPACT:
On-heap → GC'd (pauses)
Off-heap → No GC (manual)

CHOICE:
Normal data → On-heap (easy)
Large/long-lived → Off-heap (complex)
```

**Common optimizations:**

```
1. Replace Integer[] with int[]
   → 6x less memory, 2-5x faster

2. Replace ArrayList<Integer> with IntArrayList
   → No boxing, much faster

3. Use SoftReference for caches
   → Auto cleanup, no leaks

4. Use memory-mapped files for large files
   → 2-10x faster than traditional I/O

5. Avoid autoboxing in loops
   → Each iteration creates object

6. Intern duplicate strings
   → Share instances, save memory
```
