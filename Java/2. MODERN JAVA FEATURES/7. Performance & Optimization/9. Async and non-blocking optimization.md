# Async and Non-blocking Optimization

## Overview

Asynchronous and non-blocking programming transforms application performance and scalability by eliminating thread blocking—instead of waiting idly for I/O (network, disk, database), threads remain active and handle other work. A synchronous application with 10ms database queries serves 100 req/sec per thread (blocking 99% of the time), while an async application serves 10,000 req/sec per thread by never blocking. However, async programming introduces complexity: callback hell, error handling across async boundaries, thread pool exhaustion, queue overflow, and backpressure problems when producers overwhelm consumers. This guide covers async optimization: CompletableFuture best practices (composition, error handling, avoiding blocking operations), identifying and eliminating blocking operations (the silent killers of async systems), reactive programming benefits (Project Reactor for true non-blocking streams), thread pool sizing (Little's Law, formulas for different workloads, measurement strategies), queue selection (bounded vs unbounded, ArrayBlockingQueue vs LinkedBlockingQueue trade-offs), and backpressure handling (flow control strategies to prevent memory overflow and service degradation). Whether building a high-throughput microservice, reactive system, or just trying to improve responsiveness, understanding these patterns enables you to harness the power of async while avoiding its pitfalls.

```java
/**
 * COMPLETABLEFUTURE BEST PRACTICES
 */

import java.util.*;
import java.util.concurrent.*;
import java.util.stream.*;

public class CompletableFutureBestPractices {
    
    /**
     * Basic async pattern
     */
    public static void basicAsync() {
        /**
         * BLOCKING WAY:
         * 
         * Result result = expensiveOperation();
         * process(result);
         * 
         * Thread blocked for entire operation!
         * If operation takes 1 second, thread idle 1 second.
         * 
         * 
         * ASYNC WAY:
         * 
         * CompletableFuture<Result> future = 
         *   CompletableFuture.supplyAsync(() -> expensiveOperation());
         * 
         * future.thenAccept(result -> process(result));
         * 
         * Thread returns immediately!
         * Operation runs on different thread.
         * When complete, callback executed.
         */
        
        // Example
        CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> {
            // Expensive operation
            sleep(1000);
            return "Result";
        });
        
        future.thenAccept(result -> {
            System.out.println("Got: " + result);
        });
        
        /**
         * THROUGHPUT COMPARISON:
         * 
         * Synchronous (1 thread):
         * - 1 operation per second
         * - Thread blocks entire time
         * 
         * Asynchronous (1 thread + thread pool):
         * - 1000s of operations per second
         * - Thread never blocks
         * - Handles callbacks
         */
    }
    
    /**
     * Chaining operations
     */
    public static void chainingOperations() {
        // Sequential chain
        CompletableFuture<String> future = CompletableFuture
            .supplyAsync(() -> getUserId())
            .thenApply(id -> fetchUser(id))
            .thenApply(user -> user.getName())
            .thenApply(name -> name.toUpperCase());
        
        /**
         * CHAINING METHODS:
         * 
         * thenApply(Function):
         * - Transform result
         * - Returns new value
         * - Use for pure transformations
         * 
         * thenAccept(Consumer):
         * - Process result
         * - Returns void
         * - Use for side effects
         * 
         * thenCompose(Function<T, CompletableFuture<U>>):
         * - Flatten nested futures
         * - Async operation on result
         * - Use for dependent async calls
         * 
         * thenRun(Runnable):
         * - Execute after completion
         * - No access to result
         * - Use for cleanup
         */
    }
    
    /**
     * Composing dependent operations
     */
    public static void composingDependentOps() {
        // WRONG: Nested futures
        CompletableFuture<CompletableFuture<Order>> wrong = 
            CompletableFuture.supplyAsync(() -> getUserId())
                .thenApply(id -> fetchUserAsync(id))  // Returns CF!
                .thenApply(userFuture -> loadOrderAsync(userFuture.join().getId()));
        
        // RIGHT: Flat composition
        CompletableFuture<Order> right = 
            CompletableFuture.supplyAsync(() -> getUserId())
                .thenCompose(id -> fetchUserAsync(id))  // Flattens!
                .thenCompose(user -> loadOrderAsync(user.getId()));
        
        /**
         * KEY RULE:
         * 
         * If function returns CompletableFuture:
         * → Use thenCompose (flattens)
         * 
         * If function returns regular value:
         * → Use thenApply (transforms)
         * 
         * 
         * COMPARISON TO STREAMS:
         * 
         * Stream:
         * - map() for transformation
         * - flatMap() for flattening
         * 
         * CompletableFuture:
         * - thenApply() for transformation
         * - thenCompose() for flattening
         */
    }
    
    /**
     * Combining multiple futures
     */
    public static void combiningFutures() {
        // Parallel execution
        CompletableFuture<User> userFuture = fetchUserAsync(123);
        CompletableFuture<Order> orderFuture = fetchOrderAsync(456);
        CompletableFuture<Product> productFuture = fetchProductAsync(789);
        
        // Combine results
        CompletableFuture<Response> response = userFuture
            .thenCombine(orderFuture, (user, order) -> 
                new Pair<>(user, order))
            .thenCombine(productFuture, (pair, product) ->
                new Response(pair.user, pair.order, product));
        
        /**
         * COMBINING METHODS:
         * 
         * thenCombine(other, BiFunction):
         * - Wait for both
         * - Combine results
         * - Returns new value
         * 
         * thenAcceptBoth(other, BiConsumer):
         * - Wait for both
         * - Process results
         * - Returns void
         * 
         * runAfterBoth(other, Runnable):
         * - Wait for both
         * - No access to results
         * - Returns void
         */
        
        // All of (wait for all)
        CompletableFuture<Void> all = CompletableFuture.allOf(
            userFuture, orderFuture, productFuture
        );
        
        // Any of (wait for first)
        CompletableFuture<Object> any = CompletableFuture.anyOf(
            userFuture, orderFuture, productFuture
        );
        
        /**
         * PERFORMANCE:
         * 
         * Sequential:
         * Time = t1 + t2 + t3
         * Example: 100ms + 100ms + 100ms = 300ms
         * 
         * Parallel (combined):
         * Time = max(t1, t2, t3)
         * Example: max(100ms, 100ms, 100ms) = 100ms
         * 
         * 3x faster!
         */
    }
    
    /**
     * Error handling
     */
    public static void errorHandling() {
        CompletableFuture<String> future = CompletableFuture
            .supplyAsync(() -> {
                if (Math.random() > 0.5) {
                    throw new RuntimeException("Failed!");
                }
                return "Success";
            })
            .exceptionally(ex -> {
                // Handle error, return default
                System.err.println("Error: " + ex.getMessage());
                return "Default";
            })
            .thenApply(result -> result.toUpperCase());
        
        // Alternative: handle both success and error
        CompletableFuture<String> handled = CompletableFuture
            .supplyAsync(() -> riskyOperation())
            .handle((result, ex) -> {
                if (ex != null) {
                    return "Error: " + ex.getMessage();
                }
                return "Success: " + result;
            });
        
        // Alternative: whenComplete (peek at result/error)
        CompletableFuture<String> logged = CompletableFuture
            .supplyAsync(() -> riskyOperation())
            .whenComplete((result, ex) -> {
                if (ex != null) {
                    System.err.println("Failed: " + ex);
                } else {
                    System.out.println("Succeeded: " + result);
                }
                // Result/error propagated to next stage
            });
        
        /**
         * ERROR HANDLING METHODS:
         * 
         * exceptionally(Function):
         * - Catch errors
         * - Return default value
         * - Like catch block
         * 
         * handle(BiFunction):
         * - Process result OR error
         * - Must handle both cases
         * - Like finally that can transform
         * 
         * whenComplete(BiConsumer):
         * - Peek at result/error
         * - Cannot transform
         * - Like finally
         * 
         * 
         * BEST PRACTICES:
         * 
         * 1. Always handle errors
         * 2. Use exceptionally for recovery
         * 3. Use handle for transformation
         * 4. Use whenComplete for logging
         * 5. Don't ignore exceptions!
         */
    }
    
    /**
     * Custom executor
     */
    public static void customExecutor() {
        // Default executor (ForkJoinPool.commonPool())
        CompletableFuture<String> defaultExec = 
            CompletableFuture.supplyAsync(() -> work());
        
        // Custom executor
        ExecutorService executor = Executors.newFixedThreadPool(10);
        
        CompletableFuture<String> customExec = 
            CompletableFuture.supplyAsync(() -> work(), executor);
        
        /**
         * WHY CUSTOM EXECUTOR:
         * 
         * Common pool:
         * - Shared across application
         * - CPU-bound tasks
         * - Size = CPU cores - 1
         * 
         * Problem:
         * - Blocking operations exhaust pool
         * - One slow task blocks others
         * - No isolation
         * 
         * 
         * CUSTOM EXECUTOR:
         * 
         * ✓ Dedicated threads
         * ✓ Sized for workload
         * ✓ Isolation from other tasks
         * ✓ Better monitoring
         * 
         * 
         * GUIDELINES:
         * 
         * CPU-bound:
         * - Use common pool
         * - Or: Executors.newWorkStealingPool()
         * 
         * I/O-bound:
         * - Use custom pool
         * - Size = (cores * target_utilization) / (1 - blocking_coefficient)
         * - Example: 8 cores, 0.9 util, 0.9 blocking = 72 threads
         * 
         * Mixed:
         * - Separate pools
         * - CPU pool for computation
         * - I/O pool for blocking operations
         */
        
        executor.shutdown();
    }
    
    /**
     * Timeouts
     */
    public static void timeouts() throws Exception {
        // Timeout (Java 9+)
        CompletableFuture<String> future = CompletableFuture
            .supplyAsync(() -> slowOperation())
            .orTimeout(5, TimeUnit.SECONDS);  // Throws TimeoutException
        
        // Timeout with default (Java 9+)
        CompletableFuture<String> withDefault = CompletableFuture
            .supplyAsync(() -> slowOperation())
            .completeOnTimeout("Default", 5, TimeUnit.SECONDS);
        
        /**
         * TIMEOUT STRATEGIES:
         * 
         * orTimeout():
         * - Throws TimeoutException
         * - Use when timeout is error
         * 
         * completeOnTimeout():
         * - Returns default value
         * - Use when timeout is acceptable
         * 
         * 
         * BEST PRACTICES:
         * 
         * 1. Always set timeouts for external calls
         * 2. Choose timeout based on SLA
         * 3. Log timeouts for monitoring
         * 4. Consider circuit breaker for repeated timeouts
         */
    }
    
    /**
     * Real-world example: API aggregation
     */
    public static class ApiAggregator {
        private final ExecutorService ioPool =

        public CompletableFuture<DashboardData> getDashboard(long userId) {

                    () -> fetchUser(userId), ioPool)

            CompletableFuture<List<Order>> ordersFuture =

                .thenCombine(ordersFuture, (user, orders) ->

                .thenCombine(notifsFuture, (pair, notifs) ->

        /**
         * BENEFITS:
         * 
         * Sequential:
         * - Time = 1s + 2s + 0.5s = 3.5s
         * 
         * Parallel:
         * - Time = max(1s, 2s, 0.5s) = 2s
         * 
         * 1.75x faster!
         * 
         * Plus:
         * - Graceful degradation (notifications optional)
         * - Timeouts prevent hanging

        private User fetchUser(long id) { return null; }

    }
```

## 2. Avoiding Blocking Operations

### The Silent Killers of Async Systems

```java
/**
 * AVOIDING BLOCKING OPERATIONS
 */

import java.util.concurrent.*;
import java.io.*;
import java.net.*;

public class AvoidingBlocking {
    
    /**
     * Understanding blocking
     */
    public static void understandingBlocking() {
        /**
         * BLOCKING OPERATIONS:
         * 
         * Thread waits for operation to complete
         * Cannot do other work
         * Wastes resources
         * 
         * Common blocking operations:
         * 1. Synchronous I/O (file, network)
         * 2. Thread.sleep()
         * 3. Object.wait()
         * 4. Lock.lock()
         * 5. BlockingQueue.take()
         * 6. CountDownLatch.await()
         * 7. Future.get()
         * 8. Synchronized blocks (contention)
         * 
         * 
         * PROBLEM IN ASYNC SYSTEMS:
         * 
         * Thread pool size: 10 threads
         * Blocking operation: 1 second
         * 
         * All 10 threads blocked:
         * → No threads available
         * → New tasks queued
         * → System appears hung
         * → Cascading failures
         */
    }
    
    /**
     * Blocking I/O example
     */
    public static class BlockingIO {
        // BAD: Blocking I/O in CompletableFuture
        public CompletableFuture<String> readFileBad() {
            return CompletableFuture.supplyAsync(() -> {
                try {
                    // BLOCKS thread!
                    BufferedReader reader = new BufferedReader(
                        new FileReader("data.txt"));
                    return reader.readLine();
                } catch (IOException e) {
                    throw new RuntimeException(e);
                }
            });
        }
        
        // GOOD: Use dedicated I/O thread pool
        private final ExecutorService ioPool = 
            Executors.newFixedThreadPool(20);
        
        public CompletableFuture<String> readFileGood() {
            return CompletableFuture.supplyAsync(() -> {
                try {
                    BufferedReader reader = new BufferedReader(
                        new FileReader("data.txt"));
                    return reader.readLine();
                } catch (IOException e) {
                    throw new RuntimeException(e);
                }
            }, ioPool);  // Dedicated pool!
        }
        
        // BEST: Use async I/O (NIO)
        public CompletableFuture<String> readFileAsync() {
            // Use AsynchronousFileChannel
            // Truly non-blocking
            return null; // Implementation omitted
        }
        
        /**
         * IMPACT:
         * 
         * Common pool: 7 threads (8 core machine)
         * 
         * With blocking I/O on common pool:
         * - 7 blocking calls exhaust pool
         * - All threads stuck
         * - System deadlock
         * 
         * With dedicated I/O pool:
         * - Common pool free for CPU work
         * - I/O pool handles blocking
         * - No interference
         */
    }
    
    /**
     * Database calls
     */
    public static class DatabaseCalls {
        private final ExecutorService dbPool = 
            Executors.newFixedThreadPool(50);

                ResultSet rs = stmt.executeQuery();  // BLOCKS!
                return mapUser(rs);
            });  // Uses common pool - BAD!
        }
        
        // GOOD: Dedicated DB pool

            return CompletableFuture.supplyAsync(() -> {
                Connection conn = getConnection();
                PreparedStatement stmt = conn.prepareStatement(
                    "SELECT * FROM users WHERE id = ?");
                stmt.setLong(1, id);
                ResultSet rs = stmt.executeQuery();
                return mapUser(rs);
            }, dbPool);  // Dedicated pool!
        }
        
        // BEST: Use reactive database driver
        public CompletableFuture<User> findUserReactive(long id) {
            // Use R2DBC (Reactive Relational Database Connectivity)
            // Truly non-blocking
            return null; // Implementation with R2DBC
        }
        
        /**
         * DB POOL SIZING:
         * 
         * Formula:
         * pool_size = connections * (avg_query_time / avg_request_time)
         * 
         * Example:
         * - 10 DB connections
         * - Query: 10ms

         * JDBC:
         * - Blocks thread during query

         * - High memory usage

         * R2DBC:
         * - Non-blocking
         * - Few threads needed
         * - Low memory usage

         * 10x-100x more efficient!

        private User mapUser(ResultSet rs) { return null; }

    /**
     * HTTP calls
     */
    public static class HttpCalls {
        // BAD: HttpURLConnection blocks
        public CompletableFuture<String> callApiBad(String url) {
            return CompletableFuture.supplyAsync(() -> {
                try {
                    HttpURLConnection conn = 
                        (HttpURLConnection) new URL(url).openConnection();
                    conn.connect();  // BLOCKS!
                    BufferedReader reader = new BufferedReader(
                        new InputStreamReader(conn.getInputStream()));
                    return reader.readLine();
                } catch (IOException e) {
                    throw new RuntimeException(e);
                }
            });
        }
        
        // GOOD: HttpClient (Java 11+)
        private final HttpClient client = HttpClient.newBuilder()
            .executor(Executors.newFixedThreadPool(20))
            .build();
        
        public CompletableFuture<String> callApiGood(String url) {
            HttpRequest request = HttpRequest.newBuilder()
                .uri(URI.create(url))
                .build();
            
            return client.sendAsync(request, HttpResponse.BodyHandlers.ofString())
                .thenApply(HttpResponse::body);
            
            // Truly async!
            // No thread blocking
        }
        
        /**
         * HTTP CLIENT COMPARISON:
         * 
         * HttpURLConnection:
         * - Blocking
         * - Thread per request
         * - 100 requests = 100 threads blocked
         * 
         * HttpClient (async):
         * - Non-blocking
         * - Few threads
         * - 100 requests = ~5-10 threads
         * 
         * 10x-20x more efficient!
         */
    }

         *

         *    - Identify blocking points
         * 
         * 2. Profiler:
         *    - YourKit, JProfiler
         *    - Thread state timeline
         *    - Blocking time analysis
         * 
         * 3. Metrics:
         *    - Thread pool queue size
         *    - Active thread count
         *    - Task wait time
         * 
         * 4. Code review:
         *    - Search for:
         *      * Thread.sleep()
         *      * .get() on Future
         *      * Synchronous I/O
         *      * JDBC calls
         *      * synchronized blocks
         * 
         * 
         * WARNING SIGNS:
         * 
         * - Thread pool queue growing
         * - Response times increasing
         * - Thread count maxed out
         * - CPU idle while under load
         */
    }
    
    /**
     * Strategies to avoid blocking
     */
    public static void avoidanceStrategies() {
        /**
         * STRATEGY 1: Separate thread pools
         * 
         * - CPU-bound: Common pool or work-stealing
         * - I/O-bound: Dedicated pool (sized for latency)
         * - Database: Dedicated pool (sized for connections)
         * 
         * 
         * STRATEGY 2: Use async APIs
         * 
         * - HttpClient instead of HttpURLConnection
         * - AsynchronousFileChannel instead of FileInputStream
         * - R2DBC instead of JDBC
         * 
         * 
         * STRATEGY 3: Reactive programming
         * 
         * - Project Reactor
         * - RxJava
         * - Vert.x
         * 
         * 
         * STRATEGY 4: Never call .get()
         * 
         * BAD:
         * CompletableFuture<String> future = fetchAsync();
         * String result = future.get();  // BLOCKS!
         * 
         * GOOD:
         * CompletableFuture<String> future = fetchAsync();
         * future.thenAccept(result -> process(result));
         * 
         * 
         * STRATEGY 5: Timeouts
         * 
         * Prevent indefinite blocking
         * Use orTimeout() or completeOnTimeout()
         */
    }
```

## 3. Reactive Programming Benefits

### True Non-blocking Streams

```java
/**
 * REACTIVE PROGRAMMING
 */

import reactor.core.publisher.*;
import reactor.core.scheduler.Schedulers;
import java.time.Duration;

public class ReactiveProgramming {
    
    /**
     * Why reactive?
     */
    public static void whyReactive() {
        /**
         * REACTIVE STREAMS:
         * 
         * Publisher → Subscriber
         * Push-based, non-blocking
         * Backpressure support
         * 
         * 
         * KEY CONCEPTS:
         * 
         * Mono<T>: 0-1 element
         * Flux<T>: 0-N elements
         * 
         * 
         * BENEFITS:
         * 
         * 1. Non-blocking I/O:
         *    - Thread never blocks
         *    - High concurrency
         *    - Low thread count
         * 
         * 2. Backpressure:
         *    - Consumer controls rate
         *    - Prevents overflow
         *    - Automatic flow control
         * 
         * 3. Composable:
         *    - Chain operations
         *    - Transform streams
         *    - Combine publishers
         * 
         * 4. Resilient:
         *    - Retry strategies
         *    - Timeout handling
         *    - Fallback support
         * 
         * 
         * WHEN TO USE:
         * 
         * ✓ High concurrency needs
         * ✓ I/O-heavy workloads
         * ✓ Streaming data
         * ✓ Microservices
         * ✓ Real-time systems
         * 
         * ✗ CPU-bound tasks
         * ✗ Simple CRUD apps
         * ✗ Team unfamiliar with reactive
         */
    }
    
    /**
     * Basic reactive operations
     */
    public static void basicOperations() {
        // Mono (single value)
        Mono<String> mono = Mono.just("Hello")
            .map(String::toUpperCase)
            .map(s -> s + " World");
        
        mono.subscribe(System.out::println);
        
        // Flux (stream)
        Flux<Integer> flux = Flux.range(1, 10)
            .filter(i -> i % 2 == 0)
            .map(i -> i * i);
        
        flux.subscribe(System.out::println);
        
        /**
         * REACTIVE OPERATORS:
         * 
         * Transformation:
         * - map: Transform each element
         * - flatMap: Async transformation
         * - filter: Select elements
         * 
         * Combination:
         * - zip: Combine multiple publishers
         * - merge: Merge into one stream
         * - concat: Append streams
         * 
         * Error handling:
         * - onErrorReturn: Default value
         * - onErrorResume: Fallback publisher
         * - retry: Retry on error
         * 
         * Timing:
         * - timeout: Time limit
         * - delayElements: Rate limiting
         * - buffer: Batch elements
         */
    }
    
    /**
     * Async database access
     */
    public static class ReactiveDatabase {
        // Reactive repository (Spring Data R2DBC)
        interface UserRepository {
            Mono<User> findById(Long id);
            Flux<User> findAll();
            Mono<User> save(User user);
        }
        
        private final UserRepository repository = null;
        
        public Mono<User> getUser(Long id) {
            return repository.findById(id)
                .timeout(Duration.ofSeconds(5))
                .doOnError(ex -> log("Database error: " + ex))
                .onErrorReturn(User.empty());
        }
        
        /**
         * REACTIVE DB BENEFITS:
         *

         *
```

```java
         * - Increased latency
         * - Underutilized resources
         * - Poor throughput
         *

         * - Cache thrashing
         * - CPU overhead
         * - Diminishing returns

         * - Maximize throughput
         * - Minimize latency
         * - Efficient resource use
         */
    }
    
    /**
     * CPU-bound tasks
     */
    public static void cpuBoundSizing() {
        /**
         * CPU-BOUND FORMULA:
         * 
         * pool_size = number_of_cores + 1
         * 
         * OR
         * 
         * pool_size = number_of_cores * (1 + ε)
         * where ε = small overhead factor (0.1-0.2)
         * 
         * 
         * RATIONALE:
         * 
         * - CPU-bound tasks use 100% CPU
         * - More threads = context switching
         * - Optimal = 1 thread per core
         * - +1 for occasional page fault
         * 
         * 
         * EXAMPLE:
         * 
         * 8-core machine:
         * pool_size = 8 + 1 = 9 threads
         * 
         * 
         * MEASUREMENT:
         * 
         * Test with different sizes:
         * - 8 threads: 1000 ops/sec
         * - 9 threads: 1010 ops/sec (best)
         * - 10 threads: 1005 ops/sec
         * - 16 threads: 950 ops/sec (context switching)
         */
        
        int cores = Runtime.getRuntime().availableProcessors();
        ExecutorService cpuBound = Executors.newFixedThreadPool(cores + 1);
    }
    
    /**
     * I/O-bound tasks
     */
    public static void ioBoundSizing() {
        /**
         * I/O-BOUND FORMULA:
         * 
         * pool_size = number_of_cores * target_cpu_utilization * (1 + wait_time / compute_time)
         * 
         * OR
         * 
         * pool_size = number_of_cores / (1 - blocking_coefficient)
         * 
         * where blocking_coefficient = wait_time / (wait_time + compute_time)
         * 
         * 
         * EXAMPLE 1:
         * 
         * 8 cores
         * Target utilization: 100%
         * Wait time: 90ms (I/O)
         * Compute time: 10ms (CPU)
         * 
         * pool_size = 8 * 1.0 * (1 + 90/10)
         *          = 8 * 1.0 * 10
         *          = 80 threads
         * 
         * 
         * EXAMPLE 2:
         * 
         * 8 cores
         * Blocking coefficient: 0.9 (90% time waiting)
         * 
         * pool_size = 8 / (1 - 0.9)
         *          = 8 / 0.1
         *          = 80 threads
         * 
         * 
         * RATIONALE:
         * 
         * - Threads mostly waiting
         * - Can have many threads
         * - While one waits, others run
         * - High concurrency needed
         */
        
        int cores = Runtime.getRuntime().availableProcessors();
        double blockingCoefficient = 0.9;  // 90% blocking
        int poolSize = (int) (cores / (1 - blockingCoefficient));
        
        ExecutorService ioBound = Executors.newFixedThreadPool(poolSize);
    }

    public static void littlesLaw() {

         * LITTLE'S LAW:
         * 
         * L = λ * W
         *

         */
    }
```

```java
import java.util.concurrent.*;
import reactor.core.publisher.*;

public class BackpressureHandling {
    
    /**
     * Understanding backpressure
     */
    public static void understandingBackpressure() {
        /**
         * BACKPRESSURE PROBLEM:
         * 
         * Producer: 10,000 items/second
         * Consumer: 1,000 items/second
         * 
         * Without control:
         * - Queue grows 9,000 items/second
         * - Memory exhausted
         * - OutOfMemoryError
         * 
         * 
         * SOLUTION STRATEGIES:
         * 
         * 1. Slow down producer
         * 2. Buffer (limited)
         * 3. Drop items
         * 4. Request-based pull
         * 
         * 
         * REACTIVE STREAMS:
         * 
         * Built-in backpressure
         * Consumer controls rate
         * Producer respects signals
         */
    }
    
    /**
     * CompletableFuture backpressure
     */
    public static class CompletableFutureBackpressure {
        private final Semaphore semaphore = new Semaphore(100);
        
        public CompletableFuture<Result> processWithBackpressure(Request req) {
            return CompletableFuture
                .runAsync(() -> {
                    try {
                        semaphore.acquire();  // Limit concurrency
                    } catch (InterruptedException e) {
                        throw new RuntimeException(e);
                    }
                })
                .thenApply(v -> process(req))
                .whenComplete((result, ex) -> semaphore.release());
        }
        
        /**
         * SEMAPHORE PATTERN:
         * 
         * Limits concurrent operations
         * - Acquire permit before processing
         * - Release after completion
         * 
         * Benefits:
         * ✓ Bounded concurrency
         * ✓ Prevents overload
         * ✓ Simple
         * 
         * Drawbacks:
         * ✗ Blocking (acquire)
         * ✗ Manual management
         */
        
        private Result process(Request req) { return null; }
    }
    
    /**
     * Thread pool backpressure
     */
    public static class ThreadPoolBackpressure {
        // CallerRunsPolicy provides backpressure
        private final ThreadPoolExecutor pool = new ThreadPoolExecutor(
            10, 20,
            60L, TimeUnit.SECONDS,
            new ArrayBlockingQueue<>(100),
            new ThreadPoolExecutor.CallerRunsPolicy()  // Backpressure!
        );
        
        /**
         * HOW IT WORKS:
         * 
         * Normal:
         * - Task submitted
         * - Runs on pool thread
         * - Caller continues
         * 
         * Overload:
         * - Queue full
         * - Pool at max
         * - Task runs on caller thread
         * - Caller slowed down (backpressure!)
         * 
         * 
         * EFFECT:
         * 
         * Producer trying to submit 10,000/sec
         * Pool capacity: 1,000/sec
         * 
         * Result:
         * - Caller slows to 1,000/sec
         * - Natural rate limiting
         * - No queue overflow
         * 
         * 
         * ALTERNATIVE: AbortPolicy
         * 
         * Throws exception on overflow
         * Caller must handle
         * Explicit backpressure signal
         */
    }
    
    /**
     * Reactive backpressure
     */
    public static class ReactiveBackpressure {
        public Flux<Result> processStream(Flux<Request> requests) {
            return requests
                .flatMap(req -> processAsync(req), 10)  // Concurrency limit!
                .onBackpressureBuffer(100);  // Buffer overflow
        }
        
        private Mono<Result> processAsync(Request req) {
            return Mono.fromCallable(() -> process(req));
        }
        
        private Result process(Request req) { return null; }
        
        /**
         * REACTIVE STRATEGIES:
         * 
         * 1. onBackpressureBuffer(size):
         *    - Buffer excess items
         *    - Up to size limit
         *    - Error on overflow
         * 
         * 2. onBackpressureDrop():
         *    - Drop excess items
         *    - No error
         *    - Data loss
         * 
         * 3. onBackpressureLatest():
         *    - Keep only latest
         *    - Discard buffered
         *    - Data loss
         * 
         * 4. onBackpressureError():
         *    - Error immediately
         *    - No buffering
         *    - Fail fast
         * 
         * 
         * CONCURRENCY LIMIT:
         * 
         * flatMap(mapper, concurrency)
         * - Limits parallel operations
         * - Built-in backpressure
         * - Most effective!
         */

        private final ScheduledExecutorService refiller = 
            Executors.newSingleThreadScheduledExecutor();
        
        public RateLimiting() {
            // Refill permits every second
            refiller.scheduleAtFixedRate(
                () -> {
                    int available = rateLimiter.availablePermits();
                    if (available < 100) {
                        rateLimiter.release(Math.min(10, 100 - available));
                    }
                },
                1, 1, TimeUnit.SECONDS
            );

            return CompletableFuture

    /**
     * Circuit breaker pattern
     */
    public static class CircuitBreakerPattern {
        private enum State { CLOSED, OPEN, HALF_OPEN }
        
        private State state = State.CLOSED;
        private int failureCount = 0;
        private final int failureThreshold = 5;
        private long openedAt;
        
        public CompletableFuture<Result> callWithCircuitBreaker(Request req) {
            if (state == State.OPEN) {
                // Check if should try again
                if (System.currentTimeMillis() - openedAt > 60000) {
                    state = State.HALF_OPEN;
                } else {
                    return CompletableFuture.failedFuture(
                        new RuntimeException("Circuit breaker OPEN"));
                }
            }
            
            return callService(req)
                .whenComplete((result, ex) -> {
                    if (ex != null) {
                        onFailure();
                    } else {
                        onSuccess();
                    }
                });
        }
        
        private void onFailure() {
            failureCount++;
            if (failureCount >= failureThreshold) {
                state = State.OPEN;
                openedAt = System.currentTimeMillis();
            }
        }
        
        private void onSuccess() {
            if (state == State.HALF_OPEN) {
                state = State.CLOSED;
            }
            failureCount = 0;
        }
        
        private CompletableFuture<Result> callService(Request req) {
            return CompletableFuture.completedFuture(null);
        }
        
        /**
         * CIRCUIT BREAKER:
         * 
         * States:
         * - CLOSED: Normal operation
         * - OPEN: Failing, reject requests
         * - HALF_OPEN: Testing recovery
         * 
         * Behavior:
         * - Failures increase count
         * - Threshold reached → OPEN
         * - After timeout → HALF_OPEN
         * - Success → CLOSED
         * 
         * 
         * BENEFITS:
         * 
         * ✓ Fast failure
         * ✓ Prevents cascade
         * ✓ Allows recovery
         * ✓ Protects downstream
         * 
         * 
         * USE WITH:
         * 
         * - External services
         * - Database calls
         * - Microservices
         */
    }
```

**Async and Non-blocking Optimization:**

```
1. CompletableFuture Best Practices:
   - thenApply for transformation
   - thenCompose for flattening async
   - thenCombine for parallel execution
   - exceptionally for error handling

2. Avoid Blocking:
   - Separate thread pools (CPU, I/O, DB)
   - Use async APIs (HttpClient, R2DBC)
   - Never call Future.get()
   - Set timeouts on all external calls
   Result: 10x-100x higher concurrency

3. Reactive Programming:
   - Mono/Flux for non-blocking streams
   - Built-in backpressure
   - 10-100x less memory
   - 10-100x higher throughput

4. Thread Pool Sizing:
   - CPU-bound: cores + 1
   - I/O-bound: cores / (1 - blocking_coefficient)
   - Measure and adjust
   Result: Optimal resource utilization

5. Queue Selection:
   - Bounded queues (prevent OOM)
   - CallerRunsPolicy (backpressure)
   - Size: core_pool_size * 10
   Result: Stable under load

   - Semaphore for limiting concurrency
   - CallerRunsPolicy for thread pools
   - Reactive operators for streams
   - Circuit breaker for failures
   Result: Prevents system overload
```
