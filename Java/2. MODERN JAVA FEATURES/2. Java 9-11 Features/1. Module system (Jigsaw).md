# Java Module System (Project Jigsaw)

## Overview

The Java Platform Module System (JPMS), introduced in Java 9 as Project Jigsaw, represents a fundamental change in how Java applications are structured and deployed. Modules provide strong encapsulation, reliable configuration, and explicit dependencies at the language level. This guide covers module-info.java syntax, the core directives (requires, exports, opens), the service provider interface pattern (provides, uses), how the JDK itself is modularized, compatibility features (unnamed and automatic modules), practical migration strategies, and an honest assessment of benefits and challenges.

## 1. module-info.java Syntax

### Declaring Modules

```java
/**
 * MODULE-INFO.JAVA SYNTAX
 * 
 * The module declaration file
 */

/**
 * Basic module declaration
 */

// File: module-info.java (must be at source root)
module com.example.myapp {
    // Module directives go here
}

/**
 * BASIC SYNTAX:
 * 
 * module <module.name> {
 *   // directives
 * }
 * 
 * Location:
 * - Must be named module-info.java
 * - Must be at source root (not in package)
 * - One per module
 * 
 * Module name:
 * - Recommended: reverse DNS (com.example.module)
 * - Can use underscores, numbers
 * - Cannot use Java keywords
 * - Conventionally matches main package
 * 
 * Example structure:
 * src/
 *   module-info.java
 *   com/
 *     example/
 *       myapp/
 *         Main.java
 */

/**
 * Module with dependencies
 */
module com.example.app {
    // Depend on other modules
    requires java.base;        // Implicit, can be omitted
    requires java.sql;
    requires java.logging;
    
    // Export packages (make them public)
    exports com.example.app.api;
    exports com.example.app.model;
    
    // Don't export internal packages
    // (com.example.app.internal is private)
}

/**
 * MODULE DIRECTIVES:
 * 
 * requires:  Declares dependency on another module
 * exports:   Makes package available to other modules
 * opens:     Opens package for reflection
 * uses:      Declares service consumption
 * provides:  Declares service implementation
 * 
 * Modifiers:
 * transitive:    Requires is transitive (exported to dependents)
 * static:        Compile-time only dependency
 * to:            Qualified export/open (specific modules only)
 */

/**
 * Complete example
 */
module com.example.library {
    // Dependencies
    requires java.logging;
    requires transitive java.sql;  // Transitive dependency
    
    // Public API
    exports com.example.library.api;
    exports com.example.library.model;
    exports com.example.library.util to com.example.app;  // Qualified
    
    // Reflection support
    opens com.example.library.entity;
    
    // Services
    uses com.example.library.api.Plugin;
    provides com.example.library.api.Plugin 
        with com.example.library.impl.DefaultPlugin;
}

/**
 * NAMING CONVENTIONS:
 * 
 * Good names:
 * - com.example.myapp
 * - org.project.module
 * - io.github.username.library
 * 
 * Bad names:
 * - myapp (too generic)
 * - com-example-myapp (use dots, not dashes)
 * - class (keyword)
 * 
 * Match to packages:
 * If main package is com.example.myapp
 * Module should be com.example.myapp
 */

/**
 * File location and structure
 */
public class ModuleStructure {
    /**
     * TYPICAL MODULE STRUCTURE:
     * 
     * mymodule/
     *   src/
     *     module-info.java           ← Module declaration
     *     com/
     *       example/
     *         mymodule/
     *           Main.java
     *           api/
     *             PublicAPI.java     ← Exported
     *           internal/
     *             Helper.java        ← Not exported
     *   
     * MAVEN STRUCTURE:
     * 
     * mymodule/
     *   pom.xml
     *   src/
     *     main/
     *       java/
     *         module-info.java
     *         com/example/mymodule/...
     * 
     * GRADLE STRUCTURE:
     * 
     * mymodule/
     *   build.gradle
     *   src/
     *     main/
     *       java/
     *         module-info.java
     *         com/example/mymodule/...
     */
}

/**
 * Common patterns
 */
module com.example.patterns {
    /**
     * PATTERN 1: Library module
     * Provides API, hides implementation
     */
    exports com.example.patterns.api;
    // Internal packages not exported
    
    /**
     * PATTERN 2: Application module
     * Requires dependencies, doesn't export
     */
    requires java.sql;
    requires com.example.library;
    // No exports (application, not library)
    
    /**
     * PATTERN 3: Framework module
     * Uses services for extensibility
     */
    uses com.example.patterns.spi.Extension;
    
    /**
     * PATTERN 4: Plugin module
     * Provides service implementation
     */
    provides com.example.patterns.spi.Extension
        with com.example.patterns.impl.MyExtension;
}
```

## 2. requires, exports, opens

### Core Module Directives

```java
/**
 * REQUIRES, EXPORTS, OPENS
 * 
 * The fundamental module directives
 */

/**
 * requires - Declare dependencies
 */
module com.example.app {
    // Basic require
    requires java.sql;
    
    // Transitive require
    requires transitive java.logging;
    
    // Static require (compile-time only)
    requires static org.junit.jupiter.api;
}

/**
 * REQUIRES DIRECTIVE:
 * 
 * Basic form:
 * requires <module.name>;
 * 
 * Purpose:
 * - Declares dependency on another module
 * - Makes that module's exported packages available
 * - Creates module dependency graph
 * 
 * At runtime:
 * - Required module must be present
 * - Transitive dependencies resolved automatically
 * 
 * Example:
 * requires java.sql;
 * → Can use java.sql.* packages
 */

/**
 * requires transitive
 */
module com.example.database {
    requires transitive java.sql;
    exports com.example.database;
}

module com.example.app {
    requires com.example.database;
    // Automatically gets java.sql (transitive)
}

/**
 * REQUIRES TRANSITIVE:
 * 
 * Syntax:
 * requires transitive <module.name>;
 * 
 * Purpose:
 * - Dependency is re-exported
 * - Modules requiring you get this dependency too
 * 
 * Use when:
 * - Your API exposes types from dependency
 * - Dependents will need the same module
 * 
 * Example:
 * // In library module
 * requires transitive java.sql;
 * 
 * public Connection getConnection() {
 *   // Returns java.sql.Connection
 *   // Callers need java.sql module
 * }
 * 
 * Without transitive:
 * - Callers would need: requires java.sql
 * 
 * With transitive:
 * - Callers automatically get java.sql
 */

/**
 * requires static
 */
module com.example.library {
    // Optional dependency - only needed at compile time
    requires static lombok;
    requires static org.junit.jupiter.api;
}

/**
 * REQUIRES STATIC:
 * 
 * Syntax:
 * requires static <module.name>;
 * 
 * Purpose:
 * - Compile-time only dependency
 * - Not required at runtime
 * 
 * Use when:
 * - Annotation processors (Lombok)
 * - Test frameworks (JUnit)
 * - Optional features
 * 
 * Example:
 * requires static lombok;
 * → Can use @Data etc at compile time
 * → Not needed in runtime classpath
 */

/**
 * exports - Make packages public
 */
module com.example.library {
    // Public API - available to all modules
    exports com.example.library.api;
    exports com.example.library.model;
    
    // Internal - NOT exported (private to module)
    // package com.example.library.internal
}

/**
 * EXPORTS DIRECTIVE:
 * 
 * Syntax:
 * exports <package.name>;
 * exports <package.name> to <module1>, <module2>;
 * 
 * Purpose:
 * - Makes package visible to other modules
 * - Strong encapsulation boundary
 * 
 * Rules:
 * - Only exports types (classes, interfaces)
 * - NOT module itself
 * - Unexported packages are private
 * 
 * Example:
 * exports com.example.api;
 * → Other modules can access com.example.api.*
 * 
 * Not exported:
 * com.example.internal.*
 * → Other modules CANNOT access
 * → Compile error if they try
 */

/**
 * Qualified exports
 */
module com.example.library {
    // Only visible to specific modules
    exports com.example.library.internal 
        to com.example.app,
           com.example.test;
}

module com.example.app {
    requires com.example.library;
    // Can access com.example.library.internal
}

module com.example.other {
    requires com.example.library;
    // CANNOT access com.example.library.internal
}

/**
 * QUALIFIED EXPORTS:
 * 
 * Syntax:
 * exports <package> to <module1>, <module2>;
 * 
 * Purpose:
 * - Selective visibility
 * - Friend modules pattern
 * - Test-only exports
 * 
 * Use cases:
 * - Internal APIs for specific modules
 * - Test utilities
 * - Framework internals
 * 
 * Example:
 * exports com.example.internal to com.example.test;
 * → Only test module can access internal package
 */

/**
 * opens - Reflection access
 */
module com.example.app {
    // Open for reflection at runtime
    opens com.example.app.entity;
    opens com.example.app.dto to com.fasterxml.jackson.databind;
}

/**
 * OPENS DIRECTIVE:
 * 
 * Syntax:
 * opens <package>;
 * opens <package> to <module1>, <module2>;
 * 
 * Purpose:
 * - Allow reflection access
 * - Needed for frameworks
 * 
 * What it enables:
 * - setAccessible(true)
 * - Access private fields/methods
 * - Needed by: JPA, Jackson, Spring, etc.
 * 
 * vs exports:
 * exports: Compile-time access
 * opens:   Runtime reflection access
 * 
 * Can combine:
 * exports com.example.api;      // Public API
 * opens com.example.entity;     // For JPA
 */

/**
 * open module
 */
open module com.example.framework {
    requires java.sql;
    exports com.example.framework.api;
}

/**
 * OPEN MODULE:
 * 
 * Syntax:
 * open module <name> { ... }
 * 
 * Effect:
 * - ALL packages open for reflection
 * - No need for individual 'opens' directives
 * 
 * Use when:
 * - Framework that uses heavy reflection
 * - Migration from classpath
 * - Testing modules
 * 
 * Trade-off:
 * - Convenience vs encapsulation
 * - Prefer specific 'opens' for libraries
 */

/**
 * Complete example showing all directives
 */
module com.example.complete {
    // Dependencies
    requires java.logging;
    requires transitive java.sql;
    requires static lombok;
    
    // Public API
    exports com.example.complete.api;
    exports com.example.complete.model;
    
    // Friend modules
    exports com.example.complete.internal 
        to com.example.complete.test;
    
    // Reflection access
    opens com.example.complete.entity;
    opens com.example.complete.dto 
        to com.fasterxml.jackson.databind;
}

/**
 * Strong encapsulation example
 */
module com.example.encapsulation {
    exports com.example.encapsulation.api;
    
    // These packages are PRIVATE to the module:
    // - com.example.encapsulation.impl
    // - com.example.encapsulation.internal
    // - com.example.encapsulation.util
    
    // Other modules CANNOT access them
    // Even with reflection (unless opened)
}

/**
 * Common patterns
 */
public class DirectivePatterns {
    /**
     * PATTERN 1: Library module
     */
    // module com.example.library {
    //     requires transitive java.sql;  // API exposes Connection
    //     exports com.example.library.api;
    // }
    
    /**
     * PATTERN 2: Application module
     */
    // module com.example.app {
    //     requires com.example.library;
    //     requires java.logging;
    //     // No exports (it's an application)
    // }
    
    /**
     * PATTERN 3: Persistence module
     */
    // module com.example.persistence {
    //     requires java.sql;
    //     requires java.persistence;
    //     exports com.example.persistence.dao;
    //     opens com.example.persistence.entity;  // For JPA
    // }
    
    /**
     * PATTERN 4: Test module
     */
    // module com.example.test {
    //     requires com.example.library;
    //     requires static org.junit.jupiter.api;
    //     opens com.example.test to org.junit.platform.commons;
    // }
}
```

## 3. Services (provides, uses)

### Service Provider Interface Pattern

```java
/**
 * SERVICES - provides and uses
 * 
 * Service provider interface pattern in modules
 */

/**
 * Service interface (SPI)
 */
// File: src/module-info.java
module com.example.api {
    exports com.example.api;
}

// File: src/com/example/api/MessageService.java
package com.example.api;

public interface MessageService {
    void sendMessage(String message);
    String getProviderName();
}

/**
 * Service consumer
 */
module com.example.app {
    requires com.example.api;
    uses com.example.api.MessageService;
}

// File: src/com/example/app/Main.java
package com.example.app;

import com.example.api.MessageService;
import java.util.ServiceLoader;

public class Main {
    public static void main(String[] args) {
        ServiceLoader<MessageService> loader = 
            ServiceLoader.load(MessageService.class);
        
        for (MessageService service : loader) {
            System.out.println("Provider: " + service.getProviderName());
            service.sendMessage("Hello, World!");
        }
    }
}

/**
 * USES DIRECTIVE:
 * 
 * Syntax:
 * uses <service.interface>;
 * 
 * Purpose:
 * - Declares that module uses a service
 * - Enables ServiceLoader.load()
 * 
 * Example:
 * uses com.example.api.Plugin;
 * 
 * Then:
 * ServiceLoader<Plugin> loader = ServiceLoader.load(Plugin.class);
 * 
 * Note:
 * - Service interface must be visible (exported)
 * - Implementations discovered at runtime
 */

/**
 * Service provider
 */
module com.example.email {
    requires com.example.api;
    provides com.example.api.MessageService 
        with com.example.email.EmailService;
}

// File: src/com/example/email/EmailService.java
package com.example.email;

import com.example.api.MessageService;

public class EmailService implements MessageService {
    @Override
    public void sendMessage(String message) {
        System.out.println("Sending email: " + message);
    }
    
    @Override
    public String getProviderName() {
        return "Email Service";
    }
}

/**
 * PROVIDES DIRECTIVE:
 * 
 * Syntax:
 * provides <service.interface> with <implementation.class>;
 * 
 * Purpose:
 * - Declares service implementation
 * - Makes implementation discoverable
 * 
 * Example:
 * provides com.example.api.Plugin 
 *   with com.example.impl.MyPlugin;
 * 
 * Requirements:
 * - Implementation must have public no-arg constructor
 * - OR provide static 'provider()' method
 * 
 * Multiple implementations:
 * provides Service 
 *   with Impl1, Impl2, Impl3;
 */

/**
 * Multiple service providers
 */
module com.example.sms {
    requires com.example.api;
    provides com.example.api.MessageService 
        with com.example.sms.SmsService;
}

module com.example.slack {
    requires com.example.api;
    provides com.example.api.MessageService 
        with com.example.slack.SlackService;
}

// Application discovers all three:
// - EmailService
// - SmsService  
// - SlackService

/**
 * MULTIPLE PROVIDERS:
 * 
 * Can have many modules providing same service
 * ServiceLoader finds all of them
 * 
 * module provider1 {
 *   provides Service with Impl1;
 * }
 * 
 * module provider2 {
 *   provides Service with Impl2;
 * }
 * 
 * ServiceLoader.load(Service.class)
 * → Returns both Impl1 and Impl2
 */

/**
 * Service provider with factory method
 */
module com.example.factory {
    requires com.example.api;
    provides com.example.api.MessageService 
        with com.example.factory.FactoryService;
}

// File: src/com/example/factory/FactoryService.java
package com.example.factory;

import com.example.api.MessageService;

public class FactoryService implements MessageService {
    private FactoryService() {
        // Private constructor
    }
    
    // ServiceLoader calls this instead of constructor
    public static FactoryService provider() {
        return new FactoryService();
    }
    
    @Override
    public void sendMessage(String message) {
        System.out.println("Factory service: " + message);
    }
    
    @Override
    public String getProviderName() {
        return "Factory Service";
    }
}

/**
 * PROVIDER METHODS:
 * 
 * Two ways to instantiate:
 * 
 * 1. Public no-arg constructor:
 *    public class MyService implements Service {
 *      public MyService() { ... }
 *    }
 * 
 * 2. Static provider() method:
 *    public class MyService implements Service {
 *      private MyService() { ... }
 *      public static MyService provider() {
 *        return new MyService();
 *      }
 *    }
 * 
 * Use provider() for:
 * - Singleton pattern
 * - Complex initialization
 * - Dependency injection
 */

/**
 * Complete plugin system example
 */

// API module
module com.example.plugin.api {
    exports com.example.plugin.api;
}

// package com.example.plugin.api;
interface Plugin {
    String getName();
    void execute();
}

// Application module
module com.example.plugin.app {
    requires com.example.plugin.api;
    uses com.example.plugin.api.Plugin;
}

// Plugin implementation 1
module com.example.plugin.logger {
    requires com.example.plugin.api;
    provides com.example.plugin.api.Plugin 
        with com.example.plugin.logger.LoggerPlugin;
}

// Plugin implementation 2
module com.example.plugin.metrics {
    requires com.example.plugin.api;
    provides com.example.plugin.api.Plugin 
        with com.example.plugin.metrics.MetricsPlugin;
}

/**
 * PLUGIN SYSTEM PATTERN:
 * 
 * Structure:
 * 1. API module: Defines interface
 * 2. App module: Uses service
 * 3. Plugin modules: Provide implementations
 * 
 * Benefits:
 * - Loose coupling
 * - Runtime discovery
 * - Extensibility
 * - No compile-time dependency on plugins
 * 
 * Use cases:
 * - Plugin architectures
 * - Driver systems (JDBC)
 * - Extension points
 * - Framework providers
 */

/**
 * Service vs traditional dependency
 */
public class ServiceVsDependency {
    /**
     * TRADITIONAL DEPENDENCY:
     * 
     * module app {
     *   requires library;
     * }
     * 
     * → Compile-time dependency
     * → Must be present at compile time
     * → Tightly coupled
     * 
     * 
     * SERVICE PATTERN:
     * 
     * module app {
     *   uses Service;
     * }
     * 
     * module provider {
     *   provides Service with Impl;
     * }
     * 
     * → Runtime discovery
     * → Provider can be added/removed
     * → Loosely coupled
     * 
     * 
     * WHEN TO USE SERVICES:
     * 
     * ✓ Plugin systems
     * ✓ Multiple implementations
     * ✓ Runtime extensibility
     * ✓ Decoupled architecture
     * 
     * ✗ Single implementation
     * ✗ Tight coupling acceptable
     * ✗ Simple applications
     */
}
```

## 4. Modular JDK

### Understanding the Modular Java Platform

```java
/**
 * MODULAR JDK
 * 
 * How the JDK itself is modularized
 */

/**
 * JDK module structure
 */
public class ModularJDK {
    /**
     * CORE JDK MODULES:
     * 
     * java.base
     * - Core classes: Object, String, Integer, etc.
     * - Collections: List, Map, Set
     * - I/O: InputStream, OutputStream
     * - Concurrency: Thread, Lock, Executor
     * - Implicitly required by ALL modules
     * 
     * java.sql
     * - JDBC API
     * - Connection, Statement, ResultSet
     * 
     * java.xml
     * - XML processing
     * - DOM, SAX, StAX
     * 
     * java.logging
     * - java.util.logging API
     * 
     * java.desktop
     * - AWT, Swing
     * - java.awt.*, javax.swing.*
     * 
     * java.net.http
     * - HTTP Client API (Java 11+)
     */
}

 * Implicit dependency on java.base
```

## 5. Unnamed and Automatic Modules

### Compatibility with Classpath-based Code

```java
/**
 * UNNAMED AND AUTOMATIC MODULES
 * 
 * Migration compatibility features
 */

/**
 * Unnamed module
 */
public class UnnamedModule {
    /**
     * UNNAMED MODULE:
     * 
     * What is it:
     * - Code on the classpath (not module path)
     * - Legacy JARs without module-info.java
     * - Default for non-modular code
     * 
     * Characteristics:
     * - All packages exported
     * - Reads all other modules
     * - Cannot be read by named modules
     * 
     * Example:
     * $ java -cp myapp.jar com.example.Main
     * 
     * → myapp.jar is in unnamed module
     * → Can access all platform modules
     * → Other modules cannot access it
     * 
     * Use when:
     * - Running legacy code
     * - Not ready to modularize
     * - Quick migration
     */
    
    /**
     * UNNAMED MODULE RULES:
     * 
     * Can do:
     * ✓ Access all JDK modules
     * ✓ Access other classpath code
     * ✓ Use reflection freely
     * 
     * Cannot do:
     * ✗ Be required by named modules
     * ✗ Use services from modules (without --add-modules)
     * ✗ Have strong encapsulation
     * 
     * Mixing:
     * Named modules can't see unnamed
     * Unnamed can see named (with --add-modules)
     */
}

/**
 * Automatic module
 */
public class AutomaticModule {
    /**
     * AUTOMATIC MODULE:
     * 
     * What is it:
     * - JAR on module path without module-info.java
     * - Bridge between classpath and modules
     * - Automatically becomes a module
     * 
     * Module name derived from:
     * 1. Automatic-Module-Name in MANIFEST.MF (preferred)
     * 2. JAR filename (fallback)
     * 
     * Characteristics:
     * - All packages exported
     * - Reads all other modules
     * - CAN be required by named modules
     * 
     * Example:
     * $ java --module-path libs:mods \
     *        --module com.example.app
     * 
     * If libs/ contains oldlib.jar (no module-info):
     * → Becomes automatic module named "oldlib"
     * → Can be required by modules
     */
    
    /**
     * MANIFEST.MF APPROACH (Recommended):
     * 
     * In library's MANIFEST.MF:
     * Automatic-Module-Name: com.example.library
     * 
     * Then:
     * module com.example.app {
     *   requires com.example.library;  // Use MANIFEST name
     * }
     * 
     * Benefits:
     * - Stable name (not dependent on filename)
     * - Library can choose name
     * - Forward compatibility
     * 
     * Many popular libraries added this:
     * - SLF4J: org.slf4j
     * - Jackson: com.fasterxml.jackson.core
     * - Guava: com.google.common
     */
    
    /**
     * FILENAME-DERIVED NAME:
     * 
     * If no Automatic-Module-Name:
     * 
     * JAR filename:          Module name:
     * commons-lang3-3.12.jar → commons.lang3
     * guava-30.1-jre.jar     → guava
     * slf4j-api-1.7.30.jar   → slf4j.api
     * 
     * Rules:
     * 1. Remove .jar extension
     * 2. Remove version numbers
     * 3. Replace invalid chars with '.'
     * 4. Remove dots before/after digits
     * 
     * Problems:
     * - Filename changes = name changes
     * - Unstable
     * - Avoid if possible
     */
}

/**
```
```java
 * Bottom-up migration
 */
public class BottomUpMigration {
    /**
     * BOTTOM-UP STRATEGY:
     * 
     * Start with leaf dependencies, work up
     * 
     * Step 1: Identify dependency tree
     * 
     *     app
     *      ├─ service
     *      │   └─ utils
     *      └─ dao
     *          └─ utils
     * 
     * Step 2: Modularize leaves first
     * 
     *     utils → module
     * 
     * Step 3: Move up the tree
     * 
     *     dao → module (requires utils)
     *     service → module (requires utils)
     * 
     * Step 4: Finally, root
     * 
     *     app → module (requires dao, service)
     * 
     * Benefits:
     * ✓ Dependencies already modular
     * ✓ No circular dependencies
     * ✓ Gradual progress
     * 
     * Challenges:
     * ✗ Requires control of dependencies
     * ✗ Slower for large trees
     */
    
    /**
     * BOTTOM-UP EXAMPLE:
     * 
     * Week 1: Modularize utilities
     * // utils/module-info.java
     * module com.example.utils {
     *   exports com.example.utils;
     * }
     * 
     * Week 2: Modularize data layer
     * // dao/module-info.java
     * module com.example.dao {
     *   requires com.example.utils;
     *   requires java.sql;
     *   exports com.example.dao;
     * }
     * 
     * Week 3: Modularize service layer
     * // service/module-info.java
     * module com.example.service {
     *   requires com.example.utils;
     *   requires com.example.dao;
     *   exports com.example.service;
     * }
     * 
     * Week 4: Modularize application
     * // app/module-info.java
     * module com.example.app {
     *   requires com.example.service;
     *   requires java.logging;
     * }
     */
}

/**
 * Top-down migration
 */
public class TopDownMigration {
    /**
     * TOP-DOWN STRATEGY:
     * 
     * Start with application, dependencies become automatic
     * 
     * Step 1: Modularize application
     * 
     * module com.example.app {
     *   requires utils;      // Automatic module
     *   requires dao;        // Automatic module
     *   requires service;    // Automatic module
     * }
     * 
     * Step 2: Move dependencies to module path
     * 
     * $ java --module-path lib:mods --module com.example.app

     * Benefits:
     * ✓ Faster initial migration
     * ✓ Application modular quickly
     * ✓ Dependencies can wait
     * 
     * Challenges:
     * ✗ Relies on automatic modules
     * ✗ Less encapsulation initially
     * ✗ May have issues with split packages
     */
    
    /**
     * TOP-DOWN EXAMPLE:
     * 
     * Day 1: Add module-info.java to app
     * module com.example.app {
     *   requires utils;
     *   requires dao;
     *   requires service;
     * }
     * 
     * Week 1: utils.jar is automatic module
     * Week 2: dao.jar is automatic module
     * Week 3: service.jar is automatic module

     * Month 2: Convert utils to named module
     * // utils/module-info.java
     * module com.example.utils {
     *   exports com.example.utils;
     * }
     * 
     * Month 3: Convert dao to named module
     * Month 4: Convert service to named module
     *

     */
}

/**
 * Common migration issues
 */
public class MigrationIssues {
    /**
     * ISSUE 1: Split packages
     * 
     * Problem:
     * Same package in multiple JARs
     * 
     * Example:
     * myapp.jar:     com.example.util.StringUtils
     * mylib.jar:     com.example.util.DateUtils
     * 
     * Error:
     * "Package com.example.util in both module1 and module2"
     * 
     * Solutions:
     * 1. Merge into one module
     * 2. Rename packages
     * 3. Use --patch-module (temporary)
     * 
     * $ java --patch-module com.example.util=myapp.jar:mylib.jar
     */
    
    /**
     * ISSUE 2: Reflection access
     * 
     * Problem:
     * Framework uses reflection on private members
     * 
     * Example:
     * JPA, Jackson, Spring need reflection
     * 
     * Error:
     * "Unable to make field private X accessible"
     * 
     * Solutions:
     * 1. Add opens directive:
     *    opens com.example.entity;
     * 
     * 2. Qualified opens:
     *    opens com.example.entity to org.hibernate.orm.core;
     * 
     * 3. Use --add-opens (temporary):
     *    $ java --add-opens com.example/com.example.entity=ALL-UNNAMED
     */
    
    /**
     * ISSUE 3: Missing dependencies
     * 
     * Problem:
     * Dependencies not declared (implicit on classpath)
     * 
     * Example:
     * Code uses SLF4J but doesn't declare it
     * 
     * Error:
     * "Package org.slf4j is not visible"
     * 
     * Solution:
     * Add requires directive:
     * requires org.slf4j;
     */
    
    /**
     * ISSUE 4: Removed modules
     * 
     * Problem:
     * Java EE modules removed in Java 11
     * 
     * Example:
     * Code uses JAXB (java.xml.bind)
     * 
     * Error (Java 11+):
     * "Module java.xml.bind not found"
     * 
     * Solution:
     * Add external dependency:
     * 
     * Maven:
     * <dependency>
     *   <groupId>jakarta.xml.bind</groupId>
     *   <artifactId>jakarta.xml.bind-api</artifactId>
     * </dependency>
     * 
     * Module:
     * requires jakarta.xml.bind;
     */
    
    /**
     * ISSUE 5: Circular dependencies
     * 
     * Problem:
     * Module A requires B, B requires A
     * 
     * Error:
     * "Cyclic dependence involving ModuleA"
     * 
     * Solutions:
     * 1. Refactor to remove cycle
     * 2. Extract common code to new module
     * 3. Use services instead of requires
     * 
     * Before:
     * module A { requires B; }
     * module B { requires A; }  // ERROR
     * 
     * After:
     * module common {}
     * module A { requires common; }
     * module B { requires common; }
     */
}
```

```java
    // Public packages
    exports com.example.myapp.api;
    exports com.example.myapp.util to com.example.test;
    
    // Reflection
    opens com.example.myapp.entity;
    
    // Services
    uses com.example.api.Plugin;
    provides com.example.api.Plugin with com.example.impl.MyPlugin;
}
```

**Key Directives:**

```
requires          - Depend on module
requires trans    - Transitive dependency
requires static   - Compile-time only
exports           - Make package public
exports...to      - Qualified export
opens             - Allow reflection
uses              - Use service
provides...with   - Provide service
```

**Module Types:**

```
Named:      Has module-info.java, on module path
Automatic:  No module-info.java, on module path
Unnamed:    On classpath (legacy)
```

**Resolution Rules:**

```
Rule 1: Named modules can require named/automatic
Rule 2: Named cannot require unnamed
Rule 3: Unnamed can access everything
Rule 4: Automatic exports all, reads all
```

**Key insights:**

- **Strong encapsulation**: Unexported packages truly private
- **Reliable configuration**: Missing deps fail fast
- **Smaller runtimes**: jlink creates ~30MB JREs
- **Migration**: Use automatic modules as bridge
- **Optional**: Classpath still works, modules not mandatory

**The revolution:**

- Java 9+: JDK itself is modular (90+ modules)
- Enables custom runtimes, better architecture
- But: Migration can be complex, ecosystem still adapting
