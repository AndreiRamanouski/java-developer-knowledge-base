# Concurrency Design Patterns

## Overview

Concurrency patterns solve the fundamental problem of thread safety and efficient parallel execution—how to coordinate multiple threads, share data safely, and leverage multi-core processors without deadlocks, race conditions, or performance bottlenecks. Instead of error-prone manual synchronization (synchronized blocks everywhere, deadlocks, race conditions), concurrency patterns provide proven solutions: Thread Pool manages worker threads efficiently (ExecutorService instead of new Thread()), Producer-Consumer decouples production and consumption with queues (BlockingQueue handles synchronization), Read-Write Lock allows concurrent reads (ReentrantReadWriteLock), Double-Checked Locking enables thread-safe lazy initialization (requires volatile), Future Pattern enables asynchronous computation (CompletableFuture for non-blocking), and Active Object decouples method invocation from execution (message queue + worker thread). These patterns are essential in modern Java—ExecutorService is the standard for thread management, BlockingQueue powers concurrent pipelines, CompletableFuture enables reactive programming, and proper volatile usage prevents subtle bugs. Understanding concurrency patterns means understanding how to build scalable, correct multi-threaded applications that leverage modern hardware without the common pitfalls of race conditions, deadlocks, and performance degradation. This guide covers Thread Pool (types, sizing, rejection policies), Producer-Consumer (BlockingQueue implementations, backpressure), Read-Write Lock (concurrent reads, write locks), Double-Checked Locking (broken vs correct with volatile), Thread-Safe Lazy Initialization (lazy holder idiom), Future Pattern (CompletableFuture, parallel streams), and Active Object (asynchronous method execution).

## 1. Thread Pool Pattern

### Efficient Thread Management

```java
/**
 * THREAD POOL PATTERN
 * 
 * Reuses fixed number of threads for executing tasks
 * Avoids overhead of creating new threads repeatedly
 */

// ========================================
// PROBLEM: CREATING THREADS DIRECTLY
// ========================================
class BadThreadManagement {
    public void processTasks(List<Task> tasks) {
        for (Task task : tasks) {
            new Thread(() -> processTask(task)).start();  // ❌ Creates new thread each time!
        }
    }
    
    /**
     * PROBLEMS:
     * ✗ Thread creation is expensive (1-2ms per thread)
     * ✗ Unbounded thread creation (can exhaust resources)
     * ✗ No control over concurrency level
     * ✗ Hard to manage thread lifecycle
     * ✗ 10,000 tasks = 10,000 threads!
     */
    
    private void processTask(Task task) { /* ... */ }
}

// ========================================
// SOLUTION: THREAD POOL
// ========================================

import java.util.concurrent.*;

class TaskProcessor {
    private final ExecutorService executor;
    
    public TaskProcessor(int poolSize) {
        this.executor = Executors.newFixedThreadPool(poolSize);
    }
    
    public void processTasks(List<Task> tasks) {
        for (Task task : tasks) {
            executor.submit(() -> processTask(task));  // ✓ Reuses threads!
        }
    }
    
    public void shutdown() {
        executor.shutdown();
        try {
            if (!executor.awaitTermination(60, TimeUnit.SECONDS)) {
                executor.shutdownNow();
            }
        } catch (InterruptedException e) {
            executor.shutdownNow();
            Thread.currentThread().interrupt();
        }
    }
    
    private void processTask(Task task) { /* ... */ }
}

/**
 * BENEFITS:
 * 
 * ✓ Thread reuse (no creation overhead)
 * ✓ Controlled concurrency
 * ✓ Resource management
 * ✓ Task queuing when pool full
 * 
 * 
 * PERFORMANCE:
 * 
 * 10,000 tasks:
 * - Direct threads: ~20 seconds + OutOfMemoryError
 * - Thread pool (10 threads): ~2 seconds, stable memory
 */

// ========================================
// EXECUTOR SERVICE TYPES
// ========================================

class ExecutorExamples {
    
    // 1. Fixed Thread Pool - Most common
    public void fixedThreadPool() {
        ExecutorService executor = Executors.newFixedThreadPool(10);
        
        /**
         * - Fixed number of threads (10)
         * - Unbounded queue
         * - Good for: CPU-bound tasks, known workload
         * - Threads stay alive waiting for work
         */
        
        executor.shutdown();
    }
    
    // 2. Cached Thread Pool - Elastic
    public void cachedThreadPool() {
        ExecutorService executor = Executors.newCachedThreadPool();
        
        /**
         * - Creates threads as needed
         * - Threads idle for 60s are terminated
         * - Good for: Many short-lived async tasks
         * - Bad for: Long-running tasks (unbounded growth)
         */
        
        executor.shutdown();
    }
    
    // 3. Single Thread Executor - Sequential
    public void singleThreadExecutor() {
        ExecutorService executor = Executors.newSingleThreadExecutor();
        
        /**
         * - Only one thread
         * - Tasks executed sequentially
         * - Good for: Maintaining order, event loop
         * - Guarantees happen-before relationship
         */
        
        executor.shutdown();
    }
    
    // 4. Scheduled Thread Pool - Delayed/Periodic
    public void scheduledThreadPool() {
        ScheduledExecutorService executor = 
            Executors.newScheduledThreadPool(5);
        
        // One-time delayed task
        executor.schedule(() -> {
            System.out.println("Executed after 5 seconds");
        }, 5, TimeUnit.SECONDS);
        
        // Periodic task (fixed rate)
        executor.scheduleAtFixedRate(() -> {
            System.out.println("Executed every 10 seconds");
        }, 0, 10, TimeUnit.SECONDS);
        
        // Periodic task (fixed delay)
        executor.scheduleWithFixedDelay(() -> {
            System.out.println("Executed 5 seconds after previous completion");
        }, 0, 5, TimeUnit.SECONDS);
        
        /**
         * scheduleAtFixedRate: Fixed period regardless of execution time
         * scheduleWithFixedDelay: Fixed delay between executions
         */
        
        executor.shutdown();
    }
    
    // 5. Custom Thread Pool - Full control
    public void customThreadPool() {
        ThreadPoolExecutor executor = new ThreadPoolExecutor(
            5,                              // Core pool size
            10,                             // Maximum pool size
            60,                             // Keep-alive time
            TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(100), // Queue (bounded!)
            new ThreadPoolExecutor.CallerRunsPolicy()  // Rejection policy
        );
        
        /**
         * Core pool: Always alive
         * Max pool: Created when queue full
         * Keep-alive: Extra threads timeout
         * Queue: Holds pending tasks
         * Rejection policy: What to do when queue full
         */
        
        executor.shutdown();
    }
}

// ========================================
// REJECTION POLICIES
// ========================================
/**
 * When thread pool + queue is full:
 * 
 * 1. AbortPolicy (default)
 *    - Throws RejectedExecutionException
 *    - Fail fast
 * 
 * 2. CallerRunsPolicy
 *    - Submitting thread executes task
 *    - Natural backpressure
 * 
 * 3. DiscardPolicy
 *    - Silently drops task
 *    - Use with caution!
 * 
 * 4. DiscardOldestPolicy
 *    - Drops oldest queued task
 *    - Submits new task
 * 
 * 5. Custom
 *    - Implement RejectedExecutionHandler
 */

// ========================================
// THREAD POOL SIZING
// ========================================
class ThreadPoolSizing {
    
    /**
     * SIZING FORMULAS:
     * 
     * CPU-bound tasks:
     *   threads = cores + 1
     *   (Little benefit from more threads)
     * 
     * I/O-bound tasks:
     *   threads = cores * (1 + wait_time / compute_time)
     *   
     * Example:
     *   8 cores
     *   90% I/O wait (wait_time/compute_time = 9)
     *   threads = 8 * (1 + 9) = 80 threads
     */
    
    public static int calculatePoolSize() {
        int cores = Runtime.getRuntime().availableProcessors();
        
        // CPU-bound
        int cpuBoundPoolSize = cores + 1;
        
        // I/O-bound (assuming 90% wait)
        int ioBoundPoolSize = cores * 10;
        
        return ioBoundPoolSize;
    }
}

// REAL-WORLD EXAMPLE: WEB SERVER
// ========================================
class WebServer {
    private final ExecutorService requestExecutor;
    private final ScheduledExecutorService maintenanceExecutor;
    
    public WebServer() {
        // Request processing pool
        int cores = Runtime.getRuntime().availableProcessors();
        this.requestExecutor = new ThreadPoolExecutor(
            cores * 2,                      // Core threads
            cores * 4,                      // Max threads

        // Maintenance tasks pool

        // Request processing logic
        return new HttpResponse();
    }
    
    private void sendResponse(HttpResponse response) { }
    private void sendErrorResponse(Exception e) { }
    private void cleanupExpiredSessions() { }
    
    public void shutdown() {
        requestExecutor.shutdown();
        maintenanceExecutor.shutdown();
    }
}
```

## 2. Producer-Consumer Pattern

### Decoupling Production and Consumption

```java
/**
 * PRODUCER-CONSUMER PATTERN
 * 
 * Producers generate data, consumers process it
 * Queue decouples them (thread-safe handoff)
 */

// ========================================
// CLASSIC PRODUCER-CONSUMER
// ========================================

import java.util.concurrent.*;

class DataProducer implements Runnable {
    private final BlockingQueue<String> queue;
    private final int id;
    
    public DataProducer(BlockingQueue<String> queue, int id) {
        this.queue = queue;
        this.id = id;
    }
    
    @Override
    public void run() {
        try {
            for (int i = 0; i < 10; i++) {
                String data = "Data-" + id + "-" + i;
                System.out.println("Producer " + id + " producing: " + data);
                queue.put(data);  // Blocks if queue full
                Thread.sleep(100);
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
}

class DataConsumer implements Runnable {
    private final BlockingQueue<String> queue;
    private final int id;
    
    public DataConsumer(BlockingQueue<String> queue, int id) {
        this.queue = queue;
        this.id = id;
    }
    
    @Override
    public void run() {
        try {
            while (true) {
                String data = queue.take();  // Blocks if queue empty
                System.out.println("Consumer " + id + " consuming: " + data);
                Thread.sleep(200);  // Simulate processing
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
}

class ProducerConsumerDemo {
    public static void main(String[] args) {
        BlockingQueue<String> queue = new ArrayBlockingQueue<>(10);
        
        // Start producers
        ExecutorService executorService = Executors.newCachedThreadPool();
        executorService.submit(new DataProducer(queue, 1));
        executorService.submit(new DataProducer(queue, 2));
        
        // Start consumers
        executorService.submit(new DataConsumer(queue, 1));
        executorService.submit(new DataConsumer(queue, 2));
        executorService.submit(new DataConsumer(queue, 3));
        
        /**
         * 2 producers, 3 consumers
         * Queue size = 10
         * BlockingQueue handles all synchronization!
         */
    }
}

/**
 * BENEFITS:
 * 
 * ✓ Decoupling (producers/consumers independent)
 * ✓ Thread-safe (BlockingQueue handles sync)
 * ✓ Backpressure (queue.put() blocks when full)
 * ✓ Load balancing (multiple consumers)
 */

// BLOCKINGQUEUE IMPLEMENTATIONS
// ========================================

class BlockingQueueTypes {
    
    // 1. ArrayBlockingQueue - Bounded, array-backed
    public void arrayBased() {
        BlockingQueue<String> queue = new ArrayBlockingQueue<>(100);
        
        /**
         * - Fixed capacity (100)
         * - FIFO ordering
         * - Fair/unfair lock option
         * - Good for: Known bounded workload
         */
    }
    
    // 2. LinkedBlockingQueue - Optionally bounded, linked nodes
    public void linkedBased() {
        BlockingQueue<String> queue = new LinkedBlockingQueue<>(100);
        // or unbounded: new LinkedBlockingQueue<>();
        
        /**
         * - Optionally bounded
         * - Separate locks for put/take (better throughput!)
         * - Good for: Producer-consumer pipeline
         */
    }
    
    // 3. PriorityBlockingQueue - Unbounded, priority-ordered
    public void priorityBased() {
        BlockingQueue<Task> queue = new PriorityBlockingQueue<>(
            10,
            Comparator.comparingInt(Task::getPriority)
        );
        
        /**
         * - Unbounded (grows as needed)
         * - Elements ordered by priority
         * - Good for: Task scheduling by priority
         */
    }
    
    // 4. SynchronousQueue - Zero capacity
    public void synchronous() {
        BlockingQueue<String> queue = new SynchronousQueue<>();
        
        /**
         * - No storage capacity
         * - Direct handoff (producer waits for consumer)
         * - Good for: Thread coordination
         */
    }
    
    // 5. DelayQueue - Elements available after delay
    public void delayed() {
        BlockingQueue<DelayedTask> queue = new DelayQueue<>();
        
        /**
         * - Elements available after delay expires
         * - Implements Delayed interface
         * - Good for: Scheduled tasks, cache expiration
         */
    }
}

// REAL-WORLD: IMAGE PROCESSING PIPELINE
// ========================================

class Image {
    private String path;
    private byte[] data;
    
    public Image(String path) {
        this.path = path;
    }
    
    public String getPath() { return path; }
    public byte[] getData() { return data; }
    public void setData(byte[] data) { this.data = data; }
}

class ImageProcessingPipeline {
    private final BlockingQueue<Image> loadQueue;
    private final BlockingQueue<Image> processQueue;
    private final BlockingQueue<Image> saveQueue;
    
    private final ExecutorService loaders;
    private final ExecutorService processors;
    private final ExecutorService savers;
    
    public ImageProcessingPipeline() {
        this.loadQueue = new LinkedBlockingQueue<>(50);
        this.processQueue = new LinkedBlockingQueue<>(50);
        this.saveQueue = new LinkedBlockingQueue<>(50);
        
        // Create worker pools
        this.loaders = Executors.newFixedThreadPool(2);
        this.processors = Executors.newFixedThreadPool(4);  // CPU-intensive
        this.savers = Executors.newFixedThreadPool(2);
        
        // Start workers
        startWorkers();
    }
    
    private void startWorkers() {
        // Loaders: Read images from disk
        for (int i = 0; i < 2; i++) {
            loaders.submit(() -> {
                try {
                    while (!Thread.interrupted()) {
                        Image image = loadQueue.take();
                        // Load image data
                        image.setData(loadImageData(image.getPath()));
                        processQueue.put(image);
                    }
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
            });
        }
        
        // Processors: Apply transformations
        for (int i = 0; i < 4; i++) {
            processors.submit(() -> {
                try {
                    while (!Thread.interrupted()) {
                        Image image = processQueue.take();
                        // Process image (resize, filter, etc.)
                        byte[] processed = processImage(image.getData());
                        image.setData(processed);
                        saveQueue.put(image);
                    }
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
            });
        }
        
        // Savers: Write processed images
        for (int i = 0; i < 2; i++) {
            savers.submit(() -> {
                try {
                    while (!Thread.interrupted()) {
                        Image image = saveQueue.take();
                        // Save image to disk
                        saveImage(image);
                    }
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
            });
        }
    }
    
    public void submitImage(String path) {
        try {
            loadQueue.put(new Image(path));
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
    
    public void shutdown() {
        loaders.shutdownNow();
        processors.shutdownNow();
        savers.shutdownNow();
    }
    
    private byte[] loadImageData(String path) {
        // Load image from disk
        return new byte[0];
    }
    
    private byte[] processImage(byte[] data) {
        // Apply transformations
        return data;
    }
    
    private void saveImage(Image image) {
        // Save to disk
    }
}

/**
 * PIPELINE BENEFITS:
 * 
 * ✓ Stages process concurrently
 * ✓ Automatic load balancing
 * ✓ Backpressure (queues bound growth)
 * ✓ Easy to scale (add more workers)
 * 
 * 
 * PERFORMANCE:
 *
```

## 3. Read-Write Lock Pattern

### Concurrent Reads, Exclusive Writes

```java
/**
 * READ-WRITE LOCK PATTERN
 * 
 * Multiple threads can read simultaneously
 * Only one thread can write (excludes all readers)
 */

// ========================================
// PROBLEM: synchronized EVERYWHERE
// ========================================
class BadCache {
    private final Map<String, String> cache = new HashMap<>();
    
    // Read
    public synchronized String get(String key) {
        return cache.get(key);
    }
    
    // Write
    public synchronized void put(String key, String value) {
        cache.put(key, value);
    }
    
    /**
     * PROBLEM:
     * ✗ Reads block each other (only one thread can read!)
     * ✗ Wastes parallelism (most operations are reads)
     * ✗ Poor scalability
     */
}

// ========================================
// SOLUTION: READ-WRITE LOCK
// ========================================

import java.util.concurrent.locks.*;

class OptimizedCache {
    private final Map<String, String> cache = new HashMap<>();
    private final ReadWriteLock lock = new ReentrantReadWriteLock();
    private final Lock readLock = lock.readLock();
    private final Lock writeLock = lock.writeLock();
    
    // Read (concurrent)
    public String get(String key) {
        readLock.lock();
        try {
            return cache.get(key);
        } finally {
            readLock.unlock();
        }
    }
    
    // Write (exclusive)
    public void put(String key, String value) {
        writeLock.lock();
        try {
            cache.put(key, value);
        } finally {
            writeLock.unlock();
        }
    }
    
    /**
     * BENEFITS:
     * 
     * ✓ Multiple readers simultaneously
     * ✓ Writers get exclusive access
     * ✓ Better scalability for read-heavy workloads
     * 
     * 
     * PERFORMANCE:
     * 
     * 90% reads, 10 threads:
     * - synchronized: 100 ops/sec
     * - ReadWriteLock: 900 ops/sec (9x faster!)
     */
}

// ========================================
// REENTRANTREADWRITELOCK FEATURES
// ========================================

class ReadWriteLockFeatures {
    private final ReadWriteLock lock = new ReentrantReadWriteLock();
    private final Lock readLock = lock.readLock();
    private final Lock writeLock = lock.writeLock();
    private int data = 0;
    
    // 1. Reentrant - Same thread can acquire again
    public void reentrantExample() {
        writeLock.lock();
        try {
            writeLock.lock();  // ✓ Same thread, OK
            try {
                data++;
            } finally {
                writeLock.unlock();
            }
        } finally {
            writeLock.unlock();
        }
    }
    
    // 2. Downgrade - Write lock → Read lock
    public void downgradeExample() {
        writeLock.lock();
        try {
            data++;  // Write
            
            readLock.lock();  // Acquire read lock while holding write
        } finally {
            writeLock.unlock();  // Release write lock
        }
        
        try {
            // Now only read lock held
            System.out.println(data);
        } finally {
            readLock.unlock();
        }
    }
    
    // 3. Fair vs Unfair
    public void fairnessExample() {
        // Fair - Threads served in order (prevents starvation)
        ReadWriteLock fairLock = new ReentrantReadWriteLock(true);
        
        // Unfair (default) - Better throughput, possible starvation
        ReadWriteLock unfairLock = new ReentrantReadWriteLock(false);
    }
    
    // 4. tryLock - Non-blocking attempt
    public boolean tryWriteExample() {
        if (writeLock.tryLock()) {
            try {
                data++;
                return true;
            } finally {
                writeLock.unlock();
            }
        }
        return false;  // Lock not available
    }
    
    // 5. tryLock with timeout
    public boolean tryWriteWithTimeoutExample() {
        try {
            if (writeLock.tryLock(100, TimeUnit.MILLISECONDS)) {
                try {
                    data++;
                    return true;
                } finally {
                    writeLock.unlock();
                }
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
        return false;
    }
}

// ========================================
// REAL-WORLD: CONFIGURATION MANAGER
// ========================================

class ConfigurationManager {
    private final Map<String, String> config = new HashMap<>();
    private final ReadWriteLock lock = new ReentrantReadWriteLock();
    
    // Read configuration (concurrent)
    public String getProperty(String key) {
        lock.readLock().lock();
        try {
            return config.get(key);
        } finally {
            lock.readLock().unlock();
        }
    }
    
    // Read multiple properties (concurrent)
    public Map<String, String> getProperties(Set<String> keys) {
        lock.readLock().lock();
        try {
            Map<String, String> result = new HashMap<>();
            for (String key : keys) {
                result.put(key, config.get(key));
            }
            return result;
        } finally {
            lock.readLock().unlock();
        }
    }
    
    // Update configuration (exclusive)
    public void setProperty(String key, String value) {
        lock.writeLock().lock();
        try {
            config.put(key, value);
            notifyListeners(key, value);
        } finally {
            lock.writeLock().unlock();
        }
    }
    
    // Reload from file (exclusive)
    public void reloadConfiguration(Map<String, String> newConfig) {
        lock.writeLock().lock();
        try {
            config.clear();
            config.putAll(newConfig);
        } finally {
            lock.writeLock().unlock();
        }
    }
    
    // Safe iteration (read lock held during iteration)
    public void printAllProperties() {
        lock.readLock().lock();
        try {
            config.forEach((key, value) ->
                System.out.println(key + " = " + value));
        } finally {
            lock.readLock().unlock();
        }
    }
    
    private void notifyListeners(String key, String value) {
        // Notify change listeners
    }
}

/**
 * WHEN TO USE READ-WRITE LOCK:
 * 
 * ✓ Read-heavy workloads (> 70% reads)
 * ✓ Long-running read operations
 * ✓ Multiple readers need concurrent access
 * 
 * 
 * WHEN NOT TO USE:
 * 
 * ✗ Write-heavy workloads (overhead not worth it)
 * ✗ Short critical sections (synchronized is faster)
 * ✗ Single reader (no benefit)
 */
```

## 4. Double-Checked Locking Pattern

### Thread-Safe Lazy Initialization (Correct Implementation)

```java
/**
 * DOUBLE-CHECKED LOCKING
 * 
 * Lazy initialization with minimal synchronization
 * CRITICAL: Requires volatile!
 */

// ========================================
// BROKEN IMPLEMENTATION (DON'T USE!)
// ========================================
class BrokenSingleton {
    private static BrokenSingleton instance;  // ❌ NOT volatile!
    
    public static BrokenSingleton getInstance() {
        if (instance == null) {                  // First check (no lock)
            synchronized (BrokenSingleton.class) {
                if (instance == null) {          // Second check (with lock)
                    instance = new BrokenSingleton();  // ❌ PROBLEM!
                }
            }
        }
        return instance;
    }
    
    /**
     * WHY THIS IS BROKEN:
     * 
     * 1. new BrokenSingleton() is NOT atomic:
     *    a) Allocate memory
     *    b) Initialize object
     *    c) Assign to instance
     * 
     * 2. Compiler/CPU can reorder to:
     *    a) Allocate memory
     *    c) Assign to instance  (instance != null now!)
     *    b) Initialize object   (but object not initialized!)
     * 
     * 3. Another thread sees instance != null
     *    Returns partially-constructed object!
     * 
     * RESULT: Subtle, rare bugs (appears to work 99.9% of time)
     */
}

// ========================================
// CORRECT IMPLEMENTATION
// ========================================
class CorrectSingleton {
    private static volatile CorrectSingleton instance;  // ✓ volatile!
    
    public static CorrectSingleton getInstance() {
        if (instance == null) {                     // First check (no lock)
            synchronized (CorrectSingleton.class) {
                if (instance == null) {             // Second check (with lock)
                    instance = new CorrectSingleton();  // ✓ Safe with volatile
                }
            }
        }
        return instance;
    }
    
    /**
     * WHY VOLATILE FIXES IT:
     * 
     * volatile prevents reordering:
     * - Writes to volatile happen-before reads
     * - No reordering around volatile write
     * - Thread sees fully-constructed object
     * 
     * 
     * PERFORMANCE:
     * 
     * - First access: synchronized overhead
     * - Subsequent accesses: Just volatile read (fast!)
     * - No synchronization after initialization
     */
}

// ========================================
// ALTERNATIVE: LAZY HOLDER (BETTER!)
// ========================================
class LazyHolderSingleton {
    private LazyHolderSingleton() { }
    
    // Inner class not loaded until first access
    private static class Holder {
        static final LazyHolderSingleton INSTANCE = new LazyHolderSingleton();
    }
    
    public static LazyHolderSingleton getInstance() {
        return Holder.INSTANCE;
    }
    
    /**
     * WHY THIS IS BETTER:
     * 
     * ✓ Thread-safe (JVM guarantees)
     * ✓ Lazy loading (Holder not loaded until first call)
     * ✓ No synchronization
     * ✓ No volatile needed
     * ✓ Simpler code
     * 
     * USE THIS instead of double-checked locking!
     */
}

// ========================================
// WHEN TO USE DOUBLE-CHECKED LOCKING
// ========================================
/**
 * MODERN RECOMMENDATION:
 * 
 * DON'T use double-checked locking!
 * 
 * Alternatives:
 * 1. Lazy Holder idiom (best for singletons)
 * 2. Enum singleton (best overall)
 * 3. Early initialization (if lazy not needed)
 * 
 * 
 * ONLY use double-checked locking if:
 * - You have a specific non-singleton use case
 * - You understand volatile semantics fully
 * - You've profiled and it's actually faster
 */

// ========================================
// DOUBLE-CHECKED LOCKING FOR NON-SINGLETON
// ========================================
class ExpensiveResource {
    private volatile Resource resource;  // Must be volatile!
    private final Object lock = new Object();
    
    public Resource getResource() {
        Resource r = resource;  // Local copy (optimization)
        if (r == null) {
            synchronized (lock) {
                r = resource;
                if (r == null) {
                    r = new Resource();  // Expensive initialization
                    resource = r;
                }
            }
        }
        return r;
    }
    
    /**
     * PATTERN:
     * 
     * 1. Read volatile into local variable
     * 2. Check local variable (fast)
     * 3. Synchronize if null
     * 4. Double-check inside lock
     * 5. Create and assign
     */
}

class Resource {
    public Resource() {
        // Expensive initialization
        try {
            Thread.sleep(1000);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
}
```

## 5. Thread-Safe Lazy Initialization

### Multiple Safe Approaches

```java
/**
 * THREAD-SAFE LAZY INITIALIZATION
 * 
 * Four approaches from simplest to most complex
 */

// ========================================
// 1. SYNCHRONIZED METHOD (SIMPLE, SLOW)
// ========================================
class SynchronizedInit {
    private ExpensiveObject instance;
    
    public synchronized ExpensiveObject getInstance() {
        if (instance == null) {
            instance = new ExpensiveObject();
        }
        return instance;
    }
    
    /**
     * ✓ Thread-safe
     * ✓ Simple
     * ✗ Synchronized every call (slow!)
     * 
     * Use when: Simplicity > performance
     */
}

// ========================================
// 2. LAZY HOLDER IDIOM (BEST!)
// ========================================
class LazyHolderInit {
    
    private static class Holder {
        static final ExpensiveObject INSTANCE = new ExpensiveObject();
    }
    
    public static ExpensiveObject getInstance() {
        return Holder.INSTANCE;
    }
    
    /**
     * ✓ Thread-safe (JVM guarantees)
     * ✓ Lazy (loaded on first access)
     * ✓ No synchronization overhead
     * ✓ Simple
     * 
     * Use when: You need lazy singleton
     */
}

// ========================================
// 3. DOUBLE-CHECKED LOCKING (COMPLEX)
// ========================================
class DoubleCheckedInit {
    private volatile ExpensiveObject instance;
    
    public ExpensiveObject getInstance() {
        ExpensiveObject result = instance;
        if (result == null) {
            synchronized (this) {
                result = instance;
                if (result == null) {
                    instance = result = new ExpensiveObject();
                }
            }
        }
        return result;
    }
    
    /**
     * ✓ Thread-safe
     * ✓ Fast after initialization
     * ✗ Complex
     * ✗ Easy to get wrong
     * 
     * Use when: Lazy Holder not applicable
     */
}

// ========================================
// 4. ENUM SINGLETON (BEST FOR SINGLETONS)
// ========================================
enum EnumSingleton {
    INSTANCE;
    
    private final ExpensiveObject object;
    
    EnumSingleton() {
        object = new ExpensiveObject();
    }
    
    public ExpensiveObject getObject() {
        return object;
    }
    
    /**
     * ✓ Thread-safe (JVM guarantees)
     * ✓ Serialization-safe
     * ✓ Reflection-proof
     * ✓ Simplest code
     * ✗ Not lazy (loaded on class load)
     * 
     * Use when: Singleton, don't need lazy
     */
}

// ========================================
// 5. EARLY INITIALIZATION (SIMPLEST)
// ========================================
class EarlyInit {
    private static final ExpensiveObject INSTANCE = new ExpensiveObject();
    
    public static ExpensiveObject getInstance() {
        return INSTANCE;
    }
    
    /**
     * ✓ Thread-safe (static initialization)
     * ✓ Simplest code
     * ✓ No overhead
     * ✗ Not lazy (always created)
     * 
     * Use when: Don't need lazy initialization
     */
}

// ========================================
// DECISION TREE
// ========================================
/**
 * Need lazy initialization?
 *   NO  → Early initialization (static final)
 *   YES ↓
 * 
 * Is it a singleton?
 *   NO  → Double-checked locking (with volatile)
 *   YES ↓
 * 
 * Need serialization-safe?
 *   YES → Enum singleton
 *   NO  ↓
 * 
 * Need lazy loading?
 *   YES → Lazy Holder idiom
 *   NO  → Enum singleton or early init
 */

class ExpensiveObject {
    public ExpensiveObject() {
        // Expensive initialization
    }
}
```

## 6. Future Pattern

### Asynchronous Computation

```java
/**
 * FUTURE PATTERN
 * 
 * Represents result of asynchronous computation
 * Can be retrieved later
 */

import java.util.concurrent.*;
import java.util.concurrent.CompletableFuture;

// ========================================
// BASIC FUTURE WITH EXECUTOR
// ========================================
class BasicFutureExample {
    
    public void futureExample() throws Exception {
        ExecutorService executor = Executors.newFixedThreadPool(2);
        
        // Submit task, get Future
        Future<String> future = executor.submit(() -> {
            Thread.sleep(2000);  // Simulate long computation
            return "Result from async task";
        });
        
        // Do other work while task executes
        System.out.println("Doing other work...");
        
        // Wait for result (blocking)
        String result = future.get();  // Blocks until complete
        System.out.println(result);
        
        executor.shutdown();
    }
    
    // With timeout
    public void futureWithTimeout() throws Exception {
        ExecutorService executor = Executors.newSingleThreadExecutor();
        
        Future<String> future = executor.submit(() -> {
            Thread.sleep(5000);
            return "Result";
        });
        
        try {
            String result = future.get(2, TimeUnit.SECONDS);  // ❌ Timeout!
        } catch (TimeoutException e) {
            System.out.println("Task timed out!");
            future.cancel(true);  // Cancel task
        }
        
        executor.shutdown();
    }
}

// ========================================
// COMPLETABLEFUTURE (MODERN APPROACH)
// ========================================
class CompletableFutureExamples {
    
    // 1. Basic async computation
    public void basicAsync() {
        CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> {
            sleep(1000);
            return "Hello";
        });
        
        // Non-blocking callback
        future.thenAccept(result -> 
            System.out.println("Got: " + result));
        
        // Or block for result
        // String result = future.join();
    }
    
    // 2. Chaining transformations
    public void chaining() {
        CompletableFuture.supplyAsync(() -> "Hello")
            .thenApply(s -> s + " World")         // Transform
            .thenApply(String::toUpperCase)       // Transform
            .thenAccept(System.out::println);     // Consume
        
        // Output: HELLO WORLD
    }
    
    // 3. Combining futures
    public void combining() {
        CompletableFuture<String> future1 = 
            CompletableFuture.supplyAsync(() -> "Hello");
        
        CompletableFuture<String> future2 = 
            CompletableFuture.supplyAsync(() -> "World");
        
        // Combine results
        CompletableFuture<String> combined = future1.thenCombine(
            future2,
            (s1, s2) -> s1 + " " + s2
        );
        
        System.out.println(combined.join());  // Hello World
    }
    
    // 4. Parallel execution
    public void parallel() {
        CompletableFuture<Integer> f1 = CompletableFuture.supplyAsync(() -> {
            sleep(1000);
            return 10;
        });
        
        CompletableFuture<Integer> f2 = CompletableFuture.supplyAsync(() -> {
            sleep(1000);
            return 20;
        });
        
        CompletableFuture<Integer> f3 = CompletableFuture.supplyAsync(() -> {
            sleep(1000);
            return 30;
        });
        
        // Wait for all
        CompletableFuture<Void> all = CompletableFuture.allOf(f1, f2, f3);
        all.join();  // Blocks until all complete
        
        // Total time: ~1 second (parallel!), not 3
    }
    
    // 5. Exception handling
    public void exceptionHandling() {
        CompletableFuture.supplyAsync(() -> {
            if (true) throw new RuntimeException("Error!");
            return "Result";
        })
        .exceptionally(ex -> {
            System.out.println("Exception: " + ex.getMessage());
            return "Default value";
        })
        .thenAccept(System.out::println);
    }
    
    // 6. Timeout
    public void timeout() {
        CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> {
            sleep(5000);
            return "Result";
        });
        
        try {
            String result = future.orTimeout(2, TimeUnit.SECONDS)
                .join();
        } catch (CompletionException e) {
            System.out.println("Timed out!");
        }
    }
    
    private void sleep(long ms) {
        try {
            Thread.sleep(ms);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
}

// ========================================
// REAL-WORLD: ASYNC API CALLS
// ========================================
class ApiClient {
    
    // Fetch user info from multiple services
    public CompletableFuture<UserProfile> getUserProfile(String userId) {
        // Parallel API calls
        CompletableFuture<User> userFuture =

        CompletableFuture<List<Order>> ordersFuture =

            .thenApply(v -> {
                User user = userFuture.join();
                List<Order> orders = ordersFuture.join();
                Preferences prefs = prefsFuture.join();
                return new UserProfile(user, orders, prefs);
            });
    }
    
    private CompletableFuture<User> fetchUser(String userId) {
```

## 7. Active Object Pattern

### Decoupling Method Execution from Invocation

```java
/**
 * ACTIVE OBJECT PATTERN
 * 
 * Method invocation and execution in different threads
 * Caller doesn't block
 */

// ========================================
// ACTIVE OBJECT IMPLEMENTATION
// ========================================
class ActiveObject {
    private final ExecutorService executor = Executors.newSingleThreadExecutor();
    private final BlockingQueue<Runnable> requestQueue = new LinkedBlockingQueue<>();
    private int state = 0;
    
    public ActiveObject() {
        // Start dispatcher thread
        executor.submit(() -> {
            while (!Thread.interrupted()) {
                try {
                    Runnable request = requestQueue.take();
                    request.run();  // Execute in active object's thread
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    break;
                }
            }
        });
    }
    
    // Async method - returns immediately
    public Future<Integer> increment() {
        CompletableFuture<Integer> future = new CompletableFuture<>();
        requestQueue.offer(() -> {
            state++;
            future.complete(state);
        });
        return future;
    }
    
    // Async method - returns immediately
    public Future<Integer> getState() {
        CompletableFuture<Integer> future = new CompletableFuture<>();
        requestQueue.offer(() -> {
            future.complete(state);
        });
        return future;
    }
    
    public void shutdown() {
        executor.shutdownNow();
    }
}

/**
 * USAGE:
 * 
 * ActiveObject obj = new ActiveObject();
 * 
 * Future<Integer> f1 = obj.increment();  // Returns immediately!
 * Future<Integer> f2 = obj.increment();  // Returns immediately!
 * 
 * // Do other work...
 * 
 * System.out.println(f1.get());  // 1
 * System.out.println(f2.get());  // 2
 * 
 * 
 * BENEFITS:
 * 
 * ✓ Callers don't block
 * ✓ Thread-safe (single thread executes)
 * ✓ Decouples invocation from execution
 * ✓ Simplified synchronization
 */

// ========================================
// REAL-WORLD: LOGGER
// ========================================
class AsyncLogger {
    private final ExecutorService logThread = Executors.newSingleThreadExecutor();
    private final BlockingQueue<LogEntry> logQueue = new LinkedBlockingQueue<>();
    
    public AsyncLogger() {
        logThread.submit(() -> {
            while (!Thread.interrupted()) {
                try {
                    LogEntry entry = logQueue.take();
                    writeToFile(entry);
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    break;
                }
            }
        });
    }
    
    // Returns immediately (non-blocking)
    public void log(String level, String message) {
        LogEntry entry = new LogEntry(
            System.currentTimeMillis(),
            level,
            Thread.currentThread().getName(),
            message
        );
        logQueue.offer(entry);
    }
    
    private void writeToFile(LogEntry entry) {
        // Write to file (I/O operation)
        System.out.println("[" + entry.timestamp + "] " + 
            entry.level + " " + entry.message);
    }
    
    public void shutdown() {
        logThread.shutdownNow();
    }
}

class LogEntry {
    long timestamp;
    String level;
    String threadName;
    String message;
    
    LogEntry(long timestamp, String level, String threadName, String message) {
        this.timestamp = timestamp;
        this.level = level;
        this.threadName = threadName;
        this.message = message;
    }
}

/**
 * USAGE:
 * 
 * AsyncLogger logger = new AsyncLogger();
 * 
 * logger.log("INFO", "Application started");  // Returns immediately!
 * logger.log("WARN", "Low memory");           // Returns immediately!
 * logger.log("ERROR", "Connection failed");   // Returns immediately!
 * 
 * // Main thread continues without blocking
 * 
 * 
 * BENEFITS:
 * 
 * ✓ Logging doesn't block application threads
 * ✓ Batch I/O operations
 * ✓ Better throughput
 * ✓ Simple thread safety (single writer)
 */
```

## Summary

### Quick Reference

**Concurrency Patterns:**

```
1. Thread Pool: Reuse threads efficiently
   - ExecutorService (fixed, cached, scheduled)
   - Sizing: CPU-bound (cores+1), I/O-bound (cores*10+)

2. Producer-Consumer: Decouple production/consumption
   - BlockingQueue (array, linked, priority)
   - Pipeline parallelism

3. Read-Write Lock: Concurrent reads, exclusive writes
   - ReentrantReadWriteLock
   - Read-heavy workloads (>70% reads)

4. Double-Checked Locking: Lazy init with minimal sync
   - Requires volatile!
   - Better: Use Lazy Holder idiom

5. Thread-Safe Lazy Init: Safe singleton patterns
   - Lazy Holder (best for lazy)
   - Enum (best for singleton)
   - Early init (simplest)

6. Future: Asynchronous computation
   - CompletableFuture (modern)
   - Parallel execution, chaining

7. Active Object: Decouple invocation from execution
   - Message queue + worker thread
   - Non-blocking methods
```

**Pattern Comparison:**

```
┌──────────────────┬─────────────────────┬──────────────────────┐
│ Pattern          │ Problem Solved      │ Key Technique        │
├──────────────────┼─────────────────────┼──────────────────────┤
│ Thread Pool      │ Thread overhead     │ Reuse workers        │
│ Producer-Consumer│ Decouple stages     │ BlockingQueue        │
│ Read-Write Lock  │ Concurrent reads    │ Separate read/write  │
│ Double-Checked   │ Lazy + performance  │ volatile + sync      │
│ Future           │ Async computation   │ Promise/callback     │
│ Active Object    │ Non-blocking methods│ Message queue        │
└──────────────────┴─────────────────────┴──────────────────────┘
```

**Decision Guide:**

```
Need to execute many tasks?
→ Thread Pool (ExecutorService)

Need to decouple producers and consumers?
→ Producer-Consumer (BlockingQueue)

Read-heavy workload (>70% reads)?
→ Read-Write Lock

Need lazy singleton?
→ Lazy Holder idiom (NOT double-checked locking!)

Need asynchronous computation?
→ Future (CompletableFuture)

Need non-blocking method calls?
→ Active Object
```

---


**Key insights:**

- **Thread Pool** manages workers efficiently (ExecutorService)
- **Producer-Consumer** decouples with BlockingQueue
- **Read-Write Lock** enables concurrent reads
- **Double-Checked Locking** needs volatile (or use Lazy Holder!)
- **Thread-Safe Lazy Init** has multiple safe approaches
- **Future Pattern** enables async computation (CompletableFuture)
- **Active Object** decouples invocation from execution
