# Testing Patterns

## Overview

Testing patterns make tests easier to write, read, and maintain. Poor tests are brittle (break on small changes), unclear (hard to understand what's being tested), and duplicative (same setup repeated everywhere). This guide covers six essential patterns: Test Doubles (mocks, stubs, spies, fakes—how to isolate code under test), Builder Pattern for Tests (fluent API for creating test objects), Object Mother Pattern (factory methods for common test data), Test Data Builders (combining builders with meaningful defaults), Fixture Setup Patterns (@Before, @BeforeEach, and when to avoid them), and Parameterized Tests (testing multiple cases without duplication). The key insight: test code is production code—it needs the same care. DRY applies to tests but differently: don't duplicate test logic, but DO duplicate test data when it clarifies intent. Good tests are FIRST: Fast (run quickly), Independent (no order dependencies), Repeatable (same result every time), Self-Validating (pass/fail, no manual inspection), Timely (written with or before code). Understanding testing patterns means understanding trade-offs: Mocks are powerful but couple tests to implementation; Stubs are simple but limited; Fakes are realistic but complex to maintain; Builders are flexible but verbose; Object Mother is convenient but hides details; Shared fixtures are DRY but create dependencies. This guide shows when to use each pattern, common mistakes (over-mocking, shared mutable fixtures, brittle assertions), and how patterns combine (Builder + Object Mother, Stub + Spy for verification). Whether writing unit tests (fast, isolated, use mocks/stubs) or integration tests (slower, realistic, use fakes), these patterns help.

## 1. Test Doubles

### Mock, Stub, Spy, Fake

```java
/**
 * TEST DOUBLES
 * 
 * Objects that replace real dependencies in tests
 * Four types: Mock, Stub, Spy, Fake
 */

// ========================================
// PRODUCTION CODE TO TEST
// ========================================

interface EmailService {
    void sendEmail(String to, String subject, String body);
    boolean isConnected();
}

interface UserRepository {
    void save(User user);
    Optional<User> findByEmail(String email);
    List<User> findAll();
}

class UserService {
    private final UserRepository repository;
    private final EmailService emailService;
    
    public UserService(UserRepository repository, EmailService emailService) {
        this.repository = repository;
        this.emailService = emailService;
    }
    
    public void registerUser(String email, String name) {
        // Check if user exists
        if (repository.findByEmail(email).isPresent()) {
            throw new IllegalArgumentException("User already exists");
        }
        
        // Save user
        User user = new User(email, name);
        repository.save(user);
        
        // Send welcome email
        emailService.sendEmail(email, "Welcome!", "Welcome to our service");
    }
}

// 1. STUB: Returns pre-programmed responses
// ========================================

/**
 * Stub provides canned answers
 * No behavior verification
 */
class StubExample {
    
    @Test
    void testWithStub() {
        // Create stub that always returns empty
        UserRepository stubRepo = new UserRepository() {
            @Override
            public void save(User user) {
                // Do nothing
            }
            
            @Override
            public Optional<User> findByEmail(String email) {
                return Optional.empty();  // Always return empty
            }
            
            @Override
            public List<User> findAll() {
                return Collections.emptyList();
            }
        };
        
        // Stub email service
        EmailService stubEmail = new EmailService() {
            @Override
            public void sendEmail(String to, String subject, String body) {
                // Do nothing
            }
            
            @Override
            public boolean isConnected() {
                return true;  // Always connected
            }
        };
        
        // Test with stubs
        UserService service = new UserService(stubRepo, stubEmail);
        service.registerUser("test@example.com", "Test User");
        
        // ✓ Test passes (no exceptions)
    }
    
    /**
     * MOCKITO STUB
     */
    @Test
    void testWithMockitoStub() {
        UserRepository stubRepo = mock(UserRepository.class);
        EmailService stubEmail = mock(EmailService.class);
        
        // Configure stub behavior
        when(stubRepo.findByEmail("test@example.com")).thenReturn(Optional.empty());
        when(stubEmail.isConnected()).thenReturn(true);
        
        // Test
        UserService service = new UserService(stubRepo, stubEmail);
        service.registerUser("test@example.com", "Test User");
        
        // ✓ Stubs provide necessary behavior
    }
    
    /**
     * WHEN TO USE STUBS:
     * 
     * ✓ Need simple return values
     * ✓ Don't care about interactions
     * ✓ Simulating queries (read operations)
     * ✓ Providing test data
     */
}

// 2. MOCK: Verifies interactions
// ========================================

/**
 * Mock verifies behavior was called correctly
 */
class MockExample {
    
    @Test
    void testWithMock() {
        // Create mocks
        UserRepository mockRepo = mock(UserRepository.class);
        EmailService mockEmail = mock(EmailService.class);
        
        // Stub findByEmail to return empty
        when(mockRepo.findByEmail("test@example.com")).thenReturn(Optional.empty());
        
        // Test
        UserService service = new UserService(mockRepo, mockEmail);
        service.registerUser("test@example.com", "Test User");
        
        // ✓ VERIFY: Check interactions happened
        verify(mockRepo).save(any(User.class));
        verify(mockEmail).sendEmail(
            eq("test@example.com"),
            eq("Welcome!"),
            eq("Welcome to our service")
        );
    }
    
    /**
     * VERIFY CALL COUNT
     */
    @Test
    void testVerifyCallCount() {
        UserRepository mockRepo = mock(UserRepository.class);
        EmailService mockEmail = mock(EmailService.class);
        
        when(mockRepo.findByEmail(anyString())).thenReturn(Optional.empty());
        
        UserService service = new UserService(mockRepo, mockEmail);
        service.registerUser("user1@example.com", "User 1");
        service.registerUser("user2@example.com", "User 2");
        
        // Verify called exactly twice
        verify(mockRepo, times(2)).save(any(User.class));
        verify(mockEmail, times(2)).sendEmail(anyString(), anyString(), anyString());
    }
    
    /**
     * VERIFY NO INTERACTIONS
     */
    @Test
    void testVerifyNoInteractions() {
        UserRepository mockRepo = mock(UserRepository.class);
        EmailService mockEmail = mock(EmailService.class);
        
        // User already exists
        when(mockRepo.findByEmail("existing@example.com"))
            .thenReturn(Optional.of(new User("existing@example.com", "Existing")));
        
        UserService service = new UserService(mockRepo, mockEmail);
        
        try {
            service.registerUser("existing@example.com", "Test");
        } catch (IllegalArgumentException e) {
            // Expected exception
        }
        
        // Verify save and email never called
        verify(mockRepo, never()).save(any(User.class));
        verify(mockEmail, never()).sendEmail(anyString(), anyString(), anyString());
    }
    
    /**
     * WHEN TO USE MOCKS:
     * 
     * ✓ Verify method was called
     * ✓ Verify call arguments
     * ✓ Verify call count
     * ✓ Commands (write operations)
     * ✓ Side effects (email, logging)
     */
}

// 3. SPY: Partial mock (real object + verification)
// ========================================

/**
 * Spy wraps real object, can verify and override
 */
class SpyExample {
    
    /**
     * Real implementation we'll spy on
     */
    static class InMemoryUserRepository implements UserRepository {
        private final Map<String, User> users = new HashMap<>();
        
        @Override
        public void save(User user) {
            users.put(user.getEmail(), user);
        }
        
        @Override
        public Optional<User> findByEmail(String email) {
            return Optional.ofNullable(users.get(email));
        }
        
        @Override
        public List<User> findAll() {
            return new ArrayList<>(users.values());
        }
    }
    
    @Test
    void testWithSpy() {
        // Create spy of real object
        UserRepository spyRepo = spy(new InMemoryUserRepository());
        EmailService mockEmail = mock(EmailService.class);
        
        // Spy uses real implementation by default
        UserService service = new UserService(spyRepo, mockEmail);
        service.registerUser("test@example.com", "Test User");
        
        // ✓ Can verify calls (like mock)
        verify(spyRepo).save(any(User.class));
        
        // ✓ Can check real state
        assertTrue(spyRepo.findByEmail("test@example.com").isPresent());
    }
    
    /**
     * Override specific methods on spy
     */
    @Test
    void testSpyWithOverride() {
        UserRepository spyRepo = spy(new InMemoryUserRepository());
        EmailService mockEmail = mock(EmailService.class);
        
        // Override one method
        when(spyRepo.findByEmail("special@example.com"))
            .thenReturn(Optional.of(new User("special@example.com", "Special")));
        
        UserService service = new UserService(spyRepo, mockEmail);
        
        // Overridden method returns stubbed value
        try {
            service.registerUser("special@example.com", "Test");
        } catch (IllegalArgumentException e) {
            // Expected - user "exists"
        }
        
        // Other methods use real implementation
        service.registerUser("real@example.com", "Real User");
        assertTrue(spyRepo.findByEmail("real@example.com").isPresent());
    }
    
    /**
     * WHEN TO USE SPY:
     * 
     * ✓ Test real object with verification
     * ✓ Override one method, keep others real
     * ✓ Legacy code (can't easily inject dependencies)
     * 
     * ✗ USUALLY PREFER: Full mocks or real objects
     */
}

// 4. FAKE: Real implementation (simplified)
// ========================================

/**
 * Fake has working implementation (simpler than production)
 */
class FakeExample {
    
    /**
     * Fake repository (in-memory, no database)
     */
    static class FakeUserRepository implements UserRepository {
        private final Map<String, User> users = new ConcurrentHashMap<>();
        
        @Override
        public void save(User user) {
            if (users.containsKey(user.getEmail())) {
                throw new IllegalStateException("Duplicate user");
            }
            users.put(user.getEmail(), user);
        }
        
        @Override
        public Optional<User> findByEmail(String email) {
            return Optional.ofNullable(users.get(email));
        }
        
        @Override
        public List<User> findAll() {
            return new ArrayList<>(users.values());
        }
        
        public void clear() {
            users.clear();
        }
    }
    
    /**
     * Fake email service (captures emails, doesn't send)
     */
    static class FakeEmailService implements EmailService {
        private final List<Email> sentEmails = new ArrayList<>();
        private boolean connected = true;
        
        @Override
        public void sendEmail(String to, String subject, String body) {
            if (!connected) {
                throw new IllegalStateException("Not connected");
            }
            sentEmails.add(new Email(to, subject, body));
        }
        
        @Override
        public boolean isConnected() {
            return connected;
        }
        
        public List<Email> getSentEmails() {
            return new ArrayList<>(sentEmails);
        }
        
        public void disconnect() {
            connected = false;
        }
        
        public void clear() {
            sentEmails.clear();
        }
    }
    
    record Email(String to, String subject, String body) {}
    
    @Test
    void testWithFakes() {
        // Create fakes (no mocking framework needed)
        FakeUserRepository fakeRepo = new FakeUserRepository();
        FakeEmailService fakeEmail = new FakeEmailService();
        
        // Test
        UserService service = new UserService(fakeRepo, fakeEmail);
        service.registerUser("test@example.com", "Test User");
        
        // ✓ Verify using real queries (not mock verification)
        Optional<User> savedUser = fakeRepo.findByEmail("test@example.com");
        assertTrue(savedUser.isPresent());
        assertEquals("Test User", savedUser.get().getName());
        
        // ✓ Check captured state
        List<Email> emails = fakeEmail.getSentEmails();
        assertEquals(1, emails.size());
        assertEquals("test@example.com", emails.get(0).to());
        assertEquals("Welcome!", emails.get(0).subject());
    }
    
    /**
     * WHEN TO USE FAKES:
     * 
     * ✓ Integration tests (realistic behavior)
     * ✓ Fast alternative to real implementation
     * ✓ Complex interactions (hard to mock)
     * ✓ Reusable across many tests
     * 
     * Examples:
     * • In-memory database (H2, HSQLDB)
     * • Fake file system
     * • Fake HTTP server (WireMock)
     * • Fake message queue
     */
}

// CHOOSING THE RIGHT TEST DOUBLE
// ========================================

/**
 * Decision guide
 */
class TestDoubleDecisionGuide {
    
    /**
     * STUB:
     * • Need return values
     * • Don't care about interactions
     * • Query operations
     * 
     * Example: when(repo.findById("123")).thenReturn(user);
     */
    
    /**
     * MOCK:
     * • Verify interactions
     * • Check method calls
     * • Command operations
     * 
     * Example: verify(emailService).sendEmail(...)
     */
    
    /**
     * SPY:
     * • Mostly real, some mocking
     * • Legacy code
     * • Partial verification
     * 
     * Example: spy(realObject); verify(realObject).method()
     */
    
    /**
     * FAKE:
     * • Realistic behavior needed
     * • Reusable across tests
     * • Complex interactions
     * 
     * Example: new FakeUserRepository() (no mocking framework)
     */
}
```

### Fluent Test Data Creation

## 3. Object Mother Pattern

### Pre-configured Test Objects

```java
/**
 * OBJECT MOTHER PATTERN
 * 
 * Factory methods that create commonly-used test objects
 * Named after "Mother's helper" - provides ready-made objects
 */

// ========================================
// OBJECT MOTHER IMPLEMENTATION
// ========================================

/**
 * Object Mother for test users
 */
class TestUsers {
    
    /**
     * Standard test user
     */
    public static User johnDoe() {
        return new User("john@example.com", "John", "Doe");
    }
    
    /**
     * Premium user
     */
    public static User premiumUser() {
        User user = new User("premium@example.com", "Premium", "User");
        user.setMembershipType(MembershipType.PREMIUM);
        return user;
    }

     * Admin user
     */

class ObjectMotherTests {
    
    @Test

        Order order = TestOrders.internationalOrder();
```

```java
// ========================================
// PATTERN 1: INLINE SETUP (PREFERRED)

/**

class InlineSetupTests {

    void testUserRegistration() {
        // Setup inline (visible in test)
        UserRepository repository = new FakeUserRepository();
        EmailService emailService = mock(EmailService.class);
        UserService service = new UserService(repository, emailService);
        
        // Test
        User user = new User("test@example.com", "Test", "User");

 */

    private UserRepository repository;
    private EmailService emailService;
    private UserService userService;
    
    @BeforeEach
    void setUp() {
        // Runs before EACH test
        repository = new FakeUserRepository();
        emailService = mock(EmailService.class);
        userService = new UserService(repository, emailService);
    }
    
    @Test
    void testUserRegistration() {
        User user = new User("test@example.com", "Test", "User");
        userService.register(user);
        
        assertTrue(repository.findByEmail("test@example.com").isPresent());
    }
    
    @Test
    void testDuplicateUserRegistration() {
        User user1 = new User("test@example.com", "Test", "User");
        userService.register(user1);
        
        User user2 = new User("test@example.com", "Another", "User");
        assertThrows(IllegalArgumentException.class, () -> {

        });
    }

     * BENEFITS:
     * 
     * ✓ Reduces duplication (DRY)
     * ✓ Consistent setup across tests
     * 
     * DRAWBACKS:
     * 
     * ✗ Hides setup (must look elsewhere)
     * ✗ Tests share state (potential coupling)
     * ✗ Harder to understand what test needs
     * 
     * Use when: All tests need same setup

        // ...

        // ❌ Depends on execution order!
```

```java
// DECISION GUIDE
// ========================================

/**
 * CHOOSING TEST DOUBLE:
 * 
 * Stub: Need return values, don't verify
 * Mock: Verify method called correctly
 * Spy: Mostly real, some mocking
 * Fake: Realistic behavior, reusable
 */

/**
 * TEST DATA CREATION:
 * 
 * Simple data: Inline in test
 * Complex data: Builder pattern
 * Common scenarios: Object Mother
 * Flexible scenarios: Object Mother returning Builders
 */

/**
 * FIXTURE SETUP:
 * 
 * Different per test: Inline setup
 * Same for all tests: @BeforeEach
 * Expensive one-time: @BeforeAll
 * NEVER: Shared mutable state
 */

/**
 * MULTIPLE CASES:
 * 
 * Same test logic, different inputs: @ParameterizedTest
 * Different test logic: Separate tests
 */

// ========================================
// BEST PRACTICES
// ========================================

/**
 * ALWAYS:
 * ✓ Keep tests independent
 * ✓ Use descriptive test names
 * ✓ Test one thing per test
 * ✓ Make tests readable (arrange-act-assert)
 * 
 * PREFER:
 * ✓ Inline setup (visible in test)
 * ✓ Fakes for integration tests
 * ✓ Builders for complex objects
 * ✓ Parameterized for multiple cases
 * 
 * AVOID:
 * ✗ Shared mutable state
 * ✗ Order-dependent tests
 * ✗ Over-mocking (test implementation, not behavior)
 * ✗ Magic values (use constants or builders)
 */

// ========================================

/**
 * Unit Tests (70%):
 * • Fast
 * • Use mocks/stubs
 * • Test individual classes
 * 
 * Integration Tests (20%):
```

**Key Takeaways:**

```
Test Double Hierarchy:
Stub: Returns data (no verification)
Mock: Verifies interactions
Spy: Real + verification
Fake: Working implementation

Test Data Patterns:
Simple: Inline in test
Complex: Builder pattern
Common: Object Mother
Flexible: Object Mother + Builder

Setup Patterns:
Prefer: Inline setup (visible)
OK: @BeforeEach (common setup)
Avoid: Shared mutable state

The Golden Rule:
Tests are production code—treat them well!
Make tests: Fast, Independent, Repeatable,
Self-Validating, Timely (FIRST)
```
