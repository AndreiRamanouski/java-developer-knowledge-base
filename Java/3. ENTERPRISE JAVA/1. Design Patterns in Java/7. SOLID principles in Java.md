# SOLID Principles in Java

## Overview

SOLID is an acronym for five design principles that make software designs more understandable, flexible, and maintainable. These principles were introduced by Robert C. Martin (Uncle Bob) and form the foundation of clean object-oriented design. When followed together, SOLID principles help you create systems that are easy to maintain and extend over time. Each principle addresses a specific aspect of design: Single Responsibility (cohesion), Open/Closed (extensibility), Liskov Substitution (polymorphism correctness), Interface Segregation (interface design), and Dependency Inversion (decoupling). Violating SOLID principles leads to rigid, fragile code that's hard to change, test, and maintain—symptoms include God Objects (violates SRP), modification cascades (violates OCP), broken polymorphism (violates LSP), fat interfaces (violates ISP), and tight coupling (violates DIP). Understanding SOLID means understanding the "why" behind design patterns—most patterns exist to satisfy one or more SOLID principles. For example, Strategy pattern satisfies OCP (add new strategies without modifying context), Factory pattern satisfies DIP (depend on abstractions, not concrete classes), and Adapter pattern satisfies ISP (adapt fat interfaces to specific needs). This guide provides deep examples of each principle, shows common violations in real-world code, demonstrates refactoring techniques to fix violations, and connects SOLID to patterns you already know. Whether you're designing a new system or refactoring legacy code, SOLID principles provide a roadmap for better design decisions.

## 1. Single Responsibility Principle (SRP)

### "A class should have one, and only one, reason to change"

```java
/**
 * SINGLE RESPONSIBILITY PRINCIPLE
 * 
 * A class should have only ONE reason to change
 * Each class should do ONE thing well
 * High cohesion, focused responsibility
 */

        StringBuilder invoice = new StringBuilder();
        invoice.append("Invoice for Order: ").append(order.getId()).append("\n");
        invoice.append("Customer: ").append(order.getCustomer().getName()).append("\n");
        invoice.append("Date: ").append(order.getOrderDate()).append("\n\n");

            invoice.append(String.format("%s x%d: $%.2f\n",

                item.getQuantity(),

        invoice.append(String.format("Total: $%.2f\n", pricingService.calculateGrandTotal(order)));

    }
```

## 2. Open/Closed Principle (OCP)

### "Open for extension, closed for modification"

```java
/**
 * OPEN/CLOSED PRINCIPLE
 * 
 * Classes should be open for extension but closed for modification
 * Add new functionality by adding new code, not changing existing code
 */

// ========================================
// ❌ VIOLATION: Modification Required
// ========================================

/**
 * Adding new discount types requires modifying this class
 */
class DiscountCalculator {
    
    public double calculateDiscount(String discountType, double amount) {
        if (discountType.equals("PERCENTAGE")) {
            return amount * 0.1;  // 10% discount
        } else if (discountType.equals("FIXED")) {
            return 10.0;  // $10 discount
        } else if (discountType.equals("NONE")) {
            return 0.0;
        }
        // Want to add "SEASONAL" discount? Must modify this method!
        // Want to add "LOYALTY" discount? Must modify this method!
        // Want to add "BULK" discount? Must modify this method!
        
        return 0.0;
    }
    
    /**
     * PROBLEMS:
     * 
     * ✗ Every new discount type requires code modification
     * ✗ Risk of breaking existing functionality
     * ✗ Growing if-else chain
     * ✗ Hard to test (all types in one method)
     * ✗ Violates Open/Closed Principle
     */
}

// ========================================
// ✓ SOLUTION: Extension Through Polymorphism
// ========================================

/**
 * Discount interface (abstraction)
 */
interface Discount {
    double calculate(double amount);
}

/**
 * Concrete discount implementations (extensions)
 */
class PercentageDiscount implements Discount {
    private final double percentage;
    
    public PercentageDiscount(double percentage) {
        this.percentage = percentage;
    }
    
    @Override
    public double calculate(double amount) {
        return amount * percentage;
    }
}

class FixedDiscount implements Discount {
    private final double fixedAmount;
    
    public FixedDiscount(double fixedAmount) {
        this.fixedAmount = fixedAmount;
    }
    
    @Override
    public double calculate(double amount) {
        return fixedAmount;
    }
}

class NoDiscount implements Discount {
    @Override
    public double calculate(double amount) {
        return 0.0;
    }
}

/**
 * Want seasonal discount? Just add new class (no modification!)
 */
class SeasonalDiscount implements Discount {
    private final double summerDiscount;
    private final double winterDiscount;
    
    public SeasonalDiscount(double summerDiscount, double winterDiscount) {
        this.summerDiscount = summerDiscount;
        this.winterDiscount = winterDiscount;
    }
    
    @Override
    public double calculate(double amount) {
        int month = LocalDate.now().getMonthValue();
        if (month >= 6 && month <= 8) {
            return amount * summerDiscount;  // Summer
        } else if (month >= 12 || month <= 2) {
            return amount * winterDiscount;  // Winter
        }
        return 0.0;
    }
}

/**
 * Want loyalty discount? Just add new class (no modification!)
 */
class LoyaltyDiscount implements Discount {
    private final int customerYears;
    
    public LoyaltyDiscount(int customerYears) {
        this.customerYears = customerYears;
    }
    
    @Override
    public double calculate(double amount) {
        // 1% discount per year, max 20%
        double discountPercentage = Math.min(customerYears * 0.01, 0.20);
        return amount * discountPercentage;
    }
}

/**
 * Calculator now closed for modification!
 */
class ImprovedDiscountCalculator {
    private final Discount discount;
    
    public ImprovedDiscountCalculator(Discount discount) {
        this.discount = discount;
    }
    
    public double calculateDiscount(double amount) {
        return discount.calculate(amount);
    }
}

/**
 * BENEFITS:
 * 
 * ✓ Add new discounts without modifying existing code
 * ✓ Each discount is independent and testable
 * ✓ No risk of breaking existing discounts
 * ✓ Follows Strategy pattern
 * ✓ Open for extension (new Discount implementations)
 * ✓ Closed for modification (ImprovedDiscountCalculator unchanged)
 */

// Usage
class DiscountExample {
    public void example() {
        // Use percentage discount
        Discount discount1 = new PercentageDiscount(0.15);
        ImprovedDiscountCalculator calc1 = new ImprovedDiscountCalculator(discount1);
        double saving1 = calc1.calculateDiscount(100.0);  // $15
        
        // Use seasonal discount (added without modifying calculator!)
        Discount discount2 = new SeasonalDiscount(0.20, 0.10);
        ImprovedDiscountCalculator calc2 = new ImprovedDiscountCalculator(discount2);
        double saving2 = calc2.calculateDiscount(100.0);  // Depends on season
    }
}
```

## 3. Liskov Substitution Principle (LSP)

### "Objects of a superclass should be replaceable with objects of a subclass without breaking the application"

```java
/**
 * LISKOV SUBSTITUTION PRINCIPLE
 * 
 * Subtypes must be substitutable for their base types
 * If S is a subtype of T, then objects of type T can be replaced with objects of type S
 */

// ========================================
// ❌ VIOLATION: Square-Rectangle Problem
// ========================================

/**
 * Classic LSP violation: Square IS-A Rectangle?
 */
class Rectangle {
    protected int width;
    protected int height;
    
    public void setWidth(int width) {
        this.width = width;
    }
    
    public void setHeight(int height) {
        this.height = height;
    }
    
    public int getWidth() { return width; }
    public int getHeight() { return height; }
    
    public int getArea() {
        return width * height;
    }
}

/**
 * Square inherits from Rectangle
 * But violates LSP!
 */
class Square extends Rectangle {
    
    @Override
    public void setWidth(int width) {
        this.width = width;
        this.height = width;  // Square: width == height
    }
    
    @Override
    public void setHeight(int height) {
        this.width = height;  // Square: width == height
        this.height = height;
    }
}

/**
 * Test that works for Rectangle but FAILS for Square
 */
class LSPViolationTest {
    public void testRectangle(Rectangle rect) {
        rect.setWidth(5);
        rect.setHeight(4);
        
        // Expected: 5 * 4 = 20
        int area = rect.getArea();
        assert area == 20 : "Expected area 20, got " + area;
    }
    
    public void demo() {
        Rectangle rect = new Rectangle();
        testRectangle(rect);  // ✓ PASSES (area = 20)
        
        Rectangle square = new Square();
        testRectangle(square);  // ❌ FAILS! (area = 16, not 20)
        
        /**
         * PROBLEM:
         * Square cannot substitute Rectangle!
         * Violates LSP!
         */
    }
}

// ========================================
// ✓ SOLUTION: Separate Hierarchies
// ========================================

/**
 * Common interface (not inheritance)
 */
interface Shape {
    int getArea();
}

/**
 * Independent implementations
 */
class GoodRectangle implements Shape {
    private final int width;
    private final int height;
    
    public GoodRectangle(int width, int height) {
        this.width = width;
        this.height = height;
    }
    
    @Override
    public int getArea() {
        return width * height;
    }
    
    public int getWidth() { return width; }
    public int getHeight() { return height; }
}

class GoodSquare implements Shape {
    private final int side;
    
    public GoodSquare(int side) {
        this.side = side;
    }
    
    @Override
    public int getArea() {
        return side * side;
    }
    
    public int getSide() { return side; }
}

/**
 * BENEFITS:
 * 
 * ✓ No false inheritance relationship
 * ✓ Each shape behaves correctly
 * ✓ Can substitute Shape without issues
 * ✓ Follows LSP
 */

/**
 * ❌ VIOLATION: Not all birds can fly!
 */
class Bird {
    public void fly() {
        System.out.println("Flying...");
    }
}

class Sparrow extends Bird {
    // ✓ Can fly
}

class Penguin extends Bird {
    @Override
    public void fly() {
        // ❌ Penguins can't fly!

    }
}

class Ostrich extends Bird {
    @Override
    public void fly() {
        // ❌ Ostriches can't fly!

/**
 * Code expecting Bird breaks with Penguin/Ostrich
 */
class BirdWatcher {
    public void watchBirdFly(Bird bird) {

}

/**
 * ✓ SOLUTION: Separate flying capability
 */
interface BirdInterface {
    void eat();
    void sleep();
}

interface Flyable {
    void fly();
}

class FlyingBird implements BirdInterface, Flyable {
    @Override
    public void eat() {
        System.out.println("Eating...");
    }
    
    @Override
    public void sleep() {
        System.out.println("Sleeping...");
    }
    
    @Override
    public void fly() {
        System.out.println("Flying...");
    }
}

class FlightlessBird implements BirdInterface {
    @Override
    public void eat() {
        System.out.println("Eating...");
    }
    
    @Override
    public void sleep() {
        System.out.println("Sleeping...");
    }
}

class GoodSparrow extends FlyingBird {
    // Inherits fly capability
}

class GoodPenguin extends FlightlessBird {
    // No fly method (can't accidentally call it!)
}

/**
 * Now code is correct
 */
class GoodBirdWatcher {
    public void watchBirdFly(Flyable bird) {
        bird.fly();  // ✓ Only accepts birds that can fly!
    }
    
    public void watchBird(BirdInterface bird) {
        bird.eat();   // ✓ All birds can eat
        bird.sleep(); // ✓ All birds can sleep
    }
}

 * BENEFITS:
 * 
 * ✓ GoodPenguin can substitute FlightlessBird
 * ✓ GoodSparrow can substitute FlyingBird

class Account {
    protected double balance;
    
    public void deposit(double amount) {
        balance += amount;
    }
    
    public void withdraw(double amount) {
        if (balance >= amount) {
            balance -= amount;
        }
    }
    
    public double getBalance() {
        return balance;
    }
}

class ReadOnlyAccount extends Account {
    @Override
    public void deposit(double amount) {
        // ❌ Can't deposit in read-only account!
        throw new UnsupportedOperationException("Read-only account");
    }
    
    @Override
    public void withdraw(double amount) {
        // ❌ Can't withdraw from read-only account!
        throw new UnsupportedOperationException("Read-only account");
    }
}

/**
 * Code expecting Account breaks with ReadOnlyAccount
 */
class AccountService {
    public void transfer(Account from, Account to, double amount) {
        from.withdraw(amount);  // ❌ Fails if 'from' is ReadOnlyAccount!
        to.deposit(amount);     // ❌ Fails if 'to' is ReadOnlyAccount!
    }
}

/**
 * ✓ SOLUTION: Separate interfaces
 */
interface ReadableAccount {
    double getBalance();
}

interface WritableAccount extends ReadableAccount {
    void deposit(double amount);
    void withdraw(double amount);
}

class StandardAccount implements WritableAccount {
    private double balance;
    
    @Override
    public double getBalance() {
        return balance;
    }
    
    @Override
    public void deposit(double amount) {
        balance += amount;
    }
    
    @Override
    public void withdraw(double amount) {
        if (balance >= amount) {
            balance -= amount;
        }
    }
}

class AccountSnapshot implements ReadableAccount {
    private final double balance;
    
    public AccountSnapshot(double balance) {
        this.balance = balance;
    }
    
    @Override
    public double getBalance() {
        return balance;
    }
}

/**
 * Service now type-safe
 */
class GoodAccountService {
    public void transfer(WritableAccount from, WritableAccount to, double amount) {
        from.withdraw(amount);  // ✓ Both must be writable!
        to.deposit(amount);
    }
    
    public double checkBalance(ReadableAccount account) {
        return account.getBalance();  // ✓ Works for all accounts
    }
}
```

## 5. Dependency Inversion Principle (DIP)

### "Depend on abstractions, not concretions"

```java
/**
 * DEPENDENCY INVERSION PRINCIPLE
 * 
 * High-level modules should not depend on low-level modules
 * Both should depend on abstractions
 * Abstractions should not depend on details
 * Details should depend on abstractions
 */

// ========================================
// ❌ VIOLATION: Tight Coupling
// ========================================

/**
 * High-level class depends on low-level concrete classes
 */
class MySQLDatabase {
    public void save(String data) {
        System.out.println("Saving to MySQL: " + data);
    }
    
    public String get(String id) {
        System.out.println("Getting from MySQL: " + id);
        return "data";
    }
}

class UserService {
    private MySQLDatabase database;  // ❌ Depends on concrete class!
    
    public UserService() {
        this.database = new MySQLDatabase();  // ❌ Tight coupling!
    }
    
    public void createUser(String userData) {
        database.save(userData);
    }
    
    public String getUser(String userId) {
        return database.get(userId);
    }
    
    /**
     * PROBLEMS:
     * 
     * ✗ Tightly coupled to MySQL
     * ✗ Can't switch to PostgreSQL without modifying UserService
     * ✗ Can't test without real MySQL database
     * ✗ Violates Dependency Inversion Principle
     */
}

// ========================================
// ✓ SOLUTION: Depend on Abstractions
// ========================================

/**
 * Abstraction (interface)
 */
interface Database {
    void save(String data);
    String get(String id);
}

/**
 * Low-level implementations
 */
class MySQLDatabaseImpl implements Database {
    @Override
    public void save(String data) {
        System.out.println("Saving to MySQL: " + data);
    }
    
    @Override
    public String get(String id) {
        System.out.println("Getting from MySQL: " + id);
        return "data";
    }
}

class PostgreSQLDatabase implements Database {
    @Override
    public void save(String data) {
        System.out.println("Saving to PostgreSQL: " + data);
    }
    
    @Override
    public String get(String id) {
        System.out.println("Getting from PostgreSQL: " + id);
        return "data";
    }
}

class MongoDBDatabase implements Database {
    @Override
    public void save(String data) {
        System.out.println("Saving to MongoDB: " + data);
    }
    
    @Override
    public String get(String id) {
        System.out.println("Getting from MongoDB: " + id);
        return "data";
    }
}

/**
 * High-level class depends on abstraction
 */
class GoodUserService {
    private final Database database;  // ✓ Depends on interface!
    
    public GoodUserService(Database database) {  // ✓ Dependency injection!
        this.database = database;
    }
    
    public void createUser(String userData) {
        database.save(userData);
    }
    
    public String getUser(String userId) {
        return database.get(userId);
    }
    
    /**
     * BENEFITS:
     * 
     * ✓ Loosely coupled
     * ✓ Can switch databases without modifying UserService
     * ✓ Easy to test (mock Database interface)
     * ✓ Follows Dependency Inversion Principle
     */
}

// Usage
class DependencyInversionExample {
    public void example() {
        // Use MySQL
        Database mysql = new MySQLDatabaseImpl();
        GoodUserService service1 = new GoodUserService(mysql);
        service1.createUser("John");
        
        // Switch to PostgreSQL (no code change in UserService!)
        Database postgres = new PostgreSQLDatabase();
        GoodUserService service2 = new GoodUserService(postgres);
        service2.createUser("Jane");
        
        // Switch to MongoDB (no code change in UserService!)
        Database mongo = new MongoDBDatabase();
        GoodUserService service3 = new GoodUserService(mongo);
        service3.createUser("Bob");
    }
}

// REAL-WORLD EXAMPLE: Notification System

/**
 * ❌ VIOLATION: Depends on concrete email class
 */

    public void send(String to, String message) {

class NotificationService {

 * ✓ SOLUTION: Notification abstraction

    }

    @Override

        System.out.println("Sending push notification to " + recipient + ": " + message);

    }

            new EmailChannel(),
            new SMSChannel(),

    }
```

## Summary: SOLID Principles Together

```java
/**
 * HOW SOLID PRINCIPLES WORK TOGETHER
 */

// Example: E-Commerce Order System

// SRP: Separate responsibilities
class OrderEntity { /* Data only */ }
class OrderRepository { /* Database only */ }
class OrderValidator { /* Validation only */ }
class OrderPricingService { /* Pricing only */ }

// OCP: Open for extension
interface PaymentMethod {
    void process(double amount);
}
class CreditCard implements PaymentMethod { /* ... */ }
class PayPal implements PaymentMethod { /* ... */ }
// Add new payment: just create new class!

// LSP: Substitutable subtypes
interface Account {
    double getBalance();
}
interface WritableAccount extends Account {
    void deposit(double amount);
}
// Both can substitute Account correctly

// ISP: Focused interfaces
interface Readable { /* ... */ }
interface Writable { /* ... */ }
interface Searchable { /* ... */ }
// Clients depend only on what they need

// DIP: Depend on abstractions
class OrderService {
    private final OrderRepository repository;  // Interface!
    private final PaymentMethod payment;       // Interface!
    
    public OrderService(OrderRepository repository, PaymentMethod payment) {
        this.repository = repository;
        this.payment = payment;
    }
}
```

### Benefits of Following SOLID

```
Maintainability:
✓ Easy to understand
✓ Easy to modify
✓ Easy to extend

Testability:
✓ Easy to mock dependencies
✓ Easy to write unit tests
✓ Easy to isolate components

Flexibility:
✓ Easy to swap implementations
✓ Easy to add new features
✓ Easy to refactor

Reusability:
✓ Components are independent
✓ Can reuse in other projects
✓ Clear interfaces
```

### Quick Reference

```
S - Single Responsibility
    One class = One reason to change
    Example: Separate data, database, validation

O - Open/Closed
    Open for extension, closed for modification
    Example: Strategy pattern for discounts

L - Liskov Substitution
    Subtypes must be substitutable
    Example: Don't inherit if behavior differs

I - Interface Segregation
    Many specific interfaces > One fat interface
    Example: Separate Printer, Scanner, Fax

D - Dependency Inversion
    Depend on abstractions, not concretions
    Example: Inject Database interface, not MySQL class
```

### Violations vs Solutions

```
VIOLATION                     SOLUTION
────────────────────────────────────────────────────
God Object                 →  Separate by responsibility
if-else chain              →  Strategy pattern (OCP)
Square extends Rectangle   →  Separate hierarchies (LSP)
Fat interface              →  Split into focused interfaces (ISP)
new MySQLDatabase()        →  Inject Database interface (DIP)
```
