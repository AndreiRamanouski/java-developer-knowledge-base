# DRY, KISS, YAGNI Principles

## Overview

DRY, KISS, and YAGNI are three fundamental principles that guide everyday coding decisions beyond the formal SOLID principles. While SOLID focuses on object-oriented design structure, DRY/KISS/YAGNI address pragmatic development choices: how to avoid duplication, when to add abstraction, and what features to build. DRY (Don't Repeat Yourself) eliminates knowledge duplication—every piece of knowledge should have a single, authoritative representation. KISS (Keep It Simple, Stupid) favors simplicity over cleverness—the simplest solution is usually the best. YAGNI (You Aren't Gonna Need It) prevents over-engineering—don't build features until they're needed. These principles seem simple but are deceptively hard to apply correctly. DRY isn't just about code duplication—duplicating knowledge is the issue, not duplicating code structure. KISS doesn't mean avoiding complexity when it's necessary—it means avoiding accidental complexity. YAGNI doesn't mean no planning—it means no speculative implementation. The real challenge is balance: DRY vs premature abstraction, KISS vs necessary complexity, YAGNI vs good architecture. This guide shows when to apply each principle, when they conflict, and when violating them is the right choice. You'll see that experienced developers know these principles deeply but also know when to break them—copy-paste can be better than wrong abstraction (violate DRY), complex code can be simpler than "simple" magic (violate KISS), and building infrastructure early can prevent pain later (violate YAGNI). Understanding these tradeoffs separates junior from senior developers.

### "Every piece of knowledge must have a single, unambiguous, authoritative representation within a system"

## 2. Keep It Simple, Stupid (KISS)

### "Simplicity should be a key goal in design, and unnecessary complexity should be avoided"

```java
/**
 * KISS PRINCIPLE
 * 
 * Simple solutions are better than complex ones
 * Avoid over-engineering
 * Write code humans can understand
 */

// ========================================
// ❌ VIOLATION: Unnecessary Complexity
// ========================================

/**
 * Over-engineered calculator
 */
interface Operation {
    double execute(double a, double b);
}

class AddOperation implements Operation {
    @Override
    public double execute(double a, double b) {
        return a + b;
    }
}

class SubtractOperation implements Operation {
    @Override
    public double execute(double a, double b) {
        return a - b;
    }
}

class MultiplyOperation implements Operation {
    @Override
    public double execute(double a, double b) {
        return a * b;
    }
}

class DivideOperation implements Operation {
    @Override
    public double execute(double a, double b) {
        if (b == 0) throw new ArithmeticException("Division by zero");
        return a / b;
    }
}

class OperationFactory {
    public Operation createOperation(String operator) {
        switch (operator) {
            case "+": return new AddOperation();
            case "-": return new SubtractOperation();
            case "*": return new MultiplyOperation();
            case "/": return new DivideOperation();
            default: throw new IllegalArgumentException("Unknown operator");
        }
    }
}

class ComplexCalculator {
    private final OperationFactory factory;
    
    public ComplexCalculator() {
        this.factory = new OperationFactory();
    }
    
    public double calculate(double a, String operator, double b) {
        Operation operation = factory.createOperation(operator);
        return operation.execute(a, b);
    }
}

/**
 * PROBLEMS:
 * 
 * ✗ 6 classes for simple calculator
 * ✗ Over-engineered (Strategy + Factory for basic math)
 * ✗ Hard to understand at a glance
 * ✗ More code to maintain
 * ✗ Violates KISS principle
 */

// ========================================
// ✓ SOLUTION: Simple and Clear
// ========================================

class SimpleCalculator {
    
    public double calculate(double a, String operator, double b) {
        switch (operator) {
            case "+": return a + b;
            case "-": return a - b;
            case "*": return a * b;
            case "/":
                if (b == 0) throw new ArithmeticException("Division by zero");
                return a / b;
            default:
                throw new IllegalArgumentException("Unknown operator: " + operator);
        }
    }
    
    /**
     * BENEFITS:
     * 
     * ✓ One class (was 6)
     * ✓ Easy to understand
     * ✓ Less code to maintain
     * ✓ Simple and effective
     * 
     * 
     * WHEN TO USE COMPLEX VERSION:
     * 
     * • Need to add operations at runtime
     * • Operations have complex state
     * • Operations need dependency injection
     * 
     * For simple calculator? KISS wins!
     */
}

// ========================================

/**
 * IMPORTANT: Simple ≠ Easy
 * 
 * Simple: Few concepts, few dependencies
 * Easy: Familiar, convenient
 * 
 * Simple code might not be easy at first,
 * but it's better in the long run!
 */

/**
 * Example: Streams vs Loops
 */
class SimpleVsEasy {
    
    /**
     * EASY (but not simple - many concepts)
     */
    public List<String> processNamesComplex(List<User> users) {
        List<String> result = new ArrayList<>();
        for (User user : users) {
            if (user.isActive()) {
                String name = user.getName();
                if (name != null) {
                    String upper = name.toUpperCase();
                    result.add(upper);
                }
            }
        }
        return result;
    }
    
    /**
     * SIMPLE (few concepts, declarative)
     */
    public List<String> processNamesSimple(List<User> users) {
        return users.stream()
            .filter(User::isActive)
            .map(User::getName)
            .filter(Objects::nonNull)
            .map(String::toUpperCase)
            .collect(Collectors.toList());
    }
    
    /**
     * Stream version is simpler:
     * • Declarative (what, not how)
     * • No mutable state
     * • No intermediate variables
     * • Clear intent
     * 
     * Even if loops feel "easier" because you're used to them!
     */
}
```

## 3. You Aren't Gonna Need It (YAGNI)

### "Always implement things when you actually need them, never when you just foresee that you need them"

```java
/**
 * YAGNI PRINCIPLE
 * 
 * Don't build features until they're needed
 * Don't add "just in case" functionality
 * Avoid speculative generality
 */

// ========================================
// ❌ VIOLATION: Speculative Generality
// ========================================

/**
 * Over-engineered for "future" requirements
 */
class OverEngineeredUserService {
    
    // Current requirement: Save user to database
    // But "we might need to save to file or cloud later"...
    
    private final StorageStrategy strategy;
    
    public OverEngineeredUserService(StorageStrategy strategy) {
        this.strategy = strategy;
    }
    
    public void saveUser(User user) {
        strategy.save(user);
    }
}

interface StorageStrategy {
    void save(User user);
    void delete(User user);
    void update(User user);
    User findById(String id);
    List<User> findAll();
    void backup();      // "We might need backup"
    void restore();     // "We might need restore"
    void export();      // "We might need export"
    void import_();     // "We might need import"
}

    public void backup() {

// ========================================
// ✓ SOLUTION: Build What You Need Now
// ========================================

/**
 * Simple repository (what we actually need)
 */
class SimpleUserRepository {
    private final JdbcTemplate jdbcTemplate;
    
    public SimpleUserRepository(JdbcTemplate jdbcTemplate) {
        this.jdbcTemplate = jdbcTemplate;
    }
    
    public void save(User user) {
        String sql = "INSERT INTO users (id, name, email) VALUES (?, ?, ?)";
        jdbcTemplate.update(sql, user.getId(), user.getName(), user.getEmail());
    }
    
    public User findById(String id) {
        String sql = "SELECT * FROM users WHERE id = ?";
        return jdbcTemplate.queryForObject(sql, this::mapUser, id);
    }
    
    private User mapUser(ResultSet rs, int rowNum) throws SQLException {
        return new User(
            rs.getString("id"),
            rs.getString("name"),
            rs.getString("email")
        );
    }
    
    /**
     * BENEFITS:
     * 
     * ✓ Simple and direct
     * ✓ Only what we need now
     * ✓ Easy to refactor later if needed
     * ✓ No wasted effort
     * 
     * 
     * WHEN FILE STORAGE IS NEEDED:
     * 
     * Then we can extract interface and add FileRepository
     * But not before!
     */
}

 * 1. KNOWN REQUIREMENT (NOT SPECULATION)
 */

     * If requirement is in current sprint/milestone,
```

```java
// YAGNI Example: Build only what's needed
class SimpleOrderProcessor {
    private final CreditCardProcessor creditCard;
    private final EmailService email;
    
    public void processOrder(Order order) {
        creditCard.charge(order.getTotal(), order.getCard());
        email.sendConfirmation(order.getCustomer().getEmail());
    }
    // Works in 2 hours, not 2 weeks of abstraction!
}

// DRY Example: Extract duplicated logic
class PriceCalculator {
    public double calculatePrice(double basePrice, int quantity) {
        double price = basePrice * quantity;
        if (price > 100) {
            price = price * 0.9;  // ONE place
        }
        return price;
    }
}

// KISS Example: Keep it simple
class SimpleValidator {
    public void validateEmail(String email) {
        if (email == null || !email.contains("@")) {
            throw new ValidationException("Invalid email");
        }
    }
    // No need for 10 classes!
}
```

## Summary: The Pragmatic Approach

```
QUICK REFERENCE:

DRY - Don't Repeat Yourself
What: Knowledge in one place
✓ DO: Extract common logic
✗ DON'T: Abstract too early

KISS - Keep It Simple
What: Simple over complex
✓ DO: Straightforward solutions
✗ DON'T: Over-engineer

YAGNI - You Aren't Gonna Need It
What: Build what's needed now
✓ DO: Current requirements only
✗ DON'T: Speculative features

Priority: YAGNI > KISS > DRY

Final Wisdom:
"Make it work, make it right, make it fast"
"Duplication is cheaper than wrong abstraction"
```
