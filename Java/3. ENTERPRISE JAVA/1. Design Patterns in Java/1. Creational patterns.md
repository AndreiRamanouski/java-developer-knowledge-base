# Creational Design Patterns

## Overview

Creational patterns solve the fundamental problem of object creation—how to create objects in a flexible, reusable way that decouples the client from concrete implementations. Instead of `new MyClass()` scattered throughout code (tight coupling, inflexibility, testing difficulties), creational patterns provide abstractions: Singleton ensures single instance (database connections, configuration), Factory decouples creation logic (different implementations based on runtime conditions), Builder constructs complex objects step-by-step (avoiding telescoping constructors with 10+ parameters), Prototype creates copies efficiently (avoiding expensive initialization), Object Pool reuses expensive objects (database connections, threads), and Dependency Injection inverts control (framework provides dependencies, enabling loose coupling and testability). These patterns are foundational to modern Java—Spring's entire architecture relies on Singleton, Factory, and DI; builders appear in Java's own APIs (StringBuilder, Stream.Builder); pools manage JDBC connections and thread pools. Understanding creational patterns means understanding how professional Java applications are structured, tested, and maintained at scale. This guide covers thread-safe Singleton implementations (DCL, lazy holder, enum), Factory Method vs Abstract Factory (when to use each), Builder pattern (fluent API design, telescoping constructor problem), Prototype and cloning (deep vs shallow, clone pitfalls), Object Pool (connection pools, thread pools, implementation), and Dependency Injection (constructor vs setter vs field, DI containers, testing benefits).

## 1. Singleton Pattern

### Ensuring Single Instance

```java
/**
 * SINGLETON PATTERN
 * 
 * Ensures a class has only one instance
 * Provides global point of access
 */

// ========================================
// 1. EAGER INITIALIZATION
// ========================================
public class EagerSingleton {
    // Instance created at class loading
    private static final EagerSingleton INSTANCE = new EagerSingleton();
    
    private EagerSingleton() {
        // Private constructor prevents instantiation
    }
    
    public static EagerSingleton getInstance() {
        return INSTANCE;
    }
    
    /**
     * CHARACTERISTICS:
     * 
     * Pros:
     * ✓ Thread-safe (JVM guarantees)
     * ✓ Simple implementation
     * ✓ No synchronization overhead
     * 
     * Cons:
     * ✗ Eager initialization (always created)
     * ✗ Cannot handle initialization failure
     * ✗ No lazy loading
     * 
     * Use when:
     * - Instance always needed
     * - Initialization is cheap
     * - No initialization parameters needed
     */
}

// ========================================
// 2. LAZY INITIALIZATION (NOT THREAD-SAFE)
// ========================================
public class LazySingletonUnsafe {
    private static LazySingletonUnsafe instance;
    
    private LazySingletonUnsafe() {}
    
    public static LazySingletonUnsafe getInstance() {
        if (instance == null) {  // NOT THREAD-SAFE!
            instance = new LazySingletonUnsafe();
        }
        return instance;
    }
    
    /**
     * PROBLEM:
     * 
     * Thread 1: Checks instance == null (true)
     * Thread 2: Checks instance == null (true)
     * Thread 1: Creates instance
     * Thread 2: Creates instance (SECOND INSTANCE!)
     * 
     * Result: Multiple instances possible!
     * 
     * NEVER USE THIS IN PRODUCTION!
     */
}

// ========================================
// 3. SYNCHRONIZED METHOD (THREAD-SAFE BUT SLOW)
// ========================================
public class SynchronizedSingleton {
    private static SynchronizedSingleton instance;
    
    private SynchronizedSingleton() {}
    
    public static synchronized SynchronizedSingleton getInstance() {
        if (instance == null) {
            instance = new SynchronizedSingleton();
        }
        return instance;
    }
    
    /**
     * CHARACTERISTICS:
     * 
     * Pros:
     * ✓ Thread-safe
     * ✓ Lazy initialization
     * 
     * Cons:
     * ✗ Synchronized on every call (slow!)
     * ✗ 100x slower than unsynchronized
     * 
     * Performance:
     * Unsynchronized: 10ns per call
     * Synchronized: 1000ns per call
     * 
     * Use when:
     * - Rarely accessed singleton
     * - Performance not critical
     * 
     * Usually NOT recommended!
     */
}

// ========================================
// 4. DOUBLE-CHECKED LOCKING (DCL)
// ========================================
public class DoubleCheckedLockingSingleton {
    // MUST be volatile!
    private static volatile DoubleCheckedLockingSingleton instance;
    
    private DoubleCheckedLockingSingleton() {}
    
    public static DoubleCheckedLockingSingleton getInstance() {
        // First check (no locking)
        if (instance == null) {
            synchronized (DoubleCheckedLockingSingleton.class) {
                // Second check (with locking)
                if (instance == null) {
                    instance = new DoubleCheckedLockingSingleton();
                }
            }
        }
        return instance;
    }
    
    /**
     * WHY VOLATILE IS REQUIRED:
     * 
     * Without volatile:
     * 1. Thread 1 allocates memory
     * 2. Thread 1 sets instance to point to memory
     * 3. Thread 1 initializes object
     * 
     * Thread 2 might see partially constructed object!
     * 
     * With volatile:
     * - Happens-before guarantee
     * - No reordering
     * - Full initialization before visibility
     * 
     * 
     * PERFORMANCE:
     * 
     * After initialization:
     * - No synchronization (fast path)
     * - Performance ~= eager singleton
     * 
     * Before initialization:
     * - Synchronization only once
     * 
     * 
     * CHARACTERISTICS:
     * 
     * Pros:
     * ✓ Thread-safe
     * ✓ Lazy initialization
     * ✓ Fast after initialization
     * 
     * Cons:
     * ✗ Complex code
     * ✗ Requires Java 5+ (volatile fix)
     * ✗ Easy to get wrong
     * 
     * Use when:
     * - Lazy loading needed
     * - Singleton frequently accessed
     * - Initialization is expensive
     */
}

// ========================================
// 5. LAZY HOLDER IDIOM (BEST APPROACH!)
// ========================================
public class LazyHolderSingleton {
    
    private LazyHolderSingleton() {}
    
    // Inner class not loaded until getInstance() called
    private static class Holder {
        private static final LazyHolderSingleton INSTANCE = 
            new LazyHolderSingleton();
    }
    
    public static LazyHolderSingleton getInstance() {
        return Holder.INSTANCE;
    }
    
    /**
     * HOW IT WORKS:
     * 
     * 1. LazyHolderSingleton class loaded
     * 2. Holder class NOT loaded yet
     * 3. First call to getInstance()
     * 4. Holder class loaded and initialized
     * 5. INSTANCE created
     * 6. Subsequent calls return same INSTANCE
     * 
     * 
     * THREAD SAFETY:
     * 
     * JVM guarantees:
     * - Class initialization is thread-safe
     * - Happens once
     * - Synchronized by JVM
     * 
     * 
     * CHARACTERISTICS:
     * 
     * Pros:
     * ✓ Thread-safe (JVM guarantee)
     * ✓ Lazy initialization
     * ✓ Fast (no synchronization after init)
     * ✓ Simple code
     * ✓ No volatile needed
     * 
     * Cons:
     * (None significant!)
     * 
     * 
     * RECOMMENDED APPROACH!
     * 
     * Best of all worlds:
     * - Safe as synchronized
     * - Fast as eager
     * - Lazy like DCL
     * - Simple code
     */
}

// ========================================
// 6. ENUM SINGLETON (MODERN BEST PRACTICE!)
// ========================================
public enum EnumSingleton {
    INSTANCE;
    
    // Fields and methods
    private int value;
    
    public void setValue(int value) {
        this.value = value;
    }
    
    public int getValue() {
        return value;
    }
    
    public void doSomething() {
        System.out.println("Singleton doing something");
    }
    
    /**
     * CHARACTERISTICS:
     * 
     * Pros:
     * ✓ Thread-safe (JVM guarantee)
     * ✓ Serialization-safe (automatically)
     * ✓ Reflection-proof
     * ✓ Simplest code
     * ✓ Eager initialization
     * 
     * Cons:
     * ✗ No lazy initialization
     * ✗ Cannot extend class
     * ✗ Unusual syntax
     * 
     * 
     * SERIALIZATION SAFE:
     * 
     * Regular singleton:
     * - Deserialization creates new instance!
     * - Need readResolve() method
     * 
     * Enum singleton:
     * - JVM prevents duplicate instances
     * - Automatically serialization-safe
     * 
     * 
     * REFLECTION PROOF:
     * 
     * Regular singleton:
     * - Reflection can call private constructor
     * - Can create multiple instances
     * 
     * Enum singleton:
     * - Cannot instantiate enum via reflection
     * - JVM enforces single instance
     * 
     * 
     * RECOMMENDED FOR:
     * 
     * ✓ Most singleton needs
     * ✓ When serialization required
     * ✓ When reflection attacks possible
     * 
     * 
     * USAGE:
     * 
     * EnumSingleton.INSTANCE.doSomething();
     * EnumSingleton.INSTANCE.setValue(42);
     */
}

// ========================================
// 7. COMPARISON EXAMPLE
// ========================================
public class SingletonComparison {
    
    public static void demonstrateAll() {
        // Eager - simple but always created
        EagerSingleton eager = EagerSingleton.getInstance();
        
        // Synchronized - thread-safe but slow
        SynchronizedSingleton sync = SynchronizedSingleton.getInstance();
        
        // DCL - thread-safe and fast but complex
        DoubleCheckedLockingSingleton dcl = 
            DoubleCheckedLockingSingleton.getInstance();
        
        // Lazy Holder - BEST for lazy loading
        LazyHolderSingleton holder = LazyHolderSingleton.getInstance();
        
        // Enum - BEST for most cases
        EnumSingleton.INSTANCE.doSomething();
    }
    
    /**
     * DECISION TREE:
     * 
     * Need lazy loading?
     *   NO  → Use Enum Singleton
     *   YES → Use Lazy Holder Idiom
     * 
     * Need serialization safety?
     *   → Use Enum Singleton
     * 
     * Need to extend class?
     *   → Use Lazy Holder Idiom
     *   (Enums cannot be extended)
     * 
     * 
     * AVOID:
     * 
     * ✗ Lazy Singleton (not thread-safe)
     * ✗ Synchronized Method (too slow)
     * ✗ DCL (too complex, easy to get wrong)
     */
}

// ========================================
// 8. REAL-WORLD EXAMPLE: DATABASE CONNECTION
// ========================================
public class DatabaseConnectionManager {
    
    private DatabaseConnectionManager() {
        // Initialize connection pool
        System.out.println("Initializing database connection pool...");
    }
    
    private static class Holder {
        private static final DatabaseConnectionManager INSTANCE = 
            new DatabaseConnectionManager();
    }
    
    public static DatabaseConnectionManager getInstance() {
        return Holder.INSTANCE;
    }
    
    public Connection getConnection() {
        // Return connection from pool
        return null;
    }
    
    /**
     * WHY SINGLETON:
     * 
     * - Single connection pool needed
     * - Expensive to create (lazy loading beneficial)
     * - Accessed from multiple places
     * - Thread-safe access required
     * 
     * Lazy Holder Idiom chosen because:
     * ✓ Lazy initialization (pool created on first use)
     * ✓ Thread-safe
     * ✓ Fast
     * ✓ Simple
     */
}

// ========================================
// 9. BREAKING SINGLETON (PITFALLS)
// ========================================
public class SingletonPitfalls {
    
    /**
     * PITFALL 1: REFLECTION ATTACK
     */
    public static void reflectionAttack() throws Exception {
        LazyHolderSingleton instance1 = LazyHolderSingleton.getInstance();
        
        // Reflection attack
        Constructor<LazyHolderSingleton> constructor = 
            LazyHolderSingleton.class.getDeclaredConstructor();
        constructor.setAccessible(true);
        LazyHolderSingleton instance2 = constructor.newInstance();
        
        // Two different instances!
        System.out.println(instance1 == instance2);  // false!
        
        /**
         * SOLUTION: Throw exception in constructor
         * 
         * private LazyHolderSingleton() {
         *   if (Holder.INSTANCE != null) {
         *     throw new IllegalStateException("Already instantiated");
         *   }
         * }
         * 
         * OR use Enum (reflection-proof)
         */
    }
    
    /**
     * PITFALL 2: SERIALIZATION
     */
    public static void serializationAttack() {
        // Regular singleton loses uniqueness on deserialization!
        
        /**
         * SOLUTION: Implement readResolve()
         * 
         * protected Object readResolve() {
         *   return getInstance();
         * }
         * 
         * OR use Enum (serialization-safe)
         */
    }
    
    /**
     * PITFALL 3: CLONING
     */
    public static void cloningAttack() {
        /**
         * SOLUTION: Override clone()
         * 
         * @Override
         * protected Object clone() throws CloneNotSupportedException {
         *   throw new CloneNotSupportedException();
         * }
         */
    }
}
```

## 2. Factory Method vs Abstract Factory

### Decoupling Object Creation

```java
/**
 * FACTORY METHOD PATTERN
 * 
 * Defines interface for creating object
 * Subclasses decide which class to instantiate
 */

// ========================================
// PRODUCT HIERARCHY
// ========================================
interface Vehicle {
    void drive();
    int getWheels();
}

class Car implements Vehicle {
    @Override
    public void drive() {
        System.out.println("Driving a car");
    }
    
    @Override
    public int getWheels() {
        return 4;
    }
}

class Motorcycle implements Vehicle {
    @Override
    public void drive() {
        System.out.println("Riding a motorcycle");
    }
    
    @Override
    public int getWheels() {
        return 2;
    }
}

class Truck implements Vehicle {
    @Override
    public void drive() {
        System.out.println("Driving a truck");
    }
    
    @Override
    public int getWheels() {
        return 6;
    }
}

// ========================================
// FACTORY METHOD PATTERN
// ========================================
abstract class VehicleFactory {
    
    // Factory method - subclasses implement
    protected abstract Vehicle createVehicle();
    
    // Template method using factory method
    public Vehicle orderVehicle() {
        Vehicle vehicle = createVehicle();  // Factory method
        
        // Common processing
        System.out.println("Preparing vehicle...");
        vehicle.drive();
        
        return vehicle;
    }
}

class CarFactory extends VehicleFactory {
    @Override
    protected Vehicle createVehicle() {
        return new Car();
    }
}

class MotorcycleFactory extends VehicleFactory {
    @Override
    protected Vehicle createVehicle() {
        return new Motorcycle();
    }
}

/**
 * FACTORY METHOD CHARACTERISTICS:
 * 
 * Structure:
 * - One product hierarchy (Vehicle)
 * - Factory method in base class
 * - Subclasses provide concrete implementation
 * 
 * When to use:
 * ✓ Single product type
 * ✓ Creation logic varies by subclass
 * ✓ Framework needs extension point
 * 
 * Benefits:
 * ✓ Open/Closed Principle (extend without modifying)
 * ✓ Single Responsibility (creation separate from use)
 * ✓ Flexibility (new products via subclasses)
 * 
 * Example usage:
 * VehicleFactory factory = new CarFactory();
 * Vehicle vehicle = factory.orderVehicle();
 */

// ========================================
// ABSTRACT FACTORY PATTERN
// ========================================

/**
 * ABSTRACT FACTORY PATTERN
 * 
 * Creates families of related objects
 * Without specifying concrete classes
 */

// Product families
interface Button {
    void click();
}

interface Checkbox {
    void check();
}

// Windows family
class WindowsButton implements Button {
    @Override
    public void click() {
        System.out.println("Windows button clicked");
    }
}

class WindowsCheckbox implements Checkbox {
    @Override
    public void check() {
        System.out.println("Windows checkbox checked");
    }
}

// Mac family
class MacButton implements Button {
    @Override
    public void click() {
        System.out.println("Mac button clicked");
    }
}

class MacCheckbox implements Checkbox {
    @Override
    public void check() {
        System.out.println("Mac checkbox checked");
    }
}

// Abstract Factory
interface GUIFactory {
    Button createButton();
    Checkbox createCheckbox();
}

// Concrete Factories
class WindowsFactory implements GUIFactory {
    @Override
    public Button createButton() {
        return new WindowsButton();
    }
    
    @Override
    public Checkbox createCheckbox() {
        return new WindowsCheckbox();
    }
}

class MacFactory implements GUIFactory {
    @Override
    public Button createButton() {
        return new MacButton();
    }
    
    @Override
    public Checkbox createCheckbox() {
        return new MacCheckbox();
    }
}

// Client code
class Application {
    private Button button;
    private Checkbox checkbox;
    
    public Application(GUIFactory factory) {
        button = factory.createButton();
        checkbox = factory.createCheckbox();
    }
    
    public void render() {
        button.click();
        checkbox.check();
    }
}

/**
 * ABSTRACT FACTORY CHARACTERISTICS:
 * 
 * Structure:
 * - Multiple product hierarchies (Button, Checkbox)
 * - Factory creates entire family
 * - Products work together
 * 
 * When to use:
 * ✓ Multiple related products
 * ✓ Products must work together
 * ✓ Need to switch entire family
 * 
 * Benefits:
 * ✓ Consistency (all products from same family)
 * ✓ Easy to swap families
 * ✓ Isolates concrete classes
 * 
 * Drawbacks:
 * ✗ Adding new products difficult
 * ✗ More complex than Factory Method
 * 
 * Example usage:
 * GUIFactory factory = new WindowsFactory();
 * Application app = new Application(factory);
 * app.render();
 */

// ========================================
// FACTORY METHOD VS ABSTRACT FACTORY
// ========================================
public class FactoryComparison {
    
    /**
     * DECISION GUIDE:
     * 
     * Use Factory Method when:
     * ✓ Creating single product type
     * ✓ Subclasses choose implementation
     * ✓ Simple creation logic
     * 
     * Example: Document framework
     * - Document.open() calls createDocument()
     * - Subclasses create specific doc types
     * 
     * 
     * Use Abstract Factory when:
     * ✓ Creating families of related products
     * ✓ Products must work together
     * ✓ Need to switch entire family
     * 
     * Example: UI toolkit
     * - Button, Checkbox, Menu all from same theme
     * - Windows vs Mac vs Linux
     * - All components must match
     * 
     * 
     * KEY DIFFERENCES:
     * 
     * Factory Method:
     * - One product hierarchy
     * - Uses inheritance
     * - Method in abstract class
     * 
     * Abstract Factory:
     * - Multiple product hierarchies
     * - Uses composition
     * - Interface with multiple methods
     */
    
    public static void main(String[] args) {
        // Factory Method
        VehicleFactory carFactory = new CarFactory();
        Vehicle car = carFactory.orderVehicle();
        
        // Abstract Factory
        GUIFactory windowsFactory = new WindowsFactory();
        Application app = new Application(windowsFactory);
        app.render();
    }
}

// REAL-WORLD: DATABASE CONNECTION FACTORY
// ========================================
interface DatabaseConnection {
    void connect();
    void disconnect();
    void executeQuery(String sql);
}

class MySQLConnection implements DatabaseConnection {
    @Override
    public void connect() {
        System.out.println("Connecting to MySQL...");
    }
    
    @Override
    public void disconnect() {
        System.out.println("Disconnecting from MySQL...");
    }
    
    @Override
    public void executeQuery(String sql) {
        System.out.println("Executing MySQL query: " + sql);
    }
}

class PostgreSQLConnection implements DatabaseConnection {
    @Override
    public void connect() {
        System.out.println("Connecting to PostgreSQL...");
    }
    
    @Override
    public void disconnect() {
        System.out.println("Disconnecting from PostgreSQL...");
    }
    
    @Override
    public void executeQuery(String sql) {
        System.out.println("Executing PostgreSQL query: " + sql);
    }
}

// Simple Factory (variant of Factory Method)
class DatabaseConnectionFactory {
    
    public static DatabaseConnection createConnection(String type) {
        switch (type.toLowerCase()) {
            case "mysql":
                return new MySQLConnection();
            case "postgresql":
                return new PostgreSQLConnection();
            default:
                throw new IllegalArgumentException("Unknown database type: " + type);
        }
    }
    
    /**
     * SIMPLE FACTORY:
     * 
     * Simpler variant of Factory Method
     * - Static method
     * - No subclasses
     * - switch/if logic
     * 
     * Pros:
     * ✓ Very simple
     * ✓ Centralizes creation
     * 
     * Cons:
     * ✗ Violates Open/Closed Principle
     * ✗ Adding types requires modification
     * 
     * Use for:
     * - Small, stable set of types
     * - Simplicity over extensibility
     */
}
```

## 3. Builder Pattern

### Constructing Complex Objects

```java
/**
 * BUILDER PATTERN
 * 
 * Separates construction from representation
 * Builds complex objects step by step
 */

// ========================================
// PROBLEM: TELESCOPING CONSTRUCTOR
// ========================================
class UserBad {
    private final String username;
    private final String email;
    private final String firstName;
    private final String lastName;
    private final int age;
    private final String phone;
    private final String address;
    
    // Constructor hell!
    public UserBad(String username, String email) {
        this(username, email, null, null, 0, null, null);
    }
    
    public UserBad(String username, String email, String firstName, String lastName) {
        this(username, email, firstName, lastName, 0, null, null);
    }
    
    public UserBad(String username, String email, String firstName, String lastName, int age) {
        this(username, email, firstName, lastName, age, null, null);
    }
    
    // And so on... 10+ constructors!
    public UserBad(String username, String email, String firstName, 
                   String lastName, int age, String phone, String address) {
        this.username = username;
        this.email = email;
        this.firstName = firstName;
        this.lastName = lastName;
        this.age = age;
        this.phone = phone;
        this.address = address;
    }
    
    /**
     * PROBLEMS:
     * 
     * ✗ Many constructors
     * ✗ Hard to read: new User("john", "john@email.com", null, null, 25, null, "123 Main St")
     * ✗ Parameter order confusion
     * ✗ Cannot have different constructors with same parameter types
     * ✗ Adding new field = modify all constructors
     */
}

// ========================================
// SOLUTION: BUILDER PATTERN
// ========================================
class User {
    // All fields final (immutable!)
    private final String username;
    private final String email;
    private final String firstName;
    private final String lastName;
    private final int age;
    private final String phone;
    private final String address;
    
    // Private constructor - only Builder can call
    private User(Builder builder) {
        this.username = builder.username;
        this.email = builder.email;
        this.firstName = builder.firstName;
        this.lastName = builder.lastName;
        this.age = builder.age;
        this.phone = builder.phone;
        this.address = builder.address;
        
        // Validation in one place
        if (username == null || username.isEmpty()) {
            throw new IllegalArgumentException("Username required");
        }
        if (email == null || !email.contains("@")) {
            throw new IllegalArgumentException("Valid email required");
        }
    }
    
    // Static nested Builder class
    public static class Builder {
        // Required parameters
        private final String username;
        private final String email;
        
        // Optional parameters - initialized to default values
        private String firstName = "";
        private String lastName = "";
        private int age = 0;
        private String phone = "";
        private String address = "";
        
        // Constructor for required parameters only
        public Builder(String username, String email) {
            this.username = username;
            this.email = email;
        }
        
        // Fluent setters (return Builder)
        public Builder firstName(String firstName) {
            this.firstName = firstName;
            return this;
        }
        
        public Builder lastName(String lastName) {
            this.lastName = lastName;
            return this;
        }
        
        public Builder age(int age) {
            this.age = age;
            return this;
        }
        
        public Builder phone(String phone) {
            this.phone = phone;
            return this;
        }
        
        public Builder address(String address) {
            this.address = address;
            return this;
        }
        
        // Build method - creates User
        public User build() {
            return new User(this);
        }
    }
    
    // Getters only (immutable)
    public String getUsername() { return username; }
    public String getEmail() { return email; }
    public String getFirstName() { return firstName; }
    // ... other getters
    
    /**
     * USAGE:
     * 
     * User user = new User.Builder("john", "john@email.com")
     *     .firstName("John")
     *     .lastName("Doe")
     *     .age(25)
     *     .phone("555-1234")
     *     .build();
     * 
     * 
     * BENEFITS:
     * 
     * ✓ Readable code
     * ✓ Named parameters (self-documenting)
     * ✓ Immutable objects
     * ✓ Validation in one place
     * ✓ Optional parameters easy
     * ✓ Parameter order doesn't matter
     * 
     * 
     * WHEN TO USE:
     * 
     * ✓ Class has many parameters (> 4-5)
     * ✓ Many optional parameters
     * ✓ Need immutability
     * ✓ Complex construction logic
     */
}

// ========================================
// ADVANCED: METHOD CHAINING VARIANTS

class Pizza {
    private final String size;
    private final boolean cheese;
    private final boolean pepperoni;
    private final boolean mushrooms;
    
    private Pizza(Builder builder) {
        this.size = builder.size;
        this.cheese = builder.cheese;
        this.pepperoni = builder.pepperoni;
        this.mushrooms = builder.mushrooms;
    }
    
    public static class Builder {
        // Required
        private final String size;
        
        // Optional
        private boolean cheese = false;
```

### Cloning Objects Efficiently

```java
/**
 * PROTOTYPE PATTERN
 * 
 * Creates new objects by copying existing ones
 * Avoids expensive initialization
 */

// ========================================
// SHALLOW COPY VS DEEP COPY
// ========================================
class Address {
    private String street;
    private String city;
    
    public Address(String street, String city) {
        this.street = street;
        this.city = city;
    }
    
    // Getters and setters
    public String getStreet() { return street; }
    public void setStreet(String street) { this.street = street; }
    public String getCity() { return city; }
    public void setCity(String city) { this.city = city; }
}

class PersonShallowCopy implements Cloneable {
    private String name;
    private Address address;
    
    public PersonShallowCopy(String name, Address address) {
        this.name = name;
        this.address = address;
    }
    
    @Override
    protected PersonShallowCopy clone() {
        try {
            return (PersonShallowCopy) super.clone();
        } catch (CloneNotSupportedException e) {
            throw new AssertionError(); // Can't happen
        }
    }
    
    /**
     * PROBLEM WITH SHALLOW COPY:
     * 
     * PersonShallowCopy p1 = new PersonShallowCopy("John", 
     *     new Address("123 Main", "NYC"));
     * 
     * PersonShallowCopy p2 = p1.clone();
     * 
     * p2.address.setStreet("456 Oak");
     * 
     * System.out.println(p1.address.getStreet()); // "456 Oak" - PROBLEM!
     * 
     * Both objects share same Address reference!
     * Changes to p2 affect p1!
     */
    
    public Address getAddress() { return address; }
}

class PersonDeepCopy implements Cloneable {
    private String name;
    private Address address;
    
    public PersonDeepCopy(String name, Address address) {
        this.name = name;
        this.address = address;
    }
    
    @Override
    protected PersonDeepCopy clone() {
        try {
            PersonDeepCopy cloned = (PersonDeepCopy) super.clone();
            // Deep copy the address
            cloned.address = new Address(
                this.address.getStreet(),
                this.address.getCity()
            );
            return cloned;
        } catch (CloneNotSupportedException e) {
            throw new AssertionError();
        }
    }
    
    /**
     * DEEP COPY SOLUTION:
     * 
     * PersonDeepCopy p1 = new PersonDeepCopy("John",
     *     new Address("123 Main", "NYC"));
     * 
     * PersonDeepCopy p2 = p1.clone();
     * 
     * p2.address.setStreet("456 Oak");
     * 
     * System.out.println(p1.address.getStreet()); // "123 Main" - CORRECT!
     * 
     * Each object has its own Address copy
     */
    
    public Address getAddress() { return address; }
}

// ========================================
// COPY CONSTRUCTOR (BETTER ALTERNATIVE)
// ========================================
class PersonCopyConstructor {
    private String name;
    private Address address;
    
    public PersonCopyConstructor(String name, Address address) {
        this.name = name;
        this.address = address;
    }
    
    // Copy constructor
    public PersonCopyConstructor(PersonCopyConstructor other) {
        this.name = other.name;
        // Deep copy address
        this.address = new Address(
            other.address.getStreet(),
            other.address.getCity()
        );
    }
    
    /**
     * BENEFITS OVER CLONE:
     * 
     * ✓ Explicit and clear
     * ✓ No CloneNotSupportedException
     * ✓ No Cloneable interface
     * ✓ Works with final fields
     * ✓ Type-safe (no casting)
     * 
     * USAGE:
     * 
     * PersonCopyConstructor p1 = new PersonCopyConstructor("John", address);
     * PersonCopyConstructor p2 = new PersonCopyConstructor(p1);
     * 
     * RECOMMENDED over clone()!
     */
}

// SERIALIZATION-BASED DEEP COPY
// ========================================
class SerializationCopy {
    
    public static <T> T deepCopy(T object) {
        try {
            ByteArrayOutputStream bos = new ByteArrayOutputStream();
            ObjectOutputStream oos = new ObjectOutputStream(bos);
            oos.writeObject(object);
            oos.flush();
            
            ByteArrayInputStream bis = new ByteArrayInputStream(bos.toByteArray());
            ObjectInputStream ois = new ObjectInputStream(bis);
            
            return (T) ois.readObject();
        } catch (Exception e) {
            throw new RuntimeException(e);
        }
    }
    
    /**
     * SERIALIZATION DEEP COPY:
     * 
     * Pros:
     * ✓ Automatic deep copy (no manual field copying)
     * ✓ Handles complex object graphs
     * 
     * Cons:
     * ✗ Much slower than clone()
     * ✗ Requires Serializable
     * ✗ May have serialization issues
     * 
     * Use for:
     * - Complex object graphs
     * - When performance not critical
     */
}
```

```java
/**
 * OBJECT POOL PATTERN
 * 
 * Reuses expensive objects instead of creating new ones
 * Improves performance for costly initialization
 */

import java.util.concurrent.*;
import java.util.*;

// ========================================
// BASIC OBJECT POOL
// ========================================
class BasicObjectPool<T> {
    private final Queue<T> availableObjects;
    private final int maxSize;
    private final ObjectFactory<T> factory;
    
    interface ObjectFactory<T> {
        T create();
    }
    
    public BasicObjectPool(int maxSize, ObjectFactory<T> factory) {
        this.maxSize = maxSize;
        this.factory = factory;
        this.availableObjects = new LinkedList<>();
        
        // Pre-create objects
        for (int i = 0; i < maxSize; i++) {
            availableObjects.offer(factory.create());
        }
    }
    
    public T acquire() {
        T object = availableObjects.poll();
        if (object == null) {
            throw new IllegalStateException("Pool exhausted");
        }
        return object;
    }
    
    public void release(T object) {
        availableObjects.offer(object);
    }
    
    /**
     * PROBLEMS WITH BASIC POOL:
     * 
     * ✗ Not thread-safe
     * ✗ No timeout on acquire
     * ✗ No validation
     * ✗ Fixed size (no growth)
     */
}

// ========================================
// THREAD-SAFE OBJECT POOL
// ========================================
class ThreadSafeObjectPool<T> {
    private final BlockingQueue<T> availableObjects;
    private final int maxSize;
    private final ObjectFactory<T> factory;
    private final ObjectValidator<T> validator;
    
    interface ObjectFactory<T> {
        T create();
    }
    
    interface ObjectValidator<T> {
        boolean isValid(T object);
    }
    
    public ThreadSafeObjectPool(int maxSize, ObjectFactory<T> factory, 
                                ObjectValidator<T> validator) {
        this.maxSize = maxSize;
        this.factory = factory;
        this.validator = validator;
        this.availableObjects = new LinkedBlockingQueue<>(maxSize);
        
        // Pre-create objects
        for (int i = 0; i < maxSize; i++) {
            availableObjects.offer(factory.create());
        }
    }
    
    public T acquire(long timeout, TimeUnit unit) throws InterruptedException {
        T object = availableObjects.poll(timeout, unit);
        if (object == null) {
            throw new IllegalStateException("Pool exhausted - timeout");
        }
        
        // Validate object
        if (!validator.isValid(object)) {
            // Object invalid, create new one
            object = factory.create();
        }
        
        return object;
    }
    
    public void release(T object) {
        if (object != null) {
            availableObjects.offer(object);
        }
    }
    
    /**
     * IMPROVEMENTS:
     * 
     * ✓ Thread-safe (BlockingQueue)
     * ✓ Timeout on acquire
     * ✓ Object validation
     * ✓ Handles null returns
     */
}

// REAL-WORLD: DATABASE CONNECTION POOL
// ========================================
class DatabaseConnection {
    private final String id;
    private boolean inUse;
    
    public DatabaseConnection(String id) {
        this.id = id;
        this.inUse = false;
        // Simulate expensive initialization
        System.out.println("Creating connection: " + id);
        try {
            Thread.sleep(1000); // 1 second to create
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
    
    public void executeQuery(String sql) {
        System.out.println(id + " executing: " + sql);
    }
    
    public void close() {
        System.out.println("Closing connection: " + id);
    }
    
    public boolean isValid() {
        // Check if connection still alive
        return !inUse;
    }
    
    public void markInUse() { inUse = true; }
    public void markAvailable() { inUse = false; }
}

class DatabaseConnectionPool {
    private final BlockingQueue<DatabaseConnection> pool;
    private final int maxConnections;
    
    public DatabaseConnectionPool(int maxConnections) {
        this.maxConnections = maxConnections;
        this.pool = new LinkedBlockingQueue<>(maxConnections);
        
        // Pre-create connections
        for (int i = 0; i < maxConnections; i++) {
            pool.offer(new DatabaseConnection("Connection-" + (i + 1)));
        }
        
        System.out.println("Pool initialized with " + maxConnections + " connections");
    }
    
    public DatabaseConnection getConnection() throws InterruptedException {
        DatabaseConnection conn = pool.poll(5, TimeUnit.SECONDS);
        if (conn == null) {
            throw new IllegalStateException("No connections available");
        }
        
        if (!conn.isValid()) {
            // Connection invalid, create new one
            conn = new DatabaseConnection("Connection-new");
        }
        
        conn.markInUse();
        return conn;
    }
    
    public void releaseConnection(DatabaseConnection conn) {
        if (conn != null) {
            conn.markAvailable();
            pool.offer(conn);
        }
    }
    
    public void shutdown() {
        for (DatabaseConnection conn : pool) {
            conn.close();
        }
        pool.clear();
    }
    
    /**
     * USAGE:
     * 
     * DatabaseConnectionPool pool = new DatabaseConnectionPool(5);
     * 
     * try {
     *     DatabaseConnection conn = pool.getConnection();
     *     try {
     *         conn.executeQuery("SELECT * FROM users");
     *     } finally {
     *         pool.releaseConnection(conn);
     *     }
     * } catch (InterruptedException e) {
     *     Thread.currentThread().interrupt();
     * }
     * 
     * 
     * BENEFITS:
     * 
     * Without pool:
     * - 1 second per connection creation
     * - 100 queries = 100 seconds creating connections!
     * 
     * With pool:
     * - 5 seconds upfront (5 connections)
     * - 100 queries reuse connections
     * - Queries complete in ~10 seconds
     * 
     * 10x faster!
     */
}
```

## Summary

### Quick Reference

**Creational Patterns:**

```
1. Singleton: One instance
   - Enum Singleton (best, serialization-safe)
   - Lazy Holder (best for lazy loading)
   - Avoid: DCL (complex), Synchronized (slow)

2. Factory Method vs Abstract Factory:
   - Factory Method: Single product type
   - Abstract Factory: Product families
   - Simple Factory: Static method variant

3. Builder: Complex objects
   - Solves telescoping constructor
   - Fluent API (method chaining)
   - Immutable objects

4. Prototype: Clone objects
   - Copy constructor (recommended)
   - Deep vs shallow copy
   - Avoid: Cloneable interface

5. Object Pool: Reuse expensive objects
   - Database connections
   - Thread pools
   - Use: Apache Commons Pool, HikariCP

6. Dependency Injection: Loose coupling
   - Constructor injection (best)
   - Setter injection (optional deps)
   - Avoid: Field injection
```

**When to Use:**

```
Singleton: Database manager, configuration, logger
Factory: Different implementations (DB types, file formats)
Builder: Objects with many parameters (HTTP request, User)
Prototype: Expensive object creation (game characters, documents)
Object Pool: Limited expensive resources (connections, threads)
DI: All services, repositories, controllers
```
