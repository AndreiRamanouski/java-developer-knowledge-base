# Dependency Injection Patterns

## Overview

Dependency Injection (DI) is a design pattern that implements Inversion of Control (IoC) for resolving dependencies. Instead of a class creating its own dependencies (new MyDependency()), dependencies are provided from outside (injected). This simple idea has profound implications: loose coupling, testability, flexibility, and maintainability. DI is the foundation of modern Java frameworks—Spring is essentially a DI container with extras, and understanding DI deeply means understanding how these frameworks work under the hood. There are three main injection types: Constructor Injection (dependencies passed via constructor—preferred for required dependencies), Setter Injection (dependencies set via setter methods—used for optional dependencies), and Interface Injection (dependencies provided through interface methods—rarely used in Java). Beyond manual DI, frameworks like Spring, Guice, and Dagger automate dependency resolution: Spring uses reflection and runtime wiring (flexible but slower), Guice uses runtime binding with type safety (middle ground), and Dagger uses compile-time code generation (fastest, no reflection). The Service Locator pattern looks like DI but is actually an anti-pattern—it hides dependencies, makes testing hard, and creates tight coupling to the locator. Understanding DI patterns means understanding tradeoffs: Constructor Injection is preferred (dependencies explicit, immutable, null-safe) but verbose; Setter Injection is flexible but allows invalid states; Framework DI is convenient but adds complexity and magic; Manual DI is simple but repetitive. This guide shows when to use each approach, common mistakes (circular dependencies, over-injection, framework abuse), and practical patterns for real applications.

## 1. Constructor Injection

### The Preferred Approach

```java
/**
 * CONSTRUCTOR INJECTION
 * 
 * Dependencies provided through constructor
 * Preferred for required dependencies
 */

// ========================================
// ❌ WITHOUT DEPENDENCY INJECTION
// ========================================

/**
 * Tight coupling - creates its own dependencies
 */
class BadUserService {
    private final UserRepository repository;
    private final EmailService emailService;
    private final PasswordEncoder encoder;
    
    public BadUserService() {
        this.repository = new MySQLUserRepository();  // ❌ Hard-coded!
        this.emailService = new SmtpEmailService();   // ❌ Hard-coded!
        this.encoder = new BCryptPasswordEncoder();   // ❌ Hard-coded!
    }
    
    public void createUser(String username, String password, String email) {
        String hashedPassword = encoder.encode(password);
        User user = new User(username, hashedPassword, email);
        repository.save(user);
        emailService.sendWelcome(email);
    }
    
    /**
     * PROBLEMS:
     * 
     * ✗ Tightly coupled to concrete implementations
     * ✗ Can't test without real database/email
     * ✗ Can't swap implementations (no flexibility)
     * ✗ Violates Dependency Inversion Principle
     * ✗ Hard to change configuration
     */
}

// ========================================
// ✓ WITH CONSTRUCTOR INJECTION
// ========================================

/**
 * Dependencies injected through constructor
 */
class UserService {
    private final UserRepository repository;
    private final EmailService emailService;
    private final PasswordEncoder encoder;
    
    // ✓ Dependencies provided from outside
    public UserService(UserRepository repository,
                       EmailService emailService,
                       PasswordEncoder encoder) {
        // Validation (fail fast)
        if (repository == null) {
            throw new IllegalArgumentException("Repository required");
        }
        if (emailService == null) {
            throw new IllegalArgumentException("EmailService required");
        }
        if (encoder == null) {
            throw new IllegalArgumentException("PasswordEncoder required");
        }
        
        this.repository = repository;
        this.emailService = emailService;
        this.encoder = encoder;
    }
    
    public void createUser(String username, String password, String email) {
        String hashedPassword = encoder.encode(password);
        User user = new User(username, hashedPassword, email);
        repository.save(user);
        emailService.sendWelcome(email);
    }
    
    /**
     * BENEFITS:
     * 
     * ✓ Loosely coupled
     * ✓ Easy to test (inject mocks)
     * ✓ Dependencies explicit and visible
     * ✓ Immutable (final fields)
     * ✓ Null-safe (validated in constructor)
     * ✓ Follows Dependency Inversion Principle
     */
}

// REAL-WORLD EXAMPLE: Order Service

/**
 * Complete example with constructor injection
 */
interface OrderRepository {
    void save(Order order);
    Optional<Order> findById(String id);
}

interface PaymentProcessor {
    PaymentResult process(PaymentRequest request);
}

interface InventoryService {
    void reserveItems(List<OrderItem> items);
    void releaseItems(List<OrderItem> items);
}

interface NotificationService {
    void sendOrderConfirmation(String email, Order order);
}

class OrderService {
    private final OrderRepository orderRepository;
    private final PaymentProcessor paymentProcessor;
    private final InventoryService inventoryService;
    private final NotificationService notificationService;
    
    // ✓ All dependencies injected via constructor
    public OrderService(OrderRepository orderRepository,
                        PaymentProcessor paymentProcessor,
                        InventoryService inventoryService,
                        NotificationService notificationService) {
        this.orderRepository = Objects.requireNonNull(orderRepository);
        this.paymentProcessor = Objects.requireNonNull(paymentProcessor);
        this.inventoryService = Objects.requireNonNull(inventoryService);
        this.notificationService = Objects.requireNonNull(notificationService);
    }
    
    public void placeOrder(Order order) {
        // 1. Reserve inventory
        inventoryService.reserveItems(order.getItems());
        
        try {
            // 2. Process payment
            PaymentRequest paymentRequest = new PaymentRequest(
                order.getCustomer().getPaymentMethod(),
                order.calculateTotal()
            );
            PaymentResult result = paymentProcessor.process(paymentRequest);
            
            if (!result.isSuccessful()) {
                throw new PaymentException("Payment failed: " + result.getError());
            }
            
            // 3. Save order
            order.setStatus(OrderStatus.CONFIRMED);
            orderRepository.save(order);
            
            // 4. Send confirmation
            notificationService.sendOrderConfirmation(
                order.getCustomer().getEmail(),
                order
            );
            
        } catch (Exception e) {
            // Release inventory on failure
            inventoryService.releaseItems(order.getItems());
            throw new OrderProcessingException("Failed to place order", e);
        }
    }
    
    /**
     * BENEFITS OF CONSTRUCTOR INJECTION HERE:
     * 
     * ✓ All dependencies explicit in constructor signature
     * ✓ Easy to test (mock all 4 dependencies)
     * ✓ Can't create OrderService without dependencies
     * ✓ Dependencies are final (thread-safe)
     * ✓ Clear what OrderService needs to function
     */
}
```

## 2. Setter Injection

### For Optional Dependencies

```java
/**
 * SETTER INJECTION
 * 
 * Dependencies set via setter methods
 * Used for optional dependencies
 */

// ========================================
// WHEN TO USE SETTER INJECTION
// ========================================

/**
 * Setter injection appropriate for optional dependencies
 */
class ReportGenerator {
    private final DataSource dataSource;  // Required
    private Logger logger;                 // Optional
    private CacheService cache;            // Optional
    
    // ✓ Required dependency via constructor
    public ReportGenerator(DataSource dataSource) {
        this.dataSource = Objects.requireNonNull(dataSource);
    }
    
    // ✓ Optional dependency via setter
    public void setLogger(Logger logger) {
        this.logger = logger;
    }
    
    // ✓ Optional dependency via setter
    public void setCacheService(CacheService cache) {
        this.cache = cache;
    }
    
    public String generateReport(String reportId) {
        // Check cache (optional)
        if (cache != null) {
            String cached = cache.get(reportId);
            if (cached != null) {
                log("Cache hit for report: " + reportId);
                return cached;
            }
        }
        
        // Generate report from data source
        String report = fetchAndGenerateReport(reportId);
        
        // Store in cache (optional)
        if (cache != null) {
            cache.put(reportId, report);
        }
        
        return report;
    }
    
    private void log(String message) {
        if (logger != null) {
            logger.info(message);
        }
    }
    
    private String fetchAndGenerateReport(String reportId) {
        // Use required dataSource
        return "Report data";
    }
}

/**
 * BENEFITS:
 * 
 * ✓ Optional dependencies clear
 * ✓ Can create object without optional deps
 * ✓ Can reconfigure at runtime
 * 
 * 
 * DRAWBACKS:
 * 
 * ✗ Object can be in invalid state (before setters called)
 * ✗ Not immutable (mutable state)
 * ✗ Null checks needed
 * ✗ Dependencies not visible in constructor
 */

// ========================================
// SETTER INJECTION ANTI-PATTERNS
// ========================================

/**
 * ❌ ANTI-PATTERN 1: Required dependency via setter
 */
class BadSetterInjection {
    private DatabaseService database;  // Required but via setter!
    
    public BadSetterInjection() {
        // No parameters
    }
    
    public void setDatabase(DatabaseService database) {
        this.database = database;
    }
    
    public void saveData(String data) {
        database.save(data);  // ❌ NullPointerException if setter not called!
    }
    
    /**
     * PROBLEM: Can create invalid object
     * 
     * BadSetterInjection obj = new BadSetterInjection();
     * obj.saveData("data");  // ❌ NPE! Forgot to call setter
     */
}

/**
 * ✓ BETTER: Required dependency via constructor
 */
class GoodConstructorInjection {
    private final DatabaseService database;
    
    public GoodConstructorInjection(DatabaseService database) {
        this.database = Objects.requireNonNull(database);
    }
    
    public void saveData(String data) {
        database.save(data);  // ✓ Safe - database guaranteed non-null
    }
}

/**
 * ❌ ANTI-PATTERN 2: Circular dependencies
 */
class ServiceA {
    private ServiceB serviceB;
    
    public void setServiceB(ServiceB serviceB) {
        this.serviceB = serviceB;
    }
}

class ServiceB {
    private ServiceA serviceA;
    
    public void setServiceA(ServiceA serviceA) {
        this.serviceA = serviceA;
    }
}

/**
 * Setup code:
 * ServiceA a = new ServiceA();
 * ServiceB b = new ServiceB();
 * a.setServiceB(b);  // A → B
 * b.setServiceA(a);  // B → A (circular!)
 * 
 * PROBLEM: Circular dependency
 * 
 * Solutions:
 * 1. Redesign to break cycle
 * 2. Introduce interface/event bus
 * 3. Use lazy initialization
 */

// ========================================
// REAL-WORLD EXAMPLE: Cache Configuration
// ========================================

/**
 * Cache with optional features
 */
class CacheManager {
    private final String cacheName;
    private final int maxSize;
    
    // Optional dependencies
    private CacheLoader loader;
    private CacheEvictionListener evictionListener;
    private CacheStatistics statistics;
    
    // Required via constructor
    public CacheManager(String cacheName, int maxSize) {
        this.cacheName = Objects.requireNonNull(cacheName);
        if (maxSize <= 0) {
            throw new IllegalArgumentException("Max size must be positive");
        }
        this.maxSize = maxSize;
    }
    
    // Optional via setters
    public void setLoader(CacheLoader loader) {
        this.loader = loader;
    }
    
    public void setEvictionListener(CacheEvictionListener listener) {
        this.evictionListener = listener;
    }
    
    public void setStatistics(CacheStatistics statistics) {
        this.statistics = statistics;
    }
    
    public Object get(String key) {
        Object value = getFromCache(key);
        
        if (value == null && loader != null) {
            value = loader.load(key);
            if (value != null) {
                put(key, value);
            }
        }
        
        if (statistics != null) {
            statistics.recordHit(value != null);
        }
        
        return value;
    }
    
    public void put(String key, Object value) {
        boolean evicted = putInCache(key, value);
        
        if (evicted && evictionListener != null) {
            evictionListener.onEviction(key);
        }
    }
    
    private Object getFromCache(String key) { return null; }
    private boolean putInCache(String key, Object value) { return false; }
}

// Usage
class CacheExample {
    public void setupCache() {
        // Create with required dependencies
        CacheManager cache = new CacheManager("userCache", 1000);
        
        // Optionally configure loader
        cache.setLoader(key -> loadFromDatabase(key));
        
        // Optionally enable statistics
        cache.setStatistics(new CacheStatistics());
        
        // Use cache
        Object value = cache.get("user:123");
    }
    
    private Object loadFromDatabase(String key) { return null; }
}

interface CacheLoader {
    Object load(String key);
}

interface CacheEvictionListener {
    void onEviction(String key);
}

class CacheStatistics {
    public void recordHit(boolean hit) {}
}
```

## 3. Interface Injection

### Rarely Used in Java

```java
/**
 * INTERFACE INJECTION
 * 
 * Dependencies provided through interface methods
 * Rarely used in Java (common in some other languages)
 */

// ========================================
// INTERFACE INJECTION EXAMPLE
// ========================================

/**
 * Interface defining injection method
 */
interface DatabaseInjector {
    void injectDatabase(DatabaseConnection connection);
}

/**
 * Class implements injection interface
 */
class UserService implements DatabaseInjector {
    private DatabaseConnection connection;
    
    @Override
    public void injectDatabase(DatabaseConnection connection) {
        this.connection = connection;
    }
    
    public void saveUser(User user) {
        connection.execute("INSERT INTO users ...");
    }
}

/**
 * Framework calls injection method
 */
class Framework {
    public void configure(DatabaseInjector service) {
        DatabaseConnection conn = new DatabaseConnection();
        service.injectDatabase(conn);  // Inject via interface method
    }
}

/**
 * WHY RARELY USED IN JAVA:
 * 
 * ✗ More verbose than constructor/setter injection
 * ✗ Requires interface for each injectable type
 * ✗ Not as clear as constructor injection
 * ✗ Java developers prefer constructor/setter
 * 
 * WHERE IT'S USED:
 * 
 * • Some older frameworks (rare)
 * • Plugin systems (optional)
 * • More common in languages like ActionScript, some JS frameworks
 */
```

## 4. Service Locator Anti-Pattern

### Why It's Considered an Anti-Pattern

```java
/**
 * SERVICE LOCATOR PATTERN
 * 
 * Central registry that provides dependencies
 * Looks like DI but is actually an anti-pattern
 */

// ========================================
// THE SERVICE LOCATOR PATTERN
// ========================================

/**
 * Service locator implementation
 */
class ServiceLocator {
    private static final Map<Class<?>, Object> services = new ConcurrentHashMap<>();
    
    public static <T> void registerService(Class<T> serviceClass, T implementation) {
        services.put(serviceClass, implementation);
    }
    
    @SuppressWarnings("unchecked")
    public static <T> T getService(Class<T> serviceClass) {
        T service = (T) services.get(serviceClass);
        if (service == null) {
            throw new IllegalStateException("Service not registered: " + serviceClass.getName());
        }
        return service;
    }
}

/**
 * Using service locator (looks convenient!)
 */
class BadUserService {
    
    public void createUser(String username, String password, String email) {
        // Get dependencies from locator
        UserRepository repository = ServiceLocator.getService(UserRepository.class);
        EmailService emailService = ServiceLocator.getService(EmailService.class);
        PasswordEncoder encoder = ServiceLocator.getService(PasswordEncoder.class);
        
        // Use dependencies
        String hashedPassword = encoder.encode(password);
        User user = new User(username, hashedPassword, email);
        repository.save(user);
        emailService.sendWelcome(email);
    }
    
    /**
     * ❌ PROBLEMS WITH SERVICE LOCATOR:
     * 
     * 1. HIDDEN DEPENDENCIES
     *    • Dependencies not visible in constructor
     *    • Can't see what UserService needs
     *    • Must read entire class to find dependencies
     * 
     * 2. HARD TO TEST
     *    • Must set up ServiceLocator before each test
     *    • Global state (tests affect each other)
     *    • Can't easily inject mocks
     * 
     * 3. RUNTIME ERRORS
     *    • Forget to register service? Runtime exception!
     *    • No compile-time safety
     * 
     * 4. TIGHT COUPLING TO LOCATOR
     *    • Every class coupled to ServiceLocator
     *    • Can't use class without ServiceLocator
     *    • Defeats purpose of decoupling!
     * 
     * 5. VIOLATES DEPENDENCY INVERSION PRINCIPLE
     *    • Depends on concrete ServiceLocator
     *    • Not on abstractions
     */
}

// Setup (must be done before using any service)
class ServiceLocatorSetup {
    public void setup() {
        ServiceLocator.registerService(UserRepository.class, new MySQLUserRepository());
        ServiceLocator.registerService(EmailService.class, new SmtpEmailService());
        ServiceLocator.registerService(PasswordEncoder.class, new BCryptPasswordEncoder());
        // Easy to forget one!
    }
}

// Testing nightmare
class BadUserServiceTest {
    
    @Test
    void testCreateUser() {
        // ❌ Must set up entire ServiceLocator
        ServiceLocator.registerService(UserRepository.class, mockRepository);
        ServiceLocator.registerService(EmailService.class, mockEmail);
        ServiceLocator.registerService(PasswordEncoder.class, mockEncoder);
        
        BadUserService service = new BadUserService();
        service.createUser("john", "pass", "email");
        
        // ❌ Global state - might affect other tests!
    }
}

// ========================================
// PROPER DI INSTEAD OF SERVICE LOCATOR
// ========================================

/**
 * ✓ BETTER: Constructor injection
 */
class GoodUserService {
    private final UserRepository repository;
    private final EmailService emailService;
    private final PasswordEncoder encoder;
    
    // ✓ Dependencies explicit
    public GoodUserService(UserRepository repository,
                           EmailService emailService,
                           PasswordEncoder encoder) {
        this.repository = repository;
        this.emailService = emailService;
        this.encoder = encoder;
    }
    
    public void createUser(String username, String password, String email) {
        String hashedPassword = encoder.encode(password);
        User user = new User(username, hashedPassword, email);
        repository.save(user);
        emailService.sendWelcome(email);
    }
    
    /**
     * ✓ BENEFITS OVER SERVICE LOCATOR:
     * 
     * 1. VISIBLE DEPENDENCIES
     *    • Constructor shows exactly what's needed
     *    • Clear API
     * 
     * 2. EASY TO TEST
     *    • Just pass mocks to constructor
     *    • No global state
     * 
     * 3. COMPILE-TIME SAFETY
     *    • Can't compile without dependencies
     *    • Type-safe
     * 
     * 4. NO COUPLING TO INFRASTRUCTURE
     *    • No dependency on ServiceLocator
     *    • Pure domain code
     */
}

/**
 * WHEN SERVICE LOCATOR IS (BARELY) ACCEPTABLE:
 * 
 * • Legacy code integration (temporary bridge)
 * • Plugin systems (dynamic loading)
 * • Framework code (not application code)
 * 
 * But even then, consider alternatives!
 */
```

## 5. DI Containers

### Spring, Guice, Dagger

```java
/**
 * DI CONTAINERS
 * 
 * Frameworks that automate dependency injection
 * Handle object creation and wiring
 */

// ========================================
// SPRING FRAMEWORK
// ========================================

/**
 * Spring IoC Container
 * 
 * Features:
 * • Reflection-based
 * • XML, Java config, or annotations
 * • Extensive ecosystem
 * • Runtime wiring
 */

/**
 * Spring annotation-based configuration
 */
@Component
class SpringUserRepository implements UserRepository {
    private final DataSource dataSource;
    
    @Autowired  // Spring injects DataSource
    public SpringUserRepository(DataSource dataSource) {
        this.dataSource = dataSource;
    }
    
    @Override
    public void save(User user) {
        // Save to database
    }
}

@Service
class SpringUserService {
    private final UserRepository repository;
    private final EmailService emailService;
    private final PasswordEncoder encoder;
    
    @Autowired  // Spring injects all three
    public SpringUserService(UserRepository repository,
                             EmailService emailService,
                             PasswordEncoder encoder) {
        this.repository = repository;
        this.emailService = emailService;
        this.encoder = encoder;
    }
    
    public void createUser(String username, String password, String email) {
        String hashedPassword = encoder.encode(password);
        User user = new User(username, hashedPassword, email);
        repository.save(user);
        emailService.sendWelcome(email);

/**
 * Spring Java configuration
 */
@Configuration
class AppConfig {
    
    @Bean
    public DataSource dataSource() {

        ds.setJdbcUrl("jdbc:mysql://localhost:3306/mydb");
        ds.setUsername("user");
        ds.setPassword("pass");
        return ds;
    }
    
    @Bean
    public UserRepository userRepository(DataSource dataSource) {
        return new MySQLUserRepository(dataSource);
    }
    
    @Bean
    public EmailService emailService() {

    }
    
    @Bean
    public PasswordEncoder passwordEncoder() {

    // Spring automatically wires UserService constructor

/**
 * Using Spring container
 */
class SpringApplication {
    public static void main(String[] args) {
        // Create Spring container
        ApplicationContext context = 
            new AnnotationConfigApplicationContext(AppConfig.class);
        
        // Get fully-wired service (Spring resolved all dependencies)

/**
 * SPRING BENEFITS:
 * ✓ Automatic wiring
 * ✓ Lifecycle management
 * ✓ Extensive features (transactions, security, etc.)
```

```java
// ========================================

/**
 * Application without any DI framework

class ManualDIApplication {
    
    public static void main(String[] args) {
        // 1. Create infrastructure dependencies
        DataSource dataSource = createDataSource();
        
        // 2. Create repositories
        UserRepository userRepository = new MySQLUserRepository(dataSource);
        OrderRepository orderRepository = new MySQLOrderRepository(dataSource);
        
        // 3. Create services (inner layer)
        EmailService emailService = new SmtpEmailService();
        PasswordEncoder passwordEncoder = new BCryptPasswordEncoder();
        
        // 4. Create services (outer layer)
        UserService userService = new UserService(
            userRepository,
            emailService,
            passwordEncoder
        );
        
        PaymentProcessor paymentProcessor = new StripePaymentProcessor();
        InventoryService inventoryService = new InventoryService(dataSource);
        NotificationService notificationService = new NotificationService(emailService);
        
        OrderService orderService = new OrderService(
            orderRepository,
            paymentProcessor,
            inventoryService,
            notificationService
        );
        
        // 5. Create controllers/handlers
        UserController userController = new UserController(userService);
        OrderController orderController = new OrderController(orderService);
        
        // 6. Start application
        startServer(userController, orderController);
    }
    
    private static DataSource createDataSource() {
        HikariConfig config = new HikariConfig();
        config.setJdbcUrl("jdbc:mysql://localhost:3306/mydb");

        config.setPassword("pass");
        config.setMaximumPoolSize(10);

    }
    
    private static void startServer(UserController userController, 
                                     OrderController orderController) {
        // Start HTTP server with controllers
    }

/**
 * Use factories to organize manual DI
 */
class ServiceFactory {
    private final DataSource dataSource;
    
    // Singletons
    private UserService userService;
    private OrderService orderService;
    
    public ServiceFactory(DataSource dataSource) {
        this.dataSource = dataSource;
    }
    
    public UserService getUserService() {
        if (userService == null) {
            userService = new UserService(
                getUserRepository(),
                getEmailService(),
                getPasswordEncoder()
            );
        }
        return userService;
    }
    
    public OrderService getOrderService() {
        if (orderService == null) {
            orderService = new OrderService(
                getOrderRepository(),
                getPaymentProcessor(),
                getInventoryService(),
                getNotificationService()
            );
        }
        return orderService;
    }
    
    // Helper factory methods
    private UserRepository getUserRepository() {
        return new MySQLUserRepository(dataSource);
    }
    
    private OrderRepository getOrderRepository() {
        return new MySQLOrderRepository(dataSource);
    }
    
    private EmailService getEmailService() {
        return new SmtpEmailService();
    }
    
    private PasswordEncoder getPasswordEncoder() {
        return new BCryptPasswordEncoder();
    }
    
    private PaymentProcessor getPaymentProcessor() {
        return new StripePaymentProcessor();
    }
    
    private InventoryService getInventoryService() {
        return new InventoryService(dataSource);
    }
    
    private NotificationService getNotificationService() {
        return new NotificationService(getEmailService());
    }
}

/**
 * Using factory
 */
class FactoryBasedApplication {
    public static void main(String[] args) {
        DataSource dataSource = createDataSource();
        ServiceFactory factory = new ServiceFactory(dataSource);
        
        // Get services from factory
        UserService userService = factory.getUserService();
        OrderService orderService = factory.getOrderService();
        
        // Use services
        userService.createUser("john", "pass", "john@example.com");

    private static DataSource createDataSource() {
        return new HikariDataSource();
    }
}
```

```java
// ========================================
// INJECTION TYPES

/**
 * 1. CONSTRUCTOR INJECTION (PREFERRED)
 * 
 * ✓ Use for: Required dependencies
 * ✓ Benefits: Immutable, explicit, null-safe
 * ✓ Best practice for most cases
 */
class ConstructorInjectionExample {
    private final ServiceA serviceA;
    
    public ConstructorInjectionExample(ServiceA serviceA) {
        this.serviceA = Objects.requireNonNull(serviceA);
    }
}

/**
 * 2. SETTER INJECTION
 * 
 * ✓ Use for: Optional dependencies
 * ✗ Drawback: Mutable, can be invalid state
 * ✓ OK for: Optional configuration
 */
class SetterInjectionExample {
    private Logger logger;  // Optional
    
    public void setLogger(Logger logger) {
        this.logger = logger;
    }
}
```

**Key Takeaways:**

```
Prefer: Constructor injection (explicit, immutable, safe)
Avoid: Service Locator (hidden dependencies, hard to test)
Choose: Manual DI for small apps, frameworks for large apps

Spring: Enterprise, feature-rich
Guice: Lightweight, simple
Dagger: Android, fastest

Always: Depend on interfaces, validate dependencies, use final
```
