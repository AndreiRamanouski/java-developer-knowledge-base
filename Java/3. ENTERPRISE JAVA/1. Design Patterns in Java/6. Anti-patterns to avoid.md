# Design Anti-Patterns to Avoid - Guide

## The Grand Finale: Anti-Patterns Every Java Developer Must Know

Anti-patterns are common solutions that initially seem reasonable but lead to poor design, unmaintainable code, and technical debt. Unlike design patterns which are proven solutions, anti-patterns are proven problems—recurring mistakes that developers make despite good intentions. This final guide covers the seven most damaging anti-patterns that will haunt your codebase if left unchecked. Understanding what NOT to do is as important as knowing what TO do!

**Why This Matters**:

- Technical debt compounds exponentially
- Anti-patterns are easier to prevent than fix
- Recognition is the first step to improvement

## 1. God Object Anti-Pattern

### "The Class That Does Everything"

```java
/**
 * GOD OBJECT: One class with too many responsibilities
 * 
 * WARNING SIGNS:
 * • Class > 500 lines
 * • Class name: Manager, Controller, Handler, Util
 * • Many unrelated imports
 * • Changes break unrelated functionality
 */

// ========================================
// ❌ THE ANTI-PATTERN
// ========================================
class UserManager {
    // Database
    private Connection connection;
    
    // Email
    private String smtpServer;
    
    // Validation
    private Pattern emailPattern;
    
    // Caching
    private Map<String, User> cache;
    
    // Logging
    private Logger logger;
    
    public void createUser(String username, String password, String email) {
        // 1. Validate
        if (!emailPattern.matcher(email).matches()) {
            logger.severe("Invalid email");
            throw new IllegalArgumentException();
        }
        
        // 2. Hash password
        String hash = BCrypt.hashpw(password, BCrypt.gensalt());
        
        // 3. Save to database
        try (PreparedStatement stmt = connection.prepareStatement(
                "INSERT INTO users VALUES (?, ?, ?)")) {
            stmt.setString(1, username);
            stmt.setString(2, hash);
            stmt.setString(3, email);
            stmt.executeUpdate();
        } catch (SQLException e) {
            logger.severe("DB error: " + e);
            throw new RuntimeException(e);
        }
        
        // 4. Add to cache
        cache.put(username, new User(username, email));
        
        // 5. Send email
        sendEmail(email, "Welcome!", "Thanks for joining!");
        
        // 6. Log
        logger.info("User created: " + username);
    }
    
    // + 50 more methods doing everything!
    private void sendEmail(String to, String subject, String body) { }
    private void updateDatabase() { }
    private void clearCache() { }
    // ... database methods
    // ... email methods
    // ... validation methods
    // ... logging methods
    // ... caching methods
}

/**
 * PROBLEMS:
 * ✗ 500+ lines of code
 * ✗ Multiple responsibilities (DB, email, validation, cache, logging)
 * ✗ Hard to test (everything coupled)
 * ✗ Hard to maintain (changes ripple everywhere)
 * ✗ Hard to reuse (can't use just email logic)
 * ✗ Violates Single Responsibility Principle
 */

// ========================================
// ✓ THE SOLUTION: Single Responsibility
// ========================================

// 1. User Entity (data only)
record User(String username, String email, String passwordHash) { }

// 2. UserRepository (database only)
class UserRepository {
    private final DataSource dataSource;
    
    public void save(User user) {
        try (Connection conn = dataSource.getConnection();
             PreparedStatement stmt = conn.prepareStatement(
                 "INSERT INTO users VALUES (?, ?, ?)")) {
            stmt.setString(1, user.username());
            stmt.setString(2, user.passwordHash());
            stmt.setString(3, user.email());
            stmt.executeUpdate();
        } catch (SQLException e) {
            throw new DataAccessException(e);
        }
    }
    
    public Optional<User> findByUsername(String username) {
        // Query logic
        return Optional.empty();
    }
}

// 3. UserValidator (validation only)
class UserValidator {
    private static final Pattern EMAIL = 
        Pattern.compile("^[A-Za-z0-9+_.-]+@[A-Za-z0-9.-]+$");
    
    public void validate(String username, String password, String email) {
        if (username == null || username.isEmpty()) {
            throw new ValidationException("Username required");
        }
        if (password.length() < 8) {
            throw new ValidationException("Password too short");
        }
        if (!EMAIL.matcher(email).matches()) {
            throw new ValidationException("Invalid email");
        }
    }
}

// 4. PasswordService (password operations only)
class PasswordService {
    public String hash(String password) {
        return BCrypt.hashpw(password, BCrypt.gensalt());
    }
    
    public boolean verify(String password, String hash) {
        return BCrypt.checkpw(password, hash);
    }
}

// 5. EmailService (email only)
class EmailService {
    private final JavaMailSender mailSender;
    
    public void sendWelcome(String to) {
        SimpleMailMessage msg = new SimpleMailMessage();
        msg.setTo(to);
        msg.setSubject("Welcome!");
        msg.setText("Thanks for joining!");
        mailSender.send(msg);
    }
}

// 6. UserCache (caching only)
class UserCache {
    private final Cache<String, User> cache = CacheBuilder.newBuilder()
        .expireAfterWrite(10, TimeUnit.MINUTES)
        .build();
    
    public Optional<User> get(String username) {
        return Optional.ofNullable(cache.getIfPresent(username));
    }
    
    public void put(String username, User user) {
        cache.put(username, user);
    }
}

// 7. UserService (orchestration only - still focused!)
class UserService {
    private final UserRepository repository;
    private final UserValidator validator;
    private final PasswordService passwordService;
    private final EmailService emailService;
    private final UserCache cache;
    
    public void createUser(String username, String password, String email) {
        validator.validate(username, password, email);
        String hash = passwordService.hash(password);
        User user = new User(username, email, hash);
        repository.save(user);
        cache.put(username, user);
        emailService.sendWelcome(email);
    }
}

/**
 * BENEFITS:
 * ✓ Each class: 50-100 lines (was 500+)
 * ✓ Single responsibility
 * ✓ Easy to test (mock dependencies)
 * ✓ Easy to maintain (isolated changes)
 * ✓ Easy to reuse (use EmailService anywhere)
 * ✓ Clear purpose
 */
```

## 2. Spaghetti Code Anti-Pattern

### "Tangled Control Flow You Can't Follow"

```java
/**
 * SPAGHETTI CODE: Control flow is impossible to follow
 * 
 * WARNING SIGNS:
 * • Deep nesting (> 3 levels)
 * • Many boolean flags
 * • Goto-like constructs
 * • Can't explain in 30 seconds
 * • Test coverage < 50%
 */

// ========================================
// ❌ THE ANTI-PATTERN
// ========================================
class OrderProcessor {
    public void processOrder(Order order) {
        boolean valid = false;
        boolean inventoryChecked = false;
        boolean paymentProcessed = false;
        String error = null;
        
        // Nested validation hell
        if (order != null) {
            if (order.getItems() != null) {
                if (!order.getItems().isEmpty()) {
                    if (order.getCustomer() != null) {
                        if (order.getCustomer().getEmail() != null) {
                            valid = true;
                        } else {
                            error = "No email";
                            goto_error();
                        }
                    } else {
                        error = "No customer";
                        goto_error();
                    }
                } else {
                    error = "No items";
                    goto_error();
                }
            } else {
                error = "Items null";
                goto_error();
            }
        } else {
            error = "Order null";
            goto_error();
        }
        
        if (valid) {
            // Check inventory
            for (OrderItem item : order.getItems()) {
                int available = checkInventory(item.getProductId());
                if (available < item.getQuantity()) {
                    error = "Insufficient inventory";
                    goto_error();
                }
                inventoryChecked = true;
            }
        }
        
        if (inventoryChecked) {
            // Process payment
            try {
                processPayment(order);
                paymentProcessed = true;
            } catch (PaymentException e) {
                error = "Payment failed";
                // Should we restore inventory? Maybe? Who knows!
                if (inventoryChecked) {
                    restoreInventory(order); // Maybe?
                }
                goto_error();
            }
        }
        
        if (paymentProcessed) {
            // Deduct inventory
            try {
                deductInventory(order);
            } catch (Exception e) {
                // Payment processed, inventory failed!
                // Should we refund? Maybe?
                if (paymentProcessed) {
                    refundPayment(order); // Maybe?
                }
                goto_error();
            }
        }
        
        // More tangled logic...
        
        goto_error: {
            if (error != null) {
                System.err.println("Failed: " + error);
                // What state are we in? No idea!
            }
        }
    }
    
    /**
     * PROBLEMS:
     * ✗ Arrow anti-pattern (deep nesting)
     * ✗ Boolean flags everywhere
     * ✗ Unclear state (what executed before error?)
     * ✗ Inconsistent error handling
     * ✗ Can't add transaction rollback
     * ✗ Impossible to test (too many paths)
     */
    
    private int checkInventory(String id) { return 0; }
    private void processPayment(Order order) { }
    private void deductInventory(Order order) { }
    private void restoreInventory(Order order) { }
    private void refundPayment(Order order) { }
}

// ========================================
// ✓ THE SOLUTION: Structured Flow
// ========================================
class RefactoredOrderProcessor {
    private final InventoryService inventory;
    private final PaymentService payment;
    private final ShippingService shipping;
    
    public void processOrder(Order order) {
        // 1. Validate (fail fast)
        validateOrder(order);
        
        // 2. Reserve inventory (reversible)
        ReservationId reservation = inventory.reserve(order.getItems());
        
        try {
            // 3. Process payment (reversible)
            PaymentId paymentId = payment.process(
                order.getCustomer(),
                order.calculateTotal()
            );
            
            try {
                // 4. Confirm inventory (irreversible)
                inventory.confirm(reservation);
                
                // 5. Arrange shipping
                shipping.arrange(order);
                
            } catch (Exception e) {
                // Payment succeeded, inventory failed → refund
                payment.refund(paymentId);
                throw new OrderProcessingException("Inventory failed", e);
            }
            
        } catch (PaymentException e) {
            // Payment failed → release inventory
            inventory.release(reservation);
            throw new OrderProcessingException("Payment failed", e);
        }
    }
    
    private void validateOrder(Order order) {
        if (order == null) {
            throw new ValidationException("Order is null");
        }
        if (order.getItems() == null || order.getItems().isEmpty()) {
            throw new ValidationException("No items");
        }
        if (order.getCustomer() == null) {
            throw new ValidationException("No customer");
        }
        if (order.getCustomer().getEmail() == null) {
            throw new ValidationException("No email");
        }
    }
    
    /**
     * BENEFITS:
     * ✓ Linear flow (easy to follow)
     * ✓ Clear transaction boundaries
     * ✓ Proper rollback on errors
     * ✓ Early validation (fail fast)
     * ✓ Testable
     * ✓ Self-documenting
     */
}
```

## 3. Lava Flow Anti-Pattern

### "Dead Code Nobody Dares Touch"

```java
/**
 * LAVA FLOW: Code that's commented out or unused but kept "just in case"
 * 
 * WARNING SIGNS:
 * • Large blocks of commented code
 * • "// TODO: Remove this after testing"
 * • Methods never called
 * • Dead branches (if (false))
 * • Fear to delete anything
 */

// ========================================
// ❌ THE ANTI-PATTERN
// ========================================
class ReportGenerator {
    
    public String generateReport(List<Data> data) {
        // Old implementation (don't delete, we might need it!)
        /*
        StringBuilder result = new StringBuilder();
        for (Data d : data) {
            result.append(d.toString());
            result.append("\n");
        }
        return result.toString();
        */
        
        // New implementation (but keep old one just in case)
        /*
        return data.stream()
            .map(Data::toString)
            .collect(Collectors.joining("\n"));
        */
        
        // Actually used implementation
        return data.stream()
            .map(d -> formatData(d))
            .collect(Collectors.joining("\n", "Report:\n", "\nEnd"));
        
        // Old formatting (might need this later!)
        /*
        private String oldFormat(Data d) {
            return String.format("%s: %s", d.getId(), d.getValue());
        }
        */
    }
    
    // Unused method (was for debugging)
    /*
    private void debugPrint(List<Data> data) {
        for (Data d : data) {
            System.out.println("DEBUG: " + d);
        }
    }
    */
    
    // Experimental feature (never worked, but don't delete!)
    /*
    private String generatePdfReport(List<Data> data) {
        // This was supposed to generate PDF
        // But we switched to HTML
        // Keep it here in case we need PDF again
        return "";
    }
    */
    
    private String formatData(Data d) {
        // Legacy format (don't delete!)
        // return d.getId() + ": " + d.getValue();
        
        // New format
        return String.format("[%s] %s", d.getId(), d.getValue());
    }
    
    /**
     * PROBLEMS:
     * ✗ Hard to understand what's actually used
     * ✗ Commented code confuses developers
     * ✗ Increases maintenance burden
     * ✗ Takes up space, clutters codebase
     * ✗ Nobody dares delete (fear of breaking something)
     * ✗ "We might need it later" → you won't
     */
}

// ========================================
// ✓ THE SOLUTION: Delete Fearlessly
// ========================================
class CleanReportGenerator {
    
    public String generateReport(List<Data> data) {
        return data.stream()
            .map(this::formatData)
            .collect(Collectors.joining("\n", "Report:\n", "\nEnd"));
    }
    
    private String formatData(Data d) {
        return String.format("[%s] %s", d.getId(), d.getValue());
    }
    
    /**
     * BENEFITS:
     * ✓ Clean, readable code
     * ✓ Only active code present
     * ✓ Version control has history (can recover if needed)
     * ✓ Easier to maintain
     * 
     * 
     * VERSION CONTROL IS YOUR SAFETY NET:
     * 
     * • Git remembers everything
     * • You can recover deleted code if needed
     * • Don't use comments as version control
     * • If you're not using it now, delete it
     * 
     * 
     * HOW TO FIX LAVA FLOW:
     * 
     * 1. Use coverage tools to find unused code
     * 2. Delete commented code (git has history)
     * 3. Delete unused methods (IDE can detect)
     * 4. Delete dead branches (if (false))
     * 5. Trust your version control
     */
}

// Tool to detect dead code
/**
 * Use IDE features:
 * - IntelliJ: Analyze → Run Inspection by Name → "Unused declaration"
 * - Eclipse: Search → Java → "Unused code"
 * 
 * Use build tools:
 * - JaCoCo for test coverage
 * - SonarQube for code quality
 * - SpotBugs for bug detection
 */
```

## 4. Golden Hammer Anti-Pattern

### "If All You Have Is a Hammer..."

```java
/**
 * GOLDEN HAMMER: Using one solution for everything
 * 
 * WARNING SIGNS:
 * • "We always use [technology X]"
 * • Forcing solutions into problems
 * • Ignoring better alternatives
 * • "We know this tool, so we'll use it"
 */

// ========================================
// ❌ THE ANTI-PATTERN
// ========================================

/**
 * Example 1: Using XML for everything
 */
class XmlHammer {
    // ❌ Using XML for configuration (verbose!)
    /*
    <configuration>
        <database>
            <host>localhost</host>
            <port>5432</port>
            <name>mydb</name>
        </database>
    </configuration>
    */
    
    // ❌ Using XML for data transfer (heavyweight!)
    public String getUserData(String userId) {
        return "<user><id>" + userId + "</id><name>John</name></user>";
    }
    
    // ❌ Using XML for caching (inefficient!)
    private Map<String, String> cache = new HashMap<>();
    public void cacheUser(User user) {
        String xml = "<user><id>" + user.getId() + "</id></user>";
        cache.put(user.getId(), xml);
    }
    
    /**
     * PROBLEMS:
     * ✗ XML is verbose (100 bytes vs 20 bytes JSON)
     * ✗ Slow to parse
     * ✗ Not human-readable for configs
     * ✗ Using same tool for different problems
     */
}

/**
 * Example 2: Using inheritance for everything
 */
class InheritanceHammer {
    
    // ❌ Using inheritance for code reuse
    class UserService extends DatabaseHelper {
        // Inherits database methods (tight coupling!)
    }
    
    // ❌ Using inheritance for behavior
    class LoggingUserService extends UserService {
        // Adds logging by extending (can't compose)
    }
    
    // ❌ Using inheritance for configuration
    class ProductionUserService extends LoggingUserService {
        // Production config by extending (hierarchy explosion!)
    }
    
    /**
     * PROBLEMS:
     * ✗ Tight coupling
     * ✗ Fragile base class problem
     * ✗ Can't compose behaviors
     * ✗ Deep inheritance hierarchies
     * 
     * SHOULD USE: Composition, Dependency Injection
     */
}

/**
 * Example 3: Using microservices for everything
 */
class MicroservicesHammer {
    
    /**
     * ❌ PROBLEM:
     * 
     * • 3-person team
     * • 20 microservices
     * • Each service: deployment, monitoring, logging
     * • Network latency between services
     * • Distributed debugging nightmare
     * 
     * SHOULD USE: Modular monolith first, split when needed
     */
}

// ========================================
// ✓ THE SOLUTION: Right Tool for the Job
// ========================================

/**
 * ✓ Use appropriate formats
 */
class RightToolForJob {
    
    // Configuration → YAML (readable)
    /*
    database:
      host: localhost
      port: 5432
      name: mydb
    */
    
    // Data transfer → JSON (lightweight)
    public String getUserData(String userId) {
        return "{\"id\":\"" + userId + "\",\"name\":\"John\"}";
    }
    
    // Caching → In-memory objects (fast)
    private Map<String, User> cache = new HashMap<>();
    public void cacheUser(User user) {
        cache.put(user.getId(), user);  // Object, not XML!
    }
}

/**
 * ✓ Use composition over inheritance
 */
class CompositionApproach {
    class UserService {
        private final UserRepository repository;  // Composition
        private final Logger logger;              // Composition
        private final Config config;              // Composition
        
        public UserService(UserRepository repository, 
                          Logger logger, 
                          Config config) {
            this.repository = repository;
            this.logger = logger;
            this.config = config;
        }
    }
    
    /**
     * BENEFITS:
     * ✓ Loose coupling
     * ✓ Easy to test (mock dependencies)
     * ✓ Flexible composition
     * ✓ No inheritance hierarchy
     */
}

/**
 * DECISION MATRIX:
 * 
 * Configuration:
 * - Simple? → .properties
 * - Hierarchical? → YAML
 * - Complex? → Java Config
 * 
 * Data Transfer:
 * - Web API? → JSON
 * - Large data? → Protocol Buffers
 * - Legacy? → XML (only if required)
 * 
 * Storage:
 * - Structured data? → SQL
 * - Documents? → MongoDB
 * - Key-value? → Redis
 * - Search? → Elasticsearch
 * 
 * Architecture:
 * - Small team? → Modular monolith
 * - Independent scaling? → Microservices
 * - Real-time? → Event-driven
 */
```

## 5. Circular Dependencies Anti-Pattern

### "A Depends on B Depends on A"

```java
/**
 * CIRCULAR DEPENDENCIES: Classes depend on each other
 * 
 * WARNING SIGNS:
 * • A imports B, B imports A
 * • Can't compile classes independently
 * • Can't test classes independently
 * • Initialization order issues
 */

// ========================================
// ❌ THE ANTI-PATTERN
// ========================================

class OrderService {
    private CustomerService customerService;
    
    public void setCustomerService(CustomerService service) {
        this.customerService = service;
    }
    
    public void placeOrder(String customerId, List<Item> items) {
        Customer customer = customerService.getCustomer(customerId);
        // Process order...
        // Update customer's order history
        customerService.addOrderToHistory(customerId, orderId);
    }
}

class CustomerService {
    private OrderService orderService;  // ❌ Circular!
    
    public void setOrderService(OrderService service) {
        this.orderService = service;
    }
    
    public Customer getCustomer(String customerId) {
        Customer customer = loadFromDatabase(customerId);
        // Load customer's orders
        List<Order> orders = orderService.getOrdersForCustomer(customerId);
        customer.setOrders(orders);
        return customer;
    }
    
    public void addOrderToHistory(String customerId, String orderId) {
        // Update customer record
    }
    
    private Customer loadFromDatabase(String customerId) {
        return new Customer();
    }
}

/**
 * PROBLEMS:
 * ✗ OrderService → CustomerService → OrderService
 * ✗ Can't instantiate one without the other
 * ✗ Can't test independently
 * ✗ Tight coupling
 * ✗ Initialization order issues
 * ✗ Hard to understand flow
 */

// ========================================
// ✓ SOLUTION 1: Extract Common Interface
// ========================================

// Extract shared responsibility to interface
interface OrderRepository {
    List<Order> getOrdersForCustomer(String customerId);
}

class RefactoredOrderService implements OrderRepository {
    private final CustomerRepository customerRepository;  // Interface, not concrete!
    
    public RefactoredOrderService(CustomerRepository customerRepository) {
        this.customerRepository = customerRepository;
    }
    
    @Override
    public List<Order> getOrdersForCustomer(String customerId) {
        // Return orders
        return List.of();
    }
    
    public void placeOrder(String customerId, List<Item> items) {
        Customer customer = customerRepository.getCustomer(customerId);
        // Process order...
    }
}

interface CustomerRepository {
    Customer getCustomer(String customerId);
}

class RefactoredCustomerService implements CustomerRepository {
    private final OrderRepository orderRepository;  // Interface!
    
    public RefactoredCustomerService(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }
    
    @Override
    public Customer getCustomer(String customerId) {
        Customer customer = loadFromDatabase(customerId);
        List<Order> orders = orderRepository.getOrdersForCustomer(customerId);
        customer.setOrders(orders);
        return customer;
    }
    
    private Customer loadFromDatabase(String customerId) {
        return new Customer();
    }
}

/**
 * Now: OrderService → CustomerRepository ← CustomerService
 *      CustomerService → OrderRepository ← OrderService
 * 
 * Both depend on interfaces, not each other!
 */

// ========================================
// ✓ SOLUTION 2: Extract Third Class
// ========================================

class OrderCustomerFacade {
    private final OrderService orderService;
    private final CustomerService customerService;
    
    public OrderCustomerFacade(OrderService orderService, 
                               CustomerService customerService) {
        this.orderService = orderService;
        this.customerService = customerService;
    }
    
    public void placeOrder(String customerId, List<Item> items) {
        Customer customer = customerService.getCustomer(customerId);
        Order order = orderService.createOrder(customer, items);
        customerService.addToOrderHistory(customerId, order.getId());
    }
}

/**
 * Now: OrderService ← Facade → CustomerService
 * 
 * No circular dependency!
 */

// ========================================
// ✓ SOLUTION 3: Event-Driven (Decoupling)
// ========================================

class EventDrivenOrderService {
    private final ApplicationEventPublisher eventPublisher;
    private final CustomerRepository customerRepository;
    
    public void placeOrder(String customerId, List<Item> items) {
        Customer customer = customerRepository.getCustomer(customerId);
        Order order = createOrder(customer, items);
        
        // Publish event instead of calling CustomerService directly
        eventPublisher.publishEvent(new OrderPlacedEvent(orderId, customerId));
    }
    
    private Order createOrder(Customer customer, List<Item> items) {
        return new Order();
    }
}

class EventDrivenCustomerService {
    
    @EventListener
    public void onOrderPlaced(OrderPlacedEvent event) {
        // Update customer's order history
        addToOrderHistory(event.getCustomerId(), event.getOrderId());
    }
    
    private void addToOrderHistory(String customerId, String orderId) {
        // Update customer
    }
}

/**
 * BENEFITS:
 * ✓ No circular dependency
 * ✓ Loose coupling
 * ✓ Easy to test
 * ✓ Easy to add new listeners
 */
```

## 6. Premature Optimization Anti-Pattern

### "Optimizing Before Measuring"

```java
/**
 * PREMATURE OPTIMIZATION: Optimizing without profiling first
 * 
 * "Premature optimization is the root of all evil" - Donald Knuth
 * 
 * WARNING SIGNS:
 * • Complex code for "performance"
 * • No benchmarks to prove it's faster
 * • Optimizing non-bottlenecks
 * • Sacrificing readability for "speed"
 */

// ========================================
// ❌ THE ANTI-PATTERN
// ========================================

class PrematurelyOptimized {
    
    // ❌ Using StringBuilder for single concatenation
    public String greet(String name) {
        StringBuilder sb = new StringBuilder();  // Overkill!
        sb.append("Hello, ");
        sb.append(name);
        return sb.toString();
        
        // Simple version: return "Hello, " + name;
        // Compiler optimizes this anyway!
    }
    
    // ❌ Object pooling for lightweight objects
    private final Queue<User> userPool = new ConcurrentLinkedQueue<>();
    
    public User createUser(String name) {
        User user = userPool.poll();
        if (user == null) {
            user = new User();  // Create if pool empty
        }
        user.setName(name);
        return user;
    }
    
    public void releaseUser(User user) {
        user.setName(null);
        userPool.offer(user);  // Return to pool
    }
    
    /**
     * PROBLEM:
     * • Modern JVMs are excellent at garbage collection
     * • Object allocation is cheap (nanoseconds)
     * • Pooling adds complexity
     * • No benchmark proving it's faster
     * • Likely slower than just creating new objects!
     */
    
    // ❌ Custom cache for rarely-accessed data
    private final Map<String, Customer> customerCache = 
        new ConcurrentHashMap<>();
    private final ScheduledExecutorService cacheEviction = 
        Executors.newScheduledThreadPool(1);
    
    public Customer getCustomer(String id) {
        Customer customer = customerCache.get(id);
        if (customer == null) {
            customer = database.loadCustomer(id);
            customerCache.put(id, customer);
            
            // Schedule eviction after 5 minutes
            cacheEviction.schedule(() -> {
                customerCache.remove(id);
            }, 5, TimeUnit.MINUTES);
        }
        return customer;
    }
    
    /**
     * PROBLEM:
     * • Complex caching logic
     * • Thread pool overhead
     * • Memory management complexity
     * • No proof customers are accessed frequently
     * • Could just use Guava Cache or Spring Cache
     */
    
    // ❌ Bit manipulation for boolean flags
    private int flags = 0;  // Store multiple booleans in bits
    
    private static final int FLAG_ACTIVE = 1;      // 0001
    private static final int FLAG_PREMIUM = 2;     // 0010
    private static final int FLAG_VERIFIED = 4;    // 0100
    
    public void setActive(boolean active) {
        if (active) {
            flags |= FLAG_ACTIVE;  // Set bit
        } else {
            flags &= ~FLAG_ACTIVE;  // Clear bit
        }
    }
    
    public boolean isActive() {
        return (flags & FLAG_ACTIVE) != 0;
    }
    
    /**
     * PROBLEM:
     * • Hard to read
     * • Saves 3 bytes (boolean → bit)
     * • Modern JVMs optimize boolean storage anyway
     * • Not worth the complexity
     */
}
```

## Summary: The Seven Deadly Anti-Patterns

```java
/**
 * ANTI-PATTERNS QUICK REFERENCE
 */

// 1. GOD OBJECT
class Bad { /* 500+ lines, does everything */ }
class Good { /* 50-100 lines, single responsibility */ }

// 2. SPAGHETTI CODE
// Bad: if (a) { if (b) { if (c) { if (d) { ... }}}}
// Good: validate(); reserve(); process(); confirm();

// 3. LAVA FLOW
// Bad: /* old code */ /* more old code */ actual code /* TODO remove */
// Good: Clean code only (git has history)

// 4. GOLDEN HAMMER
// Bad: Use XML for everything
// Good: YAML for config, JSON for API, SQL for data

// 5. CIRCULAR DEPENDENCIES
// Bad: A → B → A
// Good: A → Interface ← B

// 6. PREMATURE OPTIMIZATION
// Bad: Optimize before profiling
// Good: Measure, identify bottleneck, optimize

// 7. NOT INVENTED HERE
// Bad: Build custom JSON parser
// Good: Use Jackson/Gson
```

### Detection Checklist

```
□ God Object:
  □ Class > 500 lines?
  □ Multiple responsibilities?
  □ Name ends in Manager/Handler/Util?

□ Spaghetti Code:
  □ Nesting > 3 levels?
  □ Many boolean flags?
  □ Can't explain in 30 seconds?

□ Lava Flow:
  □ Commented code blocks?
  □ Unused methods?
  □ Dead branches?

□ Golden Hammer:
  □ Using same tool for everything?
  □ Ignoring better alternatives?
  □ "We always use [X]"?

□ Circular Dependencies:
  □ A imports B, B imports A?
  □ Can't test independently?
  □ Initialization issues?

□ Premature Optimization:
  □ Complex code for "performance"?
  □ No benchmarks?
  □ Optimizing before profiling?

□ Not Invented Here:
  □ Reimplementing standard functionality?
  □ "We can build it better"?
  □ Wasting time on infrastructure?
```
