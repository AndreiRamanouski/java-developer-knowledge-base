# Behavioral Design Patterns

## Overview

Behavioral patterns solve the fundamental problem of object interaction and responsibility assignment—how objects communicate, delegate responsibilities, and implement algorithms in flexible, maintainable ways. Instead of rigid control flow and tight coupling (objects know too much about each other, algorithms hardcoded, inflexible communication), behavioral patterns provide flexible interaction mechanisms: Strategy encapsulates algorithms (sort strategies, payment methods), Observer enables one-to-many notifications (event listeners, MVC), Command encapsulates requests as objects (undo/redo, transaction systems), Template Method defines algorithm skeleton (subclasses override steps), Chain of Responsibility passes requests along chain (servlet filters, logging), State encapsulates state-dependent behavior (TCP connection states, order workflows), and Iterator provides sequential access without exposing structure (collections, streams). These patterns are fundamental to Java—collections use Iterator for traversal, Swing uses Observer for event handling, comparators use Strategy for sorting, servlet filters use Chain of Responsibility, and template methods appear throughout frameworks. Understanding behavioral patterns means understanding how to build flexible, decoupled systems where objects collaborate without tight coupling, algorithms are interchangeable, and responsibilities are clearly assigned. This guide covers Strategy (functional interfaces, Java 8+), Observer (listeners, PropertyChangeSupport, reactive extensions), Command (undo/redo, queuing, macro commands), Template Method (framework hooks, abstract classes), Chain of Responsibility (request handling, filter chains), State (state machines, transitions), and Iterator (internal implementation, fail-fast, custom iterators).

## 1. Strategy Pattern

### Encapsulating Algorithms

```java
/**
 * STRATEGY PATTERN
 * 
 * Defines family of algorithms
 * Encapsulates each one
 * Makes them interchangeable
 */

// ========================================
// PROBLEM: HARDCODED ALGORITHMS
// ========================================
class PaymentProcessorBad {
    public void processPayment(String type, double amount) {
        if (type.equals("CREDIT_CARD")) {
            // Credit card logic
            System.out.println("Processing credit card: $" + amount);
        } else if (type.equals("PAYPAL")) {
            // PayPal logic
            System.out.println("Processing PayPal: $" + amount);
        } else if (type.equals("BITCOIN")) {
            // Bitcoin logic
            System.out.println("Processing Bitcoin: $" + amount);
        }
        
        /**
         * PROBLEMS:
         * ✗ Adding payment method = modify class (violates Open/Closed)
         * ✗ Cannot reuse payment logic
         * ✗ Hard to test individual methods
         * ✗ Growing if-else chain
         */
    }
}

// ========================================
// SOLUTION: STRATEGY PATTERN
// ========================================

// Strategy interface
interface PaymentStrategy {
    void pay(double amount);
}

// Concrete strategies
class CreditCardStrategy implements PaymentStrategy {
    private String cardNumber;
    private String cvv;
    
    public CreditCardStrategy(String cardNumber, String cvv) {
        this.cardNumber = cardNumber;
        this.cvv = cvv;
    }
    
    @Override
    public void pay(double amount) {
        System.out.println("Paid $" + amount + " with credit card " + 
            cardNumber.substring(cardNumber.length() - 4));
    }
}

class PayPalStrategy implements PaymentStrategy {
    private String email;
    
    public PayPalStrategy(String email) {
        this.email = email;
    }
    
    @Override
    public void pay(double amount) {
        System.out.println("Paid $" + amount + " with PayPal account " + email);
    }
}

class BitcoinStrategy implements PaymentStrategy {
    private String walletAddress;
    
    public BitcoinStrategy(String walletAddress) {
        this.walletAddress = walletAddress;
    }
    
    @Override
    public void pay(double amount) {
        System.out.println("Paid $" + amount + " with Bitcoin to " + walletAddress);
    }
}

// Context
class ShoppingCart {
    private List<Item> items = new ArrayList<>();
    private PaymentStrategy paymentStrategy;
    
    public void addItem(Item item) {
        items.add(item);
    }
    
    public void setPaymentStrategy(PaymentStrategy strategy) {
        this.paymentStrategy = strategy;
    }
    
    public void checkout() {
        double total = items.stream()
            .mapToDouble(Item::getPrice)
            .sum();
        
        paymentStrategy.pay(total);
    }
}

@FunctionalInterface
interface DiscountStrategy {

}

class PriceCalculator {
    private DiscountStrategy discountStrategy;
    
    public void setDiscountStrategy(DiscountStrategy strategy) {
        this.discountStrategy = strategy;
    }
    
    public double calculatePrice(double originalPrice) {
        return discountStrategy.applyDiscount(originalPrice);
    }
}

 * USAGE WITH LAMBDAS:
 *

 *

 * calculator.setDiscountStrategy(price -> price);

 * 
 * // 10% discount
 * calculator.setDiscountStrategy(price -> price * 0.9);
 * System.out.println(calculator.calculatePrice(100)); // 90

 * // $20 off

 *

 * 
 * 
 * BENEFITS OF FUNCTIONAL APPROACH:
 * 
 * ✓ No strategy classes needed

 * ✓ More concise
 * ✓ Compose strategies

class Product {

    private double price;
    private int rating;

    }
    
    // Getters

    public double getPrice() { return price; }
    public int getRating() { return rating; }
    
    @Override
    public String toString() {
        return name + " ($" + price + ", " + rating + "★)";

    private List<Product> products;

    }

    public void display() {
        products.forEach(System.out::println);

}

/**
 * USAGE:
 * 
 * List<Product> products = Arrays.asList(
 *     new Product("Laptop", 999.99, 4),
 *     new Product("Mouse", 29.99, 5),
 *     new Product("Keyboard", 79.99, 3)
 * );
 * 
 * ProductCatalog catalog = new ProductCatalog(products);
 * 
 * // Sort by price (ascending)
 * catalog.sort(Comparator.comparing(Product::getPrice));
 * catalog.display();
 * 
 * // Sort by rating (descending)
 * catalog.sort(Comparator.comparing(Product::getRating).reversed());
 * catalog.display();
 * 
 * // Sort by price then rating
 * catalog.sort(Comparator.comparing(Product::getPrice)
 *     .thenComparing(Product::getRating));
 * catalog.display();
 * 
 * 
 * JAVA EXAMPLES:
 * 
 * Comparator - sorting strategy
 * Collections.sort(list, comparator)
 * 
 * LayoutManager - layout strategy (Swing)
 * panel.setLayout(new BorderLayout())
 * 
 * ThreadPoolExecutor.RejectedExecutionHandler - rejection strategy
 * new ThreadPoolExecutor(..., new CallerRunsPolicy())
 */

// ========================================
// WHEN TO USE STRATEGY
// ========================================
/**
 * USE STRATEGY WHEN:
 * 
 * ✓ Multiple versions of an algorithm
 * ✓ Algorithm needs to be selected at runtime
 * ✓ Want to avoid large conditional statements
 * ✓ Clients need to know about different algorithms
 * ✓ Algorithms use different data structures
 * 
 * 
 * STRATEGY vs STATE:
 * 
 * Strategy:
 * - Client chooses strategy
 * - Strategies independent
 * - Focus on algorithm
 * 
 * State:
 * - State transitions automatic
 * - States know about each other
 * - Focus on state-dependent behavior
 */

}
```

## 2. Observer Pattern

### One-to-Many Notification

```java
/**
 * OBSERVER PATTERN
 * 
 * Defines one-to-many dependency
 * When one object changes state, all dependents notified automatically
 */

// ========================================
// BASIC OBSERVER PATTERN
// ========================================

// Observer interface
interface Observer {
    void update(String message);
}

// Subject (Observable)
class NewsAgency {
    private List<Observer> observers = new ArrayList<>();
    private String news;
    
    public void addObserver(Observer observer) {
        observers.add(observer);
    }
    
    public void removeObserver(Observer observer) {
        observers.remove(observer);
    }
    
    public void setNews(String news) {
        this.news = news;
        notifyObservers();
    }
    
    private void notifyObservers() {
        for (Observer observer : observers) {
            observer.update(news);
        }
    }
}

// Concrete observers
class NewsChannel implements Observer {
    private String name;
    
    public NewsChannel(String name) {
        this.name = name;
    }
    
    @Override
    public void update(String news) {
        System.out.println(name + " received news: " + news);
    }
}

/**
 * USAGE:
 * 
 * NewsAgency agency = new NewsAgency();
 * 
 * Observer cnn = new NewsChannel("CNN");
 * Observer bbc = new NewsChannel("BBC");
 * Observer fox = new NewsChannel("Fox News");
 * 
 * agency.addObserver(cnn);
 * agency.addObserver(bbc);
 * agency.addObserver(fox);
 * 
 * agency.setNews("Breaking: Important event!");
 * // Output:
 * // CNN received news: Breaking: Important event!
 * // BBC received news: Breaking: Important event!
 * // Fox News received news: Breaking: Important event!
 * 
 * 
 * BENEFITS:
 * 
 * ✓ Loose coupling (subject doesn't know observers)
 * ✓ Dynamic relationships (add/remove at runtime)
 * ✓ Supports broadcast communication
 * ✓ Open/Closed Principle
 */

// ========================================
// JAVA BUILT-IN: Observable & Observer (DEPRECATED!)
// ========================================
/**
 * DEPRECATED in Java 9:
 * 
 * class NewsAgency extends Observable {
 *     public void setNews(String news) {
 *         setChanged();
 *         notifyObservers(news);
 *     }
 * }
 * 
 * class NewsChannel implements java.util.Observer {
 *     public void update(Observable o, Object arg) {
 *         System.out.println("News: " + arg);
 *     }
 * }
 * 
 * 
 * WHY DEPRECATED:
 * 
 * ✗ Observable is a class (not interface) - limits flexibility
 * ✗ setChanged() protected - awkward
 * ✗ notifyObservers() not thread-safe
 * ✗ Notification order undefined
 * 
 * 
 * ALTERNATIVES:
 * 
 * ✓ PropertyChangeSupport (JavaBeans)
 * ✓ Custom implementation
 * ✓ Event listeners (Swing)
 * ✓ Reactive extensions (RxJava, Project Reactor)
 */

// ========================================
// PROPERTY CHANGE SUPPORT (RECOMMENDED)
// ========================================

import java.beans.*;

class Stock {
    private String symbol;
    private double price;
    private PropertyChangeSupport support;
    
    public Stock(String symbol, double price) {
        this.symbol = symbol;
        this.price = price;
        this.support = new PropertyChangeSupport(this);
    }
    
    public void addPropertyChangeListener(PropertyChangeListener listener) {
        support.addPropertyChangeListener(listener);
    }
    
    public void removePropertyChangeListener(PropertyChangeListener listener) {
        support.removePropertyChangeListener(listener);
    }
    
    public void setPrice(double newPrice) {
        double oldPrice = this.price;
        this.price = newPrice;
        
        // Fire property change event
        support.firePropertyChange("price", oldPrice, newPrice);
    }
    
    public double getPrice() {
        return price;
    }
    
    public String getSymbol() {
        return symbol;
    }
}

class StockDisplay implements PropertyChangeListener {
    private String name;
    
    public StockDisplay(String name) {
        this.name = name;
    }
    
    @Override
    public void propertyChange(PropertyChangeEvent evt) {
        if ("price".equals(evt.getPropertyName())) {
            System.out.println(name + " - Stock price changed: " +
                evt.getOldValue() + " -> " + evt.getNewValue());
        }
    }
}

 * BENEFITS OF PROPERTYCHANGESUPPORT:

// ========================================
// EVENT LISTENERS (SWING/AWT STYLE)
// ========================================

// Custom event
class OrderEvent {
    private String orderId;
    private String status;
    
    public OrderEvent(String orderId, String status) {
        this.orderId = orderId;
        this.status = status;
    }
    
    public String getOrderId() { return orderId; }
    public String getStatus() { return status; }
}

// Listener interface
@FunctionalInterface
interface OrderListener {
    void onOrderStatusChanged(OrderEvent event);
}

// Event source
class OrderService {
    private List<OrderListener> listeners = new ArrayList<>();
    
    public void addOrderListener(OrderListener listener) {
        listeners.add(listener);
    }
    
    public void removeOrderListener(OrderListener listener) {
        listeners.remove(listener);
    }
    
    public void updateOrderStatus(String orderId, String newStatus) {
        // Business logic...
        
        // Notify listeners
        OrderEvent event = new OrderEvent(orderId, newStatus);
        for (OrderListener listener : listeners) {
            listener.onOrderStatusChanged(event);
        }
    }
}

/**
 * USAGE:
 * 
 * OrderService orderService = new OrderService();
 * 
 * // Add listener (lambda)
 * orderService.addOrderListener(event ->
 *     System.out.println("Order " + event.getOrderId() + 
 *         " status: " + event.getStatus()));
 * 
 * // Add email notification listener
 * orderService.addOrderListener(event ->
 *     sendEmail("Order " + event.getOrderId() + " is " + event.getStatus()));
 * 
 * // Add logging listener
 * orderService.addOrderListener(event ->
 *     log("Order status change: " + event));
 * 
 * orderService.updateOrderStatus("ORD-123", "SHIPPED");
 * // All listeners notified!
 * 
 * 
 * SWING EXAMPLE:
 * 
 * JButton button = new JButton("Click me");
 * button.addActionListener(e -> System.out.println("Button clicked!"));
 * 
 * // Multiple listeners
 * button.addActionListener(e -> logClick());
 * button.addActionListener(e -> updateUI());
 * button.addActionListener(e -> sendAnalytics());
 */

// ========================================
// PUSH vs PULL MODEL
// ========================================

// PUSH MODEL: Subject sends all data
interface PushObserver {
    void update(String symbol, double price, int volume);
}

// PULL MODEL: Observer pulls data from subject
interface PullObserver {
    void update(Subject subject);
}

class Subject {
    private List<PullObserver> observers = new ArrayList<>();
    private String symbol;
    private double price;
    private int volume;
    
    public void notifyObservers() {
        for (PullObserver observer : observers) {
            observer.update(this);  // Observer pulls what it needs
        }
    }
    
    // Getters
    public String getSymbol() { return symbol; }
    public double getPrice() { return price; }
    public int getVolume() { return volume; }
}

/**
 * PUSH vs PULL:
 * 
 * Push Model:
 * ✓ Observer gets all data
 * ✓ Simple observer interface
 * ✗ Subject must know what data observers need
 * ✗ May send unnecessary data
 * 
 * Pull Model:
 * ✓ Observer pulls only what it needs
 * ✓ More flexible
 * ✗ Observer must know subject interface
 * ✗ More coupling
 * 
 * 
 * RECOMMENDATION:
 * 
 * Push model: When observers need most/all data
 * Pull model: When observers need different data
 */

// ========================================
// WHEN TO USE OBSERVER
// ========================================
/**
 * USE OBSERVER WHEN:
 * 
 * ✓ Change in one object requires changing others
 * ✓ Number of dependents unknown or dynamic
 * ✓ Object should notify others without knowing who they are
 * ✓ Need loose coupling between objects
 * 
 * 
 * EXAMPLES:
 * 
 * - Event handling (Swing, JavaFX)
 * - MVC pattern (Model notifies View)
 * - Data binding
 * - Publish-subscribe systems
 * - Reactive programming (RxJava, Reactor)
 * 
 * 
 * CAUTIONS:
 * 
 * ✗ Memory leaks (remove observers!)
 * ✗ Update storms (cascading updates)
 * ✗ Order of notification undefined
 * ✗ Observer exceptions affect other observers
 */

// Helper methods
private void sendEmail(String message) {}
private void log(String message) {}
```

## 3. Command Pattern

### Encapsulating Requests as Objects

```java
/**
 * COMMAND PATTERN
 * 
 * Encapsulates request as object
 * Parameterizes clients with different requests
 * Supports undo/redo operations
 */

// ========================================
// BASIC COMMAND PATTERN
// ========================================

// Command interface
interface Command {
    void execute();
    void undo();
}

// Receiver (performs actual work)
class Light {
    private boolean on = false;
    
    public void turnOn() {
        on = true;
        System.out.println("Light is ON");
    }
    
    public void turnOff() {
        on = false;
        System.out.println("Light is OFF");
    }
}

// Concrete commands
class LightOnCommand implements Command {
    private Light light;
    
    public LightOnCommand(Light light) {
        this.light = light;
    }
    
    @Override
    public void execute() {
        light.turnOn();
    }
    
    @Override
    public void undo() {
        light.turnOff();
    }
}

class LightOffCommand implements Command {
    private Light light;
    
    public LightOffCommand(Light light) {
        this.light = light;
    }
    
    @Override
    public void execute() {
        light.turnOff();
    }
    
    @Override
    public void undo() {
        light.turnOn();
    }
}

// Invoker
class RemoteControl {
    private Command command;
    
    public void setCommand(Command command) {
        this.command = command;
    }
    
    public void pressButton() {
        command.execute();
    }
    
    public void pressUndo() {
        command.undo();
    }
}

/**
 * USAGE:
 * 
 * Light light = new Light();
 * Command lightOn = new LightOnCommand(light);
 * Command lightOff = new LightOffCommand(light);
 * 
 * RemoteControl remote = new RemoteControl();
 * 
 * remote.setCommand(lightOn);
 * remote.pressButton();  // Light is ON
 * remote.pressUndo();    // Light is OFF
 * 
 * remote.setCommand(lightOff);
 * remote.pressButton();  // Light is OFF
 * remote.pressUndo();    // Light is ON
 * 
 * 
 * BENEFITS:
 * 
 * ✓ Decouples invoker from receiver
 * ✓ Commands are objects (can be stored, passed, logged)
 * ✓ Easy to add new commands
 * ✓ Supports undo/redo
 * ✓ Can create composite commands (macros)
 */

// ========================================
// TEXT EDITOR WITH UNDO/REDO
// ========================================

class TextEditor {
    private StringBuilder text = new StringBuilder();
    
    public void insertText(int position, String newText) {
        text.insert(position, newText);
    }
    
    public void deleteText(int position, int length) {
        text.delete(position, position + length);
    }
    
    public String getText() {
        return text.toString();
    }
}

class InsertTextCommand implements Command {
    private TextEditor editor;
    private int position;
    private String text;
    
    public InsertTextCommand(TextEditor editor, int position, String text) {
        this.editor = editor;
        this.position = position;
        this.text = text;
    }
    
    @Override
    public void execute() {
        editor.insertText(position, text);
    }
    
    @Override
    public void undo() {
        editor.deleteText(position, text.length());
    }
}

class DeleteTextCommand implements Command {
    private TextEditor editor;
    private int position;
    private int length;
    private String deletedText;
    
    public DeleteTextCommand(TextEditor editor, int position, int length) {
        this.editor = editor;
        this.position = position;
        this.length = length;
    }
    
    @Override
    public void execute() {
        deletedText = editor.getText().substring(position, position + length);
        editor.deleteText(position, length);
    }
    
    @Override
    public void undo() {
        editor.insertText(position, deletedText);
    }
}

class CommandManager {
    private Stack<Command> undoStack = new Stack<>();
    private Stack<Command> redoStack = new Stack<>();
    
    public void executeCommand(Command command) {
        command.execute();
        undoStack.push(command);
        redoStack.clear();  // Clear redo stack on new command
    }
    
    public void undo() {
        if (!undoStack.isEmpty()) {
            Command command = undoStack.pop();
            command.undo();
            redoStack.push(command);
        }
    }
    
    public void redo() {
        if (!redoStack.isEmpty()) {
            Command command = redoStack.pop();
            command.execute();
            undoStack.push(command);
        }
    }
}

/**
 * USAGE:
 * 
 * TextEditor editor = new TextEditor();
 * CommandManager manager = new CommandManager();
 * 
 * manager.executeCommand(new InsertTextCommand(editor, 0, "Hello"));
 * System.out.println(editor.getText());  // Hello
 * 
 * manager.executeCommand(new InsertTextCommand(editor, 5, " World"));
 * System.out.println(editor.getText());  // Hello World
 * 
 * manager.undo();
 * System.out.println(editor.getText());  // Hello
 * 
 * manager.redo();
 * System.out.println(editor.getText());  // Hello World
 * 
 * manager.executeCommand(new DeleteTextCommand(editor, 0, 6));
 * System.out.println(editor.getText());  // World
 * 
 * manager.undo();
 * System.out.println(editor.getText());  // Hello World
 */

// ========================================
// MACRO COMMAND (COMPOSITE)
// ========================================

class MacroCommand implements Command {
    private List<Command> commands = new ArrayList<>();
    
    public void addCommand(Command command) {
        commands.add(command);
    }
    
    @Override
    public void execute() {
        for (Command command : commands) {
            command.execute();
        }
    }
    
    @Override
    public void undo() {
        // Undo in reverse order
        for (int i = commands.size() - 1; i >= 0; i--) {
            commands.get(i).undo();
        }
    }
}

/**
 * USAGE:
 * 
 * Light light1 = new Light();
 * Light light2 = new Light();
 * Light light3 = new Light();
 * 
 * MacroCommand allLightsOn = new MacroCommand();
 * allLightsOn.addCommand(new LightOnCommand(light1));
 * allLightsOn.addCommand(new LightOnCommand(light2));
 * allLightsOn.addCommand(new LightOnCommand(light3));
 * 
 * RemoteControl remote = new RemoteControl();
 * remote.setCommand(allLightsOn);
 * remote.pressButton();  // All lights ON
 * remote.pressUndo();    // All lights OFF
 * 
 * 
 * WHEN TO USE COMMAND:
 * 
 * ✓ Parameterize objects with operations
 * ✓ Queue operations
 * ✓ Log operations
 * ✓ Support undo/redo
 * ✓ Transaction systems
 * ✓ Macro recording
 * 
 * 
 * EXAMPLES:
 * 
 * - Text editors (undo/redo)
 * - GUI actions
 * - Transaction systems
 * - Task schedulers
 * - Remote procedure calls
 */
```

## 4. Template Method Pattern

### Defining Algorithm Skeleton

```java
/**
 * TEMPLATE METHOD PATTERN
 * 
 * Defines skeleton of algorithm in method
 * Defers some steps to subclasses
 * Subclasses override steps without changing structure
 */

// ========================================
// BASIC TEMPLATE METHOD
// ========================================

abstract class DataProcessor {
    
    // Template method (final to prevent override)
    public final void process() {
        openFile();
        extractData();
        parseData();
        analyzeData();
        sendReport();
        closeFile();
    }
    
    // Common steps (implemented in base class)
    private void openFile() {
        System.out.println("Opening file...");
    }
    
    private void closeFile() {
        System.out.println("Closing file...");
    }
    
    // Abstract steps (subclasses must implement)
    protected abstract void extractData();
    protected abstract void parseData();
    
    // Hook methods (optional override)
    protected void analyzeData() {
        System.out.println("Performing default analysis...");
    }
    
    protected void sendReport() {
        System.out.println("Sending default report...");
    }
}

class CSVDataProcessor extends DataProcessor {
    @Override
    protected void extractData() {
        System.out.println("Extracting CSV data...");
    }
    
    @Override
    protected void parseData() {
        System.out.println("Parsing CSV format...");
    }
    
    @Override
    protected void analyzeData() {
        System.out.println("Analyzing CSV data with special rules...");
    }
}

class XMLDataProcessor extends DataProcessor {
    @Override
    protected void extractData() {
```

## 5. Chain of Responsibility

### Passing Requests Along Chain

```java
/**
 * CHAIN OF RESPONSIBILITY PATTERN
 * 
 * Passes request along chain of handlers
 * Each handler decides to process or pass to next
 * Decouples sender from receiver
 */

// ========================================
// BASIC CHAIN OF RESPONSIBILITY
// ========================================

abstract class SupportHandler {
    protected SupportHandler nextHandler;
    
    public void setNextHandler(SupportHandler nextHandler) {
        this.nextHandler = nextHandler;
    }
    
    public abstract void handleRequest(SupportRequest request);
}

class SupportRequest {
    private String type;
    private String description;
    
    public SupportRequest(String type, String description) {
        this.type = type;
        this.description = description;
    }
    
    public String getType() { return type; }
    public String getDescription() { return description; }
}

class Level1Support extends SupportHandler {
    @Override
    public void handleRequest(SupportRequest request) {
        if (request.getType().equals("BASIC")) {
            System.out.println("Level 1: Handling basic request - " + 
                request.getDescription());
        } else if (nextHandler != null) {
            nextHandler.handleRequest(request);
        }
    }
}

class Level2Support extends SupportHandler {
    @Override
    public void handleRequest(SupportRequest request) {
        if (request.getType().equals("INTERMEDIATE")) {
            System.out.println("Level 2: Handling intermediate request - " + 
                request.getDescription());
        } else if (nextHandler != null) {
            nextHandler.handleRequest(request);
        }
    }
}

class Level3Support extends SupportHandler {
    @Override
    public void handleRequest(SupportRequest request) {
        if (request.getType().equals("ADVANCED")) {
            System.out.println("Level 3: Handling advanced request - " + 
                request.getDescription());
        } else {
            System.out.println("Level 3: Cannot handle request - escalating to manager");
        }
    }
}

/**
 * USAGE:
 * 
 * // Build chain
 * SupportHandler level1 = new Level1Support();
 * SupportHandler level2 = new Level2Support();
 * SupportHandler level3 = new Level3Support();
 * 
 * level1.setNextHandler(level2);
 * level2.setNextHandler(level3);
 * 
 * // Send requests through chain
 * level1.handleRequest(new SupportRequest("BASIC", "Reset password"));
 * // Output: Level 1: Handling basic request - Reset password
 * 
 * level1.handleRequest(new SupportRequest("INTERMEDIATE", "Server down"));
 * // Output: Level 2: Handling intermediate request - Server down
 * 
 * level1.handleRequest(new SupportRequest("ADVANCED", "Security breach"));
 * // Output: Level 3: Handling advanced request - Security breach
 * 
 * 
 * BENEFITS:
 * 
 * ✓ Decouples sender from receivers
 * ✓ Dynamic chain (add/remove handlers at runtime)
 * ✓ Flexible responsibility assignment
 * ✓ Single Responsibility (each handler one task)

// ========================================
// SERVLET FILTERS (REAL-WORLD EXAMPLE)

interface Filter {
    void doFilter(Request request, Response response, FilterChain chain);
}

class Request {
    private Map<String, String> headers = new HashMap<>();
    private String body;
    
    public void setHeader(String name, String value) {
        headers.put(name, value);
    }
    
    public String getHeader(String name) {
        return headers.get(name);
    }
    
    public void setBody(String body) {
        this.body = body;
    }
    
    public String getBody() {
        return body;
    }
}

class Response {
    private int status = 200;
    private String body;
    
    public void setStatus(int status) {
        this.status = status;
    }
    
    public int getStatus() {
        return status;
    }
    
    public void setBody(String body) {
        this.body = body;
    }
    
    public String getBody() {
        return body;
    }
}

class FilterChain {
    private List<Filter> filters = new ArrayList<>();
    private int currentIndex = 0;
    
    public void addFilter(Filter filter) {
        filters.add(filter);
    }
    
    public void doFilter(Request request, Response response) {
        if (currentIndex < filters.size()) {
            Filter filter = filters.get(currentIndex);
            currentIndex++;
            filter.doFilter(request, response, this);
        }
    }
}

// Concrete filters
class AuthenticationFilter implements Filter {
    @Override
    public void doFilter(Request request, Response response, FilterChain chain) {
        String token = request.getHeader("Authorization");
        if (token != null && token.equals("valid-token")) {
            System.out.println("AuthenticationFilter: Authenticated");
            chain.doFilter(request, response);  // Pass to next
        } else {
            System.out.println("AuthenticationFilter: Authentication failed");
            response.setStatus(401);
        }
    }
}

class LoggingFilter implements Filter {
    @Override
    public void doFilter(Request request, Response response, FilterChain chain) {
        System.out.println("LoggingFilter: Request received");
        chain.doFilter(request, response);  // Pass to next
        System.out.println("LoggingFilter: Response status " + response.getStatus());
    }
}

class CompressionFilter implements Filter {
    @Override
    public void doFilter(Request request, Response response, FilterChain chain) {
        System.out.println("CompressionFilter: Compressing response");
        chain.doFilter(request, response);  // Pass to next
        // Compress response body
        response.setBody("compressed(" + response.getBody() + ")");
    }
}

abstract class Logger {
    public static int INFO = 1;
    public static int DEBUG = 2;
    public static int ERROR = 3;

        if (nextLogger != null) {
            nextLogger.logMessage(level, message);
        }
    }

}
```

## 6. State Pattern

### Encapsulating State-Dependent Behavior

```java
/**
 * STATE PATTERN
 * 
 * Allows object to alter behavior when internal state changes
 * Object appears to change its class

// ========================================
// WITHOUT STATE PATTERN (MESSY!)
// ========================================
class VendingMachineBad {
    private static final int NO_COIN = 0;
    private static final int HAS_COIN = 1;
    private static final int SOLD = 2;
    private static final int SOLD_OUT = 3;
    
    private int state = NO_COIN;
    private int count;
    
    public void insertCoin() {
        if (state == NO_COIN) {
            state = HAS_COIN;
            System.out.println("Coin inserted");
        } else if (state == HAS_COIN) {
            System.out.println("Cannot insert another coin");
        } else if (state == SOLD) {
            System.out.println("Please wait, dispensing item");
        } else if (state == SOLD_OUT) {
            System.out.println("Machine is sold out");
        }
    }
    
    // Similar messy conditionals for ejectCoin(), pressButton(), dispense()...
    
    /**
     * PROBLEMS:
     * ✗ Giant if-else chains
     * ✗ Adding new state = modify all methods
     * ✗ State transitions unclear
     * ✗ Hard to maintain
     */
}

// ========================================

// State interface
interface State {
    void insertCoin();
    void ejectCoin();
    void pressButton();
    void dispense();
}

// Context
class VendingMachine {
    private State noCoinState;
    private State hasCoinState;
    private State soldState;
    private State soldOutState;
    
    private State currentState;
    private int count;
    
    public VendingMachine(int count) {
        this.count = count;
        
        noCoinState = new NoCoinState(this);
        hasCoinState = new HasCoinState(this);
        soldState = new SoldState(this);
        soldOutState = new SoldOutState(this);
        
        if (count > 0) {
            currentState = noCoinState;
        } else {
            currentState = soldOutState;
        }
    }
    
    public void insertCoin() {
        currentState.insertCoin();
    }
    
    public void ejectCoin() {
        currentState.ejectCoin();
    }
    
    public void pressButton() {
        currentState.pressButton();
        currentState.dispense();
    }
    
    public void setState(State state) {
        this.currentState = state;
    }
    
    public void releaseItem() {
        if (count > 0) {
            count--;
            System.out.println("Item dispensed. Remaining: " + count);
        }
    }
    
    public int getCount() {
        return count;
    }
    
    // Getters for states
    public State getNoCoinState() { return noCoinState; }
    public State getHasCoinState() { return hasCoinState; }
    public State getSoldState() { return soldState; }
    public State getSoldOutState() { return soldOutState; }
}
```

## 7. Iterator Pattern

### Sequential Access Without Exposing Structure

```java
/**
 * ITERATOR PATTERN
 * 
 * Provides way to access elements sequentially
 * Without exposing underlying representation
 */

// ========================================
// CUSTOM ITERATOR IMPLEMENTATION
// ========================================

interface Iterator<T> {
    boolean hasNext();
    T next();
}

interface Collection<T> {
    Iterator<T> createIterator();
}

class BookCollection implements Collection<Book> {
    private List<Book> books = new ArrayList<>();
    
    public void addBook(Book book) {
        books.add(book);
    }
    
    @Override
    public Iterator<Book> createIterator() {
        return new BookIterator();
    }
    
    // Internal iterator implementation
    private class BookIterator implements Iterator<Book> {
        private int position = 0;
        
        @Override
        public boolean hasNext() {
            return position < books.size();
        }
        
        @Override
        public Book next() {
            if (!hasNext()) {
                throw new NoSuchElementException();
            }
            return books.get(position++);
        }
    }
}

class Book {
    private String title;
    private String author;
    
    public Book(String title, String author) {
        this.title = title;
        this.author = author;
    }
    
    @Override
    public String toString() {
        return title + " by " + author;
    }
}

/**
 * USAGE:
 * 
 * BookCollection collection = new BookCollection();
 * collection.addBook(new Book("1984", "Orwell"));
 * collection.addBook(new Book("Brave New World", "Huxley"));
 * 
 * Iterator<Book> iterator = collection.createIterator();
 * while (iterator.hasNext()) {
 *     Book book = iterator.next();
 *     System.out.println(book);
 * }
 * 
 * 
 * BENEFITS:
 * 
 * ✓ Uniform interface for traversal
 * ✓ Hides internal structure
 * ✓ Multiple iterators on same collection
 * ✓ Single Responsibility (traversal separate from collection)
 */

// ========================================
// JAVA ITERATOR INTERNALS
// ========================================

/**
 * ARRAYLIST ITERATOR (SIMPLIFIED):
 * 
 * private class Itr implements Iterator<E> {
 *     int cursor = 0;        // Index of next element
 *     int lastRet = -1;      // Index of last returned element
 *     int expectedModCount = modCount;  // For fail-fast
 *     
 *     public boolean hasNext() {
 *         return cursor != size;
 *     }
 *     
 *     public E next() {
 *         checkForComodification();
 *         int i = cursor;
 *         if (i >= size)
 *             throw new NoSuchElementException();
 *         Object[] elementData = ArrayList.this.elementData;
 *         if (i >= elementData.length)
 *             throw new ConcurrentModificationException();
 *         cursor = i + 1;
 *         return (E) elementData[lastRet = i];
 *     }
 *     
 *     public void remove() {
 *         if (lastRet < 0)
 *             throw new IllegalStateException();
 *         checkForComodification();
 *         
 *         try {
 *             ArrayList.this.remove(lastRet);
 *             cursor = lastRet;
 *             lastRet = -1;
 *             expectedModCount = modCount;
 *         } catch (IndexOutOfBoundsException ex) {
 *             throw new ConcurrentModificationException();
 *         }
 *     }
 *     
 *     final void checkForComodification() {
 *         if (modCount != expectedModCount)
 *             throw new ConcurrentModificationException();
 *     }
 * }
 * 
 * 
 * FAIL-FAST BEHAVIOR:
 * 
 * List<String> list = new ArrayList<>(Arrays.asList("A", "B", "C"));
 * Iterator<String> it = list.iterator();
 * 
 * it.next();  // "A"
 * list.add("D");  // Modifies list!
 * it.next();  // ConcurrentModificationException!
 * 
 * Why?
 * - expectedModCount != modCount
 * - Iterator detects modification
 * - Throws exception
 * 
 * Solution:
 * - Use iterator.remove() instead of list.remove()
 * - Or use CopyOnWriteArrayList (no fail-fast)
 * - Or use ConcurrentHashMap
 */

// ========================================
// CUSTOM ITERATOR WITH REMOVE
// ========================================

class NumberCollection implements Iterable<Integer> {
    private List<Integer> numbers = new ArrayList<>();
    
    public void add(Integer number) {
        numbers.add(number);
    }
    
    @Override
    public java.util.Iterator<Integer> iterator() {
        return new NumberIterator();
    }
    
    private class NumberIterator implements java.util.Iterator<Integer> {
        private int position = 0;
        private boolean canRemove = false;
        
        @Override
        public boolean hasNext() {
            return position < numbers.size();
        }
        
        @Override
        public Integer next() {
            if (!hasNext()) {
                throw new NoSuchElementException();
            }
            canRemove = true;
            return numbers.get(position++);
        }
        
        @Override
        public void remove() {
            if (!canRemove) {
                throw new IllegalStateException();
            }
            numbers.remove(--position);
            canRemove = false;
        }
    }
}

/**
 * USAGE:
 * 
 * NumberCollection numbers = new NumberCollection();
 * numbers.add(1);
 * numbers.add(2);
 * numbers.add(3);
 * numbers.add(4);
 * 
 * // Remove even numbers
 * java.util.Iterator<Integer> it = numbers.iterator();
 * while (it.hasNext()) {
 *     int num = it.next();
 *     if (num % 2 == 0) {
 *         it.remove();  // Safe removal
 *     }
 * }
 * 
 * // Result: [1, 3]
 * 
 * 
 * FOR-EACH LOOP:
 * 
 * NumberCollection numbers = new NumberCollection();
 * // ... add numbers
 * 
 * // For-each uses iterator internally
 * for (Integer num : numbers) {
 *     System.out.println(num);
 * }
 * 
 * // Equivalent to:
 * java.util.Iterator<Integer> it = numbers.iterator();
 * while (it.hasNext()) {
 *     Integer num = it.next();
 *     System.out.println(num);
 * }
 */

// ========================================
// WHEN TO USE ITERATOR
// ========================================
/**
 * USE ITERATOR WHEN:
 * 
 * ✓ Need uniform way to traverse collection
 * ✓ Want to hide internal structure
 * ✓ Support multiple simultaneous traversals
 * ✓ Provide uniform interface for different collections
 * 
 * 
 * JAVA EXAMPLES:
 * 
 * Collections:
 * - ArrayList, LinkedList, HashSet all provide iterator()
 * - Uniform interface despite different structures
 * 
 * Streams:
 * - Internally use iterators
 * - Spliterator for parallel streams
 * 
 * ResultSet (JDBC):
```

## Summary

### Quick Reference

**Behavioral Patterns:**

```
1. Strategy: Encapsulate algorithms
   - Functional interfaces (Java 8+)
   - Runtime algorithm selection
   - Use: Payment methods, sorting, validation

2. Observer: One-to-many notification
   - PropertyChangeSupport (JavaBeans)
   - Event listeners
   - Use: MVC, event systems, reactive

3. Command: Encapsulate requests
   - Undo/redo support
   - Queue operations
   - Use: Text editors, transactions, GUIs

4. Template Method: Algorithm skeleton
   - Hook methods
   - Framework design
   - Use: AbstractList, HttpServlet, workflows

5. Chain of Responsibility: Pass along chain
   - Filter chains
   - Request handling
   - Use: Servlet filters, logging, middleware

6. State: State-dependent behavior
   - Eliminate conditionals
   - Explicit transitions
   - Use: TCP, workflows, game states

7. Iterator: Sequential access
   - Fail-fast behavior
   - Hide structure
   - Use: Collections, streams, traversal
```

**Pattern Comparison:**

```
Strategy vs State:
- Strategy: Client chooses, algorithms independent
- State: Automatic transitions, states know each other

Command vs Strategy:
- Command: Encapsulates action (undo/redo)
- Strategy: Encapsulates algorithm (interchangeable)

Chain vs Decorator:
- Chain: Request may stop at any handler
- Decorator: All decorators process request
```
