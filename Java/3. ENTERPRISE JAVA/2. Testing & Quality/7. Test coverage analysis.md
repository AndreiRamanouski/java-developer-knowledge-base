# Test Coverage Analysis

## Overview

Test coverage measures which parts of your code are executed during testing—it answers "did tests run this code?" but not "did tests verify this code works?". Code coverage is essential for finding untested code (gaps in your test suite) but insufficient for measuring test quality (you can have 100% coverage with zero assertions). This guide covers six critical aspects: JaCoCo Configuration (industry-standard Java coverage tool—Maven/Gradle setup, report formats, aggregation across modules), Line vs Branch Coverage (line coverage measures executed lines, branch coverage measures executed decision paths—branch coverage is more meaningful), Coverage Goals (realistic targets: 70-80% overall, 90%+ for critical code, when 100% makes sense and when it doesn't), Coverage Limitations (coverage shows execution not verification, doesn't guarantee bug-free code, can create false confidence), Uncovered Code Analysis (systematically finding gaps, understanding why code is uncovered, deciding what to test), and Build Tool Integration (enforcing minimum coverage, failing builds, generating reports, CI/CD pipeline integration). The key insight: coverage is a necessary but not sufficient quality metric. Low coverage (< 60%) indicates insufficient testing—gaps where bugs hide. High coverage (> 80%) indicates comprehensive execution but says nothing about assertion quality or test effectiveness. Understanding coverage means understanding what it measures (execution) and what it doesn't (correctness, edge cases, mutation resistance). JaCoCo (Java Code Coverage) is the standard tool—fast, accurate, integrates seamlessly with Maven/Gradle/CI tools, generates HTML/XML/CSV reports. The workflow: write tests → run JaCoCo → review report → identify gaps → add tests for uncovered critical code → maintain coverage over time. This guide shows when to chase coverage (public APIs, critical business logic, security-sensitive code) and when to ignore it (trivial getters, generated code, defensive error handling). The relationship to other metrics: code coverage + mutation testing + code review = comprehensive quality assessment. Coverage tells you where to look, mutation testing tells you if tests are effective, code review catches design issues.

## 1. Understanding Code Coverage

### What Coverage Measures (and What It Doesn't)

```java
/**
 * WHAT IS CODE COVERAGE?
 * 
 * Coverage measures which parts of code were EXECUTED during tests
 * It does NOT measure if tests VERIFIED anything
 */

// ========================================
// THE PROBLEM: COVERAGE WITHOUT VERIFICATION
// ========================================

class Calculator {
    public int add(int a, int b) {
        return a + b;
    }
    
    public int divide(int a, int b) {
        if (b == 0) {
            throw new ArithmeticException("Division by zero");
        }
        return a / b;
    }
}

/**
 * TEST WITH 100% LINE COVERAGE (but useless!)
 */
class CalculatorBadTest {
    
    @Test
    void testCalculator() {
        Calculator calc = new Calculator();
        
        // Execute code - 100% line coverage!
        calc.add(2, 3);
        calc.divide(10, 2);
        
        // ❌ NO ASSERTIONS!
        // Coverage: 100%
        // Value: 0%
    }
    
    /**
     * JACOCO REPORT:
     * ✓ 100% line coverage
     * ✓ 100% branch coverage
     * 
     * But test verifies NOTHING!
     */
}

/**
 * TEST WITH 100% COVERAGE (and useful!)
 */
class CalculatorGoodTest {
    
    @Test
    void shouldAddTwoNumbers() {
        Calculator calc = new Calculator();
        assertEquals(5, calc.add(2, 3));
        assertEquals(0, calc.add(0, 0));
        assertEquals(-1, calc.add(-3, 2));
    }
    
    @Test
    void shouldDivideTwoNumbers() {
        Calculator calc = new Calculator();
        assertEquals(5, calc.divide(10, 2));
        assertEquals(0, calc.divide(0, 5));
    }
    
    @Test
    void shouldThrowExceptionWhenDividingByZero() {
        Calculator calc = new Calculator();
        assertThrows(ArithmeticException.class, () -> calc.divide(10, 0));
    }
    
    /**
     * JACOCO REPORT:
     * ✓ 100% line coverage
     * ✓ 100% branch coverage
     * ✓ Strong assertions
     * ✓ Tests actual behavior
     */
}

// ========================================
// COVERAGE TYPES
// ========================================

/**
 * LINE COVERAGE: Percentage of lines executed
 * BRANCH COVERAGE: Percentage of decision branches executed
 * METHOD COVERAGE: Percentage of methods executed
 * CLASS COVERAGE: Percentage of classes loaded
 */

class CoverageTypes {
    
    /**
     * LINE COVERAGE EXAMPLE
     */
    public int absolute(int n) {
        if (n < 0) {          // Line 1
            return -n;        // Line 2
        }
        return n;             // Line 3
    }
    
    /**
     * Test with 67% line coverage
     */
    @Test
    void testAbsolute_PartialCoverage() {
        assertEquals(5, absolute(5));
        // Lines covered: 1, 3
        // Lines not covered: 2
        // Line coverage: 67% (2/3)
    }
    
    /**
     * Test with 100% line coverage
     */
    @Test
    void testAbsolute_FullCoverage() {
        assertEquals(5, absolute(5));   // Covers lines 1, 3
        assertEquals(5, absolute(-5));  // Covers lines 1, 2
        // Line coverage: 100% (3/3)
    }
    
    /**
     * BRANCH COVERAGE EXAMPLE
     */
    public String classify(int age) {
        if (age < 0) {           // Branch 1: true/false
            return "Invalid";
        }
        if (age < 18) {          // Branch 2: true/false
            return "Minor";
        }
        return "Adult";
    }
    
    /**
     * Test with 50% branch coverage
     */
    @Test
    void testClassify_PartialBranches() {
        assertEquals("Adult", classify(25));
        // Branch 1: false (covered)
        // Branch 1: true (NOT covered)
        // Branch 2: false (covered)
        // Branch 2: true (NOT covered)
        // Branch coverage: 50% (2/4)
    }
    
    /**
     * Test with 100% branch coverage
     */
    @Test
    void testClassify_AllBranches() {
        assertEquals("Invalid", classify(-1));  // Branch 1: true
        assertEquals("Minor", classify(10));    // Branch 1: false, Branch 2: true
        assertEquals("Adult", classify(25));    // Branch 1: false, Branch 2: false
        // Branch coverage: 100% (4/4)
    }
}

// ========================================
// WHY COVERAGE MATTERS
// ========================================

/**
 * COVERAGE FINDS UNTESTED CODE
 */
class WhyCoverageMatters {
    
    public void processPayment(Payment payment) {
        validatePayment(payment);
        
        if (payment.getAmount() > 1000) {
            requireManagerApproval(payment);  // ← Not tested!
        }
        
        chargeCard(payment);
        sendConfirmation(payment);
    }
    
    /**
     * Test with low coverage (missed critical path)
     */
    @Test
    void testProcessPayment_MissedPath() {
        Payment payment = new Payment(50);
        processPayment(payment);
        // Coverage: 80%
        // Missing: requireManagerApproval branch!
    }
    
    /**
     * Coverage report reveals gap
     * Add test for > $1000 payments
     */
    @Test
    void testProcessPayment_FullCoverage() {
        Payment payment1 = new Payment(50);
        processPayment(payment1);
        
        Payment payment2 = new Payment(1500);
        processPayment(payment2);
        // Coverage: 100%
        // All paths tested
    }
    
    /**
     * KEY INSIGHT:
     * Coverage shows WHERE to look for gaps
     * It doesn't guarantee tests are GOOD
     * 
     * Use coverage to find untested code
     * Use mutation testing to find weak tests
     */
}

// Helper classes
class Payment {
    private double amount;
    Payment(double amount) { this.amount = amount; }
    double getAmount() { return amount; }
}

void validatePayment(Payment p) {}
void requireManagerApproval(Payment p) {}
void chargeCard(Payment p) {}
void sendConfirmation(Payment p) {}
```

## 3. Line vs Branch Coverage

### Understanding Coverage Metrics

```java
/**
 * LINE VS BRANCH COVERAGE
 * 
 * Different metrics reveal different gaps
 */

// ========================================
// LINE COVERAGE
// ========================================

class LineCoverageExample {
    
    public String greet(String name) {
        if (name == null) {      // Line 1
            return "Hello!";      // Line 2
        }
        return "Hello, " + name;  // Line 3
    }
    
    /**
     * Test with 67% line coverage
     */
    @Test
    void testGreet_PartialLines() {
        assertEquals("Hello, John", greet("John"));
        
        // Lines covered: 1, 3
        // Lines not covered: 2
        // Line coverage: 67% (2/3)
        
        /**
         * PROBLEM: We executed most lines
         * But missed important null handling!
         */
    }
    
    /**
     * Test with 100% line coverage
     */
    @Test
    void testGreet_AllLines() {
        assertEquals("Hello, John", greet("John"));
        assertEquals("Hello!", greet(null));
        
        // Line coverage: 100% (3/3)
    }
}

// ========================================
// BRANCH COVERAGE
// ========================================

class BranchCoverageExample {
    
    public int calculateDiscount(int price, boolean isPremium, boolean hasCoupon) {
        int discount = 0;
        
        if (isPremium) {           // Branch 1: true/false
            discount += 10;
        }
        
        if (hasCoupon) {           // Branch 2: true/false
            discount += 5;
        }
        
        return discount;
    }
    
    /**
     * Test with 100% LINE coverage but 50% BRANCH coverage
     */
    @Test
    void testDiscount_AllLines() {
        // Test: premium + coupon
        assertEquals(15, calculateDiscount(100, true, true));
        
        /**
         * COVERAGE:
         * - Line coverage: 100% (all lines executed)
         * - Branch coverage: 50% (2/4 branches)
         * 
         * BRANCHES COVERED:
         * ✓ isPremium: true
         * ✗ isPremium: false
         * ✓ hasCoupon: true
         * ✗ hasCoupon: false
         * 
         * PROBLEM: Didn't test non-premium or no-coupon cases!
         */
    }
    
    /**
     * Test with 100% BRANCH coverage
     */
    @Test
    void testDiscount_AllBranches() {
        assertEquals(15, calculateDiscount(100, true, true));   // Both true
        assertEquals(10, calculateDiscount(100, true, false));  // Premium only
        assertEquals(5, calculateDiscount(100, false, true));   // Coupon only
        assertEquals(0, calculateDiscount(100, false, false));  // Neither
        
        /**
         * COVERAGE:
         * - Line coverage: 100%
         * - Branch coverage: 100%
         * 
         * All combinations tested!
         */
    }
}

// ========================================
// COMPLEX BRANCHING
// ========================================

class ComplexBranchingExample {
    
    public String validatePassword(String password) {
        if (password == null || password.isEmpty()) {
            return "Password required";
        }
        
        if (password.length() < 8) {
            return "Too short";
        }
        
        boolean hasUpper = password.chars().anyMatch(Character::isUpperCase);
        boolean hasLower = password.chars().anyMatch(Character::isLowerCase);
        boolean hasDigit = password.chars().anyMatch(Character::isDigit);
        
        if (!hasUpper || !hasLower || !hasDigit) {
            return "Must have upper, lower, and digit";
        }
        
        return "Valid";
    }
    
    /**
     * Comprehensive branch coverage
     */
    @Test
    void testValidatePassword() {
        // Null
        assertEquals("Password required", validatePassword(null));
        
        // Empty
        assertEquals("Password required", validatePassword(""));
        
        // Too short
        assertEquals("Too short", validatePassword("Pass1"));
        
        // Missing uppercase
        assertEquals("Must have upper, lower, and digit", validatePassword("password1"));
        
        // Missing lowercase
        assertEquals("Must have upper, lower, and digit", validatePassword("PASSWORD1"));
        
        // Missing digit
        assertEquals("Must have upper, lower, and digit", validatePassword("Password"));
        
        // Valid
        assertEquals("Valid", validatePassword("Password1"));
        
        /**
         * BRANCH COVERAGE: 100%
         * Every decision path tested
         */
    }
}

// ========================================
// WHY BRANCH COVERAGE IS BETTER
// ========================================

/**
 * COMPARISON:
 * 
 * LINE COVERAGE:
 * ✓ Simple to understand
 * ✓ Quick to measure
 * ✗ Misses decision paths
 * ✗ Can hide gaps
 * 
 * BRANCH COVERAGE:
 * ✓ More thorough
 * ✓ Tests all paths
 * ✓ Finds more gaps
 * ✗ Harder to achieve 100%
 * 
 * RECOMMENDATION:
 * - Track BOTH metrics
 * - Prioritize branch coverage
 * - Line coverage shows execution
 * - Branch coverage shows decisions
 */
```

## 4. Coverage Goals and Realistic Targets

### Setting Appropriate Coverage Targets

```java
/**
 * COVERAGE GOALS
 * 
 * What coverage percentage should you aim for?
 */

// ========================================
// REALISTIC TARGETS
// ========================================

/**
 * OVERALL TARGETS:
 * 
 * 90-100%: Critical code (payment, security, safety)
 * 80-90%:  Core business logic
 * 70-80%:  Standard application code
 * 60-70%:  Less critical features
 * <60%:    Indicates gaps (needs improvement)
 * 
 * SWEET SPOT: 75-85% for most applications
 */

class CoverageTargets {
    
    /**
     * CRITICAL CODE: Aim for 95-100%
     * 
     * Examples:
     * - Payment processing
     * - Security authentication
     * - Data validation
     * - Financial calculations
     */
    public class PaymentProcessor {
        public boolean processPayment(Payment payment) {
            // Every line tested!
            // 100% coverage expected
        }
    }
    
    /**
     * BUSINESS LOGIC: Aim for 80-90%
     * 
     * Examples:
     * - Order processing
     * - Inventory management
     * - Pricing calculations
     */
    public class OrderService {
        public Order createOrder(OrderRequest request) {
            // Most paths tested
            // 85% coverage target
        }
    }
    
    /**
     * UTILITY CODE: Aim for 70-80%
     * 
     * Examples:
     * - String formatting
     * - Date utilities
     * - Configuration helpers
     */
    public class StringUtils {
        public String truncate(String str, int length) {
            // Common cases tested
            // 75% coverage acceptable
        }
    }
    
    /**
     * ACCEPTABLE LOW COVERAGE:
     * 
     * These don't need high coverage:
     * - Simple getters/setters
     * - DTOs (data transfer objects)
     * - Configuration classes
     * - Generated code
     * - Main method
     */
    public class UserDTO {
        private String name;
        
        public String getName() { return name; }
        public void setName(String name) { this.name = name; }
        
        // 0% coverage OK - trivial code
    }
}

        }

     * 2. Financial calculations

        }
    }
    
    /**
     * 3. Medical/safety-critical systems
     */

        }

     */

     */

    }

     * 4. Configuration/wiring code
     */
```

## 5. Coverage Limitations

### What Coverage Doesn't Tell You

```java
/**
 * COVERAGE LIMITATIONS
 * 
 * Understanding what coverage CANNOT measure
 */

// ========================================
// LIMITATION 1: EXECUTION ≠ VERIFICATION
// ========================================

class ExecutionNotVerification {
    
    public int multiply(int a, int b) {
        return a * b;
    }
    
    /**
     * 100% coverage, 0% verification
     */
    @Test
    void testMultiply_NoAssertion() {
        multiply(3, 4);  // ❌ No assertion!
        
        // Coverage: 100%
        // Tests verified: 0%
        // Bug detection: 0%
    }
    
    /**
     * 100% coverage, weak verification
     */
    @Test
    void testMultiply_WeakAssertion() {
        int result = multiply(3, 4);
        assertTrue(result > 0);  // ❌ Too weak!
        
        // Coverage: 100%
        // Would pass even if multiply returned 1!
    }
    
    /**
     * 100% coverage, proper verification
     */
    @Test
    void testMultiply_StrongAssertion() {
        assertEquals(12, multiply(3, 4));
        assertEquals(0, multiply(0, 5));
        assertEquals(-12, multiply(-3, 4));
        
        // Coverage: 100%
        // Properly verifies behavior
    }
}

// ========================================
// LIMITATION 2: DOESN'T CATCH EDGE CASES
// ========================================

class EdgeCasesNotCovered {
    
    public int divide(int a, int b) {
        return a / b;  // Bug: no zero check!
    }
    
    /**
     * 100% coverage, misses critical bug
     */
    @Test
    void testDivide() {
        assertEquals(5, divide(10, 2));
        
        // Coverage: 100%
        // But didn't test division by zero!
        // Bug in production: divide(10, 0) crashes
    }
}

// ========================================
// LIMITATION 3: DOESN'T TEST COMBINATIONS
// ========================================

class CombinationsNotTested {
    
    public boolean canVote(int age, boolean isCitizen, boolean isRegistered) {
        return age >= 18 && isCitizen && isRegistered;
    }
    
    /**
     * 100% line coverage, missed combinations
     */
    @Test
    void testCanVote() {
        assertTrue(canVote(18, true, true));
        
        // Coverage: 100% lines
        // But didn't test:
        // - age < 18
        // - not a citizen
        // - not registered
        // - Various false combinations
    }
}

// ========================================
// LIMITATION 4: DOESN'T MEASURE QUALITY
// ========================================

class QualityNotMeasured {
    
    /**
     * High coverage doesn't mean:
     * 
     * ✗ Code is bug-free
     * ✗ Code is well-designed
     * ✗ Tests are meaningful
     * ✗ Edge cases are tested
     * ✗ Integration works
     * ✗ Performance is good
     * ✗ Security is solid
     * 
     * Coverage measures EXECUTION, not QUALITY
     */
}

// ========================================
// LIMITATION 5: CAN MISLEAD
// ========================================

class CoverageMisleading {
    
    /**
     * Example: High coverage hides problems
     */
    public String formatName(String first, String last) {
        // Bug: doesn't handle null!
        return first + " " + last;
    }
    
    @Test
    void testFormatName() {
        assertEquals("John Doe", formatName("John", "Doe"));
        
        // Coverage: 100%
        // False confidence!
        // Crashes with: formatName(null, "Doe")
    }
}

// ========================================
// WHAT TO USE INSTEAD
// ========================================

/**
 * COVERAGE + MUTATION TESTING + CODE REVIEW
 * 
 * Coverage:
 * - Shows execution
 * - Finds untested code
 * 
 * Mutation Testing:
 * - Measures test effectiveness
 * - Finds weak assertions
 * 
 * Code Review:
 * - Catches design issues
 * - Verifies edge cases
 * - Ensures readability
 * 
 * ALL THREE TOGETHER = Quality
 */
```

```java
/**
```

```java
// ========================================
// KEY CONCEPTS
// ========================================

/**
 * WHAT IS COVERAGE?
 * 
 * Coverage measures which code was EXECUTED during tests
 * It does NOT measure if tests VERIFIED behavior
 * 
 * Coverage finds UNTESTED code
 * Mutation testing finds WEAK tests
 */

// ========================================
// REALISTIC TARGETS
// ========================================

/**
 * 90-100%: Critical code (payment, security)
 * 80-90%:  Core business logic
 * 70-80%:  Standard application code
 * <60%:    Needs improvement
 * 
 * SWEET SPOT: 75-85% for most applications
 */

// ========================================
// BEST PRACTICES
// ========================================

/**
 * DO:
 * ✓ Track line AND branch coverage
 * ✓ Set realistic targets (75-85%)
 * ✓ Exclude trivial code
 * ✓ Fail builds on coverage drop
 * ✓ Combine with mutation testing
 * 
 * DON'T:
 * ✗ Chase 100% everywhere
 * ✗ Write tests just for coverage
 * ✗ Trust coverage alone
 */
```
