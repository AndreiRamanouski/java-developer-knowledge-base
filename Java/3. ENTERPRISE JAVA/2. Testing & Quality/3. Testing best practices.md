# Testing Best Practices

## Overview

Good tests are as important as production code—they document behavior, enable refactoring, and catch regressions. Yet bad tests create more problems than they solve: flaky tests erode confidence, slow tests kill productivity, coupled tests make refactoring painful. This guide covers seven essential practices that separate good tests from bad: Arrange-Act-Assert (AAA) pattern (structure tests in three clear phases—setup, execution, verification), Given-When-Then (GWT) pattern (BDD-style narrative—given context, when action, then outcome), Test Naming Conventions (names that explain what's tested and expected outcome), Test Independence (tests run in any order, in parallel, without shared state), Avoiding Test Interdependencies (no test depends on another running first), Fast vs Slow Tests (unit tests in milliseconds, integration tests in seconds, categorize and run appropriately), Test Pyramid (many fast unit tests, some integration tests, few E2E tests—optimal testing strategy). The key insight: tests are specifications—they describe what your code should do. Good tests are FIRST: Fast (run quickly), Independent (no order dependencies), Repeatable (same result every time), Self-Validating (pass/fail, no manual check), Timely (written with code, not after). Understanding these practices means understanding trade-offs: AAA is clear but verbose, GWT is narrative but requires discipline, fast tests are isolated but miss integration issues, slow tests are comprehensive but delay feedback. This guide shows when to apply each practice, common mistakes (shared fixtures, unclear names, brittle assertions, test pyramids inverted), and how practices combine (AAA structure + descriptive names + independence = maintainable test suite). Whether writing unit tests (isolate, mock boundaries, verify behavior) or integration tests (use real components, test workflows, accept slower speed), these practices ensure tests add value rather than burden.

## 1. Arrange-Act-Assert (AAA) Pattern

### The Three-Phase Test Structure

```java
/**
 * ARRANGE-ACT-ASSERT (AAA) PATTERN
 * 
 * Structure tests in three distinct phases:
 * 1. Arrange: Set up test data and preconditions
 * 2. Act: Execute the code under test
 * 3. Assert: Verify the outcome
 */

import org.junit.jupiter.api.*;
import static org.junit.jupiter.api.Assertions.*;
import static org.mockito.Mockito.*;

// ========================================
// EXAMPLE PRODUCTION CODE
// ========================================

class ShoppingCart {
    private List<Item> items = new ArrayList<>();
    private DiscountService discountService;
    
    ShoppingCart(DiscountService discountService) {
        this.discountService = discountService;
    }
    
    void addItem(Item item) {
        items.add(item);
    }
    
    BigDecimal getTotal() {
        BigDecimal subtotal = items.stream()
            .map(Item::getPrice)
            .reduce(BigDecimal.ZERO, BigDecimal::add);
        
        BigDecimal discount = discountService.calculateDiscount(subtotal);
        return subtotal.subtract(discount);
    }
    
    int getItemCount() {
        return items.size();
    }
}

record Item(String name, BigDecimal price) {
    BigDecimal getPrice() { return price; }
}

interface DiscountService {
    BigDecimal calculateDiscount(BigDecimal amount);
}

// ========================================
// ❌ BAD: Unclear structure
// ========================================

class BadTestStructure {
    
    @Test
    void testShoppingCart() {
        ShoppingCart cart = new ShoppingCart(mock(DiscountService.class));
        cart.addItem(new Item("Widget", new BigDecimal("10.00")));
        assertEquals(1, cart.getItemCount());
        cart.addItem(new Item("Gadget", new BigDecimal("20.00")));
        assertEquals(2, cart.getItemCount());
        BigDecimal total = cart.getTotal();
        assertTrue(total.compareTo(BigDecimal.ZERO) > 0);
        
        /**
         * PROBLEMS:
         * ✗ No clear structure
         * ✗ Multiple actions mixed with assertions
         * ✗ Hard to understand what's being tested
         * ✗ Multiple concerns in one test
         */
    }
}

// ========================================
// ✓ GOOD: Clear AAA structure
// ========================================

class GoodTestStructure {
    
    @Test
    void shouldCalculateTotalWithTwoItems() {
        // ===== ARRANGE =====
        DiscountService discountService = mock(DiscountService.class);
        when(discountService.calculateDiscount(any())).thenReturn(BigDecimal.ZERO);
        
        ShoppingCart cart = new ShoppingCart(discountService);
        cart.addItem(new Item("Widget", new BigDecimal("10.00")));
        cart.addItem(new Item("Gadget", new BigDecimal("20.00")));
        
        // ===== ACT =====
        BigDecimal total = cart.getTotal();
        
        // ===== ASSERT =====
        assertEquals(new BigDecimal("30.00"), total);
        
        /**
         * BENEFITS:
         * ✓ Clear structure (three sections)
         * ✓ One action (getTotal)
         * ✓ One assertion (total = 30.00)
         * ✓ Easy to understand what's tested
         */
    }
    
    @Test
    void shouldApplyDiscountToTotal() {
        // Arrange
        DiscountService discountService = mock(DiscountService.class);
        when(discountService.calculateDiscount(new BigDecimal("30.00")))
            .thenReturn(new BigDecimal("3.00"));  // 10% discount
        
        ShoppingCart cart = new ShoppingCart(discountService);
        cart.addItem(new Item("Widget", new BigDecimal("10.00")));
        cart.addItem(new Item("Gadget", new BigDecimal("20.00")));
        
        // Act
        BigDecimal total = cart.getTotal();
        
        // Assert
        assertEquals(new BigDecimal("27.00"), total);
    }
    
    @Test
    void shouldReturnZeroForEmptyCart() {
        // Arrange
        DiscountService discountService = mock(DiscountService.class);
        ShoppingCart cart = new ShoppingCart(discountService);
        
        // Act
        BigDecimal total = cart.getTotal();
        
        // Assert
        assertEquals(BigDecimal.ZERO, total);
    }
}

// AAA WITH COMPLEX SETUP

class ComplexAAAExample {
    
    @Test
    void shouldProcessOrderWithMultipleItems() {
        // ===== ARRANGE =====
        // Create dependencies
        UserRepository userRepository = mock(UserRepository.class);
        PaymentService paymentService = mock(PaymentService.class);
        EmailService emailService = mock(EmailService.class);
        
        // Configure stubs
        User user = new User("john@example.com", "John Doe");
        when(userRepository.findByEmail("john@example.com"))
            .thenReturn(Optional.of(user));
        when(paymentService.charge(any(), any())).thenReturn(true);
        
        // Create service under test
        OrderService orderService = new OrderService(
            userRepository,
            paymentService,
            emailService
        );
        
        // Create test data
        List<OrderItem> items = Arrays.asList(
            new OrderItem("PROD001", 2, new BigDecimal("10.00")),
            new OrderItem("PROD002", 1, new BigDecimal("25.00"))
        );
        OrderRequest request = new OrderRequest("john@example.com", items);
        
        // ===== ACT =====
        Order order = orderService.processOrder(request);
        
        // ===== ASSERT =====
        assertNotNull(order);
        assertEquals(3, order.getTotalItems());
        assertEquals(new BigDecimal("45.00"), order.getTotal());
        assertEquals(OrderStatus.CONFIRMED, order.getStatus());
        
        // Verify side effects
        verify(paymentService).charge(user, new BigDecimal("45.00"));
        verify(emailService).sendOrderConfirmation(user.getEmail(), order);
    }
    
    /**
     * AAA GUIDELINES:
     * 
     * ARRANGE:
     * ✓ Create test data
     * ✓ Configure mocks/stubs
     * ✓ Set up preconditions
     * ✓ Can be multiple lines
     * 
     * ACT:
     * ✓ Execute ONE operation
     * ✓ Usually one line
     * ✓ The code being tested
     * 
     * ASSERT:
     * ✓ Verify outcome
     * ✓ Can have multiple assertions (related)
     * ✓ Can verify mock interactions
     */
}

     * VARIATION 2: Exception testing
     */

        // Arrange

        IllegalArgumentException exception = assertThrows(
            IllegalArgumentException.class,
            () -> userService.createUser("invalid-email", "John", "Doe", null)

        // Additional assertions on exception
        assertTrue(exception.getMessage().contains("Invalid email"));
```

## 2. Given-When-Then (GWT) Pattern

### BDD-Style Test Narrative

```java
/**
 * GIVEN-WHEN-THEN PATTERN
 * 
 * Behavior-Driven Development (BDD) style testing
 * 
 * GIVEN:  Initial context/state (Arrange)
 * WHEN:   Action/event occurs (Act)
 * THEN:   Expected outcome (Assert)
 */

// ========================================
// GWT WITH COMMENTS
// ========================================

class GivenWhenThenWithComments {
    
    @Test
    void shouldApplyDiscountForPremiumUser() {
        // GIVEN a premium user with items in cart
        User premiumUser = new User("premium@example.com", "Premium User");
        premiumUser.setMembershipType(MembershipType.PREMIUM);
        
        ShoppingCart cart = new ShoppingCart(new DiscountService());
        cart.addItem(new Item("Widget", new BigDecimal("100.00")));
        
        // WHEN calculating the total
        BigDecimal total = cart.getTotal();
        
        // THEN premium discount should be applied (10% off)
        assertEquals(new BigDecimal("90.00"), total);
    }
    
    @Test
    void shouldRejectOrderWhenInventoryInsufficient() {
        // GIVEN an order with quantity exceeding inventory
        Product product = new Product("PROD001", "Widget");
        Inventory inventory = new Inventory();
        inventory.setStock(product, 5);
        
        OrderService orderService = new OrderService(inventory);
        OrderRequest request = new OrderRequest(product, 10);  // Want 10, have 5
        
        // WHEN attempting to place the order
        OrderResult result = orderService.placeOrder(request);
        
        // THEN order should be rejected
        assertFalse(result.isSuccess());
        assertEquals("Insufficient inventory", result.getErrorMessage());
    }
}

// ========================================
// GWT WITH METHODS (CLEANER)
// ========================================

class GivenWhenThenWithMethods {
    
    @Test
    void shouldSendWelcomeEmailWhenUserRegisters() {
        // Given
        EmailService emailService = givenEmailServiceIsConfigured();
        UserService userService = givenUserService(emailService);
        
        // When
        User user = whenUserRegisters(userService, "newuser@example.com");
        
        // Then
        thenWelcomeEmailShouldBeSent(emailService, user);
    }
    
    private EmailService givenEmailServiceIsConfigured() {
        return mock(EmailService.class);
    }
    
    private UserService givenUserService(EmailService emailService) {
        UserRepository repository = new InMemoryUserRepository();
        return new UserService(repository, emailService);
    }
    
    private User whenUserRegisters(UserService service, String email) {
        return service.register(email, "John", "Doe");
    }
    
    private void thenWelcomeEmailShouldBeSent(EmailService emailService, User user) {
        verify(emailService).sendWelcomeEmail(user.getEmail());
    }
}

// ========================================
// GWT FOR COMPLEX WORKFLOWS
// ========================================

class ComplexWorkflowExample {
    
    @Test
    void shouldProcessRefundWhenOrderIsCancelled() {
        // GIVEN an order that has been paid
        User user = givenUserExists("john@example.com");
        Order order = givenOrderExists(user, OrderStatus.PAID);
        PaymentService paymentService = givenPaymentServiceIsConfigured();
        
        OrderService orderService = new OrderService(
            mock(OrderRepository.class),
            paymentService,
            mock(EmailService.class)
        );
        
        // WHEN user cancels the order
        CancellationResult result = orderService.cancelOrder(order.getId(), user);
        
        // THEN refund should be processed
        assertTrue(result.isSuccess());
        assertEquals(OrderStatus.CANCELLED, result.getOrder().getStatus());
        verify(paymentService).refund(user, order.getTotal());
    }
    
    private User givenUserExists(String email) {
        return new User(email, "John Doe");
    }
    
    private Order givenOrderExists(User user, OrderStatus status) {
        Order order = new Order(Arrays.asList(
            new OrderItem("PROD001", 1, new BigDecimal("50.00"))
        ));
        order.setStatus(status);
        order.setUser(user);
        return order;
    }
    
    private PaymentService givenPaymentServiceIsConfigured() {
        PaymentService service = mock(PaymentService.class);
        when(service.refund(any(), any())).thenReturn(true);
        return service;
    }
}

class AAAvsGWTComparison {
    
    /**
     * AAA STYLE (Technical)
     */
    @Test
    void aaa_shouldCalculateTotal() {
        // Arrange
        ShoppingCart cart = new ShoppingCart(mock(DiscountService.class));
        cart.addItem(new Item("Widget", new BigDecimal("10.00")));
        
        // Act
        BigDecimal total = cart.getTotal();
        
        // Assert
        assertEquals(new BigDecimal("10.00"), total);
    }
    
    /**
     * GWT STYLE (Business-focused)
     */
    @Test
    void gwt_shouldCalculateTotal() {
        // Given a cart with one item
        ShoppingCart cart = new ShoppingCart(mock(DiscountService.class));
        cart.addItem(new Item("Widget", new BigDecimal("10.00")));
        
        // When calculating the total
        BigDecimal total = cart.getTotal();
        
        // Then the total should equal the item price
        assertEquals(new BigDecimal("10.00"), total);
    }
    
    /**
     * WHEN TO USE EACH:
     * 
     * AAA:
     * ✓ Technical tests
     * ✓ Unit tests
     * ✓ Developer-focused
     * ✓ Quick to write
     * 
     * GWT:
     * ✓ Behavior-focused tests
     * ✓ Integration tests
     * ✓ Business stakeholder communication
     * ✓ Living documentation
     * 
     * BOTH:
     * ✓ Clear structure
     * ✓ Easy to read
     * ✓ Single responsibility
     */
}
```

## 4. Test Independence

### Each Test Stands Alone

```java
/**
 * TEST INDEPENDENCE
 * 
 * Tests should be independent - no order dependencies
 * Each test should set up its own state
 */

// ========================================
// ❌ BAD: Dependent tests
// ========================================

class DependentTests {
    private static User createdUser;  // ❌ Shared state!
    
    @Test
    void test1_createUser() {
        // ❌ Test depends on running first
        UserService service = new UserService();
        createdUser = service.createUser("test@example.com", "Test");
        assertNotNull(createdUser);
    }
    
    @Test
    void test2_updateUser() {
        // ❌ Depends on test1 running first!
        UserService service = new UserService();
        createdUser.setName("Updated");
        service.updateUser(createdUser);
    }
    
    @Test
    void test3_deleteUser() {
        // ❌ Depends on test1 and test2!
        UserService service = new UserService();
        service.deleteUser(createdUser.getEmail());
    }
    
    /**
     * PROBLEMS:
     * ✗ Tests must run in order
     * ✗ Can't run tests in parallel
     * ✗ One failure breaks all subsequent tests
     * ✗ Can't run single test in isolation
     * ✗ Hard to debug
     */
}

// ========================================
// ✓ GOOD: Independent tests
// ========================================

class IndependentTests {
    
    @Test
    void shouldCreateUserWithValidData() {
        // Arrange - fresh state
        UserService service = new UserService();
        
        // Act
        User user = service.createUser("test@example.com", "Test");
        
        // Assert
        assertNotNull(user);
        assertEquals("test@example.com", user.getEmail());
    }
    
    @Test
    void shouldUpdateExistingUser() {
        // Arrange - create own test data
        UserService service = new UserService();
        User user = service.createUser("test@example.com", "Original");
        
        // Act
        user.setName("Updated");
        User updated = service.updateUser(user);
        
        // Assert
        assertEquals("Updated", updated.getName());
    }
    
    @Test
    void shouldDeleteExistingUser() {
        // Arrange - create own test data
        UserService service = new UserService();
        User user = service.createUser("test@example.com", "Test");
        
        // Act
        boolean deleted = service.deleteUser(user.getEmail());
        
        // Assert
        assertTrue(deleted);
    }
    
    /**
     * BENEFITS:
     * ✓ Each test is self-contained
     * ✓ Can run in any order
     * ✓ Can run in parallel
     * ✓ One failure doesn't affect others
     * ✓ Easy to debug
     */
}

        @BeforeAll
        static void setUpClass() {
            database = new EmbeddedDatabase();
```

## 5. Avoiding Test Interdependencies

### No Test Depends on Another

```java
/**
 * AVOIDING TEST INTERDEPENDENCIES
 * 
 * Each test should be completely independent
 * No test should depend on another test running first

// ========================================
// ❌ ANTI-PATTERN 1: Sequential tests
// ========================================

class SequentialTestsAntiPattern {
    private static OrderService orderService;
    private static Order createdOrder;
    
    @BeforeAll
    static void setUp() {
        orderService = new OrderService();
    }
    
    @Test
    @Order(1)  // ❌ Must run first!
    void step1_createOrder() {
        createdOrder = orderService.createOrder("user@example.com");
        assertNotNull(createdOrder);
    }
    
    @Test
    @Order(2)  // ❌ Depends on step1!
    void step2_confirmOrder() {
        orderService.confirmOrder(createdOrder.getId());
        assertEquals(OrderStatus.CONFIRMED, createdOrder.getStatus());
    }
    
    @Test
    @Order(3)  // ❌ Depends on step2!
    void step3_shipOrder() {
        orderService.shipOrder(createdOrder.getId());
        assertEquals(OrderStatus.SHIPPED, createdOrder.getStatus());
    }
    
    /**
     * PROBLEMS:
     * ✗ Can't run tests independently
     * ✗ Can't run in parallel
     * ✗ Hard to debug (must run all steps)
```

## 7. Test Pyramid

### The Optimal Testing Strategy

```java
/**
 * TEST PYRAMID
 * 
 * Optimal distribution of tests:
 * 
 *           /\
 *          /E2E\        10% - End-to-End Tests
 *         /______\      (Slow, Brittle, High-level)
 *        /        \
 *       /Integration\   20% - Integration Tests
 *      /____________\   (Medium, More reliable)
 *     /              \
 *    /   Unit Tests   \ 70% - Unit Tests
 *   /__________________\ (Fast, Reliable, Low-level)
 */

// ========================================
// LAYER 1: UNIT TESTS (70%)
// ========================================

/**
 * UNIT TESTS: Test individual classes in isolation
 * 
 * Characteristics:
 * • Fast (< 10ms)
 * • Isolated (mocked dependencies)
 * • Focused (single class/method)
 * • Many tests (70% of total)
 */
class UnitTestLayer {
    
    @Test
    void shouldCalculateTotalPrice() {
        // Test single method in isolation
        PriceCalculator calculator = new PriceCalculator();
        
        BigDecimal price = calculator.calculateTotal(
            new BigDecimal("100.00"),
            0.10  // 10% discount
        );
        
        assertEquals(new BigDecimal("90.00"), price);
    }
    
    @Test
    void shouldValidateEmailFormat() {
        // Test single validation rule
        EmailValidator validator = new EmailValidator();
        
        assertTrue(validator.isValid("test@example.com"));
        assertFalse(validator.isValid("invalid"));
    }
    
    @Test
    void shouldAddItemToCart() {
        // Test single operation
        ShoppingCart cart = new ShoppingCart();
        
        cart.addItem(new Item("Widget", BigDecimal.TEN));
        
        assertEquals(1, cart.getItemCount());
    }
    
    /**
     * BENEFITS:
     * ✓ Fast feedback (seconds for entire suite)
     * ✓ Pinpoint failures (exact method that failed)
     * ✓ Easy to write (no complex setup)
     * ✓ Stable (no flakiness from I/O)
     * 
     * LIMITATIONS:
     * ✗ Don't catch integration issues
     * ✗ Don't test full workflows
     * ✗ Don't test configuration
     */
}

// ========================================
// LAYER 2: INTEGRATION TESTS (20%)
// ========================================

/**
 * INTEGRATION TESTS: Test multiple components together
```

```java
// ========================================
// 1. ARRANGE-ACT-ASSERT (AAA)
// ========================================

@Test
void testExample() {
    // ARRANGE: Set up test data
    ShoppingCart cart = new ShoppingCart();
    cart.addItem(new Item("Widget", BigDecimal.TEN));
    
    // ACT: Execute code under test
    BigDecimal total = cart.getTotal();
    
    // ASSERT: Verify outcome
    assertEquals(BigDecimal.TEN, total);
}

// ========================================
// 2. GIVEN-WHEN-THEN (GWT)
// ========================================

@Test
void shouldApplyDiscount() {
    // GIVEN a premium user with items in cart
    User user = createPremiumUser();
    ShoppingCart cart = createCartWithItems();
    
    // WHEN calculating total
    BigDecimal total = cart.getTotal();
    
    // THEN discount should be applied
    assertTrue(total.compareTo(originalTotal) < 0);
}

// ========================================
// 3. TEST NAMING
// ========================================

// ✓ GOOD NAMES
@Test void shouldCalculateTotalWhenCartHasItems() {}
@Test void shouldThrowExceptionWhenEmailIsInvalid() {}
@Test void whenUserIsPremium_shouldApplyDiscount() {}

// ❌ BAD NAMES
@Test void test1() {}
@Test void testMethod() {}
@Test void verifyBehavior() {}

// ========================================
// 4. TEST INDEPENDENCE
// ========================================

// ✓ GOOD: Each test creates own data
@Test
void test1() {
    User user = new User("test@example.com", "Test");
    // test...
}

@Test
void test2() {
    User user = new User("test@example.com", "Test");
    // test...
}

// ❌ BAD: Tests share state
private static User sharedUser;

@Test
void test1() {
    sharedUser = new User("test@example.com", "Test");
}

@Test
void test2() {
    // Uses sharedUser from test1!
}

// ========================================
// 5. NO INTERDEPENDENCIES
// ========================================

// ❌ BAD: Sequential tests
@Test @Order(1)
void step1_create() { /* creates resource */ }

@Test @Order(2)
void step2_modify() { /* modifies resource from step1 */ }

// ✓ GOOD: Independent tests
@Test
void shouldModifyResource() {
    Resource resource = createResource();  // Own setup
    modify(resource);
    // test...
}

// ========================================
// 6. FAST vs SLOW TESTS
// ========================================

// FAST (< 10ms) - Unit tests with mocks
@Tag("fast")
@Test
void unitTest() {
    Calculator calc = new Calculator();
    assertEquals(4, calc.add(2, 2));
}

// MEDIUM (10ms-1s) - Integration tests
@Tag("integration")

// 7. TEST PYRAMID

/**
 * OPTIMAL DISTRIBUTION:
 * 
 * 70% Unit Tests (fast, isolated)
 * 20% Integration Tests (medium, real components)
 * 10% E2E Tests (slow, full system)
 * 
 * RUN:
 * - Unit: Every change
 * - Integration: Before commit
 * - E2E: Before deployment
 */

// ========================================
// THE FIRST PRINCIPLE
// ========================================

/**
 * FIRST:
 * 
 * F - Fast
 *     Tests run quickly (milliseconds for unit tests)
 * 
 * I - Independent
 *     No order dependencies, can run in parallel
 * 
 * R - Repeatable
 *     Same result every time, no flakiness
 * 
 * S - Self-Validating
 *     Pass or fail, no manual inspection
 * 
 * T - Timely
 *     Written with or before code (TDD)
 */

// ========================================
// BEST PRACTICES CHECKLIST
// ========================================

/**
 * DO:
 * ✓ Use AAA/GWT structure
 * ✓ Write descriptive test names
 * ✓ Keep tests independent
 * ✓ Test one thing per test
 * ✓ Use proper test pyramid
 * ✓ Make tests fast
 * ✓ Clean up after tests
 * ✓ Test behavior, not implementation
 * 
 * DON'T:
 * ✗ Share mutable state
 * ✗ Create test dependencies
 * ✗ Use generic names (test1, testMethod)
 * ✗ Test multiple things in one test
 * ✗ Invert the pyramid (too many E2E tests)
 * ✗ Write slow unit tests
 * ✗ Leave tests broken
 * ✗ Test implementation details
 */

// ========================================
// DECISION GUIDE
// ========================================

/**
 * What are you testing?
 * 
 * Pure logic/calculation
 * → UNIT TEST (mocks, fast)
 * 
 * Database queries
 * → INTEGRATION TEST (real DB)
 * 
 * External API calls
 * → INTEGRATION TEST (test service)
 * 
 * Full user workflow
 * → E2E TEST (UI + backend)
 * 
 * How to structure?
 * → AAA (technical) or GWT (business)
 * 
 * How to name?
 * → should[Action]When[Condition]
 * → when[Condition]_should[Action]
 * → given[Context]_when[Action]_then[Outcome]
 * 
 * How many tests?
 * → Follow pyramid: 70% unit, 20% integration, 10% E2E
 */
```

**Key Takeaways:**

```
The FIRST Principle:
F - Fast (milliseconds)
I - Independent (any order)
R - Repeatable (no flakiness)
S - Self-Validating (pass/fail)
T - Timely (with code)

The Test Pyramid:
        /\
       /E2E\     10% - Slow, Brittle
      /______\   
     /Integration\ 20% - Medium
    /____________\
   /   Unit Tests  \ 70% - Fast, Reliable
  /__________________\

Structure Your Tests:
ARRANGE: Set up
ACT: Execute
ASSERT: Verify

Name Your Tests:
should[Action]When[Condition]
"shouldApplyDiscountWhenUserIsPremium"

Keep Tests Independent:
✓ Fresh data each test
✗ No shared mutable state
✗ No order dependencies
```
