# Mutation Testing

## Overview

Mutation testing measures test quality by introducing bugs (mutations) into your code and checking if your tests catch them. Traditional code coverage (line/branch) shows which code is executed but not if tests actually verify behavior—100% coverage with no assertions still passes. Mutation testing solves this: it creates modified versions (mutants) of your code with subtle bugs, runs your test suite against each mutant, and checks if tests fail. A killed mutant (test fails) is good—your tests caught the bug. A survived mutant (test passes) is bad—your tests missed the bug. This guide covers five essential concepts: PITest Configuration (setup, Maven/Gradle integration, targeting specific classes, exclusions, performance tuning), Mutation Operators (types of mutations—arithmetic operators, conditionals, return values, increments, what each reveals about tests), Interpreting Mutation Scores (coverage vs mutation score, what 80% means, when 100% is unrealistic, focusing on critical code), Improving Test Quality (using mutation results to write better tests, killing survivors, assertion strengthening, edge case discovery), and CI Integration (running mutation tests in pipeline, incremental analysis, reporting, balancing speed vs thoroughness). The key insight: code coverage measures test execution, mutation testing measures test effectiveness. You can have 100% code coverage with weak tests—tests that run code but don't verify correctness. Mutation testing finds these weak spots. A good mutation score (75-85%) indicates strong tests that actually verify behavior, catch edge cases, and would detect real bugs. Understanding mutation testing means understanding the difference between "tests run this code" and "tests verify this code works correctly". PITest is the industry-standard mutation testing tool for Java—mature, fast, integrates with build tools, supports custom mutations. The workflow: write code → write tests → run PITest → see which mutations survived → strengthen tests to kill survivors → repeat. This guide shows when mutation testing adds value (critical algorithms, complex logic, public APIs) and when it's overkill (simple getters, UI code, third-party integration).

## 1. What is Mutation Testing?

### Beyond Code Coverage

```java
/**
 * MUTATION TESTING: Testing Your Tests
 * 
 * Traditional coverage: "Did tests execute this code?"
 * Mutation testing: "Would tests catch bugs in this code?"
 */

// ========================================
// THE PROBLEM WITH CODE COVERAGE
// ========================================

class Calculator {
    public int add(int a, int b) {
        return a + b;
    }
    
    public int multiply(int a, int b) {
        return a * b;
    }
}

/**
 * TEST WITH 100% CODE COVERAGE (but weak!)
 */
class CalculatorWeakTest {
    
    @Test
    void testAdd() {
        Calculator calc = new Calculator();
        calc.add(2, 3);  // ❌ No assertion!
        // 100% line coverage, but test verifies NOTHING
    }
    
    @Test
    void testMultiply() {
        Calculator calc = new Calculator();
        int result = calc.multiply(4, 5);
        assertTrue(result > 0);  // ❌ Weak assertion
        // 100% line coverage, but test is too weak
    }
    
    /**
     * CODE COVERAGE REPORT:
     * ✓ 100% line coverage
     * ✓ 100% branch coverage
     * 
     * But these tests are USELESS!
     * They don't verify correct behavior
     */
}

// ========================================
// HOW MUTATION TESTING WORKS
// ========================================

/**
 * STEP 1: PITest creates MUTANTS (modified code)
 * 
 * Original code:
 *   return a + b;
 * 
 * Mutant 1:
 *   return a - b;  // Changed + to -
 * 
 * Mutant 2:
 *   return a * b;  // Changed + to *
 * 
 * Mutant 3:
 *   return 0;      // Return constant
 */

/**
 * STEP 2: Run tests against each mutant
 * 
 * Mutant 1 (a - b):
 *   Tests pass? → SURVIVED (BAD! Test didn't catch bug)
 *   Tests fail? → KILLED (GOOD! Test caught bug)
 * 
 * Mutant 2 (a * b):
 *   Tests pass? → SURVIVED (BAD!)
 *   Tests fail? → KILLED (GOOD!)
 */

/**
 * STEP 3: Calculate mutation score
 * 
 * Mutation Score = (Killed Mutants / Total Mutants) × 100%
 * 
 * Example:
 * - 10 mutants created
 * - 8 mutants killed (tests failed)
 * - 2 mutants survived (tests passed)
 * - Mutation Score = 80%
 */

// ========================================
// STRONG TESTS (HIGH MUTATION SCORE)
// ========================================

/**
 * TEST WITH STRONG ASSERTIONS
 */
class CalculatorStrongTest {
    
    @Test
    void addShouldReturnSumOfTwoNumbers() {
        Calculator calc = new Calculator();
        
        // Test multiple cases
        assertEquals(5, calc.add(2, 3));
        assertEquals(0, calc.add(0, 0));
        assertEquals(-1, calc.add(-3, 2));
        assertEquals(100, calc.add(50, 50));
        
        /**
         * KILLS MUTANTS:
         * ✓ return a - b;  → KILLED (5 ≠ -1)
         * ✓ return a * b;  → KILLED (5 ≠ 6)
         * ✓ return 0;      → KILLED (5 ≠ 0)
         * ✓ return a;      → KILLED (5 ≠ 2)
         */
    }
    
    @Test
    void multiplyShouldReturnProductOfTwoNumbers() {
        Calculator calc = new Calculator();
        
        assertEquals(20, calc.multiply(4, 5));
        assertEquals(0, calc.multiply(0, 5));
        assertEquals(-12, calc.multiply(-3, 4));
        assertEquals(1, calc.multiply(1, 1));
        
        /**
         * KILLS MUTANTS:
         * ✓ return a + b;  → KILLED (20 ≠ 9)
         * ✓ return a - b;  → KILLED (20 ≠ -1)
         * ✓ return a / b;  → KILLED (20 ≠ 0.8)
         */
    }
}

// ========================================
// REAL-WORLD EXAMPLE
// ========================================

class PasswordValidator {
    
    public boolean isValid(String password) {
        if (password == null || password.length() < 8) {
            return false;
        }
        
        boolean hasUpper = password.chars().anyMatch(Character::isUpperCase);
        boolean hasLower = password.chars().anyMatch(Character::isLowerCase);
        boolean hasDigit = password.chars().anyMatch(Character::isDigit);
        
        return hasUpper && hasLower && hasDigit;
    }
}

/**
 * WEAK TEST (Mutants survive)
 */
class PasswordValidatorWeakTest {
    
    @Test
    void testValidPassword() {
        PasswordValidator validator = new PasswordValidator();
        boolean result = validator.isValid("Password1");
        // ❌ No assertion! Mutant survives
    }
}

/**
 * STRONG TEST (Kills mutants)
 */
class PasswordValidatorStrongTest {
    
    @Test
    void shouldAcceptValidPassword() {
        PasswordValidator validator = new PasswordValidator();
        assertTrue(validator.isValid("Password1"));
        // ✓ Kills mutant: return true; → return false;
    }
    
    @Test
    void shouldRejectPasswordShorterThan8() {
        PasswordValidator validator = new PasswordValidator();
        assertFalse(validator.isValid("Pass1"));
        // ✓ Kills mutant: length() < 8 → length() <= 8
    }
    
    @Test
    void shouldRejectPasswordWithoutUppercase() {
        PasswordValidator validator = new PasswordValidator();
        assertFalse(validator.isValid("password1"));
        // ✓ Kills mutant: hasUpper && hasLower → hasUpper || hasLower
    }
    
    @Test
    void shouldRejectPasswordWithoutDigit() {
        PasswordValidator validator = new PasswordValidator();
        assertFalse(validator.isValid("Password"));
        // ✓ Kills mutant: hasDigit required → not checked
    }
}

// ========================================
// KEY CONCEPTS
// ========================================

/**
 * TERMINOLOGY:
 * 
 * MUTANT: Modified version of code with one small change
 * KILLED: Test fails on mutant (good - test caught bug)
 * SURVIVED: Test passes on mutant (bad - test missed bug)
 * MUTATION SCORE: Percentage of mutants killed
 * 
 * MUTATION OPERATORS:
 * - Arithmetic: + → -, * → /, etc.
 * - Conditional: < → <=, == → !=, etc.
 * - Return values: return x; → return 0;
 * - Increments: ++ → --, etc.
 * - Negation: ! condition
 * 
 * GOALS:
 * ✓ 75-85% mutation score is excellent
 * ✓ 60-75% is good
 * ✓ < 60% indicates weak tests
 * ✓ 100% is often unrealistic/unnecessary
 */
```

## 3. Mutation Operators

### Types of Code Mutations

```java

 * MUTATION OPERATORS
 * 
 * Different ways PITest mutates code to test test quality

// ========================================
// 1. ARITHMETIC OPERATORS (MATH)
// ========================================

class ArithmeticMutations {
    
    /**
     * Original code
     */
    public int calculate(int a, int b) {
        return a + b;
    }
    
    /**
     * MUTANT 1: + → -
     */
    // return a - b;
    
    /**
     * MUTANT 2: + → *
     */
    // return a * b;
    
    /**
     * MUTANT 3: + → /
     */
    // return a / b;
    
    /**
     * MUTANT 4: + → %
     */
    // return a % b;
    
    /**
     * TEST TO KILL THESE:
     */
    @Test
    void testCalculate() {
        assertEquals(5, calculate(2, 3));  // Kills all 4 mutants
    }
}

// ========================================
// 2. CONDITIONAL BOUNDARIES
// ========================================

class ConditionalBoundaryMutations {
    
    /**
     * Original code
     */
    public boolean isAdult(int age) {
        return age >= 18;
    }
    
    /**
     * MUTANT 1: >= → >
     */
    // return age > 18;
    
    /**
     * MUTANT 2: >= → ==
     */
    // return age == 18;
    
    /**
     * TESTS TO KILL THESE:
     */
    @Test
    void testBoundary() {
        assertTrue(isAdult(18));   // Kills mutant 1 (> would fail)
        assertTrue(isAdult(19));   // Kills mutant 2 (== would fail)
        assertFalse(isAdult(17));  // Kills original logic errors
    }
}

// 3. NEGATE CONDITIONALS
// ========================================

class NegateConditionalMutations {
    
    /**
     * Original code
     */
    public boolean isPositive(int n) {
        return n > 0;
    }
    
    /**
     * MUTANT 1: > → <=
     */
    // return n <= 0;
    
    /**
     * MUTANT 2: > → >=
     */
    // return n >= 0;
    
    /**
     * MUTANT 3: > → <
     */
    // return n < 0;
    
    /**
     * TESTS TO KILL THESE:
     */
    @Test
    void testPositive() {
        assertTrue(isPositive(1));    // Kills mutants 1 & 3
        assertFalse(isPositive(0));   // Kills mutant 2
        assertFalse(isPositive(-1));  // Additional verification
    }
}

     * Original code
     */
    public int increment(int n) {
        return n++;  // Post-increment
    }
    
    /**
     * MUTANT 1: n++ → n--
     */
    // return n--;
    
    /**
     * MUTANT 2: n++ → ++n (pre-increment)

    // return ++n;
    
    /**
     * TEST TO KILL THESE:
     */
    @Test

        assertEquals(5, increment(5));  // Kills mutant 1 (n-- returns 5 then decrements)
        // Mutant 2 survives (++n also returns 6)
    }
    
    /**
     * Better version to kill all mutants
     */
    public int incrementBetter(int n) {
        return n + 1;
    }
}
```

## 4. Interpreting Mutation Scores

### Understanding the Results

```java
/**
 * INTERPRETING MUTATION SCORES
 * 
 * Understanding what PITest reports mean
 */

// ========================================
// PITEST REPORT STRUCTURE
// ========================================

/**
 * SAMPLE PITEST OUTPUT:
 * 
 * ================================================================================
 * - Mutators
 * ================================================================================
 * > org.pitest.mutationtest.engine.gregor.mutators.MathMutator
 * > org.pitest.mutationtest.engine.gregor.mutators.ReturnValsMutator
 * > org.pitest.mutationtest.engine.gregor.mutators.ConditionalsBoundaryMutator
 * 
 * ================================================================================
 * - Statistics
 * ================================================================================
 * >> Generated 127 mutations Killed 98 (77%)
 * >> Ran 142 tests (1.12 tests per mutation)
 * 
 * ================================================================================
 * - Coverage
 * ================================================================================
 * Line Coverage: 95%
 * Mutation Coverage: 77%
 */

// ========================================
// MUTATION SCORE INTERPRETATION
// ========================================

class MutationScoreGuide {
    
    /**
     * SCORE RANGES:
     * 
     * 85-100%: Excellent
     * - Very strong test suite
     * - Tests verify behavior thoroughly
     * - Few weak spots
     * 
     * 75-84%: Good
     * - Strong test suite
     * - Most behavior verified
     * - Some room for improvement
     * 
     * 60-74%: Acceptable
     * - Decent test suite
     * - Missing some edge cases
     * - Should improve for critical code
     * 
     * 40-59%: Weak
     * - Tests execute code but don't verify much
     * - Many assertions missing
     * - Needs significant improvement
     * 
     * <40%: Very Weak
     * - Tests barely verify anything
     * - May have no assertions
     * - Urgent need for better tests
     */
}

// ========================================
// EXAMPLE: ANALYZING MUTATION REPORT
// ========================================

class DiscountCalculator {
    
    public double calculateDiscount(double price, boolean isPremium) {
        if (price < 0) {
            throw new IllegalArgumentException("Price cannot be negative");
        }
        
        if (isPremium) {
            return price * 0.20;  // 20% discount
        }
        
        if (price > 100) {
            return price * 0.10;  // 10% discount
        }
        
        return 0;  // No discount
    }
}

/**
 * WEAK TESTS (60% mutation score)
 */
class DiscountCalculatorWeakTest {
    
    @Test
    void testDiscount() {
        DiscountCalculator calc = new DiscountCalculator();
        
        // Only tests one path
        double discount = calc.calculateDiscount(50, true);
        assertTrue(discount > 0);  // ❌ Weak assertion
        
        /**
         * SURVIVED MUTANTS:
         * 
         * 1. price * 0.20 → price * 0.10  (SURVIVED)
         * 2. price * 0.20 → price * 0.30  (SURVIVED)
         * 3. isPremium → !isPremium       (SURVIVED)
         * 4. price > 100 → price >= 100   (SURVIVED)
         * 5. return 0 → return 1          (SURVIVED)
         * 
         * Only 4 out of 10 mutants killed = 40% score
         */
    }
}

/**
 * STRONG TESTS (95% mutation score)
 */
class DiscountCalculatorStrongTest {
    
    @Test
    void shouldCalculatePremiumDiscount() {
        DiscountCalculator calc = new DiscountCalculator();
```

**Key Takeaways:**

```
The Core Idea:
Code Coverage: "Did tests RUN this code?"
Mutation Testing: "Do tests VERIFY this code?"

The Process:
1. PITest creates mutants (code with bugs)
2. Runs tests against each mutant
3. Tests fail → KILLED (good!)
4. Tests pass → SURVIVED (bad!)

Mutation Score = Killed / Total × 100%

Score Interpretation:
85-100%: Excellent
75-84%:  Good ← Target for critical code
60-74%:  Acceptable
<60%:    Weak

Common Mutations:
+ → -         (arithmetic)
< → <=        (boundary)
return x → 0  (return value)
x && y → x || y (conditional)
```
