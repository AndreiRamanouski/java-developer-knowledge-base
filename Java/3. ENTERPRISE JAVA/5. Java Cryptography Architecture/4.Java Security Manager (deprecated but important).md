# Java Security Manager (Deprecated)

## Overview

The Java Security Manager was the original mechanism for sandboxing untrusted code in Java—important for understanding legacy systems and the evolution toward modern security architectures. This guide covers five critical Security Manager concepts: Security Policies (defining what code can do—grant permissions based on code source, signed JARs can have more permissions, policy files in grant/permission syntax, centralized security configuration), Permissions (fine-grained access control—FilePermission for file access, SocketPermission for network, RuntimePermission for JVM operations, PropertyPermission for system properties, each permission has name and actions), Sandboxing Untrusted Code (running third-party code safely—Java applets in browsers used this, prevent arbitrary file access, prevent network connections, prevent system property changes, enable/disable with -Djava.security.manager), Why It's Deprecated (JEP 411 reasoning—rarely used in practice, maintenance burden for JVM developers, difficult to configure correctly, false sense of security, better alternatives exist, deprecated Java 17, removed Java 21), and Alternatives (modern security approaches—containers provide better isolation, OS-level security (SELinux, AppArmor), process isolation, least privilege principles, security by design not security manager). The key insight: Security Manager was designed for 1990s threat model (untrusted applets) but modern applications have different security needs—containers and OS-level security provide better isolation with less complexity. Common Security Manager mistakes: overly permissive policies (grants too much), complex policies hard to maintain, conflicts with libraries expecting full permissions, performance overhead from permission checks, assuming it provides complete security (it doesn't), not testing with Security Manager enabled. The relationship between concepts: Security Policy defines rules → Permissions grant specific capabilities → Security Manager enforces at runtime → Code is sandboxed → Modern alternatives (containers) provide better isolation without JVM complexity. Real-world migration strategy: legacy apps with Security Manager → containerize for isolation → use OS security (SELinux/AppArmor) → principle of least privilege → security by design in application layer → remove Security Manager dependency → upgrade to Java 17+ with confidence.

## 2. Permissions

### Fine-Grained Access Control

```java
/**
 * PERMISSIONS: Specific capabilities granted to code
 * 
 * Common permission types:
 * - FilePermission: File system access
 * - SocketPermission: Network access
 * - RuntimePermission: JVM operations
 * - PropertyPermission: System properties
 * - SecurityPermission: Security subsystem
 */

import java.io.*;
import java.net.*;
import java.security.*;
import java.util.PropertyPermission;

// ========================================
// FILEPERMISSION
// ========================================

public class FilePermissionExample {
    
    /**
     * FilePermission syntax:
     * new FilePermission("target", "actions")
     * 
     * Targets:
     * - "/path/file.txt" - specific file
     * - "/path/*" - all files in directory
     * - "/path/-" - all files recursively
     * - "<<ALL FILES>>" - all files on system
     * 
     * Actions:
     * - read, write, delete, execute
     */
    
    public void demonstrateFilePermission() {
        SecurityManager sm = System.getSecurityManager();
        if (sm == null) {
            System.out.println("No Security Manager");
            return;
        }
        
        try {
            // Check if we can read a file
            sm.checkPermission(new FilePermission("/tmp/test.txt", "read"));
            System.out.println("READ permission granted");
            
            // Try to read the file
            File file = new File("/tmp/test.txt");
            if (file.exists()) {
                System.out.println("File exists and can be accessed");
            }
            
        } catch (SecurityException e) {
            System.out.println("READ permission denied: " + e.getMessage());
        }
        
        try {
            // Check if we can write to a file
            sm.checkPermission(new FilePermission("/etc/passwd", "write"));
            System.out.println("WRITE to /etc/passwd granted (DANGEROUS!)");
            
        } catch (SecurityException e) {
            System.out.println("WRITE to /etc/passwd denied (GOOD!)");
        }
    }
    
    /**
     * Example policy for FilePermission:
     */
    /*
    grant {
        // Allow reading from application directory
        permission java.io.FilePermission "/app/data/-", "read";
        
        // Allow read/write to temp directory

        // Allow read/write to specific log file

     * 
     * Port:
     * - "8080" - specific port

     * 
     * Actions:
     * - connect, accept, listen, resolve
     */

        SecurityManager sm = System.getSecurityManager();
        if (sm == null) {
            System.out.println("No Security Manager");
            return;

        try {
            // Check if we can connect to a host
            sm.checkPermission(new SocketPermission("example.com:80", "connect"));
            System.out.println("CONNECT to example.com:80 granted");
            
            // Try to make connection
            URL url = new URL("http://example.com");
            url.openConnection();
            System.out.println("Connection successful");
            
        } catch (SecurityException e) {
            System.out.println("CONNECT denied: " + e.getMessage());
        } catch (IOException e) {
            System.out.println("Connection failed: " + e.getMessage());

            // Check if we can listen on a port

        } catch (SecurityException e) {
            System.out.println("LISTEN denied: " + e.getMessage());
        } catch (IOException e) {
            System.out.println("Server socket failed: " + e.getMessage());
        }
    }

     * Example policy for SocketPermission:
```

## 3. Sandboxing Untrusted Code

### Running Third-Party Code Safely

```java
/**
 * SANDBOXING: Run untrusted code with restricted permissions
 * 
 * Original use case: Java applets in web browsers
 * Modern use case: Plugin systems, script engines
 */

import java.security.*;

// ========================================
// SANDBOXING UNTRUSTED CODE
// ========================================

public class SandboxExample {
    
    /**
     * Load and execute untrusted code in sandbox
     */
    @Deprecated
    public void runUntrustedCode(String jarPath) throws Exception {
        // Enable Security Manager
        if (System.getSecurityManager() == null) {
            System.setSecurityManager(new SecurityManager());
        }
        
        // Load untrusted code
        URL[] urls = {new File(jarPath).toURI().toURL()};
        URLClassLoader loader = new URLClassLoader(urls);
        
        // Load class from untrusted JAR
        Class<?> pluginClass = loader.loadClass("com.untrusted.Plugin");
        
        // Create instance
        Object plugin = pluginClass.getDeclaredConstructor().newInstance();
        
        // Execute method (with security restrictions)
        try {
            pluginClass.getMethod("execute").invoke(plugin);
            System.out.println("Plugin executed successfully");
            
        } catch (InvocationTargetException e) {
            Throwable cause = e.getCause();
            if (cause instanceof SecurityException) {
                System.out.println("Plugin tried forbidden operation: " + cause.getMessage());
            } else {
                throw e;
            }
        }
    }
    
    /**
     * Policy for untrusted code:
     */
    /*
    grant codeBase "file:/path/to/untrusted/*" {
        // Minimal permissions for untrusted code
        
        // Allow reading own directory
        permission java.io.FilePermission "/path/to/untrusted/-", "read";
        
        // Allow reading system properties
        permission java.util.PropertyPermission "java.version", "read";
        permission java.util.PropertyPermission "os.name", "read";
        
        // NO file write permissions
        // NO network permissions
        // NO runtime permissions
        // NO system operations
```

## 4. Why Security Manager is Deprecated

### The End of an Era (JEP 411)

```java
/**
 * JEP 411: Deprecation of the Security Manager for Removal
 * 
 * Timeline:
 * - Java 17 (September 2021): Deprecated for removal
 * - Java 18+: Disabled by default
 * - Java 21 (September 2023): Removed completely
 * 
 * Official reasoning from JEP 411:
 */

// ========================================
// REASONS FOR DEPRECATION
// ========================================

/**
 * 1. RARELY USED
 * 
 * - Security Manager designed for Java applets (1990s)
 * - Applets are dead (removed in Java 11)
 * - Very few applications use Security Manager
 * - Most Java applications run in trusted environments
 * 
 * Survey results:
 * - <3% of Java applications use Security Manager
 * - Most usage is accidental (default policies)
 * - Real sandboxing needs moved to containers
 */

/**
 * 2. MAINTENANCE BURDEN
 * 
 * - Every JDK feature must consider Security Manager
 * - Permission checks slow down development
 * - Complex interaction with modern Java features:
 *   - Modules (Java 9+)
 *   - Records (Java 14+)
 *   - Pattern matching
 *   - Virtual threads (Java 19+)
 * 
 * Example complexity:
 * - Every new API needs security checks
 * - Every change needs policy updates
 * - Testing matrix explodes (with/without SM)
 */

/**
 * 3. DIFFICULT TO USE CORRECTLY
 * 
 * - Policy files are complex
 * - Permissions are confusing
 * - Easy to misconfigure
 * - Hard to debug
 * 
 * Common mistakes:
 */
public class SecurityManagerMistakes {
    
    // ❌ Mistake 1: Overly permissive policy
    /*
    grant {
        permission java.security.AllPermission;  // Grants EVERYTHING!
    };
    */
    
    // ❌ Mistake 2: Missing permissions
    /*
    // Application fails at runtime with SecurityException
    // No clear indication of which permission is needed
    java.lang.SecurityException: access denied ("java.io.FilePermission" "/tmp/data" "read")
    */
    
    // ❌ Mistake 3: Conflicts with libraries
    /*
    // Many libraries assume they have full permissions
    // Fail mysteriously when Security Manager is enabled
    // Examples: Hibernate, Spring, logging frameworks
    */
    
    // ❌ Mistake 4: Debugging nightmare
    /*
    // Enable debugging: -Djava.security.debug=access,failure
    // Output is verbose and hard to parse
    // Thousands of permission checks per request
    */
}

/**
 * 4. FALSE SENSE OF SECURITY
 * 
 * - Security Manager is NOT a complete security solution
 * - Cannot prevent all attacks:
 *   - Resource exhaustion (CPU, memory)
 *   - Timing attacks
 *   - Side-channel attacks
 *   - Bugs in trusted code
 * 
 * - Modern threats require different approaches:
 *   - Container isolation
 *   - Network segmentation
 *   - Input validation
 *   - Encryption
 */

/**
 * 5. BETTER ALTERNATIVES EXIST
 * 
 * Modern security approaches:
 * 
 * a) Containers (Docker, Kubernetes)
 *    - Complete process isolation
 *    - Resource limits
 *    - Network isolation
 *    - Better than Security Manager in every way
 * 
 * b) OS-level security
 *    - SELinux (Linux)
 *    - AppArmor (Linux)
 *    - Mandatory Access Control
 *    - Proven, battle-tested
 * 
 * c) Virtualization
 *    - VM-level isolation
 *    - Strong security boundaries
 *    - Separate kernel space
 * 
 * d) Security by design
 *    - Input validation
 *    - Least privilege
 *    - Defense in depth
 *    - No reliance on runtime checks
 */

// ========================================
// MIGRATION IMPACT
// ========================================

/**

 * Java 17 (LTS):
 * - Security Manager still works

 * - Disabled by default
```

## 5. Modern Alternatives

### Better Security Approaches

```java
// ========================================
// ALTERNATIVE 1: CONTAINERS
// ========================================

/**
 * Docker provides better isolation than Security Manager
 * 
 * Dockerfile example:
 */
/*
FROM eclipse-temurin:21-jre

# Create non-root user
RUN groupadd -r appuser && useradd -r -g appuser appuser

# Copy application
COPY --chown=appuser:appuser myapp.jar /app/myapp.jar

# Run as non-root
USER appuser

# Set working directory
WORKDIR /app

# Expose port
EXPOSE 8080

# Run application
ENTRYPOINT ["java", "-jar", "myapp.jar"]
*/

/**
 * Docker run with security options:
 * 
 * docker run \
 *   --read-only \                          # Read-only filesystem
 *   --tmpfs /tmp \                         # Writable /tmp
 *   --security-opt=no-new-privileges \     # Prevent privilege escalation
 *   --cap-drop=ALL \                       # Drop all capabilities
 *   --cap-add=NET_BIND_SERVICE \           # Add only needed capabilities
 *   -m 512m \                              # Memory limit
 *   --cpus=2 \                             # CPU limit
 *   --pids-limit=100 \                     # Process limit
 *   myapp:latest
 * 
 * Benefits over Security Manager:

/**
 * Run untrusted code in separate process
 */
public class ProcessIsolation {
    
    /**
     * Execute untrusted code in isolated process
     */
    public String executeInIsolatedProcess(String script) throws Exception {
        // Create process builder
        ProcessBuilder pb = new ProcessBuilder(
            "java",
            "-Xmx128m",                    // Memory limit
            "-XX:MaxMetaspaceSize=64m",    // Metaspace limit
            "-cp", "untrusted.jar",
            "com.untrusted.ScriptRunner",
            script
        );
        
        // Set working directory
        pb.directory(new File("/tmp/sandbox"));
        
        // Limit environment
        pb.environment().clear();
        pb.environment().put("HOME", "/tmp/sandbox");
        
        // Start process
        Process process = pb.start();
        
        // Set timeout
        boolean finished = process.waitFor(10, TimeUnit.SECONDS);
        
        if (!finished) {
            process.destroyForcibly();
            throw new TimeoutException("Script execution timeout");
        }
        
        // Get result
        int exitCode = process.exitValue();
        if (exitCode != 0) {
            throw new RuntimeException("Script failed with exit code: " + exitCode);
        }
        
        return new String(process.getInputStream().readAllBytes());
    }
    
    /**
     * Benefits:
     * - Complete isolation (separate JVM)
     * - Can kill process if misbehaves
     * - Memory limits enforced by OS
     * - No impact on main application
     * 
     * Drawbacks:
     * - Higher overhead (separate JVM)
     * - Slower startup
     * - More complex IPC
     */
}
```

```java
   Check permission:
   sm.checkPermission(new FilePermission("/tmp/file", "read"));

3. SANDBOXING
   • Run untrusted code with restrictions
   • Original use: Java applets
   • Modern use: Plugin systems
   • Enable: -Djava.security.manager
   
   Result: SecurityException if forbidden operation

   • Better alternatives exist
```
