# Dependency Management Strategies

## Overview

Dependency management is the art of controlling which library versions your project uses—it's critical because most Java applications depend on dozens or hundreds of external libraries, each with their own dependencies (transitive dependencies). This guide covers seven essential dependency management concepts: Transitive Dependencies (your dependencies have dependencies—dependency depth can reach 5+ levels, Maven/Gradle automatically resolve them, diamond problem when same library appears multiple times), Dependency Conflicts Resolution (when different versions requested, Maven uses "nearest wins", Gradle has sophisticated conflict resolution, understanding which version is actually used), Version Ranges (dynamic versions like [1.0,2.0) seem flexible but break reproducibility—avoid in production, pin specific versions instead), BOM (Bill of Materials—POM that defines versions for related dependencies, Spring Boot uses this pattern, import BOM and get consistent versions without specifying each), Dependency Exclusions (exclude unwanted transitive dependencies, fix conflicts by removing older versions, be surgical—only exclude when necessary), Enforcing Versions (dependency management in Maven, dependency constraints in Gradle, strict versions prevent downgrades, platform/BOM imports), and Vulnerability Scanning (OWASP Dependency Check finds CVEs, Snyk/GitHub Dependabot for continuous monitoring, failing builds on high-severity vulnerabilities). The key insight: dependency resolution is deterministic but not always intuitive—understanding how Maven/Gradle pick versions prevents "works on my machine" and production surprises. Common confusion: dependency management (Maven) doesn't add dependencies, it defines versions; exclusions remove from dependency tree, not from repository. The relationship between concepts: transitive dependencies create conflicts → conflict resolution determines winner → BOMs prevent conflicts proactively → exclusions remove problematic transitives → version enforcement locks down versions → vulnerability scanning finds security issues. Real-world strategy: use BOMs for frameworks (Spring Boot, Jackson), pin critical library versions in dependencyManagement, exclude known problematic transitives (commons-logging), scan for vulnerabilities in CI, use dependency locking for reproducible builds.

**Key Concepts**:

```
Dependency Depth:
Your App
└── Library A (1.0)
    └── Library B (2.0)
        └── Library C (3.0)
            └── Library D (4.0)

Your app gets all four!

Diamond Problem:
Your App
├── Library A → Commons-IO 2.0
└── Library B → Commons-IO 1.4

Which version wins? (Resolution strategy determines)
```

```kotlin
// ========================================

    // Single dependency pulls in many
    implementation("org.springframework.boot:spring-boot-starter-web:3.2.0")
}

// View dependency tree
// ./gradlew dependencies

// View specific configuration
// ./gradlew dependencies --configuration compileClasspath

// Insight on specific dependency
// ./gradlew dependencyInsight --dependency jackson-databind

// ========================================
// CONTROLLING TRANSITIVES
// ========================================

dependencies {
    implementation("org.springframework.boot:spring-boot-starter-web:3.2.0") {
        // Exclude specific transitive
        exclude(group = "ch.qos.logback")
    }
    
    // Disable all transitives
    implementation("com.example:library:1.0") {
        isTransitive = false
    }
}

// ========================================
// DEPENDENCY CONSTRAINTS (Gradle)
// ========================================

dependencies {
    constraints {
        // Enforce version for ALL occurrences
        implementation("com.google.guava:guava:32.1.3-jre")
    }
}

// Any transitive Guava uses this version
```

## 2. Dependency Conflicts Resolution

### How Maven and Gradle Pick Versions

```xml
<!-- ========================================
     MAVEN: NEAREST WINS
     
     Maven uses "nearest definition" strategy
     Closer to your project = wins
     ======================================== -->

<dependencies>
    <!-- Scenario 1: Different depths -->
    
    <!-- Path 1: Depth 2 -->
    <dependency>
        <groupId>com.example</groupId>
        <artifactId>library-a</artifactId>
        <version>1.0</version>
        <!-- → library-x:1.0 → guava:30.0 (depth 2) -->
    </dependency>
    
    <!-- Path 2: Depth 1 -->
    <dependency>
        <groupId>com.example</groupId>
        <artifactId>library-b</artifactId>
        <version>1.0</version>
        <!-- → guava:31.0 (depth 1) -->
    </dependency>
</dependencies>

<!-- Winner: guava:31.0 (depth 1 is nearer than depth 2) -->

<!-- ========================================
     MAVEN: FIRST DECLARATION WINS (SAME DEPTH)
     ======================================== -->

<dependencies>
    <!-- Both at depth 1 -->
    
    <dependency>
        <groupId>com.example</groupId>
        <artifactId>library-a</artifactId>
        <version>1.0</version>
        <!-- → guava:30.0 -->
    </dependency>
    
    <dependency>
        <groupId>com.example</groupId>
        <artifactId>library-b</artifactId>
        <version>1.0</version>
        <!-- → guava:31.0 -->
    </dependency>
</dependencies>

<!-- Winner: guava:30.0 (declared first) -->

<!-- ========================================
     MAVEN: OVERRIDE TRANSITIVE VERSION
     ======================================== -->

<dependencies>
    <!-- Libraries bring older versions -->
    <dependency>
        <groupId>com.example</groupId>
        <artifactId>library-a</artifactId>
        <version>1.0</version>
        <!-- → guava:25.0 -->
    </dependency>
    
    <!-- Explicitly declare newer version (depth 0) -->
    <dependency>
        <groupId>com.google.guava</groupId>
        <artifactId>guava</artifactId>
        <version>32.1.3-jre</version>
    </dependency>
</dependencies>

<!-- Winner: guava:32.1.3-jre (depth 0 beats all) -->

<!-- ========================================
     MAVEN: DEBUGGING CONFLICTS
     ======================================== -->

<!--
Show effective dependencies:
mvn dependency:tree

Show where dependency comes from:
mvn dependency:tree -Dincludes=com.google.guava:guava

Analyze dependency conflicts:
mvn dependency:analyze

Find why dependency is included:
mvn dependency:tree -Dverbose
-->

<!-- ========================================
     MAVEN: DEPENDENCY MANAGEMENT WINS
     ======================================== -->

<dependencyManagement>
    <dependencies>
        <!-- This version ALWAYS wins -->
        <dependency>
            <groupId>com.google.guava</groupId>
            <artifactId>guava</artifactId>
            <version>32.1.3-jre</version>
        </dependency>
    </dependencies>
</dependencyManagement>

<!-- No matter what transitives request,
     dependencyManagement version is used
-->
```

```kotlin
// ========================================
// GRADLE: NEWEST WINS
// 
// Gradle picks newest version by default
// Different from Maven!
// ========================================

dependencies {
    implementation("com.example:library-a:1.0")
    // → Depends on guava:30.0
    
    implementation("com.example:library-b:1.0")
    // → Depends on guava:31.0
}

// Winner: guava:31.0 (newest version)

// ========================================
// GRADLE: CONFLICT RESOLUTION STRATEGIES
// ========================================

configurations.all {
    resolutionStrategy {
        // Fail on version conflict
        failOnVersionConflict()
        
        // Force specific version
        force("com.google.guava:guava:32.1.3-jre")
        
        // Prefer project modules
        preferProjectModules()
    }
}

// ========================================
// GRADLE: DEPENDENCY CONSTRAINTS
// ========================================

dependencies {
    constraints {
        // Prefer this version in conflicts
        implementation("com.google.guava:guava:32.1.3-jre") {
            because("Security fix in 32.1.3")
        }
    }
}

// ========================================
// GRADLE: STRICT VERSIONS
// ========================================

dependencies {
    // Strictly enforce version (reject any other)
    implementation("com.google.guava:guava") {
        version {
            strictly("32.1.3-jre")
        }
    }
}

// Build fails if any dependency needs different Guava

// ========================================
// GRADLE: DEBUGGING CONFLICTS
// ========================================

/*
View dependency tree:
./gradlew dependencies

Show conflict resolution:
./gradlew dependencyInsight --dependency guava

Show why specific version chosen:
./gradlew dependencyInsight --dependency guava --configuration compileClasspath

HTML report:
./gradlew htmlDependencyReport
*/

// ========================================
// GRADLE: PLATFORM/BOM
// ========================================

dependencies {
    // Import BOM (like Maven)
    implementation(platform("org.springframework.boot:spring-boot-dependencies:3.2.0"))
    
    // Now can omit versions
    implementation("org.springframework.boot:spring-boot-starter-web")
    // Version from BOM
}

// ========================================
// GRADLE: RESOLUTION RESULT
// ========================================

configurations.compileClasspath.get().incoming.resolutionResult.allComponents {
    if (this.id is ModuleComponentIdentifier) {
        val id = this.id as ModuleComponentIdentifier
        println("${id.group}:${id.module}:${id.version}")
    }
}

// Shows all resolved versions
```

## 3. Version Ranges (Avoid!)

### Why Not to Use Dynamic Versions

```xml
<!-- ========================================
     VERSION RANGES IN MAVEN
     
     Maven supports version ranges
     But DON'T USE THEM!
     ======================================== -->

<dependency>
    <groupId>com.google.guava</groupId>
    <artifactId>guava</artifactId>
    
    <!-- AVOID: Soft requirement -->
    <version>[30.0,)</version>  <!-- 30.0 or higher -->
    
    <!-- AVOID: Range -->
    <version>[30.0,32.0)</version>  <!-- 30.0 to 31.x -->
    
    <!-- AVOID: Latest -->
    <version>LATEST</version>
    
    <!-- AVOID: Release -->
    <version>RELEASE</version>
    
    <!-- ✓ GOOD: Specific version -->
    <version>32.1.3-jre</version>
</dependency>

<!-- ========================================
     WHY AVOID VERSION RANGES?
     ======================================== -->

<!--
1. NON-REPRODUCIBLE BUILDS
   - Build today: gets guava:31.0
   - Build tomorrow: gets guava:32.0
   - Different results!

2. BREAKS CI/CD
   - Tests pass locally
   - Tests fail in CI (different version)
   - "Works on my machine!"

3. SECURITY ISSUES
   - Can't verify what's deployed
   - Audit trail unclear
   - Compliance nightmare

4. DEBUGGING HELL
   - Hard to reproduce bugs
   - Different environments = different versions
   - Support tickets harder

5. DEPENDENCY HELL
   - Version range [1.0,2.0) might pull 1.9.9
   - Breaking changes in 1.9.9
   - Your app breaks

6. PERFORMANCE
   - Maven checks repositories for new versions
   - Slows down builds
   - Network calls
-->

<!-- ========================================
     VERSION RANGE SYNTAX (FOR REFERENCE)
     ======================================== -->

<!--
[1.0]          Exactly 1.0
[1.0,)         1.0 or higher
(1.0,)         Higher than 1.0
[1.0,2.0]      Between 1.0 and 2.0 (inclusive)
[1.0,2.0)      1.0 to 2.0 (excluding 2.0)
(1.0,2.0)      Between 1.0 and 2.0 (exclusive)

DON'T USE ANY OF THESE!
-->

<!-- ========================================
     WHAT TO USE INSTEAD
     ======================================== -->

<dependencyManagement>
    <dependencies>
        <!-- ✓ Specific versions -->
        <dependency>
            <groupId>com.google.guava</groupId>
            <artifactId>guava</artifactId>
            <version>32.1.3-jre</version>
        </dependency>
    </dependencies>
</dependencyManagement>

<!-- ✓ Use BOM for consistent versions -->
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-dependencies</artifactId>
            <version>3.2.0</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>

<!-- ✓ Dependency locking (Gradle) -->
<!-- Locks versions to specific snapshot -->
```

```kotlin
// ========================================
// DYNAMIC VERSIONS IN GRADLE
// 
// Also supported, also problematic
// ========================================

dependencies {
    // ✗ AVOID: Latest version
    implementation("com.google.guava:guava:latest.release")
    
    // ✗ AVOID: Version range
    implementation("com.google.guava:guava:[30.0,)")
    
    // ✗ AVOID: Plus notation
    implementation("com.google.guava:guava:32.+")  // Any 32.x
    
    // ✓ GOOD: Specific version
    implementation("com.google.guava:guava:32.1.3-jre")
}

// ========================================
// GRADLE: DEPENDENCY LOCKING
// 
// Use this instead of version ranges!
// ========================================

// build.gradle.kts
dependencyLocking {
    lockAllConfigurations()
}

// Generate lock file:
// ./gradlew dependencies --write-locks

// Creates: gradle.lockfile
/*
com.google.guava:guava:32.1.3-jre=compileClasspath
org.springframework:spring-core:6.1.0=compileClasspath
...
*/

// Now builds use EXACT versions from lockfile
// Reproducible builds guaranteed!

// Update locked versions:
// ./gradlew dependencies --update-locks com.google.guava:guava

// ========================================
// GRADLE: FAIL ON DYNAMIC VERSIONS
// ========================================

configurations.all {
    resolutionStrategy {
        // Fail build if dynamic version found
        failOnDynamicVersions()
        
        // Fail on changing dependencies (snapshots)
        failOnChangingVersions()
    }
}

// Enforces reproducible builds
```

## 4. BOM (Bill of Materials)

### Centralized Version Management

```xml
<!-- ========================================
     WHAT IS A BOM?
     
     BOM = POM that defines dependency versions
     Import it, get consistent versions
     No need to specify each version
     ======================================== -->

<!-- ========================================
     SPRING BOOT BOM (MOST FAMOUS EXAMPLE)
     ======================================== -->

<dependencyManagement>
    <dependencies>
        <!-- Import Spring Boot BOM -->
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-dependencies</artifactId>
            <version>3.2.0</version>
            <type>pom</type>
            <scope>import</scope>  <!-- Special scope -->
        </dependency>
    </dependencies>
</dependencyManagement>

<!-- Now use dependencies WITHOUT versions -->
<dependencies>
    <!-- Version from BOM (3.2.0) -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
        <!-- NO VERSION! -->
    </dependency>
    
    <!-- Version from BOM (matching Spring Boot) -->
    <dependency>
        <groupId>org.springframework</groupId>
        <artifactId>spring-context</artifactId>
        <!-- NO VERSION! -->
    </dependency>
    
    <!-- Jackson version from BOM -->
    <dependency>
        <groupId>com.fasterxml.jackson.core</groupId>
        <artifactId>jackson-databind</artifactId>
        <!-- NO VERSION! -->
    </dependency>
```

```kotlin
dependencies {

    implementation(platform("org.springframework.boot:spring-boot-dependencies:3.2.0"))
    implementation(platform("com.company:platform-bom:1.0.0"))
    
    // First wins for conflicts
}
```

## 5. Dependency Exclusions

### Removing Unwanted Transitives

```xml
<!-- ========================================
     WHY EXCLUDE DEPENDENCIES?
     
     1. Conflicts: Older version causes issues
     2. Unused: Transitive not needed
     3. Wrong: Pulls in wrong implementation
     4. Duplication: Same API, different impl
     ======================================== -->

<!-- ========================================
     EXCLUDING SPECIFIC TRANSITIVE
     ======================================== -->

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
    <version>3.2.0</version>
    
    <!-- Exclude Tomcat (use Jetty instead) -->
    <exclusions>
        <exclusion>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-tomcat</artifactId>
        </exclusion>
    </exclusions>
</dependency>

<!-- Add replacement -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-jetty</artifactId>
    <version>3.2.0</version>
</dependency>

<!-- ========================================
     COMMON EXCLUSION: LOGGING
     ======================================== -->

<dependency>
```

## Summary

```
// ========================================
// DEPENDENCY MANAGEMENT SUMMARY
// ========================================

1. TRANSITIVE DEPENDENCIES
   • Your dependencies have dependencies
   • Can reach 5+ levels deep
   • Maven/Gradle auto-resolve entire tree
   • View: mvn dependency:tree
   • View: ./gradlew dependencies

2. CONFLICT RESOLUTION
   • Maven: Nearest wins (closest to your project)
   • Gradle: Newest wins (highest version number)
   • Override: Add dependency at depth 0
   • Enforce: Use dependencyManagement (Maven)
   • Enforce: Use constraints (Gradle)

3. VERSION RANGES (DON'T USE!)
   • [1.0,2.0) = non-reproducible builds
   • LATEST = different every time
   • 1.+ = breaks unexpectedly
   • ✓ Use: Specific versions (1.2.3)
   • ✓ Use: Dependency locking (Gradle)

4. BOM (BILL OF MATERIALS)
   • POM that defines versions
   • Import: <scope>import</scope>
   • Example: spring-boot-dependencies
   • Benefits: Consistent versions, tested together
   • Create your own for company standards

5. EXCLUSIONS
   • Remove unwanted transitives
   • Fix conflicts by excluding old versions
   • Swap implementations (Tomcat → Jetty)
   • Global: Exclude commons-logging everywhere
   • Be surgical: Only exclude when needed

6. ENFORCING VERSIONS
   • dependencyManagement (Maven)
   • Constraints (Gradle)
   • Strict versions (Gradle)
   • Enforcer plugin (Maven)
   • Dependency locking (Gradle)
   • Verification metadata (Gradle)

7. VULNERABILITY SCANNING
   • OWASP Dependency Check
   • Snyk continuous monitoring
   • GitHub Dependabot
   • Fail builds on high severity (CVSS >= 7)
   • Suppress false positives
   • Run in CI/CD pipeline
```

**Key Takeaways:**

```
The Resolution Rules:
Maven: Nearest wins (depth 0 beats depth 1)
Gradle: Newest wins (version 2.0 beats 1.0)

Override: Add at depth 0 (your pom.xml/build.gradle)

The Diamond Problem:
      App
     /   \
  Lib A  Lib B
    |      |
  Guava  Guava
   1.0    2.0

Which wins? (Resolution strategy!)

Version Strategy:
✗ AVOID: [1.0,2.0), LATEST, 32.+
✓ USE: Specific versions (32.1.3-jre)
✓ USE: Dependency locking
✓ USE: BOM for related dependencies

BOM Pattern:
<dependencyManagement>
  <dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-dependencies</artifactId>
    <type>pom</type>
    <scope>import</scope>
  </dependency>
</dependencyManagement>

Security First:
• OWASP Dependency Check in CI
• Fail builds on high severity
• GitHub Dependabot for updates
• Snyk for continuous monitoring
```
