# Gradle Fundamentals

## Overview

Gradle is a modern, flexible build automation tool that uses a programmable DSL (Domain-Specific Language) instead of XML configuration—unlike Maven's declarative approach, Gradle is imperative and allows arbitrary code in build scripts. This makes Gradle more powerful but requires understanding its execution model. This guide covers seven essential Gradle concepts: Groovy vs Kotlin DSL (Groovy is dynamic and concise but lacks IDE support, Kotlin DSL is type-safe with excellent IDE completion—choose Kotlin for new projects), Task Dependencies (tasks declare what they depend on using dependsOn/mustRunAfter/shouldRunAfter, Gradle builds dependency graph and executes in correct order), Build Lifecycle Phases (initialization finds projects, configuration evaluates all build scripts, execution runs requested tasks—understanding phases prevents common mistakes like configuring tasks during execution), Multi-Project Builds (settings.gradle defines subprojects, shared configuration in root build.gradle, cross-project dependencies, composite builds for including other builds), Custom Tasks (extending DefaultTask, inputs/outputs for up-to-date checking, incremental tasks for performance), Plugin Development (convention plugins for shared configuration, binary plugins for reusable logic, publishing to plugin portal), and Performance Optimization (build cache stores task outputs, Gradle daemon keeps JVM warm, parallel execution, configuration cache). The key insight: Gradle has three phases that execute in order (initialization, configuration, execution)—configuration code runs every time even if task isn't executed, execution code only runs if task is executed. Understanding this distinction prevents common errors like `task.doLast { println x }` (correct, runs during execution) vs `task { println x }` (wrong, runs during configuration). Common confusion: task dependencies don't copy outputs, they only establish execution order; inputs/outputs are for incremental builds, not data flow. The relationship between concepts: lifecycle phases determine when code runs → tasks define what to do → dependencies determine order → plugins bundle reusable logic → build cache and daemon optimize performance. Real-world Gradle usage: multi-project build with shared configuration → custom tasks for code generation → convention plugins for consistent setup → build cache for CI/CD speed → configuration cache for local development speed.

**Key Files**:

```
project/
├── settings.gradle(.kts)        # Project settings
├── build.gradle(.kts)           # Build script
├── gradle.properties            # Project properties
├── gradlew                      # Gradle wrapper (Unix)
├── gradlew.bat                  # Gradle wrapper (Windows)
└── gradle/
    └── wrapper/
        └── gradle-wrapper.properties
```

## 1. Groovy vs Kotlin DSL

### Choosing Your Build Script Language

```groovy
// ========================================
// GROOVY DSL (build.gradle)
// ========================================

// Simple, dynamic, concise
// But lacks type safety and IDE support

plugins {
    id 'java'
    id 'org.springframework.boot' version '3.2.0'
}

group = 'com.example'
version = '1.0.0'

repositories {
    mavenCentral()
}

dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-web'
    testImplementation 'org.springframework.boot:spring-boot-starter-test'
}

// Configuration closure
tasks.withType(JavaCompile) {
    options.encoding = 'UTF-8'
    options.compilerArgs << '-parameters'
}

// Custom task
task hello {
    doLast {
        println 'Hello from Groovy!'
    }
}

// PROS:
// ✓ Concise syntax
// ✓ Dynamic (fewer type declarations)
// ✓ Legacy Gradle examples use it
// ✓ More flexible (can do dynamic things)
//
// CONS:
// ✗ No IDE autocomplete/suggestions
// ✗ No compile-time type checking
// ✗ Harder to refactor
// ✗ Runtime errors instead of compile errors
```

```kotlin
// ========================================
// KOTLIN DSL (build.gradle.kts)
// ========================================

// Type-safe, excellent IDE support
// Recommended for new projects

plugins {
    java
    id("org.springframework.boot") version "3.2.0"
}

group = "com.example"
version = "1.0.0"

repositories {
    mavenCentral()
}

dependencies {
    implementation("org.springframework.boot:spring-boot-starter-web")
    testImplementation("org.springframework.boot:spring-boot-starter-test")
}

// Typed configuration
tasks.withType<JavaCompile> {
    options.encoding = "UTF-8"
    options.compilerArgs.add("-parameters")
}

// Custom task (typed)
tasks.register("hello") {
    doLast {
        println("Hello from Kotlin!")
    }
}

// PROS:
// ✓ Full IDE support (autocomplete, refactoring)
// ✓ Type safety (catch errors at compile time)
// ✓ Better for large projects
// ✓ Modern, actively developed
// ✓ Kotlin knowledge transfers to app code
//
// CONS:
// ✗ More verbose (type declarations)
// ✗ Steeper learning curve
// ✗ Some Groovy examples need translation
```

```kotlin
// ========================================
// SIDE-BY-SIDE COMPARISON
// ========================================

// GROOVY                              // KOTLIN
//──────────────────────────────────────────────────────────────────

// Plugin application
plugins {                              plugins {
    id 'java'                              java
    id 'application'                       application
}                                      }

// Dependencies
dependencies {                         dependencies {
    implementation 'com.google.guava:     implementation(
        guava:32.1.3-jre'                     "com.google.guava:guava:32.1.3-jre"
    testImplementation 'junit:junit:      )
        4.13.2'                            testImplementation("junit:junit:4.13.2")
}                                      }

// Task configuration
task hello {                           tasks.register("hello") {
    doLast {                               doLast {
        println 'Hello'                        println("Hello")
    }                                      }
}                                      }

// Accessing tasks
tasks.named('test') {                  tasks.named<Test>("test") {
    useJUnitPlatform()                     useJUnitPlatform()
}                                      }

// Properties
ext {                                  val myProperty by extra("value")
    myProperty = 'value'               
}                                      

// Multi-line strings
def sql = """                          val sql = """
    SELECT * FROM users                    SELECT * FROM users
    WHERE active = true                    WHERE active = true
""".stripIndent()                      """.trimIndent()

// ========================================
// MIGRATION FROM GROOVY TO KOTLIN
// ========================================

// Key differences:
//
// 1. Strings: 'single' → "double"
// 2. Method calls: method arg → method("arg")
// 3. Maps: [key: value] → mapOf("key" to value)
// 4. Closures: { } → lambda { }
// 5. Property access: task.property → tasks.named<Type>("task").get().property
// 6. Type declarations: needed for task configuration
//
// Migration tool:
// gradle init --dsl kotlin

// ========================================
// RECOMMENDATION
// ========================================

// Use Kotlin DSL for:
// ✓ New projects
// ✓ Large codebases
// ✓ Teams familiar with Kotlin
// ✓ When IDE support is important
//
// Use Groovy DSL for:
// ✓ Small, simple projects
// ✓ Legacy projects already in Groovy
// ✓ Quick scripts
// ✓ When following Groovy examples
```

## 2. Task Dependencies

### Controlling Task Execution Order

```kotlin
// ========================================
// TASK DEPENDENCIES
// 
// Tasks declare dependencies using:
// - dependsOn: This task needs that task to run first
// - mustRunAfter: If both run, enforce order
// - shouldRunAfter: Soft ordering (best effort)
// - finalizedBy: Run cleanup task after this task
// ========================================

tasks.register("compile") {
    doLast {
        println("Compiling sources...")
    }
}

tasks.register("test") {
    // test depends on compile
    dependsOn("compile")
    
    doLast {
        println("Running tests...")
    }
}

tasks.register("package") {
    // package depends on test
    // Transitive: package → test → compile
    dependsOn("test")
    
    doLast {
        println("Creating package...")
    }
}

// Run: gradle package
// Output:
// > Task :compile
// Compiling sources...
// > Task :test
// Running tests...
// > Task :package
// Creating package...

// ========================================
// MULTIPLE DEPENDENCIES
// ========================================

tasks.register("integrationTest") {
    // Multiple dependencies
    dependsOn("compile", "startDatabase")
    
    doLast {
        println("Running integration tests...")
    }
}

tasks.register("startDatabase") {
    doLast {
        println("Starting database...")
    }
}

// ========================================
// MUST RUN AFTER
// 
// Ordering constraint WITHOUT dependency
// Only applies if both tasks run
// ========================================

tasks.register("taskA") {
    doLast {
        println("Task A")
    }
}

tasks.register("taskB") {
    // If both A and B run, B runs after A
    // But running B doesn't cause A to run
    mustRunAfter("taskA")
    
    doLast {
        println("Task B")
    }
}

// gradle taskA taskB → A then B
// gradle taskB → Only B (A doesn't run)
// gradle taskA → Only A (B doesn't run)

// Use case: Ordering without forced execution
tasks.register("build") {
    dependsOn("compile")
}

tasks.register("clean") {
    doLast {
        println("Cleaning...")
    }
}

tasks.named("build") {
    // If clean runs, run it before build
    mustRunAfter("clean")
}

// gradle clean build → Clean first, then build
// gradle build → Build only (clean not forced)

// ========================================
// SHOULD RUN AFTER
// 
// Soft ordering (best effort)
// Used to optimize build order
// ========================================

tasks.register("taskX") {
    shouldRunAfter("taskY")
    
    doLast {
        println("Task X")
    }
}

tasks.register("taskY") {
    doLast {
        println("Task Y")
    }
}

// Gradle tries to run Y before X if both are in the build
// But doesn't guarantee it (unlike mustRunAfter)

// ========================================
// FINALIZED BY
// 
// Cleanup tasks that run after main task
// Even if main task fails
// ========================================

tasks.register("deploy") {
    finalizedBy("cleanup")
    
    doLast {
        println("Deploying application...")
    }
}

tasks.register("cleanup") {
    doLast {
        println("Cleaning up temporary files...")
    }
}

// gradle deploy
// → Runs deploy
// → Always runs cleanup (even if deploy fails)

// ========================================
// DYNAMIC DEPENDENCIES
// ========================================

tasks.register("processFiles") {
    // Add dependencies dynamically
    doFirst {
        if (project.hasProperty("generateFiles")) {
            dependsOn("generateFiles")
        }
    }
    
    doLast {
        println("Processing files...")
    }
}

tasks.register("generateFiles") {
    doLast {
        println("Generating files...")
    }
}

// gradle processFiles -PgenerateFiles
// → Runs generateFiles first

// ========================================
// TASK GRAPH
// 
// Gradle builds a Directed Acyclic Graph (DAG)
// Determines execution order
// ========================================

/*
Example build graph:

        clean
          ↓
      compileJava
          ↓
    processResources
          ↓
        classes
          ↓
         jar
          ↓
        build

Running: gradle build
Executes in topological order
*/

// View task dependencies:
// gradle tasks --all
// gradle dependencies --task build

// ========================================
// CIRCULAR DEPENDENCIES
// ========================================

/*
// This fails! Circular dependency
tasks.register("taskA") {
    dependsOn("taskB")
}

tasks.register("taskB") {
    dependsOn("taskA")  // Error!
}

// Gradle detects cycles and fails build
*/

// ========================================
// BEST PRACTICES
// ========================================

// DO:
// ✓ Use dependsOn for required tasks
// ✓ Use mustRunAfter for ordering
// ✓ Use finalizedBy for cleanup
// ✓ Keep dependency chains shallow
// ✓ Document non-obvious dependencies
//
// DON'T:
// ✗ Create circular dependencies
// ✗ Overuse dependencies (makes builds slow)
// ✗ Put logic in configuration phase
// ✗ Assume task execution order without dependencies
```

## 3. Build Lifecycle Phases

### Understanding Configuration vs Execution

```kotlin
// ========================================
// THE THREE PHASES
// 
// 1. INITIALIZATION: Find all projects
// 2. CONFIGURATION: Evaluate all build scripts
// 3. EXECUTION: Run requested tasks
// ========================================

// ========================================
// INITIALIZATION PHASE
// ========================================

// settings.gradle.kts
rootProject.name = "my-project"

println("INITIALIZATION: Finding projects...")

include("subproject-a")
include("subproject-b")

// This runs first, before any build.gradle

// ========================================
// CONFIGURATION PHASE
// ========================================

// build.gradle.kts

println("CONFIGURATION: Evaluating build script...")

// THIS RUNS EVERY TIME, even if task isn't executed!
tasks.register("myTask") {
    println("CONFIGURATION: Configuring myTask")
    
    // This is CONFIGURATION code
    // Runs for EVERY gradle command
    
    doLast {
        println("EXECUTION: Running myTask")
    }
}

tasks.register("otherTask") {
    println("CONFIGURATION: Configuring otherTask")
    
    doLast {
        println("EXECUTION: Running otherTask")
    }
}

// gradle myTask
// Output:
// CONFIGURATION: Evaluating build script...
// CONFIGURATION: Configuring myTask
// CONFIGURATION: Configuring otherTask  ← Runs even though we only want myTask!
// 
// > Task :myTask
// EXECUTION: Running myTask

// ========================================
// CONFIGURATION VS EXECUTION
// ========================================

tasks.register("example") {
    // CONFIGURATION PHASE
    // Runs every time, for every Gradle command
    println("Configuration: This runs during configuration")
    
    val x = project.findProperty("myProp") ?: "default"
    println("Configuration: myProp = $x")
    
    // EXECUTION PHASE
    // Only runs if this task is executed
    doFirst {
        println("Execution: Running before task action")
    }
    
    doLast {
        println("Execution: Running task action")
    }
}

// ========================================
// COMMON MISTAKE
// ========================================

tasks.register("bad") {
    // ❌ WRONG: This runs during configuration!
    val files = file("build").listFiles()
    println("Files: ${files?.size ?: 0}")
    
    // If build/ doesn't exist during configuration,
    // this fails even if task isn't run!
}

tasks.register("good") {
    // ✓ CORRECT: Defer to execution
    doLast {
        val files = file("build").listFiles()
        println("Files: ${files?.size ?: 0}")
    }
}

// ========================================
// LIFECYCLE CALLBACKS
// ========================================

// After evaluation
gradle.projectsEvaluated {
    println("All projects evaluated")
}

// After task graph built
gradle.taskGraph.whenReady {
    println("Task graph ready")
    println("Tasks to execute: ${it.allTasks.map { task -> task.name }}")
}

// Before task execution
gradle.taskGraph.beforeTask { task ->
    println("About to execute: ${task.name}")
}

// After task execution
gradle.taskGraph.afterTask { task, state ->
    if (state.failure != null) {
        println("Task ${task.name} FAILED")
    } else {
        println("Task ${task.name} completed")
    }
}

// ========================================
// CONFIGURATION AVOIDANCE
// ========================================

// Old way (eager configuration)
tasks.create("eager") {
    // Configures immediately
    println("Configuring eagerly")
}

// New way (lazy configuration)
tasks.register("lazy") {
    // Only configures if needed
    println("Configuring lazily")
}

// Best practice: Use register() not create()
// Improves performance for large builds

// ========================================
// EXECUTION TIMELINE
// ========================================

/*
1. INITIALIZATION PHASE
   - Read settings.gradle.kts
   - Determine which projects to include
   
2. CONFIGURATION PHASE
   - Evaluate all build.gradle.kts files
   - Configure all tasks (even if not executed)
   - Build task dependency graph
   
3. EXECUTION PHASE
   - Execute requested tasks in order
   - Only code in doFirst/doLast runs
   
Example: gradle clean build

1. Initialization: Load all projects
2. Configuration: Evaluate all build scripts
3. Execution: Run clean, then build tasks
*/

// ========================================
// PERFORMANCE TIP
// ========================================

// Avoid expensive operations during configuration!

tasks.register("slow") {
    // ❌ BAD: Heavy computation during configuration
    val result = (1..1000000).sum()
    println("Sum: $result")
    
    doLast {
        println("Task executes")
    }
}

tasks.register("fast") {
    // ✓ GOOD: Defer to execution
    doLast {
        val result = (1..1000000).sum()
        println("Sum: $result")
    }
}

// With 100 tasks like "slow", configuration takes forever
// Even if you only run one task!
```

## 4. Multi-Project Builds

### Managing Multiple Subprojects

```kotlin
// ========================================
// PROJECT STRUCTURE
// ========================================

/*
my-application/
├── settings.gradle.kts          # Defines subprojects
├── build.gradle.kts              # Root build script
├── gradle.properties             # Global properties
├── common/
│   └── build.gradle.kts
├── service/
│   └── build.gradle.kts
└── web/
    └── build.gradle.kts
*/

// ========================================
// settings.gradle.kts (Root)
// ========================================

rootProject.name = "my-application"

// Include subprojects
include("common")
include("service")
include("web")

// Nested structure
// include("api:rest")
// include("api:graphql")

// ========================================
// build.gradle.kts (Root)
// 
// Configuration shared by all subprojects
// ========================================

plugins {
    java
    id("org.springframework.boot") version "3.2.0" apply false
}

// Apply to ALL subprojects
allprojects {
    group = "com.example"
    version = "1.0.0"
    
    repositories {
        mavenCentral()
    }
}

// Apply to subprojects only (not root)
subprojects {
    apply(plugin = "java")
    
    java {
        toolchain {
            languageVersion.set(JavaLanguageVersion.of(17))

    }
    
    dependencies {
        // Common dependencies for all subprojects
        testImplementation("org.junit.jupiter:junit-jupiter:5.10.0")

    tasks.withType<Test> {
        useJUnitPlatform()
    }
}

// common/build.gradle.kts

dependencies {

    id("org.springframework.boot")

dependencies {
    // Depend on service module
    // Transitively gets common
    implementation(project(":service"))

    implementation("org.springframework.boot:spring-boot-starter-thymeleaf")
}

// ========================================
// CROSS-PROJECT DEPENDENCIES
// ========================================

// Depend on another project
implementation(project(":common"))

// Depend on specific configuration

// ========================================
// SHARED BUILD LOGIC
// ========================================

// buildSrc/src/main/kotlin/common-conventions.gradle.kts

    java
}

java {
    sourceCompatibility = JavaVersion.VERSION_17
    targetCompatibility = JavaVersion.VERSION_17
}

dependencies {
    testImplementation("org.junit.jupiter:junit-jupiter:5.10.0")
}

// Use in subprojects:
// plugins {
//     id("common-conventions")
// }

// ========================================
// COMPOSITE BUILDS
// 
// Include other Gradle builds
// ========================================

// settings.gradle.kts

// Now can depend on my-library as if it's a regular dependency
// dependencies {
//     implementation("com.example:my-library:1.0.0")

// ========================================
// BUILDING MULTI-PROJECT
// ========================================

/*
From root directory:

Build all projects:
./gradlew build

Build specific project:
./gradlew :web:build

Build project and dependencies:
./gradlew :web:build --include-build common

Clean all:
./gradlew clean

Run application:
./gradlew :web:bootRun

List all tasks:
./gradlew tasks --all

Build in parallel:
./gradlew build --parallel
*/

// ========================================
// PROJECT BUILD ORDER
// ========================================

/*
Gradle determines order based on dependencies:

1. common (no dependencies)
2. service (depends on common)
3. web (depends on service)

Parallel builds:
- common builds first
- service and other independent projects build in parallel
- web builds after service completes
*/
```

## 5. Custom Tasks

### Creating Reusable Build Tasks

```kotlin
// ========================================
// SIMPLE CUSTOM TASK
// ========================================

tasks.register("hello") {
    doLast {
        println("Hello, Gradle!")
    }
}

// Run: ./gradlew hello

// ========================================
// CUSTOM TASK CLASS
// ========================================

import org.gradle.api.DefaultTask
import org.gradle.api.tasks.*

abstract class GreetingTask : DefaultTask() {
    
    @get:Input
    abstract val greeting: Property<String>
    
    @TaskAction
    fun greet() {
        println("${greeting.get()}, from ${project.name}!")
    }
}

// Register task
tasks.register<GreetingTask>("greet") {
    greeting.set("Hello")
}

// Run: ./gradlew greet
// Output: Hello, from my-project!

// ========================================
// INPUTS AND OUTPUTS
// 
// For up-to-date checking and caching
// ========================================

abstract class ProcessFilesTask : DefaultTask() {
    
    @get:InputDirectory
    abstract val sourceDir: DirectoryProperty
    
    @get:OutputDirectory
    abstract val outputDir: DirectoryProperty
    
    @get:Input
    abstract val prefix: Property<String>
    
    @TaskAction
    fun process() {
        val source = sourceDir.get().asFile
        val output = outputDir.get().asFile
        
        output.mkdirs()
        
        source.listFiles()?.forEach { file ->
            val newName = "${prefix.get()}_${file.name}"
            file.copyTo(File(output, newName), overwrite = true)
        }
        
        println("Processed ${source.listFiles()?.size ?: 0} files")
    }
}

tasks.register<ProcessFilesTask>("processFiles") {
    sourceDir.set(file("src/data"))
    outputDir.set(file("build/processed"))
    prefix.set("processed")
}

// First run: Executes
// Second run: UP-TO-DATE (inputs/outputs unchanged)

// ========================================
// INCREMENTAL TASKS
// 
// Process only changed files
// ========================================

abstract class IncrementalProcessTask : DefaultTask() {
    
    @get:InputFiles
    @get:SkipWhenEmpty
    @get:PathSensitive(PathSensitivity.RELATIVE)
    abstract val inputFiles: ConfigurableFileCollection
    
    @get:OutputDirectory
    abstract val outputDir: DirectoryProperty
    
    @TaskAction
    fun execute(inputChanges: InputChanges) {
        val output = outputDir.get().asFile
        output.mkdirs()
        
        if (!inputChanges.isIncremental) {
            // Full build
            println("Full build: processing all files")
            inputFiles.forEach { processFile(it, output) }
        } else {
            // Incremental build
            println("Incremental build")
            
            inputChanges.getFileChanges(inputFiles).forEach { change ->
                when (change.changeType) {
                    ChangeType.ADDED, ChangeType.MODIFIED -> {
                        println("Processing ${change.file.name}")
                        processFile(change.file, output)
                    }
                    ChangeType.REMOVED -> {
                        println("Removing ${change.file.name}")
                        File(output, change.file.name).delete()
                    }
                }
            }
        }
    }
    
    private fun processFile(file: File, outputDir: File) {
        val output = File(outputDir, file.name)
        file.copyTo(output, overwrite = true)
    }
}

tasks.register<IncrementalProcessTask>("incrementalProcess") {
    inputFiles.from(fileTree("src/data"))
    outputDir.set(file("build/incremental"))
}

// ========================================
// TASK WITH MULTIPLE ACTIONS
// ========================================

tasks.register("multiAction") {
    doFirst {
        println("1. First action (setup)")
    }
    
    doFirst {
        println("2. Another first action")
    }
    
    doLast {
        println("3. Last action (main work)")
    }
    
    doLast {
        println("4. Another last action (cleanup)")
    }
}

// Execution order:
// 1. All doFirst (in reverse order of declaration)
// 2. @TaskAction
// 3. All doLast (in order of declaration)

// ========================================
// TASK WITH CONFIGURATION
// ========================================

open class ConfigurableTask : DefaultTask() {
    
    @Input
    var message: String = "Default message"
    
    @Input
    var count: Int = 1
    
    @TaskAction
    fun execute() {
        repeat(count) {
            println("$message (${it + 1})")
        }
    }
}

tasks.register<ConfigurableTask>("configurable") {
    message = "Hello"
    count = 3
}

// ========================================
// REAL-WORLD EXAMPLE: CODE GENERATION
// ========================================

abstract class GenerateSourceTask : DefaultTask() {
    
    @get:Input
    abstract val packageName: Property<String>
    
    @get:Input
    abstract val className: Property<String>
    
    @get:OutputFile
    abstract val outputFile: RegularFileProperty
    
    @TaskAction
    fun generate() {
        val pkg = packageName.get()
        val cls = className.get()
        
        val code = """
            package $pkg;
            
            public class $cls {
                public String getName() {
                    return "$cls";
                }
            }
        """.trimIndent()
        
        outputFile.get().asFile.apply {
            parentFile.mkdirs()
            writeText(code)
        }
        
        println("Generated $cls.java")
    }
}

tasks.register<GenerateSourceTask>("generateSource") {
    packageName.set("com.example.generated")
    className.set("GeneratedClass")
    outputFile.set(file("build/generated/com/example/generated/GeneratedClass.java"))
}

// Make compilation depend on generation
```

**Key Takeaways:**

```
The Three Phases:
1. Initialization → Find projects (settings.gradle.kts)
2. Configuration → Evaluate ALL scripts (runs always)
3. Execution → Run tasks (doFirst/doLast)

Critical Rule:
Configuration code runs EVERY TIME
Execution code runs ONLY IF task executes

Task Dependencies:
dependsOn → Forces execution
mustRunAfter → Orders if both run
finalizedBy → Cleanup after

Multi-Project:
settings.gradle.kts → include("module")
build.gradle.kts → subprojects { }
module/build.gradle.kts → implementation(project(":other"))

Performance Multipliers:
Daemon → 2x faster
Build Cache → 10x faster clean builds
Configuration Cache → 5x faster incremental
Combined → 50-100x improvement!

gradle.properties Magic:
org.gradle.daemon=true
org.gradle.caching=true
org.gradle.configuration-cache=true
org.gradle.parallel=true
org.gradle.jvmargs=-Xmx2g
```
