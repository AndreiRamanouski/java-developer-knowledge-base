# Build Optimization

## Overview

Build optimization is the art of making builds faster—critical because developers run builds hundreds of times per day, and slow builds kill productivity and increase feedback loops. This guide covers six essential optimization strategies: Incremental Builds (only rebuild what changed—inputs/outputs tracking, up-to-date checking, task avoidance, understanding what triggers rebuilds), Parallel Execution (use multiple CPU cores—Maven -T flag, Gradle --parallel, independent module compilation, reduces build time by 2-4x), Build Cache (reuse outputs from previous builds—Gradle build cache stores task outputs locally and remotely, shares cache across team/CI, can make clean builds 90% faster), Module Splitting (break monoliths into modules—smaller compilation units, parallel builds, independent deployments, faster feedback), Fat JAR vs Thin JAR (fat/uber JAR contains all dependencies in single file vs thin JAR references external dependencies—fat JAR is self-contained but large, thin JAR is small but needs classpath), and Native Image Builds (GraalVM native-image compiles Java to native executable—instant startup, lower memory, AOT compilation, perfect for CLI tools and microservices). The key insight: build time compounds over the day—a build that takes 5 minutes instead of 30 seconds means 45 builds per day = 4 hours wasted waiting. Understanding optimization means understanding what drives build time: dependency resolution (network calls), compilation (CPU-bound), testing (often slowest), packaging (I/O). Common mistakes: not leveraging incremental builds (unnecessary recompilation), running tests serially (slow), not using build cache (rebuilding same code), large single modules (can't parallelize), overly fat JARs (slow to build/transfer). The relationship between strategies: incremental builds reduce work → parallel execution speeds remaining work → build cache eliminates repeated work → module splitting enables better parallelization → appropriate JAR strategy reduces packaging time → native images eliminate JVM startup for CLI tools. Real-world optimization path: measure baseline → enable incremental builds → parallelize execution → add build cache → split large modules → optimize test execution → consider native images for CLI/microservices. Impact: typical optimization can reduce build time from 10 minutes to 30 seconds (20x improvement).

**Key Metrics**:

```
Build Time Components:
┌──────────────────────────────────────┐
│ Dependency Resolution:    5-10%      │
│ Compilation:             20-30%      │
│ Testing:                 50-60%      │
│ Packaging:                5-10%      │
└──────────────────────────────────────┘

Optimization Target:
Before:  10 minutes (unacceptable)
Goal:    30-60 seconds (good)
Best:    5-10 seconds (excellent)
```

## 1. Incremental Builds

### Only Rebuild What Changed

```xml
<!-- ========================================
     MAVEN: INCREMENTAL COMPILATION
     
     Maven's incremental compilation is limited
     Gradle has much better support
     ======================================== -->

<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <version>3.11.0</version>
    <configuration>
        <!-- Enable incremental compilation -->
        <useIncrementalCompilation>true</useIncrementalCompilation>
        
        <!-- Track file changes -->
        <fileExtensions>
            <fileExtension>.java</fileExtension>
        </fileExtensions>
    </configuration>
</plugin>

<!-- Maven limitations:
     - Module-level incremental (not file-level)
     - If one file changes, recompiles entire module
     - Not as sophisticated as Gradle
-->

<!-- ========================================
     MAVEN: SKIP UNCHANGED MODULES
     ======================================== -->

<!--
Use Maven's reactor to skip unchanged modules:

mvn clean install -pl changed-module -am

-pl: Project list (which modules)
-am: Also make (build dependencies)

Only builds changed module and its dependencies
-->

<!-- ========================================
     MAVEN: TIMESTAMPS
     ======================================== -->

<!--
Maven checks timestamps:
- Source file newer than .class → Recompile
- Test file newer than test.class → Recompile

Keep target/ directory:
- mvn install (not mvn clean install)
- Preserves previous build outputs
- Only recompiles changed files
-->
```

```kotlin
// ========================================
// GRADLE: INCREMENTAL COMPILATION
// 
// Gradle has EXCELLENT incremental builds
// File-level, not just module-level
// ========================================

tasks.withType<JavaCompile> {
    // Incremental compilation (enabled by default)
    options.isIncremental = true
    
    // Annotation processor incremental support
    options.compilerArgs.add("-Aprocessor.incremental=true")
}

// How it works:
// 1. Gradle tracks inputs (source files)
// 2. Tracks outputs (.class files)
// 3. Hashes inputs
// 4. On next build, compares hashes
// 5. Only recompiles if changed

// First build: Compiles 100 files
// Change 1 file: Recompiles 1-5 files (with dependencies)
// Much faster!

// ========================================
// GRADLE: UP-TO-DATE CHECKING
// ========================================

// Gradle marks tasks as UP-TO-DATE if:
// - Inputs unchanged
// - Outputs present and valid
// - Task action unchanged

// Example output:
/*
> Task :compileJava UP-TO-DATE
> Task :processResources UP-TO-DATE
> Task :classes UP-TO-DATE
> Task :compileTestJava
> Task :test

Only test runs (source unchanged)
*/

// ========================================
// GRADLE: TASK INPUTS/OUTPUTS
// ========================================

abstract class ProcessFilesTask : DefaultTask() {
    
    // Declare inputs
    @get:InputDirectory
    @get:PathSensitive(PathSensitivity.RELATIVE)
    abstract val inputDir: DirectoryProperty
    
    @get:InputFile
    abstract val configFile: RegularFileProperty
    
    @get:Input
    abstract val processingMode: Property<String>
    
    // Declare outputs
    @get:OutputDirectory
    abstract val outputDir: DirectoryProperty
    
    @TaskAction
    fun process() {
        val input = inputDir.get().asFile
        val output = outputDir.get().asFile
        
        // Process files...
    }
}

// First run: Executes
// Second run (nothing changed): UP-TO-DATE
// Change input file: Executes
// Change config: Executes

// ========================================
// GRADLE: INCREMENTAL TASK
// ========================================

abstract class IncrementalTask : DefaultTask() {
    
    @get:InputFiles
    @get:PathSensitive(PathSensitivity.RELATIVE)
    abstract val inputFiles: ConfigurableFileCollection
    
    @get:OutputDirectory
    abstract val outputDir: DirectoryProperty
    
    @TaskAction
    fun execute(inputChanges: InputChanges) {
        val output = outputDir.get().asFile
        
        if (!inputChanges.isIncremental) {
            // Clean build - process all files
            println("Full rebuild")
            inputFiles.forEach { processFile(it, output) }
        } else {
            // Incremental build - only changed files
            inputChanges.getFileChanges(inputFiles).forEach { change ->
                when (change.changeType) {
                    ChangeType.ADDED, ChangeType.MODIFIED -> {
                        println("Processing ${change.file.name}")
                        processFile(change.file, output)
                    }
                    ChangeType.REMOVED -> {
                        println("Removing ${change.file.name}")
                        File(output, change.file.name).delete()
                    }
                }
            }
        }
    }
    
    private fun processFile(file: File, outputDir: File) {
        val output = File(outputDir, file.name)
        file.copyTo(output, overwrite = true)
    }
}

// Only processes changed files!
// 1000 files → Change 1 → Process 1

// ========================================
// GRADLE: AVOIDING TASK EXECUTION
// ========================================

// Use @SkipWhenEmpty to avoid running if no inputs
abstract class OptionalTask : DefaultTask() {
    
    @get:InputFiles
    @get:SkipWhenEmpty  // Don't run if no files
    abstract val inputFiles: ConfigurableFileCollection
    
    @TaskAction
    fun execute() {
        // Only runs if inputFiles has files
    }
}

// ========================================
// GRADLE: CACHE KEYS
// ========================================

// Gradle creates cache key from:
// - Task inputs (files, properties)
// - Task implementation class
// - Gradle version
// - Plugin versions

// If cache key matches → UP-TO-DATE
// If different → Execute

// Force re-execution:
// ./gradlew clean build --rerun-tasks
```

## 2. Parallel Execution

### Using Multiple CPU Cores

```xml
<!-- ========================================
     MAVEN: PARALLEL BUILDS
     
     Use -T flag for parallel execution
     ======================================== -->

<!--
Parallel build with 4 threads:
mvn clean install -T 4

Parallel build with 1 thread per core:
mvn clean install -T 1C

Parallel build with 2 threads per core:
mvn clean install -T 2C

Example:
8-core machine:
-T 1C = 8 threads
-T 2C = 16 threads
-T 4 = 4 threads
-->

<!-- ========================================
     MAVEN: HOW IT WORKS
     ======================================== -->

<!--
Maven builds dependency graph:

common → service → web

Sequential (-T 1):
1. Build common (60s)
2. Build service (60s)
3. Build web (60s)
Total: 180s

Parallel (-T 4):
1. Build common (60s)
2. Build service AND web in parallel (60s each)
Total: 120s (only if service doesn't depend on web)

Reality:
common (leaf, no deps)
service (depends on common)
api (depends on common)
web (depends on service and api)

Parallel:
1. common builds alone (60s)
2. service AND api build in parallel (60s each)
3. web builds (60s)
Total: 180s

For parallel to help, need independent modules!
-->

<!-- ========================================
     MAVEN: PARALLEL TESTS
     ======================================== -->

<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-surefire-plugin</artifactId>
    <version>3.0.0</version>
    <configuration>
        <!-- Parallel execution -->
        <parallel>methods</parallel>  <!-- or classes, both -->
        <threadCount>4</threadCount>
        
        <!-- Or use thread count per core -->
        <perCoreThreadCount>true</perCoreThreadCount>
        <threadCount>2</threadCount>  <!-- 2 per core -->
        
        <!-- Fork options -->
        <forkCount>2</forkCount>  <!-- 2 JVM processes -->
        <reuseForks>true</reuseForks>
    </configuration>
</plugin>

<!-- Parallel test execution can reduce test time 2-4x -->

<!-- ========================================
     MAVEN: PARALLEL BEST PRACTICES
     ======================================== -->

<!--
DO:
✓ Use -T for multi-module builds
✓ Structure modules to maximize parallelism
✓ Use parallel tests
✓ Profile to find optimal thread count

DON'T:
✗ Use -T 100 (too many threads)
✗ Forget about dependencies
✗ Expect 8x speedup with 8 threads
✗ Use on single module (no benefit)
-->
```

```kotlin
// ========================================
// GRADLE: PARALLEL EXECUTION
// ========================================

// gradle.properties
org.gradle.parallel=true
org.gradle.workers.max=8  // Max worker threads

// Or command line:
// ./gradlew build --parallel --max-workers=8

// How it works:
// Gradle builds task graph
// Identifies independent tasks
// Executes in parallel

// Example:
// Module A: compileJava, test
// Module B: compileJava, test
// Module C: compileJava, test (depends on A)

// Parallel execution:
// 1. A.compileJava, B.compileJava (parallel)
// 2. A.test, B.test (parallel)
// 3. C.compileJava (waits for A)
// 4. C.test

// ========================================
// GRADLE: WORKER API
// ========================================

abstract class ParallelProcessTask : DefaultTask() {
    
    @get:InputFiles
    abstract val inputFiles: ConfigurableFileCollection
    
    @get:Inject
    abstract val workerExecutor: WorkerExecutor
    
    @TaskAction
    fun process() {
        // Submit work items to worker pool
        inputFiles.forEach { file ->
            workerExecutor.noIsolation().submit(ProcessWorkAction::class) {
                inputFile.set(file)
            }
        }
        
        // Workers execute in parallel
    }
}

abstract class ProcessWorkAction : WorkAction<ProcessWorkParameters> {
    override fun execute() {
        val file = parameters.inputFile.get().asFile
        // Process file...
    }
}

interface ProcessWorkParameters : WorkParameters {
    val inputFile: RegularFileProperty
}

// ========================================
// GRADLE: TEST PARALLEL EXECUTION
// ========================================

tasks.withType<Test> {
    // Parallel test execution
    maxParallelForks = Runtime.getRuntime().availableProcessors()
    
    // Or specific number
    maxParallelForks = 4
}

// JUnit 5 parallel execution
tasks.withType<Test> {
    useJUnitPlatform()
    
    systemProperty("junit.jupiter.execution.parallel.enabled", "true")
    systemProperty("junit.jupiter.execution.parallel.mode.default", "concurrent")
    systemProperty("junit.jupiter.execution.parallel.config.strategy", "dynamic")
}

// ========================================
// GRADLE: PERFORMANCE
// ========================================

// Typical improvements:
// Single-threaded: 5 minutes
// Parallel (4 cores): 2 minutes (2.5x faster)
// + Build cache: 30 seconds (10x faster)
// + Incremental: 5 seconds (60x faster)
```

## 3. Build Cache (Gradle)

### Reusing Task Outputs

```kotlin
// ========================================
// GRADLE BUILD CACHE
// 
// Most powerful build optimization
// Reuses outputs from previous builds
// ========================================

// gradle.properties
org.gradle.caching=true

// Or in settings.gradle.kts
buildCache {
    local {
        isEnabled = true
        directory = file("$rootDir/.gradle/build-cache")
        removeUnusedEntriesAfterDays = 30
    }
}

// How it works:
// 1. Task executes, produces output
// 2. Gradle stores output in cache (keyed by inputs)
// 3. Next build with same inputs
// 4. Gradle retrieves from cache instead of executing

// Example:
// First build:
// > Task :compileJava (5s)
// > Task :test (30s)

// Second build (clean):
// > Task :compileJava FROM-CACHE (0.1s)
// > Task :test FROM-CACHE (0.2s)

// 90% faster!

// ========================================
// REMOTE BUILD CACHE
// ========================================

buildCache {
    local {
        isEnabled = true
    }
    
    remote<HttpBuildCache> {
        url = uri("https://cache.company.com/cache/")
        
        // Read from remote
        isPush = false
        
        // Only CI pushes to cache
        isPush = System.getenv("CI") == "true"
        
        credentials {
            username = System.getenv("CACHE_USERNAME")
            password = System.getenv("CACHE_PASSWORD")
        }
    }
}

// Benefits:
// ✓ Developer pulls from cache (CI built it)
// ✓ No need to build everything locally
// ✓ Team shares cache
// ✓ New developer: instant setup

// ========================================
// CACHEABLE TASKS
// ========================================

@CacheableTask
abstract class MyCacheableTask : DefaultTask() {
    
    // Properly declared inputs
    @get:InputFiles
    @get:PathSensitive(PathSensitivity.RELATIVE)
    abstract val inputFiles: ConfigurableFileCollection
    
    // Properly declared outputs
    @get:OutputDirectory
    abstract val outputDir: DirectoryProperty
    
    @TaskAction
    fun execute() {
        // Task work...
    }
}

// Requirements for caching:
// 1. Declare all inputs
// 2. Declare all outputs
// 3. Pure (same inputs → same outputs)
// 4. No side effects

// ========================================
// CACHE EFFECTIVENESS
// ========================================

// View cache statistics:
// ./gradlew build --build-cache --scan

// Example report:
/*
Build cache effectiveness:
- 45 tasks executed
- 120 tasks loaded from cache (FROM-CACHE)
- 5 tasks up-to-date (UP-TO-DATE)

Cache hit rate: 72%
Time saved: 4 minutes
*/

// ========================================
// CACHE DEBUGGING
// ========================================

// Why task not cached?
// ./gradlew build --build-cache -Dorg.gradle.caching.debug=true

// Common issues:
// 1. Absolute paths in inputs
// 2. Timestamps in inputs
// 3. Non-deterministic outputs
// 4. Missing input declarations

// Fix: Use PathSensitivity.RELATIVE
@get:InputFiles
@get:PathSensitive(PathSensitivity.RELATIVE)
abstract val sources: ConfigurableFileCollection

// ========================================
// MAVEN: NO NATIVE BUILD CACHE
// ========================================

// Maven doesn't have equivalent to Gradle build cache
// Workarounds:
// 1. Use CI artifacts
// 2. Use Gradle instead
// 3. Keep target/ directory (limited)

// Maven incremental builds are module-level
// Gradle build cache is task-level (much better)
```

## 4. Module Splitting

### Breaking Up Monoliths

```
// ========================================
// WHY SPLIT MODULES?
// ========================================

Monolith Problem:
single-module/
└── src/
    ├── main/java/
    │   ├── api/      (1000 files)
    │   ├── service/  (2000 files)
    │   ├── data/     (500 files)
    │   └── util/     (300 files)
    └── test/java/    (2000 files)

Total: 5800 files in one module

Problems:
✗ Long compilation (all files)
✗ Can't parallelize
✗ Long tests (all tests)
✗ Tight coupling
✗ Large JAR

Build time: 10 minutes

// ========================================
// MULTI-MODULE SOLUTION
// ========================================

multi-module/
├── common/        (300 files)
├── data/          (500 files)
├── service/       (2000 files)
├── api/           (1000 files)
└── web/           (200 files)

Benefits:
✓ Parallel compilation
✓ Incremental builds (only changed modules)
✓ Smaller JARs
✓ Clear boundaries
✓ Independent deployments

Build time: 2 minutes (5x faster!)
Incremental: 10 seconds (60x faster!)
```

```kotlin
// ========================================
// GRADLE: MULTI-MODULE STRUCTURE
// ========================================

// settings.gradle.kts
rootProject.name = "my-app"

include("common")
include("data")
include("service")
include("api")
include("web")

// Build order (automatic):
// 1. common (parallel with data)
// 2. service (depends on common, data)
// 3. api (depends on service)
// 4. web (depends on api)

// ========================================
// MODULE SIZING
// ========================================

// Good module sizes:
// Small: 100-500 files
// Medium: 500-2000 files
// Large: 2000-5000 files
// Too large: >5000 files (split more)

// Compilation time:
// 100 files: ~5 seconds
// 500 files: ~15 seconds
// 2000 files: ~60 seconds
// 5000 files: ~180 seconds

// Sweet spot: 500-1000 files per module

// ========================================
// SPLITTING STRATEGIES
// ========================================

// Strategy 1: By layer
app/
├── api/           (REST controllers)
├── service/       (Business logic)
├── data/          (Repositories)
└── common/        (Shared utilities)

// Strategy 2: By feature
app/
├── user/          (User management)
├── order/         (Order processing)
├── payment/       (Payment handling)
└── notification/  (Notifications)

// Strategy 3: Hybrid
app/
├── common/        (Shared)
├── user-api/      (User REST)
├── user-service/  (User logic)
├── order-api/
├── order-service/
└── shared-data/   (Common data access)

// Choose based on:
// - Team structure
// - Deployment needs
// - Code organization

// ========================================
// DEPENDENCY RULES
// ========================================

// Good: Acyclic graph
common ← data ← service ← api ← web

// Bad: Circular dependencies
service ← → api  // Error!

// Gradle detects cycles and fails build

// ========================================
// WHEN TO SPLIT
// ========================================

// Split when:
// ✓ Module >5000 files
// ✓ Build time >5 minutes
// ✓ Clear logical boundaries
// ✓ Independent deployment needed
// ✓ Team works on separate areas

// Don't split when:
// ✗ Already fast (<1 minute)
// ✗ Frequent changes across modules
// ✗ No clear boundaries
// ✗ Adds complexity without benefit
```

## 5. Fat JAR vs Thin JAR

### Packaging Strategies

```xml
<!-- ========================================
     MAVEN: FAT JAR (UBER JAR)
     
     Single JAR with all dependencies included
     ======================================== -->

<!-- Using maven-shade-plugin -->
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-shade-plugin</artifactId>
    <version>3.5.1</version>
    <executions>
        <execution>
            <phase>package</phase>
            <goals>
                <goal>shade</goal>
            </goals>
            <configuration>
                <!-- Output file name -->
                <finalName>${project.artifactId}-${project.version}-fat</finalName>
                
                <!-- Main class -->
                <transformers>
                    <transformer implementation="org.apache.maven.plugins.shade.resource.ManifestResourceTransformer">
                        <mainClass>com.example.Application</mainClass>
                    </transformer>
                    
                    <!-- Merge service files -->
                    <transformer implementation="org.apache.maven.plugins.shade.resource.ServicesResourceTransformer"/>
                </transformers>
                
                <!-- Minimize JAR (remove unused classes) -->
                <minimizeJar>false</minimizeJar>
                
                <!-- Filter dependencies -->
                <filters>
                    <filter>
                        <artifact>*:*</artifact>
                        <excludes>
                            <exclude>META-INF/*.SF</exclude>
                            <exclude>META-INF/*.DSA</exclude>
                            <exclude>META-INF/*.RSA</exclude>
                        </excludes>
                    </filter>
                </filters>
            </configuration>
        </execution>
    </executions>
</plugin>

<!-- Result: my-app-1.0.0-fat.jar (50 MB)
     Contains: Your code + Spring + Tomcat + All dependencies
     
     Run: java -jar my-app-1.0.0-fat.jar
-->

<!-- ========================================
     MAVEN: FAT JAR - PROS/CONS
     ======================================== -->

<!--
PROS:
✓ Self-contained (single file)
✓ Easy deployment (copy one file)
✓ No classpath issues
✓ Works everywhere
✓ Easy to run (java -jar app.jar)

CONS:
✗ Large file size (50-200 MB)
✗ Slow to build
✗ Slow to transfer
✗ Duplicate dependencies if multiple apps
✗ Hard to update single dependency
✗ Potential conflicts (META-INF files)
-->

<!-- ========================================
     MAVEN: THIN JAR
     
     Small JAR, dependencies separate
     ======================================== -->

<!-- Using maven-jar-plugin (default) -->
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-jar-plugin</artifactId>
    <version>3.3.0</version>
    <configuration>
        <archive>
            <manifest>
                <addClasspath>true</addClasspath>
                <classpathPrefix>lib/</classpathPrefix>
                <mainClass>com.example.Application</mainClass>
            </manifest>
        </archive>
    </configuration>
</plugin>

<!-- Copy dependencies to lib/ -->
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-dependency-plugin</artifactId>
    <version>3.6.1</version>
    <executions>
        <execution>
            <id>copy-dependencies</id>
            <phase>package</phase>
            <goals>
                <goal>copy-dependencies</goal>
            </goals>
            <configuration>
                <outputDirectory>${project.build.directory}/lib</outputDirectory>
            </configuration>
        </execution>
    </executions>
</plugin>

<!-- Result:
     my-app-1.0.0.jar (10 KB - just your code)
     lib/
       ├── spring-core-6.1.0.jar
       ├── spring-beans-6.1.0.jar
       └── ... (all dependencies)
     
     Run: java -jar my-app-1.0.0.jar
     (Classpath from MANIFEST.MF)
-->

<!-- ========================================
     MAVEN: THIN JAR - PROS/CONS
     ======================================== -->

<!--
PROS:
✓ Small JAR (KB not MB)
✓ Fast to build
✓ Fast to transfer
✓ Shared dependencies (multiple apps)
✓ Easy to update single dependency
✓ Clear dependency list

CONS:
✗ Multiple files to deploy
✗ Classpath management
✗ Must keep structure (lib/ directory)
✗ More complex deployment
-->

<!-- ========================================
     MAVEN: SPRING BOOT FAT JAR
     ======================================== -->

<plugin>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-maven-plugin</artifactId>
    <version>3.2.0</version>
    <executions>
        <execution>
            <goals>
                <goal>repackage</goal>
            </goals>
        </execution>
    </executions>
</plugin>

<!-- Creates:
     my-app-1.0.0.jar (original thin JAR)
     my-app-1.0.0.jar.original (backup)
     
     Fat JAR structure:
     BOOT-INF/
       ├── classes/       (your code)
       └── lib/           (dependencies)
     org/
       └── springframework/boot/loader/  (Spring Boot loader)
     
     Run: java -jar my-app-1.0.0.jar
-->
```

```kotlin
// ========================================
// GRADLE: FAT JAR (SHADOW JAR)
// ========================================

plugins {
    id("com.github.johnrengelman.shadow") version "8.1.1"
}

tasks.shadowJar {
    archiveBaseName.set("my-app")
    archiveClassifier.set("fat")
    archiveVersion.set("1.0.0")
    
    // Merge service files
    mergeServiceFiles()
    
    // Minimize (remove unused)
    minimize {
        exclude(dependency("org.springframework.boot:.*:.*"))
    }
    
    // Relocate dependencies (avoid conflicts)
    relocate("com.google.common", "shadow.com.google.common")
    
    manifest {
        attributes["Main-Class"] = "com.example.Application"
    }
}

// Build: ./gradlew shadowJar
// Output: build/libs/my-app-1.0.0-fat.jar

// ========================================
// GRADLE: THIN JAR
// ========================================

tasks.jar {
    manifest {
        attributes["Main-Class"] = "com.example.Application"
        attributes["Class-Path"] = configurations.runtimeClasspath.get()
            .files
            .joinToString(" ") { "lib/${it.name}" }
    }
}

// Copy dependencies
tasks.register<Copy>("copyDependencies") {
    from(configurations.runtimeClasspath)
    into("build/libs/lib")
}

tasks.named("build") {
    dependsOn("copyDependencies")
}

// ========================================
// GRADLE: SPRING BOOT FAT JAR
// ========================================

plugins {
    id("org.springframework.boot") version "3.2.0"
}

tasks.bootJar {
    archiveBaseName.set("my-app")
    archiveVersion.set("1.0.0")
    
    mainClass.set("com.example.Application")
}

// Build: ./gradlew bootJar
// Output: build/libs/my-app-1.0.0.jar (fat JAR)

// ========================================
// CHOOSING FAT VS THIN
// ========================================

// Use FAT JAR when:
// ✓ Simple deployment (single file)
// ✓ Standalone application
// ✓ Lambda/Cloud Functions
// ✓ CLI tools

```
// ========================================
// BUILD OPTIMIZATION SUMMARY
// ========================================

1. INCREMENTAL BUILDS
   • Only rebuild what changed
   • File-level (Gradle) vs module-level (Maven)
   • Declare inputs/outputs properly
   • Up-to-date checking
   
   Impact: 10-100x faster incremental builds

2. PARALLEL EXECUTION
   • Maven: -T 4 (4 threads)
   • Gradle: --parallel (automatic)
   • Parallel tests: 2-4x faster
   • Requires independent modules
   
   Impact: 2-4x faster full builds

3. BUILD CACHE (Gradle)
   • Stores task outputs
   • Reuses across builds
   • Local and remote cache
   • Team shares cache
   
   Impact: 90% faster clean builds

4. MODULE SPLITTING
   • Break large modules (>5000 files)
   • Enable parallel compilation
   • Faster incremental builds
   • Clear boundaries
   
   Impact: 5-10x faster builds

5. FAT VS THIN JAR
   • Fat: Single file, self-contained (50-200 MB)
   • Thin: Small JAR + lib/ (KB + deps)
   • Choose based on deployment
   
   Fat: Simple deployment
   Thin: Faster builds, shared deps

6. NATIVE IMAGE
   • GraalVM native-image
   • Instant startup (100x faster)
   • Lower memory (4x less)
   • Long build time
   
   Best for: CLI, serverless, microservices
