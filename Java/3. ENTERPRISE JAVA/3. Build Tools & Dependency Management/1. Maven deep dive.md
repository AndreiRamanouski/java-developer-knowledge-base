# Maven Deep Dive

## Overview

Maven is Java's most popular build and dependency management tool—it automates compilation, testing, packaging, and deployment using a declarative XML configuration (pom.xml). Unlike procedural build tools (Ant scripts), Maven follows "convention over configuration"—standard directory structure (src/main/java, src/test/java), standard lifecycle phases (compile, test, package, install, deploy), declarate dependencies (Maven resolves transitively). This guide covers seven essential Maven concepts: POM Structure and Inheritance (Project Object Model hierarchy, parent POMs, effective POM calculation, property inheritance, BOM imports), Dependency Scopes (compile = default/everywhere, provided = compile-only/container-supplied, test = test-only, runtime = execution-only not compilation, system = local JAR path, import = BOM management), Dependency Management vs Dependencies (dependencyManagement defines versions centrally without including dependencies, dependencies actually includes them—prevents version conflicts in multi-module projects), Plugin Configuration (executions, goals, phases, binding plugins to lifecycle), Multi-Module Projects (parent aggregator POM, module inheritance, reactor build order, inter-module dependencies), Profiles (environment-specific configuration—dev/staging/production, activated by properties/JDK version/OS, overrides default behavior), and Repository Management (local cache ~/.m2, Maven Central, private repositories like Nexus/Artifactory, repository mirrors, deployment configuration). The key insight: Maven's power comes from its declarative nature and convention—instead of scripting "how to build", you declare "what to build" and Maven figures out how. Understanding Maven means understanding its three core concepts: lifecycle (what to do—compile, test, package), dependencies (what you need—libraries and versions), and plugins (how to do it—compiler, surefire, jar). Common confusion: dependencyManagement doesn't add dependencies, it only sets versions; compile scope is transitive (consumers get it), provided/test are not. The relationship between concepts: parent POM defines dependencyManagement → child modules inherit versions → profiles override for environments → plugins execute build steps → dependencies are resolved from repositories. Real-world Maven usage: corporate parent POM with company standards → product-specific BOM with version alignment → multi-module project with shared versions → environment profiles for different deployments → private repository for internal artifacts.

**Key Files**:

```
project/
├── pom.xml                    # Project Object Model
├── src/
│   ├── main/
│   │   ├── java/             # Source code
│   │   └── resources/        # Resources (properties, XML)
│   └── test/
│       ├── java/             # Test code
│       └── resources/        # Test resources
└── target/                   # Build output (generated)
```

## 1. POM Structure and Inheritance

### Project Object Model Fundamentals

```xml
<!-- ========================================
     BASIC POM STRUCTURE
     ======================================== -->

<?xml version="1.0" encoding="UTF-8"?>
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0 
                             http://maven.apache.org/xsd/maven-4.0.0.xsd">
    <modelVersion>4.0.0</modelVersion>
    
    <!-- ========================================
         PROJECT COORDINATES (GAV)
         ======================================== -->
    <!-- Unique identifier for this project -->
    <groupId>com.example</groupId>           <!-- Organization/group -->
    <artifactId>my-application</artifactId>  <!-- Project name -->
    <version>1.0.0-SNAPSHOT</version>        <!-- Version -->
    <packaging>jar</packaging>                <!-- jar, war, pom -->
    
    <!-- ========================================
         PROJECT INFORMATION
         ======================================== -->
    <name>My Application</name>
    <description>A sample Maven application</description>
    <url>https://example.com/my-application</url>
    
    <!-- ========================================
         PROPERTIES
         ======================================== -->
    <properties>
        <!-- Encoding -->
        <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
        
        <!-- Java version -->
        <maven.compiler.source>17</maven.compiler.source>
        <maven.compiler.target>17</maven.compiler.target>
        
        <!-- Dependency versions -->
        <spring.version>6.1.0</spring.version>
        <junit.version>5.10.0</junit.version>
    </properties>
    
    <!-- ========================================
         DEPENDENCIES
         ======================================== -->
    <dependencies>
        <dependency>
            <groupId>org.springframework</groupId>
            <artifactId>spring-core</artifactId>
            <version>${spring.version}</version>
        </dependency>
        
        <dependency>
            <groupId>org.junit.jupiter</groupId>
            <artifactId>junit-jupiter</artifactId>
            <version>${junit.version}</version>
            <scope>test</scope>
        </dependency>
    </dependencies>
    
    <!-- ========================================
         BUILD CONFIGURATION
         ======================================== -->
    <build>
        <plugins>
            <plugin>
                <groupId>org.apache.maven.plugins</groupId>
                <artifactId>maven-compiler-plugin</artifactId>
                <version>3.11.0</version>
                <configuration>
                    <source>17</source>
                    <target>17</target>
                </configuration>
            </plugin>
        </plugins>
    </build>
</project>

<!-- ========================================
     KEY CONCEPTS:
     
     1. GAV (GroupId, ArtifactId, Version):
        - Uniquely identifies project
        - groupId: reverse domain (com.example)
        - artifactId: project name (my-app)
        - version: x.y.z or x.y.z-SNAPSHOT
     
     2. Packaging:
        - jar: Java library/application
        - war: Web application
        - pom: Parent/aggregator (no code)
        - ear: Enterprise application
     
     3. Properties:
        - Define once, use many times
        - Reference: ${property.name}
        - Can override in child POMs
     
     4. Dependencies:
        - External libraries needed
        - Maven resolves transitively
        - Downloaded from repositories
     
     5. Build:
        - Plugin configuration
        - Customizes build process
     ======================================== -->
```

### POM Inheritance

```xml
<!-- ========================================
     PARENT POM (company-parent-pom)
     
     Shared configuration for all company projects
     ======================================== -->

<project>
    <modelVersion>4.0.0</modelVersion>
    
    <groupId>com.example</groupId>
    <artifactId>company-parent-pom</artifactId>
    <version>1.0.0</version>
    <packaging>pom</packaging>  <!-- Parent POMs are packaging=pom -->
    
    <name>Company Parent POM</name>
    <description>Parent POM for all company projects</description>
    
    <!-- ========================================
         PROPERTIES (Inherited by all children)
         ======================================== -->
    <properties>
        <!-- Java version standard -->
        <java.version>17</java.version>
        <maven.compiler.source>${java.version}</maven.compiler.source>
        <maven.compiler.target>${java.version}</maven.compiler.target>
        
        <!-- Encoding standard -->
        <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
        
        <!-- Plugin versions (ensures consistency) -->
        <maven-compiler-plugin.version>3.11.0</maven-compiler-plugin.version>
        <maven-surefire-plugin.version>3.0.0</maven-surefire-plugin.version>
        
        <!-- Dependency versions -->
        <spring-boot.version>3.2.0</spring-boot.version>
        <junit.version>5.10.0</junit.version>
        <mockito.version>5.7.0</mockito.version>
    </properties>
    
    <!-- ========================================
         DEPENDENCY MANAGEMENT
         
         Defines versions WITHOUT adding dependencies
         Children inherit versions but choose which to use
         ======================================== -->
    <dependencyManagement>
        <dependencies>
            <!-- Spring Boot BOM (Bill of Materials) -->
            <dependency>
                <groupId>org.springframework.boot</groupId>
                <artifactId>spring-boot-dependencies</artifactId>
                <version>${spring-boot.version}</version>
                <type>pom</type>
                <scope>import</scope>
            </dependency>
            
            <!-- JUnit -->
            <dependency>
                <groupId>org.junit.jupiter</groupId>
                <artifactId>junit-jupiter</artifactId>
                <version>${junit.version}</version>
                <scope>test</scope>
            </dependency>
            
            <!-- Mockito -->
            <dependency>
                <groupId>org.mockito</groupId>
                <artifactId>mockito-core</artifactId>
                <version>${mockito.version}</version>
                <scope>test</scope>
            </dependency>
        </dependencies>
    </dependencyManagement>
    
    <!-- ========================================
         BUILD CONFIGURATION (Inherited by all children)
         ======================================== -->
    <build>
        <pluginManagement>
            <plugins>
                <!-- Compiler plugin -->
                <plugin>
                    <groupId>org.apache.maven.plugins</groupId>
                    <artifactId>maven-compiler-plugin</artifactId>
                    <version>${maven-compiler-plugin.version}</version>
                    <configuration>
                        <source>${java.version}</source>
                        <target>${java.version}</target>
                    </configuration>
                </plugin>
                
                <!-- Test plugin -->
                <plugin>
                    <groupId>org.apache.maven.plugins</groupId>
                    <artifactId>maven-surefire-plugin</artifactId>
                    <version>${maven-surefire-plugin.version}</version>
                </plugin>
            </plugins>
        </pluginManagement>
    </build>
</project>

<!-- ========================================
     CHILD POM (inherits from parent)
     ======================================== -->

<project>
    <modelVersion>4.0.0</modelVersion>
    
    <!-- ========================================
         PARENT REFERENCE
         ======================================== -->
    <parent>
        <groupId>com.example</groupId>
        <artifactId>company-parent-pom</artifactId>
        <version>1.0.0</version>
        <!-- relativePath: Where to find parent POM
             Default: ../pom.xml
             <empty> means search in repository only -->
        <relativePath/>
    </parent>
    
    <!-- ========================================
         CHILD COORDINATES
         
         groupId and version inherited from parent
         Only artifactId is required
         ======================================== -->
    <artifactId>my-service</artifactId>
    
    <!-- ========================================
         DEPENDENCIES
         
         Versions inherited from parent's dependencyManagement
         NO VERSION SPECIFIED = uses parent's version
         ======================================== -->
    <dependencies>
        <!-- Spring Boot Starter (version from parent) -->
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-web</artifactId>
            <!-- NO VERSION! Inherited from parent -->
        </dependency>
        
        <!-- JUnit (version from parent) -->
        <dependency>
            <groupId>org.junit.jupiter</groupId>
            <artifactId>junit-jupiter</artifactId>
            <!-- NO VERSION! Inherited from parent -->
            <!-- scope inherited too -->
        </dependency>
        
        <!-- Custom version (overrides parent) -->
        <dependency>
            <groupId>com.google.guava</groupId>
            <artifactId>guava</artifactId>
            <version>32.1.3-jre</version>  <!-- Explicit version -->
        </dependency>
    </dependencies>
    
    <!-- ========================================
         BUILD PLUGINS
         
         Plugin versions inherited from parent's pluginManagement
         ======================================== -->
    <build>
        <plugins>
            <!-- Compiler plugin (version and config from parent) -->
            <plugin>
                <groupId>org.apache.maven.plugins</groupId>
                <artifactId>maven-compiler-plugin</artifactId>
                <!-- NO VERSION! Inherited from parent -->
            </plugin>
        </plugins>
    </build>
</project>

<!-- ========================================
     INHERITANCE HIERARCHY
     
     Super POM (built into Maven)
     ↓
     Company Parent POM
     ↓
     Product BOM (optional)
     ↓
     Multi-Module Parent (optional)
     ↓
     Individual Module
     
     Each level inherits and can override
     ======================================== -->

<!-- ========================================
     EFFECTIVE POM
     
     Result of merging all POMs in hierarchy
     
     View with: mvn help:effective-pom
     
     Shows:
     - All inherited properties
     - All resolved dependencies
     - All plugin configurations
     - Final build configuration
     ======================================== -->
```

### Properties and Variable Substitution

```xml
<project>
    <!-- ========================================
         PROPERTY TYPES
         ======================================== -->
    
    <properties>
        <!-- 1. USER-DEFINED PROPERTIES -->
        <my.custom.property>value</my.custom.property>
        
        <!-- 2. PROJECT PROPERTIES (from POM) -->
        <!-- Access: ${project.xxx} -->
        <!-- project.groupId -->
        <!-- project.artifactId -->
        <!-- project.version -->
        <!-- project.name -->
        
        <!-- 3. SETTINGS PROPERTIES (from settings.xml) -->
        <!-- Access: ${settings.xxx} -->
        <!-- settings.localRepository -->
        
        <!-- 4. ENVIRONMENT VARIABLES -->
        <!-- Access: ${env.xxx} -->
        <!-- env.JAVA_HOME -->
        <!-- env.PATH -->
        
        <!-- 5. SYSTEM PROPERTIES (Java system properties) -->
        <!-- Access: ${xxx} -->
        <!-- user.home -->
        <!-- java.version -->
        
        <!-- ========================================
             COMMON PROPERTY PATTERNS
             ======================================== -->
        
        <!-- Java version -->
        <java.version>17</java.version>
        <maven.compiler.source>${java.version}</maven.compiler.source>
        <maven.compiler.target>${java.version}</maven.compiler.target>
        
        <!-- Dependency versions -->
        <spring.version>6.1.0</spring.version>
        <jackson.version>2.16.0</jackson.version>
        
        <!-- Build configuration -->
        <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
        <project.reporting.outputEncoding>UTF-8</project.reporting.outputEncoding>
        
        <!-- Skip tests (can override: mvn -DskipTests) -->
        <skipTests>false</skipTests>
    </properties>
    
    <!-- ========================================
         USING PROPERTIES
         ======================================== -->
    
    <dependencies>
        <dependency>
            <groupId>org.springframework</groupId>
            <artifactId>spring-core</artifactId>
            <version>${spring.version}</version>  <!-- Property reference -->
        </dependency>
    </dependencies>
    
    <build>
        <finalName>${project.artifactId}-${project.version}</finalName>
        
        <!-- Filter resources (replaces ${xxx} in files) -->
        <resources>
            <resource>
                <directory>src/main/resources</directory>
                <filtering>true</filtering>  <!-- Enable property replacement -->
            </resource>
        </resources>
    </build>
</project>

<!-- ========================================
     EXAMPLE: application.properties with filtering
     ======================================== -->
<!-- 
# application.properties (before filtering)
app.name=${project.name}
app.version=${project.version}
app.build.timestamp=${maven.build.timestamp}

# application.properties (after filtering)
app.name=My Application
app.version=1.0.0
app.build.timestamp=2024-01-23T10:30:00Z
-->

<!-- ========================================
     PROPERTY PRECEDENCE (highest to lowest)
     
     1. CLI: mvn -Dmy.property=value
     2. POM properties
     3. Settings properties
     4. System properties
     5. Environment variables
     ======================================== -->
```

## 2. Dependency Scopes

### Understanding Maven Scopes

```xml
<!-- ========================================
     DEPENDENCY SCOPES
     
     Scope determines:
     1. When dependency is available (compile, test, runtime)
     2. Transitivity (does consumer get it?)
     3. Classpath inclusion (where it's added)
     ======================================== -->

<project>
    <dependencies>
        
        <!-- ========================================
             COMPILE SCOPE (Default)
             
             Available: Everywhere (compile, test, runtime)
             Transitive: YES (consumers get it too)
             Classpath: All classpaths
             
             Use for: Production dependencies needed at compile time
             ======================================== -->
        <dependency>
            <groupId>org.springframework</groupId>
            <artifactId>spring-core</artifactId>
            <version>6.1.0</version>
            <scope>compile</scope>  <!-- Can omit, it's default -->
        </dependency>
        
        <!-- Example: If consumers use your API that returns Spring types,
             they need Spring on their classpath too -->
        
        <!-- ========================================
             PROVIDED SCOPE
             
             Available: Compile and test (NOT runtime)
             Transitive: NO (consumers don't get it)
             Classpath: Compile and test only
             
             Use for: APIs provided by container/JDK
             Examples: servlet-api, lombok, jakarta.*, JDK tools
             
             Container provides at runtime
             ======================================== -->
        <dependency>
            <groupId>jakarta.servlet</groupId>
            <artifactId>jakarta.servlet-api</artifactId>
            <version>6.0.0</version>
            <scope>provided</scope>  <!-- Tomcat provides this -->
        </dependency>
        
        <dependency>
            <groupId>org.projectlombok</groupId>
            <artifactId>lombok</artifactId>
            <version>1.18.30</version>
            <scope>provided</scope>  <!-- Only needed at compile time -->
        </dependency>
        
        <!-- ========================================
             TEST SCOPE
             
             Available: Test only (compile-test, test)
             Transitive: NO
             Classpath: Test classpath only
             
             Use for: Test frameworks and utilities
             Examples: JUnit, Mockito, AssertJ
             ======================================== -->
        <dependency>
            <groupId>org.junit.jupiter</groupId>
            <artifactId>junit-jupiter</artifactId>
            <version>5.10.0</version>
            <scope>test</scope>
        </dependency>
        
        <dependency>
            <groupId>org.mockito</groupId>
            <artifactId>mockito-core</artifactId>
            <version>5.7.0</version>
            <scope>test</scope>
        </dependency>
        
        <!-- ========================================
             RUNTIME SCOPE
             
             Available: Test and runtime (NOT compile)
             Transitive: YES
             Classpath: Test and runtime only
             
             Use for: Implementation details not in public API
             Examples: JDBC drivers, logging implementations
             
             Code compiles without it, but needs it to run
             ======================================== -->
        <dependency>
            <groupId>com.mysql</groupId>
            <artifactId>mysql-connector-j</artifactId>
            <version>8.2.0</version>
            <scope>runtime</scope>  <!-- Loaded via JDBC, not in code -->
        </dependency>
        
        <dependency>
            <groupId>ch.qos.logback</groupId>
            <artifactId>logback-classic</artifactId>
            <version>1.4.14</version>
            <scope>runtime</scope>  <!-- SLF4J implementation -->
        </dependency>
        
        <!-- ========================================
             SYSTEM SCOPE (Discouraged, use local install instead)
             
             Available: Like provided
             Transitive: NO
             
             Points to JAR on filesystem
             Problems: Not portable, not in repository
             ======================================== -->
        <dependency>
            <groupId>com.example</groupId>
            <artifactId>custom-lib</artifactId>
            <version>1.0</version>
            <scope>system</scope>
            <systemPath>${project.basedir}/lib/custom-lib.jar</systemPath>
        </dependency>
        <!-- DON'T USE! Install to local repo instead:
             mvn install:install-file -Dfile=lib/custom-lib.jar ... -->
        
        <!-- ========================================
             IMPORT SCOPE (Special case)
             
             Only for <dependencyManagement> with type=pom
             Imports dependency management from BOM
             ======================================== -->
    </dependencies>
    
    <dependencyManagement>
        <dependencies>
            <dependency>
                <groupId>org.springframework.boot</groupId>
                <artifactId>spring-boot-dependencies</artifactId>
                <version>3.2.0</version>
                <type>pom</type>
                <scope>import</scope>  <!-- Import BOM -->
            </dependency>
        </dependencies>
    </dependencyManagement>
</project>

<!-- ========================================
     SCOPE COMPARISON TABLE
     
     ┌──────────┬─────────┬──────┬─────────┬────────────┐
     │ Scope    │ Compile │ Test │ Runtime │ Transitive │
     ├──────────┼─────────┼──────┼─────────┼────────────┤
     │ compile  │ ✓       │ ✓    │ ✓       │ YES        │
     │ provided │ ✓       │ ✓    │ ✗       │ NO         │
     │ test     │ ✗       │ ✓    │ ✗       │ NO         │
     │ runtime  │ ✗       │ ✓    │ ✓       │ YES        │
     │ system   │ ✓       │ ✓    │ ✗       │ NO         │
     └──────────┴─────────┴──────┴─────────┴────────────┘
     
     ======================================== -->

<!-- ========================================
     REAL-WORLD EXAMPLES
     ======================================== -->

<!-- WEB APPLICATION -->
<dependencies>
    <!-- compile: Spring framework (public API) -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
    </dependency>
    
    <!-- provided: Servlet API (Tomcat provides) -->
    <dependency>
        <groupId>jakarta.servlet</groupId>
        <artifactId>jakarta.servlet-api</artifactId>
        <scope>provided</scope>
    </dependency>
    
    <!-- runtime: PostgreSQL driver (not in code) -->
    <dependency>
        <groupId>org.postgresql</groupId>
        <artifactId>postgresql</artifactId>
        <scope>runtime</scope>
    </dependency>
    
    <!-- test: Testing frameworks -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-test</artifactId>
        <scope>test</scope>
    </dependency>
</dependencies>
```

## 3. Dependency Management vs Dependencies

### Understanding the Difference

```xml
<!-- ========================================
     KEY DIFFERENCE:
     
     <dependencyManagement>:
     - Defines versions ONLY
     - Does NOT add dependencies
     - Centralizes version control
     - Used in parent POMs
     
     <dependencies>:
     - Actually includes dependencies
     - Can omit version if in dependencyManagement
     - Used in child POMs
     ======================================== -->

<!-- ========================================
     PARENT POM with dependencyManagement
     ======================================== -->
<project>
    <groupId>com.example</groupId>
    <artifactId>parent-pom</artifactId>
    <version>1.0.0</version>
    <packaging>pom</packaging>
    
    <!-- ========================================
         DEPENDENCY MANAGEMENT
         
         Defines versions for children
         Does NOT add these dependencies
         ======================================== -->
    <dependencyManagement>
        <dependencies>
            <dependency>
                <groupId>org.springframework.boot</groupId>
                <artifactId>spring-boot-starter-web</artifactId>
                <version>3.2.0</version>
            </dependency>
            
            <dependency>
                <groupId>org.junit.jupiter</groupId>
                <artifactId>junit-jupiter</artifactId>
                <version>5.10.0</version>
                <scope>test</scope>
            </dependency>
            
            <dependency>
                <groupId>com.google.guava</groupId>
                <artifactId>guava</artifactId>
                <version>32.1.3-jre</version>
            </dependency>
        </dependencies>
    </dependencyManagement>
    
    <!-- NO DEPENDENCIES IN PARENT
         Children choose what they need -->
</project>

<!-- ========================================
     CHILD MODULE 1
     ======================================== -->
<project>
    <parent>
        <groupId>com.example</groupId>
        <artifactId>parent-pom</artifactId>
        <version>1.0.0</version>
    </parent>
    
    <artifactId>web-module</artifactId>
    
    <!-- ========================================
         DEPENDENCIES
         
         Versions inherited from parent
         ======================================== -->
    <dependencies>
        <!-- Version from parent's dependencyManagement -->
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-web</artifactId>
            <!-- NO VERSION! Uses 3.2.0 from parent -->
        </dependency>
        
        <dependency>
            <groupId>org.junit.jupiter</groupId>
            <artifactId>junit-jupiter</artifactId>
            <!-- NO VERSION! Uses 5.10.0 from parent -->
            <!-- Scope inherited too -->
        </dependency>
        
        <!-- Guava NOT included here
             Only includes what's needed -->
    </dependencies>
</project>

<!-- ========================================
     CHILD MODULE 2
     ======================================== -->
<project>
    <parent>
        <groupId>com.example</groupId>
        <artifactId>parent-pom</artifactId>
        <version>1.0.0</version>
    </parent>
    
    <artifactId>util-module</artifactId>
    
    <dependencies>
        <!-- Different dependencies, same versions -->
        <dependency>
            <groupId>com.google.guava</groupId>
            <artifactId>guava</artifactId>
            <!-- Version 32.1.3-jre from parent -->
        </dependency>
        
        <dependency>
            <groupId>org.junit.jupiter</groupId>
            <artifactId>junit-jupiter</artifactId>
            <!-- Same JUnit version across all modules -->
        </dependency>
        
        <!-- Spring NOT included here
             Each module picks what it needs -->
    </dependencies>
</project>

<!-- ========================================
     BILL OF MATERIALS (BOM) PATTERN
     
     Special POM that only defines versions
     Imported using scope=import
     ======================================== -->

<!-- BOM POM -->
<project>
    <groupId>com.example</groupId>
    <artifactId>platform-bom</artifactId>
    <version>1.0.0</version>
    <packaging>pom</packaging>
    
    <dependencyManagement>
        <dependencies>
            <!-- All platform dependencies with versions -->
            <dependency>
                <groupId>org.springframework</groupId>
                <artifactId>spring-core</artifactId>
                <version>6.1.0</version>
            </dependency>
            
            <dependency>
                <groupId>com.fasterxml.jackson.core</groupId>
                <artifactId>jackson-databind</artifactId>
                <version>2.16.0</version>
            </dependency>
            
            <!-- Many more dependencies... -->
        </dependencies>
    </dependencyManagement>
</project>

<!-- IMPORTING BOM -->
<project>
    <dependencyManagement>
        <dependencies>
            <!-- Import BOM -->
            <dependency>
                <groupId>com.example</groupId>
                <artifactId>platform-bom</artifactId>
                <version>1.0.0</version>
                <type>pom</type>
                <scope>import</scope>  <!-- Special scope -->
            </dependency>
        </dependencies>
    </dependencyManagement>
    
    <dependencies>
        <!-- Now can use without versions -->
        <dependency>
            <groupId>org.springframework</groupId>
            <artifactId>spring-core</artifactId>
            <!-- Version from BOM -->
        </dependency>
    </dependencies>
</project>

<!-- ========================================
     REAL-WORLD EXAMPLE: Spring Boot BOM
     ======================================== -->
<project>
    <dependencyManagement>
        <dependencies>
            <!-- Import Spring Boot BOM -->
            <dependency>
                <groupId>org.springframework.boot</groupId>
                <artifactId>spring-boot-dependencies</artifactId>
                <version>3.2.0</version>
                <type>pom</type>
                <scope>import</scope>
            </dependency>
        </dependencies>
    </dependencyManagement>
    
    <dependencies>
        <!-- All Spring Boot starters now have managed versions -->
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-web</artifactId>
        </dependency>
        
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-data-jpa</artifactId>
        </dependency>
        
        <!-- Transitive dependencies also managed -->
    </dependencies>
</project>

<!-- ========================================
     WHY USE DEPENDENCY MANAGEMENT?
     
     1. CONSISTENCY:
        - All modules use same version
        - No version conflicts
     
     2. CENTRALIZATION:
        - Update version in one place
        - All modules get update
     
     3. FLEXIBILITY:
        - Modules choose what they need
        - No unused dependencies
     
     4. OVERRIDE WHEN NEEDED:
        - Child can specify different version
        - But discouraged for consistency
     ======================================== -->
```

**Key Takeaways:**

```
The Maven Triangle:
Lifecycle → What to do (compile, test, package)
Dependencies → What you need (libraries)
Plugins → How to do it (compiler, surefire)

Dependency Scopes:
compile  → Default, everywhere
provided → Compile-only (container supplies)
test     → Test-only
runtime  → Execution-only (JDBC drivers)

dependencyManagement vs dependencies:
dependencyManagement → Defines versions (parent)
dependencies → Actually includes (child)

Multi-Module Pattern:
Parent (pom packaging)
├── Common (shared utilities)
├── Service (business logic)
└── Web (REST API, depends on service)

Profile Activation:
mvn clean install -Pprod
<activeByDefault>true</activeByDefault>
<property><name>env</name></property>

Repository Hierarchy:
Local (~/.m2) → Check first
Central → Default public
Private → Company artifacts
```
