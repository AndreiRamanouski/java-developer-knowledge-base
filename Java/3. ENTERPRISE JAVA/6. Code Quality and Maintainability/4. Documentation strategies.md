# Documentation Strategies

## Overview

Documentation is a form of communication—it bridges the gap between what the code does and why it exists. Good documentation accelerates onboarding, prevents repeated mistakes, and captures decisions that would otherwise be lost when team members leave. This guide covers six documentation strategies: Javadoc Best Practices (document the contract not the implementation—what the method promises, not how it delivers, include preconditions and postconditions, document exceptions, use @param @return @throws consistently, avoid restating what the code already says), When to Document and When Not To (public APIs need documentation, private methods usually don't, document WHY not WHAT, don't document the obvious, stale documentation is worse than no documentation, if you need documentation to explain WHAT the code does the code is probably unclear), README Files (the first thing developers see—what the project does, how to get started quickly, how to build and run, architecture overview at high level, links to deeper documentation, kept up to date or it becomes misleading), Architecture Decision Records (capture significant decisions with context—what was decided, why that choice was made, what alternatives were considered, what the consequences are, immutable history of reasoning that survives team turnover), API Documentation with Swagger/OpenAPI (generate interactive documentation from annotations—document endpoints with @Operation, describe request/response with @Schema, include examples that actually work, Swagger UI lets consumers try the API in the browser, kept in sync because it's derived from code), and Living Documentation (documentation that evolves with the code—tests as executable documentation, annotation processors generate docs from code, BDD specs that stakeholders can read, architecture diagrams generated from code structure, documentation that cannot drift from reality). The key insight: documentation is overhead—it must be maintained or it becomes a liability. The best documentation is self-documenting code. The second-best is documentation that is automatically generated or validated. The third-best is documentation that is easy to keep in sync. Manual prose documentation is the last resort, reserved for high-level concepts that code cannot express. Common mistakes: documenting implementation details (they change), writing documentation that restates the code (waste of time), creating documentation that has no owner (it rots), treating documentation as a one-time activity (it must evolve), writing for yourself (write for the next person), over-documenting (signal-to-noise ratio drops).

**The Philosophy:**

```
"The best documentation is no documentation."

This sounds contradictory, but it means:
The best documentation is code so clear that
no external documentation is needed.

When external documentation IS needed, make it:
1. Automatically generated (cannot drift)
2. Automatically validated (catches drift)
3. Co-located with code (easy to update)
4. High-signal (only what cannot be expressed in code)

Documentation has a cost:
- Time to write
- Time to maintain
- Confusion when it's wrong
- Overhead when reading

Good documentation pays for itself.
Bad documentation is worse than nothing.
```

## 1. Javadoc Best Practices

### Documenting the Contract, Not the Implementation

```java
// ========================================
// THE RULE: Javadoc documents the CONTRACT.
//
// Contract = What the method promises:
//   - What it does (high level)
//   - What inputs it accepts
//   - What it returns
//   - What exceptions it throws
//   - What preconditions must be met
//   - What postconditions are guaranteed
//
// NOT: How the method does it.
// ========================================

// ❌ BAD JAVADOC: Restates the signature, adds no value
/**
 * Gets the user.
 * 
 * @param id the id
 * @return the user
 */
public User getUser(String id) {
    return userRepository.findById(id);
}

// ✅ GOOD JAVADOC: Documents the contract and edge cases
/**
 * Retrieves a user by their unique identifier.
 * 
 * <p>This method queries the user repository and returns the user
 * if found. The returned user includes their profile, preferences,
 * and current subscription status, but does NOT include their
 * authentication credentials for security reasons.</p>
 * 
 * @param id  the unique user identifier; must not be null or blank
 * @return    the user with the given ID, never null
 * @throws IllegalArgumentException if id is null or blank
 * @throws UserNotFoundException    if no user exists with this ID
 * @see #createUser(UserRequest)
 * @see UserRepository#findById(String)
 */
public User getUser(String id) {
    if (id == null || id.isBlank()) {
        throw new IllegalArgumentException("User ID must not be blank");
    }
    return userRepository.findById(id)
        .orElseThrow(() -> new UserNotFoundException(id));
}
```

## 2. When to Document and When Not To

### The Decision Framework

```java
// ========================================
// THE DECISION TREE
// ========================================

/**
 * Before writing documentation, ask these questions IN ORDER:
 * 
 * 1. "Can I make the CODE say this instead?"
 *    → If yes: Rename, extract method, use meaningful types.
 *              No documentation needed.
 * 
 * 2. "Does this explain WHAT the code does?"
 *    → If yes: The code should already say what it does.
 *              Delete or refactor the code.
 * 
 * 3. "Is this a public API or library method?"
 *    → If yes: Document the contract (inputs, outputs, exceptions).
 * 
 * 4. "Does this explain WHY this choice was made?"
 *    → If yes: This is valuable. Document it.
 * 
 * 5. "Is there a non-obvious constraint, gotcha, or limitation?"
 *    → If yes: Document it with an example if possible.
 * 
 * 6. "Am I documenting something that will change often?"
 *    → If yes: This documentation will go stale. Reconsider.
 *              Can it be expressed as a test instead?
 * 
 * If you answered "no" to all of the above:
 *   → Don't document it. The code is sufficient.
 */

// ========================================
// DOCUMENT: Public API (external contract)
// ========================================

/**

 * Public API that external teams depend on. Contract must be clear.
 * Changes here are breaking changes.
```

## 4. Architecture Decision Records (ADR)

### Capturing the "Why" Behind Decisions

```markdown
# ========================================
# ADR FORMAT: MADR (Markdown ADR)
# ========================================

# ADR-0005: Use PostgreSQL for Order Storage

**Status:** Accepted

**Date:** 2024-01-15

**Deciders:** Jane Doe (Tech Lead), John Smith (DBA), Alice Johnson (SRE)

---

## Context and Problem Statement

We need to choose a database for storing order data in the new order management service. The service will handle:
- 10,000 orders per day (current volume)
- Expected to grow to 100,000 orders per day within 2 years
- Complex queries for order history, analytics, and reporting
- Strong consistency requirements (orders cannot be lost or duplicated)
- ACID transactions required (inventory decrements must be atomic with order creation)

What database technology should we use?

---

## Decision Drivers

* **Consistency:** ACID transactions are non-negotiable for financial data
* **Query flexibility:** Complex joins for analytics and reporting
* **Team expertise:** Team has 5 years of PostgreSQL experience, no MongoDB experience
* **Operational maturity:** Need proven backup/restore and monitoring tools
* **Cost:** Budget constraint of $500/month for database hosting
* **Performance:** Must handle 100K writes per day with sub-100ms latency

---

## Considered Options

* **Option 1:** PostgreSQL
* **Option 2:** MongoDB  
* **Option 3:** DynamoDB
* **Option 4:** MySQL

---

## Decision Outcome

**Chosen option:** PostgreSQL

**Reasoning:**
- Provides ACID guarantees we require for financial transactions
- Excellent support for complex queries and joins (needed for analytics)
- Team has deep expertise (reduces risk and operational burden)
- JSON/JSONB support provides schema flexibility where needed
- Mature ecosystem (pgAdmin, pg_dump, WAL archiving, connection poolers)
- Can handle our current and projected scale (100K orders/day is well within Postgres capacity)

---

## Consequences

### Positive
* Strong consistency guarantees eliminate entire classes of bugs
* Rich query capabilities enable product analytics without ETL pipeline
* Team can operate the database confidently from day one
* Existing monitoring and backup infrastructure applies

### Negative
* Not horizontally scalable without additional complexity (CitusDB, pg_shard)
  - Mitigation: Current projections don't require horizontal scaling for 5+ years
* Connection pool management required for high concurrency
  - Mitigation: Use pgBouncer (team has experience with this)
* Vertical scaling costs increase with volume
  - Mitigation: Optimize queries; implement read replicas if needed

### Neutral
* Need to run database migrations (Flyway)
* Must implement connection pooling (pgBouncer)

---

## Alternatives Considered

### MongoDB
**Pros:**
- Horizontal scaling built-in
- Flexible schema

**Cons:**
- No multi-document ACID transactions in versions we'd deploy (3.6)
- Team has zero operational experience
- Complex queries (aggregation pipeline) harder to write and debug
- Higher operational risk

**Why rejected:** Lack of ACID transactions is a deal-breaker. The risk of eventual consistency bugs in financial data is unacceptable.

---

### DynamoDB
**Pros:**
- Fully managed (no operational burden)
- Automatic scaling
- Strong consistency model available

**Cons:**
- Query limitations (no joins, limited filtering)
- Would require separate analytics database
- Vendor lock-in to AWS
- Cost unpredictable at scale

**Why rejected:** Query limitations would force us to build a separate analytics pipeline. Total cost of ownership (DynamoDB + analytics database + ETL) exceeds PostgreSQL.

---

### MySQL
**Pros:**
- ACID transactions
- Similar operational profile to PostgreSQL
- Mature ecosystem

**Cons:**
- No JSON/JSONB support (we need flexible metadata fields)
- Team prefers PostgreSQL's advanced features (CTEs, window functions)
- No significant benefit over PostgreSQL

**Why rejected:** No compelling advantage over PostgreSQL. Team familiarity with PostgreSQL tips the balance.
```

## Summary

```
DOCUMENTATION TYPE       WHEN TO USE                          MAINTENANCE COST
─────────────────────────────────────────────────────────────────────────────
Self-Documenting Code    Always (first choice)                Zero (is the code)
Javadoc                  Public APIs, complex contracts       Low (co-located)
README.md                Every project (front door)           Medium (manual)
ADR                      Significant decisions                Zero (immutable history)
Swagger/OpenAPI          REST APIs                            Zero (generated)
Living Documentation     Complex business rules               Zero (generated/validated)

─────────────────────────────────────────────────────────────────────────────

THE HIERARCHY OF DOCUMENTATION VALUE:

1. CODE THAT NEEDS NO DOCUMENTATION
   - Clear names, simple functions, obvious flow
   - Cost: Zero to maintain
   - Value: Highest (always correct)

2. DOCUMENTATION THAT CANNOT DRIFT
   - Swagger (generated from code)
   - Tests as specs (executable)
   - ArchUnit (enforced by build)
   - Cost: Zero to maintain (auto-updated)
   - Value: High (always in sync)

3. DOCUMENTATION CLOSE TO CODE
   - Javadoc (in the same file)
   - Inline comments (for WHY)
   - Cost: Low (easy to update)
   - Value: Medium (can drift but less likely)

4. DOCUMENTATION SEPARATE FROM CODE
   - README, ADR, architecture docs
   - Cost: Medium-High (manual updates)
   - Value: Medium (drifts if not maintained)

5. DOCUMENTATION FAR FROM CODE
   - Wiki pages, Google Docs, Confluence
   - Cost: High (often forgotten)
   - Value: Low (drifts quickly)

─────────────────────────────────────────────────────────────────────────────

DECISION FRAMEWORK:

  "Do I need to explain this?"
    → If no: Don't document. The code is clear.

  "Is this a public API or complex contract?"
    → If yes: Write Javadoc.

  "Is this an endpoint?"
    → If yes: Use Swagger/OpenAPI annotations.

  "Is this a significant decision (database, architecture, framework)?"
    → If yes: Write an ADR.

  "Is this a business rule that needs to be traceable?"
    → If yes: Use living documentation (tests, annotations).

  "Is this the first thing a developer sees?"
    → If yes: Document in README.md.

─────────────────────────────────────────────────────────────────────────────

GOLDEN RULES:

✓ Prefer code clarity over comments
✓ Prefer generated docs over manual docs
✓ Prefer co-located docs over separate docs
✓ Document WHY, not WHAT
✓ Keep READMEs short and up-to-date
✓ ADRs are immutable (record history, don't edit)
✓ Swagger annotations live in controllers (single source of truth)
✓ Tests are executable specifications
✓ Stale documentation is worse than no documentation

✗ Don't document the obvious
✗ Don't let docs drift from code
✗ Don't create documentation without an owner
✗ Don't treat documentation as one-time work
✗ Don't over-document
```

**Key Takeaway:**

```
The best documentation is code so clear
it needs no documentation.

When documentation IS needed:
1. Generate it (Swagger, ArchUnit, annotation processors)
2. Co-locate it (Javadoc, inline comments)
3. Make it immutable (ADRs)
4. Keep it minimal (README)

Documentation is overhead.
It must be maintained or it becomes a LIABILITY.

Stale documentation is worse than no documentation.
It actively misleads.

132 guides in — you now know not just HOW to document,
but WHEN to document and WHEN to make the code
so clear that documentation becomes unnecessary.
```
