# Code Review Best Practices

## Overview

Code review is one of the highest-leverage activities in software development—it catches bugs before production, spreads knowledge across the team, maintains code quality standards, and builds shared ownership. This guide covers six critical aspects: What to Look For (priorities matter—correctness first, then security, then architecture, then readability, then style last; focus on what the machine cannot check), Giving Constructive Feedback (be kind, be specific, explain why, offer alternatives, praise good work, focus on the code not the person, ask questions instead of making demands), Automated Checks Before Review (never waste human time on what machines can check—linters, formatters, tests, security scans, coverage reports must pass before human review begins), Review Checklist (systematic approach prevents blind spots—functionality, error handling, security, performance, tests, documentation, backwards compatibility), Small Focused Changes (PRs over 400 lines have <10% chance of catching bugs, under 200 lines are reviewed in <1 hour, single responsibility principle applies to PRs too, split large changes into logical commits), and Review Turnaround Time (reviews blocked >24 hours kill productivity, aim for same-day feedback, set team SLAs, balance thoroughness with velocity, context switching is expensive). The key insight: code review is not a gate—it is a conversation. The goal is not to find every possible flaw, but to catch serious problems, ensure readability, and share knowledge. Perfect is the enemy of done. Common mistakes: nitpicking style when automated tools exist, blocking on subjective preferences, reviewing 1000-line PRs (nobody reads them carefully), leaving vague feedback ("this is unclear"), reviewing without running the code, waiting days to review (context is lost), approving without reading (rubber stamping), writing defensive comments when receiving feedback. The relationship between review quality and PR size is exponential—a 200-line PR gets careful review, a 1000-line PR gets cursory approval.

**The Philosophy:**

```
"Code review is not a gatekeeping exercise.
 It is a collaborative conversation about code quality."

The primary goals:
1. Catch bugs before production
2. Ensure code is maintainable
3. Share knowledge across the team
4. Maintain consistent standards
5. Build collective ownership

NOT the goals:
- Demonstrate your superiority
- Enforce your personal style preferences
- Rewrite it the way YOU would have done it
- Find every possible improvement
- Achieve perfection

The best code reviews are SHORT conversations
that catch serious problems and build trust.
```

## 1. What to Look For in Reviews

### The Priority Hierarchy

```
PRIORITY 1 — CORRECTNESS (Always review)
├─ Does this actually solve the problem?
├─ Are there logical errors?
├─ Are edge cases handled?
├─ Will this break in production?
└─ Are the tests sufficient?

PRIORITY 2 — SECURITY (Always review)
├─ SQL injection vulnerabilities?
├─ Authentication/authorization bypasses?
├─ Sensitive data exposure?
├─ Input validation missing?
└─ Cryptographic weaknesses?

PRIORITY 3 — ARCHITECTURE (Review if significant)
├─ Does this fit the architecture?
├─ Is this the right place for this logic?
├─ Are dependencies appropriate?
├─ Does this introduce coupling?
└─ Is the abstraction level correct?

PRIORITY 4 — READABILITY (Review if unclear)
├─ Can I understand this in 30 seconds?
├─ Are names meaningful?
├─ Is the logic straightforward?
├─ Is the code self-documenting?
└─ Are comments necessary and helpful?

PRIORITY 5 — STYLE (Let automation handle this)
├─ Formatting (spaces, indentation)
├─ Import ordering
├─ Line length
├─ Naming conventions
└─ Code organization

If automated checks catch it → Don't review it.
If it's subjective preference → Let it go.
If it's wrong or dangerous → Always comment.
```

```java
// ========================================
// PRIORITY 1: CORRECTNESS ISSUES
// ========================================

// ❌ LOGICAL ERROR: Off-by-one
public List<Order> getRecentOrders(int count) {
    return orders.subList(orders.size() - count, orders.size());
    // 🚨 BUG: What if orders.size() < count? IndexOutOfBoundsException!
}

// 💬 FEEDBACK:
// "This will throw IndexOutOfBoundsException if `count` exceeds 
// the number of orders. Consider:
// 
// int start = Math.max(0, orders.size() - count);
// return orders.subList(start, orders.size());
// 
// Also, add a test case for this edge case."

// ❌ EDGE CASE: Null not handled
public Money calculateDiscount(Customer customer) {
    if (customer.isPremium()) {
        return customer.getOrderTotal().multiply(0.15);
        // 🚨 BUG: What if getOrderTotal() returns null?
    }
    return Money.ZERO;
}

// 💬 FEEDBACK:
// "What happens if customer.getOrderTotal() returns null?
// Should we return ZERO or throw an exception? Let's add a null
// check or ensure getOrderTotal() never returns null."

// ❌ RACE CONDITION: Not thread-safe
private int counter = 0;

public int getNextId() {
    return ++counter;  // 🚨 BUG: Not atomic in concurrent scenarios
}

// 💬 FEEDBACK:
// "This has a race condition in multi-threaded environments.
// Two threads could get the same ID. Consider using AtomicInteger:
//
// private final AtomicInteger counter = new AtomicInteger(0);
// public int getNextId() { return counter.incrementAndGet(); }
//
// Or document that this class is not thread-safe."
```

```java
// ========================================
// PRIORITY 2: SECURITY ISSUES
// ========================================

// ❌ SQL INJECTION: User input not sanitized
public List<User> searchUsers(String name) {
    String sql = "SELECT * FROM users WHERE name = '" + name + "'";
    // 🚨 SECURITY: SQL injection if name = "'; DROP TABLE users; --"
    return jdbcTemplate.query(sql, userRowMapper);
}

// 💬 FEEDBACK:
// "🔴 CRITICAL: This is vulnerable to SQL injection. Never concatenate
// user input into SQL. Use parameterized queries:
//
// String sql = \"SELECT * FROM users WHERE name = ?\";
// return jdbcTemplate.query(sql, userRowMapper, name);
//
// This MUST be fixed before merging."

// ❌ AUTHENTICATION BYPASS: Missing auth check
@GetMapping("/api/users/{id}")
public ResponseEntity<User> getUser(@PathVariable String id) {
    User user = userService.findById(id);
    return ResponseEntity.ok(user);
    // 🚨 SECURITY: Any user can view any other user's data!
}

// 💬 FEEDBACK:
// "🔴 CRITICAL: No authorization check. This allows any authenticated
// user to view any other user's data. Add:
//
// if (!currentUser.canViewUser(id)) {
//     throw new ForbiddenException();
// }
//
// Or use @PreAuthorize(\"@authService.canViewUser(#id)\")."

// ❌ SENSITIVE DATA: Logging passwords
public User authenticate(String email, String password) {
    logger.info("Login attempt: email={}, password={}", email, password);
    // 🚨 SECURITY: Password logged in plain text!
    return authService.authenticate(email, password);
}

// 💬 FEEDBACK:
// "🔴 CRITICAL: Never log passwords. Remove the password from the log:
//
// logger.info(\"Login attempt: email={}\", email);
//
// This violates security policy and could lead to credential theft."
```

```java
// ========================================
// PRIORITY 3: ARCHITECTURE ISSUES
// ========================================

// ❌ WRONG LAYER: Business logic in controller
@PostMapping("/api/orders")
public ResponseEntity<Order> createOrder(@RequestBody OrderRequest req) {
    // Validation
    if (req.getItems().isEmpty()) {
        return ResponseEntity.badRequest().build();
    }
    
    // Business logic — should NOT be here!
    double total = 0;
    for (OrderItem item : req.getItems()) {
        Product product = productRepo.findById(item.getProductId());
        total += product.getPrice() * item.getQuantity();
        product.setStock(product.getStock() - item.getQuantity());
        productRepo.save(product);
    }
    
    Order order = new Order(req.getCustomerId(), total);
    orderRepo.save(order);
    return ResponseEntity.ok(order);
}

// 💬 FEEDBACK:
// "This puts business logic in the controller. Controllers should
// only handle HTTP concerns (request/response mapping). The pricing
// calculation and inventory management belong in OrderService.
//
// Consider:
// Order order = orderService.createOrder(req);
// return ResponseEntity.ok(order);
//
// This also lacks transaction management — if saving the order fails,
// the inventory is already decremented. OrderService should handle
// this in a @Transactional method."

// ❌ TIGHT COUPLING: Direct instantiation
public class OrderService {
    private final EmailService emailService = new GmailEmailService();
    // 🚨 ARCHITECTURE: Tightly coupled to Gmail implementation
    
    public void placeOrder(Order order) {
        orderRepository.save(order);
        emailService.sendConfirmation(order);  // Can't test without Gmail
    }
}

// 💬 FEEDBACK:
// "This is tightly coupled to GmailEmailService. Inject the dependency
// instead:
//
// private final EmailService emailService;
//
// public OrderService(EmailService emailService) {
//     this.emailService = emailService;
// }
//
// This makes testing easier (inject a mock) and lets us swap email
// providers without changing OrderService."
```

```java
// ========================================
// PRIORITY 4: READABILITY ISSUES
// ========================================

// ❌ UNCLEAR: Magic numbers and cryptic logic
public boolean isEligible(Customer c) {
    return c.getAge() > 18 && c.getScore() >= 650 
        && c.getDays() < 90 && c.getFlag();
    // What do these numbers mean? What is "flag"?
}

// 💬 FEEDBACK:
// "This is hard to understand. What do the magic numbers represent?
// What is 'flag'? Consider extracting to named methods:
//
// public boolean isEligibleForCredit(Customer customer) {
//     return customer.isAdult()
//         && customer.hasSufficientCreditScore()
//         && customer.isRecentlyActive()
//         && customer.hasVerifiedIdentity();
// }
//
// Each predicate method has a clear name and can be tested independently."

// ❌ COMPLEX: Too many levels of nesting
public void processOrder(Order order) {
    if (order != null) {
        if (order.getCustomer() != null) {
            if (order.getCustomer().isActive()) {
                if (order.getItems().size() > 0) {
                    for (OrderItem item : order.getItems()) {
                        if (item.getProduct() != null) {
                            if (item.getProduct().isAvailable()) {
                                // Actual logic buried 6 levels deep!
                            }
                        }
                    }
                }
            }
        }
    }
}

// 💬 FEEDBACK:
// "This is hard to follow due to deep nesting. Consider early returns
// to flatten the structure:
//
// if (order == null || order.getCustomer() == null) return;
// if (!order.getCustomer().isActive()) return;
// if (order.getItems().isEmpty()) return;
//
// for (OrderItem item : order.getItems()) {
//     if (item.getProduct() == null || !item.getProduct().isAvailable()) {
//         continue;
//     }
//     // Actual logic at top level
// }
//
// This makes the happy path easier to follow."
```

## 2. Giving Constructive Feedback

### The Art of the Code Review Comment

```java
// ========================================
// DESTRUCTIVE FEEDBACK (Don't do this)
// ========================================

// ❌ ATTACKS THE PERSON
// "You clearly don't understand how databases work."
// "Did you even test this?"
// "This is terrible code."

// ❌ VAGUE AND UNHELPFUL
// "This is confusing."
// "Fix this."
// "Bad practice."
// "Needs improvement."

// ❌ DEMANDING WITHOUT EXPLANATION
// "Change this to use a HashMap."
// "Refactor this."
// "Use a different pattern here."

// ❌ NITPICKING PERSONAL STYLE
// "I would have used a for loop instead of streams here."
// "I prefer early returns over else blocks."
// "I don't like this naming convention."

// ========================================
// CONSTRUCTIVE FEEDBACK (Do this)
// ========================================

// ✅ BE SPECIFIC + EXPLAIN WHY + OFFER SOLUTION

// Example 1: Correctness issue
public List<Order> getRecentOrders(int count) {
    return orders.subList(orders.size() - count, orders.size());
}

// ❌ "This is wrong."
// ✅ "This will throw IndexOutOfBoundsException when `count` exceeds 
//    the list size. Consider:
//
//    int start = Math.max(0, orders.size() - count);
//    return orders.subList(start, orders.size());
//
//    This handles the edge case gracefully."

// Example 2: Performance issue
public boolean contains(String word) {
    for (String w : words) {  // words is a List
        if (w.equals(word)) return true;
    }
    return false;
}

// ❌ "Use a Set."
// ✅ "This is O(n) for each lookup. If `contains` is called frequently,
//    consider using a HashSet instead of a List. This would make
//    lookups O(1) on average. Worth it if we expect many lookups.
//    
//    private final Set<String> words = new HashSet<>();"

// Example 3: Readability issue
public int calc(int x, int y, boolean f) {
    return f ? x * y * 1.08 : x * y;
}

// ❌ "Bad names."
// ✅ "The names `x`, `y`, and `f` make this hard to understand. Could
//    you rename to something like:
//
//    public int calculateTotal(int price, int quantity, boolean applyTax) {
//        int subtotal = price * quantity;
//        return applyTax ? (int)(subtotal * TAX_MULTIPLIER) : subtotal;
//    }
//
//    This makes the intent clear to future readers."
```

```java
// ========================================
// TONE MATTERS: Frame as questions when possible
// ========================================

// ❌ DEMANDING (sounds accusatory)
// "Remove this method. It's not used anywhere."

// ✅ ASKING (opens dialogue)
// "I don't see where this method is called. Is it needed for a future
//  feature, or can we remove it to reduce complexity?"

// ──────────────────────────────────────

// ❌ JUDGING
// "You're doing this wrong."

// ✅ COLLABORATING
// "Have you considered using Optional here? It might make the null
//  handling more explicit. What do you think?"

// ──────────────────────────────────────

// ❌ ABSOLUTIST
// "This must be changed."

// ✅ CONTEXTUAL
// "This could lead to issues if multiple threads access this field.
//  Should we make it final or synchronized? Or is single-threaded
//  access guaranteed here?"

// ──────────────────────────────────────

// USE "WE" NOT "YOU"
// ❌ "You need to add error handling here."
// ✅ "We should add error handling here to catch network failures."

// EXPLAIN THE IMPACT
// ❌ "This is inefficient."
// ✅ "This queries the database in a loop (N+1 problem). For 100 orders,
//     this makes 100 queries instead of 1. Consider using JOIN or
//     batch fetching."

// OFFER ALTERNATIVES
// ❌ "Don't do it this way."
// ✅ "Another approach would be to cache this result. That would avoid
//     recalculating on every request. Trade-off: slightly stale data
//     vs. better performance. What do you think fits our use case?"
```

```java
// ========================================
// PRAISE GOOD WORK (Often forgotten!)
// ========================================

// ✅ Acknowledge clever solutions
// "Nice! Using a priority queue here is a clever way to keep the top K
//  elements without sorting the entire list. Clean solution."

// ✅ Recognize learning and improvement
// "Great improvement over the previous version. The extracted methods
//  make this much easier to follow. 👍"

// ✅ Highlight good practices
// "I appreciate the thorough test coverage here. The edge cases are
//  well-covered."

// ✅ Call out good documentation
// "The Javadoc here is really helpful. The example clarifies exactly
//  when to use this method."

// ✅ Acknowledge consideration for others
// "Thanks for adding the migration script. This will make deployment
//  much smoother for the ops team."

// People remember positive feedback more than negative.
// Reinforce behaviors you want to see more of.
```

```java
// ========================================
// DISTINGUISH BLOCKING vs NON-BLOCKING
// ========================================

/**
 * Use prefixes to signal priority:
 * 
 * 🔴 [BLOCKING] or [CRITICAL]: Must be fixed before merge
 * 🟡 [SUGGESTION] or [OPTIONAL]: Nice to have, but not required
 * 🔵 [QUESTION]: Asking for clarification, not requesting change
 * 🟢 [PRAISE]: Positive feedback
 * ⚪ [NITS]: Tiny style things (consider not commenting at all)
 */

// 🔴 [BLOCKING] This has a race condition that will cause duplicate
//    orders in production. Must be fixed.

// 🟡 [SUGGESTION] Consider extracting this 20-line block into a named
//    method for readability. Not required, but would help future readers.

// 🔵 [QUESTION] Why do we need to recalculate this on every request?
//    Could we cache it?

// 🟢 [PRAISE] This is a clever use of Java's try-with-resources.
//    Ensures the connection is always closed.

// ⚪ [NITS] Typo in the comment: "teh" → "the"
//    (Consider: Is this comment even needed? Maybe just fix it yourself
//     or let the spell checker catch it.)
```

## 3. Automated Checks Before Review

### Never Waste Human Time on What Machines Can Check

```yaml
# ========================================
# GITHUB ACTIONS: CI/CD Pipeline
# ========================================

# .github/workflows/pull-request.yml
name: Pull Request Checks

on:
  pull_request:
    branches: [ main, develop ]

jobs:
  automated-checks:
    runs-on: ubuntu-latest
    
    steps:
      - uses: actions/checkout@v3
      
      # ──────────────────────────────────
      # 1. CODE FORMATTING (Fail if wrong)
      # ──────────────────────────────────
      - name: Check code formatting
        run: mvn spotless:check
        # Fails if code isn't formatted per config
        # Developer runs: mvn spotless:apply locally
      
      # ──────────────────────────────────
      # 2. STATIC ANALYSIS
      # ──────────────────────────────────
      - name: Run Checkstyle
        run: mvn checkstyle:check
        # Enforces naming, style, structure rules
      
      - name: Run PMD
        run: mvn pmd:check
        # Detects potential bugs, dead code
      
      - name: Run SpotBugs
        run: mvn spotbugs:check
        # Detects common bug patterns
      
      # ──────────────────────────────────
      # 3. SECURITY SCANNING
      # ──────────────────────────────────
      - name: Run OWASP Dependency Check
        run: mvn dependency-check:check
        # Scans for known vulnerabilities in dependencies
      
      - name: Run Snyk
        uses: snyk/actions/maven@master
        env:
          SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}
        # Detects security issues in code and dependencies
      
      # ──────────────────────────────────
      # 4. TESTS (Must pass)
      # ──────────────────────────────────
      - name: Run unit tests
        run: mvn test
      
      - name: Run integration tests
        run: mvn verify -P integration-tests
      
      # ──────────────────────────────────
      # 5. CODE COVERAGE (Enforce minimum)
      # ──────────────────────────────────
      - name: Check code coverage
        run: mvn jacoco:check
        # Fails if coverage below threshold (e.g., 80%)
      
      - name: Upload coverage to Codecov
        uses: codecov/codecov-action@v3
        with:
          files: ./target/site/jacoco/jacoco.xml
      
      # ──────────────────────────────────
      # 6. BUILD (Must succeed)
      # ──────────────────────────────────
      - name: Build
        run: mvn clean package -DskipTests
        # Ensures the code compiles
      
      # ──────────────────────────────────
      # 7. QUALITY GATES (SonarQube)
      # ──────────────────────────────────
      - name: SonarQube analysis
        run: |
          mvn sonar:sonar \
            -Dsonar.projectKey=order-service \
            -Dsonar.host.url=${{ secrets.SONAR_HOST_URL }} \
            -Dsonar.login=${{ secrets.SONAR_TOKEN }}
        # Fails if quality gate not met:
        #   - Code smells > threshold
        #   - Duplications > threshold
        #   - Coverage < threshold
      
      # ──────────────────────────────────
      # 8. ARCHITECTURE RULES (ArchUnit)
      # ──────────────────────────────────
      - name: Verify architecture
        run: mvn test -Dtest=ArchitectureTest
        # Fails if architecture rules violated:
        #   - Circular dependencies
        #   - Wrong layer access
        #   - Package structure violations

  # Only notify for human review AFTER all checks pass
  notify-reviewers:
    needs: automated-checks
    runs-on: ubuntu-latest
    steps:
      - name: Request review
        run: echo "All automated checks passed. Ready for human review."
```

## 4. Review Checklist

### A Systematic Approach

```markdown
# Code Review Checklist

## ✅ FUNCTIONALITY
- [ ] Does this solve the stated problem?
- [ ] Are all acceptance criteria met?
- [ ] Have edge cases been considered?
- [ ] Are error conditions handled properly?
- [ ] Does this introduce any regressions?

## ✅ CORRECTNESS
- [ ] Is the logic sound?
- [ ] Are calculations correct?
- [ ] Are there off-by-one errors?
- [ ] Are null/empty cases handled?
- [ ] Are there race conditions or concurrency issues?

## ✅ ERROR HANDLING
- [ ] Are exceptions caught at the right level?
- [ ] Are error messages helpful?
- [ ] Are resources cleaned up properly (try-with-resources)?
- [ ] Are errors logged appropriately?
- [ ] Does error handling maintain system invariants?

## ✅ SECURITY
- [ ] Is user input validated?
- [ ] Are SQL queries parameterized (no concatenation)?
- [ ] Are authentication/authorization checks present?
- [ ] Is sensitive data (passwords, tokens) handled securely?
- [ ] Are cryptographic operations using approved algorithms?
- [ ] Is there any XSS, CSRF, or injection risk?

## ✅ PERFORMANCE
- [ ] Are there N+1 query problems?
- [ ] Are expensive operations necessary?
- [ ] Should anything be cached?
- [ ] Are indexes present on queried columns?
- [ ] Are large collections handled efficiently?
- [ ] Are connections/resources properly pooled?

## ✅ TESTS
- [ ] Are there tests for the new functionality?
- [ ] Do tests cover edge cases?
- [ ] Are tests clear and maintainable?
- [ ] Do tests run quickly?
- [ ] Are integration tests needed?
- [ ] Is the code testable (dependencies injected)?

## ✅ MAINTAINABILITY
- [ ] Are names meaningful?
- [ ] Are functions small and focused?
- [ ] Is the code self-documenting?
- [ ] Are comments necessary and helpful (WHY not WHAT)?
- [ ] Is complexity justified?
- [ ] Can I understand this in 30 seconds?

## ✅ ARCHITECTURE
- [ ] Does this fit the existing architecture?
- [ ] Are layers properly separated?
- [ ] Are dependencies appropriate?
- [ ] Is this in the right module/package?
- [ ] Does this introduce tight coupling?
- [ ] Are abstractions at the right level?

## ✅ BACKWARDS COMPATIBILITY
- [ ] Will this break existing clients?
- [ ] Are database migrations backwards-compatible?
- [ ] Are API changes versioned?
- [ ] Is deprecated functionality handled gracefully?
- [ ] Are feature flags used for risky changes?

## ✅ DOCUMENTATION
- [ ] Is there Javadoc on public APIs?
- [ ] Is there a clear commit message?
- [ ] Is the PR description helpful?
- [ ] Are architecture changes documented (ADR)?
- [ ] Is the README updated if needed?

## ✅ OPERATIONAL CONCERNS
- [ ] Are there sufficient logs for debugging?
- [ ] Are metrics/monitoring added?
- [ ] Is there a rollback plan for risky changes?
- [ ] Are configuration changes documented?
- [ ] Are alerts configured for failures?

## 🎯 Priority Order

**Must check (always):**
1. Functionality
2. Correctness
3. Security
4. Tests

**Should check (usually):**
5. Error handling
6. Performance
7. Maintainability

**Can check (when relevant):**
8. Architecture
9. Backwards compatibility

## 5. Small, Focused Changes

### The Size-Quality Curve

```

Research (Microsoft, Google):
- PRs under 200 lines: ~70% of issues found
- PRs 200-400 lines:   ~40% of issues found
- PRs over 400 lines:  ~10% of issues found

Optimal: 50-250 lines per PR
Maximum effective: 400 lines
"LGTM" threshold: 800+ lines (nobody reads carefully)

## 6. Review Turnaround Time

### The Cost of Delayed Reviews

```
THE CONTEXT-SWITCHING TAX

Developer writes code     Day 1 ─────────●
                                          │
                                          │ Context fully loaded in brain
                                          │
Review submitted          Day 1 ─────────●
                                          ↓
                                          Waiting...
                                          (Developer switches to other task)
                                          ↓
Review feedback arrives   Day 3 ─────────●
                                          ↓
                                          Developer must:
                                          1. Stop current work
                                          2. Context switch back (15-30 min)
                                          3. Re-read the code
                                          4. Remember why they did it that way
                                          5. Address feedback
                                          6. Context switch back to other task
                                          ↓
Updated PR submitted      Day 3 ─────────●
                                          ↓
                                          Waiting again...
                                          ↓
Second review            Day 5 ─────────●

Total calendar time: 5 days
Total context switches: 4 (expensive!)
Developer frustration: High
Actual work time: 2 hours spread over 5 days


vs.


FAST REVIEW CYCLE

Developer writes code     Day 1 ─────────●
Review submitted          Day 1 ─────────● (2 PM)
Review feedback arrives   Day 1 ─────────● (3 PM)
  (Context still fresh!)
Updated PR submitted      Day 1 ─────────● (4 PM)
Approved and merged       Day 1 ─────────● (4:30 PM)

Total calendar time: 4 hours
Total context switches: 0 (stays in flow)
Developer frustration: Low
Feature shipped: Same day

Productivity improvement: ~10x
```

## Team Agreement (Example)

### Priority Levels

- Response time: Review by next morning

**⚪ LOW (< 48 hours)**

### Reviewer Responsibilities

  - Morning (9-10 AM)
  - After lunch (1-2 PM)
  - End of day (4-5 PM)

  - Comment immediately: "I see this. Will review by 3 PM."
  - Or: Reassign to someone else who can

- If a PR is too large to review:
  - Comment immediately: "This is too large. Can you split it?"
  - Don't let it sit for days

### Author Responsibilities

- Keep PRs small (< 400 lines)
- Label priority clearly
- Provide context in description
- Respond to feedback within same SLA
- Ping reviewer if blocked (don't wait silently)

✓ Automate what machines can check (style, formatting, common bugs)
✓ Review for correctness, security, and maintainability (what machines can't check)
✓ Keep PRs small (< 400 lines) — large PRs don't get careful review
✓ Review quickly (< 24 hours) — context switching kills productivity
✓ Be kind and specific — you're reviewing code, not people
✓ Distinguish blocking from optional feedback
✓ Praise good work — reinforce what you want to see more of

✗ Don't nitpick style when automation exists
✗ Don't review 1000-line PRs carefully (nobody does — split them)
✗ Don't block on subjective preferences
✗ Don't leave vague feedback ("this is unclear")
