# Clean Code Principles

## Overview

Clean code is code that reads like well-written prose — another developer can understand it quickly, trust its intent, and modify it confidently. This guide covers seven foundational principles: Meaningful Names (the single most impactful act in programming — names should reveal intent, names that require comments have failed, the name IS the documentation), Small Functions (do one thing, do it well — short enough to understand in a glance, extracted functions are reusable and independently testable, the name of the function documents what it does), Comments vs Self-Documenting Code (comments are a failure of expression — if you need a comment to explain WHAT the code does, rename or refactor first, comments are valuable only for WHY, good code eliminates 90% of comments), Error Handling Clarity (errors are not afterthoughts — handle them at the right level, fail fast with meaningful messages, never swallow exceptions, separate error handling from happy-path logic), Dependency Management (control what your code depends on — depend on abstractions not concretions, inject dependencies, keep coupling low and graphs shallow), SOLID in Practice (five principles that guide object design — Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion, each solves a specific class of maintainability problem), and The Boy Scout Rule (leave the code better than you found it — every time you touch a file improve it slightly, incremental improvement compounds into significant quality). The key insight: clean code is not written — it is refactored into existence. Nobody writes it on the first pass. The discipline is in the second pass: re-reading, renaming, restructuring, simplifying. Code is read far more often than written, so optimise for the reader.

**The Philosophy:**

```
"Clean code is code that was written by someone
 who cared." — Robert C. Martin

The ratio that matters:

  Reading code : Writing code  =  10 : 1

We spend 10x more time reading than writing.
Every decision should optimise for the reader.

The cost of dirty code:

  Day 1:  "We'll clean it up later."
  Week 2: "It's too tangled to change safely."
  Month 3: "We need to rewrite this module."
  Year 1:  "We're rewriting the whole system."

Clean code is not a luxury.
It is the single most important investment
in long-term development speed.
```

## 1. Meaningful Names

### The Most Powerful Refactoring Tool

```java
// ========================================
// THE RULE: Names should reveal intent.
// If a name requires a comment, it has failed.
// ========================================

// ❌ CRYPTIC: Every identifier is a question mark
public class Calc {
    public double calc(double x, double y, int t) {
        double r = x * y;
        if (t == 1) r *= 1.08;
        if (t == 2) r *= 1.10;
        return r;
    }
}

// ✅ INTENT-REVEALING: Reads like a sentence
public class PriceCalculator {
    public double calculateTotalWithTax(double basePrice,
                                        double quantity,
                                        TaxCategory category) {
        double subtotal = basePrice * quantity;
        return subtotal * category.getMultiplier();
    }
}

public enum TaxCategory {
    STANDARD(1.08),
    LUXURY(1.10),
    EXEMPT(1.00);

    private final double multiplier;
    TaxCategory(double multiplier) { this.multiplier = multiplier; }
    public double getMultiplier() { return multiplier; }
}
```

## 2. Small Functions/Methods

### One Thing, Done Well

```java
// ========================================
// THE RULE: Each function does ONE thing,
//           at ONE level of abstraction.
// ========================================

// ❌ DOES MULTIPLE THINGS: validates, hashes, persists, notifies, logs
public void registerUser(String email, String password, String name) {
    // --- Validation ---
    if (email == null || !email.contains("@")) {
        throw new ValidationException("Invalid email");
    }
    if (password == null || password.length() < 8) {
        throw new ValidationException("Password too short");
    }
    if (name == null || name.isBlank()) {
        throw new ValidationException("Name required");
    }

    // --- Hashing ---
    String hashedPassword = BCrypt.hashpw(password, BCrypt.gensalt(12));

    // --- Persistence ---
    User user = new User(email, hashedPassword, name);
    userRepository.save(user);

    // --- Notification ---
    String html = templateEngine.render("welcome", Map.of("name", name));
    emailClient.send(email, "Welcome!", html);

    // --- Logging ---
    logger.info("User registered: {} ({})", name, email);
}

// ✅ EACH METHOD DOES ONE THING: orchestrator reads like a story
public User registerUser(RegistrationRequest request) {
    validate(request);
    User user = buildUser(request);
    userRepository.save(user);
    welcomeNotifier.send(user);
    logger.info("User registered: {}", user.getEmail());
    return user;
}

private void validate(RegistrationRequest request) {
    if (!EmailValidator.isValid(request.getEmail())) {
        throw new ValidationException("Invalid email");
    }
    if (request.getPassword().length() < MIN_PASSWORD_LENGTH) {
        throw new ValidationException("Password too short");
    }
    if (request.getName() == null || request.getName().isBlank()) {
        throw new ValidationException("Name required");
    }
}

private User buildUser(RegistrationRequest request) {
    return new User(
        request.getEmail().trim().toLowerCase(),
        passwordEncoder.encode(request.getPassword()),
        request.getName()
    );
}
```

```java
// ========================================
// HOW TO DETECT "MORE THAN ONE THING"
// ========================================

/**
 * A method does more than one thing if:
 * 
 * 1. You need section comments to navigate it
 *    "// Validation", "// Persistence" — each section
 *    should be its own method.
 * 
 * 2. You cannot name it with a single verb phrase
 *    "validateAndProcessAndSaveOrder" = three things.
 * 
 * 3. You need to set up the entire system to test one part
 *    "I just want to test validation, but I need a DB..."
 * 
 * 4. Changing one part requires understanding the whole method
 * 
 * The test: Can you describe it in ONE sentence,
 * without the word "and"?
 *   "It validates the request."     ✅ One thing.
 *   "It validates AND saves."       ❌ Two things.
 */
```

```java
// ========================================
// ABSTRACTION LEVELS: Stay on one floor
// ========================================

/**
 * Level 4 — Business orchestration:  "place the order"
 * Level 3 — Domain operations:       "calculate the total"
 * Level 2 — Infrastructure calls:    "save to the database"
 * Level 1 — Mechanics:               "serialize to bytes"
 * 
 * A method at Level 4 should only call Level 3.
 * It should never reach down to Level 1 directly.
 */

// ❌ Mixes levels — orchestration + raw string ops + HTTP call
public Receipt checkout(Cart cart) {
    if (cart.isEmpty()) throw new EmptyCartException();        // Level 3

    byte[] serialized = ObjectOutputStream.serialize(cart);    // Level 1 — too low!
    HttpResponse resp = httpClient.post("/pay", serialized);   // Level 1 — too low!

    double total = cart.items().stream()                       // Level 3
        .mapToDouble(Item::getPrice).sum();

    return new Receipt(total, resp.getTransactionId());        // Level 3
}

// ✅ Each method stays on one level
public Receipt checkout(Cart cart) {
    validateCart(cart);                           // Level 3
    Money total = calculateTotal(cart);           // Level 3
    String transactionId = processPayment(total); // Level 3 — hides Level 1 inside
    return Receipt.create(cart, total, transactionId);
}
```

```java
// ========================================
// FLAG ARGUMENTS: A hidden split in the method
// ========================================

// ❌ Boolean flag means the method does two things
createUser("Alice", "alice@example.com", true);  // What does 'true' do??

public User createUser(String name, String email) {
    return repository.save(new User(name, email));

// Caller decides what happens next — explicit, readable
User user = userService.createUser("Alice", "alice@example.com");
notificationService.sendWelcome(user);
```

## 3. Comments vs Self-Documenting Code

### When to Write Words, When to Write Better Code

```java
// ========================================
// THE RULE: A comment is a failure to express
//           yourself in code.
//
//           Comments should explain WHY, not WHAT.
//           Code already says WHAT.
// ========================================

// ❌ NOISE: Every comment restates the code
public void processOrder(Order order) {
    // Get the customer
    Customer customer = customerRepo.findById(order.getCustomerId());

    // Check if the customer is active
    if (customer.isActive()) {
        // Calculate total price
        double total = calculateTotal(order.getItems());
        // Save the order
        orderRepo.save(order);
    }
}

// ✅ Remove the noise. The code IS the documentation.
public void processOrder(Order order) {
    Customer customer = customerRepo.findById(order.getCustomerId());
    if (customer.isActive()) {
        double total = calculateTotal(order.getItems());
        orderRepo.save(order);
    }
}
```

```java
// ========================================
// COMMENTS THAT ARE ACTUALLY RENAMES
// ========================================

// ❌ Comment exists because the name is broken
int d; // number of days since last login
// → Rename: int daysSinceLastLogin;

// ❌ Comment explains what a method does
// Checks if the user qualifies for a discount
public boolean check(User u) { /* ... */ }
// → Rename: public boolean isEligibleForDiscount(User user)

// ❌ Comment explains a block that should be a method
// Calculate shipping cost based on weight and zone
double shipping = 0;
if (weight > 50) shipping = 25.99;
else if (zone > 3) shipping = 15.99;
else shipping = 7.99;
// → Extract: double shipping = calculateShipping(weight, zone);

// ❌ Comment explains a complex conditional
if (user.getAge() >= 18 && user.hasValidId() && !user.isBlacklisted()) {
    // ...
}
// → Extract: if (isEligibleForService(user)) { ... }

/**
 * The pattern:
 *   You want to write a comment →
 *   Ask: "Can I make the code say this?" →
 *   If yes: rename or extract. Delete the comment.
 *   If no: THEN write the comment.
 */
```

```java
// ========================================
// COMMENTS THAT ARE GENUINELY VALUABLE
// ========================================

public class PaymentGateway {

    /**
     * ✅ WHY: Explains the reasoning behind a non-obvious choice
     *
     * Exponential backoff rather than fixed-interval retry because
     * the payment provider throttles at 100 req/s during peak hours.
     * Fixed retries would amplify the thundering-herd problem.
     */
    private static final long[] RETRY_DELAYS_MS = {1_000, 2_000, 4_000, 8_000};

    /**
     * ✅ WHY: Regulatory constraint that code cannot express
     *
     * PCI-DSS 3.4.1 requires card numbers be irreversibly masked
     * in all storage and logs. Only the last 4 digits may appear.
     */
    public String maskCardNumber(String cardNumber) {
        return "****-****-****-" + cardNumber.substring(cardNumber.length() - 4);
    }

    /**
     * ✅ WHY: Warning about a non-obvious gotcha
     *
     * The payment provider expects amounts in the SMALLEST currency
     * unit (cents for USD, pence for GBP). Sending dollars causes
     * silent under-charge. See incident #4821.
     */
    public void charge(String token, Money amount) {
        long smallestUnit = amount.toSmallestUnit();
        providerClient.charge(token, smallestUnit);
    }

    /**
     * ✅ WHY: Explains a temporary workaround with context
     *
     * TODO(PLAT-2847): Replace with async webhook once the vendor
     * supports it. This polling loop adds ~200ms latency. The webhook
     * API is in their beta — estimated stable in Q3.
     */
    public OrderStatus pollForStatus(String vendorOrderId) {
        return providerClient.getStatus(vendorOrderId);
    }
}
```

## 4. Error Handling Clarity

### Failures as First-Class Citizens

```java
// ========================================
// THE RULE: Error handling should not obscure
//           the main logic. Separate the two.
//           Fail fast. Be specific.
// ========================================

// ❌ ERROR HANDLING TANGLED WITH BUSINESS LOGIC
public Order placeOrder(OrderRequest request) {
    Order order = null;
    try {
        Customer customer = customerRepo.findById(request.getCustomerId());
        if (customer == null) {
            log.warn("Customer not found");
            return null;  // Silent failure — caller gets null, has no idea why
        }

        order = new Order(customer, request.getItems());

        try {
            paymentService.charge(customer, order.getTotal());
        } catch (PaymentException e) {
            log.error("Payment failed");
            order.setStatus(OrderStatus.FAILED);
            // Falls through... does it return? throw? Unclear.
        }

        orderRepo.save(order);
    } catch (Exception e) {
        log.error("Error", e);  // Catches everything. Masks real bugs.
    }
    return order;  // Could be null, could be FAILED — caller must guess
}

// ✅ HAPPY PATH IS CLEAR. ERRORS ARE DECISIVE.
public Order placeOrder(OrderRequest request) {
    Customer customer = findCustomerOrFail(request.getCustomerId());
    Order order = Order.create(customer, request.getItems());
    chargeOrFail(customer, order);
    orderRepo.save(order);
    return order;
}

private Customer findCustomerOrFail(String customerId) {
    return customerRepo.findById(customerId)
        .orElseThrow(() -> new CustomerNotFoundException(customerId));
}

private void chargeOrFail(Customer customer, Order order) {
    try {
        paymentService.charge(customer, order.getTotal());
    } catch (PaymentException e) {
        throw new OrderPaymentFailedException(order.getId(), e);
    }
}
```

```java
// ========================================
// CUSTOM EXCEPTION HIERARCHY
// ========================================

/**
 * A well-designed hierarchy lets callers catch
 * at exactly the right granularity.
 *
 *   ApplicationException
 *     ├── ValidationException         (bad input)
 *     ├── NotFoundException          (resource missing)
 *     │     ├── CustomerNotFoundException
 *     │     ├── OrderNotFoundException
 *     │     └── ProductNotFoundException
 *     ├── ConflictException           (state conflict)
 *     │     └── DuplicateEmailException
 *     ├── AuthenticationException     (identity failure)
 *     └── PermissionDeniedException   (authorisation failure)
 */

public class ApplicationException extends RuntimeException {
    private final String errorCode;

    public ApplicationException(String errorCode, String message) {
        super(message);
        this.errorCode = errorCode;
    }

    public ApplicationException(String errorCode, String message, Throwable cause) {
        super(message, cause);
        this.errorCode = errorCode;
    }

    public String getErrorCode() { return errorCode; }
}

public class NotFoundException extends ApplicationException {
    public NotFoundException(String resource, Object id) {
        super("NOT_FOUND", resource + " not found: " + id);
    }
}

public class CustomerNotFoundException extends NotFoundException {
    public CustomerNotFoundException(String customerId) {
        super("Customer", customerId);
    }
}

// Controller catches and maps to HTTP status codes:
@PostMapping("/orders")
public ResponseEntity<?> createOrder(@RequestBody OrderRequest request) {
    try {
        Order order = orderService.placeOrder(request);
        return ResponseEntity.status(HttpStatus.CREATED).body(order);

    } catch (ValidationException e) {
        return ResponseEntity.badRequest()
            .body(ErrorResponse.of(e.getErrorCode(), e.getMessage()));

    } catch (NotFoundException e) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND)
            .body(ErrorResponse.of(e.getErrorCode(), e.getMessage()));

    } catch (ConflictException e) {
        return ResponseEntity.status(HttpStatus.CONFLICT)
            .body(ErrorResponse.of(e.getErrorCode(), e.getMessage()));
    }
}
```

```java
public void processPayment(String cardToken, double amount) {

// ✅ Validate immediately — fail with context
public void processPayment(String cardToken, double amount) {

        throw new IllegalArgumentException("Card token must not be blank");
    }
    if (amount <= 0) {
        throw new IllegalArgumentException("Amount must be positive, got: " + amount);
    }
    if (amount > SINGLE_TRANSACTION_LIMIT) {
        throw new IllegalArgumentException("Amount exceeds limit of " + SINGLE_TRANSACTION_LIMIT);
```

```java
// ========================================
// NEVER SWALLOW EXCEPTIONS SILENTLY
// ========================================

public void sendEmail(String to, String subject) {
    try {
        emailClient.send(to, subject);

        emailClient.send(to, subject);

public void sendWelcomeEmail(User user) {
    try {
        emailClient.send(user.getEmail(), "Welcome!");
    } catch (EmailDeliveryException e) {
        // Welcome emails are best-effort: the account is already created.

        // The user can trigger a resend from Account Settings.
        log.warn("Welcome email not delivered to {}. Resend available in UI.", user.getEmail());
    }
}
```

## 5. Dependency Management

### Control What You Depend On

```java
// ========================================
// THE RULE: Depend on abstractions.
//           Inject dependencies — don't create them.
// ========================================

// ❌ HARD-WIRED: Creates its own dependencies
public class OrderService {
    private final OrderRepository repository     = new PostgresOrderRepository();
    private final EmailService   emailService   = new SmtpEmailService();
    private final PaymentService paymentService = new StripePaymentService();

    public void placeOrder(Order order) {
        repository.save(order);
        emailService.sendConfirmation(order);
        paymentService.charge(order);
    }
    // Cannot test without real Postgres, real SMTP, real Stripe.
}

// ✅ INJECTED: Receives abstractions via constructor
public class OrderService {
    private final OrderRepository repository;
    private final EmailService   emailService;
    private final PaymentService paymentService;

    public OrderService(OrderRepository repository,
                        EmailService emailService,
                        PaymentService paymentService) {
        this.repository     = repository;
        this.emailService   = emailService;
        this.paymentService = paymentService;
    }

    public void placeOrder(Order order) {
        repository.save(order);
        emailService.sendConfirmation(order);
        paymentService.charge(order);
    }
    // Test with fakes. Swap providers without editing this class.
```

```java
// ✅ The interface is what OrderService knows about
public interface OrderRepository {
    void save(Order order);
    Optional<Order> findById(String id);
    List<Order> findByCustomerId(String customerId);
}

// Production implementation
public class PostgresOrderRepository implements OrderRepository {
    @Override
    public void save(Order order) { /* Postgres logic */ }
    // ...
}

// Test implementation — fast, in-memory, no external dependency
public class InMemoryOrderRepository implements OrderRepository {
    private final Map<String, Order> store = new ConcurrentHashMap<>();

    @Override
    public void save(Order order) {
        store.put(order.getId(), order);
    }

    @Override
    public Optional<Order> findById(String id) {
        return Optional.ofNullable(store.get(id));
    }

    @Override
    public List<Order> findByCustomerId(String customerId) {
        return store.values().stream()
            .filter(o -> o.getCustomerId().equals(customerId))
            .toList();
    }
}
```

```java
// ❌ Adding a new payment type requires editing this class
public class PaymentProcessor {

        else if ("paypal".equals(p.getType()))      { processPayPal(p);     }
        else if ("bitcoin".equals(p.getType()))     { processBitcoin(p);    }
        // Adding Stripe = editing this file
    }
}

// ✅ New payment type = new class. Zero changes to existing code.
public interface PaymentStrategy {
    boolean supports(String type);
    void process(Payment payment);
}

public class CreditCardPayment implements PaymentStrategy {
    public boolean supports(String type) { return "credit_card".equals(type); }
    public void process(Payment p)       { /* credit card logic */ }
}

public class PayPalPayment implements PaymentStrategy {
    public boolean supports(String type) { return "paypal".equals(type); }
    public void process(Payment p)       { /* PayPal logic */ }
}

// Adding Stripe: write one new class. Nothing else changes.
public class StripePayment implements PaymentStrategy {
    public boolean supports(String type) { return "stripe".equals(type); }
    public void process(Payment p)       { /* Stripe logic */ }
}

public class PaymentProcessor {
    private final List<PaymentStrategy> strategies;  // Spring injects all

    public PaymentProcessor(List<PaymentStrategy> strategies) {
        this.strategies = strategies;

    public void process(Payment payment) {
        strategies.stream()
            .filter(s -> s.supports(payment.getType()))
            .findFirst()
            .orElseThrow(() -> new UnsupportedPaymentType(payment.getType()))
            .process(payment);
    }
```

```java
// ========================================
// L — LISKOV SUBSTITUTION PRINCIPLE
// A subtype must honour its parent's contract.
// ========================================

// ❌ Square BREAKS Rectangle's contract
public class Rectangle {
    protected double width, height;
    public void setWidth(double w)  { this.width  = w; }
    public void setHeight(double h) { this.height = h; }
    public double area()            { return width * height; }
}

public class Square extends Rectangle {
    @Override
    public void setWidth(double w) {
        this.width  = w;
        this.height = w;  // ← Surprise! Height changed too.
    }
    // Code that expects Rectangle breaks with Square:
    //   rect.setWidth(5); rect.setHeight(10); rect.area()
    //   Rectangle → 50. Square → 100. Contract violated.
}

// ✅ Neither extends the other. Both honour the Shape contract.
public abstract class Shape {
    public abstract double area();
}

public class Rectangle extends Shape {
    public Rectangle(double width, double height) { /* store */ }
    public double area() { return width * height; }
}

public class Square extends Shape {
    public Square(double side) { /* store */ }
    public double area()       { return side * side; }
}

// Both work correctly wherever Shape is expected:
public double totalArea(List<Shape> shapes) {
    return shapes.stream().mapToDouble(Shape::area).sum();
}
```

```java
// ========================================
// I — INTERFACE SEGREGATION PRINCIPLE
// Clients should not depend on methods they don't use.
// ========================================

// ❌ Fat interface forces unnecessary coupling
public interface UserRepository {
```

```java
// High-level modules depend on abstractions,
// not on low-level implementations.
// ========================================

// ❌ High-level depends on low-level concrete class
public class OrderService {

// ✅ High-level depends on abstraction (interface)
public interface OrderRepository { /* ... */ }

public class OrderService {
    private final OrderRepository repo;  // Abstraction

    public OrderService(OrderRepository repo) {
        this.repo = repo;
    }
    // Doesn't know, doesn't care: Postgres, MongoDB, in-memory, or mock.
}

/**
 * SOLID as a coherent whole:
 *
 *   SRP  → Each class has one reason to change
 *   OCP  → Extend via new classes, not by editing existing ones
 *   LSP  → Subtypes keep their parent's promises
 *   ISP  → Interfaces are small and focused
 *   DIP  → Depend on interfaces; inject implementations
 *
 * Together: low coupling, high cohesion, testability, extensibility.
 *
 * ⚠️  Do NOT over-abstract. Introduce an interface only when you have
 *      a concrete reason: a second implementation, a test double, or
 *      a visible extension point. Abstractions for the sake of
 *      abstraction add complexity without benefit.
 */
```

## 7. The Boy Scout Rule

### Leave It Better Than You Found It

```java
// ========================================
// THE RULE: "Always leave the campground
//            cleaner than you found it."
//
// In code:
//   Every time you touch a file, leave it
//   slightly better than it was.
// ========================================

// ─── BEFORE: What you found when opening the file ───
public class OrderService {

    // Cryptic name you notice while adding your feature
    public void proc(Order o) {
        // Dead code — OLD_FEATURE was removed 4 months ago
        if (System.getenv("OLD_FEATURE") != null) {
            legacyProcess(o);
            return;
        }

        // Stale comment — the formula changed last sprint
        // Calculate shipping using the old formula
        double shipping = o.getItems().stream()
            .mapToDouble(i -> i.getWeight() * 2.50)
            .sum();

        o.setShipping(shipping);
        repository.save(o);
    }

    // Unused method — dead code
    private void legacyProcess(Order o) { /* ... */ }
}

// ─── AFTER: What you leave behind ───
public class OrderService {

    private static final double SHIPPING_RATE_PER_LB = 2.50;

    // ✅ Renamed: proc → fulfillOrder, o → order
    public void fulfillOrder(Order order) {
        // ✅ Dead code block removed
        // ✅ Stale comment removed — the method name is self-documenting
        double shipping = calculateShipping(order);
        order.setShipping(shipping);
        repository.save(order);
    }

    // ✅ Extracted: shipping logic has a name and is reusable
    private double calculateShipping(Order order) {
        return order.getItems().stream()
            .mapToDouble(item -> item.getWeight() * SHIPPING_RATE_PER_LB)
            .sum();
    }

    // ✅ legacyProcess deleted — no longer referenced
}
```

```java
// ========================================
// THE BOY SCOUT CHECKLIST
// (Scan for these while you're in the file)
// ========================================

/**
 * □ Confusing variable or method name?
 *     → Rename it.
 *
 * □ Dead code? (Unused methods, unreachable branches, commented-out blocks?)
 *     → Delete it. Version control preserves history.
 *
 * □ Outdated comment?
 *     → Update or remove it.
 *
 * □ Magic number without a name?
 *     → Extract to a named constant.
 *
 * □ A method doing more than one thing?
 *     → Extract the secondary logic into its own method.
 *
 * □ Empty catch block?
 *     → Log, rethrow, or document why silence is intentional.
 *
 * □ Unused import?
 *     → Delete it.
 *
 * ─────────────────────────────────
 * SCOPE: Keep it small. 2–5 minutes max.
 * If a cleanup would take longer, open a separate ticket.
 * The rule is about incremental improvement, not heroics.
 * ─────────────────────────────────
 */
```

```java
// ========================================
// WHY IT WORKS: Compound Interest
// ========================================

/**
 * WITHOUT the Boy Scout Rule:
 *
 *   Every developer avoids the dirty code.
 *   Nobody fixes it because "it's not my job."
 *   The dirty code gets dirtier.
 *   Eventually: a rewrite sprint.

 *

 *
 *   Day 1:   Rename one variable.        (30 seconds)
```
